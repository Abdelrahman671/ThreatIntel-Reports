<h1 class="wp-block-heading">High level overview</h1>



<p>Clipsa is a multipurpose password stealer, written in Visual Basic, focusing on stealing cryptocurrencies, brute-forcing and stealing administrator credentials from unsecured WordPress websites, replacing crypto-addresses present in a clipboard, and mining cryptocurrencies on infected machines. Several versions of Clipsa also deploy an XMRig coinminer to make even more money from infected computers.</p>



<p>Clipsa spreads as a malicious executable file, likely disguised as codec pack installers for media players. Once on an infected device, Clipsa can perform multiple actions, such as searching for cryptowallet addresses present in victims’ clipboards to then replace the addresses victims want to send money to with wallet addresses owned by the bad actors behind Clipsa. Furthermore, Clipsa is capable of searching for and stealing <code>wallet.dat</code> files, and installing a cryptocurrency miner.</p>



<p>Additionally, Clipsa uses infected PCs to crawl the internet for vulnerable WordPress sites. Once it finds a vulnerable site, it attempts to brute-force its way into the site, sending the valid login credentials to Clipsa’s C&amp;C servers. While we cannot say for sure, we believe the bad actors behind Clipsa steal further data from the breached sites. We also suspect they use the infected sites as secondary C&amp;C servers to host download links for miners, or to upload and store stolen data.</p>



<h2 class="wp-block-heading">Campaign overview</h2>



<p>We estimate that the attack vector is most likely malicious codec pack installers for media players (<code>Ultra XVid Codec Pack.exe</code> or <code>Installer_x86-x64_89006.exe</code>). Users who try to install these codecs for their media players inadvertently download malicious installers instead of clean ones. Once users begin the installation process, they deploy Clipsa on their machines and the malware immediately starts its malicious behavior.</p>



<p> The campaign is most prevalent in India, where Avast has blocked more than 43,000 Clipsa infection attempts, protecting more than 28,000 users in India from the malware. We have also observed higher infection attempt rates in the Philippines, where Avast protected more than 15,000 users from Clipsa and in Brazil, protecting more than 13,000 users. In total, Avast protected more than 253,000 users more than 360,000 times, since August 1, 2018. We protect all our users against Clipsa and all of its components. </p>

<h1 class="wp-block-heading" id="analysis">Analysis</h1>



<p>Clipsa uses a single executable binary with several parameters (command line arguments). The parameters distinguish program phases which run as separate processes, simultaneously. Each phase focuses on a different functionality and is started by Clipsa’s initialization process, which does not have any parameters.</p>



<p>Clipsa uses these parameters for phases: <br> 1. <em>No parameters</em><br> 2. <code>--CLIPS</code><br> 3. <code>--CLIPSS</code><br> 4. <code>--WALLS</code><br> 5. <code>--PARSE</code><br> 6. <code>--BRUTE</code></p>



<p>Phases 2-4 are designed to steal data from users, focusing on crypto-wallet related data. Phases 5 and 6 focus on finding vulnerable WordPress websites and stealing their administrative credentials. We will focus on each of these phases in the rest of this analysis.</p>



<h2 class="wp-block-heading" id="parameterless-phase">The parameterless phase</h2>



<p>When the malware is run on an infected machine, the program intuitively starts without any parameters. This phase allows Clipsa to install and hide itself on the system. Then it continues by initializing other phases that perform malicious actions.</p>



<h3 class="wp-block-heading">Pre-installation</h3>



<p>In the pre-installation phase, Clipsa creates a message dialog box which makes it look like some kind of a setup process. However, this dialog box (see figure below) is actually just a disguise, so the user thinks the codec pack they downloaded is being installed. The truth is the dialog box only displays randomly generated numbers (incrementally summed) and prints the sum. Clipsa also adds a random sleep between increments, making the process look natural.</p>



<div class="wp-block-image"><figure class="aligncenter"><img decoding="async" src="https://lh5.googleusercontent.com/BniDt3p_-XAMargEsAu0knBw16faRT2Fa7dEc6E9QpNLNDq04aFJJ9_a8esJL7sxljKRxhbTbFea4KXTIw0lczlMAyKNbvAnG3BbNfQ1aUxJU1zCvUmxrdduO8kpHRPbSDf9d0MG" alt=""/><figcaption> <em>Figure illustrating the setup progress as a disguise</em> </figcaption></figure></div>



<p>After the sum 99% is reached, the process closes the dialog. During the imaginary setup process, the malware performs no malicious nor useful actions. We believe the purpose of this behavior is to delay the actual malicious process, thus avoiding detection in auto-sandboxing tools.</p>



<p>After the imaginary setup is finished, Clipsa checks whether Task Manager is running using Windows Management Instrumentation (WMI):<br><code> Select * from Win32_Process WHERE Name = 'taskmgr.exe'</code><br>And if it is running, terminates the program to avoid user detection.</p>



<h3 class="wp-block-heading">Installation</h3>



<p>Clipsa then copies itself to the <code>%APPDATA%\Roaming</code> directory. The specific folders and binaries are named depending on the version of Clipsa. One of the newer versions copies itself to:<br> <code>C:\Users\user\AppData\Roaming\AudioDG\condlg.exe<br> C:\Users\user\AppData\Roaming\AudioDG\zcondlg.exe </code></p>



<p>Some older versions were located in:<br><code> C:\Users\user\AppData\Roaming\WinSys\coresys.exe<br> C:\Users\user\AppData\Roaming\WinSys\xcoresys.exe</code></p>



<p>From now on, we will only consider the newer version which uses the <code>AudioDG</code> path, along with C&amp;C server <code>poly.ufxtools[.]com</code>. For further details about the other C&amp;C servers, see the <a href="#C&amp;C-servers">C&amp;C servers</a> section, below.</p>



<p>During the installation process, additional directories and files are created as well:<br><code> C:\Users\user\AppData\Roaming\AudioDG\log.dat<br> C:\Users\user\AppData\Roaming\AudioDG\obj\<br> C:\Users\user\AppData\Roaming\AudioDG\udb\</code></p>



<p>Additionally, the path to <code>condlg.exe</code> is added to registry autorun, assuring the malware’s persistence:<br><code>HCU\Software\Microsoft\Windows\CurrentVersion\Run\11f86284</code></p>



<p>The registry key name <code>11f86284</code> is created from the first four bytes of a SHA-256 hash, which was computed from the <code>H3c7K4c5:H3c7K4c5</code> hard coded string. Note that we will see the string <code>H3c7K4c5</code> many times during the analysis &#8211; mostly in the encryption functions.</p>



<p>Furthermore, a new process <code>condlg.exe</code> is created (without parameters), which serves as a dropper and also starts other malicious phases. This process is, however, started from the hidden folder AudioDG and this is exactly how Clipsa knows it is already installed on the system.</p>



<p>Last but not least, the initial Clipsa process is doomed to end. Even when the entire Clipsa installation process is successful, the malware displays an error message to the users, to make the users believe the codec installation failed, making them think nothing was installed:</p>



<div class="wp-block-image"><figure class="aligncenter"><img decoding="async" src="https://lh6.googleusercontent.com/fY-JIGMObvm8lFJJKX_Y7mWzghoZBNG40XPA6BhPhyjj-9ZC7sugEGpl-6TzCQfQo2HnINcV8DTltH0wiMZD2-PBjkSVKFFz_T35F9evDmGViJpiD5kllb_bS4ayp_IACOd4lSrI" alt=""/><figcaption> <em>Figure illustrating an invoked error message at the end of Clipsa’s successful installation</em> </figcaption></figure></div>



<h4 class="wp-block-heading" id="condlg-dll-file">The condlg.dll file</h4>



<p>During the installation process, Clipsa checks for the presence of an additional file in the directory from which the user executed the malware. This file is usually named <code>65923_VTS.vob</code> or <code>setup.dll</code>. However, it is neither a multimedia container nor a library. The file is an encrypted text file that is decrypted by Clipsa and it is saved to a new file:<br> <code>C:\Users\user\AppData\Roaming\AudioDG\condlg.dll</code> </p>



<p>The file holds several thousand Bitcoin addresses. As we will see later, this file will be used in <a href="#clips-phase">the CLIPS phase</a> during which crypto-wallet addresses are replaced in the clipboard. However, its presence is optional and Clipsa is fully functional without it.</p>



<p>Note that the names of the original files follow the assumption that Clipsa is served disguised as codec pack installers for media players.</p>



<p>The decryption process is very straightforward. Clipsa loads the file and XORs each byte with a current index in the byte array modulo 0xFF:</p>

<h3 class="wp-block-heading" id="coinmining">Coinmining</h3>



<p>Once Clipsa is successfully installed on the system, it uses the process <code>condlg.exe</code> to harvest information about the infected system (like OS version, serial number, username) and sends the information to the <code>poly.ufxtools[.]com</code> file repository. This step also serves as a heartbeat, identifying whether the server is still alive. The actual data takes on this form:<br> <code>xxxxxxxx|PING|OS version|0|0|0|0|0|0|0|0</code><br>where the first column is the first four bytes of a SHA-256 hash calculated from the drive’s (where the malware is located) serial number in the decadic form and username, creating a fingerprint for the user. The third column is an OS version in plaintext, followed by a sequence of numbers parsed from the <code>AudioDG\log.dat</code> file (implicitly zeroes). More information about the log file can be found at the end of our analysis in the <a href="#logging">logging</a> subsection.  </p>



<p>If&nbsp; <code>poly.ufxtools[.]com</code> is alive, the malware proceeds with downloading and executing an XMRig coinminer. The download is performed using this request:<br>  <code>poly.ufxtools[.]com/wp-content/plugins/WPSystem/dl.php?a=d</code><br>and the file is named as the first four bytes of a SHA-256 hash calculated from a randomly generated floating point number between 0-1 (rounded to 7 digits after the decimal point):<br><code> C:\Users\user\AppData\Local\Temp\xxxxxxxx.exe</code>  </p>



<p>This coinminer uses several layers of obfuscation. At first, its Base64-encoded overlay is decoded, creating a new PowerShell script. This script decodes a GZIP file from another Base64-encoded string. This archive is unpacked and a slightly obfuscated final script is created. This final script then decodes a few additional Base64 strings and one of them is the XMRig binary, which is executed afterwards.</p>



<h4 class="wp-block-heading">Files 65923_VTS.asx and setup.bin</h4>



<p>Last but not least, Clipsa checks whether the user’s localization matches any of the locales in this hard coded list:<br><code>ARE,AUS,AUT,BEL,BGR,CAN,CHE,CHN,CYP,CZE,DEU,DNK,ESP,EST,FIN,FRA,GBR,GEO,HKG,HUN,</code><br><code>IDN,IRL,ISR,ITA,JPN,KOR,LUX,LVA,MAC,MCO,MYS,NLD,NOR,NZL,PHL,POL,PRT,QAT,SAU,SGP,</code><br><code>SVK,SVN,SWE,SWZ,THA,TUR,TWN,USA </code></p>



<p>If the user’s locale does <strong>not</strong> match any of the locales in the hard coded list, Clipsa attempts to find two additional files in the directory from which the malware was started:<br> <code> 65923_VTS.asx<br>  setup.bin</code></p>



<p>If one of these files is present, Clipsa copies it into the local temporary folder:<br> <code>C:\Users\user\AppData\Local\Temp\xxxxxxxx.exe</code><br> where <code>xxxxxxxx</code> are four random bytes generated with the same random generator mentioned above in the <a href="#coinmining">coinmining</a> subsection. After the file is copied, it is executed as well. Note that only one of these files is expected to be found, because it is actually one file with two name variants. Clipsa doesn’t contain this file directly in its binary. It is, however, very likely that some installers do contain this file and it is dropped along with Clipsa. During our analysis, we didn’t encounter this file bundled in Clipsa installers. We did, however, discover it through other sources. </p>



<p>The file <code>65923_VTS.asx</code> is yet another coinminer. It mines the Monero cryptocurrency, sending the money to the address:<br><code>49Y3XrW9mPtAqVDmFjAWNXF5X8sEgTS23Sa6ZVvJwFEEMa5rG7Yt3zaDY2TKH1sfChjPkUqYpygyKNVy</code><br><code>hPguXU1f4WGFp2f</code><br>in a pool:<br> <code>pool.supportxmr[.]com</code></p>



<p>The miner uses several anti-debugging techniques, few obfuscations, and persistence actions. Some of the samples we found were inside a crypter, too.</p>



<p>To mention some of the coinminer properties, the coinminer is always copied and renamed into a new location:<br> <code>C:\Users\user\AppData\Roaming\Host\svchost.exe</code><br>immediately after execution (thus, the file <code>Temp\xxxxxxxx.exe</code> is truly temporary) and a new process is created from this location. This confuses the user to think the coinminer is a legitimate Windows process. Furthermore, it writes autorun entries into registry and it also registers itself into Task Scheduler with a somewhat ironic name, using the following command:<br> <code>cmd.exe /c SCHTASKS /Create /SC MINUTE /MO 2 /TN "Microsoft Malware Protection</code><br><code> Command Line Utility" /TR "C:\Users\user\AppData\Roaming\Host\svchost.exe"</code></p>



<p>The coinminer also uses multithreading, where one of the threads periodically checks the following list of active processes:<br><code> taskmgr.exe<br> procexp64.exe<br> procexp.exe<br> processhacker.exe<br> procmon.exe<br> wireshark.exe<br> vnc.exe<br> anvir.exe </code><br> and these opened windows:<br><code> Process Hacker [%s\%s]<br> Process Hacker [%s\%s]+ (Administrator)<br> Process Explorer - Sysinternals: www.sysinternals.com [%s\%s]</code><br> where the strings <code>%s</code> are respectively:<br> <code>%USERNAME%<br> %COMPUTERNAME%</code><br> If any of these windows or processes are found, the coinminer pauses its actions. This is done to persuade the user that nothing is wrong with their computer.</p>



<h3 class="wp-block-heading">The OK download</h3>



<p>After the coinminer is downloaded and executed, Clipsa downloads an additional file from the URL:<br> <code>poly.ufxtools[.]com/wp-content/plugins/WPSystem/ok.php</code><br> to a path:<br>  <code>C:\Users\user\AppData\Local\Temp\xxxxxxxx.log</code><br> However, during our analysis, this URL was returning 0 bytes and we could not verify the purpose of the file because of this.</p>



<h4 class="wp-block-heading">The tsk.dat file</h4>



<p>After a heartbeat with information about the user’s system is returned, Clipsa performs a request:<br>  <code>poly.ufxtools[.]com/wp-content/plugins/WPSystem/dl.php?a=i</code><br> During our analysis, this request returned a string <code>ogirejsorg584erg4sgef</code> which represents a “task” from the C&amp;C server.</p>



<p>The task is saved to a new file:<br> <code>C:\Users\user\AppData\Roaming\AudioDG\tsk.dat</code><br> This file contains information about the C&amp;C from which the request was received and also the task. Both values are hashed using SHA-256 (only the first four bytes are used) and prefixed by hard coded letters <code>P</code> and <code>T</code>:<br> <code>Pf66ef67b<br> Tf8ab7df4</code><br> Where <code>f66ef67b</code> is calculated from <code>http[:]//poly.ufxtools[.]com</code> (without safety brackets) and <code>f8ab7df4</code> is calculated from the task string <code>ogirejsorg584erg4sgef</code>.</p>



<h3 class="wp-block-heading">Initialization of phases</h3>



<p>At this stage, Clipsa performs the initialization of phases which are started one by one. For this purpose, the binary <code>zcondlg.exe</code> is used. Clipsa starts processes with these parameters, which run simultaneously:<br> <code>zcondlg.exe --CLIPS<br> zcondlg.exe --CLIPSS<br> zcondlg.exe --PARSE<br> zcondlg.exe --BRUTE<br> zcondlg.exe --WALLS </code></p>



<p>After this initialization, the process <code>condlg.exe</code> is not terminated. Instead, it creates a new BRUTE process every 10 seconds. These new processes are used as “brute-force workers” who take on new brute-force jobs and try to gain administrative login credentials of WordPress websites (see <a href="#brute-phase">the BRUTE phase</a> for details).</p>



<h4 class="wp-block-heading">The bool.scan file</h4>



<p>After <a href="#walls-phase">the WALLS phase</a> is started (parameter <code>--WALLS</code>), a file <code>bool.scan</code> is created:<br> <code>C:\Users\user\AppData\Roaming\AudioDG\bool.scan</code><br> This file is empty and serves as a check whether this phase was started or not. This prevents Clipsa from repeatedly scanning the entire disk for crypto-wallets.</p>



<h2 class="wp-block-heading" id="clips-phase">The CLIPS phase</h2>



<p>This phase is dedicated to modifying the user’s clipboard. The clipboard is continuously checked and validated if its contents match Bitcoin (BTC) or Ethereum (ETH) address formats. If an address in the correct format is found, Clipsa replaces the address with the most similar BTC address from a predefined list. This serves as an easy way of stealing money from users when they, for example, copy and paste their own crypto-wallet address to a friend, effectively misleading the user to send money elsewhere.</p>



<h3 class="wp-block-heading">Replacing wallet addresses</h3>



<p>Before the actual replacement process begins, Clipsa checks the existence of a file<br><code> C:\Users\user\AppData\Roaming\AudioDG\condlg.dll</code><br> This file, however, is a text file and we could see its decryption process in <a href="#condlg-dll-file">the condlg.dll file</a> subsection. If the file exists, the malware reads its content and compares it with the BTC wallet address below:<br><code><a rel="noreferrer noopener" aria-label=" (opens in a new tab)" href="https://www.blockchain.com/en/btc/address/111u5Bbmz7gmaf2NXVyciTjCfdfqejzWm" target="_blank">111u5Bbmz7gmaf2NXVyciTjCfdfqejzWm</a></code><br> This address is the first address in the <code>condlg.dll</code> file, thus Clipsa checks if the file contains valid content. If it does, Clipsa uses this huge list of addresses in the <a href="#selection-process">selection process</a>.<br> If a valid Ethereum address is found instead in the clipboard, it is exclusively replaced by:<br>  <code><a rel="noreferrer noopener" aria-label=" (opens in a new tab)" href="https://www.blockchain.com/en/eth/address/0x4966DB520B0680fC19df5d7774cA96F42E6aBD4F" target="_blank">0x4966DB520B0680fC19df5d7774cA96F42E6aBD4F</a></code><br> This means that no other ETH address is used and the <a href="#selection-process">selection process</a> is omitted.</p>



<p> During our analysis, the total amount of ETH received in this wallet was 55.059107 ETH (at the time of publishing this was ~12,632.76 USD). Note that since some funds could have come from other sources, the total amount does not necessarily reflect the actual amount of cryptocurrencies stolen by Clipsa.</p>



<p>If the file <code>condlg.dll</code> is not found or it has different contents, Clipsa decrypts two different lists of BTC wallets. These two lists are then concatenated, creating a list with 2,000 valid BTC addresses (see <a href="#app-btc-addresses-list-2000">A.1 BTC addresses list (2000)</a>). Details about received funds to the BTC addresses can be found in subsection <a href="#received-funds">received funds</a> at the end of this analysis.</p>



<p>From this list, the<strong> </strong>most similar<strong> </strong>address is selected and is used as a replacement if a valid BTC address is found in the user’s clipboard.</p>



<p>A complete list of available BTC addresses used by Clipsa can be found in <a href="#app-btc-addresses-list-complete">A.2 BTC addresses list (complete)</a>. </p>



<h4 class="wp-block-heading" id="selection-process">Selection process</h4>



<p>The selection process is designed in two stages. In the first stage, the algorithm selects a sublist from the BTC addresses, depending on whether the original address starts with a number, 1 or 3. All the other addresses from the list are omitted. Note that BTC addresses match this regular expression: <code>[13][a-km-zA-HJ-NP-Z1-9]{25,34}</code>.</p>



<p>In the second stage, a specific algorithm is used. It compares two BTC addresses (the original one and the selected one from the sublist), calculating a <strong>similarity index</strong> <em><code>SI</code></em> which is a distance between the two strings. The similarity index changes depending on the compared bytes of both strings where the outer bytes have a greater influence on the similarity index and the middle one the lowest. The bytes are compared from both sides of the addresses.</p>



<p>This calculation can be represented by the equation listed below:</p>



<div class="wp-block-image"><figure class="aligncenter is-resized"><img decoding="async" src="https://decoded.avast.io/wp-content/uploads/sites/2/2019/07/image-1.png" alt="" class="wp-image-746" width="140" height="70"/></figure></div>



<p>where <em><code>n</code></em> is a desired calculation length from the beginning of the string and from the end of the string, <em><code>i</code></em> is an index in the string and <em><code>v</code></em> is a value representing a match strength:<br> <em><code>v=0</code></em> when the bytes match<br> <em><code>v=1</code></em> when the bytes match case insensitive<br> <em><code>v=2</code></em> when the bytes don’t match<br> From this definition, we can see that the wallet with the lowest <code><em>SI</em></code> is selected from the sublist.    </p>



<p>Last but not least, note that Clipsa uses <em><code>n=2</code></em> as a parameter for calculating the <code><em>SI</em></code> value, effectively checking only the first two and last two bytes for similarity comparison. The selection process described above makes sense, because users are more likely to notice the change in the first and last <em><code>n</code></em> characters.</p>



<h3 class="wp-block-heading">Saving the result</h3>



<p>When the address in the clipboard is successfully replaced, Clipsa creates a new file:<br>  <code>C:\Users\user\AppData\Roaming\AudioDG\rep.dat</code><br> This file contains <a href="#string-encryption">encrypted</a> information about what address was replaced by which address. This information is represented by the string in the following format:<br> <code> pattern|original address|new address</code><br> where <code>pattern</code> is a randomly selected string: <code>1,2</code> or <code>2,1</code>. However, if the wallet address is the ETH address, <code>pattern</code> is set to a letter <code>E</code>. It is unclear why Clipsa chooses between the strings <code>1,2</code> and <code>2,1</code>. In our opinion, it would be sufficient to add, for example, a prefix <code>B</code> and omit the “obfuscation”.</p>



<p>In a later stage, the file <code>rep.dat</code> is then used in <a href="#clipss-phase">the CLIPSS phase</a> as a log data which is sent to the C&amp;C server.</p>



<h3 class="wp-block-heading" id="string-encryption">String encryption</h3>



<p>As mentioned above, the string saved in the <code>rep.dat</code> file is encrypted. Furthermore, almost all the strings found in the Clipsa binaries are encrypted as well. This encryption process is actually the same one the malware author used to obfuscate strings in Clipsa. Clipsa contains several types of custom encryption/decryption functions. This function, however, is the most prevalent.</p>



<p>A custom encryption function is designed to achieve the string encryption. To explain the process properly, let’s first look at the decryption function in Python code below:</p>



<figure class="wp-block-image"><img loading="lazy" decoding="async" width="288" height="277" src="https://decoded.avast.io/wp-content/uploads/sites/2/2019/08/image-4.png" alt="" class="wp-image-797"/></figure>



<p>The string <code>encrypted</code> is the input for this decryption function. Note that this string always contains only valid ASCII values.</p>



<p>The encryption function has, however, a bit of a different approach. It takes the to-be-encrypted plaintext and converts it to hexadecimal form. After that, it selects two random letters from the predefined alphabet:<br> <code>abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789</code><br>These two bytes (<code>xx</code>) are then appended to the string <code>H3c7K4c5xx</code> and Clipsa tries to <strong>decrypt</strong> it. Consider it as a single step of the decrypt function listed above. If the desired output is equal to the plaintext, the two bytes are considered valid and they are saved as a ciphertext. This process repeats, as long as all bytes of the ciphertext are generated and can be correctly decrypted to plaintext. Thus, the whole process of encryption is not deterministic, i.e. one plaintext can be encrypted to multiple ciphertexts.</p>



<p>For example, let’s take the replacement of the hard coded Ethereum address to the same address, represented as the string below:<br><code>E|0x4966DB520B0680fC19df5d7774cA96F42E6aBD4F|0x4966DB520B0680fC19df5d7774cA96F42</code><br><code>E6aBD4F</code></p>



<p>The encrypted output might look like this:<br><code>IZZNRwqRnwWUdb56NoutldDWpJkJl4s6OCWOQV9Ur5Brz1ctf3GUm0SDjq52BuoxPRY4PB9hGDmu5lVc</code><br><code>H6R0Ng0Qg4AFk2gT4xHSakkEtxrWXfBgnwnAllIruQlLGOJlt0LKLLIwgw3fjaSIng3losV8l4cfs6xq</code><br><code>WgUwmPEDN7oSGG3NZSmLDHqlLFsKthuTJqpQD8kZN7meKeafK5SD0bHK48yuFcWuElwZldwuV3S6RM7A</code><br><code>ngcWOXSfy2bmvS4Sj9015auLkziW2uBJqs5t4XFUZAfmrUH04hsyUyDPKMBHH6pQTPAK3aS6bmcqxf1A</code><br><code>9RTuLT5IWdlGS1oVq6QFfcUl3AKR</code></p>



<h2 class="wp-block-heading" id="clipss-phase">The CLIPSS phase</h2>



<p>The CLIPSS phase is closely entangled with the CLIPS phase. It continually searches for the <code>rep.dat</code> file. When it is found, Clipsa decrypts its contents (see <a href="#string-encryption">string encryption</a> for details) and parses the values. If the file is not found or has invalid (e.g. empty) content, the process sleeps for one second and then repeats.</p>



<p>After that, the malware creates a new string that contains five values separated by pipes:<br><code> fingerprint|REPL|type|original address|replaced address</code><br> Where <code>fingerprint</code> is created as the first four bytes of a SHA-256 hash calculated from the user’s serial number (in decimal form) and computer name, <code>type</code> is a “type” of the address (<code>E</code> for Ethereum or a random string <code>1,2</code> or <code>2,1</code> for BTC). The <code>original address</code> and the <code>replaced address</code> represent the user’s and attacker’s addresses respectively.</p>



<p>After this plaintext string is created, it is once again <a href="#string-encryption">encrypted</a> and Clipsa attempts to upload it to the <code>poly.ufxtools[.]com</code> C&amp;C server. After the upload is completed, the malware deletes the contents of the <code>rep.dat</code> file and repeats the entire process (i.e. waits for new content).</p>



<h2 class="wp-block-heading" id="walls-phase">The WALLS phase</h2>



<p>The WALLS phase is designed to find <code>wallet.dat</code> files on the disk (hence the name). Note that <code>wallet.dat</code> is a typical filename for cryptocurrency wallets. Stealing this file effectively results in stealing money from this wallet. Furthermore, Clipsa also finds all text files (<code>*.txt</code>) which contain <a href="https://github.com/bitcoin/bips/blob/master/bip-0039.mediawiki" target="_blank" rel="noreferrer noopener" aria-label=" (opens in a new tab)">bip-39</a> patterns. If any such files are present, they are encrypted by RC4 stream cipher and sent to the <code>poly.ufxtools[.]com</code> file repository.</p>



<p>However, not all filesystem directories are searched. The malware recursively scans the whole filesystem, excluding only a list of predefined locations (separated by pipes):<br><code>\AppData|\Boot|\PerfLogs|\Program Files|\Temporary|\AMD\|\Dell\|\HP\|\Intel\|</code><br><code>\McAfee\|\Norton\|\NortonInstaller\|\vim\|Chrome|Drivers|Ebook|lessons|Lyrics|</code><br><code>Microsoft|Sample Music|Sample Pictures|savedIndexNames|TypingMaster|Windows|Games|</code><br><code>HeroOnline|iTunes|FIFA|League|MineCraft|nDoors|SmileGate|Steam|TwelveSky|WarRock</code></p>



<h3 class="wp-block-heading" id="file-encryption-rc4">File encryption (RC4)</h3>



<p>As mentioned before, when the file of interest is found, Clipsa encrypts its contents. This is done by RC4 stream cipher.</p>



<p>First of all, Clipsa hashes a hard coded string <code>H3c7K4c5</code> using SHA-256 and uses the output as a key with a key length of 32 bytes:<br><code>4d7b290afaa14d86b4cf64fc5bcca8de99536196ee5a18f963d51f26d7956775</code><br> The rest of the cipher follows a typical RC4 implementation.</p>



<h3 class="wp-block-heading">Saving the result &#8211; wallet.dat</h3>



<p>After the content of the file is encrypted, two new files are created. The encrypted content is written into a file with the extension <code>.data.bin</code>. The second file contains a plaintext path to the file whose contents were stolen and its extension is <code>.path.bin</code>. Furthermore, both files have a name created from two hashes, separated by a dot. The first hash is created from the user’s fingerprint (see <a href="#clipss-phase">the CLIPSS phase</a> for details). The second hash is calculated from the absolute path to the stolen file &#8211; only the first two bytes are used:<br> <code>C:\Users\user\AppData\Roaming\AudioDG\xxxxxxxx.yyyy.data.bin<br> C:\Users\user\AppData\Roaming\AudioDG\xxxxxxxx.yyyy.path.bin</code><br> Clipsa then tries to upload these files to the <code>poly.ufxtools[.]com</code> C&amp;C server:<br><code>poly.ufxtools[.]com/wp-content/plugins/WPSecurity/up.php</code></p>



<h3 class="wp-block-heading">Saving the result &#8211; text files</h3>



<p>As previously mentioned, Clipsa also focuses on text files that contain words with specific patterns. These patterns form <a href="https://github.com/bitcoin/bips/blob/master/bip-0039.mediawiki" target="_blank" rel="noreferrer noopener" aria-label=" (opens in a new tab)">bip-39</a> mnemonic seed recovery phrases (or “mnemonic word sequences”) which are used as a seed for a pseudo-random generator. If a user knows the seed, they can deterministically generate the same wallet keys that were generated when the user first created their wallet. Thus, Clipsa in this phase actually focuses on stealing mnemonic word sequences to crack cryptocurrency wallets.</p>



<h4 class="wp-block-heading">Selecting the correct text files</h4>



<p>Even though each <code>wallet.dat</code> file is stolen, only text files with maximum length of 32,771 bytes are selected. Furthermore, after the contents of these text files are truncated and all redundant spaces are removed, the contents have to match specific patterns from the list below:<br> <code>#WWWWWWWWWWWW#<br> #WWWWWWWWWWWWN<br> NWWWWWWWWWWWW#<br> NWWWWWWWWWWWWN<br> #NWNWNWNWNWNWNWNWNWNWNWNW#<br> #NWNWNWNWNWNWNWNWNWNWNWNWN<br> NNWNWNWNWNWNWNWNWNWNWNWNW#<br> NNWNWNWNWNWNWNWNWNWNWNWNWN</code></p>



<p>These patterns are made up of letters. Each letter represents a word (string which is delimited by space (0x20)). There are three types of letters:<br>  <code>N</code> &#8211; Number (decadic)<br>  <code>W</code> &#8211; Word from <a href="https://github.com/bitcoin/bips/blob/master/bip-0039/english.txt" target="_blank" rel="noreferrer noopener" aria-label=" (opens in a new tab)">bip-39</a> word list<br>  <code>#</code> &#8211; Unknown word   </p>



<p>Thus, every word from the text file is tested to check whether the word is in the <a rel="noreferrer noopener" aria-label=" (opens in a new tab)" href="https://github.com/bitcoin/bips/blob/master/bip-0039/english.txt" target="_blank">bip-39</a> word list. This is done by matching every word from the file content against a hard coded encrypted word list (see <a href="#decryption-of-word-list">decryption of word list (bip-39)</a> below). If a word matches, the character <code>#</code>, <code>N</code>, or <code>W</code> is appended to a new pattern. After the new pattern is finished and if it matches any of the patterns listed above, the file is copied, its contents <a href="#file-encryption-rc4">encrypted</a>, and uploaded.</p>



<h4 class="wp-block-heading">Saving text files on disk</h4>



<p>Saving the selected text files is very similar to the stolen <code>wallet.dat</code> files. Instead of using the extension <code>.bin</code>, an extension <code>.txt</code> is used:<br> <code>C:\Users\user\AppData\Roaming\AudioDG\xxxxxxxx.yyyy.data.txt<br> C:\Users\user\AppData\Roaming\AudioDG\xxxxxxxx.yyyy.path.txt</code><br> Note that after the files are successfully uploaded, they are then deleted from this location.</p>



<h3 class="wp-block-heading" id="decryption-of-word-list">Decryption of word list (bip-39)</h3>



<p>The decryption process of the word list is very straightforward. The cipher is a simple substitution which uses a lookup table and a ciphertext.</p>



<p>For the word list, this lookup table is given:<br><code>  JyBhjP0OSI3qVQn!U4ZlvK5zsfDEdgTRx%XaF|br6YA1u87Li2mpkWNtecoGCwM9H</code><br> while using this alphabet:<br><code>  0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ|!%</code><br> Here is a decryption of the beginning of the ciphertext (truncated) to plaintext:<br><code>  3q3zQszM3qZKZgXM3qKnM3qsTgM3qsRnM<br>  abandon|ability|able|about|above|</code><br> For the whole word list in plaintext, see <a href="#app-word-list">B.1 Word list (bip-39)</a>.</p>



<p>Note that this list exactly matches the <a href="https://github.com/bitcoin/bips/blob/master/bip-0039/english.txt" target="_blank" rel="noreferrer noopener" aria-label=" (opens in a new tab)">bip-39 word list</a> for generating mnemonic phrases for crypto wallets.</p>



<h2 class="wp-block-heading" id="parse-phase">The PARSE phase</h2>



<p>In the PARSE phase, Clipsa decrypts a hard coded <a href="#decryption-of-word-list">word list</a>. From this list, it selects particular keywords which are used by Google and Bing search engines. Every site from the search engine results page is parsed and saved into separate files on the disk in obfuscated representation.</p>



<h3 class="wp-block-heading">Keywords selection</h3>



<p>Before the random selection of the keywords begins, Clipsa uses a hard coded string:<br> <code>w1|w2|w1 w2|w2 w1|w1 and w2|w1 at w2|w1 but w2|w1 else w2|w1 for w2|w1 if w2|</code><br><code>w1 in w2|w1 of w2|w1 or w2|w1 with w2</code><br> We can look at this string as a list of patterns (separated by pipes) which is used for searching in search engines. A random pattern from this string is selected. The words <code>w1</code> and <code>w2</code> are then replaced by randomly selected words from the <a href="#app-word-list">word list</a>. Thus, even though both <code>w1</code> and <code>w2</code> are selected, the malware can skip one of them (depending on the used pattern).</p>



<h3 class="wp-block-heading">Parsing</h3>



<p>When the keyword pattern is selected, it is inserted into Google search:<br><code>https://www.google.com/search?q=PATTERN&amp;start=0&amp;num=100</code><br>where <code>PATTERN</code> could, for example, be <code>abandon in above</code>.</p>



<p> If the Google search request is unsuccessful, the malware tries an additional request using Bing:<br><code>https://www.bing.com/search?q=PATTERN&amp;first=0&amp;count=50</code> </p>



<p>If a valid response is received, Clipsa parses all URLs present in all <em><code>&lt;cite&gt;</code></em> HTML tags and simplifies them by removing <em><code>http</code></em>, <em><code>https</code></em>, <em><code>&lt;strong&gt;</code></em> tags, and any text after the domain name. Furthermore, the malware hashes the URLs using SHA-256 and creates a new directory structure in the <code>AudioDG\udb\</code> folder. It takes the first two bytes from the calculated hash for a folder name and a file name (hexadecimal):<br><code>  AudioDG\udb\xx\xx.dat</code><br> and the file <code>xx.dat</code> contains additional two bytes of the hash. This way, Clipsa knows which URLs were parsed and it omits any duplicities found on the search result page.</p>



<p> All these unique URLs are then continually visited by new requests:<br><code>  http(s)://website.example/xmlrpc.php</code><br> If the <code>xmlrpc.php</code> file is accessible on the WordPress server, the page returns:<br><code>  XML-RPC server accepts POST requests only</code><br> and Clipsa tries to access<br><code>  http(s)://website.example/wp-login.php</code><br> as well, attempting to confirm that it is indeed a WordPress server with a login page.         </p>



<p>At this point, Clipsa attempts to get the WordPress site’s admin username. To achieve this, the malware executes a simple HTTP request for user enumeration (exploiting a weak configuration of the WordPress site):<br>  <code>http(s)://website.example/?author=1</code><br> which will often redirect the URL to:<br>  <code>http(s)://website.example/author/admin/</code><br> where <code>admin</code> is the first username on the WordPress server, which is usually the admin’s username (ID=1). Note that it usually matches <strong>admin</strong> literally, too…</p>



<p>However, Clipsa does not extract information from the new URL, but it instead parses the redirected site contents and searches for the following pattern:<br><code> href="http(s)://website.example/author/admin/feed/"</code><br> where the <code>admin</code> is extracted.</p>



<p>If Clipsa successfully retrieves the admin’s username, the whole PARSE phase is considered successful and the malware creates an additional file in the <code>AudioDG\obj\</code> subdirectory. The file is named using the first four bytes of the SHA-256 hash calculated from the URL address and it serves as a log file and a “brute-force job” for <a href="#brute-phase">the BRUTE phase</a>. The file contains four values separated by pipes and we will refer to it as the <code>URL file</code> for simplicity:<br><code>www.website.example|admin’s name|0|0</code><br>As we will see later in <a href="#brute-phase">the BRUTE phase</a>, the last two columns are actually the “number of brute-force attempts” and an “epoch time from the last brute-force attempt”, respectively.</p>



<h2 class="wp-block-heading" id="brute-phase">The BRUTE phase</h2>



<p>The BRUTE phase serves a single purpose: Brute-force its way into the administrative privileges of the WordPress website and send brute-forced credentials to the <code>poly.ufxtools[.]com</code> file repository.</p>



<p>However, it is strictly bound to <a href="#parse-phase">the PARSE phase</a>, because it uses the <code>URL file</code> created at the end of its last successful run. Note that a new process (<code>--BRUTE</code>) is created every 10 seconds, effectively parallelizing the brute-force process.</p>



<h3 class="wp-block-heading">Parsing the URL file and login credentials</h3>



<p>Clipsa selects a random <code>URL file</code> from the <code>\obj\</code> subdirectory. This file also has to have a timestamp (the last column in the file) lower than the present date. Clipsa then parses the file, retrieving the to-be-brute-forced URL and admin&#8217;s username.</p>



<p>After the <code>admin’s username</code> is retrieved, Clipsa uses three additional values as usernames:<code><br>  %domain%<br>  admin<br>  test</code><br>where <code>%domain%</code> is replaced with the current to-be-brute-forced domain. The rest of the strings are hard coded (and they match common usernames).</p>



<p>Moreover, the malware decrypts a list of frequently used passwords. It uses a simple substitution cipher (which actually uses the same lookup table as in <a href="#decryption-of-word-list">the WALLS phase</a>) to retrieve the list. For the full list of passwords in plaintext, see <a href="#app-passwords-list">C.1 Passwords list</a> in the appendix.</p>



<p>However, this password list is not complete. As we can see, the list starts with these two columns:<br><code>!|%domain%</code><br> The first column (exclamation mark) is replaced with the <code>admin’s username</code> and <code>%domain%</code> is replaced with the current to-be-brute-forced domain. The rest of the password list is kept as is.</p>



<h3 class="wp-block-heading">Brute-forcing</h3>



<p>Before the brute-forcing begins, Clipsa increases the penultimate number in the <code>URL file</code> by one (number of brute-force attempts) and replaces the last number by the current timestamp (from epoch), indicating a new brute-force attempt.</p>



<p>After all these preparations are finished, Clipsa begins to brute-force. First, it creates a XML-RPC request for the <code>xmlrpc.php</code> file:</p>



<p><code>&lt;methodCall&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&lt;methodName&gt;wp.getUsersBlogs&lt;/methodName&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&lt;params&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;param&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;value&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;string&gt;admin&lt;/string&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;/value&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;/param&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;param&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;value&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;string&gt;password&lt;/string&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;/value&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;/param&gt;<br> &nbsp;&nbsp;&nbsp;&nbsp;&lt;/params&gt;<br> &lt;/methodCall&gt;</code></p>



<p>where <code>admin</code> and <code>password</code> are filled in according to the selected username and the selected password (from the <a href="#app-passwords-list">list</a>).</p>



<p>If the attempt to retrieve the <code>UserBlogs</code> is successful, the response should contain a string <code>isAdmin</code>. If XML-RPC is enabled and poorly configured, an attacker can use the request above to obtain the response with the confirmation of the credentials. With the verbose response, Clipsa knows the credentials are valid and the brute-forcing is successful. If not, the next password is selected and Clipsa tries again.</p>



<p>When Clipsa successfully brute-forces its way into a WordPress admin account, it creates a string with all the information it obtained in the process:<br><code>  fingerprint|GOOD|www.website.example|admin’s username|password</code><br>This string is then <a href="#string-encryption">encrypted</a> and uploaded to the <code>poly.ufxtools[.]com</code> file repository.</p>



<h2 class="wp-block-heading" id="logging">Logging</h2>



<p>As we marginally mentioned in the beginning of the analysis, Clipsa creates and uses an additional file:<br><code> C:\Users\user\AppData\Roaming\AudioDG\log.dat</code><br>This file is used for logging purposes, which the malware author can use to debug Clipsa and obtain statistics.</p>



<p>The file contains eight columns, separated by pipes. Each column holds a different piece of information. At the beginning, when the file is created, it is empty, meaning all the columns are filled with zeros:<br><code>0|0|0|0|0|0|0|0</code><br>A curious observer could find out that the file is actually created with 10 columns (all zeros). However, these additional columns are removed after the first write into the log.</p>



<p>This file is then continually filled after each successful functionality iteration. Depending on the phase in which the functionality is located, the specific column is modified. In the table below, we present which phase affects which column(s):</p>



<table class="wp-block-table"><tbody><tr><td><strong>Phase name</strong></td><td><strong>Column in the log file</strong></td></tr><tr><td>PARSE</td><td>1, 2, 3, 6</td></tr><tr><td>BRUTE</td><td>4, 5, 6</td></tr><tr><td>CLIPS</td><td>7</td></tr><tr><td>WALLS</td><td>8</td></tr><tr><td>CLIPSS</td><td>None. Functionality of this phase is not logged.</td></tr></tbody></table>



<p>Now, let’s break down what these numbers mean, exactly, based on Clipsa phases, which we explained earlier:<br> 1. Number of parsed URLs<br> 2. Number of XML-RPC requests (a check whether the site is a WordPress site or not)<br> 3. Number of author requests (attempts to retrieve the admin’s username)<br> 4. Number of brute-force tries<br> 5. Number of successfully brute-forced sites (obtained administrative credentials)<br> 6. Current count of files in <code>\obj</code> directory<br> 7. Number of replaced crypto-wallet addresses<br> 8. Number of stolen <code>wallet.dat</code> files or <code>.txt</code> files with mnemonic phrases</p>



<p>Note that the contents of the <code>log.dat</code> file are sent to the C&amp;C server with every heartbeat,&nbsp; which is performed after Clipsa is started (and after the successful installation).</p>

<h1 class="wp-block-heading" id="C&amp;C-servers">C&amp;C servers</h1>



<p>During this analysis, we described only one C&amp;C server:<br><code> poly.ufxtools[.]com</code></p>



<p>This approach was to simplify the description of how Clipsa works. The selection of the C&amp;C addresses is actually done by reading multiple hard coded addresses from the memory,&nbsp; but of the several  addresses only one or very few of them is the C&amp;C server address. Each of these addresses is visited, and if the server responds correctly, Clipsa knows it is the C&amp;C. For example, the entire list of addresses in the analysed Clipsa sample is:<br><code> poly.ufxtools[.]com<br> industriatempo.com[.]br<br> robertholeon[.]com<br> deluxesingles[.]com<br> naijafacemodel[.]com<br> www.quanttum[.]trade<br> www.blinov-house[.]ru<br> ssgoldtravel[.]com<br> www.greenbrands[.]ir<br> new.datance[.]com</code></p>



<p>Furthermore, these sites are randomly permuted before every server interaction (e.g. an upload of stolen files). This is performed to obfuscate the network communication by randomness and non-malicious requests/responses.</p>



<p>Moreover, the list of addresses above is only an example. Nearly every Clipsa sample carries a different set of addresses that sometimes contain completely different C&amp;C addresses.</p>



<p>Without further ado, here is a full list of active C&amp;C servers that we encountered during our analysis:<br> <code>http[:]//besttipsfor[.]com<br> http[:]//chila[.]store<br> http[:]//globaleventscrc[.]com<br> http[:]//ionix.co[.]id<br> http[:]//mahmya[.]com<br> http[:]//mohanchandran[.]com<br> http[:]//mutolarahsap[.]com<br> http[:]//northkabbadi[.]com<br> http[:]//poly.ufxtools[.]com<br> http[:]//raiz[.]ec<br> http[:]//rhsgroup[.]ma<br> http[:]//robinhurtnamibia[.]com<br> http[:]//sloneczna10tka[.]pl<br> http[:]//stepinwatchcenter[.]se<br> http[:]//topfinsignals[.]com<br> http[:]//tripindiabycar[.]com<br> http[:]//videotroisquart[.]net<br> http[:]//wbbministries[.]org</code></p>

    let targetWindow = scriptOptions._targetWindow || 'prefer-popup',
        lastPopup = false;


    document.addEventListener('click', function (e) {
        if (e.target) {
            const buttonLinkElement = e.target.closest('a[data-plugin="nsl"][data-action="connect"]') || e.target.closest('a[data-plugin="nsl"][data-action="link"]');
            if (buttonLinkElement) {
                if (lastPopup && !lastPopup.closed) {
                    e.preventDefault();
                    lastPopup.focus();
                } else {

                    let href = buttonLinkElement.href,
                        success = false;
                    if (href.indexOf('?') !== -1) {
                        href += '&';
                    } else {
                        href += '?';
                    }

                    const redirectTo = buttonLinkElement.dataset.redirect;
                    if (redirectTo === 'current') {
                        href += 'redirect=' + encodeURIComponent(window.location.href) + '&';
                    } else if (redirectTo && redirectTo !== '') {
                        href += 'redirect=' + encodeURIComponent(redirectTo) + '&';
                    }

                    if (targetWindow !== 'prefer-same-window' && checkWebView()) {
                        targetWindow = 'prefer-same-window';
                    }

                    if (targetWindow === 'prefer-popup') {
                        lastPopup = NSLPopup(href + 'display=popup', 'nsl-social-connect', buttonLinkElement.dataset.popupwidth, buttonLinkElement.dataset.popupheight);
                        if (lastPopup) {
                            success = true;
                            e.preventDefault();
                        }
                    } else if (targetWindow === 'prefer-new-tab') {
                        const newTab = window.open(href + 'display=popup', '_blank');
                        if (newTab) {
                            if (window.focus) {
                                newTab.focus();
                            }
                            success = true;
                            window._nslHasOpenedPopup = true;
                            e.preventDefault();
                        }
                    }

                    if (!success) {
                        window.location = href;
                        e.preventDefault();
                    }
                }
            }
        }
    });

    let buttonCountChanged = false;

    const googleLoginButtons = document.querySelectorAll(' a[data-plugin="nsl"][data-provider="google"]');
    if (googleLoginButtons.length && checkWebView()) {
        googleLoginButtons.forEach(function (googleLoginButton) {
            if (scriptOptions._unsupportedWebviewBehavior === 'disable-button') {
                disableButtonInWebView(googleLoginButton);
            } else {
                googleLoginButton.remove();
                buttonCountChanged = true;
            }
        });
    }

    const facebookLoginButtons = document.querySelectorAll(' a[data-plugin="nsl"][data-provider="facebook"]');
    if (facebookLoginButtons.length && checkWebView() && /Android/.test(window.navigator.userAgent) && !isAllowedWebViewForUserAgent('facebook')) {
        facebookLoginButtons.forEach(function (facebookLoginButton) {
            if (scriptOptions._unsupportedWebviewBehavior === 'disable-button') {
                disableButtonInWebView(facebookLoginButton);
            } else {
                facebookLoginButton.remove();
                buttonCountChanged = true;
            }
        });
    }

    const separators = document.querySelectorAll('div.nsl-separator');
    if (buttonCountChanged && separators.length) {
        separators.forEach(function (separator) {
            const separatorParentNode = separator.parentNode;
            if (separatorParentNode) {
                const separatorButtonContainer = separatorParentNode.querySelector('div.nsl-container-buttons');
                if (separatorButtonContainer && !separatorButtonContainer.hasChildNodes()) {
                    separator.remove();
                }
            }
        })
    }
});
