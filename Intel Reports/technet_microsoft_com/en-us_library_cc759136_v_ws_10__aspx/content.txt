<h2 id="what-does-distributed-transaction-coordinator-do">What does Distributed Transaction Coordinator do?</h2>
<p>The Distributed Transaction Coordinator (DTC) service coordinates transactions that update two or more transaction-protected resources, such as databases, message queues, files systems, and so on. These transaction-protected resources may be on a single computer or distributed across many networked computers.</p>
<h2 id="who-does-this-feature-apply-to">Who does this feature apply to?</h2>
<ul>
<li><p>Users of any computers that participate in DTC transactions, either directly or through other computers.</p>
</li>
<li><p>System administrators of networks that use DTC components to perform transactions across networks.</p>
</li>
</ul>
<h2 id="what-new-functionality-is-added-to-this-feature-in-windows-server2003-service-pack1">What new functionality is added to this feature in Windows Server&nbsp;2003 Service Pack&nbsp;1?</h2>
<h3 id="securing-all-network-communication-by-default">Securing all network communication by default</h3>
<h4 id="detailed-description">Detailed description</h4>
<p>In Windows Server&nbsp;2003 Service Pack&nbsp;1, DTC provides the administrator with greater control over the network communication between computers. By default, all network communication is disabled.</p>
<p>In order to manipulate the communication settings, the DTC security settings properties page has been enhanced. To see the page, use the following procedure:</p>
<h4 id="to-open-the-dtc-security-settings-properties-page">To open the DTC security settings properties page</h4>
<ol>
<li><p>Open the <strong>Component Services</strong> snap-in Microsoft Management Console (MMC).</p>
</li>
<li><p>In the console tree, click the <strong>Computers</strong> folder.</p>
</li>
<li><p>In the results pane, right click <strong>My Computer</strong> and then click <strong>Properties</strong>.</p>
</li>
<li><p>Click the <strong>MSDTC</strong> tab, and then click <strong>Security Configuration</strong>.</p>
</li>
</ol>
<p>The table below defines the new fields in the property page, along with the registry keys affected for the different settings. All the registry keys related to MSDTC are located in the following registry key:</p>
<p><code>MyComputer\HKEY_LOCAL_MACHINE\Software\Microsoft\MSDTC</code></p>
<div class="WARNING">
<p>Warning</p>
<p>Incorrectly editing the registry may severely damage your system. Before making changes to the registry, you should back up any valued data on the computer. These registry keys might not be supported in future releases.</p>
</div>
<p>The following table tells you where to find the MSDTC key specific values.</p>
<table>
<colgroup>
<col>
<col>

<td><p>Determines whether DTC on the local computer is allowed to access the network. This setting must be enabled in combination with one of the other settings to enable network DTC transactions.</p>
<p>Default setting: Off</p></td>
<td><p>Security\NetworkDtcAccess</p>
<p>0 = Off</p>
<p>1 = On</p></td>
</tr>
<tr class="even">
<td><p>Allow inbound</p></td>
<td><p>Allows a distributed transaction that originates from a remote computer to run on this computer.</p>
<p>Default setting: Off</p></td>
<td><p>To enable this setting you must set the following registry key values to 1:</p>
<p><code>Security\NetworkDtcAccess</code></p>
<p><code>Security\NetworkDtcAccessTransactions</code></p>
<p><code>Security\NetworkDtcAccessInbound</code></p>
<p>To disable this setting, you only need to set the following registry key value to 0:</p>
<p><code>Security\NetworkDtcAccessInbound</code></p></td>
</tr>
<tr class="odd">
<td><p>Allow Outbound</p></td>
<td><p>Allows the local computer to initiate a transaction and run it on a remote computer.</p></td>
<td><p>To enable this setting, you need to set the following registry key values to 1:</p>
<p><code>Security\NetworkDtcAccess</code></p>
<p><code>Security\NetworkDtcAccessTransactions</code></p>
<p><code>Security\NetworkDtcAccessOutbound</code></p>
<p>To disable this setting, you only need to set the following registry key value to 0:</p>
<p><code>Security\NetworkDtcAccessOutbound</code></p></td>
</tr>
<tr class="even">
<td><p>Mutual Authentication Required</p></td>
<td><p>Adds support for mutual authentication in future versions and is the highest secured communication mode. In the current versions of Windows and Windows Server, it is functionally equivalent to the Incoming Caller Authentication Required setting. This is the recommended transaction mode for clients running Windows&nbsp;XP SP2 and servers running a member of the Windows Server&nbsp;2003 family.</p>

<tr class="odd">
<td>You cannot use the <strong>Mutual Authentication Required</strong> transaction mode with computers that are in a clustered environment, or any computers that are negotiating transactions with such computers. In that context, you can use the <strong>Incoming Caller Authentication Required</strong> transaction mode instead. In a clustered environment, the computer account for the Distributed Transaction Coordinator service specifies the cluster node's host name instead of the transaction node's host name, which prevents the authentication request from succeeding when the <strong>Mutual Authentication Required</strong> transaction mode is enabled.</td>

<tr class="odd">
<td><p>Incoming Caller Authentication Required</p></td>
<td><p>Requires the local DTC to communicate with a remote DTC using only encrypted messages and mutual authentication. This setting is recommended for servers running Windows Server&nbsp;2003 that are operating in a cluster.</p>
<p>Only Windows Server&nbsp;2003 and Windows&nbsp;XP SP2 support this feature, so you should only use this if you know that the DTC on the remote computer runs either the Windows Server&nbsp;2003 or Windows&nbsp;XP SP2 operating system.</p></td>
<td><p>AllowOnlySecureRpcCalls = 0</p>
<p>FallbackToUnsecureRPCIfNecessary = 1</p>
<p>TurnOffRpcSecurity = 0</p></td>
</tr>
<tr class="even">
<td><p>No Authentication Required</p></td>
<td><p>Provides system compatibility between previous versions of the Windows operating system. When enabled, communication on the network between DTCs can fall back to a non-authentication or non-encrypted communication if a secure communication channel cannot be established. This setting should be used if the DTC on the remote computer runs a Windows&nbsp;2000 operating system or a Windows&nbsp;XP operating system earlier than SP2. This setting is also useful when the DTCs that are involved are located on computers that are in domains that do not have an established trust relationship or if the computers are part of a Windows workgroup.</p></td>
<td><p>AllowOnlySecureRpcCalls = 0</p>
<p>FallbackToUnsecureRPCIfNecessary = 0</p>
<p>TurnOffRpcSecurity = 1</p></td>
</tr>
</tbody>
</table>
<h2 id="why-is-this-change-important-what-threats-does-it-help-mitigate">Why is this change important? What threats does it help mitigate?</h2>
<p>These changes are important in order to secure any communication coming into or going out from the computer. By default, after installing Windows Server&nbsp;2003 Service Pack&nbsp;1, the computer will not accept or issue any network traffic and therefore will be less vulnerable to network attacks.</p>
<p>Additionally, the online network protocol has been upgraded to support a more securely encrypted and mutually authenticated communication mode. This helps to ensure that attackers can not intercept or take over communications between DTCs.</p>
<h2 id="what-works-differently">What works differently?</h2>
<p>After installing Windows Server&nbsp;2003 Service Pack&nbsp;1, all network communication coming out of or getting into DTC is disabled. For example, if a COM+ object attempts to update a SQL database on a remote computer using a DTC transaction, the transaction fails. Conversely, if your computer is hosting a SQL database that components from remote computers try to access using a DTC transaction, their transactions fail.</p>
<h2 id="how-do-i-fix-these-issues">How do I fix these issues?</h2>
<p>If your transactions fail because of network connectivity, you can use MSDTC security properties, as described previously in this document, select the <strong>Network DTC Access</strong> check box, and then select the <strong>Allow Inbound</strong> and <strong>Allow Outbound</strong> check boxes, as appropriate.</p>
<p>If you want to change these setting programmatically as part of your Windows Server&nbsp;2003 Service Pack&nbsp;1 deployment, you can directly change the registry values that correspond to your desired setting as described in the table in âSecuring all network communication by default,â earlier in this document. After you have changed the registry settings, you must restart the MSDTC service.</p>
<p>If you are using Windows Firewall to protect the computers in your organization, you must add MSDTC into the exception list in the Windows Firewall settings. To do so, use the following steps:</p>
<ol>
<li><p>In <strong>Control Panel</strong>, open <strong>Windows Firewall</strong>.</p>
</li>
<li><p>Click the <strong>Exceptions</strong> tab, and then click <strong>Add Program</strong>.</p>
</li>
<li><p>Click <strong>Browse</strong>, and then add <strong>c:\windows\system32\msdtc.exe</strong>.</p>
</li>
<li><p>In <strong>Programs and Services</strong>, select the <strong>Msdtc.exe</strong> check box, and then click <strong>OK</strong>.</p>
</li>
</ol>
