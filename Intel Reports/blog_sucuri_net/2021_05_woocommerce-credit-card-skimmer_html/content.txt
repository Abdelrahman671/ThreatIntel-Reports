This is, of course, a tell-tale sign that there is something seriously wrong with the website and likely a case of credit card exfiltration.</p><p>Checking the source code of the website showed that there was clearly some dodgy javascript code on their checkout:</p><p><img decoding="async" class="aligncenter wp-image-27748 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image.png" alt="" width="1236" height="296" srcset="https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image.png 1236w, https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image-600x144.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image-650x156.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image-768x184.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image-390x93.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image-820x196.png 820w, https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image-1180x283.png 1180w, https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image-560x134.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/05/swiper-image-80x19.png 80w" sizes="(max-width: 1236px) 100vw, 1236px" /></p><h2>Google Tag Manager Scripts</h2><p>At first glance it appears to be a Google Tag Manager script (a popular service used on many websites). In the past we have seen how <a href="https://blog.sucuri.net/2018/04/malicious-activities-google-tag-manager.html">Google Tag Manager’s script can be used to hide malicious content</a>. Could this be the same?</p><p>Attackers commonly place code in a way that makes it look legitimate and innocent. Sometimes they even abuse GTM itself to surreptitiously exfiltrate credit card details, making it impossible to identify without using traffic inspection software or confirming which GTM scripts belong and which ones the site owner doesn&#8217;t recognise.</p><p>A regular googletagmanager script call does not include any obfuscation, base64 encoded content or concatenation. However, we see a little bit of base64 encoded content here that decodes to the following:</p><p><img loading="lazy" decoding="async" class="aligncenter wp-image-27712 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/05/a42b.png" alt="Google Analytics Snippet" width="453" height="26" srcset="https://blog.sucuri.net/wp-content/uploads/2021/05/a42b.png 453w, https://blog.sucuri.net/wp-content/uploads/2021/05/a42b-390x22.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/05/a42b-80x5.png 80w" sizes="auto, (max-width: 453px) 100vw, 453px" /></p><p>The domain a42[.]buzz has been <a href="https://labs.sucuri.net/blacklist/info/?domain=a42.buzz">blocklisted</a> by us since January so this clearly indicates that this is malicious in nature and we’ll need to find how it&#8217;s loading.</p><h2>How to Find the Malware</h2><p>In the above example our initial scans did not pick up anything. Remember: it&#8217;s the attacker&#8217;s job to evade detection so they are writing new malware all the time. In this case, querying the files and database for &#8220;Google Tag Manager&#8221; or &#8220;atob(&#8221; (atob is the javascript instruction to decode base64 encoded strings and is common on javascript injections) also returned nothing.</p><h2>Checking Recently Modified Files</h2><p>One of the first things that we will do in such a compromise is to check for recently modified files. Sometimes it can take a surprisingly long time for a compromise to be identified so I will typically start with files modified in the last few months. If nothing is found then I will go as far back as six months or a year.</p><p>You might wonder how someone would check all those files! It seems overwhelming at first but you can remove a lot of noise by parsing out all of the obvious plugin and theme updates. They will appear in batches with a very similar (if not identical) last modified date. It&#8217;s not a perfect method but proves to be useful in some cases. What you are left with is (hopefully) a small handful of files that were modified at very different times and dates.</p><p>Attackers will often spoof the last modified date on a file to make it appear as though it hasn&#8217;t been touched in years but these files should still show up when running an SSH command such as <i>mtime.</i> If you are not familiar with the SSH shell you can also spot check recently modified files through FTP with a program like Filezilla but this is significantly more cumbersome, not as reliable, and you will be relying on those last modified timestamps that I just mentioned. Attacks can sometimes even go unnoticed for multiple years before the website owner is made aware that their environment is compromised.</p><p>Even after checking many recently modified files we were left empty handed.</p><h2>Checking the Database</h2><p>So where was this hiding? It turned out to be hiding in plain sight in the database. Base64 encoding is one of the most popular encoding techniques that attackers will employ to hide their payloads, and that is exactly what they were doing here:</p><p><img loading="lazy" decoding="async" class="aligncenter wp-image-27750 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image3.png" alt="" width="957" height="446" srcset="https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image3.png 957w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image3-600x280.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image3-650x303.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image3-768x358.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image3-390x182.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image3-820x382.png 820w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image3-560x261.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image3-80x37.png 80w" sizes="auto, (max-width: 957px) 100vw, 957px" /></p><p>It&#8217;s unclear whether or not this was a deliberate attempt by the attackers to evade detection or if this is simply the way that the plugin/theme stores its data (resulting in a happy accident for the attackers). In any event it created some additional steps for us to locate the malware.</p><h2>Encoding Base64</h2><p>The way that we were able to find it eventually (big ups to @liamsmith86 for the assist here) was by base64 encoding &lt;!&#8211; Google Tag Manager &#8211;&gt; and querying some of the strings. The full encoded string is:</p><pre>PCEtLSBHb29nbGUgVGFnIE1hbmFnZXIgLS0+</pre><p>And we queried the database for a few snippets of that until we found it lodged on the checkout page.</p><p>Let&#8217;s break down this malware shall we?</p><h2>Analysis of a Woocommerce Credit Card Swiper</h2><p>Step one of course is to decode the base64 chunk, resulting in this (what we saw loading on checkout earlier):</p><p><img loading="lazy" decoding="async" class="aligncenter wp-image-27749 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image2.png" alt="" width="797" height="108" srcset="https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image2.png 797w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image2-600x81.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image2-650x88.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image2-768x104.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image2-390x53.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image2-560x76.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/06/swiper-image2-80x11.png 80w" sizes="auto, (max-width: 797px) 100vw, 797px" /></p><p>Taken at face value it appears to be GTM but of course this is bogus.
