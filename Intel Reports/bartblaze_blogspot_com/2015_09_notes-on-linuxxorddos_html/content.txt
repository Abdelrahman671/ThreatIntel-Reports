In this post we'll be focusing on a certain kind of malware: <b>Linux/Xor.DDoS</b> (also known as DDoS.XOR or Xorddos). As usual, we'll break the post down in several points:<br />
<br />
<a href="http://bartblaze.blogspot.com/2015/09/notes-on-linuxxorddos.html#Background">Background</a><br />
<a href="http://bartblaze.blogspot.com/2015/09/notes-on-linuxxorddos.html#Diagnosis">Diagnosis</a><br />

<a href="http://bartblaze.blogspot.com/2015/09/notes-on-linuxxorddos.html#Conclusion">Conclusion </a><br />
<br />
The variant discussed in this blog post is an older variant, so certain infection mechanisms may have changed, as well as C&amp;C's. The point of this post is to familiarize yourself with ELF malware in a better way - how to diagnose, analyse, remove and finally prevent malware from infecting your Linux machines. A lot of malware is going around and it's not (all) exclusively for Windows machines.<br />
<br />
<br />
<br />
<span class="" id="Background" pple-style-span="" style="font-size: large;"><b>Background</b></span>
<br />
You may have heard about Linux/Xor.DDoS already, a Linux Trojan with rootkit capabilities (belonging to the category of 'ELF malware'). What exactly is an ELF file? According to Wikipedia:<br />
<br />
<blockquote class="tr_bq">
<i>In computing, the Executable and Linkable Format (ELF, formerly called Extensible Linking Format) is a common standard file format for executables, object code, shared libraries, and core dumps.</i><br />
<a href="https://en.wikipedia.org/wiki/Executable_and_Linkable_Format" target="_blank">Source</a></blockquote>
In other words: ELF is to Linux as PE (.exe, .com, .scr, ...) is to Windows and Mach-O to OS X.<br />

If you haven't heard about Linux/Xor.DDoS itself already, be sure to read the initial post by MalwareMustDie uncovering this malware:<br />
<a href="http://blog.malwaremustdie.org/2014/09/mmd-0028-2014-fuzzy-reversing-new-china.html" target="_blank">Fuzzy reversing a new China ELF "Linux/XOR.DDoS" </a><br />
<br />
In short: Xor.DDoS is a multi-platform, polymorphic malware for Linux OS and its ultimate goal is to DDoS other machines. The name Xor.DDoS stems from the heavy usage of <a href="https://en.wikipedia.org/wiki/XOR_cipher" target="_blank">XOR encryption</a> in both malware and network communication to the C&amp;Cs (command and control servers).<br />
<br />
There have been other write-ups about this malware as well, which will be mentioned throughout this article or referenced in the <a href="http://bartblaze.blogspot.com/2015/09/notes-on-linuxxorddos.html#Resources">Resources </a>section.<br />
<br />
<br />
<br />
<span class="" id="Diagnosis" pple-style-span="" style="font-size: large;"><b>Diagnosis</b></span>
<br />
How do you know you're infected with Xor.DDoS?<br />
<br />
First and foremost (and obviously), you'll be conducting DDoS attacks from your machine(s) to targets chosen by the malware authors.<br />

<span class="" id="Analysis" pple-style-span="" style="font-size: large;"><b>Analysis</b></span>
<br />
First off, we have to identify <u>how</u> the malware entered the system. Usually, a weak root password is used (like <i>admin</i> or <i>123456</i>, see <a href="http://blogs.cisco.com/wp-content/uploads/SSHPsycho_Passwords.txt" target="_blank">here</a> for a list of tried passwords. <i><span style="font-size: x-small;">Note: huge .txt file!</span></i>) or the attackers are brute forcing their way in. (brute forcing the SSH credentials of the root user) Another, but less common possibility, is exploiting a vulnerable service that you have running (Apache for example).<br />
<br />

(SHA1: <b>b34b6f0ec42a0153c043b0665ec47bf6e5aac894</b>)<br />

This gives us more information already, for example; the <a href="https://en.wikipedia.org/wiki/List_of_file_signatures" target="_blank">magic</a> (<b>7F 45 4C 46</b> for ELF files, 4D 5A for MZ files)&nbsp; <a href="https://en.wikipedia.org/wiki/Two%27s_complement" target="_blank">2's complement</a>, <a href="https://en.wikipedia.org/wiki/Endianness" target="_blank">little endian</a>,&nbsp; the exact type of the file (an executable; other types for ELF files may be a relocatable file, a shared object, a core file or processor specific) but most importantly here being the <i>Entry point address</i>, or the start of the program.<br />
<br />
In regards to readelf, using parameter <b>-a </b>we can dump a ton of information, you can find the output of this command on our malware on Pastebin: <a href="http://pastebin.com/Vw5bst48">Xor.DDoS - "readelf -a" output</a><br />
<br />
Note that VirusTotal has added (since <a href="http://blog.virustotal.com/2014/11/virustotal-detailed-elf-information.html" target="_blank">November 2014</a>) detailed ELF information in reports as well, which is more or less similar to readelf's output. <br />
<br />

<br />
<br />
<br />
<span class="" id="Disinfection" pple-style-span="" style="font-size: large;"><b>Disinfection</b></span>
<br />
Most importantly, you'd of course like to remove/disinfect this malware completely. Some pointers:<br />
<br />
<ul>
<li><b>Identify malicious processes</b>: run <b>ps ef </b>(<a href="http://linux.die.net/man/1/ps" target="_blank">ps</a> stands for process status) to see which processes are running. Alternatively, you can use <a href="http://linux.die.net/man/1/top" target="_blank"><b>top</b></a> or again ps with other parameters, for example <b>ps ej</b> or <b>ps aux</b> for a more complete, human readable table. Look for processes with random names; in our example it started with <i>S90 </i>and random letters afterwards, linked to files with all random names, as is the case in our example malware named <i>bmtsfnlgxu</i>.<br /><br />Once you've identified the malicious process(es), you can use the following command to find related files as well: <b>for pid in $(ps -C <i><malware></malware></i> -o pid=); do ls -la /proc/$pid/fd; done</b><br />Where <i><malware></malware></i> is the name of the suspicious process. This command will display any open, related files.<b> </b>For example, for <i>bmtsfnlgxu</i> it would be: <br /><b>for pid in $(ps -C </b><b><i><i>bmtsfnlgxu </i></i> -o pid=); do ls -la /proc/$pid/fd; done</b></li>
</ul>
<br />
<br />
<ul>
<li><b>Identify malicious files</b>: look for newly created files in <b>/etc/init.d/</b>,<b> /boot/</b> and <b>/usr/bin/</b>. Again, look for files with random names. You may also use the command<b> ls -lat | head</b> to view recently changed files.<br /><br />Check your crontab (<b>/etc/crontab</b>). Delete the malicious cron jobs, more specifically the cron.hourly jobs and in the case of Xor.DDoS they will be the following:<br /><br /><i>*/3 * * * * root /etc/cron.hourly/cron.sh<br />*/3 * * * * root /etc/cron.hourly/udev.sh<br /> </i><br />Delete these two lines from your crontab. Don't forget to save. Delete the related files, located in <b>/etc/cron.hourly</b>. In our case, their content was as follows:<i><b><br /><br />cron.sh</b></i></li>
</ul>
<div class="separator" style="clear: both; text-align: center;">
<a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEhWcS74dgO67GlLKTLXTL70ltcwaJ9sq8z4k392HOT73IBgUAMhtWcVuwMIzTLt-JX6nwkcFV5f8iYJ-6s3Jv7Scy2AidSCoMR9U7Jil6XnuWOl5JBpCDOcLgs_99ML4pkWC7AJnp0DImLo/s1600/cron.sh.PNG" imageanchor="1" style="clear: left; float: left; margin-bottom: 1em; margin-right: 1em;"><img border="0" height="50" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEhWcS74dgO67GlLKTLXTL70ltcwaJ9sq8z4k392HOT73IBgUAMhtWcVuwMIzTLt-JX6nwkcFV5f8iYJ-6s3Jv7Scy2AidSCoMR9U7Jil6XnuWOl5JBpCDOcLgs_99ML4pkWC7AJnp0DImLo/s400/cron.sh.PNG" width="400" /></a><i><b><br /></b></i></div>

</div>

<br />
<br />
<br />
<br />
As said earlier, delete these files manually, as well as the file(s) mentioned in the scripts. (in this case: <b>/lib/libgcc.so.bak</b>, <b>/lib/libgcc.so</b> and <b>/lib/libgcc4.4.so</b>.) Note that these files are <u>not</u> related to <a href="http://gcc.gnu.org/onlinedocs/gccint/Libgcc.html" target="_blank">GCC's runtime library</a> and thus can be safely deleted. It's just another way how the malware tries to hide itself.<br />
<br />
Also double-check there are no malicious files or scripts in <b>/etc/rc.d</b>. If so, remove them as well.<br />
<br />
<br />
<ul>
<li><b>Stop and kill malicious processes</b>: identify the parent process; usually it will be the one consuming the most CPU (which you can verify using any of the earlier commands, <i>top</i> being the easiest). Firstly, be sure to stop the parent process and wait for the child processes to die. Use the command: <b>kill -STOP $pid</b><br /><br />When the child processes are dead, kill the parent by using: <b>kill -9 $pid<br />Note: </b>in case you see any other malicious processes, go through the last 2 commands again.<b> <br /><br /> </b></li>
<li><b>Delete any leftover malicious files</b>: locations where the malware may reside have been indicated before, but to be complete: <br /><br />
<blockquote class="tr_bq">
<b>/ </b><span style="font-size: x-small;">(<i>root</i> directory, in rare cases)</span><br />
<b>/bin/</b><br />
<b>/boot/</b><br />
<b>/etc/init.d/</b><b> </b><br />
<b>/etc/rc.d</b><br />
<b>/etc/rcX.d </b><span style="font-size: x-small;">(where X is a number)</span><br />
<b>/lib/</b><br />
<b>/lib/udev/</b><br />
<b>/sbin/</b><br />
<b>/tmp/</b><br />
<b>/usr/bin/</b></blockquote>
</li>
</ul>
<br />
That's it. Some additional tips and tricks:&nbsp;<b> </b><br />
<ul>
<li>Use <b><a href="http://linux.die.net/man/1/rm" target="_blank">rm</a> -rf</b> to permanently remove a file. Be careful with this command.</li>
<li>Having troubles removing a file? Are you root? If not, try killing a process or deleting a file using root by prepending <b>sudo </b>before your command. For example: <i>sudo kill -STOP $pid</i></li>
<li>Malicious process keeps coming back? Go over the steps again, but this time note down where the malware resides. Make that directory and its files unmodifiable by making use of the <a href="http://linux.die.net/man/1/chattr" target="_blank">chattr </a>command. For example, malware is being recreated in <i>/usr/bin/</i>. Use the command: <b>chattr -R +i /usr/bin/ </b>Then, stop the parent, wait for the children to die and kill the parent. Remove the files. Don't forget to use chattr again after you cleaned the infection. (in our example: chattr -R -i /usr/bin/)<br /><br />It's also possible the malware is temporarily storing files into <b>/tmp/</b> while you are trying to kill its processes. When that happens, use the same chattr command on the /tmp/ directory and start over. If you are in doubt, use that chattr command on all aforementioned directories and start over. <u><b>Very important</b></u>: do not forget to use chattr -R -i on them afterwards!</li>
<li>In rare cases, the attacker may still be connected to your box. If possible, cut the internet connection and go over the disinfection steps. If this is not possible, firstly stop SSH by entering the command:<br /><b>sudo /etc/init.d/ssh stop</b><br /><br />Then, use <a href="http://linux.die.net/man/8/iptables" target="_blank">iptables </a>to drop any connection to the IPs the malware is connecting to (use netstat for example, see also Diagnosis) and to drop any connection from the attacker or cybercriminal. How to do this:<br /><br />In our example, we learned that our C&amp;C's were 103.25.9.228 and 103.25.9.229. Thus, type or copy/paste these 2 commands:<br /><b>iptables -A OUTPUT -d 103.25.9.228 -j DROP<br />iptables -A OUTPUT -d 103.25.9.229 -j DROP</b><br /><br />To block connection(s) from the attacker (you can find the attacker's IP using <b>netstat </b>for example):<br /><b>iptables -A INPUT -s $attackerIP -j DROP</b><br /><br />Don't forget to save your freshly created iptables rules by using the command <br /><b>/etc/init.d/iptables save</b><br /><br />Afterwards, change all passwords. (SSH, your user, root) </li>
</ul>
<br />
Best case scenario here is obviously:<br />
<br />
<ul>
<li>restoring from a backup&nbsp;</li>
<li>if the machine is virtual, restore to a previous snapshot</li>
</ul>
When you have either of these available, don't forget to change all passwords afterwards to prevent re-infection - and patch your machine(s)!<br />
<br />
Some Xor.DDoS variants may also incorporate a rootkit. In that case, hope you have a "best case scenario" available to you. Once a box is fully compromised, it may be hard to reinstate it back to normal or its original state.<br />
<br />
For double-checking for rootkits and other malware, you may want to check out <a href="http://www.chkrootkit.org/" target="_blank">chkrootkit</a>&nbsp;or alternatively, <a href="https://sourceforge.net/projects/rkhunter/" target="_blank">rkhunter</a>. Additionally, you may download and install an antivirus, for example <a href="http://www.clamav.net/index.html" target="_blank">ClamAV</a>.<br />
<br />
If you perform manual clean-up as indicated above and have confirmed all is in order again, you can install ClamAV and perform an extra scan to be sure. Better be safe than sorry. Then, follow the prevention tips below to stay safe.<br />
<br />
<br />
<br />
<span class="" id="Prevention" pple-style-span="" style="font-size: large;"><b>Prevention</b></span>
<br />
<ul>
<li>Use strong passwords for SSH or use keys instead of passwords for authentication. You can read how to do that <a href="http://www.makeuseof.com/tag/how-to-authenticate-over-ssh-with-keys-instead-of-passwords/" target="_blank">here</a>. In the unlikely event of you not needing SSH to a particular machine, disable it on that machine by:<b><br />sudo apt-get remove openssh-server</b><br /><br />To disable it from starting up you can use:<b><br />update-rc.d -f ssh remove</b></li>
<li>Don't open the incoming SSH port (default 22) to ANY, but rather restrict it to trusted IP addresses.</li>
<li>For more information about safely using SSH, see: <a href="https://blog.0xbadc0de.be/archives/300" target="_blank">SSH: Best practices</a></li>
<li>Use a strong firewall. In Linux there are many options, iptables is a solid choice. A good basic iptables howto can be found <a href="https://help.ubuntu.com/community/IptablesHowTo" target="_blank">here</a>. In a network or if you need to protect several machines, you may want to consider a seperate hardware appliance as your firewall/UTM/... of choice.</li>
<li>Iptables can do a very decent job once properly configured. In case you want to do less manual work, I advise to check out <a href="http://www.sshguard.net/" target="_blank">sshguard </a>or <a href="https://github.com/trustedsec/artillery/" target="_blank">artillery</a>. Both can monitor and alert you when something funky happens. In the context of our blog post, it also looks for &amp; protects against SSH bruteforce attempts. Another application to consider is <a href="https://github.com/fail2ban/fail2ban" target="_blank">fail2ban</a>. An additional tool is <a href="https://www.snort.org/" target="_blank">snort</a>. For more information about these tools, refer to their pages.</li>
<li>Consider using <a href="http://linux.die.net/man/8/selinux" target="_blank">SELinux</a>. <b>S</b>ecurity-<b>E</b>nhanced Linux is a compulsory access control security mechanism provided in the kernel.<br /><br /> </li>
<li>Consider locking down cron jobs to only certain users. To deny all users from using cron you can use:<br /><b>echo ALL &gt;&gt;/etc/cron.deny</b></li>
<li>Consider&nbsp;disabling remote root login. Read how to do that <a href="http://www.howtogeek.com/howto/linux/security-tip-disable-root-ssh-login-on-linux/" target="_blank">here</a>.</li>
<li>If you browse a lot, consider using <a href="https://noscript.net/" target="_blank">NoScript </a>as well.</li>
<li>Keep your software and applications up-to-date, as on any system. </li>
<li>Consider installing an antivirus as second opinion or at least as an additional layer. This is not a necessity but may come in handy. I recommend <a href="http://www.clamav.net/index.html" target="_blank">ClamAV</a>.</li>
<li>Don't forget to protect other appliances that may be running on *nix systems, for example your router (and nowadays, <a href="https://en.wikipedia.org/wiki/Internet_of_Things" target="_blank">IoT</a> devices). Upgrade the firmware as soon as possible and change the default root/admin password(s). Install updates/patches for your particular firewall/UTM/... as well.</li>
<li>For even more (general) tips on hardening your Linux system (not against Xor.DDoS in particular):<a href="http://www.cyberciti.biz/tips/linux-security.html" target="_blank"><br />20 Linux Server Hardening Security Tips</a></li>
</ul>

<span class="" id="Conclusion" pple-style-span="" style="font-size: large;"><b>Conclusion</b></span>
<br />
Don't be fooled: Linux malware very much exists and starts to become more prevalent. Other prevalent Linux malware nowadays is:<br />
<br />
<ul>
<li>Every ELF malware made by the ChinaZ actor or group (Linux/ChinaZ.DDoS, Linux/Kluh, ...)</li>
<li>Linux/Aes.DDoS (Dofloo, MrBlack)</li>
<li>Linux/Bash0day (Shellshock, Bashdoor)</li>
<li>Linux/BillGates (Gates.B) </li>
<li>Linux/Elknot (DnsAmp)</li>
<li>Linux/GoARM (Ramgo, Goram)</li>
<li>Linux/IptabLes and Linux/IptabLex </li>
</ul>
<br />

Note that this list is not complete and new ELF malware may pop up every day. (it's not a question of <b>if</b>, but <b>when</b> it will pop up) You can find a list of (interesting) Linux malware <a href="http://www.kernelmode.info/forum/viewtopic.php?f=16&amp;t=3471" target="_blank">here</a>.<br />
