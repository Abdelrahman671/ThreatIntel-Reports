<h1>Big Game Hunting with Ryuk: Another Lucrative Targeted Ransomware</h1>
<div class="publish_info">
<p>January 10, 2019</p>
<span>|</span>
<a href="/en-us/blog/author.alexander-hanel/" title="Posts by alexander.hanel" rel="author">
alexander.hanel</a>
<span>|</span>
<a href="/en-us/blog/category.counter-adversary-operations/" title="Counter Adversary Operations">Counter Adversary Operations</a>
</div>
<div class="post_image">
<img width="1060" height="698" src="https://assets.crowdstrike.com/is/image/crowdstrikeinc/Wizard-spider-1" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt decoding="async" fetchpriority="high" srcset="/content/dam/crowdstrike/www/en-us/wp/2019/01/Wizard-spider-1.jpg 530w,https://assets.crowdstrike.com/is/image/crowdstrikeinc/Wizard-spider-1-300x198 300w" sizes="(max-width: 1060px) 100vw, 1060px" />

</div>
<div style="display: none;">
</div>

<div class="text text--blog-content aem-GridColumn aem-GridColumn--default--12">
<div class="cmp-text" id="text-c34805490b" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
<span><p><span style="	font-weight: 400;
"><b>WIZARD SPIDER is a sophisticated eCrime group that has been operating the Ryuk ransomware since August 2018</b>, targeting large organizations for a high-ransom return. This methodology, known as “<a href="https://www.crowdstrike.com/cybersecurity-101/cyber-big-game-hunting/">big game hunting</a>,” signals a shift in operations for WIZARD SPIDER. This actor is a Russia-based criminal group known for the operation of the <a href="https://www.crowdstrike.com/cybersecurity-101/malware/trickbot/">TrickBot</a> banking malware that had focused primarily on wire fraud in the past.</span></p>
<p> </p>
<p><span style="	font-weight: 400;
">The actor name GRIM SPIDER was introduced into CrowdStrike’s nomenclature in September 2018 for the group that operates the Ryuk ransomware as a distinct sub-group of the WIZARD SPIDER criminal enterprise. However, in June 2019, further evidence emerged that allowed CrowdStrike to assess with high confidence that Ryuk is in fact operated as part of the core WIZARD SPIDER actor group.</span> <span style="	font-weight: 400;
">CrowdStrike Intelligence will now solely use the actor name WIZARD SPIDER in association with TrickBot and Ryuk. The GRIM SPIDER actor name has been deprecated.</span> <span style="	font-weight: 400;
">Similar to </span><i><span style="	font-weight: 400;
">Samas</span></i><span style="	font-weight: 400;
"> and </span><i><span style="	font-weight: 400;
">BitPaymer</span></i><span style="	font-weight: 400;
">, <b>Ryuk is specifically used to target enterprise environments.</b> Code comparison between versions of Ryuk and </span><i><span style="	font-weight: 400;
">Hermes</span></i><span style="	font-weight: 400;
"> ransomware indicates that Ryuk was derived from the Hermes source code and has been under steady development since its release. Hermes is commodity <a href="/en-us/cybersecurity-101/ransomware/">ransomware</a> that has been observed for sale on forums and used by multiple threat actors. However, Ryuk is only used by WIZARD SPIDER and, unlike Hermes, Ryuk has only been used to target enterprise environments. Since Ryuk’s appearance in August, the threat actors operating it have <b>netted over 705.80 BTC across 52 transactions for a total current value of $3,701,893.98 USD.</b><p> </p>

<h2>2. How Ryuk Ransomware is Distributed</h2>

">CrowdStrike® has conducted multiple <a href="/epp-101/incident-response-ir-plan.html">incident response (IR) engagements</a> responding to Ryuk infections in which TrickBot was also identified on hosts in the victim environment. <a href="/en-us/endpoint-security-products/falcon-x-threat-intelligence/">CrowdStrike Falcon® Intelligence™</a></span><a href="/en-us/endpoint-security-products/falcon-x-threat-intelligence/"><span style="	font-weight: 400;
">®</span></a><span style="	font-weight: 400;
"> believes that the initial compromise is performed through TrickBot, which is typically distributed either via spam email or, through the use of the Emotet (developed and operated by </span><a href="/en-us/blog/meet-crowdstrikes-adversary-of-the-month-for-february-mummy-spider/"><span style="	font-weight: 400;

">) geo-based download function. Falcon</span> <span style="	font-weight: 400;

">Some of TrickBot’s modules (such as </span><code><span style="	font-weight: 400;
">pwgrab</span></code><span style="	font-weight: 400;

<ul><li style="	font-weight: 400;

"><span style="	font-weight: 400;
">PowerShell anti-logging scripts are executed on the host.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Reconnaissance of the network is conducted using standard Windows command line tools along with external uploaded tools.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Lateral movement throughout the network is enabled using Remote Desktop Protocol (RDP).</span></li><li style="	font-weight: 400;

<h2>3. From Hermes to Ryuk: Similarities &amp; Differences</h2>
<p><span style="	font-weight: 400;
"><b>Hermes ransomware, the predecessor to Ryuk, was first distributed in February 2017.</b> Only one month after its release, a decryptor was written for Hermes, followed by the release of version 2.0 in April 2017, which fixed vulnerabilities in its cryptographic implementation. Since this release, the only way for a victim to recover files is with the private encryption key, which is obtained by paying the ransom. In late August 2017, Hermes version 2.1 was released.</span> <span style="	font-weight: 400;
">Hermes was originally </span><a href="https://twitter.com/Damian1338B/status/1028940508117258240"><span style="	font-weight: 400;
">sold on forums for $300 USD</span></a><span style="	font-weight: 400;
">. When purchased, the buyer received a build that supported two email addresses, a decryptor and a unique RSA key pair. If the purchaser desired more email addresses, they were required to purchase another build for an additional $50. The seller of Hermes ransomware appears to have stopped or limited advertising on forums in 2017.</span> <span style="	font-weight: 400;
">Early versions of Hermes were reportedly installed via internet-accessible RDP servers protected by weak credentials. In October 2017, Hermes was deployed as a destructive distraction for a Society for Worldwide Interbank Financial Telecommunication (SWIFT) compromise at the Far Eastern International Bank (FEIB) in Taiwan. Hermes’ role in the SWIFT attack is described in more detail in the </span><i><span style="	font-weight: 400;
">Attribution</span></i><span style="	font-weight: 400;
"> section at the end of this blog. In March 2018, Hermes was observed targeting users in South Korea via the GreenFlash Sundown exploit kit.</span> <span style="	font-weight: 400;
"><b>In mid-August 2018, a modified version of Hermes, dubbed Ryuk, started appearing in a public malware repository.</b> Ryuk was tailored to target enterprise environments and some of the modifications include removing anti-analysis checks. These checks include querying the </span><a href="https://en.wikipedia.org/wiki/Process_Environment_Block"><span style="	font-weight: 400;
">Process Environment Block (PEB)</span></a><span style="	font-weight: 400;
"> to see if the field is <code>BeingDebugged</code>, or querying the PEB to see if the field </span><code><span style="	font-weight: 400;
">NtGlobalFlag</span></code><span style="	font-weight: 400;
"> has been set; checking to see if the host is running VirtualBox by calling the instruction CPUID; and ensuring that the host language is not Russian, Ukrainian, or Belarusian. <b>From a process and file perspective, Hermes and Ryuk target files in a similar fashion.</b> The core differences are Ryuk’s logic that handles file access, and the use of a second, embedded public RSA key.</span> <span style="	font-weight: 400;
">The following are characteristics that have not changed:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Encrypts files using RSA-2048 and AES-256</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Stores keys in the executable using the proprietary Microsoft SIMPLEBLOB format</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Encrypts mounted devices and remote hosts</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Uses a file marker of </span><code><span style="	font-weight: 400;
">HERMES</span></code><span style="	font-weight: 400;
"> to mark or check if a file has been encrypted</span></li></ul>
<p><span style="	font-weight: 400;
">Another notable difference between Hermes and Ryuk is how the encryption keys are created. Hermes starts the encryption initialization by first generating an RSA public and private key pair</span></p>
<p> </p>
<p>— referred to as a “victim key.” An AES-256 key is generated and the victim’s RSA private key is encrypted in AES-CBC mode. The attacker-controlled public RSA key is used to encrypt the AES key (previously used to encrypt the victim’s RSA private key). Then, for each file encrypted, an AES key is generated, which is used to encrypt the file. Finally, the AES key for each file is encrypted with the victim’s RSA public key, then stored at the end of the file. <span style="	font-weight: 400;
">Ryuk contains the same logic, but no longer generates the victim-specific RSA key pair. Instead, Ryuk has two public RSA keys embedded in the executable, and what was previously the victim’s RSA private key is encrypted and embedded in the executable. Because Ryuk does not generate a victim-specific RSA key pair, all hosts can be decrypted with the same decryption key. This might appear to be a design flaw but is not, since Ryuk has a unique key for each executable.</span> <span style="	font-weight: 400;
">If a single executable is used for a single victim environment, then there are no repercussions if the private keys are leaked because it will only decrypt the damage from a single Ryuk executable. Thus, it is highly likely that Ryuk pre-generates the RSA key pairs for each victim. This is arguably more secure, since the victim’s system will never have access to the unencrypted RSA key pair parameters without paying the ransom. This approach is similar to </span><a href="/en-us/blog/big-game-hunting-the-evolution-of-indrik-spider-from-dridex-wire-fraud-to-bitpaymer-targeted-ransomware/"><span style="	font-weight: 400;
">INDRIK SPIDER’s BitPaymer</span></a><span style="	font-weight: 400;
"> ransomware, which generates a victim-specific sample with a hard-coded public key.</span></p>
<p> </p>
<h2>4. Ryuk Functionality: A Technical Analysis</h2>
<p><span style="	font-weight: 400;
">There are two types of Ryuk binaries: a dropper (which is not commonly observed) and the Ryuk executable payload. Recovery of Ryuk droppers are rare, due to the Ryuk executable payload deleting the dropper when executed. Upon execution, the dropper constructs an installation folder path. The folder path is created by calling </span><code><span style="	font-weight: 400;
">GetWindowsDirectoryW</span></code><span style="	font-weight: 400;
"> and then inserting a null byte at the fourth character of the path. This is used to create a string that contains the drive letter path. If the host operating system is Windows XP or earlier, the string </span><code><span style="	font-weight: 400;
">Documents and Settings\Default User\</span></code><span style="	font-weight: 400;
"> is appended to the drive letter path. If the host is Windows Vista or newer, the string </span><code><span style="	font-weight: 400;
">users\Public\</span></code><span style="	font-weight: 400;
"> is appended to the drive letter path. For Windows XP, an example folder path would be </span><code><span style="	font-weight: 400;
">C:\Documents and Settings\Default User\</span></code><span style="	font-weight: 400;
">, and for Window Vista or higher, the path would be </span><code><span style="	font-weight: 400;
">C:\Users\Public.</span></code> <span style="	font-weight: 400;
">A random executable file name is then constructed. It is created by calling </span><code><span style="	font-weight: 400;
">_srand</span></code><span style="	font-weight: 400;
"> with a seed value returned from calling </span><code><span style="	font-weight: 400;
">GetTickCount</span></code><span style="	font-weight: 400;
">, then </span><code><span style="	font-weight: 400;
">_rand</span></code><span style="	font-weight: 400;
"> is continuously called until five alphabetic characters are concatenated together. The extension </span><code><span style="	font-weight: 400;
">.exe</span></code><span style="	font-weight: 400;
"> is then appended. The dropper checks whether the host is 32-bit or 64-bit by calling </span><code><span style="	font-weight: 400;
">IsWow64Process</span></code><span style="	font-weight: 400;
"> and writes one of two embedded payload executables corresponding to the host’s architecture. The newly written executable is then run by calling </span><code><span style="	font-weight: 400;
">ShellExecuteW</span></code><span style="	font-weight: 400;
">. The Ryuk payload executable written by the dropper is the Ryuk component that contains the core logic for encrypting files on the host.</span> <span style="	font-weight: 400;
">Ryuk is under constant development. In recent months, Ryuk binaries have continued to deviate further and further from the original Hermes source code, with the threat actors adding and removing functionality often. In November 2018, Falcon Intelligence identified new functionality added to Ryuk that included an anti-analysis infinite loop, a ping-like request to an IP address once the encryption process was completed, and the addition of an appended file extension for encrypted files. Of these three new features, only the file extension is still present in an executable compiled on Dec. 20, 2018.</span></p>
<h3>File Encryption</h3>
<p><b><span style="	font-weight: 400;
">Compared to other <a href="/en-us/epp-101/what-is-ransomware/">families of ransomware</a>, Ryuk has very few safeguards to ensure stability of the host by not encrypting system files. For example, many ransomware families contain extensive lists of file extensions or folder names that should not be encrypted (whitelisted), but Ryuk only whitelists three extensions: It will not encrypt files with the extensions </span><code><span style="	font-weight: 400;
">exe, dll</span></code><span style="	font-weight: 400;
">, or </span><code><span style="	font-weight: 400;
">hrmlog</span></code><span style="	font-weight: 400;
">. The last extension appears to be a debug log filename created by the original Hermes developer. It should be noted that absent from this list is </span><code><span style="	font-weight: 400;
">sys</span></code><span style="	font-weight: 400;
"> (system drivers), </span><code><span style="	font-weight: 400;
">ocx</span></code><span style="	font-weight: 400;
"> (OLE control extension) and other executable file types. Encrypting these files could make the host unstable. Early versions of Ryuk included the whitelisting of </span><code><span style="	font-weight: 400;
">ini</span></code><span style="	font-weight: 400;
"> and </span><code><span style="	font-weight: 400;
">lnk</span></code><span style="	font-weight: 400;
"> files, but these have been removed in recent builds. The following folder names are also whitelisted and not encrypted.</span></b></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Chrome</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Mozilla</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Recycle.bin</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Windows</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Microsoft</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">AhnLab</span></li></ul>
<p><span style="	font-weight: 400;
">This is only a small subset of folder names that should be whitelisted in order to ensure stability on the host. While doing dynamic analysis, it was not uncommon to observe Ryuk attempting to encrypt files related to the Windows Bootloader (</span><code><span style="	font-weight: 400;
">C:\Boot</span></code><span style="	font-weight: 400;
">) or other critical files and folders. Due to the absence of proper whitelisting, an infected machine can become unstable over time and unbootable if restarted.</span> <span style="	font-weight: 400;
">As mentioned in the </span><i><span style="	font-weight: 400;
">Hermes to Ryuk</span></i><span style="	font-weight: 400;
"> section, Ryuk uses a combination of symmetric (AES) and asymmetric (RSA) encryption to encrypt files. Without the private key provided by WIZARD SPIDER, the files cannot be decrypted and are unrecoverable. A thread is created for the encryption of each file and each file is encrypted with its own AES key. After the file has been encrypted, a file extension of </span><code><span style="	font-weight: 400;
">.RYK</span></code><span style="	font-weight: 400;
"> is appended to the file. All directories will have a ransom note of (</span><code><span style="	font-weight: 400;
">RyukReadMe.txt</span></code><span style="	font-weight: 400;
">) written to the directory.</span> <span style="	font-weight: 400;
">Ryuk attempts to encrypt all mounted drives and hosts that have</span><i><span style="	font-weight: 400;
"> Address Resolution Protocol </span></i><span style="	font-weight: 400;
">(ARP) entries (IP addresses) and it enumerates all mounted drives by calling </span><code><span style="	font-weight: 400;
">GetLogicalDrives</span></code><span style="	font-weight: 400;
">. For each mounted drive, Ryuk calls </span><code><span style="	font-weight: 400;
">GetDriveTypeW</span></code><span style="	font-weight: 400;
"> to determine the drive’s type. If the drive type is not a CD-ROM, files on the drive are encrypted. To retrieve IP addresses that have ARP entries, Ryuk calls </span><code><span style="	font-weight: 400;
">GetIpNetTable</span></code><span style="	font-weight: 400;
">. It iterates through all entries and then tries to enumerate files and folders on the remote host and encrypt the files.</span></p>
<h3>Persistence</h3>
<p><span style="	font-weight: 400;
">Current builds of Ryuk no longer contain persistence functionality. Previously, to remain persistent on the host, Ryuk created a registry entry under the Run key using Windows </span><code><span style="	font-weight: 400;
">cmd.exe</span></code><span style="	font-weight: 400;
"> shell. The following command line was used to write to the Registry Run Key name </span><code><span style="	font-weight: 400;
">svchos</span></code><span style="	font-weight: 400;
"> to</span></p>
<p> </p>
<p><code><span style="	font-weight: 400;
">HKEY_CURRENT_USER\SOFTWARE\Microsoft\Windows\CurrentVersion\Run</span></code><span style="	font-weight: 400;
"> with the value being the path to the Ryuk executable.</span></p>
<p> </p>
<h3>Process Injection</h3>
<p><span style="	font-weight: 400;
">Ryuk does not encrypt files from within its own process memory space, but injects into a remote process. Before injecting into a remote process, Ryuk attempts to adjust its token privileges to have the </span><code><span style="	font-weight: 400;
">SeDebugPrivilege</span></code><span style="	font-weight: 400;
">. It takes no action if the adjustment of the token privileges fails. Before injecting into a remote process, Ryuk also calls </span><code><span style="	font-weight: 400;
">CreateToolhelp32Snapshot</span></code><span style="	font-weight: 400;
"> to enumerate all running processes. If a process is found that is not named </span><code><span style="	font-weight: 400;
">csrss.exe, explorer.exe, lsaas.exe</span></code><span style="	font-weight: 400;
"><code>,</code> or is running under </span><code><span style="	font-weight: 400;
">NT AUTHORITY</span></code><span style="	font-weight: 400;
"> system account, Ryuk will inject itself into this single process. By ensuring that the process is not running under </span><code><span style="	font-weight: 400;
">NT AUTHORITY</span></code><span style="	font-weight: 400;
"><code>,</code> the developers are assuming the process is not running under another account and therefore can be written to. Ryuk uses a combination of </span><code><span style="	font-weight: 400;
">VirtualAlloc, WriteProcessMemory</span></code><span style="	font-weight: 400;
"> and </span><code><span style="	font-weight: 400;
">CreateRemoteThread</span></code><span style="	font-weight: 400;
"> to inject itself into the remote process.</span></p>
<h3>Process/Service Termination and Anti-Recovery Commands</h3>
<p><span style="	font-weight: 400;
">Unlike other families of ransomware, Ryuk does not contain process/service termination and anti-recovery functionality embedded in the executable. In the past, Ryuk did contain these capabilities, but they have been removed and are contained within two batch files.</span> <span style="	font-weight: 400;
">The batch file </span><code><span style="	font-weight: 400;
">kill.bat </span></code><span style="	font-weight: 400;
">contains commands for stopping services, disabling services and killing processes. The processes and services are stopped to ensure no open handles exist for files that will be encrypted. The following figure is a subset of each command.</span> <code><span style="	font-weight: 400;
">net stop avpsus /y net stop McAfeeDLPAgentService /y net stop mfewc /y net stop BMR Boot Service /y net stop NetBackup BMR MTFTP Service /y … sc config SQLTELEMETRY start&#61; disabled sc config SQLTELEMETRY$ECWDB2 start&#61; disabled sc config SQLWriter start&#61; disabled sc config SstpSvc start&#61; disabled … taskkill /IM mspub.exe /F taskkill /IM mydesktopqos.exe /F taskkill /IM mydesktopservice.exe /F</span></code> Figure 1. Process/Services Termination <code>kill.bat</code> Commands <span style="	font-weight: 400;
">CrowdStrike has observed another batch file, named </span><code><span style="	font-weight: 400;
">windows.bat</span></code><span style="	font-weight: 400;
">, which makes file recovery more difficult on the victim’s machine. It should be noted that file names can be arbitrarily</span></p>
<p> </p>
<p>changed by the threat actors. The contents of the batch file are shown below in Figure 2. <code><span style="	font-weight: 400;

<h3><b>Criminal Actors Operating from Russia</b></h3>
<p><span style="	font-weight: 400;
">Falcon Intelligence has medium-high confidence that the WIZARD SPIDER threat actors are operating out of Russia. Hermes was originally advertised on </span><code><span style="	font-weight: 400;
">exploit&lt;.&gt;in</span></code><span style="	font-weight: 400;
">. This Russian-speaking forum is a well-known marketplace for selling malware and related services to criminal threat actors. If Hermes was indeed related to STARDUST CHOLLIMA, it would imply that nation-state threat actors are selling their services on Russian-speaking forums, which is unlikely.</span> <span style="	font-weight: 400;
">The Russian threat actor attribution theory is also supported by an early advertisement for Hermes, which stated that their “software did not work and will on work on RU, UA, BY” . This refers to functionality implemented in Hermes to check the host to ensure that it is not running on a Russian, Ukrainian, or Belarusian system. To check the host language, it queries the registry key <code>HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Nls\Language\</code> and the value </span><code><span style="	font-weight: 400;
">InstallLanguage</span></code><span style="	font-weight: 400;
">. If the machine has the value 0419 (Russian), 0422 (Ukrainian) or 0423 (Belarusian), it call <code>ExitProcess</code></span></p>
<p> </p>
<p>to stop executing. This functionality is commonly included by malware developers and sellers who are operating in Russia to reduce their risk of attracting local law enforcement’s attention and criminal prosecution. <span style="	font-weight: 400;
">While supporting an incident response investigation involving Ryuk, Falcon Intelligence noticed files related to the investigation being uploaded to a file-scanning website from an IP address in Moscow, Russia. The file in question was a variant of </span><code><span style="	font-weight: 400;
">kill.bat</span></code><span style="	font-weight: 400;
"> that contained commands previously only observed executed by Ryuk calling </span><code><span style="	font-weight: 400;
">ShellExecute</span></code><span style="	font-weight: 400;
">. The files could have been uploaded by a victim in Russia, but the time frame between the functionality being removed from Ryuk binaries and included in </span><code><span style="	font-weight: 400;
">kill.bat</span></code><span style="	font-weight: 400;
"> was very short. The most likely scenario is that threat actors were testing whether </span><code><span style="	font-weight: 400;
">kill.bat</span></code><span style="	font-weight: 400;
"> would be detected by antivirus engines.</span> <span style="	font-weight: 400;
">Also, during forensic investigation of a network compromised by WIZARD SPIDER, CrowdStrike Services recovered artifacts with filenames in Russian. One file was named </span><code><span style="	font-weight: 400;
">!!! files dlya raboty !!!.rar</span></code><span style="	font-weight: 400;
">, which translates to “files for work.”<p> </p>
Based on these factors, there is considerably more evidence supporting the hypothesis that the WIZARD SPIDER threat actors are Russian speakers and not North Korean.</span></p>
