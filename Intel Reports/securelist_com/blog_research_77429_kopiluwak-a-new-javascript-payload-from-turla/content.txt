													<p>On 28 January 2017, John Lambert of Microsoft (@JohnLaTwC) tweeted about a malicious document that dropped a &#8220;<a href="https://twitter.com/JohnLaTwC/status/825371797767938048">very interesting .JS backdoor</a>&#8220;. Since the end of November 2016, Kaspersky Lab has observed Turla using this new JavaScript payload and specific macro variant. This is a technique we&#8217;ve observed before with Turla&#8217;s ICEDCOFFEE payloads, detailed in a private report from June 2016 (available to customers of Kaspersky APT Intelligence Services). While the delivery method is somewhat similar to ICEDCOFFEE, the JavaScript differs greatly and appears to have been created mainly to avoid detection.</p>
<p>Targeting for this new malware is consistent with previous campaigns conducted by Turla, focusing on foreign ministries and other governmental organizations throughout Europe. Popularity of the malware, however, is much lower than ICEDCOFFEE, with victim organizations numbering in the single digits as of January 2017. We assess with high confidence this new JavaScript will be used more heavily in the future as a stage 1 delivery mechanism and victim profiler. </p>
<p>The malware is fairly simplistic but flexible in its functionality, running a standard batch of profiling commands on the victim and also allowing the actors to run arbitrary commands via Wscript. </p>
<h2 id="actor-profile">Actor Profile</h2>
<p>Turla, also known as Snake / Uroburos / Venomous Bear and KRYPTON is a Russian-speaking APT group that has been active since at least 2007. Its activity can be traced to many high-profile incidents, including the 2008 attack against the US Central Command, (see <a href="https://en.wikipedia.org/wiki/2008_cyberattack_on_United_States">Buckshot Yankee</a> incident) or more recently, the attack against <a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/02/20081149/Report_Ruag-Espionage-Case.pdf">RUAG</a>, a Swiss military contractor. The Turla group has been known as an agile, very dynamic and innovative APT, leveraging many different families of malware, <a href="https://securelist.com/satellite-turla-apt-command-and-control-in-the-sky/72081/">satellite-based</a> command and control servers and malware for non-Windows OSes.</p>
<p>Targeting Ukraine, EU-related institutions, governments of EU countries, Ministries of Foreign Affairs globally, media companies and possibly corruption related targets in Russia, the group intensified their activity in 2014, which we described in our paper <a href="https://securelist.com/analysis/publications/65545/the-epic-turla-operation/">Epic Turla</a>. During 2015 and 2016 the group diversified their activities, switching from the Epic Turla waterhole framework to the Gloog Turla framework, which is still active. They also expanded their spear phishing activities with the Skipper / WhiteAtlas attacks, which leveraged new malware. Recently, the group has intensified their satellite-based C&amp;C registrations ten-fold compared to their 2015 average. </p>

<p style="text-align:center;font-weight:bold"><i>Decoy document used in the attack</i></p>
<p>The lure document above shows an official letter from the Qatar Embassy in Cyprus to the Ministry of Foreign Affairs (MoFA) in Cyprus. Based on the name of the document (National Day Reception (Dina Mersine Bosio Ambassador&#8217;s Secretary).doc, it is presumed it may have been sent from the Qatar Ambassador&#8217;s secretary to the MoFA, possibly indicating Turla already had control of at least one system within Qatar&#8217;s diplomatic network. </p>
<p>The document contains a malicious macro, very similar to previous macros used by Turla in the past to deliver Wipbot, Skipper, and ICEDCOFFEE. However, the macro did contain a few modifications to it, mainly the XOR routine used to decode the initial JavaScript and the use of a &#8220;marker&#8221; string to find the embedded payload in the document. </p>

<h3 id="second-layer-javascript">Second Layer JavaScript</h3>
<p>Once the marker is found, the macro will carve out &#8220;15387 + 1&#8221; bytes (hard coded) from the end of the marker and pass that byte array to the aforementioned decoding routine. The end result is a JavaScript file (mailform.js &#8211; MD5: 05d07279ed123b3a9170fa2c540d2919) written to &#8220;%APPDATA%MicrosoftWindows&#8221;. </p>

<p style="text-align:center;font-weight:bold"><i>mailform.js &#8211; malicious obfuscated JavaScript payload</i></p>
<p>This file is then executed using Wscript.Shell.Run() with a parameter of &#8220;NPEfpRZ4aqnh1YuGwQd0&#8221;. This parameter is an RC4 key used in the next iteration of decoding detailed below.</p>
<p>The only function of mailform.js is to decode the third layer payload stored in the JavaScript file as a Base64 string. This string is Base64 decoded, then decrypted using RC4 with the key supplied above as a parameter (&#8220;NPEfpRZ4aqnh1YuGwQd0&#8221;). The end result is yet another JavaScript which is passed to the eval() function and executed. </p>
<h3 id="third-layer-javascript">Third Layer JavaScript</h3>
<p>The third layer payload is where the C2 beaconing and system information collection is performed. This JS will begin by copying itself to the appropriate folder location based on the version of Windows running:</p>
<ol>
<li>
<p>c:Users&lt;USERNAME&gt;AppDataLocalMicrosoftWindowsmailform.js</p>
</li>
<li>
<p>c:Users&lt;USERNAME&gt;AppDataLocalTempmailform.js</p>
</li>
<li>
<p>c:Documents and Settings&lt;USERNAME&gt;Application DataMicrosoftWindowsmailform.js</p>
</li>
</ol>
<h4 id="persistence">Persistence</h4>
<p>Next, it will establish persistence on the victim by writing to the following registry key:</p>
<p>Key: HKEY_CURRENT_USERsoftwaremicrosoftwindowscurrentversionrunmailform<br />Value: wscript.exe /b &#8220;&lt;PATH_TO_JS&gt; NPEfpRZ4aqnh1YuGwQd0&#8221;</p>

<p>After establishing its persistence, it will then execute a series of commands on the victim system using &#8220;cmd.exe /c&#8221; and store them to a file named &#8220;~dat.tmp&#8221;, in the same folder where &#8220;mailform.js&#8221; is located:</p>

<p>Once the information is collected into the temporary &#8220;~dat.tmp&#8221; file, the JavaScript reads its contents into memory, RC4 encrypts it with the key &#8220;2f532d6baec3d0ec7b1f98aed4774843&#8221;, and deletes the file after a 1 second sleep, virtually eliminating storage of victim information on disk and only having an encrypted version in memory. </p>
<h4 id="network-communications">Network Communications</h4>
<p>With the victim info stored in encrypted form in memory, the JavaScript then will perform the necessary callback(s) to the C2 servers which are hard coded in the payload. The addresses seen in this payload were as follows:</p>
<ul>
<li>http://soligro[.]com/wp-includes/pomo/db.php</li>
<li>http://belcollegium[.]org/wp-admin/includes/class-wp-upload-plugins-list-table.php</li>

<p>The &#8220;exit&#8221; command will cause script to exit gracefully, thus shutting down the communications to the C2 server until next startup / login from the user. </p>
<p>The &#8220;fail&#8221; command is for uninstalling the JavaScript and its persistence. Both the &#8220;mailform.js&#8221; file and registry key created for persistence will be deleted upon receipt of this command. </p>
<p>The &#8220;work&#8221; command is used to task the victim&#8217;s system to run arbitrary commands via Wscript.shell.run(). It begins by checking to see if a file &#8220;mailform.pif&#8221; exists in the same directory as the JavaScript, and if so, it will delete it. The victim will then send a POST request to the C2 much in the same way as before with the beacon traffic, but with some slight differences. The User-Agent header will remain the same as in the beacon traffic, but the data sent to the C2 will consist of the 4-byte string &#8220;work&#8221;. If the response from the server after this acknowledgement is &#8220;200 OK&#8221;, then the system will proceed to read the response data into memory, RC4 encrypt it using the same key &#8220;2f532d6baec3d0ec7b1f98aed4774843&#8221;, then write it out to the &#8220;mailform.pif&#8221; file referenced above. The command file is run, the JavaScript will sleep for 30 seconds, and then the file is subsequently deleted. </p>
<h2 id="victims-and-sinkholing">Victims and Sinkholing</h2>
<p>One of the domains involved in this new malware (soligro[.]com) expired in July 2016 and was was available for purchase and sinkhole at the time of the analysis. Sinkhole data shows several potential victims, with one high profile victim (195.251.32.62) located within the Greek Parliament: </p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/02/07181529/kopiluwak_en_4.png" class="magnificImage"><img loading="lazy" decoding="async" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/02/07181529/kopiluwak_en_4-1024x524.png" alt="" width="604" height="309" class="aligncenter size-large wp-image-77435" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/02/07181529/kopiluwak_en_4-1024x524.png 1024w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/02/07181529/kopiluwak_en_4-300x153.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/02/07181529/kopiluwak_en_4-768x393.png 768w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2017/02/07181529/kopiluwak_en_4.png 1222w" sizes="auto, (max-width: 604px) 100vw, 604px" /></a></p>
<p>The majority of connections to the sinkhole server have been observed from IP ranges residing within Greece. This leads us to believe the main target for the specific document above was Greece, although we also have indications of targeting in Romania and Qatar based on other data.</p>
<h2 id="conclusions">Conclusions</h2>
<p>In recent months, the Turla actors have increased their activity significantly. The addition of KopiLuwak to their already existing ICEDCOFFEE JavaScript payload indicates the group continues to evolve and deliver new tools to avoid detection by known malware signatures. </p>
<p>Currently, it seems the Turla actors continue to rely heavily on embedded macros in Office documents. While this may appear to be an elementary technique to use for such a sophisticated actor, they are repeatedly successful in compromising high value targets with this method. It is advised that users disable macros in their enterprise and not allow the user to enable said content unless absolutely necessary. Furthermore, using the polymorphic obfuscation technique for the macros has caused difficulties in writing signatures for detection. </p>
