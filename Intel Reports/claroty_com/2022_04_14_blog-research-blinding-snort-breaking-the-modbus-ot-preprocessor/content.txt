        <h2>Executive Summary</h2><ul><li><p>Team82 discovered a means by which it could blind the popular Snort intrusion detection and prevention system to malicious packets.</p></li><li><p>The vulnerability, <a href="https://claroty.com/team82/disclosure-dashboard/cve-2022-20685">CVE-2022-20685</a>, is an integer-overflow issue that can cause the Snort Modbus OT preprocessor to enter an infinite while-loop.</p></li><li><p>A successful exploit keeps Snort IDS/IPS from processing new packets and generating alerts.</p></li><li><p>The vulnerability, which can be attacked remotely, has been patched by Cisco and the Snort team.</p></li><li><p>All open source Snort project releases earlier than 2.9.19 and release 3.1.11.0 are vulnerable.</p></li><li><p>Read <a target="_blank" href="https://sec.cloudapps.cisco.com/security/center/content/CiscoSecurityAdvisory/cisco-sa-snort-dos-9D3hJLuj">Cisco&#039;s advisory here</a> for commercial product patching and mitigation information.</p></li></ul><h3>Table of Contents</h3><ul><li><p><a href="#Snort-Rules-and-Alerts">Snort Rules and Alerts</a></p></li><li><p><a href="#Modbus">Modbus</a></p></li><li><p><a href="#CVE-2022-20685">CVE-2022-20685: Technical Details</a></p></li><li><p><a href="#Vulnerability-and-Exploitability">Vulnerability and Exploitability</a></p></li><li><p><a href="#Conclusion">Conclusion</a></p></li><li><p></p></li></ul><h2>What is the Snort Network Intrusion Detection System?</h2><p>Network analysis tools are integral to keeping networks secure by providing real-time logging and analysis of events and traffic. Snort is atop this list of analysis tools as the most popular network intrusion detection and prevention system. The open-source version of the Snort IDS/IPS still has an active community of contributors and developers, while Cisco has developed commercial versions of Snort since acquiring parent company Sourcefire in 2013.</p><p>Snort is largely used passively on the network, but it can also take action on malicious packets, making it a powerful detection tool for defenders. An attacker who could blind this tool to malicious traffic, however, could gain an important advantage over network defenders.</p>

        <div id="Snort-Rules-and-Alerts"></div>
    
        <h2>Snort Rules and Alerts</h2><p>Snort&#039;s open-source network-based intrusion detection/prevention system (IDS/IPS) has the ability to perform real-time traffic analysis and packet logging on internet protocol (IP) networks. Snort IDS/IPS performs protocol analysis, content searching, and matching based on a predefined rule set. Snort rules can be based on raw data or on Snort&#039;s built-in protocol parsers.</p><p>Rules may trigger one of three actions:</p><ul><li><p>Alert rules: Generate an alert</p></li><li><p>Log rules: Alert and log the alert</p></li><li><p>Pass rules: Ignore the packet</p></li></ul><p>To make the rule-writing process simpler and improve the detection capabilities, Snort comes with a set of preprocessors that are on by default and analyze and structure network traffic into objects that can be referenced later in Snort rules. Some of the preprocessors included in Snort are: ARP, DNS, SSH and some OT (operational technology) protocols, such as <a target="_blank" href="https://www.snort.org/faq/readme-modbus">MODBUS</a> / <a target="_blank" href="https://www.snort.org/faq/readme-dnp3">DNP3</a>.</p>

        <div id="CVE-2022-20685"></div>
    
        <h2>CVE-2022-20685: Technical Details</h2><p>While researching Snort OT preprocessors, we decided to focus on Modbus because it was one of the more complex OT preprocessors Snort supports. To understand what we found, we first need to examine the structure of the Modbus Write File Record function code.</p><h3>Write File Record (command 0x15)</h3><p>The Write File Record Modbus command writes multiple groups of file registers to the Modbus server. A file is an organization of records. Each file may contain up to 10,000 records, addressed 0000 to 9999 decimal or 0x0000 to 0x270F.</p><p>The Write File Record Modbus command allows writing multiple groups of references. Each group is defined in a separate sub-request field that contains 7 bytes plus the data:</p><ul><li><p><strong>Reference type</strong>: 1 byte (must be specified as 6)</p></li><li><p><strong>File number</strong>: 2 bytes</p></li><li><p><strong>Starting record number within file</strong>: 2 bytes</p></li><li><p><strong>Length of record to be written</strong>: 2 bytes</p></li><li><p><strong>Data to be written</strong>: 2 bytes of data per register</p></li></ul><p>The number of registers to be written, combined with all other fields in the request, must not exceed the allowable length of the Modbus protocol data unit (PDU), which is 253 bytes.</p><p>Here is a summary of the Write File Record Modbus command request:</p>

        <div id="Vulnerability-and-Exploitability"></div>
    
        <h2>Vulnerability and Exploitability</h2><p>The Modbus preprocessor handles multiple Modbus function codes. Snort IDS/IPS uses the  ModbusCheckRequestLengths function to calculate the expected size for each packet.</p><p>If we look at the function ModbusCheckRequestLengths in the file modbus_decode.c, we see a while-loop that goes over all of the groups in the packet, in order to calculate the total record lengths.</p>

        <h2>3 Steps to Exploiting Modbus</h2><h3>Step 1</h3><p>The tmp_count parameter is initialized using a value from the packet-&gt;payload and represents the number of bytes remaining in the payload, according to the payload_length parameter. After tmp_count is set, we enter a while-loop with the exit condition of bytes_processed &lt; tmp_count. Therefore, so far tmp_count = 10, which is constant and won&#039;t change during the loop. As long as bytes_processed remains less than 10, the while-loop will continue looping.</p><p>To do so, let&#039;s look at the content of the while-loop. We see that bytes_processed is affected by the record_length parameter, which consists of two bytes from the Modbus payload (Group → Record Length).</p>

        <h3>Step 2</h3><p>The record_length parameter is of type uint16_t, with a value from the user-controlled Modbus payload. bytes_processed is also uint16_t, and is calculated by multiplying record_length by 2 + the sub-request header size, which is 7.</p><p>However, the result of the multiplication can be more than the maximum uint16_t size, thus overflowing the value.</p><p>For example:</p><ul><li><p><strong>record_length</strong> = 0xfffe</p></li><li><p><strong>MODBUS_FILE_RECORD_SUB_REQUEST_SIZE</strong> = 7</p></li><li><p><strong>bytes_processed</strong> = 7 + (2 * 0xfffe)</p></li></ul><p>In this example, the bytes_processed will be 0x20003, which is:</p><p>0000000000000010|0000000000000011</p><p><strong>      High                                    Low</strong></p><p>In binary, when the result is cast to uint_16t, the lower 16 bits are kept, meaning the bytes_processed will be 0000000000000011, which equals 3. If the bytes_processed is 3, we do not exit the while-loop, because 3 (bytes_processed) &lt; 10 (tmp_count), and we enter another iteration.</p><h3>Step 3</h3><p>Now, the new record_length will be taken from the user-controlled payload, from a specific offset that is partially affected by the bytes_processed value. Since we fully control the value of the bytes_processed using the integer-overflow bug, we can craft the payload in such a way that the newly calculated record_length will be any number we choose.</p><p>Therefore, if the next value that is read into the record_length (allegedly the next group&#039;s record length) is 0xfffb, then the bytes_processed will be calculated as follows:</p><p>bytes_processed = bytes_processed + MODBUS_FILE_RECORD_SUB_REQUEST_SIZE + 2*record_length</p><p>bytes_processed = 3 + 7 + 2*(0xfffb) = 0</p><p>So, the bytes_processed is now 0.The next time we enter the while-loop, the bytes_processed is 0 again, so we will go through steps 2 and 3 over and over, until the process is terminated by the user. This essentially keeps the process stuck in the while-loop (steps 2 &amp; 3) &quot;blinding&quot; it forever. In this state, Snort will not process new packets and will not alert.</p><p></p>

        <div id="Conclusion"></div>
    
        <h2>Conclusion</h2><p>Successful exploits of vulnerabilities in network analysis tools such as Snort IDS/IPS can have devastating impacts on enterprise and OT networks.</p><p><a href="https://claroty.com/team82/disclosure-dashboard/cve-2022-20685">CVE-2022-20685</a>, uncovered by Team82, targeted just one facet of this popular network intrusion detection and prevention system. It can be exploited remotely to create a denial-of-service condition in Snort, keeping it from processing new packets, and generating alerts.</p><p>Team82 believes network analysis tools are an under-researched area that deserves more analysis and attention, especially as OT networks are increasingly being centrally managed by IT network analysis familiar with Snort and other similar tools.</p><p><a href="https://claroty.com/team82/disclosure-dashboard/cve-2022-20685">CVE-2022-20685</a></p><p><strong>CWE-190</strong>: Integer Overflow or Wraparound</p><p><strong>CVSSv3 score:</strong> 7.5</p><p><strong>Description:</strong> An integer overflow vulnerability in the Snort Modbus OT preprocessor enables an attacker to remotely send a crafted packet to a vulnerable system, triggering an infinite while-loop and creating a denial-of-service condition.</p>
