<p>In this post, we analyze a piece of malware that we encountered during a recent breach investigation. What caught our attention was how the malware achieved persistence, how it used ICMP tunneling for its backdoor communications, and how it operated with different modes to increase its chances of a successful attack. Malware using ICMP is not new but is relatively uncommon. Because of this, and the presence of certain strings, we decided to name this malware â€˜Pingbackâ€™. Below we demonstrate how Pingbackâ€™s protocols work and also provide sample code on how we interacted with the malware.</p>
<!--more-->
<p>We begin by looking at how Pingback achieves persistence through DLL hijacking.</p>
<h3>Persistence through DLL Hijacking</h3>
<p>DLL (Dynamic Link Library) hijacking is a technique that involves using a legitimate application to preload a malicious DLL file. Attackers commonly abuse the Windows DLL Search Order and take advantage of this to load a malicious DLL file instead of the legitimate one.</p>
<p>The file we investigated was a DLL file called <strong><em>oci.dll</em></strong>. We knew that the file was suspicious during our initial triaging, but we could not figure how it was loaded into the system because the DLL was not loaded through traditional rundll32.exe.</p>

<p>&nbsp;</p>
<p>We found out later that it got loaded through a legitimate service called <strong>msdtc</strong> (a.k.a Microsoft Distributed Transaction Coordinator). This service, as the name suggests, coordinates transactions that span multiple machines, such as databases, message queues, and file systems.</p>
<p>It turns out the <strong>msdtc</strong> service indirectly loads <em>oci.dll</em> through MSDTCTM.DLL that loads an ODBC library to support Oracle databases called MTXOCI.DLL. This library searches for and tries to load three Oracle ODBC DLLs which include <em>oci.dll</em>, <em>SqlLib80.dll</em>, and <em>xa80.dll</em>.</p>

So, in theory, an attacker with system privileges can drop a malicious DLL and save it using one of the DLL filenames that <em>MTxOCI</em> loads. We have experimented with dropping all three DLL filenames but only <strong><em>oci.dll</em></strong> was successfully loaded by the service.</p>

<p><strong>msdtc</strong> by default does not run during start-up. To remain persistent, the <strong>msdtc</strong> service needs to be configured to start automatically, so the attacker would need system privileges to reconfigure the <strong>msdtc</strong> startup type. It can be done manually using SC command, via malicious scripts, or through a malware installer.</p>
<p>Our theory is that a separate executable installed this malware. In fact, after a bit of hunting, we <a href="https://www.virustotal.com/gui/file/4ff77ea841544569e9da8aa3990724d1473731e684a162014ad1ad54e8c8cef2/detection">found</a> a sample in VirusTotal with similar IOCs that installs <strong><em>oci.dll</em></strong> into the Windows System directory and then sets <strong>msdtc</strong> service to start automatically.</p>

<p>So that is the DLL loading part. But before turning our attention to Pingback itself and its operation, let us first lay out what is ICMP and how ICMP tunneling works.</p>
<h3>ICMP Foundation</h3>
<p>The Internet Control Message Protocol (ICMP) is a network layer protocol mainly used by network devices for diagnostic and control purposes. It is used in utilities such as ping to determine reachability and roundtrip time, traceroute, and path MTU discovery to avoid packet fragmentation and enhance performance. It can also be misused by malicious actors to scan and map a targetâ€™s network environment. This is one of the reasons why there are some debates over whether ICMP should be disabled or not. In most cases, users do not pay attention to ICMP packets either as they do not manifest open ports on the machine.</p>
<p>The malware, Pingback, at the center of our investigation, oci.dll,&nbsp;uses the ICMP protocol for its main communication. This has the effect of being hidden from the user as ports cannot be listed by netstat. Below we detail how Pingback uses the ICMP protocol to pass data back and forth between the infected host and the attackerâ€™s host. A technique called ICMP tunneling.</p>
<p>To explain ICMP tunneling, let us first understand an ICMP packet:</p>

<p>An ICMP packet is built on top of the IP layer and has an 8 byte ICMP header. The packet size limit for ICMP data is a maximum allowed size of an IPv4 network packet, minus the 20 byte IP header and 8 byte ICMP header. Or approximately 64K. The ICMP data is determined by the message type. The message types are defined here: <a href="https://www.iana.org/assignments/icmp-parameters/icmp-parameters.xhtml">https://www.iana.org/assignments/icmp-parameters/icmp-parameters.xhtml</a></p>

<h3>ICMP Tunneling</h3>
<p>Now that we have laid out the foundation and how an attacker can piggyback data on the ICMP packets, we are ready to explain exactly how exactly this malware works.</p>
<p>Pingback specifically uses the echo (ping) request or type 8 ICMP message. It starts a sniffer for every &nbsp;IP address on the host, &nbsp;spawning a thread to sniff packets on each individual IP address. To distinguish between its own packets and other packets, the sniffer ignores anything else thatâ€™s not an ICMP echo packet and does not contain the ICMP sequence number 1234, 1235 or 1236. It also ignores packets not destined for the specified IP address.</p>
<p>It then interprets the data in the following format:</p>
<p><em><strong><img loading="lazy" src="https://www.trustwave.com/hs-fs/hubfs/Imported%20sitepage%20images/6a0133f264aa62970b026bdecce0d1200c-pi.png?width=605&amp;height=110&amp;name=6a0133f264aa62970b026bdecce0d1200c-pi.png" alt="6a0133f264aa62970b026bdecce0d1200c-pi" width="605" height="110" style="height: auto; max-width: 100%; width: 605px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.trustwave.com/hs-fs/hubfs/Imported%20sitepage%20images/6a0133f264aa62970b026bdecce0d1200c-pi.png?width=303&amp;height=55&amp;name=6a0133f264aa62970b026bdecce0d1200c-pi.png 303w, https://www.trustwave.com/hs-fs/hubfs/Imported%20sitepage%20images/6a0133f264aa62970b026bdecce0d1200c-pi.png?width=605&amp;height=110&amp;name=6a0133f264aa62970b026bdecce0d1200c-pi.png 605w, https://www.trustwave.com/hs-fs/hubfs/Imported%20sitepage%20images/6a0133f264aa62970b026bdecce0d1200c-pi.png?width=908&amp;height=165&amp;name=6a0133f264aa62970b026bdecce0d1200c-pi.png 908w, https://www.trustwave.com/hs-fs/hubfs/Imported%20sitepage%20images/6a0133f264aa62970b026bdecce0d1200c-pi.png?width=1210&amp;height=220&amp;name=6a0133f264aa62970b026bdecce0d1200c-pi.png 1210w, https://www.trustwave.com/hs-fs/hubfs/Imported%20sitepage%20images/6a0133f264aa62970b026bdecce0d1200c-pi.png?width=1513&amp;height=275&amp;name=6a0133f264aa62970b026bdecce0d1200c-pi.png 1513w, https://www.trustwave.com/hs-fs/hubfs/Imported%20sitepage%20images/6a0133f264aa62970b026bdecce0d1200c-pi.png?width=1815&amp;height=330&amp;name=6a0133f264aa62970b026bdecce0d1200c-pi.png 1815w" sizes="(max-width: 605px) 100vw, 605px"></strong></em></p>
<p style="text-align: center;"><a href="https://npercoco.typepad.com/.a/6a0133f264aa62970b026bdecce0d1200c-pi" class="asset-img-link" onclick="window.open( this.href, '_blank', 'width=640,height=480,scrollbars=no,resizable=no,toolbar=no,directories=no,location=no,menubar=no,status=no,left=0,top=0' ); return false"> </a><em><strong>Figure 10</strong>: Malware's ICMP data is represented by this C structure. See appendix below for detailed information of the <strong>cmd</strong> and <strong>cmd_line</strong> fields.</em></p>
<p>&nbsp;</p>
<p>The sequence is used as a message type for each ICMP data. It currently supports 3 message types:</p>
<ul>
<li>1234 â€“ the packet contains a command or data</li>
<li>1235 and 1236 â€“ used for pure ICMP packet communication only. 1235 being the data has been received at the other end, and 1236 as new data has been received by the malware.</li>
</ul>
<p>Pingback supports several commands including:</p>
<ul>
<li><strong>shell</strong> â€“ execute a shell</li>
<li><strong>download</strong> â€“ 3 different modes of download are provided:
<ul>
<li>Mode 1: Infected host connects back to the attackerâ€™s host (works well if incoming TCP connections are blocked by firewall)</li>
<li>Mode 2: Infected host opens a socket on a specified port and waits for the attacker to connect.</li>
<li>Mode 3: Purely ICMP-based, but this is very slow and the current implementation is not very reliable in terms of flow control.</li>
</ul>
</li>
<li><strong>upload</strong> â€“ also supports 3 different modes, similar to the Download command.</li>
<li><strong>exec</strong> â€“ execute a command on the infected host.</li>
</ul>
<p>This is interesting, you can see Pingback uses a combination of ICMP for initiating any of the commands and TCP for better performance and reliability. A pure ICMP mode is also provided but is not very reliable.</p>
<h3>Protocol example</h3>
<p>To download a file in mode 1, the attacker performs:</p>
<ul>
<li>Create a socket and listen on a port</li>
<li>Send the following ICMP packet:
<ul>
<li>ICMP echo request with sequence 1234</li>
<li>Payload contains: â€œdownloadâ€, name of file to download and, IP address and port of the newly created socket, this is where the malware will stream the file to</li>
</ul>
</li>
<li>Wait for connections and receive data. There is an additional protocol here which includes is reasonably simple (send and receive) and wait for â€œEND\x00â€ string.</li>
</ul>
<p>We have provided a source to demonstrate all three modes and most of the commands supported by Pingback.</p>
<p>Source available here:</p>
<p><a href="https://github.com/SpiderLabs/pingback">https://github.com/SpiderLabs/pingback</a></p>
<p>We have also prepared a video to demonstrate how our client interacts with the malware running in an isolated infected system.</p>
<h3>Final Words</h3>
<p>ICMP tunneling is not new, but this particular sample piqued our interest as a real-world example of malware using this technique to evade detection. ICMP is useful for diagnostics and performance of IP connections in the real world. It is very useful to have them enabled but must be balanced by real-world threats. While we are not suggesting that ICMP should be disabled,&nbsp; we do suggest putting in place monitoring to help detect such covert communications over ICMP.</p>
<p>For network administrators and technical audience, a rule can be implemented to check if a packet is an ICMP echo (type 8), the data size is 788 bytes or greater and check for ICMP sequence number: 1234, 1235 or 1236. &nbsp;Backdoor command strings such as â€œdownloadâ€, â€œuploadâ€, â€œexecâ€, â€œexepâ€, â€œrexecâ€, &nbsp;â€œshellâ€ that found in an ICMP data packet can also be flagged. <a href="https://www.trustwave.com/en-us/services/technology/intrusion-detection-and-prevention/">Trustwave Managed IDS</a> devices can also detect this malicious traffic.</p>
<p>Finally, this malware did not get into the network through ICMP but rather utilizes ICMP for its covert bot communications. The initial entry vector is still being investigated.</p>
<h3>Appendix</h3>
<p><strong>cmd</strong> â€“ bot commands and may be any of the following:&nbsp;</p>
<ul>
<li><em><u>exep</u></em> (execute process) â€“ execute a binary/command on the remote host</li>
<li><em><u>download</u></em> (download mode 1) - &nbsp;attackerâ€™s initial connection is done via ICMP and appears as a ping packet. The ICMP echo packet contains data that specifies the attackerâ€™s host and port to where the malware connects back. The ICMP data also contains a file path that the attacker requests. Using the host and port information, the malware creates a new socket, then transmits the requested file back to the attacker.</li>
<li><em><u>upload</u></em> (upload mode 1) â€“ attackerâ€™s initial connection is done via ICMP. The malware receives the initial connection then connects back to attackerâ€™s host and port specified in the ICMP Echo packet. It then receives the file from the attacker to be saved in the infected systemâ€™s local disk</li>
<li><em><u>download2</u></em> â€“ (download mode 2), initial ICMP packet is sent by the attacker. The ICMP echo packet contains the requested filename and path in the infected system. It also contains a port number where the malware will bind and listen to. The malware then waits for the attacker to connect, afterward, it begins transmitting the requested file.</li>
<li><em><u>upload2</u></em> â€“ (upload mode 2), initial ICMP packet is sent by the attacker. The ICMP echo packet contains the filename of the file to be received. It also contains a port number where the malware will bind and listen to. The malware then creates the file in the remote host and waits for the attacker to connect. Once connection is established, the attacker begins transmitting the file content to the remote host</li>
<li><em><u>download3</u></em> â€“ (download mode 3), a file is sent to the attacker purely through ICMP data. Although this mode is more covert as it appears as ping packets only, this is slower than using TCP directly as only 1 packet can be transmitted at a time. The malware has to wait for acknowledgment from the attackerâ€™s end.</li>
<li><em><u>upload3</u></em> â€“ (upload mode 3), same as download mode 3 â€“ although the attacker uploads the file purely through ICMP. Also slower and unreliable but more covert than other modes.</li>
<li><em><u>shell</u></em> â€“ request malware to connect back to the attacker with a shell. Initial request is done via ICMP packet containing information including attackerâ€™s host IP and port to where the malware makes a TCP connection.</li>
</ul>
<p><strong>cmd_line&nbsp; - </strong>In <em>exep</em> command, this variable holds the command to be executed on the remote host. While in <em>download</em> and <em>upload</em> command, this variable contains the remote file name.</p>
