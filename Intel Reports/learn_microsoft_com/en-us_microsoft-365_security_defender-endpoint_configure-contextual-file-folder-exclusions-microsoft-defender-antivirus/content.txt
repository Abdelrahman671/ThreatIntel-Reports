<p>This article/section describes the contextual file and folder exclusions capability for Microsoft Defender Antivirus on Windows. This capability allows you to be more specific when you define under which context Microsoft Defender Antivirus shouldn't scan a file or folder, by applying restrictions.</p>
<h2 id="overview">Overview</h2>
<p>Exclusions are primarily intended to mitigate affects on performance. They come at the penalty of reduced protection value. These restrictions allow you to limit this protection reduction by specifying circumstances under which the exclusion should apply. Contextual exclusions aren't suitable for addressing false positives in a reliable way. If you encounter a false positive, you can submit files for analysis through the <a href="https://security.microsoft.com/" data-linktype="external">Microsoft Defender XDR</a> portal (subscription required) or through the <a href="https://www.microsoft.com/wdsi/filesubmission" data-linktype="external">Microsoft Security Intelligence</a> website. For a temporary suppression method, consider creating a custom <em>allow</em> indicator in <a href="indicator-file" data-linktype="relative-path">Microsoft Defender for Endpoint</a>.</p>
<p>There are four restrictions you can apply to limit the applicability of an exclusion:</p>
<ul>
<li><strong>File/folder path type restriction</strong>. You can restrict exclusions to only apply if the target is a file, or a folder by making the intent specific. If the target is a file but the exclusion is specified to be a folder, it will not apply. Conversely, if the target is folder but the exclusion is specified to be a file, the exclusion will apply.</li>
<li><strong>Scan type restriction</strong>. Enables you to define the required scan type for an exclusion to apply. For example, you only want to exclude a certain folder from Full scans but not from a "resource" scan (targeted scan).</li>
<li><strong>Scan trigger type restriction</strong>. You can use this restriction to specify that the exclusion should only apply when the scan was initiated by a specific event:
<ul>
<li>on demand</li>
<li>on access</li>
<li>or originating from behavioral monitoring</li>
</ul>
</li>
<li><strong>Process restriction</strong>. Enables you to define that an exclusion should only apply when a file or folder is being accessed by a specific process.</li>
</ul>
<h2 id="configuring-restrictions">Configuring restrictions</h2>
<p>Restrictions are typically applied by adding the restriction type to the file or folder exclusion path.</p>
<table>

<h3 id="requirements">Requirements</h3>
<p>This capability requires Microsoft Defender Antivirus:</p>
<ul>
<li>Platform: <strong>4.18.2205.7</strong> or later</li>
<li>Engine: <strong>1.1.19300.2</strong> or later</li>
</ul>
<h3 id="syntax">Syntax</h3>
<p>As a starting point, you may already have exclusions in place that you wish to make more specific. To form the exclusion string, first define the path to the file or folder to be excluded, then add the type name and associated value, as shown in the following example.</p>
<p><code>&lt;PATH&gt;\:{TypeName:value,TypeName:value}</code></p>
<p>Keep in mind that <em>all</em> <strong>types</strong> and <strong>values</strong> are case sensitive.</p>
<div class="NOTE">
<p>Note</p>
<p>Conditions inside <code>{}</code> MUST be true for the restriction to match. For example, if you specify two scan triggers this cannot be true, and the exclusion will not apply. To specify two restrictions of the same type, create two separate exclusions.</p>
</div>
<h3 id="examples">Examples</h3>
<p>The following string excludes "c:\documents\design.doc" only if it's a file and only in on-access scans:</p>
<p><code>c:\documents\design.doc\:{PathType:file,ScanTrigger:OnAccess}</code></p>
<p>The following string excludes "c:\documents\design.doc" only if it's scanned (on-access) due to it being accessed by a process having the image name "winword.exe":</p>
<p><code>c:\documents\design.doc\:{Process:"winword.exe"}</code></p>
<p>File and folder paths may contain wildcards, as in the following example:</p>
<p><code>c:\*\*.doc\:{PathType:file,ScanTrigger:OnDemand}</code></p>
<p>The process image path may contain wildcards, as in the following example:</p>
<p><code>c:\documents\design.doc\:{Process:"C:\Program Files*\Microsoft Office\root\Office??\winword.exe"}</code></p>
<h3 id="filefolder-restriction">File/folder restriction</h3>
<p>You can restrict exclusions to only apply if the target is a file or a folder by making the intent specific. If the target is a file but the exclusion is specified to be a folder, the exclusion won't apply. Conversely, if the target is folder but the exclusion is specified to be a file, the exclusion will apply.</p>
<h4 id="filefolder-exclusions-default-behavior">File/folder exclusions default behavior</h4>
<p>If you don't specify any other options, the file/folder is excluded from all types of scans, <em>and</em> the exclusion applies regardless of whether the target is a file or a folder. For more information about customizing exclusions to only apply to a specific scan type, see <a href="#scan-type-restriction" data-linktype="self-bookmark">Scan type restriction</a>.</p>
<div class="NOTE">
<p>Note</p>
<p>Wildcards are supported in file/folder exclusions.</p>
</div>
<h4 id="folders">Folders</h4>
<p>To ensure an exclusion only applies if the target is a folder, not a file you can use the <strong>PathType:folder</strong> restriction. For example:</p>
<p><code>C:\documents\*\:{PathType:folder}</code></p>
<h4 id="files">Files</h4>
<p>To make sure an exclusion only applies if the target is a file, not a folder you can use the PathType: file restriction.</p>
<p>Example:</p>
<p><code>C:\documents\*.mdb\:{PathType:file}</code></p>
<h3 id="scan-type-restriction">Scan type restriction</h3>
<p>By default, exclusions apply to all scan types:</p>
<ul>
<li><strong>resource</strong>: a single file or folder is scanned in a targeted way (for example, right-click, Scan)</li>
<li><strong>quick</strong>: common startup locations utilized by malware, memory and certain registry keys</li>
<li><strong>full</strong>: includes quick scan locations and complete file system (all files and folders)</li>
</ul>
<p>To mitigate performance issues, you can exclude a folder or a set of files from being scanned by a specific scan type. You can also define the required scan type for an exclusion to apply.</p>
<p>To exclude a folder from being scanned only during a full scan, specify a restriction type together with the file or folder exclusion, as in the following example:</p>
<p><code>C:\documents\:{ScanType:full}</code></p>
<p>To exclude a folder from being scanned only during a quick scan, specify a restriction type together with the file or folder exclusion:</p>
<p><code>C:\program.exe\:{ScanType:quick}</code></p>
<p>If you want to make sure this exclusion only applies to a specific file and not a folder (c:\foo.exe could be a folder), also apply the PathType restriction:</p>
<p><code>C:\program.exe\:{ScanType:quick,PathType:file}</code></p>
<h3 id="scan-trigger-restriction">Scan trigger restriction</h3>
<p>By default, basic exclusions apply to all scan triggers. ScanTrigger restriction enables you to specify that the exclusion should only apply when the scan was initiated by a specific event; on demand (including quick, full and targeted scans), on access or originating from behavioral monitoring (including memory scans).</p>
<ul>
<li><strong>OnDemand</strong>: a scan was triggered by a command or admin action. Remember that scheduled quick and full scans also fall under this category.</li>
<li><strong>OnAccess</strong>: a file or folder is opened/written/read/modified (typically considered real-time protection)</li>
<li><strong>BM</strong>: a behavioral trigger causes the behavioral monitoring to scan a specific file</li>
</ul>
<p>To exclude a file or folder and its contents from being scanned only when the file is being scanned after being accessed, define a scan trigger restriction such as the following example:</p>
<p><code>c:\documents\:{ScanTrigger:OnAccess}</code></p>
<h3 id="process-restriction">Process restriction</h3>
<p>This restriction allows you to define that an exclusion should only apply when a file or folder is being accessed by a specific process. A common scenario is when you want to avoid excluding the process as that avoidance would cause Defender Antivirus to ignore other operations by that process. Wildcards are supported in the process name/path.</p>
<div class="NOTE">
<p>Note</p>
<p>Using a large amount of process exclusion restrictions on a machine can adversely affect performance. In addition, if an exclusion is restricted to a certain process or processes, other active processes (such as indexing, backup, updates) can still trigger file scans.</p>
</div>
<p>To exclude a file or folder only when accessed by a specific process, create a normal file or folder exclusion and add the process to restrict the exclusion to. For example:</p>
<p><code>c:\documents\design.doc\:{Process:"winword.exe", Process:"msaccess.exe", Process:"C:\Program Files*\Microsoft Office\root\Office??\winword.exe"}</code></p>
<h3 id="how-to-configure">How to configure</h3>
<p>After constructing your desired contextual exclusions, you can use your existing management tool to configure file and folder exclusions using the string you created.</p>
<p>See: <a href="configure-exclusions-microsoft-defender-antivirus" data-linktype="relative-path">Configure and validate exclusions for Microsoft Defender Antivirus scans</a></p>
