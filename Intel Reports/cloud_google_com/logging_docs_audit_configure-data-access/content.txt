<p>This guide explains how to enable or disable some or all
<a href="/logging/docs/audit#data-access">Data Access audit logs</a> in your
Google Cloud projects, billing accounts, folders, and organizations by
using the Google Cloud console or the API.</p>

<h2 id="before-you-begin" data-text="Before you begin" tabindex="-1">Before you begin</h2>

<p>Before you proceed with configuring Data Access audit logs, understand the
following information:</p>


<aside class="special"><strong>Important:</strong><span> Data Access audit logs volume can be large. Enabling Data Access logs
might result in your Google Cloud project being charged for the additional
logs usage. 

For pricing information, see
<a href="/stackdriver/pricing#logging-costs">Google Cloud Observability pricing: Cloud Logging</a>.
 </span></aside><aside class="special"><strong>Important:</strong><span> If you have configured Data Access logs to track access to objects,
<a href="/storage/docs/request-endpoints#cookieauth">authenticated browser downloads</a>
from <code translate="no" dir="ltr">storage.cloud.google.com</code> may result in a <code translate="no" dir="ltr">403</code> response. For solutions
to this issue, see the
<a href="/storage/docs/troubleshooting#trouble-download-storage-cloud">Cloud Storage troubleshooting guide</a>.</span></aside>
<ul>
<li><p>Data Access audit logs&mdash;except for BigQuery&mdash;are disabled
by default. If you want Data Access audit logs to be written for
Google Cloud services other than BigQuery, you must explicitly
enable them.</p></li>
<li><p>Data Access audit logs are stored in the
<a href="/logging/docs/routing/overview#default-bucket"><code translate="no" dir="ltr">_Default</code></a> bucket unless
you&#39;ve routed them elsewhere. For more information, see
<a href="/logging/docs/audit#storing_and_routing_audit_logs">Storing and routing audit logs</a>.</p></li>
<li><p>Data Access audit logs help Google Support troubleshoot issues with your
account. Therefore, we recommend enabling Data Access audit logs when
possible.</p></li>
</ul>

<h2 id="configuration_overview" data-text="Configuration overview" tabindex="-1">Configuration overview</h2>

<p>You can configure how Data Access audit logs are enabled for your
Google Cloud resources and services:</p>

<ul>
<li><p>Organizations: You can enable and configure Data Access audit logs in an
organization, which applies to all the existing and new
Google Cloud projects and folders in the organization.</p></li>
<li><p>Folders: You can enable and configure Data Access audit logs in a
folder, which applies to all the existing and new Google Cloud projects in
the folder. You can&#39;t disable a Data Access audit log that was enabled in
the project&#39;s parent organization.</p></li>
<li><p>Projects: You can configure Data Access audit logs for an individual
Google Cloud project. You can&#39;t disable a Data Access audit log that
was enabled in a parent organization or folder.</p></li>
<li><p>Billing accounts: To configure Data Access audit logs for billing
accounts, use the Google Cloud CLI. For more information about using
the gcloud CLI with Data Access audit logs and billing accounts,
see the <a href="/sdk/gcloud/reference/beta/billing/accounts/set-iam-policy"><code translate="no" dir="ltr">gcloud beta billing accounts set-iam-policy</code></a>
documentation.</p></li>
<li><p>Default configurations: You can specify a default Data Access audit log
configuration in an organization, folder, or Google Cloud project that
applies to future Google Cloud services that begin to produce Data
Access audit logs. For instructions, see
<a href="#config-console-default">Set the default configuration</a>.</p></li>
<li><p>Permission types: You can specify that Google Cloud APIs which only
check for a certain type of permission emit an audit log. For more
information, see the <a href="#permission-types">Permission types</a> section of this
page.</p></li>
<li><p>Exempted principals: You can exempt specific principals from
having their data accesses recorded. For example, you can exempt your
internal testing accounts from having their Cloud Monitoring operations
recorded. For a list of valid principals, including users and groups,
see the
<a href="/resource-manager/reference/rest/Shared.Types/Binding#FIELDS.members"><code translate="no" dir="ltr">Binding</code> type reference</a>.</p></li>
</ul>

<p>You can configure your Data Access audit logs through the IAM
<strong>Audit Logs</strong> page of the Google Cloud console, or by using the API. These methods
are explained in the sections below.</p>

<h3 id="permission-types" data-text="Permission types" tabindex="-1">Permission types</h3>

<p>API methods check IAM permissions. IAM permissions
have a <code translate="no" dir="ltr">type</code> property, whose value is one of the following permission
types:</p>

<ul>
<li><p><code translate="no" dir="ltr">ADMIN_READ</code>: IAM permissions of this type are checked for
  Google Cloud API methods that read metadata or configuration
  information.</p></li>
<li><p><code translate="no" dir="ltr">DATA_READ</code>: IAM permissions of this type are checked for
  Google Cloud API methods that read user-provided data.</p></li>
<li><p><code translate="no" dir="ltr">DATA_WRITE</code>: IAM permissions of this type are checked for
  Google Cloud API methods that write user-provided data.</p></li>
<li><p><code translate="no" dir="ltr">ADMIN_WRITE</code>: IAM permissions of this type are checked for
  Google Cloud API methods that write metadata or configuration
  information. The audit logs associated with this type,
  <a href="/logging/docs/audit#admin-activity">Admin Activity audit logs</a>,
  are on by default and can&#39;t be disabled.</p></li>
</ul>

<p>Most Google Cloud APIs only check for a single IAM
permission and enabling the type associated with the permission for the service
enables the audit log associated with the method being called.</p>

<p>You can enable or disable permission types for services <a href="#config-console">by using the
Google Cloud console</a> or
<a href="#config-api">by invoking the API</a>.</p>

<p>The following sections generally describe other ways in which Google Cloud
API methods check for IAM permissions. For service-specific
information about which methods are checked for which permission types, see
the <a href="/logging/docs/audit/services">service&#39;s audit logging documentation</a>.</p>

<h4 id="multiple-permissions-data-access" data-text="IAM permissions checking for Data Access permission types" tabindex="-1">IAM permissions checking for Data Access permission types</h4>

<p>Some Google Cloud API methods check that a principal has multiple
IAM permissions with different permission types. An audit log is
written when one of the permission types associated with an IAM
permission is enabled on the project the API call is targeting.</p>

<p>For example, an API method might check that the principal issuing an API
request has the permissions <code translate="no" dir="ltr">example.resource.get</code> (<code translate="no" dir="ltr">DATA_READ</code>) and
<code translate="no" dir="ltr">example.resource.write</code> (<code translate="no" dir="ltr">DATA_WRITE</code>). The project only needs either
<code translate="no" dir="ltr">DATA_WRITE</code> or <code translate="no" dir="ltr">DATA_READ</code> enabled for the service to emit the audit log when
issuing the call.</p>

<h4 id="multiple-permissions-admin-activity" data-text="Admin Activity and Data Access IAM permission types checked" tabindex="-1">Admin Activity and Data Access IAM permission types checked</h4>

<p>Some Google Cloud API methods check for both an IAM
permission that has the <code translate="no" dir="ltr">ADMIN_WRITE</code> permission type, and one or more
permissions that have a Data Access permission type. These types of API calls
emit <a href="/logging/docs/audit#admin-activity">Admin Activity audit logs</a>, which
are on by default and can&#39;t be disabled.</p>

<h4 id="other-service-permission" data-text="API method checking for IAM permissions not owned by service" tabindex="-1">API method checking for IAM permissions not owned by service</h4>

<p>Some Google Cloud services have API methods that check an
IAM permission owned by a different service. In this case, the
permission type must be enabled for the service that owns the IAM
permission in order to enable the audit log associated with the API method
being called.</p>

<p>For example, Cloud Billing has an API method that checks for an <code translate="no" dir="ltr">ADMIN_READ</code>
permission type that is owned by Resource Manager. <code translate="no" dir="ltr">ADMIN_READ</code> must be
enabled for the service <code translate="no" dir="ltr">cloudresourcemanager.googleapis.com</code> to enable
the audit log associated with the Cloud Billing API.</p>

<h4 id="varied-iam-permissions" data-text="Same API method checks for different IAM permissions" tabindex="-1">Same API method checks for different IAM permissions</h4>

<p>Some Google Cloud APIs check different IAM permissions
with different permission types depending on how the API methods are called. An
audit log is written when one of the IAM permission types is
enabled on the project the API call is targeting.</p>

<p>For example, Spanner has an API method that sometimes checks for an
IAM permission with the <code translate="no" dir="ltr">DATA_WRITE</code> type, and sometimes checks
for an IAM permission with the <code translate="no" dir="ltr">DATA_READ</code> type depending on how
the method is called. In this case, enabling <code translate="no" dir="ltr">DATA_WRITE</code> for Spanner
on the project the API call only enables the audit log associated with the API
when the IAM permission with the <code translate="no" dir="ltr">DATA_WRITE</code> type is checked.</p>

<h3 id="service-config" data-text="Service-specific configurations" tabindex="-1">Service-specific configurations</h3>

<p>If there is both a Google Cloud service-wide (<code translate="no" dir="ltr">allServices</code>) configuration
and a configuration for a specific Google Cloud service, then the
resulting configuration for the service is the union of the two configurations.
In other words:</p>

<ul>
<li><p>You can enable Data Access audit logs for specific Google Cloud
services, but you can&#39;t disable Data Access audit logs for
Google Cloud services that are enabled in the broader configuration.</p></li>
<li><p>You can add additional kinds of information to a Google Cloud
service&#39;s Data Access audit log, but you can&#39;t remove kinds of information
that are specified in the broader configuration.</p></li>
<li><p>You can add principals to exemption lists, but you can&#39;t remove them
from exemption lists in the broader configuration.</p></li>
<li><p>For BigQuery Data Transfer Service, Data Access audit log configuration is
inherited from your default audit log configuration.</p></li>
</ul>

<h3 id="project-and-organization" data-text="Google Cloud resource configurations" tabindex="-1">Google Cloud resource configurations</h3>

<p>You can configure Data Access audit logs for Google Cloud projects,
billing accounts, folders, and organizations. If there is a configuration for a
Google Cloud service across the hierarchy, then the resulting
configuration is the union of the configurations. In other words, at the
Google Cloud project level:</p>

<ul>
<li><p>You can enable logs for a Google Cloud service, but you can&#39;t disable
logs for a Google Cloud service that is enabled in a parent
organization or folder.</p></li>
<li><p>You can enable kinds of information, but you can&#39;t disable kinds of
information that are enabled in a parent organization or folder.</p></li>
<li><p>You can add principals to exemption lists, but you can&#39;t remove them
from exemption lists in a parent organization or folder.</p></li>
<li><p>At a parent organization or folder level, you can enable Data Access audit
logs for a Google Cloud project within that organization or folder, even
if Data Access audit logs haven&#39;t been configured in the
Google Cloud project.</p></li>
</ul>

<h2 id="access-control" data-text="Access control" tabindex="-1">Access control</h2>

<p>Identity and Access Management roles and permissions govern access to Logging data,
including viewing and managing the
<a href="/iam/docs/policies">IAM policies</a> underlying Data Access audit
logging configurations.</p>

<p>To view or set the policies associated with Data Access configuration, you need
a role with permissions at the appropriate resource level. For instructions on
granting these resource-level roles, see
<a href="/iam/docs/granting-changing-revoking-access">Manage access to Google Cloud projects, folders, and organizations</a>.</p>

<ul>
<li><p>To set IAM policies, you need a role with the
<code translate="no" dir="ltr">resourcemanager.<var translate="no">RESOURCE_TYPE</var>.setIamPolicy</code>
permission.</p></li>
<li><p>To view IAM policies, you need a role with the
<code translate="no" dir="ltr">resourcemanager.<var translate="no">RESOURCE_TYPE</var>.getIamPolicy</code>
permission.</p>
<aside class="special"><strong>Important:</strong><span> When viewing Data Access configs in the Google Cloud console at
the Google Cloud project or folder level, if you don&#39;t have the
<code translate="no" dir="ltr">getIamPolicy</code> permission for a parent organization or folder, then
any Data Access configurations present in those parent entities isn&#39;t shown
in the Google Cloud console. Those Data Access configurations still apply
to all API calls in the Google Cloud project or folder you are viewing.</span></aside></li>
</ul>

<p>For the list of the permissions and roles you need to view Data Access audit
logs, see
<a href="/logging/docs/access-control">Access control with IAM</a>.</p>

<h2 id="config-console" data-text="Configure Data Access audit logs with the Google Cloud console" tabindex="-1">Configure Data Access audit logs with the Google Cloud console</h2>

<p>This section explains how to use the Google Cloud console to configure Data
Access audit logs.</p>

<p>You can also use the API or the Google Cloud CLI to perform these tasks
programmatically; see
<a href="#config-api">Configure Data Access audit logs with the API</a> for details.</p>

<p>To access audit log configuration options in the Google Cloud console, follow
these steps:</p>

<ol>
<li>

<p>In the Google Cloud console, go to the <b>Audit Logs</b> page:
   





<p><a class="button button-primary"
        href="https://console.cloud.google.com/iam-admin/audit"
        target="console" track-name="consoleLink"
        track-type="task"
        track-metadata-position="body"
        track-metadata-event-detail="iam-admin/audit"
        >Go to <b>Audit Logs</b></a>
</p>
</p>
<p>If you use the search bar to find this page, then select the result whose subheading is
<b>IAM & Admin</b>.</p></li>
<li><p>Select an existing Google Cloud project, folder, or organization.</p></li>
</ol>

<h3 id="config-console-enable" data-text="Enable audit logs" tabindex="-1">Enable audit logs</h3>

<p>To enable Data Access audit logs, do the following:</p>

<ol>
<li><p>In the <strong>Data Access audit logs configuration</strong> table, select one or more
Google Cloud services from the <strong>Service</strong> column.</p></li>
<li><p>In the <strong>Log Types</strong> tab, select the Data Access audit log types that you
want to enable for your selected services.</p></li>
<li><p>Click <strong>Save</strong>.</p></li>
</ol>

<p>Where you have successfully enabled audit logs, the table includes a
<i class="material-icons" aria-hidden="true" translate="no">check_circle</i> <strong>Check</strong> icon.</p>

<p>In the following example, you see that, for the Access Approval service, the
<strong>Data Read</strong> audit log type is enabled:</p>

<p><img src="/static/logging/docs/images/audit-log-config.png" alt="Audit Logs config" class="screenshot"> </p>

<p>You can also enable audit logs for all Google Cloud services that produce
Data Access audit logs. In the <strong>Data Access audit logs configuration</strong> table,
select all Google Cloud services.</p>

<p>Note that this bulk configuration method applies only to the Google Cloud
services that are currently available for your resource. If a new
Google Cloud service is added, it inherits your
<a href="#config-console-default">default audit configuration</a>.</p>

<h3 id="config-console-disable" data-text="Disable Data Access audit logs" tabindex="-1">Disable Data Access audit logs</h3>

<p>To disable Data Access audit logs, do the following:</p>

<ol>
<li><p>In the <strong>Data Access audit logs configuration</strong> table, select one or more
Google Cloud services.</p></li>
<li><p>In the <strong>Log Types</strong> tab in the information panel, select the Data Access
audit log types that you want to disable for your selected services.</p></li>
<li><p>Click <strong>Save</strong>.</p></li>
</ol>

<p>Where you&#39;ve successfully disabled Data Access audit logs, the table indicates
this with a dash. Any enabled Data Access audit logs are indicated with a
<i class="material-icons" aria-hidden="true" translate="no">check_circle</i> <strong>Check</strong> icon.</p>

<h3 id="config-console-exempt" data-text="Set exemptions" tabindex="-1">Set exemptions</h3>

<p>You can set exemptions to let you control which principals generate Data
Access audit logs for particular services. When you add an exempted principal,
audit logs aren&#39;t created for them for the selected log types.</p>
<aside class="caution"><strong>Caution:</strong><span> If you add an exempted principal to a service for an audit log type,
then this enables the audit log type even if it was previously disabled.</span></aside>
<p>To set exemptions, do the following:</p>

<ol>
<li><p>In the <strong>Data Access audit logs configuration</strong> table, select a
Google Cloud service from the <strong>Service</strong> column.</p></li>
<li><p>Select the <strong>Exempted Principals</strong> tab in the information panel.</p></li>
<li><p>In <strong>Add exempted principal</strong>, enter the principal that you want to exempt
from generating Data Access audit logs for your selected service.</p>

<p>You can add multiple principals by
clicking the <strong>Add exempted principal</strong> button as many times as needed.</p>

<p>For a list of valid principals, including users and groups,
see the <a href="/resource-manager/reference/rest/Shared.Types/Binding#FIELDS.members"><code translate="no" dir="ltr">Binding</code> type reference</a>.</p></li>
<li><p>In <strong>Disabled Log Types</strong>, select the Data Access audit log types
that you want to disable.</p></li>
<li><p>Click <strong>Save</strong>.</p></li>
</ol>

<p>Where you have successfully added exempted principals to a service, the <strong>Data
Access audit logs configuration</strong> table indicates this with a number under the
<strong>Exempted principals</strong> column.</p>

<p>To remove a principal from your exemption list, do the following:</p>

<ol>
<li><p>In the <strong>Data Access audit logs configuration</strong> table, select a
Google Cloud service from the <strong>Service</strong> column.</p></li>
<li><p>Select the <strong>Exempted Principals</strong> tab in the information panel.</p></li>
<li><p>Hover over a principal name and then select the delete
icon <i class="material-icons" aria-hidden="true" translate="no"> delete </i> that appears.</p></li>
<li><p>After the principal&#39;s name is shown in strikethrough text, click <strong>Save</strong>.</p></li>
</ol>

<p>To edit the information for an exempted principal, do the following:</p>

<ol>
<li><p>In the <strong>Data Access audit logs configuration</strong> table, select a
Google Cloud service from the <strong>Service</strong> column.</p></li>
<li><p>Select the <strong>Exempted Principals</strong> tab in the information panel.</p></li>
<li><p>Expand <i class="material-icons" aria-hidden="true" translate="no"> expand_more </i> the
principal name.</p></li>
<li><p>Select or deselect the Data Access audit log types as appropriate for the
principal.</p></li>
<li><p>Click <strong>Save</strong>.</p></li>
</ol>

<h3 id="config-console-default" data-text="Set the default configuration" tabindex="-1">Set the default configuration</h3>

<p>You can set a configuration that all new and existing Google Cloud
services in your Google Cloud project, folder, or organization inherit.
Setting this default configuration applies if a new Google Cloud service
becomes available and principals in your organization begin using it: the
service inherits the audit logging policy that you have already set for other
Google Cloud services, ensuring that Data Access audit logs are captured.</p>

<p>To set or edit the default configuration, do the following:</p>

<ol>
<li><p>Click <strong>Set Default Configuration</strong>.</p></li>
<li><p>In the <strong>Log Types</strong> tab in the information panel, select the Data Access
audit log types that you want to enable or disable.</p></li>
<li><p>Click <strong>Save</strong>.</p></li>
<li><p>Select the <strong>Exempted Principals</strong> tab in the information panel.</p></li>
<li><p>In <strong>Add exempted principal</strong>, enter the principal that you want to exempt
from generating Data Access audit logs for your selected service.</p>

<p>You can add multiple principals by
clicking the <strong>Add exempted principal</strong> button as many times as needed.</p>

<p>For a list of valid principals, including users and groups,
see the <a href="/resource-manager/reference/rest/Shared.Types/Binding#FIELDS.members"><code translate="no" dir="ltr">Binding</code> type reference</a>.</p></li>
<li><p>In <strong>Disabled Log Types</strong>, select the Data Access audit log types
that you want to disable.</p></li>
<li><p>Click <strong>Save</strong>.</p></li>
</ol>

<h2 id="config-api" data-text="Configure Data Access audit logs with the API" tabindex="-1">Configure Data Access audit logs with the API</h2>

<p>This section explains how to use the API and the gcloud CLI to
configure Data Access audit logs programmatically.</p>

<p>Many of these tasks can also be performed by using the Google Cloud console;
for instructions, see
<a href="#config-console">Configure Data Access audit logs with the Google Cloud console</a>
on this page.</p>

<h3 id="policy-objects" data-text="IAM policy objects" tabindex="-1">IAM policy objects</h3>

<p>To configure Data Access audit logs using the API, you must edit the
IAM policy associated with your Google Cloud project, folder,
or organization. The audit log configuration is in the <code translate="no" dir="ltr">auditConfigs</code> section of
the policy:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">&#34;auditConfigs&#34;: [
  {
    object(AuditConfig)
  }
]
</code></pre>
<p>For details, see the IAM
<a href="/resource-manager/reference/rest/Shared.Types/Policy">Policy</a> type.</p>

<p>The following sections describe the <code translate="no" dir="ltr">AuditConfig</code> object in more detail.
For the API and gcloud CLI commands used to change the configuration,
see <a href="#getiampolicy-setiampolicy">getIamPolicy and setIamPolicy</a></p>

<h3 id="auditconfig_objects" data-text="AuditConfig objects" tabindex="-1"><code translate="no" dir="ltr">AuditConfig</code> objects</h3>

<p>The audit log configuration consists of a list of
<a href="/resource-manager/reference/rest/Shared.Types/AuditConfig"><code translate="no" dir="ltr">AuditConfig</code></a>
objects. Each object configures the logs for one service, or it establishes a
broader configuration for all services. Each object looks like the following:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">{
  &#34;service&#34;: <var translate="no">SERVICE_NAME</var>,
  &#34;auditLogConfigs&#34;: [
    {
      &#34;logType&#34;: &#34;ADMIN_READ&#34;
      &#34;exemptedMembers&#34;: [ <var translate="no">PRINCIPAL</var>,]
    },
    {
      &#34;logType&#34;: &#34;DATA_READ&#34;
      &#34;exemptedMembers&#34;: [ <var translate="no">PRINCIPAL</var>,]
    },
    {
      &#34;logType&#34;: &#34;DATA_WRITE&#34;
      &#34;exemptedMembers&#34;: [ <var translate="no">PRINCIPAL</var>,]
    },
  ]
},
</code></pre>
<p><var translate="no">SERVICE_NAME</var> has a value such as <code translate="no" dir="ltr">&quot;appengine.googleapis.com&quot;</code>, or it is the
special value, <code translate="no" dir="ltr">&quot;allServices&quot;</code>. If a configuration doesn&#39;t mention a particular
service, then the broader configuration is used for that service. If there is no
configuration, then Data Access audit logs aren&#39;t enabled for that service.
For a list of the service names, see
<a href="/logging/docs/api/v2/resource-list#service-names">Log services</a>.</p>

<p>The <code translate="no" dir="ltr">auditLogConfigs</code> section of the <code translate="no" dir="ltr">AuditConfig</code> object is a list of 0 to 3
objects, each of which configures one kind of audit log information. If you omit
one of the kinds from the list, then that kind of information isn&#39;t enabled
for the service.</p>

<p><var translate="no">PRINCIPAL</var> is a user for whom Data Access audit logs isn&#39;t collected. The
<a href="/resource-manager/reference/rest/Shared.Types/Binding#FIELDS.members"><code translate="no" dir="ltr">Binding</code> type</a> describes different kinds of principals, including
users and groups, but not all of those can be used to configure Data Access
audit logs.</p>

<p>Following is an example of an audit configuration in both JSON and YAML formats.
The YAML format is the default when using the Google Cloud CLI.</p>
<div class="ds-selector-tabs" data-ds-scope="code-sample">
<section><h3 id="json" data-text="JSON" tabindex="-1">JSON</h3><pre translate="no" dir="ltr">
"auditConfigs": [
  {
    "auditLogConfigs": [
      {
        "logType": "ADMIN_READ"
      },
      {
        "logType": "DATA_WRITE"
      },
      {
        "logType": "DATA_READ"
      }
    ],
    "service": "allServices"
  },
  {
    "auditLogConfigs": [
      {
        "exemptedMembers": [
          "499862534253-compute@developer.gserviceaccount.com"
        ],
        "logType": "ADMIN_READ"
      }
    ],
    "service": "cloudsql.googleapis.com"
  }
],
</pre></section>
<section><h3 id="yaml" data-text="YAML" tabindex="-1">YAML</h3><pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">auditConfigs:
- auditLogConfigs:
  - logType: ADMIN_READ
  - logType: DATA_WRITE
  - logType: DATA_READ
  service: allServices
- auditLogConfigs:
  - exemptedMembers:
    - 499862534253-compute@developer.gserviceaccount.com
    logType: ADMIN_READ
  service: cloudsql.googleapis.com
</code></pre></section>
</div>
<h3 id="common_configurations" data-text="Common configurations" tabindex="-1">Common configurations</h3>

<p>Following are some common audit log configurations for Google Cloud projects.</p>

<h4 id="enable" data-text="Enable all Data Access audit logs" tabindex="-1">Enable all Data Access audit logs</h4>

<p>The following <code translate="no" dir="ltr">auditConfigs</code> section enables Data Access audit logs for all
services and principals:</p>
<div class="ds-selector-tabs" data-ds-scope="code-sample">
<section><h3 id="json_1" data-text="JSON" tabindex="-1">JSON</h3><pre translate="no" dir="ltr">
"auditConfigs": [
      {
        "service": "allServices",
        "auditLogConfigs": [
          { "logType": "ADMIN_READ" },
          { "logType": "DATA_READ"  },
          { "logType": "DATA_WRITE" },
        ]
      },
    ]
</pre></section>
<section><h3 id="yaml_1" data-text="YAML" tabindex="-1">YAML</h3><pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">auditConfigs:
- auditLogConfigs:
  - logType: ADMIN_READ
  - logType: DATA_WRITE
  - logType: DATA_READ
  service: allServices
</code></pre></section>
</div>
<h4 id="enabling-one-service" data-text="Enable one service and information kind" tabindex="-1">Enable one service and information kind</h4>

<p>The following configuration enables <code translate="no" dir="ltr">DATA_WRITE</code> Data Access audit logs for
Cloud SQL:</p>
<div class="ds-selector-tabs" data-ds-scope="code-sample">
<section><h3 id="json_2" data-text="JSON" tabindex="-1">JSON</h3><pre translate="no" dir="ltr">
"auditConfigs": [
  {
    "service": "cloudsql.googleapis.com",
    "auditLogConfigs": [
      { "logType": "DATA_WRITE" },
    ]
  },
]
</pre></section>
<section><h3 id="yaml_2" data-text="YAML" tabindex="-1">YAML</h3><pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">auditConfigs:
- auditLogConfigs:
  - logType: DATA_WRITE
  service: cloudsql.googleapis.com
</code></pre></section>
</div>
<h4 id="disabling" data-text="Disable all Data Access audit logs" tabindex="-1">Disable all Data Access audit logs</h4>

<p>To disable all Data Access audit logs (except BigQuery) in a
Google Cloud project, include an empty <code translate="no" dir="ltr">auditConfigs:</code> section in your new
IAM policy:</p>
<div class="ds-selector-tabs" data-ds-scope="code-sample">
<section><h3 id="json_3" data-text="JSON" tabindex="-1">JSON</h3><pre translate="no" dir="ltr">
"auditConfigs": [],
</pre></section>
<section><h3 id="yaml_3" data-text="YAML" tabindex="-1">YAML</h3><pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">auditConfigs:
</code></pre></section>
</div>
<p>If you remove the <code translate="no" dir="ltr">auditConfigs</code> section entirely from your new policy,
then <code translate="no" dir="ltr">setIamPolicy</code> doesn&#39;t change the existing Data Access audit logs
configuration. For more information, see
<a href="#updatemask">The setIamPolicy update mask</a>.</p>

<p>BigQuery Data Access audit logs can&#39;t be disabled.</p>

<h3 id="getiampolicy-setiampolicy" data-text="getIamPolicy and setIamPolicy" tabindex="-1"><code translate="no" dir="ltr">getIamPolicy</code> and <code translate="no" dir="ltr">setIamPolicy</code></h3>

<p>You use the Cloud Resource Manager API <code translate="no" dir="ltr">getIamPolicy</code> and <code translate="no" dir="ltr">setIamPolicy</code> methods to read
and write your IAM policy. You have several choices for the
specific methods to use:</p>

<ul>
<li><p>The <a href="/resource-manager/reference/rest/v3/projects">Cloud Resource Manager API</a> has the following methods:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">projects.getIamPolicy
projects.setIamPolicy
organizations.getIamPolicy
organizations.setIamPolicy
</code></pre></li>
<li><p>The <a href="/sdk/gcloud/reference/projects">Google Cloud CLI</a> has the following
Resource Manager commands:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">gcloud projects get-iam-policy
gcloud projects set-iam-policy
gcloud resource-manager folders get-iam-policy
gcloud resource-manager folders set-iam-policy
gcloud organizations get-iam-policy
gcloud organizations set-iam-policy
gcloud beta billing accounts get-iam-policy
gcloud beta billing accounts set-iam-policy
</code></pre></li>
</ul>
<aside class="note"><strong>Note:</strong><span> When you call <code translate="no" dir="ltr">projects.getIamPolicy</code> or <code translate="no" dir="ltr">projects get-iam-policy</code>, the
result shows only the policies set in the Google Cloud project, not the
policies inherited from a parent organization or folder.</span></aside>
<p>Regardless of your choice, follow these three steps:</p>

<ol>
<li><strong>Read</strong> the current policy using one of the <code translate="no" dir="ltr">getIamPolicy</code> methods.
Save the policy to a temporary file.</li>
<li><strong>Edit</strong> the policy in the temporary file.
<strong>Change (or add) only the <code translate="no" dir="ltr">auditConfigs</code> section.</strong></li>
<li><strong>Write</strong> the edited policy in the temporary file,
using one of the <code translate="no" dir="ltr">setIamPolicy</code> methods.</li>
</ol>

<p><code translate="no" dir="ltr">setIamPolicy</code> fails if Resource Manager detects that someone
else changed the policy after you read it in the first step. If this happens,
then repeat the three steps.</p>
<aside class="caution"><strong>Caution:</strong><span> Your edited IAM policy replaces the current policy.
Changing parts of your policy not related to audit logging might make your
Google Cloud project inaccessible.</span></aside>
<h4 id="example" data-text="Examples" tabindex="-1">Examples</h4>

<p>The following examples demonstrate how to configure your project&#39;s Data Access
audit logs using the <code translate="no" dir="ltr">gcloud</code> command and the Cloud Resource Manager API.</p>

<p>To configure organization Data Access audit logs, replace the &quot;projects&quot; version
of the commands and API methods with the &quot;organizations&quot; version.</p>
<aside class="caution"><strong>Caution:</strong><span> Follow the instructions carefully. You are editing a policy object that
contains critical information about who can access your resource. Accidentally
altering that information could make your resource unusable.</span></aside><div class="ds-selector-tabs" data-ds-scope="code-sample">
<section><h3 id="gcloud" data-text="gcloud" tabindex="-1">gcloud</h3><p>To configure your Data Access audit logs using
the <a href="/sdk/gcloud/reference/projects"><code translate="no" dir="ltr">gcloud projects</code></a> command,
do the following:</p>

<ol>
<li><p><strong>Read</strong> your project&#39;s IAM policy and store it in a
file:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">gcloud projects get-iam-policy <var translate="no">PROJECT_ID</var> &gt; /tmp/policy.yaml
</code></pre>
<p>The returned policy is shown below. This policy doesn&#39;t yet have an
<code translate="no" dir="ltr">auditConfigs</code> section:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">bindings:
- members:
  - user:colleague@example.com
  role: roles/editor
- members:
  - user:myself@example.com
  role: roles/owner
etag: BwVM-FDzeYM=
version: 1
</code></pre></li>
<li><p><strong>Edit</strong> your policy in <code translate="no" dir="ltr">/tmp/policy.yaml</code>, adding or changing <em>only</em>
the Data Access audit logs configuration.</p>
<aside class="caution"><strong>Caution:</strong><span> You must preserve the <code translate="no" dir="ltr">bindings:</code> and <code translate="no" dir="ltr">etag:</code> sections without
changes. Failure to do so might cause your Google Cloud project to
become unusable. For more information, see
<a href="#updatemask">setIamPolicy update mask</a> on this page.</span></aside>
<p>An example of your edited policy, which enables Cloud SQL
data-write Data Access audit logs, is shown below. Four lines have
been added to the beginning:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">auditConfigs:
- auditLogConfigs:
  - logType: DATA_WRITE
  service: cloudsql.googleapis.com
bindings:
- members:
  - user:colleague@example.com
  role: roles/editor
- members:
  - user:myself@example.com
  role: roles/owner
etag: BwVM-FDzeYM=
version: 1
</code></pre></li>
<li><p><strong>Write</strong> your new IAM policy:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">gcloud projects set-iam-policy <var translate="no">PROJECT_ID</var> /tmp/policy.yaml
</code></pre>
<p>If the preceding command reports a conflict with another change, then
repeat these steps, starting with the first step.</p></li>
</ol>

<p><strong>JSON</strong></p>

<p>To work with your IAM policy in JSON format instead of YAML,
substitute the following <code translate="no" dir="ltr">gcloud</code> commands into the example:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">gcloud projects get-iam-policy <var translate="no">PROJECT_ID</var> --format=json &gt;/tmp/policy.json
gcloud projects set-iam-policy <var translate="no">PROJECT_ID</var> /tmp/policy.json
</code></pre></section>
<section><h3 id="api" data-text="API" tabindex="-1">API</h3><p>To configure your Data Access audit logs using the
<a href="/resource-manager/reference/rest/v3/projects">Cloud Resource Manager API</a>, do the following:</p>

<ol>
<li><p><strong>Read</strong> your project&#39;s IAM policy, specifying the
following parameters
to the <a href="/resource-manager/reference/rest/v3/projects/getIamPolicy">getIamPolicy API method</a>:</p>

<ul>
<li><strong>resource</strong>: <code translate="no" dir="ltr">projects/<var translate="no">PROJECT_ID</var></code></li>
<li><strong>Request body</strong>: <em>empty</em></li>
</ul>

<p>The method returns the current policy object, shown below. This
project&#39;s policy doesn&#39;t yet have an <code translate="no" dir="ltr">auditConfigs</code> section:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">{
  &#34;version&#34;: 1,
  &#34;etag&#34;: &#34;BwXqwxkr40M=&#34;,
  &#34;bindings&#34;: [
    {
      &#34;role&#34;: &#34;roles/owner&#34;,
      &#34;members&#34;: [
        &#34;user:myself@example.com&#34;
      ]
    }
  ]
}
</code></pre></li>
<li><p><strong>Edit</strong> the current policy:</p>

<ul>
<li><p>Change or add the <code translate="no" dir="ltr">auditConfigs</code> section.</p>

<p>To disable your Data Access audit logs, include an
empty value
for the section: <code translate="no" dir="ltr">auditConfigs:[]</code>.</p></li>
<li><p>Preserve the value of <code translate="no" dir="ltr">etag</code>.</p></li>
</ul>

<p>You can also remove all other information from the new policy
object, as long as you&#39;re careful to set <code translate="no" dir="ltr">updateMask</code> in the next step.
The edited policy, which enables Cloud SQL data-write audit
logs, is shown below:</p>
<pre class="prettyprint" translate="no" dir="ltr"><code translate="no" dir="ltr">{
  &#34;policy&#34;: {
    &#34;auditConfigs&#34;: [
      {
        &#34;auditLogConfigs&#34;: [
          {
            &#34;logType&#34;: &#34;DATA_WRITE&#34;
          }
        ],
        &#34;service&#34;: &#34;cloudsql.googleapis.com&#34;
      }
    ],
    &#34;etag&#34;: &#34;BwXqwxkr40M=&#34;
  },
  &#34;updateMask&#34;: &#34;auditConfigs,etag&#34;
}
</code></pre></li>
<li><p><strong>Write</strong> the new policy using
the <a href="/resource-manager/reference/rest/v3/projects/setIamPolicy">setIamPolicy API method</a>, specifying the following
parameters:</p>

<ul>
<li><strong>resource</strong>: <code translate="no" dir="ltr">projects/<var translate="no">PROJECT_ID</var></code></li>
<li><strong>Request body</strong>: Include the edited policy.</li>
</ul></li>
</ol>
<aside class="caution"><strong>Caution:</strong><span> You must specify the <code translate="no" dir="ltr">updateMask</code> parameter as shown. If you do not
specify <code translate="no" dir="ltr">updateMask</code>, you might lose all access to your
Google Cloud project. For more information, see the following
section, <a href="#updatemask">The setIamPolicy update mask</a>.</span></aside></section>
</div>
<h4 id="updatemask" data-text="The setIamPolicy update mask" tabindex="-1">The <code translate="no" dir="ltr">setIamPolicy</code> update mask</h4>

<p>This section explains the importance of the <code translate="no" dir="ltr">updateMask</code> parameter in the
<code translate="no" dir="ltr">setIamPolicy</code> method, and explains why you must be careful with the
gcloud CLI <code translate="no" dir="ltr">set-iam-policy</code> command so that you don&#39;t cause
accidental harm to your Google Cloud project or organization.</p>

<p>The <a href="/resource-manager/reference/rest/v3/projects/setIamPolicy"><code translate="no" dir="ltr">setIamPolicy API method</code></a> uses an <code translate="no" dir="ltr">updateMask</code> parameter to
control which policy fields are updated. For example, if the mask does not
contain <code translate="no" dir="ltr">bindings</code>, then you can&#39;t accidentally change that policy section. On
the other hand, if the mask does contain <code translate="no" dir="ltr">bindings</code>, then that section is
<em>always</em> updated. If you don&#39;t include an updated value for <code translate="no" dir="ltr">bindings</code>, then
that section is removed entirely from the policy.</p>

<p>The <code translate="no" dir="ltr">gcloud projects set-iam-policy</code> command, which calls <code translate="no" dir="ltr">setIamPolicy</code>,
doesn&#39;t let you specify the <code translate="no" dir="ltr">updateMask</code> parameter. Instead, the command
computes a value for <code translate="no" dir="ltr">updateMask</code> in the following way:</p>

<ul>
<li>The <code translate="no" dir="ltr">updateMask</code> always contains the fields <code translate="no" dir="ltr">bindings</code> and <code translate="no" dir="ltr">etag</code>.</li>
<li>If the policy object supplied in <code translate="no" dir="ltr">set-iam-policy</code> contains any other
top-level fields&mdash;such as <code translate="no" dir="ltr">auditConfigs</code>&mdash;then those fields are
added to <code translate="no" dir="ltr">updateMask</code>.</li>
</ul>

<p>As a consequence of these rules, the <code translate="no" dir="ltr">set-iam-policy</code> command has the following
behaviors:</p>

<ul>
<li><p>If you omit the <code translate="no" dir="ltr">auditConfigs</code> section in your new policy, then the previous
value of <code translate="no" dir="ltr">auditConfigs</code> section (if any) isn&#39;t changed, because that
section isn&#39;t in the update mask. This is harmless but might be confusing.</p></li>
<li><p>If you omit <code translate="no" dir="ltr">bindings</code> in your new policy object, then the <code translate="no" dir="ltr">bindings</code>
section is removed from your policy, since this  section appears in the
update mask. This is very harmful, and causes all principals to lose access
to your Google Cloud project.</p></li>
<li><p>If you omit <code translate="no" dir="ltr">etag</code> in your new policy object, this disables the checking for
concurrent changes to your policy and might result in your changes
accidentally overwriting someone else&#39;s changes.</p></li>
</ul>
