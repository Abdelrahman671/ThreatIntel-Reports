      "articleBody": "Discovered in summer of 2014, &lt;a href=&quot;https://exchange.xforce.ibmcloud.com/collection/GootKit-Ongoing-Research-Collection-422ab7991b937f35b89923f928b79702&quot;&gt;GootKit&lt;/a&gt; is widely considered one of the most sophisticated banking Trojans active in the wild. The malware is being used in online banking fraud attacks on consumer and business accounts, mostly in the U.K. and other parts of Europe.

In this blog post, I will describe my analysis of a recent GootKit sample (MD5: 60e079ec28d47ef85e93039c21afd19c) discovered by IBM X-Force research in January 2017. The sample caught our attention when we realized that GootKit&#039;s developers had modified its architecture and changed the way it operates on the infected endpoints.

My research into GootKit&#039;s inner workings unveiled its new network interception method, which now proxies internet traffic through the malware instead of placing hooks on the browser. GootKit also bypasses certificate validation by hooking other relevant APIs to continue its malicious operation unhindered.
&lt;h2&gt;GootKit Renovating the Architecture&lt;/h2&gt;
The first and most significant change I noticed in recent GootKit samples is an architectural expansion. In the past, GootKit operated on two principal modules:
&lt;ol&gt;
 	&lt;li&gt;The Loader module, which was responsible for persistence, malware updates and the injection of malicious code into the web browser and Windows OS processes; and&lt;/li&gt;
 	&lt;li&gt;The Main module, which was responsible for general malware functionality. This module is based on a node.js engine, bound with the malware&#039;s code.&lt;/li&gt;
&lt;/ol&gt;
The Main module can be injected into a svchost process, at which point it acts as a master. It can also be injected into a browser process and act as a slave, intercepting all network communications via the browser by hooking the NtDeviceIoControlFile API. The &lt;a href=&quot;https://msdn.microsoft.com/en-us/library/ms648411%28v=vs.85%29.aspx&quot;&gt;NtDeviceIoControlFile&lt;/a&gt; service is a device-dependent interface that extends the control that applications have over various devices within the system. This API provides a consistent view of the input and output data to the system while still providing the application and the driver a device-dependent method of specifying a communications interface.

That two-module setup has changed. Recent samples we collected in 2017 feature three modules instead of two:
&lt;ol&gt;
 	&lt;li&gt;The Loader module, which has the same purpose;&lt;/li&gt;
 	&lt;li&gt;The Main module, named node32.dll by GootKit&#039;s developers, which handles the malware&#039;s functionality, is based on the same node.js engine and implements the overall functionality in JavaScript code; and&lt;/li&gt;
 	&lt;li&gt;The Browser injection module, a recent addition to GootKit&#039;s modular architecture that is responsible for network interception and access to the browser.&lt;/li&gt;
&lt;/ol&gt;
GootKit now uses this module to redirect internet traffic through its own Main module instead of letting it flow directly from the browser.
&lt;p style=&quot;text-align: center;&quot;&gt;[button link=&quot;https://www-01.ibm.com/marketing/iwm/dre/signup?source=mrs-form-9451&amp;S_PKG=ov54550&quot; color=&quot;orange1&quot; icon=&quot;download&quot; size=&quot;small&quot;]Read the white paper: How to outsmart Fraudsters with Cognitive Fraud Detection[/button]&lt;/p&gt;

&lt;h2&gt;A Modified Deployment Flow&lt;/h2&gt;
GootKit changes the way it deploys on infected endpoints and the inner workings of its operations. In the sample I analyzed for this post, GootKit is fetched by a dropper. As the droppee, it is run from an already-established persistence mechanism that was set up by its loader.

GootKit&#039;s recently updated persistence is achieved through a Group Policy Object (GPO) entry in the Windows Registry. Group Policies control the user&#039;s working environment, and GootKit abuses this feature to launch whenever the computer boots up. Most financial malware families set a RunKey in the Registry, but GootKit has changed that scheme, likely to &lt;a style=&quot;background-color: initial;&quot; href=&quot;https://securityintelligence.com/gootkit-bobbing-and-weaving-to-avoid-prying-eyes/&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;avoid automated detection rules&lt;/a&gt;.

Next, GootKit creates a mstsc.exe process in suspended state and injects the Loader module into it. This process, also known as Remote Desktop Connection (RDC), is the client application for remote desktop services. It allows a user to remotely log into a networked computer running the terminal services server.

The mstsc process, in turn, runs a svchost.exe process and injects the Loader module into svchost. Next, the Main module is loaded from the svchost process.

The malware&#039;s binaries are stored in the following Windows registry keys: HKCU\Software\AppDataLow\binary&lt;strong style=&quot;background-color: initial;&quot;&gt;Image32_0&lt;/strong&gt; to HKCU\Software\AppDataLow\binary&lt;strong style=&quot;background-color: initial;&quot;&gt;Image32_7&lt;/strong&gt;.
&lt;h2&gt;GootKit as Its Own Internet Traffic Proxy&lt;/h2&gt;
Upon launching the malware, the Main module begins to set up a proxy server on the infected machines, utilizing GootKit&#039;s new browser injection module.

Two functions implemented by GootKit within its node.js engine are being used here: SpInitializeStandaloneMitm and RandomListenPortBase. Using a malicious JavaScript function, SpInitializeStandaloneMitm, the malware creates a proxy for both HTTP and HTTPS traffic on the &lt;a href=&quot;http://www.ibm.com/software/products/en/trusteer-apex-adv-malware&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;infected endpoint&lt;/a&gt;.

GootKit uses the RandomListenPortBase function to specify the standard port choice for its traffic. HTTP traffic will route via port 80 and HTTPS via port 443. As for the listening port, which is typically randomly selected in the 1024 to 5000 range, GootKit&#039;s hardcoded choice is port 6000.

&lt;img title=&quot;gootkit reverse engineering&quot; src=&quot;https://static.securityintelligence.com/uploads/2017/03/fig1.png&quot; alt=&quot;IBM X-Force Research&quot; /&gt;
&lt;em style=&quot;background-color: initial;&quot;&gt;Figure 1: JavaScript code used for GootKit&#039;s proxy server setup&lt;/em&gt;

Note that the next function called is Initialize Certificate Faker.
&lt;h2&gt;Overcoming the Browser&#039;s Certificate Verification Flags&lt;/h2&gt;
Upon the creation of a new browser process on the infected endpoint, the Loader module injects the Browser module into that process. The Browser module will, in turn, hook two types of APIs.
&lt;ol&gt;
 	&lt;li&gt;Network APIs responsible for connection creations:
&lt;ol&gt;
 	&lt;li&gt;mswsock.dllMSAFD_ConnectEx API (the mswsock.dll is a module providing extensions for Winsock; services provided by this file are not part of Winsock); and&lt;/li&gt;
 	&lt;li&gt;mswsock.dll WSPConnect API.&lt;/li&gt;
&lt;/ol&gt;
&lt;/li&gt;
 	&lt;li&gt;Certificate validation APIs:
&lt;ol&gt;
 	&lt;li&gt;crypt32.dll CertVerifyCertificateChainPolicy API; and&lt;/li&gt;
 	&lt;li&gt;crypt32.dll CertGetCertificateChain API.&lt;/li&gt;
&lt;/ol&gt;
&lt;/li&gt;
&lt;/ol&gt;
These hooks enable GootKit to redirect the internet traffic through its own module and switch up the OS flag for an invalid certificate with a valid one, thus disabling any error alerts from the browser.
&lt;h2&gt;Redirecting Through the GootKit Proxy&lt;/h2&gt;
When a new connection is initiated in the browser, GootKit&#039;s hooks change the endpoint&#039;s IP address and the listening port number to those predefined by GootKit earlier on. GootKit uses its hooks on MSAFD_ConnectEx and WSPConnect to check that the traffic type is relevant (TCP/IP). It looks at the header to ensure that sockaddr struct shows the number 16, which is the maximum length for nonhybrid structures in IPv4 addresses:

&lt;img title=&quot;gootkit traffic proxy&quot; src=&quot;https://static.securityintelligence.com/uploads/2017/03/fig2.png&quot; alt=&quot;IBM X-Force Research&quot; /&gt;&lt;em style=&quot;background-color: initial;&quot;&gt;Figure 2&lt;/em&gt;

&lt;img title=&quot;ibm x-force research&quot; src=&quot;https://static.securityintelligence.com/uploads/2017/03/fig3.png&quot; alt=&quot;GootKit reverse engineerin&quot; /&gt;&lt;em style=&quot;background-color: initial;&quot;&gt;Figure 3&lt;/em&gt;

The next verification has to do with the modification of the endpoint&#039;s IP address from an external IP to a local one. GootKit checks that the sa_family is indeed AF_INET, which represents the internet protocol address format.

&lt;img title=&quot;gootkit reverse engineering&quot; src=&quot;https://static.securityintelligence.com/uploads/2017/03/fig4.png&quot; alt=&quot;IBM X-Force research&quot; /&gt;
&lt;em style=&quot;background-color: initial;&quot;&gt;Figure 4: Sockaddr and SA_Family setup&lt;/em&gt;

GootKit modifies the endpoint&#039;s remote IP address to the local home IP (127.0.0.1) and switches the Transmission Control Protocol (TCP) port to its own selection based on a static value, to which it adds the original TCP port number.

&lt;img title=&quot;gootkit new web traffic proxy&quot; src=&quot;https://static.securityintelligence.com/uploads/2017/03/fig5.png&quot; alt=&quot;IBM X-Force Research reverse engineering&quot; /&gt;&lt;em&gt;Figure 5: The HEX value 0x100007F translates into the home IP address 127.0.0.1&lt;/em&gt;

The TCP listening port is also changed:

&lt;img title=&quot;gootkit figure 6&quot; src=&quot;https://static.securityintelligence.com/uploads/2017/03/fig6.png&quot; alt=&quot;Gootkit — Figure 6&quot; /&gt;
&lt;em style=&quot;background-color: initial;&quot;&gt;Figure 6&lt;/em&gt;

The value 1770h translates to port number 6000.
&lt;h2&gt;Faking the Certificate Flag&lt;/h2&gt;
Since traffic from the user to the bank is routed through the GootKit proxy at this point, GootKit&#039;s developers ensured that Secure Sockets Layer (SSL) error messages don&#039;t pop up and raise the victim&#039;s suspicion. To fake the validity flag, GootKit hooks a couple of APIs.

The first is called CertGetCertificateChain. This hook will ensure that every result returned from PCERT_CHAIN_CONTEXT with a pPolicyStatus parameter will change the flag to reflect a valid certificate (set to 0):

&lt;tr&gt;

&lt;/tr&gt;
&lt;/tbody&gt;
&lt;/table&gt;

&lt;h2&gt;Conclusion&lt;/h2&gt;
GootKit&#039;s developers are evidently still working on this malware project, enhancing its evasion techniques and its ability to control what victims see when they browse to their bank&#039;s website. This recent development in GootKit&#039;s operations on infected machines is consistent with the advanced, technical nature of this malware project.

Of attacks detected by IBM X-Force in the past year, GootKit was the most active in Europe in 2016, with configurations targeting the U.K., France and Italy. Per X-Force data, GootKit ranks eighth on the global list of the most active &lt;a href=&quot;https://securityintelligence.com/is-financial-malware-breaking-the-bank/&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;financial malware families&lt;/a&gt; per attack volume. Although it is very sophisticated, it keeps its infection and attack campaigns small and focused, which helps it fly under the radar and avoid detection.

<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN" "http://www.w3.org/TR/REC-html40/loose.dtd">
<html><body><p>Discovered in summer of 2014, <a href="https://exchange.xforce.ibmcloud.com/collection/GootKit-Ongoing-Research-Collection-422ab7991b937f35b89923f928b79702" target="_blank" rel="noopener nofollow" >GootKit</a> is widely considered one of the most sophisticated banking Trojans active in the wild. The malware is being used in online banking fraud attacks on consumer and business accounts, mostly in the U.K. and other parts of Europe.</p>
<p>In this blog post, I will describe my analysis of a recent GootKit sample (MD5: 60e079ec28d47ef85e93039c21afd19c) discovered by IBM X-Force research in January 2017. The sample caught our attention when we realized that GootKit&rsquo;s developers had modified its architecture and changed the way it operates on the infected endpoints.</p>
<p>My research into GootKit&rsquo;s inner workings unveiled its new network interception method, which now proxies internet traffic through the malware instead of placing hooks on the browser. GootKit also bypasses certificate validation by hooking other relevant APIs to continue its malicious operation unhindered.</p>
<h2>GootKit Renovating the Architecture</h2>
<p>The first and most significant change I noticed in recent GootKit samples is an architectural expansion. In the past, GootKit operated on two principal modules:</p>
<ol>
<li>The Loader module, which was responsible for persistence, malware updates and the injection of malicious code into the web browser and Windows OS processes; and</li>
<li>The Main module, which was responsible for general malware functionality. This module is based on a node.js engine, bound with the malware&rsquo;s code.</li>
</ol>
<p>The Main module can be injected into a svchost process, at which point it acts as a master. It can also be injected into a browser process and act as a slave, intercepting all network communications via the browser by hooking the NtDeviceIoControlFile API. The <a href="https://msdn.microsoft.com/en-us/library/ms648411%28v=vs.85%29.aspx" target="_blank" rel="noopener nofollow" >NtDeviceIoControlFile</a> service is a device-dependent interface that extends the control that applications have over various devices within the system. This API provides a consistent view of the input and output data to the system while still providing the application and the driver a device-dependent method of specifying a communications interface.</p>
<p>That two-module setup has changed. Recent samples we collected in 2017 feature three modules instead of two:</p>
<ol>
<li>The Loader module, which has the same purpose;</li>
<li>The Main module, named node32.dll by GootKit&rsquo;s developers, which handles the malware&rsquo;s functionality, is based on the same node.js engine and implements the overall functionality in JavaScript code; and</li>
<li>The Browser injection module, a recent addition to GootKit&rsquo;s modular architecture that is responsible for network interception and access to the browser.</li>
</ol>
<p>GootKit now uses this module to redirect internet traffic through its own Main module instead of letting it flow directly from the browser.</p>
<p style="text-align: center;"><a class="btn orange1 small" href="https://www-01.ibm.com/marketing/iwm/dre/signup?source=mrs-form-9451&amp;S_PKG=ov54550" >Read the white paper: How to outsmart Fraudsters with Cognitive Fraud Detection <i class="fa fa-download"></i></a></p>
<h2>A Modified Deployment Flow</h2>
<p>GootKit changes the way it deploys on infected endpoints and the inner workings of its operations. In the sample I analyzed for this post, GootKit is fetched by a dropper. As the droppee, it is run from an already-established persistence mechanism that was set up by its loader.</p>
<p>GootKit&rsquo;s recently updated persistence is achieved through a Group Policy Object (GPO) entry in the Windows Registry. Group Policies control the user&rsquo;s working environment, and GootKit abuses this feature to launch whenever the computer boots up. Most financial malware families set a RunKey in the Registry, but GootKit has changed that scheme, likely to <a href="https://securityintelligence.com/gootkit-bobbing-and-weaving-to-avoid-prying-eyes/"  target="_blank" rel="noopener nofollow" >avoid automated detection rules</a>.</p>
<p>Next, GootKit creates a mstsc.exe process in suspended state and injects the Loader module into it. This process, also known as Remote Desktop Connection (RDC), is the client application for remote desktop services. It allows a user to remotely log into a networked computer running the terminal services server.</p>
<p>The mstsc process, in turn, runs a svchost.exe process and injects the Loader module into svchost. Next, the Main module is loaded from the svchost process.</p>
<p>The malware&rsquo;s binaries are stored in the following Windows registry keys: HKCU\Software\AppDataLow\binary<strong style="background-color: initial;">Image32_0</strong> to HKCU\Software\AppDataLow\binary<strong style="background-color: initial;">Image32_7</strong>.</p>
<h2>GootKit as Its Own Internet Traffic Proxy</h2>
<p>Upon launching the malware, the Main module begins to set up a proxy server on the infected machines, utilizing GootKit&rsquo;s new browser injection module.</p>
<p>Two functions implemented by GootKit within its node.js engine are being used here: SpInitializeStandaloneMitm and RandomListenPortBase. Using a malicious JavaScript function, SpInitializeStandaloneMitm, the malware creates a proxy for both HTTP and HTTPS traffic on the <a href="http://www.ibm.com/software/products/en/trusteer-apex-adv-malware"  target="_blank" rel="noopener nofollow" >infected endpoint</a>.</p>
<p>GootKit uses the RandomListenPortBase function to specify the standard port choice for its traffic. HTTP traffic will route via port 80 and HTTPS via port 443. As for the listening port, which is typically randomly selected in the 1024 to 5000 range, GootKit&rsquo;s hardcoded choice is port 6000.</p>
<p><amp-img src="https://static.securityintelligence.com/uploads/2017/03/fig1.png" layout="intrinsic" class="" alt="IBM X-Force Research" width="" height="" lightbox="lightbox"></amp-img><br>
<em style="background-color: initial;">Figure 1: JavaScript code used for GootKit&rsquo;s proxy server setup</em></p>
<p>Note that the next function called is Initialize Certificate Faker.</p>
<h2>Overcoming the Browser&rsquo;s Certificate Verification Flags</h2>
<p>Upon the creation of a new browser process on the infected endpoint, the Loader module injects the Browser module into that process. The Browser module will, in turn, hook two types of APIs.</p>
<ol>
<li>Network APIs responsible for connection creations:
<ol>
<li>mswsock.dllMSAFD_ConnectEx API (the mswsock.dll is a module providing extensions for Winsock; services provided by this file are not part of Winsock); and</li>
<li>mswsock.dll WSPConnect API.</li>
</ol>
</li>
<li>Certificate validation APIs:
<ol>
<li>crypt32.dll CertVerifyCertificateChainPolicy API; and</li>
<li>crypt32.dll CertGetCertificateChain API.</li>
</ol>
</li>
</ol>
<p>These hooks enable GootKit to redirect the internet traffic through its own module and switch up the OS flag for an invalid certificate with a valid one, thus disabling any error alerts from the browser.</p>
<h2>Redirecting Through the GootKit Proxy</h2>
<p>When a new connection is initiated in the browser, GootKit&rsquo;s hooks change the endpoint&rsquo;s IP address and the listening port number to those predefined by GootKit earlier on. GootKit uses its hooks on MSAFD_ConnectEx and WSPConnect to check that the traffic type is relevant (TCP/IP). It looks at the header to ensure that sockaddr struct shows the number 16, which is the maximum length for nonhybrid structures in IPv4 addresses:</p>
<p><amp-img src="https://static.securityintelligence.com/uploads/2017/03/fig2.png" layout="intrinsic" class="" alt="IBM X-Force Research" width="" height="" lightbox="lightbox"></amp-img><em style="background-color: initial;">Figure 2</em></p>
<p><amp-img src="https://static.securityintelligence.com/uploads/2017/03/fig3.png" layout="intrinsic" class="" alt="GootKit reverse engineerin" width="" height="" lightbox="lightbox"></amp-img><em style="background-color: initial;">Figure 3</em></p>
<p>The next verification has to do with the modification of the endpoint&rsquo;s IP address from an external IP to a local one. GootKit checks that the sa_family is indeed AF_INET, which represents the internet protocol address format.</p>
<p><amp-img src="https://static.securityintelligence.com/uploads/2017/03/fig4.png" layout="intrinsic" class="" alt="IBM X-Force research" width="" height="" lightbox="lightbox"></amp-img><br>
<em style="background-color: initial;">Figure 4: Sockaddr and SA_Family setup</em></p>
<p>GootKit modifies the endpoint&rsquo;s remote IP address to the local home IP (127.0.0.1) and switches the Transmission Control Protocol (TCP) port to its own selection based on a static value, to which it adds the original TCP port number.</p>
<p><amp-img src="https://static.securityintelligence.com/uploads/2017/03/fig5.png" layout="intrinsic" class="" alt="IBM X-Force Research reverse engineering" width="" height="" lightbox="lightbox"></amp-img><em>Figure 5: The HEX value 0x100007F translates into the home IP address 127.0.0.1</em></p>
<p>The TCP listening port is also changed:</p>
<p><amp-img src="https://static.securityintelligence.com/uploads/2017/03/fig6.png" layout="intrinsic" class="" alt="Gootkit &mdash; Figure 6" width="" height="" lightbox="lightbox"></amp-img><br>
<em style="background-color: initial;">Figure 6</em></p>
<p>The value 1770h translates to port number 6000.</p>
<h2>Faking the Certificate Flag</h2>
<p>Since traffic from the user to the bank is routed through the GootKit proxy at this point, GootKit&rsquo;s developers ensured that Secure Sockets Layer (SSL) error messages don&rsquo;t pop up and raise the victim&rsquo;s suspicion. To fake the validity flag, GootKit hooks a couple of APIs.</p>
<p>The first is called CertGetCertificateChain. This hook will ensure that every result returned from PCERT_CHAIN_CONTEXT with a pPolicyStatus parameter will change the flag to reflect a valid certificate (set to 0):</p>
<table border="1" cellspacing="0" cellpadding="0">
<tbody>
<tr>
<td width="575"><strong>TrustStatus.dwErrorStatus to CERT_TRUST_NO_ERROR and TrustStatus.dwInfoStatus set to 0.</strong></td>
</tr>
</tbody>
</table><div class="table-scroll-help-text"><span>Scroll to view full table </span></div>
<p><amp-img src="https://static.securityintelligence.com/uploads/2017/03/fig7.png" layout="intrinsic" class="" alt="Gootkit &mdash; Figure 7" width="" height="" lightbox="lightbox"></amp-img><em style="background-color: initial;"><br>
</em></p>
<p><em style="background-color: initial;">Figure 7</em></p>
<p>The second API hooked for this purpose is CertVerifyCertificateChainPolicy. Here, too, GootKit updates the PCERT_CHAIN_POLICY_STATUS error result to 0 for a valid certificate.</p>
<p><amp-img src="https://static.securityintelligence.com/uploads/2017/03/fig8.png" layout="intrinsic" class="" alt="Gootkit &mdash; Figure 8" width="" height="" lightbox="lightbox"></amp-img><em style="background-color: initial;"><br>
</em></p>

<h2>Conclusion</h2>
<p>GootKit&rsquo;s developers are evidently still working on this malware project, enhancing its evasion techniques and its ability to control what victims see when they browse to their bank&rsquo;s website. This recent development in GootKit&rsquo;s operations on infected machines is consistent with the advanced, technical nature of this malware project.</p>
<p>Of attacks detected by IBM X-Force in the past year, GootKit was the most active in Europe in 2016, with configurations targeting the U.K., France and Italy. Per X-Force data, GootKit ranks eighth on the global list of the most active <a href="https://securityintelligence.com/is-financial-malware-breaking-the-bank/"  target="_blank" rel="noopener nofollow" >financial malware families</a> per attack volume. Although it is very sophisticated, it keeps its infection and attack campaigns small and focused, which helps it fly under the radar and avoid detection.</p>
<p><amp-img src="https://static.securityintelligence.com/uploads/2017/03/fig9.png" layout="intrinsic" class="" alt="Gootkit &mdash; Figure 9" width="" height="" lightbox="lightbox"></amp-img><br>
<em style="background-color: initial;">Figure 9: Most prevalent global financial malware families per attack volume (Source: IBM Security, 2017 YTD).<br>
</em></p>
