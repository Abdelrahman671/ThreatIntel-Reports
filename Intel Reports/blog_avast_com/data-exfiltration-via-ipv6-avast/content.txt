                                <div class="box"><p class="perex">Testing the capabilities of IPv6 and how malware could take advantage of it</p>


                                    <span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>Within the <a href="https://www.stratosphereips.org/aposemat/"><span>Aposemat Team</span></a>, we’ve been working on testing the capabilities of IPv6 and how malware could take advantage of it. One of the topics we explored was exfiltration of data via the IPv6 protocol. In this post, we share our study into this topic.</p>
<h2><!--more--><span style="font-size: 24px;">What is exfiltration?</span></h2>
<p>Exfiltration is the unauthorized exportation of sensitive data out of the network by connecting to an external destination and/or using covert channels. The latter is commonly used to exfiltrate information while being undetected or avoid any measure in place to stop the migration of data. There have been <a href="https://www.securityweek.com/researchers-devise-perfect-data-exfiltration-technique"><span>numerous studies on this topic</span></a>, and even to this day, data theft produced by breaches put exfiltration in the center of attention.</p>
<p>To exfiltrate data, networking and transportation layers (shown in Figure 1) are commonly used as are low level layers that would require deep packet inspection to find occurrences or identify that the exfiltration is happening. They also provide fields and portions of data in the packet headers that are not commonly used or zeroed out. These sections can be used to store portions of data and could be unnoticed by analyzing the packet captures.</p>
<p style="text-align: center;"><img src="https://lh5.googleusercontent.com/gZNNr_wBEQ96PViwtnVH-3QGG7iU8n2-DUrqPADuawVqKebewI9lffiiYfgAOGujCCKXfy6WFl81z2aBwKr6g8UB1U_z9-K-9uBrHWO-elOKLX9xnY7qeukO2hmhCSDQBG3Lpnq9" width="602" height="304"></p>
<p style="text-align: center;"><em><strong>Figure 1. </strong>OSI Model and description of its layers. Layers 3 and 4 are highlighted in light orange and yellow respectively. (Source: <a href="https://en.wikipedia.org/wiki/OSI_model">Wikipedia</a>)</em></p>
<h2 style="font-size: 24px;">Tools of the trade</h2>
<p>Several tools exist to carry out exfiltration via IPv6 network stack. We’ll describe <strong>IPv6teal</strong> and <strong>IPv6DNSExfil</strong>, and how these tools are used to exfiltrate data via IPv6.</p>
<h3><span style="font-size: 14pt; color: #434343;">IPv6teal</span></h3>
<p>The first one is <a href="https://github.com/christophetd/IPv6teal">IPv6teal</a> and consists of a receiver and sender (exfiltrate) script. This tools makes use of the <a href="https://tools.ietf.org/html/rfc8200#section-6">Flow Label <span style="color: #1155cc; text-decoration: underline;">field</span></a> which is used to label sequences of packets and it has a fixed size of 20 bits (detailed in Figure 2). It makes use of this specific field because it could be variable and contains custom bits without impact on the packet reaching its destination. This detail makes a good candidate for storing data that could reach an endpoint safely while being hidden in normal traffic.</p>
<p style="text-align: center;"><img src="https://lh5.googleusercontent.com/-ih7svepAGUpinellgEUnfFBuTpzyH3h-xQBhheqfL-sojbahOYw4bx0uGXZ62rvXzKBOu3yCjdivWQp3hH9RzbJcyui-GUbU3oeT7s7Mf2VATc0KV2fvmJUNCuqE1ZSjdHjF_2l" width="602" height="185"></p>
<p style="text-align: center;"><em><strong>Figure 2.</strong> IPv6 packet header structure with Flow Label field (marked red). <br>(Source: <a href="https://en.wikipedia.org/wiki/IPv6_packet">Wikipedia</a>)</em></p>
<p>To be able to fit more data in fewer packets the author decided to use GZIP compression to accomplish this. In our tests, it took approximately two seconds and 15 packets to send a plain-text file containing the string <strong>THISISASECRET</strong> across the internet. The information is transmitted with a magic value that marks the start and end of the flow of data. These magic values also add more information about the data being transmitted.</p>
<p>The flow of packets for our test end up being built this way:</p>
<p style="text-align: left;"><img src="https://lh6.googleusercontent.com/P2zQgRZ4DC1ISY3Scy9-gA7YI4fniVqSgUyj5NIB1TtJUGm1aK_-dxW7uoiQT6OLlMzTf9OdujLTgbSQzIfCjS20DidK5tTfJ-JBtINXL0b8KT04WYuDwsblV5tE16CC2bxsNWKB" width="420" height="484"><br>The packets are built over two upper layers: the IPv6 layer and a “Raw” layer, which is only data appended to the last layer. The raw layer holds the magic values, discussed earlier, and tells the receiver when a transmission starts, how many bits are going to be transmitted and how many packets will be transmitted, not counting the packet ending the transmission.</p>
<p>Another exfiltration technique, on a higher level of the OSI Model, is done via DNS AAAA records. The <a href="https://tools.ietf.org/html/rfc3596#page-3"><span>AAAA records</span></a> were designed to be used with IPv6 addresses. When a client requests the IPv6 address of a domain it will utilize this record in order to get it from a DNS server. Although TXT records were commonly used for this as they can hold human-readable data, as well as machine-readable, queries to TXT records are less common and could be caught quickly during an study of the network flow.</p>
<h3><span style="font-size: 14pt; color: #434343;">IPv6DNSExfil</span></h3>
<p>Tools like <a href="https://github.com/DShield-ISC/IPv6DNSExfil"><span>IPv6DNSExfil</span></a> make use of this technique in order to store a secret, in a pseudo-IPv6 address format, for a short period of time on AAAA records. It will make use of the <a href="https://linux.die.net/man/1/nsupdate"><span>nsupdate</span></a> tool to dynamically create said AAAA records and push them to an upstream DNS server thus exfiltrating the information. A record created this way, using the same secret that we utilized previously, will look like this:</p>
<p><strong>a.evilexample.com.</strong> <span style="color: #6aa84f;">10</span> <span style="color: #3d85c6;">AAAA</span> 2000:<span style="color: #ff0000;">5448</span>:<span style="color: #ff0000;">4953</span>:<span style="color: #ff0000;">4953</span>:<span style="color: #ff0000;">4153</span>:<span style="color: #ff0000;">4543</span>:<span style="color: #ff0000;">5245</span>:<span style="color: #ff0000;">54</span>00</p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;<strong><span style="color: #ff0000;">T H&nbsp; I S&nbsp; I S&nbsp; A S&nbsp; E C&nbsp; R E&nbsp; T</span></strong></p>
<ul>
<li><strong>DNS record</strong></li>
<li style="color: #6aa84f;"><span style="color: #6aa84f;">TTL</span></li>
<li style="color: #3d85c6;"><span style="color: #3d85c6;">Record Type</span></li>
<li style="color: #ff0000;"><span style="color: #ff0000;">Data</span></li>
</ul>
<p>Once the record is put in place the attackers can utilize this data as they please, either by using it as a C&amp;C (<a href="https://isc.sans.edu/forums/diary/Command+and+Control+Channels+Using+AAAA+DNS+Records/21301/"><span>as suggested by the author</span></a>) or to just transfer the information from one endpoint to another with DNS queries to that specific server.</p>
<h2><span style="font-size: 16pt;">Custom exfiltration methods</span></h2>
<p>Libraries like <a href="https://github.com/secdev/scapy"><span>scapy</span></a>, for Python, make it easier for developers to interact with networking abstractions at a higher level. For example, with only two lines of code we are able to send a crafted packet to an IPv6 endpoint:</p>

<p>Using this same approach we can start generating traffic dynamically using scapy instead of just sending packets without an upper transportation layer. One case would be making use of <a href="https://tools.ietf.org/html/rfc4443"><span>ICMPv6</span></a> protocol, which is an <a href="https://www.avast.com/c-ipv4-vs-ipv6-addresses" rel="noopener">improved version of its IPv4 relative</a>. A “classic” exfiltration method using this protocol is using the echo and reply messages (commonly used by <strong>ping6</strong> networking tool) to send data outside the network without establishing a connection like TCP. This way we can send specific chunks of data over IPv6 via ICMPv6 echo requests to a remote host sniffing the network. Take a look at this code, for example:</p>

<p>This script will make use of the secret string we have been sending previously, encrypt it using the XOR cipher, and send each two bytes of that secret encrypted string via an ICMPv6 echo request with an specific ID. Those two bytes are hidden in the sequence field, which is a short integer field, and can be decrypted on destination by a receiver. Also, we are setting up the packet with an specific ID (in this case <strong>0x1337</strong>) because we want to easily recognize the packet as one of ours among the flow of networking traffic. So, let’s send a secret!</p>

<h2><span style="font-size: 16pt;">In summary</span></h2>
<p>IPv6 is growing in popularity as well as the necessity for more addressing space. Although, the percentage of IPv6 adoption worldwide is <a href="https://www.google.com/intl/en/ipv6/statistics.html#tab=ipv6-adoption"><span>lower than 35%</span></a> mostly because it still requires a big effort and investment from companies and organizations. This means that the tools and techniques demonstrated in this article will take time to be fully adopted or used while leaving space to further develop more ideas and methodologies.</p></span>
