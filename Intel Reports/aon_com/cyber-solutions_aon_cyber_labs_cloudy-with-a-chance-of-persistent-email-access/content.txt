  <div class="full_width_center content_only"> 

    <p><em>How an Advanced Threat Group Leveraged Microsoft Azure to Gain Persistent Access to Emails</em></p>
<h2>Introduction</h2>
<p>Over the last few weeks, <a href="https://www.aon.com/cyber-solutions/solutions/breach-response/">Stroz Friedberg Incident Response</a> has led multiple investigations related to the nation-state threat group being tracked under several names &#8211; UNC2452, Dark Halo, StellarParticle and SolarStorm. Stroz Friedberg has tracked some of the activities tied to this threat group going back to January 2020. It has been widely reported that this threat group used the <a href="https://www.fireeye.com/blog/threat-research/2020/12/evasive-attacker-leverages-solarwinds-supply-chain-compromises-with-sunburst-backdoor.html">SUNBURST backdoor</a> to gain initial access to victim networks. However, Stroz Friedberg is investigating incidents where the same threat group gained initial access through alternative means, without any evidence of SUNBURST use.</p>
<p>Based on open-source reporting, a common theme associated with this threat group is its focus on victims’ email data – be it in <a href="https://blog.malwarebytes.com/malwarebytes-news/2021/01/malwarebytes-targeted-by-nation-state-actor-implicated-in-solarwinds-breach-evidence-suggests-abuse-of-privileged-access-to-microsoft-office-365-and-azure-environments/">O365</a> <a href="https://www.justice.gov/opa/pr/department-justice-statement-solarwinds-update">or</a> <a href="https://www.finance.senate.gov/ranking-members-news/wyden-statement-following-treasury-and-irs-briefing-on-solarwinds-hack">on-premise</a> <a href="https://www.volexity.com/blog/2020/12/14/dark-halo-leverages-solarwinds-compromise-to-breach-organizations/">Exchange</a>. On December 13, 2020, Microsoft issued an <a href="https://msrc-blog.microsoft.com/2020/12/13/customer-guidance-on-recent-nation-state-cyber-attacks/">advisory</a> on this threat group detailing the tactics that enabled persistent long-term access to users’ email in Office 365 (“O365”) – specifically by adding the application-level permissions “Mail.Read” or “Mail.ReadWrite” to cloud applications in Azure.</p>
<p>In this post, we’re going to enumerate (1) the attack chain wherein the threat group gains long-term access to all users’ mailboxes within an O365 tenant and (2) the evidence sources that track such activity.</p>
<h2>Attack Chain</h2>
<p>A crucial component needed to gain long-term persistent access to read all users’ mailboxes in O365 is access to an administrator account in Azure Active Directory. This is especially easy for a threat group to accomplish when an administrator account is not protected by multi-factor authentication (“MFA”). Even in cases where MFA is implemented, this threat group has taken <a href="https://www.volexity.com/blog/2020/12/14/dark-halo-leverages-solarwinds-compromise-to-breach-organizations/">steps to bypass MFA</a>.  In our investigations, the threat group took the following steps:</p>
<ol>
<li>Compromise administrator account credentials and/or forge authentication tokens for an administrator account and log into the account.</li>
<li>Modify existing Azure applications to add “Mail.Read” or “Mail.ReadWrite” application permissions.</li>
<li>Create a client secret to enable “<a href="https://docs.microsoft.com/en-us/azure/active-directory/develop/v2-oauth2-client-creds-grant-flow">client credentials grant</a>” authentication to modified Azure applications with the newly added permissions.</li>
<li>With access to (1) the client ID of the targeted Azure application, (2) the Azure tenant ID and (3) the client secret, connect to victim’s Azure tenant via the registered application to read email from any mailbox in the tenant.</li>
</ol>
<p>&nbsp;</p>
<figure id="attachment_7250" aria-describedby="caption-attachment-7250" style="width: 4000px" class="wp-caption aligncenter"><a href="https://www.aon.com/cyber-solutions/wp-content/uploads/attack_flow_azure-01.jpg"><img fetchpriority="high" decoding="async" class="size-full wp-image-7250" src="https://www.aon.com/cyber-solutions/wp-content/uploads/attack_flow_azure-01.jpg" alt="" width="4000" height="2251" srcset="https://www.aon.com/cyber-solutions/wp-content/uploads/attack_flow_azure-01.jpg 4000w, https://www.aon.com/cyber-solutions/wp-content/uploads/attack_flow_azure-01-300x169.jpg 300w, https://www.aon.com/cyber-solutions/wp-content/uploads/attack_flow_azure-01-768x432.jpg 768w" sizes="(max-width: 4000px) 100vw, 4000px" /></a><figcaption id="caption-attachment-7250" class="wp-caption-text">Sample threat workflow involving modification of Azure application permissions</figcaption></figure>
<p>It’s helpful to visualize this and walk through an example scenario. When an application is registered in Azure, a delegated permission “User.Read” is automatically assigned to the application. In Azure, delegated permissions differ from application permissions as <a href="https://docs.microsoft.com/en-us/azure/active-directory/develop/v2-permissions-and-consent">described by Microsoft</a> below:</p>
<p style="padding-left: 30px"><em><strong>Delegated Permissions</strong>: Delegated permissions are used by apps that have a signed-in user present. For these apps, either the user or an administrator consents to the permissions that the app requests. The app is delegated permission to act as the signed-in user when it makes calls to the target resource.</em></p>
<p style="padding-left: 30px"><em><strong>Application Permissions:</strong> Application permissions are used by apps that run without a signed-in user present, for example, apps that run as background services or daemons. Only an administrator can consent to application permissions.</em></p>
<p>In this example, suppose the organization also adds the permission “Dataset.ReadWrite.All” for their purposes. Once a threat group gains access to Azure using an administrator account, they can add an application permission like “Mail.Read” to this existing application in Azure. Adding this permission will not impact the legitimate usage of the application by the organization, and as a result may go undetected. Upon adding the “Mail.Read” permission, the threat group can then create a client secret to authenticate to this application. Armed with the client ID, tenant ID and the client secret, the threat group can now connect to Azure and read all email via an <em>external</em> application.  That external application could be a commercial product designed to export mail from O365 or could be a custom application to pull mail via the Microsoft Graph API. Alternatively, the threat group could register a new application of their own to Azure as a method of persistent access.  This method is much more likely to be detected, depending on the tenant configurations and diligence of the organization.</p>
<p>As an example, issuing an HTTP GET request like <em>GET /users/{id | userPrincipalName}/message</em>s would result in retrieving all email messages for a specific user. A list of Graph API methods that can be used is documented by Microsoft <a href="https://docs.microsoft.com/en-us/graph/api/user-list-messages?view=graph-rest-1.0&amp;tabs=http">here</a>.</p>
<p>Using the Microsoft Graph API, a threat group can also issue requests based on certain filters. For example, they can retrieve all emails within a tenant that contain specific keywords such as “PRIVILEGED AND CONFIDENTIAL” in the body of an email.</p>
<p>As a proof of concept, Stroz Friedberg created an external application called “MailFetch” that takes as inputs client ID, tenant ID and the client secret and issues Microsoft Graph API requests through the registered Azure application associated with the client ID. To manage the output and to prevent reading all email within the tenant, the application takes as an input a “username” which is used as a filter. Using MailFetch, you can verify that the &#8220;Mail.Read&#8221; permission grants read access to all users’ mailboxes. MailFetch is publicly available on our <a href="https://github.com/strozfriedberg/msgraph-mailfetch">GitHub page</a>.</p>
<p>Organizations impacted by this threat workflow may be interested to know the following data that was accessible through the Graph API permission “Mail.Read” during our testing:</p>
<ol>
<li>Message__body__content</li>

<p>As we see from the email properties exposed above, a threat group using an Azure application with the “Mail.Read” application permission has access to the body of the email, email senders/recipients and attachments in addition to other email properties.</p>
<p>As mentioned in the example scenario, the threat group is also able to filter on email properties to retrieve only emails of interest. Per Microsoft’s <a href="https://docs.microsoft.com/en-us/graph/query-parameters#search-parameter">documentation</a>, filters can be applied to the following properties:</p>

<h2>Detection Mechanisms</h2>
<p>On December 16, 2020, Microsoft released <a href="https://techcommunity.microsoft.com/t5/azure-sentinel/solarwinds-post-compromise-hunting-with-azure-sentinel/ba-p/1995095">Azure Sentinel queries</a> that detect the Techniques, Tactics and Procedures (“TTPs”) mentioned in this post, including but not limited to: (1) the creation of service principals; and (2) addition of permissions to applications. Leveraging these, Stroz Friedberg recommends organizations review the following areas to detect the attack chain detailed above.</p>
<h3>Sign-in Activity</h3>
<p>When investigating sign-in activity for this attack chain, focus on sign-ins to Office 365 and Azure Service Principals.</p>
<p>As with any investigation into malicious activity in Office 365, look out for user sign-ins that originate from anomalous IP addresses, especially those from anonymization services. Administrator accounts are of particular interest since these are the accounts that can modify cloud applications in Azure. Gaining access to an administrator account is the first step in this attack chain.</p>
<p>In addition to common methods such as phishing or credential stuffing, one potential avenue for O365 account compromise that this threat group has been known to use is SAML token forgery. If the threat group has stolen the SAML-signing certificate from the organization’s network, or added their own certificate to the tenant, they can sign their own tokens to impersonate any user in the tenant, including administrator accounts. A <a href="https://us-cert.cisa.gov/ncas/alerts/aa21-008a">January 2021 CISA alert</a> addressing ways to detect malicious activity in Azure notes a specific “UserAuthenticationValue” of 16457, which is described as an indicator of potential SAML forgery. Looking for non-guest users whose sign-in events in the Unified Audit Log contain this value could identify sign-ins that have used forged SAML tokens.</p>
<p>Additionally, Service Principal sign-ins should be reviewed for unauthorized activity. These events can be found in the Azure Active Directory portal under Sign-Ins. Within Azure Sentinel, this data is tracked in a table named “AADServicePrincipalSigninLogs”, which displays the source IP address that the sign-in attempt originated from. As with Office 365 user logins, anomalous IP addresses could be indicative of compromise.</p>
<p>Please note that AADServicePrincipalSigninLogs is only enabled in Azure Sentinel when diagnostic logging is enabled on the workspace. Microsoft has listed the steps to enable diagnostic logging <a href="https://techcommunity.microsoft.com/t5/azure-sentinel/solarwinds-post-compromise-hunting-with-azure-sentinel/ba-p/1995095">here</a>.</p>
<h3>Modifications to Azure Applications</h3>
<p>A core component of these types of attacks involves modifying existing applications with additional permissions. There are two ways to detect this, (1) via the Unified Audit Log which shows historical activity (retention will vary depending on the environment), and (2) the active application-level permissions within Azure.</p>
<h3>Unified Audit Log</h3>
<p>Organizations should review operations in the Unified Audit Log that relate to application creations, changes, and deletions. Any of the following Unified Audit Log events could indicate changes to an Azure application:</p>
<ol>
<li>Update application.</li>
<li>Consent to application.</li>
<li>Add OAuth2PermissionGrant.</li>
<li>Add app role assignment to service principal.</li>
<li>Add app role assignment grant to user.</li>
<li>Add service principal.</li>
<li>Update service principal.</li>
<li>Update application.</li>
<li>Update application – Certificates and secrets management</li>
</ol>
<p>&nbsp;</p>
<p>Although each operation is formatted slightly differently, you will most often find the Application/Service Principal name in the <strong>AdditionalDetails</strong> portion of the log; specifically, under the <strong>targetname</strong> attribute.</p>
<p>A User Agent attribute can also be found in the <strong>AdditionalDetails</strong> section of the log entry which can also be used as a potential indicator of compromise, the exact attribute name is “User-Agent”.</p>
<h3>Audit of active application-level permissions in registered Azure Apps</h3>
<p>In addition to a review of the Unified Audit Log, Stroz Friedberg also recommends an audit of <em>active</em> application-level permissions assigned to registered Azure applications to ensure there is no active compromise.</p>
<p>Apart from email access via an Azure application with “Mail.Read” or “Mail.ReadWrite” permissions, threat actors could plausibly add other application-level permissions such as “<em>Sites.Read.All” or “Files.Read.All” </em>which would grant read access to documents stored on all SharePoint/OneDrive sites in the organization.</p>
<p>To review these application permissions within Azure, one can perform the following steps:</p>
<ol>
<li>Go to the <a href="https://portal.azure.com/">Azure Portal</a> (https://portal.azure.com)</li>
<li>Search for <strong>Enterprise Applications, </strong>and navigate to the page</li>
<li>Select the application to review</li>
<li>On the left-hand side, select <strong>Permissions</strong></li>
<li>Look for permissions that are potentially overinclusive such as Mail.Read permissions. If the Type is labeled as “Application” this indicates that there is potential for global read abilities.</li>
</ol>
<p>&nbsp;</p>
<p>See below for an example of an application with malicious “Mail.Read” permissions added:</p>
<figure id="attachment_7241" aria-describedby="caption-attachment-7241" style="width: 1625px" class="wp-caption aligncenter"><a href="https://www.aon.com/cyber-solutions/wp-content/uploads/Figure_01.png"><img decoding="async" class="size-full wp-image-7241" src="https://www.aon.com/cyber-solutions/wp-content/uploads/Figure_01.png" alt="" width="1625" height="550" srcset="https://www.aon.com/cyber-solutions/wp-content/uploads/Figure_01.png 1625w, https://www.aon.com/cyber-solutions/wp-content/uploads/Figure_01-300x102.png 300w, https://www.aon.com/cyber-solutions/wp-content/uploads/Figure_01-768x260.png 768w" sizes="(max-width: 1625px) 100vw, 1625px" /></a><figcaption id="caption-attachment-7241" class="wp-caption-text">Azure application with “Mail.Read” permissions</figcaption></figure>
<p>If you have more than a couple of applications in Azure, Stroz Friedberg recommends using <a href="https://github.com/cisagov/Sparrow">Sparrow</a>, a tool created by Cybersecurity &amp; Infrastructure Security Agency (“CISA”), to perform this audit across all applications in your Azure tenant. Upon successful execution, this tool creates a report called “ApplicationGraphPermissions.csv” that contains all the application-level permissions assigned to applications in Azure.</p>
<p>A list of all application-level graph permissions can be found in Microsoft’s documentation <a href="https://docs.microsoft.com/en-us/graph/permissions-reference">here</a>. If a compromised application contains unexpected permissions that could access Mail, Sites or Files, Stroz Friedberg recommends reviewing the Unified Audit Log for associated activities. Below is advice on identifying access to mail by a compromised application.</p>
<h3>Emails Accessed through Compromised Application</h3>
<p>In some cases, organizations can determine the exact emails that were accessed via the compromised Azure application. To perform this analysis, organizations must already have premium Office 365 E5 subscriptions. Only users that were already assigned the Office 365 E5 license will be eligible for this investigation. Enabling E5 licenses at the start of the O365/Azure investigation <em>will not</em> provide historical data.</p>
<p>To start this workflow, organizations will need to get the Application ID of the application(s) deemed compromised.</p>
<p><strong>To find the Application ID:</strong></p>
<ol>
<li>Go to the <a href="https://portal.azure.com/">Azure Portal</a> (https://portal.azure.com)</li>
<li>Search for <strong>Enterprise Applications, </strong>and navigate to the page</li>
<li>Select the application you have deemed as compromised</li>
<li>Select <strong>Properties</strong> on the left-hand side of the screen</li>
<li>Note and save the Application ID; this will be needed later.</li>
</ol>
<p>&nbsp;</p>
<p>See below for an example of the Application Properties page and location of Application ID:</p>
<figure id="attachment_7243" aria-describedby="caption-attachment-7243" style="width: 837px" class="wp-caption aligncenter"><a href="https://www.aon.com/cyber-solutions/wp-content/uploads/application_ID.png"><img decoding="async" class="size-full wp-image-7243" src="https://www.aon.com/cyber-solutions/wp-content/uploads/application_ID.png" alt="" width="837" height="693" srcset="https://www.aon.com/cyber-solutions/wp-content/uploads/application_ID.png 837w, https://www.aon.com/cyber-solutions/wp-content/uploads/application_ID-300x248.png 300w, https://www.aon.com/cyber-solutions/wp-content/uploads/application_ID-768x636.png 768w" sizes="(max-width: 837px) 100vw, 837px" /></a><figcaption id="caption-attachment-7243" class="wp-caption-text">Sample of an Application ID from https://portal.azure.com</figcaption></figure>
<p>With the Application ID documented, use the ID to determine all emails accessed by the application.</p>
<p><strong>Unified Audit Log / Mailbox Audit Log Analysis</strong></p>
<p>Within the Unified Audit Log and Mailbox Audit Logs, analyze the “MailItemsAccessed” operation. Both the <a href="https://docs.microsoft.com/en-us/microsoft-365/compliance/search-the-audit-log-in-security-and-compliance?view=o365-worldwide">Unified Audit Log</a> and <a href="https://docs.microsoft.com/en-us/powershell/module/exchange/search-mailboxauditlog?view=exchange-ps">Mailbox Audit Logs</a> contain this operation, and the client ID can be found in the “ClientAppID” field.</p>
<p>See below for a sample of attributes found in one MailItemsAccessed Unified Audit Log entry (values shown below are randomized or generic).
