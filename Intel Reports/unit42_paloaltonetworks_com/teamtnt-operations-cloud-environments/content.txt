<script type="application/ld+json">{"@context":"https:\/\/schema.org","@type":"BlogPosting","headline":"TeamTNT Actively Enumerating Cloud Environments to Infiltrate Organizations","name":"TeamTNT Actively Enumerating Cloud Environments to Infiltrate Organizations","description":"TeamTNT is targeting the credentials of 16 additional applications for the purpose of enumerating cloud environments and infiltrating organizations.","url":"https:\/\/unit42.paloaltonetworks.com\/teamtnt-operations-cloud-environments\/","mainEntityOfPage":"https:\/\/unit42.paloaltonetworks.com\/teamtnt-operations-cloud-environments\/","datePublished":"June 4, 2021","articleBody":"Executive Summary\r\nTeamTNT has been evolving their cloud-focused cryptojacking operations for some time now. TeamTNT operations have targeted and, after compromise, exfiltrated AWS credentials, targeted Kubernetes clusters and created new malware called Black-T that integrates open source cloud native tools to assist in their cryptojacking operations. TeamTNT operations are now using compromised AWS credentials to enumerate AWS cloud environments, via the AWS platform\u2019s API. These actions attempt to identify all Identity and Access Management (IAM) permissions, Elastic Compute Cloud (EC2) instances, Simple Storage Service (S3) buckets, CloudTrail configurations and CloudFormation operations granted to the compromised AWS credential. TeamTNT operations are now also targeting the credentials of 16 additional applications, including those of AWS and Google Cloud credentials, which may be stored on the compromised cloud instance, if installed.\r\n\r\nThe presence of Google Cloud credentials being targeted for collections represents the first known instance of an attacker group targeting IAM credentials on compromised cloud instances outside of AWS. While it is still possible that Microsoft Azure, Alibaba Cloud, Oracle Cloud or IBM Cloud IAM credentials could be targeted using similar methods, Unit 42 researchers have yet to find evidence of credentials from these cloud service providers (CSPs) being targeted. TeamTNT first started collecting AWS credentials on cloud instances they had compromised as early as August 2020.\r\n\r\nIn addition to the targeting of 16 application credentials from cloud applications and platforms, TeamTNT has added the usage of the open-source Kubernetes and cloud penetration toolset Peirates to their reconnaissance operations. With these techniques available, TeamTNT actors are increasingly more capable of gathering enough information in target AWS and Google Cloud environments to perform additional post-exploitation operations. This could lead to more cases of lateral movement and potential privilege escalation attacks that could ultimately allow TeamTNT actors to acquire administrative access to an organization\u2019s entire cloud environment.\r\n\r\nThat said, TeamTNT operations are still focused on cryptojacking. The TeamTNT cryptojacking operations represented within this writing have collected 6.52012192 Monero coins, which at the time of this writing equaled $1,788 USD. The mining operation was found to be operating at an average speed of 77.7KH\/s across eight mining workers. Operations using this Monero wallet address have continued for 114 days as of the time of this writing.\r\n\r\nPalo Alto Networks Prisma Cloud customers are protected from these threats through the Runtime Protection feature, Cryptominer Detection feature and the Prisma Cloud Compute Kubernetes Compliance Protection, which alerts on an insufficient Kubernetes configuration and provides secure alternatives. Additionally, Palo Alto Networks VM-Series and CN-Series products offer cloud protections that can prevent network connections from cloud instances toward known malicious IP addresses and URLs.\r\nEnumeration Techniques\r\nUnit 42 researchers identified one of TeamTNT\u2019s malware repositories, hxxp:\/\/45.9.148[.]35\/chimaera\/sh\/, which contained several bash scripts designed to perform cryptojacking operations, exploitation, lateral movement and credential scraping operations, as shown in Figure 1. This malware repository, referred to as the Chimaera Repository, highlights the expanding scope of TeamTNT operations within cloud environments as well as a target set for current and future operations.\r\n\r\n[caption id=\"attachment_119109\" align=\"aligncenter\" width=\"413\"] Figure 1. TeamTNT\u2019s Chimaera Repository.[\/caption]\r\n\r\nWithin the Chimaera repository, there were three scripts that specifically highlight TeamTNT\u2019s expanding cloud targeting capabilities and intent. The first script is grab_aws-data.sh, (SHA256: a1e9cd08073e4af3256b31e4b42f3aa69be40862b3988f964e96228f91236593), which focuses on enumerating AWS cloud environments using known AWS IAM credentials. The second script, bd_aws.sh, (SHA256: de3747a880c4b69ecaa92810f4aac20fe5f6d414d9ced29f1f7ebb82cd0f3945) scrapes all known Secure Shell Protocol (SSH) keys from an AWS instance and identifies all executable programs currently running on that instance. Finally, the script search.sh (SHA256: ed40bce040778e2227c869dac59f54c320944e19f77543954f40019e2f2b0c35) performs a search for configuration files containing application credentials stored on a given host. These scripts are newly discovered and directly highlight the targeting of cloud native applications within both AWS and Google Cloud environments.\r\nEnumerating AWS Environments\r\nThe bash script, grab_aws-data.sh, contains 70 unique AWS CommandLine Interface (AWS CLI) commands designed to enumerate seven AWS services, IAM configurations, EC2 instances, S3 buckets, support cases and direct connections, in addition to any CloudTrail and CloudFormation operations available to a given AWS IAM credential. As seen in Figure 2, all enumerated values obtained through the AWS enumeration process will be stored within the local directory \/var\/tmp\/...\/...TnT...\/aws-account-data\/ on the compromised system.\r\n\r\n[caption id=\"attachment_119111\" align=\"aligncenter\" width=\"669\"] Figure 2. TeamTNT\u2019s grab_aws.sh script.[\/caption]\r\n\r\nNavigate to the Appendix for a list of all 70 unique AWS CLI commands present within the TeamTNT script grab_aws-data.sh. As a summary, the TeamTNT script contained commands for the following seven AWS services:\r\n\r\n \t44 EC2 instance commands.\r\n \t14 IAM commands.\r\n \t4 Direct Connect commands.\r\n \t4 CloudFormation commands.\r\n \t2 CloudTrail commands.\r\n \t1 S3 command.\r\n \t1 Support command.\r\n\r\nCredential Scraping\r\nTeamTNT actors have also expanded their credential scraping capabilities to include the identification and collection of 16 unique applications, which may be present on the compromised cloud endpoint and for any of the known user accounts on the cloud instance, including the root account. There has been additional research involving this particular script. These applications were listed within the script search.sh:\r\n\r\n \tSSH keys.\r\n \tAWS keys.\r\n\r\n \tS3 clients.\r\n\r\n \ts3backer\r\n \ts3proxy\r\n \ts3ql (Google Cloud capable as well)\r\n \tpasswd-s3fs\r\n \ts3cfg\r\n\r\n\r\n\r\n\r\n \tDocker.\r\n \tGitHub.\r\n \tShodan.\r\n \tNgrok.\r\n \tPidgin.\r\n \tFilezilla.\r\n \tHexchat.\r\n \tGoogle Cloud.\r\n \tProject Jupyter.\r\n \tServer Message Block (SMB) clients.\r\n\r\nSeveral of these applications are noteworthy. The presence of Google Cloud credentials tops the list as this is the first known instance of an attacker group targeting IAM credentials outside of AWS (see Figure 3). It is possible that Microsoft Azure, Alibaba Cloud, Oracle Cloud or IBM Cloud environments could be targeted using similar techniques, but Unit 42 researchers have yet to find evidence of these CSPs being targeted. Researchers believe that it is only a matter of time before TeamTNT will develop functionality similar to that of grab_aws-data.sh described above, but targeting Google Cloud environments.\r\n\r\n[caption id=\"attachment_119113\" align=\"aligncenter\" width=\"900\"] Figure 3. TeamTNT\u2019s search.sh script searching for Google Cloud credentials.[\/caption]\r\nLateral Movement Operations\r\nIn addition to the 16 applications listed above, the following applications are specifically targeted for lateral movement operations.\r\nWeaveworks\r\nWithin the search.sh script, there are several applications identified which display evolving attack patterns for TeamTNT operations. Within the Chimaera repository, Unit 42 researchers identified several scripts that single out specific applications. One of those applications is Weaveworks (see Figure 4). Weave is a microservice network mesh application developed for container infrastructures such as Docker and Kubernetes, and allows for microservices to be running on one or multiple hosts while simultaneously maintaining network connectivity. By targeting Weave installations, TeamTNT operations have the potential to move laterally within a container infrastructure using the Weave network mesh application. As can be seen within the base64 encoded code in the script setup_scope.sh, (SHA256: 584c6efed8bbce5f2c52a52099aafb723268df799f4d464bf5582a9ee83165c1), TeamTNT is targeting Docker user accounts that contain Weave container information.\r\n\r\n[caption id=\"attachment_119115\" align=\"aligncenter\" width=\"865\"] Figure 4. TeamTNT script setup_scope.sh base64 decode code.[\/caption]\r\n\r\n[caption id=\"attachment_119117\" align=\"aligncenter\" width=\"786\"] Figure 5. Local Docker image creation for Monero mining.[\/caption]\r\nProject Jupyter\r\nAdditionally, the Project Jupyter application is listed as a target of TeamTNT operations through two sources within the Chimaera repository, first within the search.sh script as the target for credential scraping, and as a beta lateral movement script, spread_jupyter_tmp.sh (SHA256: 0d7912e62bc663c9ba6bff21ae809e458b227e3ceec0abac105d20d5dc533a22).\r\n\r\nUnit 42 researchers also found reference to Project Jupyter within a known TeamTNT actor\u2019s Twitter account. The Twitter account, @HildeTnT, posted the following image (Figure 6) on their Twitter feed, replying to a potentially compromised Jupyter endpoint. The German-language text translates to \u201cHahaha we take that as a compliment ^^ btw blocking the shell alone brings 0% security \u2026\u201d The presence of this Twitter exchange highlights the fact that TeamTNT is actively using the scripts listed within the Chimaera repository and targeting these additional cloud applications.\r\n\r\n[caption id=\"attachment_119119\" align=\"aligncenter\" width=\"577\"] Figure 6. TeamTNT actor replying to a message from a potentially compromised Jupyter endpoint.[\/caption]\r\nPeirates\r\nUnit 42 researchers have identified that TeamTNT actors are using the open source container and cloud penetration tool Peirates. As seen in Figure 7, Peirates allows actors to perform several attack functions against AWS and Kubernetes instances. This tool could enable TeamTNT actors to investigate and identify misconfigurations or potential vulnerabilities within Kubernetes and Cloud environments and could allow TeamTNT to perform additional compromising actions against cloud infrastructure.\r\n\r\n[caption id=\"attachment_119121\" align=\"aligncenter\" width=\"754\"] Figure 7. Peirates penetration testing options.[\/caption]\r\nMonero Mining Operations\r\nTeamTNT operations are still focused on cryptojacking. The previous sections presented the findings of new techniques used by TeamTNT actors to expand their cryptojacking infrastructure. The following section will focus on the findings related to the processes of mining applications TeamTNT uses to perform their cryptojacking operations.\r\nLocal Docker Image\r\nOf interest is the script file docker.container.local.spread.txt, which lists the name of a local Docker image, as shown in Figure 8. The Docker image is a local Docker image, meaning it is not hosted and downloaded from an external docker repository such as Docker Hub. Researchers did search Docker Hub for the presence of this Docker image and none were found.\r\n\r\n[caption id=\"attachment_119123\" align=\"aligncenter\" width=\"598\"] Figure 8. Contents of the docker.container.local.spread.txt.[\/caption]\r\n\r\nThe Docker container is created to provide a host for TeamTNT\u2019s Monero (XMR) mining operation. Shown in Figure 5, a Docker image is created with the name mangletmpuser\/fcminer. This image is then started and directed to navigate to the Chimaera repository file setup_xmr.sh, (SHA256: 5ddd226d400cc0b49d0175ba06a7e55cb2f5e9586111464bcf7b3bd709417904), which will initiate the Docker cryptomining process, using the open source XMRig application within a Docker container.\r\nNew Monero Wallet\r\nUnit 42 researchers identified a new Monero wallet address that has never before been witnessed in relation to TeamTNT operations, 46EPFzvnX5GH61ejkPpNcRNm8kVjs8oHS9VwCkKRCrJX27XEW2y1NPLfSa54DGHxqnKfzDUVW1jzBfekk3hrCVCmAUrFd3H. This Monero wallet address was associated with the Monero public mining pool pool.supportxmr[.]com:3333, as shown in Figure 9.\r\n\r\n[caption id=\"attachment_119125\" align=\"aligncenter\" width=\"900\"] Figure 9. SupportXMR public mining pool configuration.[\/caption]In Figure 10, this mining pool address displays that the TeamTNT mining operation has collected 6.52012192 Monero coins, which at the time of this writing equaled $1,788 USD. The mining operation was found to be operating at 77.7KH\/s, across eight mining workers at the time of this writing, and operations using this Monero wallet address have continued for 114 days. At an operating speed of 77.7KH\/s, this operation is considered to be a small mining operation for a group like TeamTNT.\r\n\r\n[caption id=\"attachment_119127\" align=\"aligncenter\" width=\"900\"] Figure 10. SupportXMR mining pool dashboard.[\/caption]\r\nConclusion\r\nGiven TeamTNT\u2019s integration of tools such as Peirates, their targeting of cloud native network mesh applications such as Weave, their operations around Kubernetes and Black-T, and their targeting and subsequent taunting of organizations using Project Jupyter, TeamTNT actors are suspected to be employing all tools listed within this blog on a regular basis. TeamTNT actors are specifically targeting cloud platforms in an attempt to circumvent future security detection tools and embed themselves into the organization\u2019s cloud environment.\r\n\r\nWe recommend that organizations operating with cloud environments monitor for and block all network connections associated with TeamTNT\u2019s Chimaera repository, as well as historic Command and Control (C2) endpoints. Using a cloud native security platform will significantly reduce the cloud infrastructure\u2019s attack surface and allow organizations to monitor for risks.\r\n\r\nThe following tips are highly recommended by Unit 42 researchers to assist in the protection of cloud infrastructure.\r\n\r\n \tEnforce least-privilege IAM access policies to all cloud IAM roles and permissions. Where applicable, use short-lived or one-time-use IAM credentials for service accounts.\r\n \tMonitor and block network traffic to known malicious endpoints.\r\n \tOnly deploy vetted container images within production environments.\r\n \tImplement and use Infrastructure as Code (IaC) scanning platforms to prevent insecure cloud instances from being deployed into production environments.\r\n \tUse cloud infrastructure configuration scanning tools that enable governance, risk management and compliance (GRC) to identify potentially threatening misconfigurations.\r\n \tUse cloud endpoint agents to monitor and prevent the running of known malicious applications within cloud infrastructure.\r\n\r\nPalo Alto Networks Prisma Cloud customers are protected from these threats through the Runtime Protection feature, Cryptominer Detection feature and the Prisma Cloud Compute Kubernetes Compliance Protection, which alerts on an insufficient Kubernetes configuration and provides secure alternatives.                 <p class="wpml-ls-statics-post_translations wpml-ls">This post is also available in: 
    <span class="wpml-ls-slot-post_translations wpml-ls-item wpml-ls-item-ja wpml-ls-first-item wpml-ls-last-item wpml-ls-item-legacy-post-translations"><a href="https://unit42.paloaltonetworks.jp/teamtnt-operations-cloud-environments/" class="wpml-ls-link"><span class="wpml-ls-native" lang="ja">日本語</span><span class="wpml-ls-display"><span class="wpml-ls-bracket"> (</span>Japanese<span class="wpml-ls-bracket">)</span></span></a></span></p><h2><a id="post-119106-_z4eg6aenzfj5"></a>Executive Summary</h2>
<p>TeamTNT has been evolving their cloud-focused cryptojacking operations for some time now. TeamTNT operations have targeted and, after compromise, <a href="https://www.cadosecurity.com/post/team-tnt-the-first-crypto-mining-worm-to-steal-aws-credentials">exfiltrated AWS credentials</a>, targeted <a href="https://unit42.paloaltonetworks.com/hildegard-malware-teamtnt/">Kubernetes clusters</a> and created new malware called Black-T that <a href="https://unit42.paloaltonetworks.com/black-t-cryptojacking-variant/">integrates open source cloud native tools</a> to assist in their cryptojacking operations. TeamTNT operations are now using compromised AWS credentials to enumerate AWS cloud environments, via the AWS platform’s API. These actions attempt to identify all <a href="https://docs.aws.amazon.com/cli/latest/reference/iam/index.html">Identity and Access Management (IAM)</a> permissions, <a href="https://docs.aws.amazon.com/cli/latest/reference/ec2/index.html">Elastic Compute Cloud (EC2)</a> instances, <a href="https://docs.aws.amazon.com/cli/latest/reference/s3/index.html">Simple Storage Service (S3)</a> buckets, <a href="https://docs.aws.amazon.com/cli/latest/reference/cloudtrail/index.html">CloudTrail</a> configurations and <a href="https://docs.aws.amazon.com/cli/latest/reference/cloudformation/index.html">CloudFormation</a> operations granted to the compromised AWS credential. TeamTNT operations are now also targeting the credentials of 16 additional applications, including those of AWS and Google Cloud credentials, which may be stored on the compromised cloud instance, if installed.</p>
<p>The presence of Google Cloud credentials being targeted for collections represents the first known instance of an attacker group targeting IAM credentials on compromised cloud instances outside of AWS. While it is still possible that Microsoft Azure, Alibaba Cloud, Oracle Cloud or IBM Cloud IAM credentials could be targeted using similar methods, Unit 42 researchers have yet to find evidence of credentials from these cloud service providers (CSPs) being targeted. TeamTNT first started collecting AWS credentials on cloud instances they had compromised as early as <a href="https://www.cadosecurity.com/post/team-tnt-the-first-crypto-mining-worm-to-steal-aws-credentials">August 2020</a>.</p>
<p>In addition to the targeting of 16 application credentials from cloud applications and platforms, TeamTNT has added the usage of the open-source Kubernetes and cloud penetration toolset <a href="https://github.com/inguardians/peirates">Peirates</a> to their reconnaissance operations. With these techniques available, TeamTNT actors are increasingly more capable of gathering enough information in target AWS and Google Cloud environments to perform additional post-exploitation operations. This could lead to more cases of lateral movement and potential privilege escalation attacks that could ultimately allow TeamTNT actors to acquire administrative access to an organization’s entire cloud environment.</p>
<p>That said, TeamTNT operations are still focused on cryptojacking. The TeamTNT cryptojacking operations represented within this writing have collected 6.52012192 Monero coins, which at the time of this writing equaled $1,788 USD. The mining operation was found to be operating at an average speed of 77.7KH/s across eight mining workers. Operations using this Monero wallet address have continued for 114 days as of the time of this writing.</p>
<p>Palo Alto Networks <a href="https://www.paloaltonetworks.com/prisma/cloud">Prisma Cloud</a> customers are protected from these threats through the Runtime Protection feature, Cryptominer Detection feature and the Prisma Cloud Compute Kubernetes Compliance Protection, which alerts on an insufficient Kubernetes configuration and provides secure alternatives. Additionally, Palo Alto Networks <a href="https://www.paloaltonetworks.com/prisma/vm-series">VM-Series</a> and <a href="https://www.paloaltonetworks.com/network-security/cn-series">CN-Series</a> products offer cloud protections that can prevent network connections from cloud instances toward known malicious IP addresses and URLs.</p>
<h2><a id="post-119106-_2wdkmkdbm629"></a>Enumeration Techniques</h2>
<p>Unit 42 researchers identified one of TeamTNT’s malware repositories, <span style="font-family: 'courier new', courier, monospace;">hxxp://45.9.148[.]35/chimaera/sh/</span>, which contained several bash scripts designed to perform cryptojacking operations, exploitation, lateral movement and credential scraping operations, as shown in Figure 1. This malware repository, referred to as the Chimaera Repository, highlights the expanding scope of TeamTNT operations within cloud environments as well as a target set for current and future operations.</p>
<figure id="attachment_119109" aria-describedby="caption-attachment-119109" style="width: 413px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-119109" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2021/06/word-image.png" alt="This malware repository, referred to as the Chimaera Repository, highlights the expanding scope of TeamTNT operations within cloud environments as well as a target set for current and future operations." width="413" height="735" /><figcaption id="caption-attachment-119109" class="wp-caption-text">Figure 1. TeamTNT’s Chimaera Repository.</figcaption></figure>
<p>Within the Chimaera repository, there were three scripts that specifically highlight TeamTNT’s expanding cloud targeting capabilities and intent. The first script is <span style="font-family: 'courier new', courier, monospace;">grab_aws-data.sh</span>, (SHA256: <span style="font-family: 'courier new', courier, monospace;">a1e9cd08073e4af3256b31e4b42f3aa69be40862b3988f964e96228f91236593</span>), which focuses on enumerating AWS cloud environments using known AWS IAM credentials. The second script, <span style="font-family: 'courier new', courier, monospace;">bd_aws.sh</span>, (SHA256: <span style="font-family: 'courier new', courier, monospace;">de3747a880c4b69ecaa92810f4aac20fe5f6d414d9ced29f1f7ebb82cd0f3945</span>) scrapes all known Secure Shell Protocol (SSH) keys from an AWS instance and identifies all executable programs currently running on that instance. Finally, the script <span style="font-family: 'courier new', courier, monospace;">search.sh</span> (SHA256: <span style="font-family: 'courier new', courier, monospace;">ed40bce040778e2227c869dac59f54c320944e19f77543954f40019e2f2b0c35</span>) performs a search for configuration files containing application credentials stored on a given host. These scripts are newly discovered and directly highlight the targeting of cloud native applications within both AWS and Google Cloud environments.</p>
<h4><a id="post-119106-_hta6dcm3rdwe"></a>Enumerating AWS Environments</h4>
<p>The bash script, <span style="font-family: 'courier new', courier, monospace;">grab_aws-data.sh</span>, contains 70 unique AWS CommandLine Interface (<a href="https://aws.amazon.com/cli/">AWS CLI</a>) commands designed to enumerate seven AWS services, <a href="https://docs.aws.amazon.com/cli/latest/reference/iam/index.html">IAM configurations</a>, <a href="https://docs.aws.amazon.com/cli/latest/reference/ec2/index.html">EC2 instances</a>, <a href="https://docs.aws.amazon.com/cli/latest/reference/s3/index.html">S3 buckets</a>, <a href="https://docs.aws.amazon.com/cli/latest/reference/support/index.html">support cases</a> and <a href="https://docs.aws.amazon.com/cli/latest/reference/directconnect/index.html">direct connections</a>, in addition to any <a href="https://docs.aws.amazon.com/cli/latest/reference/cloudtrail/index.html">CloudTrail</a> and <a href="https://docs.aws.amazon.com/cli/latest/reference/cloudformation/index.html">CloudFormation</a> operations available to a given AWS IAM credential. As seen in Figure 2, all enumerated values obtained through the AWS enumeration process will be stored within the local directory <span style="font-family: 'courier new', courier, monospace;">/var/tmp/.../...TnT.../aws-account-data/</span> on the compromised system.</p>
<figure id="attachment_119111" aria-describedby="caption-attachment-119111" style="width: 669px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-119111" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2021/06/word-image-1.png" alt="All enumerated values obtained through the AWS enumeration process will be stored within the local directory /var/tmp/.../...TnT.../aws-account-data/ on the compromised system." width="669" height="354" /><figcaption id="caption-attachment-119111" class="wp-caption-text">Figure 2. TeamTNT’s grab_aws.sh script.</figcaption></figure>
<p>Navigate to the Appendix for a list of all 70 unique AWS CLI commands present within the TeamTNT script <span style="font-family: 'courier new', courier, monospace;">grab_aws-data.sh</span>. As a summary, the TeamTNT script contained commands for the following seven AWS services:</p>
<ul>
<li>44 EC2 instance commands.</li>
<li>14 IAM commands.</li>
<li>4 Direct Connect commands.</li>
<li>4 CloudFormation commands.</li>
<li>2 CloudTrail commands.</li>
<li>1 S3 command.</li>
<li>1 Support command.</li>
</ul>
<h4><a id="post-119106-_mkgtkc9mv8sz"></a>Credential Scraping</h4>
<p>TeamTNT actors have also expanded their credential scraping capabilities to include the identification and collection of 16 unique applications, which may be present on the compromised cloud endpoint and for any of the known user accounts on the cloud instance, including the root account. There has been <a href="https://www.trendmicro.com/en_us/research/21/e/teamtnt-extended-credential-harvester-targets-cloud-services-other-software.html">additional research</a> involving this particular script. These applications were listed within the script <span style="font-family: 'courier new', courier, monospace;">search.sh</span>:</p>
<ul>
<li>SSH keys.</li>
<li>AWS keys.
<ul>
<li>S3 clients.
<ul>
<li><a href="https://github.com/archiecobbs/s3backer">s3backer</a></li>
<li><a href="https://github.com/gaul/s3proxy">s3proxy</a></li>
<li><a href="https://github.com/s3ql/s3ql">s3ql</a> (Google Cloud capable as well)</li>
<li><a href="https://github.com/s3fs-fuse/s3fs-fuse">passwd-s3fs</a></li>
<li><a href="https://s3tools.org/kb/item14.htm">s3cfg</a></li>
</ul>
</li>
</ul>
</li>
<li>Docker.</li>
<li>GitHub.</li>
<li>Shodan.</li>
<li>Ngrok.</li>
<li><a href="https://developer.pidgin.im/wiki/ConfigurationFiles">Pidgin</a>.</li>
<li><a href="https://filezilla-project.org/">Filezilla</a>.</li>
<li><a href="https://hexchat.github.io/">Hexchat</a>.</li>
<li>Google Cloud.</li>
<li><a href="https://jupyter.org/">Project Jupyter</a>.</li>
<li>Server Message Block (SMB) clients.</li>
</ul>
<p>Several of these applications are noteworthy. The presence of Google Cloud credentials tops the list as this is the first known instance of an attacker group targeting IAM credentials outside of AWS (see Figure 3). It is possible that Microsoft Azure, Alibaba Cloud, Oracle Cloud or IBM Cloud environments could be targeted using similar techniques, but Unit 42 researchers have yet to find evidence of these CSPs being targeted. Researchers believe that it is only a matter of time before TeamTNT will develop functionality similar to that of <span style="font-family: 'courier new', courier, monospace;">grab_aws-data.sh</span> described above, but targeting Google Cloud environments.</p>
<figure id="attachment_119113" aria-describedby="caption-attachment-119113" style="width: 900px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-119113" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2021/06/word-image-2.png" alt="The light blue box highlights the section of code that shows TeamTNT's search.sh script searching for Google Cloud credentials, the first known instance of an attacker group targeting IAM credentials outside of AWS, potentially for the purpose of enumerating cloud environments" width="900" height="187" /><figcaption id="caption-attachment-119113" class="wp-caption-text">Figure 3. TeamTNT’s search.sh script searching for Google Cloud credentials.</figcaption></figure>
<h2><a id="post-119106-_l3b5spgvw2d9"></a>Lateral Movement Operations</h2>
<p>In addition to the 16 applications listed above, the following applications are specifically targeted for lateral movement operations.</p>
<h4><a id="post-119106-_uad9pnulpgcp"></a>Weaveworks</h4>
<p>Within the search.sh script, there are several applications identified which display evolving attack patterns for TeamTNT operations. Within the Chimaera repository, Unit 42 researchers identified several scripts that single out specific applications. One of those applications is <a href="https://www.weave.works/docs/net/latest/overview/">Weaveworks</a> (see Figure 4). Weave is a microservice network mesh application developed for container infrastructures such as Docker and Kubernetes, and allows for microservices to be running on one or multiple hosts while simultaneously maintaining network connectivity. By targeting Weave installations, TeamTNT operations have the potential to move laterally within a container infrastructure using the Weave network mesh application. As can be seen within the base64 encoded code in the script <span style="font-family: 'courier new', courier, monospace;">setup_scope.sh</span>, (SHA256: <span style="font-family: 'courier new', courier, monospace;">584c6efed8bbce5f2c52a52099aafb723268df799f4d464bf5582a9ee83165c1</span>), TeamTNT is targeting Docker user accounts that contain Weave container information.</p>
<figure id="attachment_119115" aria-describedby="caption-attachment-119115" style="width: 865px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-119115" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2021/06/word-image-3.png" alt="As can be seen within the base64 encoded code in the script setup_scope.sh, (SHA256: 584c6efed8bbce5f2c52a52099aafb723268df799f4d464bf5582a9ee83165c1), TeamTNT is targeting Docker user accounts that contain Weave container information." width="865" height="335" /><figcaption id="caption-attachment-119115" class="wp-caption-text">Figure 4. TeamTNT script setup_scope.sh base64 decode code.</figcaption></figure>
<figure id="attachment_119117" aria-describedby="caption-attachment-119117" style="width: 786px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-119117" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2021/06/word-image-4.png" alt="This shows one way TeamTNT is able to target Docker for the purpose of cryptojacking. " width="786" height="338" /><figcaption id="caption-attachment-119117" class="wp-caption-text">Figure 5. Local Docker image creation for Monero mining.</figcaption></figure>
<h4><a id="post-119106-_c6bdzsvxb79v"></a>Project Jupyter</h4>
<p>Additionally, the Project Jupyter application is listed as a target of TeamTNT operations through two sources within the Chimaera repository, first within the <span style="font-family: 'courier new', courier, monospace;">search.sh</span> script as the target for credential scraping, and as a beta lateral movement script, <span style="font-family: 'courier new', courier, monospace;">spread_jupyter_tmp.sh</span> (SHA256: <span style="font-family: 'courier new', courier, monospace;">0d7912e62bc663c9ba6bff21ae809e458b227e3ceec0abac105d20d5dc533a22</span>).</p>
<p>Unit 42 researchers also found reference to Project Jupyter within a known TeamTNT actor’s Twitter account. The Twitter account, <span style="font-family: 'courier new', courier, monospace;">@HildeTnT</span>, posted the following image (Figure 6) on their Twitter feed, replying to a potentially compromised Jupyter endpoint. The German-language text translates to “Hahaha we take that as a compliment ^^ btw blocking the shell alone brings 0% security …” The presence of this Twitter exchange highlights the fact that TeamTNT is actively using the scripts listed within the Chimaera repository and targeting these additional cloud applications.</p>
<figure id="attachment_119119" aria-describedby="caption-attachment-119119" style="width: 577px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-119119" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2021/06/word-image-5.png" alt="The German-language text translates to “Hahaha we take that as a compliment ^^ btw blocking the shell alone brings 0% security …” The presence of this Twitter exchange highlights the fact that TeamTNT is actively using the scripts listed within the Chimaera repository and targeting these additional cloud applications." width="577" height="483" /><figcaption id="caption-attachment-119119" class="wp-caption-text">Figure 6. TeamTNT actor replying to a message from a potentially compromised Jupyter endpoint.</figcaption></figure>
<h2><a id="post-119106-_wm93cmkuhden"></a>Peirates</h2>
<p>Unit 42 researchers have identified that TeamTNT actors are using the open source container and cloud penetration tool <a href="https://github.com/inguardians/peirates">Peirates</a>. As seen in Figure 7, Peirates allows actors to perform several attack functions against AWS and Kubernetes instances. This tool could enable TeamTNT actors to investigate and identify misconfigurations or potential vulnerabilities within Kubernetes and Cloud environments and could allow TeamTNT to perform additional compromising actions against cloud infrastructure.</p>

<h2><a id="post-119106-_3s15frjkf8b7"></a>Conclusion</h2>
<p>Given TeamTNT’s integration of tools such as Peirates, their targeting of cloud native network mesh applications such as Weave, their operations around Kubernetes and Black-T, and their targeting and subsequent taunting of organizations using Project Jupyter, TeamTNT actors are suspected to be employing all tools listed within this blog on a regular basis. TeamTNT actors are specifically targeting cloud platforms in an attempt to circumvent future security detection tools and embed themselves into the organization’s cloud environment.</p>
<p>We recommend that organizations operating with cloud environments monitor for and block all network connections associated with TeamTNT’s Chimaera repository, as well as historic Command and Control (C2) endpoints. Using a cloud native security platform will significantly reduce the cloud infrastructure’s attack surface and allow organizations to monitor for risks.</p>
<p>The following tips are highly recommended by Unit 42 researchers to assist in the protection of cloud infrastructure.</p>
<ul>
<li>Enforce least-privilege IAM access policies to all cloud IAM roles and permissions. Where applicable, use short-lived or one-time-use IAM credentials for service accounts.</li>
<li>Monitor and block network traffic to known malicious endpoints.</li>
<li>Only deploy vetted container images within production environments.</li>
<li>Implement and use Infrastructure as Code (IaC) scanning platforms to prevent insecure cloud instances from being deployed into production environments.</li>
<li>Use cloud infrastructure configuration scanning tools that enable governance, risk management and compliance (GRC) to identify potentially threatening misconfigurations.</li>
<li>Use cloud endpoint agents to monitor and prevent the running of known malicious applications within cloud infrastructure.</li>
</ul>
<p>Palo Alto Networks <a href="https://www.paloaltonetworks.com/prisma/cloud">Prisma Cloud</a> customers are protected from these threats through the Runtime Protection feature, Cryptominer Detection feature and the Prisma Cloud Compute Kubernetes Compliance Protection, which alerts on an insufficient Kubernetes configuration and provides secure alternatives. Additionally, Palo Alto Networks <a href="https://www.paloaltonetworks.com/prisma/vm-series">VM-Series</a> and <a href="https://www.paloaltonetworks.com/network-security/cn-series">CN-Series</a> products offer cloud protections that can prevent network connections from cloud instances toward known malicious IP addresses and URLs.</p>

			$(subscription_form_button).removeClass('is-disabled');
		}
		setTimeout(function() {
			$(email).focus();
			$('.g-recaptcha iframe').attr('tabindex', '-1');
		}, 100)
	}

		subscribeSuccess = false;
		$(subscription_form_button).addClass('is-disabled', true);
	} 
	$(subscription_form).submit(function(e) {
		e.preventDefault();
		e.stopImmediatePropagation();
		updateEmailMask();
		var success = true;
		var form = $(this);
		var mail = form.find('input[name="Email"]');
		
		if (mail.val() === '') {
			mail.addClass('has-error');
			showError(1);
			success = false;	
		}  else if (!isEmail(mail.val())){
			showError(2);
			success = false;
		}
		else {
			mail.removeClass('has-error');
			$('.error-mail').addClass('d-none');
		}

		if (!subscribeSuccess) {
			$('.error-recaptcha').removeClass('d-none');
		} else {
			$('.error-recaptcha').addClass('d-none');
		}

		if (success && subscribeSuccess) {
			$.ajax({

					form.find('.form-success-message').html('<p class="success-message">You have been successfully subscribed</p>');
					form.find('button').removeClass('is-loading');
					$(email).val('');
					clearError();

				error: function(jqXHR, textStatus, errorThrown) {
					$(subscription_form_button).addClass('is-disabled', true);
					form.find('button').removeClass('is-loading');
					
				}
			});
		}
		return false;
	});
	function showError(error_type){
		if(error_type == 1) {
			$('.error-mail').text("Please enter the email address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		} else if(error_type == 2){
			$('.error-mail').text("Please provide a valid e-mail address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		}	
		$(subscription_form_button).removeClass('is-loading');
	}
	function clearError(){
		$('.error-mail').text("").removeClass('error-show');;
		$(subscription_form_button).removeClass('is-loading');
		$(subscription_form_button).removeClass('is-disabled');
	}
	
	$(email).on('input', function (event) { 
		var email = $(this).val();
		if (isEmail(email) ) {			
			clearError();
		} else if(email == ""){
			clearError();
		} else{
			showError(2);			  
		}
    });

	function isEmail(email) {
		var re = /^(([^<>()\[\]\\.,;:\s@"]+(\.[^<>()\[\]\\.,;:\s@"]+)*)|(".+"))@((\[[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\])|(([a-zA-Z\-0-9]+\.)+[a-zA-Z]{2,}))$/;
		return re.test(String(email).toLowerCase());
	}

         if(!captcha_loaded){

		var email = $("#unit42footerSubscription_form input[name='Email']").val();

			$("#unit42footerSubscription_form input[name='emailFormMask']").val(maskedEmail);

	}

	function maskEmailAddress (emailAddress) {
		function mask(str) {
			var strLen = str.length;
			if (strLen > 4) {
				return str.substr(0, 1) + str.substr(1, strLen - 1).replace(/\w/g, '*') + str.substr(-1,1);
			}
			return str.replace(/\w/g, '*');
		}
		return emailAddress.replace(/([\w.]+)@([\w.]+)(\.[\w.]+)/g, function (m, p1, p2, p3) {
			return mask(p1) + '@' + mask(p2) + p3;
		});

		return emailAddress;
	}
