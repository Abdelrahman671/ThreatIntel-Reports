							<div class="content "><h1 id="image-file-execution-options-ifeo">Image File Execution Options (IFEO)</h1><div class="display-flex justify-content-space-between align-items-center flex-wrap-wrap page-metadata-container">
										<div class="margin-right-xxs">
											<ul class="metadata page-metadata" data-bi-name="page info" lang="en-us" dir="ltr"><li>Article</li><li>
														<time class="is-invisible" data-article-date aria-label="Article review date" datetime="2010-03-24T00:00:00.000Z" data-article-date-source="ms.date">03/24/2010</time>
													</li></ul>
										</div></div><nav id="center-doc-outline" class="doc-outline is-hidden-desktop display-none-print margin-bottom-sm" data-bi-name="intopic toc" aria-label="In this article">
											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>&nbsp;Image File Execution options provides you with a mechanism to always launch an executable directly under the debugger. This is extremely useful if you ever need to investigate issues in the executable's startup code (services especially). You can set the IFEO options directly via the registry or indirectly using the Gflags tools (available with the <a href="https://www.microsoft.com/whdc/devtools/debugging/default.mspx" data-linktype="external">Window debugging toolkit</a>).&nbsp;&nbsp;&nbsp;</p>
<p>You need to create a registry key and populate it with a value as follows -</p>
<table>
<colgroup>
<col>
<col>
</colgroup>
<tbody>
<tr class="odd">
<td><p>Key</p></td>
<td><p><strong>"HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Image File Execution Options\&lt;executable&gt;"</strong></p></td>
</tr>
<tr class="even">
<td><p>Value</p></td>
<td><p><strong>Debugger : REG_SZ : &lt;full-path to your favorite debugger&gt;</strong></p></td>
</tr>
</tbody>
</table>
<p>&nbsp;&nbsp;&nbsp;</p>
<p>You do not need the full path to the application, only the exe name will suffice. However you do need the full path to the debugger.&nbsp; As an example, we look at launching notepad under ntsd, you would be creating the following -</p>
<table>
<colgroup>
<col>
<col>
</colgroup>
<tbody>
<tr class="odd">
<td><p>Key</p></td>
<td><p><strong>"HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Image File Execution Options\notepad.exe"</strong></p></td>
</tr>
<tr class="even">
<td><p>Value</p></td>
<td><p><strong>Debugger : REG_SZ : "c:\dbg\ntsd.exe -g"</strong></p></td>
</tr>
</tbody>
</table>
<p>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;</p>
<p>&nbsp;</p>
<p>You can also use Gflags to set IFEO too -</p>
<p><img src="https://msdntnarchive.blob.core.windows.net/media/MSDNBlogsFS/prod.evol.blogs.msdn.com/CommunityServer.Blogs.Components.WeblogFiles/00/00/00/55/05/0572.gflags.gif" alt="" data-linktype="external">&nbsp;&nbsp;&nbsp;</p>
<p>&nbsp;</p>
<p><strong>How does IFEO work?</strong> &nbsp;&nbsp;&nbsp;&nbsp;</p>
<p><a href="https://msdn2.microsoft.com/en-us/library/ms682425.aspx" data-linktype="external">Kernel32!CreateProcess</a> when called without the DEBUG_PROCESS or DEBUG_ONLY_THIS_PROCESS <a href="https://msdn2.microsoft.com/en-us/library/ms684863.aspx" data-linktype="external">creation flags</a>, checks the registry to see if IFEO has been set on the executable that it is launching. If yes, then it simply prepends the debugger path to the executable name, effectively getting the executable to launch under the debugger. If you do not specify the correct path to the debugger, then you'll probably get greeted with a "file not found" error. In our notepad/ntsd example above, Kernel32!CreateProcess ends up invoking -</p>
<p><strong>"c:\dbg\ntsd.exe -g notepad.exe"</strong> &nbsp;&nbsp;&nbsp;&nbsp;</p>
<p>Now ntsd eventually launches notepad under the debugger by calling Kernel32!CreateProcess with one of the following creation flags - DEBUG_PROCESS or DEBUG_ONLY_THIS_PROCESS. The presence of any of these creation flags forces Kernel32!CreateProcess to bypass IFEO options this time around (else we would have been running into an endless loop) and actually launch the executable under the debugger.&nbsp;&nbsp;&nbsp;&nbsp;</p>
<p>&nbsp; &nbsp;</p>
<p><strong>IFEO and 64 bit -</strong> &nbsp;&nbsp;&nbsp;&nbsp;</p>
<p>A word of caution - For 32 bit executable running in the WOW on X64 machines, your natural tendency might be to create the registry key in the syswow node -</p>
<p><strong>"HKLM\SOFTWARE\Wow6432Node\Microsoft\Windows NT\CurrentVersion\Image File Execution Options\&lt;executable&gt;"</strong> &nbsp;&nbsp;&nbsp;&nbsp;</p>
<p>However Gregg Miskelly <a href="../greggm/inside-image-file-execution-options-debugging" data-linktype="relative-path">notes that</a> you should set the IFEO corresponding to the bit-ness to the application calling into kernel32!CreateProcess to launch the executable -&nbsp;&nbsp;&nbsp;&nbsp;</p>
<p><em>"On Win 64, there are two copies of HKEY_LOCAL_MACHINE\Software (one for 32-bit apps, and one for 64-bit apps), and therefore there are two copies of these options. However, where the operating system looks isn't dependant on the bit-ness of the application that is going to be debugged (which is what you would probably expect). Instead, it is dependent on the bit-ness of the application that called CreateProcess."</em></p>
<p>&nbsp;&nbsp;</p>
<p><strong>Other IFEO caveats -</strong> &nbsp;</p>
<p>Raymond Chen notes the following caveat in <a href="https://blogs.msdn.com/oldnewthing/archive/2007/07/02/3652873.aspx" data-linktype="external">his blog entry</a>&nbsp; -</p>
<p><em>"If you passed special parameters via the STARTUPINFO structure, those parameters get passed to the debugger. And the PROCESS_INFO that is returned by the CreateProcess function describes the debugger, not the process being debugged."</em></p>
<p>&nbsp;&nbsp;&nbsp;</p>
<p><strong>IFEO and Managed debuggers -</strong></p>
<p>IFEO can only be used for native or interop debugging, but not for managed debugging. Mike Stall has an <a href="../jmstall/ifeo-and-managed-debugging" data-linktype="relative-path">excellent blog entry</a> that describes in great detail exactly why. The gist is this - Managed debuggers like Mdbg/cordbg/VS.NET use ICorDebug::CreateProcess to launch managed executables under the debugger. However for managed debugging, the debugger should call ICorDebug::CreateProcess without the DEBUG_PROCESS or DEBUG_ONLY_THIS_PROCESS creation flags (<a href="https://msdn2.microsoft.com/en-us/library/ms232508.aspx" data-linktype="external">this is publicly documented</a>). This API internally ends up calling Kernel32!CreateProcess without the DEBUG_PROCESS or DEBUG_ONLY_THIS_PROCESS creation flags. This leads to the endless loop that I described above. Is this an ICorDebug API design flaw?&nbsp; Not really. Just an oversight in my opinion - The API designers missed one scenario. Maybe in the next version of the CLR this will be fixed (I do not know for sure).</p>
<p>&nbsp;&nbsp;&nbsp;</p>
<p><strong>Some excellent resources on IFEO -</strong></p>
<table>
<colgroup>
<col>
<col>
</colgroup>

<p><strong>TIP of the day -</strong></p>
<p><strong>Question</strong> - System services can launch before the user has a chance to log on. So how do you debug the startup code of these system services?</p>
<p><strong>Answer</strong> - Put the machine under kernel Debugger (KD), use IFEO to launch the service under NTSD (use ntsd's&nbsp; "-d" option to pipe the ntsd output to KD) and reboot the machine. When the system service launches, it will be launched under ntsd.&nbsp; The ntsd debugger will automatically cause it break into KD when it encounters the initial loader breakpoint. The debugging session will begin in user mode automatically (yipee!). After you are done debugging, switch control to KD by issuing ".breakin" command.</p>
