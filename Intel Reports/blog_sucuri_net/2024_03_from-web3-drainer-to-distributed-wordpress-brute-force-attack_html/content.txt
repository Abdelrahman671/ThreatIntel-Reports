</figure></div><div class="entry-container"><div class="entry-content-wrap"><div class="entry-content"><p>Two weeks ago we discussed a new development in website hacks: <a href="https://blog.sucuri.net/2024/02/web3-crypto-malware-angel-drainer.html">Web3 crypto wallet drainers</a>. We’ve been closely following the most significant variant which injects drainers using the external <b>cachingjs/turboturbo.js</b> script. Our SiteCheck <a href="https://sitecheck.sucuri.net/">website scanner</a> has already detected this version on over 1,200 sites since the beginning of February, 2024.</p><p>Since our last post, this malware campaign has seen two new iterations resulting in distributed <a href="https://sucuri.net/guides/what-is-brute-force-attack/">brute force attacks</a> against target WordPress websites from the browsers of completely innocent and unsuspecting site visitors. Sounds unrelated, right? Well, let’s take a closer look.</p><h3>Iteration 1: dynamiclink[.]lol drainer</h3><p>On <b>Feb 20, 2024</b>, the <b>turboturbo.js</b> script domain changed — this time from <b>dynamiclinks[.]cfd/cachingjs/turboturbo.js</b> to <b>dynamiclink[.]lol/cachingjs/turboturbo.js</b>.</p><p>This new wave started on the very same day the new <b>dynamiclink[.]lol</b> domain was registered and hosted on the server with IP <b>93.123.39.199</b>.</p><p>The drainer <b>settings.js</b> file also saw a small change: The <b>worker_address</b> was modified from <b>0xc5cE06FC4E2A26514afe69e25a6B36ab51F9FE42</b> to <b>0xFe8a95604CB87A9C6C5b1Ec681Bcfb4aE77F0c31</b>.</p><h3>Iteration 2: dynamic-linx[.]com script</h3><p>On <b>Feb 23, 2024</b>, attackers registered another new <b>dynamic-linx[.]com</b> domain (also hosted on <b>93.123.39.199</b> and <b>94.156.8.251</b>). that attackers are able to try more than <strong>41,800 passwords for each site</strong>.</p><p>Now that we have a simple understanding of the script’s task management features, let’s take a look at how the attacker’s use these scripts to launch their attacks against victim sites.</p><h3>Attack stages and lifecycle</h3><p>The attack consists of five key stages that allow a bad actor to leverage already compromised websites to launch distributed brute force attacks against thousands of other potential victim sites.</p><ul><li aria-level="1"><b>Stage 1: Obtain URLs of WordPress sites.</b> The attackers either crawl the internet themselves or use various search engines and databases to obtain lists of target WordPress sites.</li><li aria-level="1"><b>Stage 2: Extract author usernames.</b> Attackers then scan the target sites, extracting real usernames of authors that post on those domains.</li><li aria-level="1"><b>Stage 3: Inject malicious scripts.</b> Attackers then inject their <b>dynamic-linx[.]com/chx.js</b> script to websites that they have already compromised.</li><li aria-level="1"><b>Stage 4: Brute force credentials.</b>  As normal site visitors open infected web pages, the malicious script is loaded. Behind the scenes, the visitors’ browsers conduct a distributed brute force attack on thousands of target sites without any active involvement from attackers.</li><li aria-level="1"><b>Stage 5: Verify compromised credentials.</b> Bad actors verify brute forced credentials and gain unauthorized access to sites targeted in stage 1.</li></ul><p>So, how do attackers actually accomplish a distributed brute force attack from the browsers of completely innocent and unsuspecting website visitors? Let’s take a look at stage 4 in closer detail.</p><h4>Distributed brute force attack steps:</h4><ol><li aria-level="1">When a site visitor opens an infected web page, the user’s browser requests a task from the <b>hxxps://dynamic-linx[.]com/getTask.php</b> URL.</li><li aria-level="1">If the task exists, it parses the data and obtains the URL of the site to attack along with a valid username and a list of 100 passwords to try.</li><li aria-level="1">For every password in the list, the visitor’s browser sends the <b>wp.uploadFile</b> <a href="https://blog.sucuri.net/2023/05/what-is-xml-rpc-security-risks-how-to-disable.html">XML-RPC</a> API request to upload a file with encrypted credentials that were used to authenticate this specific request. That’s 100 API requests for each task! If authentication succeeds, a small text file with valid credentials is created in the WordPress uploads directory.</li><li aria-level="1">When all the passwords are checked, the script sends a notification to <b>hxxps://dynamic-linx[.]com/completeTask.php</b> that the task with a specific <b>taskId</b> (probably a unique site) and <b>checkId</b> (password batch) has been completed.</li><li aria-level="1">Finally, the script requests the next task and processes a new batch of passwords. And so on indefinitely while the infected page is open.</li></ol><p>This is how thousands of visitors across hundreds of infected websites unknowingly and simultaneously try to bruteforce thousands of other third-party WordPress sites. And since the requests come from the browsers of real visitors, you can imagine this is a challenge to filter and block such requests.</p><p>When the attack is over, the operators simply need to visit the target sites from their list (identified in stage 1) and try to download a specific file. If the file exists and they manage to download it, they’ll find the valid WordPress user credentials encoded within the file contents.</p><h3>Attack statistics</h3><p>In our telemetry, we see dozens of thousands of requests to thousands of unique domains checking for the file that this brute force attack tries to upload. In most cases, the response is <b>404</b> which means that the attack was not successful. However, in approximately <b>0.5%</b> of cases, we see the 200 response code which signifies the bad actors might have managed to guess the WordPress password.</p><p>Double checking the sites with the “200 OK” responses, we noticed that only one of them was actually compromised. The rest of the sites simply had non-standard configurations that made them return 200 even for “not found” pages. This further decreased the success rate of the brute force campaign below 0.02%.</p><p>In the past 4 days, we’ve recorded over 1200+ unique IPs that have tried to download the credentials file. Out of those IPs, the following <b>5</b> addresses accounted for over <b>85%</b> of all requests:</p><table><tbody><tr><td><b>IP</b></td><td><b>%</b></td><td><b>ASN</b></td></tr><tr><td>146.70.199.169</td><td>34.37%</td><td>M247, RO</td></tr><tr><td>138.199.60.23</td><td>28.13%</td><td>CDNEXT, GB</td></tr><tr><td>138.199.60.32</td><td>10.96%</td><td>CDNEXT, GB</td></tr><tr><td>138.199.60.19</td><td>6.54%</td><td>CDNEXT, GB</td></tr><tr><td>87.121.87.178</td><td>5.94%</td><td>SOUZA-AS, BR</td></tr></tbody></table><p>The last IP <b>87.121.87.178</b> is the same IP used for the <b>billlionair[.]app</b> domain where the <a href="https://blog.sucuri.net/2024/02/web3-crypto-malware-angel-drainer.html">Angel Drainer</a> was hosted.</p><p>In all cases, the attackers use these two User-Agent strings:</p><table><tbody><tr><td><b>User-Agent</b></td><td><b>%</b></td></tr><tr><td>Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/58.0.3029.110 Safari/537.3</td><td>95.8%</td></tr><tr><td>python-requests/2.25.1</td><td>4.2%</td></tr></tbody></table><p>This reveals that operators are most likely using python scripts to automate checking results of their brute force attack.</p><h3>Indicators of compromise, mitigating risk and final thoughts</h3><p>At this point, it’s not exactly clear why the attackers switched from Web 3 crypto drainers to a distributed WordPress brute force attack. Most likely, they realized that at their scale of infection (~1000 compromised sites) the crypto drainers are not very profitable yet. Moreover, they draw too much attention and their domains get blocked pretty quickly. So, it appears reasonable to switch the payload with something stealthier, that at the same time can help increase their portfolio of compromised sites for future waves of infections that they will be able to monetize in one way or another.</p><p>More than anything, this attack reminds us about the importance of using <a href="https://blog.sucuri.net/2024/01/how-to-make-strong-password.html">strong passwords</a>. With the level of technology available to bad actors now, it is pretty easy to try hundreds of thousands passwords on millions of sites within a reasonable timeframe.</p><p>In addition to secure passwords, you might want to consider restricting access to the WordPress admin interface and <a href="https://blog.sucuri.net/2023/05/what-is-xml-rpc-security-risks-how-to-disable.html">xmlrpc.php</a> file to trusted IPs only. This is quickly accomplished with the Sucuri <a href="https://sucuri.net/website-firewall/">web application firewall</a>, which makes it easy to restrict access to only certain IPs.</p><p>For DIY types who think they might be affected by this malware and want to hunt around for indicators of compromise, we recommend checking WordPress uploads directories (<b>wp-content/uploads/…</b>) for unknown files. This particular attack creates short <b>.txt</b> files that contain “<b>ActiveLamezh</b>”. You can also check directories like <b>wp-content/uploads/2024/02/</b> and <b>wp-content/uploads/2024/03</b>.</p><p>Even if you don’t find the uploaded .txt files, consider changing the passwords of all WordPress admin users to ensure that they are long, strong, and unique from other sets of credentials.</p><p>If you believe that your website may be infected with malware but aren’t sure how to tackle the issue, reach out and <a href="https://sucuri.net/live-chat/">chat with us</a>!
