<h2 id="overview">Overview</h2>
<ul>
<li><p>The pre-existing process creation audit event ID 4688 will now include audit information for command line processes.</p>
</li>
<li><p>It will also log SHA1/2 hash of the executable in the Applocker event log</p>
<ul>
<li>Application and Services Logs\Microsoft\Windows\AppLocker</li>
</ul>
</li>
<li><p>You enable via GPO, but it's disabled by default</p>
<ul>
<li>"Include command line in process creation events"</li>
</ul>
</li>
</ul>
<p><img src="media/command-line-process-auditing/gtr_adds_event4688.gif" alt="Screenshot that highlights the Process Command Line." data-linktype="relative-path"></p>
<p><strong>Figure  SEQ Figure \* ARABIC 16 Event 4688</strong></p>
<p>Review the updated event ID 4688 in  REF _Ref366427278 \h Figure 16.  Prior to this update none of the information for <strong>Process Command Line</strong> gets logged.  Because of this additional logging we can now see that not only was the wscript.exe process started, but that it was also used to execute a VB script.</p>
<h2 id="configuration">Configuration</h2>
<p>To see the effects of this update, you'll need to enable two policy settings.</p>
<h3 id="you-must-have-audit-process-creation-auditing-enabled-to-see-event-id-4688">You must have Audit Process Creation auditing enabled to see event ID 4688.</h3>
<p>To enable the Audit Process Creation policy, edit the following group policy:</p>
<p><strong>Policy location:</strong> Computer Configuration &gt; Policies &gt; Windows Settings &gt; Security Settings &gt; Advanced Audit Configuration &gt; Detailed Tracking</p>
<p><strong>Policy Name:</strong> Audit Process Creation</p>
<p><strong>Supported on:</strong> Windows 7 and above</p>
<p><strong>Description/Help:</strong></p>
<p>This security policy setting determines whether the operating system generates audit events when a process is created (starts) and the name of the program or user that created it.</p>
<p>These audit events can help you understand how a computer is being used and to track user activity.</p>
<p>Event volume: Low to medium, depending on system usage</p>
<p><strong>Default:</strong> Not configured</p>
<h3 id="in-order-to-see-the-additions-to-event-id-4688-you-must-enable-the-new-policy-setting-include-command-line-in-process-creation-events">In order to see the additions to event ID 4688, you must enable the new policy setting: Include command line in process creation events</h3>
<p><strong>Table  SEQ Table \* ARABIC 19 Command line process policy setting</strong></p>
<table>
<thead>
<tr>
<th>Policy Configuration</th>
<th>Details</th>
</tr>

<td><strong>Path</strong></td>
<td>Administrative Templates\System\Audit Process Creation</td>
</tr>
<tr>
<td><strong>Setting</strong></td>
<td><strong>Include command line in process creation events</strong></td>
</tr>
<tr>
<td><strong>Default setting</strong></td>
<td>Not Configured (not enabled)</td>

<tr>
<td><strong>Supported on:</strong></td>
<td>?</td>
</tr>
<tr>
<td><strong>Description</strong></td>
<td>This policy setting determines what information is logged in security audit events when a new process has been created.<p>This setting only applies when the Audit Process Creation policy is enabled. If you enable this policy setting the command line information for every process will be logged in plain text in the security event log as part of the Audit Process Creation event 4688, "a new process has been created," on the workstations and servers on which this policy setting is applied.</p><p>If you disable or don't configure this policy setting, the process's command line information won't be included in Audit Process Creation events.</p><p>Default: Not configured</p><p>Note: When this policy setting is enabled, any user with access to read the security events will be able to read the command line arguments for any successfully created process. Command line arguments can contain sensitive or private information such as passwords or user data.</p></td>

<p><img src="media/command-line-process-auditing/gtr_adds_includeclisetting.gif" alt="Screenshot that shows " data-linktype="relative-path"></p>
<p>When you use Advanced Audit Policy Configuration settings, you need to confirm that these settings aren't overwritten by basic audit policy settings.  Event 4719 is logged when the settings are overwritten.</p>
<p><img src="media/command-line-process-auditing/gtr_adds_event4719.gif" alt="Screenshot that shows the Include command line in process creation events dialog box." data-linktype="relative-path"></p>
<p>The following procedure shows how to prevent conflicts by blocking the application of any basic audit policy settings.</p>
<h3 id="to-ensure-that-advanced-audit-policy-configuration-settings-arent-overwritten">To ensure that Advanced Audit Policy Configuration settings aren't overwritten</h3>
<p><img src="media/command-line-process-auditing/gtr_adds_advauditpolicy.gif" alt="command-line auditing" data-linktype="relative-path"></p>
<ol>
<li><p>Open the Group Policy Management console</p>
</li>
<li><p>Right-click Default Domain Policy, and then select Edit.</p>
</li>
<li><p>Double-click Computer Configuration, double-click Policies, and then double-click Windows Settings.</p>
</li>
<li><p>Double-click Security Settings, double-click Local Policies, and then select Security Options.</p>
</li>
<li><p>Double-click Audit: Force audit policy subcategory settings (Windows Vista or later) to override audit policy category settings, and then select Define this policy setting.</p>
</li>
<li><p>Select Enabled, and then select OK.</p>
</li>
</ol>
