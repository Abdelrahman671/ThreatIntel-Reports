											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>Token protection (sometimes referred to as token binding in the industry) attempts to reduce attacks using token theft by ensuring a token is usable only from the intended device. When an attacker is able to steal a token, by hijacking or replay, they can impersonate their victim until the token expires or is revoked. Token theft is thought to be a relatively rare event, but the damage from it can be significant.</p>
<p>Token protection creates a cryptographically secure tie between the token and the device (client secret) it's issued to. Without the client secret, the bound token is useless. When a user registers a Windows 10 or newer device in Microsoft Entra ID, their primary identity is <a href="../devices/concept-primary-refresh-token#how-is-the-prt-protected" data-linktype="relative-path">bound to the device</a>. What this means: A policy can ensure that only bound sign-in session (or refresh) tokens, otherwise known as Primary Refresh Tokens (PRTs) are used by applications when requesting access to a resource.</p>
<div class="IMPORTANT">
<p>Important</p>
<p>Token protection is currently in public preview. For more information about previews, see <a href="https://www.microsoft.com/licensing/terms/product/ForOnlineServices/all" data-linktype="external">Universal License Terms For Online Services</a>.
With this preview, we're giving you the ability to create a Conditional Access policy to require token protection for sign-in tokens (refresh tokens) for specific services. We support token protection for sign-in tokens in Conditional Access for desktop applications accessing Exchange Online and SharePoint Online on Windows devices.</p>
</div>
<div class="IMPORTANT">
<p>Important</p>
<p>The following changes have been made to Token Protection since the initial public preview release:</p>
<ul>
<li><strong>Sign In logs output:</strong> The value of the string used in "enforcedSessionControls" and "sessionControlsNotSatisfied" changed from "Binding" to "SignInTokenProtection" in late June 2023. Queries on Sign In Log data should be updated to reflect this change.</li>
</ul>
</div>
<div class="NOTE">
<p>Note</p>
<p>We may interchange sign in tokens and refresh tokens in this content. If you want to provide protection for Microsoft 365 access tokens and application session cookies, consider trying <a href="/en-us/entra/global-secure-access/how-to-compliant-network" data-linktype="absolute-path">compliant network check enforcement</a>.</p>
</div>
<p><span class="mx-imgBorder">
<img src="media/concept-token-protection/complete-policy-components-session.png" alt="Screenshot showing a Conditional Access policy requiring token protection as the session control" data-linktype="relative-path">
</span>
</p>
<h2 id="requirements">Requirements</h2>
<p>This preview supports the following configurations for access to resources with Token Protection Conditional Access policies applied:</p>
<ul>
<li>Windows 10 or newer devices that are Microsoft Entra joined, Microsoft Entra hybrid joined, or Microsoft Entra registered.</li>
<li>OneDrive sync client version 22.217 or later</li>
<li>Teams native client version 1.6.00.1331 or later</li>
<li>Power BI desktop version 2.117.841.0 (May 2023) or later</li>
<li>Visual Studio 2022 or later when using the 'Windows authentication broker' Sign-in option</li>
<li>Office Perpetual clients aren't supported</li>
</ul>
<h3 id="known-limitations">Known limitations</h3>
<ul>
<li>External users (Microsoft Entra B2B) aren't supported and shouldn't be included in your Conditional Access policy.</li>
<li>The following applications don't support signing in using protected token flows and users are blocked when accessing Exchange and SharePoint:
<ul>
<li>PowerShell modules accessing Exchange, SharePoint, or Microsoft Graph scopes that are served by Exchange or SharePoint</li>
<li>PowerQuery extension for Excel</li>
<li>Extensions to Visual Studio Code which access Exchange or SharePoint</li>
<li>The new Teams 2.1 preview client gets blocked after sign out due to a bug. This bug should be fixed in a future service update.</li>
</ul>
</li>
<li>The following Windows client devices aren't supported:
<ul>
<li>Windows Server</li>
<li>Surface Hub</li>
<li>Windows-based Microsoft Teams Rooms (MTR) systems</li>
</ul>
</li>
</ul>
<h2 id="licensing-requirements">Licensing requirements</h2>
<p>Using this feature requires Microsoft Entra ID P2 licenses. To find the right license for your requirements, see <a href="https://www.microsoft.com/security/business/identity-access-management/azure-ad-pricing" data-linktype="external">Compare generally available features of Microsoft Entra ID</a>.</p>
<div class="NOTE">
<p>Note</p>
<p>Token Protection enforcement is part of Microsoft Entra ID Protection and will be part of the P2 license at general availability.</p>
</div>
<h2 id="deployment">Deployment</h2>
<p>For users, the deployment of a Conditional Access policy to enforce token protection should be invisible when using compatible client platforms on registered devices and compatible applications.</p>
<p>To minimize the likelihood of user disruption due to app or device incompatibility, we highly recommend:</p>
<ul>
<li>Start with a pilot group of users, and expand over time.</li>
<li>Create a Conditional Access policy in <a href="concept-conditional-access-report-only" data-linktype="relative-path">report-only mode</a> before moving to enforcement of token protection.</li>
<li>Capture both Interactive and Non-interactive sign in logs.</li>
<li>Analyze these logs for long enough to cover normal application use.</li>
<li>Add known good users to an enforcement policy.</li>
</ul>
<p>This process helps to assess your usersâ client and app compatibility for token protection enforcement.</p>
<h3 id="create-a-conditional-access-policy">Create a Conditional Access policy</h3>
<p>Users who perform specialized roles like those described in <a href="/en-us/security/privileged-access-workstations/privileged-access-security-levels#specialized" data-linktype="absolute-path">Privileged access security levels</a> are possible targets for this functionality. We recommend piloting with a small subset to begin.</p>
<p>The steps that follow help create a Conditional Access policy to require token protection for Exchange Online and SharePoint Online on Windows devices.</p>
<ol>
<li>Sign in to the <a href="https://entra.microsoft.com" data-linktype="external">Microsoft Entra admin center</a> as at least a <a href="../role-based-access-control/permissions-reference#conditional-access-administrator" data-linktype="relative-path">Conditional Access Administrator</a>.</li>
<li>Browse to <strong>Protection</strong> &gt; <strong>Conditional Access</strong> &gt; <strong>Policies</strong>.</li>
<li>Select <strong>New policy</strong>.</li>
<li>Give your policy a name. We recommend that organizations create a meaningful standard for the names of their policies.</li>
<li>Under <strong>Assignments</strong>, select <strong>Users or workload identities</strong>.
<ol>
<li>Under <strong>Include</strong>, select the users or groups who are testing this policy.</li>
<li>Under <strong>Exclude</strong>, select <strong>Users and groups</strong> and choose your organization's emergency access or break-glass accounts.</li>
</ol>
</li>
<li>Under <strong>Target resources</strong> &gt; <strong>Cloud apps</strong> &gt; <strong>Include</strong> &gt; <strong>Select apps</strong>
<ol>
<li><p>Under <strong>Select</strong>, select the following applications supported by the preview:</p>
<ol>
<li>Office 365 Exchange Online</li>
<li>Office 365 SharePoint Online</li>
</ol>
<div class="WARNING">
<p>Warning</p>
<p>Your Conditional Access policy should only be configured for these applications. Selecting the <strong>Office 365</strong> application group may result in unintended failures. This is an exception to the general rule that the <strong>Office 365</strong> application group should be selected in a Conditional Access policy.</p>
</div>
</li>
<li><p>Choose <strong>Select</strong>.</p>
</li>
</ol>
</li>
<li>Under <strong>Conditions</strong>:
<ol>
<li>Under <strong>Device platforms</strong>:
<ol>
<li>Set <strong>Configure</strong> to <strong>Yes</strong>.</li>
<li><strong>Include</strong> &gt; <strong>Select device platforms</strong> &gt; <strong>Windows</strong>.</li>
<li>Select <strong>Done</strong>.</li>
</ol>
</li>
<li>Under <strong>Client apps</strong>:
<ol>
<li>Set <strong>Configure</strong> to <strong>Yes</strong>.
<div class="WARNING">
<p>Warning</p>
<p>Not configuring the Client Apps condition, or leaving <strong>Browser</strong> selected may cause applications that use MSAL.js, such as Teams Web to be blocked.</p>
</div>
</li>
<li>Under Modern authentication clients, only select <strong>Mobile apps and desktop clients</strong>. Leave other items unchecked.</li>
<li>Select <strong>Done</strong>.</li>
</ol>
</li>
</ol>
</li>
<li>Under <strong>Access controls</strong> &gt; <strong>Session</strong>, select <strong>Require token protection for sign-in sessions</strong> and select <strong>Select</strong>.</li>
<li>Confirm your settings and set <strong>Enable policy</strong> to <strong>Report-only</strong>.</li>
<li>Select <strong>Create</strong> to create to enable your policy.</li>
</ol>
<p>After administrators confirm the settings using <a href="howto-conditional-access-insights-reporting" data-linktype="relative-path">report-only mode</a>, they can move the <strong>Enable policy</strong> toggle from <strong>Report-only</strong> to <strong>On</strong>.</p>
<h3 id="capture-logs-and-analyze">Capture logs and analyze</h3>
<p>Monitoring Conditional Access enforcement of token protection before and after enforcement.</p>
<h4 id="sign-in-logs">Sign-in logs</h4>
<p>Use Microsoft Entra sign-in log to verify the outcome of a token protection enforcement policy in report only mode or in enabled mode.</p>
<ol>
<li>Sign in to the <a href="https://entra.microsoft.com" data-linktype="external">Microsoft Entra admin center</a> as at least a <a href="../role-based-access-control/permissions-reference#conditional-access-administrator" data-linktype="relative-path">Conditional Access Administrator</a>.</li>
<li>Browse to <strong>Identity</strong> &gt; <strong>Monitoring &amp; health</strong> &gt; <strong>Sign-in logs</strong>.</li>
<li>Select a specific request to determine if the policy is applied or not.</li>
<li>Go to the <strong>Conditional Access</strong> or <strong>Report-Only</strong> pane depending on its state and select the name of your policy requiring token protection.</li>
<li>Under <strong>Session Controls</strong> check to see if the policy requirements were satisfied or not.</li>
</ol>

<h4 id="log-analytics">Log Analytics</h4>
<p>You can also use <a href="../monitoring-health/tutorial-configure-log-analytics-workspace" data-linktype="relative-path">Log Analytics</a> to query the sign-in logs (interactive and non-interactive) for blocked requests due to token protection enforcement failure.</p>
<p>Here's a sample Log Analytics query searching the non-interactive sign-in logs for the last seven days, highlighting <strong>Blocked</strong> versus <strong>Allowed</strong> requests by <strong>Application</strong>. These queries are only samples and are subject to change.</p>
<div class="NOTE">
<p>Note</p>
<p><strong>Sign In logs output:</strong> The value of the string used in "enforcedSessionControls" and "sessionControlsNotSatisfied" changed from "Binding" to "SignInTokenProtection" in late June 2023. Queries on Sign In Log data should be updated to reflect this change. The examples cover both values to include historical data.</p>
</div>
<pre><code class="lang-kusto">//Per Apps query 
// Select the log you want to query (SigninLogs or AADNonInteractiveUserSignInLogs ) 
//SigninLogs 
AADNonInteractiveUserSignInLogs 
// Adjust the time range below 
| where TimeGenerated &gt; ago(7d) 
| project Id,ConditionalAccessPolicies, Status,UserPrincipalName, AppDisplayName, ResourceDisplayName 
| where ConditionalAccessPolicies != "[]" 
| where ResourceDisplayName == "Office 365 Exchange Online" or ResourceDisplayName =="Office 365 SharePoint Online" 
//Add userPrinicpalName if you want to filter  
// | where UserPrincipalName =="&lt;user_principal_Name&gt;" 
| mv-expand todynamic(ConditionalAccessPolicies) 
| where ConditionalAccessPolicies ["enforcedSessionControls"] contains '["Binding"]' or ConditionalAccessPolicies ["enforcedSessionControls"] contains '["SignInTokenProtection"]' 
| where ConditionalAccessPolicies.result !="reportOnlyNotApplied" and ConditionalAccessPolicies.result !="notApplied" 
| extend SessionNotSatisfyResult = ConditionalAccessPolicies["sessionControlsNotSatisfied"] 
| extend Result = case (SessionNotSatisfyResult contains 'SignInTokenProtection' or SessionNotSatisfyResult contains 'SignInTokenProtection', 'Block','Allow')
| summarize by Id,UserPrincipalName, AppDisplayName, Result 
| summarize Requests = count(), Users = dcount(UserPrincipalName), Block = countif(Result == "Block"), Allow = countif(Result == "Allow"), BlockedUsers = dcountif(UserPrincipalName, Result == "Block") by AppDisplayName 
| extend PctAllowed = round(100.0 * Allow/(Allow+Block), 2) 
| sort by Requests desc 
</code></pre>
<p>The result of the previous query should be similar to the following screenshot:</p>
<p><span class="mx-imgBorder">
<a href="media/concept-token-protection/log-analytics-results.png#lightbox" data-linktype="relative-path">
<img src="media/concept-token-protection/log-analytics-results.png" alt="Screenshot showing example results of a Log Analytics query looking for token protection policies" data-linktype="relative-path">
</a>
</span>
</p>
<p>The following query example looks at the non-interactive sign-in log for the last seven days, highlighting <strong>Blocked</strong> versus <strong>Allowed</strong> requests by <strong>User</strong>.</p>
<pre><code class="lang-kusto">//Per users query 
// Select the log you want to query (SigninLogs or AADNonInteractiveUserSignInLogs ) 
//SigninLogs 
AADNonInteractiveUserSignInLogs 
// Adjust the time range below 
| where TimeGenerated &gt; ago(7d) 
| project Id,ConditionalAccessPolicies, UserPrincipalName, AppDisplayName, ResourceDisplayName 
| where ConditionalAccessPolicies != "[]" 
| where ResourceDisplayName == "Office 365 Exchange Online" or ResourceDisplayName =="Office 365 SharePoint Online" 
//Add userPrincipalName if you want to filter  
// | where UserPrincipalName =="&lt;user_principal_Name&gt;" 
| mv-expand todynamic(ConditionalAccessPolicies) 
| where ConditionalAccessPolicies ["enforcedSessionControls"] contains '["Binding"]' or ConditionalAccessPolicies ["enforcedSessionControls"] contains '["SignInTokenProtection"]'
| where ConditionalAccessPolicies.result !="reportOnlyNotApplied" and ConditionalAccessPolicies.result !="notApplied" 
| extend SessionNotSatisfyResult = ConditionalAccessPolicies.sessionControlsNotSatisfied 
| extend Result = case (SessionNotSatisfyResult contains 'SignInTokenProtection' or SessionNotSatisfyResult contains 'SignInTokenProtection', 'Block','Allow')
| summarize by Id, UserPrincipalName, AppDisplayName, ResourceDisplayName,Result  
| summarize Requests = count(),Block = countif(Result == "Block"), Allow = countif(Result == "Allow") by UserPrincipalName, AppDisplayName,ResourceDisplayName 
| extend PctAllowed = round(100.0 * Allow/(Allow+Block), 2) 
| sort by UserPrincipalName asc   
</code></pre>
