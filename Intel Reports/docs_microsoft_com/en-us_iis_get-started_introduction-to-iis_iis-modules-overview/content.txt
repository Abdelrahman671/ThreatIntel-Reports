<h2 id="introduction">Introduction</h2>
<p>The IIS 7 and above Web server feature set is componentized into more than thirty independent modules.</p>
<p>A module is either a Win32 DLL (native module) or a .NET 2.0 type contained within an assembly (managed module). Similar to a set of building blocks, modules are added to the server in order to provide the desired functionality for your applications. Likewise, all IIS modules can be removed, or replaced with custom modules developed using the IIS C++ APIs, or the familiar ASP.NET 2.0 APIs.</p>
<p>This article describes common IIS module management tasks, and details each module including its configuration settings and the potential effect a module removal has on the Web server. The management examples are given using both the graphical IIS Manager and the <strong>AppCmd</strong> command line tool.</p>
<p><a id="Prerequisites"></a></p>
<h2 id="prerequisites">Prerequisites</h2>
<p>To follow the steps in this document it is best to do a full install of IIS.</p>
<div class="NOTE">
<p>Note</p>
<p>Not all editions of Windows support all available IIS features. To see which features of IIS are supported on your operating system, you will need to consult the documentation which shipped with your version of Windows.</p>
</div>
<h3 id="to-perform-a-full-installation-of-iis-on-windows-vista-windows-7-windows-8-windows-10-or-windows-11">To Perform a Full Installation of IIS on Windows Vista, Windows 7, Windows 8, Windows 10 or Windows 11</h3>
<ol>
<li>Click <strong>Start</strong> and then click <strong>Control Panel</strong>.</li>
<li>In the <strong>Control Panel</strong>, click <strong>Programs</strong>.</li>

<p><a id="Getting"></a></p>
<h2 id="getting-started-with-modules">Getting Started with Modules</h2>
<p>In order to add a module to the server, you must perform two steps:</p>
<ol>
<li>Install a module on the server (native modules only).</li>
<li>Enable the module in an application.</li>
</ol>
<p>The first step registers the module globally with the server, making it available within each server worker process. It is necessary only for native modules due to the trusted nature of native code, and is available only to administrators.</p>
<div class="NOTE">
<p>Note</p>
<p>A native module has unrestricted access to any resource available to the server worker process, just like an ISAPI filter or extension in previous versions. Because of this unrestricted access, you should install only native modules that come from a trusted source.</p>
</div>
<p>The second step enables the module to execute within a particular application and effectively allows the application administrator to control the server features enabled for the application. This step allows both installed native modules and managed modules to be enabled for each application.</p>
<p><a id="install_native"></a></p>
<h3 id="to-install-a-native-module">To Install a Native Module</h3>
<p>In order to install a native module, it must be registered with the server using one of the options below:</p>
<ul>
<li>Manually editing the IIS configuration store. In IIS 7.5 and later you can use the Configuration Editor in the IIS Manager.</li>
<li>Using the IIS Manager</li>
<li>Using the <strong>AppCmd.exe</strong> command line tool</li>
</ul>
<p>All three of these options result in the module entry being added to the &lt;globalModules&gt; IIS configuration section, which can be set only at the server level. To examine the contents of this section, open the root configuration file located in <code>%windir%\system32\inetsrv\config\applicationhost.config</code>, and search for the string "&lt;globalModules&gt;".</p>
<p>After a full IIS installation, this section contains an entry for each of the native modules shipped with IIS, specifying a name and the path to the module DLL:</p>
<pre><code class="lang-xml" name="Main">&lt;globalModules&gt;
    &lt;add name="DefaultDocumentModule" image="%windir%\system32\inetsrv\defdoc.dll" /&gt;
    &lt;add name="DirectoryListingModule" image="%windir%\system32\inetsrv\dirlist.dll" /&gt;
    &lt;add name="StaticFileModule" image="%windir%\system32\inetsrv\static.dll" /&gt;
    ...

&lt;/globalModules&gt;
</code></pre>
<p>All of these modules are described in detail later in this document.</p>
<h3 id="to-uninstall-a-native-module">To Uninstall a Native Module</h3>
<p>You can uninstall a native module if that module is no longer in use on the server, or if you would like to replace it with another module. Remove the corresponding module entry from the &lt;globalModules&gt; configuration list, and the associated entry in the <code>&lt;modules&gt;</code> configuration list using one of the following options:</p>
<ul>
<li>Manually editing the IIS configuration store. In IIS 7.5 and above you can use the Configuration Editor.</li>
<li>Using the IIS Manager</li>
<li>Using the <strong>AppCmd.exe</strong> command line tool</li>
</ul>
<div class="NOTE">
<p>Note</p>
<p>Because the &lt;globalModules&gt; configuration section can be set only at the server level, you must be an administrator to uninstall a module.</p>
</div>
<p><a id="Enable_mod"></a></p>
<h3 id="enabling-a-module-for-an-application">Enabling a Module for an Application</h3>
<p>A module must be enabled before it can provide service for a particular application. In order to enable a native module, it must first be installed on the server (see the previous section, <a href="#install_native" data-linktype="self-bookmark">To Install a Native Module</a>).</p>
<p>A managed module does not require installation, and can be enabled directly for each application. This allows applications to include their managed modules directly within the application by registering them in the application's web.config file, and providing the implementation in /BIN or /App_Code directories.</p>
<p>To enable a module, do one of the following:</p>
<ul>
<li>Manually edit the IIS configuration store, either globally to enable the module for all applications on the server, or in a particular web.config file located within each application for which you would like to enable this module. In IIS 7.5 or above you can use the Configuration Editor.</li>
<li>Use the IIS Manager</li>
<li>Use the <strong>AppCmd.exe</strong> command line tool</li>
</ul>
<p>All three of these options add the module entry to the <code>&lt;modules&gt;</code> IIS configuration section, which is can be set at both the server level and application level. Examine the contents of this section by opening the root configuration file located in <code>%windir%\system32\inetsrv\config\applicationhost.config</code>, and searching for the string "&lt;modules&gt;".</p>
<p>Unlike native modules, a managed module does not require adding an entry to the &lt;globalModules&gt; configuration section.</p>
<p>After a full IIS installation, the configuration section contains an entry for each of the modules (both managed and native) that shipped with IIS. The entry indicates that all these modules are enabled by default for all applications on the server. Each entry in this section specifies either the name of a native module that was previously installed on the server, or the name and .NET type of a managed module:</p>
<pre><code class="lang-xml" name="Main">&lt;modules&gt;
&lt;add name="DefaultDocumentModule" /&gt;
&lt;add name="DirectoryListingModule" /&gt;
&lt;add name="StaticFileModule"/&gt;
 ...

&lt;/modules&gt;
</code></pre>
<p><a id="Disabling"></a></p>
<h3 id="disabling-a-module-in-an-application">Disabling a Module in an Application</h3>
<p>Disable a module if that module is no longer in use in the application, or if you want to replace it with another module. To disable a module, remove the corresponding module entry from the <code>&lt;modules&gt;</code> configuration collection for a particular application where you do not want this module to execute. If the module is enabled at the server level, remove it there to disable it in all applications on the server by default. Use one of the following options to remove the module at the server level:</p>
<ul>
<li>Manually edit the &lt;system.webServer&gt;/&lt;modules&gt; configuration section in your application.</li>
<li>Use the IIS Manager.</li>
<li>Use the <strong>AppCmd.exe</strong> command line tool.</li>
</ul>
<p>After the module is removed from the application, it will not be active in that application. However, if the module is native, it will still be loaded in the server worker process, and may be used by other applications that did not remove it.</p>
<div class="NOTE">
<p>Note</p>
<p>The configuration section is unlocked by default if ASP.NET is installed. This default unlocking allows applications to disable both native and managed modules enabled globally, and to add new managed modules. This configuration section can be locked at the server level to prevent modification entirely, or prevent removal of specific module entries if desired by the Administrator.</p>
</div>
<p><a id="Precondition"></a></p>
<h2 id="preconditions">Preconditions</h2>
<p>There is another attribute on a module entry called precondition. The IIS core engine uses preconditions to determine when to enable a particular module. Performance reasons, for example, might determine that you only want to execute managed modules for requests that also go to a managed handler. The precondition in the following example (precondition="managedHandler") only enables the forms authentication module for requests that are also handled by a managed handler, such as requests to .aspx or .asmx files:</p>
<pre><code class="lang-xml" name="Main">&lt;add name="FormsAuthentication"type="System.Web.Security.FormsAuthenticationModule" preCondition="managedHandler" /&gt;
</code></pre>
<p>If you remove the attribute precondition="managedHandler", Forms Authentication also applies to content that is not served by managed handlers, such as .html, .jpg, .doc, but also for classic ASP (.asp) or PHP (.php) extensions. See "<a href="../../application-frameworks/building-and-running-aspnet-applications/how-to-take-advantage-of-the-iis-integrated-pipeline" data-linktype="relative-path">How to Take Advantage of IIS Integrated Pipeline</a>" for an example of enabling ASP.NET modules to run for all content.</p>
<p>You can also use a shortcut to enable all managed (ASP.NET) modules to run for all requests in your application, regardless of the "managedHandler" precondition. To enable all managed modules to run for all requests without configuring each module entry to remove the "managedHandler" precondition, use the <strong>runAllManagedModulesForAllRequests</strong> property in the <code>&lt;modules&gt;</code> section:</p>
<pre><code class="lang-xml" name="Main">&lt;modules runAllManagedModulesForAllRequests="true"/&gt;
</code></pre>
<p>When you use this property, the "managedHandler" precondition has no effect and all managed modules run for all requests.</p>

<h2 id="managing-modules-from-the-command-line">Managing Modules from the Command Line</h2>
<p>To quickly manage modules from the command line, or script module management tasks, use the <strong>AppCmd.exe</strong> command line tool.</p>
<h3 id="to-install-a-module-using-appcmdexe">To install a module using AppCmd.exe</h3>
<p>At the Command Prompt, type the following command, replacing values in italics with the values for the module on your computer.</p>
<pre><code class="lang-console" name="Main">Appcmd.exe install module /name:MODULE_NAME /image:PATH_TO_DLL
</code></pre>
<div class="NOTE">
<p>Note</p>
<p>Installing a native module automatically enables it at the server level.</p>
</div>
<p>For example, the following command installs the Default Document module.</p>
<pre><code class="lang-console" name="Main">%windir%\system32\inetsrv\appcmd.exe install module /name:DefaultDocumentModule /image:%windir%\system32\inetsrv\defdoc.dll
</code></pre>
<p>If you run the command on a computer where the module is already loaded, you wil get an error message similar to the following:</p>
<p>ERROR (message:Failed to add duplicate collection element "DefaultDocumentModule".)</p>
<h3 id="to-uninstall-a-module-using-appcmdexe">To uninstall a module using AppCmd.exe</h3>
<p>At the Command Prompt, type the following command, replacing values in italics with the values for the module on your computer.</p>
<pre><code class="lang-console" name="Main">Appcmd.exe uninstall module MODULE_NAME
</code></pre>
<div class="NOTE">
<p>Note</p>
<p>Uninstalling a module automatically disables it at the server level.</p>
</div>
<p>For example, the following command uninstalls the Default Document module. This command is provided only as example. You should only run it on a test server.</p>
<pre><code class="lang-console" name="Main">%windir%\system32\inetsrv\appcmd.exe uninstall module DefaultDocumentModule
</code></pre><h3 id="to-list-the-modules-enabled-either-for-an-application-or-globally">To list the modules enabled either for an application or globally</h3>
<p>At the Command Prompt, type the following command, replacing values in italic text with the values for the module on your computer.</p>
<pre><code class="lang-console" name="Main">Appcmd.exe list modules [/app.name:APPLICATION_NAME]
</code></pre>
<p>For example, the following command lists the modules enabled for the Default Web Site.</p>
<pre><code class="lang-console" name="Main">%windir%\system32\inetsrv\appcmd.exe list modules /app.name:"Default Web Site"
</code></pre>
<p>And this command lists the modules installed globally.</p>
<pre><code class="lang-console" name="Main">%windir%\system32\inetsrv\appcmd.exe list modules
</code></pre><h3 id="to-enable-a-module-either-for-a-particular-application-or-globally">To enable a module either for a particular application or globally</h3>
<p>At the Command Prompt, type the following command, replacing values in italic text with the values for the module on your computer.</p>
<pre><code class="lang-console" name="Main">Appcmd.exe add module /name:MODULE_NAME /type:MGD_TYPE
</code></pre>
<p>For example, the following command adds the Forms Authentication module to the Default Web Site.</p>
<pre><code class="lang-console" name="Main">%windir%\system32\inetsrv\appcmd.exe add module /name:FormsAuthentication /type:System.Web.Security.FormsAuthenticationModule /app.name:"Default Web Site"
</code></pre>
<p>And this command adds the Forms Authentication Module globally.</p>
<pre><code class="lang-console" name="Main">%windir%\system32\inetsrv\appcmd.exe add module /name:FormsAuthentication /type:System.Web.Security.FormsAuthenticationModule
</code></pre><h3 id="to-disable-a-module-either-for-a-particular-application-or-globally">To disable a module either for a particular application or globally</h3>
<p>At the Command Prompt, type the following command, replacing values in italic text with the values for the module on your computer.</p>
<pre><code class="lang-console" name="Main">Appcmd.exe delete module MODULE_NAME [/app.name:APPLICATION_NAME]
</code></pre>
<p>For example, the following command removes the Forms Authentication module from the Default Web Site.</p>
<pre><code class="lang-console" name="Main">%windir%\system32\inetsrv\appcmd.exe delete module FormsAuthentication /app.name:"Default Web Site"
</code></pre>
<p>And this command removes the Forms Authentication Module globally.</p>
<pre><code class="lang-console" name="Main">%windir%\system32\inetsrv\appcmd.exe delete module FormsAuthentication
</code></pre><h3 id="to-get-more-help-on-the-syntax-for-each-of-the-app-cmdexe-commands">To get more help on the syntax for each of the App Cmd.exe commands</h3>
<p>To display the commands supported on the Module object type the following command:</p>
<pre><code class="lang-console" name="Main">Appcmd.exe module /?
