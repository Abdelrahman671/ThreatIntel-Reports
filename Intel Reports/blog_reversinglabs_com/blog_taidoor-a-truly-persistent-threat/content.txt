<div class="blog__post-content html-content">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><h2 style="text-align: center;">When malware lasts longer than your washing machine</h2>
<p><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Cover-Taidoor.png?width=700&amp;name=Blog-Cover-Taidoor.png" alt="Taidoor - a truly persistent threat" style="width: 700px;" width="700" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Cover-Taidoor.png?width=350&amp;name=Blog-Cover-Taidoor.png 350w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Cover-Taidoor.png?width=700&amp;name=Blog-Cover-Taidoor.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Cover-Taidoor.png?width=1050&amp;name=Blog-Cover-Taidoor.png 1050w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Cover-Taidoor.png?width=1400&amp;name=Blog-Cover-Taidoor.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Cover-Taidoor.png?width=1750&amp;name=Blog-Cover-Taidoor.png 1750w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Cover-Taidoor.png?width=2100&amp;name=Blog-Cover-Taidoor.png 2100w" sizes="(max-width: 700px) 100vw, 700px"></p>
<h2><strong>Introduction</strong></h2>
<!--more-->
<p>Government-supported actors usually conduct long-lasting activities in cyberspace, and to simplify such continuous processes, they often develop malicious tools with the intention of using them for a long period of time. Like any other malicious tool, they need to be stealthy, and when they get detected, some modifications are necessary to become undetectable again. Sometimes it can be as simple as changing the compromised infrastructure, while at other times creating a new version of the tool is required. When organizations put a lot of effort into creating a tool like that, they probably don't plan to use it in massive campaigns. It is expected to be used in smaller, targeted attacks; therefore, researchers won’t have too many samples for analysis at their disposal. <br><br>An example of such a tool is Taidoor RAT (remote access trojan), dating all the way back to 2008, whose new version was recently discovered and presented in a <a href="https://us-cert.cisa.gov/ncas/analysis-reports/ar20-216a" rel="noopener" target="_blank">technical report</a> released by the US government institutions. Taidoor is described as a Remote Access Trojan developed and used by cyber actors supported by the Chinese government. The new version of the RAT consists of two parts - a loader in a DLL form, and a main RAT module that comes as RC4-encrypted binary data. The loader first decrypts the encrypted main RAT module, and then executes its exported Start function. The report provides two samples for both the loader and the main encrypted RAT module. These samples come with only two C2 domains and one C2 IP.</p>
<p>In this blog, you will learn how our Titanium Platform can help you find more malware samples to extend the IOC lists and power up your defenses.</p>
<h2><strong>Taidoor</strong></h2>
<p>As already mentioned, the Taidoor RAT consists of two parts, a loader and the main RAT module. A few pivoting attempts on the loader samples didn’t produce any results, so we focused on the samples of the main RAT module. This makes sense, as they contain the malware configuration including C2 domains and IPs.<br><br>Two samples of the main RAT available from the threat report are encrypted with the RC4 algorithm, and aren’t suitable for pivoting in that form. The first step is to decrypt them and get them to their normal DLL format suitable for metadata extraction. Publicly available tools like <a href="https://gchq.github.io/CyberChef" rel="noopener" target="_blank">CyberChef</a> can be used to decrypt various encryption algorithms, including RC4. Once decrypted, the DLL is processed with Titanium Platform, and its metadata is extracted. The first thing to do is look at the files grouped into the same buckets based on the RHA1, our functional file similarity algorithm, for each of the samples.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/RHA_dll.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/RHA_dll.jpg" alt="Similar files grouped by RHA1 algorithm" style="width: 1400px;"></a> <a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/RHA_dll_plus.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/RHA_dll_plus.jpg" alt="Similar files grouped by RHA1 algorithm" style="width: 1400px;"></a>
<p style="text-align: center;"><br><strong><span style="font-size: 16px;">Similar files grouped by RHA1 algorithm</span></strong></p>
<p>The RHA1 algorithm reveals ten more samples dating back to 2018 and 2019. However, there are additional options for pivoting. Looking at the samples' exports shows a specific combination of functions and original file name.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/samples-exports.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/samples-exports.jpg" alt="Samples exports" style="width: 560px; display: block; margin: 0px auto;"></a>
<p style="font-size: 16px; text-align: center;"><strong>Samples exports</strong></p>
<p>Running a Titanium Platform cloud search query with this specific combination finds the samples already discovered using the RHA1 file similarity algorithm, and also 3 additional samples.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/payload_exports_mark.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/payload_exports_mark.jpg" alt="Pivoting on exports and original name" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-size: 16px;"><strong>Pivoting on exports and original name</strong></span></p>
<p>Analyzing these three samples shows that they are using the same AES key as described in the threat report. The significant difference is that they have another layer of encryption used to hide the part of the code responsible for configuration decryption, including the AES key and S-Box initialization.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/sample859e_decrypted_layer_sample.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/sample859e_decrypted_layer_sample.jpg" alt="Loading of AES decryption key in previously decrypted layer of code" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-size: 16px;"><strong>Loading of AES decryption key in previously decrypted layer of code</strong></span></p>
<p>Pivoting on these three samples didn’t return any new results. However, there is something uncommon in this exported file name - “mm.dll”. It is probably a shorthand for something.</p>
<p>What if there were some variations in the functionality of these samples? There is a chance that this <em>mm</em> prefix could have something appended to it. To check this assumption, a search query is constructed using the wildcard character to match samples that start with the <em>mm</em> prefix and at the same time contain the aforementioned exported functions <em>Start</em> and <em>Service</em>.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/payload_exports_mm_expand.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/payload_exports_mm_expand.jpg" alt="Pivoting on exports and original name using wildcard character" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-size: 16px;"><strong>Pivoting on exports and original name using wildcard character</strong></span></p>
<p>Ordering these search results by their size - and ignoring clean software and previously discovered samples - leaves just a few hashes we need to look at. These hashes are split in two groups based on their size. Looking at their exports reveals two new original file names - <strong>mm_tcp_dll.dll</strong> and <strong>mm_tcp_svchost.dll</strong>.<br><br></p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/samples_exports_1.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/samples_exports_1.jpg" alt="Samples exports" style="width: 1400px;"></a> <a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/samples_exports_2.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/samples_exports_2.jpg" alt="Samples exports" style="width: 1400px;"></a>
<p style="font-size: 16px; text-align: center;"><strong>Samples exports</strong></p>
<p>Analyzing these samples in disassembler confirms these are indeed Taidoor samples, since they perform the same AES decryption and use the same AES key. The 466 KB samples also have a layer of encryption used to hide the part of the code responsible for configuration decryption, as already described in the previous case.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/AES_decryption_IDA.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/AES_decryption_IDA.jpg" alt="Loading of AES decryption key in 134KB samples" style="width: 360px; display: block; margin: 0px auto;"></a>
<p style="text-align: center;"><br><span style="font-size: 16px;"><strong>Loading of AES decryption key in 134KB samples</strong></span></p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/sample859e_decrypted_layer_small.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/sample859e_decrypted_layer_small.jpg" alt="Samples exports" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-size: 16px;"><strong>Loading of AES decryption key in previously decrypted layer of code in 466KB samples</strong></span></p>
<p>They also include the same strings present in the Taidoor samples mentioned in the referenced threat report.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/strings.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/strings.jpg" alt="Some of the strings found in Taidoor samples" style="width: 560px; display: block; margin: 0px auto;"></a>
<p style="text-align: center;"><br><span style="font-size: 16px;"><strong>Some of the strings found in Taidoor samples</strong></span></p>
<p>One interesting thing that catches the eye are the pdb paths. Samples of the 134 KB size contain the pdb path <strong>“C:\Users\john\Desktop\KD17.6_20170628\Release\mm_tcp_svchost.pdb”</strong> and have the compilation timestamp set to 2017-07-04T09:20:21. Samples of the 466 KB size have their compilation timestamp set to 2016-04-06T08:44:22 and don’t include any pdb paths, but looking at the contained strings reveals one very similar to the mentioned pdb path - <strong>“c:\users\develop\desktop\kd15.1_aes_20160321\mm_tcp_dll\svchost.cpp”</strong>. We can assume that in the “KD<strong>VV.v_YYYYMMDD</strong>” part of the string, the <strong>VV.v</strong> represents the malware version, and that <strong>YYYYMMDD</strong> represents the creation date. Unfortunately, pivoting variations on these pdb paths didn’t result in any other samples besides the ones previously found.</p>
<p>There are still two discovered original file names left to pivot on. Pivoting on <em>mm_tcp_svchost.dll</em> doesn’t result in anything new. On the other hand, pivoting on <em>mm_tcp_dll.dll</em> reveals around 70 more samples. Looking at them in detail indicates they are older versions of Taidoor dating all the way back to 2011. These samples have configuration encrypted with the RC4 algorithm, and not with AES like the version from the <a href="https://us-cert.cisa.gov/ncas/analysis-reports/ar20-216a" rel="noopener" target="_blank">referenced</a> threat report. We won't describe these samples here because they are a bit outdated and are more similar to the older versions of Taidoor already covered with technical reports from <a href="https://www.fireeye.com/blog/threat-research/2013/09/evasive-tactics-taidoor-3.html" rel="noopener" target="_blank">FireEye</a> and <a href="https://www.trendmicro.de/cloud-content/us/pdfs/security-intelligence/white-papers/wp_the_taidoor_campaign.pdf" rel="noopener" target="_blank">TrendMicro</a>. <br><br>But if there is a sample named <em>mm_tcp_dll.dll</em>, is it possible that there is a sample using the http protocol for communication? We can guess that it would be named <em>mm_http_dll.dll</em> if it follows the naming convention. A search query indeed finds 17 samples with this name, and analyzing them in disassembler shows they also have the configuration encrypted with the same “<strong>0xA1 0xA2</strong>” RC4 key. But as illustrated in the following image, these samples are also 8 years old, and are not in the focus of interest of this blog post.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/exports_mm_http.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/exports_mm_http.jpg" alt="Pivoting on original name" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-size: 16px;"><strong>Pivoting on original name</strong></span></p>
<p>Another very interesting sample exists, originally named <em>mm_udt_dll.dll</em>, suggesting usage of the UDT protocol for the C2 communication. Since it is also quite old, it won’t be examined in detail, but it uses the same RC4 key for configuration encryption, and it is likely that the only difference is the network protocol used for the communication.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/taidoor/exports_mm_udt.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/taidoor/exports_mm_udt.jpg" alt="Pivoting on original name" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-size: 16px;"><strong>Pivoting on original name</strong></span></p>
<p>All collected samples have their configuration placed in the <em>.data</em> section, and use the same AES key “<strong>2B 7E 15 16 28 AE D2 A6 AB F7 15 88 09 CF 4F 3C</strong>” IV: “<strong>00</strong>”. Configurations were extracted from all the samples and subsequently decrypted with the provided key. The last thing to do was to collect decrypted C2 domains and IPs, and create an IOC list provided at the end of this blog.</p>

<p>&nbsp;</p>
<p>Two oldest groups of samples have an extra layer of encryption used for hiding the part of the code responsible for AES decryption. At some point in 2017, the malware developers decided to simplify the code and move that extra protection layer, probably to another PE artifact (the loader component). The fact that the same AES key has been used for more than 4 years is a lucky coincidence which simplifies the IOC extraction for researchers and makes correlating samples easier. Older versions of Taidoor mentioned in this blog use a different encryption algorithm, but have a highly similar set of exported functions, and also very similar high-level program logic.</p>
<h2><strong>Conclusion</strong></h2>
<p>Like any other software product, malware families require a lot of maintenance and improvement to achieve long-term operability. Even though such continuous upgrading helps malware avoid detection mechanisms, it also results in related malware versions. These related versions must have some similarities, and this is an opportunity for security researchers to establish correlations between various malware campaigns and quickly find more threat samples. <br><br>As part of this research, and based on the analysis provided in the referenced threat report, 23 related samples were found and 40 new C2 IPs and domains extracted from their configurations.<br><br>Titanium Platform helps researchers get a more detailed insight into malware samples. Various search queries can be constructed from the extracted metadata, and can be used to find related threats. With Titanium Platform you can also establish time relationships between the samples, and together with the differences in malware functionality, this can enable you a better understanding of malware development history.</p>
