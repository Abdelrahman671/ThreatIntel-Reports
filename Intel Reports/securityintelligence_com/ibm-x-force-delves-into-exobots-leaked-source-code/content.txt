When they do, they give rise to variants and variations of the same malware, lowering the bar for more criminals to enter the mobile malware scene and try their hand at mobile banking fraud.
&lt;h2&gt;Delving Into ExoBot&#039;s Inner Workings&lt;/h2&gt;
The following sections describe technical details about ExoBot as analyzed by X-Force mobile threat researcher Shahar Tavor.
&lt;h3 class=&quot;&quot;&gt;Architecture Basics&lt;/h3&gt;
Having analyzed ExoBot&#039;s leaked source code, X-Force researchers found the malware to possess some notable features, such as dynamic modules loaded at runtime and some anti-emulation tricks to evade popular sandboxes.

The image below illustrates the malware&#039;s architecture. The overall routine occurs as follows:
&lt;ol&gt;
 	&lt;li&gt;An Android mobile device user unknowingly downloads a loader app via SMS spam or directly from the official store.&lt;/li&gt;
 	&lt;li&gt;The loader app calls back to the attacker&#039;s command-and-control (C&amp;C) server and is replied with the most recent version of the malware binary.&lt;/li&gt;
 	&lt;li&gt;ExoBot is fetched and executed on the device.&lt;/li&gt;
 	&lt;li&gt;ExoBot downloads a .dex file from the C&amp;C server. This file is the malware&#039;s main module. This component is central to ExoBot&#039;s functionality, and without it, most malware features would not work.&lt;/li&gt;
 	&lt;li&gt;The attacker can now send commands to the malware; those are handled by the main module.&lt;/li&gt;
&lt;/ol&gt;
&lt;p align=&quot;center&quot;&gt;&lt;img title=&quot;exobot architecture&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/1-1.png&quot; alt=&quot;IBM X-Force Mobile Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 1: ExoBot&#039;s architecture&lt;/em&gt;&lt;/p&gt;

&lt;h2 class=&quot;&quot;&gt;ExoBot&#039;s Infection Routine&lt;/h2&gt;
ExoBot mostly spreads to new devices via SMS spam and sometimes via downloader apps that &lt;a class=&quot;SWhtmlLink&quot; href=&quot;https://securityintelligence.com/anubis-strikes-again-mobile-malware-continues-to-plague-users-in-official-app-stores/&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;make it into the official stores&lt;/a&gt;.
&lt;h3 class=&quot;&quot;&gt;First Run&lt;/h3&gt;
At its first run on a device, ExoBot removes the application icon from the home screen, minimizes all running apps, and then shows the home screen so that the infected user wouldn&#039;t notice anything new on his or her device.
&lt;h3 class=&quot;&quot;&gt;OpSec Checks&lt;/h3&gt;
To evade sandboxes and automated analysis, ExoBot performs several checks before running:
&lt;ul&gt;
 	&lt;li&gt;&lt;strong&gt;Anti-emulation checks &lt;/strong&gt;— Like other malware, Android malware is often analyzed by mobile threat researchers&#039; emulators and sandboxes. To dodge the analysis, ExoBot checks the environment&#039;s build model, International Mobile Equipment Identity (IMEI) and registered operator.&lt;/li&gt;
 	&lt;li&gt;&lt;strong&gt;Antidebugging &lt;/strong&gt;— To evade debugging by automated tools, ExoBot uses the isDebuggerConnected() function to determine if a debugger is attached to the device.&lt;/li&gt;
 	&lt;li&gt;&lt;strong&gt;Country filter &lt;/strong&gt;— To limit the reach of local law enforcement, some malware codes avoid running in Eastern European countries. ExoBot&#039;s developer included a special list of countries where the malware should not run:
&lt;ul&gt;
 	&lt;li&gt;Armenia&lt;/li&gt;
 	&lt;li&gt;Azerbaijan&lt;/li&gt;
 	&lt;li&gt;Belarus&lt;/li&gt;
 	&lt;li&gt;Canada&lt;/li&gt;
 	&lt;li&gt;Kazakhstan&lt;/li&gt;
 	&lt;li&gt;Kyrgyzstan&lt;/li&gt;
 	&lt;li&gt;Moldova&lt;/li&gt;
 	&lt;li&gt;Russia&lt;/li&gt;
 	&lt;li&gt;Serbia and Montenegro&lt;/li&gt;
 	&lt;li&gt;Slovakia&lt;/li&gt;
 	&lt;li&gt;Tajikistan&lt;/li&gt;
 	&lt;li&gt;Turkmenistan&lt;/li&gt;
 	&lt;li&gt;Ukraine&lt;/li&gt;
 	&lt;li&gt;United States&lt;/li&gt;
 	&lt;li&gt;Uzbekistan&lt;/li&gt;
&lt;/ul&gt;
&lt;/li&gt;
&lt;/ul&gt;
Interestingly, Canada and the U.S. are on the list of countries that would not be targeted.

A second filter is applied by the language set on the device. The following language specifications will make ExoBot stop running on the device:
&lt;ul&gt;
 	&lt;li class=&quot;&quot;&gt;Armenian&lt;/li&gt;
 	&lt;li class=&quot;&quot;&gt;Azeri (Latin)&lt;/li&gt;

&lt;h3 class=&quot;&quot;&gt;Device Admin Request&lt;/h3&gt;
To be able to use all of its features, especially locking the user&#039;s screen and changing the lock screen&#039;s password, ExoBot needs to run with &lt;a class=&quot;SWhtmlLink&quot; href=&quot;https://securityintelligence.com/administrator-credentials-a-security-breach-waiting-to-happen/&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;admin permissions&lt;/a&gt;. Requiring this permission level involves the infected user, and to solicit his or her help, ExoBot displays a fake screen that purports to be the GNU General Public License.
&lt;p align=&quot;center&quot;&gt;&lt;img title=&quot;exobot fake admin request&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/3.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 2: ExoBot asking for admin permissions via a fake GNU page&lt;/em&gt;&lt;/p&gt;
By gaining the admin permission level, ExoBot can also establish its persistence mechanism and ensure that the user would not be able to remove the app without disabling the malware with equal permissions. Getting rid of ExoBot involves manually removing it from the device admin apps list. Unfortunately, even after removals, ExoBot will pop up a screen asking the user to activate it.
&lt;h3 class=&quot;&quot;&gt;Obfuscation&lt;/h3&gt;
To keep its resources protected, ExoBot&#039;s developer uses two obfuscation techniques:
&lt;ol&gt;
 	&lt;li&gt;&lt;strong&gt;String protection&lt;/strong&gt; — The malware&#039;s operator can elect to replace all string characters with a key of his or her choosing.&lt;/li&gt;
&lt;/ol&gt;
&lt;p align=&quot;center&quot;&gt;&lt;img title=&quot;exobot analysis&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/8.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 3: String protection&lt;/em&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;This can be reversed by a decrypt function, which replaces the new key with the default key and reverts to the previous string.&lt;/p&gt;
&lt;img title=&quot;exobot encoded strings&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/9.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 4: Reversing the string obfuscation&lt;/em&gt;&lt;/p&gt;

&lt;ol style=&quot;text-align: left;&quot; start=&quot;2&quot;&gt;
 	&lt;li&gt;&lt;strong&gt;Advanced Encryption Standard (AES) encryption&lt;/strong&gt; of requests and responses from the C&amp;C server, as well as encrypting the malware&#039;s configuration. The encryption was implemented using a hardcoded key and then Base64 encoding.&lt;/li&gt;
&lt;/ol&gt;
&lt;h3 class=&quot;&quot;&gt;ExoBot&#039;s Continuous C&amp;C Communication&lt;/h3&gt;
With new infections, ExoBot retrieves modules from the C&amp;C server and then continually keeps in touch with it to check for new tasks assigned by the attacker. If new tasks are found, ExoBot queues them for execution in a designated file located in in a private folder on disk. All the information in the queue is saved in encrypted form. The queue file is used throughout the malware&#039;s activity and life cycle.

To keep on top of tasks, ExoBot checks them periodically and by an alarm setup. Requests and responses are kept encrypted in JavaScript Object Notation (JSON) format.
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;img title=&quot;exobot android malware analysis&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/10.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 5: Queue file with pending tasks&lt;/em&gt;&lt;/p&gt;
Until a new task is sent through, the malware will continue to update the C&amp;C server with information about the device, the installed apps and a configuration status.
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;img title=&quot;exobot communication with c2&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/13_cns.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 6: ExoBot sends a request with device information to the C&amp;C server&lt;/em&gt;&lt;/p&gt;
ExoBot&#039;s C&amp;C communication is secured with an SSL.

To identify devices, each infected device receives a unique MD5 with its identifiers: IMEI, build model, central processing unit (CPU) and Android ID. The details are transmitted to the C&amp;C server under the &quot;I&quot; field inside every JSON request.
&lt;h3 class=&quot;&quot;&gt;Dynamic Module Loading&lt;/h3&gt;
Unlike similar malware, ExoBot has a unique method to load its modules in a dynamic way. The image blow shows the code that enables this tactic to work:
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;img title=&quot;exobot analysis&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/2.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 7: Code snippet showing ExoBot&#039;s dynamic module loading&lt;/em&gt;&lt;/p&gt;
The developer here used the payload to manage module loading. The main module is the component that downloads the modules from the C&amp;C on its first run, and it&#039;s also the component that handles them later when they are required to run.

Modules are loaded to memory in reflection, giving the malware the ability to inspect and dynamically call classes, methods and attributes at runtime instead of having to do that at compile time.

The reflection is implemented using the Android DexClassLoader application programming interface (API), which is a class loader that allows an application to load .jar and APK files. This method can be used to execute code that was not previously installed as part of an existing app and load it into that app during runtime.

Note that ExoBot&#039;s .dex file is still present on disk, but the code is not packed within the APK. Using reflection in this case allows the malware to avoid static analysis of its modules and keep them out of antivirus engines&#039; sight. Moreover, loading in runtime gives ExoBot added agility. It can fetch new modules from the C&amp;C server at any time and add/update its features easily and quickly on devices that are already infected.
&lt;h2 class=&quot;&quot;&gt;Module Execution&lt;/h2&gt;
To execute modules, ExoBot&#039;s C&amp;C server sends the malware a command with relevant parameters alongside the MD5 hash of a .dex file that contains the module to ensure the correct one is executed:
&lt;p class=&quot;&quot; style=&quot;padding-left: 30px;&quot;&gt;&lt;em&gt;data/data/app_package_name/m/md5&lt;strong&gt;(module_name).dex&lt;/strong&gt;&lt;/em&gt;&lt;/p&gt;
The malware then proceeds with the required action on the infected device.

In some cases, the module may not exist in the malware&#039;s folder on the device, and it will launch a request to download it from the C&amp;C server.
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;img title=&quot;exobot communication with c2&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/11_cns.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 8: ExoBot requesting a module download from the C&amp;C server to get the victim&#039;s contacts&lt;/em&gt;&lt;/p&gt;
Next, ExoBot hashes the required module on disk and compares its MD5 to the one attributed to that module on the C&amp;C server. This step is taken to prevent tampering and errors. Upon confirming a match, the malware loads the requested module and reports the result back to the C&amp;C server.
&lt;p align=&quot;center&quot;&gt;&lt;img title=&quot;exobot communication with c2 server&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/12_cns.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 9: ExoBot&#039;s request to get contacts from the victim&#039;s texting app&lt;/em&gt;&lt;/p&gt;
A result in JSON format contains data returned from the module alongside an API status field indicating whether the module is running properly or has failed.

&lt;h2 class=&quot;&quot;&gt;ExoBot&#039;s Overlay Screen Mechanism&lt;/h2&gt;
Overlay malware has been around for a while, but the method became popular when &lt;a href=&quot;https://securityintelligence.com/gm-bot-alive-upgraded-now-android-m/&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;GM Bot&lt;/a&gt; started commercializing it in the underground. Ever since GM Bot, &lt;a href=&quot;https://securityintelligence.com/mobile-malware-competition-rises-in-underground-markets/&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;various Android banking Trojans&lt;/a&gt; use the overlay tactic, implementing its use in different ways.

In ExoBot&#039;s case, the overlay screen pop-up takes place by detecting the foreground app and then displaying a custom webpage activity to match that app. In the case of banks, cryptocurrency exchanges, payment and shopping apps, the fake screen will match their login screens.

To begin, ExoBot gets its target list from the C&amp;C server. The malware scans all the files found under the /proc virtual filesystem to map all running processes the user initiated. It does not scan apps and processes users cannot launch themselves.
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;img title=&quot;exobot process fetching&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/4.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 10: Listing all active processes from /proc&lt;/em&gt;&lt;/p&gt;
At startup, services of the Android operating system run in the bg_non_interactive state. This state means that the service is at low priority at the time and cannot use more than 5 percent of the CPU. To determine if an app of interest is running in the foreground, the malware checks if the app&#039;s process ID (PID) is &lt;em&gt;not&lt;/em&gt; in the noninteractive state list.
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;img title=&quot;exobot looks up inactive processes&quot; src=&quot;https://securityintelligence.com/wp-content/uploads/2018/09/5.png&quot; alt=&quot;IBM X-Force Mobile Threat Research&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: left;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 11: Looking for processes that run in the foreground&lt;/em&gt;&lt;/p&gt;
If a targeted app is running in the foreground, ExoBot will pop up a new activity on top of the original app, covering it and prioritizing the fake screen. In the overlay, a custom webpage impersonates the login interface of the original app to phish the victim&#039;s credentials.
&lt;h2 class=&quot;&quot;&gt;Persistent Credit Card Requests&lt;/h2&gt;
On top of targeting banking and other financial applications, social and chat apps, ExoBot possesses a module designed to steal payment card information.

This module is tasked with frequent pop-ups of fake screens that require victims to tap their payment card details into the activity. The lure is set up as a notification from Google Play, or from other services for Android-based devices.
&lt;h2 class=&quot;&quot;&gt;Mobile Malware Continues to Evolve&lt;/h2&gt;
ExoBot&#039;s code leak is just another leak among others that provide cybercriminals with a basis for malware that they can adapt to their own illicit operations and attacks. With this code leak, X-Force researchers expect to see ExoBot offspring in the coming months, as well as new actors who will use it in different geographies to target local banks through their mobile banking user base.

To mitigate the threat of mobile malware, organizations can help protect their apps with &lt;a href=&quot;https://www.ibm.com/security/fraud-protection&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;IBM Trusteer&lt;/a&gt;. Those managing employee mobile devices can learn more about IBM&#039;s unified endpoint management (UEM) solution, &lt;a href=&quot;https://www.ibm.com/security/mobile/maas360&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;MaaS360&lt;/a&gt;.

<html><body><p>Following the <a href="https://securityintelligence.com/news/unknown-actor-leaks-android-malware-exobot-source-code/"  target="_blank" rel="noopener nofollow" >recent source code leak</a> of the Android banking Trojan ExoBot, IBM X-Force research delved into the malware&rsquo;s inner workings to help uncover insights into its dynamic mechanisms and the features that help criminals use it in cross-channel bank fraud.</p>
<h2 class="">ExoBot Genesis</h2>
<p>ExoBot is Android malware that was based originally on a previous code known as <a href="https://securityintelligence.com/marcher-mobile-bot-adds-uk-targets-steps-up-banking-fraud-capabilities/"  target="_blank" rel="noopener nofollow" >Marcher</a>. This code represents a banking Trojan that uses the overlay technique &mdash; that is, popping up fake windows that hide the original app users open &mdash; to trick victims into tapping their banking credentials into a fake interface. After stealing account access details, the malware can also intercept SMS messages and phone calls, thereby enabling criminals to take over the victim&rsquo;s bank account and other financial accounts at their discretion.</p>
<p>Some of the capabilities that enable ExoBot to facilitate fraudulent activity on infected devices include gaining admin privileges, launching overlay screens, and exfiltrating SMS, data and other information from the infected device.</p>
<p>In 2016, ExoBot&rsquo;s developer was selling the malware on the clear web for a while, even <a href="https://exchange.xforce.ibmcloud.com/collection/ExoBot-Android-Malware-e112e0be1fefdcfbc013f6202a6fcaff" target="_blank" rel="noopener nofollow"  target="_blank" rel="noopener nofollow" >advertising an upgrade</a> in May 2017. In January 2018, the actor <a href="https://www.bleepingcomputer.com/news/security/exobot-author-calls-it-quits-and-sells-off-banking-trojan-source-code/" target="_blank" rel="noopener nofollow"  target="_blank" rel="noopener nofollow" >decided to sell it off</a> in the underground, but by May 2018, the source code was leaked online by an unknown actor.</p>
<p>Source code leaks, especially those of Android malware codes, have <a href="https://securityintelligence.com/android-malware-about-to-get-worse-gm-bot-source-code-leaked/"  target="_blank" rel="noopener nofollow" >happened in the recent past</a>. When they do, they give rise to variants and variations of the same malware, lowering the bar for more criminals to enter the mobile malware scene and try their hand at mobile banking fraud.</p>
<h2>Delving Into ExoBot&rsquo;s Inner Workings</h2>
<p>The following sections describe technical details about ExoBot as analyzed by X-Force mobile threat researcher Shahar Tavor.</p>
<h3 class="">Architecture Basics</h3>
<p>Having analyzed ExoBot&rsquo;s leaked source code, X-Force researchers found the malware to possess some notable features, such as dynamic modules loaded at runtime and some anti-emulation tricks to evade popular sandboxes.</p>
<p>The image below illustrates the malware&rsquo;s architecture. The overall routine occurs as follows:</p>
<ol>
<li>An Android mobile device user unknowingly downloads a loader app via SMS spam or directly from the official store.</li>
<li>The loader app calls back to the attacker&rsquo;s command-and-control (C&amp;C) server and is replied with the most recent version of the malware binary.</li>
<li>ExoBot is fetched and executed on the device.</li>
<li>ExoBot downloads a .dex file from the C&amp;C server. This file is the malware&rsquo;s main module. This component is central to ExoBot&rsquo;s functionality, and without it, most malware features would not work.</li>
<li>The attacker can now send commands to the malware; those are handled by the main module.</li>
</ol>
<p align="center"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/1-1.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Research" width="806" height="645" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 1: ExoBot&rsquo;s architecture</em></p>
<h2 class="">ExoBot&rsquo;s Infection Routine</h2>
<p>ExoBot mostly spreads to new devices via SMS spam and sometimes via downloader apps that <a class="SWhtmlLink" href="https://securityintelligence.com/anubis-strikes-again-mobile-malware-continues-to-plague-users-in-official-app-stores/"  target="_blank" rel="noopener nofollow" >make it into the official stores</a>.</p>
<h3 class="">First Run</h3>
<p>At its first run on a device, ExoBot removes the application icon from the home screen, minimizes all running apps, and then shows the home screen so that the infected user wouldn&rsquo;t notice anything new on his or her device.</p>
<h3 class="">OpSec Checks</h3>
<p>To evade sandboxes and automated analysis, ExoBot performs several checks before running:</p>
<ul>
<li><strong>Anti-emulation checks </strong>&mdash; Like other malware, Android malware is often analyzed by mobile threat researchers&rsquo; emulators and sandboxes. To dodge the analysis, ExoBot checks the environment&rsquo;s build model, International Mobile Equipment Identity (IMEI) and registered operator.</li>
<li><strong>Antidebugging </strong>&mdash; To evade debugging by automated tools, ExoBot uses the isDebuggerConnected() function to determine if a debugger is attached to the device.</li>
<li><strong>Country filter </strong>&mdash; To limit the reach of local law enforcement, some malware codes avoid running in Eastern European countries. ExoBot&rsquo;s developer included a special list of countries where the malware should not run:
<ul>
<li>Armenia</li>
<li>Azerbaijan</li>
<li>Belarus</li>
<li>Canada</li>
<li>Kazakhstan</li>
<li>Kyrgyzstan</li>
<li>Moldova</li>
<li>Russia</li>
<li>Serbia and Montenegro</li>
<li>Slovakia</li>
<li>Tajikistan</li>
<li>Turkmenistan</li>
<li>Ukraine</li>
<li>United States</li>
<li>Uzbekistan</li>
</ul>
</li>
</ul>
<p>Interestingly, Canada and the U.S. are on the list of countries that would not be targeted.</p>
<p>A second filter is applied by the language set on the device. The following language specifications will make ExoBot stop running on the device:</p>
<ul>
<li class="">Armenian</li>
<li class="">Azeri (Latin)</li>
<li class="">Belarusian</li>
<li class="">Chinese</li>
<li class="">Czech</li>
<li class="">Indo-European</li>
<li class="">Kyrgyz</li>
<li class="">Romanian</li>
<li class="">Russian</li>
<li class="">Slovak</li>
<li class="">Turkish</li>
<li class="">Ukrainian</li>
<li class="">Uzbek (Latin)</li>
</ul>
<h3 class="">Device Admin Request</h3>
<p>To be able to use all of its features, especially locking the user&rsquo;s screen and changing the lock screen&rsquo;s password, ExoBot needs to run with <a class="SWhtmlLink" href="https://securityintelligence.com/administrator-credentials-a-security-breach-waiting-to-happen/"  target="_blank" rel="noopener nofollow" >admin permissions</a>. Requiring this permission level involves the infected user, and to solicit his or her help, ExoBot displays a fake screen that purports to be the GNU General Public License.</p>
<p align="center"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/3.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="290" height="506" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 2: ExoBot asking for admin permissions via a fake GNU page</em></p>
<p>By gaining the admin permission level, ExoBot can also establish its persistence mechanism and ensure that the user would not be able to remove the app without disabling the malware with equal permissions. Getting rid of ExoBot involves manually removing it from the device admin apps list. Unfortunately, even after removals, ExoBot will pop up a screen asking the user to activate it.</p>
<h3 class="">Obfuscation</h3>
<p>To keep its resources protected, ExoBot&rsquo;s developer uses two obfuscation techniques:</p>
<ol>
<li><strong>String protection</strong> &mdash; The malware&rsquo;s operator can elect to replace all string characters with a key of his or her choosing.</li>
</ol>
<p align="center"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/8.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="390" height="108" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 3: String protection</em></p>
<p class="" style="text-align: left;" align="center">This can be reversed by a decrypt function, which replaces the new key with the default key and reverts to the previous string.</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/9.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="1572" height="211" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 4: Reversing the string obfuscation</em></p>
<ol style="text-align: left;" start="2">
<li><strong>Advanced Encryption Standard (AES) encryption</strong> of requests and responses from the C&amp;C server, as well as encrypting the malware&rsquo;s configuration. The encryption was implemented using a hardcoded key and then Base64 encoding.</li>
</ol>
<h3 class="">ExoBot&rsquo;s Continuous C&amp;C Communication</h3>
<p>With new infections, ExoBot retrieves modules from the C&amp;C server and then continually keeps in touch with it to check for new tasks assigned by the attacker. If new tasks are found, ExoBot queues them for execution in a designated file located in in a private folder on disk. All the information in the queue is saved in encrypted form. The queue file is used throughout the malware&rsquo;s activity and life cycle.</p>
<p>To keep on top of tasks, ExoBot checks them periodically and by an alarm setup. Requests and responses are kept encrypted in JavaScript Object Notation (JSON) format.</p>
<p style="text-align: center;"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/10.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="1195" height="102" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 5: Queue file with pending tasks</em></p>
<p>Until a new task is sent through, the malware will continue to update the C&amp;C server with information about the device, the installed apps and a configuration status.</p>
<p style="text-align: center;"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/13_cns.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="1710" height="189" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 6: ExoBot sends a request with device information to the C&amp;C server</em></p>
<p>ExoBot&rsquo;s C&amp;C communication is secured with an SSL.</p>
<p>To identify devices, each infected device receives a unique MD5 with its identifiers: IMEI, build model, central processing unit (CPU) and Android ID. The details are transmitted to the C&amp;C server under the &ldquo;I&rdquo; field inside every JSON request.</p>
<h3 class="">Dynamic Module Loading</h3>
<p>Unlike similar malware, ExoBot has a unique method to load its modules in a dynamic way. The image blow shows the code that enables this tactic to work:</p>
<p style="text-align: center;"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/2.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="1007" height="95" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 7: Code snippet showing ExoBot&rsquo;s dynamic module loading</em></p>
<p>The developer here used the payload to manage module loading. The main module is the component that downloads the modules from the C&amp;C on its first run, and it&rsquo;s also the component that handles them later when they are required to run.</p>
<p>Modules are loaded to memory in reflection, giving the malware the ability to inspect and dynamically call classes, methods and attributes at runtime instead of having to do that at compile time.</p>
<p>The reflection is implemented using the Android DexClassLoader application programming interface (API), which is a class loader that allows an application to load .jar and APK files. This method can be used to execute code that was not previously installed as part of an existing app and load it into that app during runtime.</p>
<p>Note that ExoBot&rsquo;s .dex file is still present on disk, but the code is not packed within the APK. Using reflection in this case allows the malware to avoid static analysis of its modules and keep them out of antivirus engines&rsquo; sight. Moreover, loading in runtime gives ExoBot added agility. It can fetch new modules from the C&amp;C server at any time and add/update its features easily and quickly on devices that are already infected.</p>
<h2 class="">Module Execution</h2>
<p>To execute modules, ExoBot&rsquo;s C&amp;C server sends the malware a command with relevant parameters alongside the MD5 hash of a .dex file that contains the module to ensure the correct one is executed:</p>
<p class="" style="padding-left: 30px;"><em>data/data/app_package_name/m/md5<strong>(module_name).dex</strong></em></p>
<p>The malware then proceeds with the required action on the infected device.</p>
<p>In some cases, the module may not exist in the malware&rsquo;s folder on the device, and it will launch a request to download it from the C&amp;C server.</p>
<p style="text-align: center;"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/11_cns.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="1092" height="229" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 8: ExoBot requesting a module download from the C&amp;C server to get the victim&rsquo;s contacts</em></p>
<p>Next, ExoBot hashes the required module on disk and compares its MD5 to the one attributed to that module on the C&amp;C server. This step is taken to prevent tampering and errors. Upon confirming a match, the malware loads the requested module and reports the result back to the C&amp;C server.</p>
<p align="center"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/12_cns.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="968" height="362" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 9: ExoBot&rsquo;s request to get contacts from the victim&rsquo;s texting app</em></p>
<p>A result in JSON format contains data returned from the module alongside an API status field indicating whether the module is running properly or has failed.</p>

<h2 class="">ExoBot&rsquo;s Overlay Screen Mechanism</h2>
<p>Overlay malware has been around for a while, but the method became popular when <a href="https://securityintelligence.com/gm-bot-alive-upgraded-now-android-m/"  target="_blank" rel="noopener nofollow" >GM Bot</a> started commercializing it in the underground. Ever since GM Bot, <a href="https://securityintelligence.com/mobile-malware-competition-rises-in-underground-markets/"  target="_blank" rel="noopener nofollow" >various Android banking Trojans</a> use the overlay tactic, implementing its use in different ways.</p>
<p>In ExoBot&rsquo;s case, the overlay screen pop-up takes place by detecting the foreground app and then displaying a custom webpage activity to match that app. In the case of banks, cryptocurrency exchanges, payment and shopping apps, the fake screen will match their login screens.</p>
<p>To begin, ExoBot gets its target list from the C&amp;C server. The malware scans all the files found under the /proc virtual filesystem to map all running processes the user initiated. It does not scan apps and processes users cannot launch themselves.</p>
<p style="text-align: center;"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/4.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="638" height="374" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 10: Listing all active processes from /proc</em></p>
<p>At startup, services of the Android operating system run in the bg_non_interactive state. This state means that the service is at low priority at the time and cannot use more than 5 percent of the CPU. To determine if an app of interest is running in the foreground, the malware checks if the app&rsquo;s process ID (PID) is <em>not</em> in the noninteractive state list.</p>
<p style="text-align: center;"><amp-img src="https://securityintelligence.com/wp-content/uploads/2018/09/5.png" layout="intrinsic" class="" alt="IBM X-Force Mobile Threat Research" width="694" height="113" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: left;" align="center"><em>Figure 11: Looking for processes that run in the foreground</em></p>
<p>If a targeted app is running in the foreground, ExoBot will pop up a new activity on top of the original app, covering it and prioritizing the fake screen. In the overlay, a custom webpage impersonates the login interface of the original app to phish the victim&rsquo;s credentials.</p>
<h2 class="">Persistent Credit Card Requests</h2>
<p>On top of targeting banking and other financial applications, social and chat apps, ExoBot possesses a module designed to steal payment card information.</p>
<p>This module is tasked with frequent pop-ups of fake screens that require victims to tap their payment card details into the activity. The lure is set up as a notification from Google Play, or from other services for Android-based devices.</p>
<h2 class="">Mobile Malware Continues to Evolve</h2>
<p>ExoBot&rsquo;s code leak is just another leak among others that provide cybercriminals with a basis for malware that they can adapt to their own illicit operations and attacks. With this code leak, X-Force researchers expect to see ExoBot offspring in the coming months, as well as new actors who will use it in different geographies to target local banks through their mobile banking user base.</p>
