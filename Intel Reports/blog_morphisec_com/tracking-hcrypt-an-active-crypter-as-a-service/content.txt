<p>In 2021 Morphisec identified increased usage of the “<span style="font-weight: bold;">HCrypt</span>” crypter. In this post, we lockpick “HCrypt”—a <span style="font-weight: bold;">crypter as a service</span> marketed as a FUD (fully undetectable) loader for the client's RAT of choice. We chose to dissect the crypter’s operations along with tracking several actors that utilize it.</p>

<p>The crypter-as-a-service model is indicative of the trend toward malware authors creating and selling code to other groups with less technical sophistication. As a result, more financially motivated threat actors can adopt better attacks if they have the money to spend. This results in many groups putting forward the bare-minimum effort required to execute sophisticated malware campaigns.</p>
<h2>Technical Introduction</h2>
<p><span style="font-size: 11px; color: #000000;"><img src="https://blog.morphisec.com/hs-fs/hubfs/Summarized%20loader%20execution%20flow.png?width=798&amp;height=449&amp;name=Summarized%20loader%20execution%20flow.png" alt="Summarized loader execution flow" width="798" height="449" loading="lazy" style="height: auto; max-width: 100%; width: 798px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Summarized%20loader%20execution%20flow.png?width=399&amp;height=225&amp;name=Summarized%20loader%20execution%20flow.png 399w, https://blog.morphisec.com/hs-fs/hubfs/Summarized%20loader%20execution%20flow.png?width=798&amp;height=449&amp;name=Summarized%20loader%20execution%20flow.png 798w, https://blog.morphisec.com/hs-fs/hubfs/Summarized%20loader%20execution%20flow.png?width=1197&amp;height=674&amp;name=Summarized%20loader%20execution%20flow.png 1197w, https://blog.morphisec.com/hs-fs/hubfs/Summarized%20loader%20execution%20flow.png?width=1596&amp;height=898&amp;name=Summarized%20loader%20execution%20flow.png 1596w, https://blog.morphisec.com/hs-fs/hubfs/Summarized%20loader%20execution%20flow.png?width=1995&amp;height=1123&amp;name=Summarized%20loader%20execution%20flow.png 1995w, https://blog.morphisec.com/hs-fs/hubfs/Summarized%20loader%20execution%20flow.png?width=2394&amp;height=1347&amp;name=Summarized%20loader%20execution%20flow.png 2394w" sizes="(max-width: 798px) 100vw, 798px"></span></p>
<p style="font-weight: bold; font-size: 12px; text-align: center;">Figure 2: Summarized loader execution flow</p>
<p>Our description of the attack chain flow follows the artifacts that are known to us. Although the initial access infection vector is missing, we have identified cases in which a VBS code is executed that leads to an .hta file execution described as Encoding.txt. The next stages involve persistence and AV evasion through PowerShell, and then the final stage consists of a standard .Net reflective loader which loads the RAT of choice. Along the way, the actors and the author use free accessible code and file sharing services such as github.com, cdn.discordapp.com, and minpic.de.</p>
<p>Within all of its versions, the crypter maintains the same execution flow with different code tweaks in an attempt to avoid detection by AV. The above diagram covers the main Crypter&nbsp;functionality for several versions that we have observed since Jan 2021.</p>
<h3>The First HCrypt Stage: Encoding.txt</h3>
<p>‘Encoding.txt’, along with the other .txt file names mentioned in the diagram and within this blog refers to the specific stage internal name within the <em>crypter application</em> (this will be presented later).&nbsp;</p>
<p>This is usually the first stage execution (sometimes wrapped in a .vbs file). Its purpose is to elevate the execution flow to PowerShell and get the additional code by downloading it from a user-defined custom URL (the user here is the ‘actor’ who uses the crypter).</p>

<h3>The Second HCrypt Stage: ALL.txt</h3>
<p>This stage’s purpose is to set up persistence along with downloading, saving, and executing the next stage on the victim’s host. Usually, it can be identified by the author fingerprint, which names the code`s function “HBar.”</p>
<ul>
<li aria-level="1">The name of the saved file, which is also one of the focal fingerprints for this crypter, is ‘Microsoft.ps1’. Usually, this file will refer either to an AV bypass or server.txt depends on the version/configuration.</li>
<li aria-level="1">If configured by the user (actor), persistence is achieved by downloading and saving a .hta or .vbs file to the victim’s “startup” directory. This script executes a 1-liner PowerShell code that executes the described Microsoft.ps1 above. Most of the observed variants download this file from a hard-coded URL within the crypter from one of the following author`s GitHub repositories.</li>
</ul>

<h3>The Third HCrypt Stage: AV Bypass</h3>
<p>This PowerShell script function, usually named “HBankers,” may appear on some versions of the <em>HCrypt</em> attack flow. As of this writing the AV identification functionality seems to be still in development. The flow of the attack doesn’t change with AV detection.</p>

<h3>The Fourth Stage: Server.txt</h3>
<p>The final PowerShell stage, often hosted as a .jpg file, decodes and executes the loader and payload.</p>
<p>The loader and payload are hard-coded in this stage within a byte array variable, while each version saves it in a different format and names the variable differently (i.e $H1, $nam2021, $brazi). We have observed that those byte arrays contain PE files embedded in Hex, Decimal, or Base64 formats and sometimes also with character swapping as a simple encoding.</p>
