tag-digital-risk-protection tag-threat-intelligence" id="post-5786"> <div class="entry-content"> <p><em>Note: this post was originally shared on </em><a href="https://squiblydoo.blog/"><em>https://squiblydoo.blog/</em></a><em> by a member of the Binary Defense Team. In order to ensure this research is visible to a broader audience, this employee agreed to let us share it here.</em></p> <p><em>Since the original post, the threat actor has been updating their malware and their tactics weekly. Some information in this post does not reflect the current version of the malware. However, the information is still important for defenders as there are still old infections that resemble the documented behavior, and the tactics can be used by other actors.</em></p> <h3 class="wp-block-heading">Part One: Persistence Script Analysis</h3> <p>Mars-Deimos-RS-2.MD5: 88E60DFE5045E7157D71D1CB4170C073<br />Mars-Deimos-RS-2:SHA256: 8A57BD2598057EE784711B47B9B61B4ECBA5311FAC800B55070D560480F86EAC</p> <h3 class="wp-block-heading">Mars Deimos Overview</h3> <p>Mars-Deimos-RS-2 is .NET binary injected into memory. It has also been documented as&nbsp;<a href="https://www.crowdstrike.com/blog/solarmarker-backdoor-technical-analysis/">Solarmarker by CrowdStrike</a>. It shares Indicators of Compromise (IOC) with a directly related malware which has been documented as&nbsp;<a href="https://blog.morphisec.com/jupyter-infostealer-backdoor-introduction">Jupyter Infostealer by Morphisec</a>. The particular binary studied here appears to be tracked internally by the author as “Mars Deimos AppVersion RS-2”. Mars Deimos is a C2 client and the binary under analysis is the backdoor stage of the malware. The Jupyter variant shares functionality with Mars Deimos but can also steal cookies from browsers.</p> <p><em>Note on naming: Though it can be confusing if there are too many names for the same malware, I want to maintain calling it “Mars Deimos” in order to assist defenders. In the original script, you can see it called “Mars Deimos” but I could not find any information about it under that name. In the Solarwinds analysis, it is called “D:M” by the malware author, so it seems like the name “Mars Deimos” is appropriate to maintain. Jupyter shares some IOC with Mars Deimos, which can complicate things, but they are maintained independently by the malware authors and have different functionality</em>.</p> <p>My goal in this post is provide analysis of the persistence script. Because Mars Deimos is a present-day threat, it is important for administrators and analysts to recognize and understand the scripts if they are found.</p> <h3 class="wp-block-heading">How the Mars Deimos malware is delivered</h3> <p>According to the analysis by CrowdStrike and Morphisec, Mars Deimos normally has several stages. The binary for this analysis appears to be the fourth or fifth stage (according to CrowdStrike) and is the persistent backdoor. My analysis began at this stage and I do not have access to the earlier stages, but it appears that a user downloaded an executable that was disguised as a Word document. The executable then dropped two files: a .bat script and an unreadable file without an extension. For persistence, shortcuts on the desktop were modified to call the .bat script.</p> <h3 class="wp-block-heading">Beginning Analysis of Mars Deimos</h3> <p>As is common when PowerShell is being executed maliciously, the .bat script is written in a way to be confusing as follows:</p> <ul class="wp-block-list"><li>The script was named “DZWhBTixXsCjSOuNobQfpImvelygwUznrLHGPtkFAaMKVYcJ.cMd”,</li><li>the script is all one line</li><li>the variable names are complicated (such as “$ab13e4b80f240fb13f8a62c3a6db5”).</li></ul> <div class="wp-block-image"><figure class="aligncenter size-large"><img alt=" class=" wp-image-32102"/ nitro-lazy-src="https://cdn-dkajddp.nitrocdn.com/rHVGjbdIfsJHeZKoyaQvWfrAvKeGdBoq/assets/images/optimized/rev-e6195f9/www.binarydefense.com/wp-content/uploads/2023/04/Picture1-1024x188.jpg" class="nitro-lazy" decoding="async" nitro-lazy-empty id="NjExOjIwNA==-1" src="data:image/svg+xml;nitro-empty-id=NjExOjIwNA==-1;base64,PHN2ZyB2aWV3Qm94PSIwIDAgMTAyNCAxODgiIHdpZHRoPSIxMDI0IiBoZWlnaHQ9IjE4OCIgeG1sbnM9Imh0dHA6Ly93d3cudzMub3JnLzIwMDAvc3ZnIj48L3N2Zz4=" /><figcaption><em>Image of the Script before alterations</em></figcaption></figure></div> <p>To be honest, looking at a script like this, it can boggle the mind quite easily.</p> <p>First, we will break up the commands into separate lines. Looking at the beginning, we can see that it is using PowerShell (spelled “poWeRsHEll” to avoid detection methods that look for the word “PowerShell”) so we will break it at some of the semi-colons to make it easier to read. (Some of the semicolons appear to be in a For-Loop, so we won’t break those up just yet.)</p> <div class="wp-block-image"><figure class="aligncenter size-large"><img alt=" class=" wp-image-32103"/ nitro-lazy-src="https://cdn-dkajddp.nitrocdn.com/rHVGjbdIfsJHeZKoyaQvWfrAvKeGdBoq/assets/images/optimized/rev-e6195f9/www.binarydefense.com/wp-content/uploads/2023/04/Picture2-1024x231.jpg" class="nitro-lazy" decoding="async" nitro-lazy-empty id="NjIzOjIwNA==-1" src="data:image/svg+xml;nitro-empty-id=NjIzOjIwNA==-1;base64,PHN2ZyB2aWV3Qm94PSIwIDAgMTAyNCAyMzEiIHdpZHRoPSIxMDI0IiBoZWlnaHQ9IjIzMSIgeG1sbnM9Imh0dHA6Ly93d3cudzMub3JnLzIwMDAvc3ZnIj48L3N2Zz4=" /><figcaption><em>Image of the Script broken into sections to improve readibility</em></figcaption></figure></div> <p>The script is still hard to read because of the variable names. At this point, the purpose of the variable in line 1 is unclear… but in line 3, a variable is assigned a value from [system.io.file], and it appears to be getting the file from a “base64string” that it is converting.</p> <p>The easiest way we can convert that string from Base64 is using&nbsp;<a href="https://gchq.github.io/CyberChef/">CyberChef</a>: we can copy and paste that string into the&nbsp;<strong>Input</strong>&nbsp;section of CyberChef, drag-and-drop “From Base64” into the&nbsp;<strong>Recipe</strong>&nbsp;section and then, see the decoded string in the&nbsp;<strong>Output</strong>. In our instance, we receive a file path to a second file: “C:UsersUserNameAppDataRoamingbYtNoKkvyXglTMfAEaPreQVGbHpIFdzCjsfklSGPyQVrJqUXvMBuanNcODhwmxgLKiYWRtAETeZo”</p> <div class="wp-block-image"><figure class="aligncenter size-large"><img alt=" class=" wp-image-32104"/ nitro-lazy-src="https://cdn-dkajddp.nitrocdn.com/rHVGjbdIfsJHeZKoyaQvWfrAvKeGdBoq/assets/images/optimized/rev-e6195f9/www.binarydefense.com/wp-content/uploads/2023/04/Picture3-1024x241.jpg" class="nitro-lazy" decoding="async" nitro-lazy-empty id="NjM1OjIwNA==-1" src="data:image/svg+xml;nitro-empty-id=NjM1OjIwNA==-1;base64,PHN2ZyB2aWV3Qm94PSIwIDAgMTAyNCAyNDEiIHdpZHRoPSIxMDI0IiBoZWlnaHQ9IjI0MSIgeG1sbnM9Imh0dHA6Ly93d3cudzMub3JnLzIwMDAvc3ZnIj48L3N2Zz4=" /><figcaption><em>Image of CyberChef output after decoding Base64 input</em></figcaption></figure></div> <p>With this information, we can conclude that a second file is being read into the variable on line 3, $a6eb2a91b614769b847e6964de33d. In the original incident response, a file was indeed found at that location and I have that file. Let’s rename every instance of the variable name to $fileTwo since we don’t have much more information at this point.</p> <div class="wp-block-image"><figure class="aligncenter size-large"><img alt=" class=" wp-image-32105"/ nitro-lazy-src="https://cdn-dkajddp.nitrocdn.com/rHVGjbdIfsJHeZKoyaQvWfrAvKeGdBoq/assets/images/optimized/rev-e6195f9/www.binarydefense.com/wp-content/uploads/2023/04/Picture4-1024x158.jpg" class="nitro-lazy" decoding="async" nitro-lazy-empty id="NjQzOjIwNA==-1" src="data:image/svg+xml;nitro-empty-id=NjQzOjIwNA==-1;base64,PHN2ZyB2aWV3Qm94PSIwIDAgMTAyNCAxNTgiIHdpZHRoPSIxMDI0IiBoZWlnaHQ9IjE1OCIgeG1sbnM9Imh0dHA6Ly93d3cudzMub3JnLzIwMDAvc3ZnIj48L3N2Zz4=" /><figcaption><em>Image of script with the variable renamed to $fileTwo</em></figcaption></figure></div> <p>Having renamed the variable, we see that after it is assigned the bytes of our second file in line 3, the variable is used a few times in the For-Loop and after the For-Loop, $fileTwo is&nbsp;<em>loaded</em>&nbsp;using [system.reflection.assembly]::load($filetwo).</p> <p>Next, we clean up the For-Loop to make it more readable. For Loops usually initialize a variable for use in the loop (an iterator variable), create a condition that that is checked to see if the variable should continue, and then (optionally) a call to increment the iterator variable. Typically when writing a program, one uses “$i” for the iterator variable in a For-Loop; if there are multiple loops, the programmer names the next iterator “$j”. Since we appear to have two For-Loops, we’ll rename the variable in the first For-Loop, “$ad066d1ff554189be1555e5c01765”, to “$i” and the variable initialized in the second For-Loop, “$ac07982da3749c86bd16117fb94f0”, to “$j”. Also, while we are at it, we will format it to look like a normal For-Loop.</p> <div class="wp-block-image"><figure class="aligncenter size-large"><img alt=" class=" wp-image-32106"/ nitro-lazy-src="https://cdn-dkajddp.nitrocdn.com/rHVGjbdIfsJHeZKoyaQvWfrAvKeGdBoq/assets/images/optimized/rev-e6195f9/www.binarydefense.com/wp-content/uploads/2023/04/Picture5.jpg" class="nitro-lazy" decoding="async" nitro-lazy-empty id="NjU1OjE5NQ==-1" src="data:image/svg+xml;nitro-empty-id=NjU1OjE5NQ==-1;base64,PHN2ZyB2aWV3Qm94PSIwIDAgOTM2IDE4OCIgd2lkdGg9IjkzNiIgaGVpZ2h0PSIxODgiIHhtbG5zPSJodHRwOi8vd3d3LnczLm9yZy8yMDAwL3N2ZyI+PC9zdmc+" /><figcaption><em>For-Loop variables renamed and formatted for structure.</em></figcaption></figure></div> <p>For someone familiar with For-Loops, that is much more readable now. The remaining obfuscated variable matches the variable from Line 1 in the script. In the For-Loop, we appear to be indexing into and manipulating $fileTwo using “-bxor” and indexes of the obfuscated variable. (To use “bxor” is to use the XOR operator on the individual bytes. Explaining XOR is beyond my goal here. However, “bxor” is common in malicious PowerShell so the reader should take note.) So it appears that the variable, $ab13e4b80f240fb13f8a62c3a6db5, is a key for decoding $fileTwo before loading it in line 7. So let’s rename the variable to “$key”. Having made those changes, the script is much more readable.</p> <div class="wp-block-image"><figure class="aligncenter size-large"><img alt=" class=" wp-image-32108"/ nitro-lazy-src="https://cdn-dkajddp.nitrocdn.com/rHVGjbdIfsJHeZKoyaQvWfrAvKeGdBoq/assets/images/optimized/rev-e6195f9/www.binarydefense.com/wp-content/uploads/2023/04/Picture6-1024x247.jpg" class="nitro-lazy" decoding="async" nitro-lazy-empty id="NjYzOjIwNA==-1" src="data:image/svg+xml;nitro-empty-id=NjYzOjIwNA==-1;base64,PHN2ZyB2aWV3Qm94PSIwIDAgMTAyNCAyNDciIHdpZHRoPSIxMDI0IiBoZWlnaHQ9IjI0NyIgeG1sbnM9Imh0dHA6Ly93d3cudzMub3JnLzIwMDAvc3ZnIj48L3N2Zz4=" /><figcaption><em>Image of the Script with obfuscation removed</em></figcaption></figure></div> <p>Looking at the script now, we can see:</p> <ul class="wp-block-list"><li>The script uses “CMD /c” to execute a PowerShell command with the Window option of “hidden”. (Line 1)</li><li>It sets a variable as a key (Line 1)</li><li>It sets a variable for a second file (Line 3)</li><li>It decodes the file using the key and two For-Loops (Lines 5-11)</li><li>It uses [system.reflection.assembly] to load the file (Line 13)</li><li>And then “interacts” with the file it now calls “[mars.deimos]” (Line 13).</li></ul> <p>Having deobfuscated the script, let’s understand better what it is doing.</p> <p>Using Google and searching for things such as “System.Reflection.Assembly used in malware” returns many results explaining that this is a method of loading .NET executable into memory.</p> <p>Putting what we’ve learned together: the binary gets loaded into memory whenever this script is called. In order to make sure the script is called, an earlier stage modified desktop shortcuts to call this script. Great. Now let us find out what the executable does in order to understand risks and possible damages.</p> <h3 class="wp-block-heading"><strong>Writing the binary to file</strong></h3> <p>Initially, I struggled finding a method of writing this binary to disk. While [system.reflection.assembly] has a load function, it didn’t have any obvious method to save the assembly as a file rather than loading it into memory. The articles I found discussed how to load assemblies into memory and did not focus on analyzing assemblies that were loaded into memory. So, I want to document my solution here as well.</p> <p>In this situation, we have access to the script that loads it into memory, additionally, the script itself does the decoding of the second file in the For-Loops. With the second file decoded, we can actually write the file to disk with one line of PowerShell; that is, after the For-Loops, we can remove the remaining commands that load it into memory and use the following command to write it to disk:</p> <p>Set-Content .backdoor.bin -Value $fileTwo -Encoding Byte</p> <p>With that, we save the decoded file as “backdoor.bin” in our current working directory. As a Malware Analyst our remaining task is to investigate the functionality of the binary. That will be in the second part of this blog post.</p> <h3 class="wp-block-heading">Alternative Analysis Tips</h3> <p>In addition to what is written above, I completed additional analysis before writing the binary to file. I want to document those activities as they may result in a quicker analysis than full script and binary analysis when a quicker analysis is needed by administrators or analysts.</p> <p>As mentioned above, the binary is injected into memory by the script and is not written to file. These are the implications of this:</p> <ul class="wp-block-list"><li>If you do not have a method of flagging PowerShell execution or Memory Injection, the injected binary will go undetected,</li><li>No binary will be written to disk for static analysis. You will not have anything to upload to a service such as VirusTotal for quick verification and the scripts will not match anything on VirusTotal as they are custom in order to point to your host’s user directory.</li><li>You will need to preform some type of dynamic analysis in order to identify the malware and activity that the malware is performing.</li></ul> <h3 class="wp-block-heading">Dynamic Analysis</h3> <p>For anyone unfamiliar, dynamic analysis refers to analyzing malware by running it. We will run the script in a virtual machine disconnected from the internet and see what happens when it runs.</p> <p>Two easy tools for this are included in the&nbsp;<a href="https://docs.microsoft.com/en-us/sysinternals/downloads/sysinternals-suite">Sysinternals Suite</a>: Process Monitor (hereafter Procmon) and Process Explorer. Procmon monitors and logs registry, file, process, and thread activity. Process Explorer displays information on open Processes and what threads and processes are opened by each process.</p> <p>For running the malware, we are simply going to use the files provided for us by the Threat Actor: we will drop the script and encoded file onto our virtual machine and modify the Base64 in the script to point to the encoded file on our virtual machine. (We can do that by decoding the Base64 in CyberChef, modifying the path, encoding the path to Base64 with CyberChef, and then putting the encoded path back into the script.).</p> <p>With ProcessExplorer open and Procmon capturing events, we can run the script and identify what process ID it runs with. I ran the script and saw a new “cmd.exe” process with a child process of “powershell.exe”. By hovering my mouse over it, I confirmed that the command line to start this process was our malicious script. I can then see that the Process ID was 3888.</p> <div class="wp-block-image"><figure class="aligncenter size-large"><img alt=" class=" wp-image-32109"/ nitro-lazy-src="https://cdn-dkajddp.nitrocdn.com/rHVGjbdIfsJHeZKoyaQvWfrAvKeGdBoq/assets/images/optimized/rev-e6195f9/www.binarydefense.com/wp-content/uploads/2023/04/Picture7-1024x755.jpg" class="nitro-lazy" decoding="async" nitro-lazy-empty id="NzQzOjIwNA==-1" src="data:image/svg+xml;nitro-empty-id=NzQzOjIwNA==-1;base64,PHN2ZyB2aWV3Qm94PSIwIDAgMTAyNCA3NTUiIHdpZHRoPSIxMDI0IiBoZWlnaHQ9Ijc1NSIgeG1sbnM9Imh0dHA6Ly93d3cudzMub3JnLzIwMDAvc3ZnIj48L3N2Zz4=" /><figcaption><em>An image of Process Explorer with the malicious script executed and the mouse hovering over the PowerShell process.</em></figcaption></figure></div> <p>I stopped Procmon capturing events and filtered Procmon to only include entries for Process ID 3888. After a few seconds and it still logged 13,000 events related to process. A majority of the events are common PowerShell events that aren’t important to analyzing our malware. To be honest, sifting through it is not an easy task.</p> <p>In this instance, we are actually able to use the “Find” feature (Ctrl+F) to look for the name of the 2nd file. I’ve filtered out some other operations, but what we find is that shortly after it access that second file, the process writes a file to disk called “solarmarker.dat”.</p> <div class="wp-block-image"><figure class="aligncenter size-large"><img alt=" class=" wp-image-32110"/ nitro-lazy-src="https://cdn-dkajddp.nitrocdn.com/rHVGjbdIfsJHeZKoyaQvWfrAvKeGdBoq/assets/images/optimized/rev-e6195f9/www.binarydefense.com/wp-content/uploads/2023/04/Picture8-1024x241.jpg" class="nitro-lazy" decoding="async" nitro-lazy-empty id="NzU1OjIwNA==-1" src="data:image/svg+xml;nitro-empty-id=NzU1OjIwNA==-1;base64,PHN2ZyB2aWV3Qm94PSIwIDAgMTAyNCAyNDEiIHdpZHRoPSIxMDI0IiBoZWlnaHQ9IjI0MSIgeG1sbnM9Imh0dHA6Ly93d3cudzMub3JnLzIwMDAvc3ZnIj48L3N2Zz4=" /><figcaption><em>An image with Procmon filtered and the solarmarker.dat WriteFile operation revealed.</em></figcaption></figure></div> <p>The file is not intelligible if we open it, but this file a high value finding. It is a high value finding because the malware is not using random strings anymore: the malware created a file with a static name “solarmarker.dat”.</p> <div class="wp-block-image"><figure class="aligncenter size-large"><img alt=" class=" wp-image-32111"/ nitro-lazy-src="https://cdn-dkajddp.nitrocdn.com/rHVGjbdIfsJHeZKoyaQvWfrAvKeGdBoq/assets/images/optimized/rev-e6195f9/www.binarydefense.com/wp-content/uploads/2023/04/Picture9.jpg" class="nitro-lazy" decoding="async" nitro-lazy-empty id="NzYzOjE5NQ==-1" src="data:image/svg+xml;nitro-empty-id=NzYzOjE5NQ==-1;base64,PHN2ZyB2aWV3Qm94PSIwIDAgNTk0IDE4MiIgd2lkdGg9IjU5NCIgaGVpZ2h0PSIxODIiIHhtbG5zPSJodHRwOi8vd3d3LnczLm9yZy8yMDAwL3N2ZyI+PC9zdmc+" /><figcaption><em>An image of the contents of solarmarker.dat</em></figcaption></figure></div> <p>With that name, we are able to Google and find anything that has been published about the malware. With that, I was able to identify quickly what the malware was and the possible behaviors that may have been executed on the system. From this published information, as an administrator, I have the most important information I need:</p> <ul class="wp-block-list"><li>The solarmarker.dat is documented as a Host Identifier sent to the threat actor</li><li>Any hosts with this file in that directory have been compromised with this malware and this malware may steal passwords</li></ul> <p>We can then take the appropriate action for our organization: removing the malware on compromised hosts and resetting passwords as deemed necessary.</p> <h3 class="wp-block-heading">Conclusion</h3> <p>In this Part 1, we’ve analyzed the malware script to understand what the script does, we have also done dynamic analysis in the event that we cannot understand the script. The dynamic analysis gave us an important string allowing us to act fast on confirming an infection and gave us a direction for remediating the malware.</p> <p>In Part 2, we will look at the binary
