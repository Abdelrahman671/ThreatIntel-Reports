    <div class="node-full__body blog-content__body">  <h2>Overview</h2>

<p>For several years, Proofpoint researchers have been tracking the use of first-stage downloaders, which are used by threat actors to install other forms of malware during and after their malicious email campaigns. In particular, over the last two years, these downloaders have become increasingly robust, providing advanced profiling and targeting capabilities.</p>

<p>More importantly, downloaders and other malware like botnets and banking Trojans have displaced ransomware as primary payloads, giving threat actors the flexibility to deploy a range of malware in secondary infections. For example, one of the most prevalent, Smoke Loader, has been used extensively to drop payloads such as Ursnif and The Trick banking Trojans, as well as using its own modules for credential and other information and data-stealing, among other malicious functions.</p>

<p>Since late August 2019, Proofpoint researchers have been tracking the development and sale of a new modular loader named Buer by its authors. Buer has features that are highly competitive with Smoke Loader, is being actively sold in prominent underground marketplaces, and is intended for use actors seeking a turn-key, off-the-shelf solution.</p>

<h2>Campaigns</h2>

<h3>August 28, 2019</h3>

<p>On August 28, Proofpoint researchers observed malicious email messages that appear to reply to earlier legitimate email conversations. They contained Microsoft Word attachments that use Microsoft Office macros to download the next stage payload.</p>

<p><img src="/sites/default/files/buerpicture1.png"></p>

<p><em>Figure 1: Example Microsoft Word attachment used in the August 28, 2019, campaign</em></p>

<p>We observed the next-stage payload being downloaded from URLs including:</p>

<p><strong>hxxp://jf8df87sdfd.yesteryearrestorations[.]net/gate.php</strong></p>

<p><strong>hxxp://93345fdd.libertycolegios[.]com/gate.php</strong></p>

<p>The dropped payload was named <strong>verinstere222.xls</strong> or <strong>verinstere33.exe</strong> (a naming convention that the actor used during that period). Instead of the <a href="https://www.proofpoint.com/us/threat-insight/post/ursnif-variant-dreambot-adds-tor-functionality">Dreambot variant of Ursnif</a> frequently associated with this actor, the payload was an undocumented loader not previously observed in the wild.</p>

<p>In the following weeks over September and October, Proofpoint researchers and other members of the infosec community [1] observed several campaigns from the same actor dropping either the Dreambot variant of Ursnif or this new loader.</p>

<h3>October 10, 2019</h3>

<p>On October 10, Proofpoint researchers observed a malvertising campaign in Australia redirecting to the Fallout Exploit Kit (EK) dropping the new loader.</p>

<p><img src="/sites/default/files/buerpicture2.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 2: HTTP network traffic trace with the Fallout EK exploiting vulnerable browsers </em></p>

<p>The loader then dropped several second-stage malware payloads including KPOT stealer, Amadey, and Smoke Loader.</p>

<h3>October 21, 2019</h3>

<p>Since the beginning of July, Proofpoint researchers observed approximately 100 campaigns involving Ostap [2] almost exclusively loading several instances of The Trick. On the 21, however, Proofpoint researchers observed malicious email messages with subject lines such as <strong>Penalty Notice # PKJWVBP</strong> containing Microsoft Word attachments. The documents contained macros that, if enabled, would execute Ostap. We observed Ostap downloading this loader from</p>

<p><strong>hxxps://185.130.104[.]187/nana/kum.php?pi=18b&amp;[redacted]</strong></p>

<p>which in turn loaded The Trick ono22 from its C&amp;C: <strong>garrisontx[.]us</strong></p>

<p><img src="/sites/default/files/buerpicture3.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 3: Network traffic observed once the macro in the malicious documents is enabled.</em></p>

<p><strong><img src="/sites/default/files/buerpicture4.png" style="border-width: 0px; border-style: solid;"></strong></p>

<p><em>Figure 4: Example Microsoft Word attachment used in the October 21 campaign</em></p>

<h2>Marketplace &amp; Feature Analysis</h2>

<p>Because we began observing this new loader in use in multiple, distinct campaigns, we expected that it was being sold in an underground marketplace to multiple actors. Moreover, we discovered an advertisement from August 16 on an underground forum describing a loader named Buer that matched the functionality of the malware observed in the above campaigns.</p>

<p>The features added and advertised in the following weeks match exactly with the evolution of the loader found in these campaigns.</p>

<p>We retrieved text from a bulletin board posting by the author, in Russian, requesting a payment of $400 for the malware, and offering their services to set up the software for prospective customers in order to get it up and running. The author also notes that updates and bug fixes are free of charge, but there is a $25 surcharge for rebuilding to new addresses.</p>

<p>The following text, which Proofpoint also extracted from the underground marketplace, and is presumed to be written by the author of the malware, is a summary of the functionality of the loader as described in the original Russian:</p>

<p><img src="/sites/default/files/buerpicture5.png" style="border-width: 1px; border-style: solid;"></p>

<p><em>Figure 5. Text from underground forum post describing Buer Loader bot functionality</em></p>

<p>Similarly, the advertisement also lists control panel functionality. The author notes that the modular bot is written entirely in C, using a control panel written in .NET Core, emphasizing higher performance in both the client and server due to the choice of programming language.</p>

<ul>
	<li>As per the description, the bot has a total payload of 55 to 60 kilobytes, functions as a native Windows executable and dynamic link library, runs entirely in resident memory, and is compatible with 32-bit and 64-bit Microsoft Windows operating systems.</li>
	<li>The bot communicates over an HTTPS connection and can be updated remotely from the control panel after the decrypt as well as the rebuild.</li>
	<li>The author also notes that the loader runs as a surrogate process of a trusted application, and functions using User level privileges.</li>
	<li>Most notably, the software will not run in the CIS (former Soviet states, such as Russia).</li>
</ul>

<p>The ad describes the following features for the server and control panel:</p>

<ul>
	<li>The control panel is advertised as also being written in .NET Core, noting easy installation on Ubuntu / Debian Linux server systems.</li>
	<li>The server provides a wide range of statistics, including counters for online, living, dead, and total bots; a real-time update for the list of bots; a file download counter; and an ability to filter systems by type of operating system, access rights of installed bots, and number of logical CPU cores.</li>
	<li>Downloaded files from the infected systems are stored in encrypted form on the server, with access granted by a token.</li>
	<li>Most importantly, like the bots themselves, the author notes that the server does not process API requests sent from within CIS-member countries.</li>
</ul>

<p>The forum post also included technical release notes for the Buer loader and control panel (version 1.1.2). In the introduction, the author noted that launching the loader now consists of three steps -- if the first two steps are unsuccessful on the infected system, and the injection into the surrogate process fails (for example, due to incompatibility with the crypt itself), the loader will execute under its own process instead.</p>

<p>The release notes call out the following for the loader:</p>

<ul>
	<li>The loader uses a <a href="https://www.proofpoint.com/us/threat-insight/post/sandiflux-another-fast-flux-infrastructure-used-malware-distribution-emerges">FastFlux</a> architecture.</li>
	<li>The loader works from under a trusted process within Microsoft Windows. The MemLoadEx process now supports x64 [.]exe as a trusted application.</li>
	<li>MemLoad has been updated and now supports native x32 [.]exe.</li>
</ul>

<p>The release notes call out the following features for the control panel:</p>

<ul>
	<li>API access is accomplished using HTTPS with support for self--signed certificates.</li>
	<li>Support for editing tasks in the panel. The user can stop the task during execution and change the payload and the number of executions.</li>
	<li>Added the ability to create a task by bot ID. Very suitable for point loads.</li>
	<li>A step-by-step window for creating tasks.</li>
	<li>A notification that allows you to learn about the necessary bots online.</li>
	<li>The uniqueness of the bot ID has been increased.</li>
	<li>Tags have been added to the panel, allowing sorting bots for subsequent actions with them.</li>
	<li>Displays the computer name in the table.</li>
	<li>Improved crypto compatibility.</li>
	<li>Added bot history.</li>
	<li>The panel now expands to Docker (Docker container support).
	<ul>
		<li><strong>Proofpoint Researcher Note:</strong> We presume this feature is for ease of integration into leased Docker hosts, simplifying installation, although potentially the panel/C&amp;C <a href="https://www.zdnet.com/article/a-hacking-group-is-hijacking-docker-systems-with-exposed-api-endpoints/?ftag=TRE49e8aa0&amp;bhid=22108669915048766929494654526268">could be installed on a compromised Docker host.</a></li>
	</ul>
	</li>
	<li>Validation on the file on the panel. Now the panel will not miss the file that the loader will not be able to download and will notify the client about this.</li>
	<li>Tasks can now be repeated.</li>
</ul>

<p>Finally, the author described the following technical changes for version 1.1.9. These are noteworthy as they demonstrate that the malware is under active, professional development.</p>

<ul>
	<li>The loader has acquired a new method for launching External for local files. The advantages of the method are uniqueness and no CreateProcess / ShellExecute through the loader. The launch produces a trusted process without any commands to it.</li>
	<li>The panel has the ability to tag all bots that have performed a specific task. This will allow the user to distribute the payload to certain groups of bots.</li>
	<li>Implemented integration API. Available documentation for it.</li>
	<li>Added the ability to send a file by reference in proxy mode. The file is transferred to the bot in encrypted form.</li>
	<li>The bug of counting bots by country has been fixed and other improvements have been added.</li>
</ul>

<h2>Malware Analysis</h2>

<p>Buer Loader is a new downloader malware that downloads and executes additional payloads.</p>

<h3>Anti-analysis features</h3>

<p>The loader contains some basic anti-analysis functionality:</p>

<ul>
	<li>Checks for debuggers by inspecting the NtGlobalFlag in the Process Environment Block (PEB) and Thread Environment Block (TEB)</li>
	<li>Checks for virtual machines using the Red Pill [4], No Pill [5], and related mechanisms</li>
	<li>Checks locale to make sure the malware is not running in specific countries (Figure 14)</li>
</ul>

<p><img src="/sites/default/files/buerpicture14.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 14: Malware check to make sure it is not running in specific countries</em></p>

<h3>Persistence</h3>

<p>Persistence is set up by configuring a Registry RunOnce entry. Depending on the version, the registry entry will execute the malware directly or schedule a task to execute it.</p>

<h3>Encrypted Strings</h3>

<p>This sample contains a function to encrypt strings.</p>

<p><img src="/sites/default/files/buerpicture15.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 15: Decryption sequence for strings</em></p>

<p>The following function is an example of how to decrypt the encrypted strings in Ghidra using Jython:</p>

<p><img src="/sites/default/files/buerpicture16.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 16: Decryption sequence for strings (Python version)</em></p>

<p><img src="/sites/default/files/buerpicture17.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 17: Example string decryptions</em></p>

<h3>Windows API Calls</h3>

<p>This sample uses a hashing algorithm to resolve most of its Windows API calls. The hashing algorithm ensures each character of the API name is a capital letter. It then rotates right (ROR) each character by 13 and adds them together.</p>

<p><img src="/sites/default/files/buerpicture18.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 18: Hashing algorithm to resolve Windows API calls</em></p>

<p>The following function is an example of how Python can be used to help resolve the API calls.</p>

<p><img src="/sites/default/files/buerpicture19.png" style="border-width: 0px; border-style: solid;"></p>

<h3>Command and Control</h3>

<p>Command and control (C&amp;C) functions are handled via HTTP(S) GET requests. An example command beacon looks like Figure 20:</p>

<p><img src="/sites/default/files/buerpicture20.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 20: Example command beacon</em></p>

<p>These requests go to the update API and contain an encrypted parameter. This parameter can be decrypted by:</p>

<ol>
	<li>Base64 decoding</li>
	<li>Hex decoding</li>
	<li>RC4 decryption (the key used in the analyzed samples was CRYPTO_KEY)</li>
</ol>

<p>An example of the plaintext parameter is:</p>

<p><strong><em>88a5e68a2047fa5ebdc095a8500d8fae565a6b225ce94956e194b4a0e8a515ae|ab21d61b35a8d1dc4ffb3cc4b75094c31b8c00de3ffaaa17ce1ad15e876dbd1f|Windows 7|x64|4|Admin|RFEZOWGZPBYYOI</em></strong></p>

<p>It contains pipe-delimited data consisting of:</p>

<ul>
	<li>Bot ID (SHA-256 hex digest of various system parameters such as hardware profile GUID and name, computer name, volume serial number, and CPUID)</li>
	<li>An SHA-256 hash of its own executable image</li>
	<li>Windows version</li>
	<li>Architecture type</li>
	<li>Number of processors</li>
	<li>User privileges</li>
	<li>Computer name</li>
</ul>

<p>An example command beacon response is shown in Figure 21:</p>

<p><img src="/sites/default/files/buerpicture21.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 21: Example command beacon response</em></p>

<p>It can be decrypted similarly to the request parameter above, except that the hex-encoded bytes are separated by dash characters. An example plaintext response is shown in&nbsp; Figure 22:</p>

<p><img src="/sites/default/files/buerpicture22.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 22: Plaintext command beacon response</em></p>

<p>The decrypted text is a JSON object containing various options on how to download and execute a payload:</p>

<ul>
	<li>type - there are two types:
	<ul>
		<li>update - update self</li>
		<li>download_and_exec - download and execute</li>
	</ul>
	</li>
	<li>options - specifies options for the payload to download:
	<ul>
		<li>Hash - only applicable to update type to determine whether a new update is available</li>
		<li>x64 - whether the payload is 64-bit</li>
		<li>FileType - not used in analyzed samples</li>
		<li>AssemblyType - not used in analyzed samples</li>
		<li>AccessToken - used to download the payload (see below)</li>
		<li>External - indicates whether the payload is downloaded from the C&amp;C or an external URL</li>
	</ul>
	</li>
	<li>method - method of execution:
	<ul>
		<li>exelocal - create process</li>
		<li>memload - inject and manually load payload</li>
		<li>memloadex - inject and manually load payload</li>
		<li>loaddllmem - inject and manually load payload</li>
	</ul>
	</li>
	<li>parameters - parameters to pass on the command line</li>
	<li>pathToDrop - not used in analyzed samples</li>
	<li>autorun - indicates whether to setup Registry RunOnce persistence for the payload</li>
	<li>modules - see Modules section below</li>
	<li>timeout - not used in analyzed samples</li>
</ul>

<p>Payloads downloaded from the C&amp;C server are done via requests to the <strong>download API </strong>as seen in Figure 23:</p>

<p><img src="/sites/default/files/buerpicture23.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 23: Downloading payload from C&amp;C</em></p>

<p>An example of the plaintext request parameter is shown below:</p>

<p><strong>88a5e68a2047fa5ebdc095a8500d8fae565a6b225ce94956e194b4a0e8a515ae|58007044-67d4-4963-9f5f-400dfbc69e74</strong></p>

<p>It contains the bots ID and <strong>AccessToken</strong> from the command beacon response. If the payload is downloaded from the C&amp;C, it is encrypted with RC4. In the analyzed samples the key was <strong>CRYPTO_KEY.</strong></p>

<h3>Modules</h3>

<p>The command beacon response contains a modules list. Proofpoint researchers have not observed Buer modules being used in the wild yet, but based on the code this list will contain module AccessTokens. The module file name is queried by sending an AccessToken to the <strong>module API </strong>of the C&amp;C. The module will then be downloaded using the <strong>downloadmodule API.</strong> Once downloaded and decrypted, it is loaded using the <strong>loaddllmem</strong> method.</p>

<h2>Conclusion</h2>

<p>A new downloader, Buer, has appeared recently in a variety of campaigns, via malvertising leading to exploit kits; as a secondary payload via Ostap; and as a primary payload downloading malware such as The Trick banking Trojan.</p>

<p>The new loader has robust geotargeting, system profiling, and anti-analysis features and is currently being marketed on underground forums with value-added setup services. The Russian-speaking author(s) is actively developing the downloader with sophisticated control panels and a rich feature set, making the malware competitive in underground markets.</p>

<p>The downloader is written in C while the control panel is written in .NET core, indicating optimization for performance and small download footprint, as well as the ability to easily install the control panel on Linux servers -- built-in support for Docker containers will further facilitate its proliferation on rented hosts used for malicious purposes, and potentially, compromised hosts as well. The latter capability is included in its advertised features and release notes.</p>
