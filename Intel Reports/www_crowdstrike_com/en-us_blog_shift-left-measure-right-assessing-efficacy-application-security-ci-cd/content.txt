        <h1>Shift Left, Measure Right: Assessing the Efficacy of Application Security in the Age of CI/CD</h1>
        
        <div class="publish_info">
            <p>November 21, 2024</p>
            <span>|</span>
            <a href="/en-us/blog/author.jamie-gale/" title="Posts by Jamie Gale" rel="author">
                Jamie Gale</a>
            <span>|</span>
                
            <a href="/en-us/blog/category.cloud-security/" title="Cloud &amp; Application Security">Cloud &amp; Application Security</a>
        </div>
        <div class="post_image">

<div class="cmp-text" id="text-987e2b5be8" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span>As organizations continue to invest in developing cloud-native applications, they must also invest in securing them. Industry data shows application security is a growing market with investments predicted to reach $13.57 billion USD by 2029, according to <a href="https://www.statista.com/outlook/tmo/cybersecurity/cyber-solutions/application-security/worldwide#revenue">Statista Market Insights</a>.</span>
    
</div>


    

</div>
<div class="image">
<div data-cmp-is="image" data-asset-id="6a06d5b4-dcca-4246-989d-0d841ea9f441" data-title="Figure 1. Investments in application security predicted to climb to $13.57 billion USD by 2029 (Source: Statista Market Insights) "/>

    </div>

<div class="text text--blog-content">

  

<div class="cmp-text" id="text-104472516f" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><p>But application security is a tricky process to navigate for many DevOps and DevSecOps teams. Today’s applications are constantly evolving with new features and updates, continuously introducing the possibility of vulnerabilities and misconfigurations that could heighten risk. Further, organizations navigating the transition from DevOps to DevSecOps may lack the metrics needed to effectively track and measure their application security posture.</p>
<p>Here we discuss the challenges in securing cloud-native applications and identify ways to measure the efficacy of an application security strategy in reducing risk. </p>
<h2>Security Starts Left but Doesn’t End There</h2>
<p>Integrating security early and frequently in the software development lifecycle and CI/CD pipeline — a process known as &#34;shifting security left&#34; — is an application security best practice. However, modern applications constantly evolve and frequently receive updates after they’re deployed. According to the <a href="https://www.crowdstrike.com/2024-state-of-application-security-report/">CrowdStrike 2024 State of Application Security Report</a>, 71% of organizations update applications weekly, and 19% do so multiple times a day.</p>
<p>Application security teams must continually review application code changes as they are developed and deployed to confirm they are free of known vulnerabilities, misconfigurations and other security problems. To accomplish this, application security teams use a variety of scanning and testing tools to uncover potential issues in the application code, configuration or runtime execution. </p>
<p>While these tools are essential for secure development, there must be a balance across the entire lifecycle of the application. Pre-production security tools can either miss issues altogether or generate an abundance of false positives. For example, software composition analysis (SCA) tools find vulnerabilities in open source libraries but are unable to identify vulnerabilities in custom code. Static application security testing (SAST) is valuable for analyzing custom code but cannot detect runtime configuration issues and business logic errors. Conversely, dynamic application security testing focuses on discovering vulnerabilities that occur during execution, such as injection attacks or misconfigurations, but it does not have visibility into the source code, meaning it may miss vulnerabilities like hardcoded credentials, logic errors or insecure coding practices that are only detectable by SAST.</p>
<p>Measuring the success of an application security strategy shouldn&#39;t focus solely on the number of detected vulnerabilities or security issues, or on remediation speed. Rather, success should be measured by managing risk and prioritizing the most impactful issues.</p>
<h2>Moving Beyond DevOps KPIs</h2>
<p>DevOps, initially focused on accelerating software development and delivery, has for many organizations evolved into DevSecOps, which is responsible for integrating security into the CI/CD process. </p>
<p>As DevOps evolves into DevSecOps, the metrics used to track and measure success must also change. Google’s DevOps research assessment (<a href="https://dora.dev/">DORA</a>) team tracks the following five metrics to determine DevOps success and maturity: deployment frequency, lead time for code changes, mean time to recovery, change failure rate, and reliability. While many industry leaders use DORA metrics to measure <a href="https://cloud.google.com/blog/products/devops-sre/announcing-the-2023-state-of-devops-report">DevOps success</a>, these tend to focus on the volume of vulnerabilities and the speed of resolution rather than the efficiency and effectiveness of resolving security risk. They lack the specific application security metrics necessary for DevSecOps.</p>
<p>There are several additional metrics DevSecOps teams must consider when measuring the security of their applications.</p>
<h2>Making Sense of Application Security and CI/CD</h2>
<p>To measure application security effectiveness, teams must identify the processes and tools embedded in their CI/CD pipeline, then evaluate the coverage and gaps spanning stages from pre-production to production.</p>
</span>
    
</div>


    

</div>
<div class="image">
<div data-cmp-is="image" data-asset-id="af9b119e-342f-4ee7-8847-d4fc95a13380" data-title="Figure 2. Application security across CI/CD"/>

<div class="text text--blog-content">

  

<div class="cmp-text" id="text-55011bf1d6" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><p>The next step is to find the KPIs to measure the effectiveness of your application security across the CI/CD pipeline. Below we highlight some KPIs that can help measure the impact that application security tools and processes have on risk. </p>
<p>In the following examples, a critical risk is any threat that is exploitable and has access to sensitive data. </p>
<table cellpadding="1" cellspacing="0" border="1"><caption>Table 1. Application security KPIs (pre-production)</caption><tbody><tr class="top-row bg-orange"><th colspan="3" style="	text-align: center;
">Pre-Production</th></tr><tr><td><b>KPI</b></td><td><b>Description</b></td><td><b>Purpose</b></td></tr><tr><td>Critical Risk Volume</td><td># of critical risks detected in pre-production</td><td>To understand the volume of critical risks found in pre-production (required for the Critical Risk Escape Rate KPI)</td></tr><tr><td>Critical Risk Remediation Queue</td><td># of critical risk tickets</td><td>To understand the stress levels and workload placed on the developers who need to fix these issues</td></tr><tr><td>Known Risk Acceptance</td><td># of known risks not remediated (accepted) and pushed to production with reason</td><td>To understand how many risks are accepted</td></tr></tbody></table>
<table cellpadding="1" cellspacing="0" border="1"><caption>Table 2. Application security KPIs (production)</caption><tbody><tr class="top-row bg-orange"><th colspan="3" style="	text-align: center;
">Production</th></tr><tr><td><b>KPI</b></td><td><b>Description</b></td><td><b>Purpose</b></td></tr><tr><td>Critical Risk Volume</td><td># of critical risks detected in deployed apps</td><td>To compare to the volume of critical risks found in pre-production app security (required for the Critical Risk Escape Rate KPI)</td></tr><tr><td>Critical Risk Escape Rate</td><td>Ratio: # of critical risks found in pre-production to # found in production</td><td>To assess effectiveness of pre-production security</td></tr><tr><td>Critical Risks Per Deployment</td><td>Average # of critical risks for each deployment</td><td>To benchmark expected critical risks</td></tr><tr><td>Mean Time Between Critical Risks</td><td>Average time between critical risk detection</td><td>To assess critical risk prevention/reduction</td></tr></tbody></table>
<p>Tracking these metrics provides valuable insights into the effectiveness of application security.</p>
<h2>Assess Qualitative Metrics Focused on Teams</h2>
<p>DevSecOps is both a technical and cultural endeavor, so it’s important to measure the sentiment of key stakeholders. Here are some questions to consider.</p>
<p><b>Developer Experience </b></p>
<ul><li><p>Do you have the information you need to fix security bugs in code efficiently?</p>
</li></ul>
<p><b>Security Experience </b></p>
<ul><li>How much time do you spend on manual security reviews and checklists?</li><li>Do you have time to take on proactive security projects? </li></ul>
<p><b>SRE/Operations Experience </b></p>
<ul><li><p>Do you have the necessary tools and processes to quickly pinpoint the root cause of an outage and restore service? </p>
</li></ul>
<p><b>CISO Experience </b></p>
<ul><li>How secure are your applications today?</li><li>In the event of a breach related to an application, do your teams have the information and tools they need to respond quickly?</li></ul>
<h2>Closing the Gap with Falcon ASPM</h2>
<p>The majority of today’s application security tools and processes take place before an application is deployed. Early inclusion of security, or shifting security left, will always be a best practice, but it’s not enough. Application security is essential for every phase of the development life cycle, including — and most importantly — production.</p>
<p>CrowdStrike Falcon® Cloud Security is helping organizations extend their cloud security to applications running in production with Falcon Application Security Posture Management (ASPM). Falcon ASPM is a risk-driven, agentless approach for securing complex, frequently changing applications built and run in the cloud. Falcon ASPM assesses risk in a unique way that helps teams identify the greatest threats in their applications. Here are some of the most important factors that drive Falcon ASPM risk scoring:</p>
<ol><li><p>Severity of the threat(s) in a given application microservice</p>
</li><li><p>Connections to sensitive or proprietary data </p>
</li><li><p>Exposure to the internet, third parties or other connections</p>
</li><li><p>Reachability (for code library vulnerabilities) of threats to the application</p>
</li></ol>
<p>With better risk scoring comes better prioritization. Instead of another application security tool that leaves you with hundreds of potentially critical issues to fix, Falcon ASPM automatically narrows down the top few real risks for you and gives your developers the information they need to fix those issues quickly.</p>
