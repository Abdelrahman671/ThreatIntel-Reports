These techniques expand beyond DNS tunneling only for command and control (C2) and virtual private network (VPN) purposes.\r\n\r\nMalicious actors occasionally employ DNS tunneling as a covert communications channel, because it can bypass conventional network firewalls. This allows C2 traffic and data exfiltration that can remain hidden from some traditional detection methods.\r\n\r\nHowever, we recently detected three recent campaigns using DNS tunneling for purposes outside of traditional C2 and VPN use: scanning and tracking. In scanning, adversaries employ DNS tunneling to scan a victim's network infrastructure and gather information useful for future attacks. In tracking, adversaries use DNS tunneling techniques to track delivery of malicious emails and monitor the use of Content Delivery Networks (CDN).\r\n\r\nThis article provides a detailed case study that reveals how adversaries have used DNS tunneling for both scanning and tracking. We aim to increase awareness of these new use cases and provide further insight that can help security professionals better protect their networks.\r\n\r\nWe have built a system to monitor for DNS tunneling, and this detection is embedded in our DNS Security solution. An example is illustrated below in Figure 1.\r\n\r\nA typical use case for DNS tunneling includes the following steps:\r\n\r\n \tAttackers first register a domain malicious[.]site and then establish a C2 server that uses DNS tunneling as a communication channel. Attackers have many options to set up this C2 channel, such as by abusing Cobalt Strike.\r\n \tAttackers can create, develop or acquire malware that communicates with the server as a client and send this malware to a compromised client machine.\r\n \tThe compromised machine is usually behind a firewall and cannot directly communicate with attackers\u2019 servers. However, the malware can encode the data into the subdomain of malicious[.]site and make a DNS query toward the DNS resolver, as shown in Figure 1.\r\n \tDue to the unique nature of tunneling fully qualified domain names (FQDNs), the DNS resolver cannot find corresponding records from its cache. As a result, the resolver will then conduct recursive DNS queries toward root nameservers, top-level domain (TLD) nameservers and attacker-controlled authoritative nameservers for this domain.\r\n \tAttackers can obtain the decoded data from DNS traffic and manipulate the DNS response to infiltrate malicious data to the client.\r\n\r\n[caption id=\"attachment_133496\" align=\"aligncenter\" width=\"700\"] Figure 1. An overview of data exfiltration and infiltration with DNS tunneling.[\/caption]\r\nHow Is DNS Tunneling Hidden?\r\nDNS tunneling is hidden due to three factors.\r\n\r\n \tTraditional firewalls can reject unauthorized traffic. However, DNS traffic over User Datagram Protocol (UDP) port 53 is ubiquitous and commonly allowed through firewalls and other network security measures.\r\n \tDNS tunneling is conducted via a logical channel between the compromised client and the attacker\u2019s server, with the implementation of DNS protocol. That means the client machine does not communicate with the attacker's server directly, adding another layer of obscurity.\r\n \tAttackers typically encode data sent during exfiltration and infiltration with their own customized methods, which disguises the data within seemingly legitimate DNS traffic.\r\n\r\nHow Do Adversaries Leverage DNS Tunneling?\r\nThe use of DNS tunneling for C2 purposes enables attackers to establish stealthy and resilient communication channels, facilitating malicious activities such as data exfiltration and infiltration. Well-known campaigns such as DarkHydrus, OilRig, xHunt, SUNBURST and Decoy Dog leverage DNS tunneling for C2.\r\n\r\nThe DNS types used by attackers include:\r\n\r\n \tIPv4 (A)\r\n \tIPv6 (AAAA)\r\n \tMail exchange (MX)\r\n \tCanonical name (CNAME)\r\n \tText (TXT) records\r\n\r\nSome VPN vendors also use DNS tunneling for legitimate purposes, such as bypassing firewalls to avoid internet censorship or network service charges.\r\n\r\nIn addition to C2 and VPN purposes, attackers can also use DNS tunneling for tracking and scanning, as we\u2019ve observed in recent tunneling campaigns.\r\n\r\n \tDNS tunneling for tracking:\r\n\r\n \tAttackers can track victims\u2019 activities with regard to spam, phishing or advertisement contents. They do so by delivering malicious domains to victims with their identity information encoded in subdomain payloads.\r\n\r\n\r\n \tDNS tunneling for scanning:\r\n\r\n \tAttackers can scan network infrastructure by encoding the IP address and timestamp in the tunneling payloads, with spoofed source IP addresses. Then, the attackers are able to discover open resolvers so that they can exploit resolver vulnerabilities to perform DNS attacks \u2013 which can lead to malicious redirection or denial of service.\r\n\r\n\r\n\r\nTo better understand these two new use cases, our next sections cover the campaigns we have discovered using DNS tunneling for tracking and for scanning.\r\nDNS Tunneling for Tracking\r\nTo track a victim's behavior in conventional C2 communications, a threat actor's malware embeds data from a user's actions in URLs that it transmits to a C2 server through web traffic. In DNS tunneling, attackers accomplish the same result by using subdomains in DNS traffic.\r\n\r\nIn this application of DNS tunneling, an attacker's malware embeds information on a specific user and that user's actions into a unique subdomain of a DNS query. This subdomain is the tunneling payload, and the DNS query for the FQDN uses an attacker-controlled domain.\r\n\r\nAn authoritative nameserver for the attacker-controlled domain receives the DNS query. This attacker-controlled nameserver stores all DNS queries for the domain. The unique subdomains and timestamps of these DNS queries provide a log of the victim's activity. This is not limited to a single victim, and attackers can use it to track multiple victims from their campaign.\r\nTrkCdn DNS Tunneling Campaign\r\nWe call this campaign \"TrkCdn\" due to the characteristics of the domain names used for its DNS tunneling. Based on our analysis, we believe the DNS tunneling technique used in the TrkCdn campaign is meant to track a victim's interaction with its email content. Our data indicates the attacker targeted 731 potential victims. This campaign used 75 IP addresses for nameservers, resolving 658 attacker-controlled domains.\r\n\r\nEach domain only uses a single nameserver IP address, while one nameserver IP address can serve up to 123 domains. These domains use the same DNS configurations and the same encoding method for their subdomains. The attacker registered all domains under [.]com or [.]info TLDs and set domain names by combining two or three root words, which is a practice attackers use to avoid domain generation algorithm (DGA) detection.\r\n\r\nA subset of these domains are as follows:\r\n\r\n \tsimitor[.]com\r\n \tvibnere[.]com\r\n \tedrefo[.]com\r\n \tpordasa[.]info\r\n \tvitrfar[.]info\r\n \tfrotel[.]info\r\n\r\nA list of these domains along with sample FQDNs, nameservers, nameserver IP addresses and registration dates are shown below in Table 1. Analysis of DNS traffic for simitor[.]com reveals how attackers could achieve this.\r\n\r\nHere, we only show the tracking-relevant DNS configurations used by this tunneling domain. 193.9.114[.]43 served as the same IP address for the root domain, nameservers and cdn.simitor[.]com. This behavior is a common indicator for tunneling domains, because attackers need to build a nameserver for themselves while also trying to reduce attack cost. Therefore, they typically use only a single IP address for both domain hosting and name server.\r\n\r\nAll *.trk.simitor[.]com are redirected to cdn.simitor[.]com via a wildcard DNS record as shown below.\r\nsimitor[.]com A 193.9.114[.]43\r\n\r\nns1.simitor[.]com A 193.9.114[.]43\r\n\r\nns2.simitor[.]com A 193.9.114[.]43\r\n\r\ncdn.simitor[.]com A 193.9.114[.]43\r\n\r\n*.trk.simitor[.]com CNAME cdn.simitor[.]com\r\nFor the TrkCdn campaign, MD5 hash values represent email addresses in the DNS traffic. These MD5 values are subdomains for the DNS queries of a tunneling payload. For example, an email address of unit42@not-a-real-domain[.]com has an MD5 value of 4e09ef9806fb9af448a5efcd60395815. Therefore, the FQDN of a DNS query for the tunneling payload would be 4e09ef9806fb9af448a5efcd60395815.trk.simitor[.]com.\r\n\r\nDNS queries for these FQDNs can act as a tracking mechanism for emails sent by the threat actor. For example, if a victim opens one of these emails, embedded content might automatically generate the DNS query, or a victim could click on a link within the email. However this happens, after an infected host generates a DNS query for the FQDN, the DNS resolver will contact the IP address for the authoritative nameserver of the FQDN. Due to its wildcard configuration, the victim's DNS resolver would obtain the following result:\r\n4e09ef9806fb9af448a5efcd60395815.trk.simitor[.]com. 3600 IN CNAME cdn.simitor[.]com.\r\n\r\ncdn.simitor[.]com. 555 IN A 193.9.114[.]43\r\nHence, even though the FQDNs vary across different targets, they are all forwarded to the same IP address used by cdn.simitor[.]com. This authoritative name server then returns a DNS result that leads to an attacker-controlled server that delivers attacker-controlled content. This content can include advertisements, spam or phishing contents.\r\n\r\nFor tracking purposes, attackers can query DNS logs from their authoritative nameservers and compare the payload with the hash values of the email addresses. This way, attackers can know when a specific victim opens one of their emails or clicks on a link, and they can monitor campaign performance.\r\n\r\nFor example, a graph showing the cumulative distribution function (CDF) of DNS queries for FQDNs from the TrkCdn campaign is shown below in Figure 2. This graph shows the total percentage of DNS queries for TrkCdn FQDNs from 0 to 30 days. The graph indicates that approximately 80% of victims view the campaign's emails only once, while an additional 10% view the messages again within approximately one week. Attackers can view this FQDN data from their authoritative nameservers in the same manner.\r\n\r\n[caption id=\"attachment_133498\" align=\"aligncenter\" width=\"600\"] Figure 2. The CDF of FQDN span days in the TrkCdn campaign.[\/caption]\r\nDomain Lifecycle\r\nBy investigating an older domain pordasa[.]info, we conclude that the TrkCdn domain lifecycle goes through four distinct phases. These four phases are as follows:\r\n\r\n \tIncubation phase (two to 12 weeks)\r\n\r\n \tAfter the domain registration, attackers only configure the DNS settings and do nothing else, attempting to avoid malicious newly registered domain detection.\r\n\r\n\r\n \tActive phase (two to three weeks)\r\n\r\n \tAttackers actively distribute thousands of FQDNs to the corresponding victims\u2019 email addresses.\r\n\r\n\r\n \tTracking phase (nine to 11 months)\r\n\r\n \tVictims query the FQDNs, while attackers track their behaviors by obtaining DNS logs.\r\n\r\n\r\n \tRetirement phase (one year after registration)\r\n\r\n \tAttackers typically stop updating the domain registration after one year.\r\n\r\n\r\n\r\nBelow, Figure 3 shows an example of this lifecycle for pordasa[.]info. An attacker used this domain for DNS tunneling-style tracking, originally registering it on Oct. 12, 2022.\r\n\r\n[caption id=\"attachment_133500\" align=\"aligncenter\" width=\"600\"] Figure 3. The lifecycle of the pordasa[.]info domain.[\/caption]\r\nTrkCdn Persistence\r\nUntil February 2024, we found adversaries using new IP addresses and registering new domains for their authoritative nameservers associated with TrkCdn activity. Attackers registered these domains between Oct. 19, 2020, and Jan. 2, 2024. We analyze the timeline of domain registration and the domain's first use across different IP addresses.\r\n\r\nFigure 4 tracks the use of TrkCdn domains associated with 49 IP addresses. As noted in Figure 4, the majority of IP addresses used for TrkCdn's authoritative nameservers lie within the 185.121.0[.]0\/16 or the 146.70.0[.]0\/16 subnets. This indicates that the threat actor behind TrkCdn tends to use specific hosting providers.\r\n\r\n[caption id=\"attachment_133502\" align=\"aligncenter\" width=\"900\"] Figure 4. The timeline of TrkCdn domain registration and usage across different IP addresses.[\/caption]\r\nSpamTracker DNS Tunneling Campaign\r\nOur second example is a campaign using DNS tunneling to track spam delivery. Because this campaign uses DNS tunneling for spam tracking, we have nicknamed this campaign \"SpamTracker.\"\r\n\r\nThis campaign uses a similar tracking mechanism as the TrkCdn campaign. This campaign is related to 44 tunneling domains that have an IP address of 35.75.233[.]210 for its authoritative nameservers.\r\n\r\nThese domains share the same DGA naming method and subdomain encoding method used by the TrkCdn campaign. Nameservers for the A records of these domains are hosted on IP addresses that fall into the 103.8.88[.]64\/27 subnet. This campaign originated from Japan, and most of the targets were part of educational institutions.\r\n\r\nThis campaign employs emails and website links to deliver spam and phishing content that covers the following subjects:\r\n\r\n \tFortune-telling services\r\n \tFake package delivery updates\r\n \tSecondary job offers\r\n \tLifetime free items\r\n\r\nFigure 5 shows an example of these emails. <p>This article presents a case study on new applications of domain name system (DNS) tunneling we have found in the wild. These techniques expand beyond DNS tunneling only for command and control (C2) and virtual private network (VPN) purposes.</p>

<h2><strong>DNS Tunneling</strong></h2>
<p><a href="https://www.paloaltonetworks.com/cyberpedia/what-is-dns-tunneling" target="_blank" rel="noopener">DNS tunneling</a> embeds information into DNS requests and responses in a manner that allows a compromised host to communicate through DNS traffic with a nameserver controlled by an attacker. An example is illustrated below in Figure 1.</p>
<p>A typical use case for DNS tunneling includes the following steps:</p>
<ul>
<li>Attackers first register a <span style="font-family: 'courier new', courier, monospace;">domain malicious[.]site</span> and then establish a C2 server that uses DNS tunneling as a communication channel. Attackers have many options to set up this C2 channel, such as by abusing <a href="https://unit42.paloaltonetworks.com/tag/cobalt-strike/" target="_blank" rel="noopener">Cobalt Strike</a>.</li>
<li>Attackers can create, develop or acquire malware that communicates with the server as a client and send this malware to a compromised client machine.</li>
<li>The compromised machine is usually behind a firewall and cannot directly communicate with attackers’ servers. However, the malware can encode the data into the subdomain of <span style="font-family: 'courier new', courier, monospace;">malicious[.]site</span> and make a DNS query toward the DNS resolver, as shown in Figure 1.</li>
<li>Due to the unique nature of tunneling fully qualified domain names (FQDNs), the DNS resolver cannot find corresponding records from its cache. As a result, the resolver will then conduct recursive DNS queries toward root nameservers, top-level domain (TLD) nameservers and attacker-controlled authoritative nameservers for this domain.</li>
<li>Attackers can obtain the decoded data from DNS traffic and manipulate the DNS response to infiltrate malicious data to the client.</li>
</ul>
<figure id="attachment_133496" aria-describedby="caption-attachment-133496" style="width: 700px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-133496" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2024/05/word-image-527205-133492-1.png" alt="Image 1 is a diagram of how data exfiltration and infiltration work with DNS tunneling. The client server communicates with the DNS resolver. Icon for read/write from the cache. From the DNS resolver the addresses communicate with the root name servers, the top level domain name servers and the authoritative name servers. " width="700" height="372" /><figcaption id="caption-attachment-133496" class="wp-caption-text">Figure 1. An overview of data exfiltration and infiltration with DNS tunneling.</figcaption></figure>
<h3><a id="post-133492-_9uzl7ckj1old"></a>How Is DNS Tunneling Hidden?</h3>
<p>DNS tunneling is hidden due to three factors.</p>
<ul>
<li>Traditional firewalls can reject unauthorized traffic. However, DNS traffic over User Datagram Protocol (UDP) port 53 is ubiquitous and commonly allowed through firewalls and other network security measures.</li>
<li>DNS tunneling is conducted via a logical channel between the compromised client and the attacker’s server, with the implementation of DNS protocol. That means the client machine does not communicate with the attacker's server directly, adding another layer of obscurity.</li>
<li>Attackers typically encode data sent during exfiltration and infiltration with their own customized methods, which disguises the data within seemingly legitimate DNS traffic.</li>
</ul>
<h3><a id="post-133492-_5zu7mpldg4ya"></a>How Do Adversaries Leverage DNS Tunneling?</h3>
<p>The use of DNS tunneling for C2 purposes enables attackers to establish stealthy and resilient communication channels, facilitating malicious activities such as data exfiltration and infiltration. Well-known campaigns such as <a href="https://unit42.paloaltonetworks.com/darkhydrus-delivers-new-trojan-that-can-use-google-drive-for-c2-communications/" target="_blank" rel="noopener">DarkHydrus</a>, <a href="https://unit42.paloaltonetworks.com/oilrig-novel-c2-channel-steganography/" target="_blank" rel="noopener">OilRig</a>, <a href="https://unit42.paloaltonetworks.com/xhunt-campaign-backdoors/" target="_blank" rel="noopener">xHunt</a>, <a href="https://unit42.paloaltonetworks.com/solarstorm-supply-chain-attack-timeline/" target="_blank" rel="noopener">SUNBURST</a> and <a href="https://www.infoblox.com/company/news-events/press-releases/decoy-dog-is-no-ordinary-pupy-infoblox-reveals-shift-in-malware-tactics-after-initial-discovery/" target="_blank" rel="noopener">Decoy Dog</a> leverage DNS tunneling for C2.</p>
<p>The DNS types used by attackers include:</p>
<ul>
<li>IPv4 (A)</li>
<li>IPv6 (AAAA)</li>
<li>Mail exchange (MX)</li>
<li>Canonical name (CNAME)</li>
<li>Text (TXT) records</li>
</ul>
<p>Some VPN vendors also use DNS tunneling for legitimate purposes, such as bypassing firewalls to avoid internet censorship or network service charges.</p>
<p>In addition to C2 and VPN purposes, attackers can also use DNS tunneling for tracking and scanning, as we’ve observed in recent tunneling campaigns.</p>
<ul>
<li>DNS tunneling for tracking:
<ul>
<li>Attackers can track victims’ activities with regard to spam, phishing or advertisement contents. They do so by delivering malicious domains to victims with their identity information encoded in subdomain payloads.</li>
</ul>
</li>
<li>DNS tunneling for scanning:
<ul>
<li>Attackers can scan network infrastructure by encoding the IP address and timestamp in the tunneling payloads, with spoofed source IP addresses. Then, the attackers are able to discover open resolvers so that they can exploit resolver vulnerabilities to perform DNS attacks – which can lead to malicious redirection or denial of service.</li>
</ul>
</li>
</ul>
<p>To better understand these two new use cases, our next sections cover the campaigns we have discovered using DNS tunneling for tracking and for scanning.</p>
<h2><a id="post-133492-_yj0yy7dg26gi"></a><strong>DNS Tunneling for Tracking</strong></h2>
<p>To track a victim's behavior in conventional C2 communications, a threat actor's malware embeds data from a user's actions in URLs that it transmits to a C2 server through web traffic. In DNS tunneling, attackers accomplish the same result by using subdomains in DNS traffic.</p>
<p>In this application of DNS tunneling, an attacker's malware embeds information on a specific user and that user's actions into a unique subdomain of a DNS query. This subdomain is the tunneling payload, and the DNS query for the FQDN uses an attacker-controlled domain.</p>
<p>An authoritative nameserver for the attacker-controlled domain receives the DNS query. This attacker-controlled nameserver stores all DNS queries for the domain. The unique subdomains and timestamps of these DNS queries provide a log of the victim's activity. This is not limited to a single victim, and attackers can use it to track multiple victims from their campaign.</p>
<h3><a id="post-133492-_lgyxeg31r22j"></a>TrkCdn DNS Tunneling Campaign</h3>
<p>We call this campaign "TrkCdn" due to the characteristics of the domain names used for its DNS tunneling. Based on our analysis, we believe the DNS tunneling technique used in the TrkCdn campaign is meant to track a victim's interaction with its email content. Our data indicates the attacker targeted 731 potential victims. This campaign used 75 IP addresses for nameservers, resolving 658 attacker-controlled domains.</p>
<p>Each domain only uses a single nameserver IP address, while one nameserver IP address can serve up to 123 domains. These domains use the same DNS configurations and the same encoding method for their subdomains. The attacker registered all domains under <span style="font-family: 'courier new', courier, monospace;">[.]com</span> or <span style="font-family: 'courier new', courier, monospace;">[.]info</span> TLDs and set domain names by combining two or three root words, which is a practice attackers use to avoid domain generation algorithm (DGA) detection.</p>

<p>Hence, even though the FQDNs vary across different targets, they are all forwarded to the same IP address used by <span style="font-family: 'courier new', courier, monospace;">cdn.simitor[.]com</span>. This authoritative name server then returns a DNS result that leads to an attacker-controlled server that delivers attacker-controlled content. This content can include advertisements, spam or phishing contents.</p>
<p>For tracking purposes, attackers can query DNS logs from their authoritative nameservers and compare the payload with the hash values of the email addresses. This way, attackers can know when a specific victim opens one of their emails or clicks on a link, and they can monitor campaign performance.</p>

Attackers can view this FQDN data from their authoritative nameservers in the same manner.</p>

<h3><a id="post-133492-_gxk7mubz9rv3"></a>Domain Lifecycle</h3>
<p>By investigating an older domain <span style="font-family: 'courier new', courier, monospace;">pordasa[.]info</span>, we conclude that the TrkCdn domain lifecycle goes through four distinct phases. These four phases are as follows:</p>
<ul>
<li>Incubation phase (two to 12 weeks)
<ul>
<li>After the domain registration, attackers only configure the DNS settings and do nothing else, attempting to avoid malicious newly registered domain detection.</li>
</ul>
</li>
<li>Active phase (two to three weeks)
<ul>
<li>Attackers actively distribute thousands of FQDNs to the corresponding victims’ email addresses.</li>
</ul>
</li>
<li>Tracking phase (nine to 11 months)
<ul>
<li>Victims query the FQDNs, while attackers track their behaviors by obtaining DNS logs.</li>
</ul>
</li>
<li>Retirement phase (one year after registration)
<ul>
<li>Attackers typically stop updating the domain registration after one year.</li>
</ul>
</li>
</ul>

<h2><a id="post-133492-_cn0fr1n73xd7"></a><strong>DNS Tunneling for Scanning</strong></h2>
<p>Network scanning, which seeks vulnerabilities within network infrastructures, is usually the first stage of cyberattacks. However, the use case of DNS tunneling for network scanning is understudied. As a result, uncovering the scanning applications of tunneling campaigns can help us prevent cyberattacks at an early stage, mitigating potential damage.</p>
<h3><a id="post-133492-_sr8qppgsss3i"></a>SecShow DNS Tunneling Campaign</h3>
<p>We found a new campaign in which threat actors leverage tunneling to periodically scan a victim's network infrastructure, and then they typically perform reflection attacks. Their malicious actions include the following:</p>
<ul>
<li>Seeking open resolvers</li>
<li>Testing resolver delays</li>
<li>Exploiting resolver vulnerabilities</li>
<li>Obtaining time-to-live (TTL) information.</li>
</ul>
<p>This campaign generally targets open resolvers. As a result, we find victims mainly come from education, high tech and government fields, where open resolvers are commonly found. This campaign contains three domains, leveraging various subdomains to achieve different network scanning.</p>
<p>We list these three domains along with examples of FQDNs, nameservers, nameserver IP addresses and registration times in Table 3. These domains share the nameserver IP address of <span style="font-family: 'courier new', courier, monospace;">202.112.47[.]45</span>. We named this campaign "SecShow" due to the domain names the attackers used.</p>

<h3><a id="post-133492-_2fbbhg4bng9v"></a>SecShow Tunneling Use</h3>
<p>SecShow uses different subdomain values for different scanning purposes. Here, we present four use cases to show how attackers scan the networks.</p>
<p><strong>Case 1:</strong> <span style="font-family: 'courier new', courier, monospace;">bc2874fb-1.c.secshow[.]net</span></p>
<p>In this FQDN, <span style="font-family: 'courier new', courier, monospace;">bc2874fb</span> is a hexadecimal encoding for IP address <span style="font-family: 'courier new', courier, monospace;">188.40.116[.]251</span> and <span style="font-family: 'courier new', courier, monospace;">-1</span> is a counter to make the FQDN unique while the nameserver domain is <span style="font-family: 'courier new', courier, monospace;">c.secshow[.]net</span>.</p>
<p>Attackers first spoof a random source IP address (e.g., <span style="font-family: 'courier new', courier, monospace;">188.40.116[.]251</span>) and make a DNS query to a candidate IP address for the encoded FQDN (<span style="font-family: 'courier new', courier, monospace;">bc2874fb-1.c.secshow[.]net</span>). Once the attackers’ authoritative nameserver (<span style="font-family: 'courier new', courier, monospace;">c.secshow[.]net</span>) receives a DNS query, they can obtain the incoming resolver’s IP address and the encoded source IP address used for this query.</p>
<p>Attackers repeat this process with different spoofed IP addresses and discover the open resolvers in the networks and the IP addresses that these open resolvers service. This can be the first step for DNS spoofing, DNS cache poisoning or DNS amplification attacks.</p>
<p><strong>Case 2:</strong> <span style="font-family: 'courier new', courier, monospace;">20240212190003.bailiwick.secshow[.]net</span></p>
<p>This FQDN type only appears every Monday at 19:00:03 UTC. The payload indicates a timestamp (e.g., on Feb. 12, 2024, at 19:00:03 UTC) that is the generation time for this FQDN.</p>
<p>Attackers spoof a source IP address and query this FQDN from a resolver IP address. Attackers can perform the following activities:</p>
<ul>
<li>Test the query delays for this resolver</li>
<li>Check whether their domain is blocked and the query is forwarded to a sinkhole</li>
<li>Exploit the vulnerabilities of this resolver</li>
</ul>

<p>In the above code, <span style="font-family: 'courier new', courier, monospace;">afusdnfysbsf[.]com</span> was a malicious domain that had been revoked. However, its record can still be cached by resolvers. Therefore, attackers might leverage some resolver’s cache vulnerabilities in older software versions (for example, <a href="https://kb.isc.org/docs/aa-00691" target="_blank" rel="noopener">CVE-2012-1033</a>) to prevent domain name revocation.</p>
<p><strong>Case 3:</strong> <span style="font-family: 'courier new', courier, monospace;">1-103-170-192-121-103-170-192-9.h.secshow[.]net</span></p>
<p>The payload starts with a counter padding of 1 followed by two IP addresses of <span style="font-family: 'courier new', courier, monospace;">103.170.192[.]121</span> and <span style="font-family: 'courier new', courier, monospace;">103.170.192[.]9</span> that are the spoofed source IP address and the resolver's destination IP address.</p>
<p>This FQDN type is similar to Case 1. However, the A record of this FQDN is a random IP address that varies along with query attempts, with a long TTL of 86400. This feature could be exploited to perform the following activities:</p>
<ul>
<li>DNS amplification distributed denial-of-service (DDoS) attacks</li>
<li>DNS cache poisoning attacks</li>
<li>Resource exhaustion attacks</li>
</ul>
<p><strong>Case 4:</strong> <span style="font-family: 'courier new', courier, monospace;">0-53ea2a3a-202401201-ans-dnssec.l-test.secdns[.]site</span></p>
<p>The payload contains a pre-padding of <span style="font-family: 'courier new', courier, monospace;">0</span> followed by a hex-encoded IP address (<span style="font-family: 'courier new', courier, monospace;">53ea2a3a</span>), a date (<span style="font-family: 'courier new', courier, monospace;">20240120</span>), and post-padding (<span style="font-family: 'courier new', courier, monospace;">1</span>). We observe that attackers use this type of FQDN to obtain the following information:</p>
<ul>
<li>Max/min TTL</li>
<li>Timeout</li>
<li>Query speed information</li>
</ul>
<p>These are useful for some DNS threats such as <a href="https://www.ndss-symposium.org/wp-content/uploads/2023/02/ndss2023_s5_paper.pdf" target="_blank" rel="noopener">Phoenix Domain [PDF]</a> and <a href="https://kb.isc.org/docs/aa-00691" target="_blank" rel="noopener">Ghost Domain Names</a>.</p>
<h3><a id="post-133492-_t2ppmxj00p8f"></a>Mitigation</h3>
<p>The DNS tunneling domains used in these campaigns can be detected by Palo Alto Networks firewall products. However, we also suggest the following measures to reduce the attack surface of DNS resolvers.</p>
<ul>
<li>Control the service range of resolvers to accept necessary queries only</li>
<li>Promptly update the resolver software version to prevent N-day vulnerabilities</li>
</ul>
<h2><a id="post-133492-_r0vj4vfdbt1a"></a><strong>Conclusion</strong></h2>
<p>DNS tunneling techniques can be leveraged by adversaries to perform various actions not normally associated with DNS tunneling. Despite the conventional impression that tunneling is used for C2 and VPN purposes, we also find that attackers can use DNS tunneling as a vehicle for victim activity tracking and network scanning.</p>

(function($) {
	// Migrated from the unit42-v5 + Modifications
	var subscribeSuccess = false;
	var email = document.getElementById('newsletter-email');
	var subscription_form = document.getElementById('unit42footerSubscription_form');
	var subscription_form_button = document.getElementById('unit42footerSubscription_form_button');
	window.captchaComplete = function() {
		subscribeSuccess = true;
		if ($(mail).val() != '' && isEmail($(mail).val())) {
			$(subscription_form_button).removeClass('is-disabled');
		}
		setTimeout(function() {
			$(email).focus();
			$('.g-recaptcha iframe').attr('tabindex', '-1');
		}, 100)
	}
	window.captchaExpires = function() {
		subscribeSuccess = false;
		$(subscription_form_button).addClass('is-disabled', true);
	} 
	$(subscription_form).submit(function(e) {
		e.preventDefault();
		e.stopImmediatePropagation();
		updateEmailMask();
		var success = true;
		var form = $(this);
		var mail = form.find('input[name="Email"]');
		
		if (mail.val() === '') {
			mail.addClass('has-error');
			showError(1);
			success = false;	
		}  else if (!isEmail(mail.val())){
			showError(2);
			success = false;
		}
		else {
			mail.removeClass('has-error');
			$('.error-mail').addClass('d-none');
		}

		if (!subscribeSuccess) {
			$('.error-recaptcha').removeClass('d-none');
		} else {
			$('.error-recaptcha').addClass('d-none');
		}

		if (success && subscribeSuccess) {
			$.ajax({
				type: 'POST',
				url: form.attr('action'),
				data: form.serialize(),
				beforeSend: function() {
                 form.find('button').addClass('is-loading');
    			},
				success: function(msg) {
					form.find('.form-success-message').html('<p class="success-message">You have been successfully subscribed</p>');
					form.find('button').removeClass('is-loading');
					$(email).val('');
					clearError();
				},
				error: function(jqXHR, textStatus, errorThrown) {
					$(subscription_form_button).addClass('is-disabled', true);
					form.find('button').removeClass('is-loading');
					
				}
			});
		}
		return false;
	});
	function showError(error_type){
		if(error_type == 1) {
			$('.error-mail').text("Please enter the email address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		} else if(error_type == 2){
			$('.error-mail').text("Please provide a valid e-mail address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		}	
		$(subscription_form_button).removeClass('is-loading');
	}
	function clearError(){
		$('.error-mail').text("").removeClass('error-show');;
		$(subscription_form_button).removeClass('is-loading');
		$(subscription_form_button).removeClass('is-disabled');
	}
	
	$(email).on('input', function (event) { 
		var email = $(this).val();
		if (isEmail(email) ) {			
			clearError();
		} else if(email == ""){
			clearError();
		} else{
			showError(2);			  
		}
    });

	function isEmail(email) {
		var re = /^(([^<>()\[\]\\.,;:\s@"]+(\.[^<>()\[\]\\.,;:\s@"]+)*)|(".+"))@((\[[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\])|(([a-zA-Z\-0-9]+\.)+[a-zA-Z]{2,}))$/;
		return re.test(String(email).toLowerCase());
	}
	
	var captcha_loaded = false;
         if(!captcha_loaded){
            // recaptcha on foucs call
            $(document).on('change paste keyup', '#newsletter-email', function () {
               if($('.g-recaptcha').hasClass('d-none')){
                $('.g-recaptcha').removeClass('d-none');
               }
               if(!captcha_loaded ){
                  captcha_loaded  = true;
                  // trigger loading api.js (recaptcha.js) script
                  var head = document.getElementsByTagName('head')[0];
                  var script = document.createElement('script');
                  script.type = 'text/javascript';
                  script.src = 'https://www.google.com/recaptcha/api.js?hl=en_US';
                  head.appendChild(script);
               }
            });
        }
	
	function updateEmailMask() {
		var email = $("#unit42footerSubscription_form input[name='Email']").val();
		if (email && email.trim() != '') {
			var maskedEmail = maskEmailAddress(email);
			$("#unit42footerSubscription_form input[name='emailFormMask']").val(maskedEmail);
		}
	}

	function maskEmailAddress (emailAddress) {
		function mask(str) {
			var strLen = str.length;
			if (strLen > 4) {
				return str.substr(0, 1) + str.substr(1, strLen - 1).replace(/\w/g, '*') + str.substr(-1,1);
			}
			return str.replace(/\w/g, '*');
		}
		return emailAddress.replace(/([\w.]+)@([\w.]+)(\.[\w.]+)/g, function (m, p1, p2, p3) {
			return mask(p1) + '@' + mask(p2) + p3;
		});

		return emailAddress;
	}

}(jQuery));
