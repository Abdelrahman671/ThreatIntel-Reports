		<h3 class="tags">
                            <a class="post-category" href="https://expel.com/blog/category/security-operations/">
                    Security operations</a>
                                        ·  7  MIN READ
                        				·  ANTHONY RANDAZZO, BRITTON MANAHAN AND SAM LIPTON											·  APR 28, 2020			                            ·  TAGS:
                                     <a href="https://expel.com/blog/tag/ciso/">
                        <span>CISO</span>
                    </a>
                                    / <a href="https://expel.com/blog/tag/company-news/">
                        <span>Company news</span>
                    </a>
                                    / <a href="https://expel.com/blog/tag/get-technical/">
                        <span>Get technical</span>
                    </a>
                                    / <a href="https://expel.com/blog/tag/heads-up/">
                        <span>Heads up</span>
                    </a>
                                    / <a href="https://expel.com/blog/tag/managed-security/">
                        <span>Managed security</span>
                    </a>
                                    </h3>
		<p>Detection and response in cloud infrastructure is a relatively new frontier. On top of that, there aren’t many compromise details publicly available to help shape the detection strategy for anyone running workloads in the cloud.</p>
<p>That’s why our team here at Expel is attempting to bridge the gap between theory and practice.</p>
<p>Over the years, we’ve detected and responded to countless Amazon Web Services (AWS) incidents, ranging from public S3 bucket exposures to compromised EC2 instance credentials and <a href="https://aws.amazon.com/rds/">RDS</a> ransomware attacks. Recently, we identified an incident involving the use of compromised AWS access keys.</p>
<p>In this post, we&#8217;ll walk you through how we caught the problem, what we observed in our response, how we kicked the bad guy out and the lessons we learned along the way.</p>
<h2>Compromised AWS access keys: How we caught ‘em</h2>
<p>We first determined there was something amiss thanks to an Expel detection using CloudTrail logs. Here at Expel, we encourage many of our customers who run on AWS to use Amazon GuardDuty. But we’ve also taken it upon ourselves to <a href="https://expel.com/blog/following-cloudtrail-generating-aws-security-signals-sumo-logic/" target="_blank" rel="noopener noreferrer">develop detection use cases against CloudTrail logs</a>. GuardDuty does a great job of identifying common attacks, and we’ve also found CloudTrail logs to be a great source of signal for additional alerting that’s more specific to an AWS service or an environment.</p>
<p>It all started with the alert below, telling us that EC2 SSH access keys were being generated (<a href="https://docs.aws.amazon.com/AWSEC2/latest/APIReference/API_CreateKeyPair.html" target="_blank" rel="noopener noreferrer">CreateKeyPair</a>/<a href="https://docs.aws.amazon.com/AWSEC2/latest/APIReference/API_ImportKeyPair.html" target="_blank" rel="noopener noreferrer">ImportKeyPair</a>) from a suspicious source IP address.</p>
<p><img decoding="async" class="aligncenter size-full wp-image-5946" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20607'%3E%3C/svg%3E" alt="" width="1308" height="607" data-lazy-src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-alert.jpg" /><noscript><img decoding="async" class="aligncenter size-full wp-image-5946" src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-alert.jpg" alt="" width="1308" height="607" /></noscript><span style="font-size: 13px;"><em>Initial lead Expel alert</em></span></p>
<p>How’d we know it was suspicious?</p>
<p>We&#8217;ve created an orchestration framework that allows us to launch actions when certain things happen. In this case, when an alert fired an Expel robot picked it up and added additional information. This robot uses a third-party enrichment service for IPs (in this case, our friends at <a href="https://ipinfo.io/" target="_blank" rel="noopener noreferrer">ipinfo.io</a>). More on our robots here shortly.</p>
<hr />
<p><strong>Keep in mind that these are not logins to AWS per se. These are authenticated API calls with valid IAM user access keys. API access can be restricted at the IP layer, but it <a href="https://aws.amazon.com/premiumsupport/knowledge-center/iam-restrict-calls-ip-addresses/" target="_blank" rel="noopener noreferrer">can be a little burdensome to manage</a> in the IAM Policy. As you can see in the alert shown above, there was no MFA enforced for this API call. Again, this was not a login, but you can also <a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_mfa_configure-api-require.html" target="_blank" rel="noopener noreferrer">enforce MFA for specific API calls through the IAM Policy</a>. We’ve observed only a few AWS customers using either of these controls.</strong></p>
<hr />
<p>Another interesting detail from this alert was the use of the <a href="https://aws.amazon.com/cli/" target="_blank" rel="noopener noreferrer">AWS Command Line Interface (CLI)</a>. This isn’t completely out of the norm, but it heightened our suspicion a bit because it’s less common than console (UI) or AWS SDK access. Additionally, we found this user hadn’t used the AWS CLI in recent history, potentially indicating a new person was using these credentials. The manual creation of an access key was also an atypical action versus leveraging infrastructure as code to manage keys (i.e. CloudFormation or Terraform).</p>
<p>Taking all of these factors into consideration, we knew we had an event worthy of additional investigation.</p>
<h3>Cue the robots, answer some questions</h3>
<p>Our orchestration workflows are critically important – they tackle highly repetitive tasks, that is answer questions an analyst would ask about an alert, on our behalf as soon as the alert fires. We call these workflows our robots.</p>
<p>When we get an AWS alert from a customer’s environment, we have three consistent questions we like to answer to help our analysts determine if it&#8217;s worthy of additional investigation (decision support):</p>
<ul>
<li>Did this IAM principal (user, role, etc.) assume any other roles?</li>
<li>What AWS services does this principal normally interact with?</li>
<li>What interesting API calls has this principal made?</li>
</ul>
<p>So, when the initial lead alert for the SSH key generation came in, we quickly understood that role assumption was not in play for this compromise. If the user had assumed roles, it would have been key to identity and include them in the investigation. Instead, we saw the image below:</p>
<p><img decoding="async" class="aligncenter size-full wp-image-5947" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20277'%3E%3C/svg%3E" alt="" width="1308" height="277" data-lazy-src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-assuemrole_robot.jpg" /><noscript><img decoding="async" class="aligncenter size-full wp-image-5947" src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-assuemrole_robot.jpg" alt="" width="1308" height="277" /></noscript><span style="font-size: 13px;"><em>Expel AWS AssumeRole Robot</em></span></p>
<p>Once we knew access was limited to this IAM user, we wanted to know what AWS services this principal generally interacts with. Understanding this helps us spot outlier activity that’s considered unusual for that principal. Seeing the very limited API calls to other services further indicated that something nefarious might be going on.</p>
<p><img decoding="async" class="aligncenter size-full wp-image-5948" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20352'%3E%3C/svg%3E" alt="" width="1308" height="352" data-lazy-src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-aws-services-robot.jpg" /><noscript><img decoding="async" class="aligncenter size-full wp-image-5948" src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-aws-services-robot.jpg" alt="" width="1308" height="352" /></noscript><span style="font-size: 13px;"><em>Expel AWS Service Interaction Robot</em></span></p>
<p>Finally, we wanted to see what interesting API calls the principal made. From a detection perspective, we define interesting API calls in this context to be mostly anything that isn’t Get*, List*, Describe* and Head*. This enrichment returned 344 calls to the AuthorizeSecurityGroupIngress API from the AWS CLI user-agent. This is really the tipping point for considering this a security incident.</p>
<p><img decoding="async" class="aligncenter size-full wp-image-5950" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20331'%3E%3C/svg%3E" alt="" width="1308" height="331" data-lazy-src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-interesting-api-robot.jpg" /><noscript><img decoding="async" class="aligncenter size-full wp-image-5950" src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-interesting-api-robot.jpg" alt="" width="1308" height="331" /></noscript><span style="font-size: 13px;"><em>Expel AWS Interesting API Robot</em></span></p>
<h2>How we responded</h2>
<p>After we spotted the attack, we needed to scope this incident and provide the measures for containment. We framed our response by asking the primary investigative questions. Our initial response was going to be limited to determining what happened in the AWS control plane (API). CloudTrail was our huckleberry for answering most of our questions.</p>
<ul>
<li>What credentials did the attacker have access to?</li>
<li>How long has the attacker had access?</li>
<li>What did the attacker do with the access?</li>
<li>How did the attacker get access?</li>
</ul>
<h3>What credentials did the attacker have access to?</h3>
<p>By querying historical CloudTrail events for signs of this attacker, Expel was able to identify that they had access to a total of eight different IAM user access keys, and was active from two different IPs. If we recall from earlier, we were able to use our robot to determine that no successful AssumeRole calls were made, limiting our response to these IAM users.</p>
<h3>How long has the attacker had access?</h3>
<p>CloudTrail indicated that most of the access keys had not been used by anyone else in the past 30 days thus we can infer that the attacker likely discovered the keys recently.</p>
<h3>What did the attacker do with the access?</h3>
<p>Based on observed API activity, the attacker had a keen interest in S3, EC2 and RDS services as we observed <strong>ListBuckets</strong>, <strong>DescribeInstances</strong> and <strong>DescribeDBInstances</strong> calls for each access key, indicating an attempt to see which of these resources was available to the compromised IAM user.</p>
<p>As soon as the attacker identified a key with considerable permissions, <strong>DescribeSecurityGroups</strong> was called to determine the level of application tier access (firewall access) into the victim’s AWS environment. Once these groups were enumerated, the attacker “backdoored” all of the security groups with a utility similar to <a href="https://github.com/dagrz/aws_pwn/blob/master/persistence/backdoor_all_security_groups.py" target="_blank" rel="noopener noreferrer">aws_pwn’s backdoor_all_security_group script</a>. This allowed for any TCP/IP access into the victim’s environment.</p>
<p>Additional <strong>AuthorizeSecurityGroupIngress</strong> calls were made for specific ingress rules for port 5432 (postgresql) and port 1253, amounting to hundreds of unique Security Group rules created. These enabled the attacker to gain network access to the environment and created additional risks by exposing many AWS service instances (EC2, RDS, etc.) to the internet.</p>
<p>A subsequent <strong>DescribeInstances</strong> call identified available EC2 instances to the IAM user. The attacker then created a SSH key pair (our initial lead alert for <strong>CreateKeyPair</strong>) for an existing EC2 instance. This instance was not running at the time so the attacker turned it on via a<strong> RunInstances</strong> call. Ultimately, this series of actions resulted in command line access to the targeted EC2 instance, at which point visibility can be a challenge without additional OS logging or security products to investigate instance activity.</p>
<h3 id="credentials">How did the attacker get credentials?</h3>
<p>While frustrating, it&#8217;s not always feasible to identify the root cause of an incident for a variety of reasons. For example, sometimes the technology simply doesn’t produce the data necessary to determine the root cause. In this case, using the tech we had available to us, we weren’t able to determine how the attacker gained credentials, but we have the following suspicions:</p>
<ul>
<li>Given multiple credentials were compromised, it&#8217;s likely they were found in a public repository such as git, an exposed database or somewhere similar.</li>
<li>It&#8217;s also possible credentials were lifted from developer machines directly, for example the AWS credentials file.</li>
</ul>
<p>We attempted to confirm these, but couldn’t get to an answer in this case. Though unfortunate, it offers an opportunity to work with the victim to improve visibility.</p>

<h2>Cloud Security Threat Containment</h2>
<p>By thoroughly scoping the attacker’s activities, we were able to deliver clear remediation steps. This included:</p>
<ul>
<li>Deleting the compromised access keys for the eight involved IAM user accounts;</li>
<li>Snapshotting (additional forensic evidence) and rebuilding the compromised EC2 instance;</li>
<li>Deleting the SSH keys generated by the attacker;</li>
<li>And deleting the hundreds of Security Group ingress rules created by the attacker.</li>
</ul>
<p><strong>Resilience: Helping our customer improve their security posture</strong></p>
<p>When we say incident response isn’t complete without fixing the root of the problem – <a href="https://expel.com/services/managed-detection-and-response/" target="_blank" rel="noopener noreferrer">we mean it</a>.</p>
<p>One of the many things that makes us different at Expel is that we don’t just throw alerts over the fence. That would only be sort of helpful to our customers and puts us in a position where we’d have to tackle the same issue on another day … and likely on many more days after that.</p>
<p>We’re all about <a href="https://expel.com/blog/7-habits-highly-effective-socs/" target="_blank" rel="noopener noreferrer">efficiency</a> here. That’s why we provide clear recommendations for how to solve issues and what actions a customer can take to prevent these kinds of attacks in the future. Everybody wins (except for the bad guys).</p>
<p>While we weren’t certain how the access keys were compromised in the first place, below are the resilience recommendations we gave our customer once the issue was resolved.</p>
<p><img decoding="async" class="aligncenter size-full wp-image-5952" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20373'%3E%3C/svg%3E" alt="" width="1308" height="373" data-lazy-src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-unusedIAM-resilience.jpg" /><noscript><img decoding="async" class="aligncenter size-full wp-image-5952" src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-unusedIAM-resilience.jpg" alt="" width="1308" height="373" /></noscript><span style="font-size: 13px;"><em>Expel AWS Resilience (1)</em></span></p>
<ul>
<li>If the IAM user is unused, then it probably doesn’t need to remain active in your account. We made this recommendation because these access keys hadn’t been in use by anyone other than the attacker in the previous 30 days.</li>
</ul>
<p><span style="font-size: 13px;"><em><img decoding="async" class="aligncenter size-full wp-image-5965" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20479'%3E%3C/svg%3E" alt="" width="1308" height="479" data-lazy-src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-IAM-keys-resilience-1.jpg" /><noscript><img decoding="async" class="aligncenter size-full wp-image-5965" src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-IAM-keys-resilience-1.jpg" alt="" width="1308" height="479" /></noscript>Expel AWS Resilience (2)</em></span></p>
<ul>
<li>Since the access keys for this IAM principal were at least 30 days old given that no activity occurred from a legitimate user, it was time to do some tidying up, so to speak. If you need that user, rotate the access keys on a regular basis.</li>
</ul>
<p><img decoding="async" class="aligncenter size-full wp-image-5949" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%201308%20479'%3E%3C/svg%3E" alt="" width="1308" height="479" data-lazy-src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-IAM-keys-resilience.jpg" /><noscript><img decoding="async" class="aligncenter size-full wp-image-5949" src="https://expel.com/wp-content/uploads/2020/04/image-Expel-finding-evil-in-AWS-IAM-keys-resilience.jpg" alt="" width="1308" height="479" /></noscript><span style="font-size: 13px;"><em>Expel AWS Resilience (3)</em></span></p>
<ul>
<li>We noticed that this IAM user had far too many EC2 permissions and thought this resilience measure was in order. We also shared that it would be far safer to delegate those EC2 permissions with an IAM role.</li>
</ul>
<h2>Lessons learned</h2>
<p>Fortunately, we were able to disrupt this attack before there was any serious damage, but it highlighted the very real fact that cloud infrastructure – whether you’re running workloads on AWS or somewhere else – is a prime target for attackers. As with every incident, we took some time to talk through what we discovered through this investigation and are sharing our key lessons here.</p>
<ul>
<li><strong>AWS customers must architect better security “in” the cloud.</strong> That is, create greater visibility into the EC2 and other infrastructure identified in the shared responsibility model.</li>
<li>You can’t find evil if the analysts don’t know what to look for <strong>– train, train some more, and then when you’re done training, train again.</strong> Special thanks to Scott Piper (<a href="https://twitter.com/0xdabbad00" target="_blank" rel="noopener noreferrer">@0xdabbad00</a>) and Rhino Security Labs (<a href="https://twitter.com/RhinoSecurity" target="_blank" rel="noopener noreferrer">@RhinoSecurity</a>) for their contributions to AWS security research.</li>
<li><strong>While security in the cloud is still relatively in its infancy, the same can be said for the attacker behaviors –</strong> much of what we observed here and in the past were elementary attack patterns.</li>
<li><strong>There are additional automated enrichment opportunities.</strong> We’ve started working on a new AWS robotic workflow to summarize historical API usage data for the IAM principal and will compare it to the access parameters of the alert.</li>
</ul>
<p>Be on the lookout for an additional blog post in the future for our automated AWS alert enrichments. Until then, check out our other <a href="https://expel.com/blog/" target="_blank" rel="noopener noreferrer">blogs</a> to learn more about how we leverage AWS cloud security for our customers, along with tips and tricks for ramping up your own org’s security when it comes to cloud.</p>
