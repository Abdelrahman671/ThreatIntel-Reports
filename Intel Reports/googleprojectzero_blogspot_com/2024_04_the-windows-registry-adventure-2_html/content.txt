 <p class="zwnMkNbCmE-c14 zwnMkNbCmE-c25"><span class="zwnMkNbCmE-c26">Posted by Mateusz Jurczyk, Google Project Zero</span></p>
 <p class="zwnMkNbCmE-c14 zwnMkNbCmE-c23"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span>Before diving into the low-level security aspects of the registry, it is important to understand its role in the operating system and a bit of history behind it. In essence, the registry is a hierarchical database made of named &quot;keys&quot; and &quot;values&quot;, used by Windows and applications to store a variety of settings and configuration data. It is represented by a tree structure, in which keys may have one or more sub-keys, and every subkey is associated with exactly one parent key. Furthermore, every key may also contain one or more values, which have a type (integer, string, binary blob etc.) and are used to store actual data in the registry. Every key can be uniquely identified by its name and the names of all of its ascendants separated by the special backslash character (&#39;\&#39;), and starting with the name of one of the top-level keys (</span><span class="zwnMkNbCmE-c3">HKEY_LOCAL_MACHINE</span><span>, </span><span class="zwnMkNbCmE-c3">HKEY_USERS</span><span>, etc.). For example, a full registry path may look like this: </span><span class="zwnMkNbCmE-c3">HKEY_CURRENT_USER\Software\Microsoft\Windows</span><span>. At a high level, this closely resembles the structure of a file system, where the top-level key is equivalent to the root of a mounted disk partition (e.g. C:\), keys are equivalent to directories, and values are equivalent to files. One important distinction, however, is that keys are the only type of securable objects in the registry, and values play a much lesser role in the database than files do in the file system. Furthermore, specific subtrees of the registry are stored on disk in binary files called </span><span class="zwnMkNbCmE-c19">registry hives</span><span>, and the hive mount points don&#39;t necessarily correspond one-to-one to the top-level keys (e.g. the </span><span class="zwnMkNbCmE-c3">C:\Windows\system32\config\SOFTWARE</span><span>&nbsp;hive is mounted under </span><span class="zwnMkNbCmE-c3">HKEY_LOCAL_MACHINE\Software</span><span class="zwnMkNbCmE-c2">, a one-level nested key).</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">Fundamentally, there are only a few basic operations that can be performed in the registry. These operations are summarized in the table below:</span></p>

 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span>Before we dive into any of them in particular, let&#39;s first trace the registry&#39;s evolution and the path that led to its current state.</span></p><h2 class="zwnMkNbCmE-c16 zwnMkNbCmE-c14" id="h.er920v1wpv7t"><span class="zwnMkNbCmE-c17">Windows 3.1</span></h2>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span>The registry was first introduced in Windows 3.1 released in 1992. It was designed as a centralized configuration store meant to address the many shortcomings of basic text configuration files from MS-DOS (e.g. </span><span class="zwnMkNbCmE-c3">config.sys</span><span>) and the slightly more structured .INI files from very early versions of Windows. But the first registry was nothing like we know it today: there was only one top-level key (an equivalent of </span><span class="zwnMkNbCmE-c3">HKEY_CLASSES_ROOT</span><span>) and only one hive (</span><span class="zwnMkNbCmE-c3">C:\windows\reg.dat</span><span>) limited to 64 KB in size, formatted in a custom binary format represented by the magic bytes &quot;SHCC3.10&quot;. There were no values (data was assigned directly to keys), and the registry was used solely for OLE/COM and file type registration. This is what the first Regedit.exe looked like when launched in </span><span class="zwnMkNbCmE-c19">advanced mode</span><span class="zwnMkNbCmE-c2">:</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c13"><span class="zwnMkNbCmE-c2"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEjIGV3RisD_k7944hSPyIBpIgkX942EoD0GwYrvsF10IK5RlafbYtJQZQujn0pHCguofBHZSjICqE74cfFQL-jQvBbY20O4wE6EjE5wub_ZJxMzyANUvd65RMHHgrWOpIwpJdQRWr6efCMj1sDf_TVWtrqU51P9HujQ0RGm07YDNbtfxvAaP2kPbivN1pI/s640/image1.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="Screenshot of the first registry editor running on Windows 3.1" border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEjIGV3RisD_k7944hSPyIBpIgkX942EoD0GwYrvsF10IK5RlafbYtJQZQujn0pHCguofBHZSjICqE74cfFQL-jQvBbY20O4wE6EjE5wub_ZJxMzyANUvd65RMHHgrWOpIwpJdQRWr6efCMj1sDf_TVWtrqU51P9HujQ0RGm07YDNbtfxvAaP2kPbivN1pI/s640/image1.png" style="max-height: 750px; max-width: 600px;" title="Screenshot of the first registry editor running on Windows 3.1" /></a></span></p>

 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span>Despite its limitations, the Windows 3.1 registry was an important milestone, as it established long-lasting concepts like its hierarchical structure and paved the way for today&#39;s advanced registry features.</span></p><h2 class="zwnMkNbCmE-c16 zwnMkNbCmE-c14" id="h.84eoez0bjso"><span class="zwnMkNbCmE-c17">Windows NT 3.1, 3.5 and 3.51</span></h2>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">One year later in 1993, a new version of Windows was released based on a completely refreshed and more robust kernel design: Windows NT 3.1. To this day, the original NT kernel continues to be the underpinning of all modern versions of Windows up to and including Windows 11 &ndash; and the same can be said for its registry implementation. The biggest functional registry changes found in Windows NT 3.x as compared to Windows 3.1 were:</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p><ul style="padding: 0;" class="c6 lst-kix_9jktwi63hg68-0 start"><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span class="zwnMkNbCmE-c2">Introducing many new top-level keys (HKLM, HKCU, HKU) and thus extending the scope of information intended to be stored in the registry.</span></li><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span>Replacing the single </span><span class="zwnMkNbCmE-c3">reg.dat</span><span>&nbsp;hive file with a number of separate hives (</span><span class="zwnMkNbCmE-c3">default</span><span>, </span><span class="zwnMkNbCmE-c3">sam</span><span>, </span><span class="zwnMkNbCmE-c3">security</span><span>, </span><span class="zwnMkNbCmE-c3">software</span><span>, </span><span class="zwnMkNbCmE-c3">system</span><span>&nbsp;located in </span><span class="zwnMkNbCmE-c3">C:\winnt\system32\config</span><span class="zwnMkNbCmE-c2">).</span></li><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span class="zwnMkNbCmE-c2">Introducing named values with several possible data types.</span></li><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span class="zwnMkNbCmE-c2">Making registry keys securable.</span></li><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span class="zwnMkNbCmE-c2">Eliminating the 64 KB registry hive limit.</span></li></ul>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span>To accommodate these new features, Windows adopted a novel binary format called &quot;regf&quot;, which was specifically designed to support the expanded functionality. The core principles behind the format remained unchanged across the NT 3.x version line, but it continued to internally evolve, as signified by the increasing version numbers encoded in the hive file headers. Specifically, pre-release builds of Windows NT 3.1 used </span><span class="zwnMkNbCmE-c0">regf v1.0</span><span>, Windows NT 3.1 RTM used </span><span class="zwnMkNbCmE-c0">regf v1.1</span><span>, and Windows NT 3.5 and 3.51 used </span><span class="zwnMkNbCmE-c0">regf v1.2</span><span class="zwnMkNbCmE-c2">.</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span>Lastly, while Regedit.exe remained the simplistic &quot;Registration Info Editor&quot;, a new utility, </span><span class="zwnMkNbCmE-c3">RegEdt32.exe</span><span class="zwnMkNbCmE-c2">, was added with far more options and unrestricted access to the system registry. Despite its dated appearance, the structure of the UI began to resemble the shape of the modern registry and the core concepts behind today&#39;s registry editor:</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c13"><span class="zwnMkNbCmE-c1"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEiXTxJuEci8Qz786crT_vx4E__oLe1xylzj7YLT6y3fcEDUcQ5xMCm1FrKGC8aDEByXq4APReP6VB4urDJvcnHaCh6phypDXehNLx8oB29gm1MV9NFoay9O-foFlQeqLCTLsph-Pa24U7Aw6-zuRCXJBHTe5ssF7h5BHKQDTsYzqt_-g03jHptRCPB0LKU/s640/image5.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="Screenshot of RegEdt32.exe running on Windows NT 3.1" border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEiXTxJuEci8Qz786crT_vx4E__oLe1xylzj7YLT6y3fcEDUcQ5xMCm1FrKGC8aDEByXq4APReP6VB4urDJvcnHaCh6phypDXehNLx8oB29gm1MV9NFoay9O-foFlQeqLCTLsph-Pa24U7Aw6-zuRCXJBHTe5ssF7h5BHKQDTsYzqt_-g03jHptRCPB0LKU/s640/image5.png" style="max-height: 750px; max-width: 600px;" title="Screenshot of RegEdt32.exe running on Windows NT 3.1" /></a></span></p>
 <p class="zwnMkNbCmE-c13"><span class="zwnMkNbCmE-c21 zwnMkNbCmE-c19">RegEdt32.exe running on Windows NT 3.1</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">Notably, Windows NT 3.1 was the first system whose parts of code are still used today in Windows 11. Based on this observation, we can now confidently claim that the registry code base is over 30 years old.</span></p><h2 class="zwnMkNbCmE-c14 zwnMkNbCmE-c16" id="h.5ow8q7kr5g6c"><span class="zwnMkNbCmE-c17">Windows 95</span></h2>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span>Not long after, in the summer of 1995, Windows 95 was officially released to the public. It quickly became a huge hit, mostly thanks to innovations in the user interface &ndash; it was the first version to feature a taskbar, the Start menu, and the general look and feel that we now associate with Windows. With regards to the registry internals, though, it wasn&#39;t particularly interesting. It continued the trend started by Windows NT 3.x of expanding the registry into an even more central part of the operating system, and borrowed many of the same high-level concepts. However, since it was based on a completely different kernel than NT, the underlying registry implementation differed, too. All of the registry data was typically stored in just two files: </span><span class="zwnMkNbCmE-c3">C:\WINDOWS\System.dat</span><span>&nbsp;and </span><span class="zwnMkNbCmE-c3">C:\WINDOWS\User.dat</span><span class="zwnMkNbCmE-c2">. They were encoded in yet another binary format indicated by the &quot;CREG&quot; signature, which was more capable than the Win3.1 format, but inferior to WinNT&#39;s regf (e.g. it didn&#39;t support security descriptors). The same format was later inherited by subsequent systems from the 9x series, namely Windows 98 and Me, but its legacy ended there. According to my knowledge, the CREG format had minimal impact on the registry&#39;s development in the NT line, so a deeper discussion of its internals isn&#39;t necessary.</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">Arguably, the one thing that had the most lasting impact in Windows 95 related to registry was the complete redesign of Regedit.exe, both functionally and visually. It gained the ability to browse the entire registry tree, read existing values and create new ones, rename keys, and search for text strings within keys, values and data. At first glance, it looks almost identical to the modern Registry Editor, with the exception of a few missing options, such as loading custom hives or managing key security. Even the program icon has remained largely unchanged and to many power users, it is synonymous with the Windows registry up to this day:</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c13"><span class="zwnMkNbCmE-c2"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgqNprR0lutls9iQfc2TyOTHMgILbRdpSizf2cILN8s0QwpYnWLlZZAhkwH90GlAry1uo_TyntkBkBW6SVG6TRosM-NbIcbopuY-5JS-hL5117mFnS3uy7P7fCh0dGtjQpj5fll8HF3hyphenhyphenlHSdfuK1DpC_6OitzGW4FZ9BqqtNLn-erUCLRRAMeft30Zn7k/s640/image2.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="Screenshot of Redesigned Regedit.exe running on Windows 95" border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgqNprR0lutls9iQfc2TyOTHMgILbRdpSizf2cILN8s0QwpYnWLlZZAhkwH90GlAry1uo_TyntkBkBW6SVG6TRosM-NbIcbopuY-5JS-hL5117mFnS3uy7P7fCh0dGtjQpj5fll8HF3hyphenhyphenlHSdfuK1DpC_6OitzGW4FZ9BqqtNLn-erUCLRRAMeft30Zn7k/s640/image2.png" style="max-height: 750px; max-width: 600px;" title="Screenshot of Redesigned Regedit.exe running on Windows 95" /></a></span></p>
 <p class="zwnMkNbCmE-c13"><span class="zwnMkNbCmE-c19 zwnMkNbCmE-c21">Redesigned Regedit.exe running on Windows 95</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p><h2 class="zwnMkNbCmE-c16 zwnMkNbCmE-c14" id="h.qafgvcbasjq0"><span class="zwnMkNbCmE-c17">Windows NT 4.0</span></h2>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span>The debut of Windows NT 4.0 in 1996 marked another important milestone for the registry, but this time mostly on the technical side. In terms of visuals, NT 4.0 adopted the same graphical interface as Windows 95, including the new and improved Regedit.exe. As a result of the Regedit addition, Windows NT 4.0 now included two competing registry editors: Regedit from Windows 95 and RegEdt32 from Windows NT 3.x. They shared some overlapping functionality (e.g. the ability to manually traverse the registry and inspect individual values), but each offered some unique features too: only Regedit was capable of searching for data in values, while only RegEdt32 supported managing the security of registry keys. I suspect that the presence of two different tools must have been confusing for users who wanted to modify the system&#39;s internal settings: not only did they have to understand the structure of the registry and how to navigate it, but also know which tool to use for a specific task. Both utilities made their way into Windows 2000, but they were finally merged in Windows XP into a single Regedit.exe program. RegEdt32.exe can still be found on modern versions of Windows in </span><span class="zwnMkNbCmE-c3">C:\Windows\system32</span><span>&nbsp;as a historical artifact</span><span class="zwnMkNbCmE-c2">, but all it currently does is just launch Regedit.exe and terminate.</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span>As mentioned earlier, the really important changes in NT 4.0 happened under the hood. Between the release of NT 3.51 and NT 4.0, the kernel developers updated some internal aspects of the regf format to simplify it and make it more efficient. Furthermore, a new optimization called &quot;fast leaves&quot; was introduced, which added special four-byte hints to the subkey lists in order to speed up key lookups. These changes were substantial and not backwards-compatible, so the version had to be increased again, leading to </span><span class="zwnMkNbCmE-c0">regf v1.3</span><span>. This is noteworthy because 1.3 is the earliest hive type that is considered a </span><span class="zwnMkNbCmE-c19">modern version</span><span>&nbsp;and that is still supported by today&#39;s Windows 10 and 11, even though newer format versions up to 1.6 exist now too. It means that one can copy a hive file off of a Windows NT 4.0 system, load it in Regedit on Windows 11, examine and modify it, copy it back, and each of these steps will work without issue. What is more, the support is not just there for reading archival hives &ndash; in documented API functions such as </span><span class="zwnMkNbCmE-c20"><a class="zwnMkNbCmE-c151" href="https://learn.microsoft.com/en-us/windows/win32/api/winreg/nf-winreg-regsavekeyexa">RegSaveKeyExA</a></span><span>, version 1.3 is represented by the </span><span class="zwnMkNbCmE-c3">REG_STANDARD_FORMAT</span><span>&nbsp;enum, indicating that it is considered the &quot;standard&quot; even as of today. And indeed, there are some core system hives in Windows 11, such as UsrClass.dat mounted at </span><span class="zwnMkNbCmE-c3">HKEY_USERS\&lt;SID&gt;_Classes</span><span>, that are still encoded in the regf v1.3 format. So in that sense, Windows NT 4.0 and 11, despite being released decades apart and representing vastly different technological eras, exhibit a fundamental connection.</span></p><h2 class="zwnMkNbCmE-c16 zwnMkNbCmE-c14" id="h.5lhzotyodb8x"><span class="zwnMkNbCmE-c17">Modern times</span></h2>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">Based on the fact that both the regf hive format and the graphical interface of Regedit have essentially remained the same between 1996 and 2024, one could assume that the internal registry implementation hasn&#39;t changed that much, either. We can try to prove or disprove this hypothesis by performing a little experiment, measuring the volume of registry-related code in each consecutive version of Windows. To ensure a consistent methodology and make the survey security-relevant, we will focus on the kernel-mode part of the Configuration Manager, which largely constitutes a local attack surface. Such an analysis is technically feasible and even relatively easy to achieve, because:</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p><ul style="padding: 0;" class="c6 lst-kix_isoepzyy3bf4-0 start"><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span>The entirety of the kernel registry-related code is compiled into a single executable image: </span><span class="zwnMkNbCmE-c3">ntoskrnl.exe</span><span class="zwnMkNbCmE-c2">.</span></li><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span class="zwnMkNbCmE-c2">Debug symbols (PDB/DBG files) for the kernels of all NT-family systems were made publicly available by Microsoft, either via the Microsoft Symbol Server, symbol packages downloadable from the Microsoft website, or symbol files bundled with the system installation media.</span></li><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span>The kernel code follows a consistent naming convention, where all function names related to the registry start with either &quot;Hv&quot; (standing for </span><span class="zwnMkNbCmE-c19">Hive</span><span>), &quot;Cm&quot; (standing for </span><span class="zwnMkNbCmE-c19">Configuration Manager</span><span>) or &quot;Vr&quot; (likely standing for </span><span class="zwnMkNbCmE-c19">Virtualized Registry</span><span class="zwnMkNbCmE-c2">), with a few minor exceptions.</span></li><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span class="zwnMkNbCmE-c2">There are some very good reverse-engineering tools available today, which can help us count the number of assembly instructions or even the number of decompiled C-like source code lines corresponding to the registry engine.</span></li></ul>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">In my case, I used IDA Pro with Hex-Rays to decompile the entire kernel of each NT-line system, and then ran a post-processing script to extract the registry related functions. After counting the numbers of lines and plotting them on a diagram, here is what we get:</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEiyJpeSxZgU24oX1-oMXNHlEHNoBE8WsLCjmZ115l-QHtYYQZkN4KZdO82Y4yJtoEa2apxSqSFf_Ha-F0FY0eS8C7nfNDqeQa_oRLw8jDOVZMl3GqyAFOek10bh-rL40bq9Oq_5ZaAMFRreYLP7fEWqAy9joPLQx2mqRo4S-aKhA1FVU9xoJoPb5K1f1Ng/s1999/image3.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="Image of a histogram showing lines of code for registry related functions per windows version, starting at around 10,000 lines of code in NT 4.0 and increasing tenfold to around 100,000 lines in Windows 11" border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEiyJpeSxZgU24oX1-oMXNHlEHNoBE8WsLCjmZ115l-QHtYYQZkN4KZdO82Y4yJtoEa2apxSqSFf_Ha-F0FY0eS8C7nfNDqeQa_oRLw8jDOVZMl3GqyAFOek10bh-rL40bq9Oq_5ZaAMFRreYLP7fEWqAy9joPLQx2mqRo4S-aKhA1FVU9xoJoPb5K1f1Ng/s1200/image3.png" style="max-height: 750px; max-width: 600px;" title="Image of a histogram showing lines of code for registry related functions per windows version, starting at around 10,000 lines of code in NT 4.0 and increasing tenfold to around 100,000 lines in Windows 11" /></a></span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c21 zwnMkNbCmE-c19"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">As we can see, there has been an enormous, steady growth of the code base, starting at around 10,000 lines of code in NT 4.0 and increasing tenfold to around 100,000 lines in Windows 11. It is important to reiterate that this only covers the kernel portions of the registry and ignores code found in user-mode libraries such as advapi32.dll, KernelBase.dll or ntdll.dll. Furthermore, I expect that the decompiled code is more dense than the original source code because it doesn&#39;t include any comments or whitespace. Taking all this into account, the total extent of the registry code managed by Microsoft is probably much bigger than the numbers shown above.</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">Going back to the kernel registry code, its expansion in time has been substantial, both in absolute and relative terms. But if these developments are invisible to the average user, what does all of the new code do? The changes can be divided into three major categories:</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p><ul style="padding: 0;" class="c6 lst-kix_qqcc7cub1y3f-0 start"><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span class="zwnMkNbCmE-c0">Optimizations</span><span class="zwnMkNbCmE-c2">: changes making the registry more efficient, e.g. introducing a &quot;hash leaf&quot; subkey index type to make key lookups even faster in regf v1.5, or adding a native system call to rename keys in-place without involving an expensive copy+delete operation on an entire subtree.</span></li><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span class="zwnMkNbCmE-c0">Backwards compatibility:</span><span>&nbsp;changes meant to make legacy applications run seamlessly on modern systems, e.g. </span><span class="zwnMkNbCmE-c20"><a class="zwnMkNbCmE-c151" href="https://learn.microsoft.com/en-us/windows/win32/sysinfo/registry-virtualization">registry virtualization</a></span><span class="zwnMkNbCmE-c2">.</span></li><li style="margin-left: 46pt;" class="c5 li-bullet-0"><span class="zwnMkNbCmE-c0">New features:</span><span>&nbsp;changes adding new functionality to the registry or adapting it to new use cases. These are either made available via a new API (thus mainly relevant to software developers), or not documented at all and only used by Windows internally. Examples include support for values larger than 1 MB, </span><span class="zwnMkNbCmE-c20"><a class="zwnMkNbCmE-c151" href="https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/wdm/nf-wdm-cmregistercallbackex">registry callbacks</a></span><span>, </span><span class="zwnMkNbCmE-c20"><a class="zwnMkNbCmE-c151" href="https://learn.microsoft.com/en-us/windows/win32/api/winreg/nf-winreg-regcreatekeytransactedw">support for transactions</a></span><span>, </span><span class="zwnMkNbCmE-c20"><a class="zwnMkNbCmE-c151" href="https://learn.microsoft.com/en-us/windows/win32/api/winreg/nf-winreg-regloadappkeyw">application hives</a></span><span class="zwnMkNbCmE-c2">, and differencing hives.</span></li></ul>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">Interestingly, the biggest changes weren&#39;t occurring with any regularity, but rather were concentrated in just four versions of Windows: NT 3.1&ndash;4.0, XP, Vista and 10 Anniversary Update (1607). This is illustrated in the timeline below:</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgP2C56s_Qy-f91J8y_mYK0kdBPBqXuZrD_6jRg8ifjU5G5tqkNGJGOobSM9tv7xmbRB6tC7o5pP6_wgaPFIqbTvHBIoXP9pSTxS93EGw2AhnY-pM_HjVsHDPjQrp-PN9EPSTClzvRrZygkur4gyMAnoQeApBm0tvcP68ZVPlxCloeeGB0ZQAqb44WMhTU/s1859/image4.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="Timeline showing versions of windows and when registry functionality was added to each version, spanning from Windows NT 3.1 in 1993 through to Windows 10/11 22H2 in 2022" border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgP2C56s_Qy-f91J8y_mYK0kdBPBqXuZrD_6jRg8ifjU5G5tqkNGJGOobSM9tv7xmbRB6tC7o5pP6_wgaPFIqbTvHBIoXP9pSTxS93EGw2AhnY-pM_HjVsHDPjQrp-PN9EPSTClzvRrZygkur4gyMAnoQeApBm0tvcP68ZVPlxCloeeGB0ZQAqb44WMhTU/s1200/image4.png" style="max-height: 750px; max-width: 600px;" title="Timeline showing versions of windows and when registry functionality was added to each version, spanning from Windows NT 3.1 in 1993 through to Windows 10/11 22H2 in 2022" /></a></span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">This is of course not an exhaustive list: it includes the features that I have found to be the most interesting during the security audit, but it is missing modifications related to incremental logging, improvements to how hive files are managed and mapped in memory, and many other optimizations, stability improvements and refactorings implemented by Microsoft throughout the years. But it goes to show that the registry is a highly complex part of the Windows kernel, and one with a lot of potential for deep, interesting bugs just waiting to be discovered.</span></p>
 <p class="zwnMkNbCmE-c8"><span class="zwnMkNbCmE-c2"></span></p>
 <p class="zwnMkNbCmE-c10 zwnMkNbCmE-c14"><span class="zwnMkNbCmE-c2">In the next post, I will share a number of useful sources of information I have discovered while researching the registry. Some of them may be more obvious than others, but all of them have significantly helped me understand certain aspects of the technology or given me the necessary context that I was missing. Until next time!</span></p>

//<![CDATA[
(function(){
  let archive_list = document.getElementById('ArchiveList');
  if (archive_list == null) return;
  let cur_year = archive_list.querySelector('.post-count-link').innerText.trim() - 0;
  let last_year = 2014;
  let elements = [];
  const MONTHS = ',Jan,Feb,Mar,Apr,May,Jun,Jul,Aug,Sep,Oct,Nov,Dec'.split(',');

  let parent = document.getElementById('ArchiveList');
  while (parent.childNodes.length) parent.removeChild(parent.childNodes[0]);

  function fetch_next_year() {
    let url = 'https://googleprojectzero.blogspot.com/?action=getTitles&widgetId=BlogArchive1&widgetType=BlogArchive&responseType=js&path=https%3A%2F%2Fgoogleprojectzero.blogspot.com%2F'+cur_year;
    fetch(url).then(resp => {
      if (!resp.ok) {
        console.log('http error');
        return;
      }
      resp.text().then(text => {
        let scope = {
          _WidgetManager: {
            _HandleControllerResult: (name, method, results) => {
              elements.push(document.createElement('hr'));
              let year_header = document.createElement('div');
              year_header.appendChild(document.createTextNode(cur_year));
              year_header.style.fontSize = 'large';
              elements.push(year_header);

              let list = document.createElement('ul');
              elements.push(list);

              for (let obj of results.posts) {
                let link_parts = obj.url.split('/');
                let year = link_parts[3];
                let month = link_parts[4];

                let el = document.createElement(/*'div'*/'li');
                el.style.listStyleType = 'square';
                el.style.listStylePosition = 'inside';
                let link = document.createElement('a');
                el.appendChild(link);
                link.appendChild(document.createTextNode(obj.title));
                link.href = obj.url;
                let date_trailer = document.createElement('span');
                el.appendChild(date_trailer);
                //date_trailer.appendChild(document.createTextNode(' ('+year+'-'+month+')'));
                date_trailer.appendChild(document.createTextNode(' ('+MONTHS[parseInt(month, 10)]+')'));
                //date_trailer.style.textAlign = 'right';
                //elements.push(el);
                list.appendChild(el);
              }
            }
          }
        };
        with (scope) { eval(text); }
        if (cur_year == last_year) {
          finish();
        } else {
          cur_year--;
          fetch_next_year();
        }
      });
    });
  }
  fetch_next_year();
  function finish() {
    for (let obj of elements) {
      parent.appendChild(obj);
    }
    console.log(elements);
  }
})();
