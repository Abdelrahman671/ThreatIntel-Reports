											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>Azure Backup provides <a href="backup-azure-monitoring-built-in-monitor" data-linktype="relative-path">built-in monitoring and alerting capabilities</a> in a Recovery Services vault. These capabilities are available without any additional management infrastructure. But this built-in service is limited in the following scenarios:</p>
<ul>
<li>If you monitor data from multiple Recovery Services vaults across subscriptions</li>
<li>If the preferred notification channel is <em>not</em> email</li>
<li>If users want alerts for more scenarios</li>
<li>If you want to view information from an on-premises component such as System Center Data Protection Manager in Azure, which the portal doesn't show in <a href="backup-azure-monitoring-built-in-monitor#backup-jobs-in-backup-center" data-linktype="relative-path"><strong>Backup Jobs</strong></a> or <a href="backup-azure-monitoring-built-in-monitor#backup-alerts-in-recovery-services-vault" data-linktype="relative-path"><strong>Backup Alerts</strong></a></li>
</ul>
<h2 id="using-log-analytics-workspace">Using Log Analytics workspace</h2>
<h3 id="create-alerts-by-using-log-analytics">Create alerts by using Log Analytics</h3>
<p>In Azure Monitor, you can create your own alerts in a Log Analytics workspace. In the workspace, you use <em>Azure action groups</em> to select your preferred notification mechanism.</p>
<div class="IMPORTANT">
<p>Important</p>
<p>For information on the cost of creating this query, see <a href="https://azure.microsoft.com/pricing/details/monitor/" data-linktype="external">Azure Monitor pricing</a>.</p>
</div>
<p>Open the <strong>Logs</strong> section of the Log Analytics workspace and create a query for your own Logs. When you select <strong>New Alert Rule</strong>, the Azure Monitor alert-creation page opens, as shown in the following image.</p>
<p><img src="media/backup-azure-monitoring-laworkspace/custom-alert.png" alt="Create an alert in a Log Analytics workspace" data-linktype="relative-path"></p>
<p>Here the resource is already marked as the Log Analytics workspace, and action group integration is provided.</p>
<p><img src="media/backup-azure-monitoring-laworkspace/inkedla-azurebackup-createalert.jpg" alt="The Log Analytics alert-creation page" data-linktype="relative-path"></p>
<h4 id="alert-condition">Alert condition</h4>
<p>The defining characteristic of an alert is its triggering condition. Select <strong>Condition</strong> to automatically load the Kusto query on the <strong>Logs</strong> page as shown in the following image. Here you can edit the condition to suit your needs. For more information, see <a href="#sample-kusto-queries" data-linktype="self-bookmark">Sample Kusto queries</a>.</p>
<p><img src="media/backup-azure-monitoring-laworkspace/la-azurebackup-alertlogic.png" alt="Setting up an alert condition" data-linktype="relative-path"></p>
<p>If necessary, you can edit the Kusto query. Choose a threshold, period, and frequency. The threshold determines when the alert will be raised. The period is the window of time in which the query is run. For example, if the threshold is greater than 0, the period is 5 minutes, and the frequency is 5 minutes, then the rule runs the query every 5 minutes, reviewing the previous 5 minutes. If the number of results is greater than 0, you're notified through the selected action group.</p>
<div class="NOTE">
<p>Note</p>
<p>To run the alert rule once a day, across all the events/logs that were created on the given day, change the value of both 'period' and 'frequency' to 1440, that is, 24 hours.</p>
</div>
<h4 id="alert-action-groups">Alert action groups</h4>
<p>Use an action group to specify a notification channel. To see the available notification mechanisms, under <strong>Action groups</strong>, select <strong>Create New</strong>.</p>
<p><img src="media/backup-azure-monitoring-laworkspace/la-azurebackup-actiongroup.png" alt="Available notification mechanisms in the &quot;Add action group&quot; window" data-linktype="relative-path"></p>
<p>You can satisfy all alerting and monitoring requirements from Log Analytics alone, or you can use Log Analytics to supplement built-in notifications.</p>
<p>For more information, see <a href="/en-us/azure/azure-monitor/alerts/alerts-log" data-linktype="absolute-path">Create, view, and manage log alerts by using Azure Monitor</a> and <a href="/en-us/azure/azure-monitor/alerts/action-groups" data-linktype="absolute-path">Create and manage action groups in the Azure portal</a>.</p>
<h3 id="sample-kusto-queries">Sample Kusto queries</h3>
<p>The default graphs give you Kusto queries for basic scenarios on which you can build alerts. You can also modify the queries to fetch the data you want to be alerted on. Paste the following sample Kusto queries on the <strong>Logs</strong> page, and then create alerts on the queries.</p>
<p>Recovery Services vaults and Backup vaults send data to a common set of tables that are listed in this article. However, there are slight differences in the schema for Recovery Services vaults and Backup vaults (<a href="backup-azure-monitoring-built-in-monitor" data-linktype="relative-path">learn more</a>). So, this section is split into multiple sub-sections that helps you to use the right queries depending on which workload or vault types you want to query.</p>
<h4 id="queries-common-across-recovery-services-vaults-and-backup-vaults">Queries common across Recovery Services vaults and Backup vaults</h4>
<ul>
<li><p>All successful backup jobs</p>
<pre><code class="lang-Kusto">AddonAzureBackupJobs
| where JobOperation=="Backup"
| summarize arg_max(TimeGenerated,*) by JobUniqueId
| where JobStatus=="Completed"
</code></pre>
</li>
<li><p>All failed backup jobs</p>
<pre><code class="lang-Kusto">AddonAzureBackupJobs
| where JobOperation=="Backup"
| summarize arg_max(TimeGenerated,*) by JobUniqueId
| where JobStatus=="Failed"
</code></pre>
</li>
</ul>
<h4 id="queries-specific-to-recovery-services-vault-workloads">Queries specific to Recovery Services vault workloads</h4>
<ul>
<li><p>All successful Azure VM backup jobs</p>
<pre><code class="lang-Kusto">AddonAzureBackupJobs
| where JobOperation=="Backup"
| summarize arg_max(TimeGenerated,*) by JobUniqueId
| where JobStatus=="Completed"
| join kind=inner
(
    CoreAzureBackup
    | where OperationName == "BackupItem"
    | where BackupItemType=="VM" and BackupManagementType=="IaaSVM"
    | distinct BackupItemUniqueId, BackupItemFriendlyName
)
on BackupItemUniqueId
</code></pre>
</li>
<li><p>All successful SQL log backup jobs</p>
<pre><code class="lang-Kusto">AddonAzureBackupJobs
| where JobOperation=="Backup" and JobOperationSubType=="Log"
| summarize arg_max(TimeGenerated,*) by JobUniqueId
| where JobStatus=="Completed"
| join kind=inner
(
    CoreAzureBackup
    | where OperationName == "BackupItem"
    | where BackupItemType=="SQLDataBase" and BackupManagementType=="AzureWorkload"
    | distinct BackupItemUniqueId, BackupItemFriendlyName
)
on BackupItemUniqueId
</code></pre>
</li>
<li><p>All successful Azure Backup agent jobs</p>
<pre><code class="lang-Kusto">AddonAzureBackupJobs
| where JobOperation=="Backup"
| summarize arg_max(TimeGenerated,*) by JobUniqueId
| where JobStatus=="Completed"
| join kind=inner
(
    CoreAzureBackup
    | where OperationName == "BackupItem"
    | where BackupItemType=="FileFolder" and BackupManagementType=="MAB"
    | distinct BackupItemUniqueId, BackupItemFriendlyName
)
on BackupItemUniqueId
</code></pre>
</li>
<li><p>Backup Storage Consumed per Backup Item</p>
<pre><code class="lang-Kusto">CoreAzureBackup
//Get all Backup Items
| where OperationName == "BackupItem"
//Get distinct Backup Items
| distinct BackupItemUniqueId, BackupItemFriendlyName
| join kind=leftouter
(AddonAzureBackupStorage
| where OperationName == "StorageAssociation"
//Get latest record for each Backup Item
| summarize arg_max(TimeGenerated, *) by BackupItemUniqueId
| project BackupItemUniqueId , StorageConsumedInMBs)
on BackupItemUniqueId
| project BackupItemUniqueId , BackupItemFriendlyName , StorageConsumedInMBs
| sort by StorageConsumedInMBs desc
</code></pre>
</li>
</ul>
<h4 id="queries-specific-to-backup-vault-workloads">Queries specific to Backup vault workloads</h4>
<ul>
<li><p>All successful Azure PostgreSQL backup jobs</p>
<pre><code class="lang-Kusto">AddonAzureBackupJobs
| where JobOperation=="Backup"
| summarize arg_max(TimeGenerated,*) by JobUniqueId
  | where DatasourceType == "Microsoft.DBforPostgreSQL/servers/databases"
| where JobStatus=="Completed"	
</code></pre>
</li>
<li><p>All successful Azure Disk restore jobs</p>
<pre><code class="lang-Kusto">AddonAzureBackupJobs
| where JobOperation == "Restore"
| summarize arg_max(TimeGenerated,*) by JobUniqueId
| where DatasourceType == "Microsoft.Compute/disks"
| where JobStatus=="Completed"
</code></pre>
</li>
<li><p>Backup Storage Consumed per Backup Item</p>
<pre><code class="lang-Kusto">CoreAzureBackup
| where OperationName == "BackupItem"
| summarize arg_max(TimeGenerated, *) by BackupItemUniqueId
| project BackupItemUniqueId, BackupItemFriendlyName, StorageConsumedInMBs
</code></pre>
</li>
</ul>
<h3 id="diagnostic-data-update-frequency">Diagnostic data update frequency</h3>
<p>The diagnostic data from the vault is pumped to the Log Analytics workspace with some lag. Every event arrives at the Log Analytics workspace <em>20 to 30 minutes</em> after it's pushed from the Recovery Services vault. Here are further details about the lag:</p>
<ul>
<li>Across all solutions, the backup service's built-in alerts are pushed as soon as they're created. So they usually appear in the Log Analytics workspace after 20 to 30 minutes.</li>
<li>Across all solutions, on-demand backup jobs and restore jobs are pushed as soon as they <em>finish</em>.</li>
<li>For all solutions except SQL and SAP HANA backup, scheduled backup jobs are pushed as soon as they <em>finish</em>.</li>
<li>For SQL and SAP HANA backup, because log backups can occur every 15 minutes, information for all the completed scheduled backup jobs, including logs, is batched and pushed every 6 hours.</li>
<li>Across all solutions, other information such as the backup item, policy, recovery points, storage, and so on, is pushed at least <em>once per day.</em></li>
<li>A change in the backup configuration (such as changing policy or editing policy) triggers a push of all related backup information.</li>
</ul>
<div class="NOTE">
<p>Note</p>
<p>The same delay applies to other destinations for diagnostics data, such as Storage accounts and Event Hubs.</p>
</div>
<h2 id="using-the-recovery-services-vaults-activity-logs">Using the Recovery Services vault's activity logs</h2>
<div class="CAUTION">
<p>Caution</p>
<p>The following steps apply only to <em>Azure VM backups.</em> You can't use these steps for solutions such as the Azure Backup agent, SQL backups within Azure, or Azure Files.</p>
</div>
<p>You can also use activity logs to get notification for events such as backup success. To begin, follow these steps:</p>
<ol>
<li>Sign in into the Azure portal.</li>
<li>Open the relevant Recovery Services vault.</li>
<li>In the vault's properties, open the <strong>Activity log</strong> section.</li>
</ol>
<p>To identify the appropriate log and create an alert:</p>
<ol>
<li><p>Verify that you're receiving activity logs for successful backups by applying the filters shown in the following image. Change the <strong>Timespan</strong> value as necessary to view records.</p>
<p><img src="media/backup-azure-monitoring-laworkspace/activitylogs-azurebackup-vmbackups.png" alt="Filtering to find activity logs for Azure VM backups" data-linktype="relative-path"></p>
</li>
<li><p>Select the operation name to see the relevant details.</p>
</li>
<li><p>Select <strong>New alert rule</strong> to open the <strong>Create rule</strong> page.</p>
</li>
<li><p>Create an alert by following the steps in <a href="/en-us/azure/azure-monitor/alerts/alerts-activity-log" data-linktype="absolute-path">Create, view, and manage activity log alerts by using Azure Monitor</a>.</p>
<p><img src="media/backup-azure-monitoring-laworkspace/new-alert-rule.png" alt="New alert rule" data-linktype="relative-path"></p>
</li>
</ol>
<p>Here the resource is the Recovery Services vault itself. Repeat the same steps for all of the vaults in which you want to be notified through activity logs. The condition won't have a threshold, period, or frequency because this alert is based on events. As soon as the relevant activity log is generated, the alert is raised.</p>
<h2 id="using-log-analytics-to-monitor-at-scale">Using Log Analytics to monitor at scale</h2>
<p>You can view all alerts created from activity logs and Log Analytics workspaces in Azure Monitor. Just open the <strong>Alerts</strong> pane on the left.</p>
<p>Although you can get notifications through activity logs, we highly recommend using Log Analytics rather than activity logs for monitoring at scale. Here's why:</p>
<ul>
<li><strong>Limited scenarios</strong>: Notifications through activity logs apply only to Azure VM backups. The notifications must be set up for every Recovery Services vault.</li>
<li><strong>Definition fit</strong>: The scheduled backup activity doesn't fit with the latest definition of activity logs. Instead, it aligns with <a href="/en-us/azure/azure-monitor/essentials/resource-logs#send-to-log-analytics-workspace" data-linktype="absolute-path">resource logs</a>. This alignment causes unexpected effects when the data that flows through the activity log channel changes.</li>
<li><strong>Problems with the activity log channel</strong>: In Recovery Services vaults, activity logs that are pumped from Azure Backup follow a new model. Unfortunately, this change affects the generation of activity logs in Azure Government, Azure Germany, and Microsoft Azure operated by 21Vianet. If users of these cloud services create or configure any alerts from activity logs in Azure Monitor, the alerts aren't triggered. Also, in all Azure public regions, if a user <a href="/en-us/azure/azure-monitor/essentials/activity-log" data-linktype="absolute-path">collects Recovery Services activity logs into a Log Analytics workspace</a>, these logs don't appear.</li>
</ul>
<p>Use a Log Analytics workspace for monitoring and alerting at scale for all your workloads that are protected by Azure Backup.</p>
