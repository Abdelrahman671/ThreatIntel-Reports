    <h2>Executive summary</h2>
<ul>
<li><p>Akamai researcher Ben Barnea found two vulnerabilities in Microsoft Windows, which were assigned <a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2023-35384" target="_blank" rel="nofollow noreferrer">CVE-2023-35384</a> and <a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2023-36710" target="_blank" rel="nofollow noreferrer">CVE-2023-36710</a>.</p>
</li>
<li><p><b>An attacker on the internet can chain the vulnerabilities together to create a full, zero-click remote code execution (RCE) exploit against Outlook clients.</b></p>
</li>
<li><p>The first vulnerability lies in the parsing of a path by the <i>MapUrlToZone</i> function. Exploiting this vulnerability requires sending a crafted email to an Outlook client, which in turn will download a special sound file from an attacker-controlled server.</p>
</li>
<li><p>The second vulnerability lies in the Audio Compression Manager (ACM). This vulnerability is exploited when the downloaded sound file is autoplayed, and it can lead to code execution on the victim machine. This vulnerability is described in detail in <a href="/blog/security-research/chaining-vulnerabilities-to-achieve-rce-part-two" target="_self">part 2</a> of this blog post.</p>
</li>
<li><p>The vulnerabilities were responsibly disclosed to Microsoft and addressed on the <a href="/blog/security-research/akamai-perspective-patch-tuesday-august-2023" target="_self">August 2023</a> and <a href="/blog/security-research/akamai-perspective-patch-tuesday-october-2023" target="_self">October 2023</a> Patch Tuesdays.</p>
</li>
<li><p>Windows machines with the October 2023 software update installed are protected from these vulnerabilities. Additionally, Outlook clients that use Exchange servers patched with March 2023 software update are protected against the abused feature.</p>
</li>
</ul>

    <h2>Overview</h2>
<p>Among the vulnerabilities addressed as part of the March 2023 Patch Tuesday was a <a href="/blog/security-research/important-outlook-vulnerability-bypass-windows-api" target="_self">critical Outlook vulnerability</a> assigned <a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2023-23397" target="_blank" rel="nofollow noreferrer">CVE-2023-23397</a>, which was exploited in the wild by Forest Blizzard, which Microsoft has identified as a Russian state-sponsored threat actor. In December 2023, Microsoft, along with the Polish Cyber Command (DKWOC), <a href="https://twitter.com/MsftSecIntel/status/1731626192300634585" target="_blank" rel="nofollow noreferrer">published</a> that they had seen recent exploitation attempts of the vulnerability by the same threat actor. The vulnerability allowed an attacker to coerce an Outlook client to connect to the attacker’s server. As part of this connection, the client sends their NTLM credentials to the attacker, who can then crack them offline, or use them in a relay attack. This vulnerability could be exploited remotely over the internet without any user interaction (zero click).</p>
<p>After the patch for this vulnerability was released, we found a bypass that we described in a <a href="/blog/security-research/important-outlook-vulnerability-bypass-windows-api" target="_self">previous blog post</a>. This bypass was fixed on the May 2023 Patch Tuesday. In that publication, we recommended that Microsoft remove the abused feature, as it introduces a vast and complex attack surface. Since the feature remains in Outlook, we decided to investigate it further.</p>
<p><b>Eventually, we were able to achieve a full RCE vulnerability  chain on Outlook</b>. We found another bypass to the original Outlook vulnerability — a bypass that once again allowed us to coerce the client to connect to an attacker-controlled server and download a malicious sound file. Then, we managed to find a vulnerability in the Windows media parsing library that is used to process and play any audio file in general and Outlook’s notification sounds in particular. An attacker chaining these vulnerabilities together can achieve a zero-click RCE on vulnerable Outlook clients.</p>
<p>This two-part blog post series will present the research we conducted to find the two vulnerabilities. This first part will focus on the previous bypass and a new one. In <a href="/blog/security-research/chaining-vulnerabilities-to-achieve-rce-part-two" target="_self">part 2</a>, we will describe the media parsing vulnerability we found.</p>

    <h2>The original vulnerability</h2>
<p>The Outlook vulnerability that was patched in March is triggered when an attacker sends an email containing a reminder with a custom notification sound. This custom sound is specified by the attacker as a path, using the extended <a href="https://learn.microsoft.com/en-us/office/client-developer/outlook/mapi/mapi-property-overview" target="_blank" rel="nofollow noreferrer">MAPI</a> property <i>PidLidReminderFileParameter</i>. An attacker can specify a UNC path that would cause the client to retrieve the sound file from any SMB server. As part of the connection to the remote SMB server, the Net-NTLMv2 hash is sent in a negotiation message (Figure 1). </p>

    <p><b>To fix the issue, the code now calls <i>MapUrlToZone</i> to classify the path as intranet, local, or internet. If the URL points to a resource on the internet, the default reminder sound is used instead of the custom one.</b></p>

    <h2>Finding a bypass</h2>
<p>After the mitigation was put in place, we asked ourselves whether it was possible to bypass it.</p>
<p>In this context, a bypass means a path that will both pass the locality test and be used by Outlook to download the sound file from a remote location. In other words, <b>we need to find a path that <i>MapUrlToZone</i> would consider as non-internet, but that <i>CreateFile</i> would treat as an internet domain</b>.</p>
<p>To find such a bypass, we needed to fully understand the inner workings of the functions, and the operations done as part of the path parsing.</p>

</div>

<div class="text text__pb-20 aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-d0d65a62ff&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-12-14T03:47:59Z&#34;,&#34;xdm:text&#34;:&#34;&lt;h3>Windows paths and URLs&lt;/h3>\r\n&lt;p>There are many different types of &lt;a href=\&#34;https://learn.microsoft.com/en-us/dotnet/standard/io/file-path-formats\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>DOS paths&lt;/a> in Windows, and much research has been conducted on them and their conversion to NT paths. We will not cover the different Windows path types here; &lt;a href=\&#34;https://googleprojectzero.blogspot.com/2016/02/the-definitive-guide-on-win32-to-nt.html\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>James Forshaw&#39;s blog post&lt;/a> thoroughly covers this topic.&lt;/p>\r\n&lt;p>Let’s go back to our functions of interest. &lt;i>CreateFile&lt;/i> receives a Windows path; &lt;i>MapUrlToZone&lt;/i> (as hinted by its name) can be passed either a URL or a path. To find a bypass, we first need to understand what path types are supported by each function (or both).&lt;/p>\r\n&lt;p>&lt;b>Note&lt;/b>: &lt;i>CreateFile&lt;/i> and &lt;i>MapUrlToZone&lt;/i> do not process paths themselves; instead, they use other WinAPI functions for this purpose. For the sake of brevity, we’ll use &lt;i>CreateFile&lt;/i> and &lt;i>MapUrlToZone&lt;/i> to refer to their underlying path-parsing functions.&lt;/p>\r\n&#34;}}">
    <h3>Windows paths and URLs</h3>
<p>There are many different types of <a href="https://learn.microsoft.com/en-us/dotnet/standard/io/file-path-formats" target="_blank" rel="nofollow noreferrer">DOS paths</a> in Windows, and much research has been conducted on them and their conversion to NT paths. We will not cover the different Windows path types here; <a href="https://googleprojectzero.blogspot.com/2016/02/the-definitive-guide-on-win32-to-nt.html" target="_blank" rel="nofollow noreferrer">James Forshaw's blog post</a> thoroughly covers this topic.</p>
<p>Let’s go back to our functions of interest. <i>CreateFile</i> receives a Windows path; <i>MapUrlToZone</i> (as hinted by its name) can be passed either a URL or a path. To find a bypass, we first need to understand what path types are supported by each function (or both).</p>
<p><b>Note</b>: <i>CreateFile</i> and <i>MapUrlToZone</i> do not process paths themselves; instead, they use other WinAPI functions for this purpose. For the sake of brevity, we’ll use <i>CreateFile</i> and <i>MapUrlToZone</i> to refer to their underlying path-parsing functions.</p>

    <h2>Detection and mitigation</h2>
<p>Microsoft published <a href="https://www.microsoft.com/en-us/security/blog/2023/03/24/guidance-for-investigating-attacks-using-cve-2023-23397/" target="_blank" rel="nofollow noreferrer">comprehensive guidance</a> for the detection and mitigation of the original Outlook vulnerability. From our observation, all the methods specified are applicable to the new vulnerability as they are not dependent on the URL specified in the <i>PidLidReminderFileParameter</i> property.</p>
<p>We recommend that organizations use <a href="/products/akamai-guardicore-segmentation" target="_self">microsegmentation</a> to block outgoing SMB connections to remote public IP addresses. Furthermore, we recommend that you either disable NTLM in your environment, or add users to the <a href="https://learn.microsoft.com/windows-server/security/credentials-protection-and-management/protected-users-security-group?ocid=magicti_ta_learndoc" target="_blank" rel="nofollow noreferrer">Protected Users group</a>, which prevents the use of NTLM as an authentication mechanism.</p>
<p>Blocking outgoing SMB connections and disabling NTLM can help prevent credentials theft. Yet, when SMB requests fail, Windows falls through to WebDAV if it is enabled. Credentials theft can not be abused through WebDAV, yet downloading the sound file is still possible, which is the second step in our RCE chain.</p>

    <h2>Why stop now?</h2>
<p>In this post, we have detailed the research process that led to the discovery of the two bypasses, including their root-cause analysis. As we’ve shown, Windows path parsing code is complex and often can lead to vulnerabilities. Security researchers who encounter path handling code are encouraged to think about the attack surface it presents.</p>
<p>Other than the bypass of <i>MapUrlToZone</i> in the context of Outlook, we can not rule out the possibility that those vulnerabilities can also lead to <a href="https://redcanary.com/threat-detection-report/techniques/mark-of-the-web-bypass/" target="_blank" rel="nofollow noreferrer">Mark-of-the-Web</a> (MotW) bypasses.</p>
<p>Other than the ability to leak NTLM credentials, we also have the ability to download and play an arbitrary sound file. Now you can read <a href="/blog/security-research/chaining-vulnerabilities-to-achieve-rce-part-two" target="_self">part 2 of this blog series</a>, which  details the sound-parsing vulnerability.</p>
