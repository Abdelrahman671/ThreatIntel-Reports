<p>We recently came across a spike of spam email samples containing GuLoader. This malware was discovered last year in 2019 and became more popular among cyber criminals during the coronavirus outbreak. GuLoader is usually attached to a spam email related to bill payments, wire transfers or COVID malspam (you can see a detailed analysis of the&nbsp;<a href="https://labs.vipre.com/using-covid-19-pandemic-to-distribute-backdoor-infection/">COVID malspam here</a>).&nbsp;GuLoader is written in VB5/6 and compressed in a .rar/.iso file. We can see on the graph below the increase of GuLoader which our customers have received:</p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-new5.png" alt="Data collected from January to April 2020 showing the increase in GuLoader related samples"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 1.0 Data collected from January to April 2020 showing the increase in GuLoader related samples</em></strong></p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-1_censored.png" alt="Spam emails containing GuLoader"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 2.0 Spam emails containing GuLoader</em></strong></p>



<p>Guloader is popular for distributing Remote Access Trojan (<em>RAT</em>) tools. These allow the attackers to control, monitor, or steal information from the infected machine. This malware downloader utilizes cloud hosting services (<em>Microsoft OneDrive or Google Drive</em>) to keep its payload encrypted.</p>



<h2 class="wp-block-heading" id="UnloadingtheGuLoader-DigDeeperInsideofGuLoader">Dig Deeper Inside of GuLoader</h2>



<p>Analyzing the GuLoader sample, the malware is indeed a VB5/6 executable. Also, a compiled Visual Basic sample can be recognized by an imported DLL called&nbsp;<em>MSVBVM60.DLL</em>.</p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-11.png" alt="GuLoader sample written in VB5/6 and the msvbvm60.dll"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 3.0 GuLoader sample written in VB5/6 and the msvbvm60.dll</em></strong></p>



<p>Analyzing further, we&#8217;ve found the malware&#8217;s encrypted malicious code. This malware allocates virtual memory and decrypts the encrypted malicious code using XOR.</p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-Capture1.png" alt="The decryption routine"/></figure></div>


<p class="has-text-align-center"><strong>Figure 4.0 The decryption routine</strong></p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-Capture2.png" alt="The encrypted malicious code (left) and the decrypted malicious code in allocated virtual memory (right)"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 5.0 The encrypted malicious code (left) and the decrypted malicious code in allocated virtual memory (right).</em></strong></p>



<p>The decrypted code will be in virtual memory 0x350000. Checking this memory in memory map, it has read, write, and execute (<em>RWE</em>) access. We&#8217;ve now dumped the decrypted code to conduct analysis.</p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-new4.png" alt="The dumped memory and the familiar strings that were found in the decrypted code"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 6.0 The dumped memory and the familiar strings that were found in the decrypted code</em></strong></p>



<p>Checking the strings on the decrypted code, we can see clearly the cloud hosting service URL that stores the encrypted payload (<em>hxxps://drive[.]google[.]com/uc?export=download&amp;id=19sVk-ZTWHVl3_ITBst1x51qX2L28yNlw</em>). We can also see familiar DLLs like wininet.dll and APIs like InternetOpenA, InternetOpenUrlA, InternetSetOptionA etc. The wininet.dll contains internet related functions like InternetOpenA and these functions will probably be used to connect to the URL that contains the encrypted payload.</p>



<p>Analyzing what&#8217;s inside of the decrypted code, we can see that the malware will find the GetProcAddress function in kernel32.dll because GetProcAddress is important in finding and calling other API functions. In order to do this, the malware will first access the&nbsp;<a href="https://www.aldeid.com/wiki/PEB-Process-Environment-Block" rel="noreferrer noopener" target="_blank">Process Environment Block (<em>PEB</em>)</a>&nbsp;-&gt; LDR data -&gt; InMemoryOrderModuleList and then get the address of the module kernel32.dll.</p>


<div class="wp-block-image">
<figure class="aligncenter is-resized"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-new7.png" alt="Accessing the PEB and getting the address of kernel32.dll" style="width:707px;height:auto"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 7.0 Accessing the PEB and getting the address of kernel32.dll</em></strong></p>



<p>After obtaining the address of kernel32.dll and finding GetProcAddress in kernel32.dll, the malware will resolve the following series of APIs:</p>



<ul>
<li>LoadLibraryA</li>



<li>TerminateProcess</li>



<li>EnumWindows</li>



<li>NtProtectVirtualMemory</li>



<li>NtSetInformationThread</li>



<li>NtAllocateVirtualMemory</li>



<li>DbgBreakPoint</li>



<li>DbgUiRemoteBreakin</li>
</ul>



<p>After this, we ran into some anti-analysis techniques. The anti-analysis was used by malware authors to make it more difficult to analyze the malware.</p>



<p>Here are some of the techniques we encountered:</p>



<ul>
<li>An anti-debugger that&nbsp;hides the thread from the debugger. In order to perform this, the API NtSetInformationThread is needed. They set the second parameter (<em>ThreadInformationClass</em>) to 0x11 which is equivalent to ThreadHideFromDebugger. It will hide the thread from the debugger so it can&#8217;t be easily debugged. For example, the thread will continue to run, but the debugger will not be able to receive any events related to the thread.</li>
</ul>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-12.png" alt="Calling of NtSetInformationThread to hide the thread from the debugger"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 8.0 Calling of NtSetInformationThread to hide the thread from the debugger</em></strong></p>



<ul>
<li>Thread attach in a debugger can be seen in the thread window. On&nbsp;<em>figure 9.0</em>, we can see the before and after the thread is hidden from the debugger. The before part is where we can see the main thread and its thread ID which is 11DC. The after part is where the main thread is hidden from the debugger.</li>
</ul>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-new16.png" alt="Before and after the hiding of thread"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 9.0 Before and after the hiding of thread</em></strong></p>



<ul>
<li>There&#8217;s another technique that will first call&nbsp;the <em>NtProtectVirtualMemory</em>&nbsp;function&nbsp;to set the permission&nbsp;of&nbsp;ntdll&#8217;s .text section as&nbsp;<em>PAGE_EXECUTE_READWRITE</em>. The ntdll.dll contains the following APIs, <em>DbgBreakPoint</em> and <em>DbgUiRemoteBreakin</em>, that will be used to perform anti-attach. The malware prevents the debugger from attaching to a process by hooking the&nbsp;<em>DbgBreakPoint</em> and <em>DbgUiRemoteBreakin</em>&nbsp;functions. For example, it will patch <em>DbgBreakPoint</em>&nbsp;and <em>DbgUIRemoteBreakin</em>&nbsp;functions that will trigger the process to exit or to designate an unknown location. Like in&nbsp;<em>figure 12.0</em>,&nbsp;<em>DbgUIRemoteBreakin</em>&nbsp;will call <em>0x00000000</em> address and exit.</li>
</ul>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-Capture_1.png" alt="Calling of NtProtectVirtualMemory to set the permission of NTDLL.DLL"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 10.0 Calling of NtProtectVirtualMemory to set the permission of NTDLL.DLL</em></strong></p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-Capture_2.png" alt="Patching of DbgBreakPoint and DbgUiRemoteBreakin for anti-attach technique"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 11.0 Patching of DbgBreakPoint and DbgUiRemoteBreakin for anti-attach technique</em></strong></p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-Capture_3.png" alt="Before and after patching the DbgUlRemote Breakin function"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 12.0 Before and after patching the DbgUlRemote Breakin function</em></strong></p>



<p>GuLoader will create a folder in the C:\Users directory and the created folder contains a copy of the malware itself. It will also achieve persistence by modifying the registry key HKCU\Software\Microsoft\Windows\CurrentVersion\RunOnce</p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-new11_censored.jpg" alt="The created folder containing the created malware copy"/></figure></div>


<p class="has-text-align-center"><strong><em>Figure 13.0&nbsp; The created folder containing the created malware copy</em></strong></p>



<p>Now GuLoader implements <a href="https://labs.vipre.com/?s=process+hollowing">process hollowing</a>:</p>


<div class="wp-block-image">
<figure class="aligncenter"><img decoding="async" src="/wp-content/uploads/2024/04/unloading-the-guloader-Untitled-Diagram1.png" alt="The child process (for this sample its RegAsm.exe) downloads and decrypts the encrypted payload"/></figure></div>


<p>The&nbsp;child process (for this sample it&#8217;s <em>RegAsm.exe</em>)&nbsp;downloads and decrypts the encrypted payload from a cloud hosting service,&nbsp;and maps the decrypted payload into memory to execute.</p>


<div class="wp-block-image">
<figure class="aligncenter size-full"><img fetchpriority="high" decoding="async" width="754" height="139" src="https://vipre.com/wp-content/uploads/2024/04/unloading-the-guloader-16.png" alt="The cloud hosting service storing the encrypted payload" class="wp-image-23976" srcset="https://vipre.com/wp-content/uploads/2024/04/unloading-the-guloader-16.png 754w, https://vipre.com/wp-content/uploads/2024/04/unloading-the-guloader-16-300x55.png 300w, https://vipre.com/wp-content/uploads/2024/04/unloading-the-guloader-16-18x3.png 18w" sizes="(max-width: 754px) 100vw, 754px" /></figure></div>


<p class="has-text-align-center"><strong><em>Figure 15.0 The cloud hosting service storing the encrypted payload</em></strong></p>



<p>The common GuLoader payloads are Formbook, NetWire, <a href="https://labs.vipre.com/business-email-compromise-img-file-attachment-contains-remcos/">Remcos</a>, Lokibot etc.</p>
