											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>Devices aren't commonly targeted in identity-based attacks, but <em>can</em> be used to satisfy and trick security controls, or to impersonate users. Devices can have one of four relationships with Microsoft Entra ID:</p>
<ul>
<li><p>Unregistered</p>
</li>
<li><p><a href="../identity/devices/concept-device-registration" data-linktype="relative-path">Microsoft Entra registered</a></p>
</li>
<li><p><a href="../identity/devices/concept-directory-join" data-linktype="relative-path">Microsoft Entra joined</a></p>
</li>
<li><p><a href="../identity/devices/concept-hybrid-join" data-linktype="relative-path">Microsoft Entra hybrid joined</a></p>
</li>
</ul>
<p>Registered and joined devices are issued a <a href="../identity/devices/concept-primary-refresh-token" data-linktype="relative-path">Primary Refresh Token (PRT),</a> which can be used as a primary authentication artifact, and in some cases as a multifactor authentication artifact. Attackers may try to register their own devices, use PRTs on legitimate devices to access business data, steal PRT-based tokens from legitimate user devices, or find misconfigurations in device-based controls in Microsoft Entra ID. With Microsoft Entra hybrid joined devices, the join process is initiated and controlled by administrators, reducing the available attack methods.</p>
<p>For more information on device integration methods, see <a href="../identity/devices/plan-device-deployment" data-linktype="relative-path">Choose your integration methods</a> in the article <a href="../identity/devices/plan-device-deployment" data-linktype="relative-path">Plan your Microsoft Entra device deployment.</a></p>
<p>To reduce the risk of bad actors attacking your infrastructure through devices, monitor</p>
<ul>
<li><p>Device registration and Microsoft Entra join</p>
</li>
<li><p>Non-compliant devices accessing applications</p>
</li>
<li><p>BitLocker key retrieval</p>
</li>
<li><p>Device administrator roles</p>
</li>
<li><p>Sign-ins to virtual machines</p>
</li>
</ul>
<h2 id="where-to-look">Where to look</h2>
<p>The log files you use for investigation and monitoring are:</p>
<ul>
<li><p><a href="../identity/monitoring-health/concept-audit-logs" data-linktype="relative-path">Microsoft Entra audit logs</a></p>
</li>
<li><p><a href="../identity/monitoring-health/concept-sign-ins" data-linktype="relative-path">Sign-in logs</a></p>
</li>
<li><p><a href="/en-us/purview/audit-solutions-overview" data-linktype="absolute-path">Microsoft 365 Audit logs</a></p>
</li>
<li><p><a href="/en-us/azure/key-vault/general/logging?tabs=Vault" data-linktype="absolute-path">Azure Key Vault logs</a></p>
</li>
</ul>
<p>From the Azure portal, you can view the Microsoft Entra audit logs and download as comma-separated value (CSV) or JavaScript Object Notation (JSON) files. The Azure portal has several ways to integrate Microsoft Entra logs with other tools that allow for greater automation of monitoring and alerting:</p>
<ul>
<li><p><strong><a href="/en-us/azure/sentinel/overview" data-linktype="absolute-path">Microsoft Sentinel</a></strong> â enables intelligent security analytics at the enterprise level by providing security information and event management (SIEM) capabilities.</p>
</li>
<li><p><strong><a href="https://github.com/SigmaHQ/sigma/tree/master/rules/cloud/azure" data-linktype="external">Sigma rules</a></strong> - Sigma is an evolving open standard for writing rules and templates that automated management tools can use to parse log files. Where Sigma templates exist for our recommended search criteria, we've added a link to the Sigma repo. The Sigma templates aren't written, tested, and managed by Microsoft. Rather, the repo and templates are created and collected by the worldwide IT security community.</p>
</li>
<li><p><strong><a href="/en-us/azure/azure-monitor/overview" data-linktype="absolute-path">Azure Monitor</a></strong> â enables automated monitoring and alerting of various conditions. Can create or use workbooks to combine data from different sources.</p>
</li>
<li><p><strong><a href="/en-us/azure/event-hubs/event-hubs-about" data-linktype="absolute-path">Azure Event Hubs</a> -integrated with a SIEM</strong>- <a href="../identity/monitoring-health/howto-stream-logs-to-event-hub" data-linktype="relative-path">Microsoft Entra logs can be integrated to other SIEMs</a> such as Splunk, ArcSight, QRadar, and Sumo Logic via the Azure Event Hubs integration.</p>
</li>
<li><p><strong><a href="/en-us/defender-cloud-apps/what-is-defender-for-cloud-apps" data-linktype="absolute-path">Microsoft Defender for Cloud Apps</a></strong> â enables you to discover and manage apps, govern across apps and resources, and check your cloud appsâ compliance.</p>
</li>
<li><p><strong><a href="../id-protection/concept-workload-identity-risk" data-linktype="relative-path">Securing workload identities with Microsoft Entra ID Protection</a></strong> - Used to detect risk on workload identities across sign-in behavior and offline indicators of compromise.</p>
</li>
</ul>
<p>Much of what you'll monitor and alert on are the effects of your Conditional Access policies. You can use the <a href="../identity/conditional-access/howto-conditional-access-insights-reporting" data-linktype="relative-path">Conditional Access insights and reporting workbook</a> to examine the effects of one or more Conditional Access policies on your sign-ins, and the results of policies including device state. This workbook enables you to view a summary, and identify the effects over a specific time period. You can also use the workbook to investigate the sign-ins of a specific user.</p>
<p>The rest of this article describes what we recommend you monitor and alert on, and is organized by the type of threat. Where there are specific pre-built solutions we link to them or provide samples following the table. Otherwise, you can build alerts using the preceding tools.</p>
<h2 id="device-registrations-and-joins-outside-policy">Device registrations and joins outside policy</h2>
<p>Microsoft Entra registered and Microsoft Entra joined devices possess primary refresh tokens (PRTs), which are the equivalent of a single authentication factor. These devices can at times contain strong authentication claims. For more information on when PRTs contain strong authentication claims, see <a href="../identity/devices/concept-primary-refresh-token" data-linktype="relative-path">When does a PRT get an MFA claim</a>? To keep bad actors from registering or joining devices, require multi-factor authentication (MFA) to register or join devices. Then monitor for any devices registered or joined without MFA. Youâll also need to watch for changes to MFA settings and policies, and device compliance policies.</p>

<td>Alert when: Any device registered or joined without MFA<br><a href="https://github.com/Azure/Azure-Sentinel/blob/master/Hunting%20Queries/SigninLogs/SuspiciousSignintoPrivilegedAccount.yaml" data-linktype="external">Microsoft Sentinel template</a><br><a href="https://github.com/SigmaHQ/sigma/tree/master/rules/cloud/azure" data-linktype="external">Sigma rules</a></td>

</code></pre>
<p>You can also use <a href="/en-us/mem/intune/protect/device-compliance-get-started" data-linktype="absolute-path">Microsoft Intune to set and monitor device compliance policies</a>.</p>
<h2 id="non-compliant-device-sign-in">Non-compliant device sign-in</h2>
<p>It might not be possible to block access to all cloud and software-as-a-service applications with Conditional Access policies requiring compliant devices.</p>

Microsoft Entra ID can <a href="/en-us/windows/client-management/azure-active-directory-integration-with-mdm" data-linktype="absolute-path">integrate with MDM</a> to enforce device compliance with corporate policies, and can report a deviceâs compliance status.</p>

<h3 id="use-loganalytics-to-query">Use LogAnalytics to query</h3>
<p><strong>Sign-ins by non-compliant devices</strong></p>
<pre><code>SigninLogs
| where DeviceDetail.isCompliant == false
| where ConditionalAccessStatus == "success"
</code></pre>
<p><strong>Sign-ins by unknown devices</strong></p>
<pre><code>
SigninLogs
| where isempty(DeviceDetail.deviceId)
| where AuthenticationRequirement == "singleFactorAuthentication"
| where ResultType == "0"
| where NetworkLocationDetails == "[]"
</code></pre>
<h2 id="stale-devices">Stale devices</h2>
<p>Stale devices include devices that haven't signed in for a specified time period. Devices can become stale when a user gets a new device or loses a device, or when a Microsoft Entra joined device is wiped or reprovisioned. Devices might also remain registered or joined when the user is no longer associated with the tenant. Stale devices should be removed so the primary refresh tokens (PRTs) cannot be used.</p>
