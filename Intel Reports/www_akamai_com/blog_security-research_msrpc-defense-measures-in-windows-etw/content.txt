    <p>We’re continuing our <a href="/blog/security-research/msrpc-security-mechanisms" target="_self">dive into MS-RPC</a>, with yet another blog post. But this time, instead of focusing on the offensive side and looking for vulnerabilities, we’ll discuss some of Windows built-in defensive capabilities. We’ll see how to utilize them to catch a glimpse of what is going on under the RPC hood, hopefully detecting nefarious activity along the way.</p>
<p>MS-RPC is integral to the Windows operating system, which consequently means that many lateral movement paths go over it. Attacks like <a href="https://attack.mitre.org/techniques/T1021" target="_blank" rel="nofollow noreferrer">PSExec</a>, <a href="https://attack.mitre.org/techniques/T1053/005/" target="_blank" rel="nofollow noreferrer">Remote Scheduled Task</a>, <a href="https://attack.mitre.org/techniques/T1003/006/" target="_blank" rel="nofollow noreferrer">DCSync</a>, and <a href="https://github.com/topotam/PetitPotam" target="_blank" rel="nofollow noreferrer">PetitPotam</a> are all carried out over MS-RPC protocols and it can be difficult to discern benign from malicious network traffic, with just traditional network metadata (source and destination port and IP address; sometimes also process information).</p>
<p>We will bolster our visibility with the Event Tracing for Windows (ETW), a built-in tracing and monitoring mechanism in the Windows operating system. ETW provides us with a treasure trove of information, specifically for RPC, especially when compared with traditional network metadata.</p>
<p>The concept of tracking RPC events by using its ETW provider isn’t new, and there are many good resources and tools already available for it (we link to some of them in our <a href="#references" target="_self">References</a> section; let us know if you’re aware of any additional ones). <b>Most existing research focuses on client-side events, which attackers can tamper with or modify the program to circumvent logging entirely</b>. Instead, we’ll focus on detecting attacks using server-side events, which are outside the attackers’ reach, and see the unique considerations we need to parse them.</p>
<p>In this post, we’ll see how to consume and parse events from the RPC ETW provider, and how to scan events for potentially malicious activity. Using the ETW provider, we can also see the exact operation that is being requested, which gives us a much finer granularity with which we can detect attacks. We’ll see exactly how that works in a bit.</p>

      <h2 class="cmp-title__text">Remind me: What’s RPC?<br />

<div class="text text__pb-20 aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-53265a539a&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-08-03T03:50:50Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>RPC stands for remote procedure call, and it’s a form of interprocess communication (IPC). Specifically, this protocol is designed to allow remote function invocation between different processes. In our case, we’ll focus on Microsoft’s implementation, &lt;a href=\&#34;https://learn.microsoft.com/en-us/windows/win32/rpc/rpc-start-page\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>MS-RPC&lt;/a>.&lt;/p>\r\n&lt;p>MS-RPC is designed as a client-server model. The server defines the interface it will expose using the interface definition language (IDL). Inside the IDL we have a universally unique identifier (UUID) for the interface, as well as the function definitions for the functions it exposes (Figure 1).&lt;/p>\r\n&#34;}}">
    <p>RPC stands for remote procedure call, and it’s a form of interprocess communication (IPC). Specifically, this protocol is designed to allow remote function invocation between different processes. In our case, we’ll focus on Microsoft’s implementation, <a href="https://learn.microsoft.com/en-us/windows/win32/rpc/rpc-start-page" target="_blank" rel="nofollow noreferrer">MS-RPC</a>.</p>
<p>MS-RPC is designed as a client-server model. The server defines the interface it will expose using the interface definition language (IDL). Inside the IDL we have a universally unique identifier (UUID) for the interface, as well as the function definitions for the functions it exposes (Figure 1).</p>

    <p>Communication is carried over certain transport protocols, and each transport has its own kind of endpoint (Figure 2). The best way to explain this is with an example: RPC communication can be carried over TCP, in which case the transport is TCP and the endpoint is the TCP socket, identified by a port number.</p>

<div class="text aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-3536956cc4&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-08-03T04:45:18Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>&lt;a href=\&#34;https://github.com/topotam/PetitPotam/blob/main/PetitPotam.py\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>PetitPotam&lt;/a> is an authentication coercion attack on the Encrypted File System (EFS) service.     <p><a href="https://github.com/topotam/PetitPotam/blob/main/PetitPotam.py" target="_blank" rel="nofollow noreferrer">PetitPotam</a> is an authentication coercion attack on the Encrypted File System (EFS) service. Basically, attackers can connect to the EFS RPC interface (<a href="https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-efsr/08796ba8-01c8-4872-9221-1000ec2eff31" target="_blank" rel="nofollow noreferrer">MS-EFSR</a>) and tell it to open a remote file specified by a UNC path. It would then trigger an outbound SMB connection with authentication that the attacker can then relay.</p>
<p>When the attack PoC was released, it used <i>EfsRpcOpenFileRaw</i> (opnum 0), which has since been patched. <a href="https://twitter.com/topotam77" target="_blank" rel="nofollow noreferrer">Topotam</a>, the researcher who found the vulnerability, also found that <i>EfsRpcEncryptFileSrv</i> (opnum 4) had the same flaw.</p>

<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-9f6b56b550&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-08-03T04:47:36Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>Even though there’s a lot of information to be gleaned from the RPC ETW provider, it is not a one-stop shop for all of our security needs. This means that lateral movement detection via the ETW provider is still only a heuristics-based approach, though with a lot more context than traditional network four-tuples.&lt;/p>\r\n&lt;p>It is also a pure detection method, and can’t be used to stop or respond to attacks. Microsoft does provide us another built-in defense mechanism for RPC — the RPC filter in the Windows firewall. You can read more about that filtering mechanism and learn how to use it in our &lt;a href=\&#34;/content/akamai/en/blog/security/guide-rpc-filter.html\&#34; target=\&#34;_self\&#34;>Definitive Guide to the Remote Procedure Call (RPC) Filter post&lt;/a>.&lt;/p>\r\n&#34;}}">
    <p>Even though there’s a lot of information to be gleaned from the RPC ETW provider, it is not a one-stop shop for all of our security needs. While knowing which operation is requested in each network flow is valuable data, the most crucial information, what data was passed in each request, isn’t logged. This means that lateral movement detection via the ETW provider is still only a heuristics-based approach, though with a lot more context than traditional network four-tuples.</p>
<p>It is also a pure detection method, and can’t be used to stop or respond to attacks. Microsoft does provide us another built-in defense mechanism for RPC — the RPC filter in the Windows firewall. You can read more about that filtering mechanism and learn how to use it in our <a href="/blog/security/guide-rpc-filter" target="_self">Definitive Guide to the Remote Procedure Call (RPC) Filter post</a>.</p>

    <p>The RPC ETW provider isn’t some new addition to Windows, but it’s been mostly neglected in consideration for network defense. There are some tools out there that interact and consume events with it, but they are mostly aimed at security researchers, and focus less on the network side of things.</p>
<p>In this post, we’ve discussed how we can use the RPC ETW provider, coupled with the TCP and SMB providers, to glean visibility into the requested RPC operations that come from the network.
