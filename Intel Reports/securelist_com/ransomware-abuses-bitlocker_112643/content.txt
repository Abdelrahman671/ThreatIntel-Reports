													<h2 id="introduction">Introduction</h2>
<p>Attackers always find creative ways to bypass defensive features and accomplish their goals. This can be done with packers, crypters, and code obfuscation. However, one of the best ways of evading detection, as well as maximizing compatibility, is to use the operating system&#8217;s own features. In the context of ransomware threats, one notable example is leveraging exported functions present in the cryptography DLL <strong>ADVAPI32.dll</strong>, such as <strong>CryptAcquireContextA</strong>, <strong>CryptEncrypt</strong>, and <strong>CryptDecrypt</strong>. In this way, the adversaries can make sure that the malware can run and simulate normal behavior in various versions of the OS that support this DLL.</p>
<p>Although this seems smart enough, another clever technique caught our attention in a recent incident response engagement: using the native BitLocker feature to encrypt entire volumes and stealing the decryption key. The original purpose of BitLocker is to address the risks of data theft or exposure from lost, stolen, or improperly decommissioned devices. Nonetheless, threat actors have found out that this mechanism can be repurposed for malicious ends to great effect.</p>
<p>In that incident, the attackers were able to deploy and run an advanced VBS script that took advantage of BitLocker for unauthorized file encryption. We spotted this script and its modified versions in Mexico, Indonesia, and Jordan. In the sections below, we analyze in detail the malicious code obtained during our incident response effort and provide tips for mitigating this kind of threat.</p>
<p>This is not the first time we have seen BitLocker used for encrypting drives and demanding a ransom. Previously, attackers <a href="https://www.microsoft.com/en-us/security/blog/2022/09/07/profiling-dev-0270-phosphorus-ransomware-operations/" target="_blank" rel="noopener">used this Microsoft utility</a> to encrypt critical systems after accessing and controlling these. In this case, however, the adversary took additional steps to maximize the damage from the attack and hinder an effective response to the incident.</p>
<h2 id="vbscript-analysis">VBScript analysis</h2>
<p>One interesting fact is that the attackers did not bother to obfuscate the bulk of the code, as threat actors typically do. The most plausible explanation for this is that they already had full control of the target system when the script was executed. It is stored at <strong>C:\ProgramData\Microsoft\Windows\Templates\</strong> as <strong>Disk.vbs. </strong>Its first lines contain a function that converts a string to its binary representation using an <strong>ADODB.Stream</strong> object. This function is later used for encoding data to be sent in an HTTP POST request.</p>
<div id="attachment_112645" style="width: 817px" class="wp-caption aligncenter"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22152755/Bitlocker_abuse_01.png" class="magnificImage"><img loading="lazy" decoding="async" aria-describedby="caption-attachment-112645" class="size-full wp-image-112645" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22152755/Bitlocker_abuse_01.png" alt="Stream_StringToBinary function" width="807" height="502" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22152755/Bitlocker_abuse_01.png 807w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22152755/Bitlocker_abuse_01-300x187.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22152755/Bitlocker_abuse_01-768x478.png 768w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22152755/Bitlocker_abuse_01-563x350.png 563w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22152755/Bitlocker_abuse_01-740x460.png 740w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22152755/Bitlocker_abuse_01-450x280.png 450w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22152755/Bitlocker_abuse_01-800x498.png 800w" sizes="(max-width: 807px) 100vw, 807px" /></a><p id="caption-attachment-112645" class="wp-caption-text">Stream_StringToBinary function</p></div>
<p>The first step by the main function of the script is to use Windows Management Instrumentation (WMI) to query information about the operating system with the help of the <strong>Win32_OperatingSystem</strong> class. For each object within the query results, the script checks if the current domain is different from the target. If it is, the script finishes automatically. After that, it checks if the name of the operating system contains &#8220;xp&#8221;, &#8220;2000&#8221;, &#8220;2003&#8221;, or &#8220;vista&#8221;, and if the Windows version matches any one of these, the script finishes automatically and deletes itself.</p>

The likely reason the malware does not try to perform same operations on network drives (DriveType = 4) is to avoid triggering detection tools on the network.</p>
<p>To resize local drives in Windows Server 2008 or 2012, the script checks the primary boot partition and saves this information. It saves the index of the different partitions and then performs the following actions using <strong>diskpart</strong>:</p>
<ul>
<li>Shrink the size of each non-boot partition by 100 MB. This creates 100 MB in unallocated space in each partition other than the boot volume;</li>
<li>Split the unallocated space into new 100 MB primary partitions;</li>
<li>Format the partitions with the override option, which forces the volume to dismount first if necessary, and assigns a file system and a drive letter to each;</li>
<li>Activate the partitions;</li>
<li>If the shrink procedure was successful, save &#8220;ok&#8221; as a variable, so the script can continue.</li>
</ul>

<p>After that, the malware disables the protectors used to secure BitLocker&#8217;s encryption key and deletes them. The deletion method may vary depending on the version of the OS. In a Windows Server 2008 or Windows 7 scenario, this is accomplished via VBS features, after which the script uses PowerShell to force the deletion of the protectors.</p>
<p>Having completed the deletion, it enables the use of a numerical password as a protector and the encryption feature.</p>

<p>As the next step, the 64-character encryption key is generated by the malware using a random multiplication and replacement of the following elements:</p>
<ul>
<li>A variable with the numbers 0–9;</li>
<li>The famous pangram, &#8220;The quick brown fox jumps over the lazy dog&#8221;, in lowercase and uppercase, which contains every letter of the English alphabet;</li>
<li>Special characters.</li>
</ul>
<p>The randomness of this password is accomplished by a seed made of various elements of the affected system, such as used memory and network statistics. Later, this information is sent to the attacker. We tested the key generation logic in our environment, and with a slight modification of the script, we were able to see the generated password.</p>

<h2 id="artifacts-and-digital-forensics">Artifacts and digital forensics</h2>
<p>As the local activity performed by the script includes cleaning up its traces, clearing some logs and the tasks created for execution, and finally, encrypting the whole drive, it was not easy to get forensic artifacts to identify the malicious activities and to find opportunities for decryption.</p>
<p>Fortunately, some of the script content and commands executed were registered and logged by a third-party service, and these were collected for analysis. This allowed us to obtain the secure strings to which the encryption keys were converted from some of the affected systems.</p>
<div id="attachment_112664" style="width: 1034px" class="wp-caption aligncenter"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20.png" class="magnificImage"><img loading="lazy" decoding="async" aria-describedby="caption-attachment-112664" class="size-large wp-image-112664" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20-1024x126.png" alt="Secure strings obtained" width="1024" height="126" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20-1024x126.png 1024w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20-300x37.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20-768x95.png 768w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20-1536x189.png 1536w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20-740x91.png 740w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20-1600x197.png 1600w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20-800x99.png 800w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154735/Bitlocker_abuse_20.png 1703w" sizes="(max-width: 1024px) 100vw, 1024px" /></a><p id="caption-attachment-112664" class="wp-caption-text">Secure strings obtained</p></div>
<p>Elsewhere, we attempted to collect network logs where the POST requests to the C2 were stored. However, the most common configuration for web activity logging includes GET but unfortunately not POST requests.</p>
<p>We did finally obtain the POST requests, but this was very challenging. The case provides justification for logging POST traffic and ensuring that all critical system activity is forwarded to a central repository with enough space for storing data for the recommended retention period (six or more months) to avoid losing evidence after attackers remove all their traces from the individual systems.</p>
<p>Finally, some systems in the customer&#8217;s infrastructure remained unencrypted and were considered unaffected at first. However, we later found out that they had, in fact, been affected, but BitLocker was not configured in these systems. This made it possible for us to obtain the script itself, analyze its behavior and collect further evidence.</p>
<h2 id="recovery">Recovery</h2>
<p>While we could obtain some of the passphrases and fixed values implemented by the threat actor to create the encryption keys, the script includes some variable values and those are different for each single affected system, making the decryption process difficult.</p>
<div id="attachment_112665" style="width: 1034px" class="wp-caption aligncenter"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154825/Bitlocker_abuse_21.png" class="magnificImage"><img loading="lazy" decoding="async" aria-describedby="caption-attachment-112665" class="size-large wp-image-112665" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154825/Bitlocker_abuse_21-1024x166.png" alt="Network information collected for use in the seed" width="1024" height="166" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154825/Bitlocker_abuse_21-1024x166.png 1024w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154825/Bitlocker_abuse_21-300x49.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154825/Bitlocker_abuse_21-768x124.png 768w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154825/Bitlocker_abuse_21-740x120.png 740w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154825/Bitlocker_abuse_21-800x130.png 800w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2024/05/22154825/Bitlocker_abuse_21.png 1370w" sizes="(max-width: 1024px) 100vw, 1024px" /></a><p id="caption-attachment-112665" class="wp-caption-text">Network information collected for use in the seed</p></div>
<h2 id="mitigations">Mitigations</h2>
<p>Companies are encouraged to use BitLocker or other encryption tools (such as VeraCrypt) to protect corporate secrets. However, a few precautions must be taken to avoid the abuse by attackers.</p>
<ul>
<li>Use robust, properly configured EPP solution to detect threats that try to abuse BitLocker;</li>
<li>Implement <a href="https://www.kaspersky.com/enterprise-security/managed-detection-and-response" target="_blank" rel="noopener">Managed Detection and Response (MDR)</a> to proactively scan for threats;</li>
<li>If BitLocker is enabled, make sure you are using a strong password and have the recovery keys stored in a secure location;</li>
<li>Ensure that users have only minimal privileges. This way, they cannot enable encryption features or change registry keys on their own;</li>
<li>Enable network traffic logging and monitoring. Configure the logging of both GET and POST requests. In case of infection, the requests made to the attacker&#8217;s domain may contain passwords or keys;</li>
<li>Monitor for events associated with VBS execution and PowerShell, and save the logged scripts and commands to an external repository storing activity that may be deleted locally;</li>
<li>Make backups frequently, store them offline, and test them.</li>
</ul>
<p>If you need assistance with investigation of a ransomware attack and recovering encrypted data, please contact us at <a href="mailto:gert@kaspersky.com" target="_blank" rel="noopener">gert@kaspersky.com</a>.</p>
<h2 id="conclusion">Conclusion</h2>
<p>Our incident response and malware analysis are evidence that attackers are constantly refining their tactics to evade detection. In this incident, we observed the abuse of the native BitLocker feature for unauthorized data encryption. The VBS script demonstrates that the malicious actor involved in this attack have an excellent understanding of Windows internals. Although the script analysis was not complicated at all, this kind of threat is difficult to detect, since unique strings inside the artifact can be easily modified to bypass YARA rules. Therefore, the best detection method in scenarios like these is behavioral analysis, which correlates different actions performed by the application to reach a verdict.</p>
<p>Kaspersky products detect the threat described in this article with the following verdicts:</p>
<ul>
<li>Trojan.VBS.SAgent.gen;</li>
<li>Trojan-Ransom.VBS.BitLock.gen;</li>
<li>Trojan.Win32.Generic.</li>
</ul>
