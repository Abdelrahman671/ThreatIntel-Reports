<p style="font-size:12px;text-align:left">On 25 March, Kaspersky researchers <a href="https://securelist.com/operation-shadowhammer/89992/">published details</a> of a supply chain compromise involving ASUS, a Taiwan-based computer manufacturer. As part of this compromise, a threat actor pushed malicious code to victims who connected to the company&#8217;s servers using the ASUS Live Update feature used to deliver drivers and other updates (this blog notes that such update platforms are common across all manufacturers). <br><br>The malicious code in question is a first-stage triage tool, and details of the second-stage code have not yet been uncovered. This post documents this first-stage functionality of one of the identified variants, which compares the victim&#8217;s MAC address to a hardcoded list prior to communicating with a C2.</p>



<span id="more-154"></span>



<p><u><strong>Malware Workflow</strong></u></p>



<p style="font-size:12px;text-align:left">Kaspersky provided a malicious hash in its original blog post; however, pivoting using <a href="https://twitter.com/craiu/status/1110180467272232961">content from a tweet from Costin Raiu</a> yields additional files. Examination of these files (either through the VirusTotal platform or manually) indicates that they come in pairs: the smaller executables are hardcoded resources within the larger executables. A full hash list is available at the end of this post, and the hash used for this analysis is below.</p>



<figure class="wp-block-image"><img fetchpriority="high" decoding="async" width="1024" height="416" src="https://norfolkinfosec.com/wp-content/uploads/2019/03/pivot-1024x416.png" alt="" class="wp-image-155" srcset="https://norfolkinfosec.com/wp-content/uploads/2019/03/pivot-1024x416.png 1024w, https://norfolkinfosec.com/wp-content/uploads/2019/03/pivot-300x122.png 300w, https://norfolkinfosec.com/wp-content/uploads/2019/03/pivot-768x312.png 768w, https://norfolkinfosec.com/wp-content/uploads/2019/03/pivot-210x85.png 210w, https://norfolkinfosec.com/wp-content/uploads/2019/03/pivot-1536x624.png 1536w, https://norfolkinfosec.com/wp-content/uploads/2019/03/pivot.png 1580w" sizes="(max-width: 980px) 100vw, 980px" /></figure>



<p style="font-size:12px;text-align:left">MD5: fa83ffde24f149f9f6d1d8bc05c0e023<br>SHA1: b0127ce307589ef48e2658784dd83ef7aa26097b<br>SHA256: c299b6dd210ab5779f3abd9d10544f9cae31cd5c6afc92c0fc16c8f43def7596<br><br>At the start of the malicious workflow for this code, the program dynamically resolves the LoadLibraryEx and GetProcAddress calls, which are in turn used to load additional APIs, including:<br>&#8211; memcpy (ntdll)<br>&#8211; memcmp (ntdll)<br>&#8211; memset (ntdll)<br>&#8211; MD5Init (ntdll)<br>&#8211; MD5Update (ntdll)<br>&#8211; MD5Final (ntdll) <br>&#8211; GetAdaptersAddresses (iphlpapi)<br>&#8211; InternetOpenA (wininit)<br>&#8211; InternetOpenURLA (wininit) <br>&#8211; InternetQueryDataAvailable (wininit)  <br>&#8211; InternetReadFile (wininit)  <br><br>The malware then begins allocating large sections of code to memory and begins the triage process. The first function in this workflow queries the device&#8217;s network adapters through the GetAdaptersAddresses API call resolved above. The network adapter information is allocated into memory. Using the MD5 Init, Update, and Final calls, the malware calculates an MD5 hash of each adapter&#8217;s MAC address and stores this in memory for later comparison. Documentation for these functions can be found <a href="http://manpages.ubuntu.com/manpages/xenial/man3/md5.3bsd.html">here</a>.  <br><br>This workflow is shown below- the orange represents the parameters for the MD5Update call, one of which is the address highlighted in blue of the stored MAC address.</p>



<figure class="wp-block-image"><img decoding="async" width="1024" height="533" src="https://norfolkinfosec.com/wp-content/uploads/2019/03/MD5_Hashing_Adapter-1024x533.png" alt="" class="wp-image-157" srcset="https://norfolkinfosec.com/wp-content/uploads/2019/03/MD5_Hashing_Adapter-1024x533.png 1024w, https://norfolkinfosec.com/wp-content/uploads/2019/03/MD5_Hashing_Adapter-300x156.png 300w, https://norfolkinfosec.com/wp-content/uploads/2019/03/MD5_Hashing_Adapter-768x400.png 768w, https://norfolkinfosec.com/wp-content/uploads/2019/03/MD5_Hashing_Adapter-210x109.png 210w, https://norfolkinfosec.com/wp-content/uploads/2019/03/MD5_Hashing_Adapter-1536x800.png 1536w, https://norfolkinfosec.com/wp-content/uploads/2019/03/MD5_Hashing_Adapter.png 1920w" sizes="(max-width: 980px) 100vw, 980px" /></figure>



<p style="font-size:12px;text-align:left">After these hash values are calculated and stored, the program compares them to a hardcoded set of MD5s. This step determines whether or not the malware will call out to its C2. By performing comparisons of the hashes of these values rather than the values themselves, the malware author increases the difficulty of identifying potential targets. <br><br>At a high level, if no match is found, the C2 routine is bypassed and the malware terminates shortly thereafter. A lower-level workflow for this is shown below:</p>



<figure class="wp-block-image"><img decoding="async" width="1024" height="478" src="https://norfolkinfosec.com/wp-content/uploads/2019/03/Memory_Compare-1024x478.png" alt="" class="wp-image-158" srcset="https://norfolkinfosec.com/wp-content/uploads/2019/03/Memory_Compare-1024x478.png 1024w, https://norfolkinfosec.com/wp-content/uploads/2019/03/Memory_Compare-300x140.png 300w, https://norfolkinfosec.com/wp-content/uploads/2019/03/Memory_Compare-768x359.png 768w, https://norfolkinfosec.com/wp-content/uploads/2019/03/Memory_Compare-210x98.png 210w, https://norfolkinfosec.com/wp-content/uploads/2019/03/Memory_Compare.png 1160w" sizes="(max-width: 980px) 100vw, 980px" /></figure>



<p style="font-size:12px;text-align:left">Should a match be found, the &#8220;test eax eax&#8221; instruction will not set the zero flag, preventing the malware from jumping over the function and initiating the C2 call-out for the next (currently undisclosed) stage of activity:</p>



<figure class="wp-block-image"><img loading="lazy" decoding="async" width="771" height="438" src="https://norfolkinfosec.com/wp-content/uploads/2019/03/InternetOpenURLA.png" alt="" class="wp-image-159" srcset="https://norfolkinfosec.com/wp-content/uploads/2019/03/InternetOpenURLA.png 771w, https://norfolkinfosec.com/wp-content/uploads/2019/03/InternetOpenURLA-300x170.png 300w, https://norfolkinfosec.com/wp-content/uploads/2019/03/InternetOpenURLA-768x436.png 768w, https://norfolkinfosec.com/wp-content/uploads/2019/03/InternetOpenURLA-210x119.png 210w" sizes="auto, (max-width: 771px) 100vw, 771px" /></figure>
