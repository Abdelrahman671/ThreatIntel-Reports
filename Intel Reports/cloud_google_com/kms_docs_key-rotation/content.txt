      Key rotation
      
    </h1>

<div class="devsite-article-body clearfix
  ">

  
    
    


<p>This page discusses key rotation in Cloud Key Management Service. Key rotation is the
process of creating new encryption keys to replace existing keys. By rotating
your encryption keys on a regular schedule or after specific events, you can
reduce the potential consequences of your key being compromised. For specific
instructions to rotate a key, see <a href="/kms/docs/rotate-key">Rotating keys</a>.</p>

<h2 id="why_rotate_keys" data-text="Why rotate keys?" tabindex="-1">Why rotate keys?</h2>

<p>For symmetric encryption, periodically and automatically rotating keys is a
recommended security practice. Some industry standards, such as
<a href="https://www.pcisecuritystandards.org/document_library?category=pcidss&amp;document=pci_dss">Payment Card Industry Data Security Standard</a> (PCI DSS), require the regular
rotation of keys.</p>

<p>Cloud Key Management Service <strong>does not</strong> support automatic rotation of asymmetric keys. See
<a href="#asymmetric">Considerations for asymmetric keys</a> in this document.</p>

<p>Rotating keys provides several benefits:</p>

<ul>
<li><p>Limiting the number of messages encrypted with the same key version helps
prevent attacks enabled by cryptanalysis. Key lifetime
recommendations depend on the key&#39;s algorithm, as well as either the number
of messages produced or the total number of bytes encrypted with the same
key version. For example, the recommended key lifetime for symmetric
encryption keys in Galois/Counter Mode (GCM) is based on the number of
messages encrypted, as noted at
<a href="https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf">https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-38d.pdf</a>.</p></li>
<li><p>In the event that a key is compromised, regular rotation limits the number of
actual messages vulnerable to compromise.</p>

<p><strong>If you suspect that a key version is compromised,
<a href="/kms/docs/enable-disable">disable it</a> and
<a href="/kms/docs/iam">revoke access to it</a> as soon as possible.</strong></p></li>
<li><p>Regular key rotation ensures that your system is resilient to manual rotation,
whether due to a security breach or the need to migrate your application to a
stronger cryptographic algorithm. <strong>Validate your key rotation procedures
before a real-life security incident occurs.</strong></p></li>
</ul>

<p>You can also manually rotate a key, either because it is compromised, or to
modify your application to use a different algorithm.</p>

<h2 id="how_often_to_rotate_keys" data-text="How often to rotate keys" tabindex="-1">How often to rotate keys</h2>

<p>We recommend that you <a href="/kms/docs/rotate-key#automatic">rotate keys automatically</a>
on a regular schedule. A rotation schedule defines the frequency of rotation,
and optionally the date and time when the first rotation occurs. The rotation
schedule can be based on either the key&#39;s age or the number or volume of
messages encrypted with a key version.</p>

<p>Some security regulations require periodic, automatic key rotation. Automatic
key rotation at a defined period, such as every 90 days, increases security with
minimal administrative complexity.</p>

<p>You should also <a href="/kms/docs/rotate-key#manual">manually rotate a key</a> if you
suspect that it has been compromised, or when security guidelines require you to
migrate an application to a stronger key algorithm. You can schedule a manual
rotation for a date and time in the future. Manually rotating a key does not
pause, modify, or otherwise impact an existing automatic rotation schedule for
the key.</p>
<aside class="note"><strong>Note:</strong><span> When you rotate a key, data encrypted with previous key versions <strong>isn&#39;t</strong>
automatically re-encrypted with the new key version. For more information, see
<a href="/kms/docs/re-encrypt-data">re-encrypting data</a>.</span></aside>
<p>Don&#39;t rely on irregular or manual rotation as a primary component of your
application&#39;s security.</p>

<h2 id="after" data-text="After you rotate keys" tabindex="-1">After you rotate keys</h2>

<p>Rotating keys creates new active key versions, but doesn&#39;t re-encrypt your data
and doesn&#39;t disable or delete previous key versions. Previous key versions
remain active and incur costs until they are destroyed. Re-encrypting data
removes your reliance on old key versions, allowing you to destroy them to avoid
incurring additional costs. To learn how to re-encrypt your data, see
<a href="/kms/docs/re-encrypt-data">Re-encrypting data</a>.</p>

<p>You must <a href="/kms/docs/destroy-restore#pre-destroy-checklist">make sure that a key version is no longer in
use</a> before destroying the key
version.</p>
<aside class="warning"><strong>Warning:</strong><span> Destroying a key version that is still in use can cause permanent data
loss. It&#39;s your responsibility to ensure that a key version is safe to destroy.
Google is not responsible for outages, loss of data, or compliance issues that
result from you destroying a key version.</span></aside>
<h2 id="asymmetric" data-text="Considerations for asymmetric keys" tabindex="-1">Considerations for asymmetric keys</h2>

<p>Cloud KMS does not support automatic rotation for asymmetric keys,
because additional steps are required before you can use the new asymmetric key
version.</p>

<ul>
<li><p>For asymmetric keys used for <strong>signing</strong>, you must distribute the public
key portion of the new key version. Afterward, you can specify the
new key version in calls to the <a href="/kms/docs/reference/rest/v1/projects.locations.keyRings.cryptoKeys.cryptoKeyVersions/asymmetricSign"><code translate="no" dir="ltr">CryptoKeyVersions.asymmetricSign</code></a> method
to create a signature, and update applications to use the new key version.</p></li>
<li><p>For asymmetric keys used for <strong>encryption</strong>, you must distribute and
incorporate the public portion of the new key version into applications that
encrypt data, and grant access to the private portion of the new key version,
for applications that decrypt data.</p></li>
</ul>

<h2 id="whats_next" data-text="What's next" tabindex="-1">What's next</h2>

<ul>

<li><a href="/kms/docs/enable-disable">Enable or disable a key</a>.</li>

</ul>
