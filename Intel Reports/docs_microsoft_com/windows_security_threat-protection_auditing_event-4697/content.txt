<p><em><strong>Required Server Roles:</strong></em> None.</p>
<p><em><strong>Minimum OS Version:</strong></em> Windows Server 2016, Windows 10.</p>
<p><em><strong>Event Versions:</strong></em> 0.</p>
<p><em><strong>Field Descriptions:</strong></em></p>
<p><strong>Subject:</strong></p>
<ul>
<li><strong>Security ID</strong> [Type = SID]<strong>:</strong> SID of account that was used to install the service. Event Viewer automatically tries to resolve SIDs and show the account name. If the SID cannot be resolved, you will see the source data in the event.</li>
</ul>
<blockquote>
<p><strong>Note</strong>&nbsp;&nbsp;A <strong>security identifier (SID)</strong> is a unique value of variable length used to identify a trustee (security principal). Each account has a unique SID that is issued by an authority, such as an Active Directory domain controller, and stored in a security database. Each time a user logs on, the system retrieves the SID for that user from the database and places it in the access token for that user. The system uses the SID in the access token to identify the user in all subsequent interactions with Windows security. When a SID has been used as the unique identifier for a user or group, it cannot ever be used again to identify another user or group. For more information about SIDs, see <a href="/en-us/windows/access-protection/access-control/security-identifiers" data-linktype="absolute-path">Security identifiers</a>.</p>
</blockquote>
<ul>
<li><p><strong>Account Name</strong> [Type = UnicodeString]<strong>:</strong> the name of the account that was used to install the service.</p>
</li>
<li><p><strong>Account Domain</strong> [Type = UnicodeString]<strong>:</strong> subjectâs domain or computer name. Formats vary, and include the following:</p>
<ul>
<li><p>Domain NETBIOS name example: CONTOSO</p>
</li>
<li><p>Lowercase full domain name: contoso.local</p>
</li>
<li><p>Uppercase full domain name: CONTOSO.LOCAL</p>
</li>
<li><p>For some <a href="/en-us/windows/security/identity-protection/access-control/security-identifiers" data-linktype="absolute-path">well-known security principals</a>, such as LOCAL SERVICE or ANONYMOUS LOGON, the value of this field is âNT AUTHORITYâ.</p>
</li>
<li><p>For local user accounts, this field will contain the name of the computer or device that this account belongs to, for example: âWin81â.</p>
</li>
</ul>
</li>
<li><p><strong>Logon ID</strong> [Type = HexInt64]<strong>:</strong> hexadecimal value that can help you correlate this event with recent events that might contain the same Logon ID, for example, â<a href="event-4624" data-linktype="relative-path">4624</a>: An account was successfully logged on.â</p>
</li>
</ul>
<p><strong>Service Information:</strong></p>
<ul>
<li><strong>Service Name</strong> [Type = UnicodeString]: the name of installed service.</li>
</ul>
<img src="images/branchcache-properties.png" alt="BrancheCache Properties illustration" width="718" height="673" data-linktype="relative-path">
<ul>
<li><p><strong>Service File Name</strong> [Type = UnicodeString]: This is the fully rooted path to the file that the Service Control Manager will execute to start the service. If command-line parameters are specified as part of the image path, those are logged.</p>
<p>Note that this is the path to the file when the service is created. If the path is changed afterwards, the change is not logged. This would have to be tracked via Process Create events.</p>
</li>
<li><p><strong>Service Type</strong> [Type = HexInt32]: Indicates the <a href="/en-us/dotnet/api/system.serviceprocess.servicetype?cs-lang=csharp&amp;cs-save-lang=1#code-snippet-1" data-linktype="absolute-path">type</a> of service that was registered with the Service Control Manager. It can be one of the following:</p>
</li>

<p>Most services installed are configured to <strong>Auto Load</strong>, so that they start automatically after Services.exe process is started.</p>
<ul>
<li><p><strong>Service Account</strong> [Type = UnicodeString]: The security context that the service will run as when started. Note that this is what was configured when the service was installed, if the account is changed later that is not logged.</p>
<p>The service account parameter is only populated if the service type is a "Win32 Own Process" or "Win32 Share Process" (displayed as "User Mode Service."). Kernel drivers do not have a service account name logged.</p>
<p>If a service (Win32 Own/Share process) is installed but no account is supplied, then LocalSystem is used.</p>
<p>The token performing the logon is inspected, and if it has a SID then that SID value is populated in the event (in the System/Security node), if not, then it is blank.</p>
</li>
</ul>
<h2 id="security-monitoring-recommendations">Security Monitoring Recommendations</h2>
<p>For 4697(S): A service was installed in the system.</p>
<blockquote>
<p><strong>Important</strong>&nbsp;&nbsp;For this event, also see <a href="appendix-a-security-monitoring-recommendations-for-many-audit-events" data-linktype="relative-path">Appendix A: Security monitoring recommendations for many audit events</a>.</p>
</blockquote>
<ul>
<li><p>We recommend monitoring for this event, especially on high value assets or computers, because a new service installation should be planned and expected. Unexpected service installation should trigger an alert.</p>
</li>
<li><p>Monitor for all events where <strong>âService File Nameâ</strong> is not located in <strong>%windir%</strong> or <strong>âProgram Files/Program Files (x86)â</strong> folders. Typically new services are located in these folders.</p>
</li>
</ul>
<!-- -->
<ul>
<li><p>Report all â<strong>Service Type</strong>â equals â<strong>0x1</strong>â, â<strong>0x2</strong>â or â<strong>0x8</strong>â. These service types start first and have almost unlimited access to the operating system from the beginning of operating system startup. These types are very rarely installed.</p>
</li>
<li><p>Report all â<strong>Service Start Type</strong>â equals â<strong>0</strong>â or â<strong>1</strong>â. These service start types are used by drivers, which have unlimited access to the operating system.</p>
</li>
<li><p>Report all â<strong>Service Start Type</strong>â equals â<strong>4</strong>â. It is not common to install a new service in the <strong>Disabled</strong> state.</p>
</li>
<li><p>Report all â<strong>Service Account</strong>â not equals â<strong>localSystem</strong>â, â<strong>localService</strong>â or â<strong>networkService</strong>â to identify services which are running under a user account.</p>
</li>
</ul>
