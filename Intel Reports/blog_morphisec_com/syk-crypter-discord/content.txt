<div class="section post-body">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>With <a href="https://influencermarketinghub.com/discord-stats/#:~:text=Discord%20has%20150%20Million%20Monthly%20Users,-Discord%20has%20now&amp;text=The%20most%20significant%20growth%20came,spend%20chatting%20on%20Discord%20servers."><span>50% more users</span></a> last year than in 2020, the number of people using the community chat platform Discord is growing at a blistering pace. This has led cybercriminals to refine and expand malicious attack use cases for the platform. In this threat research report, Morphisec reveals how threat actors are using Discord as part of an increasingly popular attack chain with a new SYK crypter designed to outwit signature and behavior-based security controls.&nbsp;</p>
<!--more-->
<p>Morphisec’s Threat Labs team is on the cutting edge of threat research in this area. Our researchers previously dissected other Discord-related threats like <a href="https://blog.morphisec.com/nft-buyers-beware-journey-of-a-crypto-scammer-how-to-stop-them"><span>Babadeda </span></a>and <a href="https://engage.morphisec.com/journey-of-a-crypto-scammer-nft-001-threat-report"><span>NFT-001</span></a>. We can report that as Discord has expanded from a gaming messaging app to broader use, it’s being used to distribute a crypter we named SYK.</p>
<p>The attack chain preceding the SYK crypter deployment demonstrates a new evolution of how threat actors abuse Discord's CDN (content delivery network). As a conduit for new, highly innovative crypters, Discord plays an important role in a campaign that starts with targeted phishing emails directed at organizations in various sectors.</p>
<p>The attack chain we saw comprises two main components; a .NET loader (which we refer to as DNetLoader) and a .NET crypter (SYK Crypter). This crypter delivers many malware families, such as AsyncRAT, njRAT, QuasarRAT, WarzoneRAT, NanoCore RAT, and RedLine Stealer, putting organizations in every sector and industry at risk.&nbsp;</p>

<p>Before diving into the analysis, let’s look at the execution chain:</p>
<p style="text-align: center;"><img src="https://blog.morphisec.com/hs-fs/hubfs/Technical%20Analysis1.png?width=406&amp;height=486&amp;name=Technical%20Analysis1.png" alt="Technical Analysis1" width="406" height="486" loading="lazy" style="height: auto; max-width: 100%; width: 406px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Technical%20Analysis1.png?width=203&amp;height=243&amp;name=Technical%20Analysis1.png 203w, https://blog.morphisec.com/hs-fs/hubfs/Technical%20Analysis1.png?width=406&amp;height=486&amp;name=Technical%20Analysis1.png 406w, https://blog.morphisec.com/hs-fs/hubfs/Technical%20Analysis1.png?width=609&amp;height=729&amp;name=Technical%20Analysis1.png 609w, https://blog.morphisec.com/hs-fs/hubfs/Technical%20Analysis1.png?width=812&amp;height=972&amp;name=Technical%20Analysis1.png 812w, https://blog.morphisec.com/hs-fs/hubfs/Technical%20Analysis1.png?width=1015&amp;height=1215&amp;name=Technical%20Analysis1.png 1015w, https://blog.morphisec.com/hs-fs/hubfs/Technical%20Analysis1.png?width=1218&amp;height=1458&amp;name=Technical%20Analysis1.png 1218w" sizes="(max-width: 406px) 100vw, 406px"></p>
<p style="text-align: center;"><em>Malware execution flow</em></p>
<p>This execution flow consists of two stages and a final payload. The first stage is the downloader. It connects to a hard coded Discord CDN endpoint and downloads encrypted data. The data, once decrypted, is the second stage—the crypter. This second stage loads into the memory and is responsible for decrypting the final payload, which is stored as a PE resource. It includes antivirus evasion, persistence setup, and injection of the final payload to a newly initiated process.</p>
<h2><span style="font-size: 16px;">Discord CDN as Malware Distributor</span></h2>
<p><strong>Steps 1-2</strong></p>
<p>If you’re unfamiliar with the Discord CDN, it enables Discord users to create and contribute to topic-based text channels. There, users share photos, videos, voice messages, and executable files, all of which are stored on Discord CDN servers—including malware masquerading as legitimate files.&nbsp;</p>
<p>The URL format for a specific file is as follows:&nbsp;</p>
<p><em>hxxps://cdn.discordapp[.]com/attachments/{ChannelID}/{AttachmentID}/{filename}</em></p>
<p>In this context, the DNetLoader is identified by the filename, a three digit number. Let’s look inside the code:</p>
<p style="text-align: center;"><img src="https://blog.morphisec.com/hs-fs/hubfs/Steps1-2.png?width=624&amp;height=615&amp;name=Steps1-2.png" alt="Steps1-2" width="624" height="615" loading="lazy" style="height: auto; max-width: 100%; width: 624px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Steps1-2.png?width=312&amp;height=308&amp;name=Steps1-2.png 312w, https://blog.morphisec.com/hs-fs/hubfs/Steps1-2.png?width=624&amp;height=615&amp;name=Steps1-2.png 624w, https://blog.morphisec.com/hs-fs/hubfs/Steps1-2.png?width=936&amp;height=923&amp;name=Steps1-2.png 936w, https://blog.morphisec.com/hs-fs/hubfs/Steps1-2.png?width=1248&amp;height=1230&amp;name=Steps1-2.png 1248w, https://blog.morphisec.com/hs-fs/hubfs/Steps1-2.png?width=1560&amp;height=1538&amp;name=Steps1-2.png 1560w, https://blog.morphisec.com/hs-fs/hubfs/Steps1-2.png?width=1872&amp;height=1845&amp;name=Steps1-2.png 1872w" sizes="(max-width: 624px) 100vw, 624px"></p>
<p style="text-align: center;"><em>First stage malicious code</em></p>
<p>The first stage is pretty straightforward. The malware downloads the next stage from Discord CDN where the file name is hardcoded and used as the decryption key. The decryption algorithm is just a subtraction of the file name from each byte in the downloaded data.</p>
<p>Once decoded, the malware loads it into memory and creates an instance of the first exported type. Then the execution moves to the next stage. In other cases, the instance name is explicitly noted, usually with type name “B”.&nbsp;</p>

<p style="text-align: center;"><em>Mapping payload to C2</em></p>

<h2><span style="font-size: 16px;">The SYK Crypter</span></h2>
<p><strong>Steps 3-5</strong></p>
<p>Before diving deeper into the .NET crypter, note that we found that the same crypter was delivered by loaders <strong><em>other than</em></strong> the DNetLoader. However, they all had a resource named SYKSBIKO in common—the encrypted payload. For this reason, we dubbed it the SYK Crypter.</p>
<p>As with other crypters, this crypter has a payload decryption method, control flow manipulation, strings and constant obfuscation, AV detection, persistence, and anti-debugging features. We examine each capability and explain how it’s implemented.</p>
<h3><span style="font-size: 14px; color: #434343;">Configuration Extraction / Strings Obfuscation</span></h3>
<p>The SYK crypter holds its configuration inside an obfuscated string represented as a byte array:</p>
<p style="text-align: center;"><img src="https://blog.morphisec.com/hs-fs/hubfs/Steps3-5.png?width=624&amp;height=180&amp;name=Steps3-5.png" alt="Steps3-5" width="624" height="180" loading="lazy" style="height: auto; max-width: 100%; width: 624px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Steps3-5.png?width=312&amp;height=90&amp;name=Steps3-5.png 312w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5.png?width=624&amp;height=180&amp;name=Steps3-5.png 624w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5.png?width=936&amp;height=270&amp;name=Steps3-5.png 936w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5.png?width=1248&amp;height=360&amp;name=Steps3-5.png 1248w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5.png?width=1560&amp;height=450&amp;name=Steps3-5.png 1560w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5.png?width=1872&amp;height=540&amp;name=Steps3-5.png 1872w" sizes="(max-width: 624px) 100vw, 624px"></p>
<p style="text-align: center;"><em>Encrypted byte array and access functions</em></p>
<p>The crypter starts with a string de-obfuscation technique. Each string can be accessed and used by a predefined function which hardcodes its length and offset in a large byte array. The de-obfuscation algorithm is just XOR with 170 and the current index, so we can use the following Python script:</p>
<p>encrypted = [231, 216, 235, …]&nbsp;</p>
<p>ba_encrypted = bytearray(encrypted)</p>
<p>ba_decrypted = bytearray(encrypted)</p>
<p>for counter, i in enumerate(ba_encrypted):</p>
<p>&nbsp;&nbsp;&nbsp;&nbsp;ba_decrypted[counter] = (i ^ counter ^ 170) &amp; 0xff</p>
<p>A similar method is used as part of an Agent Tesla <a href="https://malgamy.github.io/malware-analysis/Deep-Analysis-Agent-Tesla/"><span>delivery campaign</span></a>.</p>
<p>Among all setting strings inside the configuration, the important ones are the final payload decryption key, list of AV solutions services and process names, and a small .NET delegator (base64 encoded).</p>
<p><img src="https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_2.png?width=624&amp;height=400&amp;name=Steps3-5_2.png" alt="Steps3-5_2" width="624" height="400" loading="lazy" style="height: auto; max-width: 100%; width: 624px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_2.png?width=312&amp;height=200&amp;name=Steps3-5_2.png 312w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_2.png?width=624&amp;height=400&amp;name=Steps3-5_2.png 624w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_2.png?width=936&amp;height=600&amp;name=Steps3-5_2.png 936w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_2.png?width=1248&amp;height=800&amp;name=Steps3-5_2.png 1248w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_2.png?width=1560&amp;height=1000&amp;name=Steps3-5_2.png 1560w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_2.png?width=1872&amp;height=1200&amp;name=Steps3-5_2.png 1872w" sizes="(max-width: 624px) 100vw, 624px"></p>
<p style="text-align: center;"><em>Decrypted configuration</em></p>
<p>As you can see above, several strings are still encrypted. The crypter uses subtraction encryption for those, with the keys also stored as part of the configuration.<img src="https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_3.png?width=624&amp;height=120&amp;name=Steps3-5_3.png" alt="Steps3-5_3" width="624" height="120" loading="lazy" style="height: auto; max-width: 100%; width: 624px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_3.png?width=312&amp;height=60&amp;name=Steps3-5_3.png 312w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_3.png?width=624&amp;height=120&amp;name=Steps3-5_3.png 624w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_3.png?width=936&amp;height=180&amp;name=Steps3-5_3.png 936w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_3.png?width=1248&amp;height=240&amp;name=Steps3-5_3.png 1248w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_3.png?width=1560&amp;height=300&amp;name=Steps3-5_3.png 1560w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_3.png?width=1872&amp;height=360&amp;name=Steps3-5_3.png 1872w" sizes="(max-width: 624px) 100vw, 624px"></p>

<h3><span style="font-size: 14px; color: #434343;">Security Solutions Detection</span></h3>
<p>The crypter checks for the existence of a set of security solutions using the following two methods.&nbsp;</p>
<ul>
<li aria-level="1">By calling GetProcessByName:<br><img src="https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_4.png?width=624&amp;height=41&amp;name=Steps3-5_4.png" alt="Steps3-5_4" width="624" height="41" loading="lazy" style="height: auto; max-width: 100%; width: 624px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_4.png?width=312&amp;height=21&amp;name=Steps3-5_4.png 312w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_4.png?width=624&amp;height=41&amp;name=Steps3-5_4.png 624w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_4.png?width=936&amp;height=62&amp;name=Steps3-5_4.png 936w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_4.png?width=1248&amp;height=82&amp;name=Steps3-5_4.png 1248w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_4.png?width=1560&amp;height=103&amp;name=Steps3-5_4.png 1560w, https://blog.morphisec.com/hs-fs/hubfs/Steps3-5_4.png?width=1872&amp;height=123&amp;name=Steps3-5_4.png 1872w" sizes="(max-width: 624px) 100vw, 624px"></li>

<h3><span style="font-size: 14px; color: #434343;">Final Payload Injection</span></h3>
<h4><span style="font-size: 12px; color: #666666;">Payload Decryption and Deobfuscation</span></h4>
<p>Before moving forward, we need to understand where the final payload is located and how it’s decrypted. We can divide this process into four steps:</p>
<ol>
<li aria-level="1">Read the decryption key from the config—the first element</li>
<li aria-level="1">Read resource bytes from SYKSBIKO.Properties.Resources.a</li>
<li aria-level="1">Use the key to decrypt the resource’s bytes&nbsp;</li>
<li aria-level="1">Deflate the result&nbsp;</li>
</ol>
<p>The final payload decryption algorithm is a bit more complicated than the previous algorithms.</p>
<p>The decryption starts from initializing a new 256 unsigned integer array with its index values.</p>
<p style="text-align: center;"><img src="https://blog.morphisec.com/hs-fs/hubfs/FinalPayloadInjection1.png?width=393&amp;height=96&amp;name=FinalPayloadInjection1.png" alt="FinalPayloadInjection1" width="393" height="96" loading="lazy" style="height: auto; max-width: 100%; width: 393px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/FinalPayloadInjection1.png?width=197&amp;height=48&amp;name=FinalPayloadInjection1.png 197w, https://blog.morphisec.com/hs-fs/hubfs/FinalPayloadInjection1.png?width=393&amp;height=96&amp;name=FinalPayloadInjection1.png 393w, https://blog.morphisec.com/hs-fs/hubfs/FinalPayloadInjection1.png?width=590&amp;height=144&amp;name=FinalPayloadInjection1.png 590w, https://blog.morphisec.com/hs-fs/hubfs/FinalPayloadInjection1.png?width=786&amp;height=192&amp;name=FinalPayloadInjection1.png 786w, https://blog.morphisec.com/hs-fs/hubfs/FinalPayloadInjection1.png?width=983&amp;height=240&amp;name=FinalPayloadInjection1.png 983w, https://blog.morphisec.com/hs-fs/hubfs/FinalPayloadInjection1.png?width=1179&amp;height=288&amp;name=FinalPayloadInjection1.png 1179w" sizes="(max-width: 393px) 100vw, 393px"></p>

<p>As part of the decryption, before XORing the values there is another swapping, as seen earlier. Then an index is calculated from the sum of the swapped values. The encrypted data is XORed with the value of the array inside the index.</p>
<p>The end result is a deflated compressed representation of the final payload. So all that’s left to do is decompress the result and get the final payload.&nbsp;</p>
<h3><span style="font-size: 12px; color: #666666;">Process Hollowing Injection</span></h3>
<p>The SYK crypter uses Process Hollowing as its preferred injection method. It creates a new process—RegAsm.exe or the named process according to the configuration—and injects the decrypted final payload into it.</p>
<p>It's interesting how the WinAPI functions get loaded into memory. The SYK malware uses the .NET Delegator in its configuration to create a delegate for each function.</p>
<p><img src="https://blog.morphisec.com/hs-fs/hubfs/ProcessHollowing1.png?width=624&amp;height=174&amp;name=ProcessHollowing1.png" alt="ProcessHollowing1" width="624" height="174" loading="lazy" style="height: auto; max-width: 100%; width: 624px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/ProcessHollowing1.png?width=312&amp;height=87&amp;name=ProcessHollowing1.png 312w, https://blog.morphisec.com/hs-fs/hubfs/ProcessHollowing1.png?width=624&amp;height=174&amp;name=ProcessHollowing1.png 624w, https://blog.morphisec.com/hs-fs/hubfs/ProcessHollowing1.png?width=936&amp;height=261&amp;name=ProcessHollowing1.png 936w, https://blog.morphisec.com/hs-fs/hubfs/ProcessHollowing1.png?width=1248&amp;height=348&amp;name=ProcessHollowing1.png 1248w, https://blog.morphisec.com/hs-fs/hubfs/ProcessHollowing1.png?width=1560&amp;height=435&amp;name=ProcessHollowing1.png 1560w, https://blog.morphisec.com/hs-fs/hubfs/ProcessHollowing1.png?width=1872&amp;height=522&amp;name=ProcessHollowing1.png 1872w" sizes="(max-width: 624px) 100vw, 624px"></p>
