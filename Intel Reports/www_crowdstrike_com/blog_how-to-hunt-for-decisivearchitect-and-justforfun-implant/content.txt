<div class="cmp-text" id="text-3591a66eda" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
<span><span style="	font-weight: 400;
">The security landscape is constantly developing to provide easier ways to establish endpoint visibility across networks through the use of </span><a href="/en-us/cybersecurity-101/endpoint-security/endpoint-detection-and-response-edr/"><span style="	font-weight: 400;
">endpoint detection and response (EDR)</span></a><span style="	font-weight: 400;
"> utilities. However, certain challenges still remain, particularly as a result of many organizations&#39; need for systems running legacy or proprietary operating systems, such as Solaris. If such systems are not adequately protected using other security controls or unless they can only be accessed by systems with appropriate endpoint-based detection/prevention capabilities, this can cause a gap in visibility for an organization that an adversary could abuse.</span>
<span style="	font-weight: 400;
">On multiple occasions dating back to 2019, the CrowdStrike Services Incident Response team, CrowdStrike Intelligence team and Falcon OverWatch™ team have encountered an adversary targeting global entities, in particular telecommunications companies, to obtain targeted personal user information — for example, call detail records (CDRs) or information relating to specific phone numbers.<p> </p></span>
<span style="	font-weight: 400;
">Similar to the activity cluster reported as </span><a href="/en-us/blog/an-analysis-of-lightbasin-telecommunications-attacks/"><span style="	font-weight: 400;
">LightBasin</span></a><span style="	font-weight: 400;
">, this adversary primarily focuses on Linux and Solaris systems using a custom-built implant tracked by CrowdStrike Intelligence as JustForFun (also publicly known as BPFDoor). While this adversary does interact with Windows systems, mostly during the early stages of an intrusion, CrowdStrike has not yet identified any custom implants geared toward Windows systems. Instead, the adversary relies on publicly available tools, such as ldapdomaindump, or the post-exploitation framework Impacket, to target Windows systems from previously compromised Linux systems.</span>
<span style="	font-weight: 400;
">CrowdStrike Intelligence is currently tracking these intrusions under the DecisiveArchitect activity cluster (also publicly known as Red Menshen); however, this activity is not currently attributed by CrowdStrike to a specific country-nexus. While CrowdStrike has primarily observed the adversary targeting telecommunications companies, other isolated incidents targeting organizations such as logistics entities have also been observed.</span>
<span style="	font-weight: 400;
">DecisiveArchitect exhibits a high degree of operational security as part of their tactics to make it more difficult for defenders to identify and investigate their activity through the use of various defense evasion techniques. While other publicly available research highlights how the implant operates, this blog focuses on methods to hunt for this implant, or implants that may operate in a similar manner, while also highlighting techniques of interest across Solaris systems.</span>
<h2><span style="	font-weight: 400;
">Spoofed Command Lines</span></h2>
<span style="	font-weight: 400;
">DecisiveArchitect utilizes a custom implant tracked by CrowdStrike as JustForFun, which is typically persisted using SysVinit scripts. When executed, the implant overwrites the process command line within the process environment by randomly selecting a new command line from one of ten hard-coded options, listed in Figure 1.</span>
<pre><code>/sbin/udevd -d
/sbin/mingetty /dev/tty6
/usr/sbin/console-kit-daemon --no-daemon
hald-addon-acpi: listening on acpi kernel interface /proc/acpi/event
dbus-daemon --system
hald-runner
pickup -l -t fifo -u
avahi-daemon: chroot helper
/sbin/auditd -n
/usr/lib/systemd/systemd-journald</code></pre>
<p style="	text-align: center;
"><span style="	font-weight: 400;
">Figure 1. Hard-coded options for command-line spoofing in JustForFun</span></p>
<span style="	font-weight: 400;
">When DecisiveArchitect interacts with the implants to establish an interactive shell on a system, the </span><code><span style="	font-weight: 400;
">bash </span></code><span style="	font-weight: 400;
">process spawned by the implant process displays the following command line instead. This makes it appear as if the Postfix queue manager is executing as a way to hide itself from analysts and system administrators:</span>
<code><span style="	font-weight: 400;
">qmgr -l -t fifo -u</span></code>
<span style="	font-weight: 400;
">On Solaris systems, though the executable itself exhibits no mechanism for similar command-line spoofing, DecisiveArchitect achieves similar functionality through the use of </span><code><span style="	font-weight: 400;
">LD_PRELOAD</span></code><span style="	font-weight: 400;
">, such as the following example identified within a SysVinit script:</span>
<code><span style="	font-weight: 400;
">LD_PRELOAD&#61;/lib/librbtinfo.so.1 /usr/lib/vtdaemon -c 16</span></code>
<span style="	font-weight: 400;
">When executed, the process only shows the command line </span><code><span style="	font-weight: 400;
">/usr/lib/vtdaemon -c 16</span></code><span style="	font-weight: 400;
">, where the actual JustForFun implant is the file </span><code><span style="	font-weight: 400;
">/lib/librbtinfo.so.1</span></code><span style="	font-weight: 400;
">.</span>
<span style="	font-weight: 400;
">As recently as April 2022, CrowdStrike observed further variations with regard to DecisiveArchitect&#39;s tactics, techniques and procedures (TTPs), with the actor using the </span><code><span style="	font-weight: 400;
">LD_PRELOAD</span></code><span style="	font-weight: 400;
"> environment variable across Linux systems as well, loading the JustForFun implant, </span><code><span style="	font-weight: 400;
">/lib64/libcaac.so.1</span></code><span style="	font-weight: 400;
">, within the legitimate process </span><code><span style="	font-weight: 400;
">/sbin/agetty</span></code><span style="	font-weight: 400;
">. This highlights a deviation from the standard list of spoofed command lines in Figure 1, likely as part of a further effort to remain undetected and emphasizing the importance of behavioral-based hunting and detection methods.</span>
<span style="	font-weight: 400;
">The spoofed command line appears in commands such as </span><code><span style="	font-weight: 400;
">ps</span></code> <span style="	font-weight: 400;
">that may be used to investigate suspicious activity on the host. The spoofed command line makes it less likely that the process will be treated as suspicious.</span>
<h2><span style="	font-weight: 400;
">Solaris Privilege Escalation Vulnerability Exploitation</span></h2>
<span style="	font-weight: 400;
">DecisiveArchitect targets Solaris systems via publicly available exploit code for CVE-2019-3010, a vulnerability in </span><code><span style="	font-weight: 400;
">xscreensaver</span></code><span style="	font-weight: 400;
">. Binaries used to exploit this vulnerability have usually been observed within a few minutes of the JustForFun implant deployment. CVE-2019-3010 is a logic bug that utilizes the </span><code><span style="	font-weight: 400;
">LD_PRELOAD </span></code><span style="	font-weight: 400;
">technique to facilitate local privilege escalation to the root user on Solaris 11 systems. Proof-of-concept (POC) code is publicly available and was not modified by DecisiveArchitect.<sup>1</sup></span><span style="	font-weight: 400;
"> Table 1 lists two files observed across Solaris systems related to this privilege escalation activity.</span>
<table class="orange" width="624"><tbody><tr class="top-row bg-orange"><td><strong>File Path</strong></td><td><strong>Purpose</strong></td></tr><tr><td>/tmp/getuid.so</td><td>CVE-2019-3010 exploit binary</td></tr><tr><td>/usr/lib/secure/getuid.so</td><td>CVE-2019-3010 exploit binary or log file</td></tr></tbody></table>
<p style="	text-align: center;
"><span style="	font-weight: 400;
">Table 1. Solaris exploitation file details</span></p>
<h2><span style="	font-weight: 400;
">Persistence</span></h2>
<span style="	font-weight: 400;
">The way in which DecisiveArchitect achieves persistence across Linux systems involves the usage of SysVinit scripts (i.e., rc.d/init.d scripts). Instead of simply creating a new script that references the JustForFun implants, DecisiveArchitect uses a more operational security-conscious approach by modifying existing SysVinit scripts to reference a small script file, which then finally references the JustForFun implant. The following highlights an example, including the lines added to the legitimate SysVinit script </span><code><span style="	font-weight: 400;
">/etc/rc.d/init.d/pcscd</span></code><span style="	font-weight: 400;
"> and the script </span><code><span style="	font-weight: 400;
">/etc/sysconfig/pcscd</span></code><span style="	font-weight: 400;
"> referencing the JustForFun implant, </span><code><span style="	font-weight: 400;
">/etc/sysconfig/pcscd.conf</span></code><span style="	font-weight: 400;
">:</span>
<code><b>/etc/rc.d/init.d/pcscd:</b></code>
<ul><li><b>Line 41:</b> <code><span style="	font-weight: 400;
">if &lt; -f /etc/sysconfig/pcscd &gt; ; then</span></code></li><li><b>Line 42:</b> <span style="	font-weight: 400;
"><p> </p><code>/etc/sysconfig/pcscd</code></span></li></ul>
<code><b>/etc/sysconfig/pcscd:</b></code>
<pre><code># Source config
if &lt; -f /etc/sysconfig/pcscd.conf &gt; ; then
    /etc/sysconfig/pcscd.conf
fi</code></pre>
<span style="	font-weight: 400;
">With this method of nested persistence, if an analyst simply reviews a set of SysVinit scripts by themselves, identifying the malicious line associated with the JustForFun implant would likely prove difficult without subsequently reviewing all of the files referenced within the scripts. Additionally, as part of DecisiveArchitect’s continued commitment to operational security, the adversary modifies different legitimate SysVinit scripts across systems, and uses different file names/paths for the implant and associated persistence-related scripts, making it difficult to search across other systems for indicators identified through analysis of a single system.</span>
<h2><span style="	font-weight: 400;
">Detection and Hunting</span></h2>
<span style="	font-weight: 400;
">One of the best ways to detect or hunt for this activity is to utilize EDR technology across supported Linux systems, with machine learning capabilities to detect and prevent the malicious implants, and with hunting capabilities to identify anomalous usage of common Linux system administration utilities or processes running with spoofed command lines (such as the </span><code><span style="	font-weight: 400;
">bash</span></code><span style="	font-weight: 400;
"> process running with the command line showing the Postfix queue manager command line instead). Figures 2 and 3 highlight the CrowdStrike Falcon®<sup>®</sup> platform’s machine learning capabilities and Falcon OverWatch detections associated with the JustForFun implant and JustForFun command-line tool.</span>

<span style="	font-weight: 400;
">But, even if an organization has a significant number of legacy or proprietary systems, already has the adversary burrowed into their network, or simply does not have EDR software deployed across Linux systems, all is not lost.</span>
<h2><span style="	font-weight: 400;
">Hunting for Traffic Signaling Implants</span></h2>
<span style="	font-weight: 400;
">Given the fact the JustForFun implant opens a raw socket in order to wait for the magic packet, the built-in Linux utility </span><code><span style="	font-weight: 400;
">lsof</span></code><span style="	font-weight: 400;
"> can be used to identify running processes with a raw socket open:</span>
<code><span style="	font-weight: 400;
">lsof -RPnl | grep SOCK_RAW | grep IP</span></code>
<span style="	font-weight: 400;
">Even though this command alone cannot solely determine whether the implant is present — as there are legitimate reasons for processes to have raw sockets open — analysts can highlight processes of interest for further investigation. Of particular importance is that DecisiveArchitect’s use of spoofed command lines means that the </span><code><span style="	font-weight: 400;
">lsof </span></code><span style="	font-weight: 400;
">command will report the spoofed command line, as opposed to the actual malicious file, which may make it difficult to determine whether the process is malicious or not through this alone. However, by running the </span><code><span style="	font-weight: 400;
">lsof </span></code><span style="	font-weight: 400;
">command against the process ID without any of the </span><code><span style="	font-weight: 400;
">grep</span></code> <span style="	font-weight: 400;
">filtering, an analyst can list any open files associated with that process, which should reveal the binary. An example of a true positive can be seen below, with the start of the line (i.e., the command line) displaying the start of one of the spoofed command lines listed in Figure 1. It should be noted that DecisiveArchitect can quite easily change these spoofed command lines, so analysts should be conscious of other processes beyond those listed:</span>
<code><span style="	font-weight: 400;
">dbus-daem 1215 0 root 3u pack 11912 0t0 IP type&#61;SOCK_RAW</span></code>
<span style="	font-weight: 400;
">While the </span><code><span style="	font-weight: 400;
">lsof</span></code><span style="	font-weight: 400;
"> command is not part of a default Solaris installation, similar commands exist for obtaining additional details from processes:</span>
<pre><code>for _PIDno in /proc/*; do line&#61;$(pfiles &#34;${_PIDno}&#34;); echo $_PIDno $line | grep bpf; done
for _PIDno in /proc/*; do line&#61;$(pmap &#34;${_PIDno}&#34;); echo $_PIDno $line | grep libpcap; done
for _PIDno in /proc/*; do line&#61;$(pldd &#34;${_PIDno}&#34;); echo $_PIDno $line | grep libpcap; done</code></pre>
<span style="	font-weight: 400;
">The three commands above loop through every process, with the first command looking for a string indicative of a process running with a packet filter, and the other two searching for processes with the </span><code><span style="	font-weight: 400;
">libpcap</span></code><span style="	font-weight: 400;
"> library loaded. As with the </span><code><span style="	font-weight: 400;
">lsof</span></code><span style="	font-weight: 400;
"> command, this alone cannot solely determine whether the implant is present, so an analyst would need to further investigate the specific process to confirm the presence of the implant. DecisiveArchitect’s capability to spoof command lines across Solaris systems also needs to be taken into account when investigating these processes.</span>
<span style="	font-weight: 400;
">When investigating any of these entries, one of the key questions to ask is whether the process in question has any reason to have a raw socket open, to be using a packet filter or to be utilizing the </span><code><span style="	font-weight: 400;
">libpcap</span></code><span style="	font-weight: 400;
"> library. One of the most common false positives relates to systems running processes such as </span><span style="	font-weight: 400;
"><code>tcpdump</code></span><span style="	font-weight: 400;
"> or other packet capture utilities.</span>
<span style="	font-weight: 400;
">While these hunting techniques provide a relatively simple method for identifying DecisiveArchitect activity based on activity observed across multiple intrusions, CrowdStrike expects that DecisiveArchitect will continue development of their implant across both Linux and Solaris platforms, while also improving their techniques regarding operational security of their intrusions to further hinder the ability of a defender to identify or investigate their activity, which might include identifying ways to combat these hunting techniques.</span>
<h2><span style="	font-weight: 400;
">Conclusion</span></h2>
<span style="	font-weight: 400;
">DecisiveArchitect’s operations present a clear and present threat to telecommunications companies, as well as other organizations such as logistics entities. This blog highlights important details about DecisiveArchitect’s implant, their abilities to operate on Solaris systems, and ways to hunt down the adversary’s implants to help organizations identify whether they have fallen victim to this campaign.</span>
