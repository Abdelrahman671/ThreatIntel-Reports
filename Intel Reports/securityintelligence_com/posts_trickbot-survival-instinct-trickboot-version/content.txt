As we kick off 2021, the TrickBot Trojan and the cybercrime syndicate around it are in full swing, operating on the bank fraud front as well as high-stakes ransomware attacks.

If your security team detects TrickBot on networked devices, those should become high priority to clean or reimage as needed. This malware can be a foot in the door for a number of attacks, the worst of which can be an all-out ransomware and extortion operation against your organization.

<html><body><p>October 2020 saw the TrickBot Trojan, a prominent cybercrime gang&rsquo;s tool of choice, suffer a takedown <a href="https://www.zdnet.com/article/microsoft-says-it-took-down-94-of-trickbots-command-and-control-servers/" target="_blank" rel="noopener nofollow" >attempt</a> by security vendors and law enforcement. Unfortunately, the takedown was not effective, and beyond coming back to life shortly after, TrickBot&rsquo;s operators released a new and more persistent version of the malware.</p>
<p>In this post, IBM Trusteer examines the new TrickBot version versus <a href="https://securityintelligence.com/news/trickbot-using-bazarbackdoor-to-gain-full-access-to-targeted-networks/"  target="_blank" rel="noopener nofollow" >its precedent</a> and looks into the components its developers kept or modified.</p>
<h2>TrickBot Returns</h2>
<p>What can be said about the TrickBot banking malware and botnet that has not been covered yet by the security community? After it emerged in 2016, built upon the ruins of the Dyre Trojan, TrickBot&rsquo;s developers have not laid down their cyber-arms. Looking back at the past two years, not only has TrickBot continued to plague the online banking applications that cater to businesses, it has also ventured into collaborations with other elite cybercrime gangs. Additionally, it has worked with Ryuk and other ransomware on big game hunting attacks, deployed cryptominers and was a concerning factor in election interference in 2020.</p>
<p>Like many other botnets, TrickBot has garnered a lot of attention from law enforcement and suffered a <a href="https://www.zdnet.com/article/trickbot-botnet-survives-takedown-attempt-but-microsoft-sets-new-legal-precedent/" target="_blank" rel="noopener nofollow" >takedown attempt</a> in October 2020. Unfortunately, despite losing a large proportion of its infrastructure, TrickBot not only re-emerged, but it also launched a new, and more persistent version than ever, using a UEFI/BIOS bootkit to help it remain undetected on infected devices. This bootkit was dubbed TrickBoot by those who <a href="https://eclypsium.com/2020/12/03/trickbot-now-offers-trickboot-persist-brick-profit/" target="_blank" rel="noopener nofollow" >first analyzed</a> and reported about it.</p>
<p>Our team at IBM Trusteer research examined what changed between the two most recent versions of the TrickBot malware. Let&rsquo;s take a closer look at some of the components that keep TrickBot persistent and stealthy over time.</p>
<h2>Compared Components</h2>
<p>This comparison will look into a few major components from previous versions 1000512 and 1000513 versus the newer version 100003. Unusually, the version number jumped backward.</p>
<ul>
<li>Persistence mechanism</li>
<li>Injection technique</li>
<li>Bot configuration with a random twist</li>
<li>Mutex naming concept</li>
<li>Compromise check</li>
</ul>
<h2>Persistence Mechanism &ndash; Modified in New Version</h2>
<p>Most malicious software developers implement some sort of persistence mechanism to ensure the malware is not wiped from the device on reboot. TrickBot&rsquo;s long-standing persistence method uses a scheduled task that is set to run on set time lapses. This tactic was modified in the newer version.</p>
<p>In the previous version, the scheduled task appeared under the name Clean Master.</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666014.png" layout="intrinsic" class="" alt="Fig1.png" width="579" height="161" lightbox="lightbox"></amp-img></p>
<p><em>Figure 1: TrickBot&rsquo;s scheduled task created under the task directory</em></p>
<p>This task&rsquo;s executable file&rsquo;s path is directly in the &lsquo;Programs&rsquo; menu:</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666015.png" layout="intrinsic" class="" alt="Fig2.png" width="750" height="51" lightbox="lightbox"></amp-img></p>
<p><em>Figure 2: Task&rsquo;s executable location</em></p>
<p>The malware further copies itself into a new subdirectory as part of its stealth tactics, thereby making Clean Master appear as yet another legitimate program and not suspicious at first sight.</p>
<p>In the new version, although TrickBot maintained the persistence mechanism of using the Windows Task Scheduler as the way to reload the malware, the task&rsquo;s destination is now changed to a batch file. This is a stealthier choice by TrickBot&rsquo;s developers, helping the malware evade scans by antivirus and security tools that typically look for malicious executables in the autorun and task scheduler.</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666016.png" layout="intrinsic" class="" alt="Fig3.png" width="757" height="55" lightbox="lightbox"></amp-img></p>
<p><em>Figure 3: Autorun snippet of the new task</em></p>
<p>A quick look into the batch file location exposes TrickBot&rsquo;s executable, named &lsquo;newb&rsquo; in this case.</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666017.png" layout="intrinsic" class="" alt="Fig4.png" width="645" height="103" lightbox="lightbox"></amp-img></p>
<p><em>Figure 4: TrickBot&rsquo;s batch file along its executable</em></p>
<p>Further in the batch file, we also see an obfuscated script:</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666018.png" layout="intrinsic" class="" alt="Fig5.png" width="414" height="178" lightbox="lightbox"></amp-img></p>
<p><em>Figure 5: A part of TrickBot&rsquo;s obfuscated batch file</em></p>
<p>De-obfuscating the script reveals its more obvious purpose as running the TrickBot executable (newb.exe) from the following patch:</p>
<table border="1" cellspacing="0" cellpadding="0">
<tbody>
<tr>
<td>start C:\Users\[name here]\AppData\Roaming\Identities1793959193\newb.exe</td>
</tr>
</tbody>
</table><div class="table-scroll-help-text"><span>Scroll to view full table </span></div>
<h2></h2>
<h2>Choice of Injection Mechanism &ndash; Retained</h2>
<p>The injection of malicious code into legitimate Windows processes is one of the most common tactics malware developers use. The code injection can take <a href="https://securityintelligence.com/dridexs-cold-war-enter-atombombing/" >different forms</a>, but a highly popular way to implement code injection is process hollowing, in which legitimate content is removed and replaced by the attacker&rsquo;s code.</p>
<p>TrickBot uses process hollowing as its code injection tactic, and that has remained the same in the previous and current versions of the malware. This method has a couple of advantages:</p>
<ul>
<li>Allows for the termination of the main suspicious process</li>
<li>Presents the malware&rsquo;s operations as being performed by legitimate processes</li>
</ul>
<p>Here&rsquo;s what TrickBot&rsquo;s process hollowing does:</p>
<ol>
<li>Creates process wermgr.exe in suspend mode using CreateProcessInternalW. This Windows process is inherently responsible for handling errors.</li>
<li>Allocates memory for its payload shellcode using NtAllocateVirtualMemory</li>
<li>Writes into memory using NtWriteVirtualMemory</li>
<li>Reads remote process&rsquo;s Process Environment Block (PEB) using NtQueryInformationProcess</li>
<li>Reads first module wermgr.exe&rsquo;s address using the PEB structure</li>
<li>Goes to the 0xF0+0x28=0x118 offset in the wermgr.exe address to find its entry point:</li>
</ol>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666020.png" layout="intrinsic" class="" alt="Fig6.png" width="517" height="26" lightbox="lightbox"></amp-img></p>
<p><em>Figure 6: Wermgr&rsquo;s entry-point address</em></p>
<ol start="7">
<li>Writes a trampoline to the entry point 0x80F0 and the module&rsquo;s base address to jump to the shellcode using NtWriteVirtualMemory. A <a href="https://en.wikipedia.org/wiki/Trampoline_(computing)#:~:text=The%20trampoline%20is%20a%20small,variables%20of%20the%20enclosing%20function." target="_blank" rel="noopener nofollow" >trampoline</a> is a small piece of code which is constructed on the fly on the stack when the address of a nested function is taken.</li>
<li>Calls NtResumeThread to resume the thread.</li>
</ol>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666021.png" layout="intrinsic" class="" alt="Fig7.png" width="481" height="63" lightbox="lightbox"></amp-img></p>
<p><em>Figure 7: Example of TrickBot&rsquo;s trampoline</em></p>
<h2>Bot Configuration Receives Randomization Twist</h2>
<p>Bot configuration is the way the malware configures its own file on initial setup when a new machine is infected or when it is updated on infected devices. The initial configuration contains data such as encryption keys, IDs, credentials, URLs and communication patterns it will use to send information to its botmaster.</p>
<p>Bot configurations are easy to modify and are changed by malware operators in their various campaigns and attacks, offering agility and stealth.</p>
<p>TrickBot&rsquo;s initial configuration elements are traditionally split between its embedded XML &lsquo;mcconf&rsquo; configuration file and Base64 encrypted strings stored within the unpacked TrickBot core. The &lsquo;mcconf&rsquo; file contains a list of C2 server IP addresses, along with the TrickBot version number, Bot ID and default modules. Remaining configuration elements such as install directory and scheduled task name are stored as custom Base64 encoded strings.</p>
<p>TrickBot retained the same bot configuration scheme from its previous versions and continued to use the same one in current versions, only this time, it added some randomization to data inside the file to delay detection and analysis.</p>
<p>When we looked at the details, we saw that only task names have been modified, but all the data and purposes remain the same. We will come back to task names later.</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666022.png" layout="intrinsic" class="" alt="Fig8.png" width="689.5799086757991" height="322" lightbox="lightbox"></amp-img></p>
<p><em>Figure 8: TrickBot configuration elements from decrypted base64 strings &mdash;&nbsp;excerpt comparison from previous and current versions</em></p>
<p>TrickBot&rsquo;s configuration is embedded within the unpacked TrickBot core binaries. The extraction process takes place as follows:</p>
<ol start="9">
<li>Upon execution, TrickBot is unpacked. TrickBot makes use of a variety of different packing techniques; in this case, the malware is stored as a resource of the parent executable and decrypted using a modified implementation of the RC4 cipher with a hardcoded key embedded within the file. This technique is also commonly used by Emotet.</li>
</ol>
<p>The only difference between the standard RC4 and this modified version is the change of the standard RC4 constant value from 256 to 1147.</p>
<ol start="10">
<li>The resulting binary is the TrickBot Loader. This is a highly obfuscated binary that decrypts its other functions and payloads as required using a XOR operation. This loader is responsible for decrypting and loading the TrickBot core binaries. The loader decrypts the appropriate 32/64 bit core binary using XOR and decompresses it using an LZO algorithm.</li>
<li>Within the TrickBot core binary, the &lsquo;mcconf&rsquo; configuration file is embedded and encrypted using AES and XOR. The binary also contains a list of custom base64 encoded strings separated by the NULL byte &lsquo;\x00&rsquo;.</li>
</ol>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666023.png" layout="intrinsic" class="" alt="Fig9.png" width="599" height="98" lightbox="lightbox"></amp-img></p>
<p><em>Figure 9: Part of the encoded TrickBot configuration, separated by NULL bytes</em></p>
<ol start="12">
<li>The last string is picked to become a custom Base64 encoding index.</li>
<li>When required, every string the malware calls is then pushed to a Base64 decoding function that uses the custom index to decode the configuration. This way, the specific configuration is in plaintext only when it&rsquo;s used, which can make both dynamic and static analysis by outsiders take longer.</li>
</ol>
<p>The Base64 binary data encoding uses an index table of 64 characters, which typically includes basic characters such as: ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/</p>
<p>With TrickBot&rsquo;s configuration, the index table used in our sample was:</p>
<p>3GjYRHFAKkzMws2IuZpiJOqamWgSBh14nE0N8x9LbXQt5yUPvedVcl6ofTD/+rC7</p>
<p>A closer look revealed that TrickBot&rsquo;s encryption methods are customized by its developers. For example, Trickbot uses different implementation for the RC4 cipher and a modified Base64 decoding algorithm, incorporating a custom table in order to make analysis more difficult. These choices are likely there since most crypto detection scripts are not likely to find such functions automatically.</p>
<h2>Bot Configuration Changes</h2>
<p>In the beginning of this section, we mentioned that only task names in the bot configuration were modified. These names change frequently and, in the newer samples, task names are appended a randomized 10-digit number that modifies the name each time. For example, for the task name &lsquo;identities&rsquo; we will now see a string like &lsquo;Identities1793959193&rsquo;. The developer likely uses this random twist to eliminate the ability to search for constants with detection tools.</p>
<p>The randomization itself is achieved by a couple of additional calculations and XOR operations on two values that TrickBot fetches. In this case, it uses the result that is produced at that moment from the computer&rsquo;s processor&rsquo;s read time-stamp counter command (RDTSC<sup><a title="" href="#ftn1" >[1]</a></sup>) and the result from the Windows GetTickCount <sup><a title="" href="#ftn2" >[2]</a> </sup> function.</p>
<h2>Fully Randomized Mutex Naming &ndash; Retained</h2>
<p>The mutex &mdash; the mutual exclusion object that synchronizes access to a resource &mdash; is very commonly used by malware developers. As a result, mutexes are part of the indicators of compromise to look for when trying to determine if a device has been infected by specific malware. Searching for mutexes is also something malware does to ensure it does not re-infect a device and cause instability, or to kill the processes of a rival malware.</p>
<p>TrickBot uses a mutex naming algorithm to create random, unique mutexes across infected devices that it manages to control. Since mutexes are a definitive indicator of compromise, TrickBot&rsquo;s developers were creative in creating the algorithm used here, staying away from relying on system artifacts that can be more easily figured out.</p>
<p>In current versions, TrickBot&rsquo;s mutexes are created as fully randomized 32-digits consisting of a 16- byte globally unique identifier (GUID).</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666024.png" layout="intrinsic" class="" alt="Fig10.png" width="731" height="102" lightbox="lightbox"></amp-img></p>
<p><em>Figure 10: An example of a TrickBot mutex name</em></p>
<p>In the process of creating the mutex name, &lsquo;get_random&rsquo;, TrickBot splits it into two parts:</p>
<ol>
<li>Random [8 Random bytes]</li>
<li>Related [8 Related bytes]</li>
</ol>
<p>To randomize the strings, TrickBot uses the GetTickCount function and the RDTSC assembly instruction.</p>
<p><amp-img src="https://securityintelligence.com/wp-content/uploads/2021/01/1611666025.png" layout="intrinsic" class="" alt="Fig11.png" width="217" height="106" lightbox="lightbox"></amp-img></p>
<p><em>Figure 11: TrickBot&rsquo;s simple get_random function</em></p>
<p>The second part, or &lsquo;related part&rsquo;, is built upon the first. It&rsquo;s a combination of some XORing operations of the first, &lsquo;random part&rsquo; with the computer&rsquo;s volume C serial number. Finally, the Windows StringFromCLSID <sup><a title="" href="#ftn3" >[3]</a></sup> function is used to give the mutex its GUID form.</p>
<p>What&rsquo;s important in the mutex name here is not the value, but the correlation between its first and second halves.</p>
<h2>Previous Version Checks</h2>
<p>Before the TrickBot malware infects a device, it runs a detection process to ensure that device has not already been infected.</p>
<p>This detection takes us back to TrickBot&rsquo;s mutex, which is what it looks for on the device. Since mutex names are randomized, it cannot directly look for a name. Rather, it examines the relation between the first half and the second half of the mutex, and then infers whether the resulting GUID has or has not been generated in the past.</p>
<p>Let&rsquo;s take a closer look at the mutex enumeration process that TrickBot performs:</p>
<ol>
<li>It. begins by enumerating all system handles. In Windows-based devices, a handle is a pointer to an object, such as a mutex, a file, etc. The malware does that by calling the ZwQueryInformation Native API<sup><a title="" href="#ftn4" >[4]</a></sup> with SYSTEM_HANDLE_INFORMATION (0x10) as the parameter.</li>
<li>Next, it uses the NtQueryObject function it queries for information about each object, and in our case each handle, to search mutex names.</li>
<li>Then it uses the OpenProcess function to open the process the mutex handle belongs to.</li>
<li>Lastly, it uses the DuplicateHandle function to duplicate the handle to that mutex so it can access it.</li>
</ol>
<p>Note that the first two functions are Windows Native APIs belonging to the NTDLL.dll library. Usually, when calling a Windows API function from the KERNEL32.dll, it calls its Windows Native API counterpart function in NTDLL.dll that address the Windows kernel.</p>
<p>At this point, if the device is deemed infected, TrickBot initiates an ExitProcess to abandon a second installation of the malware.</p>
<h2>TrickBot Lives On &ndash; A Reminder to Stay Vigilant</h2>
<p>As we kick off 2021, the TrickBot Trojan and the cybercrime syndicate around it are in full swing, operating on the bank fraud front as well as high-stakes ransomware attacks.</p>
<p>If your security team detects TrickBot on networked devices, those should become high priority to clean or reimage as needed. This malware can be a foot in the door for a number of attacks, the worst of which can be an all-out ransomware and extortion operation against your organization.</p>
