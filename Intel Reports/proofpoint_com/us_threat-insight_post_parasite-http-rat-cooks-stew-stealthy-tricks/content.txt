<h1 class="blog-banner__heading">
      <span>Parasite HTTP RAT cooks up a stew of stealthy tricks</span>

  </h1>

    </div>
  </div>

  <div class="blog-content">
    <div class="blog-content__sharethis sharethis_toolbox sharethis_32x32_style">
      <div class="blog-content__sharethis_label sharethis__label">Share with your network!</div>
      <div class="blog-content__sharethis_buttons sharethis_buttons">
        <div class="sharethis-inline-share-buttons"></div>
        
<span class="addthis_button_subscribe at300b UNCONVERTED" title=Subscribe>
  <span class="at-icon-wrapper block-subscribe-button__trigger block-subscribe-button__addthis"></span>
</span>


      </div>
    </div>
        <div class="blog-content__metadata blog-content__metadata-author">
      <span class="blog-content__date">  <time datetime="2018-07-25T14:15:00Z">July 25, 2018</time>

</span>
        <span class="blog-content__author">
                Proofpoint Staff
  
        </span>
    </div>
    <div class="node-full__body blog-content__body">  <p><strong>Overview</strong></p>

<p>Proofpoint researchers recently discovered a new remote access Trojan (RAT) available for sale on underground markets. The RAT, dubbed Parasite HTTP, is especially notable for the extensive array of techniques it incorporates for sandbox detection, anti-debugging, anti-emulation, and other protections. The malware is also modular in nature, allowing actors to add new capabilities as they become available or download additional modules post infection.</p>

<p>To date, we have only observed Parasite HTTP in a single small email campaign with intended recipients primarily in the information technology, healthcare, and retail industries.</p>

<p><strong>Campaign Analysis</strong></p>

<p>On July 16, 2018, Proofpoint observed a small campaign that appeared to leverage human resources distribution lists as well as some individual recipients at a range of organizations. Specifically, the campaign targeted the following distribution lists, among others:</p>

<ul>
	<li>hr@[organization domain]</li>
	<li>recruiting@</li>
	<li>accessibility@</li>
	<li>resumes@</li>
</ul>

<p>The messages purported to be resumes or CV submissions and used subjects including:</p>

<ul>
	<li>advertised position</li>
	<li>would like to apply</li>
	<li>application</li>
</ul>

<p>The messages contained Microsoft Word attachments with names such as (Figure 1):</p>

<ul>
	<li>my_cv.doc</li>

<p>The documents contained macros that, if enabled, would download Parasite HTTP from a remote site.</p>

<p><strong>Malware Analysis</strong></p>

<p>Parasite HTTP contains an impressive collection of obfuscation and sandbox- and research environment-evasion techniques. In this section we examine the more sophisticated components.</p>

<p><em><u>String Obfuscation</u></em></p>

<p>Parasite HTTP uses the following encoding for its obfuscated strings:</p>

<ul>
	<li>4-byte XOR key</li>
	<li>2-byte string length in characters</li>
	<li>N-byte obfuscated ASCII/Unicode string</li>
</ul>

<p>This 6-byte header exists prior to the reference to the string in assembly.&nbsp;Parasite HTTP contains four routines for operating on strings preceded by a 6-byte header.&nbsp;For each type of string, ASCII or Unicode, one variant leaves the obfuscated string in place and returns a dynamically-allocated, deobfuscated version of the string.&nbsp;The other variant uses VirtualProtect to deobfuscate the string in place, setting the XOR key to 0 after the deobfuscation has been performed, which effectively skips deobfuscation during future access to the string.</p>

<p><em><u>Sandbox Detection via Sleep Manipulation</u></em></p>

<p>Parasite HTTP uses a sleep routine to delay execution and check for sandboxes or emulation. Specifically it employs the following code to perform its sleeps:</p>

<p><img src="/sites/default/files/phf2.png"></p>

<p><em>Figure 2: Routine that Parasite HTTP uses to perform sleeps in 10ms increments (decompiled and with functions renamed for readability)</em></p>

<p>The above function is called by a routine that performs a sandbox check for the expected passage of time as well as non-interference with its own handling of breakpoint instructions it has placed in the checking routine (removed from the display below for readability):</p>

<p><img src="/sites/default/files/phf3.png"></p>

<p><em>Figure 3: Function that detects sandbox environments by checking for the passage of time and non-interference with its own handling of breakpoint instructions</em></p>

<p>During the interval between the installation and removal of the vectored exception handler, several breakpoint instructions are executed in the function which triggers execution of this vectored exception handler, shown below.&nbsp;That handler sets the global flag which notifies the parent function that the handler has run and skips over the breakpoint instruction.&nbsp;This code has been copied verbatim from an example of anti-debugging code posted in a public repository [1].</p>

<p><img src="/sites/default/files/phf4.png"></p>

<p><em>Figure 4: Exception handler</em></p>

<p>The sandbox checking routine first checks to ensure that the exception handler in Figure 4 has run. It then checks whether between 900ms and two seconds elapsed in response to the routine’s 1 second sleep split into 10ms increments.&nbsp;Sandboxes using code like that available in [1] for example, would have run afoul of this particular sandbox check.</p>

<p><em><u>Obfuscation of Sandbox Detection via Skipping of Allocation of Critical Buffers</u></em></p>

<p>When Parasite HTTP actually does detect a sandbox, it attempts to hide this fact from any observers. It does not simply exit or throw an error, instead making it difficult for researchers to determine&nbsp; why the malware did not run properly and crashed. In the screenshot below, we can see how Parasite HTTP uses its sandbox detection in a clever way to result in a later crash on attempting to use a buffer whose allocation was skipped:</p>

<p><img src="/sites/default/files/phf5.png"></p>

<p><em>Figure 5: Code snippet showing mechanism by which critical buffers required for malware functioning are not allocated if a sandbox is detected</em></p>

<p><em><u>Heap Clearing Bug</u></em></p>

<p>Parasite HTTP also contains a bug caused by its manual implementation of a GetProcAddress API that results in the clearing code not executing.&nbsp;This can be seen in the following commented decompilation:</p>

<p><img src="/sites/default/files/phf6.png"></p>

<p><em>Figure 6: Commented decompilation showing a bug in Parasite HTTP</em></p>

<p>The malware attempts to resolve a function named HeapSize to its associated address within kernel32.dll. However, its manual GetProcAddress function lacks support for resolving forwarded exports.&nbsp;In this case, since HeapSize is in fact a forwarded export to NTDLL’s RtlSizeHeap function, the function will return NULL and HeapSize and the associated clearing will never be called.&nbsp;This is easily confirmed by monitoring calls to RtlFreeHeap.</p>

<p><em><u>Use of Researcher Code from Github for Detecting Sandbox Hooking via Write Watches</u></em></p>

<p>Parasite HTTP adapts code from a public repository [3] for its own sandbox detection purposes.&nbsp;The code is copied verbatim, with the API resolution replaced with its own internal code, the prints removed, and the file and environment variable names generated randomly.&nbsp;Of note is that the GetWriteWatch API in this case detects more than just writes.&nbsp;Due to demand paging in Windows, the first access, whether read or write, to the allocation created will result in the page table entries being instantiated and counted as a “write” for the purposes of GetWriteWatch.&nbsp;Since the APIs being called with invalid arguments would never read or write to the allocated buffer, hooks that read from the buffer prematurely or even in the case of failure of the API, as in [2], would fail this sandbox detection.</p>

<p><em><u>Remapping of NTDLL via KnownDlls32\ntdll.dll for Hook Evasion</u></em></p>

<p>On Microsoft Windows, versions 7 and newer that have KnownDlls functionality,<wbr>&nbsp;Parasite HTTP resolves certain critical APIs by using a DLL remapping technique that while previously documented,<wbr>&nbsp;has not,<wbr>&nbsp;to our knowledge,<wbr>&nbsp;been used recently in other major malware families.  Malware behavior hidden by this technique include process injection and the Poweliks technique of including a NUL character in registry value names.&nbsp;The malware preserves support for older versions of Windows by falling back to using the existing copy of NTDLL loaded in memory.&nbsp;This behavior can be seen in the following decompilation listing:</p>

<p><img src="/sites/default/files/phf7.png"></p>

<p><em>Figure 7: Code snippet showing NTDLL remapping</em></p>

<p>Mapping the new copy of NTDLL effectively provides it with a copy free of any hooks placed on the initial NTDLL mapping, rendering its thread injection and registry modifications invisible to most userland hooking implementations.&nbsp;Further, since this mapping is accomplished with NtOpenSection and NtMapViewOfSection, it will not involve the typical calls to filesystem APIs used by other variants of the technique to achieve the same goal.</p>

<p><em><u>Obfuscated Checking for Breakpoints within a Critical Function of the Malware</u></em></p>

<p>Parasite HTTP includes an obfuscated check for debugger breakpoints within a range of its own code.&nbsp;This functionality is only used in one location to check a single function in the malware that calls out to the sandbox detection based on GetWriteWatch checks (Figure 8):</p>

<p><img src="/sites/default/files/phf8.png"></p>

<p><em>Figure 8: Obfuscated check for debugger breakpoints</em></p>

<p>This code is also copied from [1] with the malware having implemented the added “level of indirection” mentioned in the code comment.&nbsp;It is worth noting that this technique is naive and unreliable long-term over arbitrary code, as unintentional 0xcc bytes can be found in a simple byte-by-byte scan of code through certain instruction encodings, local stack frame offsets, relative references, indirect addresses, or immediate constants.</p>

<p><em><u>Extension of Themida/Formbook Technique for Unhooking APIs in NTDLL/Kernel32/Kernelbase</u></em></p>

<p>In its initial process, Parasite HTTP removes hooks on the aforementioned DLLs by reading them in from disk and comparing the first 5 bytes of each exported function to that present in the currently mapped version in memory.&nbsp;Though this technique is naive in its implementation, not making use of any instruction decoder and limiting itself to 5 hardcoded bytes, it is effective in practice.&nbsp;Consider the case of a sandbox using an indirect jump (6 bytes) for its hooks -- the malware will restore only the first 5 bytes to the original, leaving the final byte of the hook in place, most likely resulting in a crash upon its execution.</p>

<p><strong>Conclusion</strong></p>

<p>Threat actors and malware authors continuously innovate in their efforts to evade defenses and improve infection rates. Parasite HTTP provides numerous examples of state-of-the-art techniques used to avoid detection in sandboxes and via automated anti-malware systems. For consumers, organizations, and defenders, this represents the latest escalation in an ongoing malware arms race that extends even to commodity malware like Parasite. While we have currently only observed Parasite HTTP in a small campaign, we expect to see features like those used in Parasite continue to propagate across other malware variants.</p>
