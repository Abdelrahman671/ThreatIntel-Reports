<h3 class='post-title entry-title' itemprop='name'>
<a href='https://tccontre.blogspot.com/2021/01/extracting-shellcode-in-iceid-png.html'>Extracting Shellcode in ICEID .PNG Steganography</a>
</h3>
<div class='post-header'>
<div class='post-header-line-1'></div>
</div>
<div class='post-body entry-content' id='post-body-9174043645981532144' itemprop='description articleBody'>
<p>&nbsp;In this past few days I stumble to some new and old variant of ICEID malware that uses .png steganography to hide and execute its encrypted shellcode. In this article I will share how the structure of the Iceid png payload look like and how to extract its encrypted shellcode<span style="color: grey;">.</span></p><h3 style="text-align: left;"><span style="color: #ffa400;">Loader Compression:</span></h3><div>same as the other malware, IceID Loader changes its Crypter to execute its main module in memory. From old variant where the encrypted code stub and decryption module is place in the RSRC section as a data .rsrc entry (RC4 encrypted) to applib compression like the figure below.</div><div><div class="separator" style="clear: both; text-align: center;"><br /></div><br /></div><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; margin-right: auto;"><tbody><tr><td style="text-align: center;"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgh3yVsHFjIDH6UpqG8Yy2QlzIGSUQgOR-xUE0_08-AZJbCQqTbM5M88pjMBnW_HwJeOTgEFs6ifKgppkMHHV5S-b2BhztqyFiYAAQfaNIZg9JTIGJlhWPq0xuL5jxftgyAN_gstT3ej6w/s826/apli.png" style="margin-left: auto; margin-right: auto;"><img border="0" data-original-height="826" data-original-width="544" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgh3yVsHFjIDH6UpqG8Yy2QlzIGSUQgOR-xUE0_08-AZJbCQqTbM5M88pjMBnW_HwJeOTgEFs6ifKgppkMHHV5S-b2BhztqyFiYAAQfaNIZg9JTIGJlhWPq0xuL5jxftgyAN_gstT3ej6w/s16000/apli.png" /></a></td></tr><tr><td class="tr-caption" style="text-align: center;">figure 1: The aplib decompressed module of ICEID</td></tr></tbody></table><div><br /></div><h3 style="text-align: left;"><span style="color: #ffa400;">PNG Header:</span></h3><div>Before we deal with the ICEID PNG steganography, I think it is a good idea to have some preview what PNG file header format is. It will give as a clear preview how ICEID parse the .PNG header and look for its encrypted shellcode.</div><div><br /></div><div>The PNG file format started with <b><span style="color: #f1c232;">8-bytes signature header "89  50  4e  47  0d  0a  1a  0a"</span></b>. after this header is a chunk structure or series of chunk structures that contains either a <b><span style="color: #f1c232;">"critical chunks"</span></b> like "IHDR", "IDAT" and etc... or <b><span style="color: #f1c232;">"Ancillary chunks"</span></b> that may contain some attribute related to the color, pixel or metadata of the png file like "sRGB", "gAMA" and many more.</div><div><br /></div><div>each PNG chunks layout consist of 4 parts or 4 structure member. the figure below show the 4 parts in IHDR chunk type.</div><div><br /></div><br /><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; margin-right: auto;"><tbody><tr><td style="text-align: center;"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEhT1nSkhatRahXcycSCxsDhnD4qkySMsSE6AigiweUhTcfgXF9w6C8IoCCeu7Q5huP2x37UC3m5tLoEXnBGekSGzc8kXntSjrSVHfmi5dLuO_O88bpSdye9GfmNZjCJZtICyRF4W5sSg8Q/s699/hdr.png" style="margin-left: auto; margin-right: auto;"><img border="0" data-original-height="392" data-original-width="699" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEhT1nSkhatRahXcycSCxsDhnD4qkySMsSE6AigiweUhTcfgXF9w6C8IoCCeu7Q5huP2x37UC3m5tLoEXnBGekSGzc8kXntSjrSVHfmi5dLuO_O88bpSdye9GfmNZjCJZtICyRF4W5sSg8Q/s16000/hdr.png" /></a></td></tr><tr><td class="tr-caption" style="text-align: center;">figure 2: PNG File header Format</td></tr></tbody></table><div><br /></div><h3 style="text-align: left;"><span style="color: #ffa400;">ICEID PNG Module Decryptor:</span></h3><div>Now that we have some insight how PNG file format look like, lets dive in to the ICEID PNG decryptor module <b><span style="color: #f1c232;">(<i>"let's call it PNG module"</i>)</span></b> that we already extracted earlier in the memory. This part is really interesting especially in parsing the header. :)</div><div><br /></div><div>The PNG module start by executing a thread that will do the following:</div><div>1. decrypt its data section (C&amp;C url link) using same approach how it decrypt the shellcode in PNG payload file.</div><div>2. it will check the existence of the PNG file in %appdata%&lt;randomname&gt;/, if not</div><div>3.&nbsp; it will try to download it to its C&amp;C server</div><div>4. then it will parse and check the png file to extract and execute its shellcode.</div><div><br /></div><div>The first task is decrypting the C&amp;C server URL link that are place in data section that are encrypted in RC4 algorithm. the structure of data it used in decrypting this data section can be seen before a call&nbsp; function that will do the decryption part.</div><div><br /></div><div><br /></div><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; margin-right: auto;"><tbody><tr><td style="text-align: center;"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgRoCMcEPKDsxAcKYF67kLitniJTWcKQToJx_X2Nnzua1ygofVRklp6ZqOB8dZPym5W7aloMmdW_QFqtMjn5TO2B_CZAwOKafsMJZCNyT40AjWSbDuMWr81UZ4RjlosG0mK2XYcp9epPBQ/s709/struct.png" style="margin-left: auto; margin-right: auto;"><img border="0" data-original-height="178" data-original-width="709" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgRoCMcEPKDsxAcKYF67kLitniJTWcKQToJx_X2Nnzua1ygofVRklp6ZqOB8dZPym5W7aloMmdW_QFqtMjn5TO2B_CZAwOKafsMJZCNyT40AjWSbDuMWr81UZ4RjlosG0mK2XYcp9epPBQ/s16000/struct.png" /></a></td></tr><tr><td class="tr-caption" style="text-align: center;">figure 3: Encrypted Data Structure</td></tr></tbody></table><br /><div><br /></div><div>The first 8 bytes of the encrypted data section is the RC4 key and the rest is the encrypted data.</div><div><br /></div><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; margin-right: auto;"><tbody><tr><td style="text-align: center;"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEhZxCDjfDxNIuRSQ3dOEBFWGAI0llGqxC9joFwjj0Ika67o95VZvQv85ShVfDL5iheC9NZBoDpptKjH1RnFNwHJT-KGt35oCH__Gzm8Zpa9G57myLt9EfNrw2Go_MmVljau0MO_YTN4DXM/s568/url.png" style="margin-left: auto; If the module didn't find the png payload in the said folder it will try to contact the C&amp;C server to download it.</div><div><br /></div><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; margin-right: auto;"><tbody><tr><td style="text-align: center;"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEi8etBGPD0pRKV6awmaK0I2Q317mnUtnMD99txj6MLySqdQTF1FlcjUyEaC5St_L7HH0XWSpK0PQJPjfQ2WBPJD1wptvbuLuJAYkNyf-uToA92NToUdJ8V_G__NKXvW0qjQtQ4mINTrMew/s725/pcheck.png" style="margin-left: auto; margin-right: auto;"><img border="0" data-original-height="156" data-original-width="725" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEi8etBGPD0pRKV6awmaK0I2Q317mnUtnMD99txj6MLySqdQTF1FlcjUyEaC5St_L7HH0XWSpK0PQJPjfQ2WBPJD1wptvbuLuJAYkNyf-uToA92NToUdJ8V_G__NKXvW0qjQtQ4mINTrMew/s16000/pcheck.png" /></a></td></tr><tr><td class="tr-caption" style="text-align: center;">figure 5: looking for the PNG payload file</td></tr></tbody></table><br /><h3 style="text-align: left;"><span style="color: #ffa400;">Parsing PNG Header:</span></h3><div>after reading the PNG and save it in the memory, it will start the checking in offset 0x8 (skipping the PNG header) which is the <i>"chunk_data_length"</i> of the first chunk type in the header which in our case is "IHDR". The way how it parse the header to look for "IDAT" chunk_type structure is by adding:</div><div><br /></div><div><b><span style="color: #f1c232;">next_chunk_type_struct = size(chunk_data) + chunk_data_length (4 bytes) + chunk_type (4byes) + chunk_crc32 (4 bytes)</span></b></div><div><br /></div><div>except for the start <span style="color: #f1c232;"><b>chunk_type</b> </span>structure where you need to include or add the PNG header size which 8 bytes.</div><div><br /></div><div>for this topic, I created a simple python script that will parse this header and give you the basic information about the header. it also parse the <b>chunk_data</b> but I place it in the <b>debug.log</b> of this script.</div><div><br /></div><div><a href="https://github.com/tccontre/KnowledgeBase/tree/main/malware_re_tools/iceid_stego_shell_decryptor">https://github.com/tccontre/KnowledgeBase/tree/main/malware_re_tools/iceid_stego_shell_decryptor</a></div><div><br /></div><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; 5</span></b> then it will skip the <span style="color: #f1c232;"><b>chunk_data_length</b> , <b>chunk_type</b></span> and <b><span style="color: #f1c232;">chunk_crc32</span></b> by adding <b><span style="color: #f1c232;">0x0c</span></b> to the current pointer of the "IDAT" chunk header. <span style="color: #f1c232;">the byte in this position looks like a validity flag of the PN</span>G. If this byte is zero, the PNG module will check another value which is one of the parameter to the function that parse the png header which is also zero, so in this case it will exit the flag.&nbsp;</div><table align="center" cellpadding="0" cellspacing="0" class="tr-caption-container" style="margin-left: auto; margin-right: auto;"><tbody><tr><td style="text-align: center;"><a /></a></td></tr><tr><td class="tr-caption" style="text-align: center;">figure 7: byte flag</td></tr></tbody></table><br /><div><br /></div><div><br /></div><div>after this byte is the 8 byte RC4
