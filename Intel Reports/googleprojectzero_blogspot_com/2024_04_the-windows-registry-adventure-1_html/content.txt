class="c32 doc-content">
 <p class="fhlpihoAED-c22 fhlpihoAED-c34"><span class="fhlpihoAED-c27">Posted by Mateusz Jurczyk, Google Project Zero</span></p>
 <p class="fhlpihoAED-c4"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span>In the 20-month period between May 2022 and December 2023, I thoroughly audited the Windows Registry in search of local privilege escalation bugs. It all started unexpectedly: I was in the process of developing a coverage-based Windows kernel fuzzer based on the Bochs x86 emulator (one of my favorite tools for security research: see </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://github.com/googleprojectzero/bochspwn">Bochspwn</a></span><span>, </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://github.com/googleprojectzero/bochspwn-reloaded">Bochspwn Reloaded</a></span><span>, and my earlier </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://googleprojectzero.blogspot.com/2016/07/a-year-of-windows-kernel-font-fuzzing-2.html">font fuzzing infrastructure</a></span><span>), and needed some binary formats to test it on. My first pick were PE files: they are very popular in the Windows environment, which makes it easy to create an initial corpus of input samples, and a basic fuzzing harness is equally easy to develop with just a single </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://learn.microsoft.com/en-us/windows/win32/api/winver/nf-winver-getfileversioninfosizew">GetFileVersionInfoSizeW</a></span><span>&nbsp;API call. The test was successful: even though I had previously </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://bugs.chromium.org/p/project-zero/issues/list?q=id%3A1907%2C1908%2C1909%2C1910%2C1911&can=1">fuzzed PE files in 2019</a></span><span>, the new element of code coverage guidance allowed me to discover a completely new bug: </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://bugs.chromium.org/p/project-zero/issues/detail?id=2281">issue #2281</a></span><span class="fhlpihoAED-c11">.</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span>For my next target, I chose the Windows registry. That&#39;s because arbitrary registry hives can be loaded from disk without any special privileges via the </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://learn.microsoft.com/en-us/windows/win32/api/winreg/nf-winreg-regloadappkeya">RegLoadAppKey</a></span><span>&nbsp;API (since Windows Vista). The hives use a binary format and are fully parsed in the kernel, making them a noteworthy local attack surface. Furthermore, I was also somewhat familiar with basic harnessing of the registry, </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://bugs.chromium.org/p/project-zero/issues/list?q=id%3A873%2C874%2C876%2C993&can=1">having fuzzed it in 2016</a></span><span>&nbsp;together with James Forshaw. Once again, the code coverage support proved useful, leading to the discovery of </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://bugs.chromium.org/p/project-zero/issues/detail?id=2299">issue #2299</a></span><span class="fhlpihoAED-c11">. But when I started to perform a root cause analysis of the bug, I realized that:</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p><ul style="padding: 0;" class="c8 lst-kix_4rhx73br1i2v-0 start"><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">The hive binary format is not very well suited for trivial bitflipping-style fuzzing, because it is structurally simple, and random mutations are much more likely to render (parts of) the hive unusable than to trigger any interesting memory safety violations.</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">On the other hand, the registry has many properties that make it an attractive attack surface for further research, especially for manual review. It is 30+ years old, written in C, running in kernel space but highly accessible from user-mode, and it implements much more complex logic than I had previously imagined.</span></li></ul>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span>And that&#39;s how the story starts. Instead of further refining the fuzzer, I made a detour to reverse engineer the registry implementation in the Windows kernel (internally known as the </span><span class="fhlpihoAED-c11">Configuration Manager) and learn more about its inner workings. The more I learned, the more hooked I became, and before long, I was all-in on a journey to audit as much of the registry code as possible. This series of blog posts is meant to document what I&#39;ve learned about the registry, including its basic functionality, advanced features, security properties, typical bug classes, case studies of specific vulnerabilities, and exploitation techniques.</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span>While this blog is one of the first places to announce this effort, I did already give a talk titled &quot;Exploring the Windows Registry as a powerful LPE attack surface&quot; at Microsoft BlueHat Redmond in October 2023 (see </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://j00ru.vexillium.org/slides/2023/bluehat.pdf">slides</a></span><span>&nbsp;and </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://www.youtube.com/watch?v=8mJLe8sZki8">video recording</a></span><span class="fhlpihoAED-c11">). The upcoming blog posts will go into much deeper detail than the presentation, but if you&#39;re particularly curious and can&#39;t wait to find out more, feel free to check these resources as a starter. &#128578;</span></p><h2 class="fhlpihoAED-c31 fhlpihoAED-c22" id="h.aiavocjlsfe2"><span class="fhlpihoAED-c26">Research results</span></h2>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span>In the course of the research, I filed </span><span class="fhlpihoAED-c15 fhlpihoAED-c24"><a class="fhlpihoAED-c51" href="https://bugs.chromium.org/p/project-zero/issues/list?q=finder%3Amjurczyk%20product%3Akernel%20opened%3E2022-05-01%20opened%3C2024-01-01&can=1">39 bug reports</a></span><span>&nbsp;in the Project Zero bug tracker, which have been fixed by Microsoft as </span><span class="fhlpihoAED-c24">44 CVEs</span><span class="fhlpihoAED-c11">. There are a few reasons for the discrepancy between these numbers:</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p><ul style="padding: 0;" class="c8 lst-kix_2wnifzn4nxg9-0 start"><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span>Some single reports included information about multiple problems, e.g. </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://bugs.chromium.org/p/project-zero/issues/detail?id=2375">issue #2375</a></span><span class="fhlpihoAED-c11">&nbsp;was addressed by four CVEs,</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span>Some groups of reports were fixed with a single patch, e.g. issues </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://bugs.chromium.org/p/project-zero/issues/detail?id=2392">#2392</a></span><span>&nbsp;and </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://bugs.chromium.org/p/project-zero/issues/detail?id=2408">#2408</a></span><span class="fhlpihoAED-c11">&nbsp;as CVE-2023-23420,</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span>One bug report was closed as WontFix and not addressed in a security bulletin at all (</span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://bugs.chromium.org/p/project-zero/issues/detail?id=2508">issue #2508</a></span><span class="fhlpihoAED-c11">).</span></li></ul>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span class="fhlpihoAED-c11">All of the reports were submitted under the Project Zero 90-day disclosure deadline policy, and Microsoft successfully met the deadline in all cases. The average time from report to fix was 81 days.</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span>Furthermore, between November 2023 and January 2024, I reported 20 issues that had low or unclear security impact, but I believed the vendor should nevertheless be made aware of them. They were sent without a disclosure deadline and weren&#39;t put on the PZ tracker; I have since published them on our team&#39;s </span><span class="fhlpihoAED-c15"><a class="fhlpihoAED-c51" href="https://github.com/googleprojectzero/p0tools/tree/master/WinRegLowSeverityBugs">GitHub</a></span><span>. Upon assessment, Microsoft decided to fix </span><span class="fhlpihoAED-c24">6</span><span class="fhlpihoAED-c11">&nbsp;of them in a security bulletin in March 2024, while the other 14 were closed as WontFix with the option of being addressed in a future version of Windows.</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span>This sums up to a total of </span><span class="fhlpihoAED-c24">50 CVEs</span><span class="fhlpihoAED-c11">, classified by Microsoft as:</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p><ul style="padding: 0;" class="c8 lst-kix_5uedwcx8bazf-0 start"><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">39 &times; Windows Kernel Elevation of Privilege Vulnerability</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">9 &times; Windows Kernel Information Disclosure Vulnerability</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span>1 &times; Windows Kernel Memory Information Disclosure Vulnerability</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span>1 &times;</span><span class="fhlpihoAED-c11">&nbsp;Windows Kernel Denial of Service Vulnerability</span></li></ul>

 <p class="fhlpihoAED-c19"><span class="fhlpihoAED-c9">2023-Dec-1</span></p></td></tr><tr class="fhlpihoAED-c1"><td class="fhlpihoAED-c16" colspan="1" rowspan="1">
 <p class="fhlpihoAED-c3"><span class="fhlpihoAED-c9">None (MSRC-84263)</span></p></td><td class="fhlpihoAED-c21" colspan="1" rowspan="1">
 <p class="fhlpihoAED-c25"><span class="fhlpihoAED-c9">CVE-2024-26181</span></p></td><td class="fhlpihoAED-c14" colspan="1" rowspan="1">
 <p class="fhlpihoAED-c12"><span class="fhlpihoAED-c15 fhlpihoAED-c17"><a class="fhlpihoAED-c51" href="https://github.com/googleprojectzero/p0tools/tree/master/WinRegLowSeverityBugs/Reports/15_Registry_quota_exhausted_SAM_corruption">Windows Kernel registry quota exhaustion may lead to permanent corruption of the SAM database</a></span></p></td><td class="fhlpihoAED-c20" colspan="1" rowspan="1">
 <p class="fhlpihoAED-c19"><span class="fhlpihoAED-c9">2023-Dec-11</span></p></td></tr></table><h2 class="fhlpihoAED-c22 fhlpihoAED-c31" id="h.uwjgp3q5uwtq"><span class="fhlpihoAED-c26">Exploitability</span></h2>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span class="fhlpihoAED-c11">Software bugs are typically only interesting to either the offensive/defensive sides of the security community if they have practical security implications. Unfortunately, it is impossible to give a blanket statement regarding the exploitability of all registry-related vulnerabilities due to their sheer diversity on a number of levels:</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p><ul style="padding: 0;" class="c8 lst-kix_cyvdtdmfjjuy-0 start"><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c24">Affected platforms:</span><span class="fhlpihoAED-c11">&nbsp;Windows 10, Windows 11, various Windows Server versions (32/64-bit)</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c24">Attack targets:</span><span class="fhlpihoAED-c11">&nbsp;the kernel itself, drivers implementing registry callbacks, privileged user-mode applications/services</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c24">Entry points:</span><span class="fhlpihoAED-c11">&nbsp;direct registry operations, hive loading, transaction log recovery</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c24">End results:</span><span class="fhlpihoAED-c11">&nbsp;memory corruption, broken security guarantees, broken API contracts, memory/pointer disclosure, out-of-bounds reads, invalid/controlled cell index accesses</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c24">Root cause of issues:</span><span class="fhlpihoAED-c11">&nbsp;C-specific, logic errors, bad reference counting, locking problems</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c24">Nature of memory corruption:</span><span class="fhlpihoAED-c11">&nbsp;temporal (use-after-free), spatial (buffer overflows)</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c24">Types of corrupted memory:</span><span class="fhlpihoAED-c11">&nbsp;kernel pools, hive data</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c24">Exploitation time:</span><span class="fhlpihoAED-c11">&nbsp;instant, up to several hours</span></li></ul>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span class="fhlpihoAED-c11">As we can see, there are multiple factors at play that determine how the bugs came to be and what state they leave the system in after being triggered. However, to get a better understanding of the impact of the findings, I have performed a cursory analysis of the exploitability of each bug, trying to classify it as either &quot;easy&quot;, &quot;moderate&quot; or &quot;hard&quot; to exploit according to my current knowledge and experience (this is of course highly subjective). The proportions of these exploitability ratings are shown in the chart below:</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span class="fhlpihoAED-c11"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEhloWiWiOwV3eDMCPov3ipAo0WtRl6c3pgqK75My89_x4hCs_XoJ667sbLgsiEA2HIjxVWl39EPlTUiOnApwCQ9keYCdV056X5506_pR3GVVtogqLr4wVQuIkZS3LJiVQ8OHepmy4sXAEi2tPcp0cBu-hpqCwGAwKcb9t-HLU3K1IPGfHi3CSKwyu9pHjM/s1426/image2.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="A histogram showing the difficulty of exploitability for the registry issues: 18 were considered easy to exploit, 10 considered moderate and 22 considered hard " border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEhloWiWiOwV3eDMCPov3ipAo0WtRl6c3pgqK75My89_x4hCs_XoJ667sbLgsiEA2HIjxVWl39EPlTUiOnApwCQ9keYCdV056X5506_pR3GVVtogqLr4wVQuIkZS3LJiVQ8OHepmy4sXAEi2tPcp0cBu-hpqCwGAwKcb9t-HLU3K1IPGfHi3CSKwyu9pHjM/s1200/image2.png" style="max-height: 750px; max-width: 600px;" title="A histogram showing the difficulty of exploitability for the registry issues: 18 were considered easy to exploit, 10 considered moderate and 22 considered hard " /></a></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span class="fhlpihoAED-c11">The ratings were largely based on the following considerations:</span></p><ul style="padding: 0;" class="c8 lst-kix_4m34njm6c6og-0 start"><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">Hive-based memory corruption is generally considered easy to exploit, while pool-based memory corruption is considered moderate/hard depending on the specifics of the bug.</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">Triggering OOM-type conditions in the hive space is easy, but completely exhausting the kernel pools is more difficult and intrusive.</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">Logic bugs are typically easier and more reliable to exploit than memory corruption.</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">The kernel itself is typically easier to attack than other user-mode processes (system services etc.).</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">Direct information disclosure (leaking kernel pointers / uninitialized memory via various channels) is usually straightforward to exploit.</span></li><li style="margin-left: 46pt;" class="c7 li-bullet-0"><span class="fhlpihoAED-c11">However, random out-of-bounds reads, as well as read access to invalid/controlled cell indexes is generally hard to do anything useful with.</span></li></ul>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span class="fhlpihoAED-c11">Overall, it seems that more than half of the findings can be feasibly exploited for information disclosure or local privilege escalation (rated easy or moderate). What is more, many of them exhibit registry-specific bug classes which can enable particularly unique exploitation primitives. For example, hive-based memory corruption can be effectively transformed into both a KASLR bypass and a fully reliable arbitrary read/write capability, making it possible to use a single bug to compromise the kernel with a data-only attack. To demonstrate this, I have successfully developed exploits for CVE-2022-34707 and CVE-2023-23420. The outcome of running one of them to elevate privileges to SYSTEM on Windows 11 is shown on the screenshot below:</span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span class="fhlpihoAED-c11"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEitmLGFepjddh9x73DYyZyh6goMXPJIpEoePgtIjxKsBEr-jBzoAIZH-tMnb349iSak9UfL0ZVow5WEg9IbAYPNvv5_sCs41Rs1owOEdTMyNNhqGcRYtvi4-_wvHO1Ufm-cA5JLLHd-fkxP1Mby6OhzNdEJ0yuiArfVDiWD4Q4GzhTULD4l20OGZx0jQVU/s979/image1.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="Screenshot of windows terminal showing successful exploitation for CVE-2022-34707 and CVE-2023-23420" border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEitmLGFepjddh9x73DYyZyh6goMXPJIpEoePgtIjxKsBEr-jBzoAIZH-tMnb349iSak9UfL0ZVow5WEg9IbAYPNvv5_sCs41Rs1owOEdTMyNNhqGcRYtvi4-_wvHO1Ufm-cA5JLLHd-fkxP1Mby6OhzNdEJ0yuiArfVDiWD4Q4GzhTULD4l20OGZx0jQVU/s979/image1.png" style="max-height: 750px; max-width: 600px;" title="Screenshot of windows terminal showing successful exploitation for CVE-2022-34707 and CVE-2023-23420" /></a></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c13"><span class="fhlpihoAED-c11"></span></p>
 <p class="fhlpihoAED-c12 fhlpihoAED-c22"><span>Upcoming posts in this series will introduce you to the Windows registry as a system mechanism and as an attack surface, and will dive deeper into practical exploitation using hive memory corruption, out-of-bounds cell indexes and other amusing techniques. Stay tuned!</span></p>
