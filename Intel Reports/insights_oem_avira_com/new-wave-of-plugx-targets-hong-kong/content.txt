col-xs-12"><p>Mustang Panda is a well-known APT with a long history of targeting non-governmental organisations (NGOs). It utilizes shared malware like Poison Ivy, PlugX and Cobalt Strike payloads in order to gather intelligence. Since 2008, PlugX as a RAT (Remote Access Trojan) malware family has been used as a backdoor to control the victim&#8217;s machine fully. Once the device is infected, an attacker can remotely execute several kinds of commands on the affected system to retrieve machine information, capture the screen, manage services, and manage processes.</p><h2><span lang=EN-GB>Overview</span></h2><p>Avira&#8217;s Advanced Threat Research team, has been tracking Mustang Panda APT for a while. According to Avira&#8217;s telemetry data, Mustang Panda mostly targets Asia-Pacific (APAC) countries and uses Cobalt or PlugX as payload.</p><p>Avira&#8217;s Advanced Threat Research team discovered a new version of PlugX from the Mustang Panda APT that is used to spy on some targets in Hong Kong and Vietnam. The way that the APT actor infects the target, and launches the malicious payload is similar to <a href=https://blog.malwarebytes.com/threat-analysis/2016/08/unpacking-the-spyware-disguised-as-antivirus/ >previous versions</a>—but with some differences.</p><p>PlugX executes DLL hijacking with benign applications such as ESET antivirus, Adobe Update etc.  However, the way the PlugX loader launches the payload is different from how it was done for the previous versions. Also, the PlugX that Mustang Panda APT uses has some extra features, including spreading through USB, gathering information, and stealing documents in air-gaped networks via USB.</p><p>Mustang Panda APT uses a package of binaries to load the actual payload and it is intentionally designed this way to bypass file scanners and sandboxes. PlugX contains 3 files: benign EXE file for DLL hijacking, DLL (just a loader to execute the payload), and the encrypted payload (usually with &#8220;.dat&#8221; extension).  The Advanced Threat Research team at Avira, have found different types of loaders (PlugX loader), and we will discuss it in more detail below. Obviously, file scanners or sandboxes can&#8217;t detect the PlugX payload without the encrypted DAT file.</p><p>Anomali&#8217;s Threat Research Team published a <a href=https://www.anomali.com/blog/china-based-apt-mustang-panda-targets-minority-groups-public-and-private-sector-organizations>post</a> about this campaign with the focus on the initial infection. In this article, we are going to dig into the PlugX loader and new PlugX payloads that Mustang Panda APT is using to spy on the targets.</p><h2><span lang=EN-GB>First stage: Loader</span></h2><p><span lang=EN-GB>The Loader is a tiny DLL file, the payload is an encrypted DAT file. </span></p><p><span lang=EN-GB>The DLL is responsible for decrypting the DAT file and executing it like a shellcode in memory. It reads the encrypted payload from disk by calling CreateFile and ReadFile APIs, and then allocates memory via VirtualAlloc. After decryption, it changes the protection via VirtualProtect and at the end executes it via direct call instruction.</span></p><p><span lang=EN-GB>DLL utilizes junk codes and a simple obfuscation to hide the API calls via stackstrings and load them dynamically.</span></p><p><span lang=EN-GB>The DAT file contains the decryption key inside of itself. The size of the key can vary. The loader reads offset zero of encrypted DAT file until it reaches null and takes that as an XOR key to decrypt the rest of the file.</span></p><p><img fetchpriority=high decoding=async class=" wp-image-1289 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_decrypt_payload.png?x15938 alt width=349 height=427></p><p style="text-align: center;">Figure 1: decryption routine for loader</p><p><span lang=EN-GB>Some of the loaders have hardcoded time inside the binary and they simply decrypt and execute the payload in a specific period of time.</span></p><p><span lang=EN-GB>For example, look at the following picture from one of the loader samples with time checking. The date is hardcoded &#8220;20190929.&#8221;</span></p><p><img decoding=async class=" wp-image-1199 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_decrypt_payload.png?x15938 alt width=478 height=475></p><p><span lang=EN-GB>We have found 15 unique payloads.</span></p><p class=auto-cursor-target><span lang=EN-GB>We observed these DAT file names from our telemetry:</span></p><p><strong>File Name</strong></p><p>adobeupdate.dat<br> clntcon.dat<br> http_dll.dat<br> log.dat<br> mpsvc.ui<br> mp.dat</p><h2>Payload</h2><p>As explained previously, the payload is executed like a shellcode but surprisingly, it&#8217;s a full PE binary. The loader will start the payload from the zero offset of the payload that it means from MZ. The picture below shows a very small crafted shellcode to call the entrypoint.</p><p><span lang=EN-GB>The decrypted payload is just loaded in memory and there are no footprints on the disk. This is the customized version of PlugX with simple obfuscations to bypass static detection and hide the strings. </span></p><p><img decoding=async class=" wp-image-1114 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_APT4.jpg?x15938 alt width=487 height=78></p><p style="text-align: center;">Figure 2: crafted shellcode to call the entrypoint</p><p><span lang=EN-GB>The payload is a DLL with one export function called &#8220;Loader.&#8221; The shellcode at the beginning calls the &#8220;Loader&#8221; export function.</span></p><p><span lang=EN-GB>The export function does the Windows loader job to load the DLL correctly and then executes the entrypoint.</span></p><p><span lang=EN-GB>Payload uses the wrapper function for every API call to obfuscate the API calling mechanism. In the wrapper function, there are stackstrings for the API function name and module name. In the picture below, you can see an example.</span></p><p><img loading=lazy decoding=async class=" wp-image-1117 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_APT5.jpg?x15938 alt width=265 height=300></p><p class=auto-cursor-target><span lang=EN-GB>It can be executed in 3 different ways:</span></p><table><tbody><tr><td><strong>Argument</strong></td><td style="text-align: center;"><strong>Description</strong></td></tr><tr><td>no argument</td><td>Set a registry key for persistence and create a directory for files and run the malware again with -app</td></tr><tr><td>-app</td><td><ul> <li>USB infection</li> <li>Stealing documents and storing in USB for air gap network</li> <li>Connect to the C&amp;C address and execute the commands</li></ul></td></tr><tr><td>-net</td><td><ul> <li>set the value of key registry &#8220;System\CurrentControlSet\Control\Network\Version&#8221; to &#8220;1&#8221;</li> <li>Set a registry key for persistence and create a directory for files and run with -app</li></ul></td></tr></tbody></table><p>For the persistence, it uses the registry run key:<br> <code>HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Run<br> HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Run</code></p><p class=auto-cursor-target><span lang=EN-GB>It creates %appdata%\Intel directory for additional modules and stolen information.</span></p><p class=auto-cursor-target><span lang=EN-GB>It uses a set of registry keys for storing data:</span></p><p><code>"System\CurrentControlSet\Control\Network\Version" → internet access status<br> "Software\CLASSES\ms-pu\PROXY" → C&amp;C server version</code></p><p class=auto-cursor-target><span lang=EN-GB>The PlugX TLS value names in the binary:</span></p><table><tbody><tr><td style="text-align: left;"><strong>Thread name</strong></td><td style="text-align: left;"><strong>Functionality</strong></td></tr><tr><td>CXOnline::OlStartProc</td><td>starts CXOnline::OlStartProcPipe</p><p>Initializes C&amp;C communication</td></tr><tr><td>CXOnline::OlStartProcPipe</td><td>Initializes pipe communication objects<br> Parses C&amp;C commands</td></tr><tr><td>CXFuncShell::ShellT1</p><p>CXFuncShell::ShellT2</td><td>Remote Shell</td></tr><tr><td>CXSalvation::SalExceptionHandler</td><td>Global exception handler for logging in SS.LOG file</td></tr><tr><td style="text-align: left;">CXSoHttp::SoWorkProc</td><td style="text-align: left;">Sends requests to the C&amp;C server</td></tr></tbody></table><h3>USB infection</h3><p>One of the features that distinguishes this variant from other PlugX variants is the capability of spreading through a USB stick. This part of the code is different from other parts of PlugX as we don&#8217;t see any TLS value name for this functionality.</p><p>After detecting the removable USB volume drive, the PlugX variant creates a hidden folder with name &#8220;RECYCLE.BIN&#8221; and copies the EXE file, the loader DLL, and encrypted DAT file. Then, it hides all of the folders in the root directory and creates LNK for each one in order to deceive the victim to click on the LNK files.</p><h3>Air gap network</h3><p><code>PlugX checks whether there's no internet access, then tries to find the USB stick and creates the BAT file in the following path, &lt;usb volume&gt;\RECYCLE.BIN\&lt;plugX clsid&gt;\tmp.bat, and executes it. At the end, it deletes the BAT file.</code></p><p><span lang=EN-GB>By running this BAT file, it gathers local and network information about the infected system and tries to send it out via USB.</span></p><pre><em><span lang=EN-GB>c</span><span lang=EN-GB>md.exe /c systeminfo &gt; &lt;usb volume&gt;\RECYCLE.BIN\&lt;plugX clsid&gt;\sys.info</span></em></pre><pre><em><span lang=EN-GB>cmd.exe /c ipconfig /all &gt;&gt; &lt;usb volume&gt;\RECYCLE.BIN\&lt;plugX clsid&gt;\sys.info</span></em></pre><pre><em><span lang=EN-GB>cmd.exe /c netstat -ano &gt;&gt; &lt;usb volume&gt;\RECYCLE.BIN\&lt;plugX clsid&gt;\sys.info</span></em></pre><pre><em><span lang=EN-GB>cmd.exe /c arp -a &gt;&gt; &lt;usb volume&gt;\RECYCLE.BIN\&lt;plugX clsid&gt;\sys.info</span></em></pre><pre><em><span lang=EN-GB>cmd.exe /c tasklist /v &gt;&gt; &lt;usb volume&gt;\RECYCLE.BIN\&lt;plugX clsid&gt;\sys.info


</span><span lang=EN-GB><img loading=lazy decoding=async class=" wp-image-1288 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_system_info.png?x15938 alt width=503 height=407 /></span></em></pre><p style="text-align: center;">Figure 3: run BAT file for gathering info</p><p><span lang=EN-GB>It searches the whole system for the following file extensions: </span></p><p><strong>document extensions<br> </strong>.doc<br> .docx<br> .ppt<br> .pptx<br> .xls<br> .xlsx<br> .pdf</p><p><span lang=EN-GB>It encrypts the mentioned documents on the fly via RC4, then copies them in its specific folder in the usb: &#8220;&lt;usb volume&gt;\RECYCLE.BIN\&lt;plugX clsid&gt;&#8221;.</span></p><p><span lang=EN-GB>The RC4 key is hardcoded in the payload binary.</span></p><p><span lang=EN-GB>From our telemetry data, we found a different way of stealing documents. It abuses RAR to search document files and compress them locally in order to send it to its CNC server.</span></p><p><span lang=EN-GB>The command line looks like this:</span></p><p><span lang=EN-GB>Rar.exe a -r -ed -m3 -dh -tk -hp&lt;password&gt; -v100m -ta&lt;file modified date after&gt; -n*.doc* -n*.xls* -n*.pdf &lt;compressed file path&gt; &lt;searching directory&gt;</span></p><p><span lang=EN-GB>As you can see, it&#8217;s looking for a specific date to compress .xls* and .pdf extensions. It also uses a password to encrypt file data and header.</span></p><h3><span lang=EN-GB>Configuration</span></h3><p><span lang=EN-GB>The config can be encrypted or be plaintext in the binary. If the config data starts with &#8220;XXXXXXXX,&#8221; it means the config is not encrypted.</span></p><p><span lang=EN-GB>If the config is encrypted, the config data is decryped via XORring with the hardcoded key inside of the binary. In different payloads that we could find, the key was &#8220;123456789&#8221;.</span></p><p><img loading=lazy decoding=async class=" wp-image-1287 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_checkxxxx.png?x15938 alt width=451 height=333></p><p style="text-align: center;">Figure 4: check config data before decryption</p><p><img loading=lazy decoding=async class=" wp-image-1285 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_config_decrypt1.png?x15938 alt width=369 height=380></p><p style="text-align: center;"><em>Figure </em><em>5</em><em>:</em> hardcoded key for config decryption</p><p><img loading=lazy decoding=async class=" wp-image-1286 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_config_decrypt2.png?x15938 alt width=405 height=496></p><p style="text-align: center;">Figure 6: config decryption routine</p><p><span lang=EN-GB>Inside the config, there is information about the folder name for PlugX modules, mutex name, C&amp;C IP and domain address, and port number.</span></p><p class=auto-cursor-target><span lang=EN-GB>We found 17 unique pieces of config data. This is the information we extracted from them:</span></p><p><strong>C&amp;C domains</strong></p><p>www.apple-net.com<br> www.mmfhlele.com<br> www.olk4.com<br> update.olk4.com<br> infosecvn.com<br> aridndvn.ccom<br> www.freesmadav.com<br> update.freesmadav.com<br> www.lameers.com</p><p><strong>C&amp;C IP address</strong></p><p>154.223.150.105<br> 43.251.182.114<br> 185.239.226.61<br> 167.88.180.132<br> 45.251.240.55</p><p><strong>Folder Names</strong></p><p>ESET Malware ProtectionrYF<br> ESET Malware ProtectionWgM<br> ESET Malware ProtectionTCp<br> ESET Malware ProtectionOWT<br> ESET Malware Protectionmld</p><pre><span lang=EN-GB>Microsoft Malware ProtectionGCg</span></pre><pre><span lang=EN-GB>Microsoft Malware Protectionfpx</span></pre><pre><span lang=EN-GB>Microsoft Malware ProtectionwSA</span></pre><pre><span lang=EN-GB>Microsoft Malware ProtectionpiW</span></pre><pre><span lang=EN-GB>Microsoft Malware ProtectionSaG</span></pre><pre><span lang=EN-GB>Microsoft Malware ProtectionDQA</span></pre><pre><span lang=EN-GB>Microsoft Malware ProtectioncdB</span></pre><pre><span lang=EN-GB>Microsoft Malware ProtectionNSP</span></pre><pre><span lang=EN-GB>Microsoft Malware ProtectionGHQ</span></pre><pre><span lang=EN-GB>Adobe Update HelpereQU</span></pre><pre><span lang=EN-GB>Adobe Update HelperzGI</span></pre><pre><span lang=EN-GB>Adobe Update HelperatX

So, when it initializes the connection and gets the C&amp;C version, it can decide how to communicate.</span></p><p class=auto-cursor-target><span lang=EN-GB>The CNC commands in this PlugX variant are similar to other versions with some minor differences.</span></p><p class=auto-cursor-target><span lang=EN-GB>It has 2 different groups of commands: group IDs 0x1001 and 0x1002. Once it gets the C&amp;C group ID, it decides how to respond to the commands.</span></p><p class=auto-cursor-target><span lang=EN-GB>For command group ID 0x1001:</span></p><table><tbody><tr><td><strong>Command</strong></td><td><strong>Description</strong></td></tr><tr><td>0x1001</td><td>get system information → memory status, computer name, username, OS version, PlugX CLSID, cpu speed, cpu architecture, width and height of screen</td></tr><tr><td>0x1002</td><td>start pipe communication → CXOnline::OlStartProcPipe</td></tr><tr><td>0x1003</td><td>echo input back</td></tr><tr><td>0x1005</td><td>exit process</td></tr></tbody></table><p class=auto-cursor-target><span lang=EN-GB>For command group ID 0x1002:</span></p><table><tbody><tr><td><strong>Command</strong></td><td><strong>Description</strong></td></tr><tr><td>0x3000</td><td>get disk information</td></tr><tr><td>0x3001</td><td>find file</td></tr><tr><td>0x3004</td><td>read file</td></tr><tr><td>0x3007</td><td>write file</td></tr><tr><td>0x300A</td><td>create directory</td></tr><tr><td>0x300B</td><td>check if the file existed</td></tr><tr><td>0x300C</td><td>create process in hidden desktop</td></tr><tr><td>0x300D</td><td>file operation (rename, move, delete, copy)</td></tr><tr><td>0x300E</td><td>get expanded environment information</td></tr><tr><td>0x300F</td><td>get plugX directory</td></tr><tr><td>0x7002</td><td>remote shell</td></tr></tbody></table><h4><span lang=EN-GB style="color: #1d1c1d;">Old communication mechanism</span></h4><p><span lang=EN-GB style="color: #1d1c1d;">It uses HTTP post request to start the communication. The URL it uses to send the request to the server is &#8220;/update?wd=&lt;random 8 digit number&gt;&#8221;</span></p><p><span lang=EN-GB style="color: #1d1c1d;">It uses this hardcoded user-agent:</span></p><pre><span lang=EN-GB>Mozilla/4.0 (compatible; MSIE 6.0; Windows NT 5.1;SV1;

</span></pre><p>It adds these parameters to the HTTP request header</p><pre><span lang=EN-GB>x-debug
x-request
x-content
x-storage

</span></pre><h4>New communication mechanism</h4><p><span lang=EN-GB style="color: #1d1c1d;">PlugX</span> sends an encrypted random ID to initialize the connection via raw TCP and then receives a response from server.</p><p>The first 16 bytes are the header:</p><ul> <li>The first 4 bytes are the encryption key.</li> <li>The second 4 bytes are CNC command code (the 29th bit shows whether the data is compressed and 30th bits shows whether the data is encrypted).</li> <li>The next 2 bytes show the size of data (16 bytes header is not included).</li> <li>The header is encrypted as well with the first DWORD sent by the server.</li></ul><p style="text-align: center;"><img loading=lazy decoding=async class="size-full wp-image-1145 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_APT10.jpg?x15938 alt width=304 height=168>Figure 7: structure of C&amp;C communication header</p><p class=auto-cursor-target><span lang=EN-GB>The encryption algorithm used for communication is XOR but adds a hardcoded value to the key for every round of encryption. The hardcoded value for the binaries that we found was &#8220;6666&#8221;.</span></p><p><img loading=lazy decoding=async class="size-full wp-image-1147 aligncenter" src=https://www.webassetscdn.com/avira/prod-blog/wp-content/uploads/2020/08/plug_APT11.jpg?x15938 alt width=304 height=306></p><p style="text-align: center;">Figure 8: decryption routine for communication data</p><h2><span lang=EN-GB>Conclusion</span></h2><p class=auto-cursor-target><span lang=EN-GB>The Mustang Panda APT actor uses PlugX with minor changes, in an attempt to evade detection.
