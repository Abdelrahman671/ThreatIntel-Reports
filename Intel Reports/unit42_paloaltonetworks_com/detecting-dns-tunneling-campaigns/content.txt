Many organizations leave UDP and TCP port 53 open in their firewalls, which is the port DNS uses for communication. They also often leave this port unmonitored, making it an attractive target for attackers to use as a covert communications channel.\r\n\r\nDNS tunneling leverages the DNS protocol to encode data within DNS queries and responses. Therefore, cyberattackers tend to use DNS tunneling for data exfiltration, command and control (C2), or bypassing security measures.\r\n\r\n[caption id=\"attachment_136983\" align=\"alignnone\" width=\"900\"] Figure 1. The hidden information is transmitted via DNS tunneling attacks.[\/caption]\r\n\r\nFigure 1 illustrates an example of covert communications using DNS tunneling techniques.\r\n\r\nAttackers first infect a client system with malware that is able to steal the user\u2019s data. Then, the stolen data is encoded and embedded within subdomains and is transmitted over DNS queries. The attackers can receive and decode the data by hosting an authoritative DNS (aDNS) server of the root domain (e.g., helloworld[.]com in Figure 1).\r\n\r\nDNS tunneling can achieve stealthiness because the recursive DNS servers enable indirect communications between client systems and attacker-controlled authoritative DNS servers. Attackers are also able to transmit commands by encoding data into DNS responses.\r\n\r\nMalicious actors also abuse DNS tunneling in attack campaigns. For example, the Iranian threat group Evasive Serpens (aka OilRig) employed DNS tunneling to communicate between infected hosts and their C2 servers, targeting critical infrastructure in the Middle East. Another Iranian threat group we track as Obscure Serpens (aka DarkHydrus) also used DNS tunneling when it targeted government agencies and educational institutions in the Middle East in 2016.\r\n\r\nThe detection of DNS tunneling has typically focused on the attributes of individual domains, such as lexical or infrastructure patterns. However, defenders often overlook additional attributes.\r\n\r\nFor example, correlations between tunneling domains can help us monitor significant clusters for tunneling detection and discover emerging campaigns. In this article, we reveal common attributes shared within individual campaigns that could facilitate automated detection of new DNS tunneling campaigns.\r\n\r\nBased on these attributes, we have designed and deployed the first campaign monitoring system in our products. This system aims to automatically determine if a couple of historically detected tunneling domains originate from an undisclosed campaign, or if a new tunneling domain belongs to any existing well-documented campaign.\r\nAttributes for Campaign Monitoring\r\nThis section presents methods that can identify new tunneling campaigns from daily tunneling detection results. Our detection relies on the fact that domains used in each individual tunneling campaign typically exhibit similar attributes because the same malicious actors registered the domains, or the campaigns used the same tunneling methods.\r\n\r\nWe discuss the common attributes we identified within each tunneling campaign. To the best of our knowledge, these distinctive intercorrelations among tunneling campaigns have not been fully studied previously.\r\n\r\nThe following attributes relate to infrastructure, configurations, lexical patterns and targets that could be shared across the tunneling domains in each campaign:\r\n\r\n \tAuthoritative nameserver: The nameservers used by tunneling campaigns are usually owned and controlled by attackers themselves, as attackers should have unrestricted access to fetch query logs and manipulate the response. To limit the attack cost, attackers tend to use a single self-hosted authoritative DNS server. This centralized server enables them to efficiently manage and process DNS traffic of multiple tunneling domains.\r\n \tDNS configurations: In a single tunneling campaign, the DNS configurations used for each domain are usually similar or even identical. This is because the same attacker group sets and stores the configuration in the same aDNS server. The shared infrastructure settings allow attackers to effectively deploy and control the various domains in the campaign, but they also provide a significant indicator for security researchers. For instance, RussianSite and 8NS tunneling campaigns have their own DNS configuration pattern with a single aDNS IP address.\r\n \tPayload encoding: To launch a tunneling campaign, attackers usually employ consistent tunneling tools or encoding methods to encapsulate their payload data into subdomains. Therefore, within a single campaign, the patterns of subdomains across various root domains often show significant similarities. This consistency helps attackers manage their C2 or data exfiltration processing. For example, RussianSite and NSfinder use the same encoding method within each campaign.\r\n \tDomain registration: Domains used in individual campaigns frequently share the same top-level domain (TLD). Also, the associated root domains use a notably similar naming scheme, indicating a consistent choice made by the attacker group. Moreover, the registration time of these domains tends to align closely from their WHOIS records.\r\n \tAttack target: Each tunneling campaign tends to target one or a few specific victim types (e.g., governments, public infrastructures, finance\/banking\/investments or health\/medical care).\r\n\r\nBased on the above criteria, we used machine learning techniques to explore the potential new campaigns among the detected tunneling domains.\r\nNew Campaigns\r\nBy focusing on the correlations between DNS tunneling domains, we detected four previously undiscovered campaigns. Each campaign shares some attributes from infrastructure, payloads, domain registration or targeted victims. This campaign targets the finance and healthcare industries, thus we named it FinHealthXDS.\r\n\r\nBy analyzing the passive DNS records, we discovered this customized Cobalt Strike DNS beaconing format.                 <h2>Executive Summary</h2>
<p>This article reviews four previously undisclosed domain name system (DNS) tunneling campaigns that occurred in recent months. We identified these new campaigns through our recently deployed campaign monitoring system, which can identify new, potentially malicious campaigns from daily tunneling detection.</p>
<p>DNS tunneling is a technique threat actors use to encode non-DNS traffic data within DNS packet traffic. This allows the information to bypass traditional network firewalls and establish covert communication channels for data exfiltration and infiltration.</p>
<p>Our new campaign monitoring system is designed to detect tunneling domains based on the techniques and attributes commonly used in malicious campaigns. These unique intercorrelations among individual tunneling campaigns can reveal new tunneling domains.</p>
<p>For example, can attackers connect a new domain to any existing well-documented tunneling campaign? Do a couple of recently detected tunneling domains originate from an undisclosed campaign?</p>
<p>Hence, instead of investigating the tunneling domains individually, we seek to analyze these domains from the perspective of common attributes and quickly uncover new campaigns.</p>
<p>We have deployed this tunneling campaign monitoring system in our <a href="https://www.paloaltonetworks.com/network-security/advanced-dns-security">Advanced DNS Security</a> service. This system allows organizations to secure their DNS traffic by identifying and blocking new potential campaigns from tunneling domains. It also provides detailed campaign information for customers who can log in to our <a href="https://urlfiltering.paloaltonetworks.com/">Test A Site</a> service.</p>
<p>Palo Alto Networks <a href="https://docs.paloaltonetworks.com/ngfw" target="_blank" rel="noopener">Next-Generation Firewall</a> customers can access the tunneling campaign information with the <a href="https://www.paloaltonetworks.com/network-security/advanced-dns-security" target="_blank" rel="noopener">Advanced DNS Security</a> subscription and receive protections against malicious indicators (domains and IP addresses) mentioned in this article via <a href="https://docs.paloaltonetworks.com/advanced-url-filtering/administration" target="_blank" rel="noopener">Advanced URL Filtering</a>. The <a href="https://www.paloaltonetworks.com/network-security/advanced-wildfire" target="_blank" rel="noopener">Advanced WildFire</a> machine-learning models and analysis techniques have been reviewed and updated in light of the IoCs shared in this research.</p>
<p>The <a href="https://docs.paloaltonetworks.com/ngfw" target="_blank" rel="noopener">Next-Generation Firewall</a> with the <a href="https://docs.paloaltonetworks.com/advanced-threat-prevention/administration" target="_blank" rel="noopener">Advanced Threat Prevention</a> security subscription can help block the attacks with best practices by identifying the corresponding malware samples and command and control (C2) traffic.</p>
<p>Palo Alto Networks <a href="https://docs-cortex.paloaltonetworks.com/r/Cortex-XDR/Cortex-XDR-Pro-Administrator-Guide/Analytics-Concepts" target="_blank" rel="noopener">Cortex Analytics</a> customers receive protection against DNS tunneling techniques mentioned in this article via the DNS tunneling analytics detector.</p>
<p><a href="https://www.paloaltonetworks.com/cortex" target="_blank" rel="noopener">Cortex</a> also helps protect against malware from the Hiloti family and others through features such as prevention for shellcode injection.</p>
<p>If you think you might have been compromised or have an urgent matter, contact the <a href="https://start.paloaltonetworks.com/contact-unit42.html" target="_blank" rel="noopener">Unit 42 Incident Response team</a>.</p>

<h2><a id="post-136979-_4lt92rr5muov"></a><strong>DNS Tunneling: An Overview </strong></h2>
<p>DNS is a critical component of the internet infrastructure responsible for translating human-readable domain names into IP addresses. Many organizations leave UDP and TCP port 53 open in their firewalls, which is the port DNS uses for communication. They also often leave this port unmonitored, making it an attractive target for attackers to use as a covert communications channel.</p>
<p>DNS tunneling leverages the DNS protocol to encode data within DNS queries and responses. Therefore, cyberattackers tend to use DNS tunneling for data exfiltration, command and control (C2), or bypassing security measures.</p>
<figure id="attachment_136983" aria-describedby="caption-attachment-136983" style="width: 900px" class="wp-caption alignnone"><img decoding="async" class="wp-image-136983" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2024/10/word-image-78751-136979-1.png" alt="Illustration depicting the DNS query process involving a client system, an ISP DNS server, and a malicious server, with steps labeled using technical notations and domains." width="900" height="247" srcset="https://unit42.paloaltonetworks.com/wp-content/uploads/2024/10/word-image-78751-136979-1.png 2048w, https://unit42.paloaltonetworks.com/wp-content/uploads/2024/10/word-image-78751-136979-1-786x216.png 786w, https://unit42.paloaltonetworks.com/wp-content/uploads/2024/10/word-image-78751-136979-1-1920x528.png 1920w, https://unit42.paloaltonetworks.com/wp-content/uploads/2024/10/word-image-78751-136979-1-768x211.png 768w, https://unit42.paloaltonetworks.com/wp-content/uploads/2024/10/word-image-78751-136979-1-1536x422.png 1536w" sizes="(max-width: 900px) 100vw, 900px" /><figcaption id="caption-attachment-136983" class="wp-caption-text">Figure 1. The hidden information is transmitted via DNS tunneling attacks.</figcaption></figure>
<p>Figure 1 illustrates an example of covert communications using DNS tunneling techniques.</p>
<p>Attackers first infect a client system with malware that is able to steal the user’s data. Then, the stolen data is encoded and embedded within subdomains and is transmitted over DNS queries. The attackers can receive and decode the data by hosting an authoritative DNS (aDNS) server of the root domain (e.g., helloworld[.]com in Figure 1).</p>
<p>DNS tunneling can achieve stealthiness because the recursive DNS servers enable indirect communications between client systems and attacker-controlled authoritative DNS servers. Attackers are also able to transmit commands by encoding data into DNS responses.</p>
<p>Malicious actors also abuse DNS tunneling in attack campaigns. For example, the Iranian threat group <a href="https://unit42.paloaltonetworks.com/tag/evasive-serpens/">Evasive Serpens</a> (aka OilRig) <a href="https://unit42.paloaltonetworks.com/oilrig-novel-c2-channel-steganography/" target="_blank" rel="noopener">employed DNS tunneling</a> to communicate between infected hosts and their C2 servers, targeting critical infrastructure in the Middle East. Another Iranian threat group we track as Obscure Serpens (aka <a href="https://unit42.paloaltonetworks.com/tag/darkhydrus/" target="_blank" rel="noopener">DarkHydrus</a>) also used DNS tunneling when it targeted government agencies and educational institutions <a href="https://unit42.paloaltonetworks.com/unit42-new-threat-actor-group-darkhydrus-targets-middle-east-government/" target="_blank" rel="noopener">in the Middle East in 2016</a>.</p>
<p>The detection of DNS tunneling has typically focused on the attributes of individual domains, such as lexical or infrastructure patterns. However, defenders often overlook additional attributes.</p>
<p>For example, correlations between tunneling domains can help us monitor significant clusters for tunneling detection and discover emerging campaigns. In this article, we reveal common attributes shared within individual campaigns that could facilitate automated detection of new DNS tunneling campaigns.</p>
<p>Based on these attributes, we have designed and deployed the first campaign monitoring system in our products. This system aims to automatically determine if a couple of historically detected tunneling domains originate from an undisclosed campaign, or if a new tunneling domain belongs to any existing well-documented campaign.</p>
<h2><a id="post-136979-_hx7gmwky9div"></a><strong>Attributes for Campaign Monitoring</strong></h2>
<p>This section presents methods that can identify new tunneling campaigns from daily tunneling detection results. Our detection relies on the fact that domains used in each individual tunneling campaign typically exhibit similar attributes because the same malicious actors registered the domains, or the campaigns used the same tunneling methods.</p>
<p>We discuss the common attributes we identified within each tunneling campaign. To the best of our knowledge, these distinctive intercorrelations among tunneling campaigns have not been fully studied previously.</p>
<p>The following attributes relate to infrastructure, configurations, lexical patterns and targets that could be shared across the tunneling domains in each campaign:</p>
<ul>
<li><strong>Authoritative nameserver: </strong>The nameservers used by tunneling campaigns are usually owned and controlled by attackers themselves, as attackers should have unrestricted access to fetch query logs and manipulate the response. To limit the attack cost, attackers tend to use a single self-hosted authoritative DNS server. This centralized server enables them to efficiently manage and process DNS traffic of multiple tunneling domains.</li>
<li><strong>DNS configurations: </strong>In a single tunneling campaign, the DNS configurations used for each domain are usually similar or even identical. This is because the same attacker group sets and stores the configuration in the same aDNS server. The shared infrastructure settings allow attackers to effectively deploy and control the various domains in the campaign, but they also provide a significant indicator for security researchers. For instance, RussianSite and 8NS tunneling campaigns have their own DNS configuration pattern with a single aDNS IP address.</li>
<li><strong>Payload encoding: </strong>To launch a tunneling campaign, attackers usually employ consistent tunneling tools or encoding methods to encapsulate their payload data into subdomains. Therefore, within a single campaign, the patterns of subdomains across various root domains often show significant similarities. This consistency helps attackers manage their C2 or data exfiltration processing. For example, RussianSite and NSfinder use the same encoding method within each campaign.</li>
<li><strong>Domain registration: </strong>Domains used in individual campaigns frequently share the same top-level domain (TLD). Also, the associated root domains use a notably similar naming scheme, indicating a consistent choice made by the attacker group. Moreover, the registration time of these domains tends to align closely from their WHOIS records.</li>
<li><strong>Attack target: </strong>Each tunneling campaign tends to target one or a few specific victim types (e.g., governments, public infrastructures, finance/banking/investments or health/medical care).</li>
</ul>
<p>Based on the above criteria, we used machine learning techniques to explore the potential new campaigns among the detected tunneling domains.</p>
<h2><a id="post-136979-_z9w6i4m4xoc2"></a><strong>New Campaigns</strong></h2>
<p>By focusing on the correlations between DNS tunneling domains, we detected four previously undiscovered campaigns. Each campaign shares some attributes from infrastructure, payloads, domain registration or targeted victims. We show the detailed case studies on these campaigns in this section.</p>
<h3><a id="post-136979-_vma2a3mu6w4a"></a>FinHealthXDS</h3>
<p>The first campaign contains 12 domains that use a customized <a href="https://hstechdocs.helpsystems.com/manuals/cobaltstrike/current/userguide/content/topics/listener-infrastructue_beacon-dns.htm" target="_blank" rel="noopener">DNS beaconing</a> format for Cobalt Strike C2 communications. While Cobalt Strike has a default configuration for <a href="https://hstechdocs.helpsystems.com/manuals/cobaltstrike/current/userguide/content/topics/malleable-c2_dns-beacons.htm" target="_blank" rel="noopener">DNS C2 communications through a malleable C2 profile</a>, this campaign leverages a customized format.</p>
<p>In this format, the associated DNS queries use three-letter prefixes (e.g., xds) to indicate the function of the queries used in this tunneling traffic. This campaign targets the finance and healthcare industries, thus we named it <strong>FinHealthXDS</strong>.</p>
<p>By analyzing the passive DNS records, we discovered this customized Cobalt Strike DNS beaconing format. This campaign uses the xds prefix to indicate command request queries and resolves to 40.112.72[.]205 as the IP address.</p>
<p>After obtaining the returned IP address such as 40.112.72[.]62, malware will calculate the XOR value of the last byte to determine the returned command. For example, 205 XOR 62 = 243, which typically indicates transferring data with TXT records (mode dns-txt command).</p>
<p>Below, we find two examples of DNS A record queries for transferring subsequent data using TXT records.</p>
<ul>
<li><span style="font-family: 'courier new', courier, monospace;">xds.5af195b6.gear.&lt;rootdom&gt; A 40.112.72[.]205</span></li>
<li><span style="font-family: 'courier new', courier, monospace;">xds.5af195b6.gear.&lt;rootdom&gt; A 40.112.72[.]62</span></li>
</ul>
<p><a id="post-136979-_ytosnju69woa"></a> This campaign can use either A records or TXT records for an infected host to receive data. When using A records, the DNS queries follow a format with a customized prefix of pro (compared to the default value of cdn when using <a href="https://hstechdocs.helpsystems.com/manuals/cobaltstrike/current/userguide/content/topics/malleable-c2_dns-beacons.htm" target="_blank" rel="noopener">malleable C2 profiles for DNS beaconing</a> in Cobalt Strike).</p>
<p style="padding-left: 40px;"><span style="font-family: 'courier new', courier, monospace;">pro.[hex-counter][8-digit-hex-random-number].[beacon-id].[subdomain-padding].&lt;rootdom&gt; A [hex-infiltration-data]</span></p>
<p>When using TXT records for data infiltration, the DNS queries use the prefix snd instead of the default api prefix used in the malleable C2 for DNS beaconing in Cobalt Strike. An example of the format is shown below.</p>
<p style="padding-left: 40px;"><span style="font-family: 'courier new', courier, monospace;">snd.[hex-counter][8-digit-hex-random-number].[beacon-id].[subdomain-padding].&lt;rootdom&gt; TXT [infiltration-data]</span></p>
<p>To exfiltrate data to a C2 server, the DNS queries leverage two prefixes (i.e., txt for short messages and del for long messages, not the default prefix of post). The format is as follows.</p>
<p style="padding-left: 40px;"><span style="font-family: 'courier new', courier, monospace;">[txt|del].[num-of-message][encoded-message].[counter][random-number].[beacon-id].[subdomain-padding].&lt;rootdom&gt; A 40.112.72[.]205</span></p>

<h3><a id="post-136979-_1518bkjdn1na"></a>RussianSite</h3>
<p>The second tunneling campaign contains over 100 domains, all of which share the same nameserver IP address of <span style="font-family: 'courier new', courier, monospace;">185.161.248[.]253</span> from Russia. Most domains in this campaign end with the TLD <span style="font-family: 'courier new', courier, monospace;">.site</span>, while only a very few of these domains use <span style="font-family: 'courier new', courier, monospace;">.website</span>. Therefore, based on the aDNS IP address and the TLD patterns, we named this campaign <strong>RussianSite</strong>.</p>
<p>The subdomain contains two parts: a 5-character alphanumeric payload and a 1-letter or 2-letter padding that is specific in each domain. The A records of these campaign domains are distributed worldwide. However, to receive tunneling information from aDNS servers, configuring a valid aDNS IP address is mandatory for attackers, while A records are discretionary and could be invalid.</p>
<p>In Table 2 we listed examples of the domains, along with a query sample, nameservers and nameserver IP address. We have obfuscated the listed FQDN queries so no customer data is revealed.</p>
<p>We observed this campaign targeting 10 organizations from higher education. Of these 10 targets, five are governments and public infrastructure, four are medical/health institutions and one is a public accounting firm.</p>

<h3><a id="post-136979-_yf1dr3oajey4"></a>8NS</h3>

<p><a id="post-136979-_iimwo95tp6n5"></a> This campaign is driven by multiple Trojans. For example, malware from the Hiloti family <span style="font-family: 'courier new', courier, monospace;">0b99db286f3708fedf7e2bb8f24df1af13811fe46b017b6c3e7e002852479430</span>, can contact the domain <span style="font-family: 'courier new', courier, monospace;">011807dd0303[.]lantzel[.]com.</span></p>
<p><a id="post-136979-_dkd08z8t4b4n"></a> This Hiloti sample secretly downloads malicious files from a remote server and then injects the unpacked malware into <span style="font-family: 'courier new', courier, monospace;">explore[.]exe</span>. The malware creates three registry entries under <span style="font-family: 'courier new', courier, monospace;">HKEY_CURRENT_USER\SOFTWARE\Microsoft\Windows\Bfetipi</span> or <span style="font-family: 'courier new', courier, monospace;">HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\Bfetipi</span>, which a C2 domain generation algorithm uses to send client online messages to the C2 server through DNS queries, as depicted in Figure 3.</p>

<li><span style="font-family: 'courier new', courier, monospace;">n</span>: Indicator of a new process (<span style="font-family: 'courier new', courier, monospace;">n</span>) or existing process (<span style="font-family: 'courier new', courier, monospace;">e</span>), determined by mutex</li>

<li><span style="font-family: 'courier new', courier, monospace;">887</span>: Hard-coded value</li>

<h3><a id="post-136979-_8mjv23pjhuuw"></a>NSfinder</h3>
<p>The fourth campaign contains over 50 domains. All domains are named by combining three words with the last word of finder (e.g., <span style="font-family: 'courier new', courier, monospace;">unlimitedpartnersfinder[.]com</span>). The subdomains are composed of multiple similar segments, each of which contains a prefix <span style="font-family: 'courier new', courier, monospace;">ns500</span> and a number. Based on the root domain and subdomain patterns, we named this campaign <strong>NSfinder</strong>.</p>
<p>NSfinder is related to multiple malicious IP addresses, mainly from Europe. Each domain typically uses three IP addresses each time for both aDNS IP addresses and resolved IP addresses.</p>
<p>The IP addresses used by different domains have high overlapping. Also, we observe that attackers periodically change the IP address set for each domain, and the time to live (TTL) is only 60 seconds.</p>
<p>NSfinder performs attacks by setting up adult websites, and it lures victims to enter their credit card information. This campaign also correlates with multiple Trojans, such as IcedID.</p>
<p>For example, attackers frequently used the nameserver IP address <span style="font-family: 'courier new', courier, monospace;">206.188.197[.]111</span> in the campaign to communicate with a malware sample with the SHA256 hash <span style="font-family: 'courier new', courier, monospace;">dfb3e5f557a17c8cdebdb5b371cf38c5a7ab491b2aeaad6b4e76459a05b44f28</span>, identified as IcedID by different sources in VirusTotal. The nameserver IP address <span style="font-family: 'courier new', courier, monospace;">185.81.114[.]183</span> communicated with the malware <span style="font-family: 'courier new', courier, monospace;">c22d25107e48962b162c935a712240c0a4486b38891855f0e53d5eb972406782</span>, <a href="https://bazaar.abuse.ch/sample/c22d25107e48962b162c935a712240c0a4486b38891855f0e53d5eb972406782/" target="_blank" rel="noopener">identified as RedLine stealer</a>.</p>
<p>We observed this campaign having massive traffic within one month in 2023. The ns500 tokens have over a hundred variants, where the top tokens follow the distribution of English letters. We observed thousands of victims related to this campaign, mainly coming from high tech, education and manufacturing fields.</p>

<h2><a id="post-136979-_zbqi7sveh2o7"></a><strong>Conclusion</strong></h2>
<p>The domains within each tunneling campaign share some common attributes, such as the following:</p>
<ul>
<li>Infrastructure</li>
<li>DNS configurations</li>
<li>Payload encoding</li>
<li>Domain registration patterns</li>
<li>Attack targets</li>
</ul>
<p>These attributes enable us to identify significant clusters among the tunneling detection results so that we can discover the emerging tunneling campaigns.</p>
<p>This method of detection has revealed four new campaigns that we have described in this article and have named:</p>
<ul>
<li>FinHealthXDS</li>
<li>RussianSite</li>
<li>8NS</li>
<li>NSfinder</li>
</ul>
<p>We have implemented these techniques as a new campaign monitoring system into our <a href="https://www.paloaltonetworks.com/network-security/advanced-dns-security" target="_blank" rel="noopener">Advanced DNS Security</a> service, providing campaign information and descriptions to our customers. With this system, we have detected four new campaigns from daily tunneling detection results.</p>

	window.captchaComplete = function() {
		subscribeSuccess = true;
		if ($(mail).val() != '' && isEmail($(mail).val())) {
			$(subscription_form_button).removeClass('is-disabled');
		}
		setTimeout(function() {
			$(email).focus();
			$('.g-recaptcha iframe').attr('tabindex', '-1');
		}, 100)
	}
	window.captchaExpires = function() {
		subscribeSuccess = false;
		$(subscription_form_button).addClass('is-disabled', true);
	} 
	$(subscription_form).submit(function(e) {
		e.preventDefault();
		e.stopImmediatePropagation();
		updateEmailMask();
		var success = true;
		var form = $(this);
		var mail = form.find('input[name="Email"]');
		
		if (mail.val() === '') {
			mail.addClass('has-error');
			showError(1);
			success = false;	
		}  else if (!isEmail(mail.val())){
			showError(2);
			success = false;
		}
		else {
			mail.removeClass('has-error');
			$('.error-mail').addClass('d-none');
		}

		if (!subscribeSuccess) {
			$('.error-recaptcha').removeClass('d-none');
		} else {
			$('.error-recaptcha').addClass('d-none');
		}

		if (success && subscribeSuccess) {
			$.ajax({

					form.find('button').removeClass('is-loading');
					$(email).val('');
					clearError();
				},
				error: function(jqXHR, textStatus, errorThrown) {
					$(subscription_form_button).addClass('is-disabled', true);
					form.find('button').removeClass('is-loading');
					
				}
			});
		}
		return false;
	});
	function showError(error_type){
		if(error_type == 1) {
			$('.error-mail').text("Please enter the email address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		} else if(error_type == 2){
			$('.error-mail').text("Please provide a valid e-mail address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		}	
		$(subscription_form_button).removeClass('is-loading');
	}
	function clearError(){
		$('.error-mail').text("").removeClass('error-show');;
		$(subscription_form_button).removeClass('is-loading');
		$(subscription_form_button).removeClass('is-disabled');
	}

		var email = $(this).val();
		if (isEmail(email) ) {			
			clearError();
		} else if(email == ""){
			clearError();
		} else{
			showError(2);			  
		}
    });

	function isEmail(email) {
		var re = /^(([^<>()\[\]\\.,;:\s@"]+(\.[^<>()\[\]\\.,;:\s@"]+)*)|(".+"))@((\[[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\])|(([a-zA-Z\-0-9]+\.)+[a-zA-Z]{2,}))$/;

	}

	}

	function maskEmailAddress (emailAddress) {
		function mask(str) {
			var strLen = str.length;
			if (strLen > 4) {
				return str.substr(0, 1) + str.substr(1, strLen - 1).replace(/\w/g, '*') + str.substr(-1,1);
			}
			return str.replace(/\w/g, '*');
		}
		return emailAddress.replace(/([\w.]+)@([\w.]+)(\.[\w.]+)/g, function (m, p1, p2, p3) {
			return mask(p1) + '@' + mask(p2) + p3;
		});

		return emailAddress;
	}
