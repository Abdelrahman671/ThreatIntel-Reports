                     <p>This new ransomware was discovered by Michael Gillespie on 8 February 2019 and it is still improving over time. This blog will explain the technical details and share information about how this new ransomware family is working. There are some variants of the Clop ransomware but in this report, we will focus on the main version and highlight part of those variations. The main goal of Clop is to encrypt all files in an enterprise and request a payment to receive a decryptor to decrypt all the affected files. To achieve this, we observed some new techniques being used by the author that we have not seen before. Clearly over the last few months we have seen more innovative techniques appearing in ransomware.</p>
<h3>Clop Overview</h3>
<p>The Clop ransomware is usually packed to hide its inner workings. The sample we analyzed was also signed with the following certificate in the first version (now revoked):</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96084" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/revoked-certificate.png" alt="" width="694" height="243" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/revoked-certificate.png 694w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/revoked-certificate-300x105.png 300w" sizes="(max-width: 694px) 100vw, 694px" /></p>
<p style="text-align: center;">FIGURE 1. Packer signed to avoid av programs and mislead the user</p>
<p>Signing a malicious binary, in this case ransomware, may trick security solutions to trust the binary and let it pass. Although this initial certificate was revoked in a few days, another version appeared soon after with another certificate:</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96085" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/new-certificate.png" alt="" width="418" height="523" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/new-certificate.png 418w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/new-certificate-240x300.png 240w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/new-certificate-400x500.png 400w" sizes="(max-width: 418px) 100vw, 418px" /></p>
<p style="text-align: center;">FIGURE 2. New certificate in new version</p>
<p>This sample was discovered by MalwareHunterTeam (<a href="https://twitter.com/malwrhunterteam" target="_blank" rel="noopener noreferrer">https://twitter.com/malwrhunterteam</a>) on the 26 February, 2019.</p>
<p>We discovered the following Clop ransomware samples which were signed with a certificate:</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96086" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/signed-samples.png" alt="" width="781" height="384" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/signed-samples.png 781w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/signed-samples-300x148.png 300w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/signed-samples-768x378.png 768w" sizes="(max-width: 781px) 100vw, 781px" /></p>
<p>This malware is prepared to avoid running under certain conditions, for example in the first version it requests to be installed as a service; if that will not succeed, it will terminate itself.</p>
<p>The malware’s first action is to compare the keyboard of the victim computer using the function “GetKeyboardLayout”  against the hardcoded values.</p>
<p>This function returns the user keyboard input layout at the moment the malware calls the function.</p>
<p>The malware checks that the layout is bigger than the value 0x0437 (Georgian), makes some calculations with the Russian language (0x0419) and with the Azerbaijan language (0x082C). This function will return 1 or 0, 1 if it belongs to Russia or another CIS country, or 0 in every other case.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96087" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/keyboard-layout-check.png" alt="" width="671" height="316" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/keyboard-layout-check.png 671w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/keyboard-layout-check-300x141.png 300w" sizes="(max-width: 671px) 100vw, 671px" /></p>
<p style="text-align: center;">FIGURE 3. Checking the keyboard layout</p>
<p>If the function returns 0, it will go to the normal flow of the malware, otherwise it will get the device context of the entire screen with the function “GetDC”. Another condition will come from the function “GetTextCharset” that returns the font used in the system if it does not have the value 0xCC (RUSSIAN_CHARSET). If it is the charset used, the malware will delete itself from the disk and terminate itself with “TerminateProcess” but if it is not this charset, it will continue in the normal flow This double check circumvents users with a multisystem language, i.e. they have the Russian language installed but not active in the machine to avoid this type of malware.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96088" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/check-charset.png" alt="" width="707" height="229" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/check-charset.png 707w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/check-charset-300x97.png 300w" sizes="(max-width: 707px) 100vw, 707px" /></p>
<p style="text-align: center;">FIGURE 4. Check the text charset and compare with Russian charset</p>
<p>The code that is supposed to delete the ransomware from the disk contains an error. It will call directly to the prompt of the system without waiting for the malware to finish.  This means that the execution of the command will be correct but, as the malware is still running, it will not delete it from the disk. This happens because the author did not use a “timeout” command.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96089" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/deletion.png" alt="" width="766" height="273" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/deletion.png 766w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/deletion-300x107.png 300w" sizes="(max-width: 766px) 100vw, 766px" /></p>
<p style="text-align: center;">FIGURE 5. Deletion of the malware itself</p>
<p>The next action of the malware is to create a new thread that will start all processes. With the handle of this thread, it will wait for an infinite amount of time to finish with the “WaitForSingleObject” function and later return to the winMain function and exit.</p>
<p>This thread’s first action is to create a file called “Favorite” in the same folder as the malware. Later, it will check the last error with “GetLastError” and, if the last error was 0,  it will wait with the function “Sleep” for 5 seconds.</p>
<p>Later the thread will make a dummy call to the function “EraseTape” with a handle of 0, perhaps to disturb the emulators because the handle is put at 0 in a hardcoded opcode, and later a call to the function “DefineDosDeviceA” with an invalid name that returns another error. These operations will make a loop for 666000 times.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96090" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/disturbing-the-analysis.png" alt="" width="717" height="326" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/disturbing-the-analysis.png 717w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/disturbing-the-analysis-300x136.png 300w" sizes="(max-width: 717px) 100vw, 717px" /></p>
<p style="text-align: center;">FIGURE 6. Loop to disturb the analysis</p>
<p>The next action is to search for some processes with these names:</p>
<ul>
<li>SBAMTray.exe (Vipre antivirus product)</li>
<li>SBPIMSvc.exe (Sunbelt AntiMalware antivirus product)</li>
<li>SBAMSvc.exe (GFI AntiMalware antivirus product)</li>
<li>VipreAAPSvc.exe (Vipre antivirus product)</li>
<li>WRSA.exe (WebRoot antivirus product)</li>
</ul>
<p>If some of these processes are discovered, the malware will wait 5 seconds using “Sleep” and later another 5 seconds. After those &#8220;sleep&#8221;, the malware will continue with their normal flow. If these processes are not detected, it will access to their own resources and extract it with the name “OFFNESTOP1”. That resource is encrypted but has inside a “.bat” file.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96091" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/first-resource.png" alt="" width="924" height="276" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/first-resource.png 924w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/first-resource-300x90.png 300w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/first-resource-768x229.png 768w" sizes="(max-width: 924px) 100vw, 924px" /></p>
<p style="text-align: center;">FIGURE 7. Access to the first resource crypted</p>
<p>The decryption is a simple XOR operation with bytes from this string:</p>
<p>“Po39NHfwik237690t34nkjhgbClopfdewquitr362DSRdqpnmbvzjkhgFD231ed76tgfvFAHGVSDqhjwgdyucvsbCdigr1326dvsaghjvehjGJHGHVdbas”.</p>
<p>The next action is to write this batch file in the same folder where the malware stays with the function “CreateFileA”.  The file created has the name “clearsystems-11-11.bat”. Later will launch it with “ShellExecuteA”, wait for 5 seconds to finish and delete the file with the function “DeleteFileA”.</p>
<p>It is clear that the authors are not experienced programmers because they are using a .bat file for the next actions:</p>
<ul>
<li>Delete the shadow volumes with vssadmin (“vssadmin Delete Shadows /all /quiet”).</li>
<li>Resize the shadow storage for all units starting from C to H units’ letters (hardcoded letters) to avoid the shadow volumes being made again.</li>
<li>Using bcedit program to disable the recovery options in the boot of the machine and set to ignore any failure in the boot warning the user.</li>
</ul>
<p>All these actions could have been performed in the malware code itself, without the need of an external file that can be detected and removed.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96092" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/disabling-shadow-volumes.png" alt="" width="584" height="294" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/disabling-shadow-volumes.png 584w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/disabling-shadow-volumes-300x151.png 300w" sizes="(max-width: 584px) 100vw, 584px" /></p>
<p style="text-align: center;">FIGURE 8. The BAT file to disable the shadow volumes and more security</p>
<p>The next action is to create a mutex with the name hardcoded “Fany&#8212;Fany&#8212;6-6-6” and later make a call to the function “WaitForSingleObject” and check the result with 0.  If the value is 0 it means that the mutex was created for this instance of the malware but if it gets another value, it means that the mutex was made from another instance or vaccine and, in this case, it will finish the execution of the malware.</p>
<p>After this, it will make 2 threads, one of them to search for processes and the another one to crypt files in the network shares that it has access to.</p>
<p>The first thread enumerates all processes of the system and creates the name of the process in upper case and calculates a hash with the name and compares it with a big list of hashes. This hash algorithm is a custom algorithm. It is typical in malware that tries to hide what processes they are looking for. If it finds one of them it will terminate it with “TerminateProcess” function after opening with the rights to make this action with “OpenProcess” function.</p>
<p>The malware contains 61 hard-coded hashes of programs such as “STEAM.EXE”, database programs, office programs and others.</p>
<p>Below, the first 38 hashes with the associated process names. These 38 processes are the most usual processes to close as we have observed with other ransomwares families such as GandCrab, Cerber, etc.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96094" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/hashes-1.png" alt="" width="376" height="886" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/hashes-1.png 376w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/hashes-1-127x300.png 127w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/hashes-1-212x500.png 212w" sizes="(max-width: 376px) 100vw, 376px" /></p>
<p>This thread runs in an infinite loop with a wait using the function “Sleep” per iteration of 30 minutes.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96095" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/killing-critical-processes.png" alt="" width="698" height="301" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/killing-critical-processes.png 698w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/killing-critical-processes-300x129.png 300w" sizes="(max-width: 698px) 100vw, 698px" /></p>
<p style="text-align: center;">FIGURE 9. Thread to kill critical processes to unlock files</p>
<p>The second thread created has the task of enumerating all network shares and crypts files in them if the malware has access to them.</p>
<p>For executing this task, it uses the typical API functions of the module “MPR.DLL”:</p>
<ul>
<li>WNetOpenEnumW</li>
<li>WNetEnumResourceW</li>
<li>WNetCloseEnum</li>
</ul>
<p>This thread starts creating a reserve of memory with “GlobalAlloc” function to keep the information of the “MPR” functions.</p>
<p>For each network share that the malware discovers, it will prepare to enumerate more shares and crypt files.</p>
<p>For each folder discovered, it will enter it and search for more subfolders and files. The first step is to check the name of the folder/file found against a hardcoded list of hashes with the same algorithm used to detect the processes to close.</p>
<p>Below are the results of 12 of the 27 hashes with the correct names:</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96096" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/12-hashes.png" alt="" width="372" height="289" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/12-hashes.png 372w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/12-hashes-300x233.png 300w" sizes="(max-width: 372px) 100vw, 372px" /></p>
<p>If it passes, it will check that the file is not a folder, and in this case compare the name with a list of hardcoded names and extensions that are in plain text rather than in hash format:</p>
<ul>
<li>ClopReadMe.txt</li>
<li>ntldr</li>
<li>NTDLR</li>
<li>boot.ini</li>
<li>BOOT.INI</li>
<li>ntuser.ini</li>
<li>NTUSER.INI</li>
<li>AUTOEXEC.BAT</li>
<li>autoexec.bat</li>
<li>.Clop</li>
<li>NTDETECT.COM</li>
<li>ntdetect.com</li>
<li>.dll</li>
<li>.DLL</li>
<li>.exe</li>
<li>.EXE</li>
<li>.sys</li>
<li>.SYS</li>
<li>.ocx</li>
<li>.OCX</li>
<li>.LNK</li>
<li>.lnk</li>
<li>desktop.ini</li>
<li>autorun.inf</li>
<li>ntuser.dat</li>
<li>iconcache.db</li>
<li>bootsect.bak</li>
<li>ntuser.dat.log</li>
<li>thumbs.db</li>
<li>DESKTOP.INI</li>
<li>AUTORUN.INF</li>
<li>NTUSER.DAT</li>
<li>ICONCACHE.DB</li>
<li>BOOTSECT.BAK</li>
<li>NTUSER.DATA.LOG</li>
<li>THUMBS.DB</li>
</ul>
<p>This check is done with a custom function that checks character per character against all the list. It is the reason for having the same names in both upper and lower case, instead of using the function “lstrcmpiA,” for example, to avoid some hook in this function preventing the file from being affected. The check of the extension at the same time is to make the process of crypto quicker. Of course, the malware checks that the file does not have the name of the ransom note and the extension that it will put in the crypted file. Those blacklisted extensions will help the system avoid crashing during the encryption compared with other ransomware families.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96097" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/check-file-names-and-extensions.png" alt="" width="802" height="341" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/check-file-names-and-extensions.png 802w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/check-file-names-and-extensions-300x128.png 300w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/check-file-names-and-extensions-768x327.png 768w" sizes="(max-width: 802px) 100vw, 802px" /></p>
<p style="text-align: center;">FIGURE 10. Check of file names and extensions</p>
<p>This behavior is normal in ransomware but the previous check against hardcoded hashes based on the file/folder name is weird because later, as we can see in the above picture, the next check is against plain text strings.</p>
<p>If it passes this check, the malware will make a new thread with a struct prepared with a hardcoded key block, the name of the file, and the path where the file exists. In this thread the first action is to remove the error mode with “SetErrorMode” to 1 to avoid an error dialog being shown to the user if it crashes. Later, it will prepare the path to the file from the struct passed as argument to the thread and change the attributes of the file to ARCHIVE with the function “SetFileAttributesW”, however the malware does not check if it can make this action with success or not.</p>
<p>Later it will generate a random AES key and crypt each byte of the file with this key, next it will put the mark “Clop^_” at the end of the file, after the mark it will put the key used to crypt the file ciphered with the master RSA key that has hardcoded the malware to protect it against third party free decryptors.</p>
<p>The malware can use 2 different public RSA keys: one exported using the crypto api in a public blob or using the embedded in base64 in the malware. The malware will only use the second one if it cannot create the crypto context or has some problem with the crypto api functions.</p>
<p>The malware does not have support for Windows XP in its use with the crypto functions, because the CSP used in Windows XP has another name, but if run in another operating system starting with Windows Vista, it can change the name in the debugger to acquire the context later and will generate a RSA public blob.</p>
<p>Another difference with other ransomware families is that Clop will only cipher the disk that is a physical attached/embedded disk (type 3, FIXED or removable (type 2)). The malware ignores the REMOTE type (4)).</p>
<p>Anyways, the shares can be affected using the “MPR.DLL” functions without any problem.</p>

<h3>Vaccine</h3>
<p>The function to check a file or a folder name using the custom hash algorithm can be a problem for the malware execution due if one of them is found in execution, the malware will avoid it. If this happens with a folder, all the files inside that folder will be skipped as well.</p>
<p>As the algorithm and the hash is based on 32bits and only in upper case characters, it is very easy to create a collision as we know the target hashes and the algorithm</p>
<p>It cannot be used as vaccine on itself, but it can be useful to protect against the malware if the most critical files are inside of a collision folder name.</p>
<p><img decoding="async" loading="lazy" class="aligncenter size-full wp-image-96108" src="https://securingtomorrow.mcafee.com/wp-content/uploads/2019/07/collision-of-hashes.png" alt="" width="614" height="571" srcset="https://www.mcafee.com/blogs/wp-content/uploads/2019/07/collision-of-hashes.png 614w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/collision-of-hashes-300x279.png 300w, https://www.mcafee.com/blogs/wp-content/uploads/2019/07/collision-of-hashes-538x500.png 538w" sizes="(max-width: 614px) 100vw, 614px" /></p>
<p style="text-align: center;">FIGURE 17. Collision of hashes</p>
<p>In the screenshot “BOOT” is a correct name for the hash, but the others are collisions.</p>
<p>This malware has a lot of changes per version that avoid making a normal vaccine using mutex, etc.</p>
<h3>The Odd One in the Family</h3>
<p>That not all ransomware is created equally, especially goes for Clop. Earlier in this blog we have highlighted some interesting choices the developers made when it came to detecting language settings, processes and the use of batch files to delete the shadow volume copies. We found in the analysis some unique functions compared with other ransomware families.</p>
<p>However, Clop does embrace some of the procedures we have seen with other ransomware families by not listing the ransom amount or mentioning a bitcoin address.</p>
<p>Victims must communicate via email instead of with a central command and control server hosting decryption keys. In the newer versions of Clop, victims are required to state their company name and site in the email communications. We are not absolutely sure why this is, but it might be an effort to improve victim tracking.</p>
<p>Looking at the Clop ransom note, it shares TTPs with other ransomware families; e.g. it mimics the Ryuk ransomware and contains similarities with BitPaymer, however the code and functions are quite different between them.</p>

<h3>Indicators of Compromise</h3>
<p>The samples use the following MITRE ATT&amp;CK&#x2122; techniques:</p>
<ul>
<li>Execution through API (Batch file for example).</li>
<li>Application processes discovery with some procedures as the hashes of the name, and directly for the name of the process.</li>
<li>File and directory discovery: to search files to encrypt.</li>
<li>Encrypt files.</li>
<li>Process discovery: enumerating all processes on the endpoint to kill some special ones.</li>
<li>Create files.</li>
<li>Create mutants.</li>
</ul>
<h3>Conclusion</h3>
<p>Clop ransomware shows some characteristics that enterprises are its intended targets instead of end consumers. The authors displayed some creative technical solutions, to detect the victim’s language settings and installed programs. On the other hand, we also noticed some weird decisions when it came to coding certain functionalities in the ransomware. Unfortunately, it is not the first time that criminals will make money with badly programmed malware.</p>
<p>Clop is constantly evolving and even though we do not know what new changes will be implemented in the future, McAfee ATR will keep a close watch.</p>
