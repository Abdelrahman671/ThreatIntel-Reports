                    <div class="article-body">
    <p>ESET researchers have discovered a Linux variant of the SideWalk backdoor, one of the multiple custom implants used by the SparklingGoblin APT group. This variant was deployed against a Hong Kong university in February 2021, the same university that had already been targeted by SparklingGoblin during the student protests in May 2020. We originally named this backdoor StageClient, but now refer to it simply as SideWalk Linux. We also discovered that a previously known Linux backdoor – the Specter RAT, first <a href="https://blog.netlab.360.com/ghost-in-action-the-specter-botnet/" target="_blank" rel="noopener">documented</a> by 360 Netlab – is also actually a SideWalk Linux variant, having multiple commonalities with the samples we identified.</p>
<p>SparklingGoblin is an APT group whose tactics, techniques, and procedures (TTPs) partially overlap with APT41 and BARIUM. It makes use of Motnug and ChaCha20-based loaders, the CROSSWALK and SideWalk backdoors, along with Korplug (aka PlugX) and Cobalt Strike. While the group targets mostly East and Southeast Asia, we have also seen SparklingGoblin targeting a broad range of organizations and verticals around the world, with a particular focus on the academic sector. SparklingGoblin is one of the groups with access to the ShadowPad backdoor.</p>
<p>This blogpost documents SideWalk Linux, its victimology, and its numerous similarities with the originally discovered SideWalk backdoor.</p>
<h2>Attribution</h2>
<p>The SideWalk backdoor is exclusive to SparklingGoblin. In addition to the multiple code similarities between the Linux variants of SideWalk and various SparklingGoblin tools, one of the SideWalk Linux samples uses a C&amp;C address (<span style="font-family: 'courier new', courier, monospace;">66.42.103[.]222</span>) that was <a href="https://www.ptsecurity.com/ww-en/analytics/pt-esc-threat-intelligence/higaisa-or-winnti-apt-41-backdoors-old-and-new/" target="_blank" rel="noopener">previously used by SparklingGoblin</a>.</p>
<p>Considering all of these factors, we attribute with high confidence SideWalk Linux to the SparklingGoblin APT group.</p>
<h2>Victimology</h2>
<p>Even though there are various SideWalk Linux samples, as we now know them, on VirusTotal, in our telemetry we have found only one victim compromised with this SideWalk variant: a Hong Kong university that, amidst student protests, had previously been targeted by both <a href="https://www.welivesecurity.com/2020/07/29/eset-threat-report-q22020/" target="_blank" rel="noopener">SparklingGoblin</a> (using the Motnug loader and the CROSSWALK backdoor) and <a href="https://www.welivesecurity.com/2020/01/31/winnti-group-targeting-universities-hong-kong/" target="_blank" rel="noopener">Fishmonger</a> (using the ShadowPad and Spyder backdoors). Note that at that time we put those two different clusters of activity under the broader Winnti Group denomination.</p>
<p>SparklingGoblin first compromised this particular university in May 2020, and we first detected the Linux variant of SideWalk in that university’s network in February 2021. The group continuously targeted this organization over a long period of time, successfully compromising multiple key servers, including a print server, an email server, and a server used to manage student schedules and course registrations.</p>
<h2>The road to Sidewalk Linux</h2>
<p>SideWalk, which we first described in its Windows form in <a href="https://www.welivesecurity.com/2021/08/24/sidewalk-may-be-as-dangerous-as-crosswalk/" target="_blank" rel="noopener">our blogpost</a> on August 24<sup>th</sup>, 2021, is a multipurpose backdoor that can load additional modules sent from the C&amp;C server. It makes use of Google Docs as a dead-drop resolver, and Cloudflare workers as its C&amp;C server. It can properly handle communication behind a proxy.</p>
<p>The compromise chain is currently unknown, but we think that the initial attack vector could have been exploitation. This hypothesis is based on the 360 Netlab article describing the Specter botnet targeting IP cameras, and NVR and DVR devices, and the fact that the Hong Kong victim used a vulnerable WordPress server, since there were many attempts to install various webshells.</p>
<p>We first documented the Linux variant of SideWalk as <a href="https://twitter.com/esetresearch/status/1410864752948043778">StageClient</a> on July 2<sup>nd</sup>, 2021, without making the connection at that time to SparklingGoblin and its custom SideWalk backdoor. The original name was used because of the repeated appearances of the string <span style="font-family: 'courier new', courier, monospace;">StageClient</span> in the code.</p>
<p>While researching StageClient further, we found a blogpost about the <a href="https://blog.netlab.360.com/ghost-in-action-the-specter-botnet/" target="_blank" rel="noopener">Specter botnet</a> described by 360 Netlab. That blogpost describes a modular Linux backdoor with flexible configuration that uses a ChaCha20 encryption variant – basically a subset of StageClient’s functionality. Further inspection confirmed this hypothesis; we additionally found a huge overlap in functionality, infrastructure, and symbols present in all the binaries.</p>
<p>We compared the StageClient sample <span style="font-family: 'courier new', courier, monospace;">E5E6E100876E652189E7D25FFCF06DE959093433</span> with Specter samples <span style="font-family: 'courier new', courier, monospace;">7DF0BE2774B17F672B96860D013A933E97862E6C</span> and found numerous similarities, some of which we list below.</p>
<p>First, there is an overlap in C&amp;C commands. Next, the samples have the same structure of configuration and encryption method (see Figure 1 and Figure 2).</p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-1.-StageClient’s-configuration-with-modified-symbols.png"><img class="wp-image-165321" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-1.-StageClient’s-configuration-with-modified-symbols.png" alt="" width="700" height="551" /></a><p class="caption-text"><em>Figure 1. StageClient’s configuration with modified symbols</em></p></div></p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-2.-Specter’s-configuration-with-modified-symbols.png"><img class="wp-image-165322" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-2.-Specter’s-configuration-with-modified-symbols.png" alt="" width="700" height="448" /></a><p class="caption-text"><em>Figure 2. Specter’s configuration with modified symbols</em></p></div></p>
<p>Additionally, the samples’ modules are managed in almost the same way, and the majority of the interfaces are identical; modules of StageClient only need to implement one additional handler, which is for closing the module. Three out of the five known modules are almost identical.</p>
<p>Lastly, we could see striking overlaps in the network protocols of the compared samples. A variant of ChaCha20 is used twice for encryption with LZ4 compression in the very same way. Both StageClient and Specter create a number of threads (see Figure 3 and Figure 4) to manage sending and receiving asynchronous messages along with heartbeats.</p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-3.-A-part-of-StageClient’s-StageClient-StartNetwork-function.png"><img class="wp-image-165323" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-3.-A-part-of-StageClient’s-StageClient-StartNetwork-function.png" alt="" width="700" height="119" /></a><p class="caption-text"><em>Figure 3. A part of StageClient’s </em><span style="font-family: courier new, courier, monospace;">StageClient::StartNetwork</span> <em>function</em></p></div></p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-4.-A-part-of-Specter’s-StartNetwork-function.png"><img class="wp-image-165324" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-4.-A-part-of-Specter’s-StartNetwork-function.png" alt="" width="700" height="151" /></a><p class="caption-text"><em>Figure 4. A part of Specter’s </em><span style="font-family: courier new, courier, monospace;">StartNetwork</span> <em>function</em></p></div></p>
<p>Despite all these striking similarities, there are several changes. The most notable ones are the following:</p>
<ul>
<li>The authors switched from the C language to C++. The reason is unknown, but it should be easier to implement such modular architecture in C++ due to its polymorphism support.</li>
<li>An option to exchange messages over HTTP was added (see Figure 5 and Figure 6).</li>
</ul>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-5.-Sending-a-message-in-StageClient_2.png"><img class="wp-image-165417" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-5.-Sending-a-message-in-StageClient_2.png" alt="" width="600" height="226" /></a><p class="caption-text"><em>Figure 5. Sending a message in StageClient</em></p></div></p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-6.-Sending-a-message-in-Specter.png"><img class="wp-image-165326" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-6.-Sending-a-message-in-Specter.png" alt="" width="600" height="250" /></a><p class="caption-text"><em>Figure 6. Sending a message in Specter</em></p></div></p>
<ul>
<li>Downloadable plugins were replaced with precompiled modules that fulfill the same purpose; a number of new commands and two new modules were added.</li>
<li>Added the module <span style="font-family: 'courier new', courier, monospace;">TaskSchedulerMod</span>, which operates as a built-in cron utility. Its cron table is stored in memory; the jobs are received over the network and executed as shell commands.</li>
<li>Added the module <span style="font-family: 'courier new', courier, monospace;">SysInfoMgr</span>, which provides information about the underlying system such as the list of installed packages and hardware details.</li>
</ul>
<p>These similarities convince us that Specter and StageClient are from the same malware family. However, considering the numerous code overlaps between the StageClient variant used against the Hong Kong university in February 2021 and SideWalk for Windows, as described in the next section, we now believe that Specter and StageClient are both Linux variants of SideWalk, so we have decided to refer to them as SideWalk Linux.</p>
<h3>Similarities with the Windows variant</h3>
<p>SideWalk Windows and SideWalk Linux share too many similarities to describe within the confines of this blogpost, so here we only cover the most striking ones.</p>
<h4><strong>ChaCha20</strong></h4>
<p>An obvious similarity is noticeable in the implementations of ChaCha20 encryption: both variants use a counter with an initial value of <span style="font-family: 'courier new', courier, monospace;">0x0B</span>, which was previously mentioned in our blogpost as a specificity of SideWalk’s ChaCha20 implementation.</p>
<h4><strong>Software architecture</strong></h4>
<p>One SideWalk particularity is the use of multiple threads to execute one specific task. We noticed that in both variants there are exactly five threads executed simultaneously, each of them having a specific task. The following list describes the function of each; the thread names are from the code:</p>
<ul>
<li><span style="font-family: 'courier new', courier, monospace;">StageClient::ThreadNetworkReverse</span><br />
If a connection to the C&amp;C server is not already established, this thread periodically attempts to retrieve the local proxy configuration and the C&amp;C server location from the dead-drop resolver. If the previous step was successful, it attempts to initiate a connection to the C&amp;C server.</li>
<li><span style="font-family: 'courier new', courier, monospace;">StageClient::ThreadHeartDetect</span><br />
If the backdoor did not receive a command in the specified amount of time, this thread can terminate the connection with the C&amp;C server or switch to a “nap” mode that introduces minor changes to the behavior.</li>
<li><span style="font-family: 'courier new', courier, monospace;">StageClient::ThreadPollingDriven</span><br />
If there is no other queued data to send, this thread periodically sends a heartbeat command to the C&amp;C server that can additionally contain the current time.</li>
<li><span style="font-family: 'courier new', courier, monospace;">StageClient::ThreadBizMsgSend</span><br />
This thread periodically checks whether there is data to be sent in the message queues used by all the other threads and, if so, processes it.</li>
<li><span style="font-family: 'courier new', courier, monospace;">StageClient::ThreadBizMsgHandler</span><br />
This thread periodically checks whether there are any pending messages received from the C&amp;C server and, if so, handles them.</li>
</ul>
<h4><strong>Configuration</strong></h4>
<p>As in SideWalk Windows, the configuration is decrypted using ChaCha20.</p>
<h4><strong><em>Checksum</em></strong></h4>
<p>First, before decrypting, there is a data integrity check. This check is similar in both implementations of SideWalk (see Figure 7 and Figure 8): an MD5 hash is computed on the ChaCha20 nonce concatenated to the encrypted configuration data. This hash is then checked against a predefined value, and if not equal, SideWalk exits.</p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-7.-SideWalk-Linux-Configuration-integrity-check.png"><img class="wp-image-165327" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-7.-SideWalk-Linux-Configuration-integrity-check.png" alt="" width="600" height="62" /></a><p class="caption-text"><em>Figure 7. SideWalk Linux: Configuration integrity check</em></p></div></p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-8.-SideWalk-Windows-Configuration-integrity-check.png"><img class="wp-image-165328" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-8.-SideWalk-Windows-Configuration-integrity-check.png" alt="" width="800" height="45" /></a><p class="caption-text"><em>Figure 8. SideWalk Windows: Configuration integrity check</em></p></div></p>
<h4><strong><em>Layout</em></strong></h4>
<p>Figure 9 presents excerpts of decrypted configurations from the samples that we analyzed.</p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-9.-Configuration-parts-from-E5E6E100876E652189E7D25FFCF06DE959093433-left-and-FA6A40D3FC5CD4D975A01E298179A0B36AA02D4E-right.png"><img class="wp-image-165329" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-9.-Configuration-parts-from-E5E6E100876E652189E7D25FFCF06DE959093433-left-and-FA6A40D3FC5CD4D975A01E298179A0B36AA02D4E-right.png" alt="" width="600" height="144" /></a><p class="caption-text"><em>Figure 9. <p>The SideWalk Linux config contains less information than the SideWalk Windows one. This makes sense because the majority of the configuration artifacts in SideWalk Windows are used as cryptography and network parameters, whereas most of these are internal in SideWalk Linux.</p>
<h4><strong><em>Decryption using ChaCha20</em></strong></h4>
<p>As previously mentioned, SideWalk uses a main global structure to store its configuration. This configuration is first decrypted using the modified implementation of ChaCha20, as seen in Figure 10.</p>
<p><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-10a.-ChaCha20-decryption-call-in-SideWalk-Windows-left-and-in-SideWalk-Linux-right.png"><img class="alignleft wp-image-165330" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-10a.-ChaCha20-decryption-call-in-SideWalk-Windows-left-and-in-SideWalk-Linux-right.png" alt="" width="495" height="179" /></a> <a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-10b.-ChaCha20-decryption-call-in-SideWalk-Windows-left-and-in-SideWalk-Linux-right.png"><img class="aligncenter wp-image-165331" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-10b.-ChaCha20-decryption-call-in-SideWalk-Windows-left-and-in-SideWalk-Linux-right.png" alt="" width="434" height="180" /></a></p>
<p style="text-align: center;"><em>Figure 10. ChaCha20 decryption call in SideWalk Windows (left) and in SideWalk Linux (right)</em></p>
<p>Note that the ChaCha20 key is exactly the same in both variants, strengthening the connection between the two.</p>
<h4><strong>Dead-drop resolver</strong></h4>
<p>The dead-drop resolver payload is identical in both samples. As a reminder from our blogpost on SideWalk, Figure 11 depicts the format of the payload that is fetched from the dead-drop resolver.</p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-11.-Format-of-the-string-hosted-in-the-Google-Docs-document.png"><img class="wp-image-165332" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-11.-Format-of-the-string-hosted-in-the-Google-Docs-document.png" alt="" width="800" height="169" /></a><p class="caption-text"><em>Figure 11. Format of the string hosted in the Google Docs document</em></p></div></p>
<p>For the first delimiter, we notice that the <span style="font-family: 'courier new', courier, monospace;">PublicKey</span>: part of the string is ignored; the string <span style="font-family: 'courier new', courier, monospace;">AE68[…]3EFF</span> is directly searched, as shown in <em>Figure 12</em>.</p>
<p><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-12a.-SideWalk-Linux’s-firs-delimiter-routine-left-end-delimiter-and-middle-delimiter-routines-right-1.png"><img class="alignleft wp-image-165337" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-12a.-SideWalk-Linux’s-firs-delimiter-routine-left-end-delimiter-and-middle-delimiter-routines-right-1.png" alt="" width="550" height="134" /></a> <a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-12b.-SideWalk-Linux’s-firs-delimiter-routine-left-end-delimiter-and-middle-delimiter-routines-right-1.png"><img class="aligncenter wp-image-165338" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-12b.-SideWalk-Linux’s-firs-delimiter-routine-left-end-delimiter-and-middle-delimiter-routines-right-1.png" alt="" width="385" height="133" /></a></p>
<p style="text-align: center;"><em>Figure 12. SideWalk Linux’s first delimiter routine (left), end delimiter and middle delimiter routines (right)</em></p>
<p>The delimiters are identical, as well as the whole decoding algorithm.</p>
<h4><strong>Victim fingerprinting</strong></h4>
<p>In order to fingerprint the victim, different artifacts are gathered on the victim’s machine. We noticed that the fetched information is exactly the same, to the extent of it even being fetched in the same order.</p>
<p>As the boot time in either case is a Windows-compliant time format, we can hypothesize that the operators’ controller runs under Windows, and that the controller is the same for both Linux and Windows victims. Another argument supporting this hypothesis is that the ChaCha20 keys used in both implementations of SideWalk are the same.</p>
<h4><strong>Communication protocol</strong></h4>
<h4><strong><em>Data serialization</em></strong></h4>
<p>The communication protocol between the infected machine and the C&amp;C is HTTP or HTTPS, depending on the configuration, but in both cases, the data is serialized in the same manner. Not only is the implementation very similar, but the identical encryption key is used in both implementations, which, again, accentuates the similarity between the two variants.</p>
<h4><strong><em>POST requests</em></strong></h4>
<p>In the POST requests used by SideWalk to fetch commands and payloads from the C&amp;C server, one noticeable point is the use of the two parameters <span style="font-family: 'courier new', courier, monospace;">gtsid</span> and <span style="font-family: 'courier new', courier, monospace;">gtuvid</span>, as seen in Figure 13. Identical parameters are used in the Linux variant.</p>
<pre class="language-bash" title=""><code>POST /M26RcKtVr5WniDVZ/5CDpKo5zmAYbTmFl HTTP/1.1
Cache-Control: no-cache
Connection: close
Pragma: no-cache
User-Agent: Mozilla/5.0 Chrome/72.0.3626.109 Safari/537.36
gtsid: zn3isN2C6bWsqYvO
gtuvid: 7651E459979F931D39EDC12D68384C21249A8DE265F3A925F6E289A2467BC47D

Host: update.facebookint.workers[.]dev</code></pre>
<p><em>Figure 13. Example of a POST request used by SideWalk Windows</em></p>
<p>Another interesting point is that the Windows variant runs as fully position-independent shellcode, whereas the Linux variant is a shared library. However, we think the malware’s authors could have just taken an extra step, using a tool such as <a href="https://github.com/monoxgas/sRDI" target="_blank" rel="noopener">sRDI</a> to convert a compiled SideWalk PE to shellcode instead of manually writing the shellcode.</p>

<h4><strong>Versioning</strong></h4>
<p>In the Linux variant, we observed a specificity that was not found in the Windows variant: a version number is computed (see Figure 14).</p>
<p><div class="caption center"><a href="https://web-assets.esetstatic.com/wls/2022/09/Figure-14.-Versioning-function-in-SideWalk-Linux.png"><img class="wp-image-165335" src="https://web-assets.esetstatic.com/wls/2022/09/Figure-14.-Versioning-function-in-SideWalk-Linux.png" alt="" width="550" height="536" /></a><p class="caption-text"><em>Figure 14. Versioning function in SideWalk Linux</em></p></div></p>
<p>The hardcoded date could be the beginning or end of development of this version of SideWalk Linux. The final computation is made out of the year, day, and month, from the value <span style="font-family: 'courier new', courier, monospace;">Oct 26 2020</span>. In this case, the result is <span style="font-family: 'courier new', courier, monospace;">1171798691840</span>.</p>
<h4><strong>Plugins</strong></h4>
<p>In SideWalk Linux variants, modules are built in; they cannot be fetched from the C&amp;C server. That is a notable difference from the Windows variant. Some of those built-in functionalities, like gathering system information (<span style="font-family: 'courier new', courier, monospace;">SysInfoMgr</span>, for example) such as network configuration, are done directly by dedicated functions in the Windows variant. In the Windows variant, some plugins can be added through C&amp;C communication.</p>
<h4><strong>Defense evasion</strong></h4>
<p>The Windows variant of SideWalk goes to great lengths to conceal the objectives of its code. It trimmed out all data and code that was unnecessary for its execution and encrypted the rest. On the other hand, the Linux variants contain symbols and leave some unique authentication keys and other artifacts unencrypted, which makes the detection and analysis significantly easier.</p>
<p>Additionally, the much higher number of inlined functions in the Windows variant suggests that its code was compiled with a higher level of compiler optimizations.</p>
<h2>Conclusion</h2>
<p>The backdoor that was used to attack a Hong Kong university in February 2021 is the same malware family as the SideWalk backdoor, and actually is a Linux variant of the backdoor. This Linux version exhibits several similarities with its Windows counterpart along with various novelties.</p>
