<h2 class="wp-block-heading" id="h-summary">Summary</h2>



<p>QakBot has been updated with more evasion techniques. QakBot’s configuration is now stored in a registry key instead of a file. The run key for persistence is not permanently present in the registry but only written right before shutdown or reboot, and deleted immediately after QakBot is executed again. QakBot’s executable is also not stored permanently on the file system anymore, but similarly to the run key registry entry, dropped onto the file system before reboots and deleted afterwards. This way security software can only detect QakBot artifacts on disk, right before system shutdown, and shortly after system boot. However, at that time security software itself is shutting down and booting up, hence may not detect QakBot’s new persistence method.</p>



<p>Other changes include dynamic just-in-time decoding and destruction of strings at runtime. So any string used in the malware is only decoded at runtime into memory only and destroyed right afterwards.</p>



<p>The delivery method for the observed QakBot campaigns identified via the regular expression pattern of&nbsp;<code>abc[0-9]+</code>&nbsp;is still XLM macro documents as reported previously.</p>



<div style="height:var(--wp--preset--spacing--20);width:100%" aria-hidden="true" class="wp-block-spacer"></div>



<h2 class="wp-block-heading" id="h-background">Background</h2>



<p>QakBot (also known as QBot, QuakBot, Pinkslipbot) has been around since 2008. It is distributed via Emotet, i.e., Emotet will download QakBot onto victims that are already infected with Emotet but it is also distributed directly via email. To this end, it uses email conversation thread hijacking in its campaigns<sup><a href="#h-references">1</a></sup>, i.e., it will reply to emails that it finds in its victim’s mailboxes. QakBot is known to escalate intrusions by downloading the ProLock ransomware<sup><a href="#h-references">2</a></sup>&nbsp;or lately the Egregor ransomware.</p>



<p>The observed QakBot campaigns identified by campaign ID&nbsp;<code>abc</code>&nbsp;use XLM macro documents for infection. We previously reported on their low detection.<sup><a href="#h-references">3</a></sup></p>



<p>An overview of the current chain of infection used by the QakBot campaign with identifiers following the regular expression pattern of&nbsp;<code>abc[0-9]+</code>&nbsp;can be seen in the following flow graph.</p>

<h4 class="wp-block-heading" id="h-strings-only-decoded-at-runtime">Strings only decoded at runtime</h4>



<p>QakBot will decode its strings only at runtime into memory. After usage the decoded strings are removed from memory again.</p>

<p>QakBot will set specific bits in a bit mask for each running process it finds. Depending on the resulting bit mask the further infection path is altered, e.g., if&nbsp;<code>avp.exe</code>&nbsp;has been encountered. QakBot will later inject its code into&nbsp;<code>mobsync.exe</code>&nbsp;instead of&nbsp;<code>explorer.exe</code>. Because the searched process names are related to security solutions, we believe that this way QakBot tailors its execution path to evade detection by specific vendors.</p>



<p>Then in another loop, again using&nbsp;<code>CreateToolhelp32Snapshot</code>&nbsp;and&nbsp;<code>Process32{First,Next}W</code>, it checks for:</p>



<ul class="wp-block-list">
<li><code>srvpost.exe</code></li>



<li><code>frida-winjector-helper-32.exe</code></li>



<li><code>frida-winjector-helper-64.exe</code></li>
</ul>



<p>If it detects any of those processes the execution flow will run into a loop continuously calling&nbsp;<code>WaitForSingleObject(handle, 0x1fa)</code>&nbsp;on a&nbsp;<code>handle</code>&nbsp;previously generated via&nbsp;<code>CreateEvent(NULL, FALSE, FALSE, ...)</code>, i.e., it runs in an infinite loop.</p>



<div style="height:var(--wp--preset--spacing--10);width:100%" aria-hidden="true" class="wp-block-spacer"></div>



<h4 class="wp-block-heading" id="h-device-drivers">Device drivers</h4>



<p>Next, QakBot uses&nbsp;<code>SetupDiGetDeviceRegistryPropertyA</code>&nbsp;(querying properties&nbsp;<code>SPDRP_DEVICEDESC</code>&nbsp;and&nbsp;<code>SPDRP_SERVICE</code>) to check for device drivers containing the following strings:</p>



<ul class="wp-block-list">
<li><code>VBoxVideo</code></li>



<li><code>Red Hat VirtIO</code></li>



<li><code>QEMU</code></li>



<li><code>A3E64E55_pr</code></li>
</ul>



<p>We believe the search for <code>A3E64E55_pr</code> is used to detect an artifact of the ANY.RUN sandbox.<sup>3</sup> Alternatively, but unlikely, it could be used to detect an artifact of the long ago defunct xCore Complex Protection AV solution using a similar driver with the name <code>A3E64E55_pr.sys</code>.</p>



<p>If it detects any of those device drivers the execution flow will run into the same infinite loop continuously calling&nbsp;<code>WaitForSingleObject(handle, 0x1fa)</code>&nbsp;on a&nbsp;<code>handle</code>&nbsp;previously generated via&nbsp;<code>CreateEvent(NULL, FALSE, FALSE, ...)</code>, as previously mentioned.</p>

<h3 class="wp-block-heading" id="h-persistence">Persistence</h3>



<p>The persistence mechanism of QakBot has also changed. While it still uses a run key registry entry under&nbsp;<code>HKCU\Software\Microsoft\Windows\CurrentVersion\Run</code>, this key is only set right before the system is shutdown, rebooted or put to sleep. The corresponding DLL is also only dropped to disk right before shutdown, rebooted or sleep.</p>



<p>After the system boots up again, QakBot is started via the run key. The execution tree also starts via&nbsp;<code>regsvr32.exe -s ...</code>&nbsp;like the initial execution from Excel. QakBot follows the same steps as previously outlined resulting in process injection into&nbsp;<code>explorer.exe</code>.</p>

<h2 class="wp-block-heading" id="h-conclusion-and-countermeasures">Conclusion and Countermeasures</h2>



<p>From our analysis we can conclude that QakBot is trying to avoid persistent file artifacts. In previous version the configuration and QakBot executable were permanently stored on disk. This made it easy for security tools to detect them. The new version tries to avoid permanently leaving its artifacts on disk. While QakBot is not going fully fileless, it new tactics will sure lower its detection.</p>



<p>But even though QakBot has changed, the delivery mechanism behind the QakBot “<code>abc[A-Z]+</code>” campaign did not. Hence, an infection by this threat actor can be successfully prevented by blocking the initial emails.</p>
