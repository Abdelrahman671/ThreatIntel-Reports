We noticed that the file jscript9.dll has changed, so we assume that the vulnerability that was fixed is in the &lt;a href=\&#34;https://en.wikipedia.org/wiki/JScript\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>JScript&lt;/a> component.&lt;/p>\r\n&lt;p>JScript is Microsoft&#39;s implementation of ECMAScript, the same standard that’s behind JavaScript and is used by Internet Explorer. Chakra is also a JavaScript engine, forked from JScript, and is used by Microsoft Edge Legacy (newer versions of Edge are based on Chromium).     <h3><a href="https://msrc.microsoft.com/update-guide/en-US/vulnerability/CVE-2024-38178" target="_blank" rel="nofollow noreferrer">CVE-2024-38178</a> — Scripting Engine Memory Corruption (CVSS 7.5)</h3>
<p>Windows scripting languages can refer to multiple implementations of scripting languages inside Windows, such as VBScript, JavaScript, or others. We noticed that the file jscript9.dll has changed, so we assume that the vulnerability that was fixed is in the <a href="https://en.wikipedia.org/wiki/JScript" target="_blank" rel="nofollow noreferrer">JScript</a> component.</p>
<p>JScript is Microsoft's implementation of ECMAScript, the same standard that’s behind JavaScript and is used by Internet Explorer. Chakra is also a JavaScript engine, forked from JScript, and is used by Microsoft Edge Legacy (newer versions of Edge are based on Chromium). Both are implemented as DLLs, conveniently named jscript9.dll and chakra.dll.</p>
<p>According to the advisory, an attack requires the victim to click on a link, and to use Edge in Internet Explorer mode. We theorize this vulnerability might be triggered in other ways, as well, possibly through Office.</p>

In that scenario, Windows SmartScreen is responsible for displaying a warning screen. The vulnerability requires the attacker to craft a file in a specific way that will cause Windows SmartScreen to fail to check the file and, therefore, to bypass the warning screen.&lt;/p>\r\n&lt;p>This is not the first time SmartScreen bypass vulnerabilities have been actively exploited by threat actors. The November 2023 and February 2024 patches also included fixes for bypass vulnerabilities that were reported as actively exploited.&lt;/p>\r\n&lt;p>There were also four other vulnerabilities exploited in the wild.&lt;/p>\r\n&#34;}}">
    <h3><a href="https://msrc.microsoft.com/update-guide/en-US/vulnerability/CVE-2024-38213" target="_blank" rel="nofollow noreferrer">CVE-2024-38213</a> — Windows Mark of the Web (CVSS 6.5)</h3>
<p>It’s important to note that this <b>vulnerability was fixed in June 2024</b>, but the advisory was not released until August 2024.</p>
<p>This is a security bypass vulnerability, which skips the SmartScreen security warning when opening potentially malicious files that were downloaded from the web. In that scenario, Windows SmartScreen is responsible for displaying a warning screen. The vulnerability requires the attacker to craft a file in a specific way that will cause Windows SmartScreen to fail to check the file and, therefore, to bypass the warning screen.</p>
<p>This is not the first time SmartScreen bypass vulnerabilities have been actively exploited by threat actors. The November 2023 and February 2024 patches also included fixes for bypass vulnerabilities that were reported as actively exploited.</p>
<p>There were also four other vulnerabilities exploited in the wild.</p>

<div class="text text__pb-20 aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-da3cae6184&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2024-08-15T13:41:45Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>The TCP/IP stack in Windows is a core component of the operating system&#39;s networking functionality. It handles the transport of data across networks using the TCP/IP protocol suite, which is the foundation of communication over the internet and most local networks.&lt;/p>\r\n&lt;p>In the Windows operating system, the TCP/IP stack is implemented in tcpip.sys driver. This kernel module is responsible for handling network protocols, including TCP over transport layers such as IPv4 and IPv6.&lt;/p>\r\n&lt;p>&lt;a href=\&#34;https://msrc.microsoft.com/update-guide/en-US/vulnerability/CVE-2024-38063\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>CVE-2024-38063&lt;/a> (CVSS 9.8) is a critical vulnerability in this network stack that allows an &lt;b>unauthenticated attacker&lt;/b> to send specially crafted IPv6 packets that could lead to remote code execution.&lt;/p>\r\n&lt;p>A remarkable blog post by the &lt;a href=\&#34;https://msrc.microsoft.com/blog/2021/02/multiple-security-updates-affecting-tcp-ip/\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>Microsoft Security Response Center&lt;/a> team discussed previously founded CVEs in this component, where remote code execution (RCE) and denial-of-service (DoS) attacks can be achieved using malcrafted packets. The CVSS indicates that the attack complexity is low (AC:L) in contrast with the previous findings.&lt;/p>\r\n&lt;p>According to Microsoft, machines with IPv6 disabled are protected from this vulnerability.     <p>The TCP/IP stack in Windows is a core component of the operating system's networking functionality. It handles the transport of data across networks using the TCP/IP protocol suite, which is the foundation of communication over the internet and most local networks.</p>
<p>In the Windows operating system, the TCP/IP stack is implemented in tcpip.sys driver. This kernel module is responsible for handling network protocols, including TCP over transport layers such as IPv4 and IPv6.</p>
<p><a href="https://msrc.microsoft.com/update-guide/en-US/vulnerability/CVE-2024-38063" target="_blank" rel="nofollow noreferrer">CVE-2024-38063</a> (CVSS 9.8) is a critical vulnerability in this network stack that allows an <b>unauthenticated attacker</b> to send specially crafted IPv6 packets that could lead to remote code execution.</p>
<p>A remarkable blog post by the <a href="https://msrc.microsoft.com/blog/2021/02/multiple-security-updates-affecting-tcp-ip/" target="_blank" rel="nofollow noreferrer">Microsoft Security Response Center</a> team discussed previously founded CVEs in this component, where remote code execution (RCE) and denial-of-service (DoS) attacks can be achieved using malcrafted packets. The CVSS indicates that the attack complexity is low (AC:L) in contrast with the previous findings.</p>
<p>According to Microsoft, machines with IPv6 disabled are protected from this vulnerability. We believe that this vulnerability may be used by malicious actors because of the widespread use of IPV6, the low complexity of the attack, and the fact that it is accessible to unauthenticated attackers — so patch your machines!</p>

    <p>Windows Network Virtualization (WNV) is a feature within the Hyper-V Network Virtualization framework that enables virtual machines (VMs) to communicate as if they are on the same subnet, even when they are on different subnets or physical networks. This is accomplished by decoupling virtual networks from the physical network, allowing VMs to move across physical hosts without altering network configurations.</p>
<p>WNV uses Network Virtualization Generic Routing Encapsulation (NVGRE) or Virtual Extensible LAN (VXLAN) for packet encapsulation, providing isolation, scalability, and multi-tenant support in cloud environments.</p>
<p><a href="https://msrc.microsoft.com/update-guide/en-US/vulnerability/CVE-2024-38159" target="_blank" rel="nofollow noreferrer">CVE-2024-38159</a> and <a href="https://msrc.microsoft.com/update-guide/en-US/vulnerability/CVE-2024-38160" target="_blank" rel="nofollow noreferrer">CVE-2024-38160</a> could both lead to guest-to-host escape, which in turn could end in RCE. Although both the vulnerabilities have a CVSS of 9.1, they only affect Windows Server 2016 and Windows 10 (1607).</p>
<p>Microsoft suggests mitigating the vulnerabilities by either running the Hyper-V machines inside another virtual platform such as VMware, or by giving up the Hyper-V based virtualization completely.</p>
