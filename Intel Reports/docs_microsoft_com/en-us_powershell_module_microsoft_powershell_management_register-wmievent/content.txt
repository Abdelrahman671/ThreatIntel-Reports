When the subscribed WMI event is raised, it is added to the event queue in your local session even if the event occurs on a remote computer. To get events in the event queue, use the Get-Event cmdlet. You can use the parameters of Register-WmiEvent to subscribe to events on remote computers and to specify the property values of the events that can help you identify the event in the queue. You can also use the Action parameter to specify actions to take when a subscribed event is raised. When you subscribe to an event, an event subscriber is added to your session. To get the event subscribers in the session, use the Get-EventSubscriber cmdlet. To cancel the subscription, use the Unregister-Event cmdlet, which deletes the event subscriber from the session. New Common Information Model (CIM) cmdlets, introduced Windows PowerShell 3.0, perform the same tasks as the WMI cmdlets. The CIM cmdlets comply with WS-Management (WSMan) standards and with the CIM standard, which enables the cmdlets to use the same techniques to manage computers that run the Windows operating system and those that run other operating systems. Instead of using Register-WmiEvent, consider using the Register-CimIndicationEvent cmdlet.   <h2 class="title is-6 margin-block-xs"></h2>
</nav>

	<div class="margin-block-sm">
		<p>Subscribes to a Windows Management Instrumentation (WMI) event.</p>

	</div>

	<h2 id="syntax">Syntax</h2>

	<h2 id="description">Description</h2>
	<p>The <code>Register-WmiEvent</code> cmdlet subscribes to Windows Management Instrumentation (WMI) events on
the local computer or on a remote computer.</p>
<p>When the subscribed WMI event is raised, it is added to the event queue in your local session even
if the event occurs on a remote computer. To get events in the event queue, use the <code>Get-Event</code>
cmdlet.</p>
<p>You can use the parameters of <code>Register-WmiEvent</code> to subscribe to events on remote computers and to
specify the property values of the events that can help you identify the event in the queue. You can
also use the <strong>Action</strong> parameter to specify actions to take when a subscribed event is raised.</p>
<p>When you subscribe to an event, an event subscriber is added to your session. To get the event
subscribers in the session, use the <code>Get-EventSubscriber</code> cmdlet. To cancel the subscription, use
the <code>Unregister-Event</code> cmdlet, which deletes the event subscriber from the session.</p>
<p>New Common Information Model (CIM) cmdlets, introduced Windows PowerShell 3.0, perform the same
tasks as the WMI cmdlets. The CIM cmdlets comply with WS-Management (WSMan) standards and with the
CIM standard, which enables the cmdlets to use the same techniques to manage computers that run the
Windows operating system and those that run other operating systems. Instead of using
<code>Register-WmiEvent</code>, consider using the
<a href="../cimcmdlets/register-cimindicationevent?view=powershell-5.1" data-linktype="relative-path">Register-CimIndicationEvent</a> cmdlet.</p>


	<h2 id="examples">Examples</h2>
	<h3 id="example-1-subscribe-to-events-generated-by-a-class">Example 1: Subscribe to events generated by a class</h3>
	<p>This command subscribes to the events generated by the <strong>Win32_ProcessStartTrace</strong> class. This class
raises an event whenever a process starts.</p>

	<pre><code class="lang-powershell">Register-WmiEvent -Class 'Win32_ProcessStartTrace' -SourceIdentifier "ProcessStarted"</code></pre>
	
	<h3 id="example-2-subscribe-to-creation-events-for-a-process">Example 2: Subscribe to creation events for a process</h3>
	<p>This command uses a query to subscribe to Win32_process instance creation events.</p>

	<pre><code class="lang-powershell">$wmiParameters = @{
  Query = "select * from __instancecreationevent within 5 where targetinstance isa 'win32_process'"
  SourceIdentifier = "WMIProcess"
  MessageData = "Test 01"
  TimeOut = 500
}
Register-WmiEvent @wmiParameters</code></pre>
	
	<h3 id="example-3-use-an-action-to-respond-to-an-event">Example 3: Use an action to respond to an event</h3>
	<p>This example shows how to use an action to respond to an event. In this case, when a process starts,
any <code>Start-Process</code> commands in the current session are written to an XML file.</p>

	<pre><code class="lang-powershell">$action = { Get-History | where { $_.commandline -like "*start-process*" } | export-cliXml "commandHistory.clixml" }
Register-WmiEvent -Class 'Win32_ProcessStartTrace' -SourceIdentifier "ProcessStarted" -Action $action

Id    Name            State      HasMoreData   Location  Command
--    ----            -----      -----------   --------  -------
1     ProcessStarted  NotStarted False                   get-history | where {...</code></pre>
	<p>When you use the <strong>Action</strong> parameter, <code>Register-WmiEvent</code> returns a background job that represents
the event action. You can use the <strong>Job</strong> cmdlets, such as <code>Get-Job</code> and <code>Receive-Job</code>, to manage
the event job.</p>
<p>For more information, see <a href="../microsoft.powershell.core/about/about_jobs?view=powershell-5.1" data-linktype="relative-path">about_Jobs</a>.</p>

	<h3 id="example-4-register-for-events-on-a-remote-computer">Example 4: Register for events on a remote computer</h3>
	<p>This example registers for events on the Server01 remote computer.</p>

	<pre><code class="lang-powershell">Register-WmiEvent -Class 'Win32_ProcessStartTrace' -SourceIdentifier "Start" -Computername Server01
Get-Event -SourceIdentifier "Start"</code></pre>
	<p>WMI returns the events to the local computer and stores them in the event queue in the current
session. To retrieve the events, run a local <code>Get-Event</code> command.</p>

<h3 id="-action" class="font-size-md font-family-monospace">-Action</h3>
<div class="parameterInfo">
	<p>Specifies commands that handle the events. The commands in the <strong>Action</strong> parameter run when an
event is raised instead of sending the event to the event queue. Enclose the commands in braces
(<code>{}</code>) to create a script block.</p>
<p>The value of <strong>Action</strong> can include the <code>$Event</code>, <code>$EventSubscriber</code>, <code>$Sender</code>, <code>$EventArgs</code>, and
<code>$Args</code> automatic variables, which provide information about the event to the <strong>Action</strong> script
block. For more information, see <a href="../microsoft.powershell.core/about/about_automatic_variables?view=powershell-5.1" data-linktype="relative-path">about_Automatic_Variables</a>.</p>
<p>When you specify an action, <code>Register-WmiEvent</code> returns an event job object that represents that
action. You can use the cmdlets that contain the <strong>Job</strong> noun (the <strong>Job</strong> cmdlets) to manage the
event job.</p>
