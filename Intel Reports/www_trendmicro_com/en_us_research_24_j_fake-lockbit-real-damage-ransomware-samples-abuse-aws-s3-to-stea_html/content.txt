	<h1 class="article-details__title">Fake LockBit, Real Damage: Ransomware Samples Abuse AWS S3 to Steal Data</h1>
	<p class="article-details__description">This article uncovers a Golang ransomware abusing AWS S3 for data theft, and masking as LockBit to further pressure victims. The discovery of hard-coded AWS credentials in these samples led to AWS account suspensions.</p>
	<p class="article-details__author-by">By: Jaromir Horejsi, Nitesh Surana

		<div>
			<h2><span class="body-subhead-title">Key Takeaways</span></h2>
<ul>
<li><span class="rte-red-bullet" style="font-weight: normal;">We found Golang ransomware samples that abuse Amazon S3 (Simple Storage Service) Transfer Acceleration feature to exfiltrate the victim’s files and upload them to the attacker-controlled S3 buckets.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Amazon Web Services (AWS) credentials hard coded in the samples were used to track the associated AWS Account IDs linked to malicious activities, serving as valuable Indicators of Compromise (IOCs).</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Attempts were made to disguise the Golang ransomware as the notorious LockBit ransomware. This was done presumably to use the ransomware family’s notoriety to further pressure victims.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">We shared our findings with the AWS Security team. It is important to note that our finding is not a vulnerability in any of AWS Services. We confirmed with AWS the behavior we identified for this threat actor's activity and it was found to violate the AWS acceptable use policy (<a href="https://aws.amazon.com/aup/">https://aws.amazon.com/aup/</a>). The reported AWS access keys and account have been suspended.</span></li>
</ul>
<h2><span class="body-subhead-title">Introduction</span></h2>
<p>From <a href="/en_us/research/23/e/rust-based-info-stealers-abuse-github-codespaces.html">infostealer</a> <a href="/en_us/research/23/e/info-stealer-abusing-codespaces-puts-discord-users--data-at-risk.html">development</a> to data exfiltration, cloud service providers are <a href="https://www.trendmicro.com/vinfo/us/security/news/virtualization-and-cloud/uncover-cloud-attacks-with-trend-vision-one-and-cloudtrail%20%20?utm_source=trendmicroresearch&amp;utm_medium=smk&amp;utm_campaign=082024_CloudTrail">increasingly being abused</a> by threat actors for malicious schemes. While in this case the ransomware samples we examined contained hard coded AWS credentials, this is specific to this single threat actor and in general, ransomware developers leverage other online services as part of their tactics. In line with this, we examined <a href="/en_us/what-is/ransomware.html">ransomware</a> samples written in <a href="https://go.dev/" target="_blank">Go language</a> (aka Golang), targeting Windows and MacOS environments. Most of the samples contained hard-coded AWS credentials, and the stolen data were uploaded to an Amazon S3 bucket controlled by the threat actor.</p>
<p>By the tail end of the attack, the device’s wallpaper is changed into an image mentioning <a href="/en_us/research/24/b/lockbit-attempts-to-stay-afloat-with-a-new-version.html">LockBit</a>. This might lead affected users to think that LockBit is to be blamed for the incident, especially since this ransomware family had been active in recent years and even had the <a href="https://www.trendmicro.com/vinfo/us/security/research-and-analysis/threat-reports/roundup/pushing-the-outer-limits-trend-micro-2024-midyear-cybersecurity-threat-report">highest file detections</a> during the first half of this year. However, such is not the case, and the attacker only seems to be capitalizing on LockBit’s notoriety to further tighten the noose on their victims.</p>
<p>We suspect the ransomware author to be either using their own AWS account or a compromised AWS account. We came across more than thirty samples possibly from the same author, signaling that this ransomware is being actively developed and tested prior to AWS taking action to suspend the Access Keys and the AWS Account. Furthermore, using the hard-coded credential consisting of the AWS Access Key ID, one can <a href="https://hackingthe.cloud/aws/enumeration/get-account-id-from-keys/" target="_blank">find</a> the associated AWS Account ID. This finding offers an alternative perspective of considering malicious or compromised AWS Account IDs as possible IOCs in case of cross-account activities.</p>
<p>This blog describes the samples, their capabilities, and how they abuse Amazon S3 features in their attack.</p>
<h2><span class="body-subhead-title">Technical Analysis</span></h2>
<p>Golang provides developers with a single code base that can compile with dependencies for multiple different platforms. This creates a binary for each platform, making the project multiplatform and dependency-free. Threat actors capitalize on these benefits by creating malicious files with Golang such as the <a href="/en_us/research/22/h/new-golang-ransomware-agenda-customizes-attacks.html">Agenda ransomware</a> as well as the newly-discovered <a href="/en_us/research/24/i/earth-lusca-ktlvdoor.html">KTLVdoor backdoor used by Earth Lusca</a>.<b></b></p>
<p>For the ransomware samples we analyzed, most of the samples have AWS Access Key IDs and the Secret Access Keys hard-coded. While examining the inner workings of the sample, we found that it abuses a feature of AWS S3 known as S3 Transfer Acceleration (<a href="https://aws.amazon.com/s3/transfer-acceleration/" target="_blank">S3TA</a>).</p>
<p>Our analysis is based mainly on the following samples:</p>
<ol>
<li><span style="font-weight: normal;">14fe0071e76b23673569115042a961136ef057848ad44cf35d9f2ca86bd90d31</span></li>
<li><span style="font-weight: normal;">0c54e79e8317e73714f6e88df01bda2c569ec84893a7a33bb6e8e4cf96980430</span></li>
</ol>

		<div>
			<p>When executed on the infected machine, the ransomware first performs initialization through the following steps:</p>
<ol>
<li><span style="font-weight: normal;">Get the host machine universal unique identifier (UUID)</span></li>
<li><span style="font-weight: normal;">Import the hard-coded public key</span></li>
</ol>
<p>The public key is encoded in <a href="https://jpassing.com/2021/11/30/using-pem-to-encode-public-keys/" target="_blank">Privacy Enhanced Mail (PEM) format</a>.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/fig2.png" alt="Hard-coded public key in PEM format"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 2. Hard-coded public key in PEM format</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Decoding the values of the public key reveals RSA encryption and the modulus size of 2048 bits.</p>
<ol start="3">
<li><span style="font-weight: normal;">A random master key is generated and encrypted using the previously imported RSA public key. (This means that only the threat actor who owns the private key can use it to decrypt the master key.)<br />
 </span></li>
<li><span style="font-weight: normal;">Write the encrypted master key to the readme text file (<i>README.txt</i>).<br />
 </span></li>
<li><span style="font-weight: normal;">Use <a href="https://github.com/aws/aws-sdk-go-v2" target="_blank">AWS SDK for Go v2</a> library’s StaticCredentialsProvider to load static credentials. Static credentials include hard-coded AccessKeyID, SecretAccessKey, and AWS_REGION.</span></li>
</ol>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/Fig3.png" alt="Hard-coded AWS credentials"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 3. Hard-coded AWS credentials</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>After the initialization, the ransomware starts enumerating all files available in / (root directory for the macOS variant) by calling the <a href="https://pkg.go.dev/path/filepath#Walk" target="_blank">filepath.Walk</a> function. Each enumerated file is checked to confirm if it is in the exclusion folder. If yes, such files will not be encrypted.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/fig4.png" alt="Exclusion folders, macOS variant"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 4. Exclusion folders, macOS variant</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The ransomware contains a list of file extensions (usually for documents, photos, and data files) that will be encrypted.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/fig5.png" alt="Targeted file extensions"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 5. Targeted file extensions</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The <i>README.txt</i> file name is excluded from encryption.</p>
<h2><span class="body-subhead-title">Exfiltration</span></h2>
<p>Based on the acquired host machine UUID, the sample creates an Amazon S3 bucket on the attacker-controlled AWS account using the hard-coded pair of credentials.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/fig6.png" alt="Creation of Amazon S3 bucket based on host machine UUID"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 6. Creation of Amazon S3 bucket based on host machine UUID</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Once the bucket has been created, the S3TA feature is enabled by modifying the configuration.</p>
<p>The last step is encryption of the file from the beginning to the end. The encryption algorithm is AES-CTR, with password being md5 hash file name concatenated with master key.</p>
<p>As an example, ransomware generates random 16-byte master key 20 60 A3 EA 54 84 C9 27 57 76 1E CC 1F FC 12. Name of the encrypted file is text.txt.</p>
<p>So the concatenated byte sequence is 74 65 78 74 2E 74 78 74 63 20 60 A3 EA 54 84 C9 27 57 76 1E CC 1F FC 12 and its MD5 hash is 23 a3 ec c5 58 2d 97 41 07 3c 3b dc 31 7d 49 30.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/fig7.png" alt="S3TA is enabled"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 7. S3TA is enabled</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Files are then uploaded from the victim’s machine to the attacker-controlled AWS account.</p>
<p>S3TA enables users to achieve faster data transfer over long distances. It leverages the globally distributed edge locations in Amazon CloudFront. To use this feature, it must be enabled on the bucket. The bucket name should be Domain Name System (DNS) compliant and must not have periods. An S3 bucket with S3TA enabled can be accessed by the following endpoints, depending on the type of AWS environment:</p>
<ol>
<li><span style="font-weight: normal;">bucketname[.]s3-accelerate.amazonaws.com</span></li>
<li><span style="font-weight: normal;">bucketname[.]s3-accelerate.dualstack.amazonaws.com</span></li>
</ol>
<p>Each file, which passed the previous file extension checks and is smaller than 100 mebibytes (MiB), is uploaded to AWS by calling the <a href="https://pkg.go.dev/github.com/aws/aws-sdk-go-v2/feature/s3/manager#Uploader.Upload" target="_blank">Uploader.Upload</a> function. This is due to saving AWS space and funds, as uploading big files will cost attackers more money.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/fig8.png" alt="Uploading only files smaller than 100MiB"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 8. Uploading only files smaller than 100MiB</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The last step is encryption of the file from beginning to end. The encryption algorithm is AES-CTR, with the password being the MD5 hash of the file name concatenated with the master key.</p>
<p>The ransomware generates a random 16-byte master key (for example 63 20 60 A3 EA 54 84 C9 27 57 76 1E CC 1F FC 12). The name of the encrypted file is text.txt.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/fig9.png" alt="Ransomware generates a master key"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 9. Ransomware generates a master key</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Correspondingly, the concatenated byte sequence is 74 65 78 74 2E 74 78 74 63 20 60 A3 EA 54 84 C9 27 57 76 1E CC 1F FC 12 and its MD5 hash is 23 a3 ec c5 58 2d 97 41 07 3c 3b dc 31 7d 49 30. This is shown in the screenshot below (generated via <a href="https://gchq.github.io/CyberChef/" target="_blank">CyberChef</a>, used here for visualization purposes only).</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/fig10.png" alt="Process of generating an AES key"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 10. Process of generating an AES key</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>This resulting hash is used as AES key parameter of <a href="https://pkg.go.dev/crypto/aes#NewCipher" target="_blank">crypto.AES.NewCipher function</a>. The initialization vector is a randomly generated 16-bytes and is passed into crypto.cipher.NewCTR function.</p>
<p>After the encryption, the file is renamed according to the following format: &lt;original file name&gt;.&lt;initialization vector&gt;.abcd. For instance, the file <i>text.txt</i> was renamed to <i>text.txt.e5c331611dd7462f42a5e9776d2281d3.abcd</i>.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/fake-lockbit-real-damage-ransomware-samples-abuse-aws-s3-to-steal-data/fig11.png" alt="Appending an .abcd extension to the encrypted files"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 11. Appending an .abcd extension to the encrypted files</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>We ran the ransomware sample in the debugger and dump master key. Then we verified that we can correctly decrypt the previously encrypted file by choosing the proper cipher and passing the correct parameters, as shown in the screenshot below (generated via <a href="https://gchq.github.io/CyberChef/" target="_blank">CyberChef</a>, used here for visualization purposes only).</p>

		</div>

    <div class="richText">
	
    


	
		<div>
			<p>The <i>README.txt</i> file contains base64 encoded content. Decoding it reveals the master key encrypted by crypto.rsa.EncryptPKCS1v15 with a hard-coded public key as its parameter, then encoded by base64. This base64 encoded blob is followed by hostname, OS version, and infected machine identifier. To decrypt the master key, we would need the ransomware developer’s private key, which we do not have.</p>

		</div>
	

</div>

		<div>
			<p>After all files are processed, the ransomware changes the device’s wallpaper. We observed two different wallpapers in use, and both have been stolen or copied either from LockBit attacks or from a <a href="https://blogs.blackberry.com/en/2021/08/threat-spotlight-lockbit-2-0-ransomware-takes-on-top-consulting-firm" target="_blank">security blog</a> mentioning the ransomware family. It should be noted however that 2.0 is not the <a href="/en_us/research/24/b/lockbit-attempts-to-stay-afloat-with-a-new-version.html">latest LockBit version</a>. Furthermore, key figures behind the ransomware operations have just been apprehended <a href="https://www.justice.gov/opa/pr/us-charges-russian-national-developing-and-operating-lockbit-ransomware" target="_blank">earlier this year</a>.</p>

		</div>

    <div class="richText">
	
    


	
		<div>
			<p>On macOS, the ransomware uses <a href="https://ss64.com/mac/osascript.html" target="_blank">osascript</a> to change the wallpaper. The osascript command is as follows:<br />
“tell application &quot;System Events&quot; to tell every desktop to set picture to &quot;%s”.</p>
<p>On Windows, the ransomware calls <a href="https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-systemparametersinfow" target="_blank">SystemParametersInfoW</a> with uiAction parameter set to SPI_SETDESKWALLPAPER to change the wallpaper.</p>
<p>In some Windows samples, we can also find code for deleting backups (<a href="https://learn.microsoft.com/en-us/windows/win32/vss/shadow-copies-and-shadow-copy-sets" target="_blank">shadow copy</a>). Interestingly, the ransomware developer likely, without understanding, copied <i>shadowcopy.go</i> from a <a href="https://github.com/NextronSystems/ransomware-simulator/blob/master/lib/shadowcopy/shadowcopy.go" target="_blank">ransomware-simulator</a> project and left the  <a href="https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/vssadmin-delete-shadows" target="_blank">parameter /for</a> unchanged.</p>

		<div>
			<h2><span class="body-subhead-title">Conclusion</span></h2>
<p>Attackers are increasingly leveraging cloud services and features to further their malicious activities. In this blog, we analyzed a Golang ransomware that abuses Amazon S3's Transfer Acceleration feature to upload victim files to attacker-controlled S3 buckets. Such advanced capabilities enable attackers to efficiently exfiltrate data as they take advantage of cloud service providers.</p>
<p>Furthermore, account identifiers of cloud providers such as AWS Account IDs linked to malicious activities can serve as valuable IOCs. By tracking these IDs, defenders can better identify and mitigate threats within their cloud environments, underscoring the need for vigilant monitoring of cloud resources.</p>
<p>Threat actors might also disguise their ransomware sample as another more publicly known variant, and it is not difficult to see why: the infamy of high-profile ransomware attacks further pressures victims into doing the attacker’s bidding.</p>
<p>To further boost security, organizations can also employ security solutions such as <a href="/en_us/business/products/one-platform.html">Vision One</a> to detect and stop threats early and no matter where they are in the system.</p>

<p>AWS can confirm that AWS services are operating as intended. The activity identified violates the AWS <a href="https://aws.amazon.com/aup/" target="_blank">acceptable use policy</a> and the reported AWS access keys and account have been suspended.<br />
<br />
Ransomware is not specific to any computing environment in particular. However, AWS provides customers with increased visibility into and control over their security posture with respect to <a href="https://aws.amazon.com/security/protecting-against-ransomware/" target="_blank">malware</a>.<br />

</p>
