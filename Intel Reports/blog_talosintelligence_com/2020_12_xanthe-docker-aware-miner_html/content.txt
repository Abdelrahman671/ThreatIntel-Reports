    <h1 class="text-center">Xanthe - Docker aware miner</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/vanja-svajcer/">Vanja Svajcer</a>

        <br/>
        <time class="post-datetime" datetime="December 1, 2020 11:04">
            Tuesday, December 1, 2020 11:04
        </time>

        <div class="m-3">

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-0C_ptfC5pT4/X8YUlp-c6BI/AAAAAAAAAiU/pGEAlBkdNTc2HO57Dl0SohqwdchzxibwwCLcBGAsYHQ/s16000/image2.png" class="kg-image" alt loading="lazy"></figure><p>By <a href="https://twitter.com/vanjasvajcer">Vanja Svajcer </a>and Adam Pridgen, Cisco Incident Command</p><h3 id="news-summary">NEWS SUMMARY <br></h3><ul><li>Ransomware attacks and big-game hunting making the headlines, but adversaries use plenty of other methods to monetize their efforts in less intrusive ways.</li><li>Cisco Talos recently discovered a cryptocurrency-mining botnet attack we're calling "Xanthe," which attempted to compromise one of Cisco's security honeypots for tracking Docker-related threats.</li><li>These threats demonstrate several techniques of the <a href="https://attack.mitre.org/">MITRE ATT&amp;CK framework</a>, most notably Disabling Security Tools - <a href="https://attack.mitre.org/techniques/T1089/">T1089</a>, External Remote Services - <a href="https://attack.mitre.org/techniques/T1133/">T1133</a>, Exploit Public-Facing Application - <a href="https://attack.mitre.org/techniques/T1190/">T1190</a>, Resource Hijacking - <a href="https://attack.mitre.org/techniques/T1496/">T1496</a>, Scheduled Task - <a href="https://attack.mitre.org/techniques/T1053/">T1053</a>, Bash History - <a href="https://attack.mitre.org/techniques/T1552/003/">T1139</a>, SSH Hijacking - <a href="https://attack.mitre.org/techniques/T1563/001/">T1184</a> and Rootkit - <a href="https://attack.mitre.org/techniques/T1014/">T1014</a>.</li></ul><p>Attackers are constantly reinventing ways of monetizing their tools. Cisco Talos recently discovered an interesting campaign affecting Linux systems employing a multi-modular botnet with several ways to spread and a payload focused on providing financial benefits for the attacker by mining Monero online currency.</p><p>The actor employs various methods to spread across the network, like harvesting client-side certificates for spreading to known hosts using ssh, or spreading to systems with an incorrectly configured Docker API.</p><h4 id="what-s-new-we-believe-this-is-the-first-time-anyone-s-documented-xanthe-s-operations-the-actor-is-actively-maintaining-all-the-modules-and-has-been-active-since-march-this-year-">What's new? <br> We believe this is the first time anyone's documented Xanthe's operations. The actor is actively maintaining all the modules and has been active since March this year.</h4><h4 id="how-did-it-work-the-infection-starts-with-the-downloader-module-which-downloads-the-main-installer-module-which-is-also-tasked-with-spreading-to-other-systems-on-the-local-and-remote-networks-the-main-module-attempts-to-spread-to-other-known-hosts-by-stealing-the-client-side-certificates-and-connecting-to-them-without-the-requirement-for-a-password-">How did it work? <br> The infection starts with the downloader module, which downloads the main installer module, which is also tasked with spreading to other systems on the local and remote networks. The main module attempts to spread to other known hosts by stealing the client-side certificates and connecting to them without the requirement for a password.</h4><p>Two additional bash scripts terminate security services, removing competitor's botnets and ensuring persistence by creating scheduled cron jobs and modifying one of the system startup scripts.</p><p>The main payload is a variant of the XMRig Monero mining program that is protected with a shared object developed to hide the presence of the miner's process from various tools for process enumeration.</p><h4 id="so-what-defenders-need-to-be-constantly-vigilant-and-monitor-the-behavior-of-systems-within-their-network-attackers-are-like-water-they-look-for-the-smallest-crack-to-seep-in-like-we-see-in-xanthe-s-potential-to-spread-using-systems-with-exposed-docker-api-while-organizations-need-to-be-focused-on-protecting-their-most-valuable-assets-they-should-not-ignore-threats-that-are-not-specifically-targeted-at-their-infrastructure-">So what? <br> Defenders need to be constantly vigilant and monitor the behavior of systems within their network. Attackers are like water  they look for the smallest crack to seep in, like we see in Xanthe's potential to spread using systems with exposed Docker API. While organizations need to be focused on protecting their most valuable assets, they should not ignore threats that are not specifically targeted at their infrastructure.</h4><h3 id="technical-case-overview">Technical case overview <br></h3><h4 id="introduction-honeypots-are-one-of-the-most-commonly-used-systems-for-collecting-new-threats-and-threat-intelligence-with-a-number-of-open-source-implementations-cisco-talos-and-the-cisco-security-trust-organization-deploy-numerous-honeypots-tracking-attacks-on-many-platforms-protocols-and-applications-">Introduction <br> Honeypots are one of the most commonly used systems for collecting new threats and threat intelligence, with a number of open-source implementations. Cisco Talos and the Cisco Security &amp; Trust organization deploy numerous honeypots tracking attacks on many platforms, protocols and applications.</h4><p>Docker has been one of the most popular platforms for managing containers. Over time it became a de facto standard for development and deployment of web and cloud applications. However, anybody who has worked with Docker quickly realises that the learning curve is quite steep. Therefore Docker installations can be easily misconfigured and Docker daemon exposed to external networks with a minimal level of security.</p><p>According to Shodan, there are over 6,000 incorrectly-configured Docker implementations exposed to the internet, and attackers have been actively finding ways to exploit those exposed servers.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-17s854cfQeU/X8YVNZUsoQI/AAAAAAAAAic/f1LQ7GB9J4MMeAgthgLCqX_pdjeiXg42ACLcBGAsYHQ/s16000/image4.jpg" class="kg-image" alt loading="lazy"></figure><p><i>Incorrectly configured Docker installations with exposed, unprotected APIs according to Shodan</i> <br> Docker daemons have been targets of attacks for several years now, and our security teams decided to keep tracking activities related to various actors attempting to exploit them.</p><h4 id="docker-honeypot-the-cisco-security-and-trust-organization-developed-an-implementation-of-a-docker-honeypot-the-docker-honeypot-is-a-simple-server-that-emulates-some-aspects-of-the-docker-http-api-the-server-will-respond-to-">Docker honeypot <br> The Cisco Security and Trust organization developed an implementation of a <a href="https://github.com/ciscocsirt/dhp">Docker honeypot</a>. The Docker honeypot is a simple server that emulates some aspects of the Docker HTTP API. The server will respond to:</h4><ul><li>HTTP GET version</li><li>HTTP GET ping</li><li>HTTP POST create image</li><li>HTTP Error Code 500 in almost all other cases</li></ul><p>The assumption is that this service is running in a cloud provider somewhere. Currently, the service is a script that runs in a shell. When a recon event or the creation of a container is detected, the server will log the event to any of the following services:</p><ul><li>Webex Teams</li><li>Slack</li><li>Mongodb</li><li>HTTP Collector</li></ul><p>As a part of reconnaissance, the honeypots log suspicious activity related to the potential Docker attacks. This is where we recently spotted the following attempt:</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-YVd7Vy7eTFA/X8YVw8C6_DI/AAAAAAAAAio/8I16lcK-tgsFafW-I1bBTT97QSKmXVvkQCLcBGAsYHQ/s16000/image1.png" class="kg-image" alt loading="lazy"></figure><h4 id="xanthe-a-docker-aware-crypto-miner-the-attempt-belongs-to-a-relatively-unknown-crypto-mining-botnet-which-we-will-call-xanthe-based-on-the-file-name-of-the-main-spreading-script-when-investigating-open-source-information-we-found-earlier-samples-of-the-bot-on-virustotal-detected-by-a-very-few-anti-malware-engines-in-fact-despite-xanthe-having-been-active-since-march-2020-only-one-of-the-scripts-had-been-detected-">Xanthe - a Docker-aware crypto miner <br> The attempt belongs to a relatively unknown crypto mining botnet, which we will call Xanthe, based on the file name of the main spreading script. When investigating open-source information, we found earlier samples of the bot on VirusTotal detected by a very few anti-malware engines. In fact, despite Xanthe having been active since March 2020, only one of the scripts had been detected.</h4><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-h-7hT8zehgQ/X8YXAl42kaI/AAAAAAAAAi0/ecTeHoDluy8ua2eO7bdIr0uFv2ZcIqWbgCLcBGAsYHQ/s16000/image3.png" class="kg-image" alt loading="lazy"></figure><p><i>Older Xanthe versions and the VirusTotal detections</i> <br> The initial script, pop.sh, is a simple downloader script which downloads and runs the main bot module xanthe.sh, which then downloads and runs all additional modules. It is also tasked with scanning the network and spreading to other systems over SSH and by exploiting Docker daemon installations with exposed web API.</p><p>There are four modules that are downloaded and launched:</p><ul><li>Process-hiding module libprocesshider.so</li><li>Shell script to disable other miners and security services</li><li>Shell script to remove Docker containers of competing Docker-targeting crypto mining trojans</li><li>XMRig binary together with a downloaded JSON configuration file, config.json <br></li></ul><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-_mDPevdGRAc/X8YYAM-EVzI/AAAAAAAAAjA/fbKLV9qiy1MwVArBNi0Z7auBYHKhz49ngCLcBGAsYHQ/s16000/image10.jpg" class="kg-image" alt loading="lazy"></figure><p><i>Xanthe, its modules and the spreading vectors</i> <br></p><h4 id="main-module-xanthe-sh-xanthe-sh-as-well-as-most-of-the-other-xanthe-modules-is-a-shell-script-tasked-to-load-additional-modules-and-spread-the-bot-to-other-systems-">Main module - xanthe.sh <br> Xanthe.sh, as well as most of the other Xanthe modules, is a shell script tasked to load additional modules and spread the bot to other systems.</h4><p>The main Xanthe module starts with an initialization procedure that attempts to download and run two additional modules: xesa.txt and fczyo. We describe both in detail in the next section.</p><p>The next procedure downloads a variant of the XMRig miner as java_c, its JSON configuration file and the libprocesshider shared library used for hiding the miner process name in memory. The integrity of every downloaded file is verified with a hardcoded MD5 checksum value. If the MD5 checksum is not verified, the download is reattempted.</p><p>The URLs for download are hardcoded in an array per URL, with direct IP addresses 139[.]162[.]124[.]27 and 34[.]92[.]166[.]158 used for download. At the moment of writing this post, the IP address 34[.]92[.]166[.]158 runs an NginX version 1.19.2, which was released in August this year. The address 139[.]162[.]124[.]27 runs a Debian Linux distribution with an older Apache version 2.4.10 serving the files. Both of the server directories, /files/, used to host the malicious files are open and potentially misconfigured.</p><p>Xanthe uses curl to download its modules and to communicate with the logging URLs. Iplogger.org is extensively used throughout the code, with each procedure communicating with a unique logging URL. Curl's user agent strings are manually specified for each request depending on the phase and the functionality currently being executed.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-97XayaogC0o/X8YYSdJPuJI/AAAAAAAAAjI/uNCf2FXZDjA6Og4Geeze6tgClfCZ4f5PgCLcBGAsYHQ/s16000/image11.jpg" class="kg-image" alt loading="lazy"></figure><p><i>User-agent strings and the associated functionality</i></p><p>The execution of the main module continues with making sure that the /tmp directory is mounted and configured to allow execution of the files within it.</p><p>Next, the procedure checks if the miner is already running in memory by running the ps command. First, the loading of the libprocesshider is disabled, then ps is used to make sure the java_c process is running.</p><h4 id="sshd-configuration-and-spreading-the-main-infection-vector-for-this-variant-of-xanthe-seems-to-be-ssh-which-is-achieved-by-specifying-client-side-certificates-stored-for-known-hosts-">sshd configuration and spreading <br> The main infection vector for this variant of Xanthe seems to be SSH, which is achieved by specifying client-side certificates stored for known hosts.</h4><p>First, Xanthe configures the SSH daemon to make its configuration less secure and enable some functionality. The daemon is configured to run on ports 22 and 33768 with the root account login, password, client public key and <a href="https://www.ssh.com/manuals/clientserver-product/52/Secureshell-gssapiuserauthentication.html">GSSAPIAuthentication</a> (such as <a href="https://en.wikipedia.org/wiki/Kerberos_(protocol)">Kerberos</a> authentication) enabled. Once the configuration file is modified, the SSH service is restarted.</p><p>Finally, we come to the spreading function, localgo. The function starts with a fetch of the externally-visible IP address of the infected host by connecting to icanhazip.com. Next, the script uses the find utility to search for instances of client-side certificates, which will be used for authentication to remote hosts. The find is used to search for any filenames starting with the string 'id_rsa' and for files with the '.pem' file extension. The script also uses a combination of 'cat' and 'grep' to find other keys by parsing the SSH configuration files and the bash history file.</p><p>Once all possible keys have been found, the script proceeds with finding known hosts, TCP ports and usernames used to connect to those hosts. Finally, a loop is entered which iterates over the combination of all known usernames, hosts, keys and ports in an attempt to connect, authenticate on the remote host and launch the command lines to download and execute the main module on the remote system.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-BNA6Cbek7m0/X8YYuLeh7tI/AAAAAAAAAjQ/7YC2F5vyBA0tK0u4M8245CME1-YpYNkTwCLcBGAsYHQ/s16000/image5.png" class="kg-image" alt loading="lazy"></figure><p><i>SSH Xanthe-based spreading command</i> <br></p><h4 id="available-functionality-commented-out-although-we-have-seen-this-bot-hitting-our-docker-honeypots-with-an-attempt-to-spread-at-the-time-of-the-analysis-the-routine-for-docker-scanning-and-spreading-was-commented-out-the-author-s-programming-style-is-similar-in-all-scripts-first-the-functions-are-defined-and-implemented-with-all-the-flow-defined-towards-the-bottom-of-the-script-where-the-actor-can-choose-which-functionality-to-exhibit-by-commenting-out-undesirable-function-calls-">Available functionality commented out <br> Although we have seen this bot hitting our Docker honeypots with an attempt to spread, at the time of the analysis the routine for Docker scanning and spreading was commented out. The author's programming style is similar in all scripts. First the functions are defined and implemented, with all the flow defined towards the bottom of the script, where the actor can choose which functionality to exhibit by commenting out undesirable function calls.</h4><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-EyVotazriFo/X8YY_XiDokI/AAAAAAAAAjY/1Xqam7G6D50zuEZMxVTBDg_kVxXFiJxywCLcBGAsYHQ/s16000/image9.png" class="kg-image" alt loading="lazy"></figure><p><i>Functions are enabled/disabled by commenting a line using the character #</i></p><p>The Docker spreading functionality is contained within the function 'scangogo'. The function first attempts to install the mass port-scanning utility masscan to scan the local network, as well as the /24 subnet of the external IP address, for the hosts with incorrectly configured Docker API exposed on TCP port 2375.</p><p>If a suitable exposed Docker API is found, Xanthe will attempt to run Docker with the parameter pointing to the remote host, create a new container based on the busybox image and run the command to launch the downloader bash script pop.sh.</p><h4 id="auxiliary-shell-scripts-as-soon-as-the-main-module-is-launched-it-downloads-and-runs-the-two-auxiliary-modules-xesa-txt-and-fczyo-">Auxiliary shell scripts As soon as the main module is launched, it downloads and runs the two auxiliary modules, xesa.txt and fczyo.</h4><h5 id="xesa-txt-the-killer-module-xesa-starts-initially-with-removing-the-system-log-file-and-adding-a-number-of-ip-addresses-to-the-etc-hosts-file-so-that-they-are-resolved-locally-by-the-dns-resolver-to-0-0-0-0-and-blocked-the-ip-addresses-are-related-to-competing-botnets-and-security-services-such-as-the-alibaba-cloud-security-center-agent-xesa-also-attempts-to-delete-several-accounts-which-seem-to-be-related-to-competing-botnets-such-as-kinsing-">Xesa.txt - the killer module Xesa starts initially with removing the system log file and adding a number of IP addresses to the /etc/hosts file so that they are resolved locally by the DNS resolver to 0.0.0.0 and blocked. The IP addresses are related to competing botnets and security services such as the <a href="https://www.alibabacloud.com/product/security-center">Alibaba cloud security center</a> agent. Xesa also attempts to delete several accounts, which seem to be related to competing botnets such as <a href="https://threatpost.com/self-propagating-malware-docker-ports/154453/">Kinsing</a>.</h5><p>The execution continues with running a large number of ps commands to determine if competitive miners are running on the infected system. If any of the miners are found, their processes will be terminated and files removed from the system.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-EOIEVx3e8qA/X8YZ7pLykDI/AAAAAAAAAjk/FBPFsjVt5lMlneEspqnLawAzIcCI3CEKgCLcBGAsYHQ/s16000/image8.png" class="kg-image" alt loading="lazy"></figure><p><i>Docker images of competitor's miners being removed from the infected system</i></p><p>One of the main Xesa functions is terminating competitor's Docker containers if they are running and removing their images from the local Docker repository. The script finishes with another check that the java_c process is running.</p><h5 id="fczyo-persistence-and-remote-access-setup-the-second-auxiliary-module-fczyo-is-similar-to-xesa-txt-but-it-covers-slightly-different-areas-of-the-system-for-termination-the-execution-starts-with-adding-two-of-the-public-dns-servers-1-1-1-1-and-8-8-8-8-to-the-beginning-of-the-etc-resolve-conf-file-and-restarting-the-dns-resolver-to-use-those-public-servers-it-then-continues-with-terminating-competing-docker-containers-if-they-are-running-">Fczyo - persistence and remote access setup <br> The second auxiliary module, fczyo, is similar to xesa.txt, but it covers slightly different areas of the system for termination. The execution starts with adding two of the public DNS servers, 1.1.1.1 and 8.8.8.8, to the beginning of the /etc/resolve.conf file and restarting the DNS resolver to use those public servers. It then continues with terminating competing Docker containers if they are running.</h5><p>The script makes sure that the security processes are not running on the infected system and ensures that crond is running so the scheduled jobs can be executed.</p><p>Fczyo attempts to add the following users to the infected system:</p><ul><li>sysall</li><li>system</li><li>logger</li><li>autoupdater</li></ul><p>Once the users are created, the script adds them to the list of sudoers so that they can run commands with administrative privileges. The script also adds a public key to the SSH configuration to allow those users access to the host using the client-side private key. Iptables are used to configure the firewall to allow connections coming from the IP address 64[.]225[.]46[.]44 and to drop all incoming connections to ports 2375 and 2376, presumably to avoid reinfection of the system by other bots scanning for misconfigured Docker API installations.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-4ZxOcZUAL4M/X8Ya9gtT9GI/AAAAAAAAAjw/NVzgWY1AytYycMCq7Qg3SH3r0OVgXeqvACLcBGAsYHQ/s16000/image6.png" class="kg-image" alt loading="lazy"></figure><p><br></p><p>Fczyo checking if the cron job is already configured</p><p>The next major block of functions is tasked with setting up scheduled cron jobs to ensure that Xanthe modules are downloaded and run from the download server every 30 minutes. It also modifies the file /etc/rc.d/rc.local so Xanthe can be downloaded and run every time the system is started.</p><p>Fczyo also attempts to clear bash history, disable and remove <a href="https://help.ubuntu.com/community/UFW">ufw</a> and any file belonging to the <a href="https://www.sandflysecurity.com/">Sandfly security</a> products.</p><h4 id="process-hiding-module-xanthe-and-its-auxiliary-modules-use-a-shared-object-libprocesshider-which-is-downloaded-and-installed-into-usr-local-lib-libprocesshider-so-the-loading-of-the-shared-object-is-enabled-by-adding-the-path-of-the-library-to-the-file-etc-ld-so-preload-ld-so-preload-contains-a-list-of-user-specified-elf-shared-libraries-to-be-loaded-before-all-others-this-can-be-used-to-selectively-override-functions-in-other-shared-libraries-">Process-hiding module <br> Xanthe and its auxiliary modules use a shared object, libprocesshider, which is downloaded and installed into /usr/local/lib/libprocesshider.so. The loading of the shared object is enabled by adding the path of the library to the file /etc/ld.so.preload. Ld.so.preload contains a list of user-specified ELF shared libraries to be loaded before all others. This can be used to selectively override functions in other shared libraries.</h4><p>Process tools such as ps use the /proc file system and the readdir function to enumerate all the running processes. If a malicious shared object is used to implement a malicious copy of the readdir function, this function will be called before the standard library function. The malicious function then has the ability to modify the results of the readdir to omit details of some processes. The source code of the library is publicly available on Github, so the attacker only had to modify a hardcoded string containing the name of the process executable that needs to be hidden, in our case java_c.</p><h4 id="conclusion-in-this-post-we-documented-the-previously-undocumented-monero-mining-botnet-xanthe-which-attempts-to-spread-over-ssh-using-existing-private-keys-for-authentication-we-discovered-this-botnet-when-it-registered-in-a-docker-honeypot-system-the-main-module-contains-functions-disabled-in-this-variant-to-spread-by-exploiting-incorrectly-configured-docker-api-installations-">Conclusion <br> In this post, we documented the previously-undocumented Monero mining botnet Xanthe, which attempts to spread over SSH using existing private keys for authentication. We discovered this botnet when it registered in a Docker honeypot system. The main module contains functions, disabled in this variant, to spread by exploiting incorrectly configured Docker API installations.</h4><p>While Docker remains an essential tool for development and deployment of applications, it is worth remembering that its learning curve is steep. The installation is not secure by default, and it is easy to leave its API exposed to attackers on a lookout for 'free' resources they can use to run custom containers and conduct attacks.</p><p>When setting up organizations' defences, administrators and devops engineers need to make sure all components in their product development and deployment systems are properly configured and secure.</p><h3 id="coverage-ways-our-customers-can-detect-and-block-this-threat-are-listed-below-">Coverage <br> Ways our customers can detect and block this threat are listed below.</h3><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-DsVkfIBZ3_I/X8Ydw4HuyAI/AAAAAAAAAj8/LGj1Xe65Cw0mLZFsPEzAKUWaWEZLQUT6wCLcBGAsYHQ/w393-h400/image7.jpg" class="kg-image" alt loading="lazy"></figure><p>Advanced Malware Protection (<a href="https://www.cisco.com/c/en/us/products/security/advanced-malware-protection">AMP</a>) is ideally suited to prevent the execution of the malware used by these threat actors.
