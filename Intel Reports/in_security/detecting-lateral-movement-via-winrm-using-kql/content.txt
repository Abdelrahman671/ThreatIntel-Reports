<p>Over the past few months we’ve been looking a little more into the detection methods we might use to identify strange activity within a given environment.</p>



<p>A lot of this research stems from questions asked by our clients following a technical engagement, or questions from students that have taken our <a href="https://in.security/technical-training/hacking-enterprises/" target="_blank" rel="noreferrer noopener">Hacking Enterprises training</a>. With the arrival of our new <a href="https://in.security/technical-training/defending-enterprises/" target="_blank" rel="noreferrer noopener">Defending Enterprises training</a> this year where we look at detection methods from our ‘Top 10 in-the-field attacks’, a lot of this research naturally evolved and we found this element particularly interesting.</p>



<p><a href="https://azure.microsoft.com/en-gb/services/azure-sentinel/" target="_blank" rel="noreferrer noopener nofollow">Microsoft Azure Sentinel</a> is fast becoming our go-to SIEM as it not only brings the accessibility of cloud services, but a wealth of functionality at a relatively low price point. In this blog we’re going to take a brief look at the power of the <a href="https://docs.microsoft.com/en-us/azure/data-explorer/kql-quick-reference" target="_blank" rel="noreferrer noopener nofollow">Kusto Query Language (KQL)</a>.</p>



<div style="height:30px" aria-hidden="true" class="wp-block-spacer"></div>



<h2 class="wp-block-heading"><strong><u>Detecting Lateral Movement Through WinRM</u></strong></h2>



<p>Under the context of internal network monitoring, we wanted a quick and easy method to identify when a WinRM or PowerShell Remoting session had been instigated, but we also wanted an idea of where the user both had originated from, and targeted. As PowerShell Remoting uses WinRM to establish connections, the same Indictor of Attack (IOA) could be used.</p>



<p>When a WinRM connection is initialised <em>EventID 6</em> will be recorded (the source host) and when a WinRM connection is received <em>EventID 91</em> is recorded (the target host). Both events will be logged in <strong><em>Microsoft-Windows-WinRM/Operational</em></strong> (Windows Remote Management through the GUI).</p>



<p>Therefore, to chain together such events we can use a <a href="https://docs.microsoft.com/en-us/azure/data-explorer/kusto/query/join-timewindow" target="_blank" rel="noreferrer noopener nofollow">Time Window Join operation</a> to map <em>source &gt; target</em> using a query such as the following:</p>



<p>In short, this query will look for <em>Event ID 6 </em>in the <em>Microsoft-Windows-WinRM </em>log and, if found, a second query is executed that looks for <em>Event ID 91</em>, again in the <em>Microsoft-Windows-WinRM </em>log, but the events have to occur within 1 minute of each other (in larger, busy environments this timing may need to be tuned). If both events are identified (matched on username), details of the event are displayed, as shown in the example below.</p>



<pre class="wp-block-code"><code>Event
| where EventID == 6
| where Source == "Microsoft-Windows-WinRM"
| project SourceEvent = EventID, SourceTime=TimeGenerated, UserName, SourceComputer = Computer
| join kind=inner
    (
    Event 
    | where EventID == 91
    | where Source == "Microsoft-Windows-WinRM"
    | project TargetEvent = EventID, TargetTime=TimeGenerated, UserName, TargetComputer = Computer
    ) on UserName
| where (TargetTime - SourceTime) between (0min .. 1min)
| project SourceEvent, TargetEvent, SourceComputer, TargetComputer, UserName, SourceTime, TargetTime
| sort by SourceTime desc
</code></pre>



<figure class="wp-block-image size-full"><img fetchpriority="high" decoding="async" width="1024" height="317" src="https://in.security/wp-content/uploads/2022/04/WinRM_6_91-1024x317-1.png" alt="" class="wp-image-989" srcset="https://in.security/wp-content/uploads/2022/04/WinRM_6_91-1024x317-1.png 1024w, https://in.security/wp-content/uploads/2022/04/WinRM_6_91-1024x317-1-300x93.png 300w, https://in.security/wp-content/uploads/2022/04/WinRM_6_91-1024x317-1-768x238.png 768w" sizes="(max-width: 1024px) 100vw, 1024px" /></figure>



<div style="height:30px" aria-hidden="true" class="wp-block-spacer"></div>



<h2 class="wp-block-heading"><strong><u>Identifying Further Activity Through PowerShell Logging</u></strong></h2>



<p>We can further investigate any activities by collating <em>EventID 91</em> (a WinRM connection has been received – i.e. the target host) with PowerShell logging <em>EventID 4103</em> (Module Logging), to see what may have been executed after the connection was made.</p>



<p>As per the previous query, we’re using a <em>join </em>statement with much of the same underlying logic. In this instance if <em>Event ID 91</em> is identified, a second query is executed that looks for <em>Event ID 4103</em> within 1 minute of the initial connection. If both events are identified, details are displayed, as shown in the example below.</p>



<pre class="wp-block-code"><code>Event
| where EventID == 91
| where Source == "Microsoft-Windows-WinRM"
| project SourceEvent = EventID, SourceTime=TimeGenerated, UserName, Computer
| join kind=inner
    (
    Event 
    | where EventID == 4103
    | where Source == "Microsoft-Windows-PowerShell"
    | project TargetEvent = EventID, TargetTime=TimeGenerated, UserName, Computer, RenderedDescription
    ) on Computer
| where (TargetTime - SourceTime) between (0min .. 1min)
| project SourceEvent, TargetEvent, Computer, RenderedDescription , UserName, SourceTime, TargetTime
| sort by SourceTime desc
</code></pre>

<figure class="aligncenter size-full"><img decoding="async" width="1024" height="215" src="https://in.security/wp-content/uploads/2022/04/WinRM_91_4103-1024x215-1.png" alt="" class="wp-image-988" srcset="https://in.security/wp-content/uploads/2022/04/WinRM_91_4103-1024x215-1.png 1024w, https://in.security/wp-content/uploads/2022/04/WinRM_91_4103-1024x215-1-300x63.png 300w, https://in.security/wp-content/uploads/2022/04/WinRM_91_4103-1024x215-1-768x161.png 768w" sizes="(max-width: 1024px) 100vw, 1024px" /></figure></div>
