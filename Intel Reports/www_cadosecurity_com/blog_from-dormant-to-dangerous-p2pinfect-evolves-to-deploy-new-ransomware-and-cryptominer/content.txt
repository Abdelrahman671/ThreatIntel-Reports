<p><span style="color: #000000;">P2Pinfect is a rust-based malware covered extensively by Cado Security in the past. It is a fairly sophisticated malware sample that uses a peer-to-peer (P2P) botnet for its command and control mechanism. Upon initial discovery, the malware appeared mostly dormant. It would spread primarily via Redis and a limited SSH spreader, but ultimately did not seem to have an objective other than to spread. Recently, we have observed a new update to P2Pinfect that introduces a ransomware and crypto miner payload.</span></p>

<!--more-->
<p style="font-weight: bold;"><span style="font-size: 20px; color: #000000;">Recap</span></p>
<p><span style="color: #000000;">Let’s recap what we know so far about P2Pinfect. Cado Security researchers first discovered it during triage of honeypot telemetry in July of 2023. Based on our findings, we determined the campaign began on June 23rd based on the TLS certificate used for C2 communications.</span></p>
<p style="font-weight: bold;"><span style="font-size: 16px; color: #000000;">Initial Access</span></p>
<p><span style="color: #000000;">The malware spreads by exploiting the replication features in Redis - where Redis runs in a distributed cluster of many nodes, using a leader/follower topology. This allows follower nodes to become an exact replica of the leader nodes, allowing for reads to be spread across the whole cluster to balance load, and provide some resilience in case a node goes down.</span></p>
<p><span style="color: #000000;">This is frequently exploited by attackers, as leaders can instruct followers to load arbitrary modules, which can in turn be used to gain code execution on the follower nodes. P2Pinfect exploits this by using the <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">SLAVEOF</span> command to turn discovered opened Redis nodes into a follower node of the attacker server. It then uses a series of commands to write out a shared object (.so) file, and then instructs the follower to load it. Once this is done, the attacker can send arbitrary commands to the follower for it to execute.</span></p>

<p><span style="color: #000000;">P2Pinfect also utilises another Redis initial access vector where it abuses the config commands to write a cron job to the cron directory:</span></p>

<h2><span style="font-size: 16px; color: #000000;">Main Payload</span></h2>
<p><span style="color: #000000;">P2Pinfect is a worm, so all infected machines will scan the internet for more servers to infect with the same vector described above. P2Pinfect also features a basic SSH password sprayer, where it will try a few common passwords with a few common users, but the success of this infection vector seems to be a lot less than with Redis, likely as it is oversaturated.</span></p>
<p><span style="color: #000000;">Upon launch it drops an SSH key into the authorised key file for the current user and runs a series of commands to prevent access to the Redis instance apart from IPs belonging to existing connections. This is done to prevent other attackers from discovering and exploiting the server. It also tries to update the SSH configuration and restart SSH service to allow root login with password. It will also try changing passwords of other users, and will use sudo (if it has permission to) to perform privilege escalation.</span></p>
<p><span style="color: #000000;">The botnet is the most notable feature of P2Pinfect. As the name suggests, it is a peer-to-peer botnet, where every infected machine acts as a node in the network, and maintains a connection to several other nodes. This results in the botnet forming a huge mesh network, which the malware author makes use of to push out updated binaries across the network, via a gossip mechanism. The author simply needs to notify one peer, and it will inform all its peers and so on until the new binary is fully propagated across the network. When a new peer joins the network, non-expired commands are replayed to the peer by the network.</span></p>
<p style="font-weight: bold;"><span style="font-size: 20px; color: #000000;">Updated Main Payload</span></p>
<p><span style="color: #000000;">The main binary appears to have undergone a rewrite. It now appears to be entirely written using tokio, an async framework for rust, and packed with UPX. Since we first examined the payload, the internals have changed drastically. The binary is stripped and partially obfuscated, making static analysis difficult.</span></p>
<p><span style="color: #000000;">P2Pinfect used to feature persistence by adding itself to <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">.bash_logout</span> as well as a cron job, but it appears to no longer do either of these. The rest of its behaviours, such as the initial setup outlined previously, are the same.</span></p>
<p style="font-weight: bold;"><span style="font-size: 20px; color: #000000;">Updated Bash Behaviour</span></p>
<p><span style="color: #000000;">P2Pinfect drops a secondary binary at <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">/tmp/bash</span> and executes it. This process sets its command line args to <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">[kworker/1:0H] </span>in order to blend in on the process listing. <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">/tmp/bash</span> serves as a health check for the main binary. As previously documented, the main binary listens on a random port between 60100 to 60150 that other botnet peers will connect to. <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">/tmp/bash</span> periodically sends a request to the port to check it is alive, and assumedly will respawn the main binary if it goes down.</span></p>
<p><span style="color: #000000;"><img src="https://lh7-us.googleusercontent.com/docsz/AD_4nXengne3sUbGmduk1wX6UisqZvJKuYubMcHG6mvRTp9prQtHRsmFljNR_vQEuI4Sg_78aGaVbX4cSfdUrdEuL3N1hDp_9OpZAx3E-VdFJ4zzpACWDRCcizVw9hVtrqfdziyPh9SMtZj5XpOsuQiuaZhRcHU?key=Wiyo22OUKvdFSqJW24HloQ" width="602" height="640" loading="lazy"></span></p>
<p><span style="color: #000000;"><em>Sysmon logs for the /tmp/bash payload</em></span></p>
<p style="font-weight: bold;"><span style="font-size: 20px; color: #000000;">Miner Payload Becomes Active</span></p>
<p><span style="color: #000000;">Previously, we have observed a binary called <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">miner</span> that is embedded in P2Pinfect, however this appeared to never be used. We have now observed the main binary dropping the miner binary to a <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">mktmp</span> file (<span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">mktmp</span> creates a file in <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">/tmp</span> with some random characters as the name) and executing it. It features a built-in configuration, with the monero wallet and pool preconfigured. The miner is only activated after approximately five minutes has elapsed since the main payload was started.</span></p>
<p><span style="color: #000000;"><img src="https://lh7-us.googleusercontent.com/docsz/AD_4nXcq_AyppeCQXOjKdKcJ4vn-5OOvwNmTyPl3ZPvsH8YO0-tGfahhGqL5o2aoGzTvRT1odnRRckz7fn5knqyVm_2plG89dp7aJdAfBAICDSSt5xKrSnsG_oPz8ya-UJyiMQG_QyFfSqvzykGs5MfKQdixP7PE?key=Wiyo22OUKvdFSqJW24HloQ" width="602" height="140" loading="lazy"></span></p>
<p><span style="color: #000000;"><em>Wallet details for the attacker’s supposed wallet - 4BDcc1fBZ26HAzPpYHKczqe95AKoURDM6EmnwbPfWBqJHgLEXaZSpQYM8pym2Jt8JJRNT5vjKHAU1B1mmCCJT9vJHaG2QRL</em></span></p>
<p><span style="color: #000000;">The attacker has made around 71 XMR, equivalent to roughly £9,660. Interestingly, the mining pool only shows 1 worker active at 22 KH/s (which generates around £15 a month) which doesn’t seem to match up with the size of the botnet nor how much they have made.</span></p>
<p><span style="color: #000000;">Upon reviewing the actual traffic from the miner, it appears to be trying to make a connection to various Hetzner IPs on TCP port 19999 and does not start mining until this is successful. These IPs appear to belong to the <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">c3pool </span>mining pool and not the <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">supportxmr </span>pool, suggesting that the config may have been left as a red herring. Checking <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">c3pool </span>for the wallet address, there is no activity for the above wallet address beyond September 2023. It is likely that there is another wallet address being used.</span></p>
<p style="font-weight: bold;"><span style="font-size: 20px; color: #000000;">New ransomware payload</span></p>
<p><span style="color: #000000;">Upon joining the botnet, P2Pinfect receives a command instructing it to download and run a new binary called <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">rsagen</span>, which is a ransomware payload.</span></p>

<p><span style="color: #000000;">It is interesting to note that across all detonations, the download URL has not changed, and the command JSON is identical. This suggests that the command was issued directly by the malware operator, and the download server may be an attacker controlled server used to host additional payloads.</span></p>
<p><span style="color: #000000;">This JSON structure is typical of a command from the botnet. As mentioned previously, when a new botnet peer joins the network, it is replayed non-expired commands. The <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">c</span> and <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">e</span> parameters contain timestamps that are likely to be command creation and expiry times, we can determine the command to start the ransomware was issued on the 16th May 2024, and will continue to be active until 17th December 2024. We can also see other interesting parameters, such as type 5 (exec on linux, exec on windows is type 6), as well as retry parameters. Clearly a large amount of thought and effort has been put into designing P2Pinfect, far exceeding the majority of malware in sophistication.</span></p>
<p><span style="color: #000000;">The base64 args of the binary cleanly decode to “<span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">encargs 1 besttrcovery@firemail.cc,randbnothing@tutanota.com</span>” - which are the email addresses used in the ransom note for where to send payment confirmations to. It’s unknown what the encargs 1 part is for.</span></p>
<p><span style="color: #000000;">The main binary obediently downloads and executes the file:</span></p>
<p><span style="color: #000000;"><img src="https://lh7-us.googleusercontent.com/docsz/AD_4nXdJuTLWna9x5K-bwKftPDoERgvUyOrGvyVB3pPGsFoDIPd9sdP1s1Nd9m13cHbuhlaaS1fQRF6Ec8cS9mCmvQNGVQ0XLsNknA1FQEDojjylIPCnQ4L2_WfOYyyNAe3nE5BdOqw8HxD5lc-pK6v7LYWrIls?key=Wiyo22OUKvdFSqJW24HloQ" width="602" height="380" loading="lazy"></span></p>
<p><span style="color: #000000;">Upon launch, <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">rsagen</span> checks if the ransom note already exists in either the current working directory (<span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">/tmp</span>), or the home directory of the user the process is running under. If it does, it exits immediately. Otherwise, it will instead begin the encryption process. The exact cryptographic process is not known, however our assumption is that it generates a public key used to encrypt files, and encrypts the corresponding private key using the attacker’s public key, which is then added to the ransom note. This allows the attacker to then decrypt the private key and return it to the user after they pay, without needing to include any secrets or C2 on the client machine.</span></p>
<p><span style="color: #000000;"><img src="https://lh7-us.googleusercontent.com/docsz/AD_4nXelomLI2sIe8TV3EVGyaOJpEESjdU5XaB6Ulw-yPwyOZv-RZIkfuu6PiPxIaLawvAb909SgHP9NwvHzZn1rcQUVUnQ7Dp54K2gA4oSLKJAdSk6IlJZLjJJJXR9JGjUfC3laPP2LB83JrClmTqyqSimNHpWV?key=Wiyo22OUKvdFSqJW24HloQ" width="602" height="171" loading="lazy"></span></p>
<p><span style="color: #000000;"><em>The ransom note, titled “Your data has been locked!.txt”</em></span></p>
<p><span style="color: #000000;">As they are using monero, it is impossible to figure out how much they have earned so far from the campaign. 1 XMR is currently £136 as of writing, which is on the cheaper end of ransomware. As this is an untargeted and opportunistic attack, it is likely the victims are to be low value, so having a low price is to be expected.</span></p>
<p><span style="color: #000000;">After writing out the note, the ransomware iterates through all directories on the file system, and overwrites the contents with an encrypted version. It then appends <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">.encrypted</span> to the end of the file name.</span></p>
<p><span style="color: #000000;">Linux does not require file extensions on files, however the malware seems to only target files that have specific extensions. Instead of checking for particular extensions, it instead has a massive string which it then checks if the extension is contained in.</span><br><br><span style="color: #000000; font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">mdbmdfmydldfibdmyidbdbfwdbfrmaccdbsqlsqlite3msgemltxtcsv123docwpsxlsetpptppsdpsonevsdjpgpngziprar7ztarbz2tbkgztgzbakbackupdotxlwxltxlmxlcpotpubmppodtodsodpodgodfodbwpdqpwshwpdfaip64xpsrptrtfchmmhthtmurlswfdatrbaspphpjsppashcppccspyshclassjarvbvbsps1batcmdjsplsuoslnbrdschdchdipbmpgificopsdabrmaxcdrdwgdxfmbpspdgnexbjnbdcdqcdtowqxpqptsdrsdtpzfemfociiccpcbtpfgjdaniwmfvfbsldprtdbxpstdwtvalcadfabbsfccfudfftfpcfdocicaascgengcmostwkswk1onetoc2sntedbhwp602sxistivdivmxgpgaespaoisovcdrawcgmtifnefsvgm4um3umidwmaflv3g2mkv3gpmp4movaviasfvobmpgwmvflawavmp3laymmlsxmotguopstdsxdotpwb2slkdifstcsxcots3dm3dsuotstwsxwottpemp12csrcrtkeypfxder</span></p>
<p><span style="color: #000000;">This makes it quite difficult to pick out a complete list of extensions, however going through it we can see many file formats, such as py, sqlite3, sql, mkv, doc, xls, db, key, pfx, wav, mp3, and lots more.</span></p>
<p><span style="color: #000000;">The ransomware stores a database of the files it encrypted in a <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">mktmp</span> file with <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">.lockedfiles</span> appended. The user is then expected to run the <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">rsagen</span> binary again with a decryption token in order to have their files decrypted. Cado Security does not possess a decryption token as this would require paying the attackers.</span></p>
<p><span style="color: #000000;">As the ransomware runs with the privilege level of its parent, it is likely that it will be running as the Redis user in the wild since the main initial access vector is Redis. In a typical deployment, this user has limited permissions and will only be able to access files saved by Redis. It also should not have sudo privileges, so would not be able to use it for privilege escalation.</span></p>
<p><span style="color: #000000;">Redis by default doesn’t save any data to disk and is typically used for in-memory only caching or key value store, so it’s unclear what exactly the ransomware could ransom other than its config files. Redis can be configured to save data to files - but the extension for this is typically rdb, which is not included in the list of extensions that P2Pinfect will ransom.</span></p>
<p><span style="color: #000000;">With that in mind, it’s unclear what the ransomware is actually designed to ransom. As mentioned in the recap, P2Pinfect does have a limited ability to spread via SSH, which would likely compromise higher privilege users with actual files to encrypt. The spread of P2Pinfect over SSH is far more limited compared to Redis however, so the impact is much less widespread.</span></p>
<h1><span style="font-size: 20px; color: #000000;">New usermode rootkit</span></h1>
<p><span style="color: #000000;">P2Pinfect now features a usermode rootkit. It will seek out <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">.bashrc </span>files it has permission to modify in user home directories, and append <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">export LD_PRELOAD=/home/&lt;user&gt;/.lib/libs.so.1</span> to it. This results in the <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">libs.so.1 </span>file being preloaded whenever a linkable executable (such as the ls or cat commands) is run.</span></p>
<p><span style="color: #000000;">The shared object features definitions for the following methods, which hijack legitimate calls to it in order to hide specific information:</span></p>
<ul>
<li aria-level="1"><span style="color: #000000;"><span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">fopen</span> &amp; <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">fopen64</span></span></li>
<li aria-level="1"><span style="color: #000000;"><span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">open</span> &amp; <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">open64</span></span></li>
<li aria-level="1"><span style="color: #000000;"><span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">lsta</span>t &amp; <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">lstat64</span></span></li>
<li aria-level="1"><span style="color: #000000;"><span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">unlink</span> &amp; <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">unlinkat</span></span></li>
<li aria-level="1"><span style="color: #000000;"><span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">readdir</span> &amp; <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">readdir64</span></span></li>
</ul>
<p><span style="color: #000000;">When a call to <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">open</span> or <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">fopen</span> is hijacked, it checks if the argument passed is one of the PIDs associated with the main file, <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">/tmp/bash, </span>or the miner. If it is one of these, it sets errno to 2 (file not found) and returns. Otherwise, it passes the call to the respective original function. If it is a request to open <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">/proc/net/tcp</span> or <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">/proc/net/tcp6,</span> it will filter out any ports between 60100 and 60150 from the return stream.</span></p>
<p><span style="color: #000000;">Similarly with hijacked calls captured to <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">lstat</span> or <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">unlink</span>, it checks if the argument passed is the main process’ binary. It does this by using <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">ends_with</span> string function on the file name, so any file with the same random name will be hidden from stat and unlink, regardless of if it is in the right directory or is the actual main file.</span></p>
<p><span style="color: #000000;">Finally with <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">readdir,</span> it will run the original function, but remove any of the process PIDs or the main file from the returned results.</span></p>
<p><span style="color: #000000;"><img src="https://lh7-us.googleusercontent.com/docsz/AD_4nXfitbiy2QZ8Sz_ZHI7972SKG5ulRxEgVZeRc-HTNGJ8hrcgxorDvj1iiZNaU2LWSotPH_4bX07jnrC7_38wJ25pKOXW3MLOG6Bwxt9oXuGAThiWolFboesOku_sTtdEQxkF4-NmUlraASkh8P8lnZPqUQy-?key=Wiyo22OUKvdFSqJW24HloQ" width="518" height="300" loading="lazy"></span></p>
<p><span style="color: #000000;"><em>The decompiled pseudocode for the hijacked </em><span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;"><em>readdir</em></span><em> function</em></span></p>
<p><span style="color: #000000;">It is interesting to note that when a specific environment variable is set, it will bypass all of the checks. This is likely used to allow shell commands from the other malware binaries to be run without interference by the rootkit.</span></p>
<p><span style="color: #000000;"><img src="https://lh7-us.googleusercontent.com/docsz/AD_4nXcOztp9ZwXqD-JD5FAhVko1FJgizCe5rvyrDznuIfX4RDTMHyoez293hsFwhwKHteonRSZ0_k0SkjEpoBNId34kya90SZPYiGam4vVEwn5DIikLKSwePL9AmdnyAO3XBqfxDlufHiDtjPv65x3__yJUVVkX?key=Wiyo22OUKvdFSqJW24HloQ" width="492" height="272" loading="lazy"></span></p>
<p><span style="color: #000000;"><em>The decompiled pseudocode for the </em><span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;"><em>env_var</em></span><em> check</em></span></p>
<p><span style="color: #000000;">The rootkit is dynamically generated by the main binary at runtime, with it choosing a random <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">env_var</span> to set as the bypass string, and adding its own file name plus PIDs to the SO before writing it to disk.</span></p>
<p><span style="color: #000000;">Like the ransomware, the usermode rootkit suffers from a fatal flaw; if the initial access is Redis, it is likely that it will only affect the Redis user as the Redis user is only used to run the Redis server and won’t have access to other user’s home directories.</span></p>
<p style="font-weight: bold;"><span style="font-size: 20px; color: #000000;">Botnet for hire?</span></p>
<p><span style="color: #000000;">It is speculated that P2Pinfect might be a botnet for hire. This is primarily due to how the new ransomware payload is being delivered from a fixed URL by command, compared to the other payloads which are baked into the main payload. This extensibility would make sense for the attacker to use in order to deploy arbitrary payloads onto botnet nodes on a whim. This suggests that P2Pinfect may accept money for deploying other attackers' payloads onto their botnet.</span></p>
<p><span style="color: #000000;">This theory is also supported by the following factors:</span></p>
<ul>
<li aria-level="1"><span style="color: #000000;">The miner wallet address is different from the ransomware wallet address, suggesting they might be separate entities.</span></li>
<li aria-level="1"><span style="color: #000000;">The built in miner uses as much CPU as it can, which often has interfered with the operation of the ransomware. It doesn’t make sense for an attacker motivated by ransomware to deploy a miner as well.</span></li>
<li aria-level="1"><span style="color: #000000;">The <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">rsagen</span> payload is not protected by any of P2Pinfect’s defensive features, such as the usermode rootkit.</span></li>
<li aria-level="1"><span style="color: #000000;">As discussed, the command to run <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">rsagen</span> is a generic download and run command, whereas the miner has its own custom command set.</span></li>
<li aria-level="1"><span style="color: #000000;"><span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">main</span> is written using tokio and packed with UPX, <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">rsagen</span> is not packed and does not use tokio.</span></li>
</ul>
<p><span style="color: #000000;">On the other hand, the following factors seem to contradict the idea that the distribution of <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">rsagen</span> could be evidence of a botnet for hire:</span></p>
<ul>
<li aria-level="1"><span style="color: #000000;">For both the main P2Pinfect binary and rsagen, the compiler string is <span style="font-family: 'Andale Mono', AndaleMono, 'Lucida Console', monospace;">GCC(4.8.5 20150623 (Red Hat 4.8.5-44))</span>. This shows that the author of P2Pinfect almost certainly compiled it, assuming that the strings have not been tampered with</span></li>
<li aria-level="1"><span style="color: #000000;">Both of the payloads are written in Rust. It’s certainly possible that a 3rd party attacker could also have chosen Rust for the project, but combined with the above point, it seems less likely.</span></li>
</ul>
<p><span style="color: #000000;">While it’s possible that P2Pinfect might be engaging in initial access brokerage, the facts of the matter seem to point to it most likely not being the case.</span></p>
<p style="font-weight: bold;"><span style="font-size: 20px; color: #000000;">Conclusion</span></p>
<p><span style="color: #000000;">P2Pinfect is still a highly ubiquitous malware, which has spread to many servers. With its latest updates to the crypto miner, ransomware payload, and rootkit elements, it demonstrates the malware author’s continued efforts into profiting off their illicit access and spreading the network further, as it continues to worm across the internet.</span></p>
<p><span style="color: #000000;">The choice of a ransomware payload for malware primarily targeting a server that stores ephemeral in-memory data is an odd one, and P2Pinfect will likely see far more profit from their miner than their ransomware due to the limited amount of low-value files it can access due to its permission level.</span></p>
<p><span style="color: #000000;">The introduction of the usermode rootkit is a “good on paper” addition to the malware - while it is effective at hiding the main binaries, a user that becomes aware of its existence can easily remove the LD preload or the binary. If the initial access is Redis, the usermode rootkit will also be completely ineffective as it can only add the preload for the Redis service account, which other users will likely not log in as.</span></p>
