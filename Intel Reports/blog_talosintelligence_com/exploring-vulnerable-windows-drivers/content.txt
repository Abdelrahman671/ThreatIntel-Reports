    <h1 class="text-center">Exploring vulnerable Windows drivers</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/vanja-svajcer/">Vanja Svajcer</a>,

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <p>This post is the result of research into the real-world application of the Bring Your Own Vulnerable Driver (BYOVD) technique along with Cisco Talos’ series of posts about &nbsp;<a href="https://malicious%20windows%20drivers./" rel="noreferrer noopener"><u>malicious Windows drivers.</u></a> Some of this research was presented at the <a href="https://aavar.org/cybersecurity-conference/" rel="noreferrer noopener"><u>AVAR conference</u></a> in Chennai at the beginning of December 2024.&nbsp;</p><p>We would like to send a special thanks to <a href="https://connormcgarr.github.io/" rel="noreferrer noopener"><u>Connor McGarr</u></a>, <a href="https://www.linkedin.com/in/russell-sanford-759b3561/" rel="noreferrer noopener"><u>Russell Sanford</u></a>, <a href="https://www.linkedin.com/in/ryan-warns-7a8415b3/" rel="noreferrer noopener"><u>Ryan Warns</u></a>, <a href="https://www.linkedin.com/in/tharrison0/" rel="noreferrer noopener"><u>Tim Harrison</u></a> and <a href="https://www.welivesecurity.com/en/our-experts/michal-poslusny/" rel="noreferrer noopener"><u>Michal Poslušný</u></a> for their previous work on analyzing vulnerabilities in drivers. &nbsp;</p><p>During our research into vulnerable Windows drivers, we investigated classes of vulnerabilities typically exploited by threat actors as well as the payloads they typically deploy post-exploitation. The attacks in which attackers are deliberately installing known vulnerable drivers only to later exploit them is a technique referred to as Bring Your Own Vulnerable Driver (BYOVD).&nbsp;</p><h1 id="how-are-threat-actors-using-byovd">How are threat actors using BYOVD?&nbsp;</h1><p>Malicious actors use these drivers to perform a myriad of actions that help them achieve their goals. In our research, we identified three major payloads used, which we describe below. &nbsp;Along with these payloads, we also identified recent activity linked to ransomware groups, which demonstrates real-world cases of malicious actors exploiting vulnerable Windows drivers to achieve their objectives.&nbsp;</p><h2 id="vulnerable-drivers-and-common-payloads">Vulnerable drivers and common payloads&nbsp;</h2><h3 id="local-escalation-of-privileges-admin-to-kernelsystem">Local escalation of privileges (admin to kernel/system)&nbsp;</h3><p>One of the most common payloads, when we consider vulnerable drivers with arbitrary kernel memory write vulnerabilities, is escalating the privileges of a malicious process. The access privileges for any process are stored in the primary access token structure, which is contained at an undocumented offset in the _EPROCESS structure, the kernel mode structure used to maintain information about each individual process by the Windows kernel. Vergilius Project contains the documentation and offsets of almost all undocumented Windows structures, including <a href="https://_eprocess/" rel="noreferrer noopener"><u>_EPROCESS</u></a>, and can be used as a reference, equally by offensive researchers and defenders. &nbsp; &nbsp;</p><p>A common strategy for escalating privileges of an unprivileged process is to find the _EPROCESS structure of a higher privileged process in kernel memory and replace the access token of the unprivileged process with the access token of the privileged process, which is relatively simple if a vulnerable drivers can be used for reading and writing kernel memory space.&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-fdf49e55-a18d-497c-8eab-4bf1e3941fc7.png" class="kg-image" alt="" loading="lazy" width="1436" height="460" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/12/data-src-image-fdf49e55-a18d-497c-8eab-4bf1e3941fc7.png 600w, below:&nbsp;</p><ol><li>Find one _EPROCESS structure/object&nbsp;</li><li>For example, load ntoskernel.exe in user mode and calculate RVA to PsInitialSystemProcess, which points to the System process NULL))) // 11 = SystemModuleInformation to find ntoskernel VA – use the vuln driver to read the offset, add the RVA to find the _EPROCESS structure in kernel memory.&nbsp;</li><li>Read the token from the known offset using the vulnerable driver read or memory copy functionality.&nbsp;</li><li>Parse _EPROCESS to find the &nbsp;ActiveProcess links member that points to a linked list of other _EPROCESSES and iterate until the low privilege process is found.&nbsp;</li><li>Overwrite the unprivileged process access token with the one previously saved from the SYSTEM process, using a vulnerable driver kernel memory write functionality.&nbsp;&nbsp;</li></ol><h3 id="loading-of-unsigned-kernel-code">Loading of unsigned kernel code&nbsp;</h3><p>Arbitrary kernel memory write There are several open-source unsigned device drivers loading utilities. The dropper drops an executable “nthandlecallback.exe”, a vulnerable Dell binary utilities driver “dbutil_2_3.sys”, and a ZIP file with the name “tree.exe”. developed for the purpose of disabling endpoint security software, and they are used by both threat actors and in red team-based exercises.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-04069ca3-7712-4b91-84be-f16c87808585.png" class="kg-image" alt="" loading="lazy" width="1600" style="white-space: pre-wrap;">Dbutil_2_3.sys is one of the drivers supported for disabling EDR tools by RealBlindingEDR</em></i></figcaption></figure><h3 id="miscellaneous-other-payloads">Miscellaneous other payloads&nbsp;</h3><p>Vulnerable drivers, mostly in the category of drivers with insufficient access controls, have been used in some advanced attacks. For example, in the Shamoon campaign, a RawDisk driver from Eldos was used to overwrite hard drives, while in February 2022, <a href="https://blog.talosintelligence.com/threat-advisory-hermeticwiper/" rel="noreferrer noopener"><u>HermeticWiper</u></a> used a proxy physical disk writing driver from&nbsp;“EaseUS Partition Master” driver&nbsp;partition manager “empntdrv.sys” for overwriting drives. HermeticWiper contained four embedded resources, which are compressed copies of drivers used by the wiper, depending on the Windows version and the default word memory size for the operating system.&nbsp;</p><p>&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-cfa3f9ca-f542-4299-9758-cebb7eb82335.png" class="kg-image" alt="" loading="lazy" width="1243" height="476" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/12/data-src-image-cfa3f9ca-f542-4299-9758-cebb7eb82335.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/12/data-src-image-cfa3f9ca-f542-4299-9758-cebb7eb82335.png 1000w, https://blog.talosintelligence.com/content/images/2024/12/data-src-image-cfa3f9ca-f542-4299-9758-cebb7eb82335.png 1243w" sizes="(min-width: 720px) 720px"><figcaption><i><em class="italic" style="white-space: pre-wrap;">Different versions of “EaseUS Partition Master” partition manager driver are embedded as resources into HermeticWiper code</em></i></figcaption></figure><p>&nbsp;</p><h2 id="ransomware-examples-of-malicious-actors-use-of-byod">Ransomware examples of malicious actors' use of BYOD&nbsp;</h2><p>With the wide availability of EDR bypassing tools exploiting vulnerable drivers, it is not a surprise that the exploitation moved from the domain of advanced threat actors into the domain of commodity threats, primarily ransomware. We document here some of the known ransomware groups employing the BYOVD technique. &nbsp;</p><h3 id="januarykasseika">January - Kasseika&nbsp;&nbsp;</h3><p>In January 2024, Kasseika ransomware operators abused a vulnerable driver, “<a href="https://www.bleepingcomputer.com/news/security/kasseika-ransomware-uses-antivirus-driver-to-kill-other-antiviruses/" rel="noreferrer noopener"><u>viragt64.sys”</u></a>, which is part of the legitimate VirIT antivirus software, to disable a pre-determined list of 991 processes related to security tools and system utilities. The ransomware-as-a-service (RaaS) operation has been active since 2023 and uses double extortion techniques but does not operate a data leak site. In recent attacks, the ransomware first executes a script to load various tools, such as a malicious executable named “Martini.exe” and the vulnerable driver that is renamed “<a href="https://www.bleepingcomputer.com/news/security/kasseika-ransomware-uses-antivirus-driver-to-kill-other-antiviruses/" rel="noreferrer noopener"><u>Martini.sys</u></a>”. Next, Kasseika will create and start a new service whereby the driver is loaded into the malicious executable. &nbsp;&nbsp;</p><p>The executable starts scanning the environment for the hard-coded list of processes and, if detected, a control code is sent to the driver enabling it to terminate processes. &nbsp;</p><h3 id="march-%E2%80%93-akira">March – Akira&nbsp;&nbsp;</h3><p>In March 2024, Akira has been observed abusing the legitimate, signed Zemana anti-malware kernel driver “<a href="https://www.bleepingcomputer.com/news/security/terminator-antivirus-killer-is-a-vulnerable-windows-driver-in-disguise/" rel="noreferrer noopener"><u>zamguard64.sys</u></a>” via PowerTool to disable EDR at the kernel level. The exploitation of the Zemana zamguard driver was a main component of the popular Terminator EDR killer tool listed for sale on illicit marketplaces beginning May 2023. &nbsp;</p><h3 id="july-%E2%80%93-qilin">July – Qilin&nbsp;&nbsp;&nbsp;</h3><p>In July 2024, the Qilin ransomware group, another group operating under a Raas model, was observed using a new malware dubbed “<a href="https://trustedsec.com/blog/technical-analysis-killer-ultra-malware-targeting-edr-products-in-ransomware-attacks" rel="noreferrer noopener"><u>Killer Ultra</u></a>” within an attack. Killer Ultra has a plethora of capabilities, including the ability to terminate security tools with a BYOVD technique, abusing a known arbitrary process termination vulnerability impacting Zemana Anti-Keylogger driver “	”, tracked as CVE-2024-1853. The vulnerability enables attackers with the ability to terminate processes. Upon execution, Killer Ultra unpacks the vulnerable driver and creates a new service to looks for and disable a list of security tools. &nbsp;&nbsp;&nbsp;</p><h3 id="july-%E2%80%93-blackbyte">July – BlackByte&nbsp;&nbsp;</h3><p>Talos recently observed and documented developments in recent BlackByte attacks in July 2024 leveraging BYOVD to facilitate host encryption. The newer encryptor variant was observed <a href="https://blog.talosintelligence.com/blackbyte-blends-tried-and-true-tradecraft-with-newly-disclosed-vulnerabilities-to-support-ongoing-attacks/" rel="noreferrer noopener"><u>dropping four vulnerable drivers</u></a> as part of BlackByte’s usual BYOVD attack chain, which is an increase from the two or three drivers described in previous reports.These drivers consisted of <a href="https://www.loldrivers.io/drivers/e32bc3da-4db1-4858-a62c-6fbe4db6afbd/" rel="noreferrer noopener"><u>RtCore64.sys</u></a>, a driver originally used by MSI Afterburner a system overclocking utility, <a href="https://www.loldrivers.io/drivers/a4eabc75-edf6-4b74-9a24-6a26187adabf/" rel="noreferrer noopener"><u>DBUtil_2_3.sys</u></a>, a driver that is part of the Dell Client firmware update utility, <a href="https://www.loldrivers.io/drivers/e5f12b82-8d07-474e-9587-8c7b3714d60c/" rel="noreferrer noopener"><u>zamguard64.sys</u></a>, a part of the previously mentioned Zemana Anti-Malware (ZAM) application exploited by other threat actors, and <a href="https://www.loldrivers.io/drivers/2bea1bca-753c-4f09-bc9f-566ab0193f4a/" rel="noreferrer noopener"><u>gdrv.sys</u></a>, a component of is the GIGABYTE tools software package for GIGABYTE motherboards.&nbsp;</p><p>These four drivers were renamed and dropped by the encryptor binary in all BlackByte attacks investigated by Cisco Talos Incident Response (Talos IR), each with a similar naming convention. The nomenclature for the vulnerable drivers consisted of eight random alphanumeric characters followed by an underscore and an iterating number value. &nbsp;</p><h3 id="augustransomhub">August - RansomHub&nbsp;&nbsp;</h3><p>In August 2024, RansomHub ransomware actors were observed using a new malware known as EDRKillShifter to disable security tools prior to executing the ransomware binary. The EDRKillShifter can act as a loader for a vulnerable legitimate driver that, once exploited, can facilitate persistent defense evasion. Recent exploits used by the adversary are related to POCs found on Github leveraging <a href="https://github.com/keowu/BadRentdrv2" rel="noreferrer noopener"><u>RentDrv2</u></a>, while the other exploited a driver called <a href="https://github.com/BlackSnufkin/BYOVD/tree/main/TfSysMon-Killer" rel="noreferrer noopener"><u>ThreatFireMonitor.</u></a> The adversary initiated the process by launching the password-protected <a href="https://symantec-enterprise-blogs.security.com/threat-intelligence/ransomhub-knight-ransomware" rel="noreferrer noopener"><u>EDRKillShifter</u></a> binary, which decrypts and executes an embedded resource in memory, unpacking and executing a payload to exploit the target vulnerable legitimate driver to escalate privileges and disable active EDR processes. &nbsp;</p><p>The malware then created and started a new service for the driver, loading it into the system. Finally, it continuously scanned for and terminated processes that match a hardcoded list of targets, for persistent defense evasion even on reboot. &nbsp;</p><p>The adoption of the BYOVD technique by RansomHub and Qilin may be linked to members of the financially motivated threat group <a href="https://thehackernews.com/2024/07/scattered-spider-adopts-ransomhub-and.html" rel="noreferrer noopener"><u>Scattered Spider</u></a> joining forces with these ransomware groups. &nbsp;The new partnership was identified and disclosed in public reporting in July 2024, but it is possible the relationship was already well established before then. Scattered Spider members are known for <a href="https://www.crowdstrike.com/blog/scattered-spider-attempts-to-avoid-detection-with-bring-your-own-vulnerable-driver-tactic/" rel="noreferrer noopener"><u>employing BYOVD tactics</u></a> since at least December 2022. &nbsp;</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-6140891b-a17d-46c8-84f4-dfdfd7a310db.jpeg" class="kg-image" alt="" loading="lazy" width="1600" height="817" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/12/data-src-image-6140891b-a17d-46c8-84f4-dfdfd7a310db.jpeg 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/12/data-src-image-6140891b-a17d-46c8-84f4-dfdfd7a310db.jpeg 1000w, https://blog.talosintelligence.com/content/images/2024/12/data-src-image-6140891b-a17d-46c8-84f4-dfdfd7a310db.jpeg 1600w" sizes="(min-width: 720px) 720px"></figure><p>&nbsp;</p><h1 id="windows-drivers-and-vulnerabilities">Windows drivers and vulnerabilities&nbsp;</h1><h3 id="creating-malicious-windows-drivers-is-increasingly-difficult">Creating malicious Windows drivers is increasingly difficult&nbsp;</h3><p>Creating a new malicious Windows kernel driver is becoming increasingly difficult. New Windows drivers must be signed with a valid extended validation (EV) certificate by the developer, pass the Microsoft Hardware Lab Kit (HLK) compatibility tests, and be signed by the Microsoft Dev Portal. &nbsp;</p><p>However, this complex process, introduced for any newly created Windows kernel or user mode driver, does not apply to existing drivers, which means that legacy drivers signed with valid certificates will still be loaded into the Windows kernel space. &nbsp;</p><p>Installing and exploiting existing legacy vulnerable drivers may be one of the very few ways to make changes to kernel data structures or execute code in kernel, as drivers have the same permissions as any other Windows kernel component.&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-86a67a5b-c4cf-4118-ab46-2fcf59751ed0.jpeg" class="kg-image" alt="" loading="lazy" width="1173" height="1600" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/12/data-src-image-86a67a5b-c4cf-4118-ab46-2fcf59751ed0.jpeg 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/12/data-src-image-86a67a5b-c4cf-4118-ab46-2fcf59751ed0.jpeg 1000w, https://blog.talosintelligence.com/content/images/2024/12/data-src-image-86a67a5b-c4cf-4118-ab46-2fcf59751ed0.jpeg 1173w" sizes="(min-width: 720px) 720px"><figcaption><i><em class="italic" style="white-space: pre-wrap;">Exploiting vulnerability in a legacy driver is the same as exploiting any kernel vulnerability</em></i></figcaption></figure><p>Microsoft introduced a blocklist of known vulnerable drivers to tackle this issue. At the beginning, the list was included into the Windows Defender Application Control feature and was superseded by the Windows Security application in newer Windows versions. &nbsp;</p><p>Although the vulnerable drivers block list is turned on by default in systems running the Windows 11 2022 update or with systems with hardware virtualization code integrity (HVCI) turned on, there are still many systems which can be attacked by deploying a vulnerable driver or any newly discovered vulnerable driver that is not already on the blocklist. &nbsp;&nbsp;</p><h2 id="common-classes-of-vulnerabilities-in-byovd-drivers">Common classes of vulnerabilities in BYOVD drivers&nbsp;</h2><p>While investigating vulnerable Windows kernel drivers commonly used by threat actors for BYOVD campaigns, we identified three classes of vulnerabilities that are typically exploited: arbitrary MSR writes, arbitrary kernel memory writes, and insufficient access controls to driver’s functionality. This classification is not strict, and one driver can belong to multiple classes of vulnerabilities. &nbsp;</p><h3 id="arbitrary-msr-readwrite-vulnerabilities">Arbitrary MSR read/write vulnerabilities&nbsp;</h3><p>To consider this class of vulnerabilities, we first need to introduce CPU model specific registers (MSRs). MSRs are additional CPU registers that are used by the CPU and the operating system for various purposes, including regulation of caching mechanism, regulation of fan speed, or transition from user mode into kernel mode. The MSRs can be addressed by their specific number, and some of them also have human readable names. &nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-3c6171a5-eaa8-44e9-a7f2-f2b6a65cdec5.jpeg" class="kg-image" alt="" loading="lazy" width="1495" height="1600" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/12/data-src-image-3c6171a5-eaa8-44e9-a7f2-f2b6a65cdec5.jpeg 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/12/data-src-image-3c6171a5-eaa8-44e9-a7f2-f2b6a65cdec5.jpeg 1000w, https://blog.talosintelligence.com/content/images/2024/12/data-src-image-3c6171a5-eaa8-44e9-a7f2-f2b6a65cdec5.jpeg 1495w" sizes="(min-width: 720px) 720px"><figcaption><i><em class="italic" style="white-space: pre-wrap;">A specific MSR is key for making transition from user to kernel modes after calling a win32 API function</em></i></figcaption></figure><p>&nbsp;As a reminder, the transition from kernel to user mode happens in the lowest user mode DLL layer, usually “ntdll.dll”, when a system call number is placed into register rax and the syscall or the “int 0x2e” instruction is executed. During the transition, the syscall instruction updates the Instruction Pointer (RIP) and sets it to the address of the system call handler in the kernel as well as the Stack Pointer (RSP) to point to a stack in kernel space.&nbsp;</p><p>The first function to run is “KiSystemCall64”, and a question one can ask is how do Windows know where to start the execution in kernel mode? The answer lies in a MSR specifically used during user to kernel mode transition. For 64-bit Windows systems, it is the IA32_LSTAR (MSR 0xC0000082), which contains the address of the kernel-mode entry point for the syscall instruction, typically the KiSystemCall64 function.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-783a6c93-6ef0-4c49-8ba9-6cef75dda49b.png" class="kg-image" alt="" loading="lazy" width="588" height="237"><figcaption><i><em class="italic" style="white-space: pre-wrap;">MSR 0xc0000082 contains the address of the first instruction to execute in kernel mode</em></i></figcaption></figure><p>By having the ability to write content into arbitrary MSRs, attackers may be able to replace the pointer to KiSystemCall64 with the pointer to a malicious function that can run code in the kernel context. &nbsp;</p><p>As an example of a driver vulnerable to arbitrary MSR modifications, we chose WinRing0 driver, which is commonly used by XMRig cryptocurrency mining software to disable some processor features such as caching, to increase the performance of the miner. WinRing0 is also included in many open and closed source programs. Unfortunately, the driver is also exposed to an arbitrary MSR write vulnerability which can lead to kernel mode code execution in versions of Windows prior to Windows 8 or to <a href="https://talosintelligence.com/vulnerability_reports/TALOS-2020-1116" rel="noreferrer noopener"><u>escalation of privileges</u></a> in later Windows versions. This method is mitigated in the latest Windows versions with the latest exploit mitigations, such as Virtualization Based Security (which will be discussed later in the post), which is enabled by default. &nbsp;&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-2622e0d5-e33e-4274-8ef5-e09789714609.png" class="kg-image" alt="" loading="lazy" width="1244" height="912" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/12/data-src-image-2622e0d5-e33e-4274-8ef5-e09789714609.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/12/data-src-image-2622e0d5-e33e-4274-8ef5-e09789714609.png 1000w, https://blog.talosintelligence.com/content/images/2024/12/data-src-image-2622e0d5-e33e-4274-8ef5-e09789714609.png 1244w" sizes="(min-width: 720px) 720px"><figcaption><i><em class="italic" style="white-space: pre-wrap;">WinRing0 driver is vulnerable to an arbitrary MSR write vulnerability</em></i></figcaption></figure><h3 id="arbitrary-kernel-physical-memory-readwrite-vulnerabilities">Arbitrary kernel physical memory read/write vulnerabilities&nbsp;</h3><p>The second class of vulnerabilities in frequently used BYOVD drivers is the arbitrary kernel memory write class. Here, a driver functionality to write arbitrary memory is used as a write primitive to deploy shellcode into kernel memory or change important kernel data structures to achieve escalation of privileges for a malicious user mode process.&nbsp;&nbsp;</p><p>A significant number of drivers with this class of vulnerability exists, and most of them are well documented. Readers are referred to the <a href="https://www.loldrivers.io/" rel="noreferrer noopener"><u>loldrivers</u></a> project to find examples of vulnerable drivers allowing kernel memory write. &nbsp;</p><p>Any driver that uses one of the following kernel functions for may be regarded as a candidate for this class of vulnerabilities, although further analysis is almost always required to conclude that a user buffer and the target address can be supplied to the driver through a user-accessible device I/O control code (IOCTL):&nbsp;</p>

<p>A good example of this vulnerability group is CVE-2022-3699, a vulnerability in a Lenovo driver that allows arbitrary memory reading and writing. &nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-f9cc3add-0d57-4a16-99e4-fa239b87eedf.png" class="kg-image" alt="" loading="lazy" width="555" height="558"><figcaption><i><em class="italic" style="white-space: pre-wrap;">CVE-2022-3699 - memory write via exposed MmMapIoSpace function in a Lenovo driver</em></i></figcaption></figure><p>&nbsp;</p><h3 id="misusing-existing-functionality-in-windows-drivers-with-insufficient-access-controls">Misusing existing functionality in Windows drivers with insufficient access controls&nbsp;</h3><p>The third and the last class of vulnerabilities used by threat actors in attacks using BYOVD drivers is misusing existing driver functionality caused by insufficient access controls.&nbsp;&nbsp;</p><p>INF files are files used during a driver’s installation, and among other things, they also contain permissions for the driver, specified using the SDDL language. <a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/driversecurity/windows-security-model" rel="noreferrer noopener"><u>The Security Descriptor Definition Language</u></a> (SDDL) is a domain specific language that allows components to generate access control lists (ACLs) using a string format. It is utilized in both user-mode and kernel-mode programming. The diagram below illustrates how SDDL strings are structured for device objects.&nbsp;</p><p>The access value specifies the type of access allowed. The SID value specifies a security identifier that determines to whom the access value applies (for example, a user or group). For example, string “D:P(A;;GA;;;SY)(A;;GR;;;WD)” allows the system (SY) access to everything and allows everyone else (WD) only read access. &nbsp;</p><p>&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-36788191-02d8-4465-bdda-d466ae5c16a5.gif" class="kg-image" alt="Diagram showing the format of SDDL strings for device objects." loading="lazy" width="526" height="182"><figcaption><i><em class="italic" style="white-space: pre-wrap;">Security Descriptor Definition Language string format manages access permissions to driver objects</em></i></figcaption></figure><p>Programming Windows kernel drivers has a steep learning curve and, as a consequence, many drivers contain code that is copied from templates and example drivers, including their SDDL access permissions. When a driver is created, it is likely that its access permissions will be inadequate and will allow unprivileged users access to functionality that should otherwise be available to users with higher privilege levels. &nbsp;</p><p>A good example of a vulnerable driver with insufficient permissions would be an old version of an antimalware software driver “viragt64.sys” (VirIT Agent System) developed by TG Soft, which exposes the functionality of terminating a process from the kernel mode to users with lower levels of privileges. This driver is used by ransomware threat actors such as <a href="https://www.trendmicro.com/en_gb/research/24/a/kasseika-ransomware-deploys-byovd-attacks-abuses-psexec-and-expl.html" rel="noreferrer noopener"><u>Kasseika</u></a> to terminate other antimalware and EDR products. &nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img used ZwTerminateProcess to terminate arbitrary process, which can be misused by threat actors due to insufficient access permissions</em></i></figcaption></figure><p>In addition to documenting different classes of vulnerabilities in frequently used BYOVD drivers, we also investigated the most common payloads delivered by threats and potentially unwanted applications after exploiting vulnerable drivers and classified them into several groups including local escalation privileges, loading of unsigned code and bypassing EDR functionality. &nbsp;&nbsp;</p><h2 id="modern-windows-mitigations-and-vulnerable-drivers">Modern Windows mitigations and vulnerable drivers&nbsp;</h2><p>Loading malicious code into kernel memory is one of the most powerful payloads attackers can use. This approach was frequently employed in the early days of Windows, prior to Windows Vista, when there were no requirements to sign drivers. The ability to load unsigned code into kernel mode was an incentive for the creation of several Windows kernel rootkits, such as Sinowal or TDL4, designed to hide the presence of malicious payloads from defenders by modifying kernel programs and data structures. &nbsp;</p><p>To respond to those threats and kernel exploitation in general, Microsoft introduced kernel patch protection (KPP), better known as Patch Guard, in x64 versions of Windows XP SP3. This was followed by the requirement for drivers to be signed in x64 Windows Vista. &nbsp;</p><p>The introduction of the mitigations into the Windows kernel sparked a race between threat actors and Microsoft. Attackers quickly responded to newly introduced mitigations by showing how digital signature enforcement can be turned off in a race with the Patch Guard, and Microsoft responded with more mitigations. Over time, the exploitation of Windows kernels became increasingly challenging. &nbsp; Next, we will briefly describe only four significant anti-exploitation features implemented with Windows 10 and 11. &nbsp;</p><h3 id="virtualization-based-security-vbs">Virtualization-Based Security (VBS)&nbsp;</h3><p>Virtual Trust Levels (VTLs) are a key concept within <a href="https://learn.microsoft.com/en-us/windows-hardware/design/device-experiences/oem-vbs" rel="noreferrer noopener"><u>Virtualization-Based Security (VBS)</u></a>, designed to enhance system security by creating isolated execution environments. VTLs leverage hardware virtualization to separate and protect sensitive processes from potentially less secure code running in the main operating system.&nbsp;</p><p>VTLs are essentially different security levels or "worlds" within the same physical machine, each providing a different level of trust. The main goal of VTLs is to isolate trusted operations and data from the rest of the system to prevent tampering. In Windows, there are two main VTL levels.&nbsp;&nbsp;</p><p>•	VTL0: This is the standard trust level, where the traditional operating system and all user-mode and kernel-mode applications run.&nbsp;&nbsp;</p><p>•	VTL1: This is a higher trust level used to execute sensitive security functions and store critical data. It is isolated from VTL0, meaning that operations in VTL0 cannot directly access or modify the code and data in VTL1. VTL1 is used to store sensitive information like encryption keys, password hashes, and security tokens (credentials guard).&nbsp;&nbsp;</p><p>&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/12/data-src-image-deafd43e-9554-4140-8583-e829ca367cdf.jpeg" class="kg-image" alt="" loading="lazy" width="400" height="382"><figcaption><i><em class="italic" style="white-space: pre-wrap;">High level architecture of Virtualization-based security concepts, credit: Windows Internals 7</em></i><i><sup style="white-space: pre-wrap;"><em class="italic">th</em></sup></i><i><em class="italic" style="white-space: pre-wrap;"> edition, part 1</em></i></figcaption></figure><p>&nbsp;By running different parts of the kernel in different trust levels, effectively different virtual machines, Windows can use <a href="https://en.wikipedia.org/wiki/Second_Level_Address_Translation" rel="noreferrer noopener"><u>Second Level Address Translation</u></a> (SLAT) to create different access permissions for memory pages depending on the source of access.&nbsp;&nbsp;</p><p>Essentially, in a process similar to shadowing page tables, VBS enforces exclusive write or execute page access permission. In other words, if a code from VTL0 attempts to change its own page table permissions from writable to executable this will be detected by the VTL1 and the data in the page still won’t be able to execute.&nbsp;&nbsp;</p><p>This mechanism is one of the key features of another important mitigation, Hypervisor-Protected Code Integrity (HVCI).&nbsp;</p><h3 id="hypervisor-protected-code-integrity-hvci">Hypervisor-Protected Code Integrity (HVCI)&nbsp;</h3><p>When <a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/bringup/device-guard-and-credential-guard" rel="noreferrer noopener"><u>Hypervisor-protected Code Integrity (HVCI)</u></a> is enabled on a Windows system, it enforces control over memory page permissions to mitigate executable code injection. HVCI is designed so that only verified and trusted code is executed in kernel mode, and it applies policies to manage how memory pages can be used and modified.&nbsp;</p><p>One of the important features enforced by HVCI (and supported by modern CPUs) is the prevention of pages being simultaneously writable and executable. This policy is known as Write XOR Execute (W^X), which prevents memory pages from being both writable and executable at the same time.&nbsp;&nbsp;</p><p>HVCI prevents direct execution of code from pages that were recently writable, unless specific security checks are passed. Before any code can execute from a page that has had its permissions altered, it must pass a code integrity check, ensuring it is signed by a trusted certificate. If the code does not meet these integrity requirements, execution will be blocked. HVCI attempts to ensure that any code running in kernel mode is signed with a valid certificate.&nbsp;&nbsp;</p><h3 id="kernel-control-flow-guard-kcfg">Kernel Control Flow Guard (kCFG)&nbsp;</h3><p>Kernel Control Flow Guard (kCFG) is a security feature in Windows designed to protect the operating system's kernel from certain types of attacks that attempt to manipulate the control flow of kernel-mode code. It builds on the principles of Control Flow Guard (CFG), used to secure user-mode applications.&nbsp;</p><p>kCFG aims to prevent exploits that involve redirecting the control flow of kernel code to unintended or malicious locations which should prevent exploits that hijack the control flow by overwriting function pointers and other data used for indirect code execution.&nbsp;&nbsp;</p><p>During the compilation of the Windows kernel, kCFG instruments the code to create valid address bitmap and any indirect call must finish at a target known at compile time. If the call is directed outside know target the system will cause a security check failure.&nbsp;&nbsp;&nbsp;</p><h3 id="kernel-shadow-stack">Kernel shadow stack&nbsp;</h3><p>The primary purpose of the Windows kernel shadow stack is to ensure that the return addresses on the call stack cannot be tampered with, specifically to mitigate exploitation using Return Oriented Programming (ROP).&nbsp;</p><p>The shadow stack maintains a separate, copy of return addresses parallel to the regular call stack. When a function call occurs, the return address is pushed onto both the regular stack and the shadow stack. Upon function return, the system verifies the return address against the shadow stack to ensure it has not been altered. The shadow stack in Windows is hardware assisted for better performance through Intel Control-Flow Enforcement Technology (CET) and AMD Shadow Stacks.&nbsp;&nbsp;</p><h1 id="conclusion">Conclusion&nbsp;</h1><p>In recent years, Windows platform security has improved to effectively prevent deployment of newly developed malicious drivers. However, kernel mode threats of vulnerable legacy drivers remain a concern. Luckily there are a few things we can do to mitigate the risks and detect potential campaigns using BYOVD technique.&nbsp;</p><p>This could include enforcement of Extended Validation (EV) and Windows Hardware Quality Labs (WHQL) certified drivers, preventing risks associated with legacy drivers. If the blocking of all legacy drivers is not possible, employing the Windows Defender Application Control (Windows Security) drivers blocklist is recommended way to prevent the execution of known vulnerable drivers.&nbsp;</p><p>Apart from the above, for threat detection and response, it recommended to develop a capability to monitor driver load events, such as those recorded by <a href="https://learn.microsoft.com/en-us/sysinternals/downloads/sysmon" rel="noreferrer noopener"><u>Sysmon’s event ID 6</u></a>.&nbsp;&nbsp;</p><p>In summary, while Windows security has improved, maintaining vigilance against kernel mode threats requires adoption of best practices and monitoring techniques to protect against known and unknown driver vulnerabilities.&nbsp;&nbsp;</p><h1 id="references-and-further-reading">References and further reading&nbsp;</h1><h4 id="posts-and-papers"><em>Posts and papers&nbsp;</em></h4><ol><li><a href="https://blog.talosintelligence.com/exploring-malicious-windows-drivers-part-1-introduction-to-the-kernel-and-drivers/" rel="noreferrer noopener"><u>Exploring Malicious Drivers Part 1</u></a> - Cisco Talos&nbsp;</li><li><a href="https://blog.talosintelligence.com/exploring-malicious-windows-drivers-part-2/" rel="noreferrer noopener"><u>Exploring Malicious Drivers Part 2</u></a> - Cisco Talos&nbsp;</li><li><a href="https://www.crowdstrike.com/en-us/blog/state-of-exploit-development-part-1/" rel="noreferrer noopener"><u>The Current State of Exploit Development, Part 1</u></a> – Connor McGarr, Crowdstrike&nbsp;</li><li><a href="https://www.crowdstrike.com/en-us/blog/state-of-exploit-development-part-2/" rel="noreferrer noopener"><u>The Current State of Exploit Development, Part 2</u></a> – Connor McGarr, Crowdstrike&nbsp;</li><li><a href="https://connormcgarr.github.io/hvci/" rel="noreferrer noopener"><u>No Code Execution?
