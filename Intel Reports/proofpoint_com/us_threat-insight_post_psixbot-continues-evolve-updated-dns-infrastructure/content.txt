<h1 class="blog-banner__heading">
      <span>PsiXBot Continues to Evolve with Updated DNS Infrastructure </span>

  </h1>

    </div>
  </div>

  <div class="blog-content">
    <div class="blog-content__sharethis sharethis_toolbox sharethis_32x32_style">
      <div class="blog-content__sharethis_label sharethis__label">Share with your network!</div>
      <div class="blog-content__sharethis_buttons sharethis_buttons">
        <div class="sharethis-inline-share-buttons"></div>
        
<span class="addthis_button_subscribe at300b UNCONVERTED" title=Subscribe>
  <span class="at-icon-wrapper block-subscribe-button__trigger block-subscribe-button__addthis"></span>
</span>


      </div>
    </div>
        <div class="blog-content__metadata blog-content__metadata-author">
      <span class="blog-content__date">  <time datetime="2019-08-13T13:00:00Z">August 13, 2019</time>

</span>
        <span class="blog-content__author">
                Proofpoint Threat Insight Team
  
        </span>
    </div>
    <div class="node-full__body blog-content__body">  <p><strong>Overview</strong></p>

<p>Earlier this year, FoxIT published research regarding the evolution of a .NET based malware known as “PsiXBot.”[1] We have continued to observe the malware in both <a href="https://www.proofpoint.com/us/threat-insight/post/sload-and-ramnit-pairing-sustained-campaigns-against-uk-and-italy">malicious email</a> and exploit kit campaigns.</p>

<p>Since the publication of this initial research, Proofpoint researchers have observed another evolution of PsiXbot (v1.0.2) which exhibits some key differences, including a new and unique method of dynamically fetching its own DNS infrastructure by utilizing a URL shortening service to gather the server IP addresses required to resolve the .bit domains used for command and control (C&amp;C).</p>

<p><img alt="Newly observed Global Variable Version - 1.0.2" src="/sites/default/files/psixpicture1.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 1: Newly Observed Global Variable Version - 1.0.2</em></p>

<p><strong>Analysis</strong></p>

<p>Proofpoint researchers have observed a new version of PsiXBot in the wild. It has been historically delivered in both malicious spam campaigns and as a payload for the Spleevo and RIG-v exploit-kits with indiscriminate geographical targeting.</p>

<p>Analysts noted that in this version, the malware continued to check the infected machine’s installed language to determine if it is in Russia (RU) or not. If it is found to be RU, the PsiXbot will exit.</p>

<p>Historically, PsiXbot has made use of .bit domain addresses which are associated with the NameCoin cryptocurrency[2]. The .bit domain is not resolved in the same way as a more common TLDs, such as “.com” or “.net”. Rather, it requires a special DNS server to provide resolution from domain to IP address.[3] In previous versions of PsiXbot, an OpenNIC DNS server was hardcoded in the binary. Now, the authors are utilizing a URL created with the URL shortening service “tiny[.]cc” to gather the current DNS server for each C&amp;C domain:</p>

<p><img alt="Initial GET request for hex-encoded domain name using tiny[.]cc shortener service" src="/sites/default/files/psixpicture2.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 2: The initial GET request for a hex-encoded domain name using tiny[.]cc shortener service</em></p>

<p>This shortened URL is a simple hex stream, which, upon decoding, provides a C&amp;C server domain. This is also hardcoded in the executable. After this initial HTTP request to tiny[.]cc, the connection is upgraded to HTTPS. Once the connection is upgraded to HTTPS, it performs this same request, only this time an “HTTP 303 See Other” response is given.</p>

<p>With this request, the returned “Location” header contains another hex-encoded domain, similar to what we observed in the URL shortener request. This is a bit different: when the hex-encoded domain provided in the Location header is decoded, it is not a domain but rather an IP address. So far, we have observed two specific attackers provided IP addresses returned as hex-encoded domains.</p>

<p><img alt="GET request containing hex-encoded C&amp;C domain (adm4[.]bit) followed by HTTP 303" src="/sites/default/files/psixpicture3.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 3: The GET request containing a hex-encoded C&amp;C domain (adm4[.]bit) followed by an HTTP 303 See Other response revealing the hex-encoded DNS server IP address as a domain.</em></p>

<p><img alt="GET request containing hex-encoded C&amp;C domain (adm2[.]bit) followed by HTTP 303" src="/sites/default/files/psixpicture4.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 4: The GET request containing a hex-encoded C&amp;C domain (adm2[.]bit) followed by an HTTP 303 See Other response revealing another hex-encoded DNS server IP Address as a domain.</em></p>

<p>Upon learning of the intended DNS server to be used for C&amp;C redirection, the malware will ping the IP address gathered from the hex-encoded domain in the Location header. This code for gathering the C&amp;C and IP address can be observed in the code snippet below.</p>

<p><img alt="Code snippet for gathering DNS Server address" src="/sites/default/files/psixpicture6.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 5: Code snippet for gathering the DNS Server address</em></p>

<p>If the malware receives a response back, indicating the DNS server is up, it will then send a DNS query for the C&amp;C domain using the gathered DNS server. The following annotated code shows this process.</p>

<p><img alt="Annotated code walking through process of retrieving C&amp;C IP Address via attacker’s DNS server" src="/sites/default/files/psixpicture7.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 6: Annotated code walking through the process of retrieving the C&amp;C IP Address via the attacker’s DNS server.</em></p>

<p>From the network level (Figure 8), the process is as follows:</p>

<p><img alt="Annotated network traffic showing process of gathering DNS server to C&amp;C traffic" src="/sites/default/files/psixpicture8.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 7: Annotated network traffic showing the process of gathering the DNS server to C&amp;C traffic.</em></p>

<ol>
	<li>Unintended GET request to the hex-encoded IP Address of the DNS server to be used (185.228.234.204)</li>
	<li>Ping to the DNS server IP Address to obtain connectivity status (185.228.234.204)</li>
	<li>DNS query to the .bit domain (hardcoded in the sample) which returns the C&amp;C IP Address (185.159.129.37)</li>
	<li>Ping to the C&amp;C IP Address for connectivity status (185.159.129.37)</li>
	<li>HTTPS traffic to the C&amp;C domain (185.159.129.37)</li>
</ol>

<p>An example of the HTTPS-based command and control traffic appears below:</p>

<p><img alt="Initial C&amp;C request containing system information" src="/sites/default/files/psixpicture9.png" style="border-width: 0px; border-style: solid;"></p>

<p><em>Figure 8: A look at the initial C&amp;C request containing system information</em></p>

<p>The data in the POST body is encrypted with RC4. In this case, the hardcoded RC4 key was “63a6a2eea47f74b9d25d50879214997a”, which is the same key used in previous versions of PsiXBot. Using this key, we can decrypt the C&amp;C traffic to show particular checks in an infected system’s information:</p>

<p><strong>action=call&amp;user_name=test&amp;bot_id=B4DCF733C9C43D10C80120CF7760B564&amp;av=N&amp;os_major=Microsoft Windows 7 Ultimate &amp;permissions=Admin&amp;os_bit=64&amp;cpu=Intel(R) Core(TM) i3-2100 CPU&amp;gpu=NVIDIA GeForce 8800 Ultra 768&amp;version=1.0.2&amp;user_group=Admin</strong></p>

<p>The bot_id in this sample is created by taking the MD5 hash of the following system details in this order:</p>

<ul>
	<li>CPU</li>
	<li>User Name</li>
	<li>GPU</li>
	<li>Machine Name</li>
	<li>OS Version</li>
	<li>User Domain Name</li>
</ul>

<p>Based on previous analysis, it appears this version of PsiXBot chose not to include some system information it previously gathered, such as .NET version and HDD information. Upon successful C&amp;C check in, the server will reply with a JSON blob containing a “result_code”:</p>

<p><strong>{result_code:[{"result_code":"200"}]}</strong></p>

<p><strong>action=command&amp;bot_id=B4DCF733C9C43D10C80120CF7760B564</strong></p>

<p><strong>New Module Analysis</strong></p>

<p><strong><em>SelfDelete</em></strong></p>

<p>The “SelfDelete” module runs a command using the <strong>cmd [.] exe</strong> shell in a hidden window to delete the running bot process and remove it from the infected system.</p>

<p><strong><em>StartCryptoModule</em></strong></p>

<p>The “StartCryptoModule”, assembly name <strong>“LESHI”</strong>, has a new module name (possibly to account for the various cryptocurrencies included), but appears to have the same functionality as the previously analyzed module. This module will monitor the clipboard for text matching a Bitcoin, Etherium, Monero, Ripple, or Litecoin wallet address, and if found, replace it with a self-configured wallet address.</p>

<p><strong><em>StartFGModule</em></strong></p>

<p>The “StartFGModule”, assembly name <strong>“omg228”</strong>, is a newly implemented “form grabbing” module. It appears rudimentary, as it is not targeting any traffic or domains specifically, such as banking or financial websites. This module will store GET or POST requests in a log file titled “temp.log” stored in the User’s %TEMP% directory. This log is subsequently sent to the C&amp;C server.</p>

<p><strong><em>StartSpam</em></strong></p>

<p>The “StartSpam” module, assembly name <strong>“Spam”</strong>, is another newly implemented module which has the ability to send outbound email using Microsoft Outlook to send messages with varying content, crafted by the attacker based on command line switches provided:</p>

<ul>
	<li>Subject</li>
	<li>Body</li>
	<li>Name</li>
	<li>Attachment</li>
</ul>

<p>The StartSpam module is configured to delete any outbound messages after sending. In addition to this, it will harvest any saved Outlook email signatures to be used inside any messages sent by this module.</p>

<p><strong>Conclusion</strong></p>

<p>PsiXBot continues to evolve with new ways of evading detection and new features to steal information. The .bit domains remain in use; however, it utilizes a new technique to retrieve the DNS servers required to connect. PsiXBot continues to operate as a bot with various stealer actions, but this new version expands its list of modules, which pushes the bot’s capabilities in new directions. Since PsiXBot first emerged in 2017, it has undergone several changes, making it a competent and relevant stealer. The changes observed here demonstrate that the author or group behind this malware is committed to evolving this malware to compete in the threat landscape.</p>
