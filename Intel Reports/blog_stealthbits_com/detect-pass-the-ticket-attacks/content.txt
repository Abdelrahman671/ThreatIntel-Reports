<p><a href="https://blog.Netwrix.com/2021/11/30/how-to-detect-pass-the-hash-attacks/">In our first post</a> of the series, we looked at ways to detect <a href="https://blog.netwrix.com/2021/11/30/how-to-detect-pass-the-hash-attacks/" data-internallinksmanager029f6b8e52c="434" title="https://blog.netwrix.com/2021/11/30/how-to-detect-pass-the-hash-attacks/">pass-the-hash attacks</a>, which exploit NTLM authentication within <a href="https://blog.netwrix.com/2017/01/31/active-directory-domain/" data-internallinksmanager029f6b8e52c="45" title="https://blog.netwrix.com/2017/01/31/active-directory-domain/">an Active Directory domain</a>. Pass-the-ticket is a related attack that which leverages Kerberos authentication to perform lateral movement.</p>

<p>In this post, we will dive into how the pass-the-ticket attack works and what you can do to detect it.</p>



<h2 class="wp-block-heading"><span id="How_Pass-the-Ticket_Works">How Pass-the-Ticket Works</span></h2>



<p>In a pass-the-ticket attack, an attacker extracts a Kerberos Ticket Granting Ticket (TGT) from LSASS memory on a system and then uses this valid ticket on another system to request Kerberos service tickets (TGS) to gain access to network resources.</p>



<p>One primary difference between&nbsp;<a href="https://blog.Netwrix.com/2021/11/30/how-to-detect-pass-the-hash-attacks/">pass-the-hash</a>&nbsp;and <a href="https://www.Netwrix.com/pass_the_ticket.html?cID=70170000000kgEZ" target="_blank">pass-the-ticket</a> is that Kerberos TGT tickets expire (10 hours by default), whereas NTLM hashes change only when the user changes their password.&nbsp;So a TGT ticket must be used within its lifetime, or it can be renewed for a longer period of time (7 days).</p>



<p><a href="https://github.com/gentilkiwi/mimikatz" class="external" rel="nofollow" target="_blank">Mimikatz</a>&nbsp;can be used to perform pass-the-ticket, but in this post, we wanted to show how to execute the attack using another tool,&nbsp;<a href="https://github.com/GhostPack/Rubeus" class="external" rel="nofollow" target="_blank">Rubeus</a>, lets you perform Kerberos based attacks.&nbsp;Rubeus is a C# toolset written by&nbsp;<a href="https://twitter.com/harmj0y" class="external" rel="nofollow" target="_blank">harmj0y</a> and is based on the&nbsp;<a href="https://github.com/gentilkiwi/kekeo/" class="external" rel="nofollow" target="_blank">Kekeo</a>&nbsp;project by Benjamin Delpy, the author of <a href="https://blog.netwrix.com/2021/11/30/passing-the-hash-with-mimikatz/" data-internallinksmanager029f6b8e52c="437" title="https://blog.netwrix.com/2021/11/30/passing-the-hash-with-mimikatz/">Mimikatz</a>.</p>



<h2 class="wp-block-heading"><span id="Step_1_Extract_the_TGT">Step 1. Extract the TGT.</span></h2>



<p>To perform a pass-the-ticket attack with Rubeus, the first step is to obtain a TGT.&nbsp;TGTs and NTLM hashes may or may not be stored on a system after a user logs off, based on security settings.&nbsp;One of the fun/scary features of Rubeus is Monitor, which will look for 4624 logon events and dump the TGT data for any new logon sessions on a system.</p>



<p>If we use the following command, Rubeus will start monitoring for logon sessions every 30 seconds:</p>



<p>Rubeus.exe monitor /interval:30</p>

<h2 class="wp-block-heading"><span id="Detection">Detection</span></h2>



<p>You can detect pass-the-ticket at the endpoint or on your domain controllers.</p>



<h3 class="wp-block-heading"><span id="Detecting_Pass-the-Ticket_at_the_Endpoint">Detecting Pass-the-Ticket at the Endpoint</span></h3>



<p>In researching detection of pass-the-ticket, we came across a very interesting&nbsp;<a href="https://jblog.javelin-networks.com/blog/detecting-pass-ticket-pass-hash-attack-using-simple-wmi-commands/" class="external" rel="nofollow" target="_blank">approach posted by a researcher Eyal Neemany at Javelin Networks</a>.&nbsp;It advises that when you want to investigate for pass-the-ticket activity, you can take the following steps for any endpoint in your environment:</p>



<ol class="wp-block-list">
<li>Look at the current logon sessions on that system.</li>



<li>Use the klist command to inspect the Kerberos tickets associated with a session.</li>



<li>Look for Kerberos tickets that do not match the user associated with the session, which would mean they were injected into memory and a pass-the-ticket attack is afoot.</li>
</ol>



<p>Let&rsquo;s take a deeper dive into these steps.</p>



<p><strong>Step 1. </strong>To output all of the logon sessions, we can use this script adapted from the Get-<a href="https://github.com/tmmtsmith/Powershell/blob/master/Get-LoggedOnUsers.ps1" class="external" rel="nofollow" target="_blank">LoggedOnUsers</a> function on GitHub:</p>



<pre class="wp-block-preformatted">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; $regexa = '.+Domain="(.+)",Name="(.+)"</pre>



<p><strong>Step 2. </strong>Now we can use the&nbsp;<em>klist &ndash;li</em>&nbsp;command and pass in a session ID to see the tickets associated with that session.</p>



<p><strong>Step 3. </strong>We inspected a session for the user Michael, but we see a Kerberos TGT for the user Gene.&nbsp;Pass-the-ticket detected!</p>

<h4 class="wp-block-heading">Finding Events that Indicate Pass-the-Ticket Attacks</h4>



<p>So what&rsquo;s different in the event logs when there&rsquo;s pass-the-ticket activity? What should look for?&nbsp;Well it&rsquo;s likely that the attacker will harvest TGTs and then use them on a different system, so you can look for TGS requests or TGT renewals using a particular Account/Client pair that have no associated TGT request from that Account/Client pair.&nbsp;You would have to look at a TGS request or TGT renewal and then scan back the previous 10 hours to see if there was a TGT request that matches that user and computer.&nbsp;That is because in pass-the-ticket the attacker will never request a TGT; they will always steal it from LSASS.&nbsp;They may renew it, and they definitely may use it to request TGS service tickets.</p>



<p>Now, that detection goes above and beyond event log filtering, and doing it at scale likely requires a <a href="https://blog.netwrix.com/2021/06/03/siem-security-information-and-event-management/" data-internallinksmanager029f6b8e52c="17" title="https://blog.netwrix.com/2021/06/03/siem-security-information-and-event-management/">SIEM</a> or third-party product. If you&rsquo;re looking for a way to detect this, check out <a rel="noreferrer noopener" href="https://www.Netwrix.com/threat_detection_software.html?cID=70170000000kgEZ" data-type="link" data-id="https://www.Netwrix.com/threat_detection_software.html?cID=70170000000kgEZ" target="_blank">Netwrix Threat Manager</a> and see how it can help with this and other Active Directory attacks, such as <a rel="noopener" href="https://www.Netwrix.com/how_golden_ticket_attack_works.html?cID=70170000000kgEZ" target="_blank">Golden Ticket</a>, <a rel="noopener" href="https://www.Netwrix.com/pass_the_hash_attack_explained.html?cID=70170000000kgEZ" target="_blank">Pass the Hash</a> and <a rel="noopener" href="https://www.Netwrix.com/cracking_kerberos_tgs_tickets_using_kerberoasting.html?cID=70170000000kgEZ" target="_blank">Kerberoasting</a>.</p>

<h4 class="wp-block-heading">Finding Events that Indicate Pass-the-Ticket Attacks</h4>



<p>So what&rsquo;s different in the event logs when there&rsquo;s pass-the-ticket activity? What should look for?&nbsp;Well it&rsquo;s likely that the attacker will harvest TGTs and then use them on a different system, so you can look for TGS requests or TGT renewals using a particular Account/Client pair that have no associated TGT request from that Account/Client pair.&nbsp;You would have to look at a TGS request or TGT renewal and then scan back the previous 10 hours to see if there was a TGT request that matches that user and computer.&nbsp;That is because in pass-the-ticket the attacker will never request a TGT; they will always steal it from LSASS.&nbsp;They may renew it, and they definitely may use it to request TGS service tickets.</p>



<p>Now, that detection goes above and beyond event log filtering, and doing it at scale likely requires a SIEM or third-party product. If you&rsquo;re looking for a way to detect this, check out <a rel="noreferrer noopener" href="https://www.Netwrix.com/threat_detection_software.html?cID=70170000000kgEZ" data-type="link" data-id="https://www.Netwrix.com/threat_detection_software.html?cID=70170000000kgEZ" target="_blank">Netwrix Threat Manager</a> and see how it can help with this and other Active Directory attacks, such as <a rel="noopener" href="https://www.Netwrix.com/how_golden_ticket_attack_works.html?cID=70170000000kgEZ" target="_blank">Golden Ticket</a>, <a rel="noopener" href="https://www.Netwrix.com/pass_the_hash_attack_explained.html?cID=70170000000kgEZ" target="_blank">Pass the Hash</a> and <a rel="noopener" href="https://www.Netwrix.com/cracking_kerberos_tgs_tickets_using_kerberoasting.html?cID=70170000000kgEZ" target="_blank">Kerberoasting</a>.</p>

<h3 class="wp-block-heading"><span id="Detecting_Pass-the-Ticket_on_Domain_Controllers2">Detecting Pass-the-Ticket on Domain Controllers</span></h3>



<p>There is also a way to look for pass-the-ticket behavior on your domain controllers. It may not be quite as reliable, but it&rsquo;s always good to have a detection you can get from your DC logs.</p>



<h4 class="wp-block-heading">Event Logs for Legitimate Kerberos Authentication</h4>



<p>To understand what to look for, let&rsquo;s review the event logs we would see for normal Kerberos authentication on the network.</p>



<p><strong>4768 &ndash; A Kerberos authentication ticket (TGT) was requested</strong></p>



<p>The first event you should see is a&nbsp;<a href="https://www.ultimatewindowssecurity.com/securitylog/encyclopedia/event.aspx?eventID=4768" class="external" rel="nofollow" target="_blank">4768</a>&nbsp;event.&nbsp;This is the TGT request and is the first thing that must happen for a user to leverage Kerberos to access a network resource.&nbsp;You will get one of these for each user for every endpoint they access your domain from. If a user account logs in from two separate workstations, they will request a TGT from each.</p>



<p>The most relevant information in this event is the user who requested the TGT and the computer they requested it from:</p>

<h4 class="wp-block-heading">Finding Events that Indicate Pass-the-Ticket Attacks</h4>



<p>So what&rsquo;s different in the event logs when there&rsquo;s pass-the-ticket activity? What should look for?&nbsp;Well it&rsquo;s likely that the attacker will harvest TGTs and then use them on a different system, so you can look for TGS requests or TGT renewals using a particular Account/Client pair that have no associated TGT request from that Account/Client pair.&nbsp;You would have to look at a TGS request or TGT renewal and then scan back the previous 10 hours to see if there was a TGT request that matches that user and computer.&nbsp;That is because in pass-the-ticket the attacker will never request a TGT; they will always steal it from LSASS.&nbsp;They may renew it, and they definitely may use it to request TGS service tickets.</p>



<p>Now, that detection goes above and beyond event log filtering, and doing it at scale likely requires a SIEM or third-party product. If you&rsquo;re looking for a way to detect this, check out <a rel="noreferrer noopener" href="https://www.Netwrix.com/threat_detection_software.html?cID=70170000000kgEZ" data-type="link" data-id="https://www.Netwrix.com/threat_detection_software.html?cID=70170000000kgEZ" target="_blank">Netwrix Threat Manager</a> and see how it can help with this and other Active Directory attacks, such as <a rel="noopener" href="https://www.Netwrix.com/how_golden_ticket_attack_works.html?cID=70170000000kgEZ" target="_blank">Golden Ticket</a>, <a rel="noopener" href="https://www.Netwrix.com/pass_the_hash_attack_explained.html?cID=70170000000kgEZ" target="_blank">Pass the Hash</a> and <a rel="noopener" href="https://www.Netwrix.com/cracking_kerberos_tgs_tickets_using_kerberoasting.html?cID=70170000000kgEZ" target="_blank">Kerberoasting</a>.</p>
