                                <p><i class="updated-at">Last updated at Fri, 16 Aug 2024 18:36:18 GMT</i></p>
                            <!--kg-card-begin: markdown--><h2 id="executivesummary">Executive Summary</h2>
<p>On June 20, 2024, Rapid7 identified multiple intrusion attempts by threat actors utilizing techniques, tactics, and procedures (TTPs) that are consistent with an ongoing social engineering campaign being tracked by Rapid7. Rapid7 observed a meaningful shift in the tools used by the threat actors during the investigations of these recent incidents. For more information about the social engineering strategies and tools that have been used, <a href="https://www.rapid7.com/blog/post/2024/05/10/ongoing-social-engineering-campaign-linked-to-black-basta-ransomware-operators/">please refer to the previous blog.</a></p>
<!--kg-card-end: markdown--><!--kg-card-begin: markdown--><h2 id="overview">Overview</h2>
<p>The initial lure being utilized by the threat actors remains the same: an email bomb followed by an attempt to call impacted users and offer a fake solution. In the recent cases handled by Rapid7, external calls were typically made to the impacted users via Microsoft Teams. Once on the phone, the threat actor would convince the user to download and install AnyDesk, a popular remote access tool that allows the threat actor to take control of the user’s computer. Threat actors typically use this connection to upload and execute payloads on the user’s system as well as to exfiltrate stolen data, during the initial stages of the attack. Rapid7 did not observe attempts to use Microsoft’s Quick Assist in recent cases, a feature that previously facilitated numerous intrusions in cases handled by Rapid7.</p>
<p>Where threat actors previously ran a credential harvesting batch script, which typically utilized several native Windows binaries, Rapid7 has now observed the usage of the 32-bit .NET executable <code>AntiSpam.exe</code>.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.09.21-PM.png" class="kg-image"><figcaption><em>Figure 1. Fake update console window spawned by "AntiSpam.exe".</em></figcaption></figure><!--kg-card-begin: markdown--><p>During execution, <code>AntiSpam.exe</code> will pretend to download email spam filters and then prompt the user to enter their credentials into a pop-up window.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.12.26-PM.png" class="kg-image" alt="*[Figure 2. Credential harvester prompt spawned by `AntiSpam.exe`.](figure_prompt.png)*"><figcaption><em>Figure 2. Credential harvester prompt spawned by "AntiSpam.exe".</em></figcaption></figure><!--kg-card-begin: markdown--><p>The executable is presented as a spam filter updater for consistency with the initial social engineering lure, where the threat actor has already spammed the user with benign emails. The credentials entered into the prompt are saved to disk after validation along with system enumeration information. If incorrect credentials are entered by the user they are also logged to disk, but the user will be prompted to try again until they are successful.</p>
<p>Following the execution of the credential harvester, threat actors also executed a series of binaries and PowerShell scripts to attempt to establish a connection with the threat actor’s command and control (C2) servers. Rapid7 has observed follow-on payloads with the following names, which all stay consistent with the social engineering lure to avoid suspicion:</p>

<p>These payloads include: SystemBC malware, which acts as a dropper and socks proxy; Golang HTTP beacons, which seem to serve as a C2 framework; Socks proxy beacons, which can route connections; and a Beacon Object File (BOF), that was converted from a Cobalt Strike module to a standalone executable.</p>
<p>Of special note, once executed, the payload <code>update6.exe</code> will attempt to exploit <code>CVE-2022-26923</code> to add a machine account (functionality that is publicly available as a BOF), which can then be used by a threat actor for <a href="https://attack.mitre.org/techniques/T1558/003/">Kerberoasting</a>. This technique was observed in multiple cases handled by Rapid7, where it was used as a means of privilege escalation if there were vulnerable domain controllers within the impacted environment.</p>
<p>In addition to these secondary payloads, Rapid7 has observed usage of reverse SSH tunnels and the Level Remote Monitoring and Management (RMM) tool to facilitate lateral movement and retain access within compromised environments.</p>
<!--kg-card-end: markdown--><!--kg-card-begin: markdown--><h2 id="technicalanalysis">Technical analysis</h2>
<p>To gain a better understanding of the actions performed by the secondary payloads, we will analyze several of the payloads in more depth. Rapid7 determined that many of the compiled payloads recently observed have been signed with the same certificate, which is associated with the thumbprint <code>B55DAD8DA97FA6AF0272102ED0E55E76E753FD04</code>.</p>
<p>A second signature was identified that corresponds to a second batch of similar payloads (~8 files), though subsequent analysis will be limited to those signed with the thumbprint previously mentioned (~11 files). More information on these signatures is available in the <em>Indicators of Compromise</em> section.</p>
<!--kg-card-end: markdown--><!--kg-card-begin: markdown--><h3 id="antispamexe">AntiSpam.exe</h3>
<p>Although the credential harvester has been re-written as a .NET application, <code>AntiSpam.exe</code> , originally named <code>update3.exe</code>, still allocates a console window via <code>AllocConsole</code> to display messages to the user. The “filter updates” that the program pretends to download are the result of a simple loop that prints the same message 1023 times to the console window, with only the iteration number changing.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.14.09-PM.png" class="kg-image"><figcaption><em>Figure 3. Fake filters download loop within "AntiSpam.exe".</em></figcaption></figure><!--kg-card-begin: markdown--><p>Once the fake loop is completed, the program initializes a window that prompts the user to enter their credentials to complete the update. The username field in the window is automatically populated with the current user’s domain and username. When the user enters their password, their credentials are validated using the <a href="https://learn.microsoft.com/en-us/dotnet/api/system.directoryservices.accountmanagement.principalcontext.validatecredentials?view=net-8.0">ValidateCredentials</a> method.</p>
<p>The first time the user enters their credentials, <code>AntiSpam.exe</code> will spawn three enumeration commands via <code>cmd.exe</code>: <code>systeminfo</code>, <code>route print</code>, and <code>ipconfig /all</code>. The output of these commands is saved to a file at <code>%TEMP%\qwertyuio.txt</code>, a filename that results from simply running your finger across the top alphabetical row on a QWERTY keyboard.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.15.13-PM.png" class="kg-image"><figcaption><em>Figure 4. Enumeration commands spawned from "AntiSpam.exe" after the first set of credentials are entered.</em></figcaption></figure><!--kg-card-begin: markdown--><p>As seen above, there are several versions of <code>AntiSpam.exe</code> that contain functionality to write a list of usernames to <code>%TEMP%\qwertyuio.txt</code>, though a complete implementation was only observed in the most recent version (other versions only write a single new-line as the array is empty). The full implementation uses a WMI query for <code>Win32_UserAccount</code> to populate a username list. The changes across versions of the malware also indicate that development of the tool was actively in progress at the time of distribution. In what appears to be the most complete version, the username of the current user is also displayed without their domain, a format that is likely more familiar to targets of the social engineering (and thus, less suspicious).</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.16.00-PM.png" class="kg-image"><figcaption><em>Figure 5. A list of generic usernames stored within "AntiSpam.exe" that are not logged.</em></figcaption></figure><!--kg-card-begin: markdown--><p>If the user enters the wrong password, the oldest version of <code>AntiSpam.exe</code> will still log their credentials to the file at <code>%TEMP%\qwertyuio.txt</code>, and present the prompt again for the user to retry until they are successful. This approach was likely used as a fail safe method to at least inform some future password guesses by the threat actor if the user grows suspicious or gives up before successfully validating their password. Newer versions have had this functionality removed. Once valid credentials are entered all versions will display a success message for the fake update in a pop-up window and exit after appending the credentials to the end of <code>%TEMP%\qwertyuio.txt</code>.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.16.46-PM.png" class="kg-image"><figcaption><em>Figure 6. The “success” message pop-up once valid credentials are entered.</em></figcaption></figure><!--kg-card-begin: markdown--><h3 id="updatexexe">updatex.exe</h3>
<p>The typical next step for threat actors after collecting environment information and the user’s credentials is to run a series of executables, where each filename typically begins with <code>update</code> and ends with a number. Like <code>AntiSpam.exe</code>, all of these executables have been signed by the same certificate, although they each have different functions. Many of the malware payloads also appear to be inserted into the original contents of the legitimate program they are pretending to be, which can be identified based on the included metadata of the executable.</p>
<h2 id="update1exe">update1.exe</h2>
<p>The file <code>update1.exe</code> presents itself as an installer for Yandex Disk, a cloud storage and file sharing service created by Yandex. Instead, the payload will load, decrypt (XOR key: <code>58 4C 4B 61 58 55 71 4F 58 4A 45 36 4A 34 75 57 66 65 2A 35 45 24 3F 75 55 4C 00</code>), and execute a second executable from an embedded resource, via local PE injection. Local PE injection is the process by which a portable executable (PE) file can be executed in the virtual memory of a process without using the Windows loader to read it from disk, though the process must then perform the functions of the Windows loader instead.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.17.48-PM.png" class="kg-image"><figcaption><em>Figure 7. <p><code>hxxp://xx.xx.xx[.]xx:xxxxxx/api/helper-first-register?buildVersion=&lt;build_version&gt;&amp;md5=&lt;md5_hash&gt;&amp;proxyPassword=&lt;proxy_password&gt;&amp;proxyUsername=&lt;proxy_username&gt;&amp;userId=&lt;user_id&gt;</code></p>

<p>Other observed executable payloads that have similar functionality includes <code>update2.dll</code>, which is a fake AMD DirectX driver library that also loads a second stage executable payload, via local PE injection. The second stage payload, once executed, will also reach out to several C2 addresses using a Golang HTTP library. The file <code>update2.dll</code> was compiled using an obfuscation framework that hinders analysis by obfuscating the control flow of the program, for example, by replacing assembly instructions with unnecessarily complex substitutes and inserting a large amount of jumps between instructions. Unlike several of the other payloads, <code>update2.dll</code> uses a custom decryption loop and a custom decompression loop to extract the second stage payload before execution.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.19.25-PM.png" class="kg-image"><figcaption><em>Figure 8. <!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.20.09-PM.png" class="kg-image"><figcaption><em>Figure 9. The contents of the payload as it is unpacked by "update2.dll".</em></figcaption></figure><!--kg-card-begin: markdown--><h2 id="update4exe">update4.exe</h2>
<p>The metadata for <code>update4.exe</code> presents the file as a copy of APEX Scan, an anti-virus scanner created by Trend Micro. The legitimate functionality of the original program has not been removed. Instead, the threat actor has added malicious code ahead of the legitimate entry point.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.21.43-PM.png" class="kg-image"><figcaption><em>Figure 10. A malicious function is called before the APEX Scan program at the Entry Point for "update4.exe".</em></figcaption></figure><!--kg-card-begin: markdown--><p>When <code>update4.exe</code> is executed, it will ultimately load, decrypt, and execute shellcode. The shellcode will then reach out to the C2 IPv4 address <code>91.196.70[.]160:443</code> and serve as a socks proxy between <code>91.196.70[.]160</code> and other IPv4 addresses that can be specified by the threat actor.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.22.25-PM.png" class="kg-image"><figcaption><em>Figure 11. The initial malware functionality within "update4.exe" that decrypts, loads, and executes shellcode.</em></figcaption></figure><!--kg-card-begin: markdown--><p>As seen above, the malware uses Windows Native API calls (NTAPI) to <code>ZwAllocateVirtualMemory</code> and <code>NtVirtualProtect</code> to facilitate execution of the shellcode. Malware developers use lower level calls to the NTAPI (which are limited in official Microsoft documentation) for multiple reasons, including the evasion of security solutions for which detections are based on high-level API calls.</p>
<p>In this case, the virtual memory permissions are set to <code>PAGE_EXECUTE_READWRITE</code> due to the call to <code>NtVirtualProtect</code> passing <code>0x40</code> for the new protection value. Most virtual memory pages do not typically possess all three permissions at the same time, so this is a very obvious indicator that the executable is performing suspicious actions, one that is present in several of the other payloads as well. If errors are encountered during the setup process a function call is made to an invalid address, causing the process to crash due to an access violation exception rather than exiting gracefully.</p>
<p>Before being executed the shellcode is decrypted with a simple XOR loop and the key <code>&lt;i1GiPEQ?V56^uh!m&lt;iUZF!yW?</code>. However, this password is only 26 characters long and we can see that the XOR loop uses a key with 27 characters. As a result, the full XOR key includes the string terminating null byte as part of the key. Is this intentional, or is it an <a href="https://en.wikipedia.org/wiki/Off-by-one_error">off by one error</a>? Based on analysis across other payloads, this type of loop has been implemented multiple times, so it could be an attempt to impede recovery of the XOR key, even if only briefly.<br>

<p>After resolving the necessary functions, the shellcode will send two sequences of null bytes (16 null bytes then 4, after connecting) to <code>91.196.70[.]160</code> and then wait in an infinite loop, without pausing, for a response.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.23.34-PM.png" class="kg-image"><figcaption><em>Figure 12. The destination to route data to can be specified by the socks server as either an IPv4 address or domain as the destination is parsed as a <a href="https://en.wikipedia.org/wiki/SOCKS#SOCKS5">SOCKS5 address structure</a>. So, in effect, the payload <code>update4.exe</code> serves as a socks proxy during execution.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.26.06-PM.png" class="kg-image"><figcaption><em>Figure 13. <!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.33.25-PM.png" class="kg-image"><figcaption><em>Figure 14. <!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.34.05-PM.png" class="kg-image"><figcaption><em>Figure 15. <!--kg-card-end: markdown--><!--kg-card-begin: markdown--><h2 id="update6exe">update6.exe</h2>
<p>When executed, <code>update6.exe</code> will attempt to exploit CVE-2022-26923 to add a machine account if the domain controller used within the environment is vulnerable. The debugging symbols database path has been left intact and indicates this: <code>C:\Users\lfkmf\source\repos\AddMachineAccount\x64\Release\AddMachineAccount.pdb</code>. The original source code was likely copied from the <a href="https://github.com/outflanknl/C2-Tool-Collection/blob/main/BOF/CVE-2022-26923/SOURCE/CVE-2022-26923.c">publicly available Cobalt Strike module created by Outflank</a>.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2024/07/Screenshot-2024-07-29-at-1.35.13-PM.png" class="kg-image"><figcaption><em>Figure 16. <!--kg-card-end: markdown--><!--kg-card-begin: markdown--><h2 id="mitigationguidance">Mitigation guidance</h2>
<p>Rapid7 recommends baselining your environment for all installed remote monitoring and management solutions and utilizing application allowlisting solutions, such as AppLocker or ​​Microsoft Defender Application Control to block all unapproved RMM solutions from executing within the environment. For example, the Quick Assist tool, quickassist.exe, can be <a href="https://learn.microsoft.com/en-us/windows/security/application-security/application-control/windows-defender-application-control/applocker/applocker-overview">blocked from execution via AppLocker</a>.  As an additional precaution, Rapid7 recommends blocking domains associated with all unapproved RMM solutions. A public GitHub repo containing a catalog of RMM solutions, their binary names, and associated domains can be found <a href="https://github.com/0x706972686f/RMM-Catalogue/blob/main/rmm.csv">here</a>.</p>
<p>Rapid7 recommends ensuring users are aware of established IT channels and communication methods to identify and prevent common social engineering attacks. We also recommend ensuring users are empowered to report suspicious phone calls and texts purporting to be from internal IT staff.</p>
<p>Rapid7 recommends regularly updating software used within the organization to prevent the exploitation of known software vulnerabilities. A <a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2022-26923">patch was released for CVE-2022-26923</a> in May of 2022 that can prevent vulnerable domain controllers from being exploited for privilege escalation.</p>
<p>Many of the C2 addresses identified within the malware payloads from this threat actor can be attributed to low cost server hosting providers (e.g., VDSINA, ASNs: <code>AS48282</code>,<code>AS216071</code>). If access to low cost VPN/VPS/VDS services is not necessary for business purposes within the environment, then they should be blocked to limit risk.</p>
