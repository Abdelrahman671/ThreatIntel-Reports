<h2 class="post-header__secondary-head blog__post-secondary-h2">Malware authors upped their game, using homoglyphs to impersonate a protected NuGet prefix and IL weaving to inject malicious code, RL researchers found.</h2>

<div class="blog__post-content html-content">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><h2><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp?width=1400&amp;height=732&amp;name=Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp" width="1400" height="732" loading="lazy" alt="Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs" style="height: auto; max-width: 100%; width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp?width=700&amp;height=366&amp;name=Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp?width=1400&amp;height=732&amp;name=Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp?width=2100&amp;height=1098&amp;name=Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp?width=2800&amp;height=1464&amp;name=Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp?width=3500&amp;height=1830&amp;name=Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp?width=4200&amp;height=2196&amp;name=Malicious-NuGet-campaign-uses-homoglyphs-and-IL-weaving-to-fool-devs.webp 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></h2>
<p>ReversingLabs has been actively tracking a malicious campaign targeting the NuGet package manager since the beginning of August, 2023. This report presents the findings of that research, which shows how malicious actors are continuously improving their techniques and responding to disruption of their campaigns.&nbsp;<br><br>Since the beginning of the campaign, threat actors behind this campaign refined their attacks, moving from exploiting NuGet’s MSBuild <a href="/blog/iamreboot-malicious-nuget-packages-exploit-msbuild-loophole" rel="noopener"></a>integrations to a strategy that uses simple, obfuscated downloaders that are inserted into legitimate PE binary files using Intermediary Language (IL) weaving, a .NET programming technique for modifying an application’s code after compilation.</p>
<!--more--><span class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><span class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"></span></span>
<h2 style="font-weight: normal; font-size: 24px;">Discussion</h2>
<p>In October 2023, the RL research team <a href="/blog/iamreboot-malicious-nuget-packages-exploit-msbuild-loophole" rel="noopener">reported on a malicious campaign targeting the NuGet repository</a>, a popular open source package manager for .NET code. The campaign has been active since the early August 2023 and included more than 700 malicious packages.&nbsp;<br><br>The threat actors behind this campaign have since evolved their methods to achieve execution of their malicious code. Attacks started with simple initialization and post-installation Powershell scripts in the affected packages which contained malicious downloader functionality. The hundreds of malicious NuGet packages the RL research team detected weren’t hard to spot — they contained links within their install scripts (unusual) and, upon closer inspection, contained no non-malicious functionality. Following the installation and execution of the downloader, second stage malware was retrieved from an attacker-controlled C2 server.&nbsp;<br><br>In the later stages of this campaign, the threat actors moved from initialization scripts to a lesser-known technique that exploits NuGet’s MSBuild integrations feature. Malicious functionality was added to build tasks inside a <span style="font-style: italic;">*.targets</span> file ensuring that malicious code gets executed every time the project is built. Additionally, a lot of effort was put into making malicious packages look as trustworthy as possible, by choosing suitable impersonation packages, making clever typo-squats, using icons from impersonated packages and by pumping up the package download counts.</p>
<h2 style="font-weight: normal; font-size: 24px;">Patching .NET binaries with malicious content</h2>
<p>As already concluded, the threat actor behind this campaign is determined to push malware into NuGet repository and has been doing it regularly for more than the last six months. It can be concluded that this threat actor is quite skilled also: they are constantly improving their arsenal with edgy execution techniques that can be used in .NET projects.&nbsp;<br><br>The latest malicious packages published by this threat actor are using a technique called “IL weaving” to add malicious functionality to legitimate Portable Executable (PE) .NET binaries. In these attacks, the attacker takes a compiled .NET binary from a legitimate NuGet package and patches it in order to inject a module initializer into it.&nbsp;<br><br><a href="https://github.com/dotnet/runtime/blob/main/docs/design/specs/Ecma-335-Augments.md#module-initializer" rel="noopener" target="_blank">Dotnet documentation</a> says that module initializers are “<span style="font-style: italic;">executed at, or sometime before, first access to any static field or first invocation of any method defined in the module.</span>” Simply said, they are the first thing that is executed when a .NET module is loaded. They can be added to the source code and created at the compile time, but a much more interesting way to add them is by patching already compiled .NET binaries using Intermediate Language (IL) weaving. Here, the original binary is first decompiled, then the desired functionality is added (“weaved”) to it. Finally it is compiled back to an executable file. Libraries like <a href="https://www.mono-project.com/docs/tools+libraries/libraries/Mono.Cecil/" rel="noopener" target="_blank">Mono.Cecil </a>can be used to handle MSIL (Microsoft Intermediate Language) or CIL compiling and decompiling. There are open source tools like <a href="https://github.com/Fody/Fody" rel="noopener" target="_blank">Fody</a> that encapsulate MSIL construction and simplify IL weaving.&nbsp;</p>
<p style="text-align: center;"><img src="https://www.reversinglabs.com/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-1.svg" width="860" height="395" loading="lazy" alt="Blog-Malicious-NuGet-campaign-goes-on-Figure-1" style="height: auto; max-width: 100%; width: 860px; margin-left: auto; margin-right: auto; display: block;"><span style="font-weight: bold; font-size: 16px;">Figure 1: IL weaving</span></p>
<p>Since the end of October, NuGet packages containing three types of patched, malicious .NET DLL libraries have been observed. All of them implement the same downloader functionality, and download second stage malware seen in the previous phases of this campaign: the obfuscated SeroXen RAT.&nbsp;<br><br>Using binary patching, threat actors tried to make their malware downloader packages harder to detect. Analyzing compiled binaries is a bit harder task than analyzing PowerShell scripts, which contain commands in plaintext format that are easier to scan. What also made detection harder is the fact that all of the patched binaries were DLL files, which aren’t executed on their own or by default. Rather, DLLs are libraries that are called and executed under specific circumstances, which complicates dynamic analysis. But there are tools like ReversingLabs Spectra Assure that are capable of extracting behavior indicators during static analysis, making it possible to detect the presence of suspicious functionality inside compiled binary files.</p>
<h2 style="font-weight: normal; font-size: 24px;">These aren’t the Gunas you’re looking for: homoglyphs in action</h2>
<p>The threat actors copied <a href="https://www.nuget.org/packages/Guna.UI2.WinForms" rel="noopener" target="_blank"><span style="font-style: italic;">Guna.UI2.WinForms</span></a>, a popular open source package that is used for creating desktop UIs. Using IL weaving, the&nbsp;threat actor modified the code to inject a module initializer and named the modified package <span style="font-style: italic;">Gսոa.UI3.Wіnfօrms</span>, an obvious effort to “typo squat” on the name of the legitimate package. They then published it to the NuGet repository.&nbsp;<br><br>What’s interesting here is that <span style="font-style: italic;">Guna</span> is actually a reserved prefix name on the NuGet repository. That means that only the authors that own and manage the Guna prefix can publish packages starting with that specific name — a security feature of sorts. So while it looks like a high-level NuGet account compromise, it's actually not.&nbsp;</p>
<span class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><span class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><a href="https://blog.reversinglabs.com/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png?width=1400&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png" alt="Package with a reserved prefix" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png?width=700&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png?width=1400&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png?width=2100&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png?width=2800&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png?width=3500&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png?width=4200&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-2.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a></span></span>
<p style="font-weight: bold; font-size: 16px; text-align: center;">Figure 2: Package with a reserved prefix</p>
<p>As it turns out, this “Guna” isn’t what it seems. That’s because the threat actor exploited a NuGet feature — its support for <em>homoglyphs —</em>&nbsp;to create a package that convincingly mimics those that use the reserved “Guna” prefix.&nbsp;<br><br>Homoglyphs are unique characters that look identical, while having totally different features (sounds) and digital identifiers. So, for example, the latin lower case “u” and the Armenian character “ս” (“seh”) are visually identical. However, each has a unique sound associated with it and a unique unicode value — \u0075 for the latin “u” and \u057D for the Armenian “seh.” While humans may not be able to tell them apart when visually scanning text, computers see and treat them as completely different characters.&nbsp;<br><br>Looking more closely at the name of the imposter NuGet package <em>G<span style="color: #000000;">un</span>a.UI3.<span style="color: #000000;">WinForms</span></em> package, RL found four of these homoglyph characters mimicking their latinate doppelgangers: ս(\u057D), ո(\u0578), і(\u0456) and օ(\u0585).<br><br>Using RL Spectra Assure platform to analyze the two packages revealed important differences between the legitimate <em>Guna.UI2.WinForms</em> and the imposter <em>Guna.UI3.WinForms</em> packages. Specifically, it revealed that two .NET binaries were tampered with to include a module initializer.</p>

<p>When RL researchers compared the two .NET binaries using third-party tools that can decompile and diff the code to show us the differences between them, our research team noticed that the two packages have the same contents, with the biggest difference being the module initializer inside the modified, malicious package.</p>
<span class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><span class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><a href="https://blog.reversinglabs.com/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png?width=1400&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png" alt="Package diff &amp; obfuscated Initialize method" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png?width=700&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png?width=1400&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png?width=2100&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png?width=2800&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png?width=3500&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png?width=4200&amp;name=Blog-Malicious-NuGet-campaign-goes-on-Figure-4.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a></span></span>

<p>The strings inside the module initializer (Figure 4) are obfuscated by a simple method. Adjacent to every character the threat actor tries to “hide”, they’ve placed a dummy string (in this case <em> {#}{$}{&amp;}{(}{_}{#}{%}{^}{^}{#}{!}  </em>that is removed at runtime by searching for the dummy string and removing it. It’s very easy to get the deobfuscated code and see what this DLL does.</p>

<h2 style="font-weight: normal; font-size: 24px;">Detection: YARA won’t do it</h2>
<p>Since the code for the module initializer is located inside the pseudo-class <span style="font-style: italic;">&lt;Module&gt;</span>, it can’t be detected by YARA out of the box, because it’s <a href="https://github.com/VirusTotal/yara/blob/4fc1ff822c11988ef616882a1a32e787359fdef4/libyara/modules/dotnet/dotnet.c#L1528" rel="noopener" target="_blank">skipping parsing that class</a>.<br><br>RL Spectra Assure provide a behavioral indicator for it. Module initializers are not purely used by malware. They are used for many legitimate reasons such as setting up code before the main execution starts (in other words, initializing a module) and by obfuscators — tools used to protect the intellectual property of developers.<br><br>However, RL can combine this indicator with others in order to create a good threat hunting heuristics for identifying suspicious packages.</p>

<h2 style="font-weight: normal; font-size: 24px;">Evolve your AppSec to stay up to speed with threats</h2>
<p>Threat actors are constantly evolving the methods and tactics they use to compromise and infect their victims with malicious code that is used to extract sensitive data or provide attackers with control over IT assets. Any methods that give them an edge over defenders and allow their actions to remain undetected have a high value and are likely to be adopted by malicious actors. That’s why defenders have to evolve alongside threat actors in order to catch any possible threats on time and stay safe.&nbsp;<br><br>And that’s what happened in this latest campaign, which saw attackers using a variety of new — or rarely seen — techniques to escape detection and increase the changes of their malicious code being downloaded and used. Those techniques include binary patching of DLLs via IL weaving to make malicious downloader code harder to detect. They also include the attackers’ use of “homoglyphs” in typosquatting attacks. Those are glyphs or characters with an identical visual appearance to a common character (like “u” or “o”), but a different logical representation. To date, RL researchers have identified approximately 60 packages and 290 package versions that are part of this same campaign abusing module initializers. Some of the packages can be seen in the IOCs list below.&nbsp;<br><br>All the identified packages were reported to NuGet administrators and have since been removed from the NuGet platform.&nbsp;<br><br>This latest campaign highlights new ways in which malicious actors are scheming to fool developers as well as security teams into downloading and using malicious or tampered with packages from popular open source package managers like NuGet. With scores of packages and hundreds of malicious versions, the latest campaign highlights the need for development organizations to up their game and stay abreast of a fast evolving software supply chain threat landscape.</p>
