                            <p><em>*Rapid7 Incident Response consultants Noah Hemker, Tyler Starks, and malware analyst Tom Elkins contributed analysis and insight to this blog.*</em></p><!--kg-card-begin: markdown--><p>Rapid7 Incident Response was engaged to investigate an incident involving unauthorized access to two publicly-facing Confluence servers that were the source of multiple malware executions. Rapid7 identified evidence of exploitation for <a href="https://www.rapid7.com/blog/post/2024/01/19/etr-critical-cves-in-outdated-versions-of-atlassian-confluence-and-vmware-vcenter-server/">CVE-2023-22527</a> within available Confluence logs. During the investigation, Rapid7 identified cryptomining software and a Sliver Command and Control (C2) payload on in-scope servers. Sliver is a modular C2 framework that provides adversarial emulation capabilities for red teams; however, it’s also frequently abused by threat actors. The Sliver payload was used to action subsequent threat actor objectives within the environment. Without proper security tooling to monitor system network traffic and firewall communications, this activity would have progressed undetected leading to further compromise.</p>
<!--kg-card-end: markdown--><h2 id="rapid7-customers">Rapid7 customers</h2><!--kg-card-begin: markdown--><p>Rapid7 consistently monitors <a href="https://www.rapid7.com/blog/tag/emergent-threat-response/">emergent threats</a> to identify areas for new detection opportunities. The recent appearance of Sliver C2 malware prompted Rapid7 teams to conduct a thorough analysis of the techniques being utilized and the potential risks. Rapid7 InsightIDR has an alert rule <code>Suspicious Web Request - Possible Atlassian Confluence CVE-2023-22527 Exploitation</code> available for all IDR customers to detect the usage of the <code>text-inline.vm</code> consistent with the exploitation of CVE-2023-22527. A <a href="https://www.rapid7.com/db/vulnerabilities/atlassian-confluence-cve-2023-22527/">vulnerability check</a> is also available to InsightVM and Nexpose customers. A Velociraptor artifact to hunt for evidence of Confluence CVE-2023-22527 exploitation is available on the Velociraptor Artifact Exchange <a href="https://docs.velociraptor.app/exchange/artifacts/pages/confluence_cve_2023_22527/">here</a>. Read Rapid7’s blog on <a href="https://www.rapid7.com/blog/post/2024/01/19/etr-critical-cves-in-outdated-versions-of-atlassian-confluence-and-vmware-vcenter-server/">CVE-2023-22527</a>.</p>
<!--kg-card-end: markdown--><h2 id="observed-attacker-behavior">Observed Attacker Behavior</h2><!--kg-card-begin: markdown--><p>Rapid7 IR began the investigation by triaging available forensic artifacts on the two affected publicly-facing Confluence servers. These servers were both running vulnerable Confluence software versions that were abused to obtain Remote Code Execution (RCE) capabilities. Rapid7 reviewed server access logs to identify the presence of suspicious <code>POST</code> requests consistent with known vulnerabilities, including <code>CVE-2023-22527</code>. This vulnerability is a critical OGNL injection vulnerability that abuses the <code>text-inline.vm</code> component of Confluence by sending a modified POST request to the server.</p>
<p>Evidence showed multiple instances of exploitation of this CVE, however, evidence of an embedded command would not be available within the standard header information logged within access logs. Packet Capture (PCAP) was not available to be reviewed to identify embedded commands, but the identified <code>POST</code> requests are consistent with the exploitation of the CVE.<br>
The following are a few examples of the exploitation of the Confluence CVE found within access logs:</p>

<!--kg-card-end: markdown--><!--kg-card-begin: markdown--><p>Evidence showed the execution of a <code>curl</code> command post-exploitation of the CVE resulting in the dropping of cryptomining malware to the system. The IP addresses associated with the malicious POST requests to the Confluence servers matched the IP addresses of the identified <code>curl</code> command. This indicates that the dropped cryptomining malware was directly tied to Confluence CVE exploitation.<br>
As a result of the executed <code>curl</code> command, file <code>w.sh</code> was written to the <code>/tmp/</code> directory on the system. This file is a bash script used to enumerate the operating system, download cryptomining installation files, and then execute the cryptomining binary. The bash script then executed the <code>wget</code> command to download <code>javs.tar.gz</code> from the IP address <code>38.6.173[.]11</code> over port <code>80</code>. This file was identified to be the <code>XMRigCC</code> cryptomining malware which caused a spike in system resource utilization consistent with cryptomining activity. Service <code>javasgs_miner.service</code> was created on the system and set to run as root to ensure persistence.</p>
<p>The following is a snippet of code contained within <code>w.sh</code> defining communication parameters for the downloading and execution of the XMRigCC binary.</p>

This response registered as ‘invalid’ as it contained characters that could not be accurately interpreted. Evidence confirmed the successful download and execution of the XMRigCC miner, so the above Catalina log may prove useful for analysts to identify additional proof of attempted or successful exploitation.</p>

<!--kg-card-end: markdown--><!--kg-card-begin: markdown--><p>Rapid7 then shifted focus to begin a review of system network connections on both servers. Evidence showed an active connection with known-abused IP address <code>193.29.13[.]179</code> communicating over port <code>8888</code> from both servers. <code>netstat</code> command output showed that the network connection’s source program was called <code>X-org</code> and was located within the system’s <code>/tmp</code> directory. According to firewall logs, the first identified communication from this server to the malicious IP address aligned with the timestamps of the identified <code>X-org</code> file creation. Rapid7 identified another malicious file residing on the secondary server named <code>X0</code> Both files shared the same SHA256 hash, indicating that they are the same binary. The hash for these files has been provided below in the IOCs section.</p>
<p>A review of firewall logs provided a comprehensive view of the communications between affected systems and the malicious IP address. Firewall logs filtered on traffic between the compromised servers and the malicious IP address showed inbound and outbound data transfers consistent with known C2 behavior. Rapid7 decoded and debugged the Sliver payload to extract any available Indicators of Compromise (IOCs). Within the Sliver payload, Rapid7 confirmed the following IP address <code>193.29.13[.]179</code> would communicate over port <code>8888</code> using the <code>mTLS</code> authentication protocol.</p>
<!--kg-card-end: markdown--><figure class="kg-card kg-image-card"><img src="https://blog.rapid7.com/content/images/2024/02/Screenshot-2024-02-15-at-11.38.04-AM.png" class="kg-image"></figure><!--kg-card-begin: markdown--><p>After Sliver first communicated with the established C2, it checked the username associated with the current session on the local system, read <code>etc/passwd</code> and <code>etc/machine-id</code> and then communicated back with the C2 again. The contents of <code>passwd</code> and <code>machine-id</code> provide system information such as the hostname and any account on the system. Cached credentials from the system were discovered to be associated with outbound C2 traffic further supporting this credential access. This activity is consistent with the standard capabilities available within the GitHub release of Sliver hosted <a href="https://github.com/BishopFox/sliver/tree/master">here</a>.</p>
<p>The Sliver C2 connection was later used to execute <code>wget</code> commands used to download <code>Kerbrute</code>, <code>Traitor</code>, and <code>Fscan</code> to the servers. <code>Kerbute</code> was executed from <code>dev/shm</code> and is commonly used to brute-force and enumerate valid Active Directory accounts through Kerberos pre-authentications. The <code>Traitor</code> binary was executed from the <code>var/tmp</code> directory which contains the functionality to leverage <code>Pwnkit</code> and <code>Dirty Pipe</code> as seen within evidence on the system. <code>Fscan</code> was executed from the <code>var/tmp</code> directory with the file name <code>f</code> and performed scanning to enumerate systems present within the environment. Rapid7 performed containment actions to deny any further threat actor activity. No additional post-exploitation objectives were identified within the environment.</p>
<!--kg-card-end: markdown--><h2 id="mitigation-guidance">Mitigation guidance</h2><!--kg-card-begin: markdown--><p>To mitigate the attacker behavior outlined in this blog, the following mitigation techniques should be considered:</p>
<ul>
<li>
<p>Ensure that unnecessary ports and services are disabled on publicly-facing servers.</p>
</li>
<li>
<p>All publicly-facing servers should regularly be patched and remain up-to-date with the most recent software releases.</p>
</li>
<li>
<p>Environment firewall logs should be aggregated into a centralized security solution to allow for the detection of abnormal network communications.</p>
</li>
<li>
<p>Firewall rules should be implemented to deny inbound and outbound traffic from unapproved geolocations.</p>
</li>
<li>
<p>Publicly-facing servers hosting web applications should implement a restricted shell, where possible, to limit the capabilities and scope of commands available when compared to a standard bash shell.</p>
</li>
</ul>
