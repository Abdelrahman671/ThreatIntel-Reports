<h3><strong>Summary</strong></h3><br />
<p> </p><br />
<p>REvil aka Sodinokibi, Sodin is a ransomware family operated as a ransomware-as-a-service (RaaS). Deployments of REvil first were observed in April 2019, where attackers leveraged a vulnerability in Oracle WebLogic servers tracked as CVE-2019-2725.</p><br />
<p> </p><br />
<p>REvil is highly configurable and allows operators to customize the way it behaves on the infected host. Some of its features include:</p><br />
<p> </p><br />
<ul><li>Exploits a kernel privilege escalation vulnerability to gain SYSTEM privileges using CVE-2018-8453.</li><li>Whitelists files, folders and extensions from encryption.</li><li>Kills specific processes and services prior to encryption.</li><li>Encrypts files on local and network storage.</li><li>Customizes the name and body of the ransom note, and the contents of the background image.</li><li>Exfiltrates encrypted information on the infected host to remote controllers.</li><li>REvil uses Hypertext Transfer Protocol Secure (HTTPS) for communication with its controllers.</li></ul><p> </p><br />
<h3><strong>Capabilities Overview</strong></h3><br />
<p> </p><br />
<ul><li>Ransomware.</li></ul><p> </p><br />
<h3><strong>Behaviour Overview</strong></h3><br />
<p> </p><br />
<ul><li>Uses a persistence mechanism.</li><li>Encrypts additional resources.</li><li>Supports privilege escalation.</li></ul><p> </p><br />
<h3><strong>Adversary intelligence</strong></h3><br />
<p> </p><br />
<h4><strong>Developers</strong></h4><br />
<p> </p><br />
<p>REvil ransomware first was advertised on a Russian language cybercrime forum in June 2019. The main actor associated with advertising and promoting REvil ransomware is called <strong>Unknown </strong>aka <strong>UNKN</strong>. The RaaS is operated as an affiliate service, where affiliates spread the malware by acquiring victims and the REvil operators maintain the malware and payment infrastructure. Affiliates receive 60% to 70% of the ransom payment.</p><br />
<p> </p><br />
<p>Due to source code and behavior similarities between REvil and GandCrab, it was suggested there might be a connection tying the developers of the two ransomware families together. In addition to the similarities in the code, supplemental evidence tying GandCrab and REvil together is that GandGrab officially "retired" just before REvil appeared in the wild. REvil is maintained actively and is under constant development, just as GandCrab was. The most recent REvil ransomware at the time of this report was version 2.1.</p><br />
<p> </p><br />
<p>The actor <strong>Unknown</strong> acknowledged the public reports linking REvil to GandCrab with the following statement:</p><br />
<p> </p><br />
<p><em><code>“We used to be affiliates of the (GandCrab) affiliate program. We bought the source code and started our own business. We developed custom features for our purposes”</code></em></p><br />
<p> </p><br />
<p>Despite the plausible alibi given by actor <strong>Unknown</strong>, the evidence suggests that REvil is a continuation of the GandCrab RaaS operation with new software, but operated by the same individuals.</p><br />
<p> </p><br />
<p>The actors behind REvil include their master public key in all REvil binaries. Therefore, they are able to decrypt the files independently of the affiliates running the campaigns.</p><br />
<p> </p><br />
<h4><strong>Operators</strong></h4><br />
<p> </p><br />
<p>REvil is a RaaS, in the sense that a single group operates and manages the development of the ransomware while access is sold to affiliates. A field in the configuration file named <strong>pid</strong> is used to identify the affiliate that the sample belongs to. We have confirmed that a field in the malware configuration named <strong>sub</strong> is used to identify affiliate campaigns, and not the affiliates themselves, as frequently is reported. In addition, the attacker's public key can identify the same operator when used across multiple samples.</p><br />
<p> </p><br />
<p>Two prominent users on the aforementioned Russian language cybercrime forum vouched for <strong>Unknown's</strong> ransomware service:</p><br />
<p> </p><br />
<ul><li><strong>kerberos</strong> - A moderator on the aforementioned forum and long-standing cybercriminal.</li><li><strong>lalartu</strong> - A cybercrime actor known to participate in GandCrab and REvil ransomware affiliate programs.</li></ul><p> </p><br />
<p>The actor <strong>lalartu's</strong> personal information possibly was released for malicious purposes or "doxxed" by an information security researcher known as UnderTheBreach (see: <a href="https://medium.com/@underthebreach/tracking-down-revils-lalartu-by-utilizing-multiple-osint-methods-2bf3a6c65a80" target="_blank" rel="noreferrer noopener">https://medium.com/@underthebreach/tracking-down-revils-lalartu-by-utilizing-multiple-osint-methods-2bf3a6c65a80</a>). We have not been able to verify the conclusions asserted in this post.</p><br />
<p> </p><br />
<h3><strong>Infrastructure</strong></h3><br />
<p> </p><br />
<p>REvil ransomware configurations contain more than 1,000 controllers. Interestingly, the live domains we verified all were WordPress websites, so it is probable they might be compromised by the operators or purchased from other cybercriminals.</p><br />
<p> </p><br />
<p>The configuration also contains domain names that were not registered at the time of this report, for example:</p><br />
<p> </p><br />
<table><tbody><tr><td><p>$ whois andersongilmour[.]co[.]uk</p><br />
<p>No match for "andersongilmour[.]co[.]uk."</p></td></tr></tbody></table><p> </p><br />
<p>This domain name was not registered at the time of this report.</p><br />
<p> </p><br />
<p>It is likely most of the domains present in the configuration are decoys, with only a few real REvil controllers scattered inside the list.</p><br />
<p> </p><br />
<h3><strong>Detection</strong></h3><br />
<p> </p><br />
<p>This section describes the methods used to detect a REvil sample.</p><br />
<p> </p><br />
<h4><strong>Static Detection</strong></h4><br />
<p> </p><br />
<p>Unpacked REvil samples can be detected statically by looking up patterns in the code and in cryptographic functions used by the ransomware.</p><br />
<p> </p><br />
<h4><strong>Dynamic Detection</strong></h4><br />
<p> </p><br />
<p>Dynamic detection of REvil samples is possible by running Yara signatures on the memory resident image. In addition, the ransomware creates the following interesting artifacts:</p><br />
<p> </p><br />
<ul><li>A <strong>.txt</strong> file with a ransom note on each directory encrypted by the ransomware.</li><li>A <strong>.bmp</strong> image in the temporary directory set as the desktop background after the encryption.</li><li>A registry key <strong>SOFTWARE </strong>which can be present under either<strong> HKEY_LOCAL_MACHINE</strong> or <strong>HKEY_CURRENT_USER</strong>.</li><li>An alphanumeric file extension between 5 and 10 characters in length appended to the original extension of an encrypted file.</li></ul><p> </p><br />
<h4><strong>Yara Signatures</strong></h4><br />
<p> </p><br />
<p>We decided not to release Yara signatures to avoid burning them and affecting the work of other researchers. However, we will release them to network defenders and infosec professionals upon request. Email us at revil-yara-req<em>REMOVEME</em>@intel471[.]com (remove the "<em>REMOVEME</em>" text from the email address) from your corporate email address (for validation purposes only) and we will send through our Yara signatures.</p><br />
<p> </p><br />
<h3><strong>Exploits</strong></h3><br />
<p> </p><br />
<p>REvil ransomware exploits a kernel privilege escalation vulnerability in win32k.sys tracked as CVE-2018-8453 to gain SYSTEM privileges on the infected host. If the configuration instructs a sample to execute this exploit, it will allocate executable memory, decrypt the exploit code in the newly allocated region and invoke it.</p><br />
<p> </p><br />
<p>The screenshot below shows the window name <strong>sysshadow</strong> and the application programming interface (API) <strong>DestroyWindow</strong> inside the embedded exploit, proving it is the CVE-2018-8453 vulnerability.</p><br />

<h3><strong>Static Analysis</strong></h3><br />
<p> </p><br />
<p>REvil ransomware incorporates techniques to make the task of static analysis more difficult for an analyst. Most of the strings used during execution are decrypted at runtime only when needed. In addition, the imports are resolved dynamically from 32-bit hashes and placed into global variables early in execution.</p><br />
<p> </p><br />
<h4><strong>Hard-coded Strings</strong></h4><br />
<p> </p><br />
<p>Very few strings are hard coded inside REvil ransomware samples and the most interesting are:</p><br />
<p> </p><br />
<ul><li><strong>L“ServicesActive”</strong>: This string is passed to the <strong>OpenSCManagerW</strong> API to retrieve active services.</li><li><strong>“expand 32-byte kexpand 16-byte”</strong>: The constants used by the Salsa20 symmetric encryption algorithm.</li></ul><p> </p><br />
<h4><strong>String Encryption</strong></h4><br />
<p> </p><br />
<p>REvil ransomware decrypts most of the strings it uses during execution at runtime.</p><figure><img src="https://d39ec1uo9ktrut.cloudfront.net/image7-300x199.png" alt="" /></figure><p>As the screenshot above reveals, these strings are Rivest Cipher 4 (RC4) encrypted and are decrypted using a function we renamed to <strong>rc4_decrypt_string</strong>. Below is its prototype and the meaning of each parameter:</p><br />
<p> </p><br />
<p><code>void rc4_decrypt_string(BYTE *rc4_array, int rc4_key_offset, int rc4_key_length, int buffer_length, BYTE *out_buffer)</code></p><br />
<p> </p><br />
<ul><li><strong>rc4_array</strong>: A pointer to a contiguous array in the .data section containing the RC4 keys and the encrypted strings. Each RC4 key is followed directly by the string it decrypts.</li><li><strong>rc4_key_offset</strong>: The offset to the RC4 key within the array.</li><li><strong>rc4_key_length</strong>: The length of the RC4 key.</li><li><strong>buffer_length</strong>: The length of the RC4 encrypted buffer.</li><li><strong>out_buffer</strong>: Pre-allocated memory or stack space where the decrypted string is copied.</li></ul><p> </p><br />
<p>The image below shows the layout of two adjacent elements of the array:</p><br />
<figure><img src="https://d39ec1uo9ktrut.cloudfront.net/image2-300x49.png" alt="" /></figure><p>The decrypted strings are not terminated by NULL characters, therefore, the code must terminate these strings explicitly.</p><br />
<p> </p><h3><strong>Malware Behavior</strong></h3><br />
<p> </p><br />
<p>The behavior of the following samples was analyzed for this report:</p><br />
<p> </p><table><tbody><tr><td><strong>Sample</strong></td><td><strong>SHA256</strong></td></tr><tr><td>REvil packed</td><td>6953d86d09cb8ed34856b56f71421471718ea923cd12c1e72224356756db2ef1</td></tr><tr><td>REvil not packed</td><td>372c8276ab7cad70ccf296722462d7b8727e8563c0bfe4344184e1bc3afc27fc</td></tr><tr><td>REvil not packed</td><td>ec0c653d5e10fec936dae340bf97c88f153cc0cdf7079632a38a19c876f3c4fe</td></tr></tbody></table><p> </p><h4><strong>Process Execution</strong></h4><br />
<p> </p><br />
<p>During its initialization phase, REvil starts by dynamically resolving the imports it needs to function correctly. This is accomplished in a loop that reads hard-coded 32-bit values from an array in the .data section, then each value is decoded and resolved to the correct API by the responsible function <strong>rvl_resolve_api</strong>. Then the 32-bit value is overwritten with the pointer to the API.</p><figure><img src="https://d39ec1uo9ktrut.cloudfront.net/image4-300x227.png" alt="" /></figure><p>As shown in the disassembly above, additional APIs are resolved by their names with the help of the <strong>GetProcAddress</strong> API.</p><br />
<p> </p><br />
<p>With everything in place, REvil creates a global mutual exclusion object (mutex) with a hard-coded name i.e., <strong>Global\1DE3C565-E22C-8190-7A66-494816E6C5F5</strong>. This is used to ensure only a single instance of the ransomware sample is running.</p><br />
<p> </p><br />
<p>The REvil ransomware always attempts to run with elevated privileges and does so using two techniques. One technique relies on exploiting the CVE-2018-8453 vulnerability to gain SYSTEM privileges on the host. However, to determine whether it should execute the exploit or not, REvil checks its configuration. Another technique always is executed if the process is not elevated. It relies on calling <strong>ShellExecuteW</strong> to prompt the user to run the sample as an administrator. This is accomplished in an infinite loop until the user agrees to run the elevated process. Starting with version 2.1, the privilege escalation exploit code was removed.</p><br />
<p> </p><br />
<p>REvil’s configuration is in JavaScript Object Notation (JSON) and is initially RC4 encrypted. It is stored in the following fashion at the beginning of a portable executable (PE) section named <strong>.yhwfq9</strong>:</p><br />
<figure><img src="https://d39ec1uo9ktrut.cloudfront.net/image6-300x67.png" alt="" /></figure><p>It is important to mention the cyclic redundancy check (CRC32) value is calculated and validated for the encrypted configuration.</p><br />
<p> </p><br />
<p>After decrypting the configuration in dynamically allocated memory, REvil parses it using the open-source json-parser C library (see: <a href="https://github.com/udp/json-parser" target="_blank" rel="noreferrer noopener">https://github.com/udp/json-parser</a>). For an example of a REvil configuration, see the Appendix section below.</p><br />
<p> </p><br />
<p>The first JSON field checked is <strong>exp</strong>, which can be <strong>true</strong> or <strong>false</strong> and indicates whether the CVE-2018-8453 vulnerability should be exploited. If the exploit isn’t executed or failed, REvil resorts to the second technique previously described in this section.</p><br />
<p> </p><br />
<p>When REvil executes with higher privileges, it starts its main initialization phase where it reads the necessary JSON fields into the .data section and initializes registry values inside a new subkey named <strong>SOFTWARE</strong>. When doing registry operations in general, REvil first tries to use the <strong>HKEY_LOCAL_MACHINE</strong> hive and it switches to use <strong>HKEY_CURRENT_USER</strong> only if that fails. The configuration fields, registry keys and their values are described in the below Configuration section.</p><br />
<p> </p><br />
<p>The other important task in this phase is filling the ransom note template, which is present in base64 inside the configuration field <strong>nbody</strong>.</p><br />

<p> </p><br />
<p>REvil ransomware implements encryption schemes involving an elliptic curve called Curve25519, Salsa20, SHA-3, CRC32 and Advanced Encryption Standard (AES). We identified the specific open-source implementations utilized by REvil:</p><br />
<p> </p><ul><li>Curve25519: <a href="https://github.com/vstakhov/opt-cryptobox/tree/master/curve25519" target="_blank" rel="noreferrer noopener">https://github.com/vstakhov/opt-cryptobox/tree/master/curve25519</a> .</li><li>Salsa20: <a href="https://cr.yp.to/snuffle/salsa20/merged/salsa20.c" target="_blank" rel="noreferrer noopener">https://cr.yp.to/snuffle/salsa20/merged/salsa20.c</a> .</li></ul><p> </p><br />
<p>This subsection delves into the details of how these algorithms are used to encrypt data and files.</p><br />
<p> </p><h5><strong>Encryption keys</strong></h5><br />
<p> </p><br />
<p>REvil ransomware relies on Curve25519 to generate public and secret key pairs and to create shared keys for encryption. What follows defines the Curve25519 keys referred to throughout this subsection:</p><br />
<p> </p><table><tbody><tr><td><strong>Key-pair name</strong></td><td><strong>Description</strong></td></tr><tr><td>crypt (crypt_public/crypt_secret)</td><td>The victim’s main key pair used for file encryption.</td></tr><tr><td>file (file_public / file_secret)</td><td>A random key pair generated for each encrypted file.</td></tr><tr><td>attacker (attacker_public / attacker_secret)</td><td>The attacker’s key pair. attacker_public is present under the <strong>pk</strong> configuration field.</td></tr><tr><td>master (master_public / master_secret)</td><td>The master key pair. master_public is hard coded inside the binary code and is the same across all REvil samples.</td></tr><tr><td>user_config (user_config_public / user_config_secret)</td><td>The user configuration key pair. user_config_public is hard coded in the binary code and is used to encrypt the user key we find encrypted in the ransom note.</td></tr></tbody></table><p> </p><h5><strong>Data encryption</strong></h5><br />
<p> </p><br />
<p>REvil ransomware encrypts all important data it stores in the registry. For instance, the user key present in the ransom note and in the registry is a JSON dictionary encrypted using the method we describe in the paragraphs that follow.</p><br />
<p> </p><br />
<p>To encrypt a buffer, REvil invokes the following function:</p><br />
<p> </p><br />
<p><code>BYTE* rvl_encrypt_data(BYTE *hispublic, BYTE *buffer, int buffer_length, BYTE *out_buffer_length)</code></p><br />
<p> </p><br />
<p>This function requires a 32-byte Curve25519 public key, a buffer to encrypt and its length. The end result is a pointer to the buffer in the return value and its length in the output parameter <strong>out_buffer_length</strong>.</p><br />
<p> </p><br />
<p>Internally, this function performs the following actions:</p><br />
<p> </p><ul><li>Allocate buffer_length bytes plus an extra 56 bytes to hold the final data.</li><li>Zero out the first 4 bytes of the allocated buffer.</li><li>Copy the buffer to encrypt after the zeroed double word.</li><li>Generate a new Curve25519 key pair we call new_public and new_secret.</li><li>Calculate a Curve25519 shared key between the hispublic key supplied in the arguments and the new_secret. The recipient of the message can use the secret key and new_public to obtain the same shared key.</li><li>Hash the shared key using the SHA-3 algorithm.</li><li>Clear the shared key from memory.</li><li>Clear new_secret from memory.</li><li>Generate a random 128-bit AES initialization vector.</li><li>Encrypt the buffer using AES with the SHA-3 hash as a key. The buffer to encrypt includes the prepended 32-bit NULL value.</li><li>Clear the SHA-3 hash from memory.</li><li>Calculate the CRC32 of the encrypted buffer.</li><li>Append the following elements to the encrypted buffer in this order:<ul><li>The generated new_public key (32 bytes).</li><li>The initialization vector (16 bytes).</li><li>CRC32 of the encrypted buffer (4 bytes).</li></ul></li></ul><p> </p><br />
<p>The end result looks like:</p><figure><img src="https://d39ec1uo9ktrut.cloudfront.net/image8-300x71.png" alt="" /></figure><p>As an example, after decoding the user key present in the ransom note using base64, we see it respects this same format:</p><figure><img src="https://d39ec1uo9ktrut.cloudfront.net/image10-300x117.png" alt="" /></figure><p>In order for the attackers to decrypt this data and retrieve the JSON dictionary and as a result the encrypted victim’s crypt_secret key, they must do the following:</p><br />
<p> </p><br />
<ul><li>Verify the CRC32 hash of the encrypted data.</li><li>Calculate a Curve25519 shared key using the new_public key in the data and the attacker’s secret user_config_secret.</li><li>Hash the shared key using SHA-3.</li><li>Retrieve the AES initialization vector and decrypt the buffer using AES.</li><li>Remove the NULL double word at the start of the buffer.</li></ul><p> </p><br />
<p>It is important to mention the victim’s <em>crypt_secret</em> key is encrypted the same way using the attacker’s <em>attacker_public</em> key (registry value “1TfXk”) and the <em>master_public</em> key (registry value “2YEdLY”).</p><br />
<p> </p><br />
<h5><strong>File encryption</strong></h5><br />
<p> </p><br />
<p>REvil ransomware encryption works by encrypting the files in 1 MB increments unless the file size is smaller, the remaining bytes to encrypt are less than 1 MB or when the encryption type specified is not full encryption. Contents of the file are read, encrypted and written back to the file overwriting the original content. After the encryption is complete, metadata is written to the end of file and the file is renamed to include the generated file extension (registry value <strong>AaZW1s3</strong>).</p><br />
<p> </p><br />
<p>For each given file, REvil ransomware starts encryption by initializing a structure used throughout the process. Its first part includes the Windows <strong>OVERLAPPED</strong> structure used for asynchronous I/O followed by custom fields REvil uses. The most interesting part of this structure was reverse engineered to the following:</p><br />
<p> </p><br />
<p><code>&lt;strong&gt;typedef&lt;/strong&gt; &lt;strong&gt;struct&lt;/strong&gt; _rvl_filecrypt_struct<br />
<br />
{<br />
<br />
OVERLAPPED Overlapped;<br />
<br />
&lt;em&gt;/*..SNIP..*/&lt;/em&gt;<br />
<br />
&lt;strong&gt;struct&lt;/strong&gt; _rvl_metadata<br />
<br />
{<br />
<br />
BYTE crypt_secret_w_attacker_pub[88]; &lt;em&gt;// same as registry value "1TfXk".&lt;/em&gt;<br />
<br />
BYTE crypt_secret_w_master_pub[88]; &lt;em&gt;// same as registry value "2YEdLY".&lt;/em&gt;<br />
<br />
BYTE file_public[32]; &lt;em&gt;// The file_public key.&lt;/em&gt;<br />
<br />

<br />

<p> </p><br />
<p>This metadata substructure is appended to the end of every encrypted file, is used by the decrypter and by REvil to verify a file previously was encrypted.</p><br />
<p> </p><br />
<p>The Salsa20 symmetric encryption key is the SHA-3 hash of a shared key derived from the victim’s <em>crypt_public</em> key and the secret of a generated key pair: <em>file_secret</em>.</p><br />
<br />
<p>To decrypt a file, both the victim’s <em>crypt_secret</em> key and the <em>file_public</em> key must be known:</p><br />
<ul><li>It is possible to decrypt the <em>crypt_secret</em> knowing either the operators’ <em>attacker_secret</em> or <em>master_secret</em> keys and by applying the same logic described in the Data encryption subsection.</li><li>It is possible to retrieve the per file <em>file_public</em> key from the end of an encrypted file after validating its CRC32 hash.</li><li>Having attained both a public and a secret key from previous steps, generate a shared Curve25519 key and hash it using SHA-3.</li><li>Copy the value of the Salsa20 initialization vector from the metadata structure at the end of the file.</li><li>Test decrypting the encrypted NULL double word with the SHA-3 hash as a key.</li><li>Determine the encryption type used to perform correct decryption of the file, then start decrypting accordingly.</li></ul><figure><img src="https://d39ec1uo9ktrut.cloudfront.net/image9-300x105.png" alt="" /></figure><p>By following this scheme, the attackers are sure not to divulge their secret keys when sending decrypters to victims. On their side, attackers only have to decrypt the victim’s <em>crypt_secret</em> key and send back a decrypter embedding this key.</p><h4><strong>Persistence mechanism</strong></h4><br />
<p>Prior to version 2.1, REvil ransomware persists on the machine if the <strong>arn</strong> configuration field is set to <strong>true</strong>. It writes its path to the registry key <strong>SOFTWARE</strong> for persistence. The value name of the entry for the analyzed sample is <strong>k51299BQXH</strong>.</p><br />
<p>Before terminating execution, REvil marks its executable file for deletion during the next reboot. As a result, the persistence path will become invalid.</p><br />
<p>The persistence mechanism was removed from REvil version 2.1.</p><h3><strong>Protections</strong></h3><br />
<p>REvil ransomware does not implement any protections.</p><h3><strong>Communications</strong></h3><br />
<p>REvil ransomware only initiates communication with its controllers when the configuration field <strong>net</strong> is set to <strong>true</strong>. In that case, REvil extracts the controller list from the <strong>dmn</strong> configuration string and proceeds to build a custom URL in a loop for each controller before initiating communications.</p><br />
<p>Each controller is preceded by the string <strong>https://</strong>, followed by a random path chosen from hard-coded values and then terminated by a random file name. The regular expression below matches all random URL paths generated by REvil ransomware:</p><br />
<p><code>\/(wp-content|static|content|include|uploads|news|data|admin)\/(images|pictures|image|temp|tmp|graphic|assets|pics|game)\/([a-z]{2}){1,10}\.(jpg|png|gif)</code></p><br />
<p>REvil then proceeds to send a POST request containing the victim’s key stored in the registry value <strong>QaUXNv2P</strong> to the controllers.
