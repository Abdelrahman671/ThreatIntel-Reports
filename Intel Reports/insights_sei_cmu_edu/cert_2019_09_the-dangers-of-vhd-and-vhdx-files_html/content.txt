                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="e6id5">Recently, I gave a presentation at <a href="https://www.bsidespgh.com">BSidesPGH</a> 2019 called <a href="https://www.bsidespgh.com/2019#dormann">Death By Thumb Drive: File System Fuzzing with CERT BFF</a>. (The <a href="https://resources.sei.cmu.edu/library/asset-view.cfm?assetid=551450">slides from my presentation</a> are available in the SEI Digital Library.) Although my primary goal was to find bugs in kernel file-system-parsing code, a notable part of my research was investigating attack vectors. In particular, I focused on VHD and VHDX files on Windows systems. In this post, I describe some of the risks associated with these two file types.</p>
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h2>VHD and VHDX Files</h2>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="hivw3">The VHD (Virtual Hard Disk) file format, originally introduced with Connectix Virtual PC, can store the contents of a hard disk drive. Eventually, <a href="https://docs.microsoft.com/en-us/windows-server/virtualization/hyper-v/hyper-v-technology-overview">Microsoft Hyper-V</a> adopted this disk image format. Windows 7 and newer systems include the ability to manually mount VHD files. Starting with Windows 8, a user can mount a VHD by simply double-clicking on the file. Once mounted, a VHD disk image appears to Windows as a normal hard disk that's physically connected to the system. VHDX (Virtual Hard Disk v2) images are functionally equivalent to VHD images, but they include more modern features, such as support for larger sizes and disk resizing.</p>
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h2>VHD/VHDX and File System Corruption</h2>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="fdru2">After fuzzing file system images with BFF, I was able to find several different ways to crash Windows as the result of it mounting a corrupted disk. Physically plugging in a USB mass storage device with a corrupted file system was the obvious attack vector. However, many security concepts are negated when <a href="https://www.kb.cert.org/vuls/id/789985/">physical access to a system</a> is granted. VHD and VHDX files eliminate the requirement for physical access to a victim system. If a user simply double-clicks on a VHD or VHDX file that contains a specially crafted file system, they risk crashing Windows or worse, as illustrated below.</p>
                    
                  </section>
                
                  <section class="block-image_block">
                    
                    
                      <a href="/media/images/win10-fsfuzz-bsod_x9tj7EA.original.png" class="image_large_view cursor-pointer">
                        
                          <img alt="win10-fsfuzz-bsod.png" decoding="async" height="480" loading="lazy" src="/media/images/win10-fsfuzz-bsod_x9tj7EA.max-1280x720.format-webp.webp" width="640">
                        
                      </a>
                      
                    
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h2>Mark of the Web</h2>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="5ht2d"><a href="https://blogs.msdn.microsoft.com/ie/2005/03/07/mark-of-the-web/">Mark of the Web</a> (MOTW) was introduced in Windows XP SP2 and allowed Windows to tag files on the local file system with information about the <a href="https://blogs.msdn.microsoft.com/ie/2005/01/26/ie-security-zones/">Internet Explorer security zone</a> from which the files originated. This MOTW feature has evolved to handle more and more file types and scenarios. The recurring theme is that files that came from the Internet (e.g., a web page or an email) may be dangerous, and therefore should be treated with more caution.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="p2ghk">For example, starting with Microsoft Office 2010, documents tagged with an MOTW that indicated that they came from the Internet are opened in Microsoft Office <a href="https://blogs.technet.microsoft.com/office2010/2009/08/13/protected-view-in-office-2010/">Protected View</a>. Documents in Protected View are restricted in what they can do, thus reducing the attack surface of potentially dangerous documents. Here's what a user might see when opening a document in Protected View:</p>
                    
                  </section>
                
                  <section class="block-image_block">
                    
                    
                      <a href="/media/images/doc_protected_view.original.png" class="image_large_view cursor-pointer">
                        
                          <img alt="some file" decoding="async" height="281" loading="lazy" src="/media/images/doc_protected_view.max-1280x720.format-webp.webp" width="729">
                        
                      </a>
                      
                    
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="rqosj">Starting with Windows 10, <a href="https://docs.microsoft.com/en-us/windows/security/threat-protection/windows-defender-smartscreen/windows-defender-smartscreen-overview">Windows Defender SmartScreen</a> restricts the execution of certain file types if they originated from the Internet. Here's what a user might see when SmartScreen blocks an unsafe executable:</p>
                    
                  </section>
                
                  <section class="block-image_block">
                    
                    
                      <a href="/media/images/smartscreen.original.png" class="image_large_view cursor-pointer">
                        
                          <img alt="some file" decoding="async" height="500" loading="lazy" src="/media/images/smartscreen.max-1280x720.format-webp.webp" width="534">
                        
                      </a>
                      
                    
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="thp3o">How does Windows know if a file originated from the Internet? It uses the MOTW tag associated with the file in question. If Windows Explorer or other compliant ZIP utilities are used to extract the contents of a ZIP file, <a href="https://blogs.msdn.microsoft.com/ieinternals/2011/03/23/understanding-local-machine-zone-lockdown/">each file contained within a ZIP file carries the MOTW of the ZIP file container</a>.</p>
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h2>VHD/VHDX Files and MOTW</h2>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="1ry3b">From a user experience perspective, starting with Windows 8, VHD and VHDX files can have a function similar to ZIP files. That is, the user double-clicks on the file to show its contents in Windows Explorer. The important difference is that the files contained within a VHD or VHDX container do <b>not</b> retain the MOTW of the container file.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="j5bhm">What does this mean from the end user's perspective? Any file contained within a VHD or VHDX file will not receive the same protections that Windows provides against files that originated from the Internet. To help understand what that means, I created a video that demonstrates several differences between a MOTW-tagged (in a ZIP) file and one that does not contain the MOTW tag (in a VHD):</p>
                    
                  </section>
                
                  <section class="block-youtube_block">

  width="100%"
  height="auto"
  class="max-w-2xl aspect-video"
  
    src="https://www.youtube.com/embed/09GDJjBufdQ"
  
  frameborder="0"
  allow="accelerometer;autoplay;encrypted-media;gyroscope;picture-in-picture"
  allowfullscreen
></iframe>

                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h2>VHD/VHDX Files and Antivirus</h2>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="g94pi">I have found no evidence that any currently deployed antivirus software will scan the <b>files contained within</b> a VHD or VHDX file. However, for those running an enterprise, the lack of the ability to scan these files leaves a blind spot for certain files until they arrive at the endpoint. If the contents of VHD and VHDX files are not scanned by email and web gateway security products, those products have no hope of detecting malware contained within VHD or VHDX files.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="rvdiq">I created a VHD that contains the <a href="https://www.eicar.org/?page_id=3950">EICAR anti malware testfile</a> and uploaded that file to VirusTotal. Here are <a href="https://www.virustotal.com/gui/file/41f9fe7478de00caf10b21a6e1f5d77e46c337f2aff5e8a2b07b45607c494a57/detection">the results</a>:</p>
                    
                  </section>
                
                  <section class="block-image_block">
                    
                    
                      <a href="/media/images/eicar_vhd.original.png" class="image_large_view cursor-pointer">
                        
                          <img alt="some file" decoding="async" height="370" loading="lazy" src="/media/images/eicar_vhd.max-1280x720.format-webp.webp" width="1280">
                        
                      </a>
                      
                    
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="pbal1">There is no evidence that any of the scanners configured in VirusTotal scanned the contents of a VHD file.</p>
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h2>ISO and IMG Files</h2>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="irdw9">Malware spread via <a href="https://en.wikipedia.org/wiki/ISO_9660">ISO</a> files is <a href="https://blog.emsisoft.com/en/32373/beware-new-wave-of-malware-spreads-via-iso-file-email-attachments/">already happening in the wild</a>. Just like VHD and VHDX files, the contents of ISO or IMG files do not carry the MOTW of the containing file. And just like VHD and VHDX files, starting with Windows 8, ISO and IMG files can be opened with a double click. Unlike VHD and VHDX files, however, there's a better chance that a deployed antivirus product may detect malware contained in an ISO or IMG file.</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="q71rm">I performed the same EICAR test as above with VirusTotal, but this time the eicar.com file was detected within an ISO file. Here are <a href="https://www.virustotal.com/gui/file/1dd776bd47bcd5a20bb0a4bd3efe6ac856efb847e31c95649df9876af542db26/detection">the results</a>:</p>
                    
                  </section>
                
                  <section class="block-image_block">
                    
                    
                      <a href="/media/images/eicar_iso.original.png" class="image_large_view cursor-pointer">
                        
                          <img alt="some file" decoding="async" height="363" loading="lazy" src="/media/images/eicar_iso.max-1280x720.format-webp.webp" width="1280">
                        
                      </a>
                      
                    
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="6bqft">While these results are not great, there is at least some evidence that some security products will scan the file contents of an ISO file.</p>
                    
                  </section>
                
                  <section class="block-heading_block">
                    
                    
                      

    <h2>Conclusion and Recommendations</h2>

                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="ur1zy">VHD and VHDX files can be dangerous. Due to the combination of kernel-level file system parsing and also lack of MOTW tagging to their contents, allowing VHD or VHDX files to arrive at endpoints increases the risk presented to those systems. The following strategies can help minimize this risk:</p>
                    
                  </section>
                
                  <section class="block-paragraph_block">
                    
                    
                      <p data-block-key="4otc5"></p><ul><li data-block-key="4tjzr">Block VHD, VHDX, IMG, and ISO files at email gateways.</li><li data-block-key="zfp98"><a href="https://gist.github.com/wdormann/fca29e0dcda8b5c0472e73e10c78c3e7">Unregister the VHD, VHDX, IMG, and ISO file extensions in Microsoft Windows Explorer</a>.</li><li data-block-key="2dwma">Restrict VHD, VHDX, IMG, and ISO files at web gateways. (There are some legitimate reasons for these files to be downloaded, so ensure that any restrictions do not block legitimate business needs.)</li></ul>
                    
                  </section>
                
              
            </div>
