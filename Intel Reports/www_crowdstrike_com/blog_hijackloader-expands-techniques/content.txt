<p><span style="	font-weight: 400;
">CrowdStrike researchers have identified a HijackLoader (aka IDAT Loader) sample that employs sophisticated evasion techniques to enhance the complexity of the threat. HijackLoader, an increasingly popular tool among adversaries for deploying additional payloads and tooling, continues to evolve as its developers experiment and enhance its capabilities.</span></p>
<p> </p>
<p><span style="	font-weight: 400;
">In their analysis of a recent HijackLoader sample, CrowdStrike researchers discovered new techniques designed to increase the defense evasion capabilities of the loader. The malware developer used a standard process hollowing technique coupled with an additional trigger that was activated by the parent process writing to a pipe. This new approach has the potential to make defense evasion stealthier.</span></p>
<p> </p>
<p><span style="	font-weight: 400;
">The second technique variation involved an uncommon combination of process doppelgänging and process hollowing techniques. This variation increases the complexity of analysis and the defense evasion capabilities of HijackLoader. Researchers also observed additional unhooking techniques used to hide malicious activity<br/>
.<br/>
This blog focuses on the various evasion techniques employed by HijackLoader at multiple stages of the malware.</span></p>
<p> </p>
<p> </p>
<h2><span style="	font-weight: 400;
">HijackLoader Analysis</span></h2>
<h3><span style="	font-weight: 400;
">Infection Chain Overview</span></h3>
<p><span style="	font-weight: 400;
">The HijackLoader sample CrowdStrike analyzed implements complex multi-stage behavior in which the first-stage executable (<code>streaming_client.exe</code>) deobfuscates an embedded configuration partially used for dynamic API resolution (using <code>PEB_LDR_DATA</code> structure without other API usage) to harden against static analysis.</span></p>
<p><span style="	font-weight: 400;
"><br/>
Afterward, the malware uses WinHTTP APIs to check if the system has an active internet connection by connecting to <code>https&lt;:&gt;//nginx&lt;.&gt;org</code>. If the initial connectivity check succeeds, then execution continues, and it connects to a remote address to download the second-stage configuration blob. If the first URL indicated below fails, the malware iterates through the following list:</span></p>
<p><code> </code></p>
<p><code> </code></p>
<ul><li><code>https[:]//gcdnb[.]pbrd[.]co/images/62DGoPumeB5P.png?o&#61;1</code></li><li><code>https[:]//i[.]imgur[.]com/gyMFSuy.png;</code></li><li><code>https[:]//bitbucket[.]org/bugga-oma1/sispa/downloads/574327927.png</code></li></ul>
<p><span style="	font-weight: 400;
"><br/>
Upon successfully retrieving the second-stage configuration, the malware iterates over the downloaded buffer, checking for the initial bytes of a PNG header. It then proceeds to search for the magic value</span></p>
<p> </p>
<p><code>C6 A5 79 EA</code>, which precedes the <code>XOR</code> key (<code>32 B3 21 A5</code> in this sample) used to decrypt the rest of the configuration blob.</p>
<p> </p>

HijackLoader key retrieving and decrypting (click to enlarge)</div>
<p> </p>
<p><span style="	font-weight: 400;
">Following XOR decryption, the configuration undergoes decompression using the <code>RtlDecompressBuffer</code> API with <code>COMPRESSION_FORMAT_LZNT1</code>. After decompressing the configuration, the malware loads a legitimate Windows DLL specified in the configuration blob (in this sample,</span> <span style="	font-weight: 400;
"><code>C:\Windows\SysWOW64\mshtml.dll</code>).</span></p>
<p><span style="	font-weight: 400;
"><br/>
The second-stage, position-independent shellcode retrieved from the configuration blob is written to the <code>.text</code> section of the newly loaded DLL before being executed. The HijackLoader second-stage, position-independent shellcode then performs some evasion activities (further detailed below) to bypass user mode hooks using Heaven’s Gate and injects subsequent shellcode into <code>cmd.exe</code>.The injection of the third-stage shellcode is accomplished via a variation of process hollowing that results in an injected hollowed <code>mshtml.dll</code> into the newly spawned <code>cmd.exe</code> child process.</span></p>
<p><span style="	font-weight: 400;
"><br/>
The third-stage shellcode implements a user mode hook bypass before injecting the final payload (a Cobalt Strike beacon for this sample) into the child process <code>logagent.exe</code>. The injection mechanism used by the third-stage shellcode leverages the following techniques:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><code>Process Doppelgänging</code> Primitives: This technique is used to hollow a <code>Transacted Section</code> (<code>mshtml.dll</code>) in the remote process to contain the final payload.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><code>Process/DLL Hollowing</code>: This technique is used to inject the fourth-stage shellcode that is responsible for performing evasion prior to passing execution to the final payload within the transacted section from the previous step.</span></li></ul>

"><br/>

<h3><span style="	font-weight: 400;
">Main Evasion Techniques Used by HijackLoader and Shellcode</span></h3>
<p><span style="	font-weight: 400;
">The primary evasion techniques employed by HijackLoader include hook bypass methods such as Heaven&#39;s Gate and unhooking by remapping system DLLs monitored by security products. Additionally, the malware implements variations of process hollowing and an injection technique that leverages transacted hollowing, which combines the transacted section and process doppelgänging techniques with DLL hollowing.</span></p>
<h4><span style="	font-weight: 400;
">Hook Bypass: Heaven’s Gate and Unhooking</span></h4>
<p><span style="	font-weight: 400;
">Like other variants of HijackLoader, this sample implements a user mode hook bypass using Heaven&#39;s Gate (when run in <code>SysWOW64</code>) — this is similar to </span><a href="https://gist.github.com/Cr4sh/76b66b612a5d1dc2c614#file-wow64_call-cpp-L61"><span style="	font-weight: 400;
">existing</span></a><span style="	font-weight: 400;
"> (<code>x64_Syscall</code> function) implementations.</span></p>
<p><span style="	font-weight: 400;
"><br/>
This implementation of Heaven&#39;s Gate is a powerful technique that leads to evading user mode hooks placed in <code>SysWOW64 ntdll.dll</code> by directly calling the syscall instruction in the x64 version of <code>ntdll</code>.</span></p>
<p><span style="	font-weight: 400;
"><br/>
Each call to Heaven&#39;s Gate uses the following as arguments:</span></p>
<p> </p>
<p> </p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The syscall number</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The number of parameters of the syscall</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The parameters (according to the syscall)</span></li></ul>
<p><span style="	font-weight: 400;
">This variation of the shellcode incorporates an additional hook bypass mechanism to elude any user mode hooks that security products may have placed in the x64 <code>ntdll</code>. These hooks are typically used for monitoring both the x32 and x64 <code>nt<br/>
dll</code>.<br/>
<br/>
During this stage, the malware remaps the <code>.text</code> section of x64 <code>ntdll</code> by using Heaven’s Gate to call <code>NtWriteVirtualMemory</code> and <code>NtProtectVirtualMemory</code> to replace the in-memory mapped <code>ntdll</code> with the <code>.text</code> from a fresh <code>ntdll</code> read from the file <code>C:\windows\system32\ntdll.dll</code>. This unhooking technique is also used on the process hosting the final Cobalt Strike payload (<code>logagent.exe</code>) in a final attempt to evade detection.</span></p>
<h4><span style="	font-weight: 400;
">Process Hollowing Variation</span></h4>
<p><span style="	font-weight: 400;
">To inject the subsequent shellcode into the child process <code>cmd.exe</code>, the malware utilizes common process hollowing techniques. This involves mapping the legitimate Windows DLL <code>mshtml.dll</code> into the target process and then replacing its <code>.text</code> section with shellcode. An additional step necessary to trigger the execution of the remote shellcode is detailed in a later section.</span></p>
<p> </p>
<p><span style="	font-weight: 400;
">To set up the hollowing, the sample creates two pipes that are used to redirect the <code>Standard Input</code> and the <code>Standard Output</code> of the child process (specified in the aforementioned configuration blob, <code>C:\windows\syswow64\cmd.exe</code>) by placing the pipes’ handles in a <code>STARTUPINFOW</code> structure spawned with <code>CreateProcessW</code> API.</span></p>
<p> </p>
<p><i><span style="	font-weight: 400;
">One key distinction between this implementation and the typical &#34;standard&#34; process hollowing can be observed here: In standard process hollowing, the child process is usually created in a suspended state. In this case, the child is </span><b>not</b><span style="	font-weight: 400;
"> explicitly created in a suspended state, making it appear less suspicious. Since the child process is waiting for an input from the pipe created previously, its execution is hanging on receiving data from it. Essentially, we can call this an interactive process hollowing variation.</span></i></p>
<p> </p>
<p><span style="	font-weight: 400;
">As a result, the newly spawned <code>cmd.exe</code> will read input from the <code>STDIN</code> pipe, effectively waiting for new commands. At this point, its <code>EIP</code> (<code>Extended Instruction Pointer</code>) is directed toward the return from the <code>NtReadFile</code> syscall.</span></p>
<p> </p>
<p><span style="	font-weight: 400;
">The following section details the steps taken by the second-stage shellcode to set up the child process <code>cmd.exe</code> ultimately used to perform the subsequent injections used to execute the final payload.</span></p>
<p><span style="	font-weight: 400;
"><br/>
The parent process <code>streaming_client.exe</code> initiates an <code>NtDelayExecution</code> to sleep, waiting for <code>cmd.exe</code> to finish loading. Afterward, it reads the legitimate Windows DLL <code>mshtml.dll</code> from the file system and proceeds to load this library into cmd.exe as a shared section. This is accomplished using the Heaven&#39;s Gate technique for:</span></p>
<p> </p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Creating a shared section object using <code>NtCreateSection</code><p> </p>
</span></li><li><span style="	font-weight: 400;
">Mapping that section in the remote <code>cmd.exe</code> using <code>NtMapViewOfSection</code><p> </p>
</span></li></ul>
<p><span style="	font-weight: 400;
">It then replaces the <code>.text</code> section of the <code>mshtml</code> DLL with malicious shellcode by using:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Heaven’s Gate to call <code>NtProtectVirtualMemory</code> on <code>cmd.exe</code> to set <code>RWX</code> permissions on the .text section of the previously mapped section <code>mshtml.dll</code></span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Heaven’s Gate to call <code>NtWriteVirtualMemory</code> on the DLL’s <code>.text</code> section to stomp the module and write the third-stage shellcode<p> </p>
</span></li></ul>
<p><span style="	font-weight: 400;
">Finally, to trigger the execution of the remote injected shellcode, the malware uses:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Heaven’s Gate to suspend (<code>NtSuspendThread</code>) the remote main thread </span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">A new <code>CONTEXT</code> (by using <code>NtGetContextThread</code> and <code>NtSetContextThread</code>) to modify the <code>EIP</code> to point to the previously written shellcode</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Heaven’s Gate to resume (<code>NtResumeThread</code>) the remote main thread of <code>cmd.exe</code></span></li></ul>
<p><span style="	font-weight: 400;
">However, because <code>cmd.exe</code> is waiting for user input from the <code>STDINPUT</code> pipe, the injected shellcode in the new process isn&#39;t actually executed upon the resumption of the thread. The loader must take an additional step:</span></p>
<p> </p>
<p> </p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The parent process <code>streaming_client.exe</code> needs to write (<code>WriteFile</code>) <code>\r\n</code> string to the <code>STDINPUT</code> pipe created previously to send an input to <code>cmd.exe</code> after calling <code>NtResumeThread</code>. This effectively resumes execution of the primary thread at the shellcode’s entry point in the child process <code>cmd.exe</code>.</span></li></ul>
<h4><span style="	font-weight: 400;
">Interactive Process Hollowing Variation: Tradecraft Analysis</span></h4>
<p><span style="	font-weight: 400;
">We have successfully replicated the threadless process hollowing technique to understand how the pipes trigger it. Once the shellcode has been written as described, it needs to be activated. This activation is based on the concept that when a program makes a syscall, the thread waits for the kernel to return a value.</span></p>
<p> </p>
<p><span style="	font-weight: 400;
">In essence, the interactive process hollowing technique involves the following steps:</span></p>
<p> </p>
<ul><li style="	font-weight: 400;
"><b>CreateProcess: </b><span style="	font-weight: 400;
">This step involves spawning the <code>cmd.exe</code> process to inject the malicious code by redirecting <code>STDIN</code> and <code>STDOUT</code> to pipes. Notably, this process isn&#39;t suspended, making it appear less suspicious. Waiting to read input from the pipe, the <code>NtReadFile</code> syscall sets its main thread&#39;s state to <code>Waiting</code> and <code>_KWAIT_REASON</code> to <code>Executive</code>, signifying that it&#39;s awaiting the execution of kernel code operations and their return.<p> </p>
<p> </p>
<p> </p>
</span></li><li style="	font-weight: 400;
"><b>WriteProcessMemory:</b><span style="	font-weight: 400;
"> This is where the shellcode is written into the <code>cmd.exe</code> child process.</span></li><li style="	font-weight: 400;
"><b>SetThreadContext:</b><span style="	font-weight: 400;
"> In this phase, the parent sets the conditions to redirect the execution flow of the <code>cmd.exe</code> child process to the previously written shellcode’s address by modifying the <code>EIP/RIP</code> in the remote thread <code>CONTEXT</code>.</span></li><li style="	font-weight: 400;
"><b>WriteFile:</b><span style="	font-weight: 400;
"> Here, data is written to the <code>STDIN</code> pipe, sending an input to the <code>cmd.exe</code> process. This action resumes the execution of the child process from the <code>NtReadFile</code> operation, thus triggering the execution of the shellcode. Before returning to user space, the kernel is reading and restoring the values saved in the <code>_KTRAP_FRAME</code> structure (containing the <code>EIP/RIP</code> register value) to resume from where the syscall was called. By modifying the <code>CONTEXT</code> in the previous step, the loader hijacks the resuming of the execution toward the shellcode address without the need to suspend and resume the thread, which this technique usually requires.</span></li></ul>
<h4><span style="	font-weight: 400;
">Transacted Hollowing² (Transacted Section/Doppelgänger &#43; Hollowing)</span></h4>
<p><span style="	font-weight: 400;
">The malware writes the final payload in the child process <code>logagent.exe</code> spawned by the third-stage shellcode in <code>cmd.exe</code> by creating a transacted section to be mapped in the remote process. Subsequently, the malware injects fourth-stage shellcode into <code>logagent.exe</code> by loading and hollowing another instance of <code>mshtml.dll</code> into the target process. The injected fourth-stage shellcode performs the aforementioned hook bypass technique before executing the final payload previously allocated by the transacted section.</span></p>
<h4><span style="	font-weight: 400;
">Transacted Section Hollowing</span></h4>
<p><span style="	font-weight: 400;
">Similarly to process doppelgänging, the goal of a transacted section is to create a stealthy malicious section inside a remote process by overwriting the memory of the legitimate process with a transaction.</span> <span style="	font-weight: 400;
">In this sample, the third-stage shellcode executed inside <code>cmd.exe</code> places a malicious transacted section used to host the final payload in the target child process <code>logagent.exe</code>. The shellcode uses the following:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><code>NtCreateTransaction</code> to create a transaction</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><code>RtlSetCurrentTransaction</code> and <code>CreateFileW</code> with a dummy file name to replace the documented<p> </p>
<code>CreateFileTransactedW</code></span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Heaven’s Gate to call <code>NtWriteFile</code> in a loop, writing the final shellcode to the file in 1,024-byte chunks</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Creation of a section backed by that file (Heaven’s Gate call <code>NtCreateSection</code>)</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">A rollback of the previously created section by using Heaven’s Gate to call<p> </p>
<code>NtRollbackTransaction</code></span></li></ul>
<p><span style="	font-weight: 400;
">Existing similar implementations have publicly been observed in </span><a href="https://github.com/hasherezade/transacted_hollowing/blob/main/transacted_file.cpp#L13C29"><span style="	font-weight: 400;
">this project</span></a><span style="	font-weight: 400;
"> that implements transaction hollowing.</span> <span style="	font-weight: 400;
">Once the transacted section has been created, the shellcode generates a function stub at runtime to hide from static analysis. This stub contains a call to the <code>CreateProcessW</code> API to spawn a suspended child process <code>logagent.exe</code> (<code>c50bffbef786eb689358c63fc0585792d174c5e281499f12035afa1ce2ce19c8</code>) that was previously dropped by <code>cmd.exe</code></span></p>
<p> </p>
<p>under the <code>%TEMP%</code> folder. <span style="	font-weight: 400;
">After the target process has been created, the sample uses Heaven’s Gate to:</span></p>
<p> </p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Read its <code>PEB</code> by calling <code>NtReadVirtualMemory</code> to retrieve its base address (<code>0x400000</code>)<p> </p>
</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Unmap the <code>logagent.exe</code> image in the <code>logagent.exe</code> process by using <code>NtUnMapViewofSection</code><p> </p>
</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Hollow the previously created transacted section inside the remote process by remapping the section at the same base address (<code>0x400000</code>) with <code>NtMapViewofSection</code><p> </p>
</span></li></ul>
<h4><span style="	font-weight: 400;
">Process Hollowing</span></h4>
<p><span style="	font-weight: 400;
">After the third-stage shellcode within <code>cmd.exe</code> injects the final Cobalt Strike payload inside the transacted section of the <code>logagent.exe</code> process, it continues by process hollowing the target process to write the fourth shellcode stage ultimately used to execute the final payload (loaded in the transacted section) in the remote process. The third-stage shellcode maps the legitimate Windows DLL <code>C:\Windows\SysWOW64\mshtml.dll</code> in the target process prior to replacing its <code>.text</code> with the fourth-stage shellcode and executing it via <code>NtResumeThread</code>.</span></p>
<p> </p>
<p><span style="	font-weight: 400;
">This additional fourth-stage shellcode written to <code>logagent.exe</code> performs similar evasion activities to the third-stage shellcode executed in <code>cmd.exe</code> (as indicated in the hook bypass section) before passing execution to the final payload.</span></p>

<code>NtUnMapViewOfSection</code>, <code>NtWriteFile</code>, <code>NtReadFile</code>, <code>NtCreateFile</code> and <code>CreatePipe</code>.</td></tr><tr><td>T1562.001</td><td>Impair Defenses: Disable or Modify Tools</td><td>HijackLoader and its stages use Heaven’s Gate and remap x64 ntdll to bypass user space hooks.</td></tr><tr><td>T1055.012</td><td>Process Injection: Process Hollowing</td><td>HijackLoader and its stages implement a process hollowing technique variation to inject in <code>cmd.exe</code> and <code>logagent.exe</code>.</td></tr><tr><td>T1055.013</td><td>Process Injection: Process Doppelgänging</td><td>The HijackLoader shellcode implements a process doppelgänging technique variation
