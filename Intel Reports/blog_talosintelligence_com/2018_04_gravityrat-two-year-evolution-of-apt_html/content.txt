    <h1 class="text-center">GravityRAT - The Two-Year Evolution Of An APT Targeting India</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/paul-rascagneres/">Paul Rascagneres</a>
        </div>

        <br/>
        <time class="post-datetime" datetime="April 26, 2018 11:11">
            Thursday, April 26, 2018 11:11
        </time>

        <div class="m-3">

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <p><i>This blog post is authored by <a href="https://twitter.com/securitybeard">Warren Mercer</a> and <a href="https://twitter.com/r00tbsd">Paul Rascagneres</a>.</i></p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-5E4KVZuTj5E/WuAYwbmY4_I/AAAAAAAAAY8/yCEYZxdvDe0fqeXPtmrrvHiKdDSBT7dIwCLcBGAs/s640/file_733462E4-BF82-4065-B7FF-6A3871E8A954.jpg" class="kg-image" alt loading="lazy"></figure><p><br> <i>Update: 4/30 Since the publication of the blog post, one of the anti-VM capability was commented a lot on Twitter: the detection of Virtual Machines by checking the temperature of the system. We decided to add more details and clarifications concerning this feature. GravityRAT uses a WMI request in order to get the current temperature of the hardware. Here is the output of the query on a physical machine (a Surface Book):</i><br></p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-n20-E-BfwiQ/WucvFfiORUI/AAAAAAAAAag/aWbLDunys9UhfrnvecIOy5z-BeYOLo1JQCLcBGAs/s640/image13.png" class="kg-image" alt loading="lazy"></figure><p><i> The query returns the temperature of 7 thermal zones.</i><br><i> Here is the output on a Virtual Machine executed by Hyper-V on the same hardware:</i><br></p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-ifue71YyAug/WucvJRzXCNI/AAAAAAAAAak/8XC3YzAxV1YQ-bvkJUI3fcxvMcVPP3mEgCLcBGAs/s640/image5.png" class="kg-image" alt loading="lazy"></figure><p><i> The feature is not supported. The malware author used this behavior in order to identify VM (such as Sandboxes).</i></p><p>From our tests and the feedback from several researchers, this monitoring is not supported on Hyper-V, VMWare Fusion, VirtualBox, KVM and XEN. It's important to notice that several recent physical systems do not support it (a researcher reported some Lenovo and Dell hosts did not support this). It means that GravityRAT will consider this physical machine as VMs. Importantly to note this check is not foolproof as we have identified physical hosts which do not report back the temperature, however, it should also be considered a check that is identifying a lot of virtual environments. This is particularly important due to the amount of sandboxing &amp; malware detonation being carried out within virtual environments by researchers.  <br>  <br></p><h2 id="summary-today-cisco-talos-is-uncovering-a-new-piece-of-malware-which-has-remained-under-the-radar-for-the-past-two-years-while-it-continues-to-be-developed-several-weeks-ago-we-identified-the-use-of-the-latest-version-of-this-rat-remote-access-tool-in-this-article-we-will-discuss-the-technical-capabilities-the-evolution-development-and-potential-attribution-of-what-we-are-calling-gravityrat-">Summary <br> Today, Cisco Talos is uncovering a new piece of malware, which has remained under the radar for the past two years while it continues to be developed. Several weeks ago, we identified the use of the latest version of this RAT (Remote Access Tool). In this article, we will discuss the technical capabilities, the evolution, development and potential attribution of what we are calling GravityRAT.</h2><p>GravityRAT has been under ongoing development for at least 18 months, during which the developer has implemented new features. We've seen file exfiltration, remote command execution capability and anti-vm techniques added throughout the life of GravityRAT. This consistent evolution beyond standard remote code execution is concerning because it shows determination and innovation by the actor.</p><p>Throughout our investigation, we observed several malicious documents used to attack victims, which we will discuss. These malicious documents were used by the developer to run several tests on the popular analysis platform VirusTotal. Using VirusTotal allowed the developer to make changes in an attempt to decrease antivirus detection.</p><p>Although GravityRAT has not been previously published or discussed, there was some information from the National Computer Emergency Response Team (CERT) of India describing GravityRAT as being used in <a href="https://nic-cert.nic.in/NIC_CERT/pdf/13-Advisory%20for%20Malicious%20Targeted%20Attack%20Campaign.pdf">targeted attacks against India</a>. Finally, we will discuss specific attribution elements discovered during our research into GravityRAT as we identify specific information, which we believe to be leaked by the developer, such as location, and potentially their first name.</p><h2 id="infection-vectors">Infection Vectors <br></h2><h3 id="malicious-office-documents-the-majority-of-the-malicious-documents-crafted-by-the-malware-author-are-microsoft-office-word-documents-the-attacker-uses-an-embedded-macro-in-order-to-execute-malicious-code-on-the-victim-s-system-the-document-opens-and-appears-as-such-">Malicious Office Documents <br> The majority of the malicious documents crafted by the malware author are Microsoft Office Word documents. The attacker uses an embedded macro in order to execute malicious code on the victim's system. This, however, is a known tactic that a lot of Office-based malware uses. It is an attempt to trick any users who are using Protected Mode on their systems. With this approach, the attacker ensures that there is no direct execution (the executable is executed thanks to scheduled tasks), there's no download of an additional payload, and finally, the author uses the fact that the docx format is an archive in order to include its executable (GravityRAT).</li></ul><h3 id="testing-by-the-author-during-our-tracking-we-identified-several-malicious-documents-submitted-from-this-actor-on-virustotal-for-testing-purposes-they-tested-the-detection-on-macros-by-modifying-them-or-by-executing-the-calc-instead-of-the-malicious-payload-and-the-developers-tried-dynamic-data-exchange-dde-execution-in-the-office-document-this-is-abusing-the-dde-protocol-which-exists-within-microsoft-office-documents-whilst-this-is-a-feature-microsoft-provide-it-is-also-a-feature-that-an-attacker-can-leverage-for-malicious-activity-microsoft-published-mitigation-information-here-previously-the-developer-crafted-office-word-and-excel-documents-to-see-the-detection-in-virustotal-the-authors-tried-to-hide-the-dde-object-in-a-different-part-of-the-document-in-the-main-object-and-the-header-for-example-the-dde-object-simply-executes-microsoft-calc-in-the-detected-sample-here-is-an-example-">Testing By The Author <br> During our tracking, we identified several malicious documents submitted from this actor on VirusTotal for testing purposes. They tested the detection on macros (by modifying them, or by executing the calc instead of the malicious payload) and the developers tried dynamic data exchange (DDE) execution in the Office document. This is abusing the DDE protocol which exists within Microsoft Office documents. Whilst this is a feature Microsoft provide it is also a feature that an attacker can leverage for malicious activity, Microsoft published mitigation information <a href="https://docs.microsoft.com/en-us/security-updates/securityadvisories/2017/4053440">here</a> previously. The developer crafted Office Word and Excel documents to see the detection in VirusTotal. The authors tried to hide the DDE object in a different part of the document — in the main object and the header, for example. The DDE object simply executes Microsoft calc in the detected sample. Here is an example:</h3><pre><code>&lt;?xml version="1.0" encoding="UTF-8" standalone="yes"?&gt;

As explained previously, this Word document had various macros to deliver a final payload.Considering that this was the most recent version of the malware, we decided to ascertain how long this actor had been active, and how their attacks had evolved. We were able to discover four distinct versions of GravityRAT, developed over two years. Next, we will go through what we believe is the development life cycle and feature-addition mission carried out by this developer.<br></h2><h3 id="version-g1-the-malware-author-uses-a-versioning-system-starting-by-the-g-letter-the-oldest-version-we-identified-is-g1-here-is-the-pdb-path-of-the-sample-">Version G1 <br> The malware author uses a versioning system starting by the G letter. The oldest version we identified is G1. Here is the PDB path of the sample:<br></h3><pre><code>f:\F\Windows Work\G1\Adeel's Laptop\G1 Main Virus\systemInterrupts\gravity\obj\x86\Debug\systemInterrupts.pdb</code></pre><p>You can notice the potential first name of the developers: Adeel. Of course, this information can be manipulated by the malware author. This sample was compiled in December 2016. The original filename of the sample was resume.exe.</p><p>The purpose of this version was to steal information on the compromised system:</p><ul><li>MAC Address</li><li>Computer name</li><li>Username</li><li>IP address</li><li>Date</li><li>Steal files with the following extensions: .docx, .doc, .pptx, .ppt, .xlsx, .xls, .rtf and .pdf</li><li>The volumes mapped on the system <br> All this information was then sent to one of the following domains:</li></ul><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-UcHtAEOU3h0/Wt7LxSthMbI/AAAAAAAAAXM/h3Z-qsaR8o0zn2Z4w3WL6wTuA69fN1pigCLcBGAs/s1600/image13.png" class="kg-image" alt loading="lazy"></figure><p>G1 also had the ability to execute commands remotely on the infected host machine at the author's will.</p><h3 id="version-g2-we-identified-a-new-variant-used-in-july-2017-named-g2-here-is-the-pdb-of-the-sample-">Version G2 <br> We identified a new variant used in July 2017 named G2. Here is the PDB of the sample:<br></h3><pre><code>e:\Windows Work\G2\G2 Main Virus\Microsoft Virus Solutions (G2 v5) (Current)\Microsoft Virus Solutions\obj\Debug\Windows Wireless 802.11.pdb</code></pre><p>For this version, the developer modified the architecture of the malware. The main code aims to load and execute two additional .NET binaries stored in the resources of the file:<br></p><ul><li>The first resource is a legitimate open-source library available on <a href="https://github.com/dahall/TaskScheduler">GitHub</a>. It's a .NET wrapper for the Windows Task Scheduler</li><li>The second is the G2 version of GravityRAT <br> This variant shares the same command and control (C2) servers as G1, however, we have an additional 'payload' variable added to G2.</li></ul><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-vxszBdqixjI/Wt7L3rW-XgI/AAAAAAAAAXU/8RDRoFQ-LQMW9pwhjeCGNEBTQOK5KoI0gCLcBGAs/s1600/image11.png" class="kg-image" alt loading="lazy"></figure><p><br> This variant has almost identical capabilities as the previous, except one additional functionality: It collects the CPU information in the Win32_Processor entry via WMI request (Processor ID, Name, Manufacturer and the clock speed). The attacker is most likely using this information as part of an anti-vm attempt within this malware. This is used to try and thwart analysis in virtual environments.</p><p>In a slight change to the previous variant, the new payloads are executed with a Windows Scheduled Task. This would explain the inclusion of the .NET wrapper.</p><p>The analysed sample contained a decoy picture document in the resource section:</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-_c2bthVW74M/Wt7L9esEUBI/AAAAAAAAAXY/Q08XG-usdkk0x0-7gNXacU-IiGmY4WD-gCLcBGAs/s640/image14.jpg" class="kg-image" alt loading="lazy"></figure><p><br></p><h3 id="version-g3-in-august-2017-the-author-of-gravityrat-used-a-new-variant-of-its-malware-g3-here-is-the-pdb-">Version G3 <br> In August 2017, the author of GravityRAT used a new variant of its malware, G3. Here is the PDB:<br></h3><pre><code>F:\Projects\g3\G3 Version 4.0\G3\G3\obj\Release\Intel Core.pdb</code></pre><p>This variant uses the same method as G2, and includes a legitimate library in the resource section. The developers also added additional language support to the library:<br></p><ul><li>German</li><li>Spanish</li><li>French</li><li>Italian</li><li>Chinese</li></ul><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-Eu6_hxb07dg/Wt7MFLv_JHI/AAAAAAAAAXg/0oUTL3M6Z90piQX55oUTJwqo5kNj5QMqQCLcBGAs/s640/image16.png" class="kg-image" alt loading="lazy"></figure><p>The author changed the backend of the C2 server with this variant. The URI changed too, it contains the GravityRAT variant name:</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-PTtQK0yUxyI/Wt7MLUjT11I/AAAAAAAAAXk/2O4wnfCkU1QB73cEDvEeJgBihOOFT533ACLcBGAs/s1600/image6.png" class="kg-image" alt loading="lazy"></figure><p><br> August was also the same month NIC CERT notified potential victims that GravityRAT had been used in a targeted campaign. Given the ongoing development nature of this malware, it meant another variant was most likely due.</p><h3 id="version-gx-the-latest-version-of-gravityrat-was-created-in-december-2017-named-gx-here-is-the-pdb-">Version GX <br> The latest version of GravityRAT was created in December 2017 named GX. Here is the PDB:<br></h3><pre><code>C:\Users\The Invincible\Desktop\gx\gx-current-program\LSASS\obj\Release\LSASS.pdb</code></pre><p>This version is the most advanced variant of GravityRAT. Throughout the evolution, we saw this malware embedding open-source legitimate .NET libraries (for schedule tasks, compression, encryption, .NET loading). It contains a resource named "important." This is an archive with a password.</p><p>This variant has the same features as before, but this time, some new features are added:</p><ul><li>It collects open ports on the victim host by running the netstat command</li><li>It lists all the running processes</li><li>It lists available services on the system</li><li>It exfiltrates .ppt and .pptx file, in addition to the extension mentioned in the G1 variant</li><li>If a USB key is connected on the system, the malware steals the file based on an extension list</li><li>It supports file encryption (AES with the key "lolomycin2017")</li><li>It collects information on the account (account type, description, domain name, full name, SID and status)</li><li>It checks if the system is a virtual machine with several techniques</li></ul><p>The developer implemented a total of seven techniques to identify if the compromised system is a virtual machine.</p><p>The first technique consists of looking at any additional tools used by the hypervisor that are installed on the system (by checking a registry key):</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-sfrVtRiOK9Q/Wt7MR1S3QjI/AAAAAAAAAXs/RhwXts3IMcoqu6yWifrss8EAYoAPEJoiACLcBGAs/s640/image10.png" class="kg-image" alt loading="lazy"></figure><p><br> The second technique uses a WMI request to the BIOS version (Win32_BIOS entry). If the response contains: "VMware", "Virtual", "XEN", "Xen" or "A M I" the system is considered as a virtual machine. Additionally, the malware checks the SerialNumber and the version of the BIOS.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-RHN_i1EGAkw/Wt7Mb5NhnXI/AAAAAAAAAX0/cuEBsYTBZTExwsIqyjIofDN9AHmFuweTwCLcBGAs/s640/image5.png" class="kg-image" alt loading="lazy"></figure><p><br> The third technique uses the Win32_Computer entry in WMI. It checks if the manufacturer contains "VIRTUAL", "VMWARE" or "VirtualBox".</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-_bZLneKsNUQ/Wt7MhjEWrMI/AAAAAAAAAX8/jABBWa-hl58U6j0Wg703RBXWeZx4kRKFQCLcBGAs/s640/image8.png" class="kg-image" alt loading="lazy"></figure><p>The fourth technique checks the Processor ID of the system.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-mS_Iu7onLK8/Wt7MmiSTQFI/AAAAAAAAAYA/Gud1lq9dLokn83EnajpcOtOy4MQt2BPgQCLcBGAs/s640/image12.png" class="kg-image" alt loading="lazy"></figure><p>The fifth technique counts the number of cores in the infected system (the author expects more than one core)</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-Zt5D7x7XDyg/Wt7Ms97CSLI/AAAAAAAAAYE/leAGaijJe5sl3yIhY4p2CiF2K8wkNWbHwCLcBGAs/s640/image9.png" class="kg-image" alt loading="lazy"></figure><p>The sixth technique checks the current CPU temperature of the system (the MSAcpi_ThermalZoneTemperature entry). Indeed, some hypervisors (VMWare, VirtualBox and Hyper-V) do not support temperature check. The WMI request simply replies "not supported". This behaviour can be used to detect if the targeted system is a real machine.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-lyyUYEU8QMc/Wt7MyhgiXZI/AAAAAAAAAYM/ZnRY4H8SgfEu0ozJngwLhx48VEYeRBuzQCLcBGAs/s640/image7.png" class="kg-image" alt loading="lazy"></figure><p>The last technique uses the MAC Address of the infected system. If the MAC Address starts by a well-known hexadecimal number, the system is identified as a virtual machine.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-AG6M5oChPus/Wt7M2wcMVyI/AAAAAAAAAYU/NtT04MdbZmYit7d7wBZPVZZYfBiBsPNBwCLcBGAs/s1600/image15.png" class="kg-image" alt loading="lazy"></figure><p>The C2 servers communication is performed in HTTP as it did previously. The variant version of GX is used in the URI. The C2 servers we can see are shared with the previous variants:</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-bS3NywC79sg/Wt7M98eaciI/AAAAAAAAAYc/a7RaSeWNRNwNx8eGavpn8-Bk94SCc4vRQCLcBGAs/s1600/image1.png" class="kg-image" alt loading="lazy"></figure><h2 id="what-we-know-about-the-author-below-we-will-present-evidence-that-we-have-obtained-regarding-the-attacker-and-the-associated-malware-obviously-attribution-is-a-complex-field-the-developers-could-be-using-a-proxy-or-a-vpn-in-order-to-fake-the-origin-of-the-submission-but-we-will-still-simply-present-some-facts-concerning-this-actor-">What we know about the author <br> Below, we will present evidence that we have obtained regarding the attacker and the associated malware. Obviously, attribution is a complex field. The developers could be using a proxy or a VPN in order to fake the origin of the submission. But, we will still simply present some facts concerning this actor.</h2><p>The developer used at least two different usernames in the past two years: "The Invincible" and "TheMartian." In the oldest version of GravityRAT, the attacker potentially leaked his or her first name in the PDB: "Adeel" — the path contained "Adeel's Laptop". Additionally, all the malicious Office documents, and more specifically the documents used to test anti-virus on VirusTotal, were submitted from Pakistan. One of the four PE files in the IOCs section was sent from Pakistan, too.</p><p>In August 2017, NIC CERT published an <a href="https://nic-cert.nic.in/NIC_CERT/pdf/13-Advisory%20for%20Malicious%20Targeted%20Attack%20Campaign.pdf">advisory </a>about malicious targeted campaigns. This advisory mentions the C2 server infrastructure of GravityRAT, which means the GravityRAT author likely targeted Indian entities/organisations. By leveraging Cisco Umbrella and using the Investigate tool, we were able to determine that across all of the C2 domains listed, we saw a large influx of traffic originating from India, as evidenced by NIC CERT, all of the C2 domains were at least 50 percent requested by Indian IP infrastructure. It is possible that some of the non-Indian IP space requests may artefacts be due to our own research.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-xsP6ao_8HoI/Wt7NIpkvteI/AAAAAAAAAYk/CLLFsL828_AzDbFbfaMyh_dbDiOpho4IgCLcBGAs/s640/image3.png" class="kg-image" alt loading="lazy"></figure><h2 id="conclusion-this-actor-is-probably-not-the-most-advanced-actor-we-ve-seen-but-he-or-she-managed-to-stay-under-the-radar-since-2016-they-worked-on-malicious-code-and-produced-four-variants-each-new-variant-included-new-features-the-developer-used-the-same-c2-infrastructure-all-this-time-the-developer-was-clever-enough-to-keep-this-infrastructure-safe-and-not-have-it-blocklisted-by-a-security-vendor-the-actor-took-their-time-to-ensure-they-were-not-within-a-virtual-environment-to-avoid-analysis-however-they-did-not-take-any-time-at-all-to-attempt-to-obfuscate-their-net-code-the-code-was-largely-trivial-to-reverse-engineer-which-meant-static-analysis-was-an-easy-option-for-this-piece-of-malware-">Conclusion <br> This actor is probably not the most advanced actor we've seen. But he or she managed to stay under the radar since 2016. They worked on malicious code, and produced four variants. Each new variant included new features. The developer used the same C2 infrastructure all this time. The developer was clever enough to keep this infrastructure safe, and not have it blocklisted by a security vendor. The actor took their time to ensure they were not within a virtual environment to avoid analysis. However, they did not take any time at all to attempt to obfuscate their .NET code. The code was largely trivial to reverse engineer, which meant static analysis was an easy option for this piece of malware.</h2><p>NIC CERT published an advisory about this actor, which suggest they targeted Indian entities and organizations.</p><p>The author leaked information within the samples (i.e. Adeel) and on the VirusTotal platform. Thanks to this information, we we able to understand how they tested malicious documents in order to decrease detection ratios across many popular engines. During this testing period, all the samples were uploaded from Pakistan to VirusTotal.</p><h2 id="coverage-additional-ways-our-customers-can-detect-and-block-this-threat-are-listed-below-">Coverage
