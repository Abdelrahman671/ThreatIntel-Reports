<h1><span id="hs_cos_wrapper_name" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="text">Malicious npm package targets AWS users</span></h1>
<div id="hs_cos_wrapper_module_1577203992502740" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module">
<h2 class="post-header__secondary-head blog__post-secondary-h2">The history of the package is a lesson in why tracking open source threats is such a challenge — and highlights the value of RL's new Spectra Assure Community.</h2>
</div>
</div>

<div class="blog__post-content html-content">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/blog-A-Lurking-NPM-Package.webp?width=1400&amp;height=732&amp;name=blog-A-Lurking-NPM-Package.webp" width="1400" height="732" loading="lazy" alt="blog-A-Lurking-NPM-Package" style="height: auto; max-width: 100%; width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/blog-A-Lurking-NPM-Package.webp?width=700&amp;height=366&amp;name=blog-A-Lurking-NPM-Package.webp 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/blog-A-Lurking-NPM-Package.webp?width=1400&amp;height=732&amp;name=blog-A-Lurking-NPM-Package.webp 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/blog-A-Lurking-NPM-Package.webp?width=2100&amp;height=1098&amp;name=blog-A-Lurking-NPM-Package.webp 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/blog-A-Lurking-NPM-Package.webp?width=2800&amp;height=1464&amp;name=blog-A-Lurking-NPM-Package.webp 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/blog-A-Lurking-NPM-Package.webp?width=3500&amp;height=1830&amp;name=blog-A-Lurking-NPM-Package.webp 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/blog-A-Lurking-NPM-Package.webp?width=4200&amp;height=2196&amp;name=blog-A-Lurking-NPM-Package.webp 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></p>
<p>ReversingLabs researchers have made it a priority to monitor public, open source repositories for malicious packages that may lurk on them in recent years. The number and frequency of malicious packages has increased steadily as malicious actors turn to software supply chains for an easy route into hundreds, thousands or even tens of thousands of protected IT environments.&nbsp;<br><br>Many of the packages the RL threat research team has found are what we term “infostealers” — malicious software that is designed to grab sensitive information. For example, our team recently <a href="/blog/suspicious-nuget-package-grabs-data-from-industrial-systems" rel="noopener">wrote about a package discovered on the NuGet repository</a>, <span style="font-style: italic;">SqzrFramework480.dll</span>, that contained features for capturing a screenshot, sending ping packets, as well as opening a socket and sending data over it. The package contained information — including a graphic — that suggested it was targeting BOZHON Precision Industry Technology Co., a China-based firm that does industrial and digital equipment manufacturing.&nbsp;<br><br>RL researchers frequently discover downloaders and malicious applications designed to fetch other software — so called second (or third) “stages” that are often open source "infostealers" or simple “beacon-backs” that red team operators use to plant a flag within target environments to prove they were successful in breaching defenses.&nbsp;<br><br>Whatever their purpose, all of the packages we detect as malicious are reported to the relevant maintainers of the affected repositories. In our team's experience, they are removed shortly after. That’s good for developers and development teams, as it removes potentially damaging code from public repositories where it might find its way onto developers’ systems, or even integrated into legitimate applications and services.&nbsp;<br><br>But there’s a problem with this approach: The sheer scale of the open source ecosystem is so large that it is impossible for any single group of researchers to keep abreast of and police massive repositories like Python Package Index (PyPI), npm, NuGet, VS Code Marketplace and others. On any given day, RL researchers, using RL <a href="/products/software-supply-chain-security" rel="noopener" target="_blank">Spectra Assure platform</a>, identify dozens of malicious packages spread across these repositories, but far more packages may lurk undetected — suspicious but not overtly malicious.&nbsp;</p>
<!--more-->
<h2 style="font-size: 24px; font-weight: normal;">Wanted: A health warning for open source packages</h2>
<p>What developers and development organizations need is something like a health warning for open source packages — an easy way to check dangerous ingredients such as malicious or suspicious content or behaviors before the packages are deployed.&nbsp;<br><br>That’s why RL is introducing <a href="https://secure.software/" rel="noopener" target="_blank">Spectra Assure Community</a>, which provides a free open source risk assessment for the most popular package repositories, including npm, PyPI, and RubyGems. The portal provides a comprehensive risk assessment for software packages, offering visibility into threats, security, and compliance issues.&nbsp;<br><br>Spectra Assure Community helps prevent software supply chain breaches and attacks by providing unprecedented visibility into your software packages. A recent suspicious package the research team discovered on npm's Registry, provides a great example of how it will work.</p>
<h2 style="font-size: 24px; font-weight: normal;">Discovered: A downloading uploader</h2>
<p>A few days ago, our research team identified a &nbsp;malicious package lurking on npm. The package was named <a href="https://secure.software/npm/packages/legacyreact-aws-s3-typescript" rel="noopener" style="font-style: italic;">legacyreact-aws-s3-typescript</a> and “lurking” is really the right word to use. Why? Well, the <a href="https://secure.software/npm/packages/legacyreact-aws-s3-typescript" rel="noopener" style="font-style: italic;">legacyreact-aws-s3-typescript</a> had been publicly accessible on npm but dormant for about four months. It mimics another popular legitimate npm package with over a quarter of a million downloads: <a href="https://secure.software/npm/packages/react-aws-s3-typescript" rel="noopener" style="font-style: italic;">react-aws-s3-typescript</a>, which is designed to facilitate the uploading of files to Amazon S3 Buckets (AWS) directly using a <a href="https://reactnative.dev/" rel="noopener" target="_blank">React</a> typescript template. (React is an open source user interface software framework.) &nbsp;</p>
<h2 style="font-size: 24px; font-weight: normal;">Post install scripts demand a closer look</h2>
<p>The package <span style="font-style: italic;">legacyreact-aws-s3-typescript@1.2.4</span> was found as part of daily threat hunting activities using <a href="/products/software-supply-chain-security" rel="noopener">RL Spectra Assure platform</a>. The package was initially flagged as suspicious because of unusual activity found inside the package’s postinstall script (Figure 1). Specifically, after the package is installed on the user’s computer, a postinstall script runs and downloads an ELF format file and executes it.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/a-lurking-npm-blog-figure_1.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_1.png?width=1400&amp;name=a-lurking-npm-blog-figure_1.png" alt="postinstall script from version 1.2.4. fetching second stage ELF file" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_1.png?width=700&amp;name=a-lurking-npm-blog-figure_1.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_1.png?width=1400&amp;name=a-lurking-npm-blog-figure_1.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_1.png?width=2100&amp;name=a-lurking-npm-blog-figure_1.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_1.png?width=2800&amp;name=a-lurking-npm-blog-figure_1.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_1.png?width=3500&amp;name=a-lurking-npm-blog-figure_1.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_1.png?width=4200&amp;name=a-lurking-npm-blog-figure_1.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>
<p style="text-align: center; font-weight: bold; font-size: 16px;">Figure 1: A postinstall script from version 1.2.4. fetching second stage ELF file</p>
<p>Post- and pre-install scripts are not uncommon and do not always connect with malicious activities. There are many legitimate packages present on npm that use install scripts for common software development purposes. However, packages that have install scripts demand heavier scrutiny as there is a possibility the scripts might be malicious.&nbsp;<br><br>That was the case with this package as well. The ELF file downloaded by the script was still fetchable at the time the package was detected. When we analyzed that second stage software, it turned out to be a simple backdoor. It opened a socket, connected to the IP address 91[.]238[.]181[.]250, and then executed and received commands through /bin/sh. RL researchers immediately classified the package as malicious, and reported it to npm.&nbsp;</p>
<h2 style="font-size: 24px; font-weight: normal;">A legacy of deception</h2>
<p>The behaviors observed in the <em>legacyreact-aws-s3-typescript </em>package aren’t uncommon among the malicious open source packages RL has uncovered and documented. But the history of the package is a lesson in why tracking open source threats is such a challenge.&nbsp;<br><br>That’s because the latest version of the package, version 1.2.4, &nbsp;is not the only one published to npm. In February 2024, three versions of this package were published: 1.1.5, 1.1.6 and 1.1.7. Based on our analysis, all of them were completely clean, without any malicious activity that was present in the version 1.2.4. Those earlier editions of the package contained postinstall scripts as well. Even though some of them were unusual like invoking Windows 10’s calculator or being completely empty, nothing out of the ordinary happened when those packages were installed and those scripts were executed.&nbsp;<br><br>Things changed three months later, however. In May, three versions of <em>legacyreact-aws-s3-typescript</em>, 1.1.9, 1.2.1, and 1.2.2 were published to npm briefly, after which they were removed almost immediately.&nbsp;<br><br>It is unclear why they were removed. We suspect they were removed by the author of the packages, though we are unaware of the reason they were published and quickly removed.<br><br>However, RL researchers managed to get a hold of versions 1.2.1 and 1.2.2 and found that both were malicious, and had the same malicious functionality as was later published version 1.2.4. The team can’t say with any certainty whether version 1.1.9 was also malicious, but we suspect that it was, as it was published and removed alongside two other, malicious updates.&nbsp;<br><br>A few weeks later and four months after the original, non-malicious <em>legacyreact-aws-s3-typescript</em> package was first published, version 1.2.4 was published and contained the malicious code described above.This update wasn’t removed and we detected it shortly after it was published to npm.</p>
<h2 style="font-size: 24px; font-weight: normal;">Typosquatting: A red flag for open source</h2>
<p>Code analysis aside, there were some other features of the <em>legacyreact-aws-s3-typescript</em> package that became apparent while looking at the package's npm page and suggested that something was amiss.<br><br>At the top of that list was the appearance of typosquatting, in which a malicious or suspicious package adopts a name that is close — or identical to — a popular, widely used package or file. Looking at the <em>legacyreact-aws-s3-typescript’s</em> npm landing page and digging into the <em>README.md</em> and linked GitHub page both pointed to a legitimate npm package, named <em>react-aws-s3-typescript</em>. With around 255.29K downloads, that package is also advertised as a tool to upload files into AWS S3 Buckets directly using the react typescript template. Upon closer examination, it was clear that the code from the legitimate package was copied and pasted into the malicious one. That resulted in two npm landing pages, one belonging to a legitimate package and another to the malicious package, with both looking exactly the same.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/a-lurking-npm-blog-figure_2.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_2.png?width=1400&amp;name=a-lurking-npm-blog-figure_2.png" alt="legacyreact-aws-s3-typescript’s npm page" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_2.png?width=700&amp;name=a-lurking-npm-blog-figure_2.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_2.png?width=1400&amp;name=a-lurking-npm-blog-figure_2.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_2.png?width=2100&amp;name=a-lurking-npm-blog-figure_2.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_2.png?width=2800&amp;name=a-lurking-npm-blog-figure_2.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_2.png?width=3500&amp;name=a-lurking-npm-blog-figure_2.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_2.png?width=4200&amp;name=a-lurking-npm-blog-figure_2.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>
<p style="font-weight: bold; text-align: center; font-size: 16px;">Figure 2: legacyreact-aws-s3-typescript’s npm page</p>
<p>The malicious actor behind the <em>legacyreact-aws-s3-typescript</em> package took further steps to try to convince developers their malicious package is legitimate and safe to use. For example, the last published version of <em>react-aws-s3-typescript</em> is 1.1.5 and it was published two years ago. The <em>first </em>version of malicious package <em>legacyreact-aws-s3-typescript </em>is also 1.1.5 and it was published about four&nbsp;months ago. What’s at work is an effort by the malicious actors to pass their package off as a continuation of — and update to the <em>react-aws-s3-typescript</em> package. Having the word “legacy” in the name only further validates that theory.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/a-lurking-npm-blog-figure_3.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_3.png?width=1400&amp;name=a-lurking-npm-blog-figure_3.png" alt="legacyreact-aws-s3-typescript’s versions previously available on npm" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_3.png?width=700&amp;name=a-lurking-npm-blog-figure_3.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_3.png?width=1400&amp;name=a-lurking-npm-blog-figure_3.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_3.png?width=2100&amp;name=a-lurking-npm-blog-figure_3.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_3.png?width=2800&amp;name=a-lurking-npm-blog-figure_3.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_3.png?width=3500&amp;name=a-lurking-npm-blog-figure_3.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/a-lurking-npm-blog-figure_3.png?width=4200&amp;name=a-lurking-npm-blog-figure_3.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>

<h2 style="font-size: 24px; font-weight: normal;">Spectra Assure Community: A portal for OSS truth</h2>
<p>Of course, picking out these subtle, suspicious indicators isn’t easy — especially for developers and development teams under pressure to meet aggressive development and release deadlines. That’s why RL is introducing <a href="https://secure.software/" rel="noopener">Spectra Assure Community,</a> a free portal where developers can access all the information on packages we find on platforms like npm, PyPI and RubyGems through the RL research team's regular scanning and threat hunting efforts.&nbsp;<br><br>For the packages like <span style="font-style: italic;">legacyreact-aws-s3-typescript</span>, Spectra Assure Community provides a vital assessment of the trustworthiness of the underlying code in the package.&nbsp;<br><br>A portion of information Spectra Assure Community provides can be seen in Figure 4. As you can see, the scan of the package immediately detected the presence of malware in the latest update as well as a critical, unpatched vulnerability.</p>

<p style="font-weight: bold; font-size: 16px; text-align: center;">Figure 4: <a href="https://secure.software/npm/packages/legacyreact-aws-s3-typescript/1.2.4" rel="noopener" target="_blank">legacyreact-aws-s3-typescript page on Spectra Assure Community page</a></p>
<p>Malicious packages found on public repositories are unfortunately nothing new — and they’re becoming more common each day.&nbsp;<br><br>RL's latest research uncovered a package on the npm repository mimicking a very popular legitimate npm package that was dormant for 4 months before the maintainer published a string of new, malicious versions containing downloader functionality, culminating in version 1.2.4, which RL downloaded and analyzed.&nbsp;<br><br>The RL research team's latest discovery shows that threats lurking on public repositories are still prevalent, and developers and software publishers alike need to pay close attention to the risk brought in by third-party, open source components. While that includes the risks posed by vulnerable code, that’s not the full extent of open source risk. As our latest research shows, hasty decisions by developers risk introducing malicious code into otherwise clean products. That’s why it’s necessary for developers to conduct security assessments that can verify the integrity and quality of public, open source libraries for safety before they are used.&nbsp;<br><br>With the launch of the <a href="https://secure.software/" rel="noopener">Spectra Assure Community</a> developers and software publishers now have a powerful new tool to help them do just that quickly, and easily. Leveraging RL’s award-winning Spectra Assure software supply chain security solution, Spectra Assure Community enables developers, maintainers, repo managers, and engineering teams among to assess more than five million code packages from open source repositories for malicious code, code tampering, suspicious behaviors, known vulnerabilities, license compliance issues, exposed secrets, and overall package health.&nbsp;<br><br><span style="font-style: italic;">To learn more about Spectra Assure Community or check your open source software package for any threat, visit <a href="https://secure.software/" rel="noopener" target="_blank">secure.software</a>.</span></p>
