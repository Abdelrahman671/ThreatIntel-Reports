 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13" id="h.onqcwsq9k09z"><span class="WuvYfpVbPj-c4 WuvYfpVbPj-c22">Guest post by Nick Galloway, Senior Security Engineer, 20% time on Project Zero</span></p><br><br>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span>Late in 2023, while working on a 20% project with Project Zero, I found an integer overflow in the </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://code.videolan.org/videolan/dav1d&amp;sa=D&amp;source=editors&amp;ust=1727869833018829&amp;usg=AOvVaw0OeT8v646Gvjwd0m7jBsCT">dav1d</a></span><span>&nbsp;AV1 video decoder. That integer overflow leads to </span><span>an out-of-bounds write</span><span>&nbsp;to memory. </span><span>Dav1d</span><span>&nbsp;1.4.0 </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://code.videolan.org/videolan/dav1d/-/commit/2b475307dc11be9a1c3cc4358102c76a7f386a51&amp;sa=D&amp;source=editors&amp;ust=1727869833019194&amp;usg=AOvVaw3804XIY0C7CGB-UUn5NwNA">patched</a></span><span>&nbsp;this, and it was assigned </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://bugs.chromium.org/p/project-zero/issues/detail?id%3D2502&amp;sa=D&amp;source=editors&amp;ust=1727869833019361&amp;usg=AOvVaw2q-_2_XpPz6efrZdkggxY_">CVE-2024-1580</a></span><span>. After the disclosure, I received some questions about how this issue was discovered, since dav1d is already being fuzzed by at least </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://github.com/google/oss-fuzz&amp;sa=D&amp;source=editors&amp;ust=1727869833019505&amp;usg=AOvVaw1deHOW6xxNlIZyxGlGjMct">oss-fuzz</a></span><span>.</span><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">&nbsp;This blog post explains what happened. It&rsquo;s a useful case study in how to construct fuzzers to exercise as much code as possible. But first, some background...</span></p><h1 class="BvmDwwzBKL-c3" id="h.x6s3sza9d3l"><span class="BvmDwwzBKL-c19 BvmDwwzBKL-c9">Background</span></h1><h2 class="BvmDwwzBKL-c10" id="h.ir0rv4c0p8xk"><span class="BvmDwwzBKL-c21 BvmDwwzBKL-c9">Dav1d</span></h2>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://code.videolan.org/videolan/dav1d&amp;sa=D&amp;source=editors&amp;ust=1727869833020012&amp;usg=AOvVaw1NMeiqpYWceoZIYBeSFatd">Dav1d</a></span><span>&nbsp;is a highly-optimized AV1 decoder. </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://en.wikipedia.org/wiki/AV1&amp;sa=D&amp;source=editors&amp;ust=1727869833020113&amp;usg=AOvVaw1ETEEXIx_hheFr4FUEaMVU">AV1</a></span><span>&nbsp;is a royalty-free video coding format developed by the Alliance for Open Media, and achieves improved data compression compared to older formats. AV1 is </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://caniuse.com/av1&amp;sa=D&amp;source=editors&amp;ust=1727869833020194&amp;usg=AOvVaw3h5W2dCTCZKbFY_sl8V3t0">widely supported</a></span><span>&nbsp;by web browsers, and a significant parsing vulnerability in AV1 decoders could be used as part of an attack to gain remote code execution. In the right context, where AV1 is parsed in a received message, this could allow a 0-click exploit. Testing some popular messaging clients by sending AV1 videos and AVIF images (which uses the AV1 codec) yielded the following results:</span></p><ul style="padding: 0;" class="lst-kix_mfum7ompb01z-0 start"><li style="margin-left: 46pt;" class="c2 c6 li-bullet-0"><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">AVIF images are displayed in iMessage</span></li><li style="margin-left: 46pt;" class="c2 c6 li-bullet-0"><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">AVIF images are NOT displayed in Android Messages when sent as an MMS</span></li><li style="margin-left: 46pt;" class="c2 c6 li-bullet-0"><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">AVIF images are displayed in Google Chat</span></li><li style="margin-left: 46pt;" class="c2 c6 li-bullet-0"><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">AV1 videos are not immediately displayed in Google Chat, but can be downloaded by the receiver and eventually can be played after being downscaled</span></li></ul>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13 BvmDwwzBKL-c12"><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9"></span></p>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span>Dav1d is written primarily in C and notably has different code paths for different architectures. There are x86, x86-64, ppc, riscv, arm32, and arm64 code paths in the repository, most of these containing optimized assembly. As noted in their roadmap, support for some of these is ongoing work, but at least ARMv7, ARMv8, and x86-64 have been thoroughly tested in the field. Based on this being a library written in C and assembly, as well as dav1d&rsquo;s ubiquitous support in web browsers, I</span><span>&nbsp;might expect it already has excellent fuzzing coverage from multiple sources</span><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">.</span></p><h2 class="BvmDwwzBKL-c10" id="h.xkeplzph3tva"><span class="BvmDwwzBKL-c9 BvmDwwzBKL-c21">The integer overflow</span></h2>

 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">Those pointers are later written to, resulting in an out of bounds write to memory. Two test cases are provided with the bug, the first of which (poc1.obu) will result in an address which is outside the valid range of addresses, and so might not be exploitable. The other test case (poc2.obu) enables high bit depth mode and so has higher memory requirements, but results in pointers that are within the normal range of addresses, and so is more likely to be useful in an exploit.</span></p><h2 class="BvmDwwzBKL-c10" id="h.otfogf9v5l5c"><span class="BvmDwwzBKL-c21 BvmDwwzBKL-c9">Fuzzing Space Definition</span></h2>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span>A fuzzer&rsquo;s success is typically measured by &ldquo;coverage&rdquo;, where the fuzz target&#39;s execution is traced to examine which lines of assembly code have been covered. When I talk about the &ldquo;fuzzing space&rdquo;, I specifically mean space in the sense of a </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://en.wikipedia.org/wiki/Space_(mathematics)&amp;sa=D&amp;source=editors&amp;ust=1727869833031893&amp;usg=AOvVaw0_mQO6SzJFsxVzIRqaI_Ph">mathematical space</a></span><span>, where the set of lines of code that are executed by a given set of test cases is something we would like to maximize. In other words, a good fuzzer will execute as many lines of code as possible with the smallest possible set of test cases. To fully define the space we would also consider the fuzzing engine that generates test cases, the initial seed corpus, and the various configurations and architectures supported by the code to be fuzzed. </span></p><h1 class="BvmDwwzBKL-c3" id="h.ku7mn4tierjd"><span>Modified Dav1d </span><span>Fuzzer</span></h1>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span>The dav1d fuzzer in oss-fuzz at the time I was looking at dav1d is </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://github.com/google/oss-fuzz/tree/dc543015c47f5efe6a23fa9d555a10c20ec531bf/projects/dav1d&amp;sa=D&amp;source=editors&amp;ust=1727869833032315&amp;usg=AOvVaw1rCRY6jwO3dC0odT9NmnTM">visible on GitHub</a></span><span>. This contains build instructions and a dockerfile for oss-fuzz to run this at scale. The fuzzer implementation is in the dav1d source repository. The </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://code.videolan.org/videolan/dav1d/-/blob/8ddb28e5a726f185186c7bbd749ca3499c639c15/tests/libfuzzer/meson.build&amp;sa=D&amp;source=editors&amp;ust=1727869833032482&amp;usg=AOvVaw2JbV-f3E9v_VebU3Loe3dI">meson.build file</a></span><span>&nbsp;s</span><span>hows a couple of configurations, one for building dav1d_fuzzer and the other for building dav1d_fuzzer_mt, which additionally defines </span><span class="BvmDwwzBKL-c11">DAV1D_MT_FUZZING</span><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">.</span></p>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13 BvmDwwzBKL-c12"><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9"></span></p>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span>The fuzzing code is written in C, found in </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://code.videolan.org/videolan/dav1d/-/blob/8ddb28e5a726f185186c7bbd749ca3499c639c15/tests/libfuzzer/dav1d_fuzzer.c%23L93&amp;sa=D&amp;source=editors&amp;ust=1727869833032844&amp;usg=AOvVaw1PIWcyouvclPaJEhZrhXmx">dav1d_fuzzer.c</a></span><span>. The fuzzer implements LLVMFuzzerTestOneInput, </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://llvm.org/docs/LibFuzzer.html%23fuzz-target&amp;sa=D&amp;source=editors&amp;ust=1727869833032939&amp;usg=AOvVaw2b2IgFaVnTqQSdA36JTZGc">the standard way to use libfuzzer</a></span><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">.  <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span class="BvmDwwzBKL-c9 BvmDwwzBKL-c14">There&rsquo;s an understandable desire to avoid reporting errors when a fuzzer triggers huge allocations.  <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span>I also tested a number of other avenues that did not lead to discovering vulnerabilities. One example is fuzzing on ARM, which I had expected might result in a vulnerability due to it </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://github.com/google/oss-fuzz/blob/dc543015c47f5efe6a23fa9d555a10c20ec531bf/projects/dav1d/project.yaml%23L18&amp;sa=D&amp;source=editors&amp;ust=1727869833049213&amp;usg=AOvVaw3RM5rWPeDjKeykKUR6xmK3">not being covered by OSS-Fuzz</a></span><span>. Despite this not uncovering anything, I still believe that it&rsquo;s worthwhile to run fuzz tests on other architectures </span><span>when possible</span><span>, especially when the target has different code paths and optimized assembly for different architectures, as is the case with dav1d.</span></p>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13 BvmDwwzBKL-c12"><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9"></span></p><h1 class="BvmDwwzBKL-c3" id="h.a2p7c3czxq8x"><span class="BvmDwwzBKL-c19 BvmDwwzBKL-c9">Conclusion</span></h1>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span>The ultimate lesson I took away from this is that a fruitful area to look for vulnerabilities is artificial limits within a fuzzer</span><span>.</span><span>&nbsp;By setting a relatively small </span><span class="BvmDwwzBKL-c11">frame_size_limit</span><span>, the dav1d fuzzer missed the integer overflow. There is a good reason for this limit, which is that </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://github.com/google/oss-fuzz/blob/master/docs/faq.md%23what-are-the-specs-on-your-machines&amp;sa=D&amp;source=editors&amp;ust=1727869833049749&amp;usg=AOvVaw0l6rO0pb_D5DmlkvkIUgv8">oss-fuzz only supports 2.5GB of RAM</a></span><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">. This highlights a tradeoff for fuzzers. By limiting the amount of RAM we can hope to increase coverage overall by fitting more fuzzers within the machines we have. Unfortunately this means limited coverage for the part of the fuzzing space that requires more memory.</span></p>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13 BvmDwwzBKL-c12"><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9"></span></p>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span>Until </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://github.com/memorysafety/rav1d&amp;sa=D&amp;source=editors&amp;ust=1727869833049988&amp;usg=AOvVaw21ZyX_NLPMleV9Bd7e1WoB">memory safe parsers</a></span><span>&nbsp;are available and widely used, memory corruption issues will continue to present a serious threat to users</span><span>.</span><span>&nbsp;For now, perhaps we can create fuzzers that are configured to occasionally explore parts of the fuzzing space that require more memory.</span></p><h1 class="BvmDwwzBKL-c3" id="h.mhw22d7c6izt"><span class="BvmDwwzBKL-c19 BvmDwwzBKL-c9">P.S.: OSS-Fuzz Bughunters Reward Program</span></h1>
 <p class="BvmDwwzBKL-c2 BvmDwwzBKL-c13"><span>Finally, I would like to mention that as of writing there is a Google bughunters reward program for improving </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://oss-fuzz-introspector.storage.googleapis.com/index.html&amp;sa=D&amp;source=editors&amp;ust=1727869833050268&amp;usg=AOvVaw3nvIvq5yFCtLWrfet-ti0e">fuzzing coverage</a></span><span>&nbsp;in critical OSS projects. </span><span class="BvmDwwzBKL-c7"><a href="https://www.google.com/url?q=https://bughunters.google.com/open-source-security/oss-fuzz&amp;sa=D&amp;source=editors&amp;ust=1727869833050344&amp;usg=AOvVaw2Ck2jEAKUCherhIIqlHJCZ">See the bughunters site</a></span><span class="BvmDwwzBKL-c14 BvmDwwzBKL-c9">&nbsp;for more details.</span></p>
