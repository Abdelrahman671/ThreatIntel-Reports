<span><ul><li><span style="	font-weight: 400;
">In December 2022, CrowdStrike </span><a href="/en-us/blog/analysis-of-intrusion-campaign-targeting-telecom-and-bpo-companies/"><span style="	font-weight: 400;
">reported on a campaign</span></a><span style="	font-weight: 400;
"> by SCATTERED SPIDER, targeting organizations within the telecom and business process outsourcing (BPO) sectors with an end objective of gaining access to mobile carrier networks.</span></li><li><span style="	font-weight: 400;
">In the weeks since that post, the CrowdStrike Falcon<sup>®</sup> platform prevented a novel attempt by SCATTERED SPIDER to deploy a malicious kernel driver through a vulnerability (CVE-2015-2291) in the Intel Ethernet diagnostics driver.</span></li><li><span style="	font-weight: 400;
">The activity exploits a well known and pervasive deficiency in Windows security that enables adversaries to bypass Windows kernel protections with the Bring-Your-Own-Vulnerable-Driver tactic. </span></li><li><span style="	font-weight: 400;
">CrowdStrike Services has observed the actor attempting to bypass other endpoint tools including Microsoft Defender for Endpoint, Palo Alto Networks Cortex XDR and SentinelOne using more traditional defense evasion techniques targeting Windows registry hives.</span></li></ul>
<h2><span style="	font-weight: 400;
">Introduction</span></h2>
<p><span style="	font-weight: 400;
">In December, CrowdStrike reported that beginning in June 2022, the CrowdStrike Services, CrowdStrike<sup>®</sup> Falcon OverWatch™ and CrowdStrike Intelligence teams observed an increase in the targeting of telco and BPO industries. CrowdStrike Intelligence attributed this campaign with low confidence to the SCATTERED SPIDER eCrime adversary.</span></p>
<p><span style="	font-weight: 400;
">S</span>CATTERED SPIDER (aka Roasted 0ktapus, UNC3944) leverages a combination of credential phishing and social engineering to capture one-time-password (OTP) codes or overwhelms targets using multifactor authentication (MFA) notification fatigue tactics. Having obtained access, the adversary avoids using unique malware, instead favoring a wide range of legitimate remote management tools to maintain persistent access.</p>
<p>In the weeks since <a href="/en-us/blog/analysis-of-intrusion-campaign-targeting-telecom-and-bpo-companies/">that blog post</a>, the Falcon platform detected a novel attempt by this adversary to deploy a malicious kernel driver through a vulnerability (CVE-2015-2291) in the Intel Ethernet diagnostics driver for Windows (<code style="	font-size: 14.0px;
">iqvw64.sys</code>).</p>
<h2><span style="	font-weight: 400;
">Microsoft Windows Security Deficiencies Leave Devices Vulnerable</span></h2>
<p><span style="	font-weight: 400;
">This vulnerability has been used by adversaries for several years to deploy malicious drivers into the Windows kernel. This technique is known as “Bring Your Own Vulnerable Driver” (BYOVD) and is a tactic that has persisted due to a gap in Windows security.</span></p>
<p><span style="	font-weight: 400;
">I</span>n an attempt to limit the amount of capabilities that malware can gain access to on a Windows system, starting with 64-bit Windows Vista, Windows does not allow unsigned kernel-mode drivers to run by default. BYOVD “makes it easy for an attacker with administrative control <a href="https://arstechnica.com/information-technology/2022/10/how-a-microsoft-blunder-opened-millions-of-pcs-to-potent-malware-attacks.html">to bypass Windows kernel protections</a>,” allowing an adversary to install a legitimately signed but malicious driver to execute an attack. Publicly available tools, such as KDMapper, allow adversaries to easily take advantage of BYOVD to map non-signed drivers into memory.</p>
<p>In 2021, <a href="https://www.microsoft.com/en-us/security/blog/2021/12/08/improve-kernel-security-with-the-new-microsoft-vulnerable-and-malicious-driver-reporting-center.html">Microsoft stated</a> that “Increasingly, adversaries are leveraging legitimate drivers in the ecosystem and their security vulnerabilities to run malware,” and that “drivers with confirmed security vulnerabilities will be blocked on Windows 10 devices in the ecosystem using Microsoft Defender for Endpoint attack surface reduction (ASR) and Microsoft Windows Defender Application Control (WDAC) technologies to protect devices against exploits involving vulnerable drivers to gain access to the kernel.”</p>
<p>However, as noted by multiple security researchers (e.g., <a href="https://news.sophos.com/en-us/2022/10/04/blackbyte-ransomware-returns/amp.html">here</a>, <a href="https://eclypsium.com/2019/08/10/screwed-drivers-signed-sealed-delivered.html">here</a> and <a href="https://www.welivesecurity.com/2022/01/11/signed-kernel-drivers-unguarded-gateway-windows-core.html">here</a>) over the past two years, the issue continues to persist as Microsoft fails to block vulnerable drivers by default.</p>
<p>In this instance, the adversary attempted to load a malicious driver that was prevented from running and quarantined by CrowdStrike Falcon<sup>®</sup> Prevent machine learning (ML) and identified by Falcon OverWatch. This driver is designed to use the privileged driver space provided by the vulnerable Intel driver to overwrite specific routines in the CrowdStrike Falcon sensor driver with adversary-created trampoline code. While this was prevented by the Falcon sensor and immediately escalated to the customer with human analysis, a series of recommendations to protect Microsoft kernel memory can be found in the &#34;Recommendations&#34; section below.</p>
<p>In the past months, CrowdStrike Services has observed the actor attempting to bypass other endpoint tools including Microsoft Defender for Endpoint, Palo Alto Networks Cortex XDR and SentinelOne.</p>
<h2><span style="	font-weight: 400;
">Technical Analysis</span></h2>
<p><span style="	font-weight: 400;
">CrowdStrike has identified various versions of a malicious driver that are signed by different certificates and authorities — including stolen certificates originally issued to NVIDIA and Global Software LLC, as well as a self-signed test certificate. The intent of the adversary is to disable the endpoint security products visibility and prevention capabilities so the actor can further their actions on objectives.</span></p>
<p><span style="	font-weight: 400;
">Versions of the sample are small, 64-bit Windows kernel drivers with less than 35 relatively simple functions.</span></p>
<p><span style="	font-weight: 400;
">An example driver with SHA256 hash</span></p>
<p><code><span style="	font-weight: 400;
">b6e82a4e6d8b715588bf4252f896e40b766ef981d941d0968f29a3a444f68fef</span></code><span style="	font-weight: 400;
"> has its build time set to </span><code><span style="	font-weight: 400;
">1970-01-01 00:01:35 UTC</span></code><span style="	font-weight: 400;
">. It contains various status messages and calls to </span><code><span style="	font-weight: 400;
">DbgPrintEx()</span></code><span style="	font-weight: 400;
">, as a means to provide status messages to the threat actor. The file is signed using a certificate with the following parameters:</span></p>
<p><span style="	font-weight: 400;
">serial: </span><code><span style="	font-weight: 400;
">31 11 00 fb 8d ee 5e 09 37 6b 69 a8 f6 23 e0 ee</span></code></p>
<p><span style="	font-weight: 400;
">issued to: </span><code><span style="	font-weight: 400;
">Global Software, LLC</span></code></p>
<p><span style="	font-weight: 400;
">valid from: </span><code><span style="	font-weight: 400;
">2018-05-14</span></code> <span style="	font-weight: 400;
">valid to: </span><code><span style="	font-weight: 400;
">2021-06-18</span></code></p>
<p><span style="	font-weight: 400;
">The same certificate has been observed signing other malicious files dating back to at least 2018, suggesting that other threat actors have copies of it.</span></p>
<p><span style="	font-weight: 400;
">The driver walks the list of loaded kernel modules, searching for </span><code><span style="	font-weight: 400;
">csagent.sys</span></code><span style="	font-weight: 400;
"> (the CrowdStrike Falcon kernel component), and scans the identified module for a hard-coded pattern of 64 bytes. In addition to the search pattern, the scanning function retrieves the mask string &#34;</span><code><span style="	font-weight: 400;
">xxxxxxxxxxxxxxxxxxx??xxxxxxxxxxxxxxxxxxxxxxxxxxxx??xx</span></code><span style="	font-weight: 400;
">&#34; as an argument that masks out bytes corresponding to absolute memory addresses. These change when the driver code is mapped to a different base address, and should therefore be excluded. The mask string is only 61 bytes long, 3 bytes shorter than the code pattern, possibly due to incomplete adjustments.</span></p>
<p><span style="	font-weight: 400;
">A second sample with SHA256 hash </span><code><span style="	font-weight: 400;
">e23283e75ed2bdabf6c703236f5518b4ca37d32f78d3d65b073496c12c643cfe</span></code><span style="	font-weight: 400;
"> has a PE build timestamp of </span><code><span style="	font-weight: 400;
">2022-12-23 15:11:27 UTC</span></code><span style="	font-weight: 400;
"> that matches the signing date. This file is digitally signed with what appears to be a test certificate with the following parameters:</span></p>
<p><span style="	font-weight: 400;
">serial: </span><code><span style="	font-weight: 400;
">23 43 9d 9d d3 2a a7 b2 4b bb 6e 31 64 fb 47 53</span></code></p>
<p><code></code><span style="	font-weight: 400;
">issued to: </span><code><span style="	font-weight: 400;
">WDKTestCert guid0,133162475712847553</span></code></p>
<p><span style="	font-weight: 400;
">valid from: </span><code><span style="	font-weight: 400;
">2022-12-23</span></code></p>
<p><span style="	font-weight: 400;
">valid to: </span><code><span style="	font-weight: 400;
">2032-12-23</span></code></p>
<p><span style="	font-weight: 400;
">This sample is intended to be loaded using BYOVD techniques and hence does not require a verifiable digital signature. No other files signed with this test certificate were available at the time of analysis.</span></p>
<p><span style="	font-weight: 400;
">U</span>pon startup, the driver decrypts a hard-coded string of targeted security products using a basic XOR loop:</p>
<p><code style="	font-size: 14.0px;
">for i in len(name): name<i> ^&#61; i % 56 &#43; 49</i></code></p>
<p><i>The malicious driver then finds the target driver using the same method and patches it, in memory, at hard-coded offsets. The patching routine operates on a list where each element represents a hook structure that contains a pointer to the target function, a pointer to the malware routine and trampoline code to invoke that routine. The installed malware routines signal success to the Falcon sensor in every case even though the routines perform no operation.</i></p>
<p><i>While the outlined activity appears to target specific industries, organizations of all types should apply the lessons learned to harden defenses against such threats. CrowdStrike recommends that organizations employ a rigorous, defense-in-depth approach that monitors endpoints, cloud workloads, identities and networks to defend against advanced, persistent adversaries. The holistic deployment of security tooling paired with a high operational tempo in responding to alerts and incidents are critical to success.</i></p>
<h2><i><span style="	font-weight: 400;
">Recommendations</span></i></h2>
<p><i><span style="	font-weight: 400;
">The described activity will be prevented and quarantined by the Falcon platform if configured as outlined in our prevention policy </span><a href="https://supportportal.crowdstrike.com/s/article/Prevention-Policy-Best-Practice-Guidelines"><span style="	font-weight: 400;
">best practices recommendations</span></a><span style="	font-weight: 400;
">.</span></i></p>
<h3><i><span style="	font-weight: 400;
">Prevention Policy Settings</span></i></h3>
<p><i> <span style="	font-weight: 400;
">With regard to the malicious activity detailed here, in particular confirm the Windows prevention policy settings listed are set as follows:</span></i></p>
<ul><li><i><span style="	font-weight: 400;
">Sensor Tampering Protection </span><i><span style="	font-weight: 400;
">enabled</span></i></i></li><li><i><span style="	font-weight: 400;
">Cloud Anti-malware </span><i><span style="	font-weight: 400;
">Prevention</span></i><span style="	font-weight: 400;
"> slider at Moderate or higher</span></i></li><li><i><span style="	font-weight: 400;
">Sensor Anti-malware </span><i><span style="	font-weight: 400;
">Prevention</span></i><span style="	font-weight: 400;
"> slider at Moderate or higher</span></i></li><li><i><span style="	font-weight: 400;
">Suspicious Processes </span><i><span style="	font-weight: 400;
">enabled</span></i></i></li><li><i><span style="	font-weight: 400;
">Suspicious Kernel Drivers </span><i><span style="	font-weight: 400;
">enabled</span></i></i></li></ul>
<p><i><span style="	font-weight: 400;
">The complete list of our best practices recommendations for prevention policies may be found in the support portal here: </span><a href="https://supportportal.crowdstrike.com/s/article/Prevention-Policy-Best-Practice-Guidelines"><span style="	font-weight: 400;
">Prevention Policy Best Practice Guidelines</span></a><h3><span style="	font-weight: 400;
">Mitigate CVE-2015-2291</span></h3>
<span style="	font-weight: 400;
">CrowdStrike customers should ensure they have the ability to locate and patch the vulnerable Intel Ethernet Driver specified in CVE-2015-2291. Prioritizing the patching of vulnerable drivers can help mitigate this and similar attack vectors involving signed driver abuse.</span> </i></p>

<h3><i><span style="	font-weight: 400;
">Evaluate Enabling Microsoft Memory Integrity Capabilities</span></i></h3>
<p><i> <span style="	font-weight: 400;
">Due to the inherent flaws in Microsoft protection of kernel memory permitting these types of BYOVD attacks, additional functionality has been incorporated into modern Windows Operating Systems. </span><a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/bringup/device-guard-and-credential-guard"><span style="	font-weight: 400;
">Hypervisor-Protected Code Integrity (HVCI)</span></a><span style="	font-weight: 400;
">, a component of Virtualization-Based Security (VBS) is designed to prevent users with elevated privilege from being able to read and write to kernel memory. The protections were implemented in order to address the security flaw of not enforcing kernel memory protections.</span> </i></p>
<p> </p>
<p><i><span style="	font-weight: 400;
">While CrowdStrike has not identified any issues with the Falcon platform running in environments with these features enabled, there have been intermittent reports of performance issues in various other environments tied to specific applications and some processors.</span> <span style="	font-weight: 400;
">CrowdStrike Falcon<sup>®</sup> Insight XDR customers can view the status of HVCI by navigating to: Host setup and management &gt; Zero Trust Assessment.</span></i></p>
