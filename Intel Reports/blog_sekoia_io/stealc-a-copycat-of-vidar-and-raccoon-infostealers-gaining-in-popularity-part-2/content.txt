<h2 class="wp-block-heading" id="h-context">Context</h2>



<p>This report is a follow up of the <a href="https://blog.sekoia.io/stealc-a-copycat-of-vidar-and-raccoon-infostealers-gaining-in-popularity-part-1/">previous blog post</a> on Stealc. Stealc is an <strong>information stealer</strong> advertised on the <strong>underground forums</strong> XSS, Exploit and BHF by the <em>Plymouth </em>threat actor. In this blog post, we focus on the <strong>technical analysis</strong> of a <strong>standalone</strong> sample. <strong>Similarities </strong>were observed with <strong>Vidar</strong>, <strong><a href="https://blog.sekoia.io/fr/raccoon-stealer-v2-partie-1-de-retour-des-morts/" target="_blank" rel="noreferrer noopener">Raccoon</a></strong> and <strong>Mars</strong> stealers during the <strong>reverse engineering </strong>phase.</p>



<p>Functionalities implemented in <strong>Stealc</strong>, including environment detection, anti-analysis, strings obfuscation, dynamic API resolution, a significant list of targeted browsers, extensions, wallets and installed software makes it a top-tier threat within the infostealer ecosystem.&nbsp;</p>



<h2 class="wp-block-heading" id="h-malware-analysis">Malware analysis</h2>



<p>The next sections list the different techniques observed during the reverse engineering of Stealc to provide information and detailed explanations on Stealc operations and behaviors.&nbsp;</p>



<p><em>All details of the infection chain, distribution and tracking of this threat were provided in </em><a href="https://blog.sekoia.io/stealc-a-copycat-of-vidar-and-raccoon-infostealers-gaining-in-popularity-part-1/" target="_blank" rel="noreferrer noopener"><em>part 1</em></a><em>.</em></p>



<p>Stealc sample SHA-256 used for the analysis is: <a href="https://bazaar.abuse.ch/sample/77d6f1914af6caf909fa2a246fcec05f500f79dd56e5d0d466d55924695c702d/" target="_blank" rel="noreferrer noopener">77d6f1914af6caf909fa2a246fcec05f500f79dd56e5d0d466d55924695c702d</a></p>



<p>Stealc sample SHA-256 with a next stage configured is: <a href="https://bazaar.abuse.ch/sample/1587857ad744c322a2b32731cdd48d98eac13f8aa8ff2f2afb01ebba88d15359/" target="_blank" rel="noreferrer noopener">1587857ad744c322a2b32731cdd48d98eac13f8aa8ff2f2afb01ebba88d15359</a></p>



<h3 class="wp-block-heading" id="h-anti-analysis">Anti analysis</h3>



<p>The malware implements anti-analysis techniques by adding unconditional jump to a nearby offset, confusing the decompiler that cannot grasp the pointed function.&nbsp;</p>



<p>As shown in figure 1, the decompiler analyzed a function with multiple jump instructions (<span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">jz</span>, <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">jnz</span> opcodes), with a destination address defined to the next address plus an offset of 1 or 2 (depending on the case). This results in the decompiler not making a correct assumption and avoiding decompiling the function.&nbsp;</p>

<p>The execution flow of Stealc is straightforward, it first <strong>deobfuscates strings</strong> used for further <strong>dynamic API resolution</strong>. Then, it performs various <strong>checks</strong> on the infected host to <strong>exit on particular conditions</strong>, it also checks the amount of RAM and whether it is executed by an <strong>antivirus solution</strong>. Finally, it verifies that the current date is <strong>preceding </strong>the <strong>hardcoded</strong> one.<br>After this <strong>initial setup</strong> and <strong>detection</strong>, the malware goes to the function responsible for the C2 <strong>interaction</strong>, in which the stealer configuration is downloaded, and data are exfiltrated.</p>

<h3 class="wp-block-heading" id="h-dynamic-api-resolution">Dynamic API resolution</h3>



<p>To reduce its detection rate by antivirus solutions, Stealc uses the Dynamic API Resolution (T1027.007) technique. To do so, the malware searches for the <em>kernel32</em> base address using the PE header structure and goes through <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">LDR_DATA_TABLE_ENTRY</span>. Then, it iterates over the table until it matches the <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">GetProcAddress</span> function and returns the address of the dedicated entry.</p>

<h3 class="wp-block-heading" id="h-environment-detection-amp-checks">Environment detection &amp; checks</h3>



<p>Stealc attempts to detect its environment for two purposes:</p>



<ol class="wp-block-list">
<li>Exit in particular conditions (sandbox environment, unwanted location, <em>etc</em>.)</li>



<li>Host fingerprinting</li>
</ol>



<p>The malware implements the following exit conditions:</p>



<ul class="wp-block-list">
<li>Username is <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">JohnDoe</span> (Windows Defender emulator default username);</li>

<li>Configured language is Russian;</li>

<li>Section <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">.text</span> should be writable (by default Stealc configures its .text with write permission).</li>

<li>No display is configured.</li>
</ul>



<h3 class="wp-block-heading" id="h-miscellaneous-functionalities">Miscellaneous functionalities</h3>



<p>Stealc also implements functionalities common to other malware of the stealer family. It has the capability to take a screenshot of the infected host and to fingerprint the infected machine. To retrieve this information, the sample queries the suitable registry keys and interacts with the Windows API.</p>



<p>The fingerprinted information are:</p>



<ul class="wp-block-list">
<li>Public IP address;</li>



<li>Geolocation;</li>



<li>Hardware ID;</li>



<li>Operating System version;</li>



<li>Architecture;</li>



<li>Username;</li>



<li>Computer name;</li>



<li>Local time;</li>



<li>Language;</li>



<li>Keyboard layout;</li>



<li>Physical resources: CPU (core, name), RAM, number of threads, display resolution and GPU driver;</li>



<li>List of running processes;</li>



<li>List of installed applications.</li>
</ul>

<p>The malware communicates over HTTP, data are sent in POST requests that use multi-form structure whose forms are the stolen data encoded in base64.</p>



<p>In the first interaction, the infected host sends its HWID (hardware ID) and its build name (the value is &#8220;<em>default</em>&#8220;).</p>



<p>The server responds with the following <strong>base64</strong> string:</p>

<p>The first hash is, in fact, an identifier used as a <strong>token</strong> for all communications, and sent in a dedicated form for each message.&nbsp;</p>

<p>The stealer <strong>gets</strong> its <strong>configuration</strong> from the <strong>C2</strong>, with a POST request whose two forms are sent. <p>It repeats this same operation for each browser, their extensions, for the wallets and installed applications.</p>

<h4 class="wp-block-heading">File grabber</h4>



<p>After stealing data from targeted browsers and their extensions, the stealer uses its file grabber functionality. The grabber configuration is received from the C2 and is formatted as follow:</p>



<p><span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">standart|%DESKTOP%\\|*.txt,*.doc,*.docx,*.xls|7000|1|0|</span></p>



<p>The structure of the configuration is the following one. First a <em>name,</em> then a <em>directory</em> or a <em>shortcut</em> to a directory (here the desktop), thirdly a list of <em>file extensions</em> that the malware wants to exfiltrate and finally the <em>maximum size</em>. We also identified 2 extra parameters that were not useful for the analysis.</p>



<p>In case the file name and path match the grabber filters, it is exfiltrated in a POST request to the C2 with three forms.</p>

<h3 class="wp-block-heading" id="h-trace-removal">Trace removal</h3>



<p>Stealc attempts to reduce its infection traces by removing itself and its downloaded DLLs (T1070.004) with the following one-line command:</p>



<p><code>cmd.exe /c timeout /t 5 &amp; del /f /q "$STEALERPATH" &amp; del "C:\ProgramData\*.dll" &amp; exit</code></p>



<p>The command is executed with a basic <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">ShellExecuteA</span> function from <em>Shell32.dll</em>.</p>



<h2 class="wp-block-heading" id="h-conclusion">Conclusion</h2>



<p><strong>Stealc</strong> displays all functionalities and behaviors to be a <strong>viable tool in the </strong>information stealer catalog, and will almost certainly be incorporated in multiple intrusion sets’ toolsets, either as a shift or an expansion of their capabilities. Based on observed similarities between <strong>Stealc</strong> and other malware of the <a href="https://blog.sekoia.io/overview-of-the-russian-speaking-infostealer-ecosystem-the-logs/" target="_blank" rel="noreferrer noopener">infostealer</a> family, notably <strong>Raccoon</strong> and <strong>Mars</strong> stealer, SEKOIA.IO analysts assess it is likely a confirmation of a transmission and circulation of knowledge, including source code, and of human resources, in the Russian-speaking cybercriminal ecosystem.</p>

<h2 class="wp-block-heading" id="h-annex-1-configuration-extraction">Annex 1 &#8211; Configuration extraction</h2>



<p>As introduced in the <em>strings obfuscation </em>section, Stealc embeds the address of the C2 and its different URLs in the <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">rdata</span> section of the PE.</p>



<p>Based on our observation, the script should meet the following requirements:</p>



<ol class="wp-block-list">
<li>Retrieve the RC4 key in rdata;</li>



<li>Deobfuscate the strings until all patterns related to the C2 are spotted.</li>
</ol>



<p>The RC4 key is hardcoded in the PE in cleartext and by definition RC4 keys are 20 bytes long. Stealc C2 information are stored with the following structure:</p>



<ul class="wp-block-list">
<li>C2 base URL: <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">http://&lt;ip or domain&gt;</span> or <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">https://&lt;ip or domain&gt;</span>;</li>



<li>C2 URL resource which is a random string ending by <span style="background-color:#FAFAFA;border-radius:3px;padding3px;border:1px solid #CCC;padding-right:5px;padding-left:5px;font-family:monospace">.php</span> extension;</li>



<li>C2 directory name where the DLLs are hosted (<em>nss3.dll, sqlite3.dll, etc</em>&#8230;).</li>
</ul>



<p>The provided configuration extractor simply loops over that section to find the patterns described previously.</p>



<pre class="wp-block-code"><code>from base64 import b64decode
from pefile import PE, SectionStructure
from cryptography.hazmat.primitives.ciphers import Cipher, algorithms


class Stealc:

    """Stealc configuration"""

    rc4_key: bytes = b""
    base_url: str = ""
    endpoint_url: str = ""
    dlls_directory: str = ""

    def __str__(self):
        out = f"Stealc RC4 key: {self.rc4_key}\n"
        out += f"SteaC Command and Control:\n"
        out += f"\t- {''.join(&#91;self.base_url, self.endpoint_url])}\n"
        out += f"\t- {''.join(&#91;self.base_url, self.dlls_directory])}\n"
        return out

    def rc4_decrypt(self, data: bytes) -&gt; bytes:
        """decrypt RC4 data with the provided key."""

        algorithm = algorithms.ARC4(self.rc4_key)
        cipher = Cipher(algorithm, mode=None)
        decryptor = cipher.decryptor()
        return decryptor.update(data)


def get_section(pe: PE, section_name: str) -&gt; SectionStructure:
    """return section by name, if not found raise KeyError exception."""

    for section in filter(
        lambda x: x.Name.startswith(section_name.encode()), pe.sections
    ):
        return section

    available_sections = ", ".join(
        &#91;_sec.Name.replace(b"\x00", b"").decode() for _sec in pe.sections]
    )
    raise KeyError(
        f"{section_name} not found in the PE, available sections: {available_sections}"
    )


def get_rdata(pe_path: str) -&gt; SectionStructure:
    """Extract Stealc radata section"""

    pe = PE(pe_path)
    section_rdata = get_section(pe, ".rdata")
    return section_rdata


def is_valid_string(data: bytes) -&gt; bool:
    return True if all(map(lambda x: x &gt;= 43 and x &lt;= 122, data)) else False


def search_Command_and_Control(stealc: Stealc, rdata_section: SectionStructure):
    """
    Search two types of strings in rdata section of Stealc:
    1. The RC4 key which is 20 bytes long;
    2. Strings matching the way Stealc stores its C2 configuration (these strings are decoded (base64 decode + RC4 decryption),
       This works for the Stealc version at least until 15 Feb 2023 but could change in new versions...
        2.1 base url (`http://something...` or `https://something...`)
        2.2 endpoint which ends with `.php`
        2.3 DLLs directory starts and ends with `/` (eg: `/something_random/`)
    """

    for string in filter(
        lambda x: x and is_valid_string(x), rdata_section.get_data().split(b"\x00" * 2)
    ):
        if len(string) == 20 and not stealc.rc4_key:
            # Hopefully the RC4 key is stored as the beginning of the rdata section
            stealc.rc4_key = string
            print(f"&#91;+] RC4 key found: {stealc.rc4_key}")
        if stealc.rc4_key and string != stealc.rc4_key:
            try:
                cleartext = stealc.rc4_decrypt(b64decode(string))
                # print(f"{string.decode():&lt;40} {cleartext}")
                if cleartext.startswith(b"http://") or cleartext.startswith(
                    b"https://"
                ):
                    print(f"&#91;+] Found StealC Command and Control")
                    stealc.base_url = cleartext.decode()
                elif cleartext.startswith(b"/") and cleartext.endswith(b"/"):
                    print(f"&#91;+] Found DLLs URL directory name")
                    stealc.dlls_directory = cleartext.decode()
                elif cleartext.endswith(b".php"):
                    print(f"&#91;+] Found StealC endpoint")
                    stealc.endpoint_url = cleartext.decode()

            except Exception:
                pass


if __name__ == "__main__":
    import sys
    if len(sys.argv) &lt; 2:
        print(f"not enough parameter, please provide as argument the path to stealc sample.")
    stealc = Stealc()
    rdata = get_rdata(sys.argv&#91;1])
    search_Command_and_Control(stealc, rdata)
    print(stealc)
</code></pre>
