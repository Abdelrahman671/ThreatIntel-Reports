											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>It's common for multiple versions of the same dynamic-link library (DLL) to exist in different file system locations within an operating system (OS). You <em>can</em> control the specific location from which any given DLL is loaded by specifying a full path. But if you don't use that method, then the system searches for the DLL at load time as described in this topic. The <em>DLL loader</em> is the part of the operating system (OS) that loads DLLs and/or resolves references to DLLs.</p>
<div class="TIP">
<p>Tip</p>
<p>For definitions of <em>packaged</em> and <em>unpackaged</em> apps, see <a href="/en-us/windows/apps/package-and-deploy/#advantages-and-disadvantages-of-packaging-your-app" data-linktype="absolute-path">Advantages and disadvantages of packaging your app</a>.</p>
</div>
<h2 id="factors-that-affect-searching">Factors that affect searching</h2>
<p>Here are some special search factors that are discussed in this topicâyou can consider them to be part of the DLL search order. Later sections in this topic list these factors in the appropriate search order for certain app types, together with other search locations. This section is just to introduce the concepts, and to give them names that we'll use to refer to them later in the topic.</p>
<ul>
<li><strong>DLL redirection</strong>. For details, see <a href="/en-us/windows/win32/dlls/dynamic-link-library-redirection" data-linktype="absolute-path">Dynamic-link library redirection</a>.</li>
<li><strong>API sets</strong>. For details, see <a href="/en-us/windows/win32/apiindex/windows-apisets" data-linktype="absolute-path">Windows API sets</a>.</li>
<li><strong>Side-by-side (SxS) manifest redirection</strong>âdesktop apps only (not UWP apps). You can redirect by using an application manifest (also known as a side-by-side application manifest, or a fusion manifest). For details, see <a href="/en-us/windows/win32/sbscs/manifests" data-linktype="absolute-path">Manifests</a>.</li>
<li><strong>Loaded-module list</strong>. The system can check to see whether a DLL with the same module name is already loaded into memory (no matter which folder it was loaded from).</li>
<li><strong>Known DLLs</strong>. If the DLL is on the list of known DLLs for the version of Windows on which the application is running, then the system uses its copy of the known DLL (and the known DLL's dependent DLLs, if any). For a list of known DLLs on the current system, see the registry key <code>HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager\KnownDLLs</code>.</li>
</ul>
<p>If a DLL has dependencies, then the system searches for the dependent DLLs as if they were loaded by using only their module names. That's true even if the first DLL was loaded by specifying a full path.</p>
<h2 id="search-order-for-packaged-apps">Search order for packaged apps</h2>
<p>When a packaged app loads a packaged module (specifically, a library moduleâa <code>.dll</code> file) by calling the <a href="/en-us/windows/win32/api/Winbase/nf-winbase-loadpackagedlibrary" data-linktype="absolute-path"><strong>LoadPackagedLibrary</strong></a> function, the DLL must be in the package dependency graph of the process. For more information, see <strong>LoadPackagedLibrary</strong>. When a packaged app loads a module by other means, and doesn't specify a full path, the system searches for the DLL and its dependencies at load time as described in this section.</p>
<p>When the system searches for a module or its dependencies, it always uses the search order for packaged apps; even if a dependency is not packaged app code.</p>
<h3 id="standard-search-order-for-packaged-apps">Standard search order for packaged apps</h3>
<p>The system searches in this order:</p>
<ol>
<li>DLL redirection.</li>
<li>API sets.</li>
<li><strong>Desktop apps only (not UWP apps)</strong>. SxS manifest redirection.</li>
<li>Loaded-module list.</li>
<li>Known DLLs.</li>
<li>The package dependency graph of the process. This is the application's package plus any dependencies specified as <code>&lt;PackageDependency&gt;</code> in the <code>&lt;Dependencies&gt;</code> section of the application's package manifest. Dependencies are searched in the order they appear in the manifest.</li>
<li>The folder the calling process was loaded from (the executable's folder).</li>
<li>The system folder (<code>%SystemRoot%\system32</code>).</li>
</ol>
<p>If a DLL has dependencies, then the system searches for the dependent DLLs as if they were loaded with just their module names (even if the first DLL was loaded by specifying a full path).</p>
<h3 id="alternate-search-order-for-packaged-apps">Alternate search order for packaged apps</h3>
<p>If a module changes the standard search order by calling the <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a> function with <strong>LOAD_WITH_ALTERED_SEARCH_PATH</strong>, then the search order is the same as the standard search order except that in step 7 the system searches the folder that the specified module was loaded from (the top-loading module's folder) instead of the executable's folder.</p>
<h2 id="search-order-for-unpackaged-apps">Search order for unpackaged apps</h2>
<p>When an unpackaged app loads a module and doesn't specify a full path, the system searches for the DLL at load time as described in this section.</p>
<div class="IMPORTANT">
<p>Important</p>
<p>If an attacker gains control of one of the directories that's searched, then it can place a malicious copy of the DLL in that folder. For ways to help prevent such attacks, see <a href="/en-us/windows/win32/dlls/dynamic-link-library-security" data-linktype="absolute-path">Dynamic-link library security</a>.</p>
</div>
<h3 id="standard-search-order-for-unpackaged-apps">Standard search order for unpackaged apps</h3>
<p>The standard DLL search order used by the system depends on whether or not <em>safe DLL search mode</em> is enabled.</p>
<p>Safe DLL search mode (which is enabled by default) moves the user's current folder later in the search order. To disable safe DLL search mode, create the <code>HKEY_LOCAL_MACHINE\System\CurrentControlSet\Control\Session Manager\SafeDllSearchMode</code> registry value, and set it to 0. Calling the <a href="/en-us/windows/win32/api/winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> function effectively disables safe DLL search mode (while the specified folder is in the search path), and changes the search order as described in this topic.</p>
<p>If safe DLL search mode is enabled, then the search order is as follows:</p>
<ol>
<li>DLL Redirection.</li>
<li>API sets.</li>
<li>SxS manifest redirection.</li>
<li>Loaded-module list.</li>
<li>Known DLLs.</li>
<li><strong>Windows 11, version 21H2 (10.0; Build 22000), and later</strong>. The package dependency graph of the process. This is the application's package plus any dependencies specified as <code>&lt;PackageDependency&gt;</code> in the <code>&lt;Dependencies&gt;</code> section of the application's package manifest. Dependencies are searched in the order they appear in the manifest.</li>
<li>The folder from which the application loaded.</li>
<li>The system folder. Use the <a href="/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-getsystemdirectorya" data-linktype="absolute-path"><strong>GetSystemDirectory</strong></a> function to retrieve the path of this folder.</li>
<li>The 16-bit system folder. There's no function that obtains the path of this folder, but it is searched.</li>
<li>The Windows folder. Use the <a href="/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-getwindowsdirectorya" data-linktype="absolute-path"><strong>GetWindowsDirectory</strong></a> function to get the path of this folder.</li>
<li>The current folder.</li>
<li>The directories that are listed in the <code>PATH</code> environment variable. This doesn't include the per-application path specified by the <strong>App Paths</strong> registry key. The <strong>App Paths</strong> key isn't used when computing the DLL search path.</li>
</ol>
<p>If safe DLL search mode is <em>disabled</em>, then the search order is the same except that <em>the current folder</em> moves from position 11 to position 8 in the sequence (immediately after step <em>7. The folder from which the application loaded</em>).</p>
<h3 id="alternate-search-order-for-unpackaged-apps">Alternate search order for unpackaged apps</h3>
<p>To change the standard search order used by the system, you can call the <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a> function with <strong>LOAD_WITH_ALTERED_SEARCH_PATH</strong>. You can also change the standard search order by calling the <a href="/en-us/windows/win32/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> function.</p>
<div class="NOTE">
<p>Note</p>
<p>The standard search order of the process will also be affected by calling the <a href="/en-us/windows/win32/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> function in the parent process before the start of the current process.</p>
</div>
<p>If you specify an alternate search strategy, then its behavior continues until all associated executable modules have been located. After the system starts processing DLL initialization routines, the system reverts to the standard search strategy.</p>
<p>The <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a> function supports an alternate search order if the call specifies <strong>LOAD_WITH_ALTERED_SEARCH_PATH</strong>, and the <em>lpFileName</em> parameter specifies an absolute path.</p>
<ul>
<li>The standard search strategy begins (after the initial steps) in the calling application's folder.</li>
<li>The alternate search strategy specified by <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a> with <strong>LOAD_WITH_ALTERED_SEARCH_PATH</strong> begins (after the initial steps) in the folder of the executable module that <strong>LoadLibraryEx</strong> is loading.</li>
</ul>
<p>That's the only way in which they differ.</p>
<p>If safe DLL search mode is enabled, then the alternate search order is as follows:</p>
<p>Steps 1-6 are the same as the standard search order.</p>
<ol start="7">
<li>The folder specified by <em>lpFileName</em>.</li>
<li>The system folder. Use the <a href="/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-getsystemdirectorya" data-linktype="absolute-path"><strong>GetSystemDirectory</strong></a> function to retrieve the path of this folder.</li>
<li>The 16-bit system folder. There's no function that obtains the path of this folder, but it is searched.</li>
<li>The Windows folder. Use the <a href="/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-getwindowsdirectorya" data-linktype="absolute-path"><strong>GetWindowsDirectory</strong></a> function to get the path of this folder.</li>
<li>The current folder.</li>
<li>The directories that are listed in the <code>PATH</code> environment variable. This doesn't include the per-application path specified by the <strong>App Paths</strong> registry key. The <strong>App Paths</strong> key isn't used when computing the DLL search path.</li>
</ol>
<p>If safe DLL search mode is <em>disabled</em>, then the alternate search order is the same except that <em>the current folder</em> moves from position 11 to position 8 in the sequence (immediately after step <em>7. The folder specified by lpFileName</em>).</p>
<p>The <a href="/en-us/windows/win32/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> function supports an alternate search order if the <em>lpPathName</em> parameter specifies a path. The alternate search order is as follows:</p>
<p>Steps 1-6 are the same as the standard search order.</p>
<ol start="7">
<li>The folder from which the application loaded.</li>
<li>The folder specified by the <em>lpPathName</em> parameter of <a href="/en-us/windows/win32/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a>.</li>
<li>The system folder.</li>
<li>The 16-bit system folder.</li>
<li>The Windows folder.</li>
<li>The directories listed in the <code>PATH</code> environment variable.</li>
</ol>
<p>If the <em>lpPathName</em> parameter is an empty string, then the call removes the current folder from the search order.</p>
<p><a href="/en-us/windows/win32/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> effectively disables safe DLL search mode while the specified folder is in the search path. To restore safe DLL search mode based on the <strong>SafeDllSearchMode</strong> registry value, and restore the current folder to the search order, call <strong>SetDllDirectory</strong> with <em>lpPathName</em> as NULL.</p>
<h3 id="search-order-using-load_library_search-flags">Search order using LOAD_LIBRARY_SEARCH flags</h3>
<p>You can specify a search order by using one or more <strong>LOAD_LIBRARY_SEARCH</strong> flags with the <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a> function. You can also use <strong>LOAD_LIBRARY_SEARCH</strong> flags with the <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-setdefaultdlldirectories" data-linktype="absolute-path"><strong>SetDefaultDllDirectories</strong></a> function to establish a DLL search order for a process. You can specify additional directories for the process DLL search order by using the <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-adddlldirectory" data-linktype="absolute-path"><strong>AddDllDirectory</strong></a> or <a href="/en-us/windows/win32/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> functions.</p>
<p>The directories that are searched depend on the flags specified with <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-setdefaultdlldirectories" data-linktype="absolute-path"><strong>SetDefaultDllDirectories</strong></a> or <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a>. If you use more than one flag, then the corresponding directories are searched in this order:</p>
<ol>
<li><strong>LOAD_LIBRARY_SEARCH_DLL_LOAD_DIR</strong>. The folder that contains the DLL is searched. This folder is searched only for dependencies of the DLL to be loaded.</li>
<li><strong>LOAD_LIBRARY_SEARCH_APPLICATION_DIR</strong>. The application folder is searched.</li>
<li><strong>LOAD_LIBRARY_SEARCH_USER_DIRS</strong>. Paths explicitly added with the <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-adddlldirectory" data-linktype="absolute-path"><strong>AddDllDirectory</strong></a> function or the <a href="/en-us/windows/win32/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> function are searched. If you add more than one path, then the order in which the paths are searched is unspecified.</li>
<li><strong>LOAD_LIBRARY_SEARCH_SYSTEM32</strong>. The System folder is searched.</li>
</ol>
<p>If you call <a href="/en-us/windows/win32/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a> with no <strong>LOAD_LIBRARY_SEARCH</strong> flags, or you establish a DLL search order for the process, then the system searches for DLLs using either the standard search order or the alternate search order.</p>
