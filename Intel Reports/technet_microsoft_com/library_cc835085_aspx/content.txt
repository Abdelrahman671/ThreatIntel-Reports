							<div class="content "><h1 id="netdom-trust">Netdom trust</h1><div class="display-flex justify-content-space-between align-items-center flex-wrap-wrap page-metadata-container">
										<div class="margin-right-xxs">
											<ul class="metadata page-metadata" data-bi-name="page info" lang="en-us" dir="ltr"><li>Article</li><li>
														<time class="is-invisible" data-article-date aria-label="Article review date" datetime="2020-02-27T00:00:00.000Z" data-article-date-source="ms.date">02/27/2020</time>
													</li></ul>
										</div></div><nav id="center-doc-outline" class="doc-outline is-hidden-desktop display-none-print margin-bottom-sm" data-bi-name="intopic toc" aria-label="In this article">
											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>&nbsp;</p>
<p>Applies To: Windows Server 2008, Windows Server 2008 R2, Windows Server 2012, Windows 8</p>
<p>Establishes, verifies, or resets a trust relationship between domains.</p>
<p><strong>Netdom</strong> is a command-line tool that is built into Windows Server&nbsp;2008 and Windows Server&nbsp;2008&nbsp;R2. It is available if you have the Active&nbsp;Directory Domain Services (AD&nbsp;DS) server role installed. It is also available if you install the Active&nbsp;Directory Domain Services Tools that are part of the Remote Server Administration Tools (RSAT). For more information, see <a href="https://go.microsoft.com/fwlink/?linkid=177813" data-linktype="external">How to Administer Microsoft Windows Client and Server Computers Locally and Remotely</a> (<a href="https://go.microsoft.com/fwlink/?LinkID=177813" data-linktype="external">https://go.microsoft.com/fwlink/?LinkID=177813</a>).</p>
<p>To use <strong>netdom</strong>, you must run the <strong>netdom</strong> command from an elevated command prompt. To open an elevated command prompt, click <strong>Start</strong>, right-click <strong>Command Prompt</strong>, and then click <strong>Run as administrator</strong>.</p>
<p>For examples of how to use this command, see <a href="">Examples</a>.</p>
<div class="IMPORTANT">
<p>Important</p>
<p><strong>Netdom</strong> cannot be used to create a forest trust between two AD DS forests. To create an across-forest trust between two AD DS forests, you can either use a scripting solution or the Active Directory Domains and Trusts snap-in.</p>
</div>
<h2 id="syntax">Syntax</h2>
<pre><code>netdom trust &lt;TrustingDomainName&gt; {/d: | /domain:} &lt;TrustedDomainName&gt; [{/ud: | /userd:}[&lt;Domain&gt;\]&lt;User&gt; [{/pd: | /passwordd:}{&lt;Password&gt;|*}] [{/uo: | /usero:}&lt;User&gt;] [{/po: | /passwordo:}{&lt;Password&gt;|*}] [/verify] [/reset] [/passwordt:&lt;NewRealmTrustPassword&gt;] [/add [/realm]] [/remove [/force]] [/twoway] [/kerberos] [/transitive[:{YES|NO}]] [/oneside:{TRUSTED | TRUSTING}] [/force] [/quarantine[:{YES | NO}]] [/namesuffixes:&lt;TrustName&gt; [/togglesuffix:#]] [/EnableSIDHistory] [/ForestTRANsitive] [/SelectiveAUTH][/AddTLN][/AddTLNEX][/RemoveTLN] [/RemoveTLNEX][/SecurePasswordPrompt] [/EnableTgtDelegation] [{/help | /?}]
</code></pre>
<h2 id="parameters">Parameters</h2>
<table>
<colgroup>
<col>
<col>
</colgroup>
<thead>
<tr class="header">
<th><p>Parameter</p></th>
<th><p>Description</p></th>
</tr>
</thead>
<tbody>
<tr class="odd">
<td><p>&lt;TrustingDomainName&gt;</p></td>
<td><p>Specifies the name of the trusting domain.</p></td>
</tr>
<tr class="even">
<td><p>{/d: | /domain:}&lt;TrustedDomainName&gt;</p></td>
<td><p>Specifies the name of the trusted domain. If you do not specify this parameter, then <strong>netdom trust</strong> uses the domain to which the current computer belongs.</p></td>
</tr>
<tr class="odd">
<td><p>{/ud: | /userd:}[&lt;Domain&gt;\]&lt;User&gt;]</p></td>
<td><p>Specifies the user account to use to make the connection with the domain that you specify in the <strong>/d</strong> or <strong>/domain</strong> parameter. If you do not specify this parameter, then <strong>netdom trust</strong> uses the current user account.</p></td>
</tr>
<tr class="even">
<td><p>/pd:{&lt;Password&gt;|*}</p></td>
<td><p>Specifies the password of the user account that you specify in the <strong>/ud</strong> or <strong>/userd</strong>: parameter. If you specify the value of this parameter as a wildcard character (*), this parameter prompts you for the password.</p></td>
</tr>
<tr class="odd">
<td><p>{/uo: | /usero:}&lt;User&gt;</p></td>
<td><p>Specifies the user account to use to make the connection with the trusting domain. If you do not specify this parameter, then <strong>netdom trust</strong> uses the current user account.</p></td>
</tr>
<tr class="even">
<td><p>{/po: | /passwordo:}{&lt;Password&gt;|*}</p></td>
<td><p>Specifies the password of the user account that you specify in the <strong>/uo</strong> or <strong>/usero</strong> parameter. If you specify the value of this parameter as a wildcard character (*), this parameter prompts you for the password.</p></td>
</tr>
<tr class="odd">
<td><p>/verify</p></td>
<td><p>Verifies the secure channel secrets upon which a specific trust relationship is based.</p></td>
</tr>
<tr class="even">
<td><p>/reset</p></td>
<td><p>Resets the trust secret between trusted domains or between the domain controller and the workstation.</p></td>
</tr>
<tr class="odd">
<td><p>/passwordt:&lt;NewRealmTrustPassword&gt;</p></td>
<td><p>Specifies a new trust password. This parameter is valid only if you specify the <strong>/add</strong> parameter, and only if one of the domains that you specify is a non-Windows, Kerberos realm. You set the trust password on the Windows domain only, which means that you do not need credentials for the non-Windows domain.</p></td>
</tr>
<tr class="even">
<td><p>/add</p></td>
<td><p>Specifies to create a trust.</p></td>
</tr>
<tr class="odd">
<td><p>/realm</p></td>
<td><p>Specifies to create the trust for a non-Windows, Kerberos realm. This parameter is valid only if you specify the <strong>/add</strong> and <strong>/passwordt</strong> parameters.</p></td>
</tr>
<tr class="even">
<td><p>/remove</p></td>
<td><p>Specifies to break a trust.</p></td>
</tr>
<tr class="odd">
<td><p>/force</p></td>
<td><p>Removes from the forest both the trusted domain object and the cross-reference object for the domain that you specify. You use this parameter to clean up decommissioned domains that you no longer use and that you cannot remove by using the Active&nbsp;Directory Installation Wizard. This problem can occur if the domain controller for a decommissioned domain is disabled or damaged and there are no additional domain controllers, or if you cannot recover a decommissioned domain controller from backup media. This parameter is valid only if you specify the <strong>/remove</strong> parameter.</p></td>
</tr>
<tr class="even">
<td><p>/twoway</p></td>
<td><p>Specifies to establish a two-way trust relationship rather than a one-way trust relationship.</p></td>
</tr>
<tr class="odd">
<td><p>/kerberos</p></td>
<td><p>Specifies to exercise the Kerberos protocol between a workstation and a destination domain. This parameter is valid only if you specify the <strong>/verify</strong> parameter.</p></td>
</tr>
<tr class="even">
<td><p>/transitive[:{YES|NO}]</p></td>
<td><p>Specifies to set either a transitive or non-transitive trust. This parameter is valid only for a non-Windows, Kerberos realm. <strong>Netdom trust</strong> creates non-Windows, Kerberos trusts that are non-transitive. If you do not specify a value for this parameter, then <strong>netdom trust</strong> displays the current transitivity state. The following list shows the values that you can specify.</p>
<ul>
<li><p>YES: Sets the realm to a transitive trust.</p></li>
<li><p>NO: Sets the realm to a non-transitive trust.</p></li>
</ul><p><b>Note</b><br>On systems that use a locale other than English, use the translated values for âYESâ and âNOâ that correspond to the language locale that the system uses.</p></td>
</tr>
<tr class="odd">
<td><p>/oneside:{TRUSTED| TRUSTING}</p></td>
<td><p>Specifies to create or remove the trust object on only one domain. The following list shows the values that you can specify.</p>
<ul>
<li><p>TRUSTED: Specifies to create or remove the trust object on the trusted domain that you specify in the <strong>/d</strong> or <strong>/domain</strong> parameter.</p></li>
<li><p>TRUSTING: Specifies to create or remove the trust object on the trusting domain. This value is valid only if you specify the <strong>/add</strong> or <strong>/remove</strong> parameter. The <strong>/passwordt</strong> parameter is required when you use the <strong>/add</strong> or <strong>/remove</strong> parameter.</p></li>
</ul></td>
</tr>
<tr class="even">
<td><p>/quarantine[:{YES | NO}]</p></td>
<td><p>Sets or clears the domain quarantine attribute. If you do not specify a value for this parameter, then <strong>netdom trust</strong> displays the current quarantine state. The following list shows the values that you can specify.</p>
<ul>
<li><p>YES: Specifies to accept only security identifiers (SIDs) from the directly-trusted domain for authorization data that <strong>netdom trust</strong> returns during authentication. <strong>Netdom trust</strong> removes SIDS from any other domains.</p></li>
<li><p>NO: Specifies to accept any SID for authorization data that <strong>netdom trust</strong> returns during authentication. This is the default value.</p></li>
</ul></td>
</tr>
<tr class="odd">
<td><p>/namesuffixes:&lt;TrustName&gt;</p></td>
<td><p>Lists the routed name suffixes for <em>TrustName</em> on the domain that <em>TrustingDomainName</em> names. You can use the <strong>/usero</strong> and <strong>/passwordo</strong> parameters for authentication. The <strong>/domain</strong> parameter is not required.</p></td>
</tr>
<tr class="even">
<td><p>/togglesuffix:#</p></td>
<td><p>Changes the status of a name suffix. Used with the <strong>/namesuffixes</strong> parameter. The number of the name entry specified by the <strong>/namesuffixes</strong> parameter must be provided to indicate which name will have its status changed. Names that are in conflict cannot have their status changed until the name in the conflicting trust is disabled. Always precede this command with the <strong>/namesuffixes</strong> parameter because LSA will not always return the names in the same order.</p></td>
</tr>
<tr class="odd">
<td><p>/EnableSIDhistory</p></td>
<td><p>Specifying yes allows users who migrate to the trusted forest from any other forest to use SID history to access resources in this forest. Valid only for an outbound forest trust.</p>
<p><strong>Note</strong>&nbsp;&nbsp;&nbsp;Allow migrated users to use SID history only if you can trust the trusted forest administrators to specify SIDs of this forest in the SID history attribute of their users appropriately.</p>
<p>Specifying <strong>no</strong> would disable the ability of the migrated users in the trusted forest to use SID history to access resources in this forest. Specifying <strong>/EnableSIDHistory</strong> without yes or no will display the current state.</p></td>
</tr>
<tr class="even">
<td><p>/ForestTRANsitive</p></td>
<td><p>Specifying yes marks this trust as forest transitive. Specifying <strong>no</strong> marks this trust as not forest transitive. Specifying <strong>/ForestTRANsitive</strong> without yes or <strong>no</strong> will display the current state of this trust attribute. Valid only for non-Windows realm trusts and can only be performed on the root domain for a forest.</p></td>
</tr>
<tr class="odd">
<td><p>/SelectiveAUTH</p></td>
<td><p>Specifying <strong>no</strong> disables selective authentification across this trust. Specifying <strong>/SelectiveAUTH</strong> without yes or no displays the currrent state of this trust attribute. Specifying yes enables selective authentification across this trust. Valid only on outbound forest and external trusts.</p></td>
</tr>
<tr class="even">
<td><p>/AddTLN</p></td>
<td><p>Adds the specified top level name (DNS name suffix) to the forest trust info for the specified trust. Valid only for a forest transitive non-Windows realm trust and can only be performed on the root domain for a forest. Refer to the <strong>/NameSuffixes</strong> operation for a list of name suffixes.</p></td>
</tr>
<tr class="odd">
<td><p>/AddTLNEX</p></td>
<td><p>Adds the specified top level name exclusion(DNS name suffix) to the forest trust info for the specified trust. Valid only for a forest transitive non-Windows realm trust and can only be performed on the root domain for a forest. Refer to the <strong>/NameSuffixes</strong> operation for a list of name suffixes.</p></td>
</tr>
<tr class="even">
<td><p>/RemoveTLN</p></td>
<td><p>Removes the specified top level name (DNS name suffix) from the forest trust info from the specified trust. Valid only for a forest transitive non-Windows realm trust and can only be performed on the root domain for a forest. Refer to the <strong>/NameSuffixes</strong> operation for a list of name suffixes.</p></td>
</tr>
<tr class="odd">
<td><p>/RemoveTLNEX</p></td>
<td><p>Removes the specified top level name exclusion (DNS Name Suffix) from the forest trust info from the specified trust. Valid only for a forest transitive non-Windows realm trust and can only be performed on the root domain for a forest. Refer to the <strong>/NameSuffixes</strong> operation for a list of name suffixes.</p></td>
</tr>
<tr class="even">
<td><p>/SecurePasswordPrompt</p></td>
<td><p>Use secure credentials popup to specify credentials. This option should be used when smart card credentials are required. This option is only in effect when the password valued is supplied as *.</p></td>
</tr>
<tr class="odd">
<td><p>/EnableTgtDelegation</p></td>
<td><p>Set to no to disable Kerberos full delegation on outbound forest trusts. This prevents services in the other forests from receiving&nbsp;forwarded TGTs. Warning: By setting EnableTgtDelegation to No, services in the other forests with âTrust this computer/user for delegation to any serviceâ configured will not be able to use Kerberos full delegation with any account in this forest to any service.</p></td>

<h2 id="examples">Examples</h2>
<p>When used with the Trust operation, the <strong>/d:</strong> parameter always refers to the trusted domain.</p>
<p>To set the Windows&nbsp;NT&nbsp;4.0 resource domain USA-Chicago to trust the Windows&nbsp;NT&nbsp;4.0 account domain Northamerica, type the following command at the command prompt:</p>
<pre><code>netdom trust /d:Northamerica USA-Chicago /add /Ud:Northamerica\admin /Pd:* /Uo:USA-Chicago\admin /Po:*
</code></pre>
<p>When you press ENTER, you see the following prompt:</p>
<pre><code>Password for Northamerica\admin:
</code></pre>
<p>Enter the password for Northamerica\admin. When you press ENTER, you see the following prompt:</p>
<pre><code>Password for USA-Chicago\admin:
</code></pre>
<p>Type the password for USA-Chicago\admin, and then press ENTER.</p>
<p>The user must have credentials for both domains. You can use the <strong>/pd</strong> parameter to specify the password for Northamerica\admin and the <strong>/po</strong> parameter to specify the password for USA-Chicago\admin. If the user does not provide passwords at the command prompt, the user is prompted for both.</p>
<p>If you want to specify a two-way trust, type the following command at the command prompt</p>
<pre><code>netdom trust /d:marketing.contoso.com engineering.contoso.com /add /twoway /Uo:admin@engineering.contoso.com /Ud:admin@marketing.contoso.com
</code></pre>
<p>To establish a one-way trust where Northamerica trusts the non-Windows, Kerberos realm ATHENA, type the following command at the command prompt:</p>
<pre><code>netdom trust /d:ATHENA Northamerica /add /PT:password /realm
</code></pre>
<p>The <strong>/d</strong> parameter specifies the trusted domain and the <strong>/realm</strong> parameter indicates that this is a non-Windows, Kerberos realm. The order of the domains is not important. You can supply credentials to the Windows&nbsp;2000 domain, if needed.</p>
<div class="NOTE">
<p>Note</p>
<p>Verifying a specific trust relationship requires credentials unless the user has domain administrator privileges on both domains.</p>
</div>
<p>If you want to set the Kerberos realm ATHENA to trust the Northamerica domain, type the following command at the command prompt:</p>
<pre><code>netdom trust /d:Northamerica ATHENA /add
</code></pre>
<div class="NOTE">
<p>Note</p>
<p>To establish a two-way trust, you can specify the <strong>/twoway</strong> parameter.</p>
</div>
<p>Non-Windows Kerberos trusts are created as non-transitive. If you want to change the trust from ATHENA to Northamerica as transitive, type the following command at the command prompt:</p>
<pre><code>netdom trust Northamerica /d:ATHENA /trans:yes
</code></pre>
<p>To display the transitive state, type the following command at the command prompt:</p>
<pre><code>netdom trust Northamerica /d:ATHENA /trans
</code></pre>
<p>The order of these two domains is not important. Either can be the non-Windows, Kerberos domain.</p>
<p>To undo the trust that USA-Chicago has for Northamerica, type the following command at the command prompt:</p>
<pre><code>netdom trust /d:Northamerica USA-Chicago /remove
</code></pre>
<p>To break a two-way trust relationship, type the following command at the command prompt:</p>
<pre><code>netdom trust /d:marketing.contoso.com Engineering.contoso.com /remove /twoway /Uo:admin@engineering.contoso.com /Ud:admin@marketing.contoso.com
</code></pre>
<p>To verify the one-way trust that USA-Chicago has for Northamerica, type the following command at the command prompt:</p>
<pre><code>netdom trust /d:Northamerica USA-Chicago /verify
</code></pre>
<p>To verify a two-way trust between the Northamerica and Europe domains, type the following command at the command prompt:</p>
<pre><code>netdom trust /d:Northamerica EUROPE /verify /twoway
</code></pre>
<p>The <strong>/verify</strong> parameter checks that the appropriate shared secrets are synchronized between the two domains involved in the trust.</p>
<p>To reset the secure channel for the one-way trust between Northamerica and USA-Chicago, type the following command at the command prompt:</p>
<pre><code>netdom trust /d:Northamerica USA-Chicago /Ud:Northamerica\admin /reset
</code></pre>
<p>The <strong>/reset</strong> parameter synchronizes the appropriate shared secrets if they are not already synchronized.</p>
<p>To verify that Kerberos authentication occurs successfully between a workstation and a service that is located in the domain devgroup.example.com, type the following command at the command prompt:</p>
<pre><code>netdom trust /d:devgroup.example.com /verify /KERBEROS
</code></pre>
<p>When you use the <strong>netdom</strong> Trust operation with the <strong>/verify /kerberos</strong> parameters, the trust operation searches for a session ticket for the Kerberos Admin service in the target domain. If the search operation is successful, you can conclude that all Kerberos operations, such as KDC referrals, operate correctly between the workstation and the target domain.</p>
<div class="NOTE">
<p>Note</p>
<p>You cannot run this trust operation from a remote location. You must run the operation on the workstation that you want to test.</p>
</div>
<p>To list the routed name suffixes for the trust between contoso and the trustpartnerdomain, type the following command at the command prompt:</p>
<pre><code>netdom trust contoso /namesuffixes:trustpartnerdomain
</code></pre>
<div class="NOTE">
<p>Note</p>
<p>The <strong>/d</strong> parameter is not needed for this operation which is an exception from other Trust operations.</p>
</div>
<p>This lists all the routed name suffixes for the trust relationship between contoso and the trustpartnerdomain. The trust relationship must be either a Forest Trust relationship or a Non-Windows Realm trust with the <strong>Forest Transitive</strong> attribute set.</p>
<p>The following is sample output:</p>
<pre><code>C:\nt\ds\netapi\netdom\obj\i386&gt;netdom trust shasandom2 /namesuffixes:powermatic

<pre><code>netdom trust myTestDomain /namesuffixes:foresttrustpartnerdomain /togglesuffix:1

<p>Note</p>
<p>You must use the <strong>/ToggleSuffix</strong> parameter with the <strong>/NameSuffixes</strong> parameter. Use <strong>/NameSuffixes</strong> immediately before you use <strong>/ToggleSuffix</strong> because the order in which the name suffixes are listed may change.</p>
</div>
<p>The following is sample output:</p>
<div class="NOTE">
<p>Note</p>
<p>The output reflects the routed name suffix list after the Toggling operation.</p>
</div>

<p>To add the DNS name suffix contoso.com to the Forest Trust Info with trustpartnerdomain, type the following command at the command prompt:</p>
<pre><code>Netdom trust myTestDomain /d:trustPartnerDomain /AddTln:contoso.com
</code></pre>
<p>Adding the DNS name suffix is only allowed for a trust with a Forest Transitive, Non-Windows Realm Trust. This is also true for the following commands:</p>
<ul>
<li><p>Netdom trust myTestDomain /d:trustPartnerDomain /RemoteTln:contoso.com</p>
</li>
<li><p>Netdom trust myTestDomain /d:trustPartnerDomain /AddTLNEx:something.contoso.com</p>
<p>This must have a TLN entry present for the parent naming context, in this case, contoso.com, otherwise the operation is disallowed.)</p>
</li>
<li><p>Netdom trust myTestDomain /d:trustPartnerDomain /RemoveTLNEx:something.contoso.com</p>
</li>
</ul>
<p>The following code lists the name suffixes on a Non-Windows Realm Trust:</p>
<pre><code>C:\nt\ds\netapi\netdom\obj\i386&gt;netdom trust shasandom2 /ns:server.mit.org

<p>The following code adds another TLN:</p>
<pre><code>C:\nt\ds\netapi\netdom\obj\i386&gt;netdom trust shasandom2 /d:domain.mit.org /addtln:dude.com

The TLN or Exclusion was successfully added to the Forest Trust Info.

The command completed successfully.
</code></pre>
<p>The following code adds an invalid TLN exclusion:</p>
<pre><code>C:\nt\ds\netapi\netdom\obj\i386&gt;netdom trust shasandom2 /d:domain.mit.org /addtlnex:dude.com

Try "netdom HELP" for more information.
</code></pre>
<p>The following code adds a valid TLN exclusion:</p>
<pre><code>C:\nt\ds\netapi\netdom\obj\i386&gt;netdom trust shasandom2 /d:domain.mit.org /addtlnex:child.contoso.com

The command completed successfully.
</code></pre>
<p>The following code shows the result of previous operations:</p>
<pre><code>C:\nt\ds\netapi\netdom\obj\i386&gt;netdom trust shasandom2 /ns:server.mit.org
