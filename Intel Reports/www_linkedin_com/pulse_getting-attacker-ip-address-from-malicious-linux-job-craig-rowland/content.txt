              <div class="article-main__content max-w-[744px]"><p>Attackers on Linux often try to establish&nbsp;<a href="https://www.sandflysecurity.com/blog/linux-malware-persistence-with-cron/" target="_blank" rel="nofollow noopener">persistence</a>&nbsp;so they can maintain access to a host. The most common way to do this is with a&nbsp;<em>cron</em>&nbsp;job that is scheduled to run and re-install malware if it has been removed. However, an often overlooked persistence method is putting scheduled jobs in the Linux&nbsp;<em>at</em>&nbsp;job scheduler.</p><p>The&nbsp;<a href="https://linux.die.net/man/1/at" target="_blank" rel="nofollow noopener"><em>at</em></a><em>&nbsp;</em>command allows users to schedule commands to run just like with&nbsp;<em>cron</em>, but they tend to be commands to run one time in the future and not commands to run repeatedly like&nbsp;<em>cron</em>. But consider if an attacker inserts a command to run six months from the date of intrusion. You may think you got rid of them only to find they return at a future date when their command runs and rolls out the welcome mat again for them.</p><p>This is why it’s always a good idea to check your Linux hosts for&nbsp;<em>at</em>&nbsp;commands you don’t recognize with a simple&nbsp;<em>atq</em>&nbsp;command line call when on a host. But also, it’s not a bad idea to manually review&nbsp;<em>at</em>&nbsp;jobs to make sure they are not running things they shouldn’t.&nbsp;For incident responders, checking for malicious&nbsp;<em>at</em>&nbsp;jobs is <strong>mandatory</strong>.&nbsp;This is because if an attacker has inserted an&nbsp;<em>at</em>&nbsp;command you are in luck because by default many Linux systems put the IP address of the system into the entry.</p><h2>Checking for Malicious Linux At Jobs</h2><p>There are two ways to view&nbsp;<em>at</em>&nbsp;jobs on a host. You can use the&nbsp;<em>atq&nbsp;</em>command to dump all at jobs and then the&nbsp;<em>at&nbsp;</em>command to view them.</p><pre spellcheck="false">root@ubuntu-<span class="hljs-number">18</span>-<span class="hljs-number">1</span><span class="hljs-symbol">:~</span><span class="hljs-comment"># atq</span>
<span class="hljs-number">2</span>	Sat Jul <span class="hljs-number">25</span> <span class="hljs-number">21</span><span class="hljs-symbol">:</span><span class="hljs-number">56</span><span class="hljs-symbol">:</span><span class="hljs-number">00</span> <span class="hljs-number">2020</span> a root
<span class="hljs-number">3</span>	Sat Jul <span class="hljs-number">25</span> <span class="hljs-number">21</span><span class="hljs-symbol">:</span><span class="hljs-number">56</span><span class="hljs-symbol">:</span><span class="hljs-number">00</span> <span class="hljs-number">2020</span> a root
<span class="hljs-number">4</span>	Sat Jul <span class="hljs-number">25</span> <span class="hljs-number">21</span><span class="hljs-symbol">:</span><span class="hljs-number">56</span><span class="hljs-symbol">:</span><span class="hljs-number">00</span> <span class="hljs-number">2020</span> a root
<span class="hljs-number">1</span>	Sat Jul <span class="hljs-number">25</span> <span class="hljs-number">21</span><span class="hljs-symbol">:</span><span class="hljs-number">32</span><span class="hljs-symbol">:</span><span class="hljs-number">00</span> <span class="hljs-number">2020</span> a root
root@ubuntu-<span class="hljs-number">18</span>-<span class="hljs-number">1</span><span class="hljs-symbol">:~</span><span class="hljs-comment">#</span>
</pre><p>The above we see four jobs for the root user. The first column is the&nbsp;<em>at</em>&nbsp;job number which we can list individually with the&nbsp;<em>at -c &lt;jobnum&gt;</em>&nbsp;command:</p><pre spellcheck="false">root@ubuntu-18-1:~<span class="hljs-comment"># at -c 1</span>
<span class="hljs-meta">#!/bin/sh</span>
