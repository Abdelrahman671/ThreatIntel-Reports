The threat actor group JavaGhost has been active for over five years and continues to target cloud environments to send out phishing campaigns to unsuspecting targets.\r\n\r\nAccording to website defacement lists such as DefacerID, the group focused historically on defacing websites. However, according to our telemetry, in 2022, they pivoted to sending out phishing emails for financial gain.\r\n\r\nBetween 2022-24, Unit 42 has performed multiple investigations relating to the group JavaGhost, which targeted organizations\u2019 AWS environments. The group focuses on sending phishing campaigns and has not been seen stealing data for extortion during their time in organizations\u2019 AWS environments.\r\n\r\nThese attacks are not due to a vulnerability in AWS. This group takes advantage of misconfigurations in the victim organizations' environments that expose AWS credentials in the form of long-term access keys. They use these leaked keys to initiate all the actions discussed in this report.\u202c\r\n\r\nThis article covers common methodologies that JavaGhost uses to create their phishing infrastructure. We also cover other tactics employed within compromised cloud environments to establish long-term persistence.\r\n\r\nWe have recently observed JavaGhost using advanced evasion methods to cover their tracks. These methods have typically only been used by Scattered Spider, which shows the level of sophistication of this threat actor group.\r\n\r\nAll JavaGhost activities have resulted in a detectable logging footprint, which forms the basis of the alerts at the end of the article.\r\n\r\nPalo Alto Networks customers are better protected through Cortex Cloud and Cortex XSIAM.\r\n\r\nIf you think you might have been compromised or have an urgent matter, contact the Unit 42 Incident Response team.\r\n\r\n\r\n\r\nRelated Unit 42 Topics\r\nCloud Cybersecurity Research, AWS\r\n\r\n\r\n\r\nJavaGhost History\r\nHistorically, JavaGhost participated in the website defacement of numerous entities, starting in 2019. Figure 1 shows some of the websites the group defaced.\r\n\r\n[caption id=\"attachment_138459\" align=\"alignnone\" width=\"1250\"] Figure 1. Websites defaced by JavaGhost. Source: DefacerID.[\/caption]\r\n\r\nThe JavaGhost group also had two websites (shown in Figures 2 and 3). One contains the group\u2019s slogan, \u201cwe are there but not visible.\u201d The other contains text in Indonesian that translates to \u201cstop blaming everything,\u201d which matches with the language used to name some of the resources in their attacks. The site also lists the various group member handles (shown in Figure 3).\r\n\r\n[caption id=\"attachment_138461\" align=\"alignnone\" width=\"1000\"] Figure 2. Historic JavaGhost website. Source: Wayback Machine.[\/caption]\r\n\r\n[caption id=\"attachment_138463\" align=\"alignnone\" width=\"1000\"] Figure 3. Historic JavaGhost website. Source: Wayback Machine.[\/caption]\r\n\r\nBased on our investigations, the group shifted in 2022 from website defacement to sending out phishing campaigns to unsuspecting targets. Datadog reported on this activity shift back in 2023, but the group continues their work, which Unit 42 has seen as recently as December 2024.\r\nAttack Overview\r\nUnit 42 has handled numerous cases in 2022-24 associated with JavaGhost. The attack leveraged overly permissive IAM permissions allowing the victim\u2019s Amazon Simple Email Service (SES) and WorkMail services to send out phishing messages. JavaGhost benefits from using other organizations\u2019 AWS environments because they do not have to pay for any of the created resources. They can also use preexisting SES infrastructure to send out phishing emails.\r\n\r\nUsing preexisting SES infrastructure allows the threat actor\u2019s phishing emails to bypass email protections since the emails originate from a known entity from which the target organization has previously received emails.\r\nInitial Access with Defense Evasion\r\nBetween 2022-24, the group evolved their tactics to more advanced defense evasion techniques that attempt to obfuscate identities in the CloudTrail logs. This tactic has historically been exploited by Scattered Spider. AWS CloudTrail records all management events occurring within an AWS account.\r\n\r\nJavaGhost obtained exposed long-term access keys associated with identity and access management (IAM) users that allowed them to gain initial access to an AWS environment via the command-line interface (CLI). These long-term access keys come from various exposures, as discussed in a prior Unit 42 research article.\r\n\r\nUpon entry to an organization\u2019s AWS environment with the compromised access key, the threat actors do not perform the application programming interface (API) call GetCallerIdentity. Other threat actors often use GetCallerIdentity as their first API call after compromising AWS credentials to enumerate basic information about the compromised account, such as the account ID and user ID.\r\n\r\nBecause defenders frequently anticipate attackers using GetCallerIdentity during initial compromise, JavaGhost evades detection by not using this API call, thereby bypassing any alerts configured to trigger on its execution. Instead, the group performs different first API calls such as GetServiceQuota, GetSendQuota and GetAccount for their initial interaction with a compromised AWS account.\r\n\r\nGetServiceQuota returns the current quota limit for a specified AWS service while GetSendQuota returns the max number of emails that can be sent in 24 hours for SES. GetAccount returns information about the email-sending status of SES and other SES attributes.\r\n\r\nAfter confirming their access to an organization's AWS account with the long-term access key from the CLI, the threat actors behind JavaGhost generate temporary credentials and a login URL to allow themselves console access. Accessing the console via this methodology obfuscates their identity and allows them easier visibility into the resources within an AWS account.\r\n\r\nSince attackers rarely create temporary credentials to access the AWS console URL, these methods often bypass detection. The following section details these techniques, emphasizing how using the console allows attackers to sidestep the restrictions IAM imposes on temporary access keys generated through the CLI.\r\nGetFederationToken and GetSigninToken\r\nGenerating an AWS console login page from long-term access keys takes multiple steps but the entire process can be scripted. NetSPI has created a GitHub repo with an example of how to perform this process and AWS has instructions as well.\r\n\r\nThe first step in this process requires the creation of temporary credentials from the compromised long-term access key. Long-term access keys start with the four letters AKIA, while temporary access keys begin with ASIA.\r\n\r\nTo acquire temporary AWS credentials, JavaGhost uses the GetFederationToken API within the AWS Security Token Service (STS). This API call requires the following parameters:\r\n\r\n \tA name for the federated user\r\n \tAn inline or managed session policy defining the desired permissions (as illustrated in Figure 4). JavaGhost purposefully utilizes an \u201callow all\u201d inline policy to take advantage of the maximum permissions allowed to the underlying IAM user.\r\n \tThe duration for which the temporary credentials should be valid (specified in seconds)\r\n\r\n[caption id=\"attachment_138465\" align=\"alignnone\" width=\"266\"] Figure 4. Example of an inline policy from the GetFederationToken event.[\/caption]\r\n\r\nWhile the AssumeRole API call can also retrieve temporary credentials for this process, JavaGhost opts to use the GetFederationToken option instead. Of note, the inline policy provided in the request does not override the permissions associated with the long-term access key.\r\n\r\nThe permissions granted to the short-term access key result in an intersection of the IAM user permissions associated with the access key and the policies included in the GetFederationToken request. If the GetFederationToken permissions contain broader privileges than the IAM user, then the more limited permissions from the IAM user take effect. Therefore, the provided policy can only reduce and never increase the permissions already granted to the principal represented by the compromised access key and secret key.\r\n\r\nOnce the GetFederationToken request returns the temporary credentials (i.e., sessionId, sessionKey and sessionToken), an encoded URL is required before generating the sign-in token. To generate the encoded URL, the threat actor uses the Python urllib3 library.\r\n\r\nOnce the encoded URL is obtained, a GetSigninToken request returns the information needed to create the URL that allows federated users to access the AWS console. Within the CloudTrail logs associated with the GetSigninToken events, the user agent shows Python-urllib\/3.10, which is how Unit 42 inferred the Python library used by JavaGhost to perform these operations.\r\n\r\nThe generated URL grants access to the console for a default of 15 minutes, which is what the threat actor chose to do. After that, a threat actor must repeat this process to generate a new URL or specify a longer session duration during the GetSigninToken request. The temporary access key generated by the GetFederationToken actions does not need to be regenerated unless the session duration has expired.\r\n\r\nTo revoke the session associated with the compromised credentials, an IAM policy has to be attached directly to the user. The process discussed above does not require the usage of any roles, so there is no built-in way to revoke the session like AWS provides with an IAM role.\r\n\r\nTo stop an active threat actor using this console access method, attaching the AWS managed AWSDenyAll policy invalidates all the permissions for the user. It does not stop an active session in the console, but all attempted actions are blocked.\r\nSetting Up the Phishing Infrastructure\r\nRegarding the SES logging configuration, none of the customer AWS environments from our engagements had SES data events enabled. Therefore, the following analysis focuses solely on CloudTrail Management Events.\r\n\r\nJavaGhost uses SES and WorkMail to configure their phishing infrastructure. The group starts by creating various SES email identities, followed by updating DomainKeys Identified Mail (DKIM) settings. DKIM uses public key cryptography to verify the authenticity of emails.\r\n\r\nThe threat actor group also modified the SES Virtual Delivery Manager (VDM) and Mail-from attributes.\r\n\r\nTo send emails, an SES email or domain identity must exist. The creation of new SES identities appears as CreateEmailIdentity events in the CloudTrail logs and the response elements provide additional details around whether the identity type was a domain or an email address.\r\n\r\nJavaGhost creates multiple email and domain identities as well as modifying the following attributes. The DKIM settings are configured during the user creation and generate the PutEmailIdentityDkimAttributes event in CloudTrail logs.\r\n\r\nWhile DKIM settings can be configured separately from the identity creation process, this group usually updates them during the identity creation itself. The attackers also update the custom Mail-From domain configuration for the email identities. This resulted in the PutEmailIdentityMailFromAttributes event showing the attribute update in the request parameters field within the CloudTrail logs.\r\n\r\nThe group makes various changes to the SES Virtual Delivery Manager (VDM) feature, which also results in the PutAccountVdmAttributes event appearing in the CloudTrail logs.\r\n\r\nIn addition to setting up various email identities, JavaGhost configures an AWS WorkMail Organization and adds WorkMail users. Creating a WorkMail Organization results in numerous SES and AWS Directory Service (DS) events within the CloudTrail logs.\r\n\r\nUpon the creation of the WorkMail Organization seen as CreateOrganization in the CloudTrail logs, the following events appear in the CloudTrail logs associated with SES:\r\n\r\n \tCreateReceiptRule\r\n \tCreateReceiptRuleSet\r\n \tPutIdentityPolicy\r\n \tSetActiveReceiptRuleSet\r\n \tVerifyDomainDkim\r\n \tVerifyDomainIdentity\r\n\r\nIn the console, within the advanced configuration of the WorkMail Organization creation, user directories can either be created from scratch or an existing directory can be used (shown in Figure 5).\r\n\r\n[caption id=\"attachment_138467\" align=\"alignnone\" width=\"826\"] Figure 5. Selecting Create Amazon WorkMail directory generates three Directory Services events automatically in the CloudTrail logs.[\/caption]\r\n\r\nSelecting the creation of a new WorkMail directory automatically generates the following DS CloudTrail events:\r\n\r\n \tAuthorizeApplication\r\n \tCreateAlias\r\n \tCreateIdentityPoolDirectory\r\n\r\nAfter completing the WorkMail Organization creation, the threat actors create various WorkMail users. Creating a WorkMail user generates a CreateUser event (with workmail.amazonaws[.]com as the event source) and the user automatically gets registered to WorkMail with the event RegisterToWorkMail appearing in the CloudTrail logs. The WorkMail registration requires no input from the user when performed through the console.\r\n\r\nFigure 6 shows how to create a new WorkMail user.\r\n\r\n[caption id=\"attachment_138469\" align=\"alignnone\" width=\"763\"] Figure 6. Creating a new WorkMail user.[\/caption]\r\n\r\nBefore sending out the phishing emails, JavaGhosts creates new SMTP credentials. When creating the new SMTP credentials, the threat actors do not change the default username so all the new SMTP usernames start with ses-smtp-user.* Figure 7 shows an example of this.\r\n\r\n[caption id=\"attachment_138471\" align=\"alignnone\" width=\"750\"] Figure 7. Creation of default named SMTP user with default IAM group and permissions.[\/caption]\r\n\r\nCreating new SMTP credentials results in the generation of a new IAM user with the user\u2019s name matching the SMTP username and not the SMTP display name. If the AWS account has not used SES historically, the SMTP creator is prompted that a new IAM user group will be created.\r\n\r\nThis new IAM user group is called AWSSESSendingGroupDoNotRename by default, which also attaches an inline policy to the group allowing ses:SendRawEmail only. These operations appear as CreateGroup and PutGroupPolicy in the CloudTrail logs.\r\n\r\nIf the AWS account has used SMTP credentials historically, the IAM group will most likely already exist and appear in the Permissions list. Figure 8 shows an example of this.\r\n\r\n[caption id=\"attachment_138473\" align=\"alignnone\" width=\"824\"] Figure 8. IAM group details once IAM group already exists.[\/caption]\r\n\r\nAfter finishing user creation, the system displays the new IAM username, along with the SMTP username and SMTP password.\r\n\r\nThe SMTP username displays an access key ID. When reviewing the user in IAM, the SMTP username appears as an access key there as well. Figure 9 shows an example of this.\r\n\r\n[caption id=\"attachment_138475\" align=\"alignnone\" width=\"700\"] Figure 9. Example retrieval of SMTP credentials.[\/caption]\r\n\r\nThe SMTP username still resolves to the AWS account ID if decrypted. All these events appear in the CloudTrail logs as IAM CreateUser, CreateAccessKey and AddUserToGroup events.\r\n\r\nWhen organizations already have SES infrastructure in their AWS environment, JavaGhost uses the preexisting resources to send phishing attacks. Unless dataplane logging is enabled, there are few to no events to review in the CloudTrail logs. The cost for the additional emails sent will appear in the Cost and Usage Reports, but otherwise, only various reconnaissance events result in CloudTrail logs.\r\nIdentity and Access Management (IAM)\r\nThroughout the time frame of the attacks, JavaGhost creates various IAM users, some they use during their attacks and others that they never use. The unused IAM users seem to serve as long-term persistence mechanisms.\r\n\r\nAfter their creation, the threat actor only confirms access via console logins and performs no other actions. The IAM users have a variety of names. Some are meant to blend in with other IAM users that would be typical within an AWS account and others are more obviously named. The IoC section provides a full list of IAM usernames.\r\n\r\nAll the new IAM users have the AWS managed AdministratorAccess policy attached as well as access to the console. The AdministratorAccess policy allows any action against any resource within an AWS account.\r\n\r\nFigure 10 shows the permissions associated with this policy. All of these IAM events appear in the CloudTrail logs as CreateUser, AttachUserPolicy and CreateLoginProfile.\r\n\r\n[caption id=\"attachment_138477\" align=\"alignnone\" width=\"257\"] Figure 10. AWS managed AdministratorAccess policy.[\/caption]\r\n\r\nThe creation of IAM users is a common cloud technique commonly seen within many of our other investigations. JavaGhost sets themselves apart by evolving to use unique methods to access an AWS account.\r\n\r\nIn the initial attacks, this group used the original compromised access key for most of their activity. In 2024, they transitioned to using an IAM role to access the organization\u2019s AWS account from a threat actor-compromised AWS account before proceeding with the attack.\r\n\r\nTo accomplish this, the threat actors created a new IAM role with a trust policy attached, allowing access from a threat actor-controlled AWS account. A trust policy specifies what entities can assume the role.\r\n\r\nThis role creation appears in the CloudTrail logs as CreateRole with the trust policy written in the request parameters field. Figure 11 shows an example of a trust policy.\r\n\r\n[caption id=\"attachment_138479\" align=\"alignnone\" width=\"323\"] Figure 11. Example of an inline trust policy from the CreateRole CloudTrail event.[\/caption]\r\n\r\nIn the case of JavaGhost, the trusted entity belongs to an AWS account. The new role also has unlimited permissions within the environment, with the attachment of the AdministratorAccess policy as seen by the CloudTrail event AttachRolePolicy.\r\n\r\nWith the successful creation of the new administrative role, the threat actor can log into the AWS account from the trusted threat actor-owned AWS account. When the threat actor assumes their role that they created to access the compromised AWS account, CloudTrail records this event as two separate events, AssumeRole and SwitchRole, which occur simultaneously. Unlike the creation of new IAM users, the role creation does not appear suspicious until the trust policy reveals the external access and the role is uncovered as a backdoor.\r\nSecurity Group\r\nThe group continues to leave the same calling card in the middle of their attack by creating new Amazon Elastic Cloud Compute (EC2) security groups named Java_Ghost, with the group description \u201cWe Are There But Not Visible.\u201d These security groups do not contain any security rules and the group typically makes no attempt to attach these security groups to any resources. The creation of the security groups appear in the CloudTrail logs in the CreateSecurityGroup events.\r\n\r\nThis group description matches the group\u2019s slogan on their old website, shown in Figure 12.\r\n\r\n[caption id=\"attachment_138481\" align=\"alignnone\" width=\"1000\"] Figure 12. JavaGhost website. Source: Wayback Machine.[\/caption]\r\nAdditional Suspicious Activity\r\nIn addition to the main components of its phishing attacks, the group attempts two other unique tactics within attacks:\r\n\r\n \tThe group attempts to leave an Organization Unit with the event LeaveOrganization. AWS Organizations help with the management of multiple AWS accounts. They consist of features such as Service Control Policies (SCPs), which help manage IAM permissions at scale, and Organizational Units.\r\n\r\n \tOrganization Units help administrators manage multiple AWS accounts by grouping them together, and they allow for the application of SCPs at the Organization Unit level. Leaving an AWS Organization Unit removes any SCPs that apply to the AWS account and changes the security guardrails that limit activities within an AWS account.\r\n\r\n\r\n \tThe group enables all AWS regions not enabled by default.\r\n\r\n \tAfter March 20, 2019, AWS no longer enables new regions by default and JavaGhost enables those 13 disabled regions as part of their attacks to evade security controls. Enabling regions results in the EnableRegion event in the CloudTrail logs with the region name present in the request parameters field.\r\n\r\n\r\n\r\nConclusion\r\nUnit 42 has investigated multiple JavaGhost cases over the past few years and has observed the group continuously evolving its tactics. Initially, JavaGhost performed attacks using only a compromised access key, but has now advanced to employing sophisticated evasion techniques.                 <h2><a id="post-138456-_4lt92rr5muov"></a>Executive Summary</h2>
<p>Unit 42 researchers have observed phishing activity that we track as TGR-UNK-0011. We assess with high confidence that this cluster overlaps with the threat actor group JavaGhost. The threat actor group JavaGhost has been active for over five years and continues to target cloud environments to send out phishing campaigns to unsuspecting targets.</p>
<p>According to website defacement lists such as DefacerID, the group focused historically on defacing websites. However, according to our telemetry, in 2022, they pivoted to sending out phishing emails for financial gain.</p>
<p>Between 2022-24, Unit 42 has performed multiple investigations relating to the group JavaGhost, which targeted organizations’ AWS environments. The group focuses on sending phishing campaigns and has not been seen stealing data for extortion during their time in organizations’ AWS environments.</p>
<p>These attacks are not due to a vulnerability in AWS. This group takes advantage of misconfigurations in the victim organizations' environments that expose AWS credentials in the form of long-term access keys. They use these leaked keys to initiate all the actions discussed in this report.‬</p>
<p>This article covers common methodologies that JavaGhost uses to create their phishing infrastructure. We also cover other tactics employed within compromised cloud environments to establish long-term persistence.</p>
<p>We have recently observed JavaGhost using advanced evasion methods to cover their tracks. These methods have typically only been used by Scattered Spider, which shows the level of sophistication of this threat actor group.</p>
<p>All JavaGhost activities have resulted in a detectable logging footprint, which forms the basis of the alerts at the end of the article.</p>
<p>Palo Alto Networks customers are better protected through <a href="https://www.paloaltonetworks.com/cortex/cloud" target="_blank" rel="noopener">Cortex Cloud</a> and <a href="https://www.paloaltonetworks.com/cortex/cortex-xsiam" target="_blank" rel="noopener">Cortex XSIAM</a>.</p>
<p>If you think you might have been compromised or have an urgent matter, contact the <a href="https://start.paloaltonetworks.com/contact-unit42.html" target="_blank" rel="noopener">Unit 42 Incident Response team</a>.</p>

<p>Based on our investigations, the group shifted in 2022 from website defacement to sending out phishing campaigns to unsuspecting targets. <a href="https://securitylabs.datadoghq.com/articles/following-attackers-trail-in-aws-methodology-findings-in-the-wild/" target="_blank" rel="noopener">Datadog reported</a> on this activity shift back in 2023, but the group continues their work, which Unit 42 has seen as recently as December 2024.</p>
<h2><a id="post-138456-_5fdtwbu94div"></a>Attack Overview</h2>
<p>Unit 42 has handled numerous cases in 2022-24 associated with JavaGhost. The attack leveraged overly permissive IAM permissions allowing the victim’s Amazon <a href="https://docs.aws.amazon.com/ses/latest/dg/Welcome.html" target="_blank" rel="noopener">Simple Email Service</a> (SES) and <a href="https://docs.aws.amazon.com/workmail/latest/userguide/what_is.html" target="_blank" rel="noopener">WorkMail</a> services to send out phishing messages. JavaGhost benefits from using other organizations’ AWS environments because they do not have to pay for any of the created resources. They can also use preexisting SES infrastructure to send out phishing emails.</p>
<p>Using preexisting SES infrastructure allows the threat actor’s phishing emails to bypass email protections since the emails originate from a known entity from which the target organization has previously received emails.</p>
<h3><a id="post-138456-_4bz0vrd62hdi"></a>Initial Access with Defense Evasion</h3>
<p>Between 2022-24, the group evolved their tactics to more advanced defense evasion techniques that attempt to obfuscate identities in the <a href="https://docs.aws.amazon.com/awscloudtrail/latest/userguide/cloudtrail-user-guide.html" target="_blank" rel="noopener">CloudTrail</a> logs. This tactic has historically been <a href="https://www.crowdstrike.com/en-us/blog/how-adversaries-persist-with-aws-user-federation/" target="_blank" rel="noopener">exploited by Scattered Spider</a>. AWS CloudTrail records all management events occurring within an AWS account.</p>
<p>JavaGhost obtained exposed long-term access keys associated with <a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/introduction.html" target="_blank" rel="noopener">identity and access management</a> (IAM) users that allowed them to gain initial access to an AWS environment via the <a href="https://aws.amazon.com/cli/" target="_blank" rel="noopener">command-line interface</a> (CLI). These long-term access keys come from various exposures, as discussed in a prior <a href="https://unit42.paloaltonetworks.com/large-scale-cloud-extortion-operation/" target="_blank" rel="noopener">Unit 42 research article</a>.</p>
<p>Upon entry to an organization’s AWS environment with the compromised access key, the threat actors do not perform the application programming interface (API) call <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/STS/latest/APIReference/API_GetCallerIdentity.html" target="_blank" rel="noopener">GetCallerIdentity</a></span>. Other threat actors often use <span style="font-family: 'courier new', courier, monospace;">GetCallerIdentity</span> as their first API call after compromising AWS credentials to enumerate basic information about the compromised account, such as the account ID and user ID.</p>
<p>Because defenders frequently anticipate attackers using <span style="font-family: 'courier new', courier, monospace;">GetCallerIdentity</span> during initial compromise, JavaGhost evades detection by <em>not</em> using this API call, thereby bypassing any alerts configured to trigger on its execution. Instead, the group performs different first API calls such as <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/servicequotas/2019-06-24/apireference/API_GetServiceQuota.html" target="_blank" rel="noopener">GetServiceQuota</a></span>, <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/ses/latest/APIReference/API_GetSendQuota.html" target="_blank" rel="noopener">GetSendQuota</a></span> and <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/ses/latest/APIReference-V2/API_GetAccount.html" target="_blank" rel="noopener">GetAccount</a></span> for their initial interaction with a compromised AWS account.</p>
<p><span style="font-family: 'courier new', courier, monospace;">GetServiceQuota</span> returns the current quota limit for a specified AWS service while <span style="font-family: 'courier new', courier, monospace;">GetSendQuota</span> returns the max number of emails that can be sent in 24 hours for SES. <span style="font-family: 'courier new', courier, monospace;">GetAccount</span> returns information about the email-sending status of SES and other SES attributes.</p>
<p>After confirming their access to an organization's AWS account with the long-term access key from the CLI, the threat actors behind JavaGhost generate temporary credentials and a login URL to allow themselves console access. Accessing the console via this methodology obfuscates their identity and allows them easier visibility into the resources within an AWS account.</p>
<p>Since attackers rarely create temporary credentials to access the AWS console URL, these methods often bypass detection. The following section details these techniques, emphasizing how using the console allows attackers to sidestep the restrictions IAM imposes on temporary access keys generated through the CLI.</p>
<h3><a id="post-138456-_d3cz90vnle56"></a>GetFederationToken and GetSigninToken</h3>
<p>Generating an AWS console login page from long-term access keys takes multiple steps but the entire process can be scripted. NetSPI has created a GitHub repo with an <a href="https://github.com/NetSPI/aws_consoler/tree/master" target="_blank" rel="noopener">example of how to perform this process</a> and <a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_enable-console-custom-url.html" target="_blank" rel="noopener">AWS has instructions</a> as well.</p>
<p>The first step in this process requires the creation of temporary credentials from the compromised long-term access key. Long-term access keys start with the four letters <a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/reference_identifiers.html" target="_blank" rel="noopener">AKIA, while temporary access keys begin with ASIA</a>.</p>
<p>To acquire temporary AWS credentials, JavaGhost uses the <span style="font-family: 'courier new', courier, monospace;">GetFederationToken</span> API within the AWS <a href="https://docs.aws.amazon.com/STS/latest/APIReference/welcome.html" target="_blank" rel="noopener">Security Token Service</a> (STS). This API call requires the following parameters:</p>
<ul>
<li>A name for the federated user</li>
<li>An inline or managed session policy defining the desired permissions (as illustrated in Figure 4). JavaGhost purposefully utilizes an “allow all” inline policy to take advantage of the maximum permissions allowed to the underlying IAM user.</li>
<li>The duration for which the temporary credentials should be valid (specified in seconds)</li>
</ul>
<figure id="attachment_138465" aria-describedby="caption-attachment-138465" style="width: 266px" class="wp-caption alignnone"><img  class="wp-image-138465 lozad"  data-src="https://unit42.paloaltonetworks.com/wp-content/uploads/2025/02/word-image-376694-138456-4.png" alt="Text showing a JSON-format AWS IAM policy document with a version date, a statement block that includes an ID, effect set to allow, and wildcard actions and resources." width="266" height="217" /><figcaption id="caption-attachment-138465" class="wp-caption-text">Figure 4. Example of an inline policy from the <span style="font-family: 'courier new', courier, monospace;">GetFederationToken</span> event.</figcaption></figure>
<p>While the <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/STS/latest/APIReference/API_AssumeRole.html" target="_blank" rel="noopener">AssumeRole</a></span> API call can also retrieve temporary credentials for this process, JavaGhost opts to use the <span style="font-family: 'courier new', courier, monospace;">GetFederationToken</span> option instead. Of note, the inline policy provided in the request does not override the permissions associated with the long-term access key.</p>
<p>The permissions granted to the short-term access key result in an <a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_temp_control-access_getfederationtoken.html" target="_blank" rel="noopener">intersection of the IAM user permissions</a> associated with the access key and the policies included in the <span style="font-family: 'courier new', courier, monospace;">GetFederationToken</span> request. If the <span style="font-family: 'courier new', courier, monospace;">GetFederationToken</span> permissions contain broader privileges than the IAM user, then the more limited permissions from the IAM user take effect. Therefore, the provided policy can only reduce and never increase the permissions already granted to the principal represented by the compromised access key and secret key.</p>
<p>Once the <span style="font-family: 'courier new', courier, monospace;">GetFederationToken</span> request returns the temporary credentials (i.e., <span style="font-family: 'courier new', courier, monospace;">sessionId</span>, <span style="font-family: 'courier new', courier, monospace;">sessionKey</span> and <span style="font-family: 'courier new', courier, monospace;">sessionToken</span>), an encoded URL is required before generating the sign-in token. To generate the encoded URL, the threat actor uses the Python <span style="font-family: 'courier new', courier, monospace;">urllib3</span> library.</p>
<p>Once the encoded URL is obtained, a <span style="font-family: 'courier new', courier, monospace;">GetSigninToken</span> request returns the information needed to create the URL that allows federated users to access the AWS console. Within the CloudTrail logs associated with the <span style="font-family: 'courier new', courier, monospace;">GetSigninToken</span> events, the user agent shows <span style="font-family: 'courier new', courier, monospace;">Python-urllib/3.10</span>, which is how Unit 42 inferred the Python library used by JavaGhost to perform these operations.</p>
<p>The generated URL grants access to the console for a default of 15 minutes, which is what the threat actor chose to do. After that, a threat actor must repeat this process to generate a new URL or specify a longer session duration during the <span style="font-family: 'courier new', courier, monospace;">GetSigninToken</span> request. The temporary access key generated by the <span style="font-family: 'courier new', courier, monospace;">GetFederationToken</span> actions does not need to be regenerated unless the session duration has expired.</p>
<p>To revoke the session associated with the compromised credentials, an IAM policy has to be attached directly to the user. The process discussed above does not require the usage of any roles, so there is no built-in way to revoke the session like <a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_use_revoke-sessions.html" target="_blank" rel="noopener">AWS provides with an IAM role</a>.</p>
<p>To stop an active threat actor using this console access method, attaching the AWS managed <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/aws-managed-policy/latest/reference/AWSDenyAll.html" target="_blank" rel="noopener">AWSDenyAll</a></span> policy invalidates all the permissions for the user. It does not stop an active session in the console, but all attempted actions are blocked.</p>
<h3><a id="post-138456-_93h2ynwunn9"></a>Setting Up the Phishing Infrastructure</h3>
<p>Regarding the SES logging configuration, none of the customer AWS environments from our engagements had SES data events enabled. Therefore, the following analysis focuses solely on <a href="https://repost.aws/knowledge-center/cloudtrail-data-management-events" target="_blank" rel="noopener">CloudTrail Management Events</a>.</p>
<p>JavaGhost uses SES and WorkMail to configure their phishing infrastructure. The group starts by creating various SES email identities, followed by updating DomainKeys Identified Mail (DKIM) settings. DKIM uses public key cryptography to verify the authenticity of emails.</p>
<p>The threat actor group also modified the SES <a href="https://docs.aws.amazon.com/ses/latest/dg/vdm.html#:~:text=Virtual%20Deliverability%20Manager%20is%20an,affecting%20your%20delivery%20success%20rate" target="_blank" rel="noopener">Virtual Delivery Manager (VDM)</a> and Mail-from attributes.</p>
<p>To send emails, an SES email or domain identity must exist. The creation of new SES identities appears as <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/ses/latest/APIReference-V2/API_CreateEmailIdentity.html" target="_blank" rel="noopener">CreateEmailIdentity</a> </span>events in the CloudTrail logs and the response elements provide additional details around whether the identity type was a domain or an email address.</p>
<p>JavaGhost creates multiple email and domain identities as well as modifying the following attributes. The DKIM settings are configured during the user creation and generate the <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/ses/latest/APIReference-V2/API_PutEmailIdentityDkimAttributes.html" target="_blank" rel="noopener">PutEmailIdentityDkimAttributes</a></span> event in CloudTrail logs.</p>
<p>While DKIM settings can be configured separately from the identity creation process, this group usually updates them during the identity creation itself. The attackers also update the custom Mail-From domain configuration for the email identities. This resulted in the <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/ses/latest/APIReference-V2/API_PutEmailIdentityMailFromAttributes.html" target="_blank" rel="noopener">PutEmailIdentityMailFromAttributes</a></span> event showing the attribute update in the request parameters field within the CloudTrail logs.</p>
<p>The group makes various changes to the SES Virtual Delivery Manager (VDM) feature, which also results in the <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/ses/latest/APIReference-V2/API_PutAccountVdmAttributes.html" target="_blank" rel="noopener">PutAccountVdmAttributes</a></span> event appearing in the CloudTrail logs.</p>

<h3><a id="post-138456-_494dygbmch9x"></a>Identity and Access Management (IAM)</h3>
<p>Throughout the time frame of the attacks, JavaGhost creates various IAM users, some they use during their attacks and others that they never use. The unused IAM users seem to serve as long-term persistence mechanisms.</p>
<p>After their creation, the threat actor only confirms access via console logins and performs no other actions. The IAM users have a variety of names. Some are meant to blend in with other IAM users that would be typical within an AWS account and others are more obviously named. The <a href="#post-138456-_v43nde4it4br">IoC section</a> provides a full list of IAM usernames.</p>
<p>All the new IAM users have the AWS managed <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/aws-managed-policy/latest/reference/AdministratorAccess.html" target="_blank" rel="noopener">AdministratorAccess</a></span> policy attached as well as access to the console. The <span style="font-family: 'courier new', courier, monospace;">AdministratorAccess</span> policy allows any action against any resource within an AWS account.</p>
<p>Figure 10 shows the permissions associated with this policy. All of these IAM events appear in the CloudTrail logs as <span style="font-family: 'courier new', courier, monospace;">CreateUser</span>, <span style="font-family: 'courier new', courier, monospace;">AttachUserPolicy</span> and <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/IAM/latest/APIReference/API_CreateLoginProfile.html" target="_blank" rel="noopener">CreateLoginProfile</a></span>.</p>
<figure id="attachment_138477" aria-describedby="caption-attachment-138477" style="width: 257px" class="wp-caption alignnone"><img  class="wp-image-138477 lozad"  data-src="https://unit42.paloaltonetworks.com/wp-content/uploads/2025/02/word-image-393297-138456-10.png" alt="Screenshot of a code snippet with a version dated 2012-10-17, containing a statement that includes full access permissions with wildcard characters in the 'Action' and 'Resource' fields." width="257" height="198" /><figcaption id="caption-attachment-138477" class="wp-caption-text">Figure 10. <p>The creation of IAM users is a common cloud technique commonly seen within many of our other investigations. JavaGhost sets themselves apart by evolving to use unique methods to access an AWS account.</p>
<p>In the initial attacks, this group used the original compromised access key for most of their activity. In 2024, they transitioned to using an IAM role to access the organization’s AWS account from a threat actor-compromised AWS account before proceeding with the attack.</p>
<p>To accomplish this, the threat actors created a new IAM role with a <a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles.html" target="_blank" rel="noopener">trust policy</a> attached, allowing access from a threat actor-controlled AWS account. A trust policy specifies what entities can assume the role.</p>
<p>This role creation appears in the CloudTrail logs as <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/IAM/latest/APIReference/API_CreateRole.html" target="_blank" rel="noopener">CreateRole</a></span> with the trust policy written in the request parameters field. Figure 11 shows an example of a trust policy.</p>
<figure id="attachment_138479" aria-describedby="caption-attachment-138479" style="width: 323px" class="wp-caption alignnone"><img  class="wp-image-138479 lozad"  data-src="https://unit42.paloaltonetworks.com/wp-content/uploads/2025/02/word-image-395802-138456-11.png" alt="An image displaying a code snippet related to AWS IAM policy with fields for Version, Statement including Effect, Principal with an AWS account ID, and an empty Condition." width="323" height="258" /><figcaption id="caption-attachment-138479" class="wp-caption-text">Figure 11. Example of an inline trust policy from the <span style="font-family: 'courier new', courier, monospace;">CreateRole</span> CloudTrail event.</figcaption></figure>
<p>In the case of JavaGhost, the trusted entity belongs to an AWS account. The new role also has unlimited permissions within the environment, with the attachment of the <span style="font-family: 'courier new', courier, monospace;">AdministratorAccess</span> policy as seen by the CloudTrail event <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/IAM/latest/APIReference/API_AttachRolePolicy.html" target="_blank" rel="noopener">AttachRolePolicy</a></span>.</p>
<p>With the successful creation of the new administrative role, the threat actor can log into the AWS account from the trusted threat actor-owned AWS account. When the threat actor assumes their role that they created to access the compromised AWS account, CloudTrail records this event as two separate events, <span style="font-family: 'courier new', courier, monospace;">AssumeRole</span> and <span style="font-family: 'courier new', courier, monospace;">SwitchRole</span>, which occur simultaneously. Unlike the creation of new IAM users, the role creation does not appear suspicious until the trust policy reveals the external access and the role is uncovered as a backdoor.</p>

<p>In addition to the main components of its phishing attacks, the group attempts two other unique tactics within attacks:</p>
<ul>
<li>The group attempts to leave an Organization Unit with the event <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/organizations/latest/APIReference/API_LeaveOrganization.html" target="_blank" rel="noopener">LeaveOrganization</a></span>. AWS <a href="https://docs.aws.amazon.com/organizations/latest/userguide/orgs_introduction.html" target="_blank" rel="noopener">Organizations</a> help with the management of multiple AWS accounts. They consist of features such as <a href="https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps.html" target="_blank" rel="noopener">Service Control Policies</a> (SCPs), which help manage IAM permissions at scale, and Organizational Units.
<ul>
<li>Organization Units help administrators manage multiple AWS accounts by grouping them together, and they allow for the application of SCPs at the Organization Unit level. Leaving an AWS Organization Unit removes any SCPs that apply to the AWS account and changes the security guardrails that limit activities within an AWS account.</li>
</ul>
</li>
<li>The group enables all AWS regions not enabled by default.
<ul>
<li>After <a href="https://docs.aws.amazon.com/controltower/latest/userguide/opt-in-region-considerations.html" target="_blank" rel="noopener">March 20, 2019</a>, AWS no longer enables new regions by default and JavaGhost enables those 13 disabled regions as part of their attacks to evade security controls. Enabling regions results in the <span style="font-family: 'courier new', courier, monospace;"><a href="https://docs.aws.amazon.com/accounts/latest/reference/API_EnableRegion.html" target="_blank" rel="noopener">EnableRegion</a></span> event in the CloudTrail logs with the region name present in the request parameters field.</li>
</ul>
</li>
</ul>
<h2><a id="post-138456-_vnoiw4q5latn"></a>Conclusion</h2>
<p>Unit 42 has investigated multiple JavaGhost cases over the past few years and has observed the group continuously evolving its tactics. Initially, JavaGhost performed attacks using only a compromised access key, but has now advanced to employing sophisticated evasion techniques. Luckily, all of the group’s activity results in detectable events within the CloudTrail logs that organizations can hunt for and create new alerts to detect.</p>
<h3><a id="post-138456-_lsbythhc3nz9"></a>Palo Alto Networks Protection and Mitigation</h3>
<p>Palo Alto Networks customers are better protected from the threats discussed above through the following products:</p>
<p><a href="https://www.paloaltonetworks.com/cortex/cloud" target="_blank" rel="noopener">Cortex Cloud</a> and <a href="https://www.paloaltonetworks.com/cortex/cortex-xsiam" target="_blank" rel="noopener">Cortex XSIAM</a> alert on the following activities related to AWS resources:</p>
<ul>
<li>IAM actions such as new user creations, attaching of <span style="font-family: 'courier new', courier, monospace;">AdministratorAccess</span> Policy, <span style="font-family: 'courier new', courier, monospace;">getfederatedtoken</span>, and <span style="font-family: 'courier new', courier, monospace;">getsignintoken</span></li>
<li>Suspicious sending of emails through Simple Email Service (SES)</li>
<li>Use of <span style="font-family: 'courier new', courier, monospace;">getgroup</span> and <span style="font-family: 'courier new', courier, monospace;">putgroup</span> in CloudTrail</li>
</ul>

<p>To mitigate opportunities for attackers to use techniques discussed above, we recommend:</p>
<ul>
<li>Limiting access to administrative rights</li>
<li>Rotating IAM credentials regularly</li>
<li>Using short term/just-in-time (JIT) access tokens</li>
<li>Enabling multi-factor authentication (MFA)</li>
</ul>

(function($) {
	// Migrated from the unit42-v5 + Modifications
	var subscribeSuccess = false;
	var email = document.getElementById('newsletter-email');
	var subscription_form = document.getElementById('unit42footerSubscription_form');
	var subscription_form_button = document.getElementById('unit42footerSubscription_form_button');
	window.captchaComplete = function() {
		subscribeSuccess = true;
		if ($(mail).val() != '' && isEmail($(mail).val())) {
			$(subscription_form_button).removeClass('is-disabled');
		}
		setTimeout(function() {
			$(email).focus();
			$('.g-recaptcha iframe').attr('tabindex', '-1');
		}, 100)
	}
	window.captchaExpires = function() {
		subscribeSuccess = false;
		$(subscription_form_button).addClass('is-disabled', true);
	} 
	$(subscription_form).submit(function(e) {
		e.preventDefault();
		e.stopImmediatePropagation();
		updateEmailMask();
		var success = true;
		var form = $(this);
		var mail = form.find('input[name="Email"]');
		
		if (mail.val() === '') {
			mail.addClass('has-error');
			showError(1);
			success = false;	
		}  else if (!isEmail(mail.val())){
			showError(2);
			success = false;
		}
		else {
			mail.removeClass('has-error');
			$('.error-mail').addClass('d-none');
		}

		if (!subscribeSuccess) {
			$('.error-recaptcha').removeClass('d-none');
		} else {
			$('.error-recaptcha').addClass('d-none');
		}

		if (success && subscribeSuccess) {
			$.ajax({
				type: 'POST',
				url: form.attr('action'),
				data: form.serialize(),
				beforeSend: function() {
                 form.find('button').addClass('is-loading');
    			},
				success: function(msg) {
					form.find('.form-success-message').html('<p class="success-message">You have been successfully subscribed</p>');
					form.find('button').removeClass('is-loading');
					$(email).val('');
					clearError();
				},
				error: function(jqXHR, textStatus, errorThrown) {
					$(subscription_form_button).addClass('is-disabled', true);
					form.find('button').removeClass('is-loading');
					
				}
			});
		}
		return false;
	});
	function showError(error_type){
		if(error_type == 1) {
			$('.error-mail').text("Please enter the email address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		} else if(error_type == 2){
			$('.error-mail').text("Please provide a valid e-mail address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		}	
		$(subscription_form_button).removeClass('is-loading');
	}
	function clearError(){
		$('.error-mail').text("").removeClass('error-show');;
		$(subscription_form_button).removeClass('is-loading');
		$(subscription_form_button).removeClass('is-disabled');
	}
	
	$(email).on('input', function (event) { 
		var email = $(this).val();
		if (isEmail(email) ) {			
			clearError();
		} else if(email == ""){
			clearError();
		} else{
			showError(2);			  
		}
    });

	function isEmail(email) {
		var re = /^(([^<>()\[\]\\.,;:\s@"]+(\.[^<>()\[\]\\.,;:\s@"]+)*)|(".+"))@((\[[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\.[0-9]{1,3}\])|(([a-zA-Z\-0-9]+\.)+[a-zA-Z]{2,}))$/;
		return re.test(String(email).toLowerCase());
	}
	
	var captcha_loaded = false;
         if(!captcha_loaded){
            // recaptcha on foucs call
            $(document).on('change paste keyup', '#newsletter-email', function () {
               if($('.g-recaptcha').hasClass('d-none')){
                $('.g-recaptcha').removeClass('d-none');
               }
               if(!captcha_loaded ){
                  captcha_loaded  = true;
                  // trigger loading api.js (recaptcha.js) script
                  var head = document.getElementsByTagName('head')[0];
                  var script = document.createElement('script');
                  script.type = 'text/javascript';
                  script.src = 'https://www.google.com/recaptcha/api.js?hl=en_US';
                  head.appendChild(script);
               }
            });
        }
	
	function updateEmailMask() {
		var email = $("#unit42footerSubscription_form input[name='Email']").val();
		if (email && email.trim() != '') {
			var maskedEmail = maskEmailAddress(email);
			$("#unit42footerSubscription_form input[name='emailFormMask']").val(maskedEmail);
		}
	}

	function maskEmailAddress (emailAddress) {
		function mask(str) {
			var strLen = str.length;
			if (strLen > 4) {
				return str.substr(0, 1) + str.substr(1, strLen - 1).replace(/\w/g, '*') + str.substr(-1,1);
			}
			return str.replace(/\w/g, '*');
		}
		return emailAddress.replace(/([\w.]+)@([\w.]+)(\.[\w.]+)/g, function (m, p1, p2, p3) {
			return mask(p1) + '@' + mask(p2) + p3;
		});

		return emailAddress;
	}

}(jQuery));
