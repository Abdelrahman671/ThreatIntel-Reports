    <p>Today’s application programming interfaces (APIs) enable flexible and rapid integration among virtually any software, device, or data source. APIs serve a broad range of functionality and provide a foundation for innovation and digital transformation. </p>
<p>APIs have also become the de facto standard for building and connecting modern applications, especially with the increasing move to microservices-based architectures. It is important to properly secure APIs because they serve as the digital glue among systems.</p>
<p>Each of these API calls can potentially open security holes and create privacy risks ranging from poor data validation, configuration errors, and flaws in implementation to a lack of integration between security components. This is important to note when addressing the vulnerabilities defined within the Open Web Application Security Project (OWASP) Top 10 API Security Risks. </p>
<p>On June 5, 2023, OWASP issued <a href="https://owasp.org/API-Security/editions/2023/en/0x03-introduction/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">the first major update</a> to their initial list, which was released in 2019. Let’s review the changes and see which key factors are influencing today’s API vulnerabilities so you can be better informed on your journey to secure your APIs.</p>
<h2>OWASP Top 10 API Security Risks</h2>
<p><a href="https://owasp.org/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">OWASP</a> is a nongovernmental organization that creates security awareness documents based on community feedback and expert assessment, including contributions from Akamai. These documents describe the most common types of vulnerabilities found in organizations today and are an excellent resource for anyone involved with APIs, from developers to CISOs.</p>
<p>OWASP publishes a separate <a href="https://owasp.org/API-Security/editions/2023/en/0x02-foreword/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">Top 10 API Security Risks</a> document in addition to their other Top 10 lists to emphasize that <a href="/solutions/security/app-and-api-security" target="_self">API security</a> requires a unique approach. The <a href="https://owasp.org/www-project-api-security/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">OWASP API Security Project</a> “focuses on strategies and solutions to understand and mitigate the unique vulnerabilities and security risks of … APIs.” </p>
<h2>The differences</h2>
<p>Let’s review what’s different between the 2019 and 2023 editions (Figure 1).</p>

    <p><span class="pre"></span>According to <a href="https://owasp.org/API-Security/editions/2023/en/0x04-release-notes/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">the 2023 release notes</a>:</p>
<p>The OWASP Top 10 API Security Risks 2023 is a forward-looking awareness document for a fast-paced industry. It does not replace other Top 10s. In this edition:</p>
<ul>
<li>We've combined Excessive Data Exposure and Mass Assignment focusing on the common root cause: object property level authorization validation failures.</li>
<li>We've put more emphasis on resource consumption, over-focusing on the pace they are exhausted.</li>
<li>We've created a new category &quot;Unrestricted Access to Sensitive Business Flows&quot; to address new threats, including most of those that can be mitigated using rate limiting.</li>
<li>We added &quot;Unsafe Consumption of APIs&quot; to address something we've started seeing: attackers have started looking for a target's integrated services to compromise those, instead of hitting the APIs of their target directly. This is the right time to start creating awareness about this increasing risk.</li>
</ul>

    <h2>What’s new, what’s in, and what’s out<br />
</h2>
<h2>NEW | API3:2023 | Broken Object Property Level Authorization</h2>
<p>OWASP combined the former categories of Excessive Data Exposure and Mass Assignment into the new <a href="https://owasp.org/API-Security/editions/2023/en/0xa3-broken-object-property-level-authorization/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">Broken Object Property Level Authorization (BOPLA)</a>, focusing on the common root cause: object property level authorization validation failures.</p>
<p>The difference between Broken Object Property Level Authorization (BOLA) and BOPLA is that BOLA refers to a whole object, while BOPLA refers to a property inside an object (Figure 2). </p>

<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-12e5ff481e&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-06-19T12:53:13Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>Both OWASP and Akamai continue to see major risks on the object level, which explains why BOLA remains the first and most critical API security risk to be aware of.&amp;nbsp;&lt;/p>\r\n&lt;p>However, diving one level deeper and looking at the object property level, there is additional risk of oversharing information or allowing for specific properties that can be modified or deleted when that should not be the case.&lt;/p>\r\n&lt;p>Being covered for BOLA doesn’t mean you are covered for BOPLA, and combining Mass Assignment and Excessive Data Exposure into a single category emphasizes the attention required by API developers for properties in an object.&lt;/p>\r\n&lt;p>This distinction is important for those who are choosing an API security product because they need to make sure that the product covers both types of attack.&lt;/p>\r\n&lt;h2>IN | API6:2023 | Server Side Request Forgery&lt;/h2>\r\n&lt;p>OWASP lowered the Injection security risk and, in doing so, removed it from the top 10 and paved the way for &lt;a href=\&#34;https://owasp.org/API-Security/editions/2023/en/0xa7-server-side-request-forgery/\&#34;>Server-Side Request Forgery (SSRF)&lt;/a> to be added.&amp;nbsp;&lt;/p>\r\n&lt;p>SSRF is a type of attack that exploits a vulnerability in a web application or API that allows an attacker to make unauthorized requests from the server to other internal or external systems.&lt;/p>\r\n&lt;p>Here are some potential reasons why OWASP chose to make this change:&amp;nbsp;&lt;/p>\r\n&lt;ul>\r\n&lt;li>APIs are more vulnerable to SSRF attacks because they rely on modern technologies, such as Kubernetes and Docker, which rely on HTTPS API-based communication protocols.&lt;/li>\r\n&lt;li>With technologies such as Webhooks, SSO integrations, and URL file fetching/redirects through API endpoints, there is an opportunity for threat actors to apply SSRF.&lt;/li>\r\n&lt;/ul>\r\n&lt;h3>A deeper dive&lt;/h3>\r\n&lt;p>For a deeper dive into SSRF attacks, read Mike Elissen’s article &lt;a href=\&#34;https://blog.securitylevelup.eu/your-apis-are-enabling-server-side-request-forgery-ssrf-attacks\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34; referrerpolicy=\&#34;no-referrer\&#34;>Your APIs Are Enabling Server-Side Request Forgery (SSRF) Attacks&lt;/a>.&lt;/p>\r\n&lt;h2>OUT | API8:2019 | Injection&lt;/h2>\r\n&lt;p>Removing Injection attacks from the list was a bold and contentious move within the API security community, but there is a reduced threat from Injection attacks on API endpoints.&amp;nbsp;&lt;/p>\r\n&lt;p>Injection is now is essentially part of &lt;a href=\&#34;https://owasp.org/API-Security/editions/2023/en/0xa8-security-misconfiguration/\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34; referrerpolicy=\&#34;no-referrer\&#34;>API8:2023 | Security Misconfiguration&lt;/a>. Proper security configuration should include web application and API protection mechanisms, which should scan and prevent injections by default.&lt;/p>\r\n&lt;p>GraphQL is growing as an API technology. It is, at its core, a querying language that could re-open the door to a rise in Injection attacks, so API developers who rely on GraphQL should continue to stay vigilant.&lt;/p>\r\n&lt;h2>NEW | API4:2023 | Unrestricted Resource Consumption&lt;/h2>\r\n&lt;p>The original list focused specifically on understanding the risk of resource consumption that leads to API endpoints becoming overwhelmed (and potentially unavailable) harming user experience.&amp;nbsp;&lt;/p>\r\n&lt;p>API endpoints these days need to be available — but just being available isn&#39;t everything. Implementing API gateway, load balancing, or rate limiting controls is a step in the right direction.&amp;nbsp;&lt;/p>\r\n&lt;p>In recent years, we are seeing a huge shift in the “economics of API usage.” This &lt;a href=\&#34;https://owasp.org/API-Security/editions/2023/en/0xa4-unrestricted-resource-consumption/\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34; referrerpolicy=\&#34;no-referrer\&#34;>updated category&lt;/a> takes aim at creating awareness of this aspect, which will continue to grow with third-party API integrations.&lt;/p>\r\n&lt;h2>NEW | API6:2023 | Unrestricted Access to Sensitive Business Flows&lt;/h2>\r\n&lt;p>Another new addition is &lt;a href=\&#34;https://owasp.org/API-Security/editions/2023/en/0xa6-unrestricted-access-to-sensitive-business-flows/\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34; referrerpolicy=\&#34;no-referrer\&#34;>API6:2023 Unrestricted Access to Sensitive Business Flows&lt;/a>. This category has finally codified what makes security so special: to think more like a threat actor and see where potential gains can be.&lt;/p>\r\n&lt;p>The technology (the API) is just a way to execute the business logic, and having ways of bypassing or altering this business logic through the technology is unwanted and could lead to complications.&lt;/p>\r\n&lt;p>OWASP shared specific examples of how these complications can be prevented, but this security risk is very specific to the business logic that your API endpoints are supporting.&lt;/p>\r\n&lt;h3>API developers: Example&lt;/h3>\r\n&lt;p>Recently, Mike Elissen had a conversation with API developers at a streaming service. To attract a new audience, this streaming service offered a free 30-day trial for new users who shared their credit card information.&amp;nbsp;&lt;/p>\r\n&lt;p>However, the business logic was only looking at unique email addresses for new sign-ups. Special thanks to Erez Yallon, Inon Shkedy, and Paulo Silva for all their great work on the 2023 edition.&amp;nbsp;&lt;/p>\r\n&lt;p>We also want to thank all the contributors, specifically Akamai’s &lt;a href=\&#34;https://www.linkedin.com/in/zavodchik/\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34; referrerpolicy=\&#34;no-referrer\&#34;>Maxim Zavodchik&lt;/a> and &lt;a href=\&#34;https://www.linkedin.com/in/mikeelissen/\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34; referrerpolicy=\&#34;no-referrer\&#34;>Mike Elissen&lt;/a> for participating in this project and educating the larger developer community on API security.&lt;/p>\r\n&lt;h2>Additional information about APIs&lt;/h2>\r\n&lt;p>Akamai tracks the use of APIs and the amount of API traffic as part of its State of the Internet (SOTI) reports.     <p>Both OWASP and Akamai continue to see major risks on the object level, which explains why BOLA remains the first and most critical API security risk to be aware of. </p>
<p>However, diving one level deeper and looking at the object property level, there is additional risk of oversharing information or allowing for specific properties that can be modified or deleted when that should not be the case.</p>
<p>Being covered for BOLA doesn’t mean you are covered for BOPLA, and combining Mass Assignment and Excessive Data Exposure into a single category emphasizes the attention required by API developers for properties in an object.</p>
<p>This distinction is important for those who are choosing an API security product because they need to make sure that the product covers both types of attack.</p>
<h2>IN | API6:2023 | Server Side Request Forgery</h2>
<p>OWASP lowered the Injection security risk and, in doing so, removed it from the top 10 and paved the way for <a href="https://owasp.org/API-Security/editions/2023/en/0xa7-server-side-request-forgery/">Server-Side Request Forgery (SSRF)</a> to be added. </p>
<p>SSRF is a type of attack that exploits a vulnerability in a web application or API that allows an attacker to make unauthorized requests from the server to other internal or external systems.</p>
<p>Here are some potential reasons why OWASP chose to make this change: </p>
<ul>
<li>APIs are more vulnerable to SSRF attacks because they rely on modern technologies, such as Kubernetes and Docker, which rely on HTTPS API-based communication protocols.</li>
<li>With technologies such as Webhooks, SSO integrations, and URL file fetching/redirects through API endpoints, there is an opportunity for threat actors to apply SSRF.</li>
</ul>
<h3>A deeper dive</h3>
<p>For a deeper dive into SSRF attacks, read Mike Elissen’s article <a href="https://blog.securitylevelup.eu/your-apis-are-enabling-server-side-request-forgery-ssrf-attacks" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">Your APIs Are Enabling Server-Side Request Forgery (SSRF) Attacks</a>.</p>
<h2>OUT | API8:2019 | Injection</h2>
<p>Removing Injection attacks from the list was a bold and contentious move within the API security community, but there is a reduced threat from Injection attacks on API endpoints. </p>
<p>Injection is now is essentially part of <a href="https://owasp.org/API-Security/editions/2023/en/0xa8-security-misconfiguration/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">API8:2023 | Security Misconfiguration</a>. Proper security configuration should include web application and API protection mechanisms, which should scan and prevent injections by default.</p>
<p>GraphQL is growing as an API technology. It is, at its core, a querying language that could re-open the door to a rise in Injection attacks, so API developers who rely on GraphQL should continue to stay vigilant.</p>
<h2>NEW | API4:2023 | Unrestricted Resource Consumption</h2>
<p>The original list focused specifically on understanding the risk of resource consumption that leads to API endpoints becoming overwhelmed (and potentially unavailable) harming user experience. </p>
<p>API endpoints these days need to be available — but just being available isn't everything. Implementing API gateway, load balancing, or rate limiting controls is a step in the right direction. </p>
<p>In recent years, we are seeing a huge shift in the “economics of API usage.” This <a href="https://owasp.org/API-Security/editions/2023/en/0xa4-unrestricted-resource-consumption/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">updated category</a> takes aim at creating awareness of this aspect, which will continue to grow with third-party API integrations.</p>
<h2>NEW | API6:2023 | Unrestricted Access to Sensitive Business Flows</h2>
<p>Another new addition is <a href="https://owasp.org/API-Security/editions/2023/en/0xa6-unrestricted-access-to-sensitive-business-flows/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">API6:2023 Unrestricted Access to Sensitive Business Flows</a>. This category has finally codified what makes security so special: to think more like a threat actor and see where potential gains can be.</p>
<p>The technology (the API) is just a way to execute the business logic, and having ways of bypassing or altering this business logic through the technology is unwanted and could lead to complications.</p>
<p>OWASP shared specific examples of how these complications can be prevented, but this security risk is very specific to the business logic that your API endpoints are supporting.</p>
<h3>API developers: Example</h3>
<p>Recently, Mike Elissen had a conversation with API developers at a streaming service. To attract a new audience, this streaming service offered a free 30-day trial for new users who shared their credit card information. </p>
<p>However, the business logic was only looking at unique email addresses for new sign-ups. A new email address could easily access a new trial using the same credit card information, which could lead to users who create new accounts each month,  using the service for free indefinitely. </p>
<h2>NEW | API10:2023 | Unsafe Consumption of APIs</h2>
<p>As third-party <a href="https://owasp.org/API-Security/editions/2023/en/0xaa-unsafe-consumption-of-apis/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">API consumption</a> is growing rapidly, APIs often have to integrate and communicate with different internal and external (third-party) services, sending and receiving data.</p>
<p>It is important to follow the “basic” rules of security in those cases, too, and it can be complicated for security products to detect vulnerabilities and proactively defend in this space. </p>
<p>OWASP includes a range of suggestions in their document, both general and API-specific, such as:</p>
<ul>
<li>Follow redirections carefully. Build this oversight into the business logic as well as add security solutions that are continuously monitoring and inspecting the traffic flows.</li>
<li>Do not simply trust third-party APIs, even if they have a good reputation. Build defenses and limits into your application and API endpoints.</li>
</ul>
<h2>Akamai can help with API security</h2>
<p>Organizations and their security vendors must work closely together, aligning across people, processes, and technologies to institute a solid defense against the security risks outlined in the <a href="https://owasp.org/API-Security/editions/2023/en/0x11-t10/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">OWASP Top 10 API Security Risks</a>. </p>
<p>Akamai provides industry-leading security solutions, highly experienced experts, and the <a href="/solutions/cloud-computing" target="_self">Akamai Connected Cloud</a>, which gleans insight from millions of web application attacks, billions of bot requests, and trillions of API requests every single day. Akamai’s web application and API security solutions will help secure your organization against the most-advanced forms of web application, distributed denial-of-service, and API-based attacks. </p>
