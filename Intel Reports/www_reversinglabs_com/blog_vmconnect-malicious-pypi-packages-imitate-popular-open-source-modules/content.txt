<h1><span id="hs_cos_wrapper_name" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="text">VMConnect: Malicious PyPI packages imitate popular open source modules</span></h1>
<div id="hs_cos_wrapper_module_1577203992502740" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module">
<h2 class="post-header__secondary-head blog__post-secondary-h2">ReversingLabs threat researchers have identified a new malicious PyPI campaign that includes a suspicious VMConnect package published to the PyPI repo.</h2>
</div>
</div>
<div class="blog-author__short">
<div class="blog-author__short-info">

<div id="hs_cos_wrapper_module_1705062190107298" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module"></div>
<div class="blog__post-content html-content">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p style="text-align: left;"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Blog.jpg?width=1400&amp;height=732&amp;name=VMConnect-Blog.jpg" alt="VMConnect-Blog" loading="lazy" style="height: auto; max-width: 100%; width: 1400px;" width="1400" height="732" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Blog.jpg?width=700&amp;height=366&amp;name=VMConnect-Blog.jpg 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Blog.jpg?width=1400&amp;height=732&amp;name=VMConnect-Blog.jpg 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Blog.jpg?width=2100&amp;height=1098&amp;name=VMConnect-Blog.jpg 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Blog.jpg?width=2800&amp;height=1464&amp;name=VMConnect-Blog.jpg 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Blog.jpg?width=3500&amp;height=1830&amp;name=VMConnect-Blog.jpg 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Blog.jpg?width=4200&amp;height=2196&amp;name=VMConnect-Blog.jpg 4200w" sizes="(max-width: 1400px) 100vw, 1400px"><br>ReversingLabs has identified several malicious Python packages on the Python Package Index (PyPI) open source repository. In all, ReversingLabs researchers uncovered 24 malicious packages imitating three, popular open source Python tools: <span style="font-style: italic;">vConnector</span>, a wrapper module for <a href="https://github.com/vmware/pyvmomi"><span>pyVmomi VMware vSphere bindings</span></a>; as well as <span style="font-style: italic;">eth-tester</span>, a collection of tools for testing ethereum based applications; and <span style="font-style: italic;">databases</span>,<strong> </strong>a tool that gives <a href="https://docs.python.org/3/library/asyncio.html"><span>asyncro</span></a> support for a range of databases.</p>
<!--more-->
<p>Based on the research team's observations, the campaign began on or around July 28, 2023, when the first of the malicious packages were published. It continues to the current day, with new, malicious PyPI packages posted on a daily basis, as prior packages are detected and removed.</p>
<p>In contrast to other, recent supply chain campaigns, such as <a href="https://www.reversinglabs.com/blog/operation-brainleeches-malicious-npm-packages-fuel-supply-chain-and-phishing-attacks"><span>Operation Brainleeches</span></a>, the malicious packages that make up this campaign display evidence of a concerted effort to deceive developers. They achieve this by implementing the entire functionality of the modules they are imitating and standing up corresponding and linked GitHub projects that omit the malicious functionality found in the PyPI release package.&nbsp;&nbsp;</p>
<p>This is not the first time that we have observed such behavior. In June, 2022, for example, we discovered an <a href="https://www.reversinglabs.com/blog/npm-coinminer-what-you-see-is-not-always-what-you-get-0"><span>npm malicious package</span></a>, <span style="font-style: italic;">maintenancewebsite</span>, which used a similar approach to hide cryptomining features. The VMConnect campaign is the latest example of open source modules being used to propagate malicious code, and more evidence that security assessments of open source code repositories may miss these nuanced attacks.</p>
<p>The ReversingLabs research team is continuously monitoring open-source package repositories for instances of malicious code tampering, the planting of malicious packages or dependencies and other forms of software supply chain attacks. This work involves both automated and human-led scanning and analysis of packages published in the most popular public package repositories like npm, PyPI, Ruby and NuGet.&nbsp;</p>
<p>Historically, the vast majority of the malicious supply chain campaigns we have identified were found on the npm open source repository, which is home to the lion’s share of open source projects and developers. However, in recent months other platforms, in particular, the Python Package Index (PyPI) have seen increased malicious activity. In February, for example, ReversingLabs researchers discovered <a href="https://www.reversinglabs.com/blog/beware-impostor-http-libraries-lurk-on-pypi"><span>41 malicious PyPI packages posing as HTTP libraries</span></a>, with some mimicking popular and widely used libraries. In March, we <a href="https://www.reversinglabs.com/blog/package-names-repurposed-to-push-malware-on-pypi"><span>encountered a malicious PyPI package named termcolour,</span></a> a three-stage downloader published in multiple versions that co-opted the name of a defunct PyPI package. Then, in May, PyPI <a href="https://www.reversinglabs.com/blog/pypi-paused-as-automated-attack-overwhelms-admins"><span>briefly stopped accepting new submissions </span></a>after it was overwhelmed with malicious submissions.&nbsp;</p>
<p>ReversingLabs threat researchers have identified a new malicious PyPI campaign that includes a suspicious VMConnect package published to the PyPI repository.</p>
<h2 style="font-weight: bold;">Suspicious behavior detected</h2>
<p>This package was declared suspicious during routine scanning by ReversingLabs Titanium Platform, a powerful static analysis engine capable of extracting various types of metadata from a wide range of file formats.&nbsp;</p>
<p>Besides the various types of metadata, the ReversingLabs Titanium Platform is also capable of extracting behavior indicators, making it easier to understand functional capabilities of a file. And that capability is what drew our attention to the<em> _init_.py</em> file within VMConnect. Code inside VMConnect’s <em>__init__.py </em>file is capable of creating a process, decoding data using the Base64 algorithm and converting binary data to its string representation — a behavior commonly used in obfuscation. This combination of behaviors is what triggered the initial detection and prompted further investigation.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/vmconnect-blog-figure1-behaviors.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure1-behaviors.png?width=1455&amp;name=vmconnect-blog-figure1-behaviors.png" alt="Behavior indicators extracted from VMConnect package" style="width: 1455px;" width="1455" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure1-behaviors.png?width=728&amp;name=vmconnect-blog-figure1-behaviors.png 728w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure1-behaviors.png?width=1455&amp;name=vmconnect-blog-figure1-behaviors.png 1455w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure1-behaviors.png?width=2183&amp;name=vmconnect-blog-figure1-behaviors.png 2183w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure1-behaviors.png?width=2910&amp;name=vmconnect-blog-figure1-behaviors.png 2910w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure1-behaviors.png?width=3638&amp;name=vmconnect-blog-figure1-behaviors.png 3638w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure1-behaviors.png?width=4365&amp;name=vmconnect-blog-figure1-behaviors.png 4365w" sizes="(max-width: 1455px) 100vw, 1455px"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 1: Behavior indicators extracted from VMConnect package.</span></p>
<p>A detailed look at the <em>__init__.py </em>file confirmed the presence of malicious functionality inside the package. That started with that Base64 encoded string, which gets decoded and executed in another process.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/vmconnect-blog-figure2-init.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure2-init.png?width=1455&amp;name=vmconnect-blog-figure2-init.png" alt="Execution of Base64 encoded string" style="width: 1455px;" width="1455" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure2-init.png?width=728&amp;name=vmconnect-blog-figure2-init.png 728w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure2-init.png?width=1455&amp;name=vmconnect-blog-figure2-init.png 1455w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure2-init.png?width=2183&amp;name=vmconnect-blog-figure2-init.png 2183w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure2-init.png?width=2910&amp;name=vmconnect-blog-figure2-init.png 2910w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure2-init.png?width=3638&amp;name=vmconnect-blog-figure2-init.png 3638w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure2-init.png?width=4365&amp;name=vmconnect-blog-figure2-init.png 4365w" sizes="(max-width: 1455px) 100vw, 1455px"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 2: Execution of Base64 encoded string inside the <em>__init__.py</em> file.</span></p>
<p>When we decode the string, we discovered that it contains a download URL which is modified based on the information collected from the host machine. The substring <em>paperpin3902 </em>in the command and control URL is replaced with a string containing the first letter of the host’s platform name, username and a random, 6 character-long string.</p>
<p><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure3-config.png?width=570&amp;height=226&amp;name=vmconnect-blog-figure3-config.png" alt="C2 URL template, extracted from the Base64 encoded string" loading="lazy" style="height: auto; max-width: 100%; width: 570px; margin-left: auto; margin-right: auto; display: block;" width="570" height="226" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure3-config.png?width=285&amp;height=113&amp;name=vmconnect-blog-figure3-config.png 285w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure3-config.png?width=570&amp;height=226&amp;name=vmconnect-blog-figure3-config.png 570w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure3-config.png?width=855&amp;height=339&amp;name=vmconnect-blog-figure3-config.png 855w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure3-config.png?width=1140&amp;height=452&amp;name=vmconnect-blog-figure3-config.png 1140w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure3-config.png?width=1425&amp;height=565&amp;name=vmconnect-blog-figure3-config.png 1425w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure3-config.png?width=1710&amp;height=678&amp;name=vmconnect-blog-figure3-config.png 1710w" sizes="(max-width: 570px) 100vw, 570px"></p>
<p style="font-weight: bold; font-size: 16px; text-align: center;">Figure 3: C2 URL template, extracted from the Base64 encoded string, gets modified based on host machine information.</p>
<p>The decoded and executed Base64 string contains an endless execution loop which contacts the command and control (C2) server and attempts to download another Base64 encoded string with additional commands. If it succeeds, that code is executed and the loop repeats, with the C2 server polled by the infected host for new commands after a preconfigured sleep period.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/vmconnect-blog-figure4-loop.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure4-loop.png?width=1455&amp;name=vmconnect-blog-figure4-loop.png" alt="Command fetching and execution loop extracted from the Base64 encoded string" style="width: 1455px;" width="1455" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure4-loop.png?width=728&amp;name=vmconnect-blog-figure4-loop.png 728w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure4-loop.png?width=1455&amp;name=vmconnect-blog-figure4-loop.png 1455w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure4-loop.png?width=2183&amp;name=vmconnect-blog-figure4-loop.png 2183w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure4-loop.png?width=2910&amp;name=vmconnect-blog-figure4-loop.png 2910w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure4-loop.png?width=3638&amp;name=vmconnect-blog-figure4-loop.png 3638w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure4-loop.png?width=4365&amp;name=vmconnect-blog-figure4-loop.png 4365w" sizes="(max-width: 1455px) 100vw, 1455px"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 4: Command fetching and execution loop extracted from the Base64 encoded string present in the <em>__init__.py</em>file.</span></p>
<p>Although the C2 server was live at the time of this research, the research team did not observe it serving any commands. Since the command fetching is performed in an endless loop, it is possible that the operator of the C2 server uploads commands only after the infected machine is determined to be interesting to the threat actor. Alternatively, the C2 server could be performing some type of request filtering. For example, attackers may filter requests based on the IP address of the infected machine to avoid infecting targets from specific countries.</p>
<h2 style="font-weight: bold;">Github misdirection&nbsp;</h2>
<p>The VMConnect package was published on July 28th, by a developer named <em>Hushki Manager</em> — a throw-away PyPI account created the same day when its only package was published.&nbsp;</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/vmconnect-blog-figure5-PyPI_author.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure5-PyPI_author.png?width=1455&amp;name=vmconnect-blog-figure5-PyPI_author.png" alt="PyPI profile of hushki502 user" style="width: 1455px;" width="1455" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure5-PyPI_author.png?width=728&amp;name=vmconnect-blog-figure5-PyPI_author.png 728w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure5-PyPI_author.png?width=1455&amp;name=vmconnect-blog-figure5-PyPI_author.png 1455w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure5-PyPI_author.png?width=2183&amp;name=vmconnect-blog-figure5-PyPI_author.png 2183w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure5-PyPI_author.png?width=2910&amp;name=vmconnect-blog-figure5-PyPI_author.png 2910w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure5-PyPI_author.png?width=3638&amp;name=vmconnect-blog-figure5-PyPI_author.png 3638w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure5-PyPI_author.png?width=4365&amp;name=vmconnect-blog-figure5-PyPI_author.png 4365w" sizes="(max-width: 1455px) 100vw, 1455px"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 5: PyPI profile of hushki502 user.</span></p>
<p>Despite that, the actor did put a lot of effort into making the package look trustworthy. For example, it has a legitimate-looking description, and that description corresponds to the functionality present inside the package.&nbsp;Threat actors often don’t make the effort to tidy up the project's description on the PyPI website.</p>
<p>Instead, they often rely on simple typosquatting of package names in order to trick enough developers into installing their creation. For example, the threat actors behind the recently discovered <a href="https://www.reversinglabs.com/blog/operation-brainleeches-malicious-npm-packages-fuel-supply-chain-and-phishing-attacks"><span>Operation Brainleeches</span></a> campaign made minimal efforts to disguise their malicious packages, using default file names like index.html and DEMO.txt likely copied from phishing kits and minimally altered before being published.&nbsp;</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/vmconnect-blog-figure6-PyPI_project.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure6-PyPI_project.png?width=1455&amp;name=vmconnect-blog-figure6-PyPI_project.png" alt="VMConnect package description." style="width: 1455px;" width="1455" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure6-PyPI_project.png?width=728&amp;name=vmconnect-blog-figure6-PyPI_project.png 728w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure6-PyPI_project.png?width=1455&amp;name=vmconnect-blog-figure6-PyPI_project.png 1455w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure6-PyPI_project.png?width=2183&amp;name=vmconnect-blog-figure6-PyPI_project.png 2183w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure6-PyPI_project.png?width=2910&amp;name=vmconnect-blog-figure6-PyPI_project.png 2910w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure6-PyPI_project.png?width=3638&amp;name=vmconnect-blog-figure6-PyPI_project.png 3638w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure6-PyPI_project.png?width=4365&amp;name=vmconnect-blog-figure6-PyPI_project.png 4365w" sizes="(max-width: 1455px) 100vw, 1455px"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 6: VMConnect package description.</span></p>
<p>In this campaign, however, the attackers took a more studied approach. In addition to reusing the description from the actual packages, the attackers properly set links to a Github source code repository which was created by the same author on the same day. The Github project site looks trustworthy, with a description matching the one on the PyPI site and several commits and nothing obviously suspicious in the published files.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/vmconnect-blog-figure7-husky_github.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure7-husky_github.png?width=1455&amp;name=vmconnect-blog-figure7-husky_github.png" alt="Github profile of hushki502 user." style="width: 1455px;" width="1455" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure7-husky_github.png?width=728&amp;name=vmconnect-blog-figure7-husky_github.png 728w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure7-husky_github.png?width=1455&amp;name=vmconnect-blog-figure7-husky_github.png 1455w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure7-husky_github.png?width=2183&amp;name=vmconnect-blog-figure7-husky_github.png 2183w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure7-husky_github.png?width=2910&amp;name=vmconnect-blog-figure7-husky_github.png 2910w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure7-husky_github.png?width=3638&amp;name=vmconnect-blog-figure7-husky_github.png 3638w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure7-husky_github.png?width=4365&amp;name=vmconnect-blog-figure7-husky_github.png 4365w" sizes="(max-width: 1455px) 100vw, 1455px"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 7: Github profile of hushki502 user.</span></p>
<p>Finally, the malicious actors took care to hide the malicious nature of this tool by omitting malicious functionality from the <em>__init__.py</em> file that was published to the Github repository.&nbsp;</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/vmconnect-blog-figure8-init_gihub.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure8-init_gihub.png?width=1455&amp;name=vmconnect-blog-figure8-init_gihub.png" alt="Benign version of file" style="width: 1455px;" width="1455" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure8-init_gihub.png?width=728&amp;name=vmconnect-blog-figure8-init_gihub.png 728w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure8-init_gihub.png?width=1455&amp;name=vmconnect-blog-figure8-init_gihub.png 1455w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure8-init_gihub.png?width=2183&amp;name=vmconnect-blog-figure8-init_gihub.png 2183w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure8-init_gihub.png?width=2910&amp;name=vmconnect-blog-figure8-init_gihub.png 2910w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure8-init_gihub.png?width=3638&amp;name=vmconnect-blog-figure8-init_gihub.png 3638w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure8-init_gihub.png?width=4365&amp;name=vmconnect-blog-figure8-init_gihub.png 4365w" sizes="(max-width: 1455px) 100vw, 1455px"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 8: Benign version of<em>__init__.py </em>file present in Github repository.</span></p>
<p>The use of corresponding GitHub repositories to create the impression of a legitimate open source package is something we have seen before - and there’s a good explanation for why threat actors are taking the trouble to do this.&nbsp;</p>
<p>Historically, many supply chain security solutions have relied on source code reviews of third party libraries. Attackers, therefore, have an incentive to throw code scanning tools and manual code reviewers off their scent. For reviewers, PyPI projects tend to look more trustworthy if they have a corresponding link to a Github repository that doesn’t seem arbitrary.</p>
<p>In the case of the malicious PyPI packages, the attacker created a phony Github repository with an identical name as the PyPI package and copied the entire functionality from the legitimate Github project into the corresponding PyPI project. However, the malicious functionality is not present within the source code. It is only by scanning the artifacts used in the build process that this threat would have been detected.&nbsp;</p>
<p>ReversingLabs observed a similar tactic used in the case of the <a href="https://www.reversinglabs.com/blog/npm-coinminer-what-you-see-is-not-always-what-you-get-0"><span>npm coinminer</span></a> discovered a year ago. In that campaign, the content of the release package - a temporary “maintenance mode” website” — was also different from the content hosted in the corresponding source code repository. As with the current PyPI packages, malicious content was added to the legitimate public source code resulting in a PyPI release package that looks much the same as the open source code it is built on, but with some subtle (and malicious) changes that are easily overlooked.&nbsp;</p>
<p>The lesson for development and application security teams is that release packages can (and do) contain malicious functionality which isn’t present in open source repositories and therefore can’t be detected with source code scanning or manual source code reviews alone. These teams need a method for detecting suspicious content in the final release packages themselves to avoid falling victim to supply chain attacks such as this VMConnect PyPI campaign.&nbsp;</p>
<h2 style="font-weight: bold;">Imitation: The sincerest form of thievery</h2>
<p>It is also educational to take a closer look at the methods the attackers in this case used to disguise their malicious intent, including dressing their malicious wares up to imitate legitimate and widely used open source packages.&nbsp;</p>
<p>As we noted above, the packages we detected mimic a variety of well known PyPI packages with wide distributions, but serving very different ends. The common thread connecting the PyPI packages that were imitated appears to be nothing more than their popularity, measured in monthly downloads.&nbsp;</p>
<p>In each case, the attackers disguised their PyPI packages to look like these widely used and legitimate tools. That included copying the package description and pasting it into their imitation packages, simply replacing the legitimate package name with the name of their impostor package.&nbsp;</p>
<p>From the standpoint of a threat researcher or incident responder, their lack of effort makes the job of detecting the ruse easier. In our research, a Google search for the first sentence of the description immediately revealed the corresponding, legitimate package that was being mimickedAs visible in Figures 9 and 10, the descriptions of the PyPI packages and Github projects are identical, aside from the project name.&nbsp;&nbsp;</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/vmconnect-blog-figure9-VMConnect.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure9-VMConnect.png?width=1455&amp;name=vmconnect-blog-figure9-VMConnect.png" alt="Github description of the malicious VMConnect project" style="width: 1455px;" width="1455" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure9-VMConnect.png?width=728&amp;name=vmconnect-blog-figure9-VMConnect.png 728w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure9-VMConnect.png?width=1455&amp;name=vmconnect-blog-figure9-VMConnect.png 1455w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure9-VMConnect.png?width=2183&amp;name=vmconnect-blog-figure9-VMConnect.png 2183w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure9-VMConnect.png?width=2910&amp;name=vmconnect-blog-figure9-VMConnect.png 2910w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure9-VMConnect.png?width=3638&amp;name=vmconnect-blog-figure9-VMConnect.png 3638w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure9-VMConnect.png?width=4365&amp;name=vmconnect-blog-figure9-VMConnect.png 4365w" sizes="(max-width: 1455px) 100vw, 1455px"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 9: Github description of the malicious VMConnect project.</span></p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/vmconnect-blog-figure10-vconnector.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure10-vconnector.png?width=1455&amp;name=vmconnect-blog-figure10-vconnector.png" alt="Github description of the malicious VMConnect project" style="width: 1455px;" width="1455" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure10-vconnector.png?width=728&amp;name=vmconnect-blog-figure10-vconnector.png 728w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure10-vconnector.png?width=1455&amp;name=vmconnect-blog-figure10-vconnector.png 1455w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure10-vconnector.png?width=2183&amp;name=vmconnect-blog-figure10-vconnector.png 2183w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure10-vconnector.png?width=2910&amp;name=vmconnect-blog-figure10-vconnector.png 2910w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure10-vconnector.png?width=3638&amp;name=vmconnect-blog-figure10-vconnector.png 3638w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/vmconnect-blog-figure10-vconnector.png?width=4365&amp;name=vmconnect-blog-figure10-vconnector.png 4365w" sizes="(max-width: 1455px) 100vw, 1455px"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 10: Github description of the legitimate vConnector project.</span></p>
<p>In this case, vConnector is a fairly popular package, first published nine years ago, with the last modification being committed to the Github repository almost four years ago. Download stats for this package show that it has almost 40 thousand downloads per month. A large number of monthly downloads combined with the lack of recent maintenance make this package a very good target for impersonation.</p>
<p>On July 19, seven different versions of a package titled <em>osinfopkg </em>were published to the PyPI repository and used to develop and&nbsp; test the malicious functionality that eventually got included into the VMConnect package.Since this was a testing package, the malicious actor didn’t give too much care into making it look trustworthy, even though it did create a dedicated Github <a href="https://github.com/MarianaCamelia"><span>repository</span></a> containing two more projects.</p>
<p>Then, on July 31, another package named <span style="font-style: italic;">ethter</span>, was released. In this case, the threat actor impersonated the popular package <a href="https://pypi.org/project/eth-tester/#description"><span>eth-tester</span></a> — a collection of tools for testing ethereum based applications which has more than 60 thousand monthly downloads. As the ReversingLabs research team observed with the VMConnect package, the ethter description was copied verbatim from the eth-tester package, with only the package name replaced.&nbsp;</p>
<p>Here again, a phony Github repository was created and malicious functionality was hidden in the <em>utils/based.py</em> file inside the corresponding PyPI package. The malicious file isn’t present in the Github source code repository, only in the PyPI package. The threat actor also released the latest version of their malicious package with the 1.10.1b1 version number. That is in line with the version numbering convention of the impersonated package, and is higher than the latest released version of the impersonated package. The same approach is applied to the malicious <span style="font-style: italic;">quantiumbase</span><strong> </strong>package impersonating legitimate <a href="https://pypi.org/project/databases/" style="font-style: italic;"><span>databases</span></a> package.&nbsp;</p>
<h2 style="font-weight: bold;">Attackers play Whac-a-Module</h2>
<p>Fortunately for Python developers, none of the packages we detected were available for download for very long. Most, including the VMConnect package, were removed from PyPI within one to three days of being posted.&nbsp;</p>
<p>Because of limitations in the PyPI platform, we do not know the reason that these packages were removed: whether they were detected by internal systems operated by PyPI itself, whether reports from ReversingLabs or other firms led to the takedowns, or whether the malicious actors themselves removed the packages. Efforts to get a definitive answer from PyPI on the circumstances that led to the various take downs were not successful.&nbsp;</p>
<p>However, we have observed that new malicious packages get published to PyPI on a daily basis, as soon as the previous package is reported and removed from the PyPI repository. In other words, despite the detection and removal of the packages described above, this malicious PyPI campaign is alive and ongoing. The quick response from threat actors to replace malicious PyPI packages suggests that this is a well-planned and organized campaign and that the malicious actors probably have a prepared list of packages suitable for imitation.</p>
<h2 style="font-weight: bold;">Questions linger on who, what, and why</h2>
<p>There are lingering questions about key elements of this malicious, supply chain campaign. Among them: who (or what) posted the malicious packages, what the ultimate objective of the attacks is and why the campaign was launched.&nbsp;</p>
<p>On the question of who, we do not have sufficient telemetry to pinpoint a threat actor or actors as being responsible for the latest campaign. By publishing the IOCs including command and control infrastructure&nbsp; we have collected, however, we hope that others may be able to connect those with evidence of known attacks and threat actors to help fill in the full picture behind this campaign.</p>
<p>Previous supply chain attacks utilizing PyPI, npm and other public repositories range from sophisticated (<a href="/blog/iconburst-npm-software-supply-chain-attack-grabs-data-from-apps-websites" rel="noopener">IconBurst</a>) to unsophisticated (<a href="/blog/operation-brainleeches-malicious-npm-packages-fuel-supply-chain-and-phishing-attacks" rel="noopener">Operation Brainleeches</a>). While the VMConnect campaign doesn’t have many of the telltale signs of a sophisticated nation-state campaign, it is not a copy and paste “script kiddie” operation either. More data is required to reach conclusions about who or what is responsible for this PyPI malicious campaign.&nbsp;</p>
<p>Similarly, the research team was unable to obtain the second stage deliverable used in these attacks. As noted, the compromised endpoints ReversingLabs observed merely polled C2 servers waiting for further commands (and presumably downloads). But we were unable to observe any active exchanges, which could indicate that the malicious actors were not actively using the infrastructure, or that the compromised endpoints we controlled were not of interest to them.&nbsp;</p>
<p>Lacking any visibility into the later stages of this campaign, it is impossible to know what its ultimate purpose was: theft of sensitive data or intellectual property? Surveillance? Ransomware? All of the above? More data that reveals the full breadth of this campaign is needed before we can speculate on its intent.&nbsp;</p>

<h2 style="font-weight: bold;">Conclusion</h2>
<p>These latest examples of malicious packages on the PyPI platform are typical in many ways. They imitate popular and legitimate open source packages with tens of thousands monthly downloads and hide malicious functionality inside Base64 encoded strings which downloads additional commands from a C2 server. The ReversingLabs research team has seen <a href="https://www.reversinglabs.com/blog/beware-impostor-http-libraries-lurk-on-pypi"><span>many variations </span></a>on this <a href="https://www.reversinglabs.com/blog/pypi-repo-poisoned-with-colour-blind-rat"><span>attack pattern</span></a> in recent months.&nbsp;</p>
<p>What does this mean for security teams and development organizations? The VMConnect campaign is a reminder that threats lurk on open source repositories, and that standard practices like automated source code reviews are not enough to smoke them out.&nbsp;</p>
<p>Source code for the malicious modules published to GitHub was non malicious, but significantly different from the content of the release packages that eventually get compiled into the release artifacts as third party dependencies. Clues that something was amiss — like Base64 encoded strings — were easy to spot, if you know where to look for them. Still, such indicators often escape notice by conventional application security testing and manual code reviews.&nbsp;</p>
<p><span style="font-style: italic;">The </span><em>ReversingLabs Software Supply Chain Security platform</em><span style="font-style: italic;"> can assist you in security assessments, helping sniff out signs of malicious functionality, assess third party package dependencies, and provide a wide range of behavior indicators. These indicators can be extracted from various file formats and used to detect this type of threat before it results in malicious code being deployed in your environment.</span></p></span>
