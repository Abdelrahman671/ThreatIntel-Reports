			<p>A while ago we got our hands on an interesting malware sample. The sample was interesting to us for multiple reasons. On one hand it was a ransomware variant written fully in PowerShell. On the other hand, it contacted multiple domains which were not registered. The following domains were contacted by the sample:</p><ul><li>agvlmtkxmtq2.top</li><li>ahmwmtkxmtq2.top </li><li>amq1mtkxmtq2.top</li><li>bxfmmtkxmtq2.top</li><li>ehuxmtkxmtq2.top</li></ul><p>Looking at the above domains, we got the feeling that they were possibly automatically generated. A malware sample that tries to reach out a domain that is not registered is possibly interesting while doing malware research, because the malware might send interesting information to that domain. The security research team of KPN recognized the sample as FTCODE and decided to register one the domains. This blog contains all information of our FTCODE analysis :).</p><h3  id="How_is_FTCODE_being_delivered_-anchor"><strong>How is FTCODE being delivered to victims?</strong></h3><p>A message on Twitter indicate that the FTCODE ransomware was spotted on the 23th of September in 2019 for the first time in Italy. The first samples that we could find in public sandboxes are from the 1st of October 2019.</p><p>Analyzing the public sandboxes show that FTCODE mainly has been served by using two well-known techniques. The first one is by sending a document to victims containing a malicious <a href="https://docs.microsoft.com/en-us/windows/security/threat-protection/intelligence/macro-malware" class="" title="https://docs.microsoft.com/en-us/windows/security/threat-protection/intelligence/macro-malware">macro</a>. The macro starts PowerShell code to start FTCODE. The following screenshot is an example of a document that has been used:</p>        <img src="https://www.kpn.com/upload/ae43768d-0e11-45d7-a43d-9db6c01b5c05_afb1.png" class="img-responsive"  alt="" /><p>Another method we have seen is using VBS files to drop the malware. The VBS scripts launches PowerShell (the payload) and opens a file to mislead the user, such as a MP3 file or a JPG file:</p>        <img src="https://www.kpn.com/upload/24391e73-e933-4369-ae69-b67adff8f058_afb2.png" class="img-responsive"  alt="" /><p><strong>Domain Generation Algorithm (DGA)</strong></p><p>The domains we found seemed to contain some interesting pattern. We downloaded the code and searched for the code polling our domain. After diving into the code it became clear that a simple domain generating algorithm (<a href="https://en.wikipedia.org/wiki/Domain_generation_algorithm" class="" title="https://en.wikipedia.org/wiki/Domain_generation_algorithm">DGA</a>) was used. The following code was used:</p>        <img src="https://www.kpn.com/upload/39791877-c9aa-4047-9de9-bc90735eae75_afb3.png" class="img-responsive"  alt="" /><p>In the above code, a DGA is available which generates a domain based on the following elements:</p><p>a)&nbsp;&nbsp;&nbsp;&nbsp; a string "hee", "xu1", "hs0", "jd5" or "mqf" </p><p>b)&nbsp;&nbsp;&nbsp;&nbsp; the current date using the PowerShell function Get-Date -UFormat %y%m%V</p><p>The above values elements are concatenated, converted to bytes and encoded with base64. The .top top level domain is added to make it a valid .top domain. For example, the DGA would generate the following domains for week 46 in 2019:</p><p></p><ul><li>agvlmtkxmtq2.top (hee191146)</li><li>ehuxmtkxmtq2.top (xu1191146)</li><li>ahmwmtkxmtq2.top (hs0191146)</li><li>amq1mtkxmtq2.top (jd5191146)</li><li>bxfmmtkxmtq2.top (mqf191146)</li></ul><p>The DGA allowed us to generate many sinkhole domains in advance. We decided to register a few of them for multiple weeks. We registered the first .top domain generated (hee) by the code, so we can use those domains to catch interesting information.</p><p>There is one surprising thing here. Due to the padding of base64, the domain generation in the first 9 weeks of 2020 will fail. The <a href="https://docs.microsoft.com/en-us/powershell/module/microsoft.powershell.utility/get-date?view=powershell-5.1" class="" title="https://docs.microsoft.com/en-us/powershell/module/microsoft.powershell.utility/get-date?view=powershell-5.1">documentation</a> of PowerShell 5.1 states that Get-Date %V will return 01. However, in reality it returns for some unknown reason 1.&nbsp; Therefore, base64 padding is added to the domain name, resulting in the following domain being generated by the script: agvlmjawmte=.top. It is impossible to register a domain with the = character. In week 10 (March 2020), the DGA will work again. </p><p><strong>Sinkhole</strong></p><p>We have control over the .top domains, but what kind of information is being send to these<ins>&nbsp; </ins>domains? By analyzing the code, it became clear that the code is first trying to reach out the main command and control server (C2). The screenshot above shows that for this sample the C2 is the domain indu.rkeindustries.com. If it succeeds contacting this domain, it will only communicate to this domain. However, if it fails to contact this domain, it will fall back to the DGA domains, until one of them is reachable. </p><p><em>In other words: only the bots that lost contact with the main C2 of the attackers will contact our sinkhole. It seems the mechanism is there for the botnet owners to manage their network in the case their C2 is taken down.</em></p><p>So, what is in those requests from the bots? We have seen different data being sent depending on the situation. But mainly we received POST-request from different IP-addresses containing a base64 encoded string, an example request in JSON format:</p><div class="embedded-html"><pre><code>{"datetime":"Mon Nov 25 15:04:35:000 2019","clientip":"93.x.x.x","serverport":"80","request_method":"POST","request_uri":"\/","headers":{"Host":"agvlmtkxmtq4.top","Expect":"100-continue","Connection":"keep-alive"},"body":"l=dj0xMDI5LjQmZ3VpZD0wOGIzZDc3OC1mNzJjLTRlODYtYWUyMy02MzI3M2M3YWI1YTgmZXJyb3I9c2VuZHBvc3QyOmh0dHA6Ly9hZ3ZsbXRreG10cTQudG9wLzo6RWNjZXppb25lIGR1cmFudGUgbGEgY2hpYW1hdGEgZGkgIlN1YnN0cmluZyIgY29uICIyIiBhcmdvbWVudG8vaTogIkluZGV4IGUgbGVuZ3RoIGRldm9ubyBmYXJlIHJpZmVyaW1lbnRvIGEgdW5hIHBvc2l6aW9uZSBuZWxsYSBzdHJpbmdhLg0KTm9tZSBwYXJhbWV0cm86IGxlbmd0aCI="}</code></pre>
</div><p>After decoding the data, we determined the following information:</p><div class="embedded-html"><pre><code>v=1029.4&amp;guid=08b3d778-f72c-4e86-ae23-63273c7ab5a8&amp;error=sendpost2:http://agvlmtkxmtq4.top/::Eccezione durante la chiamata di "Substring" con "2" argomento/i: "Index e length devono fare riferimento a una posizione nella stringa.
Nome parametro: length"</code></pre>
</div><p>The variable v contains the version of the bot. A lot of samples are available with different version numbers. Also, a GUID is sent to us. This is an interesting value as the GUID is used on a lot of parts within the attack, which we will explain later on. It is the unique identifier of the victim. Next to this, an exception message is embedded within the request. </p><p>In some situations, the base64 contained a register call. The bots try to register itself on our backup C2 instead of the attackers C2. The attackers C2 might be already blocked by a security measurement such as an Intrusion Prevention System (IPS). The following example shows a register call:</p><div class="embedded-html"><pre><code>v=927.1&amp;guid=&amp;status=register&amp;ssid=&amp;os=10.0.15063.1387&amp;psver=5&amp;comp_name=LAPTOP-4PR702DQ&amp;domen=http://agvlmtkxmtq4.top/ </code></pre></div><p>This request contains some other interesting information such as a PowerShell version and computer name. By importing all the information in an ELK instance, we were able to get great insights into the malware activity. Below is data from 25<sup>th</sup> November 2019 until the 30<sup>th</sup> of November 2019:</p>        <img src="https://www.kpn.com/upload/5d9cb434-a79a-418f-924c-b4fd9e0c267f_afb4.png" class="img-responsive"  alt="" /><p>Above screenshot shows a timeline of all traffic between 25 November 2019 and 30 November 2019. There is an interesting spike in above timeline. Possibly we have an explanation for this spike: only the bots are connecting to our sinkhole if they are unable to reach the attackers C2. Possibly, the C2 of the attackers was down for a while (maintenance?), which resulted in the bots connecting to us instead of the real C2.</p>        <img src="https://www.kpn.com/upload/b6c356a6-cb2d-4688-a29e-b424e4152170_afb5.png" class="img-responsive"  alt="" />        <img src="https://www.kpn.com/upload/2030e37e-5ab5-4873-a5e2-d5d2c5256642_AFB5B.png" class="img-responsive"  alt="" /><p>These stats show the total of unique IP-addresses we have seen during the same period, and the different versions of samples contacting our sinkhole. </p>        <img src="https://www.kpn.com/upload/156e4f4e-3dad-41d0-a8e0-b19046e5e917_AFB6.png" class="img-responsive"  alt="" /><p>Above pie shows the countries where the bots are connecting from. By manual analyzing the IP-addresses, we have seen different type of companies that are infected by FTCODE, such as (all in Italy):</p><p>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; Banks</p><p>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; Energy companies</p><p>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; Hospitals</p><p>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; Government</p><p>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; </p>        <img src="https://www.kpn.com/upload/fc62efa9-4620-47cc-b422-fd668933c6a8_AFB7.png" class="img-responsive"  alt="" /><p>The infection map of Italy, based on Geo IP information.</p><p><strong>Attackers C2</strong></p><p>One of the behaviors we noticed during the analysis is that the C2 server of the attackers is changing domains regularly. Also, the web services behind the domains were unreachable really quickly. We have seen cases where new samples came available pointing a new domain, the web server behind the new domain that was used was offline within a few hours. </p><p>After a while, we infected one of our machines. The interesting thing is that this machine was still able to reach the domains that went offline in the past. In other words, the attackers are using IP-filters to allow already infected victims to connect to their C2. Other clients are ignored / blocked by the IP-filter. This is possibly a way to hide the C2, or mislead researchers to let them think the C2s is down?</p><p>During our analysis we were not able to reach the C2 of the attackers multiple times in a short time. Possibly, they implemented a banning mechanism that blocks IPs whenever clients have unexpected behavior. For example, we tried to contact all the C2 domains from our infected machine in a short time, which resulted in the C2 becoming unavailable. Most likely they detected this behavior and blocked our IP. </p><p><strong>Taking over the bots</strong></p><p>By analyzing the code, we noticed that a computer infected with FTCODE is continuously polling the attackers C2, and if that fails it will try to contact our sinkhole C2. The contacted C2 is able to return a client specific secret key and a piece of PowerShell code. If the secret key is correct, the PowerShell will get executed. The following code is performing the polling, and executing the PowerShell if the response is correct:</p>        <img src="https://www.kpn.com/upload/f3ddf256-06ce-413f-89c7-32362be3bf08_AFB8.png" class="img-responsive"  alt="" /><p>We contacted the attackers C2 to see how a payload exactly looks like. The returned value was a base64 value, after decoding the following value appeared:</p>        <img src="https://www.kpn.com/upload/00e9a205-87a5-44f7-a197-20c4108af2ed_AFB9.png" class="img-responsive"  alt="" /><p>Above response of the real C2 confirms our code analysis, a secret is required, and a piece of PowerShell is passed by the C2. </p><p>So, how is the secret that is sent by server validated? We dived in the code to see how the secret mechanism works. It became clear that whenever a payload is received by the server, the first 16 characters are compared with a value within the thumbcache TXT-file. If the values are equal, the PowerShell code is executed. </p><p>And are we able to obtain the secrets of all those other bots? We have a sinkhole, that is being contacted by 1000s of bots. Like earlier mentioned, they are contacting our C2 with information such as the version and GUID. We tried to forward all the information that is being sent to us to the real C2 (C2 in the middle :D), in the hope we would receive the secret and the PowerShell code that is there for the infected machine ready for being executed. The post data forwarded to the real C2:</p><div class="embedded-html"><pre><code>l=dj0xMDI5LjQmZ3VpZD0yN2VjNjJmMy1iMzc4LTRkMTQtYTgyYi0wYTk2NmY2ODQ4NGEmJmRvbWVuPWh0dHA6Ly9hZ3ZsbXRreG10cTIudG9wLw==</code></pre>
</div><p>Base64 Decoded version:</p><div class="embedded-html"><pre><code>v=1029.4&amp;guid=27ec62f3-b378-4d14-a82b-0a966f68484a&amp;&amp;domen=http://agvlmtkxmtq2.top/</code></pre>
 </div><p>This resulted in getting the infected machine specific secret and the PowerShell code being ready at the real C2. We can conclude that we can forward all the polling requests reaching our sinkhole to the real C2 server of the attackers, to disclose the secret keys. <strong>WIN! </strong>After obtaining all the secret keys, we could execute PowerShell code on all the machines trying to contact our sinkhole, by returning the infected machine secret followed by any piece of PowerShell we want to execute. </p><p><strong>Ok, but lets do it then!</strong></p><p>Lets test this theory, because there might be a condition that tackles it. First of all, we need to obtain the secret from the attackers C2 for our bot.                             dataLayerSelectors.forEach((selector) => {
                                const elements = document.querySelectorAll(selector);
                                if (!elements) {
                                    return;
                                }
                                for (const element of elements) {
                                    element.addEventListener('click', () => {
                                        const interactionEvent = {
                                            event: 'userInteraction',
                                            interaction: {
                                                eventInfo: {
                                                    eventName: isNavigatingToSeperateLink(element) ? 'nav' : 'user',
                                                    eventAction: 'click'
                                                },
                                                category: {
                                                    primaryCategory: 'home'
                                                },
                                                attributes: {
                                                    moduleName: 'body',
                                                    position: 'main body',
                                                    element: getElement(element),
                                                    label: getLabel(element)
                                                }
                                            }
                                        };
                                        if (window.adobeDataLayer) {
                                            window.adobeDataLayer.push(interactionEvent);
                                        }
                                    });
                                }
                            });
                            function isNavigatingToSeperateLink(element) {
                                const href = element.getAttribute('href');
                                if(!href) {
                                    return false;
                                }
                                return !(href.startsWith('#'));
                            }
                            function getElement(element) {
                                let el = '';
                                if (element.getAttribute('role') === 'button') {
                                    el = 'button';
                                } else if (element.innerHTML.indexOf('<img') > -1) {
                                    el = 'image';
                                } else if (element.querySelector('label') || element.querySelector('h2') || element.querySelector('b')) {
                                    el = 'accordion';
                                } else {
                                    el = 'textlink';
                                }
                                return el;
                            }
                            function getLinkLabel(element, label) {
                                let linkLabel = '';
                                let labelWithoutSpecialCharAndSpace = label ? label.replace(/[^a-zA-Z0-9]/g, '') : 'no label';
                                let linkHref = element.getAttribute('href') || '';
                                if (isValidHttpUrl(linkHref)) {
                                    linkLabel =
                                        labelWithoutSpecialCharAndSpace +
                                        '|' +
                                        new URL(linkHref).hostname +
                                        new URL(linkHref).pathname
                                            .replace(/(^\w+:|^)\/\//, '')
                                            .replace(/\.htm|\.asp/g, '')
                                            .trim();
                                } else if (!isNavigatingToSeperateLink(element)) {
                                    linkLabel =
                                        labelWithoutSpecialCharAndSpace +
                                        '|' +
                                        new URL(window.location.href).hostname +
                                        new URL(window.location.href).pathname.replace(/\.htm|\.asp/g, '').trim();
                                } else {
                                    linkLabel =
                                        labelWithoutSpecialCharAndSpace +
                                        '|' +
                                        window.location.hostname +
                                        linkHref
                                            .replace(/\.htm|\.asp/g, '')
                                            .split('?')[0]
                                            .trim();
                                }
                                return linkLabel;
                            }
                            function isValidHttpUrl(link) {
                                let url;
                                try {
                                    url = new URL(link);
                                } catch (_) {
                                    return false;
                                }
                                return url.protocol === 'http:' || url.protocol === 'https:';
                            }
                            function getLabel(element) {
                                let label = '';
                                if (element.title && !isValidHttpUrl(element.title)) {
                                    label = element.title;
                                } else if (element.querySelector('label')) {
                                    label = element.querySelector('label').innerText;
                                } else if (element.querySelector('h2')) {
                                    label = element.querySelector('h2').innerText;
                                } else {
                                    label = element.innerText;
                                }
                                return (element.getAttribute('href') ? getLinkLabel(element, label) : (label ? label.replace(/[^a-zA-Z0-9]/g, '') : 'no label')).toLowerCase();
                            }
                        })();
                    }, 1000);
