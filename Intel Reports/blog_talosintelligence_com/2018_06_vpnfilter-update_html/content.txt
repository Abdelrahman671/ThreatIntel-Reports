    <h1 class="text-center">VPNFilter Update - VPNFilter exploits endpoints, targets new devices</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/william-largent/">William Largent</a>
        </div>

        <br/>
        <time class="post-datetime" datetime="June 6, 2018 09:02">
            Wednesday, June 6, 2018 09:02
        </time>

        <div class="m-3">
        </div>
    </div>

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <h2 id="introduction">Introduction</h2><p>Cisco Talos, while working with our various intelligence partners, has discovered additional details regarding "<a href="https://blog.talosintelligence.com/VPNFilter/">VPNFilter</a>." In the days since we first published our findings on the campaign, we have seen that VPNFilter is targeting more makes/models of devices than initially thought, and has additional capabilities, including the ability to deliver exploits to endpoints. Talos recently published a blog about a <a href="https://blog.talosintelligence.com/VPNFilter/">broad campaign</a> that delivered VPNFilter to small home-office network devices, as well as network-attached storage devices. As we stated in that post, our research into this threat was, and is, ongoing. In the wake of that post, we have had a number of partners step forward with additional information that has assisted us in our work. This post is an update of our findings over the past week.</p><p>First, we have determined that additional devices are being targeted by this actor, including some from vendors that are new to the target list. These new vendors are ASUS, D-Link, Huawei, Ubiquiti, UPVEL, and ZTE. New devices were also discovered from Linksys, MikroTik, Netgear, and TP-Link. Our research currently shows that no Cisco network devices are affected. We've provided an updated device list below.</p><p>We have also discovered a new stage 3 module that injects malicious content into web traffic as it passes through a network device. At the time of our initial posting, we did not have all of the information regarding the suspected stage 3 modules. The new module allows the actor to deliver exploits to endpoints via a man-in-the-middle capability (e.g. they can intercept network traffic and inject malicious code into it without the user's knowledge). With this new finding, we can confirm that the threat goes beyond what the actor could do on the network device itself, and extends the threat into the networks that a compromised network device supports. We provide technical details on this module, named "ssler" below.</p><p>Additionally, we've discovered an additional stage 3 module that provides any stage 2 module that lacks the kill command the capability to disable the device. When executed, this module specifically removes traces of the VPNFilter malware from the device and then renders the device unusable. Analysis of this module, called "dstr," is also provided below.</p><p>Finally, we've conducted further research into the stage 3 packet sniffer, including in-depth analysis of how it looks for Modbus traffic.</p><h3 id="technical-details">Technical details</h3><h4 id="new-third-stage-modules">New third-stage modules</h4><p>'ssler' (Endpoint exploitation module — JavaScript injection)</p><p>The ssler module, which we pronounce as "Esler," provides data exfiltration and JavaScript injection capabilities by intercepting all traffic passing through the device destined for port 80. This module is expected to be executed with a parameter list, which determines the module's behavior and which websites should be targeted. The first positional parameter controls the folder on the device where stolen data should be stored. The purpose of the other named parameters are as follows:</p><ul><li><strong>dst:</strong> Used by the iptables rules created to specify a destination IP address or CIDR range that the rule should apply to.</li><li><strong>src:</strong> Used by the iptables rules created to specify a source IP address or CIDR range that the rule should apply to.</li><li><strong>dump:</strong> Any domain passed in a dump parameter will have all of its HTTP headers recorded in the reps_*.bin file.</li><li><strong>site:</strong> When a domain is provided in the "site" parameter, this domain will have its web pages targeted for JavaScript injection.</li><li><strong>hook:</strong> This parameter determines the URL of the JavaScript file for injection.</li></ul><p>The first action taken by the ssler module is to configure the device's iptables to redirect all traffic destined for port 80 to its local service listening on port 8888. It starts by using the insmod command to insert three iptables modules into the kernel (ip_tables.ko, iptable_filter.ko, iptable_nat.ko) and then executes the following shell commands:</p><ul><li>iptables -I INPUT -p tcp --dport 8888 -j ACCEPT</li><li>iptables -t nat -I PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 8888</li><li>Example: ./ssler logs src:192.168.201.0/24 dst:10.0.0.0/16 -A PREROUTING -s 192.168.201.0/24 -d 10.0.0.0/16 -p tcp -m tcp --dport 80 -j REDIRECT --to-ports 8888</li></ul><p>Note: To ensure that these rules do not get removed, ssler deletes them and then adds them back approximately every four minutes.</p><p>Any outgoing web requests on port 80 are now intercepted by ssler and can be inspected and manipulated before being sent to the legitimate HTTP service. All HTTP requests are sslstripped. That is, the following changes are made to requests before being sent to the true HTTP server:</p><ul><li>Any instances of the string https:// are replaced with http://, converting requests for secure HTTP resources to requests for insecure ones so sensitive data such as credentials can be extracted from them.</li><li>If the request contains the header Connection: keep-alive, it is replaced with Connection: close</li><li>If the request contains the header Accept-Encoding with the gzip value, this is converted to Accept-Encoding: plaintext/none so no responses will be compressed with gzip (exceptions are made for certain file types, such as images).</li></ul><p>If the host is in one of the dump: parameters, the details of the request are saved to the disk for exfiltration, including the URL, port and all of the request headers. If the host is not in a dump: parameter, it will only dump requests with an Authorization header or URLs that have credentials in them. URLs are determined to have credentials if they contain either the string assword= or ass= and one of the following strings in them:</p><ul><li>sername=</li><li>ser=</li><li>ame=</li><li>ogin=</li><li>ail=</li><li>hone=</li><li>session%5Busername</li><li>session%5Bpassword</li><li>session[password</li></ul><p>Any POST requests to accounts.google.com containing the string signin will also be dumped.</p><p>After these modifications are made, a connection to the true HTTP server is made by ssler using the modified request data over port 80. Ssler receives the response from the HTTP server and makes the following changes to the response before passing it on to the victim:</p><ul><li>A response with an https:// in its Location header value is converted to http://</li><li>The following headers are ignored, i.e. not sent to the client:</li><li>Alt-Scv</li><li>Vary</li><li>Content-MD5</li><li>content-security-policy</li><li>X-FB-Debug</li><li>public-key-pins-report-only</li><li>Access-Control-Allow-Origin</li><li>The entire response is sslstripped — that is, all instances of https:// with \x20http://.</li><li>If parameter site: is provided a domain (or part of a domain, e.g. "google"), it will attempt to inject JavaScript into all Content-Type: text/html or Content-Type: text/javascript responses. The requirement is that the string &lt;meta name= … &gt; be present and long enough to fit the string from the hook: parameter. The &lt;meta name= … &gt; tag will be replaced with &lt;script type="text/javascript" src="[hook value]"&gt;. The victim IP combined with the site is then added to an internal whitelist in ssler and will not be targeted for injection again until the whitelist is cleared (which occurs every four days). <br> Each domain that is sslstripped in the responses (e.g. domains found in links) is then added to a list of stripped domains. Subsequent requests that are intercepted by the ssler module to domains in this list will occur via HTTPS over port 443, instead of HTTP over port 80. By default, four domains are on this list, so ssler will always connect to these domains via HTTPS over port 443: www.google.com, twitter.com, www.facebook.com, or www.youtube.com.</li></ul><h3 id="-dstr-device-destruction-module-">'dstr' (device destruction module)</h3><p>The dstr modules are used to render an infected device inoperable by deleting files necessary for normal operation. It deletes all files and folders related to its own operation first before deleting the rest of the files on the system, possibly in an attempt to hide its presence during a forensic analysis.</p><p>The x86 version of the dstr module was analyzed in-depth. This module first deleted itself from the disk and then stops the execution of the parent Stage 2 process. It will then search all running process for ones named vpnfilter, security, and tor and terminate them. Next, it explicitly deletes the following files and directories:</p><ul><li>/var/tmp/client_ca.crt</li><li>/var/tmp/client.key</li><li>/var/tmp/client.crt</li><li>/var/run/vpnfilterm/htpx</li><li>/var/run/vpnfilter</li><li>/var/run/vpn.tmp</li><li>/var/run/vpn.pid</li><li>/var/run/torrc</li><li>/var/run/tord/hidden_ssh/private_key</li><li>/var/run/tord/hidden_ssh/hostname</li><li>/var/run/tor</li><li>/var/run/msvf.pid</li><li>/var/run/client_ca.crt</li><li>/var/run/client.key</li><li>/var/run/client.crt</li><li>/var/pckg/mikrotik.o</li><li>/var/pckg/.mikrotik.</li><li>/var/msvf.pid</li><li>/var/client_ca.crt</li><li>/var/client.key</li><li>/var/client.crt</li><li>/tmp/client_ca.crt</li><li>/tmp/client.key</li><li>/tmp/client.crt</li><li>/flash/nova/etc/loader/init.x3</li><li>/flash/nova/etc/init/security</li><li>/flash/nova/etc/devel-login</li><li>/flash/mikrotik.o</li><li>/flash/.mikrotik.</li><li>/var/run/vpnfilterw/</li><li>/var/run/vpnfilterm/</li><li>/var/run/tord/hidden_ssh/</li><li>/var/run/tord/</li><li>/flash/nova/etc/loader/</li><li>/flash/nova/etc/init/</li></ul><p>The dstr module clears flash memory by overwriting the bytes of all available /dev/mtdX devices with a 0xFF byte. Finally, the shell command rm -rf /* is executed to delete the remainder of the file system and the device is rebooted. At this point, the device will not have any of the files it needs to operate and fail to boot.</p><p>Additional research on the third stage packet sniffer</p><p>'ps' (stage 3 packet sniffer)</p><p>One of stage 3 packet sniffer module samples we have is the R600VPN MIPS-like (Lexra architecture) sample. This sample is a packet sniffer that is looking for basic authentication as well as monitoring ICS traffic, and is specific to the TP-LINK R600-VPN. The malware uses a raw socket to look for connections to a pre-specified IP address, only looking at TCP packets that are 150 bytes or larger (note: This is the full packet size, with headers. Depending on the size of the TCP header, the PDU could be approximately 56 to 96 bytes and still meet the criteria to get logged). It has the ability to view, but not modify, the network traffic. Very significant changes would be required to implement functionality that could modify traffic.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/2022/09/060418-VPN-filter-flowchart--edited-.jpg" class="kg-image" alt loading="lazy" width="1200" height="2854" srcset="https://blog.talosintelligence.com/content/images/size/w600/2022/09/060418-VPN-filter-flowchart--edited-.jpg 600w, https://blog.talosintelligence.com/content/images/size/w1000/2022/09/060418-VPN-filter-flowchart--edited-.jpg 1000w, https://blog.talosintelligence.com/content/images/2022/09/060418-VPN-filter-flowchart--edited-.jpg 1200w" sizes="(min-width: 720px) 720px"></figure><p>Packets that are not on port 502, are scanned for BasicAuth, and that information is logged.</p><ul><li><strong>Else: </strong>(non-Modbus traffic): sniffing HTTP basic auth credentials</li><li>Destination IP Address == command line argument IP address</li><li>Source port &gt; 1024</li><li>Source port != 8080</li><li>Source port != 8088</li><li>Packet Data length &gt; 20 bytes</li><li>Packet does not contain</li><li>&lt;/ and &gt;</li><li>&lt;?xml</li><li>Basic Og==</li><li>/tmUnblock.cgi</li><li>Password Required</li><li>&lt;div</li><li>&lt;form</li><li>&lt;input</li><li>this. and .get</li><li>{</li><li>}</li><li>200 OK</li><li>&lt;span</li><li>&lt;SPAN</li><li>&lt;DIV</li><li><strong>Packet contains 'Authorization:</strong> Basic' OR one user/pass combination</li><li>User</li><li>User=</li><li>user=</li><li>Name=</li><li>name=</li><li>Usr=</li><li>usr=</li><li>Login=</li><li>login=</li><li>Pass</li><li>Pass=</li><li>pass=</li><li>Password=</li><li>password=</li><li>Passwd=</li><li>passwd=</li><li><strong>Logging: </strong>Logs on IPs and ports, but not the packet contents on port 502. It does not validate the traffic as Modbus.</li><li><strong>Modbus:</strong> Logs SourceIP, SourcePort, DestinationIP, DestinationPort and labels it *modbus*</li><li><strong>All Other:</strong> write full packet to log file if and only if it passes basic auth check <br></li></ul><h3 id="conclusion">Conclusion </h3><p>These new discoveries have shown us that the threat from VPNFilter continues to grow. In addition to the broader threat surface found with additional targeted devices and vendors, the discovery of the malware's capability to support the exploitation of endpoint devices expands the scope of this threat beyond the devices themselves, and into the networks those devices support. If successful, the actor would be able to deploy any desired additional capability into the environment to support their goals, including rootkits, exfiltration capability and destructive malware.</p><p>Talos would like to thank all of the individual researchers, companies and intelligence partners from around the world who have stepped forward to share information and address this threat. Your actions have helped us gain a greater understanding of this campaign, and in some cases, have directly improved the situation. We recognize this is a team sport, and truly appreciate your assistance.</p><p>We will continue to monitor VPNFilter and work with our partners to understand the threat as it continues to evolve in order to ensure that our customers remain protected and the public is informed.</p><h3 id="updated-list-of-iocs">Updated List of IOCs <br></h3><p>As stated previously, we highly suspect that there are additional IOCs and versions of this malware that we are not currently aware of.
