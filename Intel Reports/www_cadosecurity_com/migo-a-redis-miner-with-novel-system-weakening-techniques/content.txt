 <h2 class="wp-block-heading">Introduction</h2> 
 <p>Cado Security Labs researchers have recently encountered a novel malware campaign targeting Redis for initial access. Whilst Redis is no stranger to exploitation by Linux and cloud-focused attackers, this particular campaign involves the use of a number of novel system weakening techniques against the data store itself.&nbsp;</p> 
 <p>The malware, named Migo by the developers, aims to compromise Redis servers for the purpose of mining cryptocurrency on the underlying Linux host.&nbsp;</p> 
 <p>Summary:</p> 
 <ul> 
  <li>New Redis system weakening commands have been observed in the wild</li> 
  <li>The campaign utilises these commands to exploit Redis to conduct a cryptojacking attack</li> 
  <li>Migo is delivered as a Golang ELF binary, with compile-time obfuscation and the ability to persist on Linux hosts</li> 
  <li>A modified version of a popular user mode rootkit is deployed by the malware to hide processes and on-disk artefacts</li> 
 </ul> 
 <p></p> 
 <h2 class="wp-block-heading">Initial Access</h2> 
 <p>Cado researchers were first alerted to the Migo campaign after noticing an unusual series of commands targeting a Redis honeypot.&nbsp;</p> 
 <p>A malicious node at the IP 103[.]79[.]118[.]221 connected to the honeypot and disabled the following configuration options using the Redis command line interface’s (CLI) config set feature:</p> 
 <ul> 
  <li><code>set protected-mode</code></li> 
  <li><code>replica-read-only</code></li> 
  <li><code>aof-rewrite-incremental-fsync</code></li> 
  <li><code>rdb-save-incremental-fsync</code></li> 
 </ul> 
 <p>Discussing each of these in turn will shed some light on the attacker’s motivation for doing so.</p> 
 <p></p> 
 <h3 class="wp-block-heading"><strong>set protected-mode</strong></h3> 
 <p><a href="https://redis.io/docs/management/security/#protected-mode">Protected mode</a> is an operating mode of the Redis server that’s designed as a mitigation for users who may have inadvertently exposed the server to external networks.&nbsp;</p> 
 <p>Introduced in version 3.2.0, protected mode is engaged when a Redis server has been deployed in the default configuration (i.e. bound to all networking interfaces) without having password authentication enabled. In this mode, the Redis server will only accept connections from the loopback interface, any other connections will receive an error.</p> 
 <p>Given that the attacker does not have access to the loopback interface and is instead attempting to connect externally, this command should automatically fail on Redis servers with protected mode enabled. It’s possible the attacker has misunderstood this feature and is trying to issue a number of system weakening commands in an opportunistic manner.&nbsp;</p> 
 <p>This feature is disabled in Cado’s honeypot environment, which is why these commands and additional actions on objective succeed.</p>

 <h3 class="wp-block-heading"><strong>replica-read-only</strong></h3> 
 <p>As the name suggests, the <a href="https://redis.io/docs/management/replication/#read-only-replica">replica-read-only</a> feature configures Redis <a href="https://redis.io/docs/management/replication">replicas</a> (exact copies of a master Redis instance) to reject all incoming write commands. This configuration parameter is enabled by default, to prevent accidental writes to replicas which could result in the master/replica topology becoming out of sync.</p> 
 <p>Cado researchers have previously <a href="../redis-p2pinfect/index.html">reported</a> on exploitation of the replication feature being used to deliver malicious payloads to Redis instances. The attackers behind Migo are likely disabling this feature to facilitate future exploitation of the Redis server.</p>

 </figure>

 <p></p> 
 <p>After disabling these configuration parameters, the attacker uses the <code>set</code> command to set the values of two separate Redis keys. One key is assigned a string value corresponding to a malicious attacker-controlled SSH key, and the other to a Cron job that retrieves the malicious primary payload from Transfer.sh (a relatively uncommon distribution mechanism previously covered by <a href="../redis-miner-leverages-command-line-file-hosting-service/index.html">Cado</a>) via Pastebin.&nbsp;</p> 
 <p>The attackers will then follow-up with a series of commands to change the working directory of Redis itself, before saving the contents of the database. If the working directory is one of the Cron directories, the file will be parsed by <code>crond</code> and executed as a normal Cron job.&nbsp;<br>This is a common attack pattern against Redis servers and has been previously documented by <a href="../kiss-a-dog-discovered-utilizing-a-20-year-old-process-hider/index.html">Cado</a> and <a href="https://www.trendmicro.com/en_ph/research/20/d/exposed-redis-instances-abused-for-remote-code-execution-cryptocurrency-mining.html">others</a>.</p>

 <p></p> 
 <p>This in-memory script proceeds to create an empty file at <code>/tmp/.xxx1</code> (an indicator to the previous stage that the host has been compromised) before retrieving the primary payload from transfer.sh. This payload is saved as <code>/tmp/.migo</code>, before being executed as a background task via <code>nohup</code>.</p> 
 <p></p> 
 <h2 class="wp-block-heading">Primary Payload – Static Properties</h2> 
 <p>The Migo primary payload (<code>/tmp/.migo</code>) is delivered as a statically-linked and stripped UPX-packed ELF, compiled from Go code for the x86_64 architecture. The sample uses vanilla UPX packing (i.e. the UPX header is intact) and can be trivially unpacked using <code>upx -d</code>.&nbsp;</p> 
 <p>After unpacking, analysis of the <code>.gopclntab </code>section of the binary highlights the attacker’s use of a compile-time obfuscator to obscure various strings relating to internal symbols. You might wonder why this is necessary when the binary is already stripped, the answer lies with a feature of the Go programming language named “Program Counter Line Table <code>(pclntab)</code>”.&nbsp;</p> 
 <p>In short, the <code>pclntab</code> is a structure located in the <code>.gopclntab</code> section of a Go ELF binary. It can be used to map virtual addresses to symbol names, for the purposes of generating stack traces. This allows reverse engineers the ability to recover symbols from the binary, even in cases where the binary is stripped. For more information on this, see Mandiant’s excellent blog on <a href="https://www.mandiant.com/resources/blog/golang-internals-symbol-recovery">Golang symbol recovery</a>.&nbsp;</p> 
 <p>The developers of Migo appear to be aware of this, and have opted to further protect these symbols by applying additional compile-time obfuscation. This is likely to prevent details of the malware’s capabilities from appearing in stack traces or being easily recovered by reverse engineers.</p>

 <p>With the miner installed and an XMRig configuration set, the malware proceeds to query some information about the system, including the number of logged-in users (via the <code>w</code> binary) and resource limits for users on the system. It also sets the number of Huge Pages available on the system to 128, using the <code>vm.nr_hugepages</code> parameter. These actions are fairly typical for cryptojacking malware.&nbsp;</p> 
 <p>Interestingly, Migo appears to recursively iterate through files and directories under <code>/etc</code>. The malware will simply read files in these locations and not do anything with the contents. One theory is this could be a (weak) attempt to confuse sandbox and dynamic analysis solutions by performing a large number of benign actions, resulting in a non-malicious classification. It’s also possible the malware is hunting for an artefact specific to the target environment that’s missing from our own analysis environment. However, there was no evidence of this recovered during our analysis.</p> 
 <p>Once this is complete, the binary is copied to <code>/tmp</code> via the <code>/proc/self/exe</code> symlink ahead of registering persistence, before a series of shell commands are executed. An example of these commands is listed below.</p>

 <p></p> 
 <p>In summary, they perform the following actions:</p> 
 <ul> 
  <li>Make the copied version of the binary executable, to be executed via a persistence mechanism</li> 
  <li>Disable SELinux and search for uninstallation scripts for monitoring agents bundled in compute instances from cloud providers such as Qcloud and Alibaba Cloud</li> 
  <li>Execute the miner and pass the dropped configuration into it</li> 
  <li>Configure <code>iptables</code> to drop outbound traffic to specific IPs</li> 
  <li>Kill competing miners and payloads from similar campaigns</li> 
  <li>Register persistence via the systemd timer <code>system-kernel.timer</code></li> 
 </ul> 
 <p>Note that these actions are consistent with prior mining campaigns targeting East Asian cloud providers analysed by Cado <a href="../watchdog-continues-to-target-east-asian-csps/index.html">researchers</a>.&nbsp;</p> 
 <p>Migo will also attempt to prevent outbound traffic to domains belonging to these cloud providers by writing the following lines to <code>/etc/hosts</code>, effectively creating a blackhole for each of these domains. It’s likely that this is to prevent monitoring agents and update software from contacting these domains and triggering any alerts that might be in place.&nbsp;</p> 
 <p>This also gives us some insight into the infrastructure targeted by the malware, as these domains belong to the same cloud service providers as we discussed previously.</p>

 <h2 class="wp-block-heading">Persistence</h2> 
 <p>As seen in the commands above, Migo achieves persistence on the target host via the use of a systemd service and associated <code>systemd</code> timer. These are named <code>system-kernel.timer</code> and <code>system-kernel.service</code> respectively.&nbsp;</p> 
 <p>The service unit is straightforward, it simply ensures the Migo payload is executable before invoking it. The malware also configures the allowed number of open file descriptors (via the <code>LimitNOFILE</code> parameter) and increases the CPU shares weighting to 1000000, allowing the miner to fully utilise the CPU.</p>

 <h2 class="wp-block-heading">Process Hiding</h2> 
 <p>Interestingly, Migo will attempt to hide on-disk artefacts dropped by itself via the use of a user mode rootkit. These artefacts include the contents <code>/tmp/.migo_worker</code> directory, where the malware stores the miner and configuration file, as well as the main payload located at <code>/tmp/.migo</code>.&nbsp;</p> 
 <p>To achieve this, the malware updates <code>/etc/ld.so.preload</code> to point at a Linux shared object file located at <code>/usr/local/lib/libsystemd.so</code>, effectively conducting <a href="../linux-attack-techniques-dynamic-linker-hijacking-with-ld-preload/index.html">Dynamic Linker hijacking</a> on the Redis host. This shared object is embedded within the Migo primary payload and is extracted at runtime.</p>

With this shared object in place, the malware intercepts invocations of file and process listing tools (<code>ls, ps, top</code> etc) and hides the appropriate lines from the tool’s output.</p>

 <h2 class="wp-block-heading">Conclusion</h2> 
 <p>Migo demonstrates that cloud-focused attackers are continuing to refine their techniques and improve their ability to exploit web-facing services. The campaign utilised a number of Redis system weakening commands, in an attempt to disable security features of the data store that may impede their initial access attempts. These commands have not previously been reported in campaigns leveraging Redis for initial access.&nbsp;</p> 
 <p>The developers of Migo also appear to be aware of the malware analysis process, taking additional steps to obfuscate symbols and strings found in the <code>pclntab</code> structure that could aid reverse engineering. Even the use of Go to produce a compiled binary as the primary payload, rather than using a series of shell scripts as seen in previous campaigns, suggests that those behind Migo are continuing to hone their techniques and complicate the analysis process.&nbsp;</p> 
 <p>In addition, the use of a user mode rootkit could complicate post-incident forensics of hosts compromised by Migo. Although libprocesshider is frequently used by cryptojacking campaigns, this particular variant includes the ability to hide on-disk artefacts in addition to the malicious processes themselves.</p>
