 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c17">Posted by James Forshaw, Google Project Zero</span><span><br><br>This is a short blog post about some recent improvements I&#39;ve been making to the </span><span class="dnwkefRIXX-c12"><a href="http://oleview.net">OleView.NET</a></span><span>&nbsp;tool which has been released as part of version 1.16. The tool is designed to discover the attack surface of Windows COM and find security vulnerabilities such as privilege escalation and remote code execution. The updates were recently presented at the Microsoft Bluehat conference in Redmond under the name &quot;</span><span class="dnwkefRIXX-c12"><a href="https://github.com/tyranid/infosec-presentations/blob/master/Bluehat/2024/DCOM%2520Research%2520for%2520Everyone!.pdf">DCOM Research for Everyone!</a></span><span class="dnwkefRIXX-c2">&quot;. This blog expands on the topics discussed to give a bit more background and detail that couldn&#39;t be fit within the 45-minute timeslot. This post assumes a knowledge of COM as I&#39;m only going to describe a limited number of terms.</span></p><h2 class="dnwkefRIXX-c13" id="h.t7jr0box29ak"><span class="dnwkefRIXX-c8 dnwkefRIXX-c14">Using the </span><a href="http://oleview.net">OleView.NET</a><span class="dnwkefRIXX-c8 dnwkefRIXX-c14">&nbsp;Tooling</span></h2>
 <p class="dnwkefRIXX-c0"><span>Before we start the discussion it&#39;s important to understand how you can get hold of the </span><span class="dnwkefRIXX-c12"><a href="http://oleview.net">OleView.NET</a></span><span>&nbsp;tool and some basic usage. The simplest way to get the tooling is to install it from the PowerShell gallery with the </span><span class="dnwkefRIXX-c4">Install-Module OleViewDotNet</span><span class="dnwkefRIXX-c2">&nbsp;command. This installs both the PowerShell module and the GUI. </span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>Next you need to parse the COM registration artifacts into an internal database. You can do this by running the </span><span class="dnwkefRIXX-c4">Get-ComDatabase</span><span>&nbsp;command. Once it&#39;s finished you&#39;re ready to go. You will notice that it can take a long time to complete, so it&#39;d be annoying to have to do this every time you want to start researching. For that reason you can use the command </span><span class="dnwkefRIXX-c4">Set-ComDatabase -Default</span><span>&nbsp;to write out the database to a default storage location. Now the next time you start PowerShell you can just run an inspection command, such as </span><span class="dnwkefRIXX-c4">Get-ComClass</span><span class="dnwkefRIXX-c2">&nbsp;and the default database will be automatically loaded. </span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>This default database is also shared with the GUI, which you can start by running the </span><span class="dnwkefRIXX-c4">Show-ComDatabase</span><span>&nbsp;command. For general research I find the GUI to be easier to use and you can click around and look at the COM registration information. For analysis, the ability to script through PowerShell is more important.</span></p><h2 class="dnwkefRIXX-c13" id="h.vgh9yp6bcxrn"><span class="dnwkefRIXX-c8 dnwkefRIXX-c14">Researching COM Services</span></h2>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">Performing security research in COM usually involves the following steps::</span></p><ul style="padding: 0;" class="lst-kix_p9mqotz9vuw2-0 start"><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">Enumerate potential COM classes of interest. These might be classes which are accessible outside of a sandbox, running at high privilege or designed to be remotely exposed.</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">Validate whether the COM classes are truly accessible from the attack position. COM has various security controls which determine what users can launch, activate and access an object. Understanding these security controls allows the list of COM classes of interest to be limited to only those that are actually part of the attack surface.</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">Enumerate exposed interfaces, determine what they do and call methods on them to test for security vulnerabilities.</span></li></ul>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">The last step is the focus of the updates to the tooling, making it easier to determine what an exposed interface does and call methods to test the behavior. The goal is to minimize the amount of reverse engineering needed (although generally some is still required) as well as avoid needing to write code outside of the tooling to interact with the COM service under test.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>To achieve this goal, </span><span class="dnwkefRIXX-c12"><a href="http://oleview.net">OleView.NET</a></span><span>&nbsp;will </span><span>pull together</span><span class="dnwkefRIXX-c2">&nbsp;any sources of interface information it has, then provide a mechanism to inspect and invoke methods on the interface through the UI or via PowerShell. The sources of information that it currently pulls together are:</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p><ol class="lst-kix_p89tpxkie3qv-0 start" start="1"><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span>Known interfaces, either defined in the base .NET framework class libraries or inside </span><span class="dnwkefRIXX-c12"><a href="http://oleview.net">OleView.NET</a></span><span class="dnwkefRIXX-c2">.</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">COM interface definitions present in the Global Assembly Cache.</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">Registered type libraries.</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">Windows Runtime interfaces.</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">Extracted proxy class marshaling information.</span></li></ol>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">One useful benefit of gathering this information, is that the tool formats the interface as &quot;source code&quot; so you can manually inspect it.</span></p><h2 class="dnwkefRIXX-c13" id="h.q3xwlois3nnq"><span class="dnwkefRIXX-c8 dnwkefRIXX-c14">Formatting Interfaces Definitions</span></h2>
 <p class="dnwkefRIXX-c0"><span>The </span><span class="dnwkefRIXX-c12"><a href="http://oleview.net">OleView.NET</a></span><span class="dnwkefRIXX-c2">&nbsp;tool uses a database object to represent all the artifacts it has analyzed on your system. The latest released version defines some of these objects to be convertible to &quot;source code&quot;. For example the following can be converted if the tool can determine some meta data that to represent the artifact:</span></p><ul style="padding: 0;" class="lst-kix_cb3mwjmcuxek-0 start"><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">COM interfaces</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">COM proxies</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">COM Windows Runtime classes.</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">Type libraries, interfaces and classes.</span></li></ul>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>How you get to this conversion depends on whether you&#39;re using the PowerShell or the UI. The simplest approach is PowerShell, using the </span><span class="dnwkefRIXX-c4">ConvertTo-ComSourceCode</span><span class="dnwkefRIXX-c2">&nbsp;command. For example, the following will convert an interface object into source code:</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span></span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">PS&gt; Get-ComInterface -Name IMyInterface | ConvertTo-ComSourceCode -Parse</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>Note that we also need to pass a </span><span class="dnwkefRIXX-c4">-Parse</span><span class="dnwkefRIXX-c2">&nbsp;option to the command.  <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c6">&nbsp; &nbsp; </span><span class="dnwkefRIXX-c5">[id(</span><span class="dnwkefRIXX-c6 dnwkefRIXX-c15">1</span><span class="dnwkefRIXX-c5">),</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">propget]</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c6">&nbsp; &nbsp; </span><span class="dnwkefRIXX-c5">HRESULT</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">Count([out,</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">retval]</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c6 dnwkefRIXX-c18">int</span><span class="dnwkefRIXX-c5">*</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">c);</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c6">&nbsp; &nbsp; </span><span class="dnwkefRIXX-c5">[id(</span><span class="dnwkefRIXX-c6 dnwkefRIXX-c15">0xFFFFFFFC</span><span class="dnwkefRIXX-c5">),</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">restricted]</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c6">&nbsp; &nbsp; </span><span class="dnwkefRIXX-c5">HRESULT</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">_NewEnum([out,</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">retval]</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">IUnknown**</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">ppvObject);</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c6">&nbsp; &nbsp; </span><span class="dnwkefRIXX-c5">[id(</span><span class="dnwkefRIXX-c6 dnwkefRIXX-c15">2</span><span class="dnwkefRIXX-c5">)]</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c6">&nbsp; &nbsp; </span><span class="dnwkefRIXX-c5">HRESULT</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c5">Refresh();</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c5">};</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">This is in the Microsoft Interface Definition Language (MIDL) format, the type library version should be pretty accurate and could even be recompiled by the MIDL compiler. For proxies some of the information is lost and so the MIDL generated isn&#39;t completely accurate, but as we&#39;ll see later there&#39;s limited reasons to take the output and recompile. </span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">Another thing to note is that proxies lose name information when compiled from MIDL to their C marshaled representation. Therefore the tool just generates placeholder names, for example, method names are of the form &quot;ProcN&quot;. If the proxy is for a type that has a known definition, such as from a Windows Runtime type or a type library then the tool will try and automatically apply the names. If not, you&#39;ll need to manually change them if you want them to be anything other than the default.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">You can change the names from PowerShell by modifying the proxy object directly. For example the &quot;IBitsTest1&quot; interface looks like the following before doing anything:</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span></span><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">[</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">&nbsp; object,</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">&nbsp; uuid(51A183DB-67E0-4472-8602-3DBC730B7EF5),</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">]</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">interface IBitsTest1 : IUnknown {</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">&nbsp; &nbsp; HRESULT Proc3([out, string] wchar_t** p0);</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">}</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">You can modify &quot;Proc3&quot; with the following script:</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span></span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">PS&gt; $proxy</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">=</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">Get-ComProxy</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">-Iid</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">51A183DB-67E0-4472-8602-3DBC730B7EF5</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">PS&gt; $proxy.Procedures[0].Name = &quot;GetBitsDllPath&quot;</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">PS&gt; $proxy.Procedures[0].Parameters[0].Name = &quot;DllPath&quot;</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">Now the formatted output looks like the following:</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span></span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">[</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c6">&nbsp; </span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">object,</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c6">&nbsp; </span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">uuid(51A183DB-67E0-4472-8602-3DBC730B7EF5),</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">]</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">interface</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">IBitsTest1</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">:</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">IUnknown</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">{</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c6">&nbsp; &nbsp; </span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">HRESULT</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">GetBitsDllPath([out,</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">string]</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">wchar_t**</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">DllPath);</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">}</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">This renaming will also be important when we come back to calling proxied methods. Obviously it&#39;d be annoying to run this script every time, so you can cache the names using the following command:</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4">PS&gt; Export-ComProxyName -Proxy $p -ToCache</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>This will write out a file describing the names to a local cache file. When the proxy is loaded again in another session this cache file will be automatically applied. The </span><span class="dnwkefRIXX-c4">Export-ComProxyName</span><span>&nbsp;and corresponding </span><span class="dnwkefRIXX-c4">Import-ComProxyName</span><span>&nbsp;commands allow you to read and write XML or JSON files representing the proxy names which you can modify in a text editor if that&#39;s easier.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>One of the quickest wins is to enumerate the interfaces for a COM object, then pass the output of that through the </span><span class="dnwkefRIXX-c4">ConvertTo-ComSource</span><span class="dnwkefRIXX-c2">&nbsp;code command. For example:</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span></span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">PS&gt; $obj</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">=</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">New-ComObject</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c7">-Clsid</span><span class="dnwkefRIXX-c6">&nbsp;</span><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">4575438f-a6c8-4976-b0fe-2f26b80d959e</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">PS&gt; Get-ComInterface -Object $obj | ConvertTo-ComSourceCode -Parse</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">This creates a new COM object based on its CLSID, enumerates the interfaces it supports and then passes them through the conversion process to get out a &quot;source code&quot; representation of the interfaces.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>

In the resulting window, there will be a tree view of artifacts. You need to open the source code viewer window by right clicking the tree and selecting the </span><span class="dnwkefRIXX-c4">Show Source Code</span><span class="dnwkefRIXX-c2">&nbsp;option in the context menu. This will result in a view similar to the following:</span></p>

 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">If you want to edit a proxy from PowerShell in a similar GUI you can use following command:</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4">PS&gt; Edit-ComSourceCode $proxy</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">This will show a dialog similar to the following where you can do edits to the proxy name information:</span></p>

 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p><h2 class="dnwkefRIXX-c13" id="h.9sqgjjxjfh4"><span class="dnwkefRIXX-c8 dnwkefRIXX-c14">Genering Interfaces from a Proxy Definition</span></h2>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">Now on to the more important side of these updates, the ability to invoke methods on the interfaces exposed by an object you want to research. The tool has always given you some ability to invoke methods as long as the object has a .NET interface to call through reflection. This could either be through a known interface type, such as a built-in one or the Windows Runtime interfaces or by converting a type library into a .NET assembly on demand.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">What&#39;s new is the ability to generate an interface based on a proxy definition and then use that to invoke methods. Initially I tried to implement this by generating an .NET interface dynamically which would then use the existing .NET interop to call the proxy methods. This worked fine for simple proxies but quickly hit problems when doing anything more complex:</span></p><ul style="padding: 0;" class="lst-kix_c3xw8wo2ynri-0 start"><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span>Some types are hard to represent in easy to use .NET types, such as pointers to structures. This is &quot;handled&quot; in the type library converter by just exporting them as </span><span class="dnwkefRIXX-c4">IntPtr</span><span class="dnwkefRIXX-c2">&nbsp;parameters which means the caller has to manually marshal the data. Get this wrong and the tool crashes.</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">Any structures need to be accurately laid out so the native marshaler can read and write to the correct field locations. Get this wrong and the tool crashes.</span></li><li style="margin-left: 46pt;" class="c0 c9 li-bullet-0"><span class="dnwkefRIXX-c2">Did I mention that if you get this wrong the tool crashes?</span></li></ul>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>Fortunately I already had a solution, my </span><span class="dnwkefRIXX-c12"><a href="https://github.com/googleprojectzero/sandbox-attacksurface-analysis-tools.v1">sandbox library</a></span><span>&nbsp;already had the ability to dynamically generate a </span><span class="dnwkefRIXX-c12"><a href="https://googleprojectzero.blogspot.com/2019/12/calling-local-windows-rpc-servers-from.html">.NET class from parsed NDR data</a></span><span class="dnwkefRIXX-c2">, in fact I was already using the library to parse the NDR data for proxies so I realized could I repurpose the existing client builder for COM proxy clients. I needed to do some simple refactoring of the code to make it build from a COM proxy instance rather than an RPC server, but I quickly had an RPC client. This RPC client doesn&#39;t directly interact with any native marshaling code, so it&#39;s unlikely to crash. Also any complex structures are built in a way which makes it easy to modify from .NET removing the problems around pointers. One issue with using the RPC client method is the same interface could be used for both in-process and out-of-process objects. Due to the way COM is designed a client usually doesn&#39;t need to care about where the object is, but in this case it must be accessible via a proxy. This isn&#39;t that big an issue, there&#39;s no security boundary between in-process COM objects and so being able to call methods on them isn&#39;t that interesting.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>The next problem was the RPC transport. COM calls have an additional input and output parameter, the </span><span class="dnwkefRIXX-c12"><a href="https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-dcom/d57e9fd9-8e8b-45b8-99ea-9b0266009676">ORPTHIS</a></span><span>&nbsp;and </span><span class="dnwkefRIXX-c12"><a href="https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-dcom/10ffacb5-c8b2-4b6a-bccc-9e515818661f">ORPCTHAT</a></span><span>&nbsp;structures, that need to be added to the call. These parameters could have been added to the RPC client, but it would seem best to make the clients agnostic of the transport. Instead as my RPC code has pluggable RPC transport I was able to reimplement a custom version over the top of the existing ALPC and TCP transports which added the additional parameters to any call. That wasn&#39;t the end of it though, ALPC needs an additional pair of parameters, </span><span class="dnwkefRIXX-c3">LocalThis</span><span>&nbsp;and </span><span class="dnwkefRIXX-c3">LocalThat</span><span>, which are potentially different depending on versions of Windows. Also you need to add support for additional services such as the OXID resolver and communication with the local DCOM activator. While I implemented all this it wasn&#39;t as reliable as I&#39;d like, however it&#39;s still present in the </span><span class="dnwkefRIXX-c12"><a href="https://github.com/tyranid/oleviewdotnet/tree/master/OleViewDotNet/Rpc/Clients">source code</a></span><span class="dnwkefRIXX-c2">&nbsp;if you want to play with it. </span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c3">As an aside, I should point out that </span><span class="dnwkefRIXX-c12 dnwkefRIXX-c3"><a href="https://x.com/hakril">Clement Rouault</a></span><span class="dnwkefRIXX-c3">,</span><span class="dnwkefRIXX-c3">&nbsp;one of the original researchers into ALPC RPC protocol of which parts of my own implementation is inspired, recently released a very </span><span class="dnwkefRIXX-c12 dnwkefRIXX-c3"><a href="https://blog.exatrack.com/STUBborn/">similar project</a></span><span class="dnwkefRIXX-c3 dnwkefRIXX-c20">&nbsp;for their Python tooling which implements the ALPC DCOM protocol.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>I decided that I&#39;d need a different approach, in the COM runtime the RPC channel used by a proxy instance is represented by the </span><span class="dnwkefRIXX-c12"><a href="https://learn.microsoft.com/en-us/windows/win32/api/objidl/nn-objidl-irpcchannelbuffer">IRpcChannelBuffer</a></span><span>&nbsp;interface. An object implementing this interface is connected to the proxy during initialization, it is then used to send and receive NDR formatted data from the client to the server. The implementation handles all the idiosyncrasies such as the additional parameters, handling OXID resolving and reference counting. If we could get hold of a proxy object&#39;s instance of the </span><span class="dnwkefRIXX-c3">IRpcChannelBuffer</span><span class="dnwkefRIXX-c2">&nbsp;object, we could use that instead of implementing our own protocol, the challenge was how to get it.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>After a bit of research I found that we can use the documented </span><span class="dnwkefRIXX-c12"><a href="https://learn.microsoft.com/en-us/windows/win32/api/rpcproxy/nf-rpcproxy-ndrproxyinitialize">NdrProxyInitialize</a></span><span>&nbsp;function to get hold of the interface from its </span><span class="dnwkefRIXX-c3">MIDL_STUB_MESSAGE</span><span class="dnwkefRIXX-c2">&nbsp;structure by passing in the interface pointer to a proxy. While it wouldn&#39;t be as flexible as a fully custom implementation this gave me an easy way to handle the transport without worrying about platform or protocol differences. It could also work from an existing COM object, just query the appropriate interface, extract the buffer and make calls to the remote server.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>Of course nothing is that simple, I discovered that while the </span><span class="dnwkefRIXX-c3">IRpcChannelBuffer</span><span>&nbsp;object is a COM object it has a broken implementation of </span><span class="dnwkefRIXX-c3">IMarshal</span><span>. As .NET&#39;s COM interop tries to query for </span><span class="dnwkefRIXX-c3">IMarshal</span><span>&nbsp;when generating a Runtime Callable Wrapper, it will immediately crash the process. I had to </span><span class="dnwkefRIXX-c12"><a href="https://github.com/tyranid/oleviewdotnet/blob/master/OleViewDotNet/Rpc/Transport/RpcChannelBuffer.cs">manually dispatch the calls</a></span><span class="dnwkefRIXX-c2">&nbsp;to the method through native delegates, but at least it works. </span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p><h2 class="dnwkefRIXX-c13" id="h.14avb18m4386"><span class="dnwkefRIXX-c8 dnwkefRIXX-c14">Calling Interface Methods</span></h2>
 <p class="dnwkefRIXX-c0"><span>Okay, so how do you use the tool to call arbitrary methods? For the GUI it works like it always has, when you create an instance of a COM object, usually by right clicking an entry in a view and selecting </span><span class="dnwkefRIXX-c4">Create Instance</span><span class="dnwkefRIXX-c2">&nbsp;you&#39;ll get a new object information window similar to the following:</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgrzj92oD3hiwhUTXO9b1RuNGzu_8kpvukMtptP4PxvvvS4AMP9WfmgT2YeURnUYePq1Vs3fj-zISIbsyhxppVysfPnJZWYSz5WZUfirDC-AKe7Tf_MF5FUYOg8GMHaKu6mwcqVPeqmV25uH0CpPikekphTEAQB3jW-9rRR8VKIawZsUIGheFDA3pZMMYo/s828/image5.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="Screenshot showing what happens when you right-click on an entry in a view and select create instance" border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEgrzj92oD3hiwhUTXO9b1RuNGzu_8kpvukMtptP4PxvvvS4AMP9WfmgT2YeURnUYePq1Vs3fj-zISIbsyhxppVysfPnJZWYSz5WZUfirDC-AKe7Tf_MF5FUYOg8GMHaKu6mwcqVPeqmV25uH0CpPikekphTEAQB3jW-9rRR8VKIawZsUIGheFDA3pZMMYo/s828/image5.png" style="max-height: 750px; max-width: 600px;" title="Screenshot showing what happens when you right-click on an entry in a view and select create instance" /></a></span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>At the bottom of the window is a list of supported interfaces. In the right column is an indicator if there&#39;s a viewer for that interface. If it&#39;s set to </span><span class="dnwkefRIXX-c3">Yes</span><span class="dnwkefRIXX-c2">, then you can double click it to bring up an invocation window like the following:</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEh3XnKVEKF2bHF_ES_5M28IcJnlezKlY6FCa3mLXeF15YcW0A1rbad_H3myrXPrgoBVGjJI0hSXRxElzIBBSQPhwB4zGEcIxUJXdpQsr9F_sG3lkih1gBVTwxVsF7RKOqeb6U9c2ehHDv4Q9NZm-SiFcZSK9wb2cEESfjNS4JbR1TZkqlng3E9VkGTMKmM/s956/image2.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="Screenshot of the OleView .NET tooling showing the invoked method " border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEh3XnKVEKF2bHF_ES_5M28IcJnlezKlY6FCa3mLXeF15YcW0A1rbad_H3myrXPrgoBVGjJI0hSXRxElzIBBSQPhwB4zGEcIxUJXdpQsr9F_sG3lkih1gBVTwxVsF7RKOqeb6U9c2ehHDv4Q9NZm-SiFcZSK9wb2cEESfjNS4JbR1TZkqlng3E9VkGTMKmM/s956/image2.png" style="max-height: 750px; max-width: 600px;" title="Screenshot of the OleView .NET tooling showing the invoked method " /></a></span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">From this window you can double click a method to bring up a new dialog where you can specify the arguments and invoke the method as shown below.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"><a href="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEjbLT1500wjMy5d1pSm2dd6fYVz2h3kUfET2i9MxeIcriv-UxVZCEa5nNKGZVtfT7J5eAtk_eK152wQPLMyrPA0QmnrHKvl7Zj9509IgsF6sEQGXzGqECv6BPLSFGXuLspf1V-3gvAKYNYHMZqgRORja-Yt4ysVupN095depSVjqdOiPRzdgIHx9_VCRt0/s1178/image1.png" style="display: block; padding: 1em 0;text-align: center;"><img alt="Screenshot of Invoke GetBitsDllPath showing that the operation completed successfully" border="0" src="https://blogger.googleusercontent.com/img/b/R29vZ2xl/AVvXsEjbLT1500wjMy5d1pSm2dd6fYVz2h3kUfET2i9MxeIcriv-UxVZCEa5nNKGZVtfT7J5eAtk_eK152wQPLMyrPA0QmnrHKvl7Zj9509IgsF6sEQGXzGqECv6BPLSFGXuLspf1V-3gvAKYNYHMZqgRORja-Yt4ysVupN095depSVjqdOiPRzdgIHx9_VCRt0/s1178/image1.png" style="max-height: 750px; max-width: 600px;" title="Screenshot of Invoke GetBitsDllPath showing that the operation completed successfully" /></a></span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2">Once invoked it&#39;ll show the resulting output parameters and if the return value is an integer will assume it&#39;s a HRESULT error code. These windows are the same for &quot;reflected&quot; interfaces such as type libraries and Windows Runtime interfaces as well as proxy clients. The names of proxy methods won&#39;t be automatically updated if you change them when the interface window is open. You&#39;ll need to go back to the object information window and double click the interface again to get it to recreate the client.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>For PowerShell you can specify an </span><span class="dnwkefRIXX-c4">Iid</span><span>&nbsp;argument when using the </span><span class="dnwkefRIXX-c4">New-ComObject</span><span>&nbsp;command or use the </span><span class="dnwkefRIXX-c4">Get-ComObjectInterface</span><span class="dnwkefRIXX-c2">&nbsp;command to query an existing COM object for a new interface. The tooling will pick the best option for calling the interface from the options available to it, including generating the RPC client dynamically. </span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span></span><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">PS&gt; $obj = New-ComObject -Clsid 4991D34B-80A1-4291-83B6-3328366B9097</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">PS&gt; $test = Get-ComObjectInterface $o -Iid 51A183DB-67E0-4472-8602-3DBC730B7EF5</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">PS&gt; $test.GetBitsDllPath()</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7"></span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">DllPath &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp;retval</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c11 dnwkefRIXX-c7">------- &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp;------</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c4 dnwkefRIXX-c7 dnwkefRIXX-c11">c:\windows\system32\qmgr.dll &nbsp; &nbsp; &nbsp;0</span></p>
 <p class="dnwkefRIXX-c0"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>To make it easier to call interface methods from PowerShell the exposed methods on the object will be modified to wrap output parameters in a single return value. You can see this in the listing above, the </span><span class="dnwkefRIXX-c3">DllPath</span><span>&nbsp;parameter was originally an output only parameter. Rather than deal with &nbsp;that in the script a return structure was automatically created containing the </span><span class="dnwkefRIXX-c3">DllPath</span><span class="dnwkefRIXX-c2">&nbsp;as well as the HRESULT return value. If the parameter is an input and output then the method signature accepts the input value and the return value contains the output value.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>If the definitions for your interface don&#39;t already exist you can import them into the tool to be used by the automatic interface selection. To do this you&#39;ll need to define the interfaces as .NET types and compile them into an assembly. Then in the GUI use the </span><span class="dnwkefRIXX-c4">File&rarr;Import Interop Assembly</span><span>&nbsp;menu option or for PowerShell use the </span><span class="dnwkefRIXX-c4">Add-ComObjectInterface</span><span class="dnwkefRIXX-c2">&nbsp;command. Both of these options allow you to specify the assembly will be automatically loaded the next time you start the tool. This will make a copy of the DLL to a central location so that it can be accessed even if you delete the library later.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>If all you have is an IDL file for a set of COM interfaces you can import them into the tool indirectly with help from the Windows SDK. First compile the IDL file using the MIDL compiler to generate a type library, then use the </span><span class="dnwkefRIXX-c12"><a href="https://learn.microsoft.com/en-us/dotnet/framework/tools/tlbimp-exe-type-library-importer">TLBIMP</a></span><span class="dnwkefRIXX-c2">&nbsp;command to generate an Assembly file from the type library. Finally you can import it using the previous paragraph&#39;s methods.</span></p>
 <p class="dnwkefRIXX-c0 dnwkefRIXX-c1"><span class="dnwkefRIXX-c2"></span></p>
 <p class="dnwkefRIXX-c0"><span>There&#39;s plenty to discover in </span><span class="dnwkefRIXX-c12"><a href="http://oleview.net">OleView.NET</a></span><span>&nbsp;which I&#39;ve not covered here. I&#39;d encourage you to play around, or check out the </span><span class="dnwkefRIXX-c12"><a href="https://github.com/tyranid/oleviewdotnet">source code</a></span><span>&nbsp;on github.</span></p>
