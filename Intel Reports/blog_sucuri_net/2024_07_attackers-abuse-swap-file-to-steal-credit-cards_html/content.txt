</figure></div><div class="entry-container"><div class="entry-content-wrap"><div class="entry-content"><p>When it comes to website security, sometimes the most innocuous features can become powerful tools in the hands of attackers. Such was the case in a recent incident we investigated, where bad actors exploited the humble swap file to maintain a persistent credit card skimmer on a Magento e-commerce site. This clever tactic allowed the malware to survive multiple cleanup attempts — that is, until our analysts wrapped up their investigation.</p><p>In this post, we&#8217;ll peel back the layers of this sophisticated ecommerce attack, offering valuable insights into how you can protect your own online store from similar threats.</p><p>Let’s take a look!</p><h2>Inspecting the malware</h2><p>When navigating to the compromised website’s checkout page, we could see an interesting script buried way down in the page source.</p><p><picture decoding="async" class="aligncenter wp-image-34208 size-full">

</p><p>The script had all the usual indicators of malware, like base64 encoded variables and hex encoded strings. But once decoded, we could see some clear indicators that the script was watching for credit card details.</p><h3>Malicious checkout page behavior</h3><p>The following snippet enables a checkout button and adds custom bindings to the normal click function on the compromised checkout page.</p><p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-34209" src="https://blog.sucuri.net/wp-content/uploads/2024/07/custom-bindings.png" alt="snippet enables a checkout button and adds custom bindings to the normal click function on the compromised checkout page" width="770" height="237" srcset="https://blog.sucuri.net/wp-content/uploads/2024/07/custom-bindings.png 770w, https://blog.sucuri.net/wp-content/uploads/2024/07/custom-bindings-600x185.png 600w, https://blog.sucuri.net/wp-content/uploads/2024/07/custom-bindings-650x200.png 650w, https://blog.sucuri.net/wp-content/uploads/2024/07/custom-bindings-768x236.png 768w, https://blog.sucuri.net/wp-content/uploads/2024/07/custom-bindings-390x120.png 390w, https://blog.sucuri.net/wp-content/uploads/2024/07/custom-bindings-560x172.png 560w" sizes="(max-width: 770px) 100vw, 770px" /></p><p>Once the checkout button is clicked, the script captures data entered into the credit card form via a <b>querySelectorAll </b>function as seen here.</p><p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-34210" src="https://blog.sucuri.net/wp-content/uploads/2024/07/querySelectorAll.png" alt="script captures data entered into the credit card form via a querySelectorAll function" width="543" height="198" srcset="https://blog.sucuri.net/wp-content/uploads/2024/07/querySelectorAll.png 543w, https://blog.sucuri.net/wp-content/uploads/2024/07/querySelectorAll-390x142.png 390w" sizes="(max-width: 543px) 100vw, 543px" /></p><p>Several other variations of the function elsewhere in the script capture sensitive information such as name, address, card number and other data needed by the attackers to utilize the stolen card details. We can also see a domain, <b>amazon-analytic[.]com</b>, used by the attackers to retrieve the stolen credit card details.</p><p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-34211" src="https://blog.sucuri.net/wp-content/uploads/2024/07/amazon-analytic-domain.png" alt="amazon-analytic[.]com used by the attackers to retrieve the stolen credit card details" width="542" height="58" srcset="https://blog.sucuri.net/wp-content/uploads/2024/07/amazon-analytic-domain.png 542w, https://blog.sucuri.net/wp-content/uploads/2024/07/amazon-analytic-domain-390x42.png 390w" sizes="(max-width: 542px) 100vw, 542px" /></p><p>This amazon-analytic[.]com domain was registered in February 2024 and has also been seen used in other previous cases of credit card theft. Note the use of the brand name; this tactic of leveraging popular products and services in domain names is often used by bad actors in an attempt to evade detection.</p><h2>Tracing the source to bootstrap.php</h2><p>Once we had the checkout page decoded, it was time to find the source. Further investigation led us to the Magento <b>app/bootstrap.php</b> file which had been completely replaced from the official version.</p><p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-34212" src="https://blog.sucuri.net/wp-content/uploads/2024/07/appbootstrap-php.png" alt="Replaced Magento app/bootstrap.php file" width="937" height="629" srcset="https://blog.sucuri.net/wp-content/uploads/2024/07/appbootstrap-php.png 937w, https://blog.sucuri.net/wp-content/uploads/2024/07/appbootstrap-php-600x403.png 600w, https://blog.sucuri.net/wp-content/uploads/2024/07/appbootstrap-php-650x436.png 650w, https://blog.sucuri.net/wp-content/uploads/2024/07/appbootstrap-php-768x516.png 768w, https://blog.sucuri.net/wp-content/uploads/2024/07/appbootstrap-php-390x262.png 390w, https://blog.sucuri.net/wp-content/uploads/2024/07/appbootstrap-php-820x550.png 820w, https://blog.sucuri.net/wp-content/uploads/2024/07/appbootstrap-php-560x376.png 560w" sizes="(max-width: 937px) 100vw, 937px" /></p><p>Decoding that base64 content revealed the exact malicious script found in the checkout page source. We also located the <b>curl</b> function used by the attackers to exfiltrate the stolen data to their external domain.</p><p>The <b>ob_filter_callback</b> function is used to add the skimmer script to web pages if their URLs contain the keyword “checkout” and the pages were requested using the GET method.</p><p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-34214" src="https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback.png" alt="ob_filter_callback function is used to add the skimmer script to web pages" width="1684" height="162" srcset="https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback.png 1684w, https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback-600x58.png 600w, https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback-650x63.png 650w, https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback-768x74.png 768w, https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback-1536x148.png 1536w, https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback-390x38.png 390w, https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback-820x79.png 820w, https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback-1180x114.png 1180w, https://blog.sucuri.net/wp-content/uploads/2024/07/ob_filter_callback-560x54.png 560w" sizes="(max-width: 1684px) 100vw, 1684px" /></p><h2>The malware removal process</h2><p>We figured that removing the malware should be as simple as replacing <b>app/bootstrap.php</b> with the correct version and then clearing caches, since we know the content there is what we are seeing in the checkout, right? Well, not in this case.</p><p>Once we replaced the file with the correct contents, we noticed the script was still loading in checkout. It’s not impossible for files to be reinfected pretty quickly, so we took another look at <b>bootstrap.php </b>with our malware cleanup tools and noticed that the malware was back.</p><p>What was especially strange was that when we looked at the file on the server directly via SSH, it was completely clean.</p><p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-34213" src="https://blog.sucuri.net/wp-content/uploads/2024/07/clean-file.png" alt="Apparently clean file" width="550" height="390" srcset="https://blog.sucuri.net/wp-content/uploads/2024/07/clean-file.png 550w, https://blog.sucuri.net/wp-content/uploads/2024/07/clean-file-390x277.png 390w" sizes="(max-width: 550px) 100vw, 550px" /></p><p>However, when we searched the file system for indicators of the malware using our cleanup tools, the malware still appeared.</p><p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-34215" src="https://blog.sucuri.net/wp-content/uploads/2024/07/malware-infection.png" alt="Malware infection" width="620" height="379" srcset="https://blog.sucuri.net/wp-content/uploads/2024/07/malware-infection.png 620w, https://blog.sucuri.net/wp-content/uploads/2024/07/malware-infection-600x367.png 600w, https://blog.sucuri.net/wp-content/uploads/2024/07/malware-infection-390x238.png 390w, https://blog.sucuri.net/wp-content/uploads/2024/07/malware-infection-560x342.png 560w" sizes="(max-width: 620px) 100vw, 620px" /></p><p>As a further test, we copied the apparently infected<b> bootstrap.php </b>file to a new file name — bootstrap.php.clean — and viewed that with our malware cleanup tools. The file contents were correct.</p><p>So, <i>somehow</i>, it seemed that the file was both clean and infected at the same time (Schrodinger&#8217;s malware?).</p><p>It doesn’t happen often, but occasionally malware will linger in memory and can hook into file session data. We restarted the Apache and PHP services, and eventually rebooted the entire server to attempt to clear that up. Surprisingly the malware was still there and we were clearly missing something.</p><h2>Back to the basics: swap files</h2><p>We went back to review the malware script and file where the script originated and noticed something interesting at the very bottom.</p><p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-34216" src="https://blog.sucuri.net/wp-content/uploads/2024/07/swapme-reference.png" alt="swapme file reference" width="317" height="70" /></p><p>At first glance, there did not appear to be any file with that name on the server.</p><p><img loading="lazy" decoding="async" class="aligncenter wp-image-34217 size-large" src="https://blog.sucuri.net/wp-content/uploads/2024/07/swapme-650x234.png" alt="Checking for swapme" width="650" height="234" srcset="https://blog.sucuri.net/wp-content/uploads/2024/07/swapme-650x234.png 650w, https://blog.sucuri.net/wp-content/uploads/2024/07/swapme-600x216.png 600w, https://blog.sucuri.net/wp-content/uploads/2024/07/swapme-768x277.png 768w, https://blog.sucuri.net/wp-content/uploads/2024/07/swapme-390x141.png 390w, https://blog.sucuri.net/wp-content/uploads/2024/07/swapme-820x296.png 820w, https://blog.sucuri.net/wp-content/uploads/2024/07/swapme-560x202.png 560w, https://blog.sucuri.net/wp-content/uploads/2024/07/swapme.png 949w" sizes="(max-width: 650px) 100vw, 650px" /></p><p>The <b>swapme</b> part of the file name clued us in that there might be some swap lingering around. When files are edited directly via ssh the server will create a temporary ‘swap’ version in case the editor crashes, which prevents the entire contents from being lost.</p><p>While we could not see any <b>~swapme</b> file with the <b>ls</b> command, running a vi command on <b>bootstrap.php-swapme</b> to directly edit the swap file revealed that the file was indeed there, and it contained the exact same contents as the infected version of <b>bootstrap.php</b>.</p><p>It became evident that the attackers were leveraging a swap file to keep the malware present on the server and evade normal methods of detection. We removed that swap file, cleared caches again, and presto: the checkout page source was finally clean.</p><h2>Mitigating further attacks</h2><p>The attacker’s misuse of the swap file system highlights the importance of security measures that go <em>beyond</em> surface-level scans and cleanups. In this case, it was clear that the attackers were able to gain access via SSH or some terminal session, otherwise we wouldn’t have found the swap file that was created when the file was edited.</p><p>To mitigate the risk of persistent ecommerce malware infections like this one, you can restrict sFTP, SSH, FTP, CPanel and any other admin access to trusted IPs. While FTP and SSH restrictions need to be configured directly on the hosting server, our <a href="https://sucuri.net/website-firewall/">website firewall</a> also adds additional layers of protection to prevent unwanted access to areas like your WordPress and Magento admin panels. It is also good practice to keep your CMS and plugins or modules up to date. Older versions of website software often contain vulnerabilities that attackers will easily exploit with automated attack tools.</p><p>If you believe that your site has been infected, our experienced security analysts are <a href="https://sucuri.net/" target="_blank" rel="noopener">available 24&#215;7</a> to help clean up malware in your website environment. If you’re more of a DIY type, we also provide free guides with detailed <a href="https://sucuri.net/guides/how-to-clean-a-hacked-website/">steps to clean up an infected website</a>.</p></div><div class="post-author"><div class="author-wrap"><div class="author"><div class="author-avatar"> <a href="https://blog.sucuri.net/author/matt-morrow" rel="author"> <img alt='' src='https://secure.gravatar.com/avatar/890e38d7466587bdcb96bd18543e2b88?s=120&#038;d=mm&#038;r=g' srcset='https://secure.gravatar.com/avatar/890e38d7466587bdcb96bd18543e2b88?s=240&#038;d=mm&#038;r=g 2x' class='avatar avatar-120 photo' height='120' width='120' loading='lazy' decoding='async'/> </a></div><div class="author-description"><h5 class="title-author"> <span class="fn"> <a href="https://blog.sucuri.net/author/matt-morrow" rel="author"> Matt Morrow </a> </span></h5><p class="note">Matt is a Cyber Security analyst who joined Sucuri in 2018.
