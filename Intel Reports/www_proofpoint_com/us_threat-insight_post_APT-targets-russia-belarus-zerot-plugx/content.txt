    <div class="node-full__body blog-content__body">  <p><strong>Overview</strong></p>

<p>Although state-sponsored attacks against the United States by Chinese threat actors have decreased dramatically since the signing of the US-China Cyber Agreement in 2016, Proofpoint researchers have continued to observe <a href="https://www.proofpoint.com/us/threat-reference/advanced-persistent-threat">advanced persistent threat (APT)</a> activity associated with Chinese actors targeting other regions. We have previously written about related activity [2][3] in which a particular China-based attack group used PlugX and NetTraveler Trojans for espionage in Europe, Russia, Mongolia, Belarus, and other neighboring countries. Most recently, we have observed the same group targeting military and aerospace interests in Russia and Belarus. Since the summer of 2016, this group began using a new downloader known as ZeroT to install the PlugX remote access Trojan (RAT) and added Microsoft Compiled HTML Help (.chm) as one of the initial droppers delivered in spear-phishing emails.</p>

<p>This blog details the function of the new malware, provides delivery details for elements of the APT activity, and describes additional changes in tactics, techniques, and procedures (TTPs) associated with this group.</p>

<p><strong>Delivery</strong></p>

<p>In previous campaigns, the group used spear-phishing emails with Microsoft Word document attachments utilizing CVE-2012-0158, or URLs linking to RAR-compressed executables. Although some of these patterns of behavior still continue, in June 2016 we observed the attackers using a new type of dropper to deliver a previously unknown malware we named "ZeroT". Specifically, the CHM file 20160621.chm (SHA256: 4ef91c17b1415609a2394d2c6c353318a2503900e400aab25ab96c9fe7dc92ff) dropped the first known sample of ZeroT.</p>

<p>The proprietary Microsoft HTML Help file format (.chm) is used for software documentation and may consist of HTML pages and other compressed files. This particular CHM contained an HTM file and an executable file. The HTM file contained the text displayed to the user and referenced the executable svchost.exe (SHA256: d1c4a51064aeec4c11a8f90f80a3b60a36c07cce2dde0756c114e477d63ce375). Thus, opening the CHM has the effect of running the executable (the UAC dialog is shown in Figure 1).</p>

<p><em>Figure 2: Listing of the files in the CHM file and their partial contents</em></p>

<p>Attackers also continued to send spear-phishing emails with Microsoft Word attachments utilizing CVE-2012-0158 to exploit the client. These documents were built with MNKit, described in detail here [6][7]. For example, the email with subject “Федеральная целевая программа 2017-2020 гг.” (translated from Russian: “Federal Target Program 2017-2020 gg.”) contained an attachment “2017-2020.doc” and was sent to a potential victim in an aerospace company in December 2016.</p>

<p><strong>Analysis</strong></p>

<p>This section provides an analysis of ZeroT, its delivery and obfuscation techniques.</p>

<p><strong>UAC Bypass and Sideloading</strong></p>

<p>Throughout our investigation, many of the analyzed ZeroT RAR SFX samples (e.g. 67693ddb6236d3ef790059409ae240212c47acfd8c1c76d65c3ef19096fdf43b) contained a file named Go.exe which performs Windows UAC bypass. This executable contains a PDB path indicating its purpose of bypassing UAC (Fig. 4).</p>

<p><a href="/sites/default/files/apt-4.png" target="_blank"><img alt="PDB path containing Chinese characters translating to 'Desktop'" src="/sites/default/files/apt-4.png"></a></p>

<p><em>Figure 4: PDB path containing Chinese characters translating to “Desktop”</em></p>

<p>This executable is obfuscated using the same technique that is reused in the sideload DLL and the ZeroT payload (described later). When run, Go.exe modifies the registry key shown in Figure 5&nbsp;to perform the UAC bypass by exploiting Event Viewer [1].</p>

<p><a href="/sites/default/files/apt-5.png" target="_blank"><img alt="Modified registry key to execute Zlh.exe exploiting UAC bypass vulnerability in eventvwr.exe" src="/sites/default/files/apt-5.png"></a></p>

<p><em>Figure 5: Modified registry key to execute Zlh.exe exploiting a UAC bypass vulnerability in eventvwr.exe</em></p>

<p>It then executes eventvwr.exe which proceeds to execute Zlh.exe using the UAC bypass vulnerability (Fig. 6).</p>

<p><a href="/sites/default/files/apt-6.png" target="_blank"><img alt="Zlh.exe is executed via the eventvwr.exe UAC bypass vulnerability" src="/sites/default/files/apt-6.png"></a></p>

<p><em>Figure 6: Zlh.exe is executed via the eventvwr.exe UAC bypass vulnerability</em></p>

<p>Zlh.exe is a legitimate, signed <a href="https://en.wikipedia.org/wiki/Norman_Safeground" target="_blank">Norman Safeground AS</a> application, which is used to sideload a malicious nflogger.dll DLL.The encrypted ZeroT payload is usually named NO.2.mui. The sideloaded DLL does not always use the same vulnerable executable, but it is always similar in functionality. Usually the DLL is not packed, but we have observed instances compressed by <a href="https://upx.github.io/" target="_blank">UPX</a>. This malicious DLL is usually obfuscated with the same junk code: dummy API calls inserted in between real instructions (Fig. 7). The same obfuscation can be found in multiple functions in ZeroT itself.</p>

<p><a href="/sites/default/files/apt-7.png" target="_blank"><img alt="Dummy API calling for obfuscation" src="/sites/default/files/apt-7.png" style="border-width: 0px; border-style: solid;"></a></p>

<p><em>Figure 7: Dummy API calls for obfuscation</em></p>

<p>This DLL has no other noticeable characteristics, as it functions like a typical malicious sideload. After loading the encrypted payload in memory, it transfers the execution to a shellcode that is located at the beginning of the file. Even if the process is similar for the PlugX RAT sideloaded later, the shellcode and obfuscation have nothing in common. Once loaded in memory, the ZeroT shellcode does not present any kind of obfuscation, unlike that for PlugX. This shellcode is charged with unpacking the encrypted and compressed payload. As in the new PlugX dropper detailed below, this is done using RC4 and RtlDecompressBuffer. As in PlugX samples, the PE header of ZeroT has been tampered with, specifically the “MZ” and “PE” constants (Fig. 8). On some PlugX versions, either “GULP” or “XV” are common as tags replacing the “MZ” constant.</p>

<p><a href="/sites/default/files/apt-8.png" target="_blank"><img alt="Altered ZeroT PE Header" src="/sites/default/files/apt-8.png" style="border-width: 0px; border-style: solid;"></a></p>

<p><em>Figure 8 : Altered ZeroT PE Header</em></p>

<p><strong>ZeroT Command and Control Protocol</strong></p>

<p>ZeroT communicates with its command and control (C&amp;C) over HTTP. A fake User-Agent is used in all the requests made by this malware: “<em>Mozilla/6.0 (compatible; MSIE 10.0; Windows NT 6.2; Tzcdrnt/6.0)</em>”, with “<em>Tzcdrnt</em>” possibly being a typo of “<em>Trident</em>.” In all the samples we observed, ZeroT first beacons to <em>index.php</em> expecting an RC4-encrypted response using a static key: “(*^GF(9042&amp;*” (Fig. 9).</p>

<p><a href="/sites/default/files/apt-9.png" target="_blank"><img alt="ZeroT initial beacon over HTTP requesting URL configuration" src="/sites/default/files/apt-9.png" style="border-width: 0px; border-style: solid;"></a></p>

<p><em>Figure 9: ZeroT initial beacon over HTTP requesting URL configuration</em></p>

<p>In the decrypted initial server response (Fig. 10,11), ZeroT expects several URLs including a location to POST system information prefixed with “w:” and a download location for any stage 2 payloads denoted with an “r:” prefix.</p>

<p><a href="/sites/default/files/apt-10.png" target="_blank"><img alt="Decrypted tassnews[.]net index.php response containing several URLs" src="/sites/default/files/apt-10.png" style="border-width: 0px; border-style: solid;"></a></p>

<p><em>Figure 10: Decrypted tassnews[.]net index.php response containing several URLs</em></p>

<p>Next, ZeroT uses HTTP POST beacons to transmit information about the infected system to the C&amp;C. The first beacon contains the following data: “<em>Cn=%s&amp;La=%s&amp;</em>” where Cn is the computername and La is the local IP address (Fig. 11). While the first beacon is transmitted in cleartext it is probable that this behavior was unintentional as subsequent POST beacons in the loop are encrypted. The first POST beacon is followed by another in the following format (Fig. 11,12): “<em>Lg=%d&amp;Pv=%d&amp;Bu=%%s&amp;Cn=%s&amp;Cu=%s&amp;Dn=%s&amp;Ki=%s&amp;La=%s&amp;Me=%s&amp;Os=%s&amp;Ov=%s&amp;Pt=%s&amp;Fl=%%d</em>” that is RC4-encrypted with the following key: “s2-18rg1-41g3j_.;”. This POST sends basic fingerprinting data including computer name, system language, domain information and Windows versioning.</p>

<p><a href="/sites/default/files/apt-11.png" target="_blank"><img alt="Initial plaintext POST beacon" src="/sites/default/files/apt-11.png" style="border-width: 0px; border-style: solid;"></a></p>

<p>Figure 11: Initial plaintext POST beacon</p>

<p><a href="/sites/default/files/apt-12.png" target="_blank"><img alt="RC4-encrypted POST beacon" src="/sites/default/files/apt-12.png" style="border-width: 0px; border-style: solid;"></a></p>

<p>Figure 12: RC4-encrypted POST beacon</p>

<p>The final piece of ZeroT’s C&amp;C protocol is to retrieve any stage-2 payloads. In the initial samples of ZeroT, the <em>tassnews[.]net </em>C&amp;C was used to distribute plain, non-encoded PE payloads (Fig. 13). Although we were not able to retrieve all the payloads that may have existed at this C&amp;C, the ones we did observe were RAR SFX archives used to deliver PlugX.</p>

<p><a href="/sites/default/files/apt-13.png" target="_blank"><img alt="ZeroT downloading non-encoded PlugX Stage 2" src="/sites/default/files/apt-13.png" style="border-width: 0px; border-style: solid;"></a></p>

<p>Figure 13: ZeroT downloading non-encoded PlugX Stage 2</p>

<p>The ZeroT samples communicating to <em>versig[.]net </em>functioned differently from samples using tassnews[.]net where Bitmap (BMP) [8] URLs (Figure 14) were provided as a stage 2 payload instead of EXEs (Fig. 14).</p>

<p><a href="/sites/default/files/apt-14.png" target="_blank"><img alt="Decrypted versig[.]net index.php response with F.bmp stage 2" src="/sites/default/files/apt-14.png" style="border-width: 0px; border-style: solid;"></a></p>

<p><em>Figure 14: Decrypted versig[.]net index.php response with F.bmp stage 2</em></p>

<p>The BMPs used for stage 2 in all the instances we analyzed looked like normal images (Fig. 15, 16) which indicated a form of steganography is being used that minimizes changes to the appearance of the image.</p>

<p><a href="/sites/default/files/apt-15.png" target="_blank"><img alt="F.bmp image containing stage 2 hidden using steganography" src="/sites/default/files/apt-15.png" style="border-width: 0px; border-style: solid;"></a></p>

<p><em>Figure 15: F.bmp image containing stage 2 hidden using steganography</em></p>

<p><em>Analysis of the F.bmp image revealed that it is indeed using Least Significant Bit (LSB) </em>Steganography [9,10]<em>, a</em> commonly used form of steganography that embeds data in an image without significantly affecting its appearance.</p>

<p><strong>Analysis of ZeroT’s Bitmap LSB Steganography</strong></p>

<p>ZeroT uses a single, large function for the custom LSB algorithm that occurs as the first function in the samples we analyzed. It may selectively choose to extract one, two, three, or four bits per pixel byte depending on the size of the embedded payload and the image being used, meaning it is capable of 1-, 2-, 3-, and 4-bit LSB (Fig. 16).</p>

<p><a href="/sites/default/files/apt-16.png" target="_blank"><img alt="Diagram depicting portions of ZeroT’s LSB steganography algorithm" src="/sites/default/files/apt-16.png" style="border-width: 0px; border-style: solid;"></a></p>

<p><em>Figure 16: Diagram depicting portions of ZeroT’s LSB steganography algorithm</em></p>

<p>The first step in the algorithm extracts the width and height from the image. A 24-bit depth Bitmap is assumed while calculating the size of memory needed (3*Width*Height) to store all the pixels in the image, which is allocated using <em>malloc</em>. Next, because Bitmaps are padded to 32-bit boundaries, ZeroT will check to see if any padding exists by AND’ing 3*Width. If that value is not zero, then it is subtracted from four which is then used as the padding.</p>

<p>The following step in the algorithm is to assemble the bitmap row by row. Technically what occurs is as follows: iteratively 3*Width bytes, beginning at the hardcoded offset of 54, are copied to the end of the previously allocated memory (malloc’d 3*Width*Height). Padding is then skipped, if any exists, followed by the next 3*Width bytes until all rows of pixels are copied.</p>

<p>Next, starting at offset 10 of the assembled bitmap, ZeroT extracts a 32-bit value from 32-bytes using 1-bit LSB that is used to indicate the size of the embedded stage 2 payload. The stage 2 file extension is then extracted starting at offset nine and working its way backwards until a period is found (e.g., “<em>.exe</em>”).</p>

<p>Finally, ZeroT checks various values such as the image length against the needed bits to complete the length of the extracted payload [8*len(extracted payload)]. Depending on the result, ZeroT will decide to use 1-, 2-, 3-, or 4-bit LSB. Lastly, once each bit is extracted, all the bits will be compiled into bytes to form the extracted stage 2 payload.</p>

<p><strong>Stage 2 Payloads</strong></p>

<p>Until the end of 2016, PlugX payloads were delivered as RAR SFX archives and used one of the usual sideload executables such as fsguidll.exe. ZeroT sets up the persistence for these samples,&nbsp; adding a new service to run PlugX during system startup (Fig 17).</p>

<p><a href="/sites/default/files/apt-17.png" target="_blank"><img alt="ZeroT configuring PlugX service to run during startup" src="/sites/default/files/apt-17.png" style="border-width: 0px; border-style: solid;"></a></p>

<p><em>Figure 17: ZeroT configures PlugX service to run during startup</em></p>

<p>Of interest, a recent ZeroT sample (SHA256: a9519d2624a842d2c9060b64bb78ee1c400fea9e43d4436371a67cbf90e611b8) downloaded a much smaller BMP payload (SHA256: 25de9c3f7bf1f0be7eb84cf90efb643d5d51ce1742da8bcc4c7db0eec79a221f). This was an example where the stage 2 payload was distributed using a custom dropper instead of RAR SFX. This dropper loads the resources (Fig. 18), decrypts them using the MD5 hash of a command line argument as the RC4 key (note: crypto API is used instead of the custom RC4 implementation) and decompresses it with LZNT1 via the RtlDecompressBuffer API. There are three Resource items contained in the payload that would eventually decrypt to: “<em>fsguidll.exe</em>”, “<em>fslapi.dll</em>”, and “<em>flsapi.dll.gui</em>.” Unfortunately, when the payload extracted from the BMP is executed, no command line argument is provided so none of the resources are properly decrypted and decompressed. Based on the file names and size of fslapi.dll.gui, it is very likely that PlugX is the intended stage 2 payload.</p>

<p><a href="/sites/default/files/apt-18.png" target="_blank"><img alt="Resources loaded by the custom dropper" src="/sites/default/files/apt-18.png" style="border-width: 0px; border-style: solid;"></a></p>

<p><em>Figure 18: Resources loaded by the custom dropper</em></p>

<p>Finally, in all other cases where a stage 2 payload was successfully retrieved, PlugX was delivered. None of the PlugX samples that we analyzed were issued from new builders (internal version 20141028), and therefore they do not present any kind of new techniques. The extracted PlugX configurations are provided in Appendices A and B [10].</p>

<p><strong>Infrastructure Links</strong></p>

<p>In addition to their similar TTPs, ZeroT infrastructure has been continuously shared with NetTraveler and both malware families share the same C&amp;C domains.</p>

<ul>
	<li>The C&amp;C domain www.tassnews[.]net was used by initial samples of ZeroT in June 2016. It was concurrently used by NetTraveler (example SHA256: b43cbc905088c08ee3b71b6e053f91f2c79d71556462eae1c13f1cc8eb5bec72) as well as long prior [3]</li>
	<li>The C&amp;C domain www.riaru[.]net was observed used by at least one sample of ZeroT (SHA256: fc2d47d91ad8517a4a974c4570b346b41646fac333d219d2f1282c96b4571478). This domain was previously tied to NetTraveler as well [3]</li>
	<li>The C&amp;C domain www.versig[.]net is used by many samples of ZeroT from September 2016 to January 2017. This domain was also used by many NetTraveler samples as well (example SHA256: 0d6d789d603d6d9ba68131592fd595c4d82c0288be309876d27a53466158b312) in the time frame from October 2016 to January 2017</li>
</ul>

<p>The PlugX samples downloaded by ZeroT exhibit infrastructure connections to PlugX samples described in our 2015 blog about this group,&nbsp; “<a href="https://www.proofpoint.com/us/threat-insight/post/PlugX-in-Russia" target="_blank">In Pursuit of Optical Fibers and Troop Intel: Targeted Attack Distributes PlugX in Russia</a>”. Specifically:</p>

<ul>
	<li>The C&amp;C domain dicemention[.]com was configured (or not removed from) in the PlugX (SHA256: 3149fb0ddd89b77ecfb797c4ab4676c63d157a6b22ba4c8f98e8478c24104dfa) downloaded by ZeroT (SHA256: d1c4a51064aeec4c11a8f90f80a3b60a36c07cce2dde0756c114e477d63ce375). This domain was also used by PlugX samples described in the 2015 blog</li>
</ul>

<p>Note also an interesting connection: the hostname www.riaru[.]net resolved to IP 103.200.31[.]110 which also responded for yandax[.]net. One of the PlugX C&amp;Cs (www[.].micrnet[.]net) resolved to 103.229.28[.]133 which also resolved to yandcx[.]com.</p>

<p><strong>Conclusion</strong></p>

<p>This APT activity represents both a change in TTPs as well as the introduction of new malware known as ZeroT by a Chinese state-sponsored attack group that we have previously associated with multiple campaigns. Proofpoint researchers have predicted that APT activity will continue to increase in the coming year and we will continue to track developments among state-sponsored actors.</p>
