											<div class="js-reading-content">
												<div class="c-wysiwyg">
													<p>Sometimes when investigating an infection and focusing on a targeted attack, we come across something we were not expecting. The case described below is one such occurrence.</p>
<p>In June 2022, we found a suspicious shellcode running in the memory of a system process. We decided to dig deeper and investigate how the shellcode was initially placed into the process and where on the infected system the threat was hidden.</p>
<h2 id="the-infection-chain">The infection chain</h2>
<p>We were unable to reproduce the whole infection procedure, but we were able to reconstruct it from the point at which PowerShell was executed, as shown in the sceme below.</p>
<p style="text-align: center"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01.png" class="magnificImage"><img loading="lazy" decoding="async" class="aligncenter size-large wp-image-109697" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01-932x1024.png" alt="General attack execution flow" width="932" height="1024" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01-932x1024.png 932w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01-273x300.png 273w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01-768x843.png 768w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01-1399x1536.png 1399w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01-159x175.png 159w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01-370x406.png 370w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01-255x280.png 255w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01-800x879.png 800w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150326/Minas_on_the_way_to_complex_01.png 1600w" sizes="(max-width: 932px) 100vw, 932px" /></a></p>
<p style="text-align: center"><strong><em>General attack execution flow</em></strong></p>
<p>In a nutshell, the infection chain is as follows:</p>
<ol>
<li>PowerShell script runs via the Task Scheduler and downloads the lgntoerr.gif file from a remote server.</li>
<li>The script decrypts lgntoerr.gif, resulting in a .NET DLL, which is then loaded.</li>
<li>The .NET DLL extracts and decrypts three files from its resources: two DLLs and an encrypted payload. The extracted files are placed in the ProgramData directory.</li>
<li>The .NET DLL creates a task to autorun the legitimate ilasm.exe component at system startup via Task Scheduler.</li>
<li>Task Scheduler starts ilasm.exe from the ProgramData directory.</li>
<li>ilasm.exe launches fusion.dll, a malicious DLL hijacker, from the same directory.</li>
<li>fusion.dll loads the second decrypted DLL.</li>
<li>That DLL creates a suspended dllhost.exe process.</li>
<li>It then decrypts the payload from the encrypted binary file.</li>
<li>The decrypted payload is loaded into the dllhost.exe process as a DLL.</li>
<li>The PID of the dllhost.exe process is saved to a file in the ProgramData directory.</li>
<li>The dllhost.exe process passes control to the decrypted payload.</li>
<li>The payload DLL extracts and launches the miner DLL in memory.</li>
</ol>
<p>We named this malware Minas. From our reconstruction of the infection chain, we determined that it originated by running an encoded PowerShell script as a task, which we believe with low confidence was created through GPO:</p>

<p style="text-align: center"><strong><em>Encoded PowerShell command</em></strong></p>
<h2 id="technical-description">Technical description</h2>
<p>The core functionality of the PowerShell script is to launch the malware installation process. To do this, the PowerShell script downloads an encrypted payload from a remote server, decrypts it using a custom XOR encryption algorithm with the key &#8220;fuckkasd9akey&#8221; and loads the  payload into memory:</p>
<p style="text-align: center"><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150447/Minas_on_the_way_to_complex_03.png" class="magnificImage"><img loading="lazy" decoding="async" class="aligncenter size-large wp-image-109699" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150447/Minas_on_the_way_to_complex_03-1024x219.png" alt="Decoded PowerShell command" width="1024" height="219" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150447/Minas_on_the_way_to_complex_03-1024x219.png 1024w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150447/Minas_on_the_way_to_complex_03-300x64.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150447/Minas_on_the_way_to_complex_03-768x164.png 768w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150447/Minas_on_the_way_to_complex_03-817x175.png 817w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150447/Minas_on_the_way_to_complex_03-370x79.png 370w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150447/Minas_on_the_way_to_complex_03-800x171.png 800w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2023/05/16150447/Minas_on_the_way_to_complex_03.png 1093w" sizes="(max-width: 1024px) 100vw, 1024px" /></a></p>
<p style="text-align: center"><strong><em>Decoded PowerShell command</em></strong></p>
<p>The payload is a .NET binary (DLL) that is launched by the PowerShell process, passing three arguments:</p>
<p>$a.GetType(&#8220;I.C&#8221;).GetMethod(&#8220;I&#8221;).Invoke($null, ([Byte[]]($d),&#8221;A5D9FD13&#8243;,0));</p>
<ol>
<li>$d: .NET DLL as an array of bytes;</li>
<li>A5D9FD13: (key for decrypting resources in the .NET DLL);</li>
<li>0: (parameter used to block the creation of more than one main payload process).</li>
</ol>
<p>This .NET binary (which we will refer to as &#8220;the installer&#8221;) is responsible for the subsequent installation of the malware components contained in the resources of .NET DLL.</p>

After this, the files are copied to the previously created directories as mentioned above (see directory structure).</p>
<p>The malware then proceeds to append the first 100 bytes of the installer to the file {RSHash(MachineName)} that was extracted from the &#8220;_64_bin&#8221; .NET DLL resource, and the resulting file is encrypted (the key is the machine name in lower case). In addition, up to 10240 of random bytes are appended to the fusion.dll and {SDBMHash(MachineName)}.dll files, thus rendering hash detection by anti-malware products ineffective.</p>
<p>After that, the installer starts the previously created task for the running ilasm.exe, which in turn loads the malicious fusion.dll library located in the same directory, under the assumption it is using the legitimate fusion.dll (DLL hijacking technique).</p>
<p>The malware execution process consists of two loaders: DLL hijacker fusion.dll and the next stage loader {SDBMHash(MachineName)}.dll.</p>
<p>The DLL hijacker does three things:</p>
<ol>
<li>Hides the console of the ilasm.exe process.</li>
<li>Determines whether the name of the process is ilasm.exe. If not, it terminates the process.</li>
<li>Based on the SDBM hash function, it builds the path C:\ProgramData\{SDBMHash(MachineName)}\{SDBMHash(MachineName)}.dll and tries to load that DLL into memory via the LoadLibraryA API function.</li>
</ol>
<p>The loaded {SDBMHash(MachineName)}.dll DLL again checks whether the name of the process is ilasm.exe.</p>
<p>It then generates the file path C:\ProgramData\{SDBMHash({SDBMHash(MachineName)})} and checks if it exists. If the file exists, it reads the value (decimal number) from it. This decimal number is the PID of the dllhost.exe process multiplied by 0x1F4 created by a previous run of Minas. The loader then tries to terminate the process with that PID.</p>
<p>It then reads and decrypts (recall that the key is the machine name in lower case) the main payload (the miner) located at C:\ProgramData\{ApHash(MachineName)}\{RSHash(MachineName)}\{RSHash(MachineName)}. In addition, the loader creates a process environment variable (SetEnvironmentVariable) configuration with values ​​{&#8220;addr&#8221;:&#8221;185.243.112.239:443&#8243;,&#8221;p&#8221;:&#8221;143e256609bcb0be5b9f9c8f79bdf8c9&#8243;} (see the configuration description below for more details).</p>

<p>After that, {SDBMHash(MachineName)}.dll creates a suspended dllhost.exe process, creates a section, maps it into the dllhost.exe process and writes the previously read and decrypted file (raw loader with the miner component). It then reads the memory of the dllhost.exe process, determines the entry point of the main thread and rewrites it to:</p>

<h2 id="conclusion">Conclusion</h2>
<p>Minas is a miner that uses a standard implementation and aims to hide its presence. The difficulty of detection is achieved due to encryption, the random generation of names and the use of hijacking and injection techniques. It also has the ability to stay on the infected system using persistence techniques.</p>
<p>Although we were unable to determine with complete certainty how the initial PowerShell command was executed, our indicators pointed to execution via GPO. This is particularly worrisome because it means the attackers have compromised the network, indicating that they are going to great lengths to install their miners.</p>
<p>Also, it is very likely that a new variant will be released in the future that avoids AV detection. As a rule of thumb, use an AV that doesn&#8217;t primarily rely on signature detection, but also looks at (processes) behavior.</p>
