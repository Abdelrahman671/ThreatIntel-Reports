	<div class="container md:my-16 xl:my-24 my-8">
	<div class="entry-content lg:prose-lg mx-auto prose max-w-4xl">
		<p>In 2022 and 2023, Sophos X-Ops published research about <a href="https://news.sophos.com/en-us/2022/12/13/signed-driver-malware-moves-up-the-software-trust-chain/" target="_blank" rel="noopener">a toolset to sabotage the functions of endpoint protection software</a> that was being developed and used in conjunction with several major ransomware gangs. Mandiant had previously named this tool Poortry, and its loader application Stonestop.</p>
<p>The creators of the Poortry tool had managed to get purpose-built, custom kernel-level drivers signed through Microsoft’s attestation signing process. After we published our research &#8212; and Microsoft closed the loophole that allowed these drivers to be signed &#8212; the tool&#8217;s creators did not just stop. They have continued to add features and functionality to the Poortry driver, in an ongoing attempt to evade detection and to find new ways to disable EDR and endpoint protection software.</p>
<p>To explain the new features in Poortry, let&#8217;s review how drivers interact with the operating system, and how the developers of this EDR killer evolved their tool over time.</p>
<h3>How Windows drivers can sabotage protection</h3>
<p>Most EDR killers rely on a device driver being loaded into the operating system&#8217;s kernel, which gives them access to the kinds of low-level functionality to be able to unhook and terminate various kinds of protection software.</p>
<p>Under Windows, which supports a multitude of peripherals and connected components, kernel-mode drivers are given wide latitude to these kinds of low-level functions. Under normal circumstances, these drivers don’t interact with software or hardware from other companies or manufacturers, but there is no enforcement of this behavior. Thus, if a signed legitimate driver does not properly validate the processes interacting with it, EDR killers can exploit some of its features to remove protective measures.</p>
<p>Microsoft has developed a variety of ways that their operating systems can control whether drivers get loaded at all, such as the Driver Signature Enforcement mechanism: Drivers must be digitally signed by a software publisher Microsoft trusts before they can load.</p>
<p>The developers of EDR killers exploit the gaps in this trust model: They may use a driver vulnerable to abuse that was once published by a legitimate software company; They also might sign their own driver with a legitimate code-signing certificate (and there are many ways to obtain stolen or leaked certificates).</p>
<p>In general there are three ways EDR killer developers abuse code signatures:</p>
<h4>Abuse of leaked certificates</h4>
<p>This is the most straightforward way to handle the problem: Find a leaked, stolen, or otherwise compromised code-signing certificate from a legitimate company, and use it to sign your driver (or to trick Root Certificate Authorities into issuing a certificate to you).</p>
<p>For all versions of Windows that came after Windows 10 version 1607, Microsoft has required all third-party developers of kernel-mode drivers to submit their driver to Microsoft&#8217;s developer portal, to be cross-signed by Microsoft. However, cross-signed drivers not signed by Microsoft are still allowed to be loaded if it fulfills one of the following :</p>
<ul>
<li>The PC was upgraded from an earlier release of Windows to Windows 10, version 1607</li>
<li>Secure Boot is switched off in the system BIOS</li>
<li>Driver was signed with an end-entity certificate issued prior to July 29<sup>,</sup> 2015 that chains to a supported cross-signed CA</li>
</ul>
<p>Even though the update lowered the danger of cross-signed drivers that had been signed by stolen certificates, the third bullet creates a loophole that enables the second method for attackers.</p>
<h4>Signature timestamp forgery</h4>
<p>In order to maintain compatibility with older drivers, Windows loads drivers signed with “an end-entity certificates issued prior to July 29, 2015 that chains to a supported cross-signed CA.”</p>
<p>When signing a kernel driver, Microsoft provides the software publisher with a tool named <a href="https://learn.microsoft.com/en-us/dotnet/framework/tools/signtool-exe" target="_blank" rel="noopener">signtool.exe</a>. In addition to signing the provided file, signtool also checks to ensure that the provided certificate is still valid. One way to ensure this is to use the function</p>
<p>Through a series of hooks to these low-level API calls inside the operating system, attackers can alter the signing process and bypass these checks to sign their own kernel driver. One of the functions being hooked in this technique is <a href="https://learn.microsoft.com/en-us/windows/win32/api/sysinfoapi/nf-sysinfoapi-getlocaltime">GetLocalTime</a> to return a forged timestamp to pass through the checks in signtool.exe.</p>
<h4>Bypassing Microsoft attestation signing</h4>
<p>The final method is to get through Microsoft&#8217;s attestation signing process, and get the kernel driver signed directly by Microsoft. This is probably the most difficult to achieve, but also provides a signature a strong WHQL certificate that was issued by Microsoft itself – almost a holy grail of digital signatures.</p>
<p>To abuse this method, attackers need:</p>
<ul>
<li>A valid EV certificate</li>
<li>Access to the Microsoft developer portal</li>
</ul>
<p>If these requirements are fulfilled, they can prepare a CAB file, which includes the driver itself, sign it with the EV certificate, and submit it to the dashboard.</p>
<p><a href="https://news.sophos.com/wp-content/uploads/2024/08/Image1update.png"><img decoding="async" class="alignnone size-full wp-image-957028" src="https://news.sophos.com/wp-content/uploads/2024/08/Image1update.png" alt="" width="640" height="380" srcset="https://news.sophos.com/wp-content/uploads/2024/08/Image1update.png 812w, https://news.sophos.com/wp-content/uploads/2024/08/Image1update.png?resize=300,178 300w, https://news.sophos.com/wp-content/uploads/2024/08/Image1update.png?resize=768,456 768w" sizes="(max-width: 640px) 100vw, 640px" /></a></p>
<p>Once submitted, the driver undergoes several checks to ensure that the driver is not malicious. If the driver passes these tests, it will carry the “Microsoft Windows Hardware Compatibility Publisher” signature.</p>
<figure id="attachment_957029" aria-describedby="caption-attachment-957029" style="width: 384px" class="wp-caption alignnone"><a href="https://news.sophos.com/wp-content/uploads/2024/08/image2_6c98a8.png"><img decoding="async" class=" wp-image-957029" src="https://news.sophos.com/wp-content/uploads/2024/08/image2_6c98a8.png" alt="an attestation signed driver" width="384" height="513" srcset="https://news.sophos.com/wp-content/uploads/2024/08/image2_6c98a8.png 303w, https://news.sophos.com/wp-content/uploads/2024/08/image2_6c98a8.png?resize=224,300 224w" sizes="(max-width: 384px) 100vw, 384px" /></a><figcaption id="caption-attachment-957029" class="wp-caption-text">One of the WHQL signed drivers from the attacks in 2022-2023</figcaption></figure>
<h3>Poortry &amp; Stonestop: A Relevant Threat Since 2022</h3>
<p>Poortry (also sometimes called BurntCigar) is a malicious kernel driver used in conjunction with a loader named Stonestop by Mandiant, who first reported on the tool&#8217;s existence. The driver bypasses Driver Signature Enforcement by using any of the three techniques described above. Both are heavily obfuscated by commercial or open-source packers, such as VMProtect, Themida or ASMGuard.</p>
<p>From the end of 2022 to mid-2023, Poortry variants carried the Microsoft WHQL certificate. However, due to joint work Between Sophos X-Ops and Microsoft, most of this attestation signed samples were found and Microsoft deactivated the accounts that were abused to get those drivers signed.</p>
<p>Poortry&#8217;s creators were not deterred; Instead, they switched to either Signature Timestamp Forging or obtaining a valid leaked certificate.</p>
<p>Over the last year, we were able to link the use of Poortry to attacks involving at least five major ransomware families:</p>
<ul>
<li>CUBA</li>
<li>BlackCat</li>
<li>Medusa</li>
<li>LockBit</li>
<li>RansomHub</li>
</ul>
<p>Since 2023, we&#8217;ve observed threat actors repeatedly use Poortry during attacks. One characteristic we observed in our earlier research is that Poortry’s creators change their packer frequently, creating a volume of slightly modified variants based off the original. In our research, we found multiple different WHQL-signed variants, packed with different commercial or non-commercial packers.</p>
<p>Since that venue was closed to them, Poortry&#8217;s makers now deploy the drivers signed by a wide variety of non-Microsoft certificates.</p>
<p>The figure below illustrates a timeline of the observed signer names used by Poortry&#8217;s payload driver over a 15 month period.</p>
<p><a href="https://news.sophos.com/wp-content/uploads/2024/08/Image3updated.png"><img loading="lazy" decoding="async" class="alignnone wp-image-957030" src="https://news.sophos.com/wp-content/uploads/2024/08/Image3updated.png" alt="BurntCigar driver signing certificates over time In the past 17 months, the threat actors swapped the signing certificates they used for their executables at least nine times" width="852" height="366" srcset="https://news.sophos.com/wp-content/uploads/2024/08/Image3updated.png 988w, https://news.sophos.com/wp-content/uploads/2024/08/Image3updated.png?resize=300,129 300w, https://news.sophos.com/wp-content/uploads/2024/08/Image3updated.png?resize=768,330 768w" sizes="(max-width: 852px) 100vw, 852px" /></a></p>
<p>It is worthwhile mentioning that sometimes we make our observations during incident response engagements, and at other times collected as telemetry. One thing we can be sure of is that the total number and variety of certificates is larger than our observation alone can determine.</p>
<h3>Playing certificate roulette</h3>
<p>Sophos, from time to time, has observed a threat actor deploy variants of Poortry on different machines within a single estate during an attack. These variants contain the same payload, but signed with a different certificate than the driver first seen used during the attack.In August 2023, during a Sophos X-Ops investigation, we found that attackers gained initial access via a remote access tool named SplashTop. As soon as the attackers were on the network, they deployed Poortry and Stonestop. But the signer name, “bopsoft,” was already known as a stolen certificate, and was blocked using a behavioral rule.</p>
<p>Within 30 seconds after the last attempt using the &#8220;Bopsoft&#8221; signed code, the attackers were loading a different Poortry driver, this one signed by &#8220;Evangel Technology (HK) Limited.&#8221; The host was quickly isolated and the attack thwarted.</p>
<p><a href="https://news.sophos.com/wp-content/uploads/2024/08/image4update.png"><img loading="lazy" decoding="async" class="alignnone wp-image-957031" src="https://news.sophos.com/wp-content/uploads/2024/08/image4update.png" alt="Threat actor swaps signed binary mid-attack While laying the ground for a ransomware deployment, the threat actor repeatedly attempts to deploy the PoorTry driver. When Sophos endpoint protection halts the attack, they switch to a driver installer signed with a different signing certificate." width="766" height="389" srcset="https://news.sophos.com/wp-content/uploads/2024/08/image4update.png 911w, https://news.sophos.com/wp-content/uploads/2024/08/image4update.png?resize=300,152 300w, https://news.sophos.com/wp-content/uploads/2024/08/image4update.png?resize=768,389 768w" sizes="(max-width: 766px) 100vw, 766px" /></a></p>
<h2>Transition from EDR killer To EDR wiper</h2>
<p>In July 2024, while engaged in an incident where adversaries attempted to deploy RansomHub ransomware, Sophos CryptoGuard thwarted the attempted data encryption as analysts closed off the attackers&#8217; points of access. A post-incident analysis revealed that two additional executables had been dropped on multiple machines prior to the final ransomware attack:</p>
<pre>&lt;d&gt;\Users\&lt;u&gt;\desktop\c7iy3d.exe
&lt;d&gt;\Users\&lt;u&gt;\appdata\local\temp\usnnr.sys</pre>
<p>Through a combination of static and dynamic analysis, we determined the files to be Poortry and Stonestop. Among the differences we observed between the prior version and this version, Poortry now can also delete critical EDR components completely, instead of simply terminating their processes.</p>
<p>Trend Micro <a href="https://www.trendmicro.com/de_de/research/23/e/blackcat-ransomware-deploys-new-signed-kernel-driver.html">reported</a> in 2023 that Poortry had developed the capability to delete files off disk, but this was the first time we observed this feature used in an attack.</p>
<h3>A closer look at the latest variants</h3>
<p>Both the Stonestop executable and the Poortry driver are heavily packed and obfuscated. This loader was obfuscated by a closed-source packer named ASMGuard, available on Github.</p>
<figure id="attachment_957035" aria-describedby="caption-attachment-957035" style="width: 640px" class="wp-caption alignnone"><a href="https://news.sophos.com/wp-content/uploads/2024/08/image5.2.png"><img loading="lazy" decoding="async" class="size-full wp-image-957035" src="https://news.sophos.com/wp-content/uploads/2024/08/image5.2.png" alt="PoorTry driver properties shown in CFF Explorer" width="640" height="547" srcset="https://news.sophos.com/wp-content/uploads/2024/08/image5.2.png 720w, https://news.sophos.com/wp-content/uploads/2024/08/image5.2.png?resize=300,256 300w" sizes="(max-width: 640px) 100vw, 640px" /></a><figcaption id="caption-attachment-957035" class="wp-caption-text">PoorTry driver properties shown in CFF Explorer. (Editor&#8217;s note: The &#8220;Created&#8221; date is misleadingly based on when the analyst downloaded the sample, not when the file was first created.)</figcaption></figure>
<p>The driver is signed with a certificate carrying the signer name “FEI XIAO.” Sophos X-Ops has high confidence the signature timestamp was forged to sign the driver. Notably, it tries to masquerade by using the same information in its properties sheet as a driver (idmtdi.sys) for a commercially available software, <a href="https://www.tonec.com/products/idm/index.html">Internet Download Manager by Tonec Inc.</a> But it isn&#8217;t this software package&#8217;s driver &#8211; the attackers merely cloned the information from it.</p>
<figure id="attachment_957033" aria-describedby="caption-attachment-957033" style="width: 451px" class="wp-caption alignnone"><a href="https://news.sophos.com/wp-content/uploads/2024/08/image5.1.png"><img loading="lazy" decoding="async" class="size-full wp-image-957033" src="https://news.sophos.com/wp-content/uploads/2024/08/image5.1.png" alt="PoorTry driver property sheet" width="451" height="568" srcset="https://news.sophos.com/wp-content/uploads/2024/08/image5.1.png 451w, https://news.sophos.com/wp-content/uploads/2024/08/image5.1.png?resize=238,300 238w" sizes="(max-width: 451px) 100vw, 451px" /></a><figcaption id="caption-attachment-957033" class="wp-caption-text">PoorTry driver property sheet with validity dates that predate its creation by more than a decade</figcaption></figure>
<p>For explanatory purposes, we divide the execution flow into three distinct phases.</p>
<h4>Initialization Phase</h4>
<p>In incidents we&#8217;ve tracked, threat actors drop Poortry and Stonestop together, into the same directory. On execution, Stonestop checks for the corresponding driver in the current directory.</p>
<figure id="attachment_957036" aria-describedby="caption-attachment-957036" style="width: 249px" class="wp-caption alignnone"><a href="https://news.sophos.com/wp-content/uploads/2024/08/image6_f10dd1.png"><img loading="lazy" decoding="async" class="wp-image-957036 size-full" src="https://news.sophos.com/wp-content/uploads/2024/08/image6_f10dd1.png" alt="an error message will be displayed that the loader failed connecting to the kernel driver." width="249" height="146" /></a><figcaption id="caption-attachment-957036" class="wp-caption-text">Error message displayed when the loader fails to connect to the kernel driver.</figcaption></figure>
<p>The filename and device name of the driver are both hardcoded into the loader. Upon start, the loader fetches the handle of the malicious kernel driver and initiates a handshake by sending a hardcoded string to the driver via the <a href="https://learn.microsoft.com/en-us/windows/win32/api/ioapiset/nf-ioapiset-deviceiocontrol">DeviceIoControl</a> API call.</p>
<p>Overall, communication between the components happens through this DeviceIoControl API. Each feature provided by the kernel-mode component is triggered via sending a different IOCTL code. Earlier variants communicated via the IRP_MJ_DEVICE_CONTROL handler. The current variant uses the IRP_MJ_MAXIMUM_FUNCTION handler now to receive I/O request packets.</p>
<p>It is worthwhile mentioning that the mappings from IOCTL code to feature has changed since our last analysis. As an example, the command to kill a specific process by process ID was triggered by sending an I/O request packet with code 0x222094. The latest sample maps the IOCTL code 0x222144 to the same functionality.</p>
<p>Since Trend Micro&#8217;s 2023 report, Poortry&#8217;s developers increased the number of receivable IOCTL codes from 10 to 22. Our analysis of all available features is still ongoing.</p>
<p>Like previous versions, a handshake is initiated by sending a hardcoded string to the driver. Once the handshake value is accepted, it sets a flag in the binary that enables the functionalities of the  malicious driver.</p>
<figure id="attachment_957038" aria-describedby="caption-attachment-957038" style="width: 640px" class="wp-caption alignnone"><a href="https://news.sophos.com/wp-content/uploads/2024/08/image7_8b7309.png"><img loading="lazy" decoding="async" class="wp-image-957038 size-full" src="https://news.sophos.com/wp-content/uploads/2024/08/image7_8b7309.png" alt="Handshake value sent to Poortry" width="640" height="33" srcset="https://news.sophos.com/wp-content/uploads/2024/08/image7_8b7309.png 862w, https://news.sophos.com/wp-content/uploads/2024/08/image7_8b7309.png?resize=300,16 300w, https://news.sophos.com/wp-content/uploads/2024/08/image7_8b7309.png?resize=768,40 768w" sizes="(max-width: 640px) 100vw, 640px" /></a><figcaption id="caption-attachment-957038" class="wp-caption-text">Handshake value sent to Poortry</figcaption></figure>
<h4>Impairment Phase</h4>
<p>The second phase is focused on disabling EDR products through a series of different techniques, such as removal or modification of kernel notify routines.</p>
<p>Security drivers make use of several different features provided by the Windows OS to register callbacks when specific events on the Windows system occur. An example would be the function <a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/ntddk/nf-ntddk-pssetcreateprocessnotifyroutine">PsSetCreateProcessNotifyRoutine</a>, which adds a driver supplied callback routine when a new process is created.</p>
<p>Eliminating these callback routines are often a critical step to render EDR products useless. In 2022, we also wrote about a <a href="https://news.sophos.com/en-us/2022/10/04/blackbyte-ransomware-returns/">similar case</a> where BlackByte ransomware abused a legitimate vulnerable driver to remove critical kernel notify routines.</p>
<p>In the second phase, we observed a total of seven distinct IOCTL codes are sent to the kernel-mode component. Only the functionality mapped to 0x222400 is executed. The other features bailed out early due to specific flags being set in the binary. We suspect that the non-triggered functionalities are either experimental, only triggered on specific type of systems, or simply disabled.</p>
<p>The IOCTL codes and their mapped behaviors are as follows:</p>
<h4>0x2220C0 (Disabled)</h4>
<p>When received, Poortry enters an additional initialization routine, fetching addresses of various critical structures and functions.</p>
<h4>0x222100 (Disabled)</h4>
<p>When received, Poortry attempts to disable or enable kernel callbacks via modification of the PspNotifyEnableMask flag. This is a common trick used by rootkits to enable or disable kernel routine callbacks, <a href="https://overlayhack.com/edr-bypass-evasion">as explained by this article</a>.</p>
<h4>0x222104 (Disabled)</h4>
<p>When it receives this IOCTL code, Poortry modifies the kernel callbacks of the PsProcess, PsThread and ExDesktopObj object types. These are kernel-mode data structures that represent specific objects in the Windows kernel. Self-explanatory, the PsProcess object type represents a process object. These object types also contain a variable pointing to the callbacks registered for the corresponding object.</p>
<p>Because this feature was disabled, we are unsure how adversaries might aim to modify these callback lists. One possible scenarios might be to either disable them entirely by setting the callbacks to a custom function without any functionality, simply returning instantly,</p>
<figure id="attachment_957039" aria-describedby="caption-attachment-957039" style="width: 610px" class="wp-caption alignnone"><a href="https://news.sophos.com/wp-content/uploads/2024/08/image8_81de8c.png"><img loading="lazy" decoding="async" class="size-full wp-image-957039" src="https://news.sophos.com/wp-content/uploads/2024/08/image8_81de8c.png" alt="Modifying the object type's callback lists" width="610" height="110" srcset="https://news.sophos.com/wp-content/uploads/2024/08/image8_81de8c.png 610w, https://news.sophos.com/wp-content/uploads/2024/08/image8_81de8c.png?resize=300,54 300w" sizes="(max-width: 610px) 100vw, 610px" /></a><figcaption id="caption-attachment-957039" class="wp-caption-text">Modifying the object type&#8217;s callback lists</figcaption></figure>
<h4>0x222108 (Disabled)</h4>
<p>When received, Poortry modifies the CmpCallbackCount variable to either enable or disable registry kernel callbacks. The variable is used to count the number of registered callbacks. We suspect that if this value is patched to zero, the callbacks will be rendered useless.</p>
<h4>0x22210C (Disabled)</h4>
<p>When received, Poortry attempts to remove the fltMgr.sys driver from the \\FileSystem\\FastFat and \\FileSystem\\Ntfs device by use of the DeviceIoDetachDevice function. The function is usually used by valid drivers to clean up during shutdown. However, rootkits can use the function to prevent targeted drivers from receiving any further I/O requests.</p>
<p>fltMgr.sys is the filter manager on Windows. This driver is used to extend or modify the functionality of existing functionalities on the Windows system. The driver is also often used by EDR products.</p>
<p>We suspect by detaching it via use of IoDetachDevice installed filters are rendered useless on the targeted system.</p>
<p>&nbsp;</p>
<h4>0x2221C0 (Disabled)</h4>
<p>When received, Poortry enters routines to fetch the address of major functions handlers of ClassPnp.sys and ntfs.sys, such as NtfsFsdClose or NtfsFsdRead of ntfs.sys. Thus, we suspect that this routine can be used as an additional initialization routine to fetch critical function addresses that are used by other features.</p>
<h4>0x222400 (Enabled)</h4>
<p>When received, Poortry disables installed kernel callbacks through a series of different techniques. The user-mode component includes the name of the targeted driver when the I/O request packet is sent.</p>
<figure id="attachment_957040" aria-describedby="caption-attachment-957040" style="width: 640px" class="wp-caption alignnone"><a href="https://news.sophos.com/wp-content/uploads/2024/08/image9_001a1d.png"><img loading="lazy" decoding="async" class="size-full wp-image-957040" src="https://news.sophos.com/wp-content/uploads/2024/08/image9_001a1d.png" alt="Overview of patching routines and handshake check" width="640" height="255" srcset="https://news.sophos.com/wp-content/uploads/2024/08/image9_001a1d.png 1241w, https://news.sophos.com/wp-content/uploads/2024/08/image9_001a1d.png?resize=300,120 300w, https://news.sophos.com/wp-content/uploads/2024/08/image9_001a1d.png?resize=768,306 768w, https://news.sophos.com/wp-content/uploads/2024/08/image9_001a1d.png?resize=1024,408 1024w" sizes="(max-width: 640px) 100vw, 640px" /></a><figcaption id="caption-attachment-957040" class="wp-caption-text">Overview of patching routines and handshake check</figcaption></figure>
<p>Kernel callbacks installed via PsSetLoadImageNotifyRoutine, PsSetCreateThreadNotifyRoutine and PsSetCreateProcessNotifyRoutine are patched. At the prologue of the callback function, Poortry modifies the first instruction to instantly return zero when entered.</p>
<figure id="attachment_957041" aria-describedby="caption-attachment-957041" style="width: 640px" class="wp-caption alignnone"><a href="https://news.sophos.com/wp-content/uploads/2024/08/image10.png"><img loading="lazy" decoding="async" class="wp-image-957041 size-full" src="https://news.sophos.com/wp-content/uploads/2024/08/image10.png" alt="Comparison before and after prologue patching" width="640" height="364" srcset="https://news.sophos.com/wp-content/uploads/2024/08/image10.png 902w, https://news.sophos.com/wp-content/uploads/2024/08/image10.png?resize=300,171 300w, https://news.sophos.com/wp-content/uploads/2024/08/image10.png?resize=768,437 768w" sizes="(max-width: 640px) 100vw, 640px" /></a><figcaption id="caption-attachment-957041" class="wp-caption-text">Comparison before and after prologue patching</figcaption></figure>
<p>So far, we identified the following techniques to render kernel callbacks and security drivers useless:</p>
<ul>
<li>Internal structures used by the corresponding functions PsSetLoadImageNotifyRoutine, PsSetCreateThreadNotifyRoutine and PsSetCreateProcessNotifyRoutine are iterated. If the callback belongs to a tagged security driver, As a consequence, the registered callback function are exiting immediately without executing any of its intended operations.</li>
<li>The Windows kernel implements important data structures such as PsProcess, PsThread and ExDesktopObject that represent fundamental elements of the Windows operating system. These structure contain a variable named CallbackList that manages all callback routines associated with the specific object. Poortry iterates this list and if the callback belongs to a tagged security driver, As a consequence, the registered callback function are exiting immediately without executing any of its intended operations.</li>
<li>An internal linked listed used by CmRegisterCallback and CmUnregisterCallback is iterated. This linked list contains function points to registered registry and object callbacks. If the callback belongs to a tagged security driver, the prologue of the function is patched.</li>
<li>Poortry uses the exported function FltEnumerateFilters from fltMgr.sys to iterate through applied filters. If the filter belongs to a tagged security driver, the prologue of the function is patched.</li>
<li>While we were not able to directly trigger the functionality, we have found evidence that Poortry can abuse the <a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/wdm/nf-wdm-iodetachdevice">IoDetachDevice</a> function to detach a device object from a system’s device stack. In contrary to the functionality provided by IOCTL code 0x22210C, it is less evasive and detaches devices only if the device name matches the input name send via DeviceIoControl.</li>
</ul>
<h3>Cleanup Phase</h3>
<p>After impairment, the EDR killer aims at terminating security-related processes and rendering the EDR agent useless by wiping critical files off disk.</p>
<p>First, the user-mode component sends multiple I/O requests with IOCTL code 0x222144 to the kernel-mode component, including the process id of the process to kill.</p>
<p>The loader contains a list of hardcoded paths pointing at the location where EDR products are installed. It iterates all sub-folders and files in the folder and deletes files critical to the EDR agent, such as EXE files or DLL files by sending an IOCTL request with code 0x222180 to the driver. The sent request includes the path of the file to delete.</p>
<p>Notably, the user-mode component can operate in two modes:</p>
<ul>
<li>Deleting files by type</li>
<li>Deleting files by name</li>
</ul>
<p>We suspect that the author added these operation modes to ensure flexibility when aiming for different targets. We also believe that the list of hardcoded paths pointing at installation folders of EDR products change depending on the target.</p>
<figure id="attachment_957042" aria-describedby="caption-attachment-957042" style="width: 640px" class="wp-caption alignnone"><a href="https://news.sophos.com/wp-content/uploads/2024/08/image11.png"><img loading="lazy" decoding="async" class="wp-image-957042 size-full" src="https://news.sophos.com/wp-content/uploads/2024/08/image11.png" alt="Implementation of deleting files by type" width="640" height="396" srcset="https://news.sophos.com/wp-content/uploads/2024/08/image11.png 930w, https://news.sophos.com/wp-content/uploads/2024/08/image11.png?resize=300,185 300w, https://news.sophos.com/wp-content/uploads/2024/08/image11.png?resize=768,475 768w" sizes="(max-width: 640px) 100vw, 640px" /></a><figcaption id="caption-attachment-957042" class="wp-caption-text">Implementation of deleting files by type</figcaption></figure>
<h3>Conclusion</h3>
<p>Poortry, and its associated loader Stonestop, have undergone a serious feature enhancement in the 20 months since Sophos and Microsoft released a joint report on the EDR killer&#8217;s abuse of the WHQL signing mechanism. What was once a relatively simple tool for unhooking &#8220;troublesome&#8221; endpoint protection components has become, in and of itself, a Swiss Army Knife of malicious capabilities abusing a virtually limitless supply of stolen or improperly used code signing certificates in order to bypass Driver Signature Verification protections.</p>
<p>Poortry&#8217;s developers made it a differentiating characteristic of their tool that it could do more than just unhook an EDR or endpoint protection anti-tamper driver. Poortry has evolved into something akin to a rootkit that also has with finite controls over a number of different API calls used to control low-level operating system functionality. It also now has the capacity to wipe its enemies – security software – right off the disk as a way to clear the path for a ransomware deployment.</p>

		},

		submit() {
		this.loading = true;
		this.resetErrors();

		const mcList = "8d6471d831";
		const mcLanguage = decodeURIComponent( 'en' );
		const mcTags = [...this.subscribeTo, mcLanguage];

		const payload = {
			email: this.email,
			listId: mcList,
			tags: mcTags,
			language: mcLanguage,
		};

		fetch(this.config.endpoints.signup, {
			method: "POST",
			headers: {
			"Content-Type": "application/json",
			},
			body: JSON.stringify(payload),
		})
			.then((r) => r.json())
			.then((data) => {
			if (data.status === "pending") {
				this.setSuccess(
					decodeURIComponent( 'You%20have%20been%20subscribed.%20Please%20check%20your%20email%20to%20confirm.' )
				);
				this.subscribed = true;
			} else if (
				data.status === "subscribed"
			) {
				this.setSuccess(
					decodeURIComponent( 'You%20are%20already%20subscribed.%20Thank%20you%21' )
				);
			} else {
				this.setError(
					decodeURIComponent( 'We%20could%20not%20subscribe%20you.%20Please%20try%20again.' )
				);
			}
			})
			.catch((e) => {
			this.setError(
				decodeURIComponent( 'We%20could%20not%20subscribe%20you.%20Please%20try%20again.' )
			);
			})
			.finally(() => {
			this.reset();
			});
		},

		buttonText() {
		return this.loading ? decodeURIComponent( 'Please%20Wait' ) : decodeURIComponent( 'Subscribe' );
		},

		buttonDisabled() {
		return (
			this.subscribeTo.length < 1 ||
			!this.emailIsValid(this.email) ||
			this.loading
		);
		},
	};
	};
</script>
