        <h1 class="nist-page__title">Data Distribution in Privacy-Preserving Federated Learning</h1>

        
  </div>

<div

<a href="/blogs/cybersecurity-insights/authors/david-darais" hreflang="en">David Darais</a>, <a href="/blogs/cybersecurity-insights/authors/joseph-near" hreflang="en">Joseph Near</a>, <a href="/blogs/cybersecurity-insights/authors/dave-buckley" hreflang="en">Dave Buckley</a> and <a href="/blogs/cybersecurity-insights/authors/mark-durkee" hreflang="en">Mark Durkee</a>

<div
   class="nist-block"
>
  
    
      <div class="text-with-summary">
  
  
  
      <div class="nist-callout"><p><em><span lang="EN">This post is part of a series on privacy-preserving federated learning. The series is a collaboration between NIST and the UK government’s Responsible Technology Adoption Unit (RTA), previously known as the Centre for Data Ethics and Innovation. Learn more and read all the posts published to date at&nbsp;</span></em><a href="https://www.nist.gov/itl/applied-cybersecurity/privacy-engineering/collaboration-space/blog-series/privacy-preserving"><em><span lang="EN">NIST’s Privacy Engineering Collaboration Space</span></em></a><em><span lang="EN">&nbsp;or </span></em><a href="https://rtau.blog.gov.uk/tag/petsblogseries/"><em><span lang="EN">RTA’s blog</span></em></a><em><span lang="EN">.&nbsp;</span></em></p></div><p class="MsoNormal"><span lang="EN">Our </span><a href="https://www.nist.gov/blogs/cybersecurity-insights/uk-us-blog-series-privacy-preserving-federated-learning-introduction"><span lang="EN">first post</span></a><span lang="EN"> in the series introduced the concept of federated learning and described how it’s different from traditional centralized learning - in federated learning, the data is distributed among participating organizations, and share model updates (instead of raw data).</span></p><p class="MsoNormal"><span lang="EN"></span></p><p class="MsoNormal"><span lang="EN">What kinds of techniques can we use to build privacy-preserving federated learning systems? It turns out to depend heavily on </span><em><span lang="EN">how the data is distributed</span></em><span lang="EN">. This post defines and explains the different ways data can be distributed, or </span><em><span lang="EN">partitioned</span></em><span lang="EN">, among participants in federated learning systems. Future posts in the series will describe specific techniques applicable in each situation.</span></p><p class="MsoNormal"><span lang="EN"></span></p><p class="MsoNormal"><span lang="EN">Data partitioning schemes describe how data is distributed among participating organizations, as compared to the centralized scheme in which one party holds all the data.</span></p><ul><li><span lang="EN"></span><span lang="EN">In a </span><em><span lang="EN">horizontal partitioning</span></em><span lang="EN"> scheme, the </span><em><span lang="EN">rows</span></em><span lang="EN"> of the data are distributed among the participants.</span></li><li><span lang="EN"></span><span lang="EN">In a </span><em><span lang="EN">vertical partitioning&nbsp;</span></em><span lang="EN">scheme, the </span><em><span lang="EN">columns</span></em><span lang="EN"> of the data are distributed among the participants.&nbsp;</span></li></ul><p class="MsoNormal"><span lang="EN">Combinations of the two are also possible—we’ll get to those at the end of this post.</span></p><p><a class="ck-anchor" id="_pl4rcapeszwc"></a><span lang="EN"><strong>Horizontal Partitioning</strong></span></p><p class="MsoNormal"><span lang="EN">Consider our example scenario from the first post in this series: a consortium of banks wants to train a model to detect fraudulent transactions. This is an example of </span><em><span lang="EN">horizontal partitioning</span></em><span lang="EN">: each bank holds complete transaction data (i.e. containing all relevant columns) for its customers, and it’s the </span><em><span lang="EN">rows</span></em><span lang="EN"> of data that differ between the banks.</span></p> 
      <figure class=" nist-image--lightbox nist-image" role="group">
<div data-lightbox="https://www.nist.gov/sites/default/files/styles/2800_x_2800_limit/public/images/2024/02/22/Privacy%20Series1.png?itok=iZgaq8ov" data-media-id="701131">
      
    <img alt="Figure 1: Horizontally partitioned data. The partitions split up the rows of the data, but not the columns. Each partition contains distinct rows, but the same set of columns. Note that the data values shown are fictitious." height="129" loading="lazy" src="/sites/default/files/styles/480_x_480_limit/public/images/2024/02/22/Privacy%20Series1.png?itok=xuwsGBy2" typeof="foaf:Image" width="480">





                    
          </div>
<figcaption class="nist-image__caption"><strong>Figure 1: Horizontally partitioned data. The partitions split up the </strong><em><strong>rows</strong></em><strong> of the data, but not the </strong><em><strong>columns</strong></em><strong>. Each partition contains distinct rows, but the same set of columns. Note that the data values shown are fictitious.</strong><div class="nist-image__credit">
          <span class="nist-image__credit-label">Credit:</span>
          
  NIST

        </div></figcaption>
</figure>

  <p class="MsoNormal"><span lang="EN">The term </span><em><span lang="EN">horizontal partitioning</span></em><span lang="EN"> comes from the fact that the centralized version of the scenario can be transformed into the distributed version by drawing horizontal lines to indicate the different partitions, as shown in the example in Figure 1.</span></p><p class="MsoNormal"><span lang="EN"></span></p><p class="MsoNormal"><span lang="EN">In general, it’s easier to build privacy-preserving federated learning systems for horizontally partitioned data than it is to build similar systems for vertically partitioned data. This is true because when the data is partitioned horizontally, each partition can be viewed as a </span><em><span lang="EN">complete dataset</span></em><span lang="EN"> (i.e. there are no columns missing), which means </span><em><span lang="EN">each participant can train a model locally</span></em><span lang="EN"> without consulting other participants.</span></p><p class="MsoNormal"><span lang="EN"></span></p><p class="MsoNormal"><span lang="EN">Some types of models can be composed after they are trained, which leads directly to a simple but elegant approach for federated learning for horizontally partitioned data: first, each participant trains a model locally on their own data; then, the trained models are composed to form a more effective final model. We’ll discuss approaches that follow this structure in the next post in the series.</span></p><p class="MsoNormal"><span lang="EN"></span></p><p class="MsoNormal"><span lang="EN">The </span><a href="https://www.drivendata.org/competitions/103/nist-federated-learning-2-pandemic-forecasting-federated/page/586/"><span lang="EN">public health track</span></a><span lang="EN"> of the US-UK pets prize challenges was an example of horizontally partitioned data. In this track, data about individuals in a synthetically-generated population were distributed across a number of health districts. Each district held information about each individual such as their demographic attributes, social contacts, and infection status. Privacy-preserving federated learning across these horizontal partitions was then used to train models to predict an individual’s future risk of infection.</span></p><p><a class="ck-anchor" id="_cx2ijx6cfnhx"></a><span lang="EN"><strong>Vertical Partitioning</strong></span></p><p class="MsoNormal"><span lang="EN">Consider an alternative scenario involving a single bank (still holding customers’ transaction data) and a credit reporting agency that holds credit ratings. The two organizations might wish to train a model that leverages </span><em><span lang="EN">both</span></em><span lang="EN"> the transaction data </span><em><span lang="EN">and</span></em><span lang="EN"> the credit score </span><em><span lang="EN">of a single customer</span></em><span lang="EN">. This is an example of </span><em><span lang="EN">vertical partitioning</span></em><span lang="EN">: the two organizations hold different kinds of data about the </span><em><span lang="EN">same individuals</span></em><span lang="EN">; in this case, it’s the </span><em><span lang="EN">columns</span></em><span lang="EN"> of data that differ between the participants.</span></p> 
      <figure class=" nist-image--lightbox nist-image" role="group">
<div data-lightbox="https://www.nist.gov/sites/default/files/styles/2800_x_2800_limit/public/images/2024/02/22/Privacy%20Series2.png?itok=CKcrM16U" data-media-id="701136">
      
    <img alt="Figure 2: Vertically partitioned data. The partitions split up the columns of the data, but not the rows. Each partition contains the same set of rows, but a distinct set of columns. Note that the data values shown are fictitious." height="182" loading="lazy" src="/sites/default/files/styles/480_x_480_limit/public/images/2024/02/22/Privacy%20Series2.png?itok=G04q3OhP" typeof="foaf:Image" width="480">





                    
          </div>
<figcaption class="nist-image__caption"><strong>Figure 2: Vertically partitioned data. The partitions split up the </strong><em><strong>columns</strong></em><strong> of the data, but not the </strong><em><strong>rows</strong></em><strong>. Each partition contains the same set of rows, but a distinct set of columns. Note that the data values shown are fictitious.</strong><div class="nist-image__credit">
          <span class="nist-image__credit-label">Credit:</span>
          
  NIST

        </div></figcaption>
</figure>

  <p class="MsoNormal"><span lang="EN">This time, transforming the centralized version of the scenario into the distributed version involves drawing </span><em><span lang="EN">vertical</span></em><span lang="EN"> lines to indicate the different partitions, as shown in the example in Figure 2.</span></p><p class="MsoNormal"><span lang="EN"></span></p><p class="MsoNormal"><span lang="EN">Privacy-preserving federated learning systems for vertically partitioned data are especially challenging, primarily due to the need to </span><em><span lang="EN">link together</span></em><span lang="EN"> data points from different partitions about the same individual or entity during training. In contrast to horizontally partitioned federated learning, it’s generally not possible to train separate models on different </span><em><span lang="EN">columns</span></em><span lang="EN"> of the data (but overlapping </span><em><span lang="EN">rows</span></em><span lang="EN">) and then compose them afterwards.</span></p><p class="MsoNormal"><span lang="EN"></span></p><p class="MsoNormal"><span lang="EN">As a result, systems for privacy-preserving federated learning with vertically partitioned data are generally more complex and challenging to build. We’ll discuss techniques for building such systems later in the series.</span><br><br><a class="ck-anchor" id="_abbqjhpb6q9k"></a><span lang="EN"><strong>Combining Vertical and Horizontal Partitioning</strong></span></p><p class="MsoNormal"><span lang="EN">In practice, federated learning will often involve training data distributed across a combination of vertical and horizontal partitions. This was a feature of the </span><a href="https://www.drivendata.org/competitions/105/nist-federated-learning-2-financial-crime-federated/page/589/"><span lang="EN">financial crime track</span></a><span lang="EN"> of the US-UK PETs prize challenges.</span></p><p class="MsoNormal"><span lang="EN"></span></p><p class="MsoNormal"><span lang="EN">The aim of this track was to train an anomaly detection model using a synthetic dataset representing transaction data from a global payment network, enriched with synthetic account metadata (e.g. credit scores) held by banks. The account metadata was horizontally partitioned, with each partner bank storing the same metadata fields (i.e. the same columns) for each of its registered accounts.</span></p> 
      <figure class=" nist-image--lightbox nist-image" role="group">
<div data-lightbox="https://www.nist.gov/sites/default/files/styles/2800_x_2800_limit/public/images/2024/02/22/Picture3.png?itok=ueaoppG-" data-media-id="701141">
      
    <img alt="Figure 3: Combined partitioning. The account metadata held by banks (the credit score for each account) is partitioned horizontally, and the transaction data held by the payment network is partitioned vertically. Note that data values shown are fictitious." height="110" loading="lazy" src="/sites/default/files/styles/480_x_480_limit/public/images/2024/02/22/Picture3.png?itok=J-A-SW1J" typeof="foaf:Image" width="480">





                    
          </div>
<figcaption class="nist-image__caption"><strong>Figure 3: Combined partitioning. The account metadata held by banks (the credit score for each account) is partitioned horizontally, and the transaction data held by the payment network is partitioned vertically. Note that data values shown are fictitious.</strong><div class="nist-image__credit">
          <span class="nist-image__credit-label">Credit:</span>
          
  NIST

        </div></figcaption>
</figure>

  <p class="MsoNormal"><span lang="EN">Each transaction held by the payment network can only be enriched with account metadata by linking on an appropriate Account ID. Banks do not have access to the transaction data held by the payment network, meaning that the data linking needs to be done in a secure and private way. Moreover, the horizontal partitioning of the account metadata means that an efficient method is needed for determining which partner bank is relevant for linking for a particular account.</span></p><p class="MsoNormal"><span lang="EN"></span></p><p class="MsoNormal"><span lang="EN">This example shows that scenarios involving both horizontal and vertical partitioning bring additional complexities. In this setting, training a model that can detect fraud with high accuracy – while also ensuring privacy – is especially challenging. Future posts in this series will describe these challenges and explore some of the techniques used to solve them in the US-UK PETs prize challenges.</span></p><p><span lang="EN"><strong>Coming Up Next</strong></span></p><p class="MsoNormal"><span lang="EN">In our next post, we’ll begin to explore practical approaches for protecting privacy in the different partitioning scenarios described above, starting with a deep dive into input privacy approaches for horizontally partitioned data.</span><span lang="EN"></span><span lang="EN"></span></p>
