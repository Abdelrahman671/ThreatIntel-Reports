<h1><span id="hs_cos_wrapper_name" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="text">A new playground: Malicious campaigns proliferate from VSCode to npm</span></h1>
<div id="hs_cos_wrapper_module_1577203992502740" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module">
<h2 class="post-header__secondary-head blog__post-secondary-h2">To avoid compromised packages being introduced as a dependency in a larger project, security teams need to keep an eye peeled for such malicious code.</h2>
</div>
</div>
<div class="blog-author__short">
<div class="blog-author__short-info">

<div class="blog__post-content html-content">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Blog.webp?width=1400&amp;height=732&amp;name=VSCode-Blog.webp" width="1400" height="732" loading="lazy" alt="A new playground: Malicious campaigns proliferate from VSCode to npm" style="height: auto; max-width: 100%; width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Blog.webp?width=700&amp;height=366&amp;name=VSCode-Blog.webp 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Blog.webp?width=1400&amp;height=732&amp;name=VSCode-Blog.webp 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Blog.webp?width=2100&amp;height=1098&amp;name=VSCode-Blog.webp 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Blog.webp?width=2800&amp;height=1464&amp;name=VSCode-Blog.webp 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Blog.webp?width=3500&amp;height=1830&amp;name=VSCode-Blog.webp 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Blog.webp?width=4200&amp;height=2196&amp;name=VSCode-Blog.webp 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></p>
<p>ReversingLabs researchers have been monitoring multiple public repositories over the past few years. Recently, our team has expanded its threat hunting efforts to VSCode Marketplace — and the researchers started to see an increasing amount of malicious activity.&nbsp;<br><br>In the past, RL researchers have <a href="/blog/vs-code-ide-hack-how-supply-chain-attacks-can-proliferate-between-developer-ecosystems" rel="noopener">observed </a>how easy and quickly it is for supply chain attacks to proliferate from the npm community to VSCode Marketplace. Using npm packages, threat actors can get malicious code into VSCode IDE as well, which is often overlooked as a potential source of compromise.<br><br>In November 2024, this proliferation changed direction. A month later, a campaign that started on VSCode emerged in the npm community with malicious npm package <a href="https://secure.software/npm/packages/etherscancontracthandler/4.0.0" rel="noopener" target="_blank"><span style="font-style: italic;">etherscancontracthandler</span></a>, bearing a striking resemblance to previously seen malicious VSCode extensions. &nbsp;</p>
<!--more-->
<h2 style="font-weight: normal; font-size: 24px;">Packages on npm vs. VSCode extensions</h2>
<p>VSCode extensions, used to add new features to VSCode IDE, are developed in Node.js and are essentially npm packages with a wrapper around them to interconnect with VSCode APIs. While developing extensions, developers have all npm packages at their disposal and can include different npm packages as dependencies. This way, npm packages are a vector of compromise, since they can be included in VSCode extensions like legitimate functions.&nbsp;<br><br>As referenced in the <a href="/blog/vs-code-ide-hack-how-supply-chain-attacks-can-proliferate-between-developer-ecosystems" rel="noopener">RL research team blog post about a similar subject</a>, VSCode extensions aren’t usually perceived as a potential source of compromise since “they are only plugins for IDE, nothing is being added to the code base.”<br><br>The campaign the RL research team found on npm highlights the insistence of malicious actors on polluting the development cycle of the community. While all malicious VSCode extensions were quickly removed from VSCode Marketplace, a malicious actor turned to npm to publish malicious npm extensions. The npm packages potentially have a bigger attack surface, since they are not only used while developing Node.js applications, but can also be included in other npm packages and VSCode extensions as well. They can, in turn, be installed in the VSCode IDE, which compromises local development environments.</p>
<h2 style="font-weight: normal; font-size: 24px;">New wave of malicious extensions hits VSCode</h2>
<p>Throughout October 2024, the RL research team saw a new wave of malicious VSCode extensions containing &nbsp;downloader functionality — all part of the same campaign. The community was first notified of this campaign taking place in <a href="https://x.com/LehmannLorenz/status/1841545179825942991" rel="noopener" target="_blank">early October</a>, and since then, the team has been steadfast in tracking it.&nbsp;<br><br>The published tally of packages that are part of this campaign stands currently at 18.<br><br>The campaign started with targeting of the crypto community, but by the end of October, extensions published were mostly <a href="https://x.com/ReversingLabs/status/1854168942991888633" rel="noopener" target="_blank">impersonating the Zoom application</a>. And each malicious &nbsp;extension published was more sophisticated than the last. Threat actors started investing more time and effort into making those extensions appear credible. Extensions <a href="https://x.com/ReversingLabs/status/1856349703991881800" rel="noopener" target="_blank" style="font-style: italic;">EVM.Blockchain-Toolkit and VoiceMod.VoiceMod</a> had artificially inflated install counts that didn’t really correspond to the download count of the same extensions. Moreover, those extensions had reviews that were clearly fabricated by their authors.<br><br></p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/VSCode-Figure-1.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Figure-1.png?width=1400&amp;name=VSCode-Figure-1.png" alt="VoiceMod.VoiceMod’s inflated number of installs and fabricated reviews" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Figure-1.png?width=700&amp;name=VSCode-Figure-1.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Figure-1.png?width=1400&amp;name=VSCode-Figure-1.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Figure-1.png?width=2100&amp;name=VSCode-Figure-1.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Figure-1.png?width=2800&amp;name=VSCode-Figure-1.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Figure-1.png?width=3500&amp;name=VSCode-Figure-1.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/VSCode-Figure-1.png?width=4200&amp;name=VSCode-Figure-1.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>
<p style="font-weight: bold; font-size: 16px; text-align: center;">Figure 1: VoiceMod.VoiceMod’s inflated number of installs and fabricated reviews</p>
<p>All of the extensions were very similar and had the same functionality. The malicious file found in them was obfuscated with Javascript Obfuscator, which was the reason those extensions raised some red flags and were investigated as a result.&nbsp;<br><br>Functionally, the extension's code was very simple. All of the extensions were downloading the second stage payload from multiple domains, out of which some were repeated more than others.</p>
<p style="text-align: center;"><img src="https://www.reversinglabs.com/hubfs/Blog/VSCode-Figure-2.svg" width="600" height="443" loading="lazy" alt="VSCode-Figure-2" style="height: auto; max-width: 100%; width: 600px; margin-left: auto; margin-right: auto; display: block;"><span style="font-weight: bold; font-size: 16px;">Figure 2: different endpoints serving second stage</span></p>
<p>Most of the extensions were marketed as a “Solidity Language support for Visual Studio Code,” and some of the domains, such as <em>hxxps[:]//microsoft-visualstudiocode[.]com</em> or even <em>hxxps[:]//captchacdn[.]com</em>, reflected it. Malicious actors probably manufactured a few of them to try and trick users into believing they are legitimate endpoints for downloading legitimate software. Some domains were interesting because of their TLD (top level domain) — <em>.lat</em> and <em>.ru</em>, which are TLDs for Latin American communities and users and Russia, respectively.</p>
<h2 style="font-weight: normal; font-size: 24px;">Malicious packages: The sequel</h2>
<p>The original package that appeared on npm was similar to the ones the RL team has already seen on VSCode Marketplace. It was targeting the crypto community as well, with striking similarities, to the point where it could be concluded that they were published by the same threat actor, or a group of threat actors.<br><br>Currently, there are five versions of the package <em>etherscancontracthandler</em> that were published to npm, and it was the only package connected to this campaign to be published. Only three versions out of five had an obfuscated malicious payload that was obfuscated by Javascript as well. All versions were downloaders, downloading from the same endpoints VSCode extensions were downloading at second-stage. Domains were the same, but the string that would download the second stage was the same in VSCode extensions and the npm package.<br><br>Another very similar feature was the structure of the code between malicious payloads that were obfuscated. The only difference was that one was made to be a VSCode extension, and the other as an npm package.</p>

<p>Once we detected the npm package, we reported it to the npm maintainers, and they took it down promptly. The package was downloaded only around 350 times, so fortunately the reach of this package wasn’t that big.</p>
<h2 style="font-weight: normal; font-size: 24px;">Combating IDE compromises</h2>
<p>The possibility of installing plugins and extending functionality of IDEs makes them very attractive targets for malicious actors. VSCode extensions are often overlooked as a security risk when installing in IDE, but the compromise of an IDE can be a landing point for further compromise of the development cycle in the enterprise.<br><br>As the <a href="https://media.defense.gov/2022/Sep/01/2003068942/-1/-1/0/ESF_SECURING_THE_SOFTWARE_SUPPLY_CHAIN_DEVELOPERS.PDF" rel="noopener" target="_blank">Enduring Security Framework working group’s guide, “Securing the Software Supply Chan”</a> states:</p>
<blockquote>
<p><em>“[To] ensure the integrity of the development process, all IDEs and their associated plug-ins used within developer environments must be preapproved, validated and scanned for vulnerabilities before being incorporated onto any developer machine.”</em></p>
</blockquote>
<p>It is important to perform security assessments regularly. New versions of existing dependencies or products can introduce new vulnerabilities, risks and other issues that can be caught with regular checks. Security assessments can also be beneficial in catching hijacked and infected libraries before they are included in production, or to make sure build pipelines weren’t compromised.</p>
<h2 style="font-weight: normal; font-size: 24px;">Conclusion</h2>
<p>In March 2023, RL researchers discussed propagation of <a href="/blog/vs-code-ide-hack-how-supply-chain-attacks-can-proliferate-between-developer-ecosystems" rel="noopener">malicious dependency published on npm through a compromised npm module into the VSCode ecosystem</a>. It was explained how easily that can be done, and how, in turn, an IDE can then be compromised as well. This time around, RL researchers identified how the repositories traded places. The malicious VSCode campaign was expanded to npm, potentially expanding the attack surface, since npm packages can be included in VSCode Extensions, other npm packages and Node.js applications.<br><br>When using packages from public repositories, developers should keep an eye peeled for possible inclusion of a malicious code to avoid a malicious package being introduced as a dependency in some larger project. The RL <a href="https://secure.software/?_gl=1*oxtjmz*_gcl_au*MTg5MTE1MDcxMC4xNzMyNTQ1NzU2" rel="noopener" target="_blank">Spectra Assure Community </a>makes that process much easier.<br><br>Development organizations should also scrutinize the features and behaviors of the open source, third-party and commercial code they are relying on, to track dependencies and detect potential malicious payloads in them. RL’s <a href="/products/software-supply-chain-security" rel="noopener">Spectra Assure &nbsp;platform </a>delivers the deep visibility teams need to discover malware.</p>
