    <p>During the past two months, the Guardicore Labs team has been closely following a China-based campaign which aimed to infect Windows MS-SQL and phpMyAdmin servers worldwide. We have taken a deep look into the inner workings of the campaign – the tools in use, the vulnerabilities exploited and the extent of damage caused.</p>
<p>Breached machines include over 50,000 servers belonging to companies in the healthcare, telecommunications, media and IT sectors. Once compromised, the targeted servers were infected with malicious payloads. These, in turn, dropped a crypto-miner and installed a sophisticated kernel-mode rootkit to prevent the malware from being terminated.</p>
<p>Guardicore Labs witnessed the release and deployment of 20 different payload versions throughout the campaign. We contacted the hosting provider of the attack servers as well as the issuer of the rootkit certificate. As a result, the attack servers were taken down and the certificate was revoked.</p>
<p>The <i>Nansh0u</i> campaign is not a typical crypto-miner attack. It uses techniques often seen in APTs such as fake certificates and privilege escalation exploits. While advanced attack tools have normally been the property of highly skilled adversaries, this campaign shows that these tools can now easily fall into the hands of less than top-notch attackers.</p>
<p>In this post, we describe the attacks in detail and provide a complete <a href="https://github.com/guardicore/labs_campaigns/tree/master/Nansh0u" target="_blank" rel="nofollow noreferrer">IoC repository</a> for the campaign, including a <a href="https://github.com/guardicore/labs_campaigns/blob/master/Nansh0u/detect_nansh0u.ps1" target="_blank" rel="nofollow noreferrer">script</a> to detect infected machines.</p>
<h4>From Unusual Incidents to Full Investigation</h4>
<p>In the beginning of April, three attacks detected in the Guardicore Global Sensor Network (GGSN) caught our attention. All three had source IP addresses originating in South-Africa and hosted by <i>VolumeDrive</i> ISP (see IoCs). The incidents shared the same attack process, focusing on the same service and using the same breach method and post-compromise steps.</p>
<p>Looking for more attacks with a similar pattern, we found attacks dating back to February 26, with over seven hundred new victims per day. During our investigation, we found 20 versions of malicious payloads, with new payloads created at least once a week and used immediately after their creation time.</p>

<h4>Privilege Escalation Exploits</h4>
<p><i>apexp.exe</i> and <i>apexp2012.exe</i> are two exploits of a known privilege escalation vulnerability (CVE-2014-4113). Passing any program to these executables will run it with <i>SYSTEM</i> privileges.</p>
<p><i>apexp.exe</i> is known as <i>Apolmy</i> exploit and it affects both Desktop and Server versions of Windows (XP to 8.1 and 2003 to 2012 R2, respectively). This is a weaponized exploit with production-level code. <i>apexp2012.exe</i>, on the other hand, resembles more of a proof-of-concept than an operational exploit, and is designed to work on Windows 8.1. We found the latter available for download on a Chinese forum seemingly used as a hacker-community platform.</p>
<p>While both versions use the same vulnerability, they execute kernel-mode code for different purposes. The <i>Apolmy</i> version copies the SYSTEM process access token to its own process. With that token, the exploiting process runs the payload with full control over the victim machine.</p>
<p>The second version uses a method popularized by Cesar Cerrudo. Here, the exploit adds the <a href="https://docs.microsoft.com/en-us/windows/desktop/secauthz/privilege-constants" target="_blank" rel="nofollow noreferrer">SeDebugPrivilege</a> to the token. Using this Windows privilege, the attacking exploit injects code into the <i>winlogon</i> process. The injected code creates a new process which inherits <i>winlogon</i>’s <i>SYSTEM</i> privileges, providing equivalent permissions as the prior version.</p>
<h4>Payloads: Droppers of Rootkits and Miners</h4>
<p>We collected 20 payload samples from both the attacker’s servers and Guardicore Global Sensor Network (GGSN). Each payload is in fact a wrapper and has several functionalities:</p>
<ul>
<li>Execute the crypto-currency miner;</li>
<li>Create persistency by writing registry run-keys;</li>
<li>Protect the miner process from termination using a kernel-mode rootkit;</li>
<li>Ensure the miner’s continuous execution using a watchdog mechanism.</li>
</ul>
<p>The payloads spawn one of two processes – <i>dllhot.exe</i> or <i>canlang.exe</i>, depending on the payload – which mine a privacy crypto-currency named <i>TurtleCoin</i> for four different mining pools.</p>

<p>Why are signed drivers exceptional?</p>
<p>Kernel-mode drivers are executable files run within the operating system’s kernel. As such, they have high-privileged access to sensitive data structures and resources.</p>
<p>Since Windows 10 and Windows Server 2016, Microsoft only allows Microsoft-signed drivers to run in kernel mode. To receive such a signature, developers must provide Microsoft a version of their driver and pass numerous tests.</p>
<p>The chances of a malicious driver being signed are very low.</p>
<p>This would have been less awkward, had the driver not been packed and obfuscated. Unlike many other malicious drivers, this driver is protected and obfuscated with VMProtect, a software tool that attempts to frustrate reverse engineers and malware researchers.</p>
<p>The driver is designed to protect processes and prevent the user from terminating them. It creates a device named <i>SA6482</i>, allowing processes to communicate with it. The device receives process IDs (PIDs) meant to be protected – in our case, the miner PID.</p>
<p>The driver protects the process by registering a callback on the Process and Thread object types. These callbacks are triggered with every access to the protected process or any of its threads and allows drivers to modify the access rights given in each access attempt.</p>

<h4>You Don’t Have to Be a Nation-State to Own Top-Level Weapons</h4>
<p>The infection process uses advanced technological capabilities. <i>apexp.exe</i>, for instance, takes advantage of a kernel-mode vulnerability to execute code with <i>SYSTEM</i> privileges. It was used in a <a href="https://www.crowdstrike.com/blog/crowdstrike-discovers-use-64-bit-zero-day-privilege-escalation-exploit-cve-2014-4113-hurricane-panda/" target="_blank" rel="nofollow noreferrer">past Chinese APT</a> by a highly-skilled adversary to target various companies.</p>
<p>Another example is the driver dropped by the different payloads. Obtaining a signed certificate for a packed driver is not at all trivial and requires serious planning and execution. In addition, the driver supports practically every version of Windows from Windows 7 to Windows 10, including beta versions. This exhaustive coverage is not the work of a hacker writing a rootkit for fun.</p>
<p>These advanced elements are opposed to several odd SecOps decisions taken by the attacker. To begin with, attackers usually do not keep their whole infrastructure on a file server with no activated authentication controls. Logs, victims lists, usernames, binary files – we had them all in a mouse click. In addition, all binary files had their original timestamps; an experienced malware author would have tampered with those to complicate the analysis process.</p>
<p>This campaign was clearly engineered from the phase of IPs scan until the infection of victim machines and mining the crypto-coin. However, various typos and mistakes imply that this was not a thoroughly-tested operation. For example, we found a mismatch in two versions of the <i>lcn.exe</i> payload. Both were running the same miner but with swapped command-line arguments, suggesting that the first one was providing the wallet address in an incorrect position.</p>

<h4>Detection &amp; Prevention</h4>
<p>What enables this attack on Windows MS-SQL servers in the first place, is having weak usernames and passwords for authentication. As trivial as it may sound, having strong credentials is the difference between an infected and a clean machine.</p>
<p>If you expose database servers to the internet, you should minimize the danger of any possible compromise. Database vendors typically provide hardening guides; Microsoft’s <a href="https://docs.microsoft.com/en-us/sql/relational-databases/security/securing-sql-server?view=sql-server-2017" target="_blank" rel="nofollow noreferrer">guide</a> for SQL server is pretty easy to implement. In addition, you should separate internet-exposed servers from internal servers as much as possible by segmenting your network and limiting the blast radius of vulnerable devices.</p>
<p>We have released an open source PowerShell <a href="https://github.com/guardicore/labs_campaigns/blob/master/Nansh0u/detect_nansh0u.ps1" target="_blank" rel="nofollow noreferrer">script</a> to provide defenders with a simple way to detect infected machines. For more details and instructions check our <a href="https://github.com/guardicore/labs_campaigns/tree/master/Nansh0u" target="_blank" rel="nofollow noreferrer">repository</a>.</p>
<h4>Conclusion</h4>
<p>The <i>Nansh0u</i> campaign is not another run-of-the-mill mining attack. It serves as a case study for a thoroughly-planned attack scheme which incorporates various modules and progressive tools. This suggests that cyber weapons are becoming readily available, and not only to nation-state organizations.</p>
<p>This campaign demonstrates once again that common passwords still comprise the weakest link in today’s attack flows. Seeing tens of thousands of servers compromised by a simple <a href="/glossary/what-is-brute-force-attack" target="_self">brute force attack</a>, we highly recommend that organizations protect their assets with strong credentials as well as network segmentation solutions.<br />
