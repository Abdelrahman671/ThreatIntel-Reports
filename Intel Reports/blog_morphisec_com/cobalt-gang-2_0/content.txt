<p>Over the past year, Morphisec and several other <a href="https://www.morphisec.com/products/advanced-endpoint-threat-protection" rel="noopener" target="_blank">endpoint protection</a> companies have been tracking a resurgence in activity from the <span style="font-weight: bold;">Cobalt Group</span>. Cobalt is one of the most notorious cybercrime operations, with attacks against more than 100 banks across 40 countries <!--more--><span style="background-color: transparent;">attributed to the group. The most recent attacks can be grouped into two types of campaigns. Many of the campaigns are based on the known and prevalent <span style="font-style: normal;">ThreadKit</span> exploit kit generation framework. Other campaigns are more sophisticated, borrowing only some functionality from <span style="font-style: italic;">ThreadKit builder</span> while incorporating additional advanced techniques from other sources.</span></p>
<p>Morphisec Labs believes that the Cobalt Group split following the arrest of one of its top leaders in Spain in March of 2018. While Cobalt Gang 1.0 uses ThreadKit extensively, Cobalt 2.0 adds sophistication to its delivery method, borrowing some of the network infrastructures used by both APT28 &nbsp;(aka Fancy Bear) and MuddyWater.</p>
<p>One of the Cobalt 2.0 Group’s latest campaigns, an attack that leads to a Cobalt Strike beacon and to JavaScript backdoor, was investigated and <a href="https://blog.talosintelligence.com/2018/07/multiple-cobalt-personality-disorder.html">presented</a> by the Talos research team. Morphisec has investigated different samples from the same campaign. The following analysis presents our findings, focusing on the additional sophistication patterns and attribution patterns.</p>
<h2><span style="font-weight: bold;">Cobalt Group</span> Technical Details</h2>
<h3>Stage 1 - Word Macro + Whitelisting Bypass</h3>
<p>As with many other campaigns, the victim received a document with malicious macro visual basic code.</p>
<p><img src="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-1.png?width=550&amp;name=CGsplit-1.png" width="550" style="width: 550px; margin-top: 0px; margin-bottom: 0px;" alt="ThreadKit Builder" srcset="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-1.png?width=275&amp;name=CGsplit-1.png 275w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-1.png?width=550&amp;name=CGsplit-1.png 550w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-1.png?width=825&amp;name=CGsplit-1.png 825w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-1.png?width=1100&amp;name=CGsplit-1.png 1100w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-1.png?width=1375&amp;name=CGsplit-1.png 1375w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-1.png?width=1650&amp;name=CGsplit-1.png 1650w" sizes="(max-width: 550px) 100vw, 550px"></p>
<p>Although the code is heavily obfuscated, the entry point is easily identifiable. The VB code is executed starting from the <strong><em>Frame1_Layout </em></strong>function – this method is used much less frequently than the obvious Document_Open or the AutoOpen.</p>
<p><img src="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-2.png?width=600&amp;name=CGsplit-2.png" alt="CGsplit-2" width="600" style="width: 600px; margin-top: 0px; margin-bottom: 0px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-2.png?width=300&amp;name=CGsplit-2.png 300w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-2.png?width=600&amp;name=CGsplit-2.png 600w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-2.png?width=900&amp;name=CGsplit-2.png 900w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-2.png?width=1200&amp;name=CGsplit-2.png 1200w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-2.png?width=1500&amp;name=CGsplit-2.png 1500w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-2.png?width=1800&amp;name=CGsplit-2.png 1800w" sizes="(max-width: 600px) 100vw, 600px"></p>
<p>The list of additional possible execution triggers is defined here: <a href="https://www.greyhathacker.net/?p=948">https://www.greyhathacker.net/?p=948</a></p>
<p>The macro is executing the legitimate Windows process <strong><em>cmstp.exe</em></strong> (connection manager Profile Installer). This technique was previously used by the <strong>MuddyWater</strong> group when attacking Middle East targets. The use of cmstp.exe whitelisting bypass was researched by <a href="https://oddvar.moe/2017/08/15/research-on-cmstp-exe/">Oddvar Moe</a>, where he showed how, by manipulating the inf file, cmstp can execute scriptlets or executables.</p>
<p><img src="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-3.png?width=600&amp;name=CGsplit-3.png" alt="CGsplit-3" width="600" style="width: 600px; margin-top: 0px; margin-bottom: 0px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-3.png?width=300&amp;name=CGsplit-3.png 300w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-3.png?width=600&amp;name=CGsplit-3.png 600w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-3.png?width=900&amp;name=CGsplit-3.png 900w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-3.png?width=1200&amp;name=CGsplit-3.png 1200w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-3.png?width=1500&amp;name=CGsplit-3.png 1500w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-3.png?width=1800&amp;name=CGsplit-3.png 1800w" sizes="(max-width: 600px) 100vw, 600px"></p>
<p>In our case the attacker abused cmstp to execute JavaScript scriptlet (XML with JS) that is downloaded from the e-dropbox[.]biz site. This way the group limited the exposure and the delivery of the JavaScript to relevant targets only.</p>

<p>As can be seen from the deobfuscated code, the JavaScript yet again bypasses whitelisting by manipulation of regsvr32.exe, another legitimate Windows process. The two dropped artifacts – a payload DLL and a Word document – are written to the “Users\&lt;Log on User&gt;\” folder (the document will replace the opened malicious document with clean stub after killing the running Word process).</p>
<h3>Stage 3 - PureBasic Legitimate Executable Mixed with Additional Malicious Functions</h3>
<p>The dropped DLL is actually a PureBasic compiled code and a legitimate application. The application is not signed (as many other PureBasic applications) and therefore easily manipulated to execute inserted malicious code. In this case, the exported function DllRegisterServer wasn’t part of the legitimate application and is perfect for application flow redirection when executed by regsvr32.exe. Because PureBasic is a full programming language that compiles to assembly and has endless possibilities and APIs to manipulate the memory, it also complicates the generation of patterns by security vendors that base their detection on static or dynamic pattern signatures. Although some security solutions will block all PureBasic programs (wrong move – there are plenty of legitimate PureBasic programs in use today), it’s a smart move made by the attacker group.</p>
<p><img src="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-6.png?width=600&amp;name=CGsplit-6.png" alt="CGsplit-6" width="600" style="width: 600px; margin-top: 0px; margin-bottom: 0px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-6.png?width=300&amp;name=CGsplit-6.png 300w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-6.png?width=600&amp;name=CGsplit-6.png 600w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-6.png?width=900&amp;name=CGsplit-6.png 900w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-6.png?width=1200&amp;name=CGsplit-6.png 1200w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-6.png?width=1500&amp;name=CGsplit-6.png 1500w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-6.png?width=1800&amp;name=CGsplit-6.png 1800w" sizes="(max-width: 600px) 100vw, 600px"></p>
<p>To function properly, the malicious injected code needs to reflectively load and map to existing core functions. The same code also applies anti-disassembly and anti-debugging techniques. It gets the following functions from Kernel32 and Advapi32:</p>
<p><img src="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-7.png?width=650&amp;name=CGsplit-7.png" alt="CGsplit-7" width="650" style="width: 650px; margin: 0px 10px 10px 0px;" srcset="https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-7.png?width=325&amp;name=CGsplit-7.png 325w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-7.png?width=650&amp;name=CGsplit-7.png 650w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-7.png?width=975&amp;name=CGsplit-7.png 975w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-7.png?width=1300&amp;name=CGsplit-7.png 1300w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-7.png?width=1625&amp;name=CGsplit-7.png 1625w, https://blog.morphisec.com/hs-fs/hubfs/Blog_images/18-Q4/CGsplit-7.png?width=1950&amp;name=CGsplit-7.png 1950w" sizes="(max-width: 650px) 100vw, 650px"></p>
<p>The code then uses the identified functions to add persistency through registry and add next stages file names identifier through the following locations:</p>
<ul>
<li>HKCU\Environment\UserInitMprLogonScript – next stage command (JavaScript downloader executed through regsvr32) is registered under UserInitMprLogonScript.</li>
</ul>
<ul>
<li>HKCU\Software\Microsoft\Notepad\<em>&lt;$UserName$&gt; </em>- The code creates a new value under Notepad Key with a pair representing randomly generated key pair. The right side of the pair is the name of the JavaScript in the next stage (stage 4) , while the left side of the pair represents the file that will be downloaded as part of stage 5.</li>
</ul>

<p>Such a combination of registry manipulation was reported a year ago as part of an attack <a href="https://cys-centrum.com/ru/news/activity_of_cobalt_summer_2017">campaign</a> executed by the <span style="font-style: italic;">Cobalt Group</span> against Ukrainian banks.</p>
<p>As part of the last execution step of the dll, the malicious code writes a JavaScript scriptlet into the Roaming directory and then it executes CreateProcess on the regsvr32 as described by the UserInitMprLogonScript.</p>

<h3>Stage 5 - JavaScript Backdoor</h3>
<p>The last stage JavaScript is downloaded from hxxps://server.vestacp[.]kz/robots.txt.</p>
<p>The JavaScript is obfuscated the same way as in the previous stages. After deobfuscation, we encounter a backdoor that was <a href="https://blog.trendmicro.com/trendlabs-security-intelligence/backdoor-carrying-emails-set-sights-on-russian-speaking-businesses/">used in attacks against Russian speaking business</a><a href="https://blog.trendmicro.com/trendlabs-security-intelligence/backdoor-carrying-emails-set-sights-on-russian-speaking-businesses/">es</a> in August 2017. This backdoor protocol of commands here is almost identical to the previously described backdoor, aside from some name changes:</p>
<ul>
<li>"d&amp;exec" – Download an executable or a dll (if it’s a dll, use regsvr32 to execute it)</li>
<li>"more_eggs" – Downloads and replace the existing backdoor script with new script</li>
<li>"gtfo" – Clean traces, remove persistency and stage 4,5 files</li>
<li>"more_onion" – Execute the Backdoor script</li>
<li>"<strong>via_x</strong>" – execute cmd / shell commands locally</li>
</ul>
