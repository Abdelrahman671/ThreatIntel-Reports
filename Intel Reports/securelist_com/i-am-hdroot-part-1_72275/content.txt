<p>It was protected by a commercial VMProtect Win64 executable signed with a known compromised certificate from Chinese entity Guangzhou YuanLuo Technology. Moreover, the properties of the executable read as if it were Microsoft&#8217;s Net Command net.exe, and even running the sample also resulted in output typical of the original net.exe utility:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202700/hdroot_1.png" class="magnificImage"><img loading="lazy" decoding="async" id="72229" class="aligncenter size-full wp-image-72229" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202700/hdroot_1.png" alt="I am HDRoot!" width="568" height="41" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202700/hdroot_1.png 568w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202700/hdroot_1-300x22.png 300w" sizes="(max-width: 568px) 100vw, 568px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Masquerading as net.exe</em></p>
<p>All this pointed to the sample being rather suspicious.</p>
<h2 id="bootkit">Bootkit</h2>
<p>Since the code of the program was protected, an analysis of its functionality would have been an arduous task. But luckily a dump revealed some unique and quite important strings and four more samples hidden inside the initial one: Win32 and Win64 versions of one library and one driver:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202655/hdroot_2.png" class="magnificImage"><img loading="lazy" decoding="async" id="72230" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202655/hdroot_2.png" alt="I am HDRoot!" width="635" height="724" class="aligncenter size-full wp-image-72230" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202655/hdroot_2.png 635w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202655/hdroot_2-263x300.png 263w" sizes="(max-width: 635px) 100vw, 635px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Strings in malware body</em></p>
<p>The strings led us to suspect that this sample was actually a bootkit installer. And thanks to some clear artifacts we found a similar sample but with no code protection that confirmed our suspicions.</p>
<p>First of all, let&#8217;s run this tool:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202650/hdroot_3.png" class="magnificImage"><img loading="lazy" decoding="async" id="72231" class="aligncenter size-full wp-image-72231" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202650/hdroot_3.png" alt="I am HDRoot!" width="498" height="99" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202650/hdroot_3.png 498w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202650/hdroot_3-300x60.png 300w" sizes="(max-width: 498px) 100vw, 498px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Original HDD Rootkit output</em></p>
<p>The program parameters are quite self-explanatory – this tool installs a bootkit that infects the operating system during the boot stage with an arbitrary backdoor specified as a parameter. The backdoor has to be a Win32 executable or dynamic link library.</p>
<p>This utility is called &#8220;HDD Rootkit&#8221;; hence the base of our verdict names HDRoot. On 22 August 2006 the version number was 1.2.</p>
<p>So, we can conclude that the protected version was the same utility modified for use on the victim side to avoid revealing the intent of the tool in case someone outside the intruders&#8217; circle discovered it.</p>
<p>HDD Rootkit maintains a bunch of resources that also have quite telling names:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202648/hdroot_4.png" class="magnificImage"><img loading="lazy" decoding="async" id="72232" class="aligncenter size-full wp-image-72232" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202648/hdroot_4.png" alt="I am HDRoot!" width="196" height="90" /></a></p>
<p style="text-align: center;font-weight: bold"><em>HDD Rootkit resources</em></p>
<p>As it reads:</p>
<p>&#8220;MBR&#8221; maintains the 1st piece of malicious code that is injected to the MBR of an infected computer;<br />
&#8220;BOOT&#8221; – 2nd piece of malicious booting code;<br />
&#8220;RKIMAGE&#8221; – 3rd piece of malicious booting code;<br />
&#8220;DLLLOAD&#8221; – Dynamic Link Library that is pushed by the malicious booting code into the file system and OS autorun.</p>
<p>Let&#8217;s try running some executable with the help of a bootkit. In our experiment the role of the executable is played by a benign program that does nothing apart from create a file in the root of the C: drive. I will try to run it using the HDD Rootkit utility with the following command line:</p>
<p align="center"><em>hdroot.exe inst write_to_c.exe c:</em></p>
<p>telling it that I&#8217;d like to install a bootkit on drive C: that will make the program <i>write_to_c.exe</i> run on system startup.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202643/hdroot_5.png" class="magnificImage"><img loading="lazy" decoding="async" id="72233" class="aligncenter size-full wp-image-72233" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202643/hdroot_5.png" alt="I am HDRoot!" width="634" height="325" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202643/hdroot_5.png 634w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202643/hdroot_5-300x154.png 300w" sizes="(max-width: 634px) 100vw, 634px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Live installing of HDRoot bootkit</em></p>
<p>The utility checks the free space left on the specified drive and refuses to install the bootkit when the value is less than 30% of overall volume.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202639/hdroot_6.png" class="magnificImage"><img loading="lazy" decoding="async" id="72234" class="aligncenter size-full wp-image-72234" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202639/hdroot_6.png" alt="I am HDRoot!" width="561" height="374" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202639/hdroot_6.png 561w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202639/hdroot_6-300x200.png 300w" sizes="(max-width: 561px) 100vw, 561px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Free space check</em></p>
<p>So, now the bootkit has been installed. Let&#8217;s take a look at what has happened. First of all, part of the code in the MBR is replaced with a malicious one from the resource &#8220;MBR&#8221;:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202635/hdroot_7.png" class="magnificImage"><img loading="lazy" decoding="async" id="72235" class="aligncenter size-full wp-image-72235" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202635/hdroot_7.png" alt="I am HDRoot!" width="746" height="559" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202635/hdroot_7.png 746w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202635/hdroot_7-300x225.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202635/hdroot_7-200x150.png 200w" sizes="(max-width: 746px) 100vw, 746px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>&#8220;MBR&#8221; resource</em></p>
<p>The first 2 bytes EB 70 mean a jump to the 72nd offset where the rest of the 1st booting code block is located. The zeros before 0x70 and after 0xB0 mean the code of the original MBR at these positions remains intact. The following image represents a patched MBR after the bootkit is installed:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202629/hdroot_8.png" class="magnificImage"><img loading="lazy" decoding="async" id="72236" class="aligncenter size-full wp-image-72236" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202629/hdroot_8.png" alt="I am HDRoot!" width="641" height="549" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202629/hdroot_8.png 641w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202629/hdroot_8-300x257.png 300w" sizes="(max-width: 641px) 100vw, 641px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Injected malicious code in MBR</em></p>
<p>This first piece loads the next booting code block that has been placed by the bootkit installer in the 11th sector (Offset: 0x1400 bytes). The 2nd block is taken from the resource &#8220;BOOT&#8221;.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202624/hdroot_9.png" class="magnificImage"><img loading="lazy" decoding="async" id="72237" class="aligncenter size-full wp-image-72237" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202624/hdroot_9.png" alt="I am HDRoot!" width="504" height="182" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202624/hdroot_9.png 504w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202624/hdroot_9-300x108.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202624/hdroot_9-500x182.png 500w" sizes="(max-width: 504px) 100vw, 504px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>2nd booting block</em></p>
<p>The byte at 8th offset of the 2nd booting block is a drive number and the next DWORD is an offset in sectors where the next booting part is located. This example has the value 0x80, meaning drive 0 and the offset 0x5FD9A0, which if multiplied by 0x200 bytes (size of sector) results in 0xBFB34000. This is the offset in bytes from the beginning of the drive where the bootkit installer has put the 3rd booting block taken from its resource &#8220;RKIMAGE&#8221;.</p>
<p>The &#8220;RKIMAGE&#8221; resource has a large piece of code that implements a DLL injection (the DLL is taken from the &#8220;DLLLOAD&#8221; resource) into the file system and makes changes in the system registry so that DLL is loaded and run during system start-up. As that piece of code is executed at the early booting stage, there is no API for accessing the file system and the code parses the file systems (FAT32 and NTFS) on its own.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202621/hdroot_10.png" class="magnificImage"><img loading="lazy" decoding="async" id="72238" class="aligncenter size-full wp-image-72238" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202621/hdroot_10.png" alt="I am HDRoot!" width="283" height="96" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202621/hdroot_10.png 283w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202621/hdroot_10-280x96.png 280w" sizes="(max-width: 283px) 100vw, 283px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Supported file systems</em></p>
<p>It searches for the hardcoded special file whose content is replaced with the DLL taken from a specified place on the disk. Most versions of HDRoot that we have found and detected use the file <i>%windir%\WMSysPr9.prx</i> for these purposes. Sometimes the DLL overwrites some existing system library which is certainly not a safe way for malware to work because it could cause OS failure in some cases and alert the user to the infection. Among other files that can be used for overwriting we have noticed:</p>
<p><i>%windir%\twain.dll<br />
%windir%\msvidc32.dll<br />
%windir%\help\access.hlp<br />
%windir%\help\winssnap.hlp<br />
%windir%\system\olesvr.dll<br />
%windir%\syswow64\C_932.NLS<br />
%windir%\syswow64\C_20949.NLS<br />
%windir%\syswow64\dssec.dat<br />
%windir%\syswow64\irclass.dll<br />
%windir%\syswow64\msvidc32.dll<br />
%windir%\syswow64\kmddsp.tsp</i></p>
<p>The code then reads the content of the file <i>%windir%\system32\config\system</i> that maintains the content of the <i>HKEY_LOCAL_MACHINE\SYSTEM</i> registry hive. Among other things the registry hive contains information about installed services. There are numerous system services that are started during OS logon as <i>ServiceDll</i> via <i>svchost.exe</i> where the path to the functional library to be run is specified in the <i>ServiceDll</i> registry value for a particular service. The malicious booting code searches in the file &#8220;<i>system</i>&#8221; for the hardcoded path to a system library associated with a system service and replaces that value with the path to the injected DLL (for example, <i>%windir%\WMSysPr9.prx)</i>. In all the versions we encountered we found that HDRoot exploited the following services:</p>

<p>So, when the operating system starts running services, instead of loading the original service DLL <i>svchost.exe</i> loads a malicious one. This malicious library does nothing apart from load and run a backdoor taken from a specified offset on the hard drive where the bootkit installer HDD Rootkit had placed it. We have found two versions of HDRoot with different methods of doing this. The first one just saves the backdoor as a file <i>%windir%\temp\svchost.exe</i> and executes it with the help of the <i>WinExec</i> API function. By all appearances the malware author later decided that this approach is not the best way to run the backdoor because it is visible to AV products and the fact that the application has started may be noticed when inspecting events in the system logs. The other version of the DLL does not drop the file but allocates a read backdoor in memory, prepares it for proper execution (loads libraries according to the import table and fixes relocations) and runs it there on its own. This approach is much more clandestine as it substantially reduces the chances of discovering the backdoor even if the DLL or poisoned MBR are detected.</p>
<p>Returning to our experiment, when the command</p>
<p align="center"><i>hdroot.exe inst write_to_c.exe c:</i></p>
<p>has been run, we restart the operating system. After the OS has loaded we can see the result of running of our program <i>write_to_c.exe</i>, which behaves as though it were a backdoor:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202617/hdroot_11.png" class="magnificImage"><img loading="lazy" decoding="async" id="72239" class="aligncenter size-full wp-image-72239" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202617/hdroot_11.png" alt="I am HDRoot!" width="281" height="228" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Created test file zzz.bin</em></p>
<p>The file <i>C:\zzz.bin</i> is seen immediately after Windows has loaded, which proves that the program <i>write_to_c.exe</i> has been successfully executed.</p>
<p>The whole process of the HDRoot infection is as follows:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202613/hdroot_12.png" class="magnificImage"><img loading="lazy" decoding="async" id="72240" class="aligncenter size-full wp-image-72240" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202613/hdroot_12.png" alt="hdroot_12" width="800" height="300" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202613/hdroot_12.png 800w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202613/hdroot_12-300x113.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202613/hdroot_12-768x288.png 768w" sizes="(max-width: 800px) 100vw, 800px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>HDRoot operation scheme</em></p>
<p>Interestingly, the malware does not have functionality to start the original service that was substituted during the boot process. Because the affected services are part of the OS, neglecting to do this could cause Windows to malfunction and reveal the infection. This is even stranger considering the malware does try to cover its tracks. Namely &#8220;tries&#8221;, because it fails to do so. The dropped DLL has a function to restore the original value of <i>ServiceDll</i> in the registry, storing the path to the DLL associated with the service. But due to flawed code in the 3rd booting block (from &#8220;RKIMAGE&#8221;), which slightly patches the content of &#8220;DLLLOAD&#8221; before injecting, DLL starts holding the wrong data at hardcoded offsets and it prevents the DLL from finding the proper registry path to <i>ServiceDll</i> to restore the original value. That&#8217;s why, for example, &#8220;<i>C:\WINDOWS\WMSysPr9.prx</i>&#8221; can still be viewed instead of &#8220;<i>C:\WINDOWS\system32\wuauserv.dll</i>&#8221; after logging on to Windows:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202609/hdroot_13.png" class="magnificImage"><img loading="lazy" decoding="async" id="72241" class="aligncenter size-full wp-image-72241" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202609/hdroot_13.png" alt="I am HDRoot!" width="528" height="214" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202609/hdroot_13.png 528w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202609/hdroot_13-300x122.png 300w" sizes="(max-width: 528px) 100vw, 528px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Path remains to injected malicious DLL in registry</em></p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202605/hdroot_14.png" class="magnificImage"><img loading="lazy" decoding="async" id="72242" class="aligncenter size-full wp-image-72242" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202605/hdroot_14.png" alt="I am HDRoot!" width="530" height="344" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202605/hdroot_14.png 530w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202605/hdroot_14-300x195.png 300w" sizes="(max-width: 530px) 100vw, 530px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Wrong registry path and value name</em></p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202600/hdroot_15.png" class="magnificImage"><img loading="lazy" decoding="async" id="72243" class="aligncenter size-full wp-image-72243" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202600/hdroot_15.png" alt="I am HDRoot!" width="543" height="185" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202600/hdroot_15.png 543w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2015/09/07202600/hdroot_15-300x102.png 300w" sizes="(max-width: 543px) 100vw, 543px" /></a></p>
<p style="text-align: center;font-weight: bold"><em>Mistakenly overwritten registry SubKey with original value of ServiceDll</em></p>
<p>As a result, we have to conclude that the malware was not created very carefully, which is not what you expect from such a serious APT actor as Winnti. However, we have noticed the malware author&#8217;s efforts to make this bootkit work properly at the booting stage to avoid completely blocking the OS from loading. But the mistakes mentioned above leave some quite conspicuous signs of infection on the compromised computer. For example, original services such as Windows Update or Task Scheduler do not work, but it appears nobody noticed them.</p>
<p>During the investigation we found several backdoors that the HDRoot bootkit used for infecting operating systems. These malicious programs will be described in the <a href="https://securelist.com/analysis/publications/72356/i-am-hdroot-part-2/" target="_blank" rel="noopener noreferrer">next part of our article</a>.</p>

<p>In terms of partition style there are 2 options to have Windows Operating Systems installed &#8211; MBR and GPT. GPT is for UEFI mode. HDRoot bootkit works on hard drives with MBR only (legacy BIOS mode). So if you have installed Windows OS in UEFI-mode, the bootkit would not work. But HDRoot bootkit installer places 2nd booting code in 11th sector which is safe assuming there is MBR style partition, but in case of GPT this area contains information about partitions and theoretically this bootkit is able to corrupt this data if you have too many ( &gt; 36) partitions on the hard drive. Also, bear in mind that simply the fact that your computer has UEFI does not mean that Windows OS is installed in UEFI-mode on a hard drive with GPT: UEFI is compatible with MBR partition style as well.</p>
