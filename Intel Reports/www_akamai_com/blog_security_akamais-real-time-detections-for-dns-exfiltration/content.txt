    <p>Internet operations rely heavily on the Domain Name System (DNS) protocol, which is used to translate memorable domain names into IP addresses. Many services count on <a href="/products/edge-dns" target="_self">DNS</a>, making it one of the most important and fundamental internet protocols.</p>
<p>Enterprise and government IT departments use various defense mechanisms — like installing antivirus software on internal machines, deploying firewalls, and monitoring network traffic flows — to protect users from malicious actors. However, since the DNS protocol is integral to giving users internet access, it’s often poorly monitored and left unblocked.</p>
<p>Malicious actors target enterprises and government organizations for data theft using various methods. As mentioned, the DNS protocol is highly vulnerable and exploitable, so attackers use it to exchange data through a covert communication channel among themselves  — and/or command and control (C2) servers — and compromised hosts. This practice is known as <b>DNS exfiltration</b>.</p>

      <h2 class="cmp-title__text">How DNS exfiltration works</h2>
    
</div>

<div class="text text__pb-20 aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-519d5edff1&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-08-16T17:43:48Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>DNS exfiltration appends data to be exfiltrated to DNS queries and responses.     <p>DNS exfiltration appends data to be exfiltrated to DNS queries and responses. This process includes five high-level steps.<br />
</p>
<ol>
<li><p><b>Data encoding</b>. The data to be exfiltrated is encoded or broken down into smaller chunks — typically binary- or Base64-encoded — to fit into DNS query or response messages. This ensures that the data can be transmitted through the limited space available in DNS queries.<br />
</p>
</li>
<li><p><b>DNS tunneling</b>. The attacker's system or malware establishes a covert communication channel between a controlled DNS server and the attacker’s system or malware. This server could be a legitimate, compromised server or a malicious DNS server established by the attacker.<br />
</p>
</li>
<li><p><b>DNS requests</b>. The attacker's system generates DNS requests containing the encoded data as part of the query or response fields. These requests are sent to the controlled DNS server, which acts as the exfiltrated data’s receiver.<br />
</p>
</li>
<li><p><b>Data extraction</b>. On the attacker's side, the controlled DNS server receives the DNS requests, extracts the encoded data from the queries or responses, and decodes it back to its original form.<br />
</p>
</li>
<li><p><b>Data reconstruction</b>. Once the attacker has received all the encoded data, they can reconstruct the original information. This could involve reassembling the data chunks or applying any necessary decryption algorithms.<br />
</p>
</li>
</ol>
<p>The attacker may also use obfuscation techniques to blend exfiltration and  legitimate DNS traffic, making detection more challenging. Most significantly, the attackers normally use a low and slow approach to avoid detection — for example, exfiltration DNS queries may only occur once every 10 seconds.</p>

    <p>To further improve our DNS exfiltration detection capabilities, Akamai’s data science team is developing a novel, real-time, information-based DNS exfiltration detection algorithm.</p>
<p>This new algorithm was inspired by the concept of Information-based Heavy Hitters (ibHH) in a DNS query stream. ibHHs are essentially domains associated with a large amount of unique information conveyed over queries’ subdomains. </p>
<p>Our algorithm helps organizations estimate the amount of distinct information conveyed from users to registered domains over subdomains. When the amount of information conveyed to a domain exceeds a predefined threshold, the algorithm generates and reports an exfiltration alert.</p>

Other notable DNS exfiltration cases include those by advanced persistent threat groups like <a href="https://attack.mitre.org/groups/G0049/" target="_blank" rel="nofollow noreferrer">OilRig</a> and <a href="https://www.cybereason.com/blog/operation-cobalt-kitty-apt" target="_blank" rel="nofollow noreferrer">Operation Cobalt Kitty</a>.</p>

      <h2 class="cmp-title__text">Illuminating exfiltration detection challenges<br />

<div class="text text__pb-none aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-7871875a3f&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-08-16T14:16:50Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>Detecting and preventing DNS exfiltration is difficult because of the DNS protocol’s ubiquity, the challenges of overblocking legitimate DNS traffic that can impact user productivity, and the prevalence of using the DNS protocol to establish bidirectional communication for legitimate purposes.&lt;/p>\r\n&lt;p>For example, &lt;a href=\&#34;/content/akamai/en/blog/security/attacks-anti-malware-dns-spoofing.html\&#34; target=\&#34;_self\&#34;>DNS anti-malware services&lt;/a> and some &lt;a href=\&#34;https://help.canary.tools/hc/en-gb\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>honeypot services&lt;/a> use the DNS protocol and have similar DNS traffic patterns, which makes it hard to distinguish between malicious and benign DNS exfiltration events.&lt;/p>\r\n&lt;p>Timing and positioning of DNS exfiltration detection is also crucial, and can occur:&lt;/p>\r\n&lt;ul>\r\n&lt;li>&lt;p>In real time on the DNS resolver as the DNS query stream processes&lt;/p>\r\n&lt;/li>\r\n&lt;li>&lt;p>Later and offline as DNS queries are consolidated, aggregated, and analyzed&lt;/p>\r\n&lt;/li>\r\n&lt;/ul>\r\n&#34;}}">
    <p>Detecting and preventing DNS exfiltration is difficult because of the DNS protocol’s ubiquity, the challenges of overblocking legitimate DNS traffic that can impact user productivity, and the prevalence of using the DNS protocol to establish bidirectional communication for legitimate purposes.</p>
<p>For example, <a href="/blog/security/attacks-anti-malware-dns-spoofing" target="_self">DNS anti-malware services</a> and some <a href="https://help.canary.tools/hc/en-gb" target="_blank" rel="nofollow noreferrer">honeypot services</a> use the DNS protocol and have similar DNS traffic patterns, which makes it hard to distinguish between malicious and benign DNS exfiltration events.</p>
<p>Timing and positioning of DNS exfiltration detection is also crucial, and can occur:</p>
<ul>
<li><p>In real time on the DNS resolver as the DNS query stream processes</p>
</li>
<li><p>Later and offline as DNS queries are consolidated, aggregated, and analyzed</p>
</li>
</ul>

      <h3 class="cmp-title__text">Real-time DNS exfiltration detection<br />
</h3>
    
</div>

    


</div>
<div class="text aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-3f162b40ab&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-08-16T14:23:10Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>Real-time exfiltration detection is preferable, as it enables faster identification and remediation. However, this approach is much more difficult, since real-time detection algorithms should be fast and memory-efficient to avoid impacting the recursive DNS resolver performance.&lt;/p>\r\n&lt;p>&lt;a href=\&#34;https://www.sciencedirect.com/science/article/abs/pii/S1389128621003248\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>Many DNS exfiltration detection algorithms&lt;/a> have been proposed in response to the volume of DNS exfiltration research conducted over the last decade. Despite the importance of real-time detection, past research typically focused solely on detection rate improvements. Earlier proposed detection methods were usually based on supervised or unsupervised machine learning algorithms. These existing design methods limit real-time detection capabilities because:&lt;/p>\r\n&lt;ul>\r\n&lt;li>&lt;p>Nontrivial feature extraction could not be applied in real time.&lt;br>\r\n &lt;/p>\r\n&lt;/li>\r\n&lt;li>&lt;p>Complicated models not designed for execution on the network perimeter required data collection in dedicated computation environments.&lt;/p>\r\n&lt;/li>\r\n&lt;/ul>\r\n&lt;ul>\r\n&lt;li>&lt;p>Supervised models are limited by their need for labeled datasets for benign and malicious &lt;a href=\&#34;/content/akamai/en/blog/security/a-deep-dive-on-malicious-dns-traffic.html\&#34; target=\&#34;_self\&#34;>DNS traffic&lt;/a>, which is problematic because no high-quality publicly available DNS query dataset for DNS exfiltration exists. This means associated detection methods can’t accurately identify new and emerging DNS exfiltration malware.&lt;br>\r\n&lt;/p>\r\n&lt;/li>\r\n&lt;/ul>\r\n&#34;}}">
    <p>Real-time exfiltration detection is preferable, as it enables faster identification and remediation. However, this approach is much more difficult, since real-time detection algorithms should be fast and memory-efficient to avoid impacting the recursive DNS resolver performance.</p>
<p><a href="https://www.sciencedirect.com/science/article/abs/pii/S1389128621003248" target="_blank" rel="nofollow noreferrer">Many DNS exfiltration detection algorithms</a> have been proposed in response to the volume of DNS exfiltration research conducted over the last decade. Despite the importance of real-time detection, past research typically focused solely on detection rate improvements. Earlier proposed detection methods were usually based on supervised or unsupervised machine learning algorithms. These existing design methods limit real-time detection capabilities because:</p>
<ul>
<li><p>Nontrivial feature extraction could not be applied in real time.<br />
 </p>
</li>
<li><p>Complicated models not designed for execution on the network perimeter required data collection in dedicated computation environments.</p>
</li>
</ul>
<ul>
<li><p>Supervised models are limited by their need for labeled datasets for benign and malicious <a href="/blog/security/a-deep-dive-on-malicious-dns-traffic" target="_self">DNS traffic</a>, which is problematic because no high-quality publicly available DNS query dataset for DNS exfiltration exists. This means associated detection methods can’t accurately identify new and emerging DNS exfiltration malware.<br />
</p>
</li>
</ul>

      <h2 class="cmp-title__text">Using ibHH for real-time DNS exfiltration detection<br />

<div class="text text__pb-none aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-c28b2a9ea8&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-08-29T12:52:55Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>Akamai’s DNS exfiltration detection algorithm can operate in real time directly on Akamai’s carrier grade recursive DNS network. Our proposed approach — ibHH — is inspired by &lt;a href=\&#34;https://cs344-stanford.github.io/lectures/Lecture-4-HHD.pdf\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>heavy hitter detection in streams&lt;/a>, specifically &lt;a href=\&#34;https://dl.acm.org/doi/10.1145/3132465.3132474\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>distinct heavy hitter detection&lt;/a>, and can process millions of queries per second. As such, it doesn’t impair the DNS resolver’s responsiveness or ability to handle DNS resolution.&lt;/p>\r\n&lt;p>Akamai has proposed the&lt;b> information heavy hitter&lt;/b> idea to describe stream elements associated with a large amount of unique information. ibHH detection is based on how the amount of information conveyed from DNS query subdomains to their domains is quantified. Domains associated with large amounts of unique information are identified, and suspected DNS exfiltration domains are reported — all with sublinear memory consumption and performance time optimizations.&lt;/p>\r\n&lt;p>The ibHH input is a DNS query stream, such that for each DNS query (e.g., &lt;b>subdomain.example.com&lt;/b>) we extract the domain and subdomain to obtain the pair (&lt;b>example.com&lt;/b>, &lt;b>subdomain&lt;/b>), where &lt;b>example.com&lt;/b> is the query’s domain and &lt;b>subdomain&lt;/b> is its subdomain.&lt;/p>\r\n&lt;p>ibHH consists of a fixed-size cache that stores, with high probability, the information heavy hitters in the stream. The cache size is an input parameter of the algorithm. ibHH also consists of ​​a random hash function (&lt;b>Hash~U[0,1]&lt;/b>) that allows us to sample the distinct DNS query stream, detection threshold algorithm parameter, and threshold value τ (initialized to one) which is the probability of including a cache domain.&lt;/p>\r\n&lt;p>Finally, the detection threshold is inputted to ibHH, and when the information count exceeds this threshold, a new threshold is raised for the corresponding domain.&lt;/p>\r\n&lt;p>Figure 2 illustrates ibHH. An open source Python implementation of our algorithmic approach will soon be available from &lt;a href=\&#34;https://github.com/akamai\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>Akamai’s GitHub&lt;/a>.&lt;/p>\r\n&#34;}}">
    <p>Akamai’s DNS exfiltration detection algorithm can operate in real time directly on Akamai’s carrier grade recursive DNS network. Our proposed approach — ibHH — is inspired by <a href="https://cs344-stanford.github.io/lectures/Lecture-4-HHD.pdf" target="_blank" rel="nofollow noreferrer">heavy hitter detection in streams</a>, specifically <a href="https://dl.acm.org/doi/10.1145/3132465.3132474" target="_blank" rel="nofollow noreferrer">distinct heavy hitter detection</a>, and can process millions of queries per second. As such, it doesn’t impair the DNS resolver’s responsiveness or ability to handle DNS resolution.</p>
<p>Akamai has proposed the<b> information heavy hitter</b> idea to describe stream elements associated with a large amount of unique information. ibHH detection is based on how the amount of information conveyed from DNS query subdomains to their domains is quantified. Domains associated with large amounts of unique information are identified, and suspected DNS exfiltration domains are reported — all with sublinear memory consumption and performance time optimizations.</p>
<p>The ibHH input is a DNS query stream, such that for each DNS query (e.g., <b>subdomain.example.com</b>) we extract the domain and subdomain to obtain the pair (<b>example.com</b>, <b>subdomain</b>), where <b>example.com</b> is the query’s domain and <b>subdomain</b> is its subdomain.</p>
<p>ibHH consists of a fixed-size cache that stores, with high probability, the information heavy hitters in the stream. The cache size is an input parameter of the algorithm. ibHH also consists of ​​a random hash function (<b>Hash~U[0,1]</b>) that allows us to sample the distinct DNS query stream, detection threshold algorithm parameter, and threshold value τ (initialized to one) which is the probability of including a cache domain.</p>
<p>Finally, the detection threshold is inputted to ibHH, and when the information count exceeds this threshold, a new threshold is raised for the corresponding domain.</p>
<p>Figure 2 illustrates ibHH. An open source Python implementation of our algorithmic approach will soon be available from <a href="https://github.com/akamai" target="_blank" rel="nofollow noreferrer">Akamai’s GitHub</a>.</p>

    <p>To locate information heavy hitter domains from a DNS query stream, we must quantify the amount of unique information encoded in subdomains and accumulate this amount per domain. <div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-1b0673fc24&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2023-08-16T14:34:50Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>To calculate the exact amount of unique information for each domain, we needed a set of associated subdomains that require linear space consumption.     <p>To calculate the exact amount of unique information for each domain, we needed a set of associated subdomains that require linear space consumption. Since our solution runs on DNS resolvers with limited memory capabilities, we can’t use precise information counters — so we employ count-distinct approximation algorithms instead.</p>
<p>The <a href="https://www.cambridge.org/core/books/abs/mining-of-massive-datasets/mining-data-streams/B903CC35456F1DC93B0F9B9742876F30" target="_blank" rel="nofollow noreferrer">count-distinct problem</a> is well-studied, and many accurate and performant approximation algorithms exist to address it.</p>
<p>One such solution is <a href="https://hal.archives-ouvertes.fr/hal-00406166/" target="_blank" rel="nofollow noreferrer">HyperLogLog</a> (HLL). Akamai uses a HLL variant called <a href="https://dl.acm.org/doi/abs/10.1145/2452376.2452456" target="_blank" rel="nofollow noreferrer">HyperLogLog++ </a>(HLL++) that’s more accurate and uses less memory. We store and use an HLL++ instance for each cached domain to estimate the amount of information conveyed to the cached domain. Finally, once a cached domain’s information count exceeds the detection threshold, an alert is raised.</p>
