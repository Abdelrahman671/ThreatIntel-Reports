<div id="content">
    <div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>Gootkit is an advanced banking trojan first discovered in mid-2014. Known for using various techniques to evade detection, the malware also has its own unique twists: its partially written in JavaScript and it incorporates the node.js runtime environment.</p> 
  <p>Gootkit employs several checks in order to identify a virtual environment and halt its propagation sequence once that happens. In this post, well demonstrate a way to tackle these anti-research methods in order to have the sample executing in a virtual environment.</p> 
  <h2 id="first_glance">First Glance</h2> 
  <p>Running the sample in a virtual machine results in the creation of several threads, but no modifications to the system are made. Without interruptions, Gootkits process would continue to run forever.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-0049258d3b" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 1: procmon view of the sample">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/figure_1.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    <div class="a02-image__caption">
        
            Figure 1: procmon view of the sample
        
    </div>
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>Running the sample in a debugger allows us to interrupt the process and inspect its execution flow. Whenever the debugger is paused, Gootkits process is sleeping.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-54b999287a" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 2: View of the stack of the sleeping thread">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/figure_2.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    <div class="a02-image__caption">
        
            Figure 2: View of the stack of the sleeping thread
        
    </div>
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>According to the return address, the call comes from the malware (not from a system library) and the parameter is -1, meaning: sleep forever.</p> 
  <p>How can we make the malware go to a more interesting execution flow in the code?</p> 
  <h2 id="diving_deeper_into_the_assembly_code">Diving Deeper into the Assembly Code</h2> 
  <p>Looking at the assembly of the function that calls the sleep API, there are three possible execution paths: #1 and #2 are paths that we could force the malware to take by patching the conditional jumps that lead to them; #3 is the path it takes before reaching the sleep API.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-253535ab8f" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 3: The calling function of the sleep API, thread 1">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/figure_3.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    <div class="a02-image__caption">
        
            Figure 3: The calling function of the sleep API, thread 1
        
    </div>
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>There are 3 possible paths of execution. Well use a shortcut in order to decide quickly which one is the most interesting one to follow. That shortcut involves checking cross-references(xrefs) from each function, which will indicate where most of the activity would take place.</p> 
  <p>Execution path #1, contains a call to sub_4047EC, which had no xrefs.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-d8151198ea" class=" a02-image__image">
    
    
    <div class="img-link">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/no_xrefs_edited.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>Execution path #2, contains a call to sub_409450, which had just a few xrefs.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-0674b135d5" class=" a02-image__image">
    
    
    <div class="img-link">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/few_xrefs_edited.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>Execution path #3 creates a thread that will execute sub_40C8C3, which had many xrefs.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-dde22b1357" class=" a02-image__image">
    
    
    <div class="img-link">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/many_xrefs_edited.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>So, execution path #3, the newly created thread, seems like the one to take because it has a massive function graph, indicating that all the activity will most likely happen there.</p> 
  <p>After placing a breakpoint at the beginning of the thread function(0x40C8C3), we start stepping through.One of the subroutines gets stuck in an endless loop, which means Gootkit is sleeping again!</p> 
  <p>Recap #1: There are two threads, one of which is sleeping forever, but it is not the interesting one. The second one is in a sleep loop, and our goal is to force it to take a different branch in order to execute the interesting functionality.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-abd5677a4a" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 4: The calling function of the sleep API, thread #2">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/figure_4_edited.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    <div class="a02-image__caption">
        
            Figure 4: The calling function of the sleep API, thread #2
        
    </div>
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>The only way to break the loop is to make the string comparison(StrStrIW) fail. The comparison parameters:</p> 
  <ul> 
   <li>Hard-coded string Xeon</li> 
   <li>Data from <i>HKEY_LOCAL_MACHINE\HARDWARE\DESCRIPTION\System\CentralProcessor\0\ProcessorNameString</i></li> 
  </ul> 
  <p>If we changed this value in the registry, we could successfully overcome this hurdle. But, would that be the end? Would it run? Sadly, after making this change, Gootkit goes back to sleep in another endless loop, this time in a different subroutine.</p> 
  <p>Recap #2: Again, there are two threads, one that is sleeping forever (but not interesting); the other is running in a sleep loop.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-a0126e48d6" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 5: Gootkits two threads in Immunity Debuggers thread view ">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/figure_5_edited.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    <div class="a02-image__caption">
        
            Figure 5: Gootkits two threads in Immunity Debuggers thread view 
        
    </div>
</figure></div>
<div class="a02-image section">




<figure id="image-e8960ad422" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 6: Another calling function of the sleep API, thread #2">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/figure_6_edited.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    <div class="a02-image__caption">
        
            Figure 6: Another calling function of the sleep API, thread #2
        
    </div>
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>After inspecting the condition that leads us to the sleep API call, it becomes clear that this branch is taken when the GetProcAddress fails. But, why would this simple API fail?</p> 
  <p>At second glance, it is evident that the first parameter of the GetProcAddress API, the HModule, is always equal to null because of the XOR instruction highlighted in red. The conclusion is that this API is meant to fail! This whole function is a trap, and our goal is to avoid entering it altogether.</p> 
  <h2 id="following_the_traps">Following the Traps</h2> 
  <p>Well rename the function to trap and then check all the function references to see how many traps are there.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-3f066801bf" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 7: All function references to the trap function">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/figure_7_edited.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    <div class="a02-image__caption">
        
            Figure 7: All function references to the trap function
        
    </div>
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>There are 17 traps to overcome and they are all in the same function, so we assume this is the function making all the environment checks. Well have to check each condition that leads to a trap and make sure to change the environment (or patch the binary) in a way that would bypass the trap.</p> 
  <p>Figure 8 shows a code snippet from the sub_409AE2 function.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-8e7c8c8fc7" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 8: The calling function for all the traps in pseudo C view ">
        <img data-src="/content/dam/f5/f5-labs/articles/20180710_gootkit_traps/figure_8_edited.png" data-component-type="platter" class="js-bg-lozad"/>
    </div>
    <div class="a02-image__caption">
        
            Figure 8: The calling function for all the traps in pseudo C view 
        
    </div>
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>Note that each while loop is performing string decryption on the sequences of bytes shown in the variables above the loop. When following the execution in a debugger, the strings are decrypted, and some meaningful indicators of VM checks are visible. (See appendix for decryption function details.)</p> 
  <p>In this code snippet, three checks are evident:</p> 
  <ul> 
   <li>MAC address check</li> 
   <li>Checking the presence of dbghelp.dll  debugger indicator</li> 
   <li>Checking the presence of sbiedll.dll  sandboxie indicator</li> 
  </ul> 
  <p>By following the traps and patching the system accordingly, the environment is prepared for Gootkit to run in.</p> 
  <p>The rest of the checks include:</p> 
  <ul> 
   <li>Compare user name to &quot;CurrentUser&quot;/&quot;Sandbox&quot;</li> 
   <li>Compare computer name to &quot;SANDBOX&quot;/&quot;7SILVIA&quot;</li> 
   <li><i>HKEY_LOCAL_MACHINE\HARDWARE\DESCRIPTION\System\SystemBiosVersion&quot; </i>compare with AMI, VirtualBox, BOCHS, INTEL 640000, 55274-640-2673064-23950, and other serials</li> 
  </ul> 
  <p>After patching a virtual machine and running the sample, its clear that it is no longer stuck in an endless loop and that the sample continues its propagation in the system.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-9b7ef0707a" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 9: Gootkit installation process">

    <div class="a02-image__caption">

<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <h2 id="the_road_not_taken">The Road Not Taken</h2> 
  <p>Patching the virtual environment by changing the MAC address and several registry values is one way to get the sample to run. It is the recommended way to go when running Gootkit samples in an automated analysis system. However, this post would not be complete without presenting the other solution: patching the binary itself.</p> 
  <p>Every time theres a conditional jump to the trap function, one could change it to a simple jump opcode, thus skipping the trap function. This would direct the flow from the jump to the next code block while skipping the trap function.</p> 
  <p>However, this patch must be applied 17 times, and thats a tedious task. Instead, since all the calls to the trap function come from one function, its preferable to patch and bypass the call to that function.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-fd95b82392" class=" a02-image__image">
    
    
    <div class="img-link" data-title="Figure 10: The JNZ instruction opcode(0x75) is changed to the JMP instruction opcode(0xEB) in Immunity Debugger ">

    </div>
    <div class="a02-image__caption">
        
            Figure 10: The JNZ instruction opcode(0x75) is changed to the JMP instruction opcode(0xEB) in Immunity Debugger 
        
    </div>
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <h2 id="conclusion">Conclusion</h2> 
  <p>When analyzing malware, especially intricate, multi-threaded ones like Gootkit, it is important to choose the interesting thread to follow and to keep in mind that even functions that look like they contain seemingly ordinary code may be used as a deception.</p> 
  <p>MD5: 47f8d6d3e4410218c4c64701cb7d7d3d</p>
