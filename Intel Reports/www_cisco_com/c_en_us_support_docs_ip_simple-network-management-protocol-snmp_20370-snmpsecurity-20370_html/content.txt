Information</a></div></div></div><div class="lia-message-template-content-zone"><p><ul =""=""></ul></p><h2 id="toc-hId--2040164502"></h2><h2 id="toc-hId-447348331">Introduction</h2><p>This document describes how to secure your Simple Network Management Protocol (SNMP).</p><h2 id="toc-hId--1360106132"></h2><h2 id="toc-hId-1127406701">Prerequisites</h2><h3 id="toc-hId-1817968175">Requirements</h3><p>There are no specific requirements for this document.</p><h3 id="toc-hId-10513712"></h3><h3 id="toc-hId--1796940751">Components Used</h3><p>The information in this document is based on these software and hardware versions:</p><ul>
<li>
<p>SNMP View — Cisco IOS® Software Release 10.3 or later.</p>
</li>
<li>
<p>SNMP version 3 — Introduced in Cisco IOS Software Release 12.0(3)T.</p>
</li>
</ul><p>The information in this document was created from the devices in a specific lab environment. All of the devices used in this document started with a cleared (default) configuration. If your network is live, ensure that you understand the potential impact of any command.</p><h3 id="toc-hId-690572082"></h3><h3 id="toc-hId--1116882381">Conventions</h3><p>For more information on document conventions, refer to the<a href="https://www.cisco.com/c/en/us/support/docs/dial-access/asynchronous-connections/17016-techtip-conventions.html" target="_blank" rel="noopener nofollow noreferrer"> Cisco Technical Tips Conventions</a>.</p><h2 id="toc-hId--1127385485"></h2><h2 id="toc-hId--1261070209">Background Information</h2><p>It is important to secure your SNMP especially when the vulnerabilities of SNMP can be repeatedly exploited to produce a denial of service (DoS).</p><h2 id="toc-hId-1226442624">Strategies to Secure&nbsp;SNMP</h2><h3 id="toc-hId--451929120">Choose a Good SNMP Community String</h3><p>It is not a good practice to use <strong> public </strong> as read-only and <strong> private </strong> as read-write community strings.</p><h3 id="toc-hId-2035583713"></h3><h3 id="toc-hId-228129250">Setup SNMP View</h3><p>The <strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> Setup SNMP view </code> </strong> command can block the user with only access to limited Management Information Base (MIB). By default, there is no <strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> SNMP view entry exists </code></strong>. This command is configured at the global configuration mode and first introduced in Cisco IOS Software version 10.3. It works similar to<strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> access-list </code></strong>in that if you have any<strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> SNMP View </code>&nbsp; </strong>on certain MIB trees, every other tree is denied inexplicably. However, the sequence is not important and it goes through the entire list for a match before it stops.</p><p>To create or update a view entry, use the<span><strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset">snmp-server view global configuration </code></strong></span>command. To remove the specified SNMP server view entry, use the<strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> no </code></strong>form of this command.</p><p><strong>Syntax</strong>:</p><blockquote>
<pre><strong>snmp-server view view-name oid-tree {included | excluded}</strong> 

<strong>no snmp-server view view-name</strong> 

</pre>
</blockquote><p><strong>Syntax Description</strong>:</p><ul>
<li>
<p><strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset">view-name</code></strong>— Label for the view record that you update or create. The name is used to reference the record.</p>
</li>
<li>
<p><strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset">oid-tree</code></strong>&nbsp;— Object identifier of the Abstract Syntax Notation One (ASN.1) subtree to be included or excluded from the view. To identify the subtree, specify a text string that consists of numbers, such as 1.3.6.2.4, or a word, such as <strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> system</code></strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset">.</code>&nbsp;Replace a single sub-identifier with the asterisk (*) wildcard to specify a subtree family; for example 1.3.*.4.</p>
</li>
<li>
<p><strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset">included | excluded</code></strong>— Type of view. You must specify either included or excluded.</p>
</li>
</ul><p>Two standard predefined views can be used when a view is required instead of a view that must be defined. One is everything, which indicates that the user can see all objects. The other is<i> restricted </i>, which indicates that the user can see three groups: <strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> system</code></strong>, <strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> snmpStats</code></strong>, and <strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> snmpParties</code></strong>. The predefined views are described in RFC 1447.</p><p style="background-image: url('https://www.cisco.com/c/dam/en/us/td/i/templates/note.gif'); background-repeat: no-repeat; background-position: 2px 4px; height: auto; width: auto; padding: 10px 5px 10px 35px; margin-top: 10px; margin-bottom: 10px; border-top: 1px solid #ccc; border-bottom: 1px solid #ccc; overflow-x: hidden;"><strong>Note</strong>: The first <strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> snmp-server </code>&nbsp;&nbsp;</strong>command that you enter enables both versions of SNMP.</p><p>This example creates a view that includes all objects in the MIB-II system group except for<strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> sysServices </code></strong>(System 7) and all objects for interface 1 in the MIB-II interfaces group:</p><blockquote>
<pre><strong>snmp-server view agon system included</strong>

<strong>snmp-server view agon system.7 excluded</strong> 

<strong>snmp-server view agon ifEntry.*.1 included</strong> 

</pre>
</blockquote><p>This is a complete example for how to apply the MIB with community string and the output of the <strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> snmpwalk </code></strong>with <strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> view </code></strong>in place. <pre><strong>snmp-server view myview mib-2 included</strong>


<strong>snmp-server view myview atEntry excluded</strong>

</blockquote><h3 id="toc-hId--1579325213"></h3><h3 id="toc-hId-908187620">Setup SNMP Community with Access-list</h3><p>The best current practices recommend that you apply Access Control Lists (ACLs) to community strings and ensure that the requests community strings are not identical to notifications community strings. Access lists provide further protection when used in combination with other protective measures.</p><p>This example sets up ACL to community string:</p><blockquote>
<pre><strong>access-list 1 permit 10.1.1.1</strong> 

<strong>snmp-server community string1 ro 1</strong>
</pre>
</blockquote><p>When you use different community strings for requests and trap messages, it reduces the likelihood of further attacks or compromises if the community string is discovered by an attacker. Otherwise, an &nbsp;attacker could compromise a remote device or sniff a trap message from the network without authorization.</p><p>Once you enable trap with a community string, the string can be enabled for SNMP access in some Cisco IOS software. You must explicitly disable this community.&nbsp;For example:</p><blockquote>
<pre><strong>access-list 10 deny any </strong>
<strong>snmp-server host 10.1.1.1 mystring1</strong> 
<strong>snmp-server community mystring1 RO 10 </strong>

</pre>
</blockquote><h3 id="toc-hId--899266843"></h3><h3 id="toc-hId-1588245990">Setup SNMP Version 3</h3><p>Do these steps to configure SNMP version 3:</p><ol type="1">

</li>

<li>

</blockquote><h3 id="toc-hId--219208473"></h3><h3 id="toc-hId--1328500995">Setup ACL on Interfaces</h3><p>The ACL feature provides security measures that prevent attacks such as IP spoofing. The ACL can be applied on incoming or outgoing interfaces on routers.</p><p>On platforms that do not have the option to use receive ACLs (rACLs), it is possible to permit User Datagram Protocol (UDP) traffic to the router from trusted IP addresses with interface ACLs.</p><p>The next extended access list can be adapted to your network. This example assumes that the router has IP addresses 192.168.10.1 and 172.16.1.1 configured on its interfaces, that all SNMP access is to be restricted to a management station with the IP address of 10.1.1.1, and that the management station need only communicate with IP address 192.168.10.1:</p><blockquote>
<pre><strong>access-list 101 permit udp host 10.1.1.1 host 192.168.10.1</strong> </pre>
</blockquote><p>The&nbsp;<span><strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset">access-list</code></strong>&nbsp;&nbsp;</span>must then be applied to all interfaces with these configuration commands:</p><blockquote>
<pre><strong>interface ethernet 0/0

ip access-group 101 in</strong>
</pre>
</blockquote><p>All devices that communicate directly with the router on UDP ports need to be specifically listed in the previous access list. Cisco IOS software uses ports in the range 49152 to 65535 as the source port for outbound sessions such as Domain Name System (DNS) queries.</p><p>For devices that have many IP addresses configured, or many hosts that need to communicate with the router, this is not always a scalable solution.</p><h3 id="toc-hId-1159011838">rACLs</h3><p>For distributed platforms, rACLs can be an option that starts in Cisco IOS Software Release 12.0(21)S2 for the Cisco 12000 Series Gigabit Switch Router (GSR) and Release 12.0(24)S for the Cisco 7500 Series. The receive access lists protect the device from harmful traffic before the traffic can impact the route processor. Receive path ACLs also are considered a network security best practice, and must be considered as a long-term addition to good network security, as well as a workaround for this specific vulnerability. The CPU load is distributed to the line card processors and helps mitigate load on the main route processor. The white paper entitled<a href="https://www.cisco.com/site/us/en/index.html" target="_blank" rel="noopener nofollow noreferrer"> GSR: Receive Access Control Lists </a>helps to identify legitimate traffic. Use that white paper to understand how to send legitimate traffic to your device and also deny all unwanted packets..</p><h3 id="toc-hId--648442625"></h3><h3 id="toc-hId-1839070208">Infrastructure ACLs</h3><p data-unlink="true">Although it is often difficult to block traffic that transits your network, it is possible to identify traffic that must never be allowed to target your infrastructure devices and block that traffic at the border of your network. Infrastructure ACLs (iACLs) are considered a network security best practice and must be considered as a long-term addition to good network security as well as a workaround for this specific vulnerability. The white paper,<a href="https://www.cisco.com/c/en/us/support/docs/ip/access-lists/43920-iacl.html" target="_blank" rel="noopener nofollow noreferrer">&nbsp;Protecting Your Core: Infrastructure Protection Access Control Lists</a>&nbsp;,presents guidelines and recommended deployment techniques for iACLs..</p><h3 id="toc-hId-31615745"></h3><h3 id="toc-hId--1775838718">Cisco Catalyst LAN Switch Security Feature</h3><p>The IP Permit List feature restricts inbound Telnet and SNMP access to the switch from unauthorized source IP addresses. Syslog messages and SNMP traps are supported to notify a management system when a violation or unauthorized access occurs.</p><p>A combination of the Cisco IOS software security features can be used to manage routers and Cisco Catalyst switches. A security policy needs to be established that limits the number of management stations that can access the switches and routers.</p><p data-unlink="true">For more information on how to increase security on IP networks, refer to Increasing Security on IP Networks&nbsp;.</p><h2 id="toc-hId-582591396"></h2><h2 id="toc-hId--1224863067">How to Check SNMP Errors</h2><p>Configure the SNMP community ACLs with the<strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> log </code></strong>keyword. Monitor<strong> <code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> syslog </code></strong>for failed attempts, as show below.</p><blockquote>
<pre><strong>access-list 10 deny any log 
snmp-server community public RO 10</strong>
</pre>
</blockquote><p>When someone tries to access the router with the community public, you see a<span> <strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset">syslog</code>&nbsp; </strong></span>similar to this:</p><blockquote>
<pre>%SEC-6-IPACCESSLOGS: list 10 denied 172.16.1.15packet</pre>
</blockquote><p>This output means that access-list 10 has denied five SNMP packets from the host 172.16.1.1.</p><p>Periodically check SNMP for errors with the <strong><code class="cCN_CmdName" style="white-space: pre-wrap; font-style: unset; font-weight: unset; font-family: unset"> show snmp </code></strong>command, as shown here:</p><blockquote>
