										</nav><!-- <content> --><p>The AKS-managed Microsoft Entra integration simplifies the Microsoft Entra integration process. Previously, you were required to create a client and server app, and the Microsoft Entra tenant had to assign <a href="/en-us/entra/identity/role-based-access-control/permissions-reference#directory-readers" data-linktype="absolute-path">Directory Readers</a> role permissions. Now, the AKS resource provider manages the client and server apps for you.</p>
<p>Cluster administrators can configure Kubernetes role-based access control (Kubernetes RBAC) based on a user's identity or directory group membership. Microsoft Entra authentication is provided to AKS clusters with OpenID Connect. OpenID Connect is an identity layer built on top of the OAuth 2.0 protocol. For more information on OpenID Connect, see the <a href="/en-us/entra/identity-platform/v2-protocols-oidc" data-linktype="absolute-path">OpenID Connect documentation</a>.</p>
<p>Learn more about the Microsoft Entra integration flow in the <a href="concepts-identity#azure-ad-integration" data-linktype="relative-path">Microsoft Entra documentation</a>.</p>
<p>This article provides details on how to enable and use managed identities for Azure resources with your AKS cluster.</p>
<h2 id="limitations">Limitations</h2>
<p>The following are constraints integrating Azure managed identity authentication on AKS.</p>
<ul>
<li>Integration can't be disabled once added.</li>
<li>Downgrades from an integrated cluster to the legacy Microsoft Entra ID clusters aren't supported.</li>
<li>Clusters without Kubernetes RBAC support are unable to add the integration.</li>
</ul>
<h2 id="before-you-begin">Before you begin</h2>
<p>The following requirements need to be met in order to properly install the AKS addon for managed identity.</p>
<ul>
<li>You have Azure CLI version 2.29.0 or later installed and configured. Run <code>az --version</code> to find the version. If you need to install or upgrade, see <a href="/en-us/cli/azure/install-azure-cli" data-linktype="absolute-path">Install Azure CLI</a>.</li>
<li>You need <code>kubectl</code> with a minimum version of <a href="https://github.com/kubernetes/kubernetes/blob/master/CHANGELOG/CHANGELOG-1.18.md#v1181" data-linktype="external">1.18.1</a> or <a href="https://github.com/Azure/kubelogin" data-linktype="external"><code>kubelogin</code></a>. With the Azure CLI and the Azure PowerShell module, these two commands are included and automatically managed. Meaning, they're upgraded by default and running <code>az aks install-cli</code> isn't required or recommended. If you're using an automated pipeline, you need to manage upgrades for the correct or latest version. The difference between the minor versions of Kubernetes and <code>kubectl</code> shouldn't be more than <em>one</em> version. Otherwise,  authentication issues occur on the wrong version.</li>
<li>If you're using <a href="https://github.com/helm/helm" data-linktype="external">helm</a>, you need a minimum version of helm 3.3.</li>
<li>This configuration requires you have a Microsoft Entra group for your cluster. This group is registered as an admin group on the cluster to grant admin permissions. If you don't have an existing Microsoft Entra group, you can create one using the <a href="/en-us/cli/azure/ad/group#az_ad_group_create" data-linktype="absolute-path"><code>az ad group create</code></a> command.</li>
</ul>
<div class="NOTE">
<p>Note</p>
<p>Microsoft Entra integrated clusters using a Kubernetes version newer than version 1.24 automatically use the <code>kubelogin</code> format. Starting with Kubernetes version 1.24, the default format of the clusterUser credential for Microsoft Entra ID clusters is <code>exec</code>, which requires <a href="https://github.com/Azure/kubelogin" data-linktype="external"><code>kubelogin</code></a> binary in the execution PATH. There is no behavior change for non-Microsoft Entra clusters, or Microsoft Entra ID clusters running a version older than 1.24.
Existing downloaded <code>kubeconfig</code> continues to work. An optional query parameter <strong>format</strong> is included when getting clusterUser credential to overwrite the default behavior change. You can explicitly specify format to <strong>azure</strong> if you need to maintain the old <code>kubeconfig</code> format .</p>
</div>
<p><a name="enable-the-integration-on-your-aks-cluster"></a></p>
<h2 id="enable-the-integration-on-your-aks-cluster">Enable the integration on your AKS cluster</h2>
<h3 id="create-a-new-cluster">Create a new cluster</h3>
<ol>
<li><p>Create an Azure resource group using the <a href="/en-us/cli/azure/group#az_group_create" data-linktype="absolute-path"><code>az group create</code></a> command.</p>
<pre><code class="lang-azurecli" data-interactive="azurecli">az group create --name myResourceGroup --location centralus
</code></pre>
</li>
<li><p>Create an AKS cluster and enable administration access for your Microsoft Entra group using the <a href="/en-us/cli/azure/aks#az_aks_create" data-linktype="absolute-path"><code>az aks create</code></a> command.</p>
<pre><code class="lang-azurecli" data-interactive="azurecli">az aks create \
    --resource-group myResourceGroup \
    --name myManagedCluster \
    --enable-aad \
    --aad-admin-group-object-ids &lt;id&gt; [--aad-tenant-id &lt;id&gt;] \
    --generate-ssh-keys
</code></pre>
<p>A successful creation of an AKS-managed Microsoft Entra ID cluster has the following section in the response body:</p>
<pre><code class="lang-output">"AADProfile": {
    "adminGroupObjectIds": [
    "5d24****-****-****-****-****afa27aed"
    ],
    "clientAppId": null,
    "managed": true,
    "serverAppId": null,
    "serverAppSecret": null,
    "tenantId": "72f9****-****-****-****-****d011db47"
}
</code></pre>
</li>
</ol>
<h3 id="use-an-existing-cluster">Use an existing cluster</h3>
<p>Enable AKS-managed Microsoft Entra integration on your existing Kubernetes RBAC enabled cluster using the <a href="/en-us/cli/azure/aks#az_aks_update" data-linktype="absolute-path"><code>az aks update</code></a> command. Make sure to set your admin group to keep access on your cluster.</p>
<pre><code class="lang-azurecli" data-interactive="azurecli">az aks update --resource-group MyResourceGroup --name myManagedCluster --enable-aad --aad-admin-group-object-ids &lt;id-1&gt;,&lt;id-2&gt; [--aad-tenant-id &lt;id&gt;]
</code></pre>
<p>A successful activation of an AKS-managed Microsoft Entra ID cluster has the following section in the response body:</p>
<pre><code class="lang-output">"AADProfile": {
    "adminGroupObjectIds": [
        "5d24****-****-****-****-****afa27aed"
    ],
    "clientAppId": null,
    "managed": true,
    "serverAppId": null,
    "serverAppSecret": null,
    "tenantId": "72f9****-****-****-****-****d011db47"
    }
</code></pre>
<p><a name="migrate-a-legacy-azure-ad-cluster-to-integration"></a></p>
<h3 id="migrate-legacy-cluster-to-integration">Migrate legacy cluster to integration</h3>
<p>If your cluster uses legacy Microsoft Entra integration, you can upgrade to AKS-managed Microsoft Entra integration through the <a href="/en-us/cli/azure/aks#az_aks_update" data-linktype="absolute-path"><code>az aks update</code></a> command.</p>
<div class="WARNING">
<p>Warning</p>
<p>Free tier clusters may experience API server downtime during the upgrade. We recommend upgrading during your nonbusiness hours.
After the upgrade, the kubeconfig content changes. You need to run <code>az aks get-credentials --resource-group &lt;AKS resource group name&gt; --name &lt;AKS cluster name&gt;</code> to merge the new credentials into the kubeconfig file.</p>
</div>
<pre><code class="lang-azurecli" data-interactive="azurecli">az aks update --resource-group myResourceGroup --name myManagedCluster --enable-aad --aad-admin-group-object-ids &lt;id&gt; [--aad-tenant-id &lt;id&gt;]
</code></pre>
<p>A successful migration of an AKS-managed Microsoft Entra ID cluster has the following section in the response body:</p>
<pre><code class="lang-output">"AADProfile": {
    "adminGroupObjectIds": [
        "5d24****-****-****-****-****afa27aed"
    ],
    "clientAppId": null,
    "managed": true,
    "serverAppId": null,
    "serverAppSecret": null,
    "tenantId": "72f9****-****-****-****-****d011db47"
    }
</code></pre>
<p><a name="access-your-enabled-cluster"></a></p>
<h2 id="access-your-enabled-cluster">Access your enabled cluster</h2>
<ol>
<li><p>Get the user credentials to access your cluster using the <a href="/en-us/cli/azure/aks#az_aks_get_credentials" data-linktype="absolute-path"><code>az aks get-credentials</code></a> command.</p>
<pre><code class="lang-azurecli" data-interactive="azurecli">az aks get-credentials --resource-group myResourceGroup --name myManagedCluster
</code></pre>
</li>
<li><p>Follow your instructions to sign in.</p>
</li>
<li><p>Set <code>kubelogin</code> to use the Azure CLI.</p>
<pre><code class="lang-azurecli" data-interactive="azurecli">kubelogin convert-kubeconfig -l azurecli
</code></pre>
</li>
<li><p>View the nodes in the cluster with the <code>kubectl get nodes</code> command.</p>
<pre><code class="lang-azurecli" data-interactive="azurecli">kubectl get nodes
</code></pre>
</li>
</ol>
<h2 id="non-interactive-sign-in-with-kubelogin">Non-interactive sign-in with kubelogin</h2>
<p>There are some non-interactive scenarios that don't support <code>kubectl</code>. In these cases, use <a href="https://github.com/Azure/kubelogin" data-linktype="external"><code>kubelogin</code></a> to connect to the cluster with a non-interactive service principal credential to perform continuous integration pipelines.</p>
<div class="NOTE">
<p>Note</p>
<p>Microsoft Entra integrated clusters using a Kubernetes version newer than version 1.24 automatically use the <code>kubelogin</code> format. Starting with Kubernetes version 1.24, the default format of the clusterUser credential for Microsoft Entra ID clusters is <code>exec</code>, which requires <a href="https://github.com/Azure/kubelogin" data-linktype="external"><code>kubelogin</code></a> binary in the execution PATH. There is no behavior change for non-Microsoft Entra clusters, or Microsoft Entra ID clusters running a version older than 1.24.
Existing downloaded <code>kubeconfig</code> continues to work. An optional query parameter <strong>format</strong> is included when getting clusterUser credential to overwrite the default behavior change. You can explicitly specify format to <strong>azure</strong> if you need to maintain the old <code>kubeconfig</code> format .</p>
</div>
<ul>
<li><p>When getting the clusterUser credential, you can use the <code>format</code> query parameter to overwrite the default behavior. You can set the value to <code>azure</code> to use the original kubeconfig format:</p>
<pre><code class="lang-azurecli" data-interactive="azurecli">az aks get-credentials --format azure
</code></pre>
</li>
<li><p>If your Microsoft Entra integrated cluster uses Kubernetes version 1.24 or lower, you need to manually convert the kubeconfig format.</p>
<pre><code class="lang-azurecli" data-interactive="azurecli">export KUBECONFIG=/path/to/kubeconfig
kubelogin convert-kubeconfig
</code></pre>
</li>
</ul>
<div class="NOTE">
<p>Note</p>
<p>If you receive the message <strong>error: The Azure auth plugin has been removed.</strong>, you need to run the command <code>kubelogin convert-kubeconfig</code> to convert the kubeconfig format manually.</p>
<p>For more information, you can refer to <a href="https://azure.github.io/kubelogin/known-issues.html" data-linktype="external">Azure Kubelogin Known Issues</a>.</p>
</div>
<p><a name="troubleshoot-access-issues"></a></p>
<h2 id="troubleshoot-access-issues">Troubleshoot access issues</h2>
<div class="IMPORTANT">
<p>Important</p>
<p>The step described in this section suggests an alternative authentication method compared to the normal Microsoft Entra group authentication. Use this option only in an emergency.</p>
</div>
<p>If you lack administrative access to a valid Microsoft Entra group, you can follow this workaround. Sign in with an account that is a member of the <a href="/en-us/azure/role-based-access-control/built-in-roles#azure-kubernetes-service-cluster-admin-role" data-linktype="absolute-path">Azure Kubernetes Service Cluster Admin</a> role and grant your group or tenant admin credentials to access your cluster.</p>
