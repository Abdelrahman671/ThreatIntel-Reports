<section class="entry-content clearfix">
<p>During <span><a href="https://jsac.jpcert.or.jp/archive/2021/en/index.html">JSAC2021</a></span> on 28 January 2021, there was a presentation about an attack group LuoYu, which targets Korean and Japanese organisations since 2014 <a href="#1">[1]</a><a href="#2">[2]</a>. Recently, JPCERT/CC came across malware WinDealer used by this group. This article introduces some findings of our analysis.</p><h3>Malware WinDealer overview</h3><p>WinDealer steals information of an infected PC and sends it to a C2 server as described in Figure 1.</p><p><img alt="" class="asset asset-image mt-image-center" height="414" src="https://blogs.jpcert.or.jp/en/.assets/thumbnail/windealer01_png_en-800wri.png" style="max-width:100%;height:auto;display:block;margin-left:auto;margin-right:auto" width="800"/></p><p style="text-align: center;">Figure 1： Malware WinDealer behaviour overview</p><p><br/>Once launched, the malware reads configuration from a file under C:\ProgramData and loads a DLL module on its memory. It steals information about the victim PC, network configuration and SNS applications etc. and saves them in a file with an “.a” extension under %TEMP%, which is then sent to a C2 server.<br/>The following points will be described in the next sections.</p><ul>
<li>Read configuration</li>
<li>Communicate with C2 servers</li>
<li>Process and send stolen data</li>
<li>Functions of modules loaded on memory</li>
</ul><h3>Read configuration</h3><p>The malware stores its configuration in several folders under C:\ProgramData and reads it when executed. The contents are encoded based on XOR with its key value “b6a7%7486”. Please refer to Appendix A for the configuration file path and its contents. Figure 2 shows a function to decode configuration.</p><p><img alt="" class="asset asset-image mt-image-center" height="" src="https://blogs.jpcert.or.jp/en/.assets/windealer2.svg" style="max-width:100%;height:auto;display:block;margin-left:auto;margin-right:auto" width="450"/></p><p style="text-align: center;">Figure 2：Function to decode a file storing configuration</p><h3><br/>Communicate with C2 servers</h3><p>If the following configuration files exist in the designated folder, WinDealer loads the C2 server information from them and starts communicating<span></span>.</p><ul>
<li>C:\ProgramData\ad5f82e8</li>
<li>C:\ProgramData\1c76cbfe</li>
<li>C:\ProgramData\9c3b6294</li>
</ul><p>If no such file exists, WinDealer communicates to a random IP address in one of the following ranges (port 6999/UDP or <span>55556/TCP</span>). It switches to an IP address in the other range at a certain interval.</p><ul>
<li>113.62.0.0 - 113.63.255.255</li>
<li>111.120.0.0 - 111.123.255.255</li>
</ul><p>Figure 3 shows the malware’s communication flow with its C2 server. First, it encrypts an AES key with RSA algorithm and sends to a C2 server. Information stolen from a victim PC is encrypted with this AES key and sent to a C2 server at a certain interval. After that, C2 server sends a command to the victim PC. The malware executes it and sends the result to the C2 server after encryption. Besides the data exchange, the malware also communicates with domains such as www[.]microsoftcom (non-existent at the moment) and icanhazip[.]com.</p><p><img alt="" class="asset asset-image mt-image-center" height="" src="https://blogs.jpcert.or.jp/en/.assets/windealer3en.svg" style="max-width:100%;height:auto;display:block;margin-left:auto;margin-right:auto" width="800"/></p><p style="text-align: center;">Figure 3: Communication flow with a C2 server</p><p><br/>Figure 4 describes the communication contents when delivering an AES key. AES key and its CRC32 checksum value are encrypted with RSA1024bit public key. The public key is hardcoded in the sample, which is also used for other samples as well.</p><p><img alt="" class="asset asset-image mt-image-center" height="" src="https://blogs.jpcert.or.jp/en/.assets/windealer4en.svg" style="max-width:100%;height:auto;display:block;margin-left:auto;margin-right:auto" width="800"/></p><p style="text-align: center;">Figure 4： Example of contents sent with AES key</p><p><br/>From the second round of communication and onwards, data is encrypted in AES128bit ECB mode based on the AES key which was dynamically generated during the initial communication. Please refer to Appendix B for the details of data format.</p><h3>Process and send stolen data</h3><p>WinDealer processes a series of stolen data as “.a” file in a folder under %TEMP%, encrypts it with AES and send it to a C2 server. The flow of event is illustrated in Figure 5. The modules steal and process the data, while WinDealer itself monitors the files under %TEMP%, encrypts the file and sends it to a C2 server.</p><p><img alt="" class="asset asset-image mt-image-center" height="" src="https://blogs.jpcert.or.jp/en/.assets/windealer5en.svg" style="max-width:100%;height:auto;display:block;margin-left:auto;margin-right:auto" width="520"/></p><p style="text-align: center;">Figure 5：Flow of events by WinDealer and modules</p><p><br/>A part of the code for generating “.a” file by a module is as follows (Figure 6):</p><p><img alt="" class="asset asset-image mt-image-center" height="" src="https://blogs.jpcert.or.jp/en/.assets/windealer6.svg" style="max-width:100%;height:auto;display:block;margin-left:auto;margin-right:auto" width="640"/></p><p style="text-align: center;">Figure 6： DLL module’s code to generate “.a” file</p><p><br/>The stolen data is first stored in a file with an “.t” extension, which is then renamed to “.a”. The series of data is stored in different directories based on the data category, and they are taken out when “.a” file is created. Please refer to Appendix E for the details of each directory.<br/>Before writing and reading the files, the data is encoded/decoded by XOR-based function with its key value “YYYY” as in Figure 7.</p><p><img alt="" class="asset asset-image mt-image-center" height="" src="https://blogs.jpcert.or.jp/en/.assets/windealer7.svg" style="max-width:100%;height:auto;display:block;margin-left:auto;margin-right:auto" width="450"/></p><p style="text-align: center;">Figure 7：A function for XOR-based encoding when accessing “.a” file</p><h3><br/>Functions of modules loaded on memory</h3><p>Once launched, WinDealer loads a DLL module in a PE format (encoded in the sample) on the memory and executes it (Figure 8).</p><p><img alt="" class="asset asset-image mt-image-center" height="" src="https://blogs.jpcert.or.jp/en/.assets/windealer8en.svg" style="max-width:100%;height:auto;display:block;margin-left:auto;margin-right:auto" width="640"/></p><p style="text-align: center;">Figure 8： Encoded module data</p><p><br/>The malware obtains its file path, searches for a byte string “0xFF3456FF00” and extracts data from its offset 0xE. Using its offset 0x4 value and a XOR-based decode routine (Figure 9), a DLL module is loaded on the memory and then executed.</p><p><img alt="" class="asset asset-image mt-image-center" height="" src="https://blogs.jpcert.or.jp/en/.assets/windealer9.svg" style="max-width:100%;height:auto;display:block;margin-left:auto;margin-right:auto" width="640"/></p><p style="text-align: center;">Figure 9： Decoding module</p><p><br/>The loaded DLL module is named as “MozillaDll.dll”. There are 3 Export functions as follows:</p><ul>
<li>AutoGetSystemInfo: Steal data</li>
<li>GetConfigInfo: Set configuration</li>
<li>partInitOpt: Set commands</li>
</ul><p>The loaded DLL module monitors the below items, saves related items in a separate file and obtains them to send out to a C2 server.</p><ul>
<li>Files stored in a USB memory</li>
<li>Files under Documents, desktop and recycle bin</li>
<li>Files under folders related to SNS applications</li>
</ul><p>Please see Appendix D for the details of commands that C2 server sends and its contents.</p><h3>In closing</h3><p>Besides WinDealer, it has been confirmed that LuoYu uses other kinds of malware that operate in various platforms. We will report if we observe a new type of malware.<br/>For your reference, SHA256 hash values of similar samples are listed in Appendix F.</p><p><br/>- Yuma Masubuchi<br/>(Translated by Yukako Uchida)</p><h2>Reference</h2><p><a name="1"></a>[1] “LuoYu” The eavesdropper sneaking in multiple platforms<br/><span><a href="https://jsac.jpcert.or.jp/archive/2021/pdf/JSAC2021_301_shui-leon_en.pdf">https://jsac.jpcert.or.jp/archive/2021/pdf/JSAC2021_301_shui-leon_en.pdf</a></span></p><p><a name="2"></a>[2] Japan Security Analyst Conference 2021 -3rd Track-<br/><a href="https://blogs.jpcert.or.jp/en/2021/02/jsac2021report1.html"><span>https://blogs.jpcert.or.jp/en/2021/02/jsac2021report1.html</span></a></p><h2>Appendix A WinDealer configuration</h2><table border="1" data-mce-border="1" style="width: 100%; border-width: 1px; border-collapse: collapse; height: 299px;"><caption>Table A：List of configuration</caption>
