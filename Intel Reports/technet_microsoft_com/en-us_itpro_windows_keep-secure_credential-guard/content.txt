										</nav><!-- <content> --><p>Credential Guard prevents credential theft attacks by protecting NTLM password hashes, Kerberos Ticket Granting Tickets (TGTs), and credentials stored by applications as domain credentials.</p>
<p>Credential Guard uses <a href="/en-us/windows-hardware/design/device-experiences/oem-vbs" data-linktype="absolute-path">Virtualization-based security (VBS)</a> to isolate secrets so that only privileged system software can access them. Unauthorized access to these secrets can lead to credential theft attacks like <em>pass the hash</em> and <em>pass the ticket</em>.</p>
<p>When enabled, Credential Guard provides the following benefits:</p>
<ul>
<li><strong>Hardware security</strong>: NTLM, Kerberos, and Credential Manager take advantage of platform security features, including Secure Boot and virtualization, to protect credentials</li>
<li><strong>Virtualization-based security</strong>: NTLM, Kerberos derived credentials, and other secrets run in a protected environment that is isolated from the running operating system</li>
<li><strong>Protection against advanced persistent threats</strong>: when credentials are protected using VBS, the credential theft attack techniques and tools used in many targeted attacks are blocked. Malware running in the operating system with administrative privileges can't extract secrets that are protected by VBS</li>
</ul>
<div class="NOTE">
<p>Note</p>
<p>While Credential Guard is a powerful mitigation, persistent threat attacks will likely shift to new attack techniques, and you should also incorporate other security strategies and architectures.</p>
</div>
<h2 id="default-enablement">Default enablement</h2>
<div class="IMPORTANT">
<p>Important</p>
<p>Windows Server 2025 is in preview. This information relates to a prerelease product that may be substantially modified before it's released. Microsoft makes no warranties, expressed or implied, with respect to the information provided here.</p>
</div>
<p>Starting in <strong>Windows 11, 22H2</strong> and <strong>Windows Server 2025 (preview)</strong>, VBS and Credential Guard are enabled by default on devices that meet the requirements.</p>
<p>The default enablement is <strong>without UEFI Lock</strong>, thus allowing administrators to disable Credential Guard remotely if needed.</p>
<p>When Credential Guard is enabled, <a href="#system-requirements" data-linktype="self-bookmark">VBS</a> is automatically enabled too.</p>
<div class="NOTE">
<p>Note</p>
<p>If Credential Guard is explicitly <a href="configure#disable-credential-guard" data-linktype="relative-path">disabled</a> <em>before</em> a device is updated to Windows 11, version 22H2 / Windows Server 2025 (preview) or later, default enablement does not overwrite the existing settings. That device will continue to have Credential Guard disabled even after updating to a version of Windows that enables Credential Guard by default.</p>
</div>
<h3 id="default-enablement-on-windows">Default enablement on Windows</h3>
<p>Devices running Windows 11, 22H2 or later have Credential Guard enabled by default if they:</p>
<ul>
<li>Meet the <a href="#windows-edition-and-licensing-requirements" data-linktype="self-bookmark">license requirements</a></li>
<li>Meet the <a href="#system-requirements" data-linktype="self-bookmark">hardware and software requirements</a></li>
<li>Aren't <a href="configure#default-enablement" data-linktype="relative-path">explicitly configured to disable Credential Guard</a></li>
</ul>
<div class="NOTE">
<p>Note</p>
<p>Devices running Windows 11 Pro/Pro Edu 22H2 or later may have Virtualization-based Security (VBS) and/or Credential Guard automatically enabled if they meet the other requirements for default enablement, and have previously run Credential Guard. For example if Credential Guard was enabled on an Enterprise device that later downgraded to Pro.</p>
<p>To determine whether the Pro device is in this state, check if the following registry key exists: <code>Computer\HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Lsa\MSV1_0\IsolatedCredentialsRootSecret</code>. In this scenario, if you wish to disable VBS and Credential Guard, follow the instructions to <a href="configure#disable-virtualization-based-security" data-linktype="relative-path">disable Virtualization-based Security</a>. If you wish to disable Credential Guard only, without disabling VBS, use the procedures to <a href="configure#disable-credential-guard" data-linktype="relative-path">disable Credential Guard</a>.</p>
</div>
<h3 id="default-enablement-on-windows-server">Default enablement on Windows Server</h3>
<p>Devices running Windows Server 2025 (preview) or later have Credential Guard enabled by default if they:</p>
<ul>
<li>Meet the <a href="#windows-edition-and-licensing-requirements" data-linktype="self-bookmark">license requirements</a></li>
<li>Meet the <a href="#system-requirements" data-linktype="self-bookmark">hardware and software requirements</a></li>
<li>Aren't <a href="configure#default-enablement" data-linktype="relative-path">explicitly configured to disable Credential Guard</a></li>
<li>Are joined to a domain</li>
<li>Aren't a domain controller</li>
</ul>
<div class="IMPORTANT">
<p>Important</p>
<p>For information about known issues related to default enablement, see <a href="considerations-known-issues#known-issues" data-linktype="relative-path">Credential Guard: known issues</a>.</p>
</div>
<h2 id="system-requirements">System requirements</h2>
<p>For Credential Guard to provide protection, the device must meet certain hardware, firmware, and software requirements.</p>
<p>Devices that exceed the minimum hardware and firmware qualifications receive additional protections and are more hardened against certain threats.</p>
<h3 id="hardware-and-software-requirements">Hardware and software requirements</h3>
<p>Credential Guard requires the features:</p>
<ul>
<li>Virtualization-based security (VBS)
<div class="NOTE">
<p>Note</p>
<p>VBS has different requirements to enable it on different hardware platforms. For more information, see <a href="/en-us/windows-hardware/design/device-experiences/oem-vbs" data-linktype="absolute-path">Virtualization-based Security requirements</a></p>
</div>
</li>
<li><a href="../../operating-system-security/system-security/secure-the-windows-10-boot-process#secure-boot" data-linktype="relative-path">Secure Boot</a></li>
</ul>
<p>While not required, the following features are recommended to provide additional protections:</p>
<ul>
<li>Trusted Platform Module (TPM), as it provides binding to hardware. TPM versions 1.2 and 2.0 are supported, either discrete or firmware</li>
<li>UEFI lock, as it prevents attackers from disabling Credential Guard with a registry key change</li>
</ul>
<p>For detailed information on protections for improved security that are associated with hardware and firmware options, see <a href="additional-mitigations#additional-security-qualifications" data-linktype="relative-path">additional security qualifications</a>.</p>
<h4 id="credential-guard-in-virtual-machines">Credential Guard in virtual machines</h4>
<p>Credential Guard can protect secrets in Hyper-V virtual machines, just as it would on a physical machine. When Credential Guard is enabled on a VM, secrets are protected from attacks <em>inside</em> the VM. Credential Guard doesn't provide protection from privileged system attacks originating from the host.</p>
<p>The requirements to run Credential Guard in Hyper-V virtual machines are:</p>
<ul>
<li>The Hyper-V host must have an IOMMU</li>
<li>The Hyper-V virtual machine must be generation 2</li>
</ul>
<div class="NOTE">
<p>Note</p>
<p>Credential Guard is not supported on Hyper-V or Azure generation 1 VMs. Credential Guard is available on generation 2 VMs only.</p>
</div>

<h2 id="application-requirements">Application requirements</h2>
<p>When Credential Guard is enabled, certain authentication capabilities are blocked. Applications that require such capabilities break. We refer to these requirements as <em>application requirements</em>.</p>
<p>Applications should be tested before deployment to ensure compatibility with the reduced functionality.</p>
<div class="WARNING">
<p>Warning</p>
<p>Enabling Credential Guard on domain controllers isn't recommended.
Credential Guard doesn't provide any added security to domain controllers, and can cause application compatibility issues on domain controllers.</p>
</div>
<div class="NOTE">
<p>Note</p>
<p>Credential Guard doesn't provide protections for the Active Directory database or the Security Accounts Manager (SAM). The credentials protected by Kerberos and NTLM when Credential Guard is enabled are also in the Active Directory database (on domain controllers) and the SAM (for local accounts).</p>
</div>
<p>Applications break if they require:</p>
<ul>
<li>Kerberos DES encryption support</li>
<li>Kerberos unconstrained delegation</li>
<li>Kerberos TGT extraction</li>
<li>NTLMv1</li>
</ul>
<p>Applications ask and expose credentials to risk if they require:</p>
<ul>
<li>Digest authentication</li>
<li>Credential delegation</li>
<li>MS-CHAPv2</li>
<li>CredSSP</li>
</ul>
<p>Applications might cause performance issues when they attempt to hook the isolated Credential Guard process <code>LSAIso.exe</code>.</p>
<p>Services or protocols that rely on Kerberos, such as file shares or remote desktop, continue to work and aren't affected by Credential Guard.</p>
