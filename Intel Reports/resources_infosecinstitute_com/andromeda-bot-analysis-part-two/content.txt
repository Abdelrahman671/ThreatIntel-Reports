                    <h1>Andromeda Bot Analysis part 2</h1>
                    <div class="author-share">
                        <div class="author-share__details">
                            <div class="author-share__details-image">

                        <div id="article-contents">
                            
<p><h1>Bot Analysis:</h1></p>
<p>Now, you get the original Andromeda build file. Load the unpacked sample at OllyDBG. As before, after the stack frame at the EP, you see that the malware is looking to load API's address using the <span style="background-color: white;">PEB_LDR_DATA structure</span>, but this time instead of kernel32.dll; the malware try to find ntdll.dll base address, then, it will parse the EAT, hash each APIs then make comparison to find the needed APIs :</p>
<p><img src="/globalassets/wysiwyg-images-folder/092415_0000_andromedabo1.png" alt="" /></p>



<article class="azone-type-4 ">
        <div class="azone__image">
<a href="/skills/learning-paths/reverse-engineering/?utm_source=resources&amp;utm_medium=infosec+network&amp;utm_campaign=reverse+learning+path&amp;utm_content=main&amp;utm_term=reverse" target="_blank" title="" class="text-decoration-none">
                <img src="/contentassets/952a4dff68ad4342983ca10be748452e/is-ad-22-07-malware-analysis-resources-ad-1200x627-1.png" alt="Reverse Engineering Malware &#x2014; Main A" class="w-100" />

</a>        </div>
    <div class="azone__content">



    </div>
</article>
<p>After getting inside the CALL, it will calculate the hash of a buffer located at 00402028:</p>
<p>def calc_hash(string):</p>
<p>return binascii.crc32(string) &amp; 0xffffffff</p>
<p>Then, it compared it with 0xBD274BDB, if not matches, it calls RtlExitUserThread, we will figure out soon what kind of data has been hashed. Afterwards, ZwAllocateVirtualMemory will be called and return for me 7FFA0000 then the CALL at 00401343 will copy the whole buffer to the allocated space. Next, we see the CALL at VA 00401351 which takes 4 arguments and one of them is a pointer to our unknown buffer. This routine is actually preforming a RC4 encryption, you could recognize that by looking at the constants, this is how basically some cryptographic algorithm is detected; RC4 have loops that go till <span style="color: black;"><span style="background-color: white;">256</span> which is 0x100 in hexadecimal.<br />
</span></p>
<p>Using ZwAllocateVirtualMemory again, we allocate a new memory space to the partially decrypted payload so it is finally decompressed using the aPLib-library. The code that follows is responsible for processing relocations and fixing imports. For instance, from NTDLL Andromeda is importing these APIs:</p>
<p>LdrLoadDll, RtlDosPathNameToNtPathName_U, RtlFreeUnicodeString, LdrProcessRelocationBlock, RtlComputeCrc32, RtlWalkHeap, RtlImageNtHeader, RtlImageDirectoryEntryToData, RtlExitUserThread, ZwSetInformationProcess, ZwUnmapViewOfSection, ZwAllocateVirtualMemory, ZwMapViewOfSection, ZwFreeVirtualMemory, ZwOpenFile, ZwQueryDirectoryFile, ZwClose, ZwQueryInformationProcess.</p>
<p><img src="/globalassets/wysiwyg-images-folder/092415_0000_andromedabo2.png" alt="" /><em><br />
</em></p>

<p><span style="font-family: Courier New;">def calc_hash(string):<br />
</span></p>
<p><span style="font-family: Courier New;"> return binascii.crc32(string) &amp; 0xffffffff<br />

<p><span style="font-family: Courier New;">def rc4crypt(data, key):<br />
</span></p>
<p><span style="font-family: Courier New;"> x = 0<br />
</span></p>
<p><span style="font-family: Courier New;"> box = bytearray(range(256))<br />
</span></p>
<p><span style="font-family: Courier New;"> for i in range(256):<br />
</span></p>
<p><span style="font-family: Courier New;"> x = (x + box[i] + key[i % len(key)]) % 256<br />
</span></p>
<p><span style="font-family: Courier New;"> box[i], box[x] = box[x], box[i]<br />
</span></p>
<p><span style="font-family: Courier New;"> x,y = 0, 0<br />
</span></p>
<p><span style="font-family: Courier New;"> out = bytearray()<br />
</span></p>
<p><span style="font-family: Courier New;"> for byte in data:<br />
</span></p>
<p><span style="font-family: Courier New;"> x = (x + 1) % 256<br />
</span></p>
<p><span style="font-family: Courier New;"> y = (y + box[x]) % 256<br />
</span></p>
<p><span style="font-family: Courier New;"> box[x], box[y] = box[y], box[x]<br />
</span></p>
<p><span style="font-family: Courier New;"> out += bytearray([byte ^ box[(box[x] + box[y]) % 256]])<br />
</span></p>
<p><span style="font-family: Courier New;"> return out<br />

<p><span style="font-family: Courier New;">def fix_payload_relocs_and_import(segment, relocs_offset):<br />
</span></p>
<p><span style="font-family: Courier New;"> current_offset = 0<br />
</span></p>
<p><span style="font-family: Courier New;"> # processing relocations<br />
</span></p>
<p><span style="font-family: Courier New;"> while True:<br />
</span></p>
<p><span style="font-family: Courier New;"> base = Dword(segment + relocs_offset + current_offset)<br />
</span></p>
<p><span style="font-family: Courier New;"> size = Dword(segment + relocs_offset + current_offset + 4)<br />
</span></p>
<p><span style="font-family: Courier New;"> if (base == 0 and current_offset != 0) or size == 0:<br />
</span></p>
<p><span style="font-family: Courier New;"> current_offset += 4<br />
</span></p>
<p><span style="font-family: Courier New;"> break<br />
</span></p>
<p><span style="font-family: Courier New;"> current_offset += 8<br />
</span></p>
<p><span style="font-family: Courier New;"> size = (size - 8) // 2<br />
</span></p>
<p><span style="font-family: Courier New;"> for i in range(size):<br />
</span></p>

</span></p>

</span></p>

<p><span style="font-family: Courier New;"> current_offset += 2<br />
</span></p>
<p><span style="font-family: Courier New;"> # processing imports<br />

</span></p>
<p><span style="font-family: Courier New;"> module_hash = Dword(segment + relocs_offset + current_offset)<br />
</span></p>
<p><span style="font-family: Courier New;"> import_offset = Dword(segment + relocs_offset + current_offset + 4)<br />
</span></p>
<p><span style="font-family: Courier New;"> current_offset += 8<br />
</span></p>
<p><span style="font-family: Courier New;"> if module_hash == 0 or import_offset == 0:<br />

<p><span style="font-family: Courier New;"> break<br />

<p><span style="font-family: Courier New;"> module = None<br />
</span></p>
<p><span style="font-family: Courier New;"> for library in iter(IMPORTS):<br />
</span></p>
<p><span style="font-family: Courier New;"> if module_hash == calc_hash(library.lower()):<br />

<p><span style="font-family: Courier New;"> current_offset += 4<br />

<p><span style="font-family: Courier New;">def decrypt_payload(encrypted_addr, rc4key, encrypted_size, unpacked_size, entry_point, relocs, relocs_size):<br />
</span></p>
<p><span style="font-family: Courier New;"> buffer = bytearray(encrypted_size)<br />
</span></p>
<p><span style="font-family: Courier New;"> for i in range(len(buffer)):<br />
</span></p>
<p><span style="font-family: Courier New;"> buffer[i] = Byte(encrypted_addr + i)<br />
</span></p>
<p><span style="font-family: Courier New;"> decrypted = rc4crypt(buffer, rc4key)<br />
</span></p>
<p><span style="font-family: Courier New;"> unpacked = decompress(str(decrypted)).do()<br />
</span></p>
<p><span style="font-family: Courier New;"> # checking for free segment address<br />
</span></p>
<p><span style="font-family: Courier New;"> seg_start = 0x10000000<br />
</span></p>
<p><span style="font-family: Courier New;"> while SegName(seg_start) != '':<br />
</span></p>
<p><span style="font-family: Courier New;"> seg_start += 0x10000000<br />
</span></p>
<p><span style="font-family: Courier New;"> AddSeg(seg_start, seg_start + unpacked_size, 0, 1, idaapi.saRelPara, idaapi.scPub)<br />
</span></p>
<p><span style="font-family: Courier New;"> # copying data to new segment<br />
</span></p>
<p><span style="font-family: Courier New;"> data = unpacked[0]<br />
</span></p>
<p><span style="font-family: Courier New;"> for i in range(len(data)):<br />
</span></p>
<p><span style="font-family: Courier New;"> PatchByte(seg_start + i, ord(data[i]))<br />
</span></p>
<p><span style="font-family: Courier New;"> fix_payload_relocs_and_import(seg_start, relocs)<br />
</span></p>
<p><span style="font-family: Courier New;"> MakeFunction(seg_start + entry_point)<br />

<p><span style="font-family: Courier New;">def main():<br />
</span></p>
<p><span style="font-family: Courier New;"> payload_addr = AskAddr(ScreenEA(), "Enter address of andromeda payload")<br />
</span></p>
<p><span style="font-family: Courier New;"> if payload_addr != idaapi.BADADDR and payload_addr is not None:<br />

<p><span style="font-family: Courier New;"> for i in range(len(payload)):<br />

<p><span style="font-family: Courier New;"> payload[i] = Byte(payload_addr + i)<br />

</span></p>
<p><span style="font-family: Courier New;"> decrypt_payload(payload_addr + 0x28, payload[:16], dwords[0], dwords[2], dwords[3], dwords[4], dwords[5])<br />

<p><span style="font-family: Courier New;">if __name__ == '__main__':<br />
</span></p>
<p><span style="font-family: Courier New;"> main()<br />
</span></p>
<p>At the end, you see the call to: 00401532 |. FFD0 CALL EAX</p>
<p>This will transfer the control to the payload. Here is a screenshot about the payload decrypted.</p>
<p><img src="/globalassets/wysiwyg-images-folder/092415_0000_andromedabo3.png" alt="" /></p>
<p>The next step shows anti-analysis tricks that are employed. The call at VA 7FF91408 is iterating through process names and computing their CRC32 hash values: if a hash value matches any of those on a list of hash values of VM processes and monitoring tools like wireshark.exe, etc., this indicates that the debugging process is inside a sandbox environment or being monitored.</p>
<p><img src="/globalassets/wysiwyg-images-folder/092415_0000_andromedabo4.png" alt="" /><span style="color: #37424a;"><br />
</span></p>
<p>Furthermore, this trick is not changed. As in version 2.07 and 2.08, the 2.09 version continues to calculate the CRC32 hash of the volume name of drive C:, which is then compared with the hardcoded value 0x20C7DD84. If you get caught, you will run in infinite loop that call ZwDelayExecution ! just patch the JNZ after the call or put RET in ZwDelayExecution.</p>
<p style="text-align: center;"><img src="/globalassets/wysiwyg-images-folder/092415_0000_andromedabo5.png" alt="" /></p>
<p>After that I think that the CALL at VA 7FF91420 is trying to setup a KiFastSystemCall hook, this API is the lowest level API available in the "usermode" layer aka Ring3, all application' calls pass from KiFastSystemCall, which redirects all those controls onto the Windows Kernel via an instruction called SYSENTER.</p>
<p>Next, because processes run by the user can't do everything like writing in explorer.exe memory, the malware is trying to use SeDebugPrivilege and calling ZwAdjustTokenPrivilege to escalate to System privileges. It calls the SetEnvironmentVariableW API to save the original bot's full path to the environment variable. Afterwards, it comes the injection process, depending if you have a 32 or a 64 bits operating system, the malware will launch a hallowed version of msiexe.exe and inject its code there:</p>
<p><h1>Code Injection:</h1></p>
<p>The injection process involves several steps:</p>
<p>As with the previous versions, the malware calls CreateFile to get the handle of the file it wants to inject. It then gets its section handle by calling ZwCreateSection, which is used by ZwMapViewOfSection to get the image of the file in memory. From this image, it extracts the size of image and the address of the entry point from the PE header.</p>
<p>A memory address with the same size as that of the image of the file that it wants to inject is created with PAGE_EXECUTE_READWRITE access. Then the image of the file is copied over to this memory address.</p>
<p>Another memory address is created with the same size as that of the image of the original bot file, also with PAGE_EXECUTE_READWRITE access. The original file is then copied over to this new memory address.</p>
<p>A suspended process of the file to be injected is created. The memory address containing the original file is unmapped. ZwMapViewOfSection is called with the bot's file handle and the process handle (acquired from creating the suspended file process). So now the injected file's process handle has a map view of the botnet file. The final step is the call to ZwResumeThread, which resume the process.</p>
<p><img src="/globalassets/wysiwyg-images-folder/092415_0000_andromedabo6.png" alt="" /></p>
<p>If the User is an admin, it checks that with CheckTokenMemberShip, it installes into "%ALLUSERPROFILE%" and autostarts using an uncommon Registry Path "softwaremicrosoftwindowscurrentversionPoliciesExplorerRun" - with a random Key name. If not it only installs into "%USERPROFILE%".</p>
<p><img src="/globalassets/wysiwyg-images-folder/092415_0000_andromedabo7.png" alt="" /></p>
<p><h1>CnC Communication:</h1></p>
<p>Before establishing a connection, the bot prepares the message to be sent to the C&amp;C server. It uses the following format: <strong>id:%lu|bid:%lu|os:%lu|la:%lu|rg:%lu<br />
</strong></p>
<p><img src="/globalassets/wysiwyg-images-folder/092415_0000_andromedabo8.png" alt="" /></p>
<p>This string is encrypted using RC4 with a hard-coded key of length 0x20 and is further encoded using base64. The message is then sent to the server. Once a message is received, the bot calculates the CRC32 hash of the message without including the first DWORD. If the calculated hash matches the first DWORD, the message is valid. Later it is decrypted using RC4 with the VolumeSerialNumber as the key. After the RC4 decryption the message is in the format gn([base64-encoded string]). This used to be just the base64-encoded string, but for some reason the author decided not to make the server backward compatible with the older bot versions. Then it decodes the base64 string inside the brackets to get the message in plain text.</p>
<p><img src="/globalassets/wysiwyg-images-folder/092415_0000_andromedabo9.png" alt="" /></p>
<p>The first DWORD of the message is used as a multiplier to multiply a value in a fixed offset. The DWORD in that offset is used as an interval to delay calling the thread again to establish another connection. The next byte indicates what action to carry out - there are seven options:</p>
<ul>
<li>Case 1 (download EXE): Connect to the domain decrypted from the message to download an EXE file. Save the file to the %tmp% location with a random name and run the process.</li>
<li>Case 2 (load plug-ins): Connect to the domain decrypted from the message, install and load plug-ins. The plug-ins are decrypted by RC4 using the same key of length 0x20h.</li>
<li>Case 3 (update case): Connect to the domain to get the update EXE file. If a file name of VolumeSerialNumber is present in theregistry, then save the PE file to the %tmp% location with a random name; else save it to the current location with the name of the file as VolumeSerialNumber. The file in %tmp% is run, while the current process terminates. It also sends the message 'kill' xor'ed by VolumeSerialNumber to terminate the older process.</li>
<li>Case 4 (download DLL): Connect to the domain and save the DLL file to the %alluserprofile% location. The file is saved as a .dat file with a random name and loaded from a specified export function. The registry is modified so it can be auto-loaded by the bot.</li>
<li>Case 5 (delete DLLs): Delete and uninstall all the DLLs loaded and installed in Case 4.</li>
<li>Case 6 (delete plug-ins): Uninstall all the plug-ins loaded in Case 3.</li>
<li>Case 7 (uninstall bot): Suspend all threads and uninstall the bot.</li>
<li>After executing the action based on which instruction it received, another message is sent to the server to notify it that the action has been completed:</li>
</ul>
<p><strong> id:%lu|tid:%lu|res:%lu<br />
</strong></p>
<ul>
<li><strong>id</strong> is the VolumeSerialNumber</li>
<li><strong>tid</strong> is the next byte (task id) after the byte displaying the case number in the message received</li>
<li><strong>res</strong> is the result of whether or not the task was carried out successfully.</li>
</ul>
<p>Once the message has been sent, the thread exits and waits for the delay interval period to pass before it reconnects to the server to receive additional instructions.</p>
<p><h1>Conclusion:</h1></p>
<p>Andromeda's current version 2.09 increased the barriers that it has set up for security researchers. The new features raise additional difficulty for analysis, but are still easy to skip.</p>
