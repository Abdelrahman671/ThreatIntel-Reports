        <h2 id="ID0EFL" class="">Group Policy Preferences </h2>
        
        
          <section aria-labelledby="ID0EDBBL" class="ocpSection">
            <h3 id="ID0EDBBL" class="">Overview</h3>
            
              <p>
                <br aria-hidden="true">Some Group Policy Preferences can store a password. This functionality is being removed because the password was stored insecurely. This article describes the user interface changes and any available workarounds.<br aria-hidden="true"><br aria-hidden="true">The following Group Policy Preferences will no longer allow user names and passwords to be saved:</p>
              <ul>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>Drive Maps</p>
                <meta itemprop="position" content="1">
      </li>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>Local Users and Groups</p>
                <meta itemprop="position" content="2">
      </li>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>Scheduled Tasks</p>
                <meta itemprop="position" content="3">
      </li>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>Services</p>
                <meta itemprop="position" content="4">
      </li>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>Data Sources</p>
                <meta itemprop="position" content="5">
      </li>
    </ul>
              <p>This will affect the behavior of any existing Group Policy Objects (GPOs) in your environment that rely on passwords that are contained in these preferences. It will also prevent creating new Group Policy Preferences by using this functionality.<br aria-hidden="true"><br aria-hidden="true">For Drive Maps, Local Users and Groups, and Services, you may be able to achieve similar goals through other, more secure functionality in Windows.<br aria-hidden="true"><br aria-hidden="true">For Scheduled Tasks and Data Sources, you will be unable to achieve the same goals that were available through the nonsecure functionality of Group Policy Preferences passwords. </p>
            
          </section>

            <h3 id="ID0EDNBJ" class="">Summary of changes </h3>
            
              <ul>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>Password fields in all affected preferences are disabled. Administrators cannot create new preferences by using these password fields.</p>
                <meta itemprop="position" content="1">
      </li>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>The username field is disabled in some preferences.</p>
                <meta itemprop="position" content="2">
      </li>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>Existing preferences that contain a password cannot be updated. They can only be deleted or disabled, as appropriate for the specific preference.</p>
                <meta itemprop="position" content="3">
      </li>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>The behavior for Delete and Disable actions have not changed for the preferences.</p>
                <meta itemprop="position" content="4">
      </li>
      <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                  <p>When an administrator opens any preference that contains the CPassword attribute, the administrator receives the following warning dialog box to inform him or her of the recent deprecation. Attempts to save changes to new or existing preferences that require the CPassword attribute will trigger the same dialog box. Only Delete and Disable actions will not trigger warning dialog boxes.</p>
                <meta itemprop="position" content="5">
      </li>
    </ul>

          <section aria-labelledby="ID0EFBBH" class="ocpSection">
            <h3 id="ID0EFBBH" class="">Removing CPassword </h3>
            
              <p>The Windows PowerShell script that is included in this Microsoft Knowledge Base article detects whether a domain contains any Group Policy Preferences that might use CPassword. If CPassword XML is detected in a given preference, it is displayed in this list. <br aria-hidden="true"><br aria-hidden="true"><br aria-hidden="true"></p>
            
            
              <section aria-labelledby="ID0EDDBBBH" class="ocpSection">
                <h4 id="ID0EDDBBBH" class="">Detecting CPassword preferences </h4>
                
                  <p>
                    <br aria-hidden="true">This script must be run from a local directory on the domain controller that you want to clean. Copy and save the contents to a new Windows PowerShell file, determine your system drive, and then run the script as indicated in the following usage.<br aria-hidden="true"><br aria-hidden="true"> &lt;#<br aria-hidden="true">.SYNOPSIS<br aria-hidden="true">Group Policy objects in your domain can have preferences that store passwords for different tasks, such as the following:<br aria-hidden="true">    1.                 <h4 id="ID0EDBBBBH" class="">Removing CPassword preferences </h4>
                
                  <p>In order to remove the preferences that contain CPassword data, we suggest that you use Group Policy Management Console (GPMC) on the domain controller or from a client that has Remote Server Administration Tools installed. You can remove any preference in five steps on these consoles. To do this, follow these steps:</p>
                  <ol type="1" itemscope="" itemtype="http://schema.org/ItemList">
        <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                      <p>In GPMC, open the preference that contains CPassword data.</p>
                    <meta itemprop="position" content="1">
        </li>
        <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                      <p>Change the action to <b class="ocpLegacyBold">Delete</b> or <b class="ocpLegacyBold">Disable</b>, as applicable to the preference.</p>
                    <meta itemprop="position" content="2">
        </li>
        <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                      <p>Click <b class="ocpLegacyBold">OK</b> to save your changes.</p>
                    <meta itemprop="position" content="3">
        </li>
        <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                      <p>Wait for one or two Group Policy refresh cycles to allow changes to propagate to clients.</p>
                    <meta itemprop="position" content="4">
        </li>
        <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                      <p>After changes are applied on all clients, delete the preference.</p>
                    <meta itemprop="position" content="5">
        </li>
        <li itemprop="itemListElement" itemscope="" itemtype="http://schema.org/ListItem">
                      <p>Repeat steps 1 through 5 as needed to clean your whole environment. When the detection script returns zero results, you are finished.</p>
                    <meta itemprop="position" content="6">
        </li>
      </ol>
