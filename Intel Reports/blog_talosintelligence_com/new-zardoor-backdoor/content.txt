    <h1 class="text-center">New Zardoor backdoor used in long-term cyber espionage operation targeting an Islamic organization</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/cisco/">Cisco Talos</a>
        </div>

        <br/>
        <time class="post-datetime" datetime="February 8, 2024 08:00">
            Thursday, February 8, 2024 08:00
        </time>

        <div class="m-3">
                            <a href="/category/threats/" class="category primary-category">
                                Threats
                            </a>
                                                    <a href="/category/rat/" class="category primary-category">
                                RAT

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <p><em>By Jungsoo An, Wayne Lee and Vanja Svajcer.</em></p><ul><li>Cisco Talos discovered a new, stealthy espionage campaign that has likely persisted since at least March 2021. The observed activity affects an Islamic non-profit organization using backdoors for a previously unreported malware family we have named “Zardoor.”&nbsp;</li><li>We believe an advanced threat actor is carrying out this attack, based on the deployment of the custom backdoor Zardoor, the use of modified reverse proxy tools, and the ability to evade detection for several years.&nbsp;</li><li>Throughout the campaign, the adversary used <a href="https://blog.talosintelligence.com/hunting-for-lolbins/"><u>living-off-the-land binaries (LoLBins)</u></a> to deploy backdoors, establish command and control (C2), and maintain persistence.&nbsp;</li><li>At this time, we have only discovered one compromised target, however, the threat actor’s ability to maintain long-term access to the victim’s network without discovery suggests there could be others.</li><li>Based on Talos’ and third-party research, the use of reverse proxy tools overlaps with TTPs employed by several threat groups originating from China. Still, we can assess the relations of the new threat actor with the existing groups only with low confidence, as open-source tools can be used by any threat actor. The choice of the compromised target does not align with the known objectives of any known threat actors originating from China.&nbsp;&nbsp;</li></ul><p>Talos discovered an ongoing espionage campaign in May 2023 targeting an Islamic charitable non-profit organization in Saudi Arabia that exfiltrates data approximately twice a month.&nbsp;</p><p>The initial access vector is unknown, however, we observed the threat actor executing a malware we are calling the “Zardoor” backdoor to gain persistence. Then we observed the threat actor establishing C2 using open-source reverse proxy tools such as Fast Reverse Proxy (FRP), sSocks and Venom, a reverse proxy socks5 server-client tool originally developed for penetration testers.&nbsp;</p><p>The threat actor customized sSocks to remove dependencies on Visual C Runtime libraries so these tools would rely only on WinAPI libraries and therefore could be executed without unexpected runtime errors.&nbsp;</p><p>Once a connection was established, the threat actor used Windows Management Instrumentation (WMI) to move laterally and spread the attacker's tools — including Zardoor — by spawning processes on the target system and executing commands received from the C2, as seen in the commands below.</p><figure class="kg-card kg-image-card"><img src="https://lh7-us.googleusercontent.com/xK3n0JialvYV_Pxjjd44_NwtttmslZAn-i4DN61mk0ouPBZtv03005yFbTJfI6jORJKDZHOhyLCfZEcWSEi7QpqkmUj6951JGHIe525khKQWOEh1TeC3e-C3yz1w2oR1ffahvMz-psKBqiFBknheunI" class="kg-image" alt="" loading="lazy" width="766" height="365"></figure><h3 id="execution-flow-of-the-zardoor-backdoor">Execution flow of the Zardoor backdoor</h3><p>To maintain persistence, the attacker deployed a previously unseen backdoor family we have named Zardoor, which we named based on the file names “zar32.dll” and “zor32.dll”. “Zar32.dll” is the main backdoor component that communicates with the attacker’s C2, and “zor32.dll” ensures “zar32.dll” has been properly deployed with admin privileges. Talos could not obtain a file sample for the dropper used in this specific campaign. However, we found and analyzed other available samples with an execution sequence and filenames identical to the malicious activity we observed and possibly related to the attack we observed.&nbsp;</p><p>Based on our analysis of these matching samples, the execution sequence has two parts:</p><p><strong>The dropper installs and executes the malicious “oci.dll”</strong></p><p>The main purpose of this dropper is to configure “msdtc.exe” to load the malicious “oci.dll” payload. Depending on the target OS architecture, the dropper locates either a 32- or 64-bit “oci.dll” and drops it in the system file path <code>C:\Windows\System32\</code>. Then, the dropper will attempt to stop the MSDTC service and use “msdtc.exe” to help register the malicious “oci.dll” with admin privileges, using the command <code>msdtc -install</code>.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/KU4dWc8gG4LOFADhflKcInM8gub2kgMpQUlZRksVC_0jkLzDT-_dlIR3ur1LphoB2tmSZtFE_pl_YyL4uugm0l6mdMOvN_b92qicdJaaQZjzEKHj0W7IgpXsn4Y_wq5oqtjzQSBHsAw4gzSfyucjR_c" class="kg-image" alt="" loading="lazy" width="697" height="414"><figcaption><i><em class="italic" style="white-space: pre-wrap;">The dropper drops a different version of oci.dll based on the OS bitness and deletes itself using a batch file.</em></i></figcaption></figure><p>However, if the MSDTC service fails to stop, the dropper patches the binary of the malicious “oci.dll” file to remove the strings <code>1ISSYSTEM</code> and <code>1ISAUTORUN</code>, and save the patched DLL to the file path, <code>%TEMP%\win_oci_41aa0d5.dll</code>. Removing the strings will later help determine where to save “zar32.dll” and “zor32.dll” on the victim’s computer.&nbsp;</p><p>The threat actor then uses Rundll32 to execute the patched “oci.dll” using this command: <code>C:\Windows\System32\rundll32.exe %TEMP%\win_oci_41aa0d5.dll MainEntry</code>. This patched “oci.dll” will extract “zar32.dll” and “zor32.dll” into the Temp Directory, and launch “zar32.dll MainEntry” using “rundll32.exe”. The MSDTC service will register the malicious “oci.dll” with the <code>msdtc -install</code> command.</p><p>If either of these two actions is successful, the dropper configures the MSDTC service to load “oci.dll” and the DLL will be executed. Finally, a cleanup batch script is created and saved to the location <code>%TEMP%\xz330ksdfg.bat</code>. The batch script deletes the dropper and then deletes itself.</p><figure class="kg-card kg-image-card"><img src="https://lh7-us.googleusercontent.com/T3zk3kQHpkKjMSacd45c3L92OiyzI5Yxr8_e26BGQqyt9kKUQF30mKL-1UPno-NjVawVQ4bNFVZSOlsY3pr83MMQO6mJ9Tu_Fdq5V9N_GoZTiI71zco8zhhuRzzPtsSmUxvZO_HTAi6s1LIRc6KmW04" class="kg-image" alt="" loading="lazy" width="706" height="215"></figure><p><strong>Malicious “oci.dll” payload</strong></p><p>The malicious loader “oci.dll” contains the backdoor payloads, “zar32.dll” and “zor32.dll” in the resource section. Oci.dll contains two exported functions: <code>ServiceMain()</code> to launch the backdoor module (“zar32.dll”) and <code>DllEntryPoint()</code> to drop the backdoor onto the victim’s machine.</p><p>The <code>ServiceMain()</code> export is executed by the MSDTC service and launches the export function <code>MainEntry</code> of “zar32.dll” using “rundll32.exe.”</p><p>The <code>DllEntryPoint()</code> function calls the DLLMain function, which determines where to dump “zar32.dll” and “zor32.dll”. This occurs by searching for the strings <code>1ISSYSTEM</code> and <code>1ISAUTORUN</code>. If the string <code>1ISSYSTEM</code> is found in “zar32.dll”, DLLMain drops “zar32.dll” and “zor32.dll” into the System32 directory.</p><p>If the string <code>1ISSYSTEM</code> is not found, then DLLMain will look up the string <code>1ISAUTORUN</code>, and if it exists, DLLMain will drop “zar32.dll” and “zor32.dll” into the <code>%userprofile% directory</code>. If neither of the strings are found, DLLMain will drop “zar32.dll” and “zor32.dll” into the “%TEMP%” directory. After the payloads are saved, the DLLs and their export function 'MainEntry()' are launched by “rundll32.exe”.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/XE36iUojOzfXKSnkM3xVpU_jufLVs9aDMxf0UZFe-BwZodA3AYivjcCkcTQhSR64EpdX_9YtigMtg6VzK6hDg0Hq0I2RcM0IMTmdF_Dlwz4bT9GKT0m8S69y6HmfdNRfjwjuDIR__h5blellLy6zeGc" class="kg-image" alt="" loading="lazy" width="609" height="361"><figcaption><span style="white-space: pre-wrap;">The oci.dll contains two Zardoor components, zar32.dll and zor32.dll as resources.</span></figcaption></figure><p>To execute “zar32.dll”, the “oci.dll” export “ServiceMain()” is executed by “msdtc.exe” which then loads “zar32.dll” using the command: <code>rundll32.exe C:\WINDOWS\system32\zar32.dll MainEntry</code>. “Zor32.dll” is subsequently loaded from the same exported method with the command <code>rundll32.exe C:\WINDOWS\system32\zor32.dll MainEntry</code>.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/9YNNjlvi8RtdbarQdSqpDmt-1fMnMm3sk7ZkG7a83lWLR40hyO-yS_yo1AiKPqVNNTd4nRUKjmG51_YFMSqUU37bkbd6tcDOx60jtB-ZH2uJSSCYeyDdOg-D7obl7KfzW9wkn21dVslz9OR_HmjcPdk" class="kg-image" alt="" loading="lazy" width="1354" height="1600"><figcaption><span style="white-space: pre-wrap;">Zardoor modules.</span></figcaption></figure><h3 id="analysis-of-the-%E2%80%9Czar32dll%E2%80%9D-and-%E2%80%9Czor32dll%E2%80%9D-backdoor-files">Analysis of the “zar32.dll” and “zor32.dll” backdoor files</h3><p>“Zar32.dll” is an HTTP/SSL remote access tool (RAT) that is capable of sending encrypted data to the attacker’s C2, executing PE payloads in fileless mode, searching for Session IDs, remote shellcode execution, and updating the C2 IP/hostname or port in memory. “Zar32.dll” contains a hardcoded debug symbol path seen below, and has two export functions: <code>MainEntry()</code> and&nbsp; <code>DllEntryPoint</code>.&nbsp;</p>
<!--kg-card-begin: html-->
<code>PDB file path : 'C:\\Users\\john\\Desktop\\RManager\\x64\\Release\\R_Run.pdb'</code>
<!--kg-card-end: html-->
<p>Once deployed, “zar32.dll” creates three mutexes with the names, <code>3e603a07-7b2d-4a15-afef-7e9a0841e4d5</code>, <code>ThreadMutex12453</code>, and <code>rrx_%d</code>, where the value of <code>%d</code> is a random seed that is based on the DLLs’ time of execution. If the mutex <code>3e603a07-7b2d-4a15-afef-7e9a0841e4d5</code> already exists, the DLL will exit because that indicates “zar32.dll” is successfully running.</p><p>To establish a C2 connection, “zar32.dll” needs a program that allows network applications to operate through a SOCKS or HTTPS proxy. The DLL connects to the following URLs:</p><ul><li>1.0.0[.]1/index.html</li><li>1.0.0[.]2/index.html</li><li>1.0.0[.]3/index.htm</li></ul><p>The IP addresses are used by Cloudflare DNS services, including the DNS over HTTPS and the communication to these IP addresses may indicate the attempt to bypass the DNS-based detections to attacker-controlled C2 servers.&nbsp;</p><p>“Zar32.dll'' attempts to connect to its C2 server using SSL with the following HTTP User-Agents:</p><ul><li>64-bit application: <code>Mozilla/5.0 (Windows NT &lt;os_majorver&gt;.&lt;os_minorver&gt;; Trident/7.0; rv:11.0) like Gecko</code></li><li>32-bit application on 64-bit OS: <code>Mozilla/5.0 (Windows NT &lt;os_majorver&gt;.&lt;os_minorver&gt;; WOW64; Trident/7.0; rv:11.0) like Gecko</code></li></ul><p>Once a connection is successfully established, “zar32.dll” supports the following C2 commands:</p><ol><li>Encrypt and send data to C2.</li><li>Execute remotely fetched PE payload.</li><li>Search for session ID.</li><li>(Plugin exit).</li><li>Remote shellcode execution.</li><li>Delete this RAT.</li><li>Update C2 IP (IP/domain_name:port).</li><li>Do nothing.</li></ol><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/s7ao4uA2wtBBT4QsolvINlWnCJazNphvrYg2sOC4cpFhB5_AIulYqTDxY_B7b9XSe0kNzWEsaDqf1KhW3CSAOJ6Z65iTiiqIyZFlYPp4RMCg5RfOrr4E-L6zLoaJReoz6i-WLBZGW6xlLlbocy2qA1Q" class="kg-image" alt="" loading="lazy" width="766" height="467"><figcaption><span style="white-space: pre-wrap;">Zardoor (zar32.dll) C2 routine handles eight different C2 commands.</span></figcaption></figure><p>We continued to observe several dependencies in the malware’s execution routine. If “zar32.dll” is running when “zor32.dll” is installed, “zor32.dll” will install the “msdtc.exe” service installer.&nbsp;</p><p>If “zar32.dll” is not running when “zor32.dll” is installed, then “zor32.dll” starts the “msdtc.exe” service and attempts to create a mutex with the name <code>6c2711b5-e736-4397-a883-0d181a3f85ae</code>.&nbsp;</p><p>Next, “zor32.dll” will check if the “oci.dll” file exists and finish the execution if it does not. If “oci.dll” exists, “zor32.dll” attempts to create another mutex with the name <code>3e603a07-7b2d-4a15-afef-7e9a0841e4d5</code>. The DLL will exit if the mutex exists, indicating“zar32.dll” is successfully running.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/BZRQWFVgDlp8ZtEofEsWOFf07tTey9wl3clQ42hgCjwkvZxGh5yYR-BhjFwo0Gt7kQXX4hGcrLDx6UkBZ-p-Eny62o0OXxq5XrqBQceIe29gzuQumzP5z39cjqvFsjgwTDiVt0Wip7MTKxomF5pVzgs" class="kg-image" alt="" loading="lazy" width="705" height="399"><figcaption><i><em class="italic" style="white-space: pre-wrap;">If the mutex does not exist, “zor32.dll” will attempt to re-launch “zar32.dll” up to 10 times.</em></i></figcaption></figure><p>We also identified “zor32.dll” performing checks to maintain persistence if the process has admin privileges using the following procedures:</p><ol><li>If the MSDTC service is not running, “zor32.dll” will configure the MSDTC service with the command <code>msdtc -install</code>. If the installation fails, it will keep attempting up to 10 times.</li><li>“zor32.dll” attempts to query MSDTC service status and if this fails, it will attempt up to 10 times.</li><li>If the MSDTC service is running, then “zor32.dll” will attempt to stop it. If this fails, it will keep attempting to install up to 10 times.</li><li>If the MSDTC service is not running, “zor32.dll” will start the service.</li></ol><h3 id="scheduled-tasks-to-maintain-persistence">Scheduled tasks to maintain persistence</h3><p>For persistence, the threat actor registers their reverse proxies as scheduled tasks, causing the reverse proxy to execute approximately every 20 minutes to communicate with the attacker’s C2 servers. To achieve this, first, the threat actor confirms if the victim already has scheduled tasks running with the names "KasperskySecurity" or "Microsoft Security Essentialss." Then, the attacker deletes the legitimate scheduled task and creates a new one with the same name for the proxy “msbuildss.exe”.</p><figure class="kg-card kg-image-card"><img src="https://lh7-us.googleusercontent.com/bOKX_2fVHU2IWYCVaxd-l3aVn48yKlYoy9rpvH9ixWoWuRLOMs9dcmqkC87PBB04Jan_NXM01Iih5WFlFHPLdh2rVsAYAElEOKZV5HY0xxMXJLh93DRI8jP0K9ApKoRiULglsJnn452PyM42giPpTTY" class="kg-image" alt="" loading="lazy" width="766" height="190"></figure><p>Talos observed the threat actor in July 2023 storing the remote server’s public key for future tasks. This helps the threat actor to access the remote (Secure Shell) SSH server and set up remote port forwarding, allowing remote servers and devices on the internet to access devices that are on a private network.&nbsp;</p><p>The attacker downloads the private SSH key and saves it to the file path <code>c:\users\[Redacted]\.ssh\</code> with the filename “id_rsa.” The threat actor also saves the file “known_hosts”, containing the public keys hosts that can be accessed using the private key stored in “id_rsa”.</p><figure class="kg-card kg-image-card"><img src="https://lh7-us.googleusercontent.com/eEmtXfUuyGzV7kNxzzd6pREWiBm19Hr-xMnFZ-qwe2mAMzlfmo7-BBI6arswrAQyTEDVmpYeTLCQD8IMpfUmuwM2w-1nGWi8lJo8kUMErT9cDlTrBbiELOd65qu_qxtrmj-BK2oXQM0mO0QeWkqBHRk" class="kg-image" alt="" loading="lazy" width="755" height="607"></figure><p>According to the above commands, the threat actor first downloads and checks the contents of the file “2.vbs” to ensure it is the script they would like to execute. The “2.vbs” file is responsible for setting up the SSH remote port forwarding from port 443 on the victim’s device to port 22 on the attacker’s server using the user name root. The file makes sure it has successfully set up the SSH forwarding server by performing the following steps:</p><ol><li><code>netstat -ano | findstr 70.34</code> is used to find part of the remote IP address 70[.]34[.]208[.]197.</li><li>The executable “shd.exe” initiates an SSH connection using the username root and the password "3My[{BK)Ni8a".</li><li>Look for a port 443 connection on the victim’s device and kill “ssh.exe” and “shd.exe” using the <code>taskkill</code> utility.</li></ol><h3 id="reverse-proxy-tools-used-by-the-threat-actor">Reverse proxy tools used by the threat actor</h3><p>As opposed to forward proxies, used to connect devices on the private network to internet services, usually HTTP-based. Reverse proxies allow a computer connected to the internet to create a tunnel and allow remote access to services on the local private network.</p><p>Reverse proxies are often used as legitimate load balancers in complex system and application architectures. However, malicious actors are using them to establish communications with otherwise unreachable systems such as RDP servers, domain controllers, files or database servers.&nbsp;</p><h4 id="fast-reverse-proxy-frp">Fast Reverse Proxy (FRP)</h4><p><a href="https://github.com/fatedier/frp"><u>Fast Reverse Proxy</u></a> (FRP) is a reverse proxy tool that can be used to make network services, often located behind a NAT or firewall, remotely accessible. FRP consists of two main components: the FRP client and the FRP server. The FRP client is responsible for forwarding local requests to the FRP server, which in turn redirects them to the internet. This allows applications running on devices behind the NAT or firewall to be accessible from the outside network.</p><p>7000 is the default port used by the FRP server components. However, these ports can be configured per the user's needs. A basic setup involves installing the FRP server on a public server with a public IP, and the FRP client on the machine you want to expose. The client and server are configured via respective INI configuration files. Once the client and server are appropriately configured and started, services on the client's machine will be accessible via the server's public IP and the specified ports.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/vHb4f2Z6V1jyu72dEyv2E-U2N6f3dFDBT4QBlUyPdWGuO-qN57xbRwiTclBvJ7ofpX62PCPGcZ8eRfB4lzeDVHK4juM0qD5iW2KzlKaj6kupvLWEDiksV-7ggrzMwzWDpPdOBswAMOn07ayOQbvdpBc" class="kg-image" alt="" loading="lazy" width="1600" height="1247"><figcaption><i><em class="italic" style="white-space: pre-wrap;">FRP components and the basic usage.</em></i></figcaption></figure><p>Fast Reverse Proxy (FRP) has been reported to be used by several, predominantly Chinese threat actors to bypass network security measures and maintain persistence within a compromised network. By leveraging FRP, these threat actors can create a tunnel from a machine within the compromised network to an external server under their control. This allows them to exfiltrate data, deploy additional malicious tools, or carry out other activities while evading detection.&nbsp;</p><p>The usage of FRP, a legitimate and widely-used tool, makes the malicious traffic harder to distinguish from the normal network traffic, thereby increasing the stealthiness of the attack. However, the presence of an FRP client in the environment may be a good indicator of potential compromise of the network where FRP is not typically used.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/7mSQH2CeQIHsrpmkVZVQslAtzgdttlyZdHpzu6KVFUPrYeH9hffGq56tds6LR0Fgj9wy_cTWXtwDQZR4keyWuN6s__1Ka4Wt8bTuFFQd_qkrJtVJkXVX4l_wmjlSZqdMwTmKtwMog1moi91utj_oskU" class="kg-image" alt="" loading="lazy" width="1600" height="1177"><figcaption><i><em class="italic" style="white-space: pre-wrap;">VirusTotal uploads for the FRP client executables.</em></i></figcaption></figure><p>FRP is a popular tool and has been increasingly used by threat actors, based on the increase in VirusTotal submissions of FRP tools over the past few years.&nbsp;</p><h4 id="venom-proxy">Venom proxy</h4><p><a href="https://github.com/Dliv3/Venom"><u>Venom</u></a> is a multi-hop proxy designed for red teaming and pentesting written in the Go language. It allows the user to create a network of proxy nodes that can act as an admin or agent. The agent connects to either another agent or the admin node. The user controls the network through the control of the administration node and can easily add additional agents to the network.&nbsp;</p><p>Venom allows the user, which could also be a malicious actor, to create a botnet of proxies that can be used to remotely control the nodes, exfiltrate data, install additional payloads, etc.&nbsp;</p><p>The Venom features are:</p><ul><li>Multi-hop socks5 proxy.</li><li>Multi-hop port forwarding.</li><li>SSH tunneling.</li><li>Interactive shell.</li><li>Uploading and downloading files.</li><li>Network traffic encryption.</li><li>Support of multiple platforms (Linux/Windows/MacOS) and multiple architectures (x86/x64/ARM/MIPS).</li></ul><h4 id="other-reverse-proxy-tools-and-their-usage-by-threat-actors">Other reverse proxy tools and their usage by threat actors</h4><p>In addition to FRP and Venom, threat actors, predominantly originating from China, based on the <a href="https://blog.talosintelligence.com/alchimist-offensive-framework/"><u>previous Talos research</u></a> and available open-source threat intelligence use several other tools supporting reverse proxying, most commonly:</p><ul><li><a href="https://github.com/HiwinCN/HTran"><u>HTran (lcx)</u></a></li><li><a href="https://github.com/tostercx/ssocks"><u>sSocks</u></a></li><li><a href="https://rootkiter.com/EarthWorm/"><u>Earthworm (ew)</u></a></li><li><a href="https://github.com/EddieIvan01/iox"><u>iox</u></a></li><li><a href="https://github.com/cw1997/NATBypass"><u>NATBypass</u></a></li><li><a href="https://github.com/ehang-io/nps"><u>NPS</u></a></li><li><a href="https://github.com/b23r0/rsocx"><u>Rsocx</u></a></li><li><a href="https://github.com/serjs/socks5-server"><u>Go socks5</u></a></li><li><a href="https://github.com/sensepost/reGeorg"><u>Regeorg</u></a></li></ul><p>We have also created a matrix that displays the active threat groups and the proxy tools they are using. Talos assesses with low confidence that the existence of one or more of the tools on a compromised system may indicate the activity of a particular group, as these tools are easily reusable and can be employed by any malicious actor.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/OAAnAu3JNYKFFYt217Qqoc0TiwTvIXIIwBEnoez8UPCp1sTQE-yaLpBRgB6i_T85U9gNyDQAicDcWCOTWnBO3EIwB7KG6apAgHsI1e-sMFp_ATiLq6ZNaoM9FVpQHCplPSnkxx4wI5ie6PxxIibzgcA" class="kg-image" alt="" loading="lazy" width="1600" height="798"><figcaption><i><em class="italic" style="white-space: pre-wrap;">Proxy tools and the threat actors utilizing them.</em></i></figcaption></figure><h3 id="zardoor-attacks-conducted-by-an-advanced-threat-actor">Zardoor attacks conducted by an advanced threat actor</h3><p>Talos assesses this campaign was conducted by an unknown and advanced threat actor. We have not been able to attribute this activity to any known, publicly reported threat actor at this time, as we have not found any overlap between the observed tools or C2 infrastructure used in this campaign.&nbsp;</p><p>The threat actor appears highly skilled based on their ability to create new tooling, such as the Zardoor backdoors, customize open-source proxy tools, and leverage several LoLBins including “msdtc.exe” to evade detection. In particular, side-loading backdoors contained in “oci.dll” via MSDTC is a very effective method of remaining undetected while maintaining long-term access to a victim’s network.&nbsp;</p><h2 id="coverage">Coverage</h2><p>Ways our customers can detect and block this threat are listed below.</p><figure class="kg-card kg-image-card"><img src="https://lh7-us.googleusercontent.com/gxpt8pMwB3yi_Ghw3YwK_vb8rWD8h5o0jahuBKpWx44EmR98T14tlTio9uUUTR5VCBKd_KYLGcVDjcIMxWpo1TGPu_KeCO1PVAcrcKioeP4A3IEFCMPJPX922X4towsjElWZRg7M1yhdIpYrVMpZb-s" class="kg-image" alt="" loading="lazy" width="1600" height="323"></figure><p><a href="https://www.cisco.com/c/en/us/products/security/amp-for-endpoints/index.html"><u>Cisco Secure Endpoint</u></a> (formerly AMP for Endpoints) is ideally suited to prevent the execution of the malware detailed in this post.
