&lt;h2&gt;Analysis: Grandoreiro Loader&lt;/h2&gt;
In line with previous campaigns, Grandoreiro&#039;s infection chain begins with a custom loader. Often, the executable is bloated to a size of more than 100MB to hinder automatic anti-virus scanning. In hopes of circumventing automated execution, it displays a small CAPTCHA pop-up imitating Adobe PDF reader, which requires a click to continue with the execution.

&lt;img src=&quot;https://images2.cmp.optimizely.com/Zz0yYWE0NGRlODEzNzkxMWVmYjM3MjdhNzcxZDQ1YmY2OQ==&quot; alt=&quot;PDF Icon.png&quot; width=&quot;423&quot; height=&quot;189&quot; /&gt;

&lt;i&gt;Fig 8: Grandoreiro fake Adobe PDF reader CAPTCHA&lt;/i&gt;

The loader has three main tasks:
&lt;ol start=&quot;1&quot; type=&quot;1&quot;&gt;
 	&lt;li&gt;Verify if the client is a legitimate victim (not a researcher or a sandbox)&lt;/li&gt;
 	&lt;li&gt;Enumerate basic victim data and send it back to its C2&lt;/li&gt;
 	&lt;li&gt;Download, decrypt and execute the Grandoreiro banking trojan&lt;/li&gt;
&lt;/ol&gt;
&lt;h3&gt;String decryption&lt;/h3&gt;
All of these tasks require more than 120 important strings, which are encrypted using an improved algorithm.

First, Grandoreiro starts by generating a large key string, which is hardcoded and triple-Base64-encoded. The key observed in these samples begins with &lt;b&gt;&quot;D9JL@2]790B{P_D}Z-MXR&amp;EZLI%3W&gt;#VQ4UF+O6XVWB16713NIO!E...&quot;. &lt;/b&gt;It then takes the encrypted string and uses a custom decoding to convert it into a series of hexadecimal characters interpreted as bytes.

&lt;img src=&quot;https://images1.cmp.optimizely.com/Zz01MDBlMWYxZTEzNzkxMWVmODdkNTAyMWMxMmYxNWNmOA==&quot; alt=&quot;Key string.png&quot; width=&quot;570&quot; height=&quot;381&quot; /&gt;

&lt;i&gt;Fig 9: Grandoreiro custom hex encoding (note that non-hex character encoding like &#039;&quot;&#039; are never used)&lt;/i&gt;

Grandoreiro decrypts the result via the old Grandoreiro algorithm using the key string. Below is a Python implementation of the decryption routine:

&lt;img src=&quot;https://images4.cmp.optimizely.com/Zz0xZTlmYTVlNDEzODYxMWVmOGU5ZmM2NTQ4N2FkMmYzNw==&quot; alt=&quot;code snippet.png&quot; width=&quot;673&quot; height=&quot;183&quot; /&gt;

Lastly, it undergoes a final round of 256-bit AES CBC decryption and unpadding to retrieve the plaintext string. Both the AES key and Initiation Vector (IV) are also stored as encrypted strings and have to be decrypted using the same algorithm as above, however skipping the AES decryption. The graph below gives an overview of the full decryption process:

&lt;img src=&quot;https://images2.cmp.optimizely.com/Zz0yYjU0NmRhYTEzN2QxMWVmOGEyZjU2NDA5MzM4MjFjMQ==&quot; alt=&quot;Loader String decryption.png&quot; width=&quot;640&quot; height=&quot;635&quot; /&gt;

&lt;h3&gt;Victim verification&lt;/h3&gt;
To verify that a victim is not part of a sandboxed environment, the Grandoreiro loader collects the following information and checks it against a list of hardcoded values (see Appendix):
&lt;ol start=&quot;1&quot; type=&quot;1&quot;&gt;
 	&lt;li&gt;Computer name&lt;/li&gt;
 	&lt;li&gt;Username&lt;/li&gt;
 	&lt;li&gt;OS version information&lt;/li&gt;
 	&lt;li&gt;Installed Antivirus solution&lt;/li&gt;
 	&lt;li&gt;Country of the victim&#039;s public IP (via http://ip-api.com/json)&lt;/li&gt;
 	&lt;li&gt;List of running processes&lt;/li&gt;
&lt;/ol&gt;
This verification step is also used to disallow victims of specific countries. One sample did not continue execution for infections with public IPs from:
&lt;ul type=&quot;disc&quot;&gt;
 	&lt;li&gt;Russia&lt;/li&gt;
 	&lt;li&gt;Czechia&lt;/li&gt;
 	&lt;li&gt;Poland&lt;/li&gt;
 	&lt;li&gt;Netherlands&lt;/li&gt;
&lt;/ul&gt;
The sample also prevented infections of Windows 7 machines based in the US without anti-virus.
&lt;h3&gt;Victim profiling&lt;/h3&gt;
The next execution step attempts to build a basic profile of the victim to display on the C2 panel. The malware enumerates the following information on the victim machine:
&lt;ul type=&quot;disc&quot;&gt;
 	&lt;li&gt;Public IP country&lt;/li&gt;
 	&lt;li&gt;Public IP region&lt;/li&gt;
 	&lt;li&gt;Public IP city&lt;/li&gt;
 	&lt;li&gt;Computer name&lt;/li&gt;
 	&lt;li&gt;Username&lt;/li&gt;
 	&lt;li&gt;OS Version information&lt;/li&gt;
 	&lt;li&gt;Installed AV solution&lt;/li&gt;
 	&lt;li&gt;Check in the registry subkey &quot;Software\Clients\Mail&quot; if the Outlook mail client is installed. If true, the value is set to &quot;SIM&quot;, which means &quot;Yes&quot; in Portuguese&lt;/li&gt;
 	&lt;li&gt;Check if crypto-wallets exist: Binance, Electrum, Coinomi, Bitbox, OPOLODesk, Bitcoin&lt;/li&gt;
 	&lt;li&gt;Check if special banking security software is installed: IBM Trusteer, Topaz OFD, Diebold&lt;/li&gt;
 	&lt;li&gt;Number of Desktop monitors&lt;/li&gt;
 	&lt;li&gt;Volume Serial Number&lt;/li&gt;
 	&lt;li&gt;Date of infection&lt;/li&gt;
 	&lt;li&gt;Time of infection&lt;/li&gt;
&lt;/ul&gt;
Grandoreiro concatenates the results using the string &quot;*~+&quot; and sends it as part of the encrypted payload request to the C2 server.
&lt;h3&gt;C2 communication and loading Grandoreiro&lt;/h3&gt;
Grandoreiro loader&#039;s C2 server can be decrypted via the same algorithm explained above. The resulting domain name is resolved via DNS over HTTPS through the URL &lt;b&gt;https://dns.google/resolve?name=&lt;C2 server&gt; &lt;/b&gt;to circumvent DNS-based blocking. After receiving the C2 IP address, the malware takes the first 4 digits of the IP and runs 4 different digit-to-digit mappings over it resulting in the 4-digit port number.

It then concatenates the victim profiling string from above together with a capitalized Portuguese message &quot;CLIENT_SOLICITA_DDS_MDL&quot; (likely translated to &quot;Client asks for module data&quot;). An example string would be:

&lt;img src=&quot;https://images3.cmp.optimizely.com/Zz0yNDcyNGViODEzOTAxMWVmYjUzYWNhZmRlZmQ2ZjQ2YQ==&quot; alt=&quot;code snip - c2 communication 1.png&quot; width=&quot;670&quot; height=&quot;96&quot; /&gt;

The string is encrypted and sent as the URL path via an HTTP GET request to the C2 server requesting the final Grandoreiro payload.

If successful, the C2 server replies with an HTTP 200 status code containing another encrypted message. It contains the following information:
&lt;ol start=&quot;1&quot; type=&quot;1&quot;&gt;
 	&lt;li&gt;Payload download URL&lt;/li&gt;
 	&lt;li&gt;C2 server&lt;/li&gt;
 	&lt;li&gt;Directory name&lt;/li&gt;
 	&lt;li&gt;Payload name&lt;/li&gt;
 	&lt;li&gt;Payload size&lt;/li&gt;
&lt;/ol&gt;
Example:

&lt;img src=&quot;https://images2.cmp.optimizely.com/Zz02Nzc3NmU3ODEzOTAxMWVmYjU2MmRhMGUyYmFmMzZkNQ==&quot; alt=&quot;code snip - c2 communication 2.png&quot; width=&quot;684&quot; height=&quot;85&quot; /&gt;

To download, Grandoreiro issues another HTTP GET request to the payload URL. The downloaded file is stored in the specified directory name under &quot;&lt;b&gt;C:\ProgramData\&lt;/b&gt;&quot;. Next, the file is decrypted via an RC4-based algorithm using the key &quot;7684223510&quot;. Finally, it is decompressed using the &quot;ZipForge&quot; Delphi library, and the originally downloaded file is deleted.

The archive may contain two files, a .EXE (Grandoreiro banking trojan) and a .CFG (config file).

Prior to execution, the loader performs an enumeration of the current process token&#039;s group membership, specifically checking for the presence of the SECURITY_NT_AUTHORITY SID. If the process possesses the required privileges, the loader utilizes the &lt;i&gt;ShellExecuteW()&lt;/i&gt; function with the &#039;runas&#039; verb to execute the Grandoreiro payload with elevated privileges. Conversely, if the necessary privileges are not available, the loader resorts to executing itself via&lt;i&gt; ShellExecuteW()&lt;/i&gt; without elevation.

During all stages of infection—the payload download, decryption, and execution—the Grandoreiro loader reports back status messages to its C2 server. Some examples are:
&lt;ul type=&quot;disc&quot;&gt;
 	&lt;li&gt;ERRO_FALHA_DOWNLOAD (&quot;Download failed error&quot;)&lt;/li&gt;
 	&lt;li&gt;ERRO_EXTRACAO (&quot;Extraction error&quot;)&lt;/li&gt;
 	&lt;li&gt;AV_COMEU_MODULO (&quot;AV ate module&quot;)&lt;/li&gt;
 	&lt;li&gt;ERRO_EXECUCAO (&quot;Execution error&quot;)&lt;/li&gt;
 	&lt;li&gt;INFECTADO (&quot;Infected&quot;)&lt;/li&gt;
&lt;/ul&gt;
&lt;h2&gt;Grandoreiro Banking Trojan&lt;/h2&gt;
The final payload is the Grandoreiro banking trojan. The latest version has undergone major updates mainly within the string decryption and DGA calculation algorithms. It has also included a vast number of global banking applications to target, support execution and enable attackers to perform banking fraud in dozens of countries. Together with a specialized Outlook spreader module and a wide range of features, it is one of the largest known banking trojans and analysis is still ongoing. The following sections present an in-depth look at Grandoreiro&#039;s most notable characteristics, highlighting its essential features and functionalities.
&lt;h3&gt;Persistence and configuration&lt;/h3&gt;
Grandoreiro begins by establishing persistence via the Windows registry. It runs the following command to create a new registry Run key and launch the malware on user login:

&lt;img src=&quot;https://images4.cmp.optimizely.com/Zz0zODFiNWMwZTEzODkxMWVmOWY5NDdhYmVlNDg4ZDcwNw==&quot; alt=&quot;code snippet - persistence.png&quot; width=&quot;655&quot; height=&quot;96&quot; /&gt;

Note that the name of the key may differ among samples, but is often related to the original filename of the downloaded payload. If Grandoreiro does not run in an elevated process, the &quot;/runas&quot; verb is omitted.

In addition to the .CFG file, Grandoreiro also creates a .XML file in the&lt;b&gt; C:\Public\ &lt;/b&gt;directory. It is encrypted via the loader&#039;s string encryption routine and stores the Grandoreiro executable filename, path and date of infection.

If Grandoreiro can&#039;t find its .CFG file, it will populate a new .CFG with default values specifying which Grandoreiro functions are enabled, the victim&#039;s country and date of infection. The .CFG file is encrypted via the Grandoreiro string encryption algorithm explained further below.
&lt;h3&gt;Targeted applications&lt;/h3&gt;
Grandoreiro operators significantly upgraded the list of targeted banking applications, now targeting more than 1500 banks worldwide. The latest variants start by first determining if the victim is on the list of targeted countries. Each country is also mapped to a larger region, which Grandoreiro uses to determine which string searches it should run on currently active windows. This means that, if the victim country for instance is identified as Belgium, it will search for all targeted banking applications associated with the Europe region. Grandoreiro internally maps countries to the region categories &lt;b&gt;Europe, North America, Central America, South America, Africa, Indo-Pacific and global islands&lt;/b&gt;, with each region having an associated Delphi class to search for bank applications. In addition, Grandoreiro has a class searching for 266 unique strings identifying cryptocurrency wallets, which is run on every infection.

&lt;h3&gt;DGA&lt;/h3&gt;
Grandoreiro has traditionally relied on domain generation algorithms (DGA) to calculate its active C2 server based on the current date. The newest iteration of Grandoreiro contains a reworked algorithm and takes it one step further by introducing multiple seeds for its DGA. These seeds are used to calculate a different domain for each mode or functionality of the banking trojan, allowing separation of C2 tasks among several operators as part of their Malware-as-a-Service operation. Each Grandoreiro sample may have a main default seed in case the config file is missing, as well as a list of function-specific seeds. The sample X-Force analyzed contained 14 different seeds, leading to 14 possible C2 domains every day. To explain the algorithm, we will calculate the domains for April 17, 2024. The following chart provides a visualization of the algorithm with an explanation below:

&lt;img src=&quot;https://images1.cmp.optimizely.com/Zz1iODQ4YThlYTEzN2IxMWVmYjBhMjZlNjkxMWEwYzQzYw==&quot; alt=&quot;Grandoreiro DGA.png&quot; width=&quot;720&quot; height=&quot;496.2659846547314&quot; /&gt;

&lt;i&gt;Fig 13: DGA visualization&lt;/i&gt;

Starting with the domain apex, Grandoreiro has one domain mapped to every day of the year. There are two of these mappings, one for the main C2 and one for all function-specific C2s. However, of the 732 apex domains, only 337 are unique. For the given day, the primary apex is &lt;b&gt;dnsfor[.]me&lt;/b&gt; and the secondary is &lt;b&gt;neat-url[.]com&lt;/b&gt;.

For the next part, Grandoreiro concatenates the seed &quot;xretsmzrb&quot; (the main seed) with the 2 digit formatted current month, replacing each digit with three hardcoded characters. The digits &quot;0&quot; and &quot;4&quot; are replaced with &quot;oit&quot; and &quot;zia&quot; respectively, resulting in the full string &quot;xretsmzrboitzia&quot;.

Finally, for each day of the month, Grandoreiro has a custom character to character replacement mapping. For the 17th, after running all 26 character replacements iteratively, the final subdomain string is &quot;wondbbhonandhnd&quot;.

After calculating the remaining domains for all hardcoded seeds, the list of C2 domains for April 17, 2024 becomes:

&lt;h3&gt;Outlook Harvest &amp; Spam&lt;/h3&gt;
One of Grandoreiro&#039;s most interesting features is its capability to spread by harvesting data from Outlook and using the victim&#039;s account to send out spam emails. There are at least 3 mechanisms implemented in Grandoreiro to harvest and exfiltrate email addresses, with each using a different DGA seed. By using the local Outlook client for spamming, Grandoreiro can spread through infected victim inboxes via email, which likely contributes to the large amount of spam volume observed from Grandoreiro.
&lt;h4&gt;Harvesting&lt;/h4&gt;
For the Outlook harvesting mode, Grandoreiro switches its C2 to DGA seed 7 which is used to exfiltrate data. Logging and status messages continue to the main C2 server. For instance, before starting the harvesting process, it sends a log back containing the same victim profiling data as well as the strings &quot;CLIENT_SOLICITA_DD_EMSOUT&quot; (Client asks for EMSOUT data) and &quot;COLHENDO&quot; (harvesting).

In order to interact with the local Outlook client, Grandoreiro uses the &lt;a href=&quot;https://www.add-in-express.com/outlook-security/&quot;&gt;Outlook Security Manager tool&lt;/a&gt;, a software used to develop Outlook add-ins. The main reason behind this is that the Outlook Object Model Guard triggers security alerts if it detects access on protected objects. Outlook Security Manager allows Grandoreiro to disable these alerts during both the harvesting and spamming behavior. Depending on system architecture, the tool requires the DLL &quot;secman.dll&quot; or &quot;secman64.dll&quot; to be registered as COM servers. It then uses MAPI to interact with Outlook.

The malware begins by locating the root mailbox folder and then recursively iterates through the email items. For each email, it checks the &quot;SenderEmailAddress&quot; property and runs a blocklist against it, to filter out unwanted email addresses for harvesting:

&lt;img src=&quot;https://images3.cmp.optimizely.com/Zz1hNDAxOWQ4YzEzOWExMWVmYjU5ZTdhNGE5NjhkNTY2Ng==&quot; alt=&quot;code snip - 9.png&quot; width=&quot;720&quot; height=&quot;507.63807285546415&quot; /&gt;

&lt;h3&gt;String encryption&lt;/h3&gt;
With Grandoreiro being such an extensively large malware, it requires a huge amount of strings, which would make detection very easy if they were left unencrypted. Grandoreiro features more than 10k strings dispersed among more than a hundred feature-specific string-loading functions. The decryption mechanism differs slightly from the loader&#039;s string decryption:

It uses the same Grandoreiro key as the loader, which it decrypts via its custom encryption and the key &quot;A&quot;. Once it has the key, it custom-decodes the encrypted string using the same encoding as the loader and then decrypts the resulting bytes via AES ECB mode using the &lt;a href=&quot;https://github.com/AArhin/delphi-oop/blob/master/Persistence/Externals/NativeXml/extlib/eldos_aes/ElAES.pas&quot;&gt;ElAES Pascal implementation&lt;/a&gt;. The AES key is a scrambled version of the previously decrypted Grandoreiro key. After another round of custom decoding, the string is finally decrypted via the old Grandoreiro algorithm and the Grandoreiro key.

&lt;h2&gt;Conclusion&lt;/h2&gt;
X-Force observed several recent phishing campaigns impersonating official government entities to deliver the Grandoreiro banking trojan. Grandoreiro distributors typically target users in Latin America; however, since the latest &lt;a href=&quot;https://www.interpol.int/en/News-and-Events/News/2024/Disrupting-a-Grandoreiro-malware-operation&quot;&gt;law enforcement action&lt;/a&gt; against Grandoreiro operators, X-Force has observed the malware being spread outside of LATAM to include regions in Central and South America, Africa, Europe, and the Pacific. The Grandoreiro banking trojan samples that X-Force has analyzed have undergone major updates within the string decryption and DGA Calculation algorithms. These newly analyzed samples now include a vast number of at least 1500 global banking applications to target, which support execution and enable attackers to perform banking fraud in over 60 countries. The updates made to the malware, in addition to the significant increase in banking applications across several nations, indicate that the Grandoreiro distributors are seeking to conduct campaigns and deliver malware on a global scale.

<html><body><p>Since March 2024, IBM X-Force has been tracking several large-scale <a href="https://www.ibm.com/topics/phishing"  target="_blank" rel="noopener nofollow" >phishing</a> campaigns distributing the Grandoreiro banking trojan, which is likely operated as a Malware-as-a-Service (MaaS). Analysis of the malware revealed major updates within the string decryption and domain generating algorithm (DGA), as well as the ability to use Microsoft Outlook clients on infected hosts to spread further phishing emails. The latest malware variant also specifically targets over 1500 global banks, enabling attackers to perform banking fraud in over 60 countries including regions of Central and South America, Africa, Europe, and the Indo-Pacific. Although campaigns have traditionally been limited to Latin America, Spain and Portugal, X-Force observed recent campaigns impersonating Mexico&rsquo;s Tax Administration Service (SAT), Mexico&rsquo;s Federal Electricity Commission (CFE), Mexico&rsquo;s Secretary of Administration and Finance, the Revenue Service of Argentina, and notably the South African Revenue Service (SARS). The reworked malware and new targeting may indicate a change in strategy since the latest <a href="https://www.interpol.int/en/News-and-Events/News/2024/Disrupting-a-Grandoreiro-malware-operation" target="_blank" rel="noopener nofollow"  target="_blank" rel="noopener nofollow" >law enforcement</a> action against Grandoreiro, likely prompting the operators to start expanding the deployment of Grandoreiro in global phishing campaigns, beginning with South Africa.</p>
<h2>Key findings</h2>
<ul type="disc">
<li>Grandoreiro is a multi-component banking trojan likely operated as a Malware-as-a-Service (MaaS).</li>
<li>It is actively deployed in phishing campaigns impersonating government entities in Mexico, Argentina and South Africa.</li>
<li>The banking trojan specifically targets over 1500 global banking applications and websites in over 60 countries including regions in Central/South America, Africa, Europe, and the Indo-Pacific.</li>
<li>The latest variant contains major updates including string decryption and DGA calculation, allowing at least 12 different C2 domains per day.</li>
<li>Grandoreiro supports harvesting email addresses from infected hosts and using their Microsoft Outlook client to send out further phishing campaigns.</li>
</ul>
<h2>Grandoreiro operators expand campaigns</h2>
<h3>LATAM-focused campaigns</h3>
<p>Since March 2024, X-Force has observed phishing campaigns impersonating Mexico&rsquo;s Tax Administration Service (SAT), Mexico&rsquo;s Federal Electricity Commission (CFE), the Secretary of Administration and Finance for the city of Mexico, and the Revenue Service of Argentina. The emails target users within Latin America, including top-level domains (TLDs) from&nbsp;Mexico, Colombia, and Chile &ldquo;<b>.mx</b>&ldquo;, &ldquo;<b>.co</b>&ldquo;, and &ldquo;<b>.cl</b>&ldquo;. Any real identities have been redacted from the images for personal privacy.</p>
<p>The first campaign appears to be an attempt to be perceived as official and urgent and informs the target that they are receiving a final notice regarding a debit to the Federal Taxpayer Registration Fee (RFC) that has not been paid. If unpaid, consequences may include penalties, fines and a block on the user&rsquo;s tax identification number impacting the target&rsquo;s ability to conduct business and access government services legally. An additional campaign impersonates Mexico&rsquo;s Federal Electricity Commission (CFE) and reminds the recipient that they subscribed to CFEMail, and therefore can access their account statement in PDF and XML format by clicking one of the embedded links. A third campaign imitating the Secretary of Administration and Finance, directs the recipient to click on a PDF to read details regarding a compliance notice. A campaign imitating the Revenue Service of Argentina instructs the user to download a new tax document and take applicable actions.</p>
<p>In each campaign, the recipients are instructed to click on a link to view an invoice or fee, account statement, make a payment, etc. depending on the impersonated entity. If the user who clicks on the links is within a specific country (depending on the campaign, Mexico, Chile, Spain, Costa Rica, Peru, or Argentina), they are redirected to an image of a PDF icon, and a ZIP file is downloaded in the background. The ZIP files contain a large executable disguised with a PDF icon, found to have been created the day prior to, or the day of the email being sent.</p>

<h3>Campaign Impersonating the South African Revenue Service</h3>
<p>Typically Grandoreiro malware is seen in campaigns that target users within Latin America; however, after recent arrests made involving Grandoreiro operators, X-Force has seen a surge in campaigns reaching areas outside of LATAM, including TLDs from Spain, Japan, the Netherlands, and Italy. X-Force observed a phishing campaign impersonating the South African Revenue Service (SARS), purporting to be from the Taxpayer Assistance Services Division. Likely executed by the same operator, X-Force also observed two campaigns impersonating the Tax Administration Service of Mexico. Emails are written in either English, or Spanish, and resemble the same format. The emails reference a Tax number and inform the recipient that they are receiving an electronic tax invoice that is in compliance with the regulations set forth by the South African Revenue Service, or in accordance with the regulations of the Tax Administration Service. The user is provided both a PDF or XML link to view the invoice which initiates a ZIP archive download containing the Grandoreiro loader executable &ldquo;<b>SARS 35183372 eFiling 32900947.exe</b>&rdquo; (digits vary between samples).</p>
<p><amp-img src="https://images4.cmp.optimizely.com/Zz1iOGZmODY5ZTEzNzgxMWVmOTM0NGM2NTQ4N2FkMmYzNw==" layout="intrinsic" class="" alt="SAT related to SARs.png" width="720" height="452.84609978917774" lightbox="lightbox"></amp-img></p>

<h2>Analysis: Grandoreiro Loader</h2>
<p>In line with previous campaigns, Grandoreiro&rsquo;s infection chain begins with a custom loader. Often, the executable is bloated to a size of more than 100MB to hinder automatic anti-virus scanning. In hopes of circumventing automated execution, it displays a small CAPTCHA pop-up imitating Adobe PDF reader, which requires a click to continue with the execution.</p>
<p><amp-img src="https://images2.cmp.optimizely.com/Zz0yYWE0NGRlODEzNzkxMWVmYjM3MjdhNzcxZDQ1YmY2OQ==" layout="intrinsic" class="" alt="PDF Icon.png" width="423" height="189" lightbox="lightbox"></amp-img></p>
<p><i>Fig 8: Grandoreiro fake Adobe PDF reader CAPTCHA</i></p>
<p>The loader has three main tasks:</p>
<ol start="1" type="1">
<li>Verify if the client is a legitimate victim (not a researcher or a sandbox)</li>
<li>Enumerate basic victim data and send it back to its C2</li>
<li>Download, decrypt and execute the Grandoreiro banking trojan</li>
</ol>
<h3>String decryption</h3>
<p>All of these tasks require more than 120 important strings, which are encrypted using an improved algorithm.</p>
<p>First, Grandoreiro starts by generating a large key string, which is hardcoded and triple-Base64-encoded. The key observed in these samples begins with <b>&ldquo;D9JL@2]790B{P_D}Z-MXR&amp;EZLI%3W&gt;#VQ4UF+O6XVWB16713NIO!E&hellip;&rdquo;.&nbsp;</b>It then takes the encrypted string and uses a custom decoding to convert it into a series of hexadecimal characters interpreted as bytes.</p>
<p><amp-img src="https://images1.cmp.optimizely.com/Zz01MDBlMWYxZTEzNzkxMWVmODdkNTAyMWMxMmYxNWNmOA==" layout="intrinsic" class="" alt="Key string.png" width="570" height="381" lightbox="lightbox"></amp-img></p>
<p><i>Fig 9: Grandoreiro custom hex encoding (note that non-hex character encoding like &lsquo;&rdquo;&lsquo; are never used)</i></p>
<p>Grandoreiro decrypts the result via the old Grandoreiro algorithm using the key string. Below is a Python implementation of the decryption routine:</p>
<p><amp-img src="https://images4.cmp.optimizely.com/Zz0xZTlmYTVlNDEzODYxMWVmOGU5ZmM2NTQ4N2FkMmYzNw==" layout="intrinsic" class="" alt="code snippet.png" width="673" height="183" lightbox="lightbox"></amp-img></p>
<p>Lastly, it undergoes a final round of 256-bit AES CBC decryption and unpadding to retrieve the plaintext string. Both the AES key and Initiation Vector (IV) are also stored as encrypted strings and have to be decrypted using the same algorithm as above, however skipping the AES decryption. The graph below gives an overview of the full decryption process:</p>
<p><amp-img src="https://images2.cmp.optimizely.com/Zz0yYjU0NmRhYTEzN2QxMWVmOGEyZjU2NDA5MzM4MjFjMQ==" layout="intrinsic" class="" alt="Loader String decryption.png" width="640" height="635" lightbox="lightbox"></amp-img></p>

<h3>Victim verification</h3>
<p>To verify that a victim is not part of a sandboxed environment, the Grandoreiro loader collects the following information and checks it against a list of hardcoded values (see Appendix):</p>
<ol start="1" type="1">
<li>Computer name</li>
<li>Username</li>
<li>OS version information</li>
<li>Installed Antivirus solution</li>
<li>Country of the victim&rsquo;s public IP (via http://ip-api.com/json)</li>
<li>List of running processes</li>
</ol>
<p>This verification step is also used to disallow victims of specific countries. One sample did not continue execution for infections with public IPs from:</p>
<ul type="disc">
<li>Russia</li>
<li>Czechia</li>
<li>Poland</li>
<li>Netherlands</li>
</ul>
<p>The sample also prevented infections of Windows 7 machines based in the US without anti-virus.</p>

<p>The next execution step attempts to build a basic profile of the victim to display on the C2 panel. The malware enumerates the following information on the victim machine:</p>
<ul type="disc">
<li>Public IP country</li>
<li>Public IP region</li>
<li>Public IP city</li>
<li>Computer name</li>
<li>Username</li>

<li>Check if crypto-wallets exist: Binance, Electrum, Coinomi, Bitbox, OPOLODesk, Bitcoin</li>

<h3>C2 communication and loading Grandoreiro</h3>
<p>Grandoreiro loader&rsquo;s C2 server can be decrypted via the same algorithm explained above. The resulting domain name is resolved via DNS over HTTPS through the URL&nbsp;<b>https://dns.google/resolve?name=&lt;C2 server&gt;&nbsp;</b>to circumvent DNS-based blocking. After receiving the C2 IP address, the malware takes the first 4 digits of the IP and runs 4 different digit-to-digit mappings over it resulting in the 4-digit port number.</p>
<p>It then concatenates the victim profiling string from above together with a capitalized Portuguese message &ldquo;CLIENT_SOLICITA_DDS_MDL&rdquo; (likely translated to &ldquo;Client asks for module data&rdquo;). An example string would be:</p>
<p><amp-img src="https://images3.cmp.optimizely.com/Zz0yNDcyNGViODEzOTAxMWVmYjUzYWNhZmRlZmQ2ZjQ2YQ==" layout="intrinsic" class="" alt="code snip - c2 communication 1.png" width="670" height="96" lightbox="lightbox"></amp-img></p>
<p>The string is encrypted and sent as the URL path via an HTTP GET request to the C2 server requesting the final Grandoreiro payload.</p>
<p>If successful, the C2 server replies with an HTTP 200 status code containing another encrypted message. It contains the following information:</p>
<ol start="1" type="1">
<li>Payload download URL</li>
<li>C2 server</li>
<li>Directory name</li>
<li>Payload name</li>
<li>Payload size</li>
</ol>
<p>Example:</p>
<p><amp-img src="https://images2.cmp.optimizely.com/Zz02Nzc3NmU3ODEzOTAxMWVmYjU2MmRhMGUyYmFmMzZkNQ==" layout="intrinsic" class="" alt="code snip - c2 communication 2.png" width="684" height="85" lightbox="lightbox"></amp-img></p>
<p>To download, Grandoreiro issues another HTTP GET request to the payload URL. The downloaded file is stored in the specified directory name under &ldquo;<b>C:\ProgramData\</b>&ldquo;. Next, the file is decrypted via an RC4-based algorithm using the key &ldquo;7684223510&rdquo;. Finally, it is decompressed using the &ldquo;ZipForge&rdquo; Delphi library, and the originally downloaded file is deleted.</p>
<p>The archive may contain two files, a .EXE (Grandoreiro banking trojan) and a .CFG (config file).</p>
<p>Prior to execution, the loader performs an enumeration of the current process token&rsquo;s group membership, specifically checking for the presence of the SECURITY_NT_AUTHORITY SID. If the process possesses the required privileges, the loader utilizes the <i>ShellExecuteW()</i> function with the &lsquo;runas&rsquo; verb to execute the Grandoreiro payload with elevated privileges. Conversely, if the necessary privileges are not available, the loader resorts to executing itself via<i> ShellExecuteW()</i> without elevation.</p>
<p>During all stages of infection&mdash;the payload download, decryption, and execution&mdash;the Grandoreiro loader reports back status messages to its C2 server. Some examples are:</p>

<li>ERRO_FALHA_DOWNLOAD (&ldquo;Download failed error&rdquo;)</li>
<li>ERRO_EXTRACAO (&ldquo;Extraction error&rdquo;)</li>

<li>ERRO_EXECUCAO (&ldquo;Execution error&rdquo;)</li>
<li>INFECTADO (&ldquo;Infected&rdquo;)</li>
</ul>
<h2>Grandoreiro Banking Trojan</h2>
<p>The final payload is the Grandoreiro banking trojan. The latest version has undergone major updates mainly within the string decryption and DGA calculation algorithms. It has also included a vast number of global banking applications to target, support execution and enable attackers to perform banking fraud in dozens of countries. Together with a specialized Outlook spreader module and a wide range of features, it is one of the largest known banking trojans and analysis is still ongoing. The following sections present an in-depth look at Grandoreiro&rsquo;s most notable characteristics, highlighting its essential features and functionalities.</p>
<h3>Persistence and configuration</h3>
<p>Grandoreiro begins by establishing persistence via the Windows registry. It runs the following command to create a new registry Run key and launch the malware on user login:</p>
<p><amp-img src="https://images4.cmp.optimizely.com/Zz0zODFiNWMwZTEzODkxMWVmOWY5NDdhYmVlNDg4ZDcwNw==" layout="intrinsic" class="" alt="code snippet - persistence.png" width="655" height="96" lightbox="lightbox"></amp-img></p>
<p>Note that the name of the key may differ among samples, but is often related to the original filename of the downloaded payload. If Grandoreiro does not run in an elevated process, the &ldquo;/runas&rdquo; verb is omitted.</p>
<p>In addition to the .CFG file, Grandoreiro also creates a .XML file in the<b> C:\Public\&nbsp;</b>directory. It is encrypted via the loader&rsquo;s string encryption routine and stores the Grandoreiro executable filename, path and date of infection.</p>
<p>If Grandoreiro can&rsquo;t find its .CFG file, it will populate a new .CFG with default values specifying which Grandoreiro functions are enabled, the victim&rsquo;s country and date of infection. The .CFG file is encrypted via the Grandoreiro string encryption algorithm explained further below.</p>
<h3>Targeted applications</h3>
<p>Grandoreiro operators significantly upgraded the list of targeted banking applications, now targeting more than 1500 banks worldwide. The latest variants start by first determining if the victim is on the list of targeted countries. Each country is also mapped to a larger region, which Grandoreiro uses to determine which string searches it should run on currently active windows. This means that, if the victim country for instance is identified as Belgium, it will search for all targeted banking applications associated with the Europe region. Grandoreiro internally maps countries to the region categories <b>Europe, North America, Central America, South America, Africa, Indo-Pacific and global islands</b>, with each region having an associated Delphi class to search for bank applications. In addition, Grandoreiro has a class searching for 266 unique strings identifying cryptocurrency wallets, which is run on every infection.</p>

<h3>DGA</h3>
<p>Grandoreiro has traditionally relied on domain generation algorithms (DGA) to calculate its active C2 server based on the current date. The newest iteration of Grandoreiro contains a reworked algorithm and takes it one step further by introducing multiple seeds for its DGA. These seeds are used to calculate a different domain for each mode or functionality of the banking trojan, allowing separation of C2 tasks among several operators as part of their Malware-as-a-Service operation. Each Grandoreiro sample may have a main default seed in case the config file is missing, as well as a list of function-specific seeds. The sample X-Force analyzed contained 14 different seeds, leading to 14 possible C2 domains every day. To explain the algorithm, we will calculate the domains for April 17, 2024. The following chart provides a visualization of the algorithm with an explanation below:</p>
<p><amp-img src="https://images1.cmp.optimizely.com/Zz1iODQ4YThlYTEzN2IxMWVmYjBhMjZlNjkxMWEwYzQzYw==" layout="intrinsic" class="" alt="Grandoreiro DGA.png" width="720" height="496.2659846547314" lightbox="lightbox"></amp-img></p>
<p><i>Fig 13: DGA visualization</i></p>
<p>Starting with the domain apex, Grandoreiro has one domain mapped to every day of the year. There are two of these mappings, one for the main C2 and one for all function-specific C2s. However, of the 732 apex domains, only 337 are unique. For the given day, the primary apex is&nbsp;<b>dnsfor[.]me</b> and the secondary is <b>neat-url[.]com</b>.</p>
<p>For the next part, Grandoreiro concatenates the seed &ldquo;xretsmzrb&rdquo; (the main seed) with the 2 digit formatted current month, replacing each digit with three hardcoded characters. The digits &ldquo;0&rdquo; and &ldquo;4&rdquo; are replaced with &ldquo;oit&rdquo; and &ldquo;zia&rdquo; respectively, resulting in the full string &ldquo;xretsmzrboitzia&rdquo;.</p>
<p>Finally, for each day of the month, Grandoreiro has a custom character to character replacement mapping. For the 17th, after running all 26 character replacements iteratively, the final subdomain string is &ldquo;wondbbhonandhnd&rdquo;.</p>

<h3>Outlook Harvest &amp; Spam</h3>
<p>One of Grandoreiro&rsquo;s most interesting features is its capability to spread by harvesting data from Outlook and using the victim&rsquo;s account to send out spam emails. There are at least 3 mechanisms implemented in Grandoreiro to harvest and exfiltrate email addresses, with each using a different DGA seed. By using the local Outlook client for spamming, Grandoreiro can spread through infected victim inboxes via email, which likely contributes to the large amount of spam volume observed from Grandoreiro.</p>
<h4>Harvesting</h4>
<p>For the Outlook harvesting mode, Grandoreiro switches its C2 to DGA seed 7 which is used to exfiltrate data. Logging and status messages continue to the main C2 server. For instance, before starting the harvesting process, it sends a log back containing the same victim profiling data as well as the strings &ldquo;CLIENT_SOLICITA_DD_EMSOUT&rdquo; (Client asks for EMSOUT data) and &ldquo;COLHENDO&rdquo; (harvesting).</p>
<p>In order to interact with the local Outlook client, Grandoreiro uses the <a href="https://www.add-in-express.com/outlook-security/" target="_blank" rel="noopener nofollow" >Outlook Security Manager tool</a>, a software used to develop Outlook add-ins. The main reason behind this is that the Outlook Object Model Guard triggers security alerts if it detects access on protected objects. Outlook Security Manager allows Grandoreiro to disable these alerts during both the harvesting and spamming behavior. Depending on system architecture, the tool requires the DLL &ldquo;secman.dll&rdquo; or &ldquo;secman64.dll&rdquo; to be registered as COM servers. It then uses MAPI to interact with Outlook.</p>
<p>The malware begins by locating the root mailbox folder and then recursively iterates through the email items. For each email, it checks the &ldquo;SenderEmailAddress&rdquo; property and runs a blocklist against it, to filter out unwanted email addresses for harvesting:</p>

<h4>Spamming</h4>
<p>To send out spam emails, Grandoreiro uses phishing templates which it receives from its C2 server. It then goes through the template and fills out placeholder fields such as:</p>
<ul type="disc">
<li>$replyto &rarr; the Reply-to value</li>
<li>$link &rarr; a link to the payload</li>
<li>$hora &rarr; formatted current time</li>
<li>$data &rarr; formatted current date</li>
<li>$email_destino &rarr; destination address</li>
<li>$valor &rarr; A randomly generated float value such as &ldquo;123,45.67&rdquo;, likely used to create random invoice values</li>
<li>$letnum_rand_branco &rarr; random string of capital letters and digits, pasted into the email HTML between white font tags &ldquo;&lt;font style=&rdquo;color: white;&rdquo;&gt;&rdquo;. Use unknown.</li>
<li>$assunto &rarr; email subject</li>
<li>$nome_saudacao &rarr; name and greeting</li>
<li>$nome_empresa &rarr; company name</li>
<li>$link_imagem &rarr; link to image, likely to support company logos, signatures or banners</li>
</ul>
<p>Just before beginning to send out emails, Grandoreiro starts a thread to detect any appearing dialog boxes and click them away by sending specific TAB and SPACEBAR key presses. After sending out the emails, the malware carefully covers its tracks by deleting the sent messages from the victim&rsquo;s mailbox. Also, for a lot of the harvesting and spamming behavior Grandoreiro makes sure that the last input on the infected machine is at least 5min ago (or in some cases longer). The developers likely wanted to make sure victims would not notice any suspicious behavior.</p>

<ul type="disc">
<li>PRONTO (&ldquo;Ready&rdquo;)</li>
<li>EM_REPOUSO (&ldquo;In rest&rdquo;)</li>

<h3>String encryption</h3>
<p>With Grandoreiro being such an extensively large malware, it requires a huge amount of strings, which would make detection very easy if they were left unencrypted. Grandoreiro features more than 10k strings dispersed among more than a hundred feature-specific string-loading functions. The decryption mechanism differs slightly from the loader&rsquo;s string decryption:</p>
<p>It uses the same Grandoreiro key as the loader, which it decrypts via its custom encryption and the key &ldquo;A&rdquo;. Once it has the key, it custom-decodes the encrypted string using the same encoding as the loader and then decrypts the resulting bytes via AES ECB mode using the <a href="https://github.com/AArhin/delphi-oop/blob/master/Persistence/Externals/NativeXml/extlib/eldos_aes/ElAES.pas" target="_blank" rel="noopener nofollow" >ElAES Pascal implementation</a>. The AES key is a scrambled version of the previously decrypted Grandoreiro key. After another round of custom decoding, the string is finally decrypted via the old Grandoreiro algorithm and the Grandoreiro key.</p>

<p>X-Force&nbsp;observed several recent phishing campaigns impersonating official government entities to deliver the Grandoreiro banking trojan. Grandoreiro distributors typically target users in Latin America; however, since the latest <a href="https://www.interpol.int/en/News-and-Events/News/2024/Disrupting-a-Grandoreiro-malware-operation" target="_blank" rel="noopener nofollow" >law enforcement action</a> against Grandoreiro operators, X-Force has observed the malware being spread outside of LATAM to include regions in Central and South America, Africa, Europe, and the Pacific. The Grandoreiro banking trojan samples that X-Force has analyzed have undergone major updates within the string decryption and DGA Calculation algorithms. These newly analyzed samples now include a vast number of at least 1500 global banking applications to target, which support execution and enable attackers to perform banking fraud in over 60 countries. The updates made to the malware, in addition to the significant increase in banking applications across several nations, indicate that the Grandoreiro distributors are seeking to conduct campaigns and deliver malware on a global scale.</p>
<p>We encourage organizations that may be impacted by these campaigns to review the following recommendations:</p>
<ul type="disc">
<li>Exercise caution with emails and PDFs prompting a file download</li>
<li>Monitor network traffic for multiple consecutive requests to&nbsp;<a href="http://ip-api.com/json" target="_blank" rel="noopener nofollow" >http://ip-api.com/json</a> as a potential indicator of a Grandoreiro infection</li>
