            <div class="blog-post__body">
              <span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p><span style="color: #cafa3c;">Coauthored by Karthickkumar Kathiresan of Uptycs Threat Research Team</span></p>
<!--more-->
<p>In a sophisticated twist to the traditional sideloading tactics, the Quasar RAT introduces a novel dual DLL sideloading technique, ingeniously utilizing two commonly trusted Microsoft files: "ctfmon.exe" and "calc.exe." Such a method not only leverages the inherent trust these files enjoy within the Windows ecosystem but also presents an increased challenge to threat detection mechanisms. This article dives deep into the meticulous design and execution of these sideloading techniques, illustrating how they stealthily introduce, deploy, and run malicious payloads under the radar.</p>
<p>&nbsp;</p>
<p><span style="color: #c092fb;"><a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.quasar_rat" style="color: #c092fb;">QuasarRAT</a></span>, also known as CinaRAT or Yggdrasil, is a lightweight remote administration tool written in C#. This tool is openly accessible as a GitHub project. This tool is capable of various functions such as gathering system data, running applications, transferring files, recording keystrokes, taking screenshots or camera captures, recovering system passwords, and overseeing operations like File Manager, Startup Manager, Remote Desktop, and executing shell commands.</p>
<p>&nbsp;</p>
<p>Windows users, system administrators, and cybersecurity professionals need to be on high alert. The use of legitimate processes to cloak malicious activities helps them bypass traditional security measures. Hence, the need for advanced threat detection and response mechanisms becomes paramount.&nbsp;</p>
<p>&nbsp;</p>
<div class="hs-cta-embed hs-cta-simple-placeholder hs-cta-embed-141040881575" style="max-width: 100%; max-height: 100%; width: 700px; height: 248.9453125px;" data-hubspot-wrapper-cta-id="141040881575"><a href="https://cta-service-cms2.hubspot.com/web-interactives/public/v1/track/redirect?encryptedPayload=AVxigLJd9SyFPwGR4Sg7Wk7mQacZvu6UmBZx3C2VxEt5t%2BrKZNjWtizIQgolW4t%2BQv7qtGqYMVj6SNkumdlUdIadeZYY61vZGIuqe6aOME6VWLqR%2B94N2kZD00wYf1qaPJSU8LTH%2BbOMtwRd%2BOcA0L3myj8aEShrESu34Znix9BZ7kmiysG8SwJCpwSSSaIbFPX2EjBm%2FV5lXw5b6DcLeHb1cvxJtqX57Q%3D%3D&amp;webInteractiveContentId=141040881575&amp;portalId=2617658" target="_blank" rel="noopener" crossorigin="anonymous"> <img alt="Q3 Threat Intelligence Report Get the latest insights on security threats affecting Linux, Windows, and macOS platforms in 2023 &nbsp;" loading="lazy" src="https://no-cache.hubspot.com/cta/default/2617658/interactive-141040881575.png" style="height: 100%; width: 100%; object-fit: fill;" onerror="this.style.display='none'"> </a></div>
<p>&nbsp;</p>
<h2><span style="color: #c092fb;">QuasarRAT sideloading execution: a closer look at the technique</span></h2>
<p>Given the prevalence of sideloading techniques in malware campaigns, it's vital to understand their mechanisms to defend against them effectively. The case of QuasarRAT provides an insightful example.<br><br><span style="font-weight: bold;">Historical context:</span><br><br>In 2022, we detected the Qbot malware employing a DLL sideloading attack using "calc.exe." Such tactics are not new but seeing them evolve and get adopted by other malware strains shows the adaptability of threat actors. Now, in 2023, a strikingly parallel method in two phases has been observed with the QuasarRAT malware.<br><br><span style="font-weight: bold;">Step-by-step breakdown:</span><br><br></p>
<p style="font-size: 16px;"><span style="background-color: transparent; color: #efeff1; font-family: Dazzed; font-weight: bold;">1</span><span style="background-color: transparent; color: #efeff1; font-family: Dazzed; font-weight: normal;">.&nbsp; <span style="font-weight: bold;">Initial contact and execution:</span></span></p>
<span style="background-color: transparent; color: #efeff1; font-family: Dazzed;">The threat actor begins by employing DLL side-loading techniques. Interestingly, they opted for two distinct Microsoft files for their attack: "ctfmon.exe" and "calc.exe."</span><br><br>
<ul style="font-size: 16px;">
<li><span style="background-color: transparent; color: #efeff1; font-family: Dazzed; font-weight: normal;">In the initial phase, the attacker harnesses "ctfmon.exe," which is an authentic Microsoft file. By doing so, they load a malicious DLL which, to the untrained eye, would seem benign because of its disguised name.</span></li>
<li><span style="background-color: transparent; color: #efeff1; font-family: Dazzed; font-weight: normal;">Upon execution of the "ctfmon.exe" binary, the stage is set as the attacker acquires a 'stage 1' payload. This initial payload is crucial, acting as the gateway for the subsequent malicious actions.</span></li>
</ul>
<p><span style="font-size: 16px;"><strong>2.&nbsp; <span style="background-color: transparent;">Payload release:</span></strong></span></p>
<p><span style="font-size: 16px;">This 'stage 1' payload plays a dual role. It is responsible for releasing both the legitimate "calc.exe" file and the malevolent DLL into the system.</span><br><br></p>
<p><span style="font-size: 16px;"><strong>3. Second phase of attack:</strong></span></p>
<ul>
<li><span style="font-size: 16px;">At this juncture, the threat actor brings into play the "calc.exe" file, which in this context, isn't just a simple calculator application. Alongside "calc.exe," the malicious DLL is also set into motion.</span></li>
<li><span style="font-size: 16px;">On executing "calc.exe," the malicious DLL is triggered. This action culminates in the infiltration of the "QuasarRAT" payload into the computer’s memory, reflecting the attacker's adeptness at circumventing security mechanisms.</span></li>
</ul>
<p style="text-align: left;"><span style="font-size: 14px;"></span><span style="font-size: 16px;"><strong>4. Process hollowing:</strong></span><br><span style="font-size: 16px;">With the "QuasarRAT" payload now residing in the computer’s memory, the payload employs a technique known as 'process hollowing.' Here, it embeds itself into a legitimate system process, further camouflaging its malicious intentions and making detection more challenging.</span><br><br><span style="font-size: 16px;">Figure 1 depicts the QuasarRAT workflow.</span></p>
<p style="text-align: center;"><br><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%201-1.jpg?width=940&amp;height=473&amp;name=Fig%201-1.jpg" width="940" height="473" loading="lazy" alt="Figure 1 – QuasarRAT execution flow" style="height: auto; max-width: 100%; width: 940px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%201-1.jpg?width=470&amp;height=237&amp;name=Fig%201-1.jpg 470w, https://www.uptycs.com/hs-fs/hubfs/Fig%201-1.jpg?width=940&amp;height=473&amp;name=Fig%201-1.jpg 940w, https://www.uptycs.com/hs-fs/hubfs/Fig%201-1.jpg?width=1410&amp;height=710&amp;name=Fig%201-1.jpg 1410w, https://www.uptycs.com/hs-fs/hubfs/Fig%201-1.jpg?width=1880&amp;height=946&amp;name=Fig%201-1.jpg 1880w, https://www.uptycs.com/hs-fs/hubfs/Fig%201-1.jpg?width=2350&amp;height=1183&amp;name=Fig%201-1.jpg 2350w, https://www.uptycs.com/hs-fs/hubfs/Fig%201-1.jpg?width=2820&amp;height=1419&amp;name=Fig%201-1.jpg 2820w" sizes="(max-width: 940px) 100vw, 940px"><span style="font-size: 14px;"><em>Figure 1 – QuasarRAT execution flow</em></span></p>
<p>&nbsp;</p>
<h2 style="font-weight: bold;"><span style="color: #c092fb;">Technical analysis</span></h2>
<p>We first verified the ISO file and after a successful extraction, we obtained three separate files:</p>
<p>&nbsp;</p>
<ol>
<li>eBill-997358806.exe - Legitimate windows file, actual name is CTFMON.EXE&nbsp;</li>
<li>monitor.ini - &nbsp; &nbsp;Legitimate windows file, actual name is (MsCtfMonitor.DLL)&nbsp;</li>
<li>MsCtfMonitor.dll - Malicious Dll</li>
</ol>
<p>Figure 2 depicts the Process flow of the new QuasarRAT.<br><br></p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%202-1.jpg?width=1329&amp;height=107&amp;name=Fig%202-1.jpg" width="1329" height="107" loading="lazy" alt="Figure 2 – QuasarRAT process tree" style="height: auto; max-width: 100%; width: 1329px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%202-1.jpg?width=665&amp;height=54&amp;name=Fig%202-1.jpg 665w, https://www.uptycs.com/hs-fs/hubfs/Fig%202-1.jpg?width=1329&amp;height=107&amp;name=Fig%202-1.jpg 1329w, https://www.uptycs.com/hs-fs/hubfs/Fig%202-1.jpg?width=1994&amp;height=161&amp;name=Fig%202-1.jpg 1994w, https://www.uptycs.com/hs-fs/hubfs/Fig%202-1.jpg?width=2658&amp;height=214&amp;name=Fig%202-1.jpg 2658w, https://www.uptycs.com/hs-fs/hubfs/Fig%202-1.jpg?width=3323&amp;height=268&amp;name=Fig%202-1.jpg 3323w, https://www.uptycs.com/hs-fs/hubfs/Fig%202-1.jpg?width=3987&amp;height=321&amp;name=Fig%202-1.jpg 3987w" sizes="(max-width: 1329px) 100vw, 1329px"><span style="font-size: 14px;"><em>Figure 2 – QuasarRAT process tree</em></span></p>
<p>&nbsp;</p>
<p>When the binary file "eBill-997358806.exe" is run, it initiates the loading of a file titled "MsCtfMonitor.dll" &nbsp;(<span style="color: #c092fb;"><a href="https://attack.mitre.org/techniques/T1036/" rel="noopener" target="_blank" style="color: #c092fb;">name masqueraded</a></span>) via <span style="color: #c092fb;"><a href="https://attack.mitre.org/techniques/T1574/002/" rel="noopener" target="_blank" style="color: #c092fb;">dll side loading technique</a></span>, within which malicious code is concealed.<br><br>Within the "MsCtfMonitor.dll" file, there exists a resource section containing encrypted data.&nbsp;<br>The resource section (RCDATA:400) has encrypted data of size 5AC00 hex bytes.</p>
<p>&nbsp;</p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%203-1.jpg?width=851&amp;height=298&amp;name=Fig%203-1.jpg" width="851" height="298" loading="lazy" alt="Figure 3 – Resource section of MsCtfMonitor.dll containing encrypted data" style="height: auto; max-width: 100%; width: 851px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%203-1.jpg?width=426&amp;height=149&amp;name=Fig%203-1.jpg 426w, https://www.uptycs.com/hs-fs/hubfs/Fig%203-1.jpg?width=851&amp;height=298&amp;name=Fig%203-1.jpg 851w, https://www.uptycs.com/hs-fs/hubfs/Fig%203-1.jpg?width=1277&amp;height=447&amp;name=Fig%203-1.jpg 1277w, https://www.uptycs.com/hs-fs/hubfs/Fig%203-1.jpg?width=1702&amp;height=596&amp;name=Fig%203-1.jpg 1702w, https://www.uptycs.com/hs-fs/hubfs/Fig%203-1.jpg?width=2128&amp;height=745&amp;name=Fig%203-1.jpg 2128w, https://www.uptycs.com/hs-fs/hubfs/Fig%203-1.jpg?width=2553&amp;height=894&amp;name=Fig%203-1.jpg 2553w" sizes="(max-width: 851px) 100vw, 851px"><span style="font-size: 14px;"><em>Figure 3 – Resource section of MsCtfMonitor.dll containing encrypted data</em></span></p>
<p>&nbsp;</p>
<p>This data is accessed via a sequence of APIs as seen in Figure 4.</p>
<p>&nbsp;</p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%204-1.jpg?width=547&amp;height=235&amp;name=Fig%204-1.jpg" width="547" height="235" loading="lazy" alt="Figure 4 – Loading of resources" style="height: auto; max-width: 100%; width: 547px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%204-1.jpg?width=274&amp;height=118&amp;name=Fig%204-1.jpg 274w, https://www.uptycs.com/hs-fs/hubfs/Fig%204-1.jpg?width=547&amp;height=235&amp;name=Fig%204-1.jpg 547w, https://www.uptycs.com/hs-fs/hubfs/Fig%204-1.jpg?width=821&amp;height=353&amp;name=Fig%204-1.jpg 821w, https://www.uptycs.com/hs-fs/hubfs/Fig%204-1.jpg?width=1094&amp;height=470&amp;name=Fig%204-1.jpg 1094w, https://www.uptycs.com/hs-fs/hubfs/Fig%204-1.jpg?width=1368&amp;height=588&amp;name=Fig%204-1.jpg 1368w, https://www.uptycs.com/hs-fs/hubfs/Fig%204-1.jpg?width=1641&amp;height=705&amp;name=Fig%204-1.jpg 1641w" sizes="(max-width: 547px) 100vw, 547px"><span style="font-size: 14px;"><em>Figure 4 – Loading of resources</em></span></p>
<p>&nbsp;</p>
<p>To decrypt the data, the key size is F2 hex bytes and <strong>Systemfunction032</strong> API is used to decrypt the encrypted data. This is an undocumented API that indirectly calls <strong>BCryptGeneratesymmetrickey, CryptEncrypt, and CryptDestroyKey </strong>to decrypt data by RC4 where the key is also stored in .rsrc section (RCDATA: 401)</p>
<p>&nbsp;</p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%205-1.jpg?width=877&amp;height=475&amp;name=Fig%205-1.jpg" width="877" height="475" loading="lazy" alt="Figure 5– Resource section of MsCtfMonitor.dll containing decryption key" style="height: auto; max-width: 100%; width: 877px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%205-1.jpg?width=439&amp;height=238&amp;name=Fig%205-1.jpg 439w, https://www.uptycs.com/hs-fs/hubfs/Fig%205-1.jpg?width=877&amp;height=475&amp;name=Fig%205-1.jpg 877w, https://www.uptycs.com/hs-fs/hubfs/Fig%205-1.jpg?width=1316&amp;height=713&amp;name=Fig%205-1.jpg 1316w, https://www.uptycs.com/hs-fs/hubfs/Fig%205-1.jpg?width=1754&amp;height=950&amp;name=Fig%205-1.jpg 1754w, https://www.uptycs.com/hs-fs/hubfs/Fig%205-1.jpg?width=2193&amp;height=1188&amp;name=Fig%205-1.jpg 2193w, https://www.uptycs.com/hs-fs/hubfs/Fig%205-1.jpg?width=2631&amp;height=1425&amp;name=Fig%205-1.jpg 2631w" sizes="(max-width: 877px) 100vw, 877px"><span style="font-size: 14px;"><em>Figure 5– Resource section of MsCtfMonitor.dll containing decryption key</em></span></p>
<p>&nbsp;</p>
<p>After decryption, the resource data is decrypted and give’s PE file which is stage 1 "FileDownloader.exe."</p>
<p>&nbsp;</p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%206-1.jpg?width=1118&amp;height=268&amp;name=Fig%206-1.jpg" width="1118" height="268" loading="lazy" alt="Figure 6– Decrypted data" style="height: auto; max-width: 100%; width: 1118px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%206-1.jpg?width=559&amp;height=134&amp;name=Fig%206-1.jpg 559w, https://www.uptycs.com/hs-fs/hubfs/Fig%206-1.jpg?width=1118&amp;height=268&amp;name=Fig%206-1.jpg 1118w, https://www.uptycs.com/hs-fs/hubfs/Fig%206-1.jpg?width=1677&amp;height=402&amp;name=Fig%206-1.jpg 1677w, https://www.uptycs.com/hs-fs/hubfs/Fig%206-1.jpg?width=2236&amp;height=536&amp;name=Fig%206-1.jpg 2236w, https://www.uptycs.com/hs-fs/hubfs/Fig%206-1.jpg?width=2795&amp;height=670&amp;name=Fig%206-1.jpg 2795w, https://www.uptycs.com/hs-fs/hubfs/Fig%206-1.jpg?width=3354&amp;height=804&amp;name=Fig%206-1.jpg 3354w" sizes="(max-width: 1118px) 100vw, 1118px"><span style="font-size: 14px;"><em>Figure 6– Decrypted data</em></span></p>

<p>This PE File is then injected into Regasm.exe by the following sequence of API : CreateProcess, GetThreadContext, ReadProcessMemory, VirtualAllocEx and WriteProcessMemory, GetThreadContext, SetThreadContext and ResumeThread.</p>

<h3><span style="color: #c092fb;">Stage 1: FileDownloader.exe</span></h3>
<p>The stage 1 payload is a 64-bit MSIL binary file, which includes a resource section containing three binaries stored in a zip archive format.&nbsp;</p>
<p>&nbsp;</p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%208-1.jpg?width=1000&amp;height=468&amp;name=Fig%208-1.jpg" width="1000" height="468" loading="lazy" alt="Fig 8-1" style="height: auto; max-width: 100%; width: 1000px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%208-1.jpg?width=500&amp;height=234&amp;name=Fig%208-1.jpg 500w, https://www.uptycs.com/hs-fs/hubfs/Fig%208-1.jpg?width=1000&amp;height=468&amp;name=Fig%208-1.jpg 1000w, https://www.uptycs.com/hs-fs/hubfs/Fig%208-1.jpg?width=1500&amp;height=702&amp;name=Fig%208-1.jpg 1500w, https://www.uptycs.com/hs-fs/hubfs/Fig%208-1.jpg?width=2000&amp;height=936&amp;name=Fig%208-1.jpg 2000w, https://www.uptycs.com/hs-fs/hubfs/Fig%208-1.jpg?width=2500&amp;height=1170&amp;name=Fig%208-1.jpg 2500w, https://www.uptycs.com/hs-fs/hubfs/Fig%208-1.jpg?width=3000&amp;height=1404&amp;name=Fig%208-1.jpg 3000w" sizes="(max-width: 1000px) 100vw, 1000px"><span style="font-size: 14px;"><em>Figure 8 – Resource section of stage 1</em></span></p>
<p>&nbsp;</p>
<p>The stage 1 payload is equipped with code to unzip this archive successfully, depositing all the files into the Public Pictures folder.</p>
<p>&nbsp;</p>
<p>The following files are placed in the Public Pictures folder:<br><br>1. Calc.exe - Legitimate windows file<br>2. Secure32.dll - Malicious DLL<br>3. Winsecu32.dll - Legitimate windows file</p>
<h3><span style="color: #c092fb;">Stage 2: Calc.exe</span></h3>
<p>Next, "Calc.exe" was run using the command "c:\Users\Public\Pictures\Calc.exe /quit."<br><br>"/quit" serves as an argument or parameter for the Calculator executable, instructing it to open and promptly close upon launch.<br><br>When you run calc.exe, it loads a malicious DLL named "Secure32.dll." again via <span style="color: #c092fb;"><a href="https://attack.mitre.org/techniques/T1574/002/" rel="noopener" target="_blank" style="color: #c092fb;">Dll-side loading technique</a></span>. This DLL contains an encrypted resource section.</p>
<p>&nbsp;</p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%2011-1.jpg?width=1076&amp;height=630&amp;name=Fig%2011-1.jpg" width="1076" height="630" loading="lazy" alt="Figure 11- &nbsp;Resource section of Secure32.dll containing encrypted data" style="height: auto; max-width: 100%; width: 1076px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%2011-1.jpg?width=538&amp;height=315&amp;name=Fig%2011-1.jpg 538w, https://www.uptycs.com/hs-fs/hubfs/Fig%2011-1.jpg?width=1076&amp;height=630&amp;name=Fig%2011-1.jpg 1076w, https://www.uptycs.com/hs-fs/hubfs/Fig%2011-1.jpg?width=1614&amp;height=945&amp;name=Fig%2011-1.jpg 1614w, https://www.uptycs.com/hs-fs/hubfs/Fig%2011-1.jpg?width=2152&amp;height=1260&amp;name=Fig%2011-1.jpg 2152w, https://www.uptycs.com/hs-fs/hubfs/Fig%2011-1.jpg?width=2690&amp;height=1575&amp;name=Fig%2011-1.jpg 2690w, https://www.uptycs.com/hs-fs/hubfs/Fig%2011-1.jpg?width=3228&amp;height=1890&amp;name=Fig%2011-1.jpg 3228w" sizes="(max-width: 1076px) 100vw, 1076px"><span style="font-size: 14px;"><em>Figure 9 - &nbsp;Resource section of Secure32.dll containing encrypted data</em></span></p>
<p>&nbsp;</p>
<p>This resource data is accessed via a sequence of APIs like previous method mentioned as in the stage 1 (Figure 4)<br><br>To decrypt the data, Key size is 2F hex bytes and <span style="font-weight: bold;">Systemfunction32</span> API is used to decrypt the encrypted data same as in the first case which again gives a PE file.&nbsp;</p>
<p>&nbsp;</p>
<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%2012-1.jpg?width=1140&amp;height=426&amp;name=Fig%2012-1.jpg" width="1140" height="426" loading="lazy" alt="Fig 12-1" style="height: auto; max-width: 100%; width: 1140px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%2012-1.jpg?width=570&amp;height=213&amp;name=Fig%2012-1.jpg 570w, https://www.uptycs.com/hs-fs/hubfs/Fig%2012-1.jpg?width=1140&amp;height=426&amp;name=Fig%2012-1.jpg 1140w, https://www.uptycs.com/hs-fs/hubfs/Fig%2012-1.jpg?width=1710&amp;height=639&amp;name=Fig%2012-1.jpg 1710w, https://www.uptycs.com/hs-fs/hubfs/Fig%2012-1.jpg?width=2280&amp;height=852&amp;name=Fig%2012-1.jpg 2280w, https://www.uptycs.com/hs-fs/hubfs/Fig%2012-1.jpg?width=2850&amp;height=1065&amp;name=Fig%2012-1.jpg 2850w, https://www.uptycs.com/hs-fs/hubfs/Fig%2012-1.jpg?width=3420&amp;height=1278&amp;name=Fig%2012-1.jpg 3420w" sizes="(max-width: 1140px) 100vw, 1140px"><em>Figure 10 - Decrypted data</em></p>
<p>This PE File is now injected into memory space Regasm.exe via <span style="color: #c092fb;"><a href="https://attack.mitre.org/techniques/T1055/012/" rel="noopener" target="_blank" style="color: #c092fb;">process hollowing</a></span>. The API sequence followed is the same as in stage 1. But here, while calling VirtualAllocEx, the default memory address of regasm.exe (0x400000) is explicitly passed to hollow the regasm.exe and replace it with the malicious PE file.</p>

<p style="text-align: center;"><img src="https://www.uptycs.com/hs-fs/hubfs/Fig%2013-1.jpg?width=1258&amp;height=638&amp;name=Fig%2013-1.jpg" width="1258" height="638" loading="lazy" alt="Figure 13- Call to VirtualAllocEx where RDX = 0X400000 is passed" style="height: auto; max-width: 100%; width: 1258px; margin-left: auto; margin-right: auto; display: block;" srcset="https://www.uptycs.com/hs-fs/hubfs/Fig%2013-1.jpg?width=629&amp;height=319&amp;name=Fig%2013-1.jpg 629w, https://www.uptycs.com/hs-fs/hubfs/Fig%2013-1.jpg?width=1258&amp;height=638&amp;name=Fig%2013-1.jpg 1258w, https://www.uptycs.com/hs-fs/hubfs/Fig%2013-1.jpg?width=1887&amp;height=957&amp;name=Fig%2013-1.jpg 1887w, https://www.uptycs.com/hs-fs/hubfs/Fig%2013-1.jpg?width=2516&amp;height=1276&amp;name=Fig%2013-1.jpg 2516w, https://www.uptycs.com/hs-fs/hubfs/Fig%2013-1.jpg?width=3145&amp;height=1595&amp;name=Fig%2013-1.jpg 3145w, https://www.uptycs.com/hs-fs/hubfs/Fig%2013-1.jpg?width=3774&amp;height=1914&amp;name=Fig%2013-1.jpg 3774w" sizes="(max-width: 1258px) 100vw, 1258px"><em>Figure 11 - Call to VirtualAllocEx where RDX = 0X400000 is passed</em></p>
<h3><span style="color: #c092fb;">Final payload</span></h3>
<p>By getting the dump of the above region(Figure 11),&nbsp; we can analyze that this PE file is an MSIL executable obfuscated by Smart assembly.&nbsp;&nbsp;</p>

<p>Looking at the copyright of the file as "Copyright © MaxXor 2020", it looks like it might be inspired by<a href="https://github.com/quasar/Quasar/blob/master/LICENSE"><span> <span style="color: #c092fb;">open-source Quasar RAT</span></span></a> by MaxXor.</p>
