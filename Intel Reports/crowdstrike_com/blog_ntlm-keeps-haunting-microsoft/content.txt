<div class="cmp-wp-headline" id="headline-en-us-04-19-2024">
    
        <h1>NTLM Keeps Haunting Microsoft</h1>
        
        <div class="publish_info">
            <p>August 26, 2021</p>
            <span>|</span>
            <a href="/en-us/blog/author.yaron-zinar/" title="Posts by Yaron Zinar" rel="author">
                Yaron Zinar</a>
            <span>|</span>
                
            <a href="/en-us/blog/category.from-the-front-lines/" title="From The Front Lines">From The Front Lines</a>
        </div>
        <div class="post_image">
            <img width="1060" height="698" src="https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-88-us" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt="" decoding="async" fetchpriority="high" srcset="/content/dam/crowdstrike/www/en-us/wp/2024/12/Blog_1060x698-88-us.jpeg 1060w" sizes="(max-width: 1060px) 100vw, 1060px"/>
        </div>
    

    
</div>

    <div style="display: none;">
        
    </div>


    
</div>
<div class="text text--blog-content aem-GridColumn aem-GridColumn--default--12">

  

<div class="cmp-text" id="text-09b83a5680" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><p><span style="	font-weight: 400;
">Two severe Windows NT LAN Manager </span><a href="/en-us/cybersecurity-101/identity-protection/windows-ntlm/"><span style="	font-weight: 400;
">(NTLM)</span></a><span style="	font-weight: 400;
"> vulnerabilities were recently disclosed: </span><a href="https://github.com/topotam/PetitPotam"><span style="	font-weight: 400;
">PetitPotam</span></a><span style="	font-weight: 400;
"> and </span><a href="https://posts.specterops.io/certified-pre-owned-d95910965cd2"><span style="	font-weight: 400;
">AD-CS relay</span></a><span style="	font-weight: 400;
"> (specifically ESC8). These vulnerabilities follow a pattern of NTLM issues in recent years. These vulnerabilities are bad on their own, but their combination can be devastating: If a network is not protected, the combination can allow an attacker with network access alone to achieve full domain admin privileges.</span> <span style="	font-weight: 400;
">This blog post skips some of the deep technical details of these two vulnerabilities (there are plenty of great resources online) and instead provides a wider and richer context for recent attacks and similar issues that lurk in Active Directory (AD) environments.</span></p>
<h2><span style="	font-weight: 400;
">Summary</span></h2>
<p><span style="	font-weight: 400;
">PetitPotam is a coerced authentication that acts as an enabler for NTLM relay. Its key strength compared to Print Spooler coerced authentication is that it works </span><i><span style="	font-weight: 400;
">unauthenticated</span></i><span style="	font-weight: 400;
"> (an attacker only needs network access). AD-CS relay (ESC8) is the more critical issue since the AD-CS default configuration does not protect from NTLM relay. Relaying privileged authentication to an AD-CS server results in full domain takeover. Microsoft partially addressed one of the issues related to PetitPotam on Aug. 10, 2021, as discussed in our August 2021 </span><a href="/en-us/blog/patch-tuesday-analysis-august-2021/"><span style="	font-weight: 400;
">Patch Tuesday blog</span></a><span style="	font-weight: 400;
">. We have not performed an independent analysis of the patch, but researchers have </span><a href="https://www.kb.cert.org/vuls/id/405600"><span style="	font-weight: 400;
">stated</span></a><span style="	font-weight: 400;
"> it does not solve all of the authentication issues abused by PetitPotam unless additional controls are also implemented. Microsoft has not fixed the AD-CS relay by default but has issued an </span><a href="https://msrc.microsoft.com/update-guide/vulnerability/ADV210003"><span style="	font-weight: 400;
">advisory</span></a><span style="	font-weight: 400;
"> to guide customers on how to enable protection.</span></p>
<div class="wp-caption" id="attachment_44175"><img class="size-full wp-image-44175" src="/content/dam/crowdstrike/www/en-us/wp/2021/08/Picture1-5.png" alt="" width="977" height="400"/>Figure 1. Combined PetitPotam and ESC8 attack flow</div>
<h2><span style="	font-weight: 400;
">Coerced Authentication</span></h2>
<p><span style="	font-weight: 400;
">PetitPotam is a coerced authentication vulnerability. Coerced authentication is a procedure where the attackers trigger a remote authentication to a compromised machine. When the attackers capture the network login on the compromised machine, they can use this authentication to perform:</span></p>
<ol><li style="	font-weight: 400;
"><b>Password Cracking</b><span style="	font-weight: 400;
">: When the attackers coerce a user account authentication and the attacked client supports NTLM, attackers can use the captured NTLM authentication to perform password cracking. When NTLMv1 is supported (less common), attackers can use the weak cryptography to extract a password hash more easily. The authentication coercion techniques described below trigger computer account authentication and thus are not susceptible to password-cracking attacks since computer accounts have strong randomly generated passwords that rotate on a monthly basis.</span></li><li style="	font-weight: 400;
"><b>Credentials Relay</b><span style="	font-weight: 400;
">: Credentials relay has two flavors:</span><ol><li style="	font-weight: 400;
"><b>NTLM Relay:</b><span style="	font-weight: 400;
"> This is the more common attack. There is plenty of material out there on NTLM Relay — for a deeper overview, start with the introduction to </span><a href="/en-us/blog/retrieve-session-key-authentication/"><span style="	font-weight: 400;
">server signing</span></a><span style="	font-weight: 400;
"> attacks and </span><a href="/en-us/blog/bypass-epa-ntlm-attacks-wia/"><span style="	font-weight: 400;
">EPA</span></a><span style="	font-weight: 400;
"> attacks. In my experience, Microsoft has treated NTLM relay attacks as actual vulnerabilities that required patching only if there was no safe configuration (for example, with </span><a href="/en-us/blog/cve-2021-1678-printer-spooler-relay-security-advisory/"><span style="	font-weight: 400;
">CVE-2021-1678</span></a><span style="	font-weight: 400;
">). However, if a safe configuration was available, Microsoft typically published an advisory while keeping the insecure defaults, possibly out of concern that changing the default configuration may have resulted in breaking various applications. This means that in many cases, defenders can be left with insecure defaults.</span></li><li style="	font-weight: 400;
"><b>Kerberos Relay</b>: In most cases, attackers cannot relay Kerberos authentications. However, if attackers are able to compromise an account with sufficient delegation privileges, they can perform Kerberos authentications on behalf of other users. A recent interesting vulnerability discovered in this space is the Bronze Bit attack</li></ol>
</li></ol>
<p><span style="	font-weight: 400;
">Two well known authentication coercion attacks are the “Printer Bug” and PetitPotam:</span></p>
<h3><span style="	font-weight: 400;
">The “Printer Bug”</span></h3>
<p><span style="	font-weight: 400;
">An important issue that exists in the print spooler service is the “Printer Bug,” which is an authentication coercion mechanism. The issue was reported by <a href="https://twitter.com/tifkin_">Lee Christensen</a> from SpectreOps in 2018, and initially Microsoft did not fix it completely, but has removed the coerced authentication in Windows Server 2019. Similar to the new PetitPotam attack, this attack enables the attackers to trigger authentication by a remote host’s computer account. If the remote machine is privileged (e.g., a domain controller), that authentication can be relayed and the computer account privileges are used.</span></p>
<h3><span style="	font-weight: 400;
">PetitPotam (MS-EFSRPC)</span></h3>
<p><span style="	font-weight: 400;
">PetitPotam is an authentication coercion mechanism that takes advantage of MS-EFSRPC (interface c681d488-d850-11d0-8c52-00c04fd90f7e) via </span><a href="https://docs.microsoft.com/en-us/openspecs/windows_protocols/ms-efsr/ccc4fb75-1c86-41d7-bbc4-b278ec13bfb8"><span style="	font-weight: 400;
">EfsRpcOpenFileRaw</span></a><span style="	font-weight: 400;
"> function. PetitPotam has two notable advantages:</span></p>
<ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The EfsRpcOpenFileRaw function works unauthenticated. This means any attacker with network access alone could potentially exploit this issue.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">PetitPotam<p> </p>
is a more recent coerced and remains viable on all Windows versions to date.</span></li></ol>
<h2><span style="	font-weight: 400;
">Missing NTLM Relay Protections</span></h2>
<p><span style="	font-weight: 400;
">The second vulnerability is a classical NTLM relay attack. NTLM relay attacks are a very old attack technique. In general, Microsoft offers two main mitigations to protect from NTLM relay:</span></p>
<p> </p>
<p> </p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Server Signing (SMB Signing / LDAP Signing)<p> </p>
</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Channel Bindings (EPA &#61; Extended Protection for Authentication).<p> </p>
</span></li></ul>
<p><span style="	font-weight: 400;
">Some of these protections are enabled by default (e.g., SMB Signing on Domain Controllers), and some require special configuration by IT administrators (e.g., EPA for LDAP over TLS).</span></p>
<h3><span style="	font-weight: 400;
">AD-CS Relay (ESC8)</span></h3>
<p><a href="https://twitter.com/harmj0y"><span style="	font-weight: 400;
">Will Schroeder</span></a><span style="	font-weight: 400;
"> and </span><a href="https://twitter.com/tifkin_"><span style="	font-weight: 400;
">Lee Christensen</span></a><span style="	font-weight: 400;
"> have published </span><a href="https://posts.specterops.io/certified-pre-owned-d95910965cd2"><span style="	font-weight: 400;
">Certified Pre-Owned</span></a><span style="	font-weight: 400;
">, a white paper detailing many attacks on AD certificate-based authentication. Along with the white paper, they have released a defensive audit tool allowing IT administrators and security professionals to find existing issues in their network. I want to focus on one specific issue: ESC8. ESC8 is the discovery that the AD certificate server (AD-CS) by default does not enforce EPA on incoming connections. This means that if attackers are able to capture a privileged enough authentication request, it can be relayed to the AD-CS server and allow the attackers to create a certificate for the relayed account and then authenticate as that user account. Since the user is privileged (e.g., the DC account), the attacker now has full domain privileges.</span></p>
<h3><span style="	font-weight: 400;
">LDAPS Relay (CVE-2017-8563)</span></h3>
<p><span style="	font-weight: 400;
">Another important attack surface for NTLM relay attacks is LDAP. The LDAP protocol allows not only querying the directory data, but also changing directory data such as adding users, adding users to security groups, creating new computers and more. LDAP has two main supported access method:</span></p>
<ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><b>LDAP over port 389:</b> This is the regular LDAP port. LDAP is protected there by server signing, which is not turned on by default.<p> </p>
</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
"><b>LDAPS - LDAP over TLS (port 636):</b> This is the same LDAP protocol protected by TLS. Prior to 2017, LDAPS was not protected from NTLM relay at all (CVE-2017-8563). To protect LDAPS, Microsoft introduced channel bindings, which are essentially EPA protection for LDAPS. Unfortunately, Microsoft has not enabled this protection by default, and thus many networks are still unprotected today.</span></li></ol>
<p><span style="	font-weight: 400;
">As you can see, LDAP protections are not enabled by default. </span><b>We see that for more than 90% of CrowdStrike Identity Protection customers, these configurations are not in a secure manner. </b><span style="	font-weight: 400;
">This is partially mitigated since when the client sets the signing flag in the NTLM message, LDAP/S relay is mitigated (more details in the </span>wonderful blog<span style="	font-weight: 400;
"> by </span><a href="https://twitter.com/HackAndDo"><span style="	font-weight: 400;
">Pixis</span></a><span style="	font-weight: 400;
">). This makes LDAP relay a slightly less common relay target. However, if an NTLM client supports SMBv1 and/or the NTLM server is vulnerable to Drop the MIC (see more in our </span><a href="/en-us/blog/active-directory-ntlm-attack-security-advisory/"><span style="	font-weight: 400;
">2019 blog</span></a><span style="	font-weight: 400;
">), LDAP/S relay can still occur.</span></p>
<h2><span style="	font-weight: 400;
">Attack Summary</span></h2>
<p><span style="	font-weight: 400;
">The following table summarizes attack feasibility for the described methods.</span></p>
<div class="wp-caption" id="attachment_44675"><img class="wp-image-44675 size-large" src="/content/dam/crowdstrike/www/en-us/wp/2021/08/Screen-Shot-2021-08-26-at-9.22.51-AM-1024x420.png" alt="" width="1024" height="420"/>Figure 2. Availability of recent NTLM relay attacks</div>
<h2><span style="	font-weight: 400;
">How to Protect Your Network</span></h2>
<p><span style="	font-weight: 400;
">In order to protect your network, we suggest following these steps:</span></p>
<h3><span style="	font-weight: 400;
">Set Secure Configuration</span></h3>
<p><span style="	font-weight: 400;
">The following secure configurations are required:</span></p>
<ol><li style="	font-weight: 400;
"><b>Enforce </b><a href="https://techcommunity.microsoft.com/t5/storage-at-microsoft/configure-smb-signing-with-confidence/ba-p/2418102?wt.mc_id&#61;modinfra-0000-abartolo"><b>SMB Server Signing</b></a><span style="	font-weight: 400;
">. SMB signing is required by default on domain controllers, which is good. However, not all other workstations and servers in the network are protected by default. An important note is that by default relaying SMBv2-&gt;SMBv2 (a very important scenario) is possible.<p> </p>
<p> </p>
</span></li><li style="	font-weight: 400;
"><b>Enforce EPA and </b><a href="https://support.microsoft.com/en-us/topic/2020-ldap-channel-binding-and-ldap-signing-requirements-for-windows-ef185fb8-00f7-167d-744c-f299a66fc00a"><b>Channel Binding</b></a><b> on all critical servers</b><span style="	font-weight: 400;
">. This is extremely important since in practically all cases, EPA protection </span><b>is not enabled</b><span style="	font-weight: 400;
"> by default. Typically, EPA protection can be </span><i><span style="	font-weight: 400;
">enabled</span></i><span style="	font-weight: 400;
"> or </span><i><span style="	font-weight: 400;
">required</span></i><span style="	font-weight: 400;
">. While setting EPA&#61;</span><i><span style="	font-weight: 400;
">required</span></i><span style="	font-weight: 400;
"> offers better protection, setting EPA&#61;</span><i><span style="	font-weight: 400;
">enabled</span></i><span style="	font-weight: 400;
"> will block most relay attacks.</span></li><li style="	font-weight: 400;
"><b>Disable NTLM</b><span style="	font-weight: 400;
">. Even though steps 1 and 2 can almost completely mitigate NTLM relay (aside from zero days), NTLM has many issues in addition to NTLM relay, and it is advisable to completely stop using NTLM. Turning off NTLM, however, is a complex IT project that requires planning and time. The first and most critical step is understanding which systems in your network are still using NTLM and why. This step has proven to be non-trivial for many organizations. Once you have your network mapped, you can start the migration process.<p> </p>
</span></li></ol>
<h3><span style="	font-weight: 400;
">Patching</span></h3>
<p><span style="	font-weight: 400;
">In some cases, there are no secure defaults, so make sure your environment is patched against the following relay vulnerabilities:</span></p>
<ol><li style="	font-weight: 400;
"><b>Drop the MIC</b><span style="	font-weight: 400;
">: </span><a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2019-1040"><span style="	font-weight: 400;
">CVE-2019-1040</span></a><span style="	font-weight: 400;
"> can enable attackers to perform successful LDAP relay attacks. While this is an old vulnerability, make sure you are fully patched against the vulnerability.<p> </p>
</span></li><li style="	font-weight: 400;
"><b>DCE/RPC Relay</b><span style="	font-weight: 400;
">: Apply patched for </span><a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2021-1678"><span style="	font-weight: 400;
">CVE-2021-1678</span></a><span style="	font-weight: 400;
"> and </span><a href="https://msrc.microsoft.com/update-guide/en-us/vulnerability/CVE-2020-1113"><span style="	font-weight: 400;
">CVE-2020-1113</span></a><span style="	font-weight: 400;
">.</span></li><li style="	font-weight: 400;
"><b>PetitPotam</b><span style="	font-weight: 400;
">: As we have suggested, the patch probably does not solve the coerced authentication issue entirely. However, it is recommended to install a patch for </span><a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2021-36942"><span style="	font-weight: 400;
">CVE-2021-36942</span></a><span style="	font-weight: 400;
">.</span></li></ol>
<h3><span style="	font-weight: 400;
">Disable Unnecessary Services</span></h3>
<ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Disable the Printer Spooler service on all critical servers.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">MS-EFSRPC: Either disable the service entirely or follow the </span><a href="https://support.microsoft.com/en-us/topic/kb5005413-mitigating-ntlm-relay-attacks-on-active-directory-certificate-services-ad-cs-3612b773-4043-4aa9-b23d-b87910cd3429"><span style="	font-weight: 400;
">Microsoft advisory</span></a><span style="	font-weight: 400;
"> and enforce EPA and AD-CS authentications.</span></li></ol>
<h3><span style="	font-weight: 400;
">Audit Your Environment</span></h3>
<p><span style="	font-weight: 400;
">It is obvious that keeping track of all configurations (NTLM, AD-CS, enabled services, EPA, signing requirements, patch status) is almost impossible manually. In order to make sure you are fully protected, an automated audit solution is required.</span></p>
<p> </p>
<p><span style="	font-weight: 400;
">Falcon Spotlight™ customers can quickly see if their environments are vulnerable using the PrintNightmare dashboard. For those who wish to trial Spotlight for free, please see the </span><a href="https://store.crowdstrike.com/en-US/apps/352/falcon-spotlight#overview"><span style="	font-weight: 400;
">CrowdStrike Store</span></a><span style="	font-weight: 400;
">.</span></p>
