				<div class="content">
					
					<p>Ever notice how news about hidden malware almost always focuses on remediation AFTER the fact? So did we. Even now, there’s yet another news story about a rash of attacks by a group called TeamTNT. They used a crypto-mining worm to steal AWS credentials from <a href="https://www.aquasec.com/cloud-native-academy/docker-container/using-docker-hub-responsibly-4-security-best-practices/">Docker Hub</a>. Well, if hijacking cloud resources is so popular, it’s time to make finding threats BEFORE the attack just as fashionable. Our investigation determined that dynamic analysis could have saved some overworked security teams a lot of time and aggravation — if these images were detected and removed from Docker Hub before being deployed — in much the same way it helps security teams with their private registries.</p>
<p><span id="more-15518"></span></p>
<p>We at Team Nautilus detected and analyzed the Docker Hub account <span style="font-family: 'courier new', courier;">hildeteamtnt</span>, which was used by TeamTNT to store their malicious images. We ran these images in a secure container sandbox and uncovered a total of eight malicious container images, with at least two of these images being used to perform attacks in the wild. Here’s what was found:</p>
<p><a href="https://www.aquasec.com/wp-content/uploads/2024/01/hildeteamtnt-image-1.jpg" target="_blank" rel="noopener" class="modal-inline-image"><img decoding="async" style="width: 850px;" src="https://www.aquasec.com/wp-content/uploads/2024/01/hildeteamtnt-image-1.jpg" alt="TeamTNT dynamic container analysis" width="850" /></a></p>
<h3>Images in the hildeteamtnt account</h3>
<p>The images were well-designed and meticulously built to fully exploit the targeted host. The adversaries hid various malware executables (<span style="font-family: 'courier new', courier;">Tsunami</span> and <span style="font-family: 'courier new', courier;">Mirai</span>), backdoors, Docker escape tools, and Potentially Unwanted Applications (PUAs) inside these images. In each image, they used scripts to execute their nefarious attacks, some of which were simple and straightforward, while others were complex and sophisticated:</p>
<ol>
<li><strong>Simple and straightforward attacks:</strong> <em>‘metal3d’</em>, ‘<em>first</em>’ and <em>‘dockerfirst’</em> are images were designed to launch a simple resource hijacking attack (Cryptominer).</li>
<li><strong>Sophisticated attacks:</strong> <em>‘Minerescape’</em> and<em> ‘Debian-service’</em> contain a binary named <span style="font-family: 'courier new', courier;">docker-escape</span>. The binary is based on a tool in GitHub <a href="https://github.com/PercussiveElbow/docker-escape-tool" target="_blank" rel="noopener">docker-escape-tool</a>. Designed to identify if it’s running in a Docker container, it then attempts Docker escape techniques.</li>
</ol>
<p>For instance, this binary contains some of the following strings:</p>
<ul>
<li><span style="font-family: 'courier new', courier; font-size: 18px;">.docker_escape network</span> &#8211; Attempts to escape via Docker TCP socket (if found on any interfaces) or Port scans (if network namespace shared with the host).</li>
<li><span style="font-family: 'courier new', courier; font-size: 18px;">.docker_escape auto</span> &#8211; Attempts automatic escape. If successful, this starts a privileged container with the host drive mounted at /hostOS.</li>
<li><span style="font-family: 'courier new', courier; font-size: 18px;">.docker_escape Unix</span> &#8211; Attempts an escape using a mounted Docker UNIX socket located at /var/run/docker.sock.</li>
</ul>
<p><span style="background-color: transparent;">Also, ‘minerescape’ contained a shell script executing a Python file &#8211; <span style="font-family: 'courier new', courier; font-size: 18px;">minedaemon.py</span>. The file <span style="font-family: 'courier new', courier; font-size: 18px;">nightminer.py</span> allows the execution of the crypto mining process.</span></p>
<p><a href="https://www.aquasec.com/wp-content/uploads/2024/01/image-2-5.jpg" target="_blank" rel="noopener" class="modal-inline-image"><span style="background-color: transparent;"><img decoding="async" style="width: 850px;" src="https://www.aquasec.com/wp-content/uploads/2024/01/image-2-5.jpg" width="850" /></span></a></p>
<p style="padding-left: 40px;">3. <strong>Highly sophisticated attacks:</strong> Since the first tool <em>‘xmrigminer’</em> was uploaded on March 28, 2020, adversaries worked to perfect it by revising their scripts and adding more and more techniques to evade detection, hide communication channels with command and control servers, and gain persistence over time. The second image <em>‘avscan’</em> is a more sophisticated version (described below) and the last and recent version <em>‘docrunker’</em> is even worse.</p>
<h3><span style="background-color: transparent;">Deep Analysis of AVscan</span></h3>
<p>The adversaries used a known technique aimed at taking over the host by mounting the host <span style="font-family: 'courier new', courier; font-size: 18px;">/</span> dir into <span style="font-family: 'courier new', courier; font-size: 18px;">/mnt</span> in the container and then chrooting into <span style="font-family: 'courier new', courier; font-size: 18px;">/mnt</span>.</p>
<p><span style="font-family: 'courier new', courier; font-size: 18px;"><strong>AVscan.sh</strong></span><br />
The entry point for the image is the script <span style="font-family: 'courier new', courier; font-size: 18px;">AVscan.sh</span>:</p>
<p><a href="https://www.aquasec.com/wp-content/uploads/2024/01/image-3-6.jpg" target="_blank" rel="noopener" class="modal-inline-image"><img decoding="async" style="width: 542px; display: block; margin: 0px auto;" src="https://www.aquasec.com/wp-content/uploads/2024/01/image-3-6.jpg" width="542" /></a></p>
<p>Following that command, the image is designed to run the scripts <span style="font-family: 'courier new', courier; font-size: 18px;">Carray.sh</span>,<span style="font-family: 'courier new', courier; font-size: 18px;"> cron.sh</span>, and execute two malicious binaries <span style="font-family: 'courier new', courier; font-size: 18px;">SystemHealt</span> and <span style="font-family: 'courier new', courier; font-size: 18px;">AVscan</span>. The command <span style="font-family: 'courier new', courier; font-size: 18px;">vm.nr_hugepages</span> is a system property that is designed to <a href="https://medium.com/@tomas_savenas/30-increase-in-cpu-mining-hash-rate-by-enabling-huge-pages-8af5eedb7d62" target="_blank" rel="noopener">increase the efficacy of the crypto-mining process</a>.</p>
<p><a href="https://www.aquasec.com/wp-content/uploads/2024/01/AVScan-attack-image-4.jpg" target="_blank" rel="noopener" class="modal-inline-image"><img decoding="async" style="width: 850px;" src="https://www.aquasec.com/wp-content/uploads/2024/01/AVScan-attack-image-4.jpg" alt="Dynamic Container Analysis" width="850" /></a></p>
<p>The scripts <span style="font-family: 'courier new', courier; font-size: 18px;">Carray.sh</span> and <span style="font-family: 'courier new', courier; font-size: 18px;">cron.sh</span> are already stored inside the image layers.</p>
<p><span style="font-family: 'courier new', courier; font-size: 18px;"><strong>Carray.sh</strong></span><br />
The shell script Carray.sh is designed to kill any competing mining processes running on the host. The script also deletes itself after it is executed.</p>
<p><span style="font-size: 18px; font-family: 'courier new', courier;"><strong>Cron.sh</strong></span><br />
The shell script <span style="font-family: 'courier new', courier; font-size: 18px;">Cron.sh</span> sets a cron job to download  <span style="font-family: 'courier new', courier; font-size: 18px;">minion.sh</span> from:<br />
<span style="font-family: 'courier new', courier; font-size: 18px;">hxxp [:] //yd6ugsklvmydbvmntk54k4dmgkuwzar3u3zzv5vcxwdywjwbwzjmdryd [.] tor2web [.] su/&#8230;/<a href="http://minion.sh/" target="_blank" rel="noopener">minion.sh</a></span></p>
<p><span style="font-family: 'courier new', courier; font-size: 18px;"><strong>Minion.sh</strong></span><br />
Some of the snippets in this script are encoded with Based64. The adversaries decode this code into new files using innocuous or misleading names such as: <span style="font-family: 'courier new', courier; font-size: 18px;">pu,ntpd.pid,.logs.c,</span> etc. They also hide some of the files they are writing to disk. Below are three examples of what this script is doing:</p>
<ul>
<li><span style="background-color: transparent;">One file (<span style="font-family: 'courier new', courier; font-size: 18px;">/usr/bin/hid</span>) is trying to mount given PID from the directory <span style="font-family: 'courier new', courier; font-size: 18px;">/proc/&lt;PID&gt;</span> into /usr/foo. This process is done stealthily without leaving traces in <span style="font-family: 'courier new', courier; font-size: 18px;">/etc/mtab</span>.</span></li>
<li><span style="background-color: transparent;">Another file (<span style="font-family: 'courier new', courier; font-size: 18px;">/usr/bin/pu</span>) is a Python script designed to conduct lateral movement in the network by exploiting the SSH port. </span></li>
<li><span style="background-color: transparent;">Last, but not least, another file (<span style="font-family: 'courier new', courier; font-size: 18px;">/tmp/.logs.c</span>) is used to clean all login and logout records on the system (WTMP, UTMP, and Lastlog for Linux). This exemplifies the level of effort that these adversaries are taking to clean their records and maintain high standards of OPSEC.</span></li>
</ul>
<p>This script also deactivates any programs competing for resources on the CPU, such as <em>xmrigMiner</em> and <em>Watchdog</em>. Once the script is done, it deletes itself and history.</p>
<p>This script downloaded and executed three other scripts (<span style="font-family: 'courier new', courier; font-size: 18px;">user.sh</span>,<span style="font-family: 'courier new', courier; font-size: 18px;"> sbs.sh</span>, <span style="font-family: 'courier new', courier; font-size: 18px;">sxc.sh</span>):</p>
<p><strong style="background-color: transparent;">The script <span style="font-family: 'courier new', courier; font-size: 18px;">user.sh</span>:</strong><span style="background-color: transparent;"> Responsible for SSH communication with the infected host:</span></p>
<ul>
<li>Verifying that SSHD exists, and if not download, installs an Openssh-server.</li>
<li>Trying to create three users (hilde, reboot, ubuntu).</li>
<li>Attempting to grant the users root privileges.</li>
<li>Appending the adversary’s RSA-keys for the above-mentioned users (hilde, reboot, ubuntu) into:<br />
<span style="font-family: 'courier new', courier; font-size: 18px;">/home/$usersname/.ssh/authorized_keys<br />
</span><span style="font-family: 'courier new', courier; font-size: 18px;">/home/$usersname/.ssh/authorized_keys2</span></li>
<li>Setting root to be able to SSH login with no password in: <span style="font-family: 'courier new', courier; font-size: 18px;">/etc/ssh/sshd_config.</span> and restart <span style="font-family: 'courier new', courier; font-size: 18px;">SSHD</span>.</li>
<li>Using a web service (iplogger[.]org)  to transmit collected data to the attacker during the discovery process, for instance, the number of cores in the CPU, its speed, system details (using <span style="font-family: 'courier new', courier; font-size: 18px;">uname −a</span>), and targeted host IP address.</li>
<li>Logging the activity and encoding it into files (using Base64).</li>
</ul>
<p><strong>The script <span style="font-family: 'courier new', courier; font-size: 18px;">sbs.sh</span>: </strong></p>
<ul>
<li style="vertical-align: baseline;">Downloading <span style="font-family: 'courier new', courier; font-size: 18px;">00.jpg</span> (as <span style="font-family: 'courier new', courier; font-size: 18px;">/usr/bin/dns_ipv4.tar.gz</span>) which is the file <span style="font-family: 'courier new', courier; font-size: 18px;">/usr/bin/bioset</span>.</li>
<li style="vertical-align: baseline;"><u>Executing Bioset</u>:
<ul>
<li style="vertical-align: baseline;">Renaming the process to be systemd.</li>
<li style="vertical-align: baseline;">Listening to any connection on port 1982. Each new connection opens a new thread.</li>
<li style="vertical-align: baseline;">Creating a child process that listens to the socket and communicates with the father using a method called ‘Named PIPE’ (also known as FIFO).</li>
<li style="vertical-align: baseline;">The father is responsible for deciphering messages and writing it back to the child on the PIPE.</li>
<li style="vertical-align: baseline;">The child receives commands which are being executed by <span style="font-family: 'courier new', courier; font-size: 18px;">/bin/sh</span>.</li>
<li style="vertical-align: baseline;">This leads us to conclude that the Bioset is serving as a bind shell that probably allows the attacker to connect to the host after he deployed the container.</li>
</ul>
</li>
<li style="vertical-align: baseline;"><u>Discovery and Command and control</u>:
<ul>
<li style="vertical-align: baseline;">Using a web service (iplogger[.]org) to transmit collected data to the attacker during the discovery process, for instance, the number of cores in the CPU, its speed, system details (using uname -a), and targeted host IP address.</li>
<li style="vertical-align: baseline;">Logging the activity and encoding it into files (using Base64).</li>
</ul>
</li>
<li style="vertical-align: baseline;"><u>Defense Evasion</u>:
<ul>
<li style="vertical-align: baseline;">Deleting command history.</li>
<li style="vertical-align: baseline;">Deleting the shell file.</li>
</ul>
</li>
</ul>
<p><strong><u>The script &#8211; &#8216;sxc.sh’</u>: </strong></p>
<ul>
<li style="list-style-type: none;">
<ul>
<li style="vertical-align: baseline;">Downloading xmrig from a remote source.</li>
<li style="vertical-align: baseline;">Writing the configuration file to disk (<span style="font-family: 'courier new', courier; font-size: 18px;">/usr/bin/ntpd.pid</span>).</li>
<li style="vertical-align: baseline;">Verifying that the Cryptominer (the file ntpd) is running.</li>
<li style="vertical-align: baseline;">Terminating competition on resources:
<ul>
<li style="vertical-align: baseline;">Stopping all Moneroocean processes.</li>
</ul>
</li>
<li style="vertical-align: baseline;">Optimizing the miner’s operation:
<ul>
<li style="vertical-align: baseline;">Stopping monitoring services that limit the mining resources, such as <span style="font-family: 'courier new', courier; font-size: 18px;">system-getty.slices.service</span>.</li>
<li style="vertical-align: baseline;">Stopping system health services, such as <span style="font-family: 'courier new', courier; font-size: 18px;">cp.syshealt</span>.</li>
</ul>
</li>
<li style="vertical-align: baseline;"><u>Discovery and Command and control</u>:
<ul>
<li style="vertical-align: baseline;">Using a web service (iplogger[.]org)   to transmit collected data to the attacker relevant to the mining processes, for instance, Monero hash rate, system details (using uname -a), and targeted host IP address.</li>
<li style="vertical-align: baseline;">Logging the activity and encoding it into files (using Base64).</li>
</ul>
</li>
<li style="vertical-align: baseline;">Defense Evasion Techniques:
<ul>
<li style="vertical-align: baseline;">Removing system logs (<span style="font-family: 'courier new', courier; font-size: 18px;">/var/log/syslog</span>).</li>
<li style="vertical-align: baseline;">Deleting command history.</li>
<li style="vertical-align: baseline;">Deleting the shell file.</li>
<li style="vertical-align: baseline;">Disabling network configuration, security, and monitoring tools, such as netfilter firewall, iptables, kernel.nmi_watchdog (responsible to check if the kernel is hung).</li>
<li style="vertical-align: baseline;">Disabling security scanners, such as Aliyun.</li>
</ul>
</li>
</ul>
</li>
</ul>

<p style="font-size: 18px;"><span style="font-family: 'courier new', courier;">SystemHealt &#8211; cb782b40757d1aba7a3ab7db57b50847 </span><span style="font-size: 16px;">(MD5)</span></p>

<h3>The modus operandi</h3>
<p>The more sophisticated attacks contained various changes in the code, implying that TeamTNT invested time and effort to improve their attacks. They added defense evasion techniques and attempted to conceal the communication with their C2 servers to try and ensure success while making the attacks last longer.</p>
<p><strong>Examples of evasion techniques:</strong></p>
<ul>
<li>Using dynamic code. The communication channels (IPs/domains) are defined as variables and use external configuration files to hide their C2 and mining infrastructure.</li>
<li>Leveraging web services and TOR servers to increase the persistence of their C2 infrastructure.</li>
<li>Putting more emphasis on OPSEC. Many stages, in recent attacks, revealed adversaries have been trying to cover their tracks by deleting their malicious tools, logs, erasing user activity on the host, etc.</li>
<li>Using a private domain teamtnt[.]red with subdomains vps and pool.</li>
<li>Disabling security tools and network security tools.</li>
<li>Encoding many snippets with base64 (the same snippet may be encoded multiple times).</li>
</ul>
<h3>To sum it up</h3>
<p>Over four months, TeamTNT uploaded various images, with some being used to perform attacks in the wild. This is still an active campaign targeting cloud native environments in general and misconfigured Docker API ports in particular. The newer images in these accounts were designed to use advanced malicious methods, such as defense evasion, lateral movement, and discovery techniques.</p>
<p>We believe changes in these newer images somewhat reflect the learning curve (developing capabilities) of TeamTNT. The simpler and more straightforward images could be detected by static <a href="https://www.aquasec.com/cloud-native-academy/cloud-attacks/malware-analysis/" rel="noopener">malware analysis</a>, but the newer ones are too smart for that. In fact, the image <span style="font-family: 'courier new', courier; font-size: 18px;">xmrigminer</span> can only be identified as malicious using behavioral profiling and monitoring in runtime, or better yet, in the pipeline using Aqua’s <a href="https://www.aquasec.com/products/container-analysis/" rel=" noopener">Dynamic Threat Analysis</a>.</p>
<p><a href="https://info.aquasec.com/container_analysis" rel=" noopener"><img decoding="async" style="width: 500px; display: block; margin: 0px auto;" src="https://www.aquasec.com/wp-content/uploads/2024/01/DTA-Blog-CTA.jpg" alt="Dynamic Container Analysis" width="500" /></a></p>
