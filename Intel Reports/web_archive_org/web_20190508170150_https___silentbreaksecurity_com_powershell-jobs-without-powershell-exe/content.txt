	  <img align=right src="http://archiveteam.org/images/e/e6/Archiveteam.jpg" width=200> Formed in 2009, the Archive Team (not to be confused with the archive.org Archive-It Team) is a rogue archivist collective dedicated to saving copies of rapidly dying or deleted websites for the sake of history and digital heritage. The group is 100% composed of volunteers and interested parties, and has expanded into a large amount of related projects for saving online and digital history.
<p>
History is littered with hundreds of conflicts over the future of a community, group, location or business that were "resolved" when one of the parties stepped ahead and destroyed what was there. With the original point of contention destroyed, the debates would fall to the wayside. Archive Team believes that by duplicated condemned data, the conversation and debate can continue, as well as the richness and insight gained by keeping the materials. Our projects have ranged in size from a single volunteer downloading the data to a small-but-critical site, to over 100 volunteers stepping forward to acquire terabytes of user-created data to save for future generations.
<p>
The main site for Archive Team is at <a href="http://www.archiveteam.org">archiveteam.org</a> and contains up to the date information on various projects, manifestos, plans and walkthroughs.
<p>
This collection contains the output of many Archive Team projects, both ongoing and completed. Thanks to the generous providing of disk space by the Internet Archive, multi-terabyte datasets can be made available, as well as in use by the <a href="http://archive.org/web/web.php">Wayback Machine</a>, providing a path back to lost websites and work. 
<p>
Our collection has grown to the point of having sub-collections for the type of data we acquire. If you are seeking to browse the contents of these collections, the Wayback Machine is the best first stop. Otherwise, you are free to dig into the stacks to see what you may find.
<p>
<b>The Archive Team Panic Downloads</b> are full pulldowns of currently extant websites, meant to serve as emergency backups for needed sites that are in danger of closing, or which will be missed dearly if suddenly lost due to hard drive crashes or server failures.

You give it a URL to start at, and it grabs all content under that URL, records it in a WARC, and then uploads that WARC to ArchiveTeam servers for eventual injection into the Internet Archive (or other archive sites).
<p>
To use ArchiveBot, drop by #archivebot on EFNet. To interact with ArchiveBot, you issue commands by typing it into the channel. Note you will need channel operator permissions in order to issue archiving jobs. The dashboard shows the sites being downloaded currently.

							</span> 							<h1 class="entry-title">The Evolution of Offensive PowerShell Invocation</h1>
							
							<div id="single-below-header">
								<span class="meta-author vcard author"><span class="fn">By <a href="https://web.archive.org/web/20190508170150/https://silentbreaksecurity.com/author/lee/" title="Posts by Lee Christensen" rel="author">Lee Christensen</a></span></span><!--

	<div class="inner-wrap animated">

		<div class="post-content">
			
						
			

			
				<div class="content-inner">
					
										
										
				   
					<p>By now, PowerShell should be in every offensive security person&#8217;s arsenal. There are a <a href="https://web.archive.org/web/20190508170150/https://github.com/PowerShellEmpire/Empire">plethora</a> <a href="https://web.archive.org/web/20190508170150/https://github.com/samratashok/nishang">of </a><a href="https://web.archive.org/web/20190508170150/https://github.com/PowerShellMafia/PowerSploit">PowerShell </a>projects now that penetration testers and red teams can use when testing Windows networks. For privilege escalation, we have <a href="https://web.archive.org/web/20190508170150/https://github.com/PowerShellMafia/PowerSploit/blob/master/Privesc/PowerUp.ps1">PowerUp</a>. For Active Directory enumeration and exploitation, we have <a href="https://web.archive.org/web/20190508170150/https://github.com/PowerShellMafia/PowerSploit/blob/master/Recon/PowerView.ps1">PowerView</a>. Want to quickly run Mimkatz in memory on a remote box? <a href="https://web.archive.org/web/20190508170150/https://github.com/PowerShellMafia/PowerSploit/blob/master/Exfiltration/Invoke-Mimikatz.ps1">Invoke-Mimikatz</a> to the rescue. Need some creative comm channels? No problem, PowerShell can <a href="https://web.archive.org/web/20190508170150/https://github.com/lukebaggett/dnscat2-powershell">help </a><a href="https://web.archive.org/web/20190508170150/https://github.com/ChrisTruncer/WMIOps">there</a> <a href="https://web.archive.org/web/20190508170150/https://github.com/obscuresec/shmoocon/blob/master/Invoke-TwitterBot">too</a>. We even have fully-featured <a href="https://web.archive.org/web/20190508170150/https://github.com/PowerShellEmpire/Empire">remote access</a> <a href="https://web.archive.org/web/20190508170150/https://github.com/samratashok/nishang">tools </a>running written completely in PowerShell! And given Microsoft&#8217;s push to make <a href="https://web.archive.org/web/20190508170150/https://technet.microsoft.com/en-us/library/mt126167.aspx">everything </a><a href="https://web.archive.org/web/20190508170150/https://ms-iot.github.io/content/en-US/win10/samples/PowerShell.htm">administrable</a> <a href="https://web.archive.org/web/20190508170150/http://blogs.msdn.com/b/powershell/archive/2015/06/03/looking-forward-microsoft-support-for-secure-shell-ssh.aspx">via </a><a href="https://web.archive.org/web/20190508170150/http://blogs.technet.com/b/virtualization/archive/2015/05/14/powershell-direct-running-powershell-inside-a-virtual-machine-from-the-hyper-v-host.aspx">PowerShell</a>, I think it&#8217;s safe to say PowerShell isn&#8217;t going anywhere anytime soon.</p>
<p>Weaponization of PowerShell begins by invoking a PowerShell script in some way. Traditionally, offensive people use some combination of the following commands to execute PowerShell on a host:</p>
<ul>
<li>Execute a script directly: powershell.exe -ExecutionPolicy Bypass -File script.ps1</li>
<li>Execute a script via the command line:
<ul>
<li>powershell.exe -ExecutionPolicy Bypass -C &lt;command&gt;</li>
<li>powershell.exe -ExecutionPolicy Bypass -EncodedCommand &lt;Base64 Encoded Command&gt;</li>
</ul>
</li>
</ul>
<p>While these techniques work in most cases, skilled defenders can easily detect these methods through process <a href="https://web.archive.org/web/20190508170150/https://technet.microsoft.com/en-us/sysinternals/sysmon">creation </a><a href="https://web.archive.org/web/20190508170150/https://technet.microsoft.com/en-us/library/dn535776.aspx">monitoring</a>. There are a few ways to get around the need to spawn powershell.exe to execute your PowerShell. At its core, powershell.exe loads the System.Management.Automation assembly and uses the functions therein to invoke PowerShell commands. Likewise, you too can create your own C# application that references the Automation assembly and then uses the Automation assembly&#8217;s functions to execute your PowerShell (see <a href="https://web.archive.org/web/20190508170150/https://github.com/PowerShellEmpire/PowerTools/tree/master/PowerPick/SharpPick">SharpPick </a>for a great example). This, however, requires you to upload a custom application to a target&#8217;s machine (an action which is not always ideal).</p>
<p>To get around this, earlier this year I developed <a href="https://web.archive.org/web/20190508170150/https://github.com/leechristensen/UnmanagedPowerShell">UnmanagedPowerShell</a>. UnmanagedPowerShell is a C++ project that starts up the .NET framework and loads (in memory) a custom C# assembly that executes PowerShell. This is useful because, assuming you already have some form of code execution on the box, then you can use this code to inject PowerShell into any process you&#8217;d like, eliminating the need to spawn powershell.exe. Public examples of this can be seen in <a href="https://web.archive.org/web/20190508170150/https://github.com/PowerShellEmpire/PowerTools/tree/master/PowerPick">ReflectivePick </a>and PowerShell Empire&#8217;s <a href="https://web.archive.org/web/20190508170150/http://www.powershellempire.com/?page_id=273">psinject </a>command (which uses ReflectivePick underneath).</p>
<p>While UnmanagedPowerShell removed the need to initially start powershell.exe, it is not exactly ideal because it is single threaded and therefore cannot handle executing multiple PowerShell scripts at once. To get around this, you could use <a href="https://web.archive.org/web/20190508170150/https://technet.microsoft.com/en-us/library/dd878288(v=vs.85).aspx">PowerShell jobs</a>, but those have the unfortunate side effect of executing each job in a new instance of powershell.exe. You could load up UnmanagedPowerShell/ReflectivePick each time you want to execute PowerShell, but this has the potential downsides of excessive network traffic and process injection. You could also solve this by creating a new <a href="https://web.archive.org/web/20190508170150/http://blogs.technet.com/b/heyscriptingguy/archive/2013/09/29/weekend-scripter-max-out-powershell-in-a-little-bit-of-time-part-2.aspx">PowerShell runspace</a> for each job. This may also lead to problems, however, due to PowerShell scripts in different runspaces trying to PInvoke the same functions (If you have multiple runspaces, they all execute in the same app domain. Any PInvoke&#8217;d functions are bound to the AppDomain as a whole, not just in the current runspace).</p>
<p>To solve these problems, I developed the library <a href="https://web.archive.org/web/20190508170150/https://github.com/leechristensen/OffensivePowerShellTasking">PowerShellTasking</a>. PowerShellTasking solves these problems by executing each new PowerShell script in a new AppDomain (an isolation mechanism in the .NET framework). Each PowerShell script runs in the background in its own AppDomain until completion (solving the single-threaded only problem), upon which the main thread receives any output and then unloads the AppDomain (preventing the PInvoke-ing problems). PowerShellTasking is compiled as a class library and anyone can easily use it by referencing the library in their own project (see the &#8220;Example&#8221; program in the GitHub project for an example). While PowerShellTasking will not prevent scripts that use PowerShell jobs from spawning powershell.exe, it does provide operators a safer way (from an opsec perspective) to execute multiple PowerShell scripts simultaneously on a host.</p>
<p>Hopefully this gives you a better idea of how PowerShell invocation has evolved over time and gives you some insights into the challenges of PowerShell weaponization. For resources on how to detect and protect yourself from malicious PowerShell usage in your network, see Lee Holmes&#8217;s excellent <a href="https://web.archive.org/web/20190508170150/http://blogs.msdn.com/b/powershell/archive/2015/06/09/powershell-the-blue-team.aspx">blog post</a> and <a href="https://web.archive.org/web/20190508170150/http://www.irongeek.com/i.php?page=videos/derbycon5/teach-me00-when-a-powerful-platform-benefits-both-attackers-and-defenders-secure-enhancements-to-scripting-hosts-in-windows-10-lee-holmes">DerbyCon talk</a>.</p>
