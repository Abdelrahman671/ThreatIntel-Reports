    <h1 class="text-center">A deep dive into Phobos ransomware, recently deployed by 8Base group</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/guilherme-venere/">Guilherme Venere</a>
        </div>

        <br/>
        <time class="post-datetime" datetime="November 17, 2023 08:01">
            Friday, November 17, 2023 08:01
        </time>

        <div class="m-3">
                            <a href="/category/threat-spotlight/" class="category primary-category">
                                Threat Spotlight
                            </a>
                                </div>
    </div>

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <ul><li>Cisco Talos has recently observed an increase in activity conducted by 8Base, a ransomware group that uses a variant of the <a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.phobos"><u>Phobos ransomware</u></a> and other publicly available tools to facilitate their operations.&nbsp;</li><li>Most of the groupâs Phobos variants are distributed by <a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.smokeloader"><u>SmokeLoader</u></a>, a backdoor trojan. This commodity loader typically drops or downloads additional payloads when deployed. In 8Base campaigns, however, it has the ransomware component embedded in its encrypted payloads, which is then decrypted and loaded into the SmokeLoader processâ memory.</li><li>8Baseâs Phobos ransomware payload contains an embedded configuration which we describe in this blog. Besides this embedded configuration, our analysis did not uncover any other significant differences between 8Baseâs Phobos variant and other Phobos samples that have been observed in the wild since 2019.&nbsp;</li><li>Our analysis of Phobosâ configuration revealed a number of interesting capabilities, including a user access control (UAC) bypass technique and reporting victim infections to an external URL.</li><li>Notably, in all samples of Phobos released since 2019 that we analyzed, the same RSA key protected the encryption key. This led us to conclude that attaining the associated private key could enable decryption of all these samples.&nbsp;</li></ul><h1 id="smokeloader%E2%80%99s-three-stage-process-to-deliver-the-phobos-payload">SmokeLoaderâs three stage process to deliver the Phobos payload</h1><p>We wonât use this space to provide a full overview of SmokeLoader (<a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.smokeloader"><u>Malpedia has the basics</u></a>), but we would like to show how reverse-engineers can reach the final payload. In this example, weâll use the sample <em>518544e56e8ccee401ffa1b0a01a10ce23e49ec21ec441c6c7c3951b01c1b19c</em>, but <a href="https://www.virustotal.com/gui/search/behaviour_files%253A%2522.8base%2522/files"><u>any recent 8base sample</u></a> will have the same ransomware binary payload in the end. This process requires the execution of the malware in a controlled environment under a debugger like <strong><em>x32dbg</em></strong>.</p><p>SmokeLoader malware decrypts its payload in three stages. The first contains many random API calls to obfuscate the execution flow, while the other two involve shellcode stored in allocated memory. The final binary is exposed in the third stage, where a binary copy of the PE (Windows Portable Executable) data in that memory block gives the final payload in its original form.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2023/11/Phobos-flow-2.jpg" class="kg-image" alt="" loading="lazy" width="2000" height="1596" srcset="https://blog.talosintelligence.com/content/images/size/w600/2023/11/Phobos-flow-2.jpg 600w, https://blog.talosintelligence.com/content/images/size/w1000/2023/11/Phobos-flow-2.jpg 1000w, https://blog.talosintelligence.com/content/images/size/w1600/2023/11/Phobos-flow-2.jpg 1600w, https://blog.talosintelligence.com/content/images/size/w2400/2023/11/Phobos-flow-2.jpg 2400w" sizes="(min-width: 720px) 720px"><figcaption><i><em class="italic" style="white-space: pre-wrap;">Decryption process for SmokeLoader embedded payload.</em></i></figcaption></figure><p>In the first stage, enabling a breakpoint at VirtualAlloc or VirtualProtect and checking its arguments should reveal the address where the next stage will be decrypted. This memory location will then be called at the end of the entry point (EP) function, as shown above.</p><p>On the second-stage shellcode, following the second call in the function called from the entry point leads to the call to the second allocated memory with the third stage. Once the third stage is reached, that memory area should contain the unpacked binary (PE) which can then be exported to a file and analyzed. </p><p>The final payload hash extracted by this method is <em>32a674b59c3f9a45efde48368b4de7e0e76c19e06b2f18afb6638d1a080b2eb3.</em></p><h1 id="overview-of-features-included-in-the-phobos-ransomware">Overview of features included in the Phobos ransomware</h1><p>Our analysis of Phobos uncovered a number of features that enable operators of the ransomware to establish persistence in a targeted system, perform speedy encryption, and remove backups, amongst other capabilities. Phobos is a typical ransomware capable of encrypting files both in local drives as well as network shares. In our <a href="https://blog.talosintelligence.com/talos-ir-q2-2023-quarterly-recap/"><u>FY23 Q2 Talos Incident Response Quarterly Report</u></a>, we detailed how the 8Base group used their Phobos variant after installing AnyDesk to enable remote access to the machine as well as perform credential dumping via LSASS. These stolen credentials were later used to elevate privileges, exfiltrate data and run the ransomware module.</p><p>Talos has observed the following features present in the malware code:</p><ul><li>Full encryption of files below 1.5MB and partial encryption of files above this threshold to improve the speed of encryption. Larger files will have smaller blocks of data encrypted throughout the file and a list of these blocks is saved in the metadata along with the key at the end of the file.</li><li>Capability to scan for network shares in the local network.</li><li>Persistence achieved via Startup folder and Run Registry key.</li><li>Generation of target list of extensions and folders to encrypt.</li><li>Process watchdog thread to kill processes that may hold target files open. This is done to improve the chances the important files will be encrypted.</li><li>Disable system recovery, backup and shadow copies and the Windows firewall.</li><li>Embedded configuration with more than 70 options available. This configuration is encrypted with the same AES function used to encrypt files, but using a hardcoded key.</li></ul><p>Based on the analysis of this configuration data, we were able to uncover additional features present in the malware binary which can be enabled by setting specific options. These features have not been documented or have only been superficially mentioned in open source reporting to date:&nbsp;</p><ul><li>UAC bypass using a .NET profiler DLL loading vulnerability.</li><li>The existence of a debugging file that enables additional features in the malware, illustrated in the following section.</li><li>List of blocklisted file extensions that point to names of other groups using the Phobos malware.</li><li>Dynamically imported API calls to avoid behavioral detection by security products.</li><li>A hardcoded RSA key used to protect the per-file AES key used in the encryption.&nbsp;</li><li>Reporting of victim infection to an external URL.</li><li>OS check for Cyrillic language to prevent the malware from running in unwanted environments.</li></ul><p>We also examined the encryption methodology used by Phobos. Versions of Phobos released after 2019 use a custom implementation of AES-256 encryption, with a different random symmetric key used for each encrypted file, instead of using the Windows Crypto API like earlier variants. Once each file is encrypted, the key used in the encryption along with additional metadata is then encrypted using RSA-1024 with a hardcoded public key, and saved to the end of the file. This algorithm has been documented before, for example in this <a href="https://www.malwarebytes.com/blog/news/2019/07/a-deep-dive-into-phobos-ransomware"><u>Malwarebytes post from 2019</u></a>, and the process still remains the same.</p><p>This makes it extremely unlikely the files can be decrypted by brute force, as each file uses a different key, even though <a href="https://cert.pl/en/posts/2023/02/breaking-phobos/"><u>attempts at brute-forcing</u></a> it have been made in the past. It implies, however, that once the private RSA key is known, any file encrypted by any Phobos variant since 2019 can reliably be decrypted. This fact is supported by an analysis of over a thousand unpacked Phobos samples available on VirusTotal since 2019, where the same public RSA key mentioned above is used.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/wzPdqePn7KQShzfr4X-YgZBEQHgHEQdOPNRGjwvxLi6GHs1Tursc3QKfWbMtfWa_TChGP9rsi845o6XGbPjFoWZ2Z04G6eAecUZ59p_TwbV_d3FZ91Fjlv3ej-pdDLkszizwnDj5cB-uF_WWJvSXBLk" class="kg-image" alt="" loading="lazy" width="624" height="267"><figcaption><i><em class="italic" style="white-space: pre-wrap;">RSA key used in all variants of Phobos since 2019 analyzed by Talos.</em></i></figcaption></figure><p>Next, we are going to take a deeper look at the configuration file and some of the features it enables.</p><h3 id="decrypting-the-phobos-configuration-file">Decrypting the Phobos configuration file</h3><p>In order to understand how the configuration data is used in Phobos, we analyzed the malicious code in <a href="https://hex-rays.com/ida-pro/"><u>IDA Pro</u></a>. This allowed us to interact with the configuration data using <a href="https://hex-rays.com/products/ida/support/idapython_docs/"><u>IDA Python</u></a> and to decrypt all the configuration data at once. But first, we are going to look at how the configuration data looks in the binary. Once the file is loaded in IDA Pro, one of the first operations we observe being executed by the code is to check its payload and load it into memory:</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/rhhNZfDAcwsldhD5hVoHgQeUSSOUVEWlMqCgmJkKfOeqFpUCIkhObeC5j27rwXp-fYzxX3ZLLe95gjpxySqXbx-3-z9hXfwfEQy-eM8xKsZBiK93xlxdrM_Dgt6Il-y4wIg-QAnYxZZcJaAz779iF2g" class="kg-image" alt="" loading="lazy" width="624" height="519"><figcaption><i><em class="italic" style="white-space: pre-wrap;">A snippet of code at the entry point with Phobos configuration data setup in memory.</em></i></figcaption></figure><p>In the code above we see the malware initially checking the CRC32 hash of the data in the last section of the PE file. In the case of our sample, this section is named â.cdataâ although different samples may use a variation of this name. We have observed samples using â.cdataâ and â.sdataâ.</p><p>The data is then loaded into a heap-allocated memory and a structure with pointers to this data is saved to a global variable, named â<em>payoad_struct_addr</em>â in the image above. This structure will be used by the decryption function in order to load requested configuration entries later throughout the code. Each entry has a specific index which is passed as a parameter to the decryption function as we can see in the last code block.</p><p>The structure used to handle the configuration data has 48 bytes and is defined as follows:</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/VifwW9JT2X7Mbd_k4AMoldoEML3ROMwnX29Et4Uqioy1TNABvWizOkCcmtskhv0WxmBPxqNbr48i6pknkax2IPMgESQ5n11Zz78hg_B6ajnh11ycyt2FINmSisryvAyS0PaoyPUZw1mB1iMe5zbfZMc" class="kg-image" alt="" loading="lazy" width="624" height="269"><figcaption><i><em class="italic" style="white-space: pre-wrap;">Structure storing pointers to configuration data.</em></i></figcaption></figure><p>The headers and the actual data are loaded in different buffers allocated in heap memory and pointers to these buffers are saved in the structure, as well as the number of entries in the configuration. The data is followed by a 16-byte AES key used to decrypt the configuration. This key is hardcoded in the binary.&nbsp;</p><p>The header contains information to find the encrypted data for each index in the configuration. The structure has 12 bytes and is defined as follows:</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/iQ4-TYWeCZku8XIUfvpRPlub79wi8Oo0Uxgh0u-W8d9UVChuWUESD8iwY-nD9os3avV74WoNADwe-0UZ8yzXx7yaLz9tLyJB1LV8iuk6bvK1NvUgWhy_sq1FnpgEqS4YdDzi1YFH2tcNf2Ig9bopgzQ" class="kg-image" alt="" loading="lazy" width="624" height="195"><figcaption><i><em class="italic" style="white-space: pre-wrap;">Structure for header entries in configuration data.</em></i></figcaption></figure><p>The offset above is relative to the start of the encrypted data buffer starting at 0x00. The index also starts at 0x00 and each index is relatively static for the type of data it contains, which means that for a given index, every sample should have the same type of configuration data in them.</p><p>We have, however, observed that some samples show small changes in the content of their indexes. Since Phobos has been used by many groups in the past, these variations could indicate they are using a different builder or variant. In our analysis of the public samples available in VirusTotal, we found that around 88% of them have 64 configuration entries, while some have as little as 40 and as many as 72 entries.</p><p>Based on our code analysis, we were also able to infer the existence of three additional options used for setting the reporting URL and custom message to be sent back to the attacker. These options were not observed in any sample we analyzed, but the code to handle them is present in all samples. A better look at this feature is shown later in this blog.</p><p>The decryption function accepts two parameters: an index to the desired entry and a pointer to a buffer thatâs used if the entry contains more than four bytes of data. Otherwise, the buffer parameter must be zero.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/Bwghu91F4CAfJ4FftJwc1pXmdw2EnktAv860cwn2H-CgFi6fK9R0nc4n59ATcb60V7OzsRstHCb4yQePxZkU7CGGlhLQ3keCPeeXL1Ktz4W5kjIrmPIXTKpZd88pE7pcWNru9LbtO0tQUtSQGQtnrXg" class="kg-image" alt="" loading="lazy" width="624" height="671"><figcaption><i><em class="italic" style="white-space: pre-wrap;">Decryption function showing the call to AES_Init and AES_Decrypt using the key present in the payload structure.</em></i></figcaption></figure><p>The code scans the header buffer for an entry matching the requested index. It then allocates enough memory to store the encrypted data and copies the data to that space, and calls the AES_Init function with the key present at â<em>payload_struct_addr + 0x10</em>â, which is hardcoded in the sample data section. The function AES_Decrypt is then called with the encrypted data and this AES object as a parameter.</p><h3 id="automating-the-configuration-decryption-with-ida-pro">Automating the configuration decryption with IDA Pro</h3><p>Once the decryption algorithm is understood, it is easy to automate the decryption of every entry in the configuration and output the details to a file. IDA Pro allows the use of Python to automate tasks inside the application, and we decided to use the <a href="https://github.com/mandiant/flare-emu"><u>Flare-Emu</u></a> Python module to emulate the malwareâs AES routines instead of re-implementing the code in Python.</p><p>Since the decryption function requires only two parameters and is therefore fairly independent, we decided to start the emulation from that point, creating a structure similar to what the malware does for the payload data:</p><p><code>eh=flare_emu.EmuHelper()<br>conf_struct=eh.allocEmuMem(0x30)<br>header_data=eh.allocEmuMem(header_size)<br>config_data=eh.allocEmuMem(config_size)</code></p><p>Then populating this structure with the appropriate values:</p><p><code>eh.writeEmuPtr(conf_struct,header_data)<br>eh.writeEmuPtr(conf_struct+4,config_entries)<br>eh.writeEmuPtr(conf_struct+8,config_data)<br>eh.writeEmuPtr(conf_struct+0xc,config_size)<br>eh.writeEmuMem(conf_struct+0x10,AES_Init_key_struct)<br><br># load buffers<br>eh.writeEmuMem(header_data,eh.getEmuBytes(enc_data_start+0x8, header_size))<br>eh.writeEmuMem(config_data,eh.getEmuBytes(config_start,config_size))<br># write addr of struct back to code<br>eh.writeEmuPtr(eh.analysisHelper.getNameAddr(CFG_STRUCT_NAME), conf_struct)</code></p><p>Once the structure is created, we can walk through each entry in the header and prepare the emulator stack and call the emulation starting at the decryption function:</p><p><code>if entry_size&lt;=4:<br>    buffer=0<br>else:<br>    buffer=eh.allocEmuMem(entry_enc_size)<br>myStack = [0xffffffff, entry_idx, buffer]<br>eh.emulateRange(eh.analysisHelper.getNameAddr(DECRYPT_FN), stack = myStack, skipCalls=False)</code></p><p>The resulting decrypted data will be present in the EAX register if itâs four bytes or less, or in the allocated buffer if larger. After our analysis of the samples found in the wild, we found the following types of configuration entries:</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2023/11/mal_config_index.png" class="kg-image" alt="" loading="lazy" width="1750" height="3210" srcset="https://blog.talosintelligence.com/content/images/size/w600/2023/11/mal_config_index.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2023/11/mal_config_index.png 1000w, https://blog.talosintelligence.com/content/images/size/w1600/2023/11/mal_config_index.png 1600w, https://blog.talosintelligence.com/content/images/2023/11/mal_config_index.png 1750w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Malware configuration index.</span></figcaption></figure><h3 id="uac-bypass-technique">UAC bypass technique</h3><p>To execute its main objective of encrypting as many files from the victim machine as possible, a ransomware needs to be executed with elevated privileges so it can access all files on disk as well as disable important system services which may hinder its objective. Elevating process privileges usually causes a warning to be displayed to the user and may prevent the malware from running. To bypass that message, many malicious programs use an UAC bypass or exploits to elevate its own process privileges.</p><p>&nbsp;The Phobos binary contains code that performs UAC bypass using a vulnerability in the .Net Profiler DLL loading process while executing â<strong><em>compmgmt.msc</em></strong>â. This technique has been documented since at least 2017 but <a href="https://offsec.almond.consulting/UAC-bypass-dotnet.html"><u>still works</u></a> in the latest version of Windows 10.</p><p><em>Note: Even though the malware contains code to bypass UAC and the exploit works when executed, this only happens once we force the execution to follow that specific code branch and is not being actively used by the malware. The code path leading to this specific function seems to be reachable only when specific parameters are present in the malware configuration. Since Phobos is generally distributed along other malware and only after the actors are able to extract all the information they need, itâs possible the UAC code is not used because itâs not needed at the point in time the actors decide to encrypt the files.</em></p><p>In this method, a DLL is dropped to a user-writable folder and the environment variables are modified to make the .Net profiler load the file even in elevated processes. In the case of Phobos, the configuration file stores a small 2KB DLL file that only contains code to create a new process with the malicious binary. We analyzed the sample <em>fc4b14250db7f66107820ecc56026e6be3e8e0eb2d428719156cf1c53ae139c6 </em>first seen in VirusTotal in August 2019. The current 8Base sample shares 89.6% of its code with the 2019 sample, according to our analysis.</p><p>There are several functions present in the current 8Base sample that did not exist in 2019:</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/VolBP0EFBe8HpCbp_5zAapVlwgi36mhvMNfD1Z9WebCbEUqFLEVyAUSz3eLBurDwxAO0qJAIdilDYx1AUdSq51jvCl8VRzLBVwfOODydlAvTK4OdNO04MomNw138_3wa32HAksaugdQKNU7wImhUM0w" class="kg-image" alt="" loading="lazy" width="624" height="657"><figcaption><i><em class="italic" style="white-space: pre-wrap;">List of functions present in 8Base sample but not in the 2019 sample.</em></i></figcaption></figure><p>There is now support for debug files and infection report capabilities, which are not present in the old samples. This implies these features were added to Phobos source code at some point in 2019 or 2020, likely the last time the Phobos source code was updated.</p><p>Another sample that caught our attention was first <a href="https://www.malwarebytes.com/blog/news/2019/07/a-deep-dive-into-phobos-ransomware"><u>described by Malwarebytes</u></a> in 2019. The sample, <em>a91491f45b851a07f91ba5a200967921bf796d38677786de51a4a8fe5ddeafd2</em>, was first observed in the wild in May 2019. This sample is considerably different from other samples from the same time frame, only sharing 47.2% of their code.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/g_Wqn6xu98oIoVN6enr6JFoFbCau-eKJBsK2bhi8o30KUVrVx5xSawlNfUqbCFymtxa5PzUdhaOLDoxm_5Fp124V5f558FGSwBFxfQwWi4ALOFE5vsjvN-GLFP_l35YDVb0Qv0O6fXIdIDSVgWA5QEQ" class="kg-image" alt="" loading="lazy" width="624" height="168"><figcaption><i><em class="italic" style="white-space: pre-wrap;">8base sample compared to 2019 sample from Malwarebytes blog</em></i><span style="white-space: pre-wrap;">.</span></figcaption></figure><p>The main difference we observed in this sample is the usage of Windows Crypto API instead of the custom cryptographic code from recent samples. Looking at the functions present in the 2019 sample, we can see the Crypto API imported by this sample:</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/ygNc2tVl9QXigllJP4Mhb-dh_dcayB83nC3T3Oe76rzLkLTcencPkDAShs7sD0nGpneAF0m4vlbpH-zIvsp5p1iUiXjNg0tFYNRq6S43raPtBrFV5T0pK-RUPVm0NtTmgczEXKY18YZ7lGPZ4FNMnIw" class="kg-image" alt="" loading="lazy" width="469" height="335"><figcaption><i><em class="italic" style="white-space: pre-wrap;">List of imported functions from 2019 sample analyzed by Malwarebytes.</em></i></figcaption></figure><p>We observed these APIs being used in some critical functions throughout the code. The code block below is related to the encryption/decryption function in 8Base, which uses the custom cryptographic library, versus the code in the 2019 sample which uses Windows Crypto API</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://lh7-us.googleusercontent.com/LkSEeqg7bLcwHVT8bxH8QL2YB7-YQ9zQ5PdoQ8aetEKF9tBUkySWNh7ufAQO2EKWMJLGG1lnthGRYqRfBkXW5BeP-Qyp0S-t7vHTLgVgaRIVP0lUDdqWsdyy0owCztPHM9TVXlvfXol5T9iUBPJAtUA" class="kg-image" alt="" loading="lazy" width="624" height="324"><figcaption><i><em class="italic" style="white-space: pre-wrap;">File encryption function comparison between 8Base and old 2019 Phobos sample.</em></i></figcaption></figure><p>There are similar differences in the function used to decrypt the configuration file, which behaves in a related fashion but using different cryptographic APIs.</p><p>These changes observed in early samples support the assumption that Phobos went through a development phase in 2019 but has remained unchanged since then.</p><p>In our second post titled
