The group is known for its destructive wiper and fake-ransomware attacks and mainly targets Israeli organizations across multiple industries and countries.\r\n\r\nThough earlier reports of these attacks mentioned ransomware and ransom notes, these turned out to be a ruse (a trend noted in the 2023 Unit 42 Ransomware and Extortion Report). In the most recent attacks, the attackers did not request ransom \u2013 rather, the potential result of the attacks was vast data loss and disruptions of business continuity.\r\n\r\nAttacks from Agonizing Serpens usually have two main goals, the first being stealing sensitive information that includes PII and intellectual property, which threat actors then publish on social media or Telegram channels. It seems likely that their motivation to publish on social media is to sow fear or inflict reputational damage. The second goal is wreaking havoc and inflicting considerable damage by wiping as many endpoints as possible.\r\n\r\nSince its emergence, the group has developed new custom tools and they have also leveraged known hacking tools and techniques to carry out their aggressive operations.\r\nTechnical Analysis\r\nThe following sections detail the different stages of a recent incident carried out by Agonizing Serpens in October 2023, as analyzed by Unit 42 researchers.\r\nEntry Vector\r\nThe attackers gained initial access to the environment by exploiting vulnerable internet facing web servers. Subsequently, they deployed multiple web shells, which granted them a foothold in the network.\r\n\r\nThe web shells that threat actors used in the described attack (shown in Figure 1) contain the same code as web shells that were observed in previous Agonizing Serpens attacks, with variations to the naming of functions. The web shells appear to be variations of ASPXSpy.\r\n\r\n[caption id=\"attachment_131014\" align=\"aligncenter\" width=\"563\"] Figure 1. Snippet from xcopy.aspx web shell.[\/caption]\r\n\r\nAnother web shell used in this attack was named Uploader.aspx. Basic reconnaissance commands via the web shells shown in Cortex XDR.[\/caption]\r\nReconnaissance\r\nTo map out the network, the attackers used various known and publicly available scanners.\r\nNbtscan\r\nThe attackers used Nbtscan, renamed as systems.txt, to scan the network for existing hosts (shown in Figure 4).\r\n\r\n[caption id=\"attachment_131020\" align=\"aligncenter\" width=\"900\"] Figure 4. WinEggdDrop scanner used for port scanning.[\/caption]\r\nNimScan\r\nNimScan is another publicly available port scanner that the attackers used in the attack, as shown in Figure 6.\r\n\r\n[caption id=\"attachment_131024\" align=\"aligncenter\" width=\"592\"] Figure 6. NimScan being used for port scanning.[\/caption]\r\nCredential Stealing\r\nA crucial phase of the attack consisted of obtaining credentials of users with administrative privileges. To do so, the attackers tried multiple methods to obtain credentials, which were prevented by the Cortex XDR platform:\r\n\r\n \tMimikatz (filename: Mimi.exe)\r\n \tSMB password spraying\r\n \tSMB password brute force (shown in Figure 7)\r\n\r\n[caption id=\"attachment_131026\" align=\"aligncenter\" width=\"900\"] Figure 7. Password brute forcing using SMB.[\/caption]\r\n\r\n \tDumping the SAM file (shown in Figure 8)\r\n\r\n[caption id=\"attachment_131028\" align=\"aligncenter\" width=\"802\"] Figure 8. Dumping the SAM file.[\/caption]\r\nLateral Movement\r\nFigure 9 shows that to move laterally in the environment, the attackers mostly used Plink (renamed as systems.exe) to create remote tunneling and establish connections to remote machines.\r\n\r\n[caption id=\"attachment_131030\" align=\"aligncenter\" width=\"900\"] Figure 9. Plink used to establish remote tunneling.[\/caption]\r\nStealing and Exfiltrating Data\r\nAttackers attempted to steal information from databases and other critical servers before executing wipers to cover their tracks. They then tried to exfiltrate this information to the attackers\u2019 C2 servers, using different publicly available tools such as WinSCP and Putty.\r\nExtracting Database Information Using Sqlextractor\r\nThe attackers used a custom tool they named sqlextractor (binary name sql.net4.exe). Its purpose is to query SQL databases and extract sensitive PII data, such as the following:\r\n\r\n \tID numbers\r\n \tPassport scans\r\n \tEmails\r\n \tFull addresses\r\n\r\nThe data was saved into CSV files as shown in Figures 10 and 11. The tool writes the data to a hard-coded staging path: C:\\windows\\temp\\s\\.\r\n\r\nFigure 10 shows the attackers then used 7za.exe to archive the extracted data in preparation for exfiltration.\r\n\r\n[caption id=\"attachment_131032\" align=\"aligncenter\" width=\"900\"] Figure 10. Sqlextractor and 7za.exe used to extract files and archive them.[\/caption]\r\n\r\n[caption id=\"attachment_131034\" align=\"aligncenter\" width=\"668\"] Figure 11. Sqlextractor writes extracted data to CSV files.[\/caption]\r\n\r\nFigure 12 shows the attackers also used 7zG.exe to archive interesting folders in the infected environment.\r\n\r\n[caption id=\"attachment_131036\" align=\"aligncenter\" width=\"576\"] Figure 12. 7zG.exe used to archive various folders.[\/caption]\r\nData Exfiltration Using WinSCP\r\nThe attackers attempted to use WinSCP to exfiltrate files from the environment, as shown in Figures 13 and 14.\r\n\r\n[caption id=\"attachment_131038\" align=\"aligncenter\" width=\"482\"] Figure 13. WinSCP being used to exfiltrate files.[\/caption]\r\n\r\n[caption id=\"attachment_131040\" align=\"aligncenter\" width=\"900\"] Figure 14. Exfiltration alerts by Cortex XDR.[\/caption]\r\nData Exfiltration Using Pscp.exe (PuTTY Secure Copy Protocol)\r\nFigure 15 shows that another tool the attackers used for exfiltration is pscp.exe (PuTTY Secure Copy Protocol).\r\n\r\nThe attackers attempted to establish a connection to the C2, then searched for .7z and .ezip files that contain stolen data, as well as .dmp files created by ProcDump.\r\n\r\n[caption id=\"attachment_131042\" align=\"aligncenter\" width=\"410\"] Figure 15. pscp.exe used for exfiltration.[\/caption]\r\nWiper Payloads\r\nDuring the incident, the attackers attempted to use three separate wipers as part of the destructive attack. While some of the wipers show code similarities to previously reported wipers the Agonizing Serpens group used, others are considered brand new. These have been used for the first time in this attack, as detailed in the following section.\r\nMultiLayer Wiper\r\nThe first wiper that the attackers used is .NET malware called MultiLayer. As its name suggests, this wiper contains multiple layers and stages.\r\n\r\nIts compilation date is Oct. 14, 2093. As this is set to a future date, it is a clear sign of metadata manipulation.\r\n\r\nFigure 16 shows that it contains two more binaries in its resources section, named MultiList and MultiWip.\r\n\r\n[caption id=\"attachment_131044\" align=\"aligncenter\" width=\"299\"] Figure 16. MultiLayer resources.[\/caption]\r\n\r\nMultiLayer drops and executes each of the aforementioned binaries, then deletes them right after their execution.\r\nThe MultiList Component - Setting the Target Files\r\nMultiList generates a list of all the files and their paths on the fixed drives on the system. It does this by enumerating all files on the infected operating system while excluding specific folders defined in a predefined list (shown in Figure 17). The attackers can define the path to which this tool should store the list via the command line.\r\n\r\n[caption id=\"attachment_131046\" align=\"aligncenter\" width=\"900\"] Figure 17. MultiList exclusion functionality.[\/caption]\r\nMultiWip - the Core Wiper Component\r\nThe MultiWip component contains the actual file wiping functionality. It relies on the previous component (MultiList) and reads the output list of files to wipe, which is passed as a command-line argument.\r\n\r\nMultiWip\u2019s main function is called DoJob() and is responsible for carrying out the file-wiping activity in the manner shown in Figure 18.\r\n\r\n[caption id=\"attachment_131048\" align=\"aligncenter\" width=\"601\"] Figure 18. Snippet from MultiWip\u2019s main DoJob() function.[\/caption]\r\n\r\nThe malware takes the following steps in the order indicated:\r\n\r\n \tFiles located on network drives are deleted immediately.\r\n \tLocally stored files are corrupted and overwritten with random data to thwart file recovery efforts, as shown in Figure 19.\r\n\r\n[caption id=\"attachment_131050\" align=\"aligncenter\" width=\"900\"] Figure 19. Snippet of MultiWip File data destruction function.[\/caption]\r\n\r\n \tThe malware changes the original timestamps in the following FileSystemInfo properties: LastAccessTime, LastWriteTime and CreationTime. This is a well-known anti-forensic timestomping technique. The malware timestomps according to the file system. If the file system is NTFS, the malware sets the timestamp to 1601.1.1. Any other file system, the malware sets it to 1980.1.1 (shown in Figure 20).\r\n\r\n[caption id=\"attachment_131052\" align=\"aligncenter\" width=\"900\"] Figure 20. MultiWip timestomp function.[\/caption]\r\n\r\n \tThe malware changes the original paths of the deleted files, using Path.GetRandomFileName, to make any recovery efforts extremely hard.\r\n \tFinally, the malware deletes the files.\r\n\r\nCovering Tracks and Rendering the System Unusable\r\nMultiLayer is designed to cover its tracks by erasing evidence of its execution. It does so by running various commands to prevent restoration of lost data and to render the disk unusable.\r\n\r\nFigure 21 shows that MultiLayer uses the DeleteLogs() function to create a scheduled task that launches a batch script only once. The script then removes all the Windows Event Logs.\r\n\r\n[caption id=\"attachment_131054\" align=\"aligncenter\" width=\"723\"] Figure 21. MultiLayer scheduled task to delete events logs.[\/caption]\r\n\r\nTo remain stealthy, MultiLayer removes all the files it uses after its execution, including itself. To remove itself, MultiLayer uses SelfDelete().\r\n\r\nThe removal is implemented by threat actors writing a batch file named remover.bat to %TEMP% and executing it. The batch file deletes the assembly file and the batch itself, and then it clears the file system cache memory, leveraging the ProcessIdleTasks export in advapi32.dll.\r\n\r\nTo further prevent data restoration, MultiLayer tries to remove all shadow copies on the system as shown in Figure 22, and then remove the Volume Shadow Copy (VSS) service itself.\r\n\r\n[caption id=\"attachment_131056\" align=\"aligncenter\" width=\"603\"] Figure 22. MultiLayer deletion of the shadow copies.[\/caption]\r\n\r\nFigure 23 shows that, to ensure that the system can no longer boot, MultiLayer opens a handle to \\\\\\\\.\\\\PhysicalDrive0 and wipes the first 512 bytes (aka the boot sector).\r\n\r\n[caption id=\"attachment_131058\" align=\"aligncenter\" width=\"900\"] Figure 23. MultiLayer wiping the boot sector.[\/caption]\r\n\r\nFinally, after the boot sector is corrupted, MultiLayer adjusts its privileges to SeShutdownPrivilege and calls the Windows API ExitWindowsEx with the EWX_REBOOT flag, which indicates system reboot. Once the system reboots, it will not be able to boot again.\r\n\r\nThe attempt to execute MultiLayer was prevented by Cortex XDR, as depicted in Figure 24 below.\r\n\r\n[caption id=\"attachment_131060\" align=\"aligncenter\" width=\"722\"] Figure 24. Blocked execution of the MultiLayer wiper by Cortex XDR platform.[\/caption]\r\nSimilarities to Apostle, IPsec Helper, and Fantasy\r\nThroughout our analysis of MultiLayer, we noticed multiple code overlaps with Apostle, IPsec Helper and Fantasy. These are custom tools previously used by Agonizing Serpens. This might be the result of a shared codebase or being written by the same team of developers. When comparing the code of the aforementioned tools, it appears that MultiLayer shares naming conventions and even entire code blocks with them.\r\n\r\nExample 1: Self-Deletion Mechanism\r\nThe self-deletion mechanism of MultiLayer is implemented in a similar manner to IPsec Helper, Apostle and Fantasy. They share the same name for the function, named SelfDelete(). They also delete themselves by writing a batch file named remover.bat to %TEMP% and executing it, using the above mentioned functions shown in Figures 25 and 26.\r\n\r\n[caption id=\"attachment_131062\" align=\"aligncenter\" width=\"651\"] Figure 25. MultiLayer\u2019s SelfDelete() function.[\/caption]\r\n\r\n[caption id=\"attachment_131064\" align=\"aligncenter\" width=\"572\"] Figure 26. Code snippet of the IPsec Helper. Source: Figure 20 of SentinalLABS report From Wiper to Ransomware: The Evolution of Agrius.[\/caption]\r\n\r\nExample 2: Directory Listing Implementation\r\n\r\nThe recursive directory listing mechanism of MultiList is implemented in a similar manner to Fantasy and Apostle. They share the same name for the function, named GetSubDirectoryFileListRecusrive.\r\n\r\nThey also both call GetSubDirectoryFileListRecusrive() and GetDirectoryFileList(), where GetSubDirectoryFileListRecusrive() is called recursively as shown in the code snippets in Figures 27 and 28.\r\n\r\n[caption id=\"attachment_131066\" align=\"aligncenter\" width=\"881\"] Figure 27. Recursive directory listing in MultiList.[\/caption]\r\n\r\n[caption id=\"attachment_131068\" align=\"aligncenter\" width=\"777\"] Figure 28. Recursive directory listing in Fantasy. Source: Figure 6 in ESET blog \"Fantasy \u2013 a new Agrius wiper deployed through a supply\u2011chain attack.\"[\/caption]\r\nPartialWasher Wiper\r\nDuring the attack, the attackers attempted to use a second wiper, which they call PartialWasher or PW. Figure 29 shows that it was compiled on Oct. 8 and, unlike other .NET wipers mentioned in this article, it is written in C++.\r\n\r\n[caption id=\"attachment_131070\" align=\"aligncenter\" width=\"568\"] Figure 29. The compilation timestamp of PartialWasher.[\/caption]\r\n\r\nPartialWasher defines itself as a crucial process by calling NtSetInformationProcess, and it supports command-line arguments. In case no arguments are provided, the default functionality would be a wiper functionality.\r\n\r\nWhen passing 1 as an argument, the attacker can then use an interactive command-line interface (CLI). There are several typos in the interface\u2019s text, indicating that the authors are likely not native English speakers.\r\n\r\nFigure 30 shows an example of the passed arguments S \/p. They trigger the malware to gather information about available drives on the infected machine.\r\n\r\n[caption id=\"attachment_131072\" align=\"aligncenter\" width=\"900\"] Figure 30. PartialWasher\u2019s CLI and typos marked in red squares.[\/caption]\r\n\r\nThe supported commands demonstrate the wiper\u2019s further capabilities to perform individual wiping tasks at the request of its operators. It does so by implementing several anti-hooking techniques, which have not been reported thus far as part of the group's known techniques. This suggests a possible upgrade of their capabilities.\r\n\r\nThe following sections list the anti-hooking functions BFG runs before its main payload.\r\nDLL Unhooking\r\nDLL unhooking is an anti-hooking technique that attempts to remove the user mode inline hooks, which various security solutions often implement. The technique works by restoring the bytes of the hooked functions to their original disk values. This technique is well known, and it is likely that the wipers\u2019 authors largely adopted the following publicly available code.\r\nImport Address Table (IAT) Unhooking\r\nIAT unhooking is an anti-hooking technique that attempts to remove the user-mode IAT hooks, which various security solutions often implement. Based on the wiper\u2019s code, it is likely that the authors largely adopted publicly available IAT unhooking code snippets.\r\nWiping the Boot Sector\r\nTo wipe the boot sector, the wiper retrieves a device handle to \\\\.\\PhysicalDrive0 as shown in Figure 33. Once the system crashes, it will not be able to boot again (shown in Figure 35).\r\n\r\n[caption id=\"attachment_131082\" align=\"aligncenter\" width=\"439\"] Figure 35. Execution of BFG Agonizer wiper blocked by the Cortex XDR platform.[\/caption]\r\nAttempted Anti-EDR Activity\r\nDuring the attack, the group specifically attempted to bypass EDR solutions to carry out their attack uninterrupted and with greater stealth.     <span class="wpml-ls-slot-post_translations wpml-ls-item wpml-ls-item-ja wpml-ls-first-item wpml-ls-last-item wpml-ls-item-legacy-post-translations"><a href="https://unit42.paloaltonetworks.jp/agonizing-serpens-targets-israeli-tech-higher-ed-sectors/" class="wpml-ls-link"><span class="wpml-ls-native" lang="ja">日本語</span><span class="wpml-ls-display"><span class="wpml-ls-bracket"> (</span>Japanese<span class="wpml-ls-bracket">)</span></span></a></span></p><h2><a id="post-131008-_5w6sdwx5mcgc"></a>Executive Summary</h2>
<p>Unit 42 researchers have investigated a series of destructive cyberattacks beginning in January 2023 and continuing as recently as October 2023, targeting the education and technology sectors in Israel.</p>
<p>The attacks are characterized by attempts to steal sensitive data, such as personally identifiable information (PII) and intellectual property. Once the attackers stole the information, they deployed various wipers intended to cover the attackers’ tracks and to render the infected endpoints unusable.</p>
<p>Our investigation revealed the perpetrators of the attacks have strong connections to an Iranian-backed APT group Unit 42 tracks as Agonizing Serpens (aka <a href="https://apt.etda.or.th/cgi-bin/showcard.cgi?g=Agrius&amp;n=1" target="_blank" rel="noopener">Agrius</a>, <a href="https://www.bleepingcomputer.com/news/security/blackshadow-hackers-extort-israeli-insurance-company-for-1-million/" target="_blank" rel="noopener">BlackShadow</a>, <a href="https://learn.microsoft.com/en-us/microsoft-365/security/intelligence/microsoft-threat-actor-naming?view=o365-worldwide" target="_blank" rel="noopener">Pink Sandstorm</a>, <a href="https://learn.microsoft.com/en-us/microsoft-365/security/intelligence/microsoft-threat-actor-naming?view=o365-worldwide" target="_blank" rel="noopener">DEV-0022</a>).</p>
<p>Unit 42 researchers were also able to identify novel new wipers and tools that Agonizing Serpens used in their most recent attacks:</p>
<ul>
<li>MultiLayer wiper</li>
<li>PartialWasher wiper</li>
<li>BFG Agonizer wiper</li>
<li>Sqlextractor - a custom tool to extract information from database servers</li>
</ul>
<p>Based on forensic evidence, it appears that the Agonizing Serpens APT group has recently upgraded their capabilities and they are investing great efforts and resources to attempt to bypass endpoint detection and response (EDR) and other security measures. To do so, they have been rotating between using different known proof of concept (PoC) and pentesting tools as well as custom tools.</p>
<p>For the attacks described below, the attacker was unsuccessful at bypassing Cortex XDR. Cortex XDR and XSIAM detect and prevent the execution flow described. They also build behavioral profiles of user activity over time with machine learning, allowing them to detect anomalous activity indicative of, for example, credential-based attacks.</p>
<p>We are sharing this research to provide detection, prevention and hunting recommendations to help organizations protect against the threats associated with Agonizing Serpens.</p>

<h2><a id="post-131008-_9qe1dt2ghwhp"></a>Who Is the Agonizing Serpens APT Group?</h2>
<p>Agonizing Serpens (aka Agrius) is an Iranian-linked APT group that has been active since 2020. The group is known for its destructive wiper and fake-ransomware attacks and mainly targets Israeli organizations across multiple industries and countries.</p>
<p>Though earlier reports of these attacks mentioned ransomware and ransom notes, these turned out to be a ruse (a trend noted in the <a href="https://start.paloaltonetworks.com/2023-unit42-ransomware-extortion-report" target="_blank" rel="noopener">2023 Unit 42 Ransomware and Extortion Report</a>). In the most recent attacks, the attackers did not request ransom – rather, the potential result of the attacks was vast data loss and disruptions of business continuity.</p>
<p>Attacks from Agonizing Serpens usually have two main goals, the first being stealing sensitive information that includes PII and intellectual property, which threat actors then publish on social media or Telegram channels. It seems likely that their motivation to publish on social media is to sow fear or inflict reputational damage. The second goal is wreaking havoc and inflicting considerable damage by wiping as many endpoints as possible.</p>
<p>Since its emergence, the group has developed new custom tools and they have also leveraged known hacking tools and techniques to carry out their aggressive operations.</p>
<h2><a id="post-131008-_dg3kemqhofes"></a>Technical Analysis</h2>

<h3><a id="post-131008-_gpm29t634ood"></a>Wiper Payloads</h3>
<p>During the incident, the attackers attempted to use three separate wipers as part of the destructive attack. While some of the wipers show code similarities to previously reported wipers the Agonizing Serpens group used, others are considered brand new. These have been used for the first time in this attack, as detailed in the following section.</p>
<h4><a id="post-131008-_qg8d6gez366v"></a>MultiLayer Wiper</h4>
<p>The first wiper that the attackers used is .NET malware called MultiLayer. As its name suggests, this wiper contains multiple layers and stages.</p>
<p>Its compilation date is Oct. 14, 2093. As this is set to a future date, it is a clear sign of metadata manipulation.</p>
<p>Figure 16 shows that it contains two more binaries in its resources section, named MultiList and MultiWip.</p>
<figure id="attachment_131044" aria-describedby="caption-attachment-131044" style="width: 299px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131044" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-16.png" alt="Image 16 is a screenshot of a file directory. MultiLayer resources.</figcaption></figure>
<p>MultiLayer drops and executes each of the aforementioned binaries, then deletes them right after their execution.</p>
<h5><a id="post-131008-_mgkz2l756wp"></a>The MultiList Component - Setting the Target Files</h5>
<p>MultiList generates a list of all the files and their paths on the fixed drives on the system. It does this by enumerating all files on the infected operating system while excluding specific folders defined in a predefined list (shown in Figure 17). The attackers can define the path to which this tool should store the list via the command line.</p>
<figure id="attachment_131046" aria-describedby="caption-attachment-131046" style="width: 900px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131046" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-17.png" alt="Image 17 is a screenshot of the code of the predefined list of folders that are excluded by MultiList." width="900" height="503" /><figcaption id="caption-attachment-131046" class="wp-caption-text">Figure 17. <h5><a id="post-131008-_ny3jmqall2io"></a>MultiWip - the Core Wiper Component</h5>
<p>The MultiWip component contains the actual file wiping functionality. It relies on the previous component (MultiList) and reads the output list of files to wipe, which is passed as a command-line argument.</p>
<p>MultiWip’s main function is called <span style="font-family: 'courier new', courier, monospace;">DoJob()</span> and is responsible for carrying out the file-wiping activity in the manner shown in Figure 18.</p>
<figure id="attachment_131048" aria-describedby="caption-attachment-131048" style="width: 601px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131048" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-18.png" alt="Image 18 is a screenshot of the code that defines MultiWip’s DoJob function." width="601" height="502" /><figcaption id="caption-attachment-131048" class="wp-caption-text">Figure 18. <ol>
<li>Files located on network drives are deleted immediately.</li>
<li>Locally stored files are corrupted and overwritten with random data to thwart file recovery efforts, as shown in Figure 19.</li>
</ol>

<h5><a id="post-131008-_ug0emwjzalh3"></a>Covering Tracks and Rendering the System Unusable</h5>
<p>MultiLayer is designed to cover its tracks by erasing evidence of its execution. It does so by running various commands to prevent restoration of lost data and to render the disk unusable.</p>
<p>Figure 21 shows that MultiLayer uses the <span style="font-family: 'courier new', courier, monospace;">DeleteLogs()</span> function to create a scheduled task that launches a batch script only once. The script then removes all the Windows Event Logs.</p>
<figure id="attachment_131054" aria-describedby="caption-attachment-131054" style="width: 723px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131054" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-21.png" alt="Image 21 is a screenshot of the code that defines MultiLayer’s capability to delete event logs." width="723" height="294" /><figcaption id="caption-attachment-131054" class="wp-caption-text">Figure 21. MultiLayer scheduled task to delete events logs.</figcaption></figure>
<p>To remain stealthy, MultiLayer removes all the files it uses after its execution, including itself. To remove itself, MultiLayer uses <span style="font-family: 'courier new', courier, monospace;">SelfDelete()</span>.</p>
<p>The removal is implemented by threat actors writing a batch file named <span style="font-family: 'courier new', courier, monospace;">remover.bat</span> to <span style="font-family: 'courier new', courier, monospace;">%TEMP%</span> and executing it. The batch file deletes the assembly file and the batch itself, and then it clears the file system cache memory, leveraging the <span style="font-family: 'courier new', courier, monospace;">ProcessIdleTasks</span> export in<span style="font-family: 'courier new', courier, monospace;"> advapi32.dll</span>.</p>
<p>To further prevent data restoration, MultiLayer tries to remove all shadow copies on the system as shown in Figure 22, and then remove the Volume Shadow Copy (VSS) service itself.</p>
<figure id="attachment_131056" aria-describedby="caption-attachment-131056" style="width: 603px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131056" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-22.png" alt="Image 22 is a screenshot of the code that defines MultilLayer’s ability to delete shadow copies." width="603" height="57" /><figcaption id="caption-attachment-131056" class="wp-caption-text">Figure 22. MultiLayer deletion of the shadow copies.</figcaption></figure>
<p>Figure 23 shows that, to ensure that the system can no longer boot, MultiLayer opens a handle to <span style="font-family: 'courier new', courier, monospace;">\\\\.\\PhysicalDrive0</span> and wipes the first 512 bytes (aka the boot sector).</p>
<figure id="attachment_131058" aria-describedby="caption-attachment-131058" style="width: 900px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131058" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-23.png" alt="Image 23 is a screenshot of the code that defines MultilLayer’s ability to wipe the boot sector." width="900" height="184" /><figcaption id="caption-attachment-131058" class="wp-caption-text">Figure 23. MultiLayer wiping the boot sector.</figcaption></figure>
<p>Finally, after the boot sector is corrupted, MultiLayer adjusts its privileges to <span style="font-family: 'courier new', courier, monospace;">SeShutdownPrivilege</span> and calls the Windows API <span style="font-family: 'courier new', courier, monospace;"><a href="https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-exitwindowsex" target="_blank" rel="noopener">ExitWindowsEx</a></span> with the <span style="font-family: 'courier new', courier, monospace;">EWX_REBOOT</span> flag, which indicates system reboot. Once the system reboots, it will not be able to boot again.</p>

<h5><a id="post-131008-_ed5xtdvh7v9q"></a>DLL Unhooking</h5>
<p>DLL unhooking is an anti-hooking technique that attempts to remove the user mode <a href="https://www.malwaretech.com/2015/01/inline-hooking-for-programmers-part-1.html" target="_blank" rel="noopener">inline hooks</a>, which various security solutions often implement. The technique works by restoring the bytes of the hooked functions to their original disk values. This technique is well known, and it is likely that the wipers’ authors largely adopted the following <a href="https://www.ired.team/offensive-security/defense-evasion/how-to-unhook-a-dll-using-c++" target="_blank" rel="noopener">publicly available code</a>.</p>
<h5><a id="post-131008-_ujmll29d790a"></a>Import Address Table (IAT) Unhooking</h5>
<p>IAT unhooking is an anti-hooking technique that attempts to remove the user-mode <a href="https://www.ired.team/offensive-security/code-injection-process-injection/import-adress-table-iat-hooking" target="_blank" rel="noopener">IAT hooks, which various security solutions often implement.</a> Based on the wiper’s code, it is likely that the authors largely adopted <a href="https://alice.climent-pommeret.red/posts/how-and-why-to-unhook-the-import-address-table/" target="_blank" rel="noopener">publicly available IAT unhooking code</a> snippets.</p>

<h2><a id="post-131008-_b8krt4nregls"></a>Attempted Anti-EDR Activity</h2>
<p>During the attack, the group specifically attempted to bypass EDR solutions to carry out their attack uninterrupted and with greater stealth. These attempts were blocked by the Cortex XDR platform. It is interesting to note that the group tried multiple tools and techniques, and each time they failed with one, they tried to leverage another.</p>
<p>While most of the techniques are known and well-documented, the group has not used them in previous publicly reported attacks. This could suggest that the group is becoming increasingly advanced and aggressive in its approach.</p>
<p>The following sections list some of the EDR bypass tools and techniques in the order they leveraged them.</p>
<h3><a id="post-131008-_kpb7q4pu4d59"></a>EDR Service Dependency Bypass</h3>
<p>The threat actor attempted their first EDR bypass technique on Oct. 6. The actor tried to manipulate the Cortex XDR service auto-start functionally. By leveraging previously obtained Administrator privileges, the attackers tried to modify the services Cortex XDR depends on, so it would not be able to auto-start upon system startup.</p>
<p>Figure 37 shows these attempts were prevented, and then the attackers shifted into using custom tools that abuse the “bring your own vulnerable driver” (<a href="https://cybernews.com/security/bring-your-own-vulnerable-driver-attack/" target="_blank" rel="noopener">BYOVD</a>) technique.</p>

<h3><a id="post-131008-_xi59rib1k4fi"></a>DrvIX: A Custom BYOVD Loader</h3>
<p>The first custom tool the attackers used was a binary named <span style="font-family: 'courier new', courier, monospace;">agmt.exe</span>, which was compiled on Oct. 7. According to its PDB path (<span style="font-family: 'courier new', courier, monospace;">C:\Users\dude\source\repos\drvix\x64\Release\drvix.pdb</span>), it appears that this tool’s original name is drvIX (shown in Figure 38).</p>
<figure id="attachment_131088" aria-describedby="caption-attachment-131088" style="width: 429px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131088" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-38.png" alt="Image 38 is a screenshot of the agmt.exe PDB path and compilation timestamp. <p>However, according to the binary help function shown in Figure 39, the name is Drvtopia.</p>
<figure id="attachment_131090" aria-describedby="caption-attachment-131090" style="width: 771px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131090" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-39.png" alt="Image 39 is a screenshot of the code with the help section that mentions DRVtopia." width="771" height="78" /><figcaption id="caption-attachment-131090" class="wp-caption-text">Figure 39. DrvIX help section mentions Drvtopia.</figcaption></figure>
<p><span style="font-family: 'courier new', courier, monospace;">Agmt.exe</span> is a custom loader and operator for the <a href="https://www.gmer.net/" target="_blank" rel="noopener">GMER</a> driver, <span style="font-family: 'courier new', courier, monospace;">gmer64.sys</span> (renamed to <span style="font-family: 'courier new', courier, monospace;">AGMT.sys</span>). GMER’s original intended purpose is to detect and remove rootkits; however, threat actors can also leverage it to remove security products. Using <span style="font-family: 'courier new', courier, monospace;">agmt.exe</span>, the attackers can specify the PID of the target process they wish to terminate via the command line.</p>
<p><span style="font-family: 'courier new', courier, monospace;">Agmt.exe</span> starts by registering GMER a new kernel driver (<span style="font-family: 'courier new', courier, monospace;">agmt.sys</span>) as a service named AGMT and starting it, which in turn causes the operating system to load the driver into the kernel.</p>
<p>To communicate and abuse the GMER functionality of terminating processes, <span style="font-family: 'courier new', courier, monospace;">agmt.exe</span> retrieves a device handle to GMER’s device object as shown in Figure 40.</p>
<figure id="attachment_131092" aria-describedby="caption-attachment-131092" style="width: 541px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131092" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-40.png" alt="Image 40 is a screenshot of the code that opens a handle to the GMER device object." width="541" height="110" /><figcaption id="caption-attachment-131092" class="wp-caption-text">Figure 40. <span style="font-family: 'courier new', courier, monospace;">Agmt.exe</span> opens a handle to the GMER device object.</figcaption></figure>
<p>Then, <span style="font-family: 'courier new', courier, monospace;">agmt.exe</span> uses <span style="font-family: 'courier new', courier, monospace;"><a href="https://learn.microsoft.com/en-us/windows/win32/api/ioapiset/nf-ioapiset-deviceiocontrol" target="_blank" rel="noopener">DeviceIoControl</a></span> Windows API with the control code <span style="font-family: 'courier new', courier, monospace;">0x9876C094</span>, while specifying the PID in the <span style="font-family: 'courier new', courier, monospace;">Input_Buffer</span> parameter of the call (shown in Figure 41).</p>
<figure id="attachment_131094" aria-describedby="caption-attachment-131094" style="width: 670px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131094" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-41.png" alt="Image 41 is a screenshot of the code that communicates with the GMER driver in order to terminate processes." width="670" height="173" /><figcaption id="caption-attachment-131094" class="wp-caption-text">Figure 41. <span style="font-family: 'courier new', courier, monospace;">Agmt.exe</span> communicates with the GMER driver for terminating processes.</figcaption></figure>
<p><span style="font-family: 'courier new', courier, monospace;"><a href="https://learn.microsoft.com/en-us/windows/win32/devio/calling-deviceiocontrol" target="_blank" rel="noopener">DeviceIoControl</a></span> allows user mode processes to directly communicate with kernel drivers, allowing the processes to request the drivers to service certain operations for them.</p>
<p>In the case of <span style="font-family: 'courier new', courier, monospace;">agmt.exe</span>, the <span style="font-family: 'courier new', courier, monospace;">DeviceIoControl</span> API call triggers a process termination operation for the GMER driver. Figure 44 shows that by inspecting the GMER driver, we can determine that the functionality of the <span style="font-family: 'courier new', courier, monospace;">0x9876C094</span> control code is to terminate the target process provided by PID.</p>
<figure id="attachment_131096" aria-describedby="caption-attachment-131096" style="width: 293px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131096" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-42.png" alt="Image 42 is a screenshot of the code that controls code functionality. <p>The function opens a handle to the target process PID using <span style="font-family: 'courier new', courier, monospace;"><a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/ntddk/nf-ntddk-zwopenprocess" target="_blank" rel="noopener">ZwOpenProcess</a></span> and then terminates it by calling <span style="font-family: 'courier new', courier, monospace;"><a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/ntddk/nf-ntddk-zwterminateprocess" target="_blank" rel="noopener">ZwTerminateProcess</a></span>.</p>
<p>The attempt to leverage the GMER driver failed, as shown in Figure 43 below.</p>
<figure id="attachment_131098" aria-describedby="caption-attachment-131098" style="width: 900px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131098" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-43.png" alt="Image 43 is a screenshot of information from Cortex XDR. It includes the source which is XDR agent, the category, which is malware, the tags, the module, which is Behavioral Threat Protection, the severity, which is high, the description, and the action." width="900" height="107" /><figcaption id="caption-attachment-131098" class="wp-caption-text">Figure 43. Loading of the GMER driver being blocked by the Cortex XDR Platform.</figcaption></figure>
<h3><a id="post-131008-_wq92bkcmcwwa"></a>Second Vulnerable Driver Attempt (Rentdrv2 Driver)</h3>
<p>As the attempt to exploit the GMER driver failed, the attackers tried arming their drvIX tool. They did so using a different vulnerable driver from a new publicly available PoC tool called <a href="https://github.com/keowu/BadRentdrv2/tree/main" target="_blank" rel="noopener">BadRentdrv2</a>, first published in the beginning of October 2023.</p>
<p>The attacker used the same project and compiled a modified version of the tool a day later, on Oct. 8 as shown in Figure 44. This time, the binary’s original name <span style="font-family: 'courier new', courier, monospace;">drvIX.exe</span> was not changed.</p>
<figure id="attachment_131100" aria-describedby="caption-attachment-131100" style="width: 427px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131100" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-44.png" alt="Image 44 is a screenshot of the PDB path and compilation for drvIX.exe. it includes the stamp and file-name. " width="427" height="43" /><figcaption id="caption-attachment-131100" class="wp-caption-text">Figure 44. PDB path and compilation for <span style="font-family: 'courier new', courier, monospace;">drvIX.exe</span>.</figcaption></figure>
<p>The loading code of the driver looks almost identical to the aforementioned drvIX version. Similarly, <span style="font-family: 'courier new', courier, monospace;">drvIX.exe</span> accepts the PID of the target process the attacker wishes to terminate via the command line.</p>
<p>DrvIX retrieves a device handle to its device object and communicates with the Rentdrv2 driver via <span style="font-family: 'courier new', courier, monospace;">DeviceIoControl</span>. DrvIX then sends the target PID by specifying it in a structure sent as the <span style="font-family: 'courier new', courier, monospace;">Input_Buffer</span> and specifies the control code as <span style="font-family: 'courier new', courier, monospace;">0x22E010</span> (shown in Figure 45).</p>
<figure id="attachment_131102" aria-describedby="caption-attachment-131102" style="width: 887px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131102" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-45.png" alt="Image 45 is the is a screenshot of the code that allows DRVIX to communicate with Rentdrv2." width="887" height="259" /><figcaption id="caption-attachment-131102" class="wp-caption-text">Figure 45. DrvIX communicates with Rentdrv2.</figcaption></figure>
<p>Similarly to the GMER driver, the <span style="font-family: 'courier new', courier, monospace;">0x220E010</span> control code terminates the target process provided by its PID, as shown in Figure 46.</p>
<figure id="attachment_131104" aria-describedby="caption-attachment-131104" style="width: 346px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131104" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-46.png" alt="Image 46 is the is a screenshot of the code that allows Rentdrv2 control code functionality. " width="346" height="138" /><figcaption id="caption-attachment-131104" class="wp-caption-text">Figure 46. Rentdrv2 0x22E010 control code functionality.</figcaption></figure>
<p>The function opens a handle to the target process PID using <span style="font-family: 'courier new', courier, monospace;"><a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/ntddk/nf-ntddk-zwopenprocess" target="_blank" rel="noopener">ZwOpenProcess</a></span> and terminates it by calling <span style="font-family: 'courier new', courier, monospace;"><a href="https://learn.microsoft.com/en-us/windows-hardware/drivers/ddi/ntddk/nf-ntddk-zwterminateprocess" target="_blank" rel="noopener">ZwTerminateProcess</a></span>.</p>
<p>Figure 47 shows this attempt was blocked and prevented by the Cortex XDR platform.</p>
<figure id="attachment_131106" aria-describedby="caption-attachment-131106" style="width: 600px" class="wp-caption aligncenter"><img decoding="async" class="wp-image-131106" src="https://unit42.paloaltonetworks.com/wp-content/uploads/2023/11/word-image-131008-47.png" alt="Image 47 is a screenshot of information from Cortex XDR. It includes the tags, the module, which is Behavioral Threat Protection, the severity, which is high, the description, and the action. Some information is redacted. " width="600" height="397" /><figcaption id="caption-attachment-131106" class="wp-caption-text">Figure 47. Prevention of an attempt to terminate the Cortex XDR service.</figcaption></figure>
<h2><a id="post-131008-_3wblh5hm1ss4"></a>Attribution</h2>
<p>Based on the Unit 42 attribution model, we assess with a high level of confidence that the attacks described in this article were carried out by the Iranian-linked Agonizing Serpens APT group.</p>
<p>This assessment is based on the following reasons and evidence:</p>
<ul>
<li><strong>Multiple code similarities in wipers: </strong>The analysis of the MultiLayer wiper in this article shows multiple code similarities and similar naming convention to previously documented Agonizing Serpens wipers known as <a href="https://www.sentinelone.com/labs/new-version-of-apostle-ransomware-reemerges-in-targeted-attack-on-higher-education/" target="_blank" rel="noopener">Apostle</a>, its successor <a href="https://www.welivesecurity.com/2022/12/07/fantasy-new-agrius-wiper-supply-chain-attack/" target="_blank" rel="noopener">Fantasy</a>, and the IPsec Helper backdoor.</li>
<li><strong>Code similarity in web shells: </strong>The attackers used web shells variants that consisted of the same code, except for variable and function names that are replaced for each sample.</li>
<li><strong>Destructive nature of the attacks: </strong>The final step of the attacks implements a “scorched earth” policy, using custom wipers to render the endpoints unusable and cover the tracks of the attackers. This is congruent with all previous reports about the group’s activity.</li>
<li><strong>Targeting Israeli organizations: </strong>Our telemetry did not detect non-Israeli organizations affected by the attacks. This APT group seems to specifically target Israeli entities.</li>
</ul>
<h2><a id="post-131008-_35isquanjx5r"></a>Conclusion</h2>
<p>In this article we provided a deep dive analysis of a recent destructive wiper attack carried out by the Iranian-linked Agonizing Serpens APT group. This attack is a part of a broader offensive campaign that targets Israeli organizations. Based on our telemetry, the most targeted organizations belong to the education and technology sectors.</p>
<p>Our investigation uncovered new tools in the group’s arsenal that include a set of three previously undocumented wipers, as well as a database extractor tool. Analysis of the new wipers revealed that the group has upgraded their capabilities, putting an emphasis on stealth and evasive techniques designed to bypass security solutions such as EDR technology.</p>
<p>As shown throughout our research, the Cortex XDR platform can detect and prevent the various stages of the attack lifecycle.</p>

	function maskEmailAddress (emailAddress) {
		function mask(str) {
			var strLen = str.length;
			if (strLen > 4) {
				return str.substr(0, 1) + str.substr(1, strLen - 1).replace(/\w/g, '*') + str.substr(-1,1);
			}
			return str.replace(/\w/g, '*');
		}
