This is not the first time &lt;a href=&quot;https://securityintelligence.com/posts/x-force-iris-overcomes-broken-decryption-mechanism-in-jest-ransomware/&quot;&gt;X-Force sees faulty encryption mechanisms&lt;/a&gt; save the day for victimized organizations. Mistakes can easily occur when malware developers use patchwork code and dabble in cryptography without appropriate expertise.

Most organized cybercrime groups do use properly configured encryption, which is almost always impossible to break. That said, the option to examine possibilities can make a difference for victimized organizations and change the course of negotiation and recovery.
&lt;h2&gt;Thanos breeds Hakbit, Prometheus, Haron and more ransomware trouble&lt;/h2&gt;
In early 2020, a new ransomware family dubbed “&lt;a href=&quot;https://exchange.xforce.ibmcloud.com/collection/Thanos-Ransomware-cee6d3536c0773ddd929d320729b1415&quot;&gt;Thanos&lt;/a&gt;” was discovered on sale in underground forums mostly frequented by cybercriminals. At the time, Thanos was advertised as a “Ransomware Affiliate Program,” available for anyone to buy. The malware saw regular updates and new features added over time. A closer look at its code revealed that it was also used at the baseline in ransomware samples that were tracked as “Hakbit” and used in &lt;a href=&quot;https://threatpost.com/hackbit-ransomware-attack-uses-guloader-malicious-microsoft-excel-attachments/156826/&quot;&gt;additional attacks&lt;/a&gt; that targeted organizations in Austria, Switzerland and Germany.

Thanos’ developer equipped it with a bootlocker in mid-2020 and was also using a somewhat novel technique of encrypting files known as “&lt;a href=&quot;https://threatpost.com/thanos-ransomware-weaponize-riplace-tactic/156438/&quot;&gt;RIPlace&lt;/a&gt;,” in which they weaponized research into ransomware evasion techniques based on file characteristics.

In September 2020, Thanos was detected in attacks on &lt;a href=&quot;https://www.saudi24news.com/2020/09/thanos-affects-government-institutions-in-the-middle-east-2.html&quot;&gt;government organizations in MEA&lt;/a&gt;. It presented the victims with a black screen that demanded money to unlock files, and while it had a supposed capability to run a &lt;a href=&quot;https://media.cert.europa.eu/static/MEMO/2020/TLP-WHITE-CERT-EU-THREAT-MEMO-Thanos-ransomware.pdf&quot;&gt;destructive&lt;/a&gt; attack, &lt;a href=&quot;https://securityintelligence.com/news/thanos-ransomware-tries-fails-overwrite-mbr/&quot;&gt;that function did not&lt;/a&gt; work and left MBR intact.

By June 2021, more of Thanos made headlines, only this time as the base code for another ransomware, Prometheus. The latter was used in double-extortion attacks that encrypted files but also stole data and threatened to release it unless a hefty ransom was paid. Prometheus’ operators claimed to be part of the REvil group, they even placed a logo of sorts on their demands for ransom but provided no proof to that effect and may have wanted to use that as a pressure tactic.
&lt;p style=&quot;text-align: center; margin: 0cm 0cm 8pt; line-height: 107%; font-size: 11pt; font-family: Calibri, sans-serif;&quot;&gt;&lt;img src=&quot;https://images-cdn.welcomesoftware.com/Zz1kZTYxYmFlZTM3ZmYxMWVjYWMxYmJlM2QyNmMxNmU0Nw==?token=eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9.eyJzdWIiOlsiZGU2MWJhZWUzN2ZmMTFlY2FjMWJiZTNkMjZjMTZlNDciXSwiZXhwIjoxNjM1NDY2NjI5fQ.GaAwgsaRpoq-h2bfteY7PLcNfiLY_1u1sZmJeJ1ZSho&quot; alt=&quot;ransom-note.png&quot; width=&quot;532&quot; height=&quot;442&quot; /&gt;&lt;/p&gt;
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;em&gt;Figure 1: Prometheus&#039; ransom note&lt;/em&gt;&lt;/p&gt;
While the original Thanos is not as active, its code does not rest. In mid-2021 it was detected in further ransomware attacks, this time used by a group going by the name “&lt;a href=&quot;https://threatpost.com/ransomware-gangs-haron-blackmatter/168212/&quot;&gt;Haron&lt;/a&gt;.”

The Thanos code itself was and is being used by multiple threat actors, some of which were suspected to have nation-state sponsored ties. The Prometheus variant has died out in recent months, but other variations can continue to rise from the same Thanos base. What changes through each variation is customization. In Prometheus’ case, its operators used social engineering well, but were not as adept at working with encryption.
&lt;h2&gt;Breaking Prometheus&lt;/h2&gt;
While working on Prometheus samples that encrypted files on infected devices, IBM Security X-Force researchers uncovered a weakness in the key generation algorithm used in the encryption process. Unlike most ransomware cases, this was good news that ended up helping a victimized organization.

Our analysis showed that to generate the seed for encryption, the algorithm Prometheus selected uses a hardcoded initialization vector (IV) and the uptime of the computer. This means that the seed value is a lot easier to guess than it should be, since certain parameters about the encrypted file and the infected device can be obtained.

Based on such parameters, X-Force wrote a decryptor that ended up working quickly to decrypt file types that had known file headers, for example: pdf, doc, xls, ppt, docx, xlsx, pptx, 7z, mp3, jpg, jpeg, zip, iso, exe, dll, sys and png. Decrypting the files was made even easier when device boot time was known. Boot times are not a parameter one would have to guess, they can be obtained via the CBS.log file in the Windows directory.

Using the decryptor was a great option for the recovery process X-Force supported, but another note is important here. Some open-source decryption tools may emerge over time and might seem like a recovery tool that can help in large-scale cases. One must consider the time it takes a decryptor to unlock each file. Some open-source tools can take around five hours per file, or more, which would be too time consuming in cases where a lot of data is no longer accessible. A reasonable amount of time to decrypt each file should be a few minutes or less.
&lt;h2&gt;Encryption methodology and weaknesses&lt;/h2&gt;
In the Prometheus variants analyzed, there are two ways the ransomware can be configured for encryption:
&lt;h3&gt;Configuration 1&lt;/h3&gt;
Encryption process per file:
&lt;ol&gt;
 	&lt;li&gt;A 32-byte string is generated using C#’s Random class. The default constructor is used, which passes Environment.TickCount as the seed.&lt;/li&gt;
 	&lt;li&gt;The string is then encrypted using a hard-coded RSA public key. PKCS#1 v1.5 padding is used.  The ciphertext is then Base64 encoded.&lt;/li&gt;
 	&lt;li&gt;The file is encrypted using a symmetric algorithm (&lt;a href=&quot;https://en.wikipedia.org/wiki/Salsa20&quot;&gt;Salsa20&lt;/a&gt;) with a hardcoded 8-byte array as the initialization vector (IV).&lt;/li&gt;
 	&lt;li&gt;The key is the 32-byte string described above. The ciphertext is written to the encrypted file.&lt;/li&gt;
 	&lt;li&gt;The encrypted, Base64 encoded key is then appended to the end of the encrypted file, along with the string ‘GotAllDone’.&lt;/li&gt;
&lt;/ol&gt;
&lt;h3&gt;Configuration 2&lt;/h3&gt;
Encryption process per file:
&lt;ol&gt;
 	&lt;li&gt;A 32-byte string is generated using C#’s Random class. The default constructor is used, which passes Environment.TickCount as the seed.&lt;/li&gt;
 	&lt;li&gt;The string is then encrypted using a hard-coded RSA public key. PKCS#1 v1.5 padding is used.  The ciphertext is then Base64 encoded.&lt;/li&gt;
 	&lt;li&gt;&lt;a href=&quot;https://docs.microsoft.com/en-us/dotnet/api/system.security.cryptography.rfc2898derivebytes?view=net-5.0&quot;&gt;RFC2898DeriveBytes&lt;/a&gt; is used to generate a 32-byte key and an 8 byte IV. The Rfc2898DeriveBytes Class implements password-based key derivation functionality, PBKDF2, by using a pseudo-random number generator. The string generated above is used as the password, and the salt is a hardcoded 8-byte array.&lt;/li&gt;
 	&lt;li&gt;The file is encrypted using a symmetric algorithm using the parameters generated above. The ciphertext is written to the encrypted file.&lt;/li&gt;
 	&lt;li&gt;The encrypted, Base64 encoded key is then appended to the end of the encrypted file, along with the string ‘GotAllDone’.&lt;/li&gt;
&lt;/ol&gt;
&lt;h3&gt;Weaknesses in this encryption methodology&lt;/h3&gt;
X-Force found this technique to be lacking in a way that allowed for finding a way to decrypt affected files.

C#’s Random class will generate the exact same bytes as long as the seed is known. In this case, the seed is the Environment.TickCount variable, which is the number of milliseconds elapsed since a computer was last started.

That seed value can be guessed given certain parameters. Moreover, the Environment.TickCount variable is also updated around every 16 milliseconds, so it is possible for multiple files to have the same key, which can make decryption even faster down the line.

The hardcoded IV provided no additional security in this case, considering it can easily be obtained and appears to be the same for every sample analyzed. To make encryption stronger, the IV should typically be random or pseudorandom.
&lt;h2&gt;Decryption requirements and issues&lt;/h2&gt;
Can all Prometheus samples be broken in the same way? X-Force&#039;s analysis indicates that any Prometheus sample that uses the C# Random class to generate keys is vulnerable. Of note, they only decrypted files that were encrypted using a &lt;a href=&quot;https://en.wikipedia.org/wiki/Salsa20&quot;&gt;Salsa20 stream cipher&lt;/a&gt;. Some Prometheus ransomware samples can be configured to use &lt;a href=&quot;https://en.wikipedia.org/wiki/Advanced_Encryption_Standard&quot;&gt;AES-256&lt;/a&gt; and while these samples are still vulnerable, X-Force did not test the decryptor on such in their current work.
&lt;h3&gt;Requirements&lt;/h3&gt;
To decrypt files, we would need the following information:
&lt;ul&gt;
 	&lt;li&gt;The hardcoded IV used in the sample. In all samples observed, the IV was an 8-byte array: 1, 2, 3, 4, 5, 6, 7, 8.&lt;/li&gt;
 	&lt;li&gt;Text or bytes to search for from the decrypted file. X-Force used known file header bytes associated with common file extensions. For example, if the encrypted file’s original extension is .pdf, the text to search for in the file to determine success is “%PDF”.&lt;/li&gt;
&lt;/ul&gt;
&lt;h3&gt;Optional data to use&lt;/h3&gt;
&lt;ul&gt;
 	&lt;li&gt;The configuration of the sample: it’s better to determine which configuration is being used for the encryption. For instance, should RFC2898DeriveBytes be used to obtain the key?&lt;/li&gt;
 	&lt;li&gt;Mtime — the file’s modification time as recorded by the infected device.&lt;/li&gt;
 	&lt;li&gt;The boot time of the infected device. This can be found in the CBS.log file, which is in the Windows directory. This parameter may not have to be exact. Note that Prometheus will not encrypt this file. This file also contains all boot times, meaning it is possible to obtain the boot time even if encryption happened months ago. If the boot time and file modification time are not provided, decryption is still possible but will take significantly longer.&lt;/li&gt;
&lt;/ul&gt;
&lt;h3&gt;File type limitation&lt;/h3&gt;
Currently, only files with known file headers can be decrypted. For example: pdf, doc, xls, ppt, docx, xlsx, pptx, 7z, mp3, jpg, jpeg, zip, iso, exe, dll, sys and png.
&lt;h2&gt;Decryption process overview&lt;/h2&gt;
The following process is what X-Force used in their current work to decrypt data encrypted by Prometheus. It focuses on the malware’s first configuration.
&lt;h3&gt;Decryption in configuration 1 mode&lt;/h3&gt;
&lt;ol&gt;
 	&lt;li&gt;Extract the encrypted text from the file intended for decryption.  This can be done by removing the junk appended to the end of the file.  The amount of junk is equivalent to BASE64_ENCODED_SIZE(RSA_KEY_SIZE) + ‘GotAllDone’.Length.&lt;/li&gt;
 	&lt;li&gt;Attempt to estimate the tick count at the time of encryption.  The tick count begins at zero on system boot, is incremented every millisecond, but only updated every 10-15 milliseconds. Tick count continuously loops after hitting INT_MAX.&lt;/li&gt;
 	&lt;li&gt;Continuously attempt to generate a key and decrypt the ciphertext using the potential seed.&lt;/li&gt;
 	&lt;li&gt;Decrement the potential seed if the plaintext is not correct.  In most cases, the estimated seed should be within 6000 values of the correct seed.  It appears that each file should take around 10 seconds to decrypt, without any optimization considered.&lt;/li&gt;
&lt;/ol&gt;
Note that during any decryption effort, whether custom-built or provided by ransomware actors, certain conditions can affect the accuracy of time estimates of the decryption. If a file takes longer than desired to unlock, it is likely that any other file from that same device will take a similar amount of time.
&lt;h3&gt;Decrypting multiple files from the same infected device&lt;/h3&gt;
If the seed value is found for the first file encrypted, that seed value can be continuously incremented in order to find the values for every other file. This may provide a slightly faster decryption process for computers with hundreds or thousands of files to decrypt.

The decryptor tool can be run against an entire directory of files or on a per file basis.
&lt;h2&gt;A true digital pandemic&lt;/h2&gt;
The ransomware problem has turned into a true pandemic for organizations. Every month new attacks are detected, and new malware families and variations arise in the commercial cybercrime arena and through closed groups. Companies are struggling to prevent ransomware infections on the one hand and prepare for incidents on the other. Paying cybercriminals has also turned into a high-stake negotiation where the leverage is almost always on the attacker’s side.

Will it ever end? With this crime being so rampant in industrialized countries, governments and &lt;a href=&quot;https://www.washingtonpost.com/national-security/ransomware-fbi-revil-decryption-key/2021/09/21/4a9417d0-f15f-11eb-a452-4da5fe48582d_story.html&quot;&gt;law enforcement agencies&lt;/a&gt; are becoming &lt;a href=&quot;https://www.wsj.com/articles/u-s-to-target-crypto-ransomware-payments-with-sanctions-11631885336&quot;&gt;increasingly involved&lt;/a&gt; in ransomware cases, especially in cases where multiple companies are hit.

Stopping attacks is hard because it only takes a small security gap for attackers to find a way in. Response goes a longer way in detecting, containing and helping organizations recover from ransomware attacks. IBM Security X-Force can help. <!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN" "http://www.w3.org/TR/REC-html40/loose.dtd">
<html><body><p>IBM Security <a href="https://www.ibm.com/security/xforce" >X-Force</a> researchers have recently reverse-engineered <a href="https://exchange.xforce.ibmcloud.com/threats/guid:29bbf4a30843c258fc99344faf0d4b09" target="_blank" rel="noopener nofollow" >Prometheus</a> <a href="https://www.ibm.com/topics/ransomware"  target="_blank" rel="noopener nofollow" >ransomware</a> samples as part of ongoing incident response operations. X-Force has found that samples that infected organizational networks featured flawed encryption. This allowed our team to develop a fast-acting decryptor and help customers recover from the attack without a decryption key.</p>
<p>While rare, ransomware developers can make mistakes in the ways they implement <a href="https://www.ibm.com/topics/encryption"  target="_blank" rel="noopener nofollow" >encryption</a>, causing unintended flaws. This is not the first time <a href="https://securityintelligence.com/posts/x-force-iris-overcomes-broken-decryption-mechanism-in-jest-ransomware/" >X-Force sees faulty encryption mechanisms</a> save the day for victimized organizations. Mistakes can easily occur when malware developers use patchwork code and dabble in cryptography without appropriate expertise.</p>
<p>Most organized cybercrime groups do use properly configured encryption, which is almost always impossible to break. That said, the option to examine possibilities can make a difference for victimized organizations and change the course of negotiation and recovery.</p>
<h2>Thanos breeds Hakbit, Prometheus, Haron and more ransomware trouble</h2>
<p>In early 2020, a new ransomware family dubbed &ldquo;<a href="https://exchange.xforce.ibmcloud.com/collection/Thanos-Ransomware-cee6d3536c0773ddd929d320729b1415" target="_blank" rel="noopener nofollow" >Thanos</a>&rdquo; was discovered on sale in underground forums mostly frequented by cybercriminals. At the time, Thanos was advertised as a &ldquo;Ransomware Affiliate Program,&rdquo; available for anyone to buy. The malware saw regular updates and new features added over time. A closer look at its code revealed that it was also used at the baseline in ransomware samples that were tracked as &ldquo;Hakbit&rdquo; and used in <a href="https://threatpost.com/hackbit-ransomware-attack-uses-guloader-malicious-microsoft-excel-attachments/156826/" target="_blank" rel="noopener nofollow" >additional attacks</a> that targeted organizations in Austria, Switzerland and Germany.</p>
<p>Thanos&rsquo; developer equipped it with a bootlocker in mid-2020 and was also using a somewhat novel technique of encrypting files known as &ldquo;<a href="https://threatpost.com/thanos-ransomware-weaponize-riplace-tactic/156438/" target="_blank" rel="noopener nofollow" >RIPlace</a>,&rdquo; in which they weaponized research into ransomware evasion techniques based on file characteristics.</p>
<p>In September 2020, Thanos was detected in attacks on <a href="https://www.saudi24news.com/2020/09/thanos-affects-government-institutions-in-the-middle-east-2.html" target="_blank" rel="noopener nofollow" >government organizations in MEA</a>. It presented the victims with a black screen that demanded money to unlock files, and while it had a supposed capability to run a <a href="https://media.cert.europa.eu/static/MEMO/2020/TLP-WHITE-CERT-EU-THREAT-MEMO-Thanos-ransomware.pdf" target="_blank" rel="noopener nofollow" >destructive</a> attack, <a href="https://securityintelligence.com/news/thanos-ransomware-tries-fails-overwrite-mbr/" >that function did not</a> work and left MBR intact.</p>
<p>By June 2021, more of Thanos made headlines, only this time as the base code for another ransomware, Prometheus. The latter was used in double-extortion attacks that encrypted files but also stole data and threatened to release it unless a hefty ransom was paid. Prometheus&rsquo; operators claimed to be part of the REvil group, they even placed a logo of sorts on their demands for ransom but provided no proof to that effect and may have wanted to use that as a pressure tactic.</p>
<p style="text-align: center; margin: 0cm 0cm 8pt; line-height: 107%; font-size: 11pt; font-family: Calibri, sans-serif;"><amp-img src="https://images-cdn.welcomesoftware.com/Zz1kZTYxYmFlZTM3ZmYxMWVjYWMxYmJlM2QyNmMxNmU0Nw==?token=eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9.eyJzdWIiOlsiZGU2MWJhZWUzN2ZmMTFlY2FjMWJiZTNkMjZjMTZlNDciXSwiZXhwIjoxNjM1NDY2NjI5fQ.GaAwgsaRpoq-h2bfteY7PLcNfiLY_1u1sZmJeJ1ZSho" layout="intrinsic" class="" alt="ransom-note.png" width="532" height="442" lightbox="lightbox"></amp-img></p>
<p style="text-align: center;"><em>Figure 1: Prometheus&rsquo; ransom note</em></p>
<p>While the original Thanos is not as active, its code does not rest. In mid-2021 it was detected in further ransomware attacks, this time used by a group going by the name &ldquo;<a href="https://threatpost.com/ransomware-gangs-haron-blackmatter/168212/" target="_blank" rel="noopener nofollow" >Haron</a>.&rdquo;</p>
<p>The Thanos code itself was and is being used by multiple threat actors, some of which were suspected to have nation-state sponsored ties. The Prometheus variant has died out in recent months, but other variations can continue to rise from the same Thanos base. What changes through each variation is customization. In Prometheus&rsquo; case, its operators used social engineering well, but were not as adept at working with encryption.</p>
<h2>Breaking Prometheus</h2>
<p>While working on Prometheus samples that encrypted files on infected devices, IBM Security X-Force researchers uncovered a weakness in the key generation algorithm used in the encryption process. Unlike most ransomware cases, this was good news that ended up helping a victimized organization.</p>
<p>Our analysis showed that to generate the seed for encryption, the algorithm Prometheus selected uses a hardcoded initialization vector (IV) and the uptime of the computer. This means that the seed value is a lot easier to guess than it should be, since certain parameters about the encrypted file and the infected device can be obtained.</p>
<p>Based on such parameters, X-Force wrote a decryptor that ended up working quickly to decrypt file types that had known file headers, for example: pdf, doc, xls, ppt, docx, xlsx, pptx, 7z, mp3, jpg, jpeg, zip, iso, exe, dll, sys and png. Decrypting the files was made even easier when device boot time was known. Boot times are not a parameter one would have to guess, they can be obtained via the CBS.log file in the Windows directory.</p>
<p>Using the decryptor was a great option for the recovery process X-Force supported, but another note is important here. Some open-source decryption tools may emerge over time and might seem like a recovery tool that can help in large-scale cases. One must consider the time it takes a decryptor to unlock each file. Some open-source tools can take around five hours per file, or more, which would be too time consuming in cases where a lot of data is no longer accessible. A reasonable amount of time to decrypt each file should be a few minutes or less.</p>
<h2>Encryption methodology and weaknesses</h2>
<p>In the Prometheus variants analyzed, there are two ways the ransomware can be configured for encryption:</p>
<h3>Configuration 1</h3>
<p>Encryption process per file:</p>
<ol>
<li>A 32-byte string is generated using C#&rsquo;s Random class. The default constructor is used, which passes Environment.TickCount as the seed.</li>
<li>The string is then encrypted using a hard-coded RSA public key. PKCS#1 v1.5 padding is used.&nbsp; The ciphertext is then Base64 encoded.</li>
<li>The file is encrypted using a symmetric algorithm (<a href="https://en.wikipedia.org/wiki/Salsa20" target="_blank" rel="noopener nofollow" >Salsa20</a>) with a hardcoded 8-byte array as the initialization vector (IV).</li>
<li>The key is the 32-byte string described above. The ciphertext is written to the encrypted file.</li>
<li>The encrypted, Base64 encoded key is then appended to the end of the encrypted file, along with the string &lsquo;GotAllDone&rsquo;.</li>
</ol>
<h3>Configuration 2</h3>
<p>Encryption process per file:</p>
<ol>
<li>A 32-byte string is generated using C#&rsquo;s Random class. The default constructor is used, which passes Environment.TickCount as the seed.</li>
<li>The string is then encrypted using a hard-coded RSA public key. PKCS#1 v1.5 padding is used.&nbsp; The ciphertext is then Base64 encoded.</li>
<li><a href="https://docs.microsoft.com/en-us/dotnet/api/system.security.cryptography.rfc2898derivebytes?view=net-5.0" target="_blank" rel="noopener nofollow" >RFC2898DeriveBytes</a> is used to generate a 32-byte key and an 8 byte IV. The Rfc2898DeriveBytes Class implements password-based key derivation functionality, PBKDF2, by using a pseudo-random number generator. The string generated above is used as the password, and the salt is a hardcoded 8-byte array.</li>
<li>The file is encrypted using a symmetric algorithm using the parameters generated above. The ciphertext is written to the encrypted file.</li>
<li>The encrypted, Base64 encoded key is then appended to the end of the encrypted file, along with the string &lsquo;GotAllDone&rsquo;.</li>
</ol>
<h3>Weaknesses in this encryption methodology</h3>
<p>X-Force found this technique to be lacking in a way that allowed for finding a way to decrypt affected files.</p>
<p>C#&rsquo;s Random class will generate the exact same bytes as long as the seed is known. In this case, the seed is the Environment.TickCount variable, which is the number of milliseconds elapsed since a computer was last started.</p>
<p>That seed value can be guessed given certain parameters. Moreover, the Environment.TickCount variable is also updated around every 16 milliseconds, so it is possible for multiple files to have the same key, which can make decryption even faster down the line.</p>
<p>The hardcoded IV provided no additional security in this case, considering it can easily be obtained and appears to be the same for every sample analyzed. To make encryption stronger, the IV should typically be random or pseudorandom.</p>
<h2>Decryption requirements and issues</h2>
<p>Can all Prometheus samples be broken in the same way? X-Force&rsquo;s analysis indicates that any Prometheus sample that uses the C# Random class to generate keys is vulnerable. Of note, they only decrypted files that were encrypted using a <a href="https://en.wikipedia.org/wiki/Salsa20" target="_blank" rel="noopener nofollow" >Salsa20 stream cipher</a>. Some Prometheus ransomware samples can be configured to use <a href="https://en.wikipedia.org/wiki/Advanced_Encryption_Standard" target="_blank" rel="noopener nofollow" >AES-256</a> and while these samples are still vulnerable, X-Force did not test the decryptor on such in their current work.</p>
<h3>Requirements</h3>
<p>To decrypt files, we would need the following information:</p>
<ul>
<li>The hardcoded IV used in the sample. In all samples observed, the IV was an 8-byte array: 1, 2, 3, 4, 5, 6, 7, 8.</li>
<li>Text or bytes to search for from the decrypted file. X-Force used known file header bytes associated with common file extensions. For example, if the encrypted file&rsquo;s original extension is .pdf, the text to search for in the file to determine success is &ldquo;%PDF&rdquo;.</li>
</ul>
<h3>Optional data to use</h3>
<ul>
<li>The configuration of the sample: it&rsquo;s better to determine which configuration is being used for the encryption. For instance, should RFC2898DeriveBytes be used to obtain the key?</li>
<li>Mtime &mdash; the file&rsquo;s modification time as recorded by the infected device.</li>
<li>The boot time of the infected device. This can be found in the CBS.log file, which is in the Windows directory. This parameter may not have to be exact. Note that Prometheus will not encrypt this file. This file also contains all boot times, meaning it is possible to obtain the boot time even if encryption happened months ago. If the boot time and file modification time are not provided, decryption is still possible but will take significantly longer.</li>
</ul>
<h3>File type limitation</h3>
<p>Currently, only files with known file headers can be decrypted. For example: pdf, doc, xls, ppt, docx, xlsx, pptx, 7z, mp3, jpg, jpeg, zip, iso, exe, dll, sys and png.</p>
<h2>Decryption process overview</h2>
<p>The following process is what X-Force used in their current work to decrypt data encrypted by Prometheus. It focuses on the malware&rsquo;s first configuration.</p>
<h3>Decryption in configuration 1 mode</h3>
<ol>
<li>Extract the encrypted text from the file intended for decryption.&nbsp; This can be done by removing the junk appended to the end of the file.&nbsp; The amount of junk is equivalent to BASE64_ENCODED_SIZE(RSA_KEY_SIZE) + &lsquo;GotAllDone&rsquo;.Length.</li>
<li>Attempt to estimate the tick count at the time of encryption.&nbsp; The tick count begins at zero on system boot, is incremented every millisecond, but only updated every 10-15 milliseconds. Tick count continuously loops after hitting INT_MAX.</li>
<li>Continuously attempt to generate a key and decrypt the ciphertext using the potential seed.</li>
<li>Decrement the potential seed if the plaintext is not correct.&nbsp; In most cases, the estimated seed should be within 6000 values of the correct seed.&nbsp; It appears that each file should take around 10 seconds to decrypt, without any optimization considered.</li>
</ol>
<p>Note that during any decryption effort, whether custom-built or provided by ransomware actors, certain conditions can affect the accuracy of time estimates of the decryption. If a file takes longer than desired to unlock, it is likely that any other file from that same device will take a similar amount of time.</p>
<h3>Decrypting multiple files from the same infected device</h3>
<p>If the seed value is found for the first file encrypted, that seed value can be continuously incremented in order to find the values for every other file. This may provide a slightly faster decryption process for computers with hundreds or thousands of files to decrypt.</p>
<p>The decryptor tool can be run against an entire directory of files or on a per file basis.</p>
<h2>A true digital pandemic</h2>
<p>The ransomware problem has turned into a true pandemic for organizations. Every month new attacks are detected, and new malware families and variations arise in the commercial cybercrime arena and through closed groups. Companies are struggling to prevent ransomware infections on the one hand and prepare for incidents on the other. Paying cybercriminals has also turned into a high-stake negotiation where the leverage is almost always on the attacker&rsquo;s side.</p>
<p>Will it ever end? With this crime being so rampant in industrialized countries, governments and <a href="https://www.washingtonpost.com/national-security/ransomware-fbi-revil-decryption-key/2021/09/21/4a9417d0-f15f-11eb-a452-4da5fe48582d_story.html" target="_blank" rel="noopener nofollow" >law enforcement agencies</a> are becoming <a href="https://www.wsj.com/articles/u-s-to-target-crypto-ransomware-payments-with-sanctions-11631885336" target="_blank" rel="noopener nofollow" >increasingly involved</a> in ransomware cases, especially in cases where multiple companies are hit.</p>
<p>Stopping attacks is hard because it only takes a small security gap for attackers to find a way in. Response goes a longer way in detecting, containing and helping organizations recover from ransomware attacks. IBM Security X-Force can help. For a ransomware readiness and response guide, download the <a href="https://www.ibm.com/downloads/cas/EV6NAQR4"  target="_blank" rel="noopener nofollow" >Definitive Guide to Ransomware here</a>. For any other assistance by IBM&rsquo;s team of experts, explore their incident response and threat intelligence services <a href="https://www.ibm.com/security/services/ibm-x-force-incident-response-and-intelligence"  target="_blank" rel="noopener nofollow" >here</a>.</p>
