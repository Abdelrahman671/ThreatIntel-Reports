    <h2>Inline JavaScript skimmer defense</h2>
<p>The primary defense against inline JavaScript attacks is the use of a CSP policy with the <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/script-src" target="_blank" rel="nofollow noreferrer">script-src</a> directive.  </p>
<h3>Disallow inline JavaScript</h3>
<p>If your web page should not have any inline JavaScript at all, then it is easy to prevent any rogue scripts from being executed by simply specifying:</p>
<pre>
Content-Security-Policy: script-src 'self'
</pre>
<p>This policy would only allow external JavaScript files to be referenced from the same host as the policy. This would prevent the inline JavaScript skimmer code example from executing.  </p>
<h3>Allowing inline JavaScript with CSP nonces</h3>
<p>If your web page has inline JavaScript that can not be refactored and externalized, then there is another CSP script-src mitigation option called <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/Sources" target="_blank" rel="nofollow noreferrer">nonce</a>:</p>
<pre>
'nonce-&lt;base64-value&gt;'
</pre>
<p>A whitelist for specific inline scripts using a cryptographic nonce (number used once). The server must generate a unique nonce value each time it transmits a policy. It is critical to provide an unguessable nonce, as bypassing a resource’s policy is otherwise trivial. See <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/script-src#Unsafe_inline_script" target="_blank" rel="nofollow noreferrer">unsafe inline script</a> for an example. Specifying nonce makes a modern browser ignore 'unsafe-inline' which could still be set for older browsers without nonce support.</p>
<p>The updated CSP policy header could look like this:</p>
<pre>
Content-Security-Policy: script-src 'nonce-1182b04d9fA988B6EcD336EEfbe'
</pre>
<p>This would mean that any JavaScript in the page (both inline and/or externally referenced) would need to include a nonce attribute that matches the one from the CSP script-src sources list:</p>
<pre>
&lt;script nonce=&quot;1182b04d9fA988B6EcD336EEfbe&quot;&gt;
  var something = 1;
&lt;/script&gt;
</pre>
<p>By adding the nonce source to the script-src directive, you could prevent the rogue inline JavaScript skimmer code from executing while still allowing legitimate inline JavaScript code to function.  Firefox would generate a CSP error similar to Figure 4 in the console.</p>

    <p>Although this certainly is an improvement in security, there is one limitation — we are not validating the contents of the JavaScript. <h3>Nonce implementation considerations</h3>
<p>CSP nonce values must be unpredictable and change with every page invocation. <a href="https://csp.withgoogle.com/docs/faq.html#generating-nonces" target="_blank" rel="nofollow noreferrer">Google has some good information on this topic</a>. This scenario is a bit reminiscent of cross-site request forgery mitigations using synchronizer tokens. The main issue organizations need to figure out is how to add these CSP nonce values to the responses. If you are using an application templating system such as <a href="https://developers.google.com/closure/templates/docs/security#content_security_policy" target="_blank" rel="nofollow noreferrer">Closure</a>, you may be able to use it to add in the CSP nonces.  </p>
<h3>Validating inline JavaScript with CSP hashes</h3>
<p>If you want to increase security by validating the contents of inline JavaScript, then you can specify the <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/Sources" target="_blank" rel="nofollow noreferrer">hash</a> option for script-src.</p>
<pre>
'&lt;hash-algorithm&gt;-&lt;base64-value&gt;'
</pre>
<p>A sha256, sha384 or sha512 hash of scripts or styles. The use of this source consists of two portions separated by a dash: the encryption algorithm used to create the hash and the base64-encoded hash of the script or style. When generating the hash, don't include the &lt;script&gt; or &lt;style&gt; tags and note that capitalization and whitespace matter, including leading or trailing whitespace. This is a good first step for blocking these JavaScript skimmer attacks, but it could be possible for an attacker to add a script on a third-party domain that you have allowlisted. In this case, you could tighten up the CSP policy by specifying script sources down to the file level.&lt;/p>\r\n&lt;p>Let’s say that you are using jquery to interact with Google’s APIs. You could update your CSP header’s script-src list to specify the exact URL for the jquery version you are using like this:&lt;/p>\r\n&lt;pre>\r\nContent-Security-Policy: script-src &#39;self&#39; https://ajax.googleapis.com/ajax/libs/jquery/1.8.3/jquery.min.js\r\n&lt;/pre>\r\n&lt;p>This would only allow loading of that exact file over HTTPS versus allowlisting the entire ajax.googleapis.com domain.&lt;/p>\r\n&lt;h3>Allowing external JavaScript with CSP nonces&lt;/h3>\r\n&lt;p>If you don’t want to have to maintain an allowlist of all script-src sources in your CSP header, then you can use nonce values as I covered in the previous section. Remember, the CSP policy header could look like this:&lt;/p>\r\n&lt;pre>\r\nContent-Security-Policy: script-src &#39;nonce-1182b04d9fA988B6EcD336EEfbe&#39;\r\n&lt;/pre>\r\n&lt;p>The externally referenced JavaScript call would need to include a nonce attribute that matches one from the CSP script-src sources list:&lt;/p>\r\n&lt;pre>\r\n&amp;lt;script src=&amp;quot;//ajax.googleapis.com/ajax/libs/jquery/1.8.3/jquery.min.js&amp;quot; nonce=&amp;quot;1182b04d9fA988B6EcD336EEfbe&amp;quot;&amp;gt;&amp;lt;script&amp;gt;\r\n&lt;/pre>\r\n&lt;p>The presence of these nonce attributes that match the CSP header nonce value prevents the rogue third-party JavaScript calls from working.&lt;/p>\r\n&lt;h3>Inheriting trust for External JavaScript with CSP strict-dynamic&lt;/h3>\r\n&lt;p>Anyone who has attempted to allowlist third-party JavaScript sources has undoubtedly run into the issue of those files calling up other files. This cascading effect of interdependencies of files makes managing a CSP allowlist challenging. It is for this reason that CSP v3 introduced the concept of &lt;a href=\&#34;https://www.w3.org/TR/CSP3/#strict-dynamic-usage\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>strict-dynamic&lt;/a>.&amp;nbsp;&amp;nbsp;&lt;/p>\r\n&lt;pre>\r\n&#39;strict-dynamic&#39;\r\n&lt;/pre>\r\n&lt;p>The strict-dynamic source expression specifies that the trust explicitly given to a script present in the markup, by accompanying it with a nonce or a hash, shall be propagated to all the scripts loaded by that root script. At the same time, any whitelist or source expressions such as &#39;self&#39; or &#39;unsafe-inline&#39; will be ignored. See &lt;a href=\&#34;https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/script-src#strict-dynamic\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>script-src&lt;/a> for an example.&lt;/p>\r\n&lt;p>The updated CSP policy header could now look like this:&lt;/p>\r\n&lt;pre>\r\nContent-Security-Policy: script-src &#39;nonce-1182b04d9fA988B6EcD336EEfbe&#39; &#39;strict-dynamic&#39;\r\n&lt;/pre>\r\n&lt;h3>Validating external JavaScript with SRI&lt;/h3>\r\n&lt;p>If you want to validate the actual contents of external/third-party JavaScript files, you would need to utilize a web browser feature called &lt;a href=\&#34;https://developer.mozilla.org/en-US/docs/Web/Security/Subresource_Integrity\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34;>Subresource Integrity (SRI)&lt;/a>. SRI allows you to validate the remote script contents by specifying an integrity attribute to your script calls. Before using the attribute, however, you first need to calculate the proper hash of the file contents. This is similar to the process that must be done with CSP hash usage.&amp;nbsp; Here is an example using cURL and openssl to calculate a sha384 hash:&lt;/p>\r\n&lt;pre>\r\n$ curl -s https://ajax.googleapis.com/ajax/libs/jquery/1.8.3/jquery.min.js | openssl dgst -sha384 -binary | openssl base64 -A\r\n&lt;/pre>\r\n&lt;pre>\r\nlifoBlbdwizTl3Yoe612uhI3AcOam/QtWkozF7SuiACaf5UJl5reOYu4MigVxrCH\r\n&lt;/pre>\r\n&lt;p>Now that you have a proper hash, you can use it in the integrity script attribute like this:&lt;/p>\r\n&lt;pre>\r\n&amp;lt;script src=&amp;quot;https://ajax.googleapis.com/ajax/libs/jquery/1.8.3/jquery.min.js&amp;quot; integrity=&amp;quot;sha384-lifoBlbdwizTl3Yoe612uhI3AcOam/QtWkozF7SuiACaf5UJl5reOYu4MigVxrCH&amp;quot; crossorigin=&amp;quot;anonymous&amp;quot;&amp;gt;&amp;lt;/script&amp;gt;\r\n&lt;/pre>\r\n&lt;p>With SRI now activated for this remote JavaScript resource, if the contents of that file ever change, the file will be blocked by the browser. Let’s take a look at SRI in action and pretend that a Magecart attacker was somehow able to modify that jquery.min.js file to inject their JavaScript skimmer code as shown in the highlighted portion of the Burp proxy screenshot in Figure 11.&lt;/p>\r\n&#34;}}">
    <p>Important note: If you are an Akamai customer, you must also include specific domains in the CSP policy to ensure proper functionality on any Akamai products you might be using. For example, if you are using <a href="/products/mpulse-real-user-monitoring/ev" target="_self">mPulse</a>, you would need to ensure that the proper Akamai domains are listed to support mPulse configuration downloads and data beaconing functionality.  </p>
<p>Once you have identified all authorized third-party script locations, including Akamai mPulse domains, advertising, and analytics networks, you can then modify your CSP header like this:</p>
<pre>
Content-Security-Policy: script-src 'self' c.go-mpulse.net s.go-mpulse.net *.mpstats.us *.akstat.io *.doubleclick.net www.google-analytics.com
</pre>
<p>Now that you have a listing of all the legitimate domains of script sources, CSP could easily block the execution of the example injection we saw in Figure 8 to the billgetstatus.com domain.</p>
<h3>Specify all script-src source URLs</h3>
<p>In the previous example, we allowlisted the authorized domains for third-party JavaScript. This is a good first step for blocking these JavaScript skimmer attacks, but it could be possible for an attacker to add a script on a third-party domain that you have allowlisted. In this case, you could tighten up the CSP policy by specifying script sources down to the file level.</p>
<p>Let’s say that you are using jquery to interact with Google’s APIs. You could update your CSP header’s script-src list to specify the exact URL for the jquery version you are using like this:</p>
<pre>
Content-Security-Policy: script-src 'self' https://ajax.googleapis.com/ajax/libs/jquery/1.8.3/jquery.min.js
</pre>
<p>This would only allow loading of that exact file over HTTPS versus allowlisting the entire ajax.googleapis.com domain.</p>
<h3>Allowing external JavaScript with CSP nonces</h3>
<p>If you don’t want to have to maintain an allowlist of all script-src sources in your CSP header, then you can use nonce values as I covered in the previous section. Remember, the CSP policy header could look like this:</p>
<pre>
Content-Security-Policy: script-src 'nonce-1182b04d9fA988B6EcD336EEfbe'
</pre>
<p>The externally referenced JavaScript call would need to include a nonce attribute that matches one from the CSP script-src sources list:</p>
<pre>
&lt;script src=&quot;//ajax.googleapis.com/ajax/libs/jquery/1.8.3/jquery.min.js&quot; nonce=&quot;1182b04d9fA988B6EcD336EEfbe&quot;&gt;&lt;script&gt;
</pre>
<p>The presence of these nonce attributes that match the CSP header nonce value prevents the rogue third-party JavaScript calls from working.</p>
<h3>Inheriting trust for External JavaScript with CSP strict-dynamic</h3>
<p>Anyone who has attempted to allowlist third-party JavaScript sources has undoubtedly run into the issue of those files calling up other files. This cascading effect of interdependencies of files makes managing a CSP allowlist challenging. It is for this reason that CSP v3 introduced the concept of <a href="https://www.w3.org/TR/CSP3/#strict-dynamic-usage" target="_blank" rel="nofollow noreferrer">strict-dynamic</a>.  </p>
<pre>
'strict-dynamic'
</pre>
<p>The strict-dynamic source expression specifies that the trust explicitly given to a script present in the markup, by accompanying it with a nonce or a hash, shall be propagated to all the scripts loaded by that root script. At the same time, any whitelist or source expressions such as 'self' or 'unsafe-inline' will be ignored. See <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/script-src#strict-dynamic" target="_blank" rel="nofollow noreferrer">script-src</a> for an example.</p>
<p>The updated CSP policy header could now look like this:</p>
<pre>
Content-Security-Policy: script-src 'nonce-1182b04d9fA988B6EcD336EEfbe' 'strict-dynamic'
</pre>
<h3>Validating external JavaScript with SRI</h3>
<p>If you want to validate the actual contents of external/third-party JavaScript files, you would need to utilize a web browser feature called <a href="https://developer.mozilla.org/en-US/docs/Web/Security/Subresource_Integrity" target="_blank" rel="nofollow noreferrer">Subresource Integrity (SRI)</a>. SRI allows you to validate the remote script contents by specifying an integrity attribute to your script calls. Before using the attribute, however, you first need to calculate the proper hash of the file contents. This is similar to the process that must be done with CSP hash usage.  Here is an example using cURL and openssl to calculate a sha384 hash:</p>
<pre>
$ curl -s https://ajax.googleapis.com/ajax/libs/jquery/1.8.3/jquery.min.js | openssl dgst -sha384 -binary | openssl base64 -A
</pre>
<pre>
lifoBlbdwizTl3Yoe612uhI3AcOam/QtWkozF7SuiACaf5UJl5reOYu4MigVxrCH
</pre>
<p>Now that you have a proper hash, you can use it in the integrity script attribute like this:</p>
<pre>
&lt;script src=&quot;https://ajax.googleapis.com/ajax/libs/jquery/1.8.3/jquery.min.js&quot; integrity=&quot;sha384-lifoBlbdwizTl3Yoe612uhI3AcOam/QtWkozF7SuiACaf5UJl5reOYu4MigVxrCH&quot; crossorigin=&quot;anonymous&quot;&gt;&lt;/script&gt;
</pre>
<p>With SRI now activated for this remote JavaScript resource, if the contents of that file ever change, the file will be blocked by the browser. Let’s take a look at SRI in action and pretend that a Magecart attacker was somehow able to modify that jquery.min.js file to inject their JavaScript skimmer code as shown in the highlighted portion of the Burp proxy screenshot in Figure 11.</p>

<h3>Data exfiltration defense</h3>
<p>In the previous sections, I discussed different vectors used by Magecart-type attackers to inject their JavaScript skimmer code into websites. I also covered many different options using CSP and SRI to help mitigate these attacks. But don’t forget about the data exfiltration component of this attack sequence! Getting their JavaScript skimmer code injected into checkout pages doesn’t do any good if the final step of exfiltrating the customer’s sensitive data to the attacker’s site is not successful.</p>
<p>Important note: There is much <a href="https://lists.w3.org/Archives/Public/public-webappsec/2016Sep/0012.html" target="_blank" rel="nofollow noreferrer">debate in the academic and practitioner communities</a> around the effectiveness of CSP to prevent data exfiltration. I am not proposing that CSP can be used as a panacea for all data exfiltration techniques. The approach that follows outlines current Magecart modus operandi tactics for data exfiltration and shows how CSP can be used to limit their effectiveness.  </p>

    <h3>Controlling XHR with CSP <span style="font-family: monospace , monospace;font-size: 13.0px;font-weight: normal;">connect-src</span></h3>
<pre>
<a href="https://www.w3.org/TR/CSP/#directive-connect-src" target="_blank" rel="nofollow noreferrer" style="font-family: adobe-clean , Helvetica , Arial , sans-serif;">CSP v3 connect-src</a><span style="font-family: adobe-clean , Helvetica , Arial , sans-serif;"> can control the following script interface APIs:</span>

</pre>
<pre>
&lt;a&gt; ping
</pre>
<pre>
Fetch
</pre>
<pre>
XMLHttpRequest
</pre>
<pre>
WebSocket
</pre>
<pre>
EventSource
</pre>
<p>If you know that your checkout page should never be using any of these features, then use the following directive: </p>
<pre>
Content-Security-Policy: connect-src 'none' 
</pre>
<p>This would prevent the Magecart code from being able to initiate one of these connection types to send snarfed data off to the attackers.</p>

<h3>CSP violation reporting</h3>
<p>Implementing these CSP plus SRI controls is the top priority for preventing Magecart attacks and protecting your website visitors. A close second priority is receiving CSP violation report telemetry. Organizations that are implementing CSP often use reporting during the initial testing phases, but then will disable it for normal operation. This is unfortunate, as this capability is critical for being able to quickly identify outbreaks of new attacks against your customers. CSP reporting is controlled via the <b>report-uri</b> or <b>report-to</b> (CSP v3) directives.</p>
<p>Akamai customers can use Property Manager to set these CSP headers as discussed previously. Figure 18 shows adding a CSP header with a report-uri directive that points back on-domain so that any violation reports sent by clients will come back through the Akamai platform.</p>

    <p>Notice the different rule options that are available. Keep in mind these rule actions are being applied to CSP violation reports that are being sent from the web client as defined by report-uri or report-to directive in the CSP response header. If you already have CSP reporting configured to send violation telemetry back to origin and you would simply like to additionally log violation report data in Kona Site Defender, you can set the rule to Alert.
