										</nav><!-- <content> --><p>Kerberos, NTLM, and Credential Manager isolate secrets by using Virtualization-based security (VBS). Previous versions of Windows stored secrets in its process memory, in the Local Security Authority (LSA) process <code>lsass.exe</code>.</p>
<section class="row">
<div class="column span2">
<p>With Credential Guard enabled, the LSA process in the operating system talks to a component called the <em>isolated LSA process</em> that stores and protects those secrets, <code>LSAIso.exe</code>. Data stored by the isolated LSA process is protected using VBS and isn't accessible to the rest of the operating system. LSA uses remote procedure calls to communicate with the isolated LSA process.</p>
<p>For security reasons, the isolated LSA process doesn't host any device drivers. Instead, it only hosts a small subset of operating system binaries that are needed for security and nothing else. All the binaries are signed with a certificate that VBS trusts, and the signatures are validated before launching the file in the protected environment.</p>
</div>
<div class="column span2">
<p><a href="images/credential-guard-architecture.png#lightbox" data-linktype="relative-path">
<img src="images/credential-guard-architecture.png" alt="Diagram of the Credential Guard architecture." data-linktype="relative-path">
</a>
</p>
</div>
</section>
<h2 id="credential-guard-protection-limits">Credential Guard protection limits</h2>
<p>Some ways to store credentials aren't protected by Credential Guard, including:</p>
<ul>
<li>When Credential Guard is enabled, NTLMv1, MS-CHAPv2, Digest, and CredSSP can't use the signed-in credentials. Thus, single sign-on doesn't work with these protocols. However, applications can prompt for credentials or use credentials stored in the Windows Vault, which aren't protected by Credential Guard with any of these protocols
<div class="CAUTION">
<p>Caution</p>
<p>It's recommended that valuable credentials, such as the sign-in credentials, aren't used with NTLMv1, MS-CHAPv2, Digest, or CredSSP protocols.</p>
</div>
</li>
<li>Software that manages credentials outside of Windows feature protection</li>
<li>Local accounts and Microsoft Accounts</li>
<li>Credential Guard doesn't protect the Active Directory database running on Windows Server domain controllers. It also doesn't protect credential input pipelines, such as Windows Server running Remote Desktop Gateway. If you're using a Windows Server OS as a client PC, it will get the same protection as it would when running a Windows client OS</li>
<li>Key loggers</li>
<li>Physical attacks</li>
<li>Doesn't prevent an attacker with malware on the PC from using the privileges associated with any credential. We recommend using dedicated PCs for high value accounts, such as IT Pros and users with access to high value assets in your organization</li>
<li>Non-Microsoft security packages</li>
<li>Supplied credentials for NTLM authentication aren't protected. If a user is prompted for and enters credentials for NTLM authentication, these credentials are vulnerable to be read from LSASS memory. These same credentials are vulnerable to key loggers as well</li>
<li>Kerberos service tickets aren't protected by Credential Guard, but the Kerberos Ticket Granting Ticket (TGT) is protected</li>
<li>When Credential Guard is enabled, Kerberos doesn't allow <em>unconstrained Kerberos delegation</em> or <em>DES encryption</em>, not only for signed-in credentials, but also prompted or saved credentials</li>
<li>When Credential Guard is enabled on a VM, it protects secrets from attacks inside the VM. However, it doesn't provide protection from privileged system attacks originating from the host</li>
<li>Windows logon cached password verifiers (commonly called <em>cached credentials</em>) don't qualify as credentials because they can't be presented to another computer for authentication, and can only be used locally to verify credentials. They're stored in the registry on the local computer and provide validation for credentials when a domain-joined computer can't connect to AD DS during user logon. These <em>cached logons</em>, or more specifically, <em>cached domain account information</em>, can be managed using the security policy setting <strong>Interactive logon: Number of previous logons to cache</strong> if a domain controller isn't available</li>
</ul>
