                  <h1 class="title topictitle1" id="GUID-484E0B5F-CF7D-4DD1-ACB5-F92F3D92D8C5__GUID-AD12F8FE-1DC3-4B7C-A70D-B904FFCDCC66">Deploying RSA Keys Within a PKI</h1>
                  <section class="body conbody">
                     
                     <p class="p"> This module explains how to set up and deploy Rivest, Shamir, and Adelman (RSA) keys within a public key infrastructure (PKI).
                        An RSA key pair (a public and a private key) is required before you can obtain a certificate for your router; that is, the
                        end host must generate a pair of RSA keys and exchange the public key with the certification authority (CA) to obtain a certificate
                        and enroll in a PKI.
                        	 
                     </p>

                           <td border="0" class="olh_note">
                              <section class="note__content">
                                 <hr>Security threats, as well as the cryptographic technologies to help protect against them, are constantly changing. For more
                                 information about the latest Cisco cryptographic recommendations, see the Next Generation Encryption (NGE) white paper.

                     <h2 class="title topictitle2" id="GUID-6DCB972D-D3AF-42D2-B0D0-43C5B108C810__GUID-106BAD57-104B-4403-8878-9DD13C8DF1A5">Information About RSA Keys Configuration</h2>
                     <section class="body conbody"></section>

                     <article class="topic concept nested2" aria-labelledby="GUID-9C0C3D7D-250A-4548-BD09-396386917382__GUID-54668CCB-096D-4DDE-A0C5-D47EB44E4EC8" lang="en-US" id="GUID-9C0C3D7D-250A-4548-BD09-396386917382">
                        <h3 class="title topictitle3" id="GUID-9C0C3D7D-250A-4548-BD09-396386917382__GUID-54668CCB-096D-4DDE-A0C5-D47EB44E4EC8">RSA Keys Overview</h3>
                        <section class="body conbody">
                           
                           <p class="p"> An RSA key pair
                              		consists of a public key and a private key. When setting up your PKI, you must
                              		include the public key in the certificate enrollment request. After the
                              		certificate has been granted, the public key will be included in the
                              		certificate so that peers can use it to encrypt data that is sent to the
                              		router. The private key is kept on the router and used both to decrypt the data
                              		sent by peers and to digitally sign transactions when negotiating with peers. 
                              	 
                           </p>
                           
                           <p class="p"> RSA key pairs contain
                              		a key modulus value. The modulus determines the size of the RSA key. The larger
                              		the modulus, the more secure the RSA key. However, keys with large modulus
                              		values take longer to generate, and encryption and decryption operations take
                              		longer with larger keys. 
                              	 
                           </p>
                           
                           
                        </section>
                        <!-- minitoc start -->
                        <nav role="navigation" class="related-links">
                           <ul class="ullinks">
                              <li class="link ulchildlink"><a href="#GUID-1EFE1DF4-2B07-4DEA-AABD-B43E05F92620">Usage RSA Keys Versus General-Purpose RSA Keys</a></li>
                              <li class="link ulchildlink"><a href="#GUID-B112313E-4A61-465D-B4D8-FFA95E9D9E97">How RSA Key Pairs are Associated with a Trustpoint</a></li>
                           </ul>

                        <article class="topic concept nested3" aria-labelledby="GUID-1EFE1DF4-2B07-4DEA-AABD-B43E05F92620__GUID-88687315-AEFB-418D-BB73-FB36258F2076" lang="en-US" id="GUID-1EFE1DF4-2B07-4DEA-AABD-B43E05F92620">
                           <h4 class="title topictitle4" id="GUID-1EFE1DF4-2B07-4DEA-AABD-B43E05F92620__GUID-88687315-AEFB-418D-BB73-FB36258F2076">Usage RSA Keys Versus General-Purpose RSA Keys</h4>
                           <section class="body conbody">
                              <p class="p">
                                 There are two mutually exclusive types of RSA key pairs--usage keys and general-purpose keys. When you generate RSA key pairs
                                 (via the <span><span class="keyword kwd">crypto</span> <span class="keyword kwd">key</span> <span class="keyword kwd">generate</span> <span class="keyword kwd">rsa</span> </span> command), you will be prompted to select either usage keys or general-purpose keys. 
                              </p>
                              <section class="section" id="GUID-1EFE1DF4-2B07-4DEA-AABD-B43E05F92620__GUID-38DB9B5D-1545-46A0-AE3B-6FCF93E5174B">
                                 <h5 class="title sectiontitle">Usage RSA Keys </h5>
                                 
                                 
                                 <p class="p">
                                    Usage keys consist of two RSA key pairs--one RSA key pair is generated and used for encryption and one RSA key pair is generated
                                    and used for signatures. With usage keys, each key is not unnecessarily exposed. (Without usage keys, one key is used for
                                    both authentication methods, increasing the exposure of that key.) 
                                 </p>
                                 
                              </section>
                              <section class="section" id="GUID-1EFE1DF4-2B07-4DEA-AABD-B43E05F92620__GUID-2BD1D000-8376-4152-A028-B42A153BDB79">
                                 <h5 class="title sectiontitle">General-Purpose RSA Keys </h5>
                                 
                                 
                                 <p class="p">
                                    General-purpose keys consist of only one RSA key pair that used for both encryption and signatures. General-purpose key pairs
                                    are used more frequently than usage key pairs.
                                 </p>
                                 
                              </section>
                           </section>
                           <!-- minitoc start -->
                           <nav role="navigation" class="related-links"></nav>

                        <article class="topic concept nested3" aria-labelledby="GUID-B112313E-4A61-465D-B4D8-FFA95E9D9E97__GUID-3C1A779E-FFA4-4793-9112-EF343E253595" lang="en-US" id="GUID-B112313E-4A61-465D-B4D8-FFA95E9D9E97">
                           <h4 class="title topictitle4" id="GUID-B112313E-4A61-465D-B4D8-FFA95E9D9E97__GUID-3C1A779E-FFA4-4793-9112-EF343E253595">How RSA Key Pairs are Associated with a Trustpoint</h4>
                           <section class="body conbody">
                              
                              <p class="p">
                                 A trustpoint, also known as the certificate authority (CA), manages certificate requests and issues certificates to participating
                                 network devices. These services provide centralized key management for the participating devices and are explicitly trusted
                                 by the receiver to validate identities and to create digital certificates. Before any PKI operations can begin, the CA generates
                                 its own public key pair and creates a self-signed CA certificate; thereafter, the CA can sign certificate requests and begin
                                 peer enrollment for the PKI. 
                              </p>
                              
                              <table class="olh_note" border="0" role="note">
                                 <tr>
                                    <td width="1%" class="olh_note" role="heading" border="0"><img src="https://www.cisco.com/content/dam/en/us/td/i/templates/caut.gif"><br><p><b>Caution</b></p>
                                    </td>
                                    <td border="0" class="olh_note">
                                       <section class="note__content">
                                          <hr>Do not manually generate an rsa keypair under trustpoint. If we want to manually generate the keys, generate the key pairs
                                          as usage-keys and not as general-purpose keys. 
                                          <hr>
                                       </section>
                                    </td>
                                 </tr>
                              </table>
                              
                              <table class="olh_note" border="0" role="note">
                                 <tr>
                                    <td width="1%" class="olh_note" role="heading" border="0"><img src="https://www.cisco.com/content/dam/en/us/td/i/templates/caut.gif"><br><p><b>Caution</b></p>
                                    </td>
                                    <td border="0" class="olh_note">
                                       <section class="note__content">
                                          <hr>Certificate renewal with regenerate option does not work with key label starting from zero ('0'), (for example, '0test').
                                          CLI allows configuring such name under trustpoint, and allows hostname starting from zero. When configuring <span><span class="keyword kwd">rsakeypair </span> </span>
                                          <var class="keyword varname">name</var> under a trustpoint, do not configure the name starting from zero. When keypair name is not configured and the default keypair
                                          is used, make sure the router hostname does not start from zero. If it does so, configure "<span><span class="keyword kwd">rsakeypair</span> </span>
                                          <var class="keyword varname">name</var> explicitly under the trustpoint with a different name.

                        <h3 class="title topictitle3" id="GUID-085E73F2-F0DF-4D88-8700-FE5F2F9E56E0__GUID-8D4D02CA-37CC-4EDB-87EC-3E30E95F3BEF">Reasons to Store Multiple RSA Keys on a Router</h3>
                        <section class="body conbody">
                           <p class="p"> Configuring multiple RSA key pairs allows the Cisco IOS software to maintain a different key pair for each CA with which
                              it is dealing or the software can maintain multiple key pairs and certificates with the same CA. As a result, the Cisco IOS
                              software can match policy requirements for each CA without compromising the requirements specified by the other CAs, such
                              as key length, key lifetime, and general-purpose versus usage keys. 
                              	 
                           </p>
                           <p class="p"> Named key pairs (which are specified via the 
                              		<span><span class="keyword kwd">label</span> </span> 
                              		<span><var>key-label</var> </span> option) allow you to have multiple RSA key pairs, enabling the Cisco IOS software to maintain a different key pair for each
                              identity certificate. 
                              	 
                           </p>
                        </section>

                        <h3 class="title topictitle3" id="GUID-1CB802D8-9DE3-447F-BECE-CF22F5E11436__GUID-01AA2D19-3E08-4617-8DC5-C057C3423338">Benefits of Exportable RSA
                           	 Keys
                        </h3>
                        <section class="body conbody">
                           
                           <table class="olh_note" border="0" role="note">
                              <tr>
                                 <td width="1%" class="olh_note" role="heading" border="0"><img src="https://www.cisco.com/content/dam/en/us/td/i/templates/caut.gif"><br><p><b>Caution</b></p>
                                 </td>
                                 <td border="0" class="olh_note">
                                    <section class="note__content">
                                       <hr> 
                                       		
                                       <p class="p"> Exportable RSA keys
                                          		  should be carefully evaluated before use because using exportable RSA keys
                                          		  introduces the risk that these keys might be exposed. Any existing RSA keys are
                                          		  not exportable. New keys are generated as nonexportable by default. It is not
                                          		  possible to convert an existing nonexportable key to an exportable key. 
                                          		
                                       </p> 
                                       	 
                                       <hr>
                                    </section>
                                 </td>
                              </tr>
                           </table>
                           
                           <p class="p"> As of Cisco IOS Release 12.2(15)T, users can share the private RSA key pair of a router with standby routers, therefore transferring
                              the security credentials between networking devices. The key pair that is shared between two routers will allow one router
                              to immediately and transparently take over the functionality of the other router. If the main router were to fail, the standby
                              router could be dropped into the network to replace the failed router without the need to regenerate keys, reenroll with the
                              CA, or manually redistribute keys. 
                           </p>
                           
                           <p class="p"> Exporting and
                              		importing an RSA key pair also enables users to place the same RSA key pair on
                              		multiple routers so that all management stations using Secure Shell (SSH) can
                              		be configured with a single public RSA key. 
                              	 
                           </p>
                           
                           <section class="section" id="GUID-1CB802D8-9DE3-447F-BECE-CF22F5E11436__GUID-938E3DE7-60AD-4544-A447-B025F07540C9">
                              <h4 class="title sectiontitle">Exportable RSA Keys in
                                 		  PEM-Formatted Files
                              </h4> 
                              		 
                              		
                              <p class="p"> Using
                                 		  privacy-enhanced mail (PEM)-formatted files to import or export RSA keys can be
                                 		  helpful for customers who are running Cisco IOS software Release 12.3(4)T or
                                 		  later and who are using secure socket layer (SSL) or secure shell (SSH)
                                 		  applications to manually generate RSA key pairs and import the keys back into
                                 		  their PKI applications. PEM-formatted files allow customers to directly use
                                 		  existing RSA key pairs on their Cisco IOS routers instead of generating new
                                 		  keys. 
                                 		
                              </p> 
                              	 
                           </section>
                           
                        </section>

                        <h3 class="title topictitle3" id="GUID-236E9CC1-4A95-41AC-9A06-0D930A5AA1FB__GUID-52ACC7ED-67A2-45A0-9471-FD12480B9DD9">Passphrase Protection While Importing and Exporting RSA Keys</h3>
                        <section class="body conbody">
                           
                           <p class="p"> You have to include a passphrase to encrypt the PKCS12 file or the PEM file that will be exported, and when the PKCS12 or
                              PEM file is imported, the same passphrase has to be entered to decrypt it. Encrypting the PKCS12 or PEM file when it is being
                              exported, deleted, or imported protects the file from unauthorized access and use while it is being transported or stored
                              on an external device. 
                              	 
                           </p>
                           
                           <p class="p"> The passphrase can be any phrase that is at least eight characters in length; it can include spaces and punctuation, excluding
                              the question mark (?), which has special meaning to the Cisco IOS parser. 
                              	 
                           </p>
                           
                           <section class="section" id="GUID-236E9CC1-4A95-41AC-9A06-0D930A5AA1FB__GUID-D7922683-CB5A-4EB2-8113-1EBEAB2B4965">
                              <h4 class="title sectiontitle">How to Convert an Exportable RSA Key Pair to a Nonexportable RSA Key Pair</h4> 
                              		 
                              		
                              <p class="p"> Passphrase protection protects the external PKCS12 or PEM file from unauthorized access and use. To prevent an RSA key pair
                                 from being exported, it must be labeled “nonexportable.” To convert an exportable RSA key pair into a nonexportable key pair,
                                 the key pair must be exported and then reimported without specifying the “exportable” keyword. 
                                 		
                              </p> 
                              	 
                           </section>

                        <h3 class="title topictitle3" id="GUID-C3635CB0-BBC3-4CF1-9ABF-838E720E8540__GUID-3733FA6B-6488-4BFC-8858-FE97F9835999">Exporting and Importing RSA Keys</h3>
                        <section class="body conbody">
                           
                           <p class="p"> This section contains the following tasks that can be used for exporting and importing RSA keys. Whether you are using PKCS12
                              files or PEM files, exportable RSA keys allow you to use existing RSA keys on Cisco IOS routers instead of having to generate
                              new RSA keys if the main router were to fail. 
                              	 
                           </p>
                           
                        </section>

                        <article class="topic task nested3" aria-labelledby="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6__GUID-DE582041-8C4B-4868-9F84-F558581CF50A" lang="en-US" id="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6">
                           <h4 class="title topictitle4" id="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6__GUID-DE582041-8C4B-4868-9F84-F558581CF50A">Exporting and Importing RSA Keys in PKCS12 Files</h4>
                           <section class="body taskbody">
                              <section class="section context" id="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6__GUID-A2E4B740-10E5-4458-AC37-A101BB4D4DA8"> 
                                 		
                                 <p class="p"> Exporting and importing RSA key pairs enables users to transfer security credentials between devices. The key pair that is
                                    shared between two devices allows one device to immediately and transparently take over the functionality of the other router.
                                    
                                    		
                                 </p> 
                                 	 
                              </section>
                              <section class="section prereq p" id="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6__GUID-CCD984DB-B497-4C37-9C44-5C74B487E593">
                                 <section class="tasklabel">
                                    <h5 font-weight="bold" class="sectiontitle tasklabel cB_Bold">Before you begin</h5>
                                 </section> 
                                 		
                                 <p class="p">You must generate an RSA key pair and mark it “exportable” as specified in the “Generating an RSA Key Pair” task. </p> 
                                 		
                                 <table class="olh_note" border="0" role="note">
                                    <tr>
                                       <td width="1%" class="olh_note" role="heading" border="0"><img src="https://www.cisco.com/content/dam/en/us/td/i/templates/note.gif"><br><p><b>Note</b></p>
                                       </td>
                                       <td border="0" class="olh_note">
                                          <section class="note__content">
                                             <hr>
                                             				
                                             <ul class="ul" id="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6__GUID-211B845A-67E1-44CA-B37C-4EB158950C85">
                                                <li class="li" id="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6__LI_A2F3AABE87F5411EACC31BB622EB4D4B">
                                                   						
                                                   <p class="p">You cannot export RSA keys that existed on the router before your system was upgraded to Cisco IOS Release 12.2(15)T or later.
                                                      You have to generate new RSA keys and label them as “exportable” after you upgrade the Cisco IOS software. 
                                                   </p>
                                                   					
                                                </li>
                                                <li class="li" id="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6__LI_1306676778234C5F9D593D95590F34EE">
                                                   						
                                                   <p class="p">When you import a PKCS12 file that was generated by a third-party application, the PKCS12 file must include a CA certificate.
                                                      
                                                   </p>
                                                   					
                                                </li>
                                                <li class="li" id="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6__LI_AD9BF55FBF074E7F8F7B44E9E6AF09DE">
                                                   						
                                                   <p class="p">If you want reexport an RSA key pair after you have already exported the key pair and imported them to a target router, you
                                                      must specify the <span><span class="keyword kwd">exportable</span> </span> keyword when you are importing the RSA key pair. 
                                                   </p>
                                                   					
                                                </li>
                                                <li class="li" id="GUID-44796FDC-72A0-4240-895A-CCA9DB62CAE6__LI_C613862563FE40A5AE4FB10875F245B3">
                                                   						
                                                   <p class="p">The largest RSA key a router may import is 2048-bits. </p>
                                                   					
                                                </li>
                                             </ul>
                                             			
                                             <hr>
                                          </section>
                                       </td>
                                    </tr>
                                 </table> 
                                 	 
                              </section>

                        <h3 class="title topictitle3" id="GUID-851F4A9A-1AA3-418A-B4B5-D221D2D37CF5__GUID-8140C232-A19C-4362-B090-AF26ADAD0BBE">Encrypting and Locking Private Keys on a Router</h3>
                        <section class="body taskbody">
                           <section class="section context" id="GUID-851F4A9A-1AA3-418A-B4B5-D221D2D37CF5__GUID-14CCFC41-8018-4A0D-8B4B-F8A3057CCF39">
                              
                              <p class="p">
                                 Digital signatures are used to authenticate one device to another device. To use digital signatures, private information (the
                                 private key) must be stored on the device that is providing the signature. The stored private information may aid an attacker
                                 who steals the hardware device that contains the private key; for example, a thief might be able to use the stolen router
                                 to initiate a secure connection to another site by using the RSA private keys stored in the router. 
                              </p>
                              
                              <table class="olh_note" border="0" role="note">
                                 <tr>
                                    <td width="1%" class="olh_note" role="heading" border="0"><img src="https://www.cisco.com/content/dam/en/us/td/i/templates/note.gif"><br><p><b>Note</b></p>
                                    </td>
                                    <td border="0" class="olh_note">
                                       <section class="note__content">
                                          <hr>
                                          <p class="p">
                                             RSA keys are lost during password recovery operations. If you lose your password, the RSA keys will be deleted when you perform
                                             the password recovery operation. (This function prevents an attacker from performing password recovery and then using the
                                             keys.)
                                          </p>
                                          <hr>
                                       </section>
                                    </td>
                                 </tr>
                              </table>
                              
                              <p class="p">
                                 To protect the private RSA key from an attacker, a user can encrypt the private key that is stored in NVRAM via a passphrase.
                                 Users can also “lock” the private key, which blocks new connection attempts from a running router and protects the key in
                                 the router if the router is stolen by an attempted attacker. 
                              </p>
                              
                              <p class="p">
                                 Perform this task to encrypt and lock the private key that is saved to NVRAM.
                              </p>
                              
                              <table class="olh_note" border="0" role="note">
                                 <tr>
                                    <td width="1%" class="olh_note" role="heading" border="0"><img src="https://www.cisco.com/content/dam/en/us/td/i/templates/note.gif"><br><p><b>Note</b></p>
                                    </td>
                                    <td border="0" class="olh_note">
                                       <section class="note__content">
                                          <hr>
                                          <p class="p">
                                             The RSA keys must be unlocked while enrolling the CA. The keys can be locked while authenticating the router with the CA because
                                             the private key of the router is not used during authentication. 
                                          </p>
                                          <hr>
                                       </section>
                                    </td>
                                 </tr>
                              </table>
                              
                           </section>
                           <section class="section prereq p" id="GUID-851F4A9A-1AA3-418A-B4B5-D221D2D37CF5__GUID-B0958F58-3DBD-42D5-A542-D4CFFE4F8447">
                              <section class="tasklabel">
                                 <h4 font-weight="bold" class="sectiontitle tasklabel cB_Bold">Before you begin</h4>
                              </section>
                              
                              <p class="p">
                                 Before encrypting or locking a private key, you should perform the following tasks:
                              </p>
                              
                              <ul class="ul" id="GUID-851F4A9A-1AA3-418A-B4B5-D221D2D37CF5__GUID-C129895F-2AC0-424E-9D12-FF69E75D279A">
                                 <li class="li">
                                    <p class="p"> Generate an RSA key pair as shown in Generating an RSA Key Pair section.</p>
                                 </li>
                                 <li class="li">
                                    <p class="p">
                                       Optionally, you can authenticate and enroll each router with the CA server. 
                                    </p>
                                 </li>
                              </ul>
                              
                              <table class="olh_note" border="0" role="note">
                                 <tr>
                                    <td width="1%" class="olh_note" role="heading" border="0"><img src="https://www.cisco.com/content/dam/en/us/td/i/templates/note.gif"><br><p><b>Note</b></p>
                                    </td>
                                    <td border="0" class="olh_note">
                                       <section class="note__content">
                                          <hr>
                                          
                                          <p class="p">
                                             <strong class="ph b">
                                                Backward Compatibility Restriction </strong>
                                             
                                          </p>
                                          
                                          <p class="p">
                                             Any image prior to Cisco IOS Release 12.3(7)T does not support encrypted keys. To prevent your router from losing all encrypted
                                             keys, ensure that only unencrypted keys are written to NVRAM before booting an image prior to Cisco IOS Release 12.3(7)T.
                                             
                                          </p>
                                          
                                          <p class="p">
                                             If you must download an image prior to Cisco IOS Release 12.3(7)T, decrypt the key and immediately save the configuration
                                             so the downloaded image does not overwrite the configuration. 
                                          </p>
                                          
                                          <p class="p">
                                             <strong class="ph b">
                                                Interaction with Applications </strong>
                                             
                                          </p>
                                          
                                          <p class="p">
                                             An encrypted key is not effective after the router boots up until you manually unlock the key (via the <span><span class="keyword kwd">crypto</span> <span class="keyword kwd">key</span> <span class="keyword kwd">unlock</span> <span class="keyword kwd">rsa</span> </span> command). Depending on which key pairs are encrypted, this functionality may adversely affect applications such as IP security
                                             (IPsec), SSH, and SSL; that is, management of the router over a secure channel may not be possible until the necessary key
                                             pair is unlocked.
                                          </p>
                                          &gt;
                                          <hr>
                                       </section>
                                    </td>
                                 </tr>
                              </table>

                        <h3 class="title topictitle3" id="GUID-E9E39D7C-7A84-4D93-82D7-D8DB7CFD9D3F__GUID-B568EFAE-E864-43C1-88D4-E80ECD7D40E4">Removing RSA Key Pair Settings</h3>
                        <section class="body taskbody">
                           <section class="section context" id="GUID-E9E39D7C-7A84-4D93-82D7-D8DB7CFD9D3F__GUID-31708F45-2ADD-446B-B8B5-FF62943A4979">
                              
                              <p class="p">
                                 An RSA key pair may need to be removed for one of the following reasons:
                              </p>
                              
                              <ul class="ul" id="GUID-E9E39D7C-7A84-4D93-82D7-D8DB7CFD9D3F__GUID-414C7F53-E573-486D-8D9F-B85569D5C45D">
                                 <li class="li">
                                    <p class="p">
                                       During manual PKI operations and maintenance, old RSA keys can be removed and replaced with new keys. 
                                    </p>
                                 </li>
                                 <li class="li">
                                    <p class="p">
                                       An existing CA is replaced and the new CA requires newly generated keys; for example, the required key size might have changed
                                       in an organization so you would have to delete the old 1024-bit keys and generate new 2048-bit keys.
                                    </p>
                                 </li>
                                 <li class="li">
                                    <p class="p">
                                       <span>
                                          <span class="keyword kwd">T</span>
                                           </span>he peer router's public keys can be deleted in order to help debug signature verification problems in IKEv1 and IKEv2. Keys
                                       are cached by default with the lifetime of the certificate revocation list (CRL) associated with the trustpoint.
                                    </p>
                                 </li>
                              </ul>
                              
                              <p class="p">
                                 Perform this task to remove all RSA keys or the specified RSA key pair that has been generated by your router.
                              </p>
                              
                           </section>
