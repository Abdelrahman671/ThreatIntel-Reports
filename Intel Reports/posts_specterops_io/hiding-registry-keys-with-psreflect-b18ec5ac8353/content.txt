and Background</h1><p id="d3e1" class="pw-post-body-paragraph nu nv gu nw b nx ny nz oa ob oc od oe of og oh oi oj ok ol om on oo op oq or gn bk">Recently, I wanted to test detection of different kinds of registry persistence used by malware and APT groups. The Windows registry is a particularly interesting area for blue team detection as “fileless” techniques become more prevalent. One technique that has stuck in my mind is a persistence trick used by the Kovter malware family as detailed in a <a class="af os" href="https://www.symantec.com/connect/blogs/kovter-malware-learns-poweliks-persistent-fileless-registry-update" rel="noopener ugc nofollow" target="_blank">September 2015 report from Symantec,</a> and analyzed by <a class="af os" href="https://blog.malwarebytes.com/threat-analysis/2016/07/untangling-kovter/" rel="noopener ugc nofollow" target="_blank">MalwareBytes</a>, <a class="af os" href="http://blog.airbuscybersecurity.com/post/2016/03/FILELESS-MALWARE-%E2%80%93-A-BEHAVIOURAL-ANALYSIS-OF-KOVTER-PERSISTENCE" rel="noopener ugc nofollow" target="_blank">Airbus Cybersecurity</a>, and <a class="af os" href="https://reaqta.com/2015/09/poweliks-file-less-malware-keeps-evolving/" rel="noopener ugc nofollow" target="_blank">Reaqta</a>.</p><p id="d880" class="pw-post-body-paragraph nu nv gu nw b nx ot nz oa ob ou od oe of ov oh oi oj ow ol om on ox op oq or gn bk">Kovter and its predecessor Poweliks use mshta to execute code stored in registry keys and values. To persist between reboots, Kovter uses a Run key value, but with a small twist: the key value name starts with a null character (\0), followed by random chars. The null character causes an error when attempting to read the value with Regedit and other techniques that expect a null-terminated string. Using a null-character in a value name to hide from Regedit has been known since at least 2005, and Mark Russinovich previously released a tool called <a class="af os" href="https://technet.microsoft.com/en-us/sysinternals/reghide.aspx" rel="noopener ugc nofollow" target="_blank">RegHide</a> as part of the Sysinternals Suite as a proof of concept.</p><p id="cd66" class="pw-post-body-paragraph nu nv gu nw b nx ot nz oa ob ou od oe of ov oh oi oj ow ol om on ox op oq or gn bk">The <a class="af os" href="https://web.archive.org/web/20091104223505/http://technet.microsoft.com/en-us/sysinternals/bb897446.aspx" rel="noopener ugc nofollow" target="_blank">old Sysinternals page</a> described why this null character trick worked:</p><blockquote class="oy oz pa"><p id="053e" class="nu nv pb nw b nx ot nz oa ob ou od oe of ov oh oi oj ow ol om on ox op oq or gn bk">“In the Win32 API strings are interpreted as NULL-terminated ANSI (8-bit) or wide character (16-bit) strings. In the Native API names are counted Unicode (16-bit) strings. While this distinction is usually not important, it leaves open an interesting situation: there is a class of names that can be referenced using the Native API, but that cannot be described using the Win32 API. […] When a key (or any other object with a name such as a named Event, Semaphore or Mutex) is created with such a name any applications using the Win32 API will be unable to open the name, even though they might seem to see it.”</p></blockquote><p id="86a7" class="pw-post-body-paragraph nu nv gu nw b nx ot nz oa ob ou od oe of ov oh oi oj ow ol om on ox op oq or gn bk">A question from StackOverflow
