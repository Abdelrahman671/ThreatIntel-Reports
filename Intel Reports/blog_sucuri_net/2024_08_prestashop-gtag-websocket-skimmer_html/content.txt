If a website is identified as a potential source to steal and sell credit card details on the black market then you can be sure that the website will be a target.</p><p>In this case our client was experiencing an antivirus warning when accessing their website (a common symptom of Magecart / card stealing malware). </p><p>But the real question was: How was it being placed there? Searching through the files and database for strings such as <strong>WebSocket</strong> turned up nothing, so this was a little trickier to locate than usual. In this post, we&#8217;ll review web sockets, skimmer malware, and how we were able to locate the source of this card stealer infection.</p><h2>What is a web socket?</h2><p>Before we get into the investigation let&#8217;s first quickly review web sockets, what they are, and why attackers use them in carding attacks. A previous <a href="https://blog.sucuri.net/2023/11/skimming-credit-cards-with-websockets.html">post</a> on this blog goes into more detail if you&#8217;d like to read more!</p><p>Essentially, a web socket is a direct connection between a client and a server, with continuous traffic flow being established in both directions. Unlike a basic http/https connection where there is a request from the client, followed by a response from the server, web socket traffic is continuous and goes in both directions at the same time:</p><p><picture loading="lazy" decoding="async" class="aligncenter">
<source type="image/webp" srcset="https://blog.sucuri.net/wp-content/uploads/2024/08/websocket-connection.jpg.webp"/>
<img loading="lazy" decoding="async" src="https://blog.sucuri.net/wp-content/uploads/2024/08/websocket-connection.jpg" alt="WebSocket Connection" width="624" height="313"/>
</picture>
</p><p>This is useful for services such as finance and online gaming which require a lot of real-time conversation between clients and servers, where a clunky old HTTP connection would not quite suffice.</p><p>Attackers have also adopted the use of web sockets for the purpose of obfuscation. Analysing web socket traffic is more challenging than regular HTTP/HTTPS traffic, and not all analysis tools even support it at all. Specialised tools and know-how are required to dig into this sort of traffic. They&#8217;re also fit for the purpose of pilfering credit card details from infected checkout pages as we will see here.</p><h2>The investigation</h2><p>As I mentioned previously, searching for strings of what we saw in the view-source turned up nothing. Many credit card skimmers can easily be found in an infected database simply by searching for things like <strong>&lt;script</strong> or other portions of the plain-text injection.</p><p><strong>Base64</strong> encoded strings are also frequently used by attackers, but even querying for <strong>PHNjcmlw</strong> (<strong>&lt;scrip</strong> in <strong>base64</strong> encoded format) left us empty handed. Taking it a step further, thinking maybe the payload was reversed using <a href="https://www.php.net/manual/en/function.strrev.php">strrev</a>, searching for <strong>wlmcjNHP</strong> (the same string but backwards) also turned up nothing.</p><p>So, we went back to the basics. Credit card skimming malware can&#8217;t be injected just anywhere on a website into random files, it needs to load on the checkout page. For this reason, skimmers are oftentimes injected into core CMS files, theme files, certain areas of the database, or &#8211; in the case of WooCommerce WordPress websites &#8211; malicious plugins.</p><p>In this case, checking the theme files and extensions did yield a webshell:</p><p><picture loading="lazy" decoding="async">
