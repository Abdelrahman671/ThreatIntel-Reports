function cleanText(input) {
    //aligns all new lines and splits text into single paragraphs
    let paragraphs = input.replace(/\r\n|\n|\r/gm, '\n');
    paragraphs = paragraphs.split('\n');
    // clears array of empty elements (double new lines)
    paragraphs = $.grep(paragraphs, function (n) { return (n); });
    // clears text of all double whitespaces
    for (let p = 0; p < paragraphs.length; p++) {
        paragraphs[p] = paragraphs[p].replace(/\s+/g, ' ');
        paragraphs[p] = $.trim(paragraphs[p]);
    }
    // joins everything to one string
    var cleanedText = paragraphs.filter((str) => str !== '');
    // gets an array of single words

    return cleanedText[0];
}

                <div class="rich-text__tag-container">
                    <h4 class="rich-text__tag-container-heading">Topics in this article</h4>
                    <div class="rich-text__tag-inner-container">
                                <div class="rich-text__tag">
                                    <a class="rich-text__tag-text" href="https://services.global.ntt/en-us/insights/blog?Facets=Security#Blog">Security</a>
                                </div>
                    </div>
                </div>
            Emotet is a threat known to use large amounts of command and control servers (C2s) in parallel in order to ensure uptime and bypass blocking.<br />
<p>This first layer of C2s, also called Tier 1 C2s, will in part forward their received traffic to Tier 2 servers. This relationship has previously been observed by Centurylink<sup>1</sup>.</p>
<br />
<p>NTT Ltd. Threat Detection decided to explore the malware infrastructure of Emotet deeper, with multiple goals in mind:</p>
<br />
<ul><br />
    <li>ensure our long-term detection capabilities for Emotet traffic in our customer environments</li>
    <br />
    <li>explore the underlying Emotet infrastructure and track the setup and longevity of the Tier 2 servers </li>
    <br />
    <li>collaborate with CERTs and notify ISPs of malicious activity</li>
    <br />
</ul>
<br />
<p>We own and operates one of the world's largest tier-1 IP backbones, giving insight into a significant portion of the global internet traffic. We&rsquo;re also consistently ranked among the top five network providers in the world. In the fall of 2018, our security division added botnet infrastructure detection capabilities to our Managed Security Services (MSS) Threat Detection services. This unique capability was use during this project in order to explore and monitor the Emotet network infrastructure.</p>
<br />
<h3>Starting the journey</h3>
<br />
<p>The assumptions of behaviour of the Tier 1 C2s were reduced to the expectation that they forward their traffic towards a Tier 2 C2. We initially monitored Tier 1 C2 traffic en masse. We extracted Tier 1 C2 lists from Emotet samples and applied these IOCs in our netflow monitoring in NTT Ltd.&rsquo;s global internet infrastructure, where one potential Tier 2 C2 was found:</p>
<br />
<p><img alt="A technical diagram" src="-/media/df414e63575a433b9c55d72232b36cb9.ashx?h=auto" style="max-width: 100%;height: auto;" /></p>
<br />
<p>The Emotet botnet is divided into separate Epochs, Epoch 1, 2 and 3, which all have their own separate Tier 1 C2 infrastructure <sup>2</sup>. The Tier 1 C2s in the picture above belong to Epoch 2.</p>
<br />
<p>A Shodan lookup of the services of the Tier 2 C2 shows the following setup of HTTP services:</p>
<br />
<p><img alt="A technical diagram" src="-/media/a4a8f6bc2fb342cbbdf3a195438493e8.ashx?h=auto" style="max-width: 100%;height: auto;" /></p>
<br />
<p>At this point in the investigation we assumed that the above was the standard setup for all Tier 2 C2s and automated the analysis of outgoing traffic from Tier 1 C2s to find new Tier 2 C2s. We found new ones by selecting traffic where:</p>
<br />
<ul><br />
    <li>the source IP is an Emotet Tier 1 C2</li>
    <br />
    <li>the destination port is TCP/80 and is running a NGINX service, with the error message &ldquo;HTTP 404 Not found&rdquo;</li>
    <br />
    <li>the destination IP is also running an Apache service which gives &ldquo;HTTP 200 OK&rdquo; status message over port TCP/8080</li>
    <br />
</ul>
<br />
<h3>Observed backends</h3>
<br />
<p>With our new workflow established we&rsquo;ve observed 16 Tier 2 C2 servers with incoming Tier 1 C2 traffic of their respective associated Epoch:</p>
<br />
<ul><br />
    <li>eight Tier 2 C2s belong to Epoch 1</li>
    <br />
    <li>five Tier 2 C2s belong to Epoch 2</li>
    <br />
    <li>three Tier 2 C2s belong to Epoch 3</li>
    <br />
</ul>
<br />
<p>With the following hosting providers used:</p>
<br />
<ul><br />
    <li>seven using <strong>Serverius Holding</strong></li>
    <br />
    <li>five using <strong>GloboTech</strong></li>
    <br />
    <li>two using <strong>Worldstream</strong></li>
    <br />
    <li>one using <strong>PnS Hostings</strong></li>
    <br />
    <li>one using <strong>RealHosters</strong></li>
    <br />
</ul>
<br />
<p>The graphs below show the timelines for when each Tier 2 C2 is receiving connections from Tier 1 C2s grouped by each Epoch. The Y-axis is the number of connecting Tier 1 C2s and the X-axis is the timeline. The coloured lines each represent a Tier 2 C2.</p>
<br />
<h4>Epoch 1:</h4>
<br />
<p><img alt="A technical diagram" src="-/media/c5197654ddc84929ade78dc400e44162.ashx?h=auto" style="max-width: 100%;height: auto;" /></p>
<br />
<h4>Comment:</h4>
<br />
<p>Based on the synchronization in time when old Tier 2 C2s disappear and new Tier 2 C2s appears there are two separate Tier 2 C2 infrastructures for Epoch 1 in use at the same time:   <strong>37.252.15.50 -&gt; 185.180.223.70 and 72.10.162.83 -&gt; 72.10.162.84-&gt; 72.10.162.85-&gt; 72.10.162.86.  </strong></p>
<br />
<h4>Epoch 2:</h4>
<br />
<p><img alt="A technical diagram" src="-/media/95cc8c53263948118a1eb08858c5b49b.ashx?h=auto" style="max-width: 100%;height: auto;" /></p>
<br />
<h4>Comment:</h4>
<br />
<p>The new Tier 2 C2s appear at the same time or a little before the old one goes down. The overlap with two Tier 2 C2 active at the same time is noteworthy.</p>
<br />
<h4>Epoch 3:</h4>
<br />
<p><img alt="A technical diagram" src="-/media/cc549e1af9e2448cb93896f39b1a3622.ashx?h=auto" style="max-width: 100%;height: auto;" /></p>
<br />
<h4>Comment:</h4>
<br />
<p>The IP 134.119.194.179 has for a very long time been utilized as a Tier 2. The IP 37.252.14.29 has a low utilization of connecting Tier 1 C2s.</p>
<br />
<p>From the analysis above the following Tier 2 IPs were excluded due to their short time-span and low amount of connecting Tier 1 C2s indicating them to possibly be inactive:</p>
<br />
<ul><br />
    <li>37.252.15.52</li>
    <br />
    <li>37.252.14.109</li>
    <br />
    <li>185.180.223.114</li>
    <br />
</ul>
<br />
<h3>Theories on administration</h3>
<br />
<p>Our analysts have three main theories on how the actor behind Emotet administers the Tier 2 C2 servers:</p>
<br />
<h4>Theory 1 &ndash; Apache TCP/8080</h4>
<br />
<p>An administration panel and/or API access exposed over Apache TCP/8080 which the actor connects to. No network traffic in the NTT Global internet infrastructure supporting this theory has been seen.</p>
<br />
<p><img alt="A technical diagram" src="-/media/97f08a785ed94cdda30b14eee02a34e9.ashx?h=auto" style="max-width: 100%;height: auto;" /></p>
<br />
<h4>Theory 2 &ndash; NGINX TCP/80</h4>
<br />
<p>The hosting of an administration panel and/or API access over the same port that they receive C2 traffic. This would make it easier for the actor to hide their tracks by possibly tunnelling their traffic through Tier 1 C2s towards Tier 2 C2s.</p>
<br />
<p><img alt="A technical diagram" src="-/media/9f40dceae81b42418463469d1f3f2ac5.ashx?h=auto" style="max-width: 100%;height: auto;" /></p>
<br />
<h4>Theory 3 - A Tier 3 C2</h4>
<br />
<p>Tier 2 C2s forwarding their traffic towards a central Tier3 C2 which has data for all three Epochs. No outgoing traffic from Tier 2 C2s inside of NTT Global internet infrastructure has been observed supporting this and it&rsquo;s seen as very unlikely.</p>
<br />
<p><img alt="A technical diagram" src="-/media/334637bcba2944699ff87522f7b1380e.ashx?h=auto" style="max-width: 100%;height: auto;" /></p>
<br />
<h3>Closing thought on theories</h3>
<br />
<p>The overlaps observed in the Tier 2 C2 infrastructure with multiple Tier 2 C2s being active at the same time indicates an additional complexity. Based on netflow monitoring of our global internet infrastructure, theory 2 is found to be most likely, but forensics of Tier 2 C2s would be needed in order to confirm this theory.</p>
<br />
<h4>Conclusion</h4>
<br />
<p>NTT Ltd.&rsquo;s Threat Detection has reliably been able to follow the changes in the use of Tier 2 servers in the Emotet network infrastructure. It&rsquo;s noteworthy how one of the largest threat actors today can largely operate undisturbed with Tier 2 servers having uptime of multiple months. This research is of use when planning takedown actions, but also in understanding threat actors' operations and improving the detection capabilities that NTT Ltd. Threat Detection has in customer environments.</p>
<br />
