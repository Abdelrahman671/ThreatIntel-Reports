<div class="blog__post-content html-content">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p style="text-align: left;"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-DotNET-Loaders.jpg?width=700&amp;name=Blog-DotNET-Loaders.jpg" alt="DotNET Loaders" style="width: 700px;" width="700" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-DotNET-Loaders.jpg?width=350&amp;name=Blog-DotNET-Loaders.jpg 350w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-DotNET-Loaders.jpg?width=700&amp;name=Blog-DotNET-Loaders.jpg 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-DotNET-Loaders.jpg?width=1050&amp;name=Blog-DotNET-Loaders.jpg 1050w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-DotNET-Loaders.jpg?width=1400&amp;name=Blog-DotNET-Loaders.jpg 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-DotNET-Loaders.jpg?width=1750&amp;name=Blog-DotNET-Loaders.jpg 1750w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-DotNET-Loaders.jpg?width=2100&amp;name=Blog-DotNET-Loaders.jpg 2100w" sizes="(max-width: 700px) 100vw, 700px"><br>Many families of remote access trojan (RAT) are .NET executables. As was observed in the blog post <a href="#references" rel="noopener"><sup>1</sup></a> from one year ago about RevengeRAT among others, much of this malware is delivered in another .NET executable with the payload encoded as an embedded text string. These RATs when they're encoded as text and posted to pastebin like sites are tracked by Scumbots <a href="#references" rel="noopener"><sup>2</sup></a>. An additional method of hunting for this type of dropper or loader is to leverage the <em>dotnet</em> module in YARA. This module is a parser for .NET executables and presents the parsed components that it finds as a dataset inside YARA that can then be leveraged in the conditions section of a rule. In the following analysis, this parsed dataset and the user strings <a href="#references" rel="noopener"><sup>3</sup></a> entry in particular are used to identify .NET executables that have another PE executable encoded as one of those user strings.</p>
<!--more--><br>
<h2 style="font-weight: bold;">Making the Rule</h2>
<p>The first step is to examine one of the .NET loaders in question <a href="#references" rel="noopener"><sup>4</sup></a>. The examples here show a sample that delivers RevengeRAT <a href="#references" rel="noopener"><sup>5</sup></a> according to Malpedia analysis of the payload contained within it. This embedded and encoded PE file can be seen in the screenshot of the loader as analyzed by dnSpy <a href="#references" rel="noopener"><sup>6</sup></a> shown in Figure 1.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig1_embedded_rat.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig1_embedded_rat.jpg" alt="PE File Encoded as Text and Embedded in .NET Code" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 1: PE File Encoded as Text and Embedded in .NET Code</span></p>
<p>This encoded string is also visible in the debug data returned by YARA on the command line. The "-D" command line switch shows all the data that any loaded modules have available <a href="#references" rel="noopener"><sup>7</sup></a>. This is an extremely powerful feature for building rules because one can essentially see what YARA sees in a file and use that to build conditions for your new rules. The output from this command line switch when applied to the .NET sample above is shown in Figure 2. The string with the encoded payload is highlighted.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig2_yara_debug.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig2_yara_debug.jpg" alt="Debugging Output from YARA's Dotnet Module" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 2: Debugging Output from YARA's Dotnet Module</span></p>
<p>As can be seen in the debugging output, the string is interspersed with null bytes, therefore these must be included when building the regular expression. Therefore, the characters to be used in the regular expression are the following.</p>
<p><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/table.svg" alt="table" style="width: 705px;" width="705"><br>Using these characters, the base of the regular expression is the following.<br><br><span style="font-family: 'courier new', courier; font-weight: bold;">&nbsp;&nbsp; /4\x00D\x005\x00A\x00/</span><br><br>In addition to the above, two constraints can be included to reduce false positives. First, the carrot metacharacter "^" is added as the first character in the regex to make sure that only matches at the beginning of the string are possible. Second, a dot wildcard that matches any character with a repetition of 186 or more times is added to the end of the regular expression. The number chosen here for the repetition is based on research done on the smallest possible PE file with that size being 97 bytes <a href="#references" rel="noopener"><sup>8</sup></a>. This number is doubled because of the interspersed null bytes and the four characters of the PE magic number along with its accompanying null bytes are subtracted from the total.</p>
<p><span style="font-family: 'courier new', courier; font-weight: bold;">&nbsp;&nbsp; 186 = (2 * 97) - 8</span><br><br>Putting these parts together, the regular expression is the following.<br><br><span style="font-weight: bold; font-family: 'courier new', courier;">&nbsp;&nbsp; /^4\x00D\x005\x00A\x00.{186,}/</span><br><br>Next we need to account for capitalization. Some of the samples use an all lowercase alphabet to encode the embedded PE file. To account for that, two small character sets must be used in place of the "D" and "A" characters in the regular expression. After making this change, the regular expression is the following.</p>
<p><span style="font-weight: bold; font-family: 'courier new', courier;">&nbsp;&nbsp; /^4\x00[dD]\x005\x00[aA]\x00.{186,}/</span><br><br>Finally, this regular expression needs to be applied to each of the user strings that YARA's dotnet module parses out of a file. With the release of YARA 4.0, the syntax for iterating over this type of structure is really simple and easy to use <a href="#references" rel="noopener"><sup>9</sup></a>. Since the user strings are a zero-based array, one can iterate over them directly. The condition that iterates over each entry in this array and applies the regular expression to each entry in the array is the following.</p>
<p><span style="font-family: 'courier new', courier; font-weight: bold;">&nbsp;&nbsp; for any str in dotnet.user_strings : (</span><br><span style="font-family: 'courier new', courier; font-weight: bold;">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; str matches /^4\x00[dD]\x005\x00[aA]\x00.{186,}/</span><br><span style="font-family: 'courier new', courier; font-weight: bold;">&nbsp;&nbsp; )</span><br><br>Putting this all together yields the YARA rule shown in Figure 3. This rule is provided at the end of the blog.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig3_yara_rule.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig3_yara_rule.jpg" alt="YARA Rule to Match Files with an Embedded PE Executable" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 3: YARA Rule to Match Files with an Embedded PE Executable</span></p>
<p>If one is using an older version of YARA, the following syntax achieves the same results.<br><br><span style="font-family: 'courier new', courier; font-weight: bold;">&nbsp;&nbsp; for any i in (0..dotnet.number_of_user_strings-1) : (</span><br><span style="font-family: 'courier new', courier; font-weight: bold;">&nbsp;&nbsp;&nbsp;&nbsp; dotnet.user_strings[i] matches </span><br><span style="font-family: 'courier new', courier; font-weight: bold;">/^4\x00[dD]\x005\x00[aA]\x00.{186,}/</span><br><span style="font-family: 'courier new', courier; font-weight: bold;">&nbsp;&nbsp; )</span></p>
<p>Running this rule as a retro hunt in the Titanium Platform results in thousands of files that are detected as malicious or suspicious with very few files that are undetected. The hunting results are shown in Figure 4.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig4_hunting_results.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig4_hunting_results.jpg" alt="YARA Retro Hunting Results" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 4: YARA Retro Hunting Results</span></p>
<p>One question to ask when presented with good results like this is whether this technique is used in legitimate software. According to a question of how to include a windows DLL in a .NET project posted on StackOverflow, this data should be packaged as a resource<a href="#references" rel="noopener"><sup>10</sup></a>. The technique analyzed in these samples is definitely non-standard.</p>
<h2 style="font-weight: bold;">Static Extraction</h2>
<p>The next step is to extract all the embedded payloads from all these files and see what is there. To do this, a fascinating feature of the yara-python package is used: the modules_callback parameter of a YARA rule match object <a href="#references" rel="noopener"><sup>11</sup></a>. The beauty of this parameter is that it allows a function with whatever code one wants to run to be executed if a rule matches. Inside that function, the data returned from a YARA module is made available to the function in the form of a data dictionary. The specific callback function used to extract many of the payloads from the hunting results dataset is shown in Figure 5.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig5_callback_func.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig5_callback_func.jpg" alt="Callback Function to Extract Payloads" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 5: Callback Function to Extract Payloads</span></p>
<p>This callback function extracts the very most basic form of encoding encountered in these files. There are additional obfuscation techniques observed in the dataset where additional character replacements are required to recover the payload in its original form. These additional techniques will be examined in a future blog post, but many of the resulting extractions from these techniques are included in the IOC data provided below.</p>
<h2 style="font-weight: bold;">Correcting the Record</h2>
<p>After as many files as can be extracted with the python callback function as possible have been analyzed along with their parent loaders, the resulting analysis dataset is loaded into an Elasticsearch instance for easier analysis. Sorting the parent loaders by the threat level reveals a few files with a zero score. The example shown in Figure 6 is a really old file, so the detection may be stale.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig6_no_detection.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig6_no_detection.jpg" alt="Zero Threat Level on Very Old File" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 6: Zero Threat Level on Very Old File</span></p>
<p>Running this file in a sandbox shows that it is some type of hacking tool and definitely something that one would consider at least suspicious if not malicious. A screenshot of this sandbox session is shown in Figure 7.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig7_hacking_tool.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig7_hacking_tool.jpg" alt="Hacking Tool User Interface in Sandbox" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 7: Hacking Tool User Interface in Sandbox</span></p>
<p>Therefore, the Titanium Platform provides a way to correct the record and freshen an old analysis result such as this. To do this, open the file in the A1000, and click the "Reanalyze" button. This button is shown in Figure 8. As can be seen also in Figure 8, the threat level is now shown to be 2 rather than zero.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig8_reanalyze.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig8_reanalyze.jpg" alt="Reanalyze Sample" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 8: Reanalyze Sample</span></p>
<h2 style="font-weight: bold;">General Analysis</h2>
<p>As mentioned above, there are a few files that use more complex obfuscation techniques in addition to the embedding analyzed here. Even with these files excluded, a group of 1,641 loaders and their corresponding payloads are identified. A few files from this dataset along with the timeframe of the results from 2011 to now is shown in Figure 9.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig9_extraction_results.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/DotNET_Loaders/fig9_extraction_results.jpg" alt="Extraction Results" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 9: Extraction Results</span></p>
<p>The cluster IDs shown above are assigned per unique payload to the payload and each of the unique parent files that have the same embedded payload. These cluster IDs are randomly generated UUIDs per payload. The full dataset including the cluster data and threat names is provided at the end of the blog. The breakdown of the embedded files by file type according to libmagic is shown in Figure 10 with the files fairly evenly split between DLLs and EXEs.</p>
<p style="text-align: center;"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/DotNET_Loaders/fig10_embedded_file_types.jpg?width=400&amp;name=fig10_embedded_file_types.jpg" alt="Libmagic Analysis of Embedded Payloads" style="width: 400px; margin-left: auto; margin-right: auto; display: block;" width="400" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/DotNET_Loaders/fig10_embedded_file_types.jpg?width=200&amp;name=fig10_embedded_file_types.jpg 200w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/DotNET_Loaders/fig10_embedded_file_types.jpg?width=400&amp;name=fig10_embedded_file_types.jpg 400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/DotNET_Loaders/fig10_embedded_file_types.jpg?width=600&amp;name=fig10_embedded_file_types.jpg 600w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/DotNET_Loaders/fig10_embedded_file_types.jpg?width=800&amp;name=fig10_embedded_file_types.jpg 800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/DotNET_Loaders/fig10_embedded_file_types.jpg?width=1000&amp;name=fig10_embedded_file_types.jpg 1000w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/DotNET_Loaders/fig10_embedded_file_types.jpg?width=1200&amp;name=fig10_embedded_file_types.jpg 1200w" sizes="(max-width: 400px) 100vw, 400px"><br><span style="font-weight: bold; font-size: 16px;">Figure 10: Libmagic Analysis of Embedded Payloads</span></p>
<h2 style="font-weight: bold;">Conclusion</h2>
<p>The technique of encoding an executable as a text string using various encoding techniques is widely used by the adversaries who post the resulting strings on pastebin-like sites. It is interesting that some of the same string encoding techniques used by these adversaries to hide payloads on download sites across the internet are also found inside the delivery binaries. This obfuscation can definitely provide some cover from detection. To identify binaries that leverage this technique, YARA's dotnet module provides the best methods available. What was shown above is just the most simple and basic encoding technique. Further research into other encodings such as Base64 will be the topic of future blog posts.</p>
