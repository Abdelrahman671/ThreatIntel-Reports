<section class="entry-content clearfix">
<p>Malware targeting Windows OS (PE format) has a variety of obfuscation and packing techniques in place so that they complicate the code analysis processes. On the other hand, there are only a few types of packing techniques for Linux-targeting malware (ELF format), and it is mainly UPX-based. This blog article explains the details of Anti-UPX Unpacking technique, which is often applied to Linux-targeting malware.</p>
<h3>Malware with Anti-UPX Unpacking Technique</h3>
<p>The most well-known malware using Anti-UPX Unpacking technique is Mirai and its variants, which target IoT devices. Figure 1 shows the headers of UPX-packed binary and Mirai. The normal UPX packing uses “UPX!” as a magic number, while Mirai assigns a different value to each sample.</p>
<p><figure class="mt-figure mt-figure-center"><a class="mt-asset-link" href="https://blogs.jpcert.or.jp/en/.assets/anti_upx_unpack-fig1.png"><img alt="UPX-packed binary and Mirai header" class="asset asset-image at-xid-2236067 mt-image-center" height="158" src="https://blogs.jpcert.or.jp/en/.assets/thumbnail/anti_upx_unpack-fig1-640wri.png" style="display: block; margin-left: auto; margin-right: auto;" width="640"/></a><figcaption>Figure 1: UPX-packed binary (left) and Mirai header (right)</figcaption></figure></p>
<p>UPX-packed binary contains the following information in the header. Normally, only “l_magic” is altered, but “p_filesize” and “p_blocksize” are also zero-padded in some samples.</p>
<pre style="padding: 10px 10px;color:#d1d1d1;background:#1f1f1f;overflow: auto;white-space: pre"><span style="color:#e66170; font-weight:bold; ">struct</span> l_info       <span style="color:#9999a9; ">// 12-byte trailer in header for loader (offset 116)</span>
<span style="color:#b060b0; ">{</span>
    uint32_t l_checksum<span style="color:#b060b0; ">;</span>
    uint32_t l_magic<span style="color:#b060b0; ">;</span>    <span style="color:#9999a9; ">// magic number = "UPX!"</span>
    uint16_t l_lsize<span style="color:#b060b0; ">;</span>
    uint8_t  l_version<span style="color:#b060b0; ">;</span>
    uint8_t  l_format<span style="color:#b060b0; ">;</span>
<span style="color:#b060b0; ">}</span><span style="color:#b060b0; ">;</span>

<span style="color:#e66170; font-weight:bold; ">struct</span> p_info       <span style="color:#9999a9; ">// 12-byte packed program header follows stub loader</span>
<span style="color:#b060b0; ">{</span>
    uint32_t p_progid<span style="color:#b060b0; ">;</span>
    uint32_t p_filesize<span style="color:#b060b0; ">;</span>
    uint32_t p_blocksize<span style="color:#b060b0; ">;</span>
<span style="color:#b060b0; ">}</span><span style="color:#b060b0; ">;</span>
</pre>
<p>Besides Mirai, there are many other types of malware using this technique, including BoSSaBot (seen around 2014), as well as some coin miners and SBIDIOT malware, more recently. This is also applied to some types of malware which were used by Lazarus group. Figure 2 shows a part of ELF-VSingle’s code, which is associated with the group. The magic number is replaced with “MEMS”.</p>
<p><figure class="mt-figure mt-figure-center"><a class="mt-asset-link" href="https://blogs.jpcert.or.jp/en/.assets/anti_upx_unpack-fig2.png"><img alt="ELF_VSingle header" class="asset asset-image at-xid-2236068 mt-image-center" height="367" src="https://blogs.jpcert.or.jp/en/.assets/thumbnail/anti_upx_unpack-fig2-640wri.png" style="display: block; margin-left: auto; margin-right: auto;" width="640"/></a><figcaption>Figure 2: ELF_VSingle header</figcaption></figure></p>
<h3>Unpacking Anti-UPX Unpacking binary</h3>
<p>Binary based on Anti-UPX Unpacking technique cannot be unpacked using the normal upx command. However, it is actually easy to unpack it. In most cases, the only change made to such binary is its magic number “UPX!”. You can unpack it with upx command by changing this value back to “UPX!”. Figure 3 shows the process of changing the magic number in order to unpack it using upx command.</p>
<p><figure class="mt-figure mt-figure-center"><a class="mt-asset-link" href="https://blogs.jpcert.or.jp/en/.assets/anti_upx_unpack-fig3.png"><img alt="Example of unpacking binary" class="asset asset-image at-xid-2236069 mt-image-center" height="904" src="https://blogs.jpcert.or.jp/en/.assets/thumbnail/anti_upx_unpack-fig3-640wri.png" style="display: block; margin-left: auto; margin-right: auto;" width="640"/></a><figcaption>Figure 3: Example of unpacking binary</figcaption></figure></p>
<p>We have created a tool that enables unpacking binary with Anti-UPX Unpacking techniques. This tool is intended for this purpose only, and it may not work otherwise.</p>
<p><strong>JPCERTCC/upx-mod - GitHub</strong> <a href="https://github.com/JPCERTCC/upx-mod/releases/tag/v4.00-beta">https://github.com/JPCERTCC/upx-mod/releases/tag/v4.00-beta</a></p>
<p><figure class="mt-figure mt-figure-center"><a class="mt-asset-link" href="https://blogs.jpcert.or.jp/en/.assets/anti_upx_unpack-fig4.png"><img alt="Sample use of upx_mod command" class="asset asset-image at-xid-2236070 mt-image-center" height="534" src="https://blogs.jpcert.or.jp/en/.assets/thumbnail/anti_upx_unpack-fig4-640wri.png" style="display: block; margin-left: auto; margin-right: auto;" width="640"/></a><figcaption>Figure 4: Sample use of upx_mod command</figcaption></figure></p>

<h3>In closing</h3>
<p>Many attack groups use malware based on Anti-UPX Unpacking technique. It is easy to unpack such malware, but you may waste your time in unpacking process unless you notice this feature beforehand. When you analyse packed ELF binary, we recommend checking first whether it uses Anti-UPX Unpacking technique.</p>
