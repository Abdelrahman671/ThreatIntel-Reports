<h3><span id="Gaining_code_execution_using_a_malicious_SQLite_database">Gaining code execution using a malicious SQLite database</span></h3>
<p><strong>Research By:</strong> Omer Gull</p>
<h2><span id="tldr">tl;dr</span></h2>
<p>SQLite is one of the most deployed software in the world. However, from a security perspective, it has only been examined through the lens of WebSQL and browser exploitation. We believe that this is just the tip of the iceberg.</p>
<p>In our long term research, we experimented with the exploitation of memory corruption issues within SQLite without relying on any environment other than the SQL language. Using our innovative techniques of Query Hijacking and Query Oriented Programming, we proved it is possible to reliably exploit memory corruptions issues in the SQLite engine. We demonstrate these techniques a couple of real-world scenarios: pwning a password stealer backend server, and achieving iOS persistency with higher privileges.</p>
<p>We hope that by releasing our research and methodology, the security research community will be inspired to continue to examine SQLite in the countless scenarios where it is available. Given the fact that SQLite is practically built-in to every major OS, desktop or mobile, the landscape and opportunities are endless. Furthermore, many of the primitives presented here are not exclusive to SQLite and can be ported to other SQL engines. Welcome to the brave new world of using the familiar Structured Query Language for exploitation primitives.</p>
<h2><span id="Motivation">Motivation</span></h2>
<p>This research started when <a href="https://twitter.com/omriher?lang=en">omriher</a> and I were looking at the leaked source code of some notorious password stealers. While there are plenty of password stealers out there (<a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.azorult">Azorult</a>, <a href="https://securelist.com/loki-bot-stealing-corporate-passwords/87595/">Loki Bot</a>, and <a href="https://www.acunetix.com/blog/articles/pony-malware-credential-theft/">Pony</a> to name a few), their modus operandi is mostly the same:</p>
<p>A computer gets infected, and the malware either captures credentials as they are used or collects stored credentials maintained by various clients.<br />
It is not uncommon for client software to use SQLite databases for such purposes.<br />
After the malware collects these SQLite files, it sends them to its C2 server where they are parsed using PHP and stored in a collective database containing all of the stolen credentials.</p>
<p>Skimming through the leaked source code of such password stealers, we started speculating about the attack surface described above.<br />
Can we leverage the load and query of an untrusted database to our advantage?<br />
Such capabilities could have much bigger implications in countless scenarios, as <a href="https://www.sqlite.org/mostdeployed.html">SQLite is one of the most widely deployed pieces of software out there</a>.</p>
<p>A surprisingly complex code base, available in almost any device imaginable. is all the motivation we needed, and so our journey began.</p>
<h2><span id="SQLite_Intro">SQLite Intro</span></h2>
<p>The chances are high that you are currently using SQLite, even if you are unaware of it.<br />
To quote its authors:</p>
<p><em>SQLite is a C-language library that implements a small, fast, self-contained, high-reliability, full-featured, SQL database engine. SQLite is the most used database engine in the world. SQLite is built into all mobile phones and most computers and comes bundled inside countless other applications that people use every day.</em></p>
<p>Unlike most other SQL databases, SQLite does not have a separate server process. SQLite reads and writes directly to ordinary disk files. A complete SQL database with multiple tables, indices, triggers, and views is contained within a single disk file.</p>
<h3><span id="Attack_Surface">Attack Surface</span></h3>
<p>The following snippet is a fairly generic example of a password stealer backend.</p>
<p><a href="/wp-content/uploads/2019/08/process_note_src.png"><img fetchpriority="high" decoding="async" width="1466" height="1066" class="alignnone size-full wp-image-22076" src="/wp-content/uploads/2019/08/process_note_src.png" alt="" /></a></p>
<p>Given the fact that we control the database and its content, the attack surface available to us can be divided into two parts: The load and initial parsing of our database, and the SELECT query performed against it.</p>
<p>The initial loading done by sqlite3_open is actually a very limited surface; it is basically a lot of setup and configuration code for opening the database. Our surface is mainly the <a href="https://www.sqlite.org/fileformat.html#the_database_header">header</a> parsing which is battle-tested against AFL.</p>
<p>Things get more interesting as we start querying the database.<br />
Using SQLite authors’ words:</p>
<p><em>“The SELECT statement is the most complicated command in the SQL language.”</em></p>
<p>Although we have no control over the query itself (as it is hardcoded in our target), studying the SELECT process carefully will prove beneficial in our quest for exploitation.</p>
<p>As SQLite3 is a virtual machine, every SQL statement must first be compiled into a byte-code program using one of the <a href="https://www.sqlite.org/c3ref/prepare.html">sqlite3_prepare*</a> routines.<br />
Among other operations, the prepare function walks and expands all SELECT subqueries. Part of this process is verifying that all relevant objects (like tables or views) actually exist and locating them in the master schema.</p>
<h3><span id="sqlite_master_and_DDL">sqlite_master and DDL</span></h3>
<p>Every SQLite database has a <code>sqlite_master</code> table that defines the schema for the database and all of its objects (such as tables, views, indices, etc.).<br />
The sqlite_master table is defined as:</p>
<p><a href="/wp-content/uploads/2019/08/sqlite_master_DDL.png"><img decoding="async" width="838" height="478" class="alignnone size-full wp-image-22057" src="/wp-content/uploads/2019/08/sqlite_master_DDL.png" alt="" /></a></p>
<p>The part that is of special interest to us is the sql column.<br />
This field is the DDL (Data Definition Language) used to describe the object.<br />
In a sense, the DDL commands are similar to C header files. DDL commands are used to define the structure, names, and types of the data containers within a database, just as a header file typically defines type definitions, structures, classes, and other data structures.</p>
<p>These DDL statements actually appear in plain-text if we inspect the database file:</p>
<p><a href="/wp-content/uploads/2019/08/xxd_db.png"><img decoding="async" width="2068" height="1104" class="alignnone size-full wp-image-22054" src="/wp-content/uploads/2019/08/xxd_db.png" alt="" /></a></p>
<p>During the query preparation, sqlite3LocateTable()  attempts to find the in-memory structure that describes the table we are interested in querying.</p>
<p>sqlite3LocateTable() reads the schema available in sqlite_master, and if this is the first time doing it, it also has a callback for every result that verifies the DDL statement is valid and build the necessary internal data structures that describe the object in question.</p>
<h2><span id="DDL_Patching">DDL Patching</span></h2>
<p>Learning about this preparation process, we asked, can we simply replace the DDL that appears in plain-text within the file? If we could inject our own SQL to the file perhaps we can affect its behaviour.</p>
<p><a href="/wp-content/uploads/2019/08/init_callback_func.png"><img loading="lazy" decoding="async" width="2078" height="660" class="alignnone size-full wp-image-22084" src="/wp-content/uploads/2019/08/init_callback_func.png" alt="" /></a></p>
<p>Based on the code snippet above, it seems that DDL statements must begin with “create “.<br />
With this limitation in mind, we needed to assess our surface.<br />
Checking SQLite’s documentation revealed that these are the possible objects we can create:</p>
<p><a href="/wp-content/uploads/2019/08/create_options.png"><img loading="lazy" decoding="async" width="2238" height="836" class="alignnone size-full wp-image-22090" src="/wp-content/uploads/2019/08/create_options.png" alt="" /></a></p>
<p>The CREATE VIEW command gave us an interesting idea. To put it very simply, VIEWs are just pre-packaged SELECT statements. If we replace the table expected by the target software with a compatible VIEW, interesting opportunities reveal themselves.</p>
<h3><span id="Hijack_Any_Query">Hijack Any Query</span></h3>
<p>Imagine the following scenario:<br />
The original database has a single TABLE called dummy that is defined as:</p>
<p><a href="/wp-content/uploads/2019/08/dummy_table.png"><img loading="lazy" decoding="async" width="618" height="272" class="alignnone size-full wp-image-22088" src="/wp-content/uploads/2019/08/dummy_table.png" alt="" /></a></p>
<p>The target software queries it with the following:</p>
<p><a href="/wp-content/uploads/2019/08/target_query_dummy.png"><img loading="lazy" decoding="async" width="888" height="100" class="alignnone size-full wp-image-22056" src="/wp-content/uploads/2019/08/target_query_dummy.png" alt="" /></a></p>
<p>We can actually hijack this query if we craft dummy as a VIEW:</p>
<p><a href="/wp-content/uploads/2019/08/dummy_query_hijacking.png"><img loading="lazy" decoding="async" width="2056" height="100" class="alignnone size-full wp-image-22089" src="/wp-content/uploads/2019/08/dummy_query_hijacking.png" alt="" /></a></p>
<p>This “trap” VIEW enables us to hijack the query &#8211; meaning we generate a completely new query that we <strong>totally control.</strong></p>
<p><a href="/wp-content/uploads/2019/08/query_hijacking_example.png"><img loading="lazy" decoding="async" width="1532" height="816" class="alignnone size-full wp-image-22062" src="/wp-content/uploads/2019/08/query_hijacking_example.png" alt="" /></a></p>
<p>This nuance greatly expands our attack surface, from the very minimal parsing of the header and an uncontrollable query performed by the loading software, to the point where we can now interact with vast parts of the SQLite interpreter by patching the DDL and creating our own views with sub-queries.</p>
<p>Now that we can interact with the SQLite interpreter, our next question was what exploitation primitives are built into SQLite? Does it allow any system commands, reading from or writing to the filesystem?</p>
<p>As we are not the first to notice the huge SQLite potential from an exploitation perspective, it makes sense to review prior work done in the field. We started from the very basics.</p>
<h3><span id="SQL_Injections">SQL Injections</span></h3>
<p>As researchers, it’s hard for us to even spell SQL without the “i”, so it seems like a reasonable place to start. After all, we want to familiarize ourselves with the internal primitives offered by SQLite. Are there any system commands? Can we load arbitrary libraries?</p>
<p><a href="http://atta.cked.me/home/sqlite3injectioncheatsheet">It seems</a> that the most straightforward trick involves attaching a new database file and writing to it using something along the lines of:</p>
<p><a href="/wp-content/uploads/2019/08/sqli_attach.png"><img loading="lazy" decoding="async" width="1548" height="184" class="alignnone size-full wp-image-22059" src="/wp-content/uploads/2019/08/sqli_attach.png" alt="" /></a></p>
<p>We attach a new database, create a single table and insert a single line of text. The new database then creates a new file (as databases are files in SQLite) with our web shell inside it.<br />
The very forgiving nature of the PHP interpreter parses our database until it reaches the PHP open tag of “&lt;?”.<br />
Writing a webshell is definitely a win in our password stealers scenario, however, as you recall, DDL cannot begin with “ATTACH”</p>
<p><a href="/wp-content/uploads/2019/08/sqli_load_module.png"><img loading="lazy" decoding="async" width="1630" height="92" class="alignnone size-full wp-image-22058" src="/wp-content/uploads/2019/08/sqli_load_module.png" alt="" /></a></p>
<p>Another relevant option is the <a href="https://www.sqlite.org/c3ref/load_extension.html">load_extension</a> function. While this function should allow us to load an arbitrary shared object, it is disabled by default.</p>
<h3><span id="Memory_Corruptions_In_SQLite">Memory Corruptions In SQLite</span></h3>
<p>Like any other software written in C, memory safety issues are definitely something to consider when assessing the security of SQLite.<br />
In his great <a href="https://lcamtuf.blogspot.com/2015/04/finding-bugs-in-sqlite-easy-way.html">blog post</a>, Michał Zalewski described how he fuzzed SQLite with AFL to achieve some impressive results: 22 bugs in just 30 minutes of fuzzing.</p>
<p>Interestingly, SQLite has since started using AFL as an integral part of their remarkable test suite.</p>
<p>These memory corruptions were all treated with the expected gravity (Richard Hip and his team deserve tons of respect). However, from an attacker’s perspective, these bugs would prove to be a difficult path to exploitation without a decent framework to leverage them.<br />
Modern mitigations pose a major obstacle in exploiting memory corruption issues and attackers need to find a more flexible environment.</p>
<p>The Security Research community would soon find the perfect target!</p>
<h3><span id="Web_SQL">Web SQL</span></h3>
<p>Web SQL Database is a web page API for storing data in databases that can be queried using a variant of SQL through JavaScript.The W3C Web Applications Working Group ceased working on the specification in November 2010, citing a lack of independent implementations other than SQLite.</p>
<p>Currently, the API is still supported by Google Chrome, Opera and Safari.<br />
All of them use SQLite as the backend of this API.</p>
<p>Untrusted input into SQLite, reachable from any website inside some of the most popular browsers, caught the security community’s attention and as a result, the number of vulnerabilities began to rise.<br />
Suddenly, bugs in SQLite could be leveraged by the JavaScript interpreter to achieve reliable browser exploitation.</p>
<p>Several impressive research reports have been published:</p>
<ul>
<li>Low hanging fruits like <a href="https://nvd.nist.gov/vuln/detail/CVE-2015-7036">CVE-2015-7036</a>
<ul>
<li>Untrusted pointer dereference <strong>fts3_tokenizer()</strong></li>
</ul>
</li>
<li><a href="https://www.blackhat.com/docs/us-17/wednesday/us-17-Feng-Many-Birds-One-Stone-Exploiting-A-Single-SQLite-Vulnerability-Across-Multiple-Software.pdf">More complex exploits</a> presented in Blackhat 17 by the <a href="https://twitter.com/chaitintech?lang=en">Chaitin team</a>
<ul>
<li>Type confusion in <strong>fts3OptimizeFunc()</strong></li>
</ul>
</li>
<li>The recent Magellan bugs <a href="https://blog.exodusintel.com/2019/01/22/exploiting-the-magellan-bug-on-64-bit-chrome-desktop">exploited by Exodus</a>
<ul>
<li>Integer overflow in <strong>fts3SegReaderNext()</strong></li>
</ul>
</li>
</ul>
<p>A clear pattern in past WebSQL research reveals that a virtual table module named ”FTS” might be an interesting target for our research.</p>
<h2><span id="FTS">FTS</span></h2>
<p>Full-Text Search  (FTS) is a virtual table module that allows textual searches on a set of documents.<br />
From the perspective of an SQL statement, the virtual table object looks like any other table or view. But behind the scenes, queries on a virtual table invoke callback methods on shadow tables instead of the usual reading and writing on the database file.</p>
<p>Some virtual table implementations, like FTS, make use of real (non-virtual) database tables to store content.</p>
<p>For example, when a string is inserted into the FTS3 virtual table, some metadata must be generated to allow for an efficient textual search. This metadata is ultimately stored in real tables named &#8220;%_segdir&#8221; and &#8220;%_segments&#8221;, while the content itself is stored in “&#8221;%_content&#8221; where &#8220;%&#8221; is the name of the original virtual table.<br />
These auxiliary real tables that contain data for a virtual table are called &#8220;shadow tables&#8221;</p>
<p><a href="/wp-content/uploads/2019/08/shadow_tables_graph.png"><img loading="lazy" decoding="async" width="2880" height="1800" class="alignnone size-full wp-image-22060" src="/wp-content/uploads/2019/08/shadow_tables_graph.png" alt="" /></a></p>
<p>Due to their trusting nature, interfaces that pass data between shadow tables provide a fertile ground for bugs. CVE-2019-8457,- a new OOB read vulnerability we found in the RTREE virtual table module, demonstrates this well.</p>
<p>RTREE virtual tables, used for geographical indexing, are expected to begin with an integer column. Therefore, other RTREE interfaces expect the first column in an RTREE to be an integer. However, if we create a table where the first column is a string, as shown in the figure below, and pass it to the <strong>rtreenode() </strong>interface, an OOB read occurs.</p>
<p><a href="/wp-content/uploads/2019/08/rtree_oob.png"><img loading="lazy" decoding="async" width="2212" height="892" class="alignnone size-full wp-image-22061" src="/wp-content/uploads/2019/08/rtree_oob.png" alt="" /></a></p>
<p>Now that we can use query hijacking to gain control over a query, and know where to find vulnerabilities, it’s time to move on to exploit development.</p>
<h2><span id="SQLite_Internals_For_Exploit_Development">SQLite Internals For Exploit Development</span></h2>
<p>Previous publications on SQLite exploitation clearly show that there has always been a necessity for a wrapping environment, whether it is the PHP interpreter seen in this awesome <a href="https://medium.com/0xcc/bypass-php-safe-mode-by-abusing-sqlite3s-fts-tokenizer-256ee2555607">blog post </a>on abusing SQLite tokenizers or the more recent work on Web SQL from the comfort of a JavaScript interpreter.</p>
<p>As SQLite is pretty much everywhere, limiting its exploitation potential sounded like low-balling to us and we started exploring the use of SQLite internals for exploitation purposes.<br />
The research community became pretty good at utilizing JavaScript for exploit development. Can we achieve similar primitives with SQL?</p>
<p>Bearing in mind that SQL is Turing complete ([<a href="http://assets.en.oreilly.com/1/event/27/High%252520Performance%252520SQL%252520with%252520PostgreSQL%252520Presentation.pdf">1</a>], [<a href="http://wiki.postgresql.org/wiki/Mandelbrot_set">2</a>]), we started creating a primitive wish-list for exploit development based on our pwning experience.<br />
A modern exploit written purely in SQL has the following capabilities:</p>
<ul>
<li>Memory leak.</li>
<li>Packing and unpacking of integers to 64-bit pointers.</li>
<li>Pointer arithmetics.</li>
<li>Crafting complex fake objects in memory.</li>
<li>Heap Spray.</li>
</ul>
<p>One by one, we will tackle these primitives and implement them using nothing but SQL.</p>
<p><a href="/wp-content/uploads/2019/08/is_it_possible.gif"><img loading="lazy" decoding="async" width="480" height="271" class="alignnone size-full wp-image-22080" src="/wp-content/uploads/2019/08/is_it_possible.gif" alt="" /></a></p>
<p>For the purpose of achieving RCE on PHP7, we will utilize the still unfixed 1-day of  <a href="https://nvd.nist.gov/vuln/detail/CVE-2015-7036">CVE-2015-7036</a>.</p>
<p>Wait, what? How come a 4-year-old bug has never been fixed? It is actually an interesting story and a great example of our argument.<br />
This feature was only ever considered vulnerable in the context of a program that allows arbitrary SQL from an untrusted source (Web SQL), and so it was mitigated accordingly.<br />
However, SQLite usage is so versatile that we can actually still trigger it in many scenarios 🙂</p>
<h3><span id="Exploitation_Game-plan">Exploitation Game-plan</span></h3>
<p>CVE-2015-7036 is a very convenient bug to work with.<br />
In a nutshell, the vulnerable <strong>fts3_tokenizer()</strong> function returns the tokenizer address when called with a single argument (like “simple”, “porter” or any other registered tokenizer).</p>
<p><a href="/wp-content/uploads/2019/08/fts_tokenizer_1arg.png"><img loading="lazy" decoding="async" width="1192" height="246" class="alignnone size-full wp-image-22087" src="/wp-content/uploads/2019/08/fts_tokenizer_1arg.png" alt="" /></a></p>
<p>When called with 2 arguments, fts3_tokenizer overrides the tokenizer address in the first argument with the address provided by a blob in the second argument.<br />
After a certain tokenizer has been overridden, any new instance of the fts table that uses this tokenizer allows us to hijack the flow of the program.</p>
<p><a href="/wp-content/uploads/2019/08/fts_tokenizer_2arg.png"><img loading="lazy" decoding="async" width="1578" height="198" class="alignnone size-full wp-image-22086" src="/wp-content/uploads/2019/08/fts_tokenizer_2arg.png" alt="" /></a></p>
<p>Our exploitation game-plan:</p>
<ul>
<li>Leak a tokenizer address</li>
<li>Compute the base address</li>
<li>Forge a fake tokenizer that will execute our malicious code</li>
<li>Override one of the tokenizers with our malicious tokenizer</li>
<li>Instantiate an fts3 table to trigger our malicious code</li>
</ul>
<p>Now back to our exploit development.</p>
<h2><span id="Query_Oriented_Programming">Query Oriented Programming ©</span></h2>
<p>We are proud to present our own unique approach for exploit development using the familiar structured query language. We share QOP with the community in the hope of encouraging researchers to pursue the endless possibilities of database engines exploitation.<br />
Each of the following primitives is accompanied by an example from the sqlite3 shell.</p>
<p>While this will give you a hint of what want to achieve, keep in mind that our end goal is to plant all those primitives in the sqlite_master table and hijack the queries issued by the target software that loads and queries our malicious SQLite db file</p>
<h3><span id="Memory_Leak_8211_Binary">Memory Leak &#8211; Binary</span></h3>
<p>Mitigations such as ASLR definitely raised the bar for memory corruptions exploitation. A common way to defeat it is to learn something about the memory layout around us.<br />
This is widely known as Memory Leak.<br />
Memory leaks are their own sub-class of vulnerabilities, and each one has a slightly different setup.</p>
<p>In our case, the leak is the return of a BLOB by SQLite.<br />
These BLOBs make a fine leak target as they sometimes hold memory pointers.</p>
<p><a href="/wp-content/uploads/2019/08/fts_tokenizer_1arg.png"><img loading="lazy" decoding="async" width="1192" height="246" class="alignnone size-full wp-image-22087" src="/wp-content/uploads/2019/08/fts_tokenizer_1arg.png" alt="" /></a></p>
<p>The vulnerable fts3_tokenizer() is called with a single argument and returns the memory address of the requested tokenizer. <a href="https://www.sqlite.org/lang_corefunc.html#hex">hex()</a> makes it readable by humans.<br />
We obviously get some memory address, but it is reversed due to little-endianity.<br />
Surely we can flip it using some SQLite built-in string operations.</p>
<p><a href="/wp-content/uploads/2019/08/qop_bin_leak.png"><img loading="lazy" decoding="async" width="1652" height="510" class="alignnone size-full wp-image-22072" src="/wp-content/uploads/2019/08/qop_bin_leak.png" alt="" /></a></p>
<p><a href="https://www.sqlite.org/lang_corefunc.html#substr">substr()</a> seems to be a perfect fit! We can read little-endian BLOBs but this raises another question: how do we store things?</p>
<h3><span id="QOP_Chain">QOP Chain</span></h3>
<p>Naturally, storing data in SQL requires an INSERT statement. Due to the hardened verification of sqlite_master, we can’t use INSERT as all of the statements must start with “CREATE “. Our approach to this challenge is to simply store our queries under a meaningful VIEW and chain them together.<br />
The following example makes it a bit clearer:</p>
<p><a href="/wp-content/uploads/2019/08/qop_vars.png"><img loading="lazy" decoding="async" width="1798" height="608" class="alignnone size-full wp-image-22063" src="/wp-content/uploads/2019/08/qop_vars.png" alt="" /></a></p>
<p>This might not seem like a big difference, but as our chain gets more complicated, being able to use pseudo-variables will surely make our life easier.</p>
<h3><span id="Unpacking_of_64-bit_pointers">Unpacking of 64-bit pointers</span></h3>
<p>If you’ve ever done any pwning challenges, the concept of packing and unpacking of pointers should not be foreign.<br />
This primitive should make it easy to convert our hexadecimal values (like the leak we just achieved) to integers. Doing so allows us to perform various calculations on these pointers in the next steps.</p>
<p><a href="/wp-content/uploads/2019/08/qop_u64.png"><img loading="lazy" decoding="async" width="2472" height="974" class="alignnone size-full wp-image-22064" src="/wp-content/uploads/2019/08/qop_u64.png" alt="" /></a></p>
<p>This query iterates a hexadecimal string char by char in a reversed fashion using substr().</p>
<p>A translation of this char is done using <a href="https://gist.github.com/ChiChou/97a53caa2c0b49c1991e">this</a> clever trick with the minor adjustment of <a href="https://www.sqlite.org/lang_corefunc.html#instr">instr()</a> which is 1-based.<br />
All that is needed now is the proper shift that is on the right of the * sign.</p>

<h3><span id="Packing_of_64-bit_pointers">Packing of 64-bit pointers</span></h3>
<p>After reading leaked pointers and manipulating them to our will, it makes sense to pack them back to their little-endian form so we can write them somewhere.<br />
SQLite <a href="https://www.sqlite.org/lang_corefunc.html%23char">char()</a>  should be of use here as its documentation states that it will “return a string composed of characters having the Unicode code point values of an integer.”<br />
It proved to work fairly well, but only on a limited range of integers</p>

<h3><span id="Heap_Spray">Heap Spray</span></h3>
<p>Now that we crafted our fake object, it is sometimes useful to spray the heap with it.<br />
This should ideally be some repetitive form of the latter.</p>
<p>Unfortunately, SQLite does not implement the REPEAT() function like MySQL.<br />
However, <a href="https://stackoverflow.com/questions/11568496/how-to-emulate-repeat-in-sqlite">this</a> thread gave us an elegant solution.</p>
<p><a href="/wp-content/uploads/2019/08/qop_heap_spray.png"><img loading="lazy" decoding="async" width="2128" height="256" class="alignnone size-full wp-image-22068" src="/wp-content/uploads/2019/08/qop_heap_spray.png" alt="" /></a></p>
<p>The <a href="https://www.sqlite.org/lang_corefunc.html#zeroblob">zeroblob(N)</a> function returns a BLOB consisting of N bytes while we use <a href="https://www.sqlite.org/lang_corefunc.html#replace">replace()</a> to replace those zeros with our fake object.</p>
<p>Searching for those 0x41s shows we also achieved a perfect consistency. Notice the repetition every 0x20 bytes.</p>
<p><a href="/wp-content/uploads/2019/08/pwndbg_spray.png"><img loading="lazy" decoding="async" width="944" height="654" class="alignnone size-full wp-image-22074" src="/wp-content/uploads/2019/08/pwndbg_spray.png" alt="" /></a></p>
<h3><span id="Memory_Leak_8211_Heap">Memory Leak &#8211; Heap</span></h3>
<p>Looking at our exploitation game plan, it seems like we are moving in the right direction.<br />
We already know where the binary image is located, we were able to deduce where the necessary functions are, and spray the heap with our malicious tokenizer.</p>
<p>Now it’s time to override a tokenizer with one of our sprayed objects. However, as the heap address is also randomized, we don’t know where our spray is allocated.<br />
A heap leak requires us to have another vulnerability.</p>
<p>Again, we will target a virtual table interface.<br />
As virtual tables use underlying shadow tables, it is quite common for them to pass raw pointers between different SQL interfaces.</p>
<p>Note: This exact type of issue was mitigated in <a href="https://www.sqlite.org/bindptr.html">SQLite 3.20</a>. Fortunately, PHP7 is compiled with an earlier version. In case of an updated version, CVE-2019-8457 could be used here as well.</p>
<p>To leak the heap address, we need to generate an fts3 table beforehand and abuse its MATCH interface.</p>
<p><a href="/wp-content/uploads/2019/08/qop_heap_leak.png"><img loading="lazy" decoding="async" width="1346" height="314" class="alignnone size-full wp-image-22069" src="/wp-content/uploads/2019/08/qop_heap_leak.png" alt="" /></a></p>
<p>Just as we saw in our first memory leak, the pointer is little-endian so it needs to be reversed. Fortunately, we already know how to do so using SUBSTR().<br />
Now that we know our heap location, and can spray properly, we can finally override a tokenizer with our malicious tokenizer!</p>

<h2><span id="iOS_Persistency">iOS Persistency</span></h2>
<p>Persistency is hard to achieve on iOS as all executable files must be signed as part of Apple&#8217;s Secure Boot. Luckily for us, SQLite databases are not signed.</p>
<p>Utilizing our new capabilities, we will replace one of the commonly used databases with a malicious version. After the device reboots and our malicious database is queried, we gain code execution.</p>
<p>To demonstrate this concept, we replace the Contacts DB “AddressBook.sqlitedb”. As done in our PHP7 exploit, we create two extra DDL statements. One DDL statement overrides the default tokenizer “simple”, and the other DDL statement triggers the crash by trying to instantiate the overridden tokenizer. Now, all we have to do is re-write every table of the original database as a view that hijacks any query performed and redirect it toward our malicious DDL.</p>
<p><a href="/wp-content/uploads/2019/08/ios_original_contacts.png"><img loading="lazy" decoding="async" width="2386" height="416" class="alignnone size-full wp-image-22081" src="/wp-content/uploads/2019/08/ios_original_contacts.png" alt="" /></a></p>

<h2><span id="Future_Work">Future Work</span></h2>

<h2><span id="Conclusion">Conclusion</span></h2>
<p>We established that simply querying a database may not be as safe as you expect. Using our innovative techniques of Query Hijacking and Query Oriented Programming, we proved that memory corruption issues in SQLite can now be reliably exploited. To demonstrate these concepts, we achieved remote code execution on a password stealer backend running PHP7 and gained persistency with higher privileges on iOS. We believe that these are just a couple of use cases in the endless landscape of SQLite.</p>

function calcScreenW(){
	var screenC = false;
	curentW= jQuery(window).width();	
	if ( curentW > 1450 ) { 
		if (screenW != 'desktop'){
			screenW = 'desktop';

		}
	} else if (curentW > 973){
		if (screenW != 'laptop'){
			screenW = 'laptop';	
			screenC = true;
		}		
	} else if (curentW >767){
		if (screenW != 'tablet'){

		if (screenW != 'mobile'){

		jQuery('.bg-change-done').removeClass('bg-change-done');

		animate_elems();

}

function animate_elems() {
	wintop = jQuery(window).scrollTop(); // calculate distance from top of window
	// loop through each item to check when it animates
	$elems.each(function(){
		$elm = jQuery(this);
		if($elm.hasClass('lazyload')|| $elm.hasClass('bg-change-done')) { return true; } // if already animated skip to the next item

		topcoords = $elm.offset().top; // element's distance from top of page in pixels

		if(wintop > (topcoords - (winheight))) {			
			$elm.addClass('bg-change-done');		
			loadImg($elm[0]);
		} 
	});	
}

	function getCookie(cname) {
	  var name = cname + "=";
	  var decodedCookie = decodeURIComponent(document.cookie);
	  var ca = decodedCookie.split(';');

		var c = ca[i];

		}
