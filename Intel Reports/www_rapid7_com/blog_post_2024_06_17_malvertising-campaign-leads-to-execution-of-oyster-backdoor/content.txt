                            <p><em>The following analysts contributed to this blog: Thomas Elkins, Daniel Thiede, Josh Lockwood, Tyler McGraw, and Sasha Kovalev.</em></p><!--kg-card-begin: markdown--><h2 id="executivesummary">Executive Summary</h2>
<p>Rapid7 has observed a recent malvertising campaign that lures users into downloading malicious installers for popular software such as Google Chrome and Microsoft Teams. The installers were being used to drop a backdoor identified as <code>Oyster</code>, aka <code>Broomstick</code>. Following execution of the backdoor, we have observed enumeration commands indicative of hands-on-keyboard activity as well as the deployment of additional payloads.</p>
<p>In this blog post, we will examine the delivery methods of the <code>Oyster</code> backdoor, provide an in-depth analysis of its components, and offer a Python script to help extract its obfuscated configuration.</p>
<h2 id="overview">Overview</h2>
<h3 id="initialaccess">Initial Access</h3>
<p>In three separate incidents, Rapid7 observed users downloading supposed Microsoft Teams installers from typo-squatted websites. Users were directed to these websites after using search engines such as Google and Bing for Microsoft Teams software downloads. Rapid7 observed that the websites were masquerading as Microsoft Teams websites, enticing users into believing they were downloading legitimate software when, in reality, they were downloading the threat actor’s malicious software.</p>
<p><img src="https://blog.rapid7.com/content/images/2024/06/Figure-1---Fake-Microsoft-Teams-Website.png" alt="Figure-1---Fake-Microsoft-Teams-Website"><br>
<strong>Figure 1 - Fake Microsoft Teams Website</strong></p>
<p>In one case, a user was observed navigating to the URL <code>hxxps://micrsoft-teams-download[.]com/</code>, which led to the download of the binary <code>MSTeamsSetup_c_l_.exe</code>. Initial analysis of the binary <code>MSTeamsSetup_c_l_.exe</code> showed that the binary was assigned by an Authenticode certificate issued to “Shanxi Yanghua HOME Furnishings Ltd”.</p>
<p><img src="https://blog.rapid7.com/content/images/2024/06/Figure-2----MSTeamsSetup_c_l_.exe--File-Information.JPG" alt="Figure-2----MSTeamsSetup_c_l_.exe--File-Information"><br>
<strong>Figure 2 - <code>MSTeamsSetup_c_l_.exe</code> File Information</strong></p>
<p>Searching VirusTotal for other files signed by “Shanxi Yanghua HOME Furnishings Ltd” showed the following:</p>
<p><img src="https://blog.rapid7.com/content/images/2024/06/Figure-3---VirusTotal-Signature-Search-Results.png" alt="Figure-3---VirusTotal-Signature-Search-Results"><br>
<strong>Figure 3 - VirusTotal Signature Search Results</strong></p>
<p>The results indicated other versions of the installer, each impersonating as a legitimate software installer. We observed that the first installer was submitted to VirusTotal around mid-May 2024.</p>
<p>In a related incident that occurred on May 29, 2024, we observed another binary posing as a Microsoft Teams setup file, <code>TMSSetup.exe</code>, which was assigned a valid certificate issued to “Shanghai Ruikang Decoration Co., Ltd”.  As of May 30, 2024, that certificate has been revoked.</p>
<p>VirusTotal analysis of the binary <code>MSTeamsSetup_c_l_.exe</code> indicates it is associated with a malware family known as Oyster, dubbed <a href="https://exchange.xforce.ibmcloud.com/malware-analysis/guid:df2b52d89c5c0edfdf7bdaa6f67dd714">Broomstick</a> by IBM.</p>
<h3 id="whatisoysterbroomstick">What is Oyster/Broomstick?</h3>
<p>Oyster aka Broomstick aka CleanUpLoader is a family of malware first spotted in September of 2023 by researchers at IBM. While not much is known about the malware, it was delivered via a loader called <code>Oyster Installer</code>, which masqueraded as a browser installer. The installer was responsible for dropping the backdoor component, <code>Oyster Main</code>. <code>Oyster Main</code> was responsible for gathering information about the compromised host, handling communication with the hard-coded command-and-control (C2) addresses, and providing the capability for remote code execution.</p>
<p>In February, researchers on Twitter observed the same backdoor component and started to name the <code>Oyster Main</code> backdoor, <a href="https://x.com/RussianPanda9xx/status/1757932257765945478"><code>CleanUpLoader</code></a>.</p>
<p>In recent incidents, Rapid7 has observed <code>Oyster Main</code> being delivered without the <code>Oyster Installer</code>.</p>
<h3 id="technicalanalysis">Technical Analysis</h3>
<p>Initial analysis of the binary <code>MSTeamsSetup_c_l_.exe</code> revealed that two binaries were stored within the resource section. During execution, a function was observed using <code>FindResourceA</code> to locate the binaries, followed by <code>LoadResource</code> to access them. These binaries were then subsequently dropped into the Temp folder. We observed that the intended names of the two binaries dropped by <code>MSTeamsSetup_c_l_.exe</code> were <code>CleanUp30.dll</code> and <code>MSTeamsSetup_c_l_.exe</code> (the legitimate Microsoft Teams installer).</p>
<p>After dropping the binary <code>CleanUp30.dll</code> into the Temp directory, the program executes the DLL, passing the string <code>rundll32.exe %s,Test</code> to the function <code>CreateProcessA</code>, where <code>%s</code> stores the value <code>CleanUp30.dll</code>.</p>
<p><img src="https://blog.rapid7.com/content/images/2024/06/Figure-4---Execution-of--CleanUp30.dll-.JPG" alt="Figure-4---Execution-of--CleanUp30.dll-"><br>
<strong>Figure 4 - Execution of <code>CleanUp30.dll</code></strong></p>
<p>After the execution of <code>CleanUp30.dll</code>, the program proceeds to initiate the legitimate Microsoft Teams installer, <code>MSTeamsSetup_c_l_.exe</code>, also located within the Temp directory. This tactic is employed to avoid raising suspicion from the user.</p>
<h3 id="cleanup30dllanalysis">CleanUp30.dll Analysis</h3>
<p>During the execution of <code>CleanUp30.dll</code>, Rapid7 observed that the binary starts by attempting to create the hard coded mutual exclusion (mutex) <code>ITrkfSaV-4c7KwdfnC-Ds165XU4C-lH6R9pk1</code>. Mutex creation is often used by programs in order to determine if the program is already running another instance. If the program is already running, the program will terminate the new instance.</p>
<p>After creating the mutex, the binary determines its execution path by calling the function <code>GetModuleFilenameA</code>. The value is stored as a string and used as a parameter for the creation of a scheduled task, <code>ClearMngs</code>. The scheduled task is created using the function <code>ShellExecuteExW</code>, passing the following as the command line:</p>
<p><code>schtasks.exe /create /tn ClearMngs /tr &quot;rundll32 '&lt;location of binary&gt;\CleanUp30.dll',Test&quot; /sc hourly /mo 3 /f</code></p>
<p>The purpose of the scheduled task <code>ClearMngs</code> is to execute the binary <code>&lt;location of binary&gt;\CleanUp30.dll</code> with the exported function of <code>Test</code> using rundll32.exe every three hours.</p>
<p>After the creation of the scheduled task, the binary then proceeds to decode its C2 servers using a unique decoding function. The decoding function begins by taking in a string of encoded characters, and its length is in bytes. The decoding function then proceeds to read in each byte, starting from the end of the encoded string.</p>
<p><img src="https://blog.rapid7.com/content/images/2024/06/Figure-5---The-DLL-s-decoding-loop.jpg" alt="Figure-5---The-DLL-s-decoding-loop"><br>
<strong>Figure 5 - The DLL’s Decoding Loop</strong></p>
<p>Each byte of the encoded string is used as an index location to retrieve the decoded byte from a hard-coded byte map. A byte map is a byte array containing 256 bytes in a randomized order, one for each possible byte value from 1 to 256. Malware authors sometimes use this technique to obfuscate strings and other data. The iteration counter (i) used within the condition for the decoding loop is compared to half of the encoded string’s length as the decoding loop swaps two bytes at a time. The bytes of the encoded string are decoded and swapped beginning at the start and end bytes of the string and the decoding loop then progresses towards the center of the string from each end.</p>
<p>The loop swaps the bytes to reverse the decoded string, as the original plaintext strings stored in the malware were reversed prior to encoding. When the center of the string is reached, the decoding process is complete. Due to this algorithm, all the encoded strings that are passed must be of even length to avoid further processing. Immediately after the decoded string is loaded onto the stack, the malware then re-encodes the string using a similar loop.  The final result for the first decoded string is a carriage return line feed (CRLF) delimited list of C2 domains.</p>
<p>We constructed a Python script that can decode all the encoded strings contained within the <code>CleanUp.dll</code> binaries, including previous versions. The Python script can be found in our <a href="https://github.com/rapid7/Rapid7-Labs/blob/main/Malware%20Config%20Extractors/clean_extract.py">GitHub repository</a>.</p>

<h3 id="followonactivity">Follow-on Activity</h3>
<p>In one of the incidents Rapid7 observed, a PowerShell script was spawned following the execution of another version of <code>CleanUp30.dll</code>, <code>CleanUp.dll</code>. <code>CleanUp.dll</code>, similar to <code>CleanUp30.dll</code>,  was originally dropped by the other fake Microsoft Teams installer, <code>TMSSetup.exe</code>, which dropped the binary into the AppData/Local/Temp directory as well.</p>
<p><img src="https://blog.rapid7.com/content/images/2024/06/Figure-10---PowerShell-Command-Creating-.lnk-File--DiskCleanUp.lnk-.JPG" alt="Figure-10---PowerShell-Command-Creating-.lnk-File--DiskCleanUp.lnk-"><br>
<strong>Figure 10 - PowerShell Command Creating .lnk File <code>DiskCleanUp.lnk</code></strong></p>
<p>The purpose of the PowerShell script was to create a shortcut LNK file named <code>DiskCleanUp.lnk</code> within <code>C:\Users\&lt;User&gt;\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Startup\</code>. By doing so, this ensured that the LNK file <code>DiskCleanUp.lnk</code> would be run each time the user logged in. The shortcut LNK file was responsible for executing the binary <code>CleanUp.dll</code> using rundll32.exe, passing the export <code>Test</code>.<br>
Following the execution of the PowerShell script, Rapid7 observed execution of additional payloads:</p>
<ul>
<li>k1.ps1</li>
<li>main.dll</li>
<li>getresult.exe</li>
</ul>
