    <h1 class="text-center">Detecting evolving threats: NetSupport RAT campaign</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/christopher-morrison/">Christopher Morrison</a>
        </div>

        <br/>
        <time class="post-datetime" datetime="August 1, 2024 06:00">
            Thursday, August 1, 2024 06:00
        </time>

        <div class="m-3">
                        <a href="https://blog.talosintelligence.com/category/the-deep-dive-with-ntdr/" class="category primary-category">
                            The Deep Dive with NTDR
                        </a>
                            </div>
    </div>

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <ul><li>Cisco Talos is actively tracking multiple malware campaigns that utilize NetSupport RAT for persistent infections.&nbsp;</li><li>These campaigns evade detection through obfuscation and updates.&nbsp;</li><li><a href="https://snort.org/" rel="noreferrer noopener"><u>Snort</u></a> can provide a strong defense before this malware reaches endpoints.&nbsp;</li><li>In this first Deep Dive with NTDR, we explore how defenders can leverage Snort for the detection of evasive malware threats.&nbsp;</li></ul><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-3cf4f7e5-c0e7-49e0-b865-729d01c4a2c4.png" class="kg-image" alt="" loading="lazy" width="1600" height="962" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/07/data-src-image-3cf4f7e5-c0e7-49e0-b865-729d01c4a2c4.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/07/data-src-image-3cf4f7e5-c0e7-49e0-b865-729d01c4a2c4.png 1000w, https://blog.talosintelligence.com/content/images/2024/07/data-src-image-3cf4f7e5-c0e7-49e0-b865-729d01c4a2c4.png 1600w" sizes="(min-width: 720px) 720px"></figure><p>In November 2023, security vendors identified a new NetSupport RAT campaign that used fake browser updates from compromised and malicious websites to trick users into downloading a stager that downloads and invokes PowerShell commands to install the NetSupport manager agent onto the victimâs machine and establish persistence.&nbsp;</p><p>In January 2024, security researchers published <a href="https://www.esentire.com/blog/smartapesg-delivering-netsupport-rat" rel="noreferrer noopener">another analysis</a> of the same campaign, although with some differences in the initial JavaScript payload, which demonstrates a threat actor re-focusing on the obfuscation of the initial stager. There are also modifications observed in the agent installation including new paths for the randomized installation.&nbsp;</p><p>So, Cisco Talos followed suit with our own in-depth analysis. We identified multiple obfuscation and evasion techniques being used by the campaign and created appropriate detection to keep users protected. By identifying specific weaknesses in the campaignâs obfuscation techniques and <a href="https://github.com/Cisco-Talos/IOCs/tree/main/2024/08" rel="noreferrer">identifying indicators of compromise (IOCs)</a>, we can create highly accurate detection of this campaign. Talos is also in a unique position of having primarily open-source detection tools such as Snort and ClamAV. To highlight these capabilities, we will provide a detailed explanation of our detection methodology.&nbsp;</p><h2 id="campaign-history">Campaign history&nbsp;</h2><p>NetSupport Manager has been commercially available for remote device administration since 1989. Like many tools in the IT remote support industry, NetSupport Manager has been weaponized by threat actors who either cannot or do not wish to develop their own RAT. Adversaries first started using NetSuport for malicious purposes in 2017, and at this point, most security vendors widely recognize this software as a RAT. The 2020s' shift to remote work for many office workers marked an increased use of NetSupport RAT in more phishing and drive-by download campaigns, as well as being used alongside other loaders. This campaign is the most notable use of NetSupport RAT in recent years, with hundreds of known stager variants across dozens of domains used in a large-scale malicious advertising campaign.&nbsp;</p><h2 id="component-summary">Component summary&nbsp;</h2><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-99182cd2-3108-4ddf-9781-5905ad5ae54c.png" class="kg-image" alt="" loading="lazy" width="1523" height="786" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/07/data-src-image-99182cd2-3108-4ddf-9781-5905ad5ae54c.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/07/data-src-image-99182cd2-3108-4ddf-9781-5905ad5ae54c.png 1000w, https://blog.talosintelligence.com/content/images/2024/07/data-src-image-99182cd2-3108-4ddf-9781-5905ad5ae54c.png 1523w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Components in VirusTotal graph.&nbsp;</span></figcaption></figure><ol><li>Stage 1 is a JavaScript file that is downloaded from one of several malicious advertisements or compromised websites. Itâs an obfuscated downloader, keeping a Stage 2 JavaScript and PowerShell payload in memory, which is the actual meat of the dropper. Stage 1 is obfuscated and embedded within otherwise benign JavaScript libraries.&nbsp;</li><li>In Stage 2, a PowerShell script is run via JavaScript. Both are obfuscated, and the PowerShell is embedded in the JavaScript, and the JavaScript is embedded within large comment blocks of random ASCII hex. The PowerShell makes another HTTP GET to retrieve the base64-encoded ZIP. On receipt, the PowerShell extracts the payload into a semi-random path, starts execution, and then establishes basic registry persistence.&nbsp;</li><li>The end payload is a completely portable install of the commercial NetSupport Manager RAT utility. Some of the installs come with additional scripts or slight filename changes to avoid more narrow detection.&nbsp;</li></ol><h2 id="stage-1-javascript-stager">Stage 1: JavaScript stager&nbsp;</h2><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-846ace0b-3d10-414b-a861-196c05a43753.png" class="kg-image" alt="" loading="lazy" width="728" height="256" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/07/data-src-image-846ace0b-3d10-414b-a861-196c05a43753.png 600w, https://blog.talosintelligence.com/content/images/2024/07/data-src-image-846ace0b-3d10-414b-a861-196c05a43753.png 728w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">From 3b587d0c311e8ebc3bb104d564235c41ef8e64592c7419f17f48e0cee9ebc878.&nbsp;</span></figcaption></figure><p>The screenshot above shows the normalized inner payload for an older version of the JavaScript Stage 1. In this older sample, the actual malicious code is embedded within an otherwise benign JavaScript library. Several libraries are used, including jQuery, moment.js and SheetJS. However, this stage's actual malicious payload is barely obfuscated. The next stage download URL is in plain text. The call to âactiveXObjectâ to create the HTTP connection is in plain text. And similarly, the eval call is in plain text.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-ab795f2c-bd62-4256-a6f3-6a62fd4deeda.png" class="kg-image" alt="" loading="lazy" width="706" height="443" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/07/data-src-image-ab795f2c-bd62-4256-a6f3-6a62fd4deeda.png 600w, https://blog.talosintelligence.com/content/images/2024/07/data-src-image-ab795f2c-bd62-4256-a6f3-6a62fd4deeda.png 706w"><figcaption><span style="white-space: pre-wrap;">From 01d867d552a06bd778c812810a476441681c4bebabf967e80f8024b3856cb03e&nbsp;</span></figcaption></figure><p>Here, we have the normalized content of a very recent (first seen 2024.03.09) sample of the stager from the same campaign. Similarly to the older version, the malicious payload is wrapped in an otherwise benign library. Improving over the previous versions, the malicious payload within is further obfuscated with the open-source tool <a href="https://github.com/javascript-obfuscator/javascript-obfuscator" rel="noreferrer noopener">javascript-obfuscator</a>. This results in the removal of the plaintext âactiveXObjectâ and âeval,â however, the output pattern of JavaScript-obfuscator is highly identifiable, as every variable and function name is a random hexadecimal value prefixed by an underscore.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-0a0bef22-998f-42c2-b7dc-d2a1cda8511b.png" class="kg-image" alt="" loading="lazy" width="732" height="271" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/07/data-src-image-0a0bef22-998f-42c2-b7dc-d2a1cda8511b.png 600w, https://blog.talosintelligence.com/content/images/2024/07/data-src-image-0a0bef22-998f-42c2-b7dc-d2a1cda8511b.png 732w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">Deobfuscated 01d867d552a06bd778c812810a476441681c4bebabf967e80f8024b3856cb03e&nbsp;</span></figcaption></figure><p>The old and new samples shared two major features: All the obfuscation is pre-computed and not done on a per-download basis. This is observable through the fact that so many of the stages can be identified with common entry point functions after having been obfuscated with javascript-obfuscator. Also, the pre-obfuscated stagers are reused by stamping them with a new random download URL. This URL then ends up being in plain text as well so from identifying the highly unique function names from one stager, we can find many more and then automatically extract the download URLs from them statically, which is much faster than throwing them all in analysis sandboxes.&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-3eeb37ea-3125-4cb7-9391-05a0dc8ba058.png" class="kg-image" alt="" loading="lazy" width="604" height="138" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/07/data-src-image-3eeb37ea-3125-4cb7-9391-05a0dc8ba058.png 600w, https://blog.talosintelligence.com/content/images/2024/07/data-src-image-3eeb37ea-3125-4cb7-9391-05a0dc8ba058.png 604w"><figcaption><span style="white-space: pre-wrap;">Fast pattern-only rule for detection.&nbsp;</span></figcaption></figure><h3 id="stage-1-detection">Stage 1 detection&nbsp;</h3><p>From a detection standpoint, we can use Snort to leverage a fast_pattern-only file rule. This simple rule will only detect the given cluster of stagers being downloaded. However, since this is a fast_pattern-only rule on highly unique text, we can put it in the more general Snort policy configurations since there is little overhead to running this rule. Additionally, the Snort 3 file rule will do all the abstraction from other protocols that can carry files, such as FTP and SMTP, letting us cast our coverage net over a broader attack surface in case this file is being transmitted through more ways than just the known malicious advertisement.&nbsp;</p><h2 id="stage-2-javascript-powershell-dropper">Stage 2: JavaScript &amp; PowerShell dropper&nbsp;</h2><p>The second stage of the campaign is an in-memory payload downloaded by the on-disk JavaScript. This stage consists of a JavaScript function that then calls PowerShell.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-e42f83b8-8290-47fb-b8e4-e9117e889a30.png" class="kg-image" alt="" loading="lazy" width="678" height="129" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/07/data-src-image-e42f83b8-8290-47fb-b8e4-e9117e889a30.png 600w, https://blog.talosintelligence.com/content/images/2024/07/data-src-image-e42f83b8-8290-47fb-b8e4-e9117e889a30.png 678w"><figcaption><span style="white-space: pre-wrap;">From 6f3681cd91f7a19c1cf2699e1f9f7b550dfe46841ab5171124e79979fae5424a&nbsp;</span></figcaption></figure><p>&nbsp;The JavaScript content is padded on either side by large, repeating, random comment blocks. The actual JavaScript is quite simple, a slightly obfuscated call to ActiveXObject(âWScript.Shellâ), which allows the PowerShell command-line to be run.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-fb6e90dd-2124-4695-bfbc-d17a45ae2925.png" class="kg-image" alt="" loading="lazy" width="672" height="764" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/07/data-src-image-fb6e90dd-2124-4695-bfbc-d17a45ae2925.png 600w, https://blog.talosintelligence.com/content/images/2024/07/data-src-image-fb6e90dd-2124-4695-bfbc-d17a45ae2925.png 672w"><figcaption><span style="white-space: pre-wrap;">Deobfuscated from 6f3681cd91f7a19c1cf2699e1f9f7b550dfe46841ab5171124e79979fae5424a&nbsp;</span></figcaption></figure><p>The PowerShell invocation is where most of the dropperâs work happens. PowerShell is run with the command line flags â-Ex Bypassâ which sets the execution policy to âBypassâ, enabling unsigned execution, â-NoPâ which is short for â-NoPolicyâ and avoids the current userâs startup script, and â-Câ which is short for â-commandâ and will run the following command line string as a PowerShell script. The commands themselves are a straightforward download of the next stage, which is a base64-encoded ZIP file that contains a portable installation of NetSupport Manager Agent. After downloading the payload, it is base64-decoded, written to disk, and then extracted to the target install folder. A check is run to make sure âclient32.exe,â the main executable of the agent, exists in the output directory and then this file is run. Finally, a registry entry is made to establish user-level persistence of the agent on login.&nbsp;</p><p>Interestingly, this stage is not obfuscated with the same techniques as the Stage 1 file. The PowerShell file is only slightly obfuscated through random variable names and string concatenations, and the JavaScript obfuscation uses the same techniques in addition to the mentioned large random comment blocks. Because this obfuscation is so limited, we can rely on static signatures in this case. If future versions of this stager are further obfuscated, using normalized content buffers in Snort such as js_data can provide a consistent view into the payload. However, this payload has remained largely unchanged besides the payload URLs from November 2023 to March 2024.&nbsp;</p><h3 id="stage-2-detection">Stage 2 detection&nbsp;</h3><p>For Snort detection, we can use a simple series of content that matches the content of the PowerShell script. The unobfuscated registry key for persistence combined with the PowerShell flags makes a great rule for potential malware dropper detection. We can narrow this even further with the inclusion of the client32.exe filename so that we know we are only alerting on this campaign and similar uses of NetSupport RAT.&nbsp;</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-7b8a7182-702e-49ee-9da3-08a23686251e.png" class="kg-image" alt="" loading="lazy" width="882" height="194" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/07/data-src-image-7b8a7182-702e-49ee-9da3-08a23686251e.png 600w, https://blog.talosintelligence.com/content/images/2024/07/data-src-image-7b8a7182-702e-49ee-9da3-08a23686251e.png 882w" sizes="(min-width: 720px) 720px"></figure><p>An additional trick would be the use of an <a href="https://docs.snort.org/rules/headers/service_rules" rel="noreferrer noopener">HTTP service rule</a>. In Snort 3, the service inspector can identify known services regardless of the port they are on, in contrast to Snort 2 where you will need to define HTTP_PORTS ahead of time. Regardless of what port the attacker is hosting the HTTP payload on, our rule can detect the malicious content.&nbsp;</p><p>This Snort rule identifies and prevents the current iteration of dropper payloads. A more holistic approach to defense would also include behavioral detection. Since we can observe this dropperâs behavior regardless of any text obfuscation, we will effectively forbid this technique on any protected endpoint and force the threat actor to change tactics.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/07/data-src-image-6afd824f-63b5-426c-9d0e-f8c5e7023357.png" class="kg-image" alt="" loading="lazy" width="536" height="290"><figcaption><span style="white-space: pre-wrap;">Example Sigma detection.&nbsp;</span></figcaption></figure><p>&nbsp;The Sigma behavioral rule language can encapsulate this detection quite well. This rule can detect most installs but will fail to identify obfuscations through renames or different registry keys.&nbsp;</p><h2 id="obfuscated-netsupport-manager-download">Obfuscated NetSupport manager download&nbsp;</h2><p>Retreading a little bit, the PowerShell invocation downloads a base64-encoded ZIP file of an entire installation of NetSupport Manager agent. For NetSupport Manager and most other legitimate software packages, this is a highly unusual way for the package to be distributed. Most applications are installed via an MSI package or an EXE installer that downloads the full application without obfuscation. The client binaries are not obfuscated by the threat actor since obfuscating an existing compiled binary is more challenging than obfuscating the source code scripts identified in stage 1 and stage 2 files. Because of this, the unique DLLs that encompass most of the NetSupport Manager Clientâs actual logic are not renamed.&nbsp;</p><p>It might seem we will have to stream decode the base64 content and unzip it in memory to scan for malicious file contents; we can exploit two specific features of the ZIP in base64 format. The highly unique DLL names will all be close together in the ZIP archive central directory, the structure that holds the information on what files are in the archive. Second, base64 is not a random encoding, and we can pre-compute all three possible base64-encoded strings that represent any one given input. <a href="https://gchq.github.io/CyberChef/#recipe=Show_Base64_offsets('A-Za-z0-9%2B/%3D',true,'Raw')&amp;input=c3VzcGljaW91cw" rel="noreferrer noopener">CyberChef has an excellent demonstration </a>of how this works. This method involves the possible two-bit alignment positions within the six-bit encoding size of a given base64 character. Putting all this together means that we can pre-compute all the combinations of the base64-encoded DLL names as high-speed rules that are only looking for static content matches.&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/09/Unknown-2.png" class="kg-image" alt="" loading="lazy" width="627" height="181" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/09/Unknown-2.png 600w, https://blog.talosintelligence.com/content/images/2024/09/Unknown-2.png 627w"><figcaption><span style="white-space: pre-wrap;">Base64 pre-encoded matches in a Snort rule.</span></figcaption></figure><p>The threat actors using NetSupport RAT in this campaign are typically looking for the fastest way to get a RAT agent into deployment. The threat actors who use NetSupport Manager are not putting a strong amount of effort into obfuscation of droppers, let alone the agents, so this is an effective rule for their non-standard download paths.&nbsp;&nbsp;</p><h2 id="additional-snort-detections">Additional Snort detections&nbsp;</h2><p>Talos has existing Snort coverage for NetSupport Manager Agentâs network activity leveraging SID <a href="https://snort.org/rule_docs/1-44678" rel="noreferrer noopener">44678</a>, which was created back in 2017. We also expanded this coverage in 2020, with the creation of SIDs <a href="https://snort.org/rule_docs/1-53541" rel="noreferrer noopener">53539 - 53544 </a>as the usage by malicious actors increased. For the latest campaign, we increased our coverage through a Snort feature that was not available before. <a href="https://docs.snort.org/rules/headers/file_rules" rel="noreferrer noopener">File Rules</a> are a feature only available in Snort 3 that allows the abstraction of file contents from the protocols that carry them, allowing us to write wider-reaching signatures.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/09/Unknown-3.png" class="kg-image" alt="" loading="lazy" width="414" height="101"><figcaption><span style="white-space: pre-wrap;">Resource Information from client32.exe detailing the NetSupport&nbsp;Manager product.&nbsp;</span></figcaption></figure><p>As the NetSupport RAT files are not typically obfuscated by threat actors and it is a legitimate commercial product with many specialized features, there are many opportunities to create detection against the files. For example, in both the EXE and DLLs, the full manufacturer and product information for NetSupport Manager Client is detailed.&nbsp;</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/2024/09/Unknown-3-1.png" class="kg-image" alt="" loading="lazy" width="752" height="233" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/09/Unknown-3-1.png 600w, https://blog.talosintelligence.com/content/images/2024/09/Unknown-3-1.png 752w" sizes="(min-width: 720px) 720px"></figure><p>As other legitimate software binaries are unlikely to include the full metadata for NetSupport Manager, this gives us a great signature to work with that lets us cleanly identify the binaries without any heavy reverse engineering needed. Although Snort does not have a modifier for Unicode strings, Unicode text byte strings can be leveraged for detection. Once again, we can use the âfileâ rule target to throw our detection net over a much larger surface than just HTTP downloads. Other campaigns using NetSupport Manager RAT will be observable from this rule through Snortâs ability to inspect file streams from multiple protocols.&nbsp;</p><p>Although NetSupport RAT continues to be used by malicious campaigns, its widespread use hinders those threat actors who are not willing to dedicate the resources to develop their own tools or more evasive techniques. And since this campaign is still active and updated, we may still observe these techniques being developed and deployed by this threat actor. For now, this gives us an example of when a threat is persistent but not advanced.&nbsp;</p><h1 id="indicators-of-compromise">Indicators of Compromise&nbsp;</h1><p>Indicators of Compromise associated with this threat can be found <a href="https://github.com/Cisco-Talos/IOCs/tree/main/2024/08"><u>here</u></a>. </p>
