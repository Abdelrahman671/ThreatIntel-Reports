<h2><span style="	font-weight: 400;
">Dero Cryptojacking Campaign Targets Kubernetes</span></h2>
<p><span style="	font-weight: 400;
">It has been estimated that the </span><a href="https://www.forbes.com/sites/billybambrough/2022/08/21/1-trillion-crypto-knife-edge-now-hinged-on-a-fed-bombshell-after-200-billion-bitcoin-ethereum-bnb-xrp-solana-cardano-and-dogecoin-price-crash/?sh&#61;2edd9f34e2f8"><span style="	font-weight: 400;
">crypto crash</span></a><span style="	font-weight: 400;
"> of last year undercut a cryptojacking campaign’s monetary reward by 50-90%. Dero, which offers larger rewards and provides the same or better anonymizing features, is a perfect match for attackers.</span></p>
<p><span style="	font-weight: 400;
">In February 2023, CrowdStrike observed the first-ever Dero cryptojacking campaign targeting Kubernetes infrastructure. Attackers carefully targeted Kubernetes clusters on non-standard ports by scanning and identifying exposed vulnerable Kubernetes clusters with authentication set as </span><code><span style="	font-weight: 400;
">--anonymous-auth&#61;true</span></code><span style="	font-weight: 400;
">, which allows anonymous access to the Kubernetes API. A user with sufficient privileges who runs ”Kubectl proxy” can unintentionally expose a secure Kubernetes API on the host where kubectl is running, which is a less obvious way to expose the secure Kubernetes cluster bypassing authentication. Kubernetes out-of-the box doesn’t allow anonymous access to the Kubernetes control plane API. But the delayed secure-by-default decision and number of ways Kubernetes can be accidentally exposed still created a legacy of exposed systems on the internet. Attackers used the following U.S.-based IPs to scan and deploy initial payloads.</span></p>
<ul><li><code><span style="	font-weight: 400;
">209&lt;.&gt;141&lt;.&gt;32&lt;.&gt;72</span></code></li><li><code><span style="	font-weight: 400;
">209&lt;.&gt;141&lt;.&gt;42&lt;.&gt;48</span></code></li><li><code><span style="	font-weight: 400;
">205&lt;.&gt;185&lt;.&gt;124&lt;.&gt;121</span></code></li></ul>
<p><span style="	font-weight: 400;
">Figure 2 shows attempts at targeting the Kubernetes cluster over a one-week period (February 3-10, 2023).</span></p>
<div class="wp-caption" id="attachment_156675"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-6.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-156675 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-6.png" alt width="2410" height="670" /></a>Figure 2. IPs targeting a honeypot Kubernetes cluster over a one-week period (February 3-10, 2023) (Click to enlarge)</div>
<p><span style="	font-weight: 400;
">Figure 3 shows the general flow of the attack, where an attacker on the internet finds an exposed vulnerable Kubernetes cluster. After initial interaction with the Kubernetes API, the attacker deploys a Kubernetes </span><a href="https://kubernetes.io/docs/concepts/workloads/controllers/daemonset.html"><span style="	font-weight: 400;
">DaemonSet</span></a><span style="	font-weight: 400;
"> named “</span><code><span style="	font-weight: 400;
">proxy-api</span></code><span style="	font-weight: 400;
">”. The DaemonSet deploys a malicious pod on each node of the Kubernetes cluster. This helps attackers engage resources of all of the nodes at the same time to run a cryptojacking operation. The mining efforts by the pods are contributed back to a community pool, which distributes the reward (i.e., Dero coin) equally among its contributors through their digital wallet.</span></p>
<p><span style="	font-weight: 400;
">Let’s take a closer look at the campaign.</span></p>
<div class="wp-caption" id="attachment_156676"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-7.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-156676 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-7.png" alt width="1600" height="1100" /></a></div>
<div class="wp-caption">Figure 3. Campaign to attack a Kubernetes cluster to mine Dero (Click to enlarge)</div>
<h3><span style="	font-weight: 400;
">Use of Docker Hub Image with “Pause” Binary</span></h3>
<p><span style="	font-weight: 400;
">The Docker image used in the cryptojacking operation is hosted by Docker Hub for easy public access. Figure 4 shows the image “</span><code><span style="	font-weight: 400;
">pauseyyf/pause:latest</span></code><span style="	font-weight: 400;
">” was uploaded in January 2023 and has over 4,000 pulls at the time of this writing. The number of pulls implies the scope of this campaign and how many potential miner instances have been deployed. Closer analysis reveals that it is a CentOS 7 base image, where attackers added two files: “</span><code><span style="	font-weight: 400;
">entrypoint.sh</span></code><span style="	font-weight: 400;
">” and “</span><code><span style="	font-weight: 400;
">pause</span></code><span style="	font-weight: 400;
">”.</span></p>
<div class="wp-caption" id="attachment_156677"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-8.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-156677" src="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-8.png" alt width="550" height="269" /></a>(Click to enlarge)</div>
<div class="wp-caption" id="attachment_156678"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-9.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-156678" src="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-9.png" alt width="550" height="162" /></a></div>
<div class="wp-caption">Figure 4. Malicious Docker image on Docker Hub with added binary “pause” and script “entrypoint.sh” (Click to enlarge)<br/>
</div>
<p><span style="	font-weight: 400;
">Figure 5 shows the content of the script “</span><code><span style="	font-weight: 400;
">entrypoint.sh</span></code><span style="	font-weight: 400;
">”, which is the actual entrypoint of the malicious image. Once the pod is deployed by the DaemonSet, the script tries to run a “</span><code><span style="	font-weight: 400;
">pause</span></code><span style="	font-weight: 400;
">” binary with a wallet address and mining pool as an argument, which signifies the “</span><code><span style="	font-weight: 400;
">pause</span></code><span style="	font-weight: 400;
">” binary being an actual </span><a href="https://github.com/deroproject/derohe/releases/tag/Release114"><span style="	font-weight: 400;
">Dero coin miner</span></a><span style="	font-weight: 400;
">. This campaign used a community pool — i.e., </span><code><span style="	font-weight: 400;
">community-pools&lt;.&gt;mysrv&lt;.&gt;cloud:10300</span></code><span style="	font-weight: 400;
"> —</span> <span style="	font-weight: 400;
">which pays out for miniscule contributions to be made to the pool in the associated digital wallet.</span></p>
<p><span style="	font-weight: 400;
">In a legitimate Kubernetes deployment, “</span><a href="https://docs.mirantis.com/mke/3.4/ref-arch/pause-containers.html"><span style="	font-weight: 400;
">pause</span></a><span style="	font-weight: 400;
">” containers are used by Kubernetes to bootstrap a pod. Attackers may have used this name to blend in to avoid obvious detection.</span></p>
<div class="wp-caption" id="attachment_156726"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/03/Screenshot-2023-03-15-at-5.18.42-AM.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-156726" src="/content/dam/crowdstrike/www/en-us/wp/2023/03/Screenshot-2023-03-15-at-5.18.42-AM.png" alt width="600" height="121" /></a></div>
<div class="wp-caption">Figure 5. Script “entrypoint.sh” initializing Dero miner with campaign wallet address (Click to enlarge)</div>
<h3><span style="	font-weight: 400;
">Use of Kubernetes DaemonSet</span></h3>
<p><span style="	font-weight: 400;
">This campaign initially listed all nodes on the vulnerable Kubernetes cluster, as shown in Figure 6, where attackers used a GET query to the Kubernetes API, and once the adequate response was received, the attackers went ahead and created a DaemonSet.</span></p>
<p><span style="	font-weight: 400;
">Figure 7 shows the yaml used by the attacker to create the DaemonSet once the vulnerable Kubernetes cluster is found. The attacker creates it under a default Kubernetes namespace “</span><code><span style="	font-weight: 400;
">kube-system</span></code><span style="	font-weight: 400;
">” and names it “</span><code><span style="	font-weight: 400;
">proxy-api</span></code><span style="	font-weight: 400;
">” to masquerade the pod. Additionally, the attacker sets pod DNS servers to a public IP (i.e., Google’s 8.8.8.8) and marks ”<code>restartPolicy: Always</code>” in case the pod on any of the nodes crashes.</span></p>
<div class="wp-caption" id="attachment_156679"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-10.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-156679" src="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-10.png" alt width="550" height="153" /></a></div>
<div class="wp-caption">Figure 6. Campaign listing all nodes on the Kubernetes cluster (Click to enlarge)</div>
<div class="wp-caption" id="attachment_156732"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/03/Screenshot-2023-03-15-at-5.30.14-AM.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-156732" src="/content/dam/crowdstrike/www/en-us/wp/2023/03/Screenshot-2023-03-15-at-5.30.14-AM.png" alt width="600" height="641" /></a></div>
<div class="wp-caption">Figure 7. Kubernetes DaemonSet yaml used by campaign</div>
<h3><span style="	font-weight: 400;
">Silently Running Operation</span></h3>
<p><span style="	font-weight: 400;
">Once the vulnerable Kubernetes cluster was compromised, attackers made no attempts to pivot either by moving laterally to attack further resources or scan the internet for discovery, which is a common pattern among many cryptojacking campaigns observed in the wild.</span></p>
<p><span style="	font-weight: 400;
">Furthermore, attackers made no attempts to delete or disrupt cluster operation. Instead, they deployed a DaemonSet to mine Dero by masquerading the DaemonSet name to “</span><code><span style="	font-weight: 400;
">proxy-api</span></code><span style="	font-weight: 400;
">” and miner’s name to “</span><code><span style="	font-weight: 400;
">pause</span></code><span style="	font-weight: 400;
">”, which are common terms in Kubernetes logs.</span></p>
<p><span style="	font-weight: 400;
">These focused behaviors seem to clarify the intent of this campaign, which is that the attackers are solely attempting to mine for Dero. Thus, we can suspect a financially motivated cryptojacking actor is behind this campaign.</span> <span style="	font-weight: 400;
">The Dero wallet address (click to scroll)</span></p>
<pre>
<code>deroi1qyr8wnk9aw9lel0xcufdj98cqtd3lc5y84nhl679nm3wknaz0ad6xq9pvfz92xnjm0ypwc9rt0v</code>
</pre>
<p><span style="	font-weight: 400;
">seems to be mining on mainnet, but due to privacy and anonymity features of Dero, it’s hard to track the balance of such a wallet address, and the overall value of this mining operation to date. According to Dero documentation, a wallet address along with a </span><a href="https://docs.dero.io/#dero-innovations"><span style="	font-weight: 400;
">TX private key</span></a><span style="	font-weight: 400;
"> generated from the same wallet is needed to look at transactions and the balance on the Dero blockchain.</span></p>
<h2><span style="	font-weight: 400;
">Modified Monero Cryptojacking Campaign Tries to Take On Dero Cryptojacking Campaign</span></h2>
<p><span style="	font-weight: 400;
">After discovering the Dero cryptojacking campaign, CrowdStrike subsequently detected in February 2023 a modified Monero campaign attacking Kubernetes and kicking out the Dero miner to mine Monero instead.</span></p>
<p><span style="	font-weight: 400;
">Once the misconfigured Kubernetes API is found, the modified Monero campaign first deliberately tries to delete existing DaemonSets named “<code>proxy-api</code>”. This signifies the Monero campaign is aware of the ongoing Dero cryptojacking campaign, and the Monero attacker wants to knock off the existing Dero cryptojacking DaemonSet before the campaign takes over the cluster and uses all of its resources.</span></p>
<div class="wp-caption" id="attachment_156680"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-11.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-156680" src="/content/dam/crowdstrike/www/en-us/wp/2023/03/Picture1-11.png" alt width="550" height="122" /></a></div>
<div class="wp-caption">Figure 8. Monero campaign deleting the Dero DaemonSet “proxy-api” (Click to enlarge)</div>
<p><span style="	font-weight: 400;
">Apart from “<code>proxy-api</code>”. the Monero campaign tries to delete the following DaemonSets. This signifies the new campaign may be aware of other campaigns, which are deploying the following DaemonSets:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">api-proxy</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">k8s-proxy</span></li></ul>
<p><span style="	font-weight: 400;

Furthermore, it creates a cronjob to trigger a payload rather than immediately executing malicious scripts by overwriting cron files on the host file system. The following table shows the differences between the two campaigns.</span></p>

<h2><span style="	font-weight: 400;
">Conclusion</span></h2>
<p><span style="	font-weight: 400;
">As Kubernetes has become the most popular container orchestrator in the world, attackers have opportunistically targeted Kubernetes and Docker misconfigurations, design weaknesses and zero-day vulnerabilities like </span><a href="/en-us/blog/cr8escape-new-vulnerability-discovered-in-cri-o-container-engine-cve-2022-0811/"><span style="	font-weight: 400;
">cr8escape</span></a><span style="	font-weight: 400;
">, which was discovered by CrowdStrike.</span></p>
<p><span style="	font-weight: 400;
">CrowdStrike discovered the first-ever Dero cryptojacking operation targeting Kubernetes infrastructure. Dero is a relatively new cryptocurrency that is claimed to have better anonymity features. The Dero cryptojacking campaign targeted exposed Kubernetes attack surface, in which attackers ran silently using three U.S.-based server IPs to carefully scan and identify vulnerable Kubernetes clusters and deploy payload. At the same time, CrowdStrike observed another Monero campaign, which is modified and is aware of the Dero campaign and targeting the same attack surface but using a more sophisticated approach. Both campaigns are trying to find undiscovered Kubernetes attack surfaces and are battling it out.</span></p>

">The Falcon platform continues to protect organizations of all sizes from sophisticated breaches, including cryptojacking campaigns, with industry-leading CNAPP capabilities.</span></p>
