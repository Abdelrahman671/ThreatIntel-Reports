<div class="cmp-text ">
    <h2 id="introduction">Introduction</h2>
<p>In early 2020, a new sophisticated malware for Android called FluBot began to appear. On infected devices, the malware can take full remote control of the device; access victim’s contact lists; send, intercept, and hide SMS messages; log the victim’s keystrokes; steal one-time passcodes; collect personal information; carry out overlay attacks and more. Originally, the malware authors mainly targeted Spanish banks but later expanded their targets to include Australian, German, Polish, and UK banks (HSBC, Santander, Lloyds, Halifax, and others).</p>
<p>The figure below shows an example of the command and control (C2) panel, which contains detailed statistics about victims’ devices.</p>

<h3 id="flubots_abuse_of_androids_accessibility_service">FluBot’s Abuse of Android’s Accessibility Service</h3>
<p><b>FluBot isn’t the first Android malware to abuse Android's Accessibility Service.</b> FluBot appears to have many of the main features of other contemporary Android banking malware:</p>
<ul class="hex-dark rte-custom">
<li><span class="hex-dark rte-custom">Full remote control of an Android device</span></li>
<li><span class="hex-dark rte-custom">Overlay attacks against multiple bank applications to steal login credentials and credit card information (see overlay images below for HSBC and Halifax)</span></li>
<li><span class="hex-dark rte-custom">Send/intercept/hide SMS messages and notifications</span></li>
<li><span class="hex-dark rte-custom">Enabling key logging (screen text-grabbing) functionality</span></li>
<li><span class="hex-dark rte-custom">Ability to steal one-time password codes</span></li>
</ul>

<h2 id="key_points">Key Points</h2>
<p>This section includes details of the main functionality of FluBot and reviews some of the interesting commands FluBot uses in version 5.0 and version 4.9.</p>
<h3 id="static_analysis">Static Analysis</h3>
<p>FluBot uses a number of sophisticated techniques to make it difficult for researchers and security solutions to achieve their goals.</p>
<p>To complicate static analysis, the malware implements many techniques, several of which are described next.</p>
<h4>String Encryption</h4>
<p>Any significant strings in the malware are encrypted using a custom encryption scheme. Each class contains a function named “ that is responsible for encrypting suspicious strings within the class. It uses XOR encryption and utilizes a pre-defined list of hex numbers that are unique to each class. This list contains the encrypted strings.</p>
<p>The first two arguments subtracted results in the length of the string; the first argument is the offset of the pre-defined list and arg6 is the XOR key. (<a href="/content/dam/f5-labs-v2/article/articles/threats/27--2022-jan-mar/20220106_flubot/string_decryption.py.txt">A tool for extracting the strings can be downloaded here</a>.)</p>

</div>

    
</div>
<div class="a02-image section">




<figure id="image-701351db4e" class="img-full a02-image__image">
    
    <link rel="preload" as="image" href="/content/dam/f5-labs-v2/article/articles/threats/27--2022-jan-mar/20220106_flubot/Figure_8.png"/>
    <a class="img-link" href="/content/dam/f5-labs-v2/article/articles/threats/27--2022-jan-mar/20220106_flubot/Figure_8.png" data-lightbox="Image preview">
        <img data-src="/content/dam/f5-labs-v2/article/articles/threats/27--2022-jan-mar/20220106_flubot/Figure_8.png" data-component-type="platter" class="js-bg-lozad"/>
    </a>
    
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <h4>MultiDex</h4> 
  <p>In Android there are limitation on DEX files. For example, you can’t reference more than 64KB of methods within a single DEX file. To overcome this limitation, the developer can set the compiler to split the DEX file into smaller DEX files and then use MultiDex to load those additional DEX files to the App.</p> 
  <p>MultiDex is a popular and valid Android module that is used to support MultiDEX files.</p> 
  <p>FluBot uses an implementation of MultiDex to hide its malicious code from static analyzers and reversers. It uses a hard-coded configuration that contains the location of the encrypted dex, location of the decrypted dex, their corresponding name, suffix, and folder, as well as a key for decryption represented as a string.</p> 
 

</div>

    
</div>
<div class="a02-image section">




<figure id="image-9a31c82be2" class="img-full a02-image__image">
    
    <link rel="preload" as="image" href="/content/dam/f5-labs-v2/article/articles/threats/27--2022-jan-mar/20220106_flubot/Figure_9.png"/>
    <a class="img-link" href="/content/dam/f5-labs-v2/article/articles/threats/27--2022-jan-mar/20220106_flubot/Figure_9.png" data-lightbox="Image preview">
        <img data-src="/content/dam/f5-labs-v2/article/articles/threats/27--2022-jan-mar/20220106_flubot/Figure_9.png" data-component-type="platter" class="js-bg-lozad"/>
    </a>
    
</figure></div>
<div class="a03-text text section">
<div class="cmp-text ">
    
 
 
  <p>An encrypted DEX file is stored in the APK’s assets, decrypted, stored as a classes.dex file, archived in a Zip file, and loaded in runtime.</p> 
 

</div>

    
</div>
<div class="a02-image section">

  <h4>DNS Tunneling over HTTPS: Updated in Version 5.0</h4> 
  <p>In version 4.9, FluBot resolved the IP addresses of DGAs and communicated directly with the server using HTTPS port 443. In FluBot version 5.0, the authors completely changed the way the malware communicates with the C2 server. In version 5.0 (and on some late builds of version 4.9), it communicates with the C2 through DNS Tunneling over HTTPS. Here’s how it works:</p> 
  <ol class="list-paragraph"> 
   <li>The attacker sets a nameserver that will act as a C2 server and receive and send data through the DNS protocol.</li> 
   <li>FluBot uses DNS resolving providers such as Google, Cloudflare, and Alibaba to infiltrate and exfiltrate the data to the C2 server.</li> 
  </ol> 
 

</div>

    
</div>

  <h4>Encrypted Communication via HTTP using RSA+RC4 Encryption</h4> 
  <p>In versions previous to 5.0, communication with the C2 is encrypted with RSA + RC4.</p> 
  <p>Each sent and received message starts with a header that contains bot id + RC4 key, which is then RSA-encrypted with a hard-coded public key and Base64 encoded. Then the payload\command is RC4-encrypted with the RC4 key in the header.</p> 
  <p>The C2 also responds with the same header that gets verified by the malware to make sure the C2 received the header correctly and with the Base64-encoded RC4-encrypted command\data.</p> 
  <p>The communication is done over HTTP and sent to <i>http://{dga}/p.php</i>.</p>

<h3 id="flubot_commands">FluBot Commands</h3>
<p>FluBot authors continue to use sophisticated commands as the malware evolves.<i></i></p>
<h4>UPDATE_DNS_SERVERS: New in Version 5.0</h4>
<p>A new command featured in version 5.0 allows FluBot to update DNS resolvers in the malware’s configuration. The C2 communication in version 5.0 is done through a DNS-Tunneling-over-HTTPS technique. This feature enables the attacker to change its DNS resolvers once they put the attacker’s nameservers on the denylist.</p>
<h4>UPDATE_ALT_SEED</h4>
<p>New in version 5.2, allows the attacker to update the DGA seed remotely.</p>
<h4>NOTIF_INT_TOGGLE - Notification Interception</h4>
<p>FluBot has abilities to log and intercept any notification the phone receives. This is primarily used to steal one-time passcodes and sensitive information.</p>

  <h4>DISABLE_PLAY_PROTECT</h4> 
  <p>FluBot uses Android’s Accessibility Service to disable Google Play Protect, a safety check mechanism for applications installed on a device. The malware starts an intent by redirecting to the Google Play Protect settings page and then clicks on the necessary buttons to disable it.</p>

<div class="cmp-text ">
    <h2 id="mitigating_flubot">Mitigating FluBot</h2>
<p>David Warburton, principal threat research evangelist with F5 Labs, offers the following suggestions for mitigating FluBot.</p>
<h3 id="prevent">Prevent</h3>
<p>FluBot relies on tricking the user into downloading a trojan hosted on an attacker-controlled server. Android phones will, by default, prevent installation from outside of the Google Play store, though attackers know this and coach the victim into bypassing this restriction. A combined approach, using people and technology, should be taken to mitigate installation of apps from untrusted sources.</p>
<p>First, educate staff members and/or customers that they should rarely, if ever, install Android apps from outside of the Google Play store. They should understand that any message asking them to bypass security for the benefit of installing an app should not be trusted.</p>
<p>Secondly, if managed corporate mobile devices are in use by employees, consider mobile device management (MDM) solutions, which can lock down the ability to install apps from outside of the Google Play store.</p>
<p>It is worth noting that since FluBot is able to capture SMS messages and grab images from the screen, multifactor authentication (MFA) solutions may not prevent the abuse of stolen credentials since one-time passwords can be stolen from the SMS message or viewed on the victim’s screen. MFA solutions which, instead, make use of push notifications to approve login attempts <i>may</i> have some more success in preventing abuse but since FluBot is able to control an Android device using the Accessibility Service, this method may also be inadequate.</p>
<h3 id="detect">Detect</h3>
<p>Preventing infection by this kind of trojan can be difficult, if not impossible, for non-corporate or unmanaged devices. It’s therefore critical to be able to identify when stolen credentials are being used by fraudsters and, additionally, detect when automated bots are launching attacks.</p>
<p>Modern application security solutions offer the ability to detect authentication abuse, including <a href="/content/f5-labs-v2/en/archive-pages/education/what-is-credential-stuffing-">credential stuffing</a> attacks. Unlike password brute forcing which can be trivial to mitigate, credential stuffing can be difficult to detect since malicious requests appear quite genuine in nature. Effective solutions should be able to match a supplied password against a list of known stolen credentials and combine this with the ability to detect when requests come from a compromised devices that may be part of a <a href="/labs/articles/threat-intelligence/how-credential-stuffing-bots-bypass-defenses">botnet</a>.</p>
<h3 id="identify">Identify</h3>
<p>With the ability to fully control the victim’s device and hide incoming SMS messages, it is essential that out-of-band methods are used to alert victims to activity on their account. Using, for example, email to alert the user of a new login or suspicious transaction, may be the only way to reliably inform them of malicious activity on their account.</p>
