<meta name="twitter:description" content="KEY TAKEAWAYS StormBamboo successfully compromised an internet service provider (ISP) in order to poison DNS responses for target organizations. Insecure software update mechanisms were targeted to surreptitiously install malware on victim machines running macOS and Windows. Malware deployed by StormBamboo includes new variants of the MACMA malware. Analysis of the newest versions of MACMA shows converged development of the MACMA and GIMMICK malware families. Post-exploitation activity included deployment of the malicious browser extension RELOADEXT to exfiltrate victim mail data. In mid-2023, Volexity detected and responded to multiple incidents involving systems becoming infected with malware linked to StormBamboo (aka Evasive Panda, and previously tracked by Volexity under “StormCloud”). In those incidents, multiple malware families were found being deployed to macOS and Windows systems across the victim organizations’ networks. The infection vector for this malware was initially difficult to establish but later proved to be the result of a DNS poisoning attack [&hellip;]">

<h5><span style="color: #4e9dd6;">KEY TAKEAWAYS</span></h5>
<ul>
<li><em>StormBamboo successfully compromised an internet service provider (ISP) in order to poison DNS responses for target organizations.</em></li>
<li><em>Insecure software update mechanisms were targeted to surreptitiously install malware on victim machines running macOS and Windows.</em></li>
<li><em>Malware deployed by StormBamboo includes new variants of the MACMA malware.</em></li>
<li><em>Analysis of the newest versions of MACMA shows converged development of the MACMA and GIMMICK malware families.</em></li>
<li><em>Post-exploitation activity included deployment of the malicious browser extension RELOADEXT to exfiltrate victim mail data.</em></li>
</ul>
<p style="font-weight: 400;">In mid-2023, Volexity detected and responded to multiple incidents involving systems becoming infected with malware linked to StormBamboo (aka Evasive Panda, and previously <a href="https://www.volexity.com/blog/2022/03/22/storm-cloud-on-the-horizon-gimmick-malware-strikes-at-macos/">tracked by Volexity under “StormCloud</a>”). In those incidents, multiple malware families were found being deployed to macOS and Windows systems across the victim organizations’ networks.</p>
<p style="font-weight: 400;">The infection vector for this malware was initially difficult to establish but later proved to be the result of a DNS poisoning attack at the internet service provider (ISP) level. Volexity determined that StormBamboo was altering DNS query responses for specific domains tied to automatic software update mechanisms. StormBamboo appeared to target software that used insecure update mechanisms, such as HTTP, and did not properly validate digital signatures of installers. Therefore, when these applications went to retrieve their updates, instead of installing the intended update, they would install malware, including but not limited to <a href="https://blog.google/threat-analysis-group/analyzing-watering-hole-campaign-using-macos-exploits/">MACMA</a> and POCOSTICK (aka <a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.mgbot">MGBot</a>). The overall workflow used by the attackers is similar to a previous incident investigated by Volexity that was attributed to DriftingBamboo, a threat actor which is possibly related to StormBamboo.</p>
<p style="font-weight: 400;">In April 2023, ESET published a <a href="https://www.welivesecurity.com/2023/04/26/evasive-panda-apt-group-malware-updates-popular-chinese-software/">blog post</a> about a malware family that Volexity has tracked since 2018 as POCOSTICK. ESET did not have direct evidence but proposed the most likely source of infection was an adversary-in-the-middle (AiTM). Volexity can now confirm this scenario in a real-world case and prove the attacker was able to control the target ISP’s DNS infrastructure in order to modify DNS responses in the victim organization’s network.</p>
<p style="font-weight: 400;">This blog post explains the infection vector and gives an example of where an automatic update was abused by StormBamboo. Note that this is just one example; the threat actor has modified installation workflows for a range of applications whose update mechanisms are vulnerable to this type of attack.</p>
<h2>Overview</h2>
<p style="font-weight: 400;">During one incident investigated by Volexity, it was discovered that StormBamboo poisoned DNS requests to deploy malware via an HTTP automatic update mechanism and poison responses for legitimate hostnames that were used as second-stage, command-and-control (C2) servers.</p>
<p style="font-weight: 400;">The DNS records were poisoned to resolve to an attacker-controlled server in Hong Kong at IP address <code>103.96.130[.]107</code>. Initially, Volexity suspected the initial victim organization’s firewall may have been compromised. However, further investigation revealed the DNS poisoning was not performed within the target infrastructure, but further upstream at the ISP level. Volexity notified and worked with the ISP, who investigated various key devices providing traffic-routing services on their network. As the ISP rebooted and took various components of the network offline, the DNS poisoning immediately stopped. During this time, it was not possible to pinpoint a specific device that was compromised, but various components of the infrastructure were updated or left offline and the activity ceased.</p>
<p style="font-weight: 400;">This is not the first case where Volexity has encountered an attacker utilizing DNS poisoning to facilitate initial access to a target network. In the <a href="https://www.volexity.com/wp-content/uploads/2023/06/Volexity-Cyber-Session-May-2023-FirewallZeroDayInvestigations.pdf">May 2023 Cyber Session</a>, Volexity presented details of a malware family it calls CATCHDNS, DNS poisoning malware used by DriftingBamboo that was deployed to a network appliance (in that instance, a Sophos XG Firewall). Volexity cannot confirm what mechanism was used by StormBamboo on the ISP’s routers to modify DNS responses; however, CATCHDNS would be a well-designed tool to achieve this goal in an ISP environment. An analysis of CATCHDNS can be found in the <a href="#Appendix">Appendix</a>.</p>
<h2>DNS Poisoning: Now with Abuse of Insecure Automatic Update Mechanisms!</h2>
<p style="font-weight: 400;">In the previously analyzed case where CATCHDNS was used to modify DNS responses, the end goal of the attacks was to modify the content of pages users browsed. This resulted in a popup JavaScript alert on the page asking the user to “update their browser”, which would download a malicious file from the attacker’s server. In this most recent case, the attacker’s method of delivering malware was more sophisticated, abusing insecure automatic update mechanisms present in software in the victim’s environment, thus requiring no user interaction.</p>
<p style="font-weight: 400;">The logic behind the abuse of automatic updates is the same for all the applications: the legitimate application performs an HTTP request to retrieve a text-based file (the format varies) containing the latest application version and a link to the installer. Since the attacker has control of the DNS responses for any given DNS name, they abuse this design, redirecting the HTTP request to a C2 server they control hosting a forged text file and a malicious installer. The AiTM workflow is shown below.</p>
<p><img decoding="async" class="aligncenter size-full wp-image-3296" src="https://www.volexity.com/wp-content/uploads/2024/08/image001.png" alt="" width="1430" height="758" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image001.png 1430w, https://www.volexity.com/wp-content/uploads/2024/08/image001-300x159.png 300w, https://www.volexity.com/wp-content/uploads/2024/08/image001-1024x543.png 1024w, https://www.volexity.com/wp-content/uploads/2024/08/image001-768x407.png 768w" sizes="(max-width: 1430px) 100vw, 1430px" /></p>
<p style="font-weight: 400;">Volexity observed StormBamboo targeting multiple software vendors, who use insecure update workflows, using varying levels of complexity in their steps for pushing malware. For example, <a href="https://www.5kplayer.com/">5KPlayer</a> uses a workflow that, for each time the application is started, the binary automatically checks if a new version of “YoutubeDL” is available. The image below shows the HTTP request to upgrade <code>Youtube.config</code>.</p>
<p><img decoding="async" class="aligncenter size-full wp-image-3297" src="https://www.volexity.com/wp-content/uploads/2024/08/image002.png" alt="" width="900" height="90" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image002.png 900w, https://www.volexity.com/wp-content/uploads/2024/08/image002-300x30.png 300w, https://www.volexity.com/wp-content/uploads/2024/08/image002-768x77.png 768w" sizes="(max-width: 900px) 100vw, 900px" /></p>
<p style="font-weight: 400;">And the following image shows the contents of upgrade <code>Youtube.config</code>.</p>
<p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-3298" src="https://www.volexity.com/wp-content/uploads/2024/08/image003.png" alt="" width="540" height="222" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image003.png 540w, https://www.volexity.com/wp-content/uploads/2024/08/image003-300x123.png 300w" sizes="(max-width: 540px) 100vw, 540px" /></p>
<p>&nbsp;</p>
<p style="font-weight: 400;">If a new version is available, it is downloaded from the specified URL and executed by the legitimate application. StormBamboo used DNS poisoning to host a modified config file indicating a new update was available. This resulted in the YoutubeDL software downloading an upgrade package from StormBamboo’s server.</p>
<p style="font-weight: 400;">As one might expect, the YoutubeDL package had been backdoored through the insertion of malicious code into the middle of the <code>YouTubeDL.py</code> file that is used as part of the upgrade process. The image below shows inserted malicious code, starting at line 164.</p>
<p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-3299" src="https://www.volexity.com/wp-content/uploads/2024/08/image004.png" alt="" width="858" height="728" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image004.png 858w, https://www.volexity.com/wp-content/uploads/2024/08/image004-300x255.png 300w, https://www.volexity.com/wp-content/uploads/2024/08/image004-768x652.png 768w" sizes="(max-width: 858px) 100vw, 858px" /></p>
<p style="font-weight: 400;">Its purpose is to download the next stage, a PNG file containing MACMA (macOS) or POCOSTICK (Windows) depending on the operating system.</p>
<p style="font-weight: 400;">MACMA was first <a href="https://blog.google/threat-analysis-group/analyzing-watering-hole-campaign-using-macos-exploits/">publicly documented</a> in 2021 by Google TAG. In the three years since, MACMA has changed, with more features added for the convenience of the operator and some of its architecture overhauled. For example, the network protocol has been completely changed. The original version used a <a href="https://en.wikipedia.org/wiki/Data_Distribution_Service">Data Distribution Server (DDS)</a> implemented in a series of custom classes prefixed by the string “CDDS”. Now, MACMA appears to use the <a href="https://github.com/juj/kNet">kNET</a> protocol UDP for network communications. During Volexity’s analysis, Volexity noticed significant code similarities between the latest MACMA version and the GIMMICK malware family <a href="https://www.volexity.com/blog/2022/03/22/storm-cloud-on-the-horizon-gimmick-malware-strikes-at-macos/">previously described by Volexity</a>.</p>
<h2>Follow-on Activity</h2>
<p style="font-weight: 400;">In one case, following successful compromise of a victim’s macOS device, Volexity observed StormBamboo deploying a Google Chrome extension to the victim’s device. Volexity tracks this malicious extension under the name RELOADEXT. The extension was installed using a custom binary (<code>ee28b3137d65d74c0234eea35fa536af</code>) developed by the attacker. The installer supports the following parameters:</p>
<table class="table table--style-2" style="height: 145px; width: 75%; font-weight: 400;">
<thead>
<tr style="height: 39px;">
<td style="height: 39px; width: 30%;" width="96"><strong>Parameter</strong></td>
<td style="height: 39px; width: 70%;" width="528"><strong>Description</strong></td>
</tr>
</thead>
<tbody>
<tr style="height: 67px;">
<td style="height: 67px; width: 30%;" width="96"><code>-p / --plugin</code></td>
<td style="height: 67px; width: 70%;" width="528">Path of the plugins (must be a ZIP archive)</td>
</tr>
<tr style="height: 39px;">
<td style="height: 39px; width: 30%;" width="96"><code>-f / --force</code></td>
<td style="height: 39px; width: 70%;" width="528">Kill Chrome and install the plugin</td>
</tr>
</tbody>
</table>
<p style="font-weight: 400;">The browser extension is deployed by modifying the <code>Secure Preferences</code> file to include the new extension. The installer also correctly fixes the <code>protections.macs</code> and <code>protections.super_mac</code> values in the newly modified <code>SecurePreferences</code>. These values are designed to prevent tampering with a user’s browser settings, but they <a href="https://www.microsoft.com/en-us/security/blog/2020/12/10/widespread-malware-campaign-seeks-to-silently-inject-ads-into-search-results-affects-multiple-browsers/">can be forged</a>. If they do not contain the expected values, Chrome will overwrite the <code>SecurePreferences</code> file.</p>
<p style="font-weight: 400;">The plugin passed to this tool is stored in the following location:</p>
<blockquote>
<p style="font-weight: 400;"><code> $HOME/Library/Application Support/Google/Chrome/Default/Default/CustomPlug1n/Reload/</code></p>
</blockquote>
<p style="font-weight: 400;">Once configured, it can be seen in the user’s <code>SecurePreferences</code> file, as shown below.</p>
<p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-3300" src="https://www.volexity.com/wp-content/uploads/2024/08/image005.png" alt="" width="867" height="903" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image005.png 867w, https://www.volexity.com/wp-content/uploads/2024/08/image005-288x300.png 288w, https://www.volexity.com/wp-content/uploads/2024/08/image005-768x800.png 768w" sizes="(max-width: 867px) 100vw, 867px" /></p>
<p style="font-weight: 400;">Finally, the plugin (<code>6abf9a7926415dc00bcb482456cc9467</code>) is activated by the installer running the following AppleScript command:</p>
<blockquote>
<p style="font-weight: 400;"><code>osascript -e tell application “Google Chrome” to activate</code></p>
</blockquote>
<p style="font-weight: 400;">The extension portrays itself as an extension that loads a page in <em>compatibility mode</em> with Internet Explorer:</p>
<p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-3301" src="https://www.volexity.com/wp-content/uploads/2024/08/image006.png" alt="" width="1026" height="1346" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image006.png 1026w, https://www.volexity.com/wp-content/uploads/2024/08/image006-229x300.png 229w, https://www.volexity.com/wp-content/uploads/2024/08/image006-781x1024.png 781w, https://www.volexity.com/wp-content/uploads/2024/08/image006-768x1008.png 768w" sizes="(max-width: 1026px) 100vw, 1026px" /></p>
<p>&nbsp;</p>
<p style="font-weight: 400;">The main JavaScript logic used by the extension is obfuscated using <a href="https://obfuscator.io/">Obfuscator.io</a>. The purpose of the extension is to exfiltrate browser cookies to a Google Drive account controlled by the attacker. The attacker’s Google Drive <code>client_id</code>, <code>client_secret</code>, and <code>refresh_token</code> are all contained in the extension. They are encrypted beyond the default encryption afforded by Obfuscator.io using AES with the key <code>chrome extension</code>.</p>
<p style="font-weight: 400;">The exfiltrated data sent to Google Drive is also encrypted using AES, using the key <code>opizmxn!@309asdf</code> and encoded with base64 prior to exfiltration.</p>
<h2>Conclusion</h2>
<p style="font-weight: 400;">StormBamboo is a highly skilled and aggressive threat actor who compromises third parties (in this case, an ISP) to breach intended targets. The variety of malware employed in various campaigns by this threat actor indicates significant effort is invested, with actively supported payloads for not only macOS and Windows, but also network appliances.</p>
<p style="font-weight: 400;">The incident described in this blog post confirms the supposition made by ESET concerning the infection vector for the POCOSTICK malware. The attacker can intercept DNS requests and poison them with malicious IP addresses, and then use this technique to abuse automatic update mechanisms that use HTTP rather than HTTPS. This method is similar to the attack vector Volexity previously observed being used by DriftingBamboo following the <a href="https://www.volexity.com/blog/2022/06/15/driftingcloud-zero-day-sophos-firewall-exploitation-and-an-insidious-breach/">0-day exploitation of Sophos Firewalls</a>.</p>

<h2><a id="Appendix"></a>Appendix</h2>
<h3>CATCHDNS Analysis</h3>
<p style="font-weight: 400;">CATCHDNS is a 32-bit ELF malware that targets Linux systems which was discovered in a case investigated by Volexity which Volexity attributes to StormBamboo. CatchDNS is designed to be deployed on systems through which most of the network traffic passes. In the specific case investigated by Volexity, this malware was discovered on a perimeter firewall device. However, CATCHDNS could be deployed on any Linux device that supports the use of <code>libpcap</code>.</p>
<p style="font-weight: 400;">After initial analysis, Volexity found that the malware is fully stripped, and the library functions are statically linked thus making further analysis more difficult. CATCHDNS stores its configuration within itself as an encrypted archive. The malware decrypts the archive and drops it on disk at runtime with the name <code>[binary_name].tty</code>. This archive is then decompressed in memory, and the copy on disk is deleted. In the example analyzed, the configuration file was in a file named <code>dns.ini</code>. The configuration follows the <a href="https://en.wikipedia.org/wiki/INI_file">INI file format</a>, which consists of various sections containing key-value pairs.</p>
<p style="font-weight: 400;">CATCHDNS configurations can have following sections:</p>
<table class="table table--style-2" style="font-weight: 400;">
<thead>
<tr>
<td width="119"><strong>Section</strong></td>
<td width="570"><strong>Description</strong></td>
</tr>
</thead>
<tbody>
<tr>
<td width="119">[LISTEN_DEV]</p>
<p>[SEND_DEV]</td>
<td width="570">The listen device and send device sections have a “dev” key under them whose value refers to the interface on which the malware intercepts the packets and sends fake packets.</td>
</tr>
<tr>
<td width="119">[DNSDomain]</td>
<td width="570">This section contains the “dns” key whose value represents the domain whose DNS is to be hijacked.</td>
</tr>
<tr>
<td width="119">[SERVER_IP]</td>
<td width="570">This section contains the “ip” key whose value is the IP address to which the hijacked domain will resolve once the malware has successfully performed hijacking.</td>
</tr>
<tr>
<td width="119">[IPLimit]</td>
<td width="570">This section contains a key named “ip”. When this is defined, the malware only hijacks requests originating from this IP address. This option only applies to HTTP requests.</td>
</tr>
<tr>
<td width="119">[HTTPConfig]</td>
<td width="570">This section is interesting, as it is the only one with multiple keys. It defines various values that are used when the malware intercepts HTTP requests.</td>
</tr>
</tbody>
</table>
<h4>Packet Interception</h4>
<p style="font-weight: 400;">Packet Interception is a key component of CATCHDNS. To intercept packets, it makes use of <code>libpcap</code>, a common library for packet monitoring on Linux. The device/interface on which the malware intercepts the packets is specified in the configuration. It uses the <code>pcap_open_live</code> library function to open the device for capturing packets. It installs a BPF filter on the device, and the filter program is compiled using the <code>pcap_compile</code> function by passing the filter string  <code>“(udp and dst port 53 ) or (tcp and dst port 80 or 8080)”</code>. The filter only captures UDP packets on port 53 and TCP packets on ports 80 or 8080. To actually install this filter, it uses the <code>pcap_setfilter</code> call.</p>
<p style="font-weight: 400;">Once everything is set up, CATCHDNS calls <code>pcap_loop</code> with a handler function as an argument. For every packet that passes the filter, the handler function is called with the packet data as an argument. This handler function is responsible for processing every filtered packet, as shown below.</p>
<p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-3302" src="https://www.volexity.com/wp-content/uploads/2024/08/image007.png" alt="" width="1316" height="544" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image007.png 1316w, https://www.volexity.com/wp-content/uploads/2024/08/image007-300x124.png 300w, https://www.volexity.com/wp-content/uploads/2024/08/image007-1024x423.png 1024w, https://www.volexity.com/wp-content/uploads/2024/08/image007-768x317.png 768w" sizes="(max-width: 1316px) 100vw, 1316px" /></p>
<p style="font-weight: 400;">The packet processing function checks the Ethernet and IPv4 headers to determine if it is a UDP or TCP packet. Depending on the IPv4 protocol of the packet, either <code>process_udp_packet</code> or <code>process_tcp_packet</code> is called.</p>
<h4>DNS Hijacking</h4>
<p style="font-weight: 400;">After analyzing the <code>process_udp_packet</code> function, it is clear the function specifically processes DNS packets. While dealing with network packets, it a good idea to create the packet structures in IDA and apply them while analyzing. This makes it easy to understand the whole logic. A DNS packet consists of the Ethernet header, IP header, and UDP header, followed by the DNS header and DNS data. Using this knowledge, these structures are applied to the processing functions to reveal the function parsing the DNS header and to perform basic sanity checks, as shown below.</p>
<p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-3303" src="https://www.volexity.com/wp-content/uploads/2024/08/image008.png" alt="" width="1281" height="308" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image008.png 1281w, https://www.volexity.com/wp-content/uploads/2024/08/image008-300x72.png 300w, https://www.volexity.com/wp-content/uploads/2024/08/image008-1024x246.png 1024w, https://www.volexity.com/wp-content/uploads/2024/08/image008-768x185.png 768w" sizes="(max-width: 1281px) 100vw, 1281px" /></p>
<p style="font-weight: 400;">Each DNS packet contains queries that appear after the DNS header in the packet. The queries contain information about the domain for which the DNS information is requested by the client. The malware parses the DNS queries and retrieves the domain name for which the DNS request is being made. Once it has the domain name, it is compared to the DNS domain(s) present in the malware’s configuration. If there is match, the DNS request is hijacked and the malware builds a fake DNS response packet. It then sends the packet back to the client, responding with the attacker-controlled (C2) IP address instead of the legitimate IP address. The following function is used to build the fake DNS packet and send it to the client:</p>
<p><img loading="lazy" decoding="async" class="aligncenter wp-image-3304 size-full" src="https://www.volexity.com/wp-content/uploads/2024/08/image009.png" alt="" width="858" height="631" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image009.png 858w, https://www.volexity.com/wp-content/uploads/2024/08/image009-300x221.png 300w, https://www.volexity.com/wp-content/uploads/2024/08/image009-768x565.png 768w" sizes="(max-width: 858px) 100vw, 858px" /></p>
<h4>HTTP Interception and Mock Response</h4>
<p style="font-weight: 400;">The <code>process_tcp_packet</code> function is used to intercept HTTP requests. An attacker can tune the interception using various configuration options. Both GET and POST requests can be intercepted by the malware. As previously mentioned, HTTP interception can also be limited to a given IP address using <code>IPLimit</code>. HTTP interception works similarly to DNS interception. If a request meets the conditions specified, the malware builds an HTTP mock response and sends it back to the client. The response can be configured via the malware configuration, where the attacker can configure a hardcoded page to return in response to specific requests.</p>
<p style="font-weight: 400;">To successfully respond with a fake HTTP response, all conditions specified in the configuration must be satisfied. The following keys can be specified in <code>HTTPConfig</code>:</p>
<ul>
<li><code>url</code></li>
<li><code>host</code></li>
<li><code>ua</code> (user-agent)</li>
<li><code>content-type</code></li>
<li><code>otherhead_%s</code></li>
<li><code>sendlimit</code></li>
<li><code>configfile</code></li>
</ul>
<p style="font-weight: 400;">Other headers to be parsed and checked can be specified using the <code>otherhead_%s</code> key, where <code>%s</code> denotes the header name. The <code>sendlimit</code> key defines how many times the malware will respond to requests satisfying the configuration. Once this limit is exceeded, the malware will no longer modify responses to requests matching the pattern. The <code>configfile</code> key contains the path to the web page to be served if all conditions are met.</p>
<h4>Configuration Example</h4>
<p style="font-weight: 400;">Volexity was able to extract all configurations from the CATCHDNS samples encountered during the intrusion by intercepting them before they were deleted from the disk. The image below shows one example of an extracted configuration.</p>
<p><img loading="lazy" decoding="async" class="aligncenter size-full wp-image-3305" src="https://www.volexity.com/wp-content/uploads/2024/08/image010.png" alt="" width="1138" height="784" srcset="https://www.volexity.com/wp-content/uploads/2024/08/image010.png 1138w, https://www.volexity.com/wp-content/uploads/2024/08/image010-300x207.png 300w, https://www.volexity.com/wp-content/uploads/2024/08/image010-1024x705.png 1024w, https://www.volexity.com/wp-content/uploads/2024/08/image010-768x529.png 768w" sizes="(max-width: 1138px) 100vw, 1138px" /></p>
<p style="font-weight: 400;">The above configuration intercepts all DNS (53) and HTTP (80 and 8080) packets on the <code>Port1</code> device. It hijacks the <code>www.msftconnecttest[.]com</code> domain and responds with IP address <code>122.10.90[.]20</code> for this domain. In <code>HttpConfig</code> , the “host” key is absent, meaning the malware would intercept an HTTP request to any host if it satisfied the other conditions. This was only one of several configurations observed; the attacker has been observed using a variety of options offered by the malware to achieve various objectives.</p>
