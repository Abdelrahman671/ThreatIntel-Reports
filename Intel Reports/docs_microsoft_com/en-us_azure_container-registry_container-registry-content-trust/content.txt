											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>Azure Container Registry implements Docker's <a href="https://docs.docker.com/engine/security/trust/content_trust" data-linktype="external">content trust</a> model, enabling pushing and pulling of signed images. This article gets you started enabling content trust in your container registries.</p>
<div class="NOTE">
<p>Note</p>
<p>Content trust is a feature of the <a href="container-registry-skus" data-linktype="relative-path">Premium service tier</a> of Azure Container Registry.</p>
</div>
<h2 id="limitations">Limitations</h2>
<ul>
<li>Token with repository-scoped permissions does not currently support docker push and pull of signed images.</li>
</ul>
<h2 id="how-content-trust-works">How content trust works</h2>
<p>Important to any distributed system designed with security in mind is verifying both the <em>source</em> and the <em>integrity</em> of data entering the system. Consumers of the data need to be able to verify both the publisher (source) of the data, as well as ensure it's not been modified after it was published (integrity).</p>
<p>As an image publisher, content trust allows you to <strong>sign</strong> the images you push to your registry. Consumers of your images (people or systems pulling images from your registry) can configure their clients to pull <em>only</em> signed images. When an image consumer pulls a signed image, their Docker client verifies the integrity of the image. In this model, consumers are assured that the signed images in your registry were indeed published by you, and that they've not been modified since being published.</p>
<div class="NOTE">
<p>Note</p>
<p>Azure Container Registry (ACR) does not support <code>acr import</code> to import images signed with Docker Content Trust (DCT). By design, the signatures are not visible after the import, and the notary v2 stores these signatures as artifacts.</p>
</div>
<h3 id="trusted-images">Trusted images</h3>
<p>Content trust works with the <strong>tags</strong> in a repository. Image repositories can contain images with both signed and unsigned tags. For example, you might sign only the <code>myimage:stable</code> and <code>myimage:latest</code> images, but not <code>myimage:dev</code>.</p>
<h3 id="signing-keys">Signing keys</h3>
<p>Content trust is managed through the use of a set of cryptographic signing keys. These keys are associated with a specific repository in a registry. There are several types of signing keys that Docker clients and your registry use in managing trust for the tags in a repository. When you enable content trust and integrate it into your container publishing and consumption pipeline, you must manage these keys carefully. For more information, see <a href="#key-management" data-linktype="self-bookmark">Key management</a> later in this article and <a href="https://docs.docker.com/engine/security/trust/trust_key_mng/" data-linktype="external">Manage keys for content trust</a> in the Docker documentation.</p>
<div class="TIP">
<p>Tip</p>
<p>This was a very high-level overview of Docker's content trust model. For an in-depth discussion of content trust, see <a href="https://docs.docker.com/engine/security/trust/content_trust" data-linktype="external">Content trust in Docker</a>.</p>
</div>
<h2 id="enable-registry-content-trust">Enable registry content trust</h2>
<p>Your first step is to enable content trust at the registry level. Once you enable content trust, clients (users or services) can push signed images to your registry. Enabling content trust on your registry does not restrict registry usage only to consumers with content trust enabled. Consumers without content trust enabled can continue to use your registry as normal. Consumers who have enabled content trust in their clients, however, will be able to see <em>only</em> signed images in your registry.</p>
<p>To enable content trust for your registry, first navigate to the registry in the Azure portal. Under <strong>Policies</strong>, select <strong>Content Trust</strong> &gt; <strong>Enabled</strong> &gt; <strong>Save</strong>. You can also use the <a href="/en-us/cli/azure/acr/config/content-trust#az_acr_config_content_trust_update" data-linktype="absolute-path">az acr config content-trust update</a> command in the Azure CLI.</p>
<p><img src="media/container-registry-content-trust/content-trust-01-portal.png" alt="Screenshot shows enabling content trust for a registry in the Azure portal." data-linktype="relative-path"></p>
<h2 id="enable-client-content-trust">Enable client content trust</h2>
<p>To work with trusted images, both image publishers and consumers need to enable content trust for their Docker clients. As a publisher, you can sign the images you push to a content trust-enabled registry. As a consumer, enabling content trust limits your view of a registry to signed images only. Content trust is disabled by default in Docker clients, but you can enable it per shell session or per command.</p>
<p>To enable content trust for a shell session, set the <code>DOCKER_CONTENT_TRUST</code> environment variable to <strong>1</strong>. For example, in the Bash shell:</p>
<pre><code class="lang-bash"># Enable content trust for shell session
export DOCKER_CONTENT_TRUST=1
</code></pre>
<p>If instead you'd like to enable or disable content trust for a single command, several Docker commands support the <code>--disable-content-trust</code> argument. To enable content trust for a single command:</p>
<pre><code class="lang-bash"># Enable content trust for single command
docker build --disable-content-trust=false -t myacr.azurecr.io/myimage:v1 .
</code></pre>
<p>If you've enabled content trust for your shell session and want to disable it for a single command:</p>
<pre><code class="lang-bash"># Disable content trust for single command
docker build --disable-content-trust -t myacr.azurecr.io/myimage:v1 .
</code></pre>
<h2 id="grant-image-signing-permissions">Grant image signing permissions</h2>
<p>Only the users or systems you've granted permission can push trusted images to your registry. To grant trusted image push permission to a user (or a system using a service principal), grant their Microsoft Entra identities the <code>AcrImageSigner</code> role. This is in addition to the <code>AcrPush</code> (or equivalent) role required for pushing images to the registry. For details, see <a href="container-registry-roles" data-linktype="relative-path">Azure Container Registry roles and permissions</a>.</p>
<div class="IMPORTANT">
<p>Important</p>
<p>You can't grant trusted image push permission to the following administrative accounts:</p>
<ul>
<li>the <a href="container-registry-authentication#admin-account" data-linktype="relative-path">admin account</a> of an Azure container registry</li>
<li>a user account in Microsoft Entra ID with the <a href="/en-us/azure/role-based-access-control/rbac-and-directory-admin-roles#classic-subscription-administrator-roles" data-linktype="absolute-path">classic system administrator role</a>.</li>
</ul>
</div>
<div class="NOTE">
<p>Note</p>
<p>Starting July 2021, the <code>AcrImageSigner</code> role includes both the         <code>Microsoft.ContainerRegistry/registries/sign/write</code> action and the <code>Microsoft.ContainerRegistry/registries/trustedCollections/write</code> data action.</p>
</div>
<p>Details for granting the <code>AcrImageSigner</code> role in the Azure portal and the Azure CLI follow.</p>

<h3 id="azure-cli">Azure CLI</h3>
<p>To grant signing permissions to a user with the Azure CLI, assign the <code>AcrImageSigner</code> role to the user, scoped to your registry. The format of the command is:</p>
<pre><code class="lang-azurecli">az role assignment create --scope &lt;registry ID&gt; --role AcrImageSigner --assignee &lt;user name&gt;
</code></pre>
<p>For example, to grant a non-administrative user the role, you can run the following commands in an authenticated Azure CLI session. Modify the <code>REGISTRY</code> value to reflect the name of your Azure container registry.</p>
<pre><code class="lang-bash"># Grant signing permissions to authenticated Azure CLI user
REGISTRY=myregistry
REGISTRY_ID=$(az acr show --name $REGISTRY --query id --output tsv)
</code></pre>
<pre><code class="lang-azurecli">az role assignment create --scope $REGISTRY_ID --role AcrImageSigner --assignee azureuser@contoso.com
</code></pre>
<p>You can also grant a <a href="container-registry-auth-service-principal" data-linktype="relative-path">service principal</a> the rights to push trusted images to your registry. Using a service principal is useful for build systems and other unattended systems that need to push trusted images to your registry. The format is similar to granting a user permission, but specify a service principal ID for the <code>--assignee</code> value.</p>
<pre><code class="lang-azurecli">az role assignment create --scope $REGISTRY_ID --role AcrImageSigner --assignee &lt;service principal ID&gt;
</code></pre>
<p>The <code>&lt;service principal ID&gt;</code> can be the service principal's <strong>appId</strong>, <strong>objectId</strong>, or one of its <strong>servicePrincipalNames</strong>. For more information about working with service principals and Azure Container Registry, see <a href="container-registry-auth-service-principal" data-linktype="relative-path">Azure Container Registry authentication with service principals</a>.</p>
<div class="IMPORTANT">
<p>Important</p>
<p>After any role changes, run <code>az acr login</code> to refresh the local identity token for the Azure CLI so that the new roles can take effect. For information about verifying roles for an identity, see <a href="/en-us/azure/role-based-access-control/role-assignments-cli" data-linktype="absolute-path">Add or remove Azure role assignments using Azure CLI</a> and <a href="/en-us/azure/role-based-access-control/troubleshooting" data-linktype="absolute-path">Troubleshoot Azure RBAC</a>.</p>
</div>
<h2 id="push-a-trusted-image">Push a trusted image</h2>
<p>To push a trusted image tag to your container registry, enable content trust and push the image with <code>docker push</code>. After push with a signed tag completes the first time, you're asked to create a passphrase for both a root signing key and a repository signing key. Both the root and repository keys are generated and stored locally on your machine.</p>
<pre><code class="lang-console">$ docker push myregistry.azurecr.io/myimage:v1
[...]
The push refers to repository [myregistry.azurecr.io/myimage]
ee83fc5847cb: Pushed
v1: digest: sha256:aca41a608e5eb015f1ec6755f490f3be26b48010b178e78c00eac21ffbe246f1 size: 524
Signing and pushing trust metadata
You are about to create a new root signing key passphrase. This passphrase
will be used to protect the most sensitive key in your signing system. Please
choose a long, complex passphrase and be careful to keep the password and the
key file itself secure and backed up. It is highly recommended that you use a
password manager to generate the passphrase and keep it safe. There will be no
way to recover this key. You can find the key in your config directory.
Enter passphrase for new root key with ID 4c6c56a:
Repeat passphrase for new root key with ID 4c6c56a:
Enter passphrase for new repository key with ID bcd6d98:
Repeat passphrase for new repository key with ID bcd6d98:
Finished initializing "myregistry.azurecr.io/myimage"
Successfully signed myregistry.azurecr.io/myimage:v1
</code></pre>
<p>After your first <code>docker push</code> with content trust enabled, the Docker client uses the same root key for subsequent pushes. On each subsequent push to the same repository, you're asked only for the repository key. Each time you push a trusted image to a new repository, you're asked to supply a passphrase for a new repository key.</p>
<h2 id="pull-a-trusted-image">Pull a trusted image</h2>
<p>To pull a trusted image, enable content trust and run the <code>docker pull</code> command as normal. To pull trusted images, the <code>AcrPull</code> role is enough for normal users. No additional roles like an <code>AcrImageSigner</code> role are required. Consumers with content trust enabled can pull only images with signed tags. Here's an example of pulling a signed tag:</p>
<pre><code class="lang-console">$ docker pull myregistry.azurecr.io/myimage:signed
Pull (1 of 1): myregistry.azurecr.io/myimage:signed@sha256:0800d17e37fb4f8194495b1a188f121e5b54efb52b5d93dc9e0ed97fce49564b
sha256:0800d17e37fb4f8194495b1a188f121e5b54efb52b5d93dc9e0ed97fce49564b: Pulling from myimage
8e3ba11ec2a2: Pull complete
Digest: sha256:0800d17e37fb4f8194495b1a188f121e5b54efb52b5d93dc9e0ed97fce49564b
Status: Downloaded newer image for myregistry.azurecr.io/myimage@sha256:0800d17e37fb4f8194495b1a188f121e5b54efb52b5d93dc9e0ed97fce49564b
Tagging myregistry.azurecr.io/myimage@sha256:0800d17e37fb4f8194495b1a188f121e5b54efb52b5d93dc9e0ed97fce49564b as myregistry.azurecr.io/myimage:signed
</code></pre>
<p>If a client with content trust enabled tries to pull an unsigned tag, the operation fails with an error similar to the following:</p>
<pre><code class="lang-console">$ docker pull myregistry.azurecr.io/myimage:unsigned
Error: remote trust data does not exist
</code></pre>
<h3 id="behind-the-scenes">Behind the scenes</h3>
<p>When you run <code>docker pull</code>, the Docker client uses the same library as in the <a href="https://docs.docker.com/notary/getting_started/" data-linktype="external">Notary CLI</a> to request the tag-to-SHA-256 digest mapping for the tag you're pulling. After validating the signatures on the trust data, the client instructs Docker Engine to do a "pull by digest." During the pull, the Engine uses the SHA-256 checksum as a content address to request and validate the image manifest from the Azure container registry.</p>
<div class="NOTE">
<p>Note</p>
<p>Azure Container Registry does not officially support the Notary CLI but is compatible with the Notary Server API, which is included with Docker Desktop. Currently Notary version <strong>0.6.0</strong> is recommended.</p>
</div>
<h2 id="key-management">Key management</h2>
<p>As stated in the <code>docker push</code> output when you push your first trusted image, the root key is the most sensitive. Be sure to back up your root key and store it in a secure location. By default, the Docker client stores signing keys in the following directory:</p>
<pre><code class="lang-sh">~/.docker/trust/private
</code></pre>
<p>Back up your root and repository keys by compressing them in an archive and storing it in a secure location. For example, in Bash:</p>
<pre><code class="lang-bash">umask 077; tar -zcvf docker_private_keys_backup.tar.gz ~/.docker/trust/private; umask 022
</code></pre>
<p>Along with the locally generated root and repository keys, several others are generated and stored by Azure Container Registry when you push a trusted image. For a detailed discussion of the various keys in Docker's content trust implementation, including additional management guidance, see <a href="https://docs.docker.com/engine/security/trust/trust_key_mng/" data-linktype="external">Manage keys for content trust</a> in the Docker documentation.</p>
<h3 id="lost-root-key">Lost root key</h3>
<p>If you lose access to your root key, you lose access to the signed tags in any repository whose tags were signed with that key. Azure Container Registry cannot restore access to image tags signed with a lost root key. To remove all trust data (signatures) for your registry, first disable, then re-enable content trust for the registry.</p>
<div class="WARNING">
<p>Warning</p>
<p>Disabling and re-enabling content trust in your registry <strong>deletes all trust data for all signed tags in every repository in your registry</strong>. This action is irreversible--Azure Container Registry cannot recover deleted trust data. Disabling content trust does not delete the images themselves.</p>
</div>
<p>To disable content trust for your registry, navigate to the registry in the Azure portal. Under <strong>Policies</strong>, select <strong>Content Trust</strong> &gt; <strong>Disabled</strong> &gt; <strong>Save</strong>. You're warned of the loss of all signatures in the registry. Select <strong>OK</strong> to permanently delete all signatures in your registry.</p>
<p><img src="media/container-registry-content-trust/content-trust-03-portal.png" alt="Disabling content trust for a registry in the Azure portal" data-linktype="relative-path"></p>

While several key points were touched on in this article, content trust is an extensive topic and is covered more in-depth in the Docker documentation.</p>
</li>

</li>
