<h1 class="blog-banner__heading">
      <span>New modular downloaders fingerprint systems, prepare for more - Part 1: Marap</span>

  </h1>

      <span class="blog-content__date">  <time datetime="2018-08-16T14:00:00Z">August 16, 2018</time>

</span>
        <span class="blog-content__author">
                Proofpoint Staff

    <div class="node-full__body blog-content__body">  <p><strong>Overview</strong></p>

<p>Proofpoint researchers recently discovered a new downloader malware in a fairly large campaign (millions of messages) primarily targeting financial institutions. The malware, dubbed Marap (param backwards), is notable for its focused functionality that includes the ability to download other modules and payloads. The modular nature allows actors to add new capabilities as they become available or download additional modules post infection. To date, we have observed it download a system fingerprinting module that performs simple reconnaissance.</p>

<p><strong>Campaign Analysis</strong></p>

<p>On August 10, 2018, we observed several large email campaigns (millions of messages) leading to the same Marap malware payload in our testing. They shared many features with previous campaigns attributed to the TA505 actor [1]. The emails contained various attachment types:</p>

<ul>
	<li>Microsoft Excel Web Query (.iqy) files</li>
	<li>Password-protected ZIP archives containing .iqy files</li>
	<li>PDF documents with embedded .iqy files</li>
	<li>Microsoft Word documents containing macros</li>
</ul>

<p>The campaigns are outlined below:</p>

<p><u>sales .iqy attachment campaign:</u> Messages purporting to be from '"sales" &lt;[random address]&gt;' with the subject "REQUEST [REF:ABCDXYZ]" (random letters) and attachment "REP_10.08.iqy" (campaigns date)</p>

<p><img alt="Sales example email message with .iqy attachment" src="/sites/default/files/marapf1.png"></p>

<p><em>Figure 1: Sales example email message with .iqy attachment</em></p>

<p><u>Major bank .iqy attachment campaign</u>: Messages purporting to be from '"[recipient name]" &lt;random_name@[major bank].com&gt;' with subject "IMPORTANT Documents - [Major Bank]" and attachment "Request 1234_10082018.iqy" (random digits, campaign's date); note that this campaign abuses the brand and name of a major US bank and has been obscured throughout the example.</p>

<p><img alt="Major bank sample message with .iqy attachment" src="/sites/default/files/marapf2.png"></p>

<p><em>Figure 2: Major bank sample message with .iqy attachment; bank branding obscured</em></p>

<p><u>PDF attachment campaign</u>: Messages purporting to be from '"Joan Doe" &lt;netadmin@[random domain]&gt;' (random display name) with subject "DOC_1234567890_10082018" (random digits, campaigns date; also "PDF", "PDFFILE", "SCN") and matching attachment "DOC_1234567890_10082018.pdf" (with embedded .iqy file)</p>

<p><img alt="Message sample with PDF attachment with embedded .iqy file" src="/sites/default/files/marapf3.png"></p>

<p><em>Figure 3: Message sample with PDF attachment with embedded .iqy file</em></p>

<p><u>Password-protected ZIP campaign</u>: Messages purporting to be from '"John" &lt;John@[random company]&gt;' (random name) with subject "Emailing: PIC12345" (random digits) and matching attachment "PIC12345.zip"</p>

<p><img alt="Message sample with password-protected zip attachment that contains .iqy file" src="/sites/default/files/marapf4.png"></p>

<p><em>Figure 4: Message sample with password-protected zip attachment that contains a .iqy file</em></p>

<p><u>Microsoft Word attachment campaign</u>: Messages purporting to be from '"Joan" &lt;Joan@[random domain]&gt;' (random name) with subject "Invoice for 12345.10/08/2018" (random digits, today's date) with matching attachment "Invoice_ 12345.10_08_2018.doc"</p>

<p><img alt="Message sample with Microsoft Word attachment with malicious macros" src="/sites/default/files/marapf5.png"></p>

<p><em>Figure 5: Message sample with Microsoft Word attachment (incorrectly described as PDF format in the message body) with malicious macros</em></p>

<p><strong>Malware Analysis</strong></p>

<p>As noted, Marap is a new downloader, named after its command and control (C&amp;C) phone home parameter param spelled backwards. The malware is written in C and contains a few notable anti-analysis features.</p>

<p><u>Anti-Analysis Features</u></p>

<p>Most of the Windows API function calls are resolved at runtime using a hashing algorithm. API hashing is common in malware to prevent analysts and automated tools from easily determining the codes purpose. This algorithm appears to be custom to Marap. Our implementation of the hashing algorithm in Python is available on Github [3]. It is likely that the XOR keys used in our code will be different in other samples.</p>

<p>The second anti-analysis technique is the use of timing checks at the beginning of important functions (Figure 6). These checks can hinder debugging and sandboxing of the malware. If the calculated sleep time is too short, the malware exits.</p>

<p><em><img alt="Anti-analysis timing checks" src="/sites/default/files/marapf6.png"></em></p>

<p><em>Figure 6: Anti-analysis timing checks</em></p>

<p>Most of the strings in the malware are obfuscated using one of three methods:</p>

<ol>
	<li>Created on the stack (stack strings)</li>
	<li>Basic XOR encoding (0xCE was the key used in the analyzed sample, but it is likely this will change from sample to sample)</li>
	<li>A slightly more involved XOR-based encoding (An IDA Pro script implementing the decryption is available on Github [4])</li>
</ol>

<p>The last anti-analysis check compares the systems MAC address to a list of virtual machine vendors. If a virtual machine is detected and a configuration flag is set, the malware may exit.</p>

<p><u>Configuration</u></p>

<p>Maraps configuration is stored in an encrypted format in the malware binary and/or in a file named Sign.bin in the malwares working directory (e.g., C:\Users\[username]\AppData\Roaming\Intel\Sign.bin). It is DES-encrypted in CBC mode using an IV of \x00\x00\x00\x00\x00\x00\x00\x00. The key is generated using the following process:</p>

<ul>
	<li>164 bytes of data are generated using a linear congruential generator (LCG) and two hardcoded seeds (it is likely the seeds are different in other samples). An implementation of the LCG in Python is available on Github [5].</li>
	<li>The data is hashed with SHA1</li>
	<li>An 8-byte DES key is created using CryptDeriveKey and the hash</li>
</ul>

<p>An example decrypted configuration looks like:</p>

<p>15|1|hxxp://185.68.93[.]18/dot.php|hxxp://94.103.81[.]71/dot.php|hxxp://89.223.92[.]202/dot.php</p>

<p>It is pipe-delimited and contains configuration parameters for:</p>

<ul>
	<li>Sleep timeout between C&amp;C communications</li>
	<li>Flag indicating whether the malware should exit if it detects that it is running on a virtual machine</li>
	<li>Up to three C&amp;C URLs</li>
</ul>

<p><u>Command and Control</u></p>

<p>Marap uses HTTP for its C&amp;C communication but first it tries a a number of legitimate WinHTTP functions to determine whether it needs to use a proxy and if so what proxy to use. An example C&amp;C beacon is shown in Figure 7 below.</p>

<p><img alt="Example C&amp;C beacon" src="/sites/default/files/marapf7_0.png"></p>

<p><em>Figure 7: Example C&amp;C beacon</em></p>

<p>The request contains one parameter -- param -- and its data is encrypted using the same method as used for the configuration, with the addition of base64 encoding. An example of the plaintext request looks like:</p>

<p>62061c6bcdec4fba|0|0</p>

<p>It is pipe-delimited and contains the following:</p>

<ul>
	<li>Bot ID (generated by hashing the hostname, username, and MAC address with the same hashing algorithm used with API function hashing described above)</li>
	<li>Hardcoded to 0</li>
	<li>Hardcoded to 0</li>
</ul>

<p>The response is encrypted similarly and an example decrypted response looks like:</p>

<p>319&amp;1&amp;0&amp;hxxp://89.223.92[.]202/mo.enc</p>

<p>It is &amp;-delimited and contains the following:</p>

<ul>
	<li>Command ID</li>
	<li>Command</li>
	<li>Flag controlling response type</li>
	<li>Command arguments (there can be two arguments separated by a #)</li>
</ul>

<p>Identified commands:</p>

<ul>
	<li>0: Sleep and beacon again</li>
	<li>1: Download URL, DES decrypt, and manually load the MZ file (allocate a buffer, copy the PE header and sections, reallocate, and resolve the import table). This command can pass back data from the downloaded module to the C&amp;C</li>
	<li>2: Update configuration and write a DES-encrypted version to the file Sign.bin</li>
	<li>3: Download URL, DES decrypt, save the MZ file to %TEMP%/evt, and execute with a command line argument</li>
	<li>4: Download URL, DES decrypt, create/hollow out a process (same executable as malware), and inject the downloaded MZ file</li>
	<li>5: Download URL, DES decrypt, save the MZ file as %TEMP%/zvt, and load it with the LoadLibrary API</li>
	<li>6: Download URL, DES decrypt, and manually load the MZ file</li>
	<li>7: Remove self and exit</li>
	<li>8: Update self</li>
</ul>

<p>After command execution a response message can be sent back to the C&amp;C. It is pipe delimited and contains the following:</p>

<ul>
	<li>Bot ID</li>
	<li>Hardcoded 1</li>
	<li>Command ID</li>
	<li>Command</li>
	<li>Flag controlling response type</li>
	<li>Command return value</li>
	<li>Command status code (various error codes)</li>
	<li>Response data
	<ul>
		<li>Either a simple status message</li>
		<li>Or verbose # delimited data from modules</li>
	</ul>
	</li>
</ul>

<p><u>System Fingerprinting Module</u></p>

<p>At the time of publication, we have only seen a system fingerprinting module being sent from a C&amp;C server. It was downloaded from hxxp://89.223.92[.]202/mo.enc and contained an internal name of mod_Init.dll. The module is a DLL written in C and gathers and sends the following system information to the C&amp;C server:</p>

<ul>
	<li>Username</li>
	<li>Domain name</li>
	<li>Hostname</li>
	<li>IP address</li>
	<li>Language</li>
	<li>Country</li>
	<li>Windows version</li>
	<li>List of Microsoft Outlook .ost files</li>
	<li>Anti-virus software detected</li>
</ul>

<p><strong>Conclusion</strong></p>

<p>As defenses become more adept at catching commodity malware, threat actors and malware authors continue to explore new approaches to increase effectiveness and decrease the footprint and inherent noisiness of the malware they distribute. We have observed ransomware distribution drop dramatically this year while banking Trojans, downloaders, and other malware have moved to fill the void, increasing opportunities for threat actors to establish persistence on devices and networks. This new downloader, along with another similar but unrelated malware that we will detail next week, point to a growing trend of small, versatile malware that give actors flexibility to launch future attacks and identify systems of interest that may lend themselves to more significant compromise.</p>
