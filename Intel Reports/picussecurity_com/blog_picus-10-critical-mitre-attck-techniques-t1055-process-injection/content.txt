  function blockOrUnblockCookies(cookiesToBlockOrUnblock) {
    const originalCookieDescriptor = Object.getOwnPropertyDescriptor(Document.prototype, 'cookie');

    Object.defineProperty(document, 'cookie', {
      set: function (value) {
        const cookieName = value.split('=')[0];




        cookiesToBlockOrUnblock.forEach(item => {

          if (item.cookieName.includes(cookieName) && item.isBlocked) {
            console.log(`Cookie Blocked: ${cookieName}`);
            return;
          }
          originalCookieDescriptor.set.call(document, value);


        });




      }
    });

  }

  function deleteCookie(domains, name) {
    const secureFlags = [true, false];

    domains.forEach(domain => {
      secureFlags.forEach(secure => {
        document.cookie = `${name}=; expires=Thu, 01 Jan 1970 00:00:00 UTC; path=/; domain=${domain};${secure ? ' Secure;' : ''}`;
        console.log(`Deleted cookie: ${name} for domain ${domain}`);
      });
    });
  }

  // Function to delete cookies by pattern
  function deleteCookiesByPattern(cookiePatterns) {   
    // Ensure document.cookie is not empty
    if (!document.cookie) {
      console.warn("No cookies available to delete.");
      return;
    }

    const allCookies = document.cookie.split('; ').map(cookie => cookie.split('=')[0].trim());

    cookiePatterns.forEach(pattern => {
      if (pattern.includes('*')) {
        const prefix = pattern.split('*')[0]; // Get the part before the '*'
        allCookies.forEach(cookieName => {
          if (cookieName.startsWith(prefix)) {
            document.cookie = `${cookieName}=; expires=Thu, 01 Jan 1970 00:00:00 UTC; path=/;`;
            // console.log(`Deleted cookie matching pattern: ${cookieName}`);
          }
        });
      }
    });
  }

  function prepareCookiesArrayToBeDeleted(){

			 <div class="promotion-box version-2">
 <h4>The Blue Report 2024</h4>
 <div class="desc"><p><span>Get a comprehensive analysis of over 136 million cyber attacks and understand the state of threat exposure management.</span></p></div>
 
 
 
 <a class="button" href="https://www.picussecurity.com/resource/report/blue-report-2024" target="_blank" rel="noopener">
	DOWNLOAD
 </a>
 
</div>			 
			 
			 <span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><blockquote>
<p>In 2019, Picus Labs analyzed 48813 malware to determine tactics, techniques, and procedures (TTPs) used by adversaries in these malicious files. Picus Labs categorized each observed TTP by utilizing the MITRE ATT&amp;CK® framework. As a result of the present research, 445018 TTPs observed in the last year were mapped to ATT&amp;CK to identify the top 10 most common techniques used by attackers.</p>
</blockquote>
<p>Our research has found that Process Injection was the most prevalent <a href="/resource/glossary/what-is-mitre-attack" rel="noopener" target="_blank">MITRE ATT&amp;CK</a> technique used by adversaries in their malware.&nbsp; Adversaries emphasize an increased level of stealth, persistence, and privilege in their advanced cyber attacks. As a mechanism that can provide these features, it is not surprising that Process Injection is the most frequently used technique.</p>
<p>The purpose of this blog post is to review:</p>
<ul>
<li>the fundamentals of the process injection technique,&nbsp;</li>
<li>the most used target processes for injection,&nbsp;</li>
<li>its use cases by threat actors, and&nbsp;</li>
<li>red, blue, and purple teaming exercises for this technique.</li>
</ul>
<p style="text-align: center; font-size: 20px;"><span style="color: #d80033;"><a href="/resource/report/red-report-2023" rel="noopener" target="_blank" style="color: #d80033; font-weight: bold;">Explore the Red Report - Top Ten MITRE ATT&amp;CK Techniques</a></span></p>
<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">Introduction</span></h2>
<p>It is easy to detect malware processes by listing the running processes and filtering out legitimate ones that are part of the operating system or installed software. If the malware can encapsulate its malicious code within a legitimate process, it will hide on the infected system. Process injection is in fact an “old but gold” technique consisting in running arbitrary code within the address space of another process. As a result, this technique enables access to the target process’s memory, system, and network resources.&nbsp;</p>
<p>On this account, the technique provides three significant benefits for adversaries:</p>
<ul>
<li>Executing code under a legitimate process may evade security controls. The legitimate process camouflages the malicious code to evade detection since it is whitelisted.</li>
<li>Since the malicious code executed inside the legitimate process’s memory space, it may also evade disk forensics.</li>
<li>If the target process has elevated privileges, this technique will enable privilege escalation. For example, if the target process has access to network resources, the malicious code can communicate legitimately over the Internet and with other computers on the same network.</li>
</ul>
<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">Processes Targeted by Adversaries for Process Injection</span></h2>
<p>Security controls may quickly detect custom processes. Therefore, threat actors use common Windows processes such as:</p>
<ul>
<li>Built-in native Windows processes including <span style="color: #d60059; background-color: #efefef;">explorer.exe</span><span style="color: #39434c;">, </span><span style="color: #d60059; background-color: #efefef;">svchost.exe</span><span style="color: #39434c;">,&nbsp; </span><span style="color: #d60059; background-color: #efefef;">regsvr32.exe</span><span style="color: #39434c;">, </span><span style="color: #d60059; background-color: #efefef;">dllhost.exe</span><span style="color: #39434c;">, </span><span style="color: #d60059; background-color: #efefef;">services.exe</span><span style="color: #39434c;">, </span><span style="color: #d60059;"><span style="background-color: #efefef;">cvtres.exe</span></span>,<span style="color: #d60059;"><span style="background-color: #efefef;">msbuild.exe</span></span><span style="color: #d60059;"><span style="background-color: #efefef;"><span style="color: #000000;">,</span></span>&nbsp;<span style="background-color: #efefef;">RegAsm.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">RegSvcs.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">rundll32.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">arp.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">PowerShell.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">vbc.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">csc.exe</span></span>, <span style="color: #d60059; background-color: #efefef;">AppLaunch.exe</span> and <span style="color: #d60059; background-color: #efefef;">cmd.exe</span></li>
<li>Processes of common<span style="color: #39434c;"> software including </span><span style="color: #d60059;"><span style="background-color: #efefef;">iexplore.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">ieuser.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">opera.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">chrome.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">firefox.exe</span></span>, <span style="color: #d60059;"><span style="background-color: #efefef;">outlook.exe</span></span><span style="color: #39434c;">, and </span><span style="color: #d60059; background-color: #efefef;">msinm.exe</span></li>
</ul>
<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">Target Process Selection Methods</span></h2>
<p>Adversaries use the following methods when picking their target process for malicious code injection:</p>
<ul>
<li>A specific target process is called out in the code. In this case, explorer.exe and svchost.exe are the most commonly used ones.</li>
<li>A list of target processes is defined in the code. For example, the Turla cyber espionage group’s Carbon backdoor includes a configuration file consisting of a list of target processes for injection<sup><span style="text-decoration: none;"><span style="font-size: 14px;">[1]</span></span></sup>. A typical list includes native Windows and browser processes.</li>
<li>In some attack scenarios, the target process is not previously defined, and a suitable host process is located at runtime in this type of attack. For example, the CopyKittens group used Windows API functions to extract a list of currently active processes and to get a handle to each target process in its campaign<span style="font-size: 14px;"><sup>[2]</sup></span>.&nbsp;</li>
</ul>
<p style="text-align: center; font-size: 20px;"><a href="https://discover.picussecurity.com/start-your-free-trial-mitre?hsLang=en-gb" rel="noopener" target="_blank" id="__hsNewLink" style="font-weight: bold;">Start your 14-day Free Trial: Simulate MITRE ATT&amp;CK Techniques&nbsp;in minutes</a></p>
<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">Use Cases by Malware and Threat Actors</span></h2>

<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">Example Process Injection Method: Reflective DLL Injection</span></h2>
<p>Reflective DLL injection (loading) is one of the most used process injection methods employed by adversaries. This method allows injecting and executing a DLL inside another process by creating a DLL that maps itself into memory when executed, instead of relying on Window’s API’s loader calls. This technique avoids storing the DLL on disk and calling the Windows API’s LoadLibrary that might be detected by security tools.</p>
<h3 style="font-weight: bold; font-size: 26px;"><span style="color: #d80033;">Red Teaming - How to simulate?</span></h3>
<p>Powersploit’s <span style="color: #d60059; background-color: #efefef;">Invoke-ReflectivePEInjection</span>&nbsp; <span>[19]</span> module can be used to simulate the reflective DLL injection technique. In addition to loading a DLL or EXE into the PowerShell,&nbsp; It can reflectively load a DLL into a remote process. Because of its capabilities, adversaries are also using this module for injection, such as the Turla APT Group <span>[20]</span>.</p>
<p>The below command is a simulation of reflective DLL injection using the <span style="color: #d60059; background-color: #efefef;">Invoke-ReflectivePEInjection</span> module. With this command, contents of the calc.exe file are read into the <span style="color: #d60059; background-color: #efefef;">$PEByte</span> byte array using the <span style="color: #d60059; background-color: #efefef;">ReadAllBytes</span> <span>[21]</span> method. Then the byte array containing the calc.exe is loaded and executed locally using the <span style="color: #d60059; background-color: #efefef;">-PEBytes</span> parameter.</p>

<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">Blue Teaming - How to detect?</span></h2>
<h3 style="font-size: 22px; font-weight: bold;">Sigma Rule</h3>
<p>To detect the reflective DLL injection technique, we need logs that include PowerShell activities. Event log entries in the <code>Microsoft-Windows-PowerShell/Operational</code> log includes such activities. The <code>Event ID 4104</code> (script block logging) records accurate blocks of code as they are executed by the PowerShell engine. Script block logging captures the de-obfuscated full contents of the code as it is executed, including scripts and commands, as shown in the following figure.</p>
