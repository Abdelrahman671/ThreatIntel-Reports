                        <div class="post-content">
                                <p><i class="updated-at">Last updated at Tue, 07 Nov 2023 16:33:39 GMT</i></p>
                            <p><em>Technical Analysis by: Thomas Elkins, Natalie Zargarov<br>Contributions: Evan McCann, Tyler McGraw</em></p><p>Recently, Rapid7 observed the Fake Browser Update lure tricking users into executing malicious binaries. While analyzing the dropped binaries, Rapid7 determined a new loader is utilized in order to execute infostealers on compromised systems including StealC and Lumma.</p><p>The IDAT loader is a new, sophisticated loader that Rapid7 first spotted in July 2023. In earlier versions of the loader, it was disguised as a 7-zip installer that delivered the SecTop RAT. Rapid7 has now observed the loader used to deliver infostealers like Stealc, Lumma, and Amadey. It implements several evasion techniques including Process Doppelgänging, DLL Search Order Hijacking, and Heaven's Gate. IDAT loader got its name as the threat actor stores the malicious payload in the IDAT chunk of PNG file format.</p><p>Prior to this technique, Rapid7 observed threat actors behind the lure utilizing malicious JavaScript files to either reach out to Command and Control (C2) servers or drop the Net Support Remote Access Trojan (RAT).</p><p>The following analysis covers the entire attack flow, which starts from a new ClearFake malware, spotted just several days ago, and ends with the stolen information in threat actors’ hands.</p><h2 id="technical-analysis">Technical Analysis</h2><p>Threat Actors (TAs) are often staging their attacks in the way security tools will not detect them and security researchers will have a hard time investigating them.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/09/minerva_blog_image-cf.png" class="kg-image"><figcaption>Figure 1: Attack Flow</figcaption></figure><p></p><h3 id="stage-1-clearfake">Stage 1 - ClearFake</h3><p><a href="https://rmceoin.github.io/malware-analysis/clearfake/">ClearFake</a> is a new malware first recognized just a few days ago. Its campaign started on July 19,2023 which aligns with the time Rapid7 spotted a new IDAT loader distribution. We first attributed that initial attack flow to the SocGolish malware, however the ClearFake seems to be less sophisticated. </p><p>In this campaign, ClearFake malware uses base64 to obfuscate malicious Javascript, which can be easily deobfuscated by using <a href="https://gchq.github.io/CyberChef/">CyberChef</a>. As spotted by <a href="https://infosec.exchange/@rmceoin">Randy McEoin</a>, the “One noticeable difference from SocGholish is that there appears to be no tracking of visits by IP or cookies. As an analyst you can you go back to the compromised site over and over coming from the same IP and not clearing your browser cache. This also means the site owner is more likely to see the infection as well.”</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.22.04-AM.png" class="kg-image"><figcaption><em>Figure 2 - Obfuscated JavaScript Embedded in the Compromised Domain</em></figcaption></figure><p></p><p>This prompt falsely presents itself as a browser update, with the added layer of credibility coming from the fact that it appears to originate from the intended domain.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.22.50-AM.png" class="kg-image"><figcaption><em>Figure 3 - Pop-up Prompting the User to Update their Browser</em></figcaption></figure><p></p><p>Once the user interacts with the “Update Chrome” button, the browser is redirected to another URL where a binary automatically downloads to the user's default download folder. After the user double clicks the fake update binary, it will proceed to download the next stage payload. In this investigation, Rapid7 identified a binary called <em><strong>ChromeSetup.exe</strong></em>, the file name widely used in previous SocGholish attacks and now adopted by ClearFake.</p><h3 id="stage-2-msi-downloader">Stage 2 - MSI Downloader</h3><p><em><strong>ChromeSetup.exe</strong></em> downloads and executes the Microsoft Software Installer (MSI) package from: <em><strong>hxxps://ocmtancmi2c5t[.]xyz/82z2fn2afo/b3/update[.]msi</strong></em>.</p><p>In similar investigations, Rapid7 observed that the initial dropper executable appearance and file name may vary depending on the user’s browser when visiting the compromised web page. In all instances, the executables contained invalid signatures and attempted to download and install an MSI package. </p><p>Rapid7 determined that the MSI package executed with several switches intended to avoid detection:</p><ul><li>/qn to avoid an installation UI</li><li>/quiet to prevent user interaction</li><li>/norestart to prevent the system from restarting during the infection process</li></ul><p>When executed, the MSI dropper will write a legitimate <em><strong>VMwareHostOpen.exe</strong></em> executable, multiple legitimate dependencies, and the malicious Dynamic-Link Library (DLL) file <em><strong>vmtools.dll</strong></em>. It will also drop an encrypted <em><strong>vmo.log</strong></em> file which has a PNG file structure and is later decrypted by the malicious DLL. <br>Rapid7 spotted an additional version of the attack where the MSI dropped a legitimate <em><strong>pythonw.exe</strong></em>, legitimate dependencies, and the malicious DLL file <em><strong>python311.dll</strong></em>.<strong> </strong>In that case, the encrypted file was named <em><strong>pz.log</strong></em>,<strong> </strong>though the execution flow remains the same.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.25.02-AM.png" class="kg-image"><figcaption><em>Figure 4 - Content of <strong>vmo.log</strong></em></figcaption></figure><p></p><h3 id="stage-3-decryptor">Stage 3 - Decryptor</h3><p>When executed, the legitimate <em><strong>VMWareHostOpen.exe</strong></em> loads the malicious <em><strong>vmtools.dll</strong></em><strong> </strong>from the same directory as from which the <em><strong>VMWareHostOpen.exe</strong></em><strong> </strong>is executed. This technique is known as <a href="https://attack.mitre.org/techniques/T1574/001/">DLL Search Order Hijacking.</a> </p><p>During the execution of <em><strong>vmtools.dll</strong></em>, Rapid7 observed that the DLL loads API libraries from <em><strong>kernel32.dll</strong></em> and <em><strong>ntdll.dll</strong></em> using API hashing and maps them to memory. After the API functions are mapped to memory, the DLL reads the hex string <em><strong>83 59 EB ED 50 60 E8</strong></em> and decrypts it using a bitwise XOR operation with the key <em><strong>F5 34 84 C3 3C 0F 8F</strong></em>, revealing the string <em><strong>vmo.log</strong></em>. The file is similar to the <em><strong>Vmo\log</strong></em> directory, where Vmware logs are stored. </p><p>The DLL then reads the contents from <em><strong>vmo.log</strong></em> into memory and searches for the string <em><strong>…IDAT</strong></em>. The DLL takes 4 bytes following <em><strong>…IDAT</strong></em> and compares them to the hex values of <em><strong>C6 A5 79 EA</strong></em>. If the 4 bytes following <em><strong>…IDAT</strong></em> are equal to the hex values <strong>C6</strong><em><strong> A5 79 EA</strong></em>, the DLL proceeds to copy all the contents following <em><strong>…IDAT</strong></em> into memory.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.25.54-AM.png" class="kg-image"><figcaption>Figure 5 - Function Searching for Hex Values <em><strong>C6 A5 79 EA</strong></em></figcaption></figure><p></p><p>Once all the data is copied into memory, the DLL attempts to decrypt the copied data using the bitwise XOR operation with key <em><strong>F4 B4 07 9A</strong></em>. Upon additional analysis of other samples, Rapid7 determined that the XOR keys were always stored as 4 bytes following the hex string <em><strong>C6 A5 79 EA</strong></em>.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.26.38-AM.png" class="kg-image"><figcaption><em>Figure 6 - XOR Keys found within PNG Files <strong>pz.log</strong> and <strong>vmo.log</strong></em></figcaption></figure><p></p><p>Once the DLL decrypts the data in memory, it is decompressed using the RTLDecompressBuffer function. The parameters passed to the function include:</p><ul><li>Compression format</li><li>Size of compressed data</li><li>Size of compressed buffer</li><li>Size of uncompressed data</li><li>Size of uncompressed buffer</li></ul><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.27.30-AM.png" class="kg-image"><figcaption><em>Figure 7 - Parameters passed to RTLDecompressBuffer function</em></figcaption></figure><p></p><p>The <em><strong>vmtools.dll</strong></em> DLL utilizes the compression algorithm LZNT1 in order to decompress the decrypted data from the <em><strong>vmo.log</strong></em> file.</p><p>After the data is decompressed, the DLL loads <em><strong>mshtml.dll</strong></em> into memory and overwrites its .text section with the decompressed code. After the overwrite, <em><strong>vmtools.dll</strong></em> calls the decompressed code.</p><h3 id="stage-4-idat-injector">Stage 4 - IDAT Injector</h3><p>Similarly to <em><strong>vmtools.dll</strong></em>,<strong> </strong>IDAT loader uses dynamic imports. The IDAT injector then expands the <em><strong>%APPDATA%</strong></em> environment variable by using the <em><strong>ExpandEnvironmentStringsW</strong></em> API call. It creates a new folder under <em><strong>%APPDATA%</strong></em>, naming it based on the <em><strong>QueryPerformanceCounter</strong></em> API call output and randomizing its value.<br></p><p>All the dropped files by MSI are copied to the newly created folder. IDAT then creates a new instance of <em><strong>VMWareHostOpen.exe</strong></em><strong> </strong>from the <em><strong>%APPDATA%</strong></em> by using <em><strong>CreateProcessW</strong></em> and exits.<br></p><p>The second instance of <em><strong>VMWareHostOpen.exe</strong></em><strong> </strong>behaves the same up until the stage where the IDAT injector code is called from <em><strong>mshtml.dll</strong></em><strong> </strong>memory space. IDAT immediately started the implementation of the Heaven’s Gate evasion technique, which it uses for most API calls until the load of the infostealer is completed. <br></p><p>Heaven's Gate is widely used by threat actors to evade security tools. It refers to a method for executing a 64-bit process within a 32-bit process or vice versa, allowing a 32-bit process to run in a 64-bit process. This is accomplished by initiating a call or jump instruction through the use of a reserved selector. The key points in analyzing this technique in our case is to change the process mode from 32-bit to 64-bit, the specification of the selector “0x0033” required and followed by the execution of a far call or far jump, as shown in Figure 8.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.28.26-AM.png" class="kg-image"><figcaption><em>Figure 8 - Heaven’s Gate technique implementation</em></figcaption></figure><figure class="kg-card kg-image-card"><img src="https://lh6.googleusercontent.com/CKIeq3iK_IHy2wNzhbU9v_3yVZ-ZI8MFxUWv0m8KDCoe-A_D5GUAeNhj9rFde177LKgK73k6yuwTWVf3A4JJQzbHWtAxWoC_UhulLm9k9TqIhH0uv8lcvgzdgH9Z8TwJI47uq0kIBj9Wb8kl92zeWzU" class="kg-image"></figure><p><br>The IDAT injector then expands the <em><strong>%TEMP%</strong></em> environment variable by using the <em><strong>ExpandEnvironmentStringsW</strong></em> API call. It creates a string based on the <em><strong>QueryPerformanceCounter</strong></em> API call output and randomizes its value.<br></p><p>Next, the IDAT loader gets the computer name by calling <em><strong>GetComputerNameW</strong></em> API call, and the output is randomized by using <em><strong>rand</strong> </em>and <em><strong>srand</strong></em> API calls. It uses that randomized value to set a new environment variable by using <em><strong>SetEnvironmentVariableW</strong></em>.<strong> </strong>This variable is set to a combination of <em><strong>%TEMP%</strong></em> path with the randomized string created previously.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.29.14-AM.png" class="kg-image"><figcaption><em>Figure 9 - New Environment variable - <strong>TCBEDOPKVDTUFUSOCPTRQFD</strong> set to <strong>%TEMP%\89680228</strong></em></figcaption></figure><p></p><p>Now, the new <em><strong>cmd.exe</strong></em> process is executed by the loader. The loader then creates and writes to the <em><strong>%TEMP%\89680228</strong></em> file. </p><p>Next, the IDAT injects code into cmd.exe process by using <a href="https://www.ired.team/offensive-security/code-injection-process-injection/ntcreatesection-+-ntmapviewofsection-code-injection"><strong>NtCreateSection + NtMapViewOfSection Code Injection</strong></a><strong> </strong>technique<strong>. </strong>Using this technique the malware:</p><ul><li>Creates a new memory section inside the remote process by using the NtCreateSection API call</li><li>Maps a view of the newly created section to the local malicious process with RW protection by using <em><strong>NtMapViewOfSection</strong></em> API call</li><li>Maps a view of the previously created section to a remote target process with RX protection by using <em><strong>NtMapViewOfSection</strong></em> API call</li><li>Fills the view mapped in the local process with shellcode by using <em><strong>NtWriteVirtualMemory</strong></em> API call</li><li>In our case, IDAT loader suspends the main thread on the <em><strong>cmd.exe</strong> </em>process by using <em><strong>NtSuspendThread</strong></em> API call and then resumes the thread by using <em><strong>NtResumeThread</strong> </em>API call<br>After completing the injection, the second instance of <em><strong>VMWareHostOpen.exe</strong></em><strong> </strong>exits.</li></ul><h3 id="stage-5-idat-loader-">Stage 5 - IDAT Loader:</h3><p>The injected loader code implements the Heaven’s Gate evasion technique in exactly the same way as the IDAT injector did. It retrieves the <em><strong>TCBEDOPKVDTUFUSOCPTRQFD</strong></em> environment variable, and reads the <em><strong>%TEMP%\89680228</strong></em> file data into the memory. The data is then recursively XORed with the <strong>3D ED C0 D3</strong> key.  </p><p>The decrypted data seems to contain configuration data, including which process the infostealer should be loaded, which API calls should be dynamically retrieved, additional code,and more. The loader then deletes the initial malicious DLL (<em><strong>vmtools.dll</strong></em>) by using <em><strong>DeleteFileW</strong></em>.<strong> </strong>The loader finally injects the infostealer code into the <em><strong>explorer.exe</strong></em> process by using the Process Doppelgänging injection technique.</p><p>The<strong> </strong>Process Doppelgänging<strong> </strong>method utilizes the Transactional NTFS feature within the Windows operating system. This feature is designed to ensure data integrity in the event of unexpected errors. For instance, when an application needs to write or modify a file, there's a risk of data corruption if an error occurs during the write process. To prevent such issues, an application can open the file in a transactional mode to perform the modification and then commit the modification, thereby preventing any potential corruption. The modification either succeeds entirely or does not commence.</p><p>Process Doppelgänging exploits this feature to replace a legitimate file with a malicious one, leading to a process injection. The malicious file is created within a transaction, then committed to the legitimate file, and subsequently executed. The Process Doppelgänging<strong> </strong>in our sample was performed by:</p><ul><li>Initiating a transaction by using <em><strong>NtCreateTransaction</strong></em> API call</li><li>Creating a new file by using <em><strong>NtCreateFile</strong></em> API call</li><li>Writing to the new file by using <em><strong>NtWriteFile</strong></em><strong> </strong>API call</li><li>Writing malicious code into a section of the local process using <em><strong>NtCreateSection</strong></em><strong> </strong>API call</li><li>Discarding the transaction by using <em><strong>NtRollbackTransaction</strong></em><strong> </strong>API call</li><li>Running a new instance of explorer.exe process by using <em><strong>NtCreateProcessEx</strong></em> API call</li><li>Running the malicious code inside explorer.exe process by using <em><strong>NtCreateThreadEx</strong></em><strong> </strong>API call<br></li></ul><p>If the file created within a transaction is rolled back (instead of committed), but the file section was already mapped into the process memory, the process injection will still be performed.</p><p>The final payload injected into the <em><strong>explorer.exe</strong></em> process was identified by Rapid7 as Lumma Stealer.</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.29.57-AM.png" class="kg-image"><figcaption><em>Figure 10 - Process Tree</em></figcaption></figure><p></p><p>Throughout the whole attack flow, the malware delays execution by using <em><strong>NtDelayExecution</strong></em>, a technique that is usually used to escape sandboxes.</p><p>As previously mentioned, Rapid7 has investigated several IDAT loader samples. The main differences were:</p><ol><li>The legitimate software that loads the malicious DLL.</li><li>The name of the staging directory created within <em><strong>%APPDATA%</strong></em>.</li><li>The process the IDAT injector injects the Loader code to.</li><li>The process into which the infostealer/RAT loaded into.</li><li>Rapid7 observed the IDAT loader has been used to load the following infostealers and RAT: Stealc, Lumma and Amadey infostealers and SecTop RAT.</li></ol><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.30.45-AM.png" class="kg-image"><figcaption><em>Figure 11 - Part of an HTTP POST request to a StealC C2 domain</em></figcaption></figure><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.rapid7.com/content/images/2023/11/Screenshot-2023-11-07-at-10.31.33-AM.png" class="kg-image"><figcaption><em>Figure 12 - An HTTP POST request to a Lumma Stealer C2 domain</em></figcaption></figure><figure class="kg-card kg-image-card"><img src="https://lh6.googleusercontent.com/xyU47bkeYKtttXdQ9SCSiGEdH4iQPzVF4GYUfqO1OPVA5Bnt3v3-xdstEcNnxIxSMzoKM6kG1Ds2dZMaJ8xvMQnsFyHSoDPnJ9Sf4iOiAYMFCpSwNmLF_2uhoqBt1TGYDBkw3Z9GqYvvs2KRNFOTXac" class="kg-image"></figure><h2 id="conclusion">Conclusion</h2><p>IDAT Loader is a new sophisticated loader that utilizes multiple evasion techniques in order to execute various commodity malware including InfoStealers and RAT’s. The Threat Actors behind the Fake Update campaign have been packaging the IDAT Loader into DLLs that are loaded by legitimate programs such as VMWarehost, Python and Windows Defender.</p><h2 id="rapid7-customers">Rapid7 Customers</h2><p>For Rapid7 MDR and InsightIDR customers, the following Attacker Behavior Analytics (ABA) rules are currently deployed and alerting on the activity described in this blog:</p><ul><li>Attacker Technique - MSIExec loading object via HTTP</li><li>Suspicious Process - FSUtil Zeroing Out a File</li><li>Suspicious Process - Users Script Spawns Cmd And Redirects Output To Temp File</li><li>Suspicious Process - Possible Dropper Script Executed From Users Downloads Directory</li><li>Suspicious Process - WScript Runs JavaScript File from Temp Or Download Directory</li></ul><h2 id="mitre-attck-techniques-">MITRE ATT&amp;CK Techniques:</h2><p></p><!--kg-card-begin: markdown--><table>
