	 <div class="article_section">
          <h1>OpenWrt Sysupgrade flaw let hackers push malicious firmware images</h1>
          <div class="cz-news-story-title-section">
            <div class="cz-news-title-left-area">
                            By <h6><a rel='author' href='https://www.bleepingcomputer.com/author/bill-toulas/' class='author'><span itemprop="author" itemscope itemtype="https://schema.org/Person"><span itemprop="name">Bill Toulas</span></span></a></h6>
            </div>
	    <meta itemprop="articleSection" content="Security"><meta itemprop="articleSection" content="Hardware">
            <div class="cz-news-title-right-area">
              <ul>
                <li class="cz-news-date">December 9, 2024</li>
                <li class="cz-news-time">05:33 PM</li>
                
                <li class="cz-news-comment"><a href="#comment_form">0</a></li>
              </ul>
            </div>
          </div>
	  <div class="articleBody">
	   
<p style="text-align:center"><img alt="Router attack" height="897" src="https://www.bleepstatic.com/content/hl-images/2024/04/30/router.jpg" width="1600"></p>

<p>A flaw in&nbsp;OpenWrt's Attended Sysupgrade feature used to build custom, on-demand&nbsp;firmware images could have allowed for the distribution of malicious firmware packages.</p>

<p>OpenWrt is a highly customizable, open-source, Linux-based operating system designed for embedded devices, particularly network devices like routers, access points, and other IoT hardware. The project is a popular alternative to a manufacturer's firmware as it offers numerous advanced features and supports routers from ASUS, Belkin,&nbsp;Buffalo, D-Link, Zyxel, and many more.</p>

<p>The command injection and hash truncation flaw was discovered by Flatt Security researcher 'RyotaK' during a routine home lab router upgrade.</p>

<p>The critical (CVSS v4 score: 9.3) flaw, tracked as <a href="https://nvd.nist.gov/vuln/detail/CVE-2024-54143" target="_blank" rel="nofollow noopener">CVE-2024-54143</a>, was <a href="https://openwrt.org/advisory/2024-12-06" target="_blank" rel="nofollow noopener">fixed within hours</a> of being disclosed to OpenWrt's developers. However, users are urged to perform checks to ensure the safety of their installed firmware.</p>

<h2>Poisoning OpenWrt images</h2>

<p>OpenWrt&nbsp;includes a service called&nbsp;<a href="https://openwrt.org/docs/guide-user/installation/attended.sysupgrade" target="_blank" rel="nofollow noopener">Attended Sysupgrade</a> that allows users to create custom, on-demand firmware builds that include previously installed packages and settings.</p>

<p>"The Attended SysUpgrade (ASU) facility allows an OpenWrt device to update to new firmware while preserving the packages and settings. This dramatically simplifies the upgrade process: just a couple clicks and a short wait lets you retrieve and install a new image built with all your previous packages," explains an <a href="https://openwrt.org/docs/guide-user/installation/attended.sysupgrade" target="_blank" rel="nofollow noopener">OpenWrt&nbsp;support page</a>.</p>

<p>"ASU eliminates the need to make a list of packages you installed manually, or fuss with opkg just to upgrade your firmware."</p>

<p>RyotaK discovered that the sysupgrade.openwrt.org service processes these inputs via commands executed in a containerized environment.</p>

<p>A flaw in the input handling mechanism originating from the insecure usage of the 'make' command in the server code allows arbitrary command injection via the package names.</p>

<p>A second problem RyotaK discovered was that the service uses a 12-character truncated SHA-256 hash to cache build artifacts, limiting the hash to only 48 bits.</p>

<p>The researcher explains that this makes brute-forcing collisions feasible, allowing an attacker to create a request that reuses a cache key found in legitimate firmware builds.</p>

<p>By combining the two problems and using the Hashcat tool on an RTX 4090 graphics card, RyotaK <a href="https://flatt.tech/research/posts/compromising-openwrt-supply-chain-sha256-collision/" target="_blank" rel="nofollow noopener">demonstrated</a> that it's possible to modify firmware artifacts to deliver malicious builds to unsuspecting users.</p>

<div style="text-align:center">
<figure class="image" style="display:inline-block"><img alt="Python script used for overwriting legitimate firmware builds" height="222" src="https://www.bleepstatic.com/images/news/u/1220909/2024/AI/2.png" width="513"><figcaption><strong>Python script used for overwriting legitimate firmware builds</strong><br><em>Source: Flatt Security</em></figcaption></figure></div>

<h2>Check your routers</h2>

<p>The OpenWrt team immediately responded to RyotaK's private report, taking down the sysupgrade.openwrt.org service, applying a fix, and getting it back up in 3 hours on December 4, 2024.</p>

<p>The team says it's highly unlikely that anyone has exploited CVE-2024-54143, and they have found no evidence that this vulnerability impacted images from downloads.openwrt.org.</p>

<p>However, since they only have visibility for what happened in the last 7 days, it is suggested that users install a newly generated image to replace any potentially insecure images currently loaded on their devices.</p>

<p>"Available build logs for other custom images were checked and NO MALICIOUS REQUEST FOUND, however due to automatic cleanups no builds older than 7 days could be checked. Affected server is reset and reinizialized from scratch," explains&nbsp;<a href="https://lists.openwrt.org/pipermail/openwrt-announce/2024-December/000061.html" target="_blank" rel="nofollow noopener">OpenWrt</a>.</p>

<p>"Although the possibility of compromised images is near 0, it is SUGGESTED to the user to make an INPLACE UPGRADE to the same version to ELIMINATE any possibility of being affected by this. If you run a public, self-hosted instance of ASU, please update it immediately."</p>

<p>This issue has existed for a while, so there are no cut-off dates, and everyone should take the recommended action out of an abundance of caution.</p>

function cz_strip_tags(input, allowed) {
  allowed = (((allowed || '') + '')
    .toLowerCase()
    .match(/<[a-z][a-z0-9]*>/g) || [])
    .join(''); // making sure the allowed arg is a string containing only tags in lowercase (<a><b><c>)
  var tags = /<\/?([a-z][a-z0-9]*)\b[^>]*>/gi,
    commentsAndPhpTags = /<!--[\s\S]*?-->|<\?(?:php)?[\s\S]*?\?>/gi;
  return input.replace(commentsAndPhpTags, '')
    .replace(tags, function($0, $1) {
      return allowed.indexOf('<' + $1.toLowerCase() + '>') > -1 ? $0 : '';
    });
}
function cz_br2nl(str) {
	var regex = /<br\s*[\/]?>/gi;
    //var pure_str = str.replace(regex,"\n");
    var pure_str = str.replace(regex,"");
    return cz_strip_tags(pure_str,'');
}
