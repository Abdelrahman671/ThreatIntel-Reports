<h1><span id="hs_cos_wrapper_name" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="text">Don’t Sleep on the New ProxyShellMiner Campaign</span></h1>
<div id="hubspot-author_data" class="hubspot-editable" data-hubspot-form-id="author_data" data-hubspot-name="Blog Author">
<span class="hs-author-label">Posted by</span>
<a class="author-link" href="https://blog.morphisec.com/author/alon-shekalim-michael-dereviashkin">Alon Shekalim &amp; Michael Dereviashkin</a> on February 15, 2023
</div>
</div>
<div class="blog-share-media">
<span id="hs_cos_wrapper_blog_social_sharing" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_blog_social_sharing" style="" data-hs-cos-general-type="widget" data-hs-cos-type="blog_social_sharing">

<div id="hs_cos_wrapper_module_1701779085953111" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module"></div>
<div class="section post-body">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p>Morphisec has recently identified a highly evasive malware campaign delivering <a href="https://www.bilibili.com/read/cv17634838?from=articleDetail" rel="noopener" target="_blank">ProxyShellMiner</a> to Windows endpoints.</p>
<!--more-->
<p>As the name suggests, ProxyShellMiner exploits the ProxyShell vulnerabilities <a href="https://nvd.nist.gov/vuln/detail/CVE-2021-34473" rel="noopener" target="_blank">CVE-2021-34473</a> and <a href="https://nvd.nist.gov/vuln/detail/CVE-2021-34523" rel="noopener" target="_blank">CVE-2021-34523</a> in Windows Exchange servers for initial access and compromise of an organization to deliver crypto miners. After successfully breaching an Exchange server and obtaining control, the attackers use the domain controller’s NETLOGON folder to ensure the miner executes throughout the domain, similar to how software is delivered through GPO. We detected four C2 servers in use by the attackers. All are legitimate, compromised mail servers which host the malware-dependent files.&nbsp;</p>
<p>Mining cryptocurrency on an organization’s network can lead to system performance degradation, increased power consumption, equipment overheating, and can stop services. Unfortunately, mining threats are often disregarded or deprioritized until the same backdoor delivers ransomware.&nbsp;</p>

<h2>Technical Analysis&nbsp;</h2>
<h3>Configuration Option&nbsp;</h3>
<p>The samples Morphisec analyzed require a command line parameter to be supplied upon execution, for example “000.” This parameter is later used as a key for the XMRig miner configuration, and as an anti-runtime analysis tactic.&nbsp;</p>
<p style="text-align: center;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-19-08-0122-PM.png" loading="lazy" alt="The parameter serves as anti-analysis technique, and as a password for the XMrig miner"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span><br><em>The parameter serves as anti-analysis technique, and as a password for the XMrig miner</em>&nbsp;</p>
<h2 style="text-align: left;">Execution and Code Decryption Behavior&nbsp;</h2>
<p style="text-align: left;">ProxyShellMiner uses an embedded dictionary, an XOR decryption algorithm, and an XOR key downloaded from a remote server. Then, it uses a C# compiler CSC.exe with “InMemory” compile parameters to execute the next embedded code modules.&nbsp;<br><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-22-12-7637-PM.png" loading="lazy" alt="The miner downloads a KEY from a remote server which is used for XOR decryption&nbsp;"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span></p>
<p style="text-align: center;"><em>The miner downloads a KEY from a remote server which is used for XOR decryption&nbsp;</em><br><i><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-23-16-6611-PM.png" loading="lazy" alt="&nbsp;Remote server https://mail.shaferglazer[.]com/resources/files&nbsp;"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span>&nbsp;Remote server https://mail.shaferglazer[.]com/resources/files&nbsp;<br></i></p>
<p style="text-align: left;">For the next stage, ProxyShellMiner downloads a file named “DC_DLL” from a remote server (see IOCs. Analyzed sample downloads from hxxps://mail.shaferglazer[.]com/resources/files/). ProxyShellMiner performs .NET reflection of the DC_DLL file and passes arguments such as task scheduler, XML, and the XMRig key.&nbsp;</p>
<p style="text-align: center;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-24-22-9543-PM.png" loading="lazy" alt="Reflective loading of the DC_DLL&nbsp;"><span></span></span><span data-contrast="auto" xml:lang="EN-US" lang="EN-US"></span> <em>Reflective loading of the DC_DLL</em>&nbsp;</p>
<p style="text-align: left;">The malware author applied a forked version of “Confuser” which obfuscates the code.&nbsp;</p>
<h2 style="text-align: left;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-25-50-6272-PM.png" loading="lazy" alt="The malware author applied a forked version of “Confuser” which obfuscates the code.&nbsp;"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span><br>&nbsp;<br>Second Downloader: Persistence and Evasion&nbsp;</h2>
<p style="text-align: left;">To gain persistence, the miner creates a scheduled task configured to run when any user logs on. For naming of the scheduled task and creating the next stage downloader path, the miner creates a hashed value of the target machine name (7826F246.exe) and gets the environment variable CommonProgramFiles. By concatenating the two variables it creates the next stage of the downloader path.&nbsp;</p>
<p style="text-align: center;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-26-58-7321-PM.png" loading="lazy" alt="Creating a path C:\Program Files\Common Files\microsoft shared\ for the next stage downloader: 7826F246.exe&nbsp;"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span><br><em>Creating a path C:\Program Files\Common Files\microsoft shared\ for the next stage downloader: 7826F246.exe&nbsp;</em></p>
<p style="text-align: center;">Then, it uses Deflate to deobfuscate the embedded task schedule .XML. &nbsp;<br><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-28-00-1646-PM.png" loading="lazy" alt="The deobfuscated scheduled task&nbsp;"><span></span></span><span data-contrast="auto" xml:lang="EN-US" lang="EN-US"></span><br><em>The deobfuscated scheduled task&nbsp;</em></p>
<p style="text-align: left;">By using PowerShell runspace, the miner adds the process path and name of the second stage downloader (7826F246.exe) to the Defender exclusion.&nbsp;<br><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-28-58-9956-PM.png" loading="lazy" alt="the miner adds the process path and name of the second stage downloader"><span></span></span><span data-contrast="auto" xml:lang="EN-US" lang="EN-US"></span><br>It sets file attributes to the file as hidden, system, and notcontentindexed so the file is not indexed by the operating system's content indexing service.&nbsp;<br><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-30-33-0504-PM.png" loading="lazy" alt="It sets file attributes to the file as hidden, system, and notcontentindexed"><span></span></span><span data-contrast="auto" xml:lang="EN-US" lang="EN-US"></span><br>It eventually writes and executes the file. Immediately upon launch, it attempts to validate the existence of the programdata\softwaredistribution path. If not found, the attacker creates the directory.&nbsp;</p>
<p style="text-align: center;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-30-55-1292-PM.png" loading="lazy" alt="Validating the existence of programdata\softwaredistribution path&nbsp;"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span><br><em>Validating the existence of programdata\softwaredistribution path&nbsp;</em></p>
<p style="text-align: left;">The second stage downloader has a key for decrypting the additional files: the second loader named LC_DLL, the XMrig payload named DATA1, and two additional configuration files. It writes the files into the generated directory with the name “%ProgramData%\SoftwareDistribution\&lt;file&gt;.&nbsp;</p>
<p style="text-align: center;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-14-2023-09-31-35-8321-PM.png" loading="lazy" alt="It downloads five additional files, among them the payload (DATA1) and second loader (LC_DLL)&nbsp;"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span><br><em>It downloads five additional files, among them the payload (DATA1) and second loader (LC_DLL)&nbsp;</em></p>
<h2 style="text-align: left;">The Second loader&nbsp;</h2>
<p style="text-align: left;">At this point the second loader decides which of the installed browsers will be hollowed by the XMrig, then randomly chooses a pool from a list of XMrig pools.&nbsp;</p>
<p style="text-align: center;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-15-2023-01-37-54-6379-AM.png" loading="lazy" alt="List of XMrig pools&nbsp;"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span><br><em>List of XMrig pools&nbsp;</em></p>
<p style="text-align: left;">Then the XMrig config file is created, along with several parameters such as the server IP used by the XMrig, user identifier, and the password passed by the command line. In addition, the second loader checks if WinRing0x64.sys already exists in the browser path. The WinRing0x64.sys is a known, signed, and vulnerable driver used as part of the XMrig randomX algorithm to achieve a higher hash rate by accessing the MSR registers. If the driver is not found, it writes the driver from the sys variable embedded in the loader. Lastly, it sets the file attributes as hidden, system, and notcontentindexed so the file is not indexed by the operating system's content indexing service.&nbsp;</p>
<p style="text-align: center;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-15-2023-01-40-45-2070-AM.png" loading="lazy" alt="Creating the XMrig config file&nbsp;"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span><br><em>Creating the XMrig config file&nbsp;</em></p>
<h2 style="text-align: left;">Final Stage and Security Evasion&nbsp;</h2>
<p style="text-align: left;">ProxyShellMiner creates a firewall rule that applies to all Windows Firewall profiles—domain, private, and public—and blocks all outgoing traffic. In the final stage, it hollows the selected browser to inject the XMrig payload via the well-known RunPE technique. It then waits at least 30 seconds while the target machine blocks any outbound connection. It does this to tamper with the process runtime behavior analysis of common security solutions.&nbsp;</p>
<p style="text-align: center;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-15-2023-01-41-32-5207-AM.png" loading="lazy"><span></span></span><span data-contrast="none" xml:lang="EN-US" lang="EN-US"></span><br><em>Creation of the firewall rule and hollowing of the selected browser&nbsp;</em></p>
<p style="text-align: center;"><span role="presentation"><img unselectable="on" src="https://blog.morphisec.com/hubfs/undefined-Feb-15-2023-01-42-13-1678-AM.png" loading="lazy" alt="‘LockOutboundConnection’ firewall rule&nbsp;"><span></span></span><span data-contrast="auto" xml:lang="EN-US" lang="EN-US"></span><br>&nbsp;<em>‘LockOutboundConnection’ firewall rule&nbsp;</em></p>
<p style="text-align: left;">We came across a thread of several Windows Exchange server owners complaining about the new <a href="https://community.spiceworks.com/topic/2331658-server-2016-cannot-access-anything-outside-itself" rel="noopener" target="_blank">“LockOutboundConnections” firewall rule</a> added to their server that blocks all outbound traffic. The reason for this behavior is due to the scheduled task triggered by a logon event, so every logon to a target machine will block outbound traffic for at least 30 seconds.&nbsp;</p>
<h2 style="text-align: left;">Stopping ProxyShellMiner&nbsp;</h2>
<p style="text-align: left;">ProxyShellMiner doesn’t just slow down organization networks, inflate power bills, overheat equipment, and prevent services from running. It allows threat actors access for even more nefarious ends. Once attackers have a foothold in a network, they have deployed web shells, backdoors, and used tunneling utilities to further compromise victim organizations.&nbsp;&nbsp;</p>
<p style="text-align: left;">Security teams should apply patches <a href="https://support.microsoft.com/en-us/topic/description-of-the-security-update-for-microsoft-exchange-server-2019-2016-and-2013-may-11-2021-kb5003435-028bd051-b2f1-4310-8f35-c41c9ce5a2f1" rel="noopener" target="_blank">KB5003435</a> (CVE-2021-31207) and <a href="https://support.microsoft.com/en-us/topic/description-of-the-security-update-for-microsoft-exchange-server-2019-2016-and-2013-april-13-2021-kb5001779-8e08f3b3-fc7b-466c-bbb7-5d5aa16ef064" rel="noopener" target="_blank">KB5001779</a> (CVE-2021-34473 and CVE-2021-34523) as a first step in preventing these vulnerabilities from being exploited. But some may not be able to do so, and what if an attacker has already established persistence in a network?</p>
<p style="text-align: left;">An effective <a href="/what-is-the-benefit-of-a-defense-in-depth-approach" rel="noopener">defense-in-depth strategy</a> can stop ProxyShellMiner, even if the relevant patches aren’t applied. But what does this entail? Use technology like Automated Moving Target Defense (AMTD) to augment detection-based tools like NGAV, EPP, and EDR/XDR, which don’t reliably stop ProxyShellMiner. MTD can function like a virtual patch by preventing ProxyShellMiner access to runtime memory. It does this by regularly morphing (randomizing) the runtime memory environment and leaving decoy traps where legitimate targets used to be. Any code that tries to engage with the decoys is instantly shut down and trapped for forensic analysis. To learn more about Moving Target Defense, read the white paper: <a href="https://engage.morphisec.com/the-ultimate-ransomware-strategy" rel="noopener"><span style="font-style: italic;">The Ultimate Ransomware Strategy: Zero Trust + Moving Target Defense</span></a>.&nbsp;<br><a href="https://engage.morphisec.com/the-ultimate-ransomware-strategy" rel="noopener"><img src="https://blog.morphisec.com/hs-fs/hubfs/image-Mar-09-2022-06-20-00-29-PM.png?width=640&amp;name=image-Mar-09-2022-06-20-00-29-PM.png" alt="The Ultimate Ransomware Defense white paper" loading="lazy"></a></p>
