                          <h2>Key takeaways</h2>

<ul>
	<li>Bitsight has uncovered a proxy botnet delivered by PrivateLoader and Amadey, two loaders frequently employed by threat actors to distribute malware and build their botnets. We've named this proxy bot malware Socks5Systemz, which is also the name associated with the unique login panel consistently present in all active proxy bot C2 servers.</li>
	<li>While this proxy malware is not new (references on <a href="https://twitter.com/search?q=Socks5systemz&amp;src=recent_search_click&amp;f=live">Twitter</a> trace back to 2016), its usage must have remained under the radar, at least until now.</li>
	<li>Upon researching and mapping the infrastructure behind this botnet, several servers associated with this malware operation were discovered, along with a Telegram user who has built a complete proxy service by leveraging this proxy botnet.</li>
	<li>The proxy service allows clients to choose a subscription ranging from $1 USD to $4000 USD, payable in full using cryptocurrency.</li>
	<li>Based on network telemetry analysis, it is estimated that this botnet has approximately 10.000 infected systems with victims spread across the globe.</li>
	<li>No infected systems communicating with the backconnect servers were observed in Russia. This, combined with various other clues uncovered during the research (such as HTML comments, server error messages, etc.) allows us to assess with medium confidence that operators of the service are based in that geography.</li>
</ul>

<h2>Introduction</h2>

<p>Proxy services offer users the ability to rent a set of IP addresses for internet use, granting a level of online anonymity. Essentially, they make your internet traffic appear as if it's coming from a regular IP address while keeping the real origin hidden.</p>

<p>Recently, our Threat Research team discovered a new malware sample, distributed by the PrivateLoader and Amadey loaders. These two loaders are often used by threat actors to spread malware and build their armies of infected computers, also known as botnets.</p>

<p>This sample, upon reverse engineering, was found to install a proxy bot on infected systems, turning them into proxies capable of forwarding traffic for someone else. We've named this proxy bot <code>Socks5Systemz</code>, a name we found to be present on a login panel in all proxy bot C2 servers.</p>

<h2>Loader analysis</h2>

<p>All samples delivered by PrivateLoader and Amadey dropped and executed a file named <code>previewer.exe</code> that we will refer to as just the loader from now on. This file is responsible for setting up the persistence and injecting the proxy bot in memory.</p>

<h3>The install option</h3>

<p>The install option is responsible for setting up the persistence on the system and to do so it will try to copy the loader to <code>C:\ProgramData\ContentDWSvc\ContentDWSvc.exe</code> and create a Windows service to run the copied loader with both the name and display name set to <code>ContentDWSvc</code>.</p>

<h3>The start option</h3>

<h2>Proxy bot analysis</h2>

<p>The proxy bot payload is a 32 bit DLL file with a size of ~300 KB. The entry point of the proxy bot saves the name of the current filename to a global variable, sets a global flag with the architecture of the system, and it creates a new thread to start the bot main function.</p>

<center>
<article class="align-center media media--type-image media--view-mode-original">
  
      
            <div class="field field--name-field-image field--type-image field--label-hidden field__item">
                            <img src="/sites/default/files/2023/11/01/socks5systemz-proxy-bot-entry.png" width="876" height="214" alt="socks5systemz proxy bot entry" loading="lazy" typeof="foaf:Image">


              </div>
      
  </article>


<p><em>Figure 7: Proxy bot entry point</em></p>
</center>

<p>Next, the bot generates a 32-bit client ID based on the creation date of the Windows directory on the infected system.</p>

<p>On its initial run, the bot collects and stores the current system time (infection time) in a file named <code>C:\ProgramData\ts.dat</code>. Additionally, it fetches a PDF file from <code>hxxp://datasheet[.]fun/manual/avon_4_2022.pdf?&lt;client_id&gt;<client_id></client_id></code>, saving it in the <code>C:\ProgramData folder. </code></p>

<center>
<article class="align-center media media--type-image media--view-mode-original">
  
      
            <div class="field field--name-field-image field--type-image field--label-hidden field__item">
                            <img src="/sites/default/files/2023/11/01/socks5systemz-downloading-pdf.jpg" width="930" height="339" alt="socks5systemz downloading pdf" loading="lazy" typeof="foaf:Image">


              </div>
      
  </article>


<p><em>Image 8: Downloading PDF file from datasheet[.]fun</em></p>
</center>

<p>Despite the fact that the downloaded PDF doesn't contain any particularly interesting data and the domain is under the control of the threat actor, this step appears to be geared towards gathering extra telemetry and potentially aiding in the validation of infected systems.</p>

<center>
<article class="align-center media media--type-image media--view-mode-original">
  
      
            <div class="field field--name-field-image field--type-image field--label-hidden field__item">
                            <img src="/sites/default/files/2023/11/01/socks5systemz-pdf-file-content.jpg" width="930" height="198" alt="socks5systemz pdf file content" loading="lazy" typeof="foaf:Image">


              </div>
      
  </article>


<p><em>Image 9: PDF file content</em></p>
</center>

<p>Finally, the bot tries to get the address of a C2 server that's online. To do so, the bot computes a domain name using a domain generation algorithm and uses a hardcoded list of DNS servers to resolve it.</p>

<center>
<article class="align-center media media--type-image media--view-mode-original">
  
      
            <div class="field field--name-field-image field--type-image field--label-hidden field__item">
                            <img src="/sites/default/files/2023/11/01/socks5systemz-hardcoded-list-dns-resolvers.png" width="716" height="110" alt="socks5systemz hardcoded list dns resolvers" loading="lazy" typeof="foaf:Image">


              </div>
      
  </article>


<p><em>Figure 10: Hardcoded list of DNS resolvers</em></p>
</center>

<p>Since all DNS servers in the list are controlled by this threat actor, any domain that we try to resolve using them will point to a valid command and control server.</p>

<center>
<article class="align-center media media--type-image media--view-mode-original">
  
      
            <div class="field field--name-field-image field--type-image field--label-hidden field__item">
                            <img src="/sites/default/files/2023/11/01/socks5systemz-resolving-a-domain.png" width="661" height="161" alt="socks5systemz resolving a domain" loading="lazy" typeof="foaf:Image">


              </div>
      
  </article>


<p><em>Figure 11: Resolving a domain</em></p>
</center>

<p>​If the bot is unable to get a valid command and control server address using that method, it will send a HTTP GET request to <code>bddns[.]cc</code> to the following endpoint <code>/sign/&lt;rc4 data hex encoded&gt;<rc4 data encoded hex></rc4></code>. The hex encoded data is the result of encrypting a string with the format <code><dga_domain></dga_domain><dga_domain>&lt;dga_domain&gt;:&lt;client_id_hex&gt;</dga_domain><dga_domain><client_id_hex></client_id_hex></dga_domain></code> using RC4 with the key <strong><code>heyfg645fdhwi</code></strong>. The response is also hex encoded and encrypted using the same RC4 key and contains the IP address of a valid command and control server.</p>

<center>
<article class="align-center media media--type-image media--view-mode-original">
  
      
            <div class="field field--name-field-image field--type-image field--label-hidden field__item">
                            <img src="/sites/default/files/2023/11/01/socks5systemz-requesting-c2-address_0.jpg" width="930" height="300" alt="socks5systemz requesting c2 address" loading="lazy" typeof="foaf:Image">


              </div>
      
  </article>


<p><em>Figure 12: Requesting C2 address from bddns[.]cc</em></p>
</center>

<p>After getting the IP address of an active command and control server, the bot is ready to start the C2 communications by doing a HTTP GET request to the following endpoint <code>/single.php?c=&lt;rc4 data hex encoded&gt;.</code></p>

<center>
<article class="align-center media media--type-image media--view-mode-original">
  
      
            <div class="field field--name-field-image field--type-image field--label-hidden field__item">
                            <img src="/sites/default/files/2023/11/01/socks5systemz-bot-request.jpg" width="930" height="86" alt="socks5systemz bot request" loading="lazy" typeof="foaf:Image">


              </div>
      
  </article>


<p><em>Figure 13: Bot request</em></p>
</center>

<p>The data that goes in the c= parameter is the hex encoded result of encrypting the beacon string using the same RC4 key heyfg645fdhwi. The plain text beacon string has the following format: <code>client_id=%.8x&amp;connected=%d&amp;server_port=%d&amp;debug=%d&amp;os=%d.%d.%04d&amp;dgt=%d&amp;dti=%d</code></p>

			<p>&nbsp;</p>
			</td>
		</tr>

<p>While subscribing to the service, the client must provide the IP address from where it will access the proxies. Once the subscription process is complete, the client IP address gets whitelisted across the botnet, and clients can download the list of available proxies which contains the IP addresses of the backconnect servers and the TCP ports assigned to the infected systems.</p>

<h3>Victims</h3>

<p>Since PrivateLoader and Amadey loaders have been one of the main distribution channels for this proxy bot, we expected to see a pretty dispersed geographic distribution for the victims of this botnet, which was exactly what we observed on our network telemetry. Since the beginning of October, we observed approximately 10.000 systems communicating over port 1074/TCP with the backconnect servers.</p>

<p>The top 10 most affected countries are, in order, India, Brasil, Colombia, South Africa, Bangladesh, Argentina, Angola, United States, Suriname, and Nigeria.</p>
