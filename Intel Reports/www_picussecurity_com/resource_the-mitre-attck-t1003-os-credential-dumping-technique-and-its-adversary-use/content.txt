<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">Where are Windows OS Credentials Stored?</span></h2>
<p>In a Windows operating system, credentials are stored in several places:</p>
<ul style="font-size: 18px;">
<li aria-level="1"><strong>Security Account Manager (SAM) database: </strong>The SAM is a protected system file located on the local machine, which stores the hashed versions of the password for all local user accounts on the system.</li>
</ul>
<ul>
<li aria-level="1"><span style="font-size: 18px;"><strong>Local Security Authority Subsystem Service (LSASS) memory:</strong> LSASS is a Windows process responsible for authenticating user logins and enforcing security policies. When a user logs in, the LSASS process retrieves the user's credentials from the SAM database and stores them in memory for the duration of the session.&nbsp;</span></li>
</ul>
<ul style="font-size: 18px;">
<li aria-level="1"><strong>NTDS.dit: </strong>NTDS.dit is a database file on domain controllers containing all of the Active Directory data. The data in the NTDS.dit file is replicated between domain controllers in a domain or forest. If a user's account is in Active Directory, the hashed passwords are stored in the NTDS.dit file. This allows users to authenticate across all domain-joined machines.</li>
</ul>
<ul style="font-size: 18px;">
<li aria-level="1"><strong>Local Security Authority (LSA) Secrets:</strong> LSA secrets is a mechanism that allows storing secrets, such as passwords, in the Windows Registry. These secrets can be used to authenticate services, schedule tasks, and other tasks that require a password.</li>
</ul>
<ul style="font-size: 18px;">
<li aria-level="1"><strong>Cached Domain Credentials: </strong>When a user logs into a Windows computer that is part of a domain, the user's domain credentials are cached on the local machine so that the user can continue to access resources on the network if the domain controller is unavailable. The cached credentials are typically stored in the LSASS memory and can be used to authenticate the user even if the domain controller is not reachable.</li>
</ul>
<ul style="font-size: 18px;">
<li aria-level="1"><strong>Credentials Manager:</strong> Credential Manager is the built-in Windows feature that allows users to store and manage their credentials, like passwords or certificates. These credentials will be used when a user wants to access a network resource, web page, or application that requires a username and password.</li>
</ul>
<ul style="font-size: 18px;">
<li aria-level="1"><strong>Group Policy:</strong> In certain situations, credentials may be stored in Group Policy to allow automatic login for a specific user or group of users. This can be useful in cases where a user needs to access a resource that requires a username and password, but the user is not present to enter the information manually.</li>
</ul>
<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">Where are Linux and macOS OS Credentials Stored?</span></h2>
<p>In Linux and macOS operating systems, user credentials are typically stored in the following places. It's important to note that the exact locations and names of these files may vary depending on the specific Linux distribution or macOS version you are using.</p>
<ul style="font-size: 18px;">
<li aria-level="1">
<p style="font-size: 18px;"><strong>/etc/passwd:</strong> This file is used to store user information, including username, user ID (UID), group ID (GID), and home directory path.</p>
</li>
</ul>
<ul style="font-size: 18px;">
<li aria-level="1">
<p style="font-size: 18px;"><strong>/etc/shadow:</strong> This file is used to store the password hashes and other information related to user authentication, such as the last time the password was changed and the date on which the account will expire. This file is only readable by the root user.</p>
</li>
</ul>
<ul style="font-size: 18px;">
<li aria-level="1">
<p style="font-size: 18px;"><strong>PAM (Pluggable Authentication Modules):</strong> PAM is a framework that allows Linux and macOS systems to use multiple authentication methods, such as local password authentication, Kerberos, and smart cards. PAM is configured through a series of files located in the /etc/pam.d directory.</p>
</li>
</ul>
<ul style="font-size: 18px;">
<li aria-level="1">
<p style="font-size: 18px;"><strong>NSS (Name Service Switch): </strong>This is a facility provided by the operating system that allows switching between different sources of information. For example, information about users, groups, and hosts. It is configured via the /etc/nsswitch.conf file. It can include the files /etc/passwd and /etc/shadow or an external database like LDAP, AD, or NIS.</p>
</li>
</ul>
<ul style="font-size: 18px;">
<li aria-level="1">
<p><span style="font-size: 18px;"><strong>Kerberos: </strong>Kerberos is an authentication protocol that uses tickets to establish secure connections between clients and servers. Kerberos is typically used in enterprise environments and is configured through the krb5.conf file, usually located in the /etc directory</span>.</p>
</li>
</ul>
<h2 style="font-weight: bold; font-size: 26px;"><span style="color: #d80033;">Adversary Use of OS Credential Dumping</span></h2>
<p>After gaining access and elevated privileges to a target system, adversaries harvest as many credentials as possible. Adversaries utilize the OS Credential Dumping technique to collect account login and password from the compromised system's operating system and utilities. These credentials could allow threat actors to gain access to other systems and services in the network with new privileges. Adversaries use the harvested credential information for:</p>
<ul>
<li aria-level="1"><span style="font-size: 18px;">accessing restricted data and critical assets</span></li>
<li aria-level="1"><span style="font-size: 18px;">moving laterally to other hosts in the network</span></li>
<li aria-level="1"><span style="font-size: 18px;">creating new accounts and removing them to impede forensic analysis</span></li>
<li aria-level="1"><span style="font-size: 18px;">figuring out password patterns and policies to harvest other credentials</span></li>
</ul>

<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">#6.1. T1003.001 LSASS Memory</span></h2>
<p>Windows operating systems store the credentials of logged-in users in the Local Security Authority Subsystem Service (LSASS). LSASS allows users and services to access network resources seamlessly without re-entering their credentials. Adversaries harvest credentials by dumping LSASS memory.</p>
<p>LSASS verifies users logging into a Windows system, handles password changes, and creates access tokens. To authenticate users, the <span style="color: #d80033;">lsass.exe</span> process stores and uses credentials in different forms, such as Kerberos tickets, reversibly encrypted plain text, LM hashes, and NT hashes. Users with <span style="color: #d80033;">SYSTEM</span> privilege can interact with the lsass.exe process and dump its memory.</p>
<h3 style="font-size: 22px; font-weight: bold;"><span style="color: #00395c;">Adversary Use of LSASS Memory</span></h3>
<p>Since LSASS memory contains valuable credentials, adversaries utilize various methods and tools to dump LSASS memory and extract credentials:</p>
<ul>
<ul style="font-size: 18px;">
<li aria-level="1">
<p><strong>Mimikatz:</strong> Mimikatz is the most common tool for credential dumping. Mimikatz can extract plaintext passwords, password hashes, PIN codes, and Kerberos tickets from memory. Adversaries also use Mimikatz to perform pass-the-hash, pass-the-ticket, and Golden tickets attacks [1].</p>
</li>
<li aria-level="1">
<p><strong style="font-family: inherit; font-size: inherit; font-style: inherit; text-decoration-line: inherit; color: var(--dark-navy-color);">gsecdump:</strong><span style="font-family: inherit; font-size: inherit; font-style: inherit; font-weight: inherit; text-decoration-line: inherit; color: var(--dark-navy-color);"> Gsecdump is a credential dumping tool that can harvest password hashes from LSA secrets, Active Directory (AD), Security Account Manager (SAM), and logon sessions [2].</span></p>
</li>
<li aria-level="1">
<p><strong>ProcDump:</strong> ProcDump is a legitimate tool that is part of the Microsoft Sysinternals suite [3]. ProcDump monitors applications for CPU spikes and generates a memory dump of processes. However, adversaries abuse ProcDump to dump LSASS memory and extract credentials from the memory dump.</p>
</li>
<li aria-level="1">
<p><strong>Windows Task Manager: </strong>Users can create memory dumps for processes using Windows Task Manager's Create Dump File feature. Adversaries with SYSTEM privilege can use this feature to dump LSASS memory.</p>
</li>
<li aria-level="1">
<p><strong>Direct System Calls and API Unhooking: </strong>Adversaries may use direct system calls to avoid security controls. By executing the system calls directly, adversaries bypass Windows and Native API and may also bypass any user-mode hooks used by security controls. For example, Dumpert can dump LSASS memory via direct system calls and API unhooking [4].</p>
</li>
</ul>
</ul>
<p>Below, you'll discover a list of APT groups and threat actors who are utilizing LSASS memory-dumping techniques.</p>
<p>For instance, as evident in CISA's cybersecurity advisory released in September 2023 (<span style="color: #d80033;">AA23-250A</span>), APT actors utilized <span style="color: #d80033;">ProcDump</span>, a tool typically employed for monitoring and creating crash dumps of processes, to execute a sophisticated cyber attack [5].</p>
<p>They placed ProcDump in the <span style="color: #d80033;">c:\windows\system32\prc64.exe</span> directory for two key purposes: enumerating running processes and applications and, more crucially, dumping credentials from the LSASS. This technique demonstrates the attackers' adeptness in repurposing legitimate system tools for malicious objectives, a tactic often employed to blend in with normal network activities and avoid detection.</p>
<p>Additionally, in another CISA advisory released in March 2023 (<span style="color: #d80033;">AA23-075A</span>), it was seen that<span style="color: #880000;"> </span><span style="color: #d80033;">LockBit 3.0 </span>ransomware group also used Microsoft Sysinternals <span style="color: #d80033;">ProcDump</span> to dump the contents of LSASS.exe [6].&nbsp;</p>
<p>Following this, a subsequent advisory in May 2023 (<span style="color: #d80033;">AA23-136A</span>) highlighted the actions of the <span style="color: #d80033;">BianLian </span>ransomware group, which similarly targeted the<span style="color: #880000;"> </span><span style="color: #d80033;">lsass.exe</span> process but chose to create a memory dump and save it as a CSV file.&nbsp;</p>
<table style="border-collapse: collapse; table-layout: fixed; margin-left: auto; margin-right: auto; border: 1px solid #99acc2;">
<tbody>

<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">#6.2. T1003.002 Security Account Manager</span></h2>
<p>The Security Account Manager (SAM) database stores information related to local accounts, including usernames and hashed passwords. This database resides on the local disk as a file, and adversaries use various methods to access the SAM file and extract credentials.</p>
<p>The SAM is used to store credentials for local accounts. It was introduced with Windows XP and is still in use for the latest versions of Windows. The SAM file is located in <span style="color: #d80033;">%systemroot%\system32\config\SAM</span> and is mounted on the <span style="color: #d80033;">HKLM/SAM</span> registry hive. Also, the same password hashes are stored in <span style="color: #d80033;">%systemroot%\system32\config\SYSTEM</span>, and backup copies can be found in <span style="color: #d80033;">%systemroot%\repair</span> directory.</p>
<p>The SAM database stores hashes of user passwords instead of plaintext versions. While the hash format used for password storage changed over time, the SAM database is still used by the latest versions of Windows.</p>
<ul>
<li style="color: #0e101a;" aria-level="1">
<p><span style="font-size: 18px;"><strong><span style="color: #00395c;">LM: </span></strong><span style="color: #00395c;">(Legacy systems): Introduced in 1987. While turned off by default since Windows Vista/Server 2008, users can enable it afterward.</span></span></p>
</li>
<li style="color: #0e101a;" aria-level="1">
<p><span style="font-size: 18px;"><span style="color: #00395c;"></span><strong><span style="color: #00395c;">NTLMv1:</span></strong><span style="color: #00395c;"> Introduced in 1993. It is an improved version of LM but still contains vulnerabilities.</span></span></p>
</li>
<li style="color: #0e101a;" aria-level="1">
<p style="font-size: 18px;"><strong><span style="color: #00395c;">NTLMv2:</span></strong><span style="color: #00395c;"> This updated version of NTLMv1 includes additional security features, such as a challenge/response mechanism to provide message integrity and replay protection. It's mainly used in Windows operating systems and older versions than Windows NT3.1 and Windows 2000.</span></p>
</li>
<li style="color: #0e101a;" aria-level="1">
<p style="font-size: 18px;"><strong><span style="color: #00395c;">NTLMv2 Session Security: </span></strong><span style="color: #00395c;">This is an update of NTLMv2 that includes additional security features, like signing and sealing the messages, more robust encryption keys, and secure channel protection.</span></p>
</li>
<li style="color: #0e101a;" aria-level="1">
<p style="font-size: 18px;"><strong><span style="color: #00395c;">Kerberos:</span></strong><span style="color: #00395c;"> This is an industry-standard authentication protocol used in Windows operating systems.</span></p>
</li>
<li style="color: #0e101a;" aria-level="1">
<p style="font-size: 18px;"><strong><span style="color: #00395c;">bcrypt:</span></strong><span style="color: #00395c;"> A more advanced password hashing algorithm designed to replace md5crypt, Blowfish-based crypt(3) algorithm.</span></p>
</li>
<li style="color: #0e101a;" aria-level="1">
<p style="font-size: 18px;"><span style="color: #00395c;"></span><strong style="font-family: inherit; font-size: inherit; font-style: inherit; text-decoration-line: inherit; color: var(--dark-navy-color);">scrypt:</strong><span style="font-family: inherit; font-size: inherit; font-style: inherit; font-weight: inherit; text-decoration-line: inherit; color: var(--dark-navy-color);"> Another advanced password hashing algorithm, designed to be more computationally expensive than bcrypt, better suited for usage with stronger user authentication.</span></p>
</li>
</ul>
<p>Since the password is stored in a one-way format (i.e., irreversible), it is not feasible to get the original password from the hashed output as long as the length and complexity of the password are not susceptible to birthday attacks.&nbsp;</p>
<p><span style="color: #d80033; font-size: 18px;"><span style="color: #00395c;">A </span><strong><span style="color: #00395c;">one-way hash function</span></strong><span style="color: #00395c;"> is a mathematical function that converts an input string of variable length into a fixed-length binary sequence. This sequence is difficult to reverse, meaning it is difficult to use the output (the hash) to determine the original input string. One-way hash functions are commonly used to securely store passwords and other sensitive data.</span></span></p>

<p>So far, we have seen several examples of SAM database dumping through registry manipulation. And even though hash functions are designed to be one-way, having an output makes it impossible to learn the input; hashing passwords does not guarantee a foolproof security measure. With a list of dumped valid account credentials, adversaries can perform offline password cracking attacks to find the cleartext password by trying many combinations of characters and comparing the resulting hashes to the stored password hash.</p>
<p>There are several ways to perform an offline password-cracking attack:</p>
<h4><span style="color: #00395c; font-size: 20px; font-family: var(--primary-font); font-weight: bold;">1. Brute-Force Attacks</span></h4>
<p>In this attack type, adversaries try all possible combinations of characters up to a certain length and character set. The length and set of characters are generally defined by adversaries through gaining knowledge of the organization's password policy. It is important to note that as a password's complexity increases, brute-force attacks become significantly time-consuming and inefficient. For instance, adversaries can crack passwords with 12 characters using ChatGPT hardware in 8 months [10]. However, it would take them 3000 years to crack 14-character passwords with the same tools<span style="font-size: 11px; color: #000000;">.</span></p>
<h4><span style="color: #00395c; font-size: 20px; font-family: var(--primary-font); font-weight: bold;">2. Dictionary Attack</span></h4>
<p>In dictionary attacks, an adversary tries a predefined list of words and phrases commonly used as passwords.These attacks can be effective, but how fast they can be achieved depends on the information about a target, such as their birthday and birthplace, the name of their children or pet, which sports team they are a fan of, etc. In some cases, adversaries leverage hybrid attacks using brute force and dictionary attacks by trying a combination of commonly used words and randomly generated characters.</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;"><span style="font-family: var(--secondary-font);">3. Rainbow Table Attacks</span></span></h4>
<p>A rainbow table is a precomputed table of hash values that can be used to speed up the process of cracking passwords. A rainbow table attack works by comparing the target password hash with the hashes in the rainbow table to see if there is a match. The corresponding password can be retrieved from the table and used to log in if a match is found.</p>
<p>A significant benefit of using rainbow tables as an adversary is that they can avoid the hash generation process. For example, if all sets of passwords of 1-8 characters, consisting of the ASCII-32-95 characters, get hashed by the NTLM hashing algorithm, the key space would become 6,704,780,954,517,120 ≈ 2^52.6, which is approximately 460 GB.&nbsp;</p>
<p>Hence, instead of trying to generate a hash of all plaintext within a specific range, attackers can directly look up the hash in the table (some algorithms speed up the checking process) and retrieve the corresponding password.</p>
<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">#6.3. T1003.003 NTDS</span></h2>
<p>The NTDS.dit file is a database that stores information about Active Directory Domain Services (AD DS), including user objects, groups, and group membership. It also contains the hashed passwords for all users within the domain, making it another juicy target from which adversaries can dump credentials.</p>

<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">#6.4. T1003.004 LSA Secrets</span></h2>
<p>Local Security Authority (LSA) Secrets are sensitive information, such as credentials and secrets, that the LSA of a Windows operating system stores. The LSA is an operating system component responsible for managing security-related functions, such as authentication and authorization. LSA Secrets may include various information, such as password hashes, security keys, and other sensitive data.</p>
<p>LSA Secrets are stored in a protected location on the system and are typically only accessible to the operating system and trusted applications. The LSA uses them to perform various security-related tasks, such as authenticating users and granting access to resources.</p>
<p>LSA Secrets may be stored in a number of locations, including the system memory and the registry. On Windows systems, LSA Secrets may be stored in the <span style="color: #d80033;">HKEY_LOCAL_MACHINE\SECURITY\Policy\Secrets</span> registry key.</p>

<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">#6.5. T1003.005 Cached Domain Credentials</span></h2>
<p>In situations where a domain-joined computer encounters difficulty connecting to AD DS during a user's logon process, the system caches domain credentials in the registry for authentication purposes. This local caching of logon information for domain accounts ensures that users can still access their accounts even when a connection to a domain controller is unavailable during subsequent logons.</p>
<p>The storage mechanism for these cached credentials is referred to as DCC2, which stands for Domain Cached Credentials version 2. DCC2 serves as a security feature within the Microsoft Windows operating system, enabling the caching of domain credentials on a system. This functionality empowers users to log in to the domain even when they are not connected to the network, enhancing the usability of Windows systems. It acts as a secondary authentication method when a connection to the domain controller cannot be established.</p>
<p>When DCC2 is activated on a system, domain credentials are stored either in the <span style="color: #d80033;">SAM database </span>or the <span style="color: #d80033;">Credential Manager</span>, depending on the Windows version in use. These cached credentials are encrypted, ensuring their security, and can only be accessed by the system when a user attempts to log in to the domain. DCC2 encompasses two types of cached domain credentials, both of which are employed by the system for authentication purposes:</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">1. mscache2</span></h4>
<p>mscache2 is a cached domain credential used by Windows systems running Windows 2000 and later. It stores the password hash of the user's domain account, salt value, and other metadata. When a user attempts to log in to the domain, the system uses the mscache2 credentials to authenticate the user to the domain controller.</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">2. mcash2</span></h4>
<p>mcash2 (Microsoft CAched haSH) is a newer version of the mscache2 credential, and it is used by Windows systems running Windows 8 and later. It stores the password hash of the user's domain account, additional metadata, and a more robust encryption key.</p>
<h3 style="font-size: 22px; font-weight: bold;"><span style="color: #00395c;">Adversary Use of Cached Domain Credentials</span></h3>

<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">#6.6. T1003.006 DCSync</span></h2>
<p>DCSync is a feature in Microsoft Domain Controllers (DC) that allows replication of the Active Directory (AD) database from a primary to a secondary DC, ensuring all DCs have the latest directory copy. It's commonly used by adversaries with sufficient permissions to extract sensitive information like credentials.&nbsp;</p>
<p>DCSync employs the Remote Procedure Call (RPC) protocol, requiring "Replicate Directory Changes" permission on the domain object in AD. It can replicate the entire database or specific parts, either in real-time or on a schedule. Primarily used by administrators to maintain AD database integrity and availability, DCSync is integral for keeping DCs updated and often works alongside other replication technologies.</p>
<h3 style="font-size: 22px; font-weight: bold;"><span style="color: #00395c;">Adversary Use of DCSync</span></h3>
<p><span style="color: #00395c;">Attackers can abuse DCSync in their attack campaigns to obtain sensitive information from the AD database. An adversary can do this with sufficient permissions and credentials, using DCSync to replicate the AD database from a primary DC to a secondary DC and then extracting sensitive information such as user passwords and other credentials.</span></p>
<p><span style="color: #00395c;">There are several ways in which attackers may abuse DCSync in their attack campaigns:</span></p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">1. Obtaining User Credentials</span></h4>
<p><span style="color: #00395c;">An attacker may use DCSync to replicate the AD database and extract user credentials, such as passwords, to gain unauthorized access to the system. This can be done without leaving any trace of the operation on the primary DC, making it difficult to detect.</span></p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">2. Conducting Lateral Movement</span></h4>
<p><span style="color: #00395c;">An attacker may use DCSync to obtain credentials for other systems and services on the network to move laterally within the network and potentially compromise other systems.</span></p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">3. Escalating Privileges</span></h4>
<p><span style="color: #00395c;">An attacker may use DCSync to obtain credentials for high-privilege accounts, such as administrator accounts, to escalate their privileges on the system. This can allow them to perform actions that would otherwise be restricted to them.&nbsp;</span></p>

<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #00395c;"><span style="color: #d80033;">#6.7. T1003.007 Proc Filesystem</span></span></h2>
<p>The proc filesystem (procfs) is a virtual filesystem in the Linux kernel that provides information about processes and other system information. It is a pseudo-filesystem, meaning it does not exist on a physical storage device but rather is generated dynamically by the kernel as needed. Adversaries may attempt to use the procfs to obtain credentials and other sensitive information from a system.&nbsp;</p>
<p>The procfs is typically mounted in the /proc directory, and it consists of a series of virtual files and directories that provide information about various aspects of the system. Some examples of the types of information that can be found in the procfs include:</p>
<ul>
<li aria-level="1"><span style="font-size: 18px;"><strong>Process information: </strong>The /proc directory contains a subdirectory for each running process on the system, with a numeric name corresponding to the process ID (PID). These directories contain virtual files with information about the process, such as its command line arguments, current working directory, and open file descriptors.</span></li>
<li aria-level="1"><span style="font-size: 18px;"><strong>Kernel information</strong>: The procfs also contains virtual files and directories with information about the kernel and its configuration. This can include information about the version of the kernel, the system architecture, and the loaded kernel modules.</span></li>
<li aria-level="1"><span style="font-size: 18px;"><strong>Hardware information:</strong> The procfs contains virtual files with information about the hardware on the system, such as the processor type and model, the amount of memory installed, and the configured interrupts and I/O ports.</span></li>
</ul>
<h3 style="font-size: 22px; font-weight: bold;"><span style="color: #00395c;">Adversary Use of Proc Filesystem</span></h3>
<p>The proc filesystem (procfs) can potentially be used by attackers to obtain credentials and other sensitive information about the operating system and its processes. There are several ways in which attackers may use the procfs for this purpose:</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">1. Extracting Command-line Arguments</span></h4>
<p>The procfs contains virtual files with the command-line arguments of each running process on the system. An attacker may attempt to read these files in order to obtain any sensitive information that may have been passed as command-line arguments, such as passwords or API keys.</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">2. Reading Environment Variables</span></h4>
<p>The procfs contains virtual files with the environment variables of each running process. An attacker may attempt to read these files in order to obtain sensitive information that may be stored in environment variables, such as credentials for external services or database servers.</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">3. Obtaining Process Information</span></h4>
<p>The procfs contains virtual files with information about the processes running on the system, including the current working directory, open file descriptors, and other details. An attacker may use this information to gather intelligence about the system and potentially identify processes that may be of interest.</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">4. Reading Kernel Information</span></h4>
<p>The procfs contains virtual files and directories with information about the kernel and its configuration. An attacker may attempt to read these files in order to obtain information about the version of the kernel, the system architecture, and the loaded kernel modules. This information may be used to tailor an attack to the specific system and potentially exploit known vulnerabilities.</p>
<p>An adversary may use the following tools to extract credentials using the proc file system:</p>
<ul>
<li aria-level="1"><span style="font-size: 18px;"><strong>MimiPenguin</strong> is an open-source tool capable of dumping process memory and harvesting passwords and hashes by searching for text strings and regular expressions.&nbsp;</span></li>
<li aria-level="1"><span style="font-size: 18px;"><strong>LaZagne</strong> can extract credential information from process memory with the memorydump.py module. It includes regex patterns for passwords of common websites, such as Gmail, Dropbox, Salesforce, PayPal, Twitter, Github, and Slack. Lazagne uses these patterns to dump cleartext passwords from the browser's memory. Its mimipy.py module is a Python port of MimiPenguin.&nbsp;</span></li>
<li aria-level="1"><span style="font-size: 18px;"><strong>Procdump</strong> for Linux is a Linux reworking of the classic ProcDump tool from the Sysinternals suite of tools for Windows. It provides Linux developers with a straightforward method for generating core dumps of their applications in response to performance triggers. Naturally, adversaries utilize this tool to dump process memory and extract credentials from dumped memory.</span></li>
</ul>
<h2 style="font-size: 26px; font-weight: bold;"><span style="color: #d80033;">#6.8. T1003.008 /etc/passwd and /etc/shadow</span></h2>
<p>The /etc/passwd and /etc/shadow files store information about user accounts on a Unix-like system. While the /etc/passwd file stores user account information, the /etc/shadow file consists of password hashes. MD5, SHA-256, and SHA-512 are hash algorithms used for these passwords. The contents of these files are dumped by adversaries for offline password cracking.</p>
<p>The /etc/passwd file stores information about each user account, including the username, user ID (UID), group ID (GID), and home directory. It does not contain the user's password, however. The /etc/shadow file stores the hashed password for each user account, along with other information, such as the password expiration date and any password reset flags. This file is typically readable only by the root user, as it contains sensitive information.</p>
<h3 style="font-size: 22px; font-weight: bold;"><span style="color: #00395c;">Adversary Use of /etc/passwd and /etc/shadow</span></h3>
<p>Adversaries may attempt to access or modify the contents of the /etc/passwd and /etc/shadow files on a Unix-like system in order to compromise user accounts and gain unauthorized access to the system. There are several ways in which attackers may use these files for malicious purposes:</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">1. Adding new user accounts</span></h4>
<p>An attacker may add a new user account to the<span style="color: #880000;"> </span><span style="color: #d80033;">/etc/passwd</span> file with a known or easily guessable password. This would allow them to log in to the system using the new account and potentially escalate their privileges.</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">2. Modifying existing accounts</span></h4>
<p>For example, an attacker may modify an existing user account in the <span style="color: #d80033;">/etc/passwd</span> file by changing the home directory or group membership to escalate their privileges. They may also modify the user's password in the <span style="color: #d80033;">/etc/shadow</span><span style="color: #880000;"> </span>file by changing it to a known password or setting it never to expire.</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">3. Gaining access to encrypted passwords</span></h4>
<p>An attacker may attempt to gain access to the<span style="color: #880000;"> </span><span style="color: #d80033;">/etc/shadow</span> file in order to obtain the encrypted passwords for offline cracking. They may then use tools such as <span style="color: #d80033;">John the Ripper</span> or <span style="color: #d80033;">Hashcat</span> to try to crack the passwords and gain access to user accounts.</p>
<h4 style="font-size: 20px; font-weight: bold;"><span style="color: #00395c;">4. Using these files as part of a larger attack</span></h4>
<p>An attacker may use the /etc/passwd and /etc/shadow files in combination with other tactics and techniques as part of a larger attack against a system, such as moving laterally within the network.</p>

<p>The Unshadow tool is a specialized utility in Linux environments designed to merge the contents of the <span style="color: #d80033;">/etc/passwd</span> and <span style="color: #d80033;">/etc/shadow</span> files. Combining these two files, Unshadow creates a single file with usernames and associated hashed passwords. The command required for this tool is as follows [14].</p>
<table style="border-collapse: collapse; table-layout: fixed; margin-left: auto; margin-right: auto; border: 1px solid #99acc2;">
<tbody>
<tr>
<td style="background-color: #f0f0f0;">
<p><span style="color: #444444; background-color: #f0f0f0;">unshadow /etc/passwd /etc/shadow &gt; password_file</span></p>
</td>
</tr>
</tbody>
</table>
<ul>
<li style="color: #000000;" aria-level="1">
<p style="font-size: 18px;"><strong><span style="color: #00395c;">LaZagne</span></strong></p>
</li>
</ul>
<p>LaZagne stands out as a versatile credential extraction tool, capable of retrieving sensitive information from various systems, including <span style="color: #d80033;">Unix-like</span> systems. Specifically, on Linux, its <span style="color: #d80033;">shadow.py</span> module, found under the <span style="color: #d80033;">/Linux/lazagne/softwares/sysadmin</span> directory, is adept at accessing credential data from the /etc/shadow file. This file is critical as it contains hashed passwords of system users. The command used in this technique is as follows [15].</p>
<table style="border-collapse: collapse; table-layout: fixed; margin-left: auto; margin-right: auto; border: 1px solid #99acc2;">
<tbody>
<tr>
<td style="background-color: #f0f0f0;">
<p><span style="color: #444444; background-color: #f0f0f0;">sudo lazagne all</span></p>
</td>
</tr>
</tbody>
</table>
<p><span style="font-size: 18px;"><span style="color: #d80033;">LaZagne</span><span style="color: #00395c;">'s capability extends to performing dictionary attacks on several hash formats stored in </span><span style="color: #d80033;">/etc/shadow</span><span style="color: #00395c;">, including </span><span style="color: #d80033;">MD5</span><span style="color: #00395c;">, </span><span style="color: #d80033;">Blowfish</span><span style="color: #00395c;">, </span><span style="color: #d80033;">SHA-256</span><span style="color: #00395c;">, and </span><span style="color: #d80033;">SHA-512</span><span style="color: #00395c;">. This functionality allows it to potentially crack and reveal user passwords, assuming it operates with the necessary root privileges.</span></span></p>
