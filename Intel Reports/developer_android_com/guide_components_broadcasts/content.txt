<p>Android apps can send or receive broadcast messages from the Android system and
other Android apps, similar to the
<a href="https://en.wikipedia.org/wiki/Publish%E2%80%93subscribe_pattern">publish-subscribe</a>
design pattern. These broadcasts are sent when an event of interest occurs.
For example, the Android system sends broadcasts when various system events
occur, such as when the system boots up or the device starts charging. Apps
can also send custom broadcasts, for example, to notify other apps of
something that they might be interested in (for example, some new data has
been downloaded).</p>

<p>The system optimizes the delivery of broadcasts in order to maintain
optimal system health. Therefore, delivery times of broadcasts are not
guaranteed. Apps that need low-latency interprocess communication should
consider <a href="/guide/components/bound-services">bound services</a>.</p>

<p>Apps can register to receive specific broadcasts. When a broadcast is sent,
the system automatically routes broadcasts to apps that have subscribed to
receive that particular type of broadcast.</p>

<p>Generally speaking, broadcasts can be used as a messaging system across apps
and outside of the normal user flow. However, you must be careful not to abuse
the opportunity to respond to broadcasts and run jobs in the background that
can contribute to a slow system performance.</p>
<aside class="note"><strong>Note:</strong><span> Whenever possible, keep broadcasts private to your app.</span></aside>
<h2 id="system-broadcasts" data-text="About system broadcasts" tabindex="-1">About system broadcasts</h2>

<p>The system automatically sends broadcasts when various system events occur,
such as when the system switches in and out of airplane mode. System
broadcasts are sent to all apps that are subscribed to receive the
event.</p>

<p>The broadcast message itself is wrapped in an <code translate="no" dir="ltr"><a href="/reference/android/content/Intent">Intent</a></code>
object whose action string identifies the event that occurred (for example
<code translate="no" dir="ltr">android.intent.action.AIRPLANE_MODE</code>).  The intent may also include
additional information bundled into its extra field. For example, the airplane
mode intent includes a boolean extra that indicates whether or not Airplane
Mode is on.</p>

<p>For more information about how to read intents and get the action string from
an intent, see <a href="/guide/components/intents-filters">Intents and Intent
Filters</a>.</p>

<p>For a complete list of system broadcast actions, see the
<code translate="no" dir="ltr">BROADCAST_ACTIONS.TXT</code> file in the Android SDK. Each broadcast action has a
constant field associated with it. For example, the value of the constant
<code translate="no" dir="ltr"><a href="/reference/android/content/Intent#ACTION_AIRPLANE_MODE_CHANGED">ACTION_AIRPLANE_MODE_CHANGED</a></code> is
<code translate="no" dir="ltr">android.intent.action.AIRPLANE_MODE</code>. Documentation for each broadcast action
is available in its associated constant field.</p>

<h3 id="changes-system-broadcasts" data-text="Changes to system broadcasts" tabindex="-1">Changes to system broadcasts</h3>

<p>As the Android platform evolves, it periodically changes how system broadcasts
behave. Keep the following changes in mind to support all versions of Android.</p>

<h4 id="android-14" data-text="Android 14" tabindex="-1">Android 14</h4>

<p>While apps are in a <a href="/guide/components/activities/process-lifecycle">cached
state</a>, broadcast delivery is
optimized for system health. For example, less important system broadcasts such
as <a href="/reference/android/content/Intent#ACTION_SCREEN_ON"><code translate="no" dir="ltr">ACTION_SCREEN_ON</code></a> are
deferred while the app is in a cached state. Once the app goes from the cached
state into an <a href="/guide/components/activities/process-lifecycle">active process
lifecycle</a>, the system delivers
any deferred broadcasts.</p>

<p>Important broadcasts that are <a href="#manifest-declared-receivers">declared in the
manifest</a> temporarily remove apps from the cached
state for delivery.</p>

<h4 id="android_9" data-text="Android 9" tabindex="-1">Android 9</h4>

<p>Beginning with Android 9 (API level 28), The
<a href="/reference/android/net/wifi/WifiManager#NETWORK_STATE_CHANGED_ACTION"><code translate="no" dir="ltr">NETWORK_STATE_CHANGED_ACTION</code></a>
broadcast doesn&#39;t receive information about the user&#39;s location or personally
identifiable data.</p>

<p>In addition, if your app is installed on a device running Android 9 or higher,
system broadcasts from Wi-Fi don&#39;t contain SSIDs, BSSIDs, connection
information, or scan results. To get this information, call
<a href="/reference/android/net/wifi/WifiManager#getConnectionInfo()"><code translate="no" dir="ltr">getConnectionInfo()</code></a>
instead.</p>

<h4 id="android_80" data-text="Android 8.0" tabindex="-1">Android 8.0</h4>

<p>Beginning with Android 8.0 (API level 26), the system imposes additional
restrictions on manifest-declared receivers.</p>

<p>If your app targets Android 8.0 or higher, you cannot use the manifest to
declare a receiver for most implicit broadcasts (broadcasts that don&#39;t target
your app specifically). You can still use a
<a href="#context-registered-receivers">context-registered receiver</a> when the
user is actively using your app.</p>

<h4 id="android_70" data-text="Android 7.0" tabindex="-1">Android 7.0</h4>

<p>Android 7.0 (API level 24) and higher don&#39;t send the following system
broadcasts:</p>

<ul>
<li><code translate="no" dir="ltr"><a href="/reference/android/hardware/Camera#ACTION_NEW_PICTURE">ACTION_NEW_PICTURE</a></code></li>
<li><code translate="no" dir="ltr"><a href="/reference/android/hardware/Camera#ACTION_NEW_VIDEO">ACTION_NEW_VIDEO</a></code></li>
</ul>

<p>Also, apps targeting Android 7.0 and higher must register the <code translate="no" dir="ltr"><a href="/reference/android/net/ConnectivityManager#CONNECTIVITY_ACTION">CONNECTIVITY_ACTION</a></code> broadcast
using <code translate="no" dir="ltr"><a href="/reference/android/content/Context#registerReceiver(android.content.BroadcastReceiver, android.content.IntentFilter)">registerReceiver(BroadcastReceiver, IntentFilter)</a></code>.
Declaring a receiver in the manifest doesn&#39;t work.</p>

<h2 id="receiving-broadcasts" data-text="Receiving broadcasts" tabindex="-1">Receiving broadcasts</h2>

<p>Apps can receive broadcasts in two ways: through manifest-declared receivers
and context-registered receivers.</p>

<h3 id="manifest-declared-receivers" data-text="Manifest-declared receivers" tabindex="-1">Manifest-declared receivers</h3>

<p>If you declare a broadcast receiver in your manifest, the system launches your
app (if the app is not already running) when the broadcast is sent.</p>
<aside class="note"><strong>Note:</strong><span> If your app targets API level 26 or higher, you cannot use the manifest
to declare a receiver for <em>implicit</em> broadcasts (broadcasts that do not target
your app specifically), except for a few implicit broadcasts that are <a href="/guide/components/broadcast-exceptions">exempted
from that restriction</a>. In most
cases, you can use <a href="/topic/performance/scheduling">scheduled jobs</a> instead.</span></aside>
<p>To declare a broadcast receiver in the manifest, perform the following steps:</p>

<ol>
<li><p>Specify the <a href="/guide/topics/manifest/receiver-element"><code translate="no" dir="ltr">&lt;receiver&gt;</code></a>
element in your app&#39;s manifest.</p>

<pre class="prettyprint lang-xml" translate="no" dir="ltr">
&lt;!-- If this receiver listens for broadcasts sent from the system or from
     other apps, even other apps that you own, set android:exported to "true". --&gt;
&lt;receiver android:name=".MyBroadcastReceiver" android:exported="false">
    &lt;intent-filter>
        &lt;action android:name="<var translate="no">APP_SPECIFIC_BROADCAST</var>" />
    &lt;/intent-filter>
&lt;/receiver>
</pre>

<p>The intent filters specify the broadcast actions your receiver subscribes to.</p></li>
<li><p>Subclass <code translate="no" dir="ltr"><a href="/reference/android/content/BroadcastReceiver">BroadcastReceiver</a></code> and implement <code translate="no" dir="ltr"><a href="/reference/android/content/BroadcastReceiver#onReceive(android.content.Context, android.content.Intent)">onReceive(Context, Intent)</a></code>. The
broadcast receiver in the following example logs and displays the contents
of the broadcast:</p>

<div>
<div class="ds-selector-tabs"><section><h3 id="kotlin" data-text="Kotlin" tabindex="-1">Kotlin</h3>
<pre class="prettyprint lang-kotlin" translate="no" dir="ltr">

    override fun onReceive(context: Context, intent: Intent) {
        StringBuilder().apply {
            append("Action: ${intent.action}\n")
            append("URI: ${intent.toUri(Intent.URI_INTENT_SCHEME)}\n")
            toString().also { log ->
                Log.d(TAG, log)

                val binding = <var translate="no">ActivityNameBinding</var>.inflate(layoutInflater)
                val view = binding.root
                setContentView(view)

                Snackbar.make(view, log, Snackbar.LENGTH_LONG).show()
            }
        }
    }
}
</pre>
</section><section><h3 id="java" data-text="Java" tabindex="-1">Java</h3>
<pre class="prettyprint lang-java" translate="no" dir="ltr">

<p>To enable view binding,
<a href="/topic/libraries/view-binding">configure viewBinding</a> in your module-level
build.gradle file.</p></li>
</ol>

<p>The system package manager registers the receiver when the app is installed.
The receiver then becomes a separate entry point into your app which means
that the system can start the app and deliver the broadcast if the app is not
currently running.</p>

<p>The system creates a new <code translate="no" dir="ltr"><a href="/reference/android/content/BroadcastReceiver">BroadcastReceiver</a></code> component
object to handle each broadcast that it receives. This object is valid only
for the duration of the call to <code translate="no" dir="ltr"><a href="/reference/android/content/BroadcastReceiver#onReceive(android.content.Context, android.content.Intent)">onReceive(Context, Intent)</a></code>. Once your code
returns from this method, the system considers the component  no longer
active.</p>

<h3 id="context-registered-receivers" data-text="Context-registered receivers" tabindex="-1">Context-registered receivers</h3>

<p>Context-registered receivers receive broadcasts as long as their registering
context is valid. For an example, if you register within an
<code translate="no" dir="ltr"><a href="/reference/android/app/Activity">Activity</a></code>
context, you receive broadcasts as long as the activity is not destroyed. If you
register with the Application context, you receive broadcasts as long as the app
is running.</p>

<p>To register a receiver with a context, perform the following steps:</p>

<ol>
<li><p>In your app&#39;s module-level build file, include version 1.9.0 or higher of
the <a href="/jetpack/androidx/releases/core">AndroidX Core library</a>:</p>

<li><p>Choose whether the broadcast receiver should be exported and visible to
other apps on the device. If this receiver is listening for broadcasts sent

<aside class="note"><p>Some system broadcasts come from highly privileged
apps, such as Bluetooth and telephony, that are part of the Android
framework but do not run under the system's unique process ID (UID). To
receive all system broadcasts, including broadcasts from highly privileged
apps, flag your receiver with <code translate="no" dir="ltr">RECEIVER_EXPORTED</code>.</p>
<p>If you flag your receiver with <code translate="no" dir="ltr">RECEIVER_NOT_EXPORTED</code>, the
receiver is able to receive some system broadcasts and broadcasts from your
app, but not broadcasts from the highly privileged apps.</p>
<p>If your app is listening for multiple broadcasts, but only some should be
flagged <code translate="no" dir="ltr">RECEIVER_NOT_EXPORTED</code> and some
<code translate="no" dir="ltr">RECEIVER_EXPORTED</code>, partition the broadcasts among different
broadcast receivers.</p></aside>

Be sure to unregister the receiver when you no longer need it or the
context is no longer valid.</p>

<p>Be mindful of where you register and unregister the receiver,  for
example, if you register a receiver in <code translate="no" dir="ltr"><a href="/reference/android/app/Activity#onCreate(android.os.Bundle)">onCreate(Bundle)</a></code> using the activity&#39;s context, you
should unregister it in <code translate="no" dir="ltr"><a href="/reference/android/app/Activity#onDestroy()">onDestroy()</a></code> to
prevent leaking the receiver out of the activity context. If you register
a receiver in <code translate="no" dir="ltr"><a href="/reference/android/app/Activity#onResume()">onResume()</a></code>, you should
unregister it in <code translate="no" dir="ltr"><a href="/reference/android/app/Activity#onPause()">onPause()</a></code> to prevent
registering it multiple times (If you don&#39;t want to receive broadcasts
when paused, and this can cut down on unnecessary system overhead). Do not
unregister in <code translate="no" dir="ltr"><a href="/reference/android/app/Activity#onSaveInstanceState(android.os.Bundle)">onSaveInstanceState(Bundle)</a></code>,
because this isn&#39;t called if the user moves back in the history stack.</p></li>
</ol>

<h3 id="effects-process-state" data-text="Effects on process state" tabindex="-1">Effects on process state</h3>

<p>Whether your <code translate="no" dir="ltr"><a href="/reference/android/content/BroadcastReceiver">BroadcastReceiver</a></code>
is operating or not affects its contained process, which can alter its
system-killing likelihood. A foreground process executes a receiver&#39;s <code translate="no" dir="ltr"><a href="/reference/android/content/BroadcastReceiver#onReceive(android.content.Context, android.content.Intent)">onReceive()</a></code> method. The
system runs the process except under extreme memory pressure.</p>

<p>The BroadcastReceiver is deactivated after <code translate="no" dir="ltr">onReceive()</code>. The receiver&#39;s host
process is only as significant as its app components. If that process hosts only
a manifest-declared receiver (a frequent occurrence for apps the user has never
or not recently interacted with), the system may kill it after <code translate="no" dir="ltr">onReceive()</code> to make
resources available for other more critical processes.</p>

<p>Thus, broadcast receivers should not initiate long-running background threads.
The system can stop the process at any moment after <code translate="no" dir="ltr">onReceive()</code> to reclaim
memory, terminating the created thread. To keep the process alive, schedule a
<code translate="no" dir="ltr"><a href="/reference/android/app/job/JobService">JobService</a></code>
from the receiver using the <code translate="no" dir="ltr"><a href="/reference/android/app/job/JobScheduler">JobScheduler</a></code>
so the system knows the process is still working.
<a href="/guide/background">Background Work Overview</a> provides more details.</p>

<h2 id="sending-broadcasts" data-text="Sending broadcasts" tabindex="-1">Sending broadcasts</h2>

<p>Android provides three ways for apps to send broadcast:</p>

<ul>
<li>The <code translate="no" dir="ltr"><a href="/reference/android/content/Context#sendOrderedBroadcast(android.content.Intent, java.lang.String)">sendOrderedBroadcast(Intent, String)</a></code>
method sends broadcasts to one receiver at a time. As each receiver executes
in turn, it can propagate a result to the next receiver, or it can
completely abort the broadcast so that it won&#39;t be passed to other
receivers. The order receivers run in can be controlled with the
android:priority attribute of the matching intent-filter; receivers with the
same priority will be run in an arbitrary order.</li>
<li>The <code translate="no" dir="ltr"><a href="/reference/android/content/Context#sendBroadcast(android.content.Intent)">sendBroadcast(Intent)</a></code> method sends
broadcasts to all receivers in an undefined order. This is called a Normal
Broadcast. This is more efficient, but means that receivers cannot read
results from other receivers, propagate data received from the broadcast, or
abort the broadcast.</li>
</ul>

<p>The following code snippet demonstrates how to send a broadcast by creating an
Intent and calling <code translate="no" dir="ltr"><a href="/reference/android/content/Context#sendBroadcast(android.content.Intent)">sendBroadcast(Intent)</a></code>.</p>

<div>
<div class="ds-selector-tabs"><section><h3 id="kotlin" data-text="Kotlin" tabindex="-1">Kotlin</h3>
<pre class="prettyprint lang-kotlin" translate="no" dir="ltr">
Intent().also { intent ->
    intent.setAction("com.example.broadcast.MY_NOTIFICATION")
    intent.putExtra("data", "Nothing to see here, move along.")
    sendBroadcast(intent)
}
</pre>
</section><section><h3 id="java" data-text="Java" tabindex="-1">Java</h3>
<pre class="prettyprint lang-java" translate="no" dir="ltr">
Intent intent = new Intent();
intent.setAction("com.example.broadcast.MY_NOTIFICATION");
intent.putExtra("data", "Nothing to see here, move along.");
sendBroadcast(intent);
</pre>
</section></div>
</div>

<p>The broadcast message is wrapped in an <code translate="no" dir="ltr"><a href="/reference/android/content/Intent">Intent</a></code> object.
The intent&#39;s action string must provide the app&#39;s Java package name syntax and
uniquely identify the broadcast event. You can attach additional information
to the intent with <code translate="no" dir="ltr"><a href="/reference/android/content/Intent#putExtra(java.lang.String, android.os.Bundle)">putExtra(String, Bundle)</a></code>.
You can also limit a broadcast to a set of apps in the same organization by
calling <code translate="no" dir="ltr"><a href="/reference/android/content/Intent#setPackage(java.lang.String)">setPackage(String)</a></code> on the intent.</p>
<aside class="note"><strong>Note:</strong><span> Although intents are used for both sending broadcasts and starting
activities with <code translate="no" dir="ltr"><a href="/reference/android/content/Context#startActivity(android.content.Intent)">startActivity(Intent)</a></code>, these
actions are completely unrelated. Broadcast receivers can&#39;t see or capture
intents used to start an activity; likewise, when you broadcast an intent, you
can&#39;t find or start an activity.</span></aside>
<h2 id="restrict-broadcasts-permissions" data-text="Restricting broadcasts with permissions" tabindex="-1">Restricting broadcasts with permissions</h2>

<p>Permissions allow you to restrict broadcasts to the set of apps that hold
certain permissions. You can enforce restrictions on either the sender or
receiver of a broadcast.</p>

<h3 id="sending-broadcasts-permissions" data-text="Sending with permissions" tabindex="-1">Sending with permissions</h3>

<p>When you call <code translate="no" dir="ltr"><a href="/reference/android/content/Context#sendBroadcast(android.content.Intent, java.lang.String)">sendBroadcast(Intent, String)</a></code> or
<code translate="no" dir="ltr"><a href="/reference/android/content/Context#sendOrderedBroadcast(android.content.Intent, java.lang.String, android.content.BroadcastReceiver, android.os.Handler, int, java.lang.String, android.os.Bundle)">sendOrderedBroadcast(Intent, String, BroadcastReceiver, Handler, int, String, Bundle)</a></code>, you can specify a
permission parameter. Only receivers who have requested that permission with
the <uses-permission> tag in their manifest (and subsequently been granted the
permission if it is dangerous) can receive the broadcast. For example, the
following code sends a broadcast:</p>

<div>
<div class="ds-selector-tabs"><section><h3 id="kotlin" data-text="Kotlin" tabindex="-1">Kotlin</h3>
<pre class="prettyprint lang-kotlin" translate="no" dir="ltr">
sendBroadcast(Intent(BluetoothDevice.ACTION_FOUND),
              Manifest.permission.BLUETOOTH_CONNECT)
</pre>
</section><section><h3 id="java" data-text="Java" tabindex="-1">Java</h3>
<pre class="prettyprint lang-java" translate="no" dir="ltr">
sendBroadcast(new Intent(BluetoothDevice.ACTION_FOUND),
              Manifest.permission.BLUETOOTH_CONNECT)
</pre>
</section></div>
</div>

<p>To receive the broadcast, the receiving app must request the permission as
shown below:</p>
<pre class="prettyprint lang-xml" translate="no" dir="ltr"><code translate="no" dir="ltr">&lt;uses-permission android:name=&#34;android.permission.BLUETOOTH_CONNECT&#34;/&gt;
</code></pre>
<p>You can specify either an existing system permission like
<code translate="no" dir="ltr"><a href="/reference/android/Manifest.permission#BLUETOOTH_CONNECT">BLUETOOTH_CONNECT</a></code>
or define a custom permission with the
<a href="/guide/topics/manifest/permission-element"><code translate="no" dir="ltr">&lt;permission&gt;</code></a> element. For
information on permissions and security in general, see the <a href="/guide/topics/security/permissions">System
Permissions</a>.</p>
<aside class="note"><strong>Note:</strong><span> Custom permissions are registered when the app is installed. The app
that defines the custom permission must be installed before the app that uses
it.</span></aside>
<h3 id="receiving-broadcasts-permissions" data-text="Receiving with permissions" tabindex="-1">Receiving with permissions</h3>

<p>If you specify a permission parameter when registering a broadcast receiver
(either  with <code translate="no" dir="ltr"><a href="/reference/android/content/Context#registerReceiver(android.content.BroadcastReceiver, android.content.IntentFilter, java.lang.String, android.os.Handler)">registerReceiver(BroadcastReceiver, IntentFilter, String, Handler)</a></code> or in
<a href="/guide/topics/manifest/receiver-element"><code translate="no" dir="ltr">&lt;receiver&gt;</code></a> tag in your
manifest), then only broadcasters who have requested the permission with the
<a href="/guide/topics/manifest/uses-permission-element"><code translate="no" dir="ltr">&lt;uses-permission&gt;</code></a> tag
in their manifest (and subsequently been granted the permission if it is
dangerous) can send an Intent to the receiver.</p>

<p>For example, assume your receiving app has a manifest-declared receiver as
shown below:</p>
<pre class="prettyprint lang-xml" translate="no" dir="ltr"><code translate="no" dir="ltr">&lt;receiver android:name=&#34;.MyBroadcastReceiver&#34;
          android:permission=&#34;android.permission.BLUETOOTH_CONNECT&#34;&gt;
    &lt;intent-filter&gt;
        &lt;action android:name=&#34;android.intent.action.ACTION_FOUND&#34;/&gt;
    &lt;/intent-filter&gt;
&lt;/receiver&gt;
</code></pre>
<p>Or your receiving app has a context-registered receiver as shown below:</p>

<div>
<div class="ds-selector-tabs"><section><h3 id="kotlin" data-text="Kotlin" tabindex="-1">Kotlin</h3>
<pre class="prettyprint lang-kotlin" translate="no" dir="ltr">
var filter = IntentFilter(Intent.ACTION_FOUND)
registerReceiver(receiver, filter, Manifest.permission.BLUETOOTH_CONNECT, null )
</pre>
</section><section><h3 id="java" data-text="Java" tabindex="-1">Java</h3>
<pre class="prettyprint lang-java" translate="no" dir="ltr">
IntentFilter filter = new IntentFilter(Intent.ACTION_FOUND);
registerReceiver(receiver, filter, Manifest.permission.BLUETOOTH_CONNECT, null );
</pre>
</section></div>
</div>

<p>Then, to be able to send broadcasts to those receivers, the sending app must
request the permission as shown below:</p>
<pre class="prettyprint lang-xml" translate="no" dir="ltr"><code translate="no" dir="ltr">&lt;uses-permission android:name=&#34;android.permission.BLUETOOTH_CONNECT&#34;/&gt;
</code></pre>
<h2 id="security-and-best-practices" data-text="Security considerations and best practices" tabindex="-1">Security considerations and best practices</h2>

<p>Here are some security considerations and best practices for sending and
receiving broadcasts:</p>

<ul>
<li><p>If many apps have registered to receive the same broadcast in their
manifest, it can cause the system to launch a lot of apps, causing a
substantial impact on both device performance and user experience. To avoid
this, prefer using context registration over manifest declaration.
Sometimes, the Android system itself enforces the use of context-registered
receivers. For example, the <code translate="no" dir="ltr"><a href="/reference/android/net/ConnectivityManager#CONNECTIVITY_ACTION">CONNECTIVITY_ACTION</a></code> broadcast is delivered
only to context-registered receivers.</p></li>
<li><p>Do not broadcast sensitive information using an implicit intent. The
information can be read by any app that registers to receive the broadcast.
There are three ways to control who can receive your broadcasts:</p>

<ul>
<li>You can specify a permission when sending a broadcast.</li>
<li>In Android 4.0 and higher, you can specify a
<a href="/guide/topics/manifest/manifest-element#package">package</a> with
<code translate="no" dir="ltr"><a href="/reference/android/content/Intent#setPackage(java.lang.String)">setPackage(String)</a></code> when sending a
broadcast. The system restricts the broadcast to the set of apps that
match the package.</li>
</ul></li>
<li><p>When you register a receiver, any app can send potentially malicious
broadcasts to your app&#39;s receiver. There are several ways to limit the
broadcasts that your app receives:</p>

<ul>
<li>You can specify a permission when registering a broadcast receiver.</li>
<li>For manifest-declared receivers, you can set the
<a href="/guide/topics/manifest/receiver-element#exported">android:exported</a>
attribute to &quot;false&quot; in the manifest. The receiver does not receive
broadcasts from sources outside of the app.</li>
</ul></li>
<li><p>The namespace for broadcast actions is global. Make sure that action names
and other strings are written in a namespace you own, or else you may
inadvertently conflict with other apps.</p></li>
<li><p>Because a receiver&#39;s <code translate="no" dir="ltr"><a href="/reference/android/content/BroadcastReceiver#onReceive(android.content.Context, android.content.Intent)">onReceive(Context, Intent)</a></code> method runs on
the main thread, it should execute and return quickly. If you need to
perform long running work, be careful about spawning threads or starting
background services because the system can kill the entire process after
<code translate="no" dir="ltr">onReceive()</code> returns. For more information, see <a href="#effects-on-process-state">Effect on process
state</a> To perform long running work, we
recommend:</p>

<ul>
<li>Calling <code translate="no" dir="ltr"><a href="/reference/android/content/BroadcastReceiver#goAsync()">goAsync()</a></code> in your
receiver&#39;s <code translate="no" dir="ltr">onReceive()</code> method and passing the <code translate="no" dir="ltr"><a href="/reference/android/content/BroadcastReceiver.PendingResult">BroadcastReceiver.PendingResult</a></code> to a background thread.
This keeps the broadcast active after returning from <code translate="no" dir="ltr">onReceive()</code>.
However, even with this approach the system expects you to finish with
the broadcast very quickly (under 10 seconds). It does allow you to move
work to another thread to avoid glitching the main thread.</li>
<li>Scheduling a job with the <code translate="no" dir="ltr"><a href="/reference/android/app/job/JobScheduler">JobScheduler</a></code>. For more
information, see <a href="https://developer.android.com/topic/performance/scheduling.html">Intelligent Job
Scheduling</a>.</li>
</ul></li>
<li><p>Do not start activities from broadcast receivers because the user experience
is jarring; especially if there is more than one receiver. Instead, consider
displaying a <a href="/develop/ui/views/notifications">notification</a>.</p></li>
</ul>
