		<h1 class="entry-title">Unpacking Visual Basic Packers &#8211;&nbsp;IcedID</h1>	</header><!-- .entry-header -->

	<div class="entry-content">
		
<p>Despite the fact that VisualBasic is an age-old programming language, it is still being used to develop malicious software &#8211; specifically packers &#8211; to this day. As a result, you will often encounter VisualBasic based packers used in a lot of “script-kiddie” malware, such as keyloggers and remote access tools being sold on forums, and more recently, IcedID! For those of you not aware, IcedID (AKA BokBot) is a banking trojan that has been around for a couple years now, which was quite infamous in the malware analysis community due to it’s novelty process injection technique of API hooking certain calls to execute code inside of a spawned svchost.exe process. If you haven’t heard of this injection technique before, don’t worry! We will be covering it in <strong>Section 3</strong> of the Zero2Automated course, along with other injection techniques &#8211; but, if you would like to check out a few articles about IcedID to get some context before reading this one, feel free to! Anyway, let’s jump straight into the analysis!</p>



<p><strong>Indicator Of Compromise:</strong></p>



<p class="has-text-align-center">MD5 Hash of Packed Sample: <em>462018be60a69aa89b539c239761248c</em></p>



<p><strong>Initial Analysis:</strong></p>



<p>So, while I do know that this sample is packed as I have looked at it before, let’s approach it as if we had no idea what it was. First things first, we want to open it up in a PE analysis tool, which in this case I will be using <strong>CFF Explorer</strong>. One of the good things about <strong>CFF Explorer </strong>is the fact that as soon as we open up the malicious executable in it, we can see the <strong>File Info</strong>, which is <em>Microsoft Visual Basic 5/6, </em>so automatically we know that VB was used to create the file.&nbsp;</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh6.googleusercontent.com/if0gPRH9jimkFPcGokhGlfFeAWGAlHa5v9xZlLetG2gLcqHkK-F29_ys_tdlOXSGX_hN0FuxHY1LTO75-yUa2pq7G-ROiqOXRfDDQ3q-26cIaHUDjZhECxwkyfgTV-UNcO1VOZ2y" alt="" /></figure></div>



<p>As we don’t want to spend too long analyzing the executable, let’s go ahead and open it up in x32dbg to begin with. The reason we are opening it up in a debugger to begin with rather than IDA Pro is due to the fact that VB is more commonly used to create malware packers than malware itself, so we are saving ourselves some time here. What we see in the debugger can tell us if the executable is packed or not, so let’s check that out first. As you can see in the image below, the entry point is a lot different to the entry points of most executables, as all there is is a push instruction and then a call to <strong>ThunRTMain()</strong>.</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh3.googleusercontent.com/9q5L7Oqa_Npo3ONjOlnZeN9poSW880ta47GcqyiCqhQjQYTn_7khKd7ExQsCs8-Var7u0Rg8SCroiLGi0d5VJ-crFooYWCCKV1fjSkxzoJ61HLtYDiMSqHsXSpAOtfJyoIvwo65Y" alt="" /></figure></div>



<p>When we view the pushed address in the dump, you’ll notice a few strings such as <em>VB5!</em> and <em>Custom Tabs</em>. This is actually a structure used to tell <strong>ThunRTMain()</strong> about the program, including where the entry point of the user code actually is. <a href="https://www.vb-decompiler.org/pcode_decompiling.htm" target="_blank" rel="noreferrer noopener"><strong>Here</strong></a> you can see a table containing information about the different values inside the structure, and the position of them. The value we want to find is referred to as <strong>aSubMain</strong>, as that is the address called by <strong>ThunRTMain()</strong> once everything has been initialized. The area highlighted in red in the memory dump <strong>should</strong> contain the address of <strong>aSubMain</strong>, however it is completely empty! This indicates that there is some obfuscation going on, or the authors have altered the compilation routine. As a result of this, we will have to rely on setting some breakpoints instead to unpack the sample, rather than statically analyze it.</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh4.googleusercontent.com/oiTQuobbuzbXuOXjkyW7TObUIdaHyjxzPMU8i29rkD5ZMHIKp7IewGaVSeRNhGQKVvMmHgtE5LHnydeCCP96jjlE_aCv7CfmXQLF_sItlGd6Lzog3ia2a9J8KsPKwTVvHDDoig_3" alt="" /></figure></div>



<p>When unpacking VB based packers, I like to put a breakpoint on 4 main API calls:</p>



<ul class="wp-block-list"><li><strong>VirtualAlloc</strong></li><li><strong>VirtualProtect</strong></li><li><strong>IsDebuggerPresent</strong></li><li><strong>CreateProcessInternalW</strong></li></ul>



<p>This allows us to; view any allocated regions of memory that may have an executable written into it, prevent any processes from being created, as well as stop the most common anti-debug method from executing, which is quite common in these VB packers. The reason we want to break on it is so we can alter the value to “hide” the debugger from the process. If a debugger is detected by the malware, it might not halt execution as many of you may have thought &#8211; it is quite common for the program to continue executing, however it will take a very different path that does nothing. This can lead you to analyze a program for hours, trying to figure out why nothing malicious is actually happening.</p>



<p>So, putting a breakpoint on these calls and running the debugger, you can see we hit <strong>IsDebuggerPresent </strong>immediately. In order to alter the return value of this call, we want to <em>Execute To Return</em> and then alter the value in <strong>EAX</strong> from <strong>1</strong> to <strong>0</strong>. Then go ahead and execute the debugger again, and wait for the next breakpoint to be hit!</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh3.googleusercontent.com/DlSpk920hUrfLVzyYVINAK6WGyE5tLl21xfEhgcDntyM_UN_3w3dpS8sjJJQE9hdMUur5Cypa_AU_x9iLtBgjv_C4d7SxazME5PygxfXwB_ToWHS79Erj4hjWZOZN92nwmJNUBAJ" alt="" /></figure></div>



<p>The next breakpoint to be hit is <strong>VirtualAlloc</strong>, so let’s execute to return again and follow the value stored in <strong>EAX</strong> in the dump &#8211; buuut on the first run you’ll notice that there is no option to follow the value in the dump. This is also quite common for some samples &#8211; what happens is it first calls <strong>VirtualAlloc</strong> in order to reserve the memory location, and then it will allocate it on the second call. Therefore, go ahead and execute the debugger again and you’ll see it breaks on <strong>VirtualAlloc</strong> once more &#8211; this time when you execute to return, you’ll be able to follow the value in the dump!</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh4.googleusercontent.com/9NcN1FLd1PtlF5FPHpH_JQKhIZ-1ItjLI_rW7WnVtt33EOLgykMUnr4C04iLWR2kJC0XDF_Y8yy2wVxRwADY7X2oi6DaE_4qjpoBm70N4PFBpFRtgTs_lltYYpa0VGES_66i4FAV" alt="" /></figure></div>



<p>After that last allocate, running the debugger once again, it’ll break on <strong>VirtualAlloc</strong>. At this point, you’ll notice the previous memory region has been filled in, however it doesn’t seem to be very useful and it definitely isn’t an executable, so let’s continue running the debugger and keep an eye on any allocated regions of memory, until something interesting happens!</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh6.googleusercontent.com/1PLu6-DmA4LzqFt16cEbMKQLvTHHuZ-MpJJ_yA31PszAAd1rv5h1N4EmSb2mHN6BtwSTQJ4hoQfQyu7Yq1asvE-2uas3UCvXdkWVIBo-dQnV-oQdd9VzzdLOzWf0f71y_FDKK50-" alt="" /></figure></div>



<p>Finally, after ignoring several more useless <strong>VirtualAlloc</strong> and <strong>VirtualProtect</strong> calls, we finally find a new region being allocated at<strong> 0x00770000</strong>! Now let’s follow this in the dump and run the debugger once more to see if anything interesting is copied over.</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh4.googleusercontent.com/XklF_aVw38DFGBCamZIbJlS2VfW1bWcsFUpsptKurU9fqTGPWVg76Ge-BaLTFPqlkYC3kEQgULw9OKtiCEvTljOWtY7poYnzBg_7jj5VAcvRH22-726biS1v2ZG7I4BHfWOrQEeX" alt="" /></figure></div>



<p>The next breakpoint that is hit is a call to <strong>VirtualProtect</strong>, which is changing the protection of the region of memory just allocated &#8211; which has also been filled in with what seems to be shellcode. If you watched the initial video in the section regarding packers, you’ll remember that packers often use shellcode to decrypt the executable and overwrite the packer (in memory) with the decrypted executable, so that could be what is happening here. Let’s go ahead and run the debugger again, until we see an executable being written to an allocated region of memory!</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh4.googleusercontent.com/iSqQ1kRZDOUvA6GAwKk4B30gjkRQCjN0LOIsMtsEWNPOEM7MUF7w2z2DSlqv4_3h7ohqZMbOPxxUC0adVeOSZos8cf5O_XacUIiOhXgHsf8wR8ceZbq_LjdMQhihT-p4cxw7rDDQ" alt="" /></figure></div>



<p>Sure enough, after a few more triggered breakpoints, we reach a call to <strong>VirtualAlloc</strong> that allocates a memory region, which shortly after has an executable written to it! This isn’t the whole executable however, if you scroll down you’ll notice only the header has been written, so we can’t dump it out just yet. The reason why it hasn’t been fully written into memory yet is due to the fact that the executable must be mapped into memory in order to execute, so the packer will go ahead and allocate memory at <strong>0x024F1000</strong> and write the <strong>.text</strong> section to it, then allocate memory at <strong>0x024F2000 </strong>and write the <strong>.rdata </strong>section to it, and so on. Therefore, let’s go ahead and run the debugger until we hit a call to <strong>VirtualProtect</strong>, as this is when the packer begins to change the protection of the different regions of memory in the executable &#8211; such as changing the protection on the <strong>.text</strong> section to <strong>RWX </strong>(read-write-execute).</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh4.googleusercontent.com/MmetWfJuy_dYoNuVeeIwlS_D_Fj8aSh4dhjjtbPKTYvj7qDgk_BchYyLxuwFobAqRfJUN17bW2GXSYCIjLlGoehFek6HuFNO-Nns-GPG2xjbHdpAy1cNO0fV4GYFy7gQRk8yTvd7" alt="" /></figure></div>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh5.googleusercontent.com/1NgL0277sdRPTM-qaibq048jbaUq2e6MPQVm__lvRxlCSo_yLRq5_3ATYVhCulAH3SYBEotg2vCEYTvKnTjQksbRyO49etBOc2Ho_eB5W70Nq0UWtz-hJsBR_Q2xayxxhLtbXe7g" alt="" /></figure></div>



<p>Once we’ve hit <strong>VirtualProtect</strong>, we can go ahead and dump out the executable from memory, as this is the fully unpacked <strong>IcedID</strong> loader! One of my favourite tools to do this is <strong>Process Hacker 2</strong>, but you can use whatever tool you like, or even use the inbuilt x32dbg memory dump functionality.</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh5.googleusercontent.com/Ukwy7vBkA-RysLbbL6uYv3hZyh9340aUpa06yVUrTgQu-pazlV2gxWc5aZ7YXo-8iJ-FiVsgu6yms8AlaXxgQQJS2vE_zXUqv14Fz3TELhC5ZyjXoOMfV4zP_uELNwDnAKw99s15" alt="" /></figure></div>



<p>With the executable dumped from memory, we now need to unmap it. As I said before, the packer maps it into memory, and so if we were to open the program up in IDA Pro, it would try and resolve values as if the executable was unmapped &#8211; so instead of finding the <strong>.text </strong>section at <strong>0x024F1000</strong>, it would look at <strong>0x00040400</strong> (as an example). Therefore, we need to unmap it. The best tool to do this is <strong>PE-Bear</strong>, and upon opening the program up in it, we want to go to the <em>Section Hdrs</em> tab, and change the <strong>Raw Addr.</strong> tab to match the values shown in the <strong>Virtual Addr.</strong> tab, and then change the <strong>Raw Size</strong> values so they match up as well (this is the difference between the sections in terms of size) &#8211; so let’s go ahead and change that!</p>



<div class="wp-block-image"><figure class="aligncenter"><img src="https://lh4.googleusercontent.com/xpSPv980k_aVqVUXqUy9l2mv863daWoBgvMJq5NcXcMCU-s-vJb8pyRZEk1D4-nsL4Eqc1CrdElGXBkTMixDu7SP3tDQvA4ty72xIR7NtHFKmUoQjIPa532F7nGAf7baOf5YIZtj" alt="" /></figure></div>



<p>If everything has gone as planned, you’ll now have something that looks like the image below! If you check the <em>Imports</em> tab, you’ll notice that they have also been filled in, so now we can view all the imports that the malware uses at some point in it’s execution! Go ahead and save the fixed dump by right clicking the filename in the top left and choosing to <em>Save Executable As…</em><strong><em> </em></strong>and congratulations! You have successfully unpacked this VB packed sample of an IcedID Loader!</p>
