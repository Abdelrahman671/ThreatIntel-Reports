        <div class="cmp-text cmp-text--v1"><div id="text-1d479f6ff2" data-cmp-is='text' class="cmp-text__rte" style="--rte-highlightcolor: ; min-height: px;">
    <p>                The cybercriminal organization behind the InterPlanetary Storm <a href="https://www.barracuda.com/glossary/malware?utm_source=blog&amp;utm_medium=41955">malware</a> has released a new variant into the wild, now targeting Mac and Android devices in addition to Windows and Linux machines. The malware is building a <a href="https://www.barracuda.com/glossary/botnet?utm_source=blog&amp;utm_medium=41955">botnet</a>, which Barracuda researchers estimate currently includes roughly 13,500 infected machines located in 84 different countries around the world, and that number continues to grow.<br><br>The majority of the machines infected by the malware are located in Asia.<br><ul><br> 	<li>59% of infected machines are in Hong Kong, South Korea, and Taiwan</li><br> 	<li>8% are in Russia and Ukraine</li><br> 	<li>6% are in Brazil</li><br> 	<li>5% are in the United States and Canada</li><br> 	<li>3% are in Sweden</li><br> 	<li>3% are in China</li><br> 	<li>All other countries are 1% or less</li><br></ul><br>Here is a closer look at this evolving threat and solutions to help detect, block, and remediate the attacks.<br><br><img class="aligncenter size-large wp-image-41970" src="/content/dam/barracuda-blog/images/2020/10/infected-machines-map_1200x628-1-1024x536.jpg" alt="Interplanetary storm" width="1024" height="536"><br><h2><strong>Highlighted Threat</strong></h2><br><strong>New variant of InterPlanetary Storm malware</strong> — This new malware variant gains access to machines by running a dictionary attack against SSH servers, similar to <a href="https:/www.guardicore.com/2020/08/fritzfrog-p2p-botnet-infects-ssh-servers/">FritzFrog</a>, another peer-to-peer (p2p) malware. It can also gain entry by accessing open ADB (Android Debug Bridge) servers. The malware detects the CPU architecture and running OS of its victims, and it can run on ARM-based machines, an architecture that is quite common with routers and other IoT devices.<br><br>The malware is called InterPlanetary Storm because it uses the <a href="https://docs.ipfs.io/">InterPlanetary File System (IPFS) p2p network</a> and its underlying <a href="https://github.com/libp2p/go-libp2p">libp2p implementation</a>. This allows infected nodes to communicate with each other directly or through other nodes (i.e. relays).<br><br>The <a href="https://www.anomali.com/blog/the-interplanetary-storm-new-malware-in-wild-using-interplanetary-file-systems-ipfs-p2p-network">first variant of Interplanetary Storm</a>, which targeted Windows machines, was uncovered by researchers at Anomali in May 2019, and <a href="https:/labs.bitdefender.com/2020/06/ssh-targeting-golang-bots-becoming-the-new-norm/">a variant capable of attacking Linux machines</a> was reported in June of this year. This new variant, which Barracuda researchers first detected in late August, is targeting <a href="https://www.barracuda.com/glossary/iot-security?utm_source=blog&amp;utm_medium=41955">IoT devices</a>, such as TVs that run on Android operating systems, and Linux-based machines, such as routers with ill-configured SSH service.<br><br>While the botnet that this malware is building does not have clear functionality yet, it gives the campaign operators a backdoor into the infected devices so they can later be used for <a href="https://blog.barracuda.com/2020/06/25/threat-spotlight-new-cryptominer-malware-variant/">cryptomining</a>, <a href="https://www.barracuda.com/glossary/ddos?utm_source=blog&amp;utm_medium=41955">DDoS</a>, or other large-scale attacks.<br><h2><strong>The Details</strong></h2><br>This variant of InterPlanetary Storm is written in Go, uses the <a href="https://github.com/libp2p/go-libp2p">Go implementation of libp2p</a>, and is packed with UPX. It spreads using SSH brute force and open ADB ports, and it serves malware files to other nodes in the network. The malware also enables reverse shell and can run bash shell.<br><br>Barracuda researchers found several unique features designed to help the malware persist and protect it once it has infected a machine.<br><ul><br> 	<li><strong>It detects honeypots.</strong> The malware looks for the string "svr04" in the default shell prompt (PS1), which was used by the Cowrie honeypot before.</li><br> 	<li><strong>It auto updates.</strong> The malware compares the version of the running instance with the latest available version and will update accordingly.</li><br> 	<li><strong>It will try to persist itself by installing a service</strong> (system/systemv), using a <a href="https://github.com/takama/daemon">Go daemon package</a>.</li><br> 	<li><strong>It kills other processes on the machine that pose a threat to the malware</strong>, such as debuggers and competing malware. It does so by looking for the following strings in process command lines:<br><ul><br> 	<li>"/data/local/tmp"</li><br> 	<li>"rig"</li><br> 	<li>"xig"</li><br> 	<li>"debug"</li><br> 	<li>"trinity"</li><br> 	<li>"xchecker"</li><br> 	<li>"zypinstall"</li><br> 	<li>"startio"</li><br> 	<li>"startapp"</li><br> 	<li>"synctool"</li><br> 	<li>"ioservice"</li><br> 	<li>"start_"</li><br> 	<li>"com.ufo.miner"</li><br> 	<li>"com.google.android.nfcguard"</li><br> 	<li>"com.example.test"</li><br> 	<li>"com.example.test2"</li><br> 	<li>"saoas"</li><br> 	<li>"skhqwensw"</li><br></ul><br></li><br></ul><br><h2><strong>Interplanetary Storm announced keys</strong></h2><br>The malware’s backend advertises the following keys into the IPFS Distributed Hash Table (DHT). Infected nodes will then try to find peers that can provide the required services:<br><table><br><tbody><br><tr><br><td width="354"><strong>Key</strong></td><br><td width="255"><strong>Purpose</strong></td><br></tr><br><tr><br><td width="354">requeBOHCHIY2XRMYXI0HCSZA</td><br><td width="255">C2</td><br></tr><br><tr><br><td width="354">proxybackendH0DHVADBCIKQ4S7YOX4X</td><br><td width="255">Proxy backend</td><br></tr><br><tr><br><td width="354">web-api:kYVhV8KQ0mA0rs9pHXoWpD</td><br><td width="255">File distribution backend</td><br></tr><br></tbody><br></table><br>Each infected node will advertise the key “fmi4kYtTp9789G3sCRgMZVG7D3uKalwtCuWw1j8LSPHQEGVBU5hfbNdnHvt3kyR1fYUlGNAO0zactmIMIZodsOha9tnfe25Xef1” in order to inform that it is part of the botnet. The ID of each infected machine will be generated once during initial infection and will be reused if the machine restarts or the malware updates.<br><br>Infected nodes will also advertise keys in the form “stfadv:&lt;cheksum&gt;” in order to notify that the node can provide a file with that checksum.<br><h2><strong>Communication protocols</strong></h2><br>Libp2p applications handle incoming connection (streams) based on a logical address (i.e. unknown to the transport layer) called <a href="https://docs.libp2p.io/concepts/protocols/">protocol ID</a>. By convention, protocol ids have a path-like structure, with a version number as the final component.<br><br>The following protocol IDs are being used by the malware:<br><table><br><tbody><br><tr><br><td width="208"><strong>Protocol ID</strong></td><br><td width="208"><strong>Purpose</strong></td><br><td width="208"><strong>Notes</strong></td><br></tr><br><tr><br><td width="208">/sbst/1.0.0</td><br><td width="208">Used for spawning reverse shell</td><br><td width="208">Hosted on nodes</td><br></tr><br><tr><br><td width="208">/sfst/1.0.0</td><br><td width="208">Used for file transfer</td><br><td width="208">Hosted on nodes, file checksum is used for the integrity of the served file</td><br></tr><br><tr><br><td width="208">/sbpcp/1.0.0</td><br><td width="208">Used for proxy, connect to backend serve</td><br><td width="208">Hosted on backend servers</td><br></tr><br><tr><br><td width="208">/sbptp/1.0.0</td><br><td width="208">Used for proxy. Forward proxy channel</td><br><td width="208">Hosted on nodes</td><br></tr><br><tr><br><td width="208">/sreque/1.0.0</td><br><td width="208">Used for scanner queue.</td><br><td width="208">Hosted on nodes, commands from the c2 contain signature.<br><br>Messages on this channel are serialized using JSON objects. Messages from the c2 will be for either “brute-ssh”or “tcp-scan”, directing the node to scan for vulnerable machines. The node will send be the results of these scans.<br><br>The “brute-ssh” messages from the c2 will include a list of Ips to attack along with the credentials that should be used.</td><br></tr><br></tbody><br></table><br><h2><strong>File distribution backend</strong></h2><br>The file distribution servers can be discovered using the “web-api:kYVhV8KQ0mA0rs9pHXoWpD” key. The relevant peers implement http over the libp2p protocol and serve the following URLs:<br><table><br><tbody><br><tr><br><td width="208"><strong>Path</strong></td><br><td peer version</td><br></tr><br><tr><br><td width="208">/files/checksum?f=&lt;file name&gt;</td><br><td width="208">GET</td><br><td width="208">Get the current checksum of the file &lt;file file</td><br></tr><br><tr><br><td width="208">POST /files/seedrs-http</td><br><td width="208">POST</td><br><td width="208">Add node info</td><br></tr><br><tr><br><td width="208">/nodes/</td><br><td width="208">POST</td><br><td width="208">Add node info</td><br></tr><br></tbody><br></table><br><h2><strong>IOC</strong></h2><br>The malware might drop some of the following files:<br><table><br><tbody><br><tr><br><td width="312">storm_android-amd64</td><br><td width="312">d4e3102b859ebfda5a276b2ce6f226e27fdcdef5e693ee7742be863236e2374a&nbsp;</td><br></tr><br><tr><br><td a few important steps you can take to protect against this malware variant.<br><ul><br> 	<li><strong>Properly configure SSH access on all devices.</strong> This means using keys instead of passwords, which will make access more secure. When password login is enabled and the service itself is accessible, the malware can exploit the ill-configured attack surface. This is an issue common with routers and IoT devices, so they make easy targets for this malware.</li><br> 	<li><strong>Use a cloud security posture management tool</strong> to monitor SSH access control to eliminate any configuration mistakes, which can be catastrophic. To provide secured access to shells if needed; instead of exposing the resource on the internet, deploy an MFA-enabled VPN connection and segment your networks for the specific needs instead of granting access to broad IP networks.</li><br></ul><br><a href="https://csg.barracudanetworks.com/scan?utm_source=blog&amp;utm_medium=41955"><img class="aligncenter size-full wp-image-39414" src="/content/dam/barracuda-blog/images/2020/02/cloudscan_social_promo.jpg" alt="Cloud security scan" width="1200" height="628"></a>            </p>
