<h3 class="wp-block-heading">Ransomware and wiper</h3>



<p>The cyberattack on the Albanian government used a common tactic of Iranian state sponsored actors by <a href="https://www.sentinelone.com/labs/from-wiper-to-ransomware-the-evolution-of-agrius/">deploying ransomware first</a>, followed by deployment of the wiper malware. The wiper and ransomware both had forensic links to Iranian state and Iran-affiliated groups. The wiper that DEV-0842 deployed in this attack used the same license key and EldoS RawDisk driver as ZeroCleare, a wiper that Iranian state actors used in an attack on a Middle East energy company in mid-2019. In that case, <a href="https://www.ibm.com/downloads/cas/OAJ4VZNJ">IBM X-Force assessed</a> that actors affiliated with EUROPIUM gained initial access nearly a year ahead of the wiper attack. The wiper attack was subsequently performed by a separate and unknown Iranian actor. This is similar to the chain of events Microsoft detected against the Albanian government.</p>

<h2 class="wp-block-heading" id="actor-activity">Observed actor activity</h2>



<p>DART and MSTIC supported the post ransom and wiper attack analysis leveraging <a href="https://www.microsoft.com/security/business/siem-and-xdr/microsoft-365-defender">Microsoft 365 Defender</a> and collection of additional forensic artifacts. <h3 class="wp-block-heading">Access and implant</h3>

<h3 class="wp-block-heading">Data collection</h3>



<p>During the period of October 2021 â€“ January 2022, the threat actors used a unique email exfiltration tool which interacted with the Exchange web services APIs to collect email in a manner that masked the actions. <h3 class="wp-block-heading">Defense evasion</h3>



<p>Prior to launching the final stage of the attack, the threat actors gained administrative access to a deployed endpoint detection and response (EDR) solution to make modifications, removing&nbsp;libraries&nbsp;that affected the agents across the enterprise. In addition, a binary to disable components of Microsoft Defender Antivirus was propagated using custom tooling. The distributed binary named <em>disable-defender.exe</em> queries for TokenElevation using the GetTokenInformation API and checks if the process is running with elevated privileges. If the token is not running with elevated privilege, the binary prints &#8220;Must run as admin!\n&#8221;. If the token is elevated, it queries TokenUser and checks if the SID is &#8220;S-1-5-18&#8221;. If the current process doesn&#8217;t run under system context, it prints &#8220;Restarting with privileges\n&#8221; and attempts to elevate the privilege.</p>



<p>To elevate the privilege, the binary checks if the TrustedInstaller service is enabled. To do this, it starts the service &#8220;SeDebugPrivilege&#8221; and &#8220;SeImpersonatePrivilege&#8221; to assign privileges to itself. It then looks for <em>winlogon.exe</em> process, acquires its token, and impersonates calling thread using ImpersonateLoggedOnUser/SetThreadToken. After impersonating as <em>winlogon.exe</em>, it opens TrustedInstaller process, acquires its token for impersonation and creates a new process with elevated privileges using CreateProcessWithTokenW.</p>
