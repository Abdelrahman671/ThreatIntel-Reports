						<h1 class="post-title single_blog_inner__Title">
							Reptile Malware Targeting Linux Systems						</h1>

												<div class="slider-post-meta-items tab-small-col-meta blog-details-meta-wrap">
							<div class="slider-meta-left">

									<img class="img-fluid" src="https://asec.ahnlab.com/wp-content/uploads/2023/07/38_security-alert_01-803x490.png?crop=1" alt="Reptile Malware Targeting Linux Systems">

					<!-- Article content -->
					<div class="entry-content clearfix">
						
						<p>Reptile is an open-source kernel module rootkit that targets Linux systems and is publicly available on GitHub.&nbsp;<a href="https://github.com/f0rb1dd3n/Reptile"><strong>[1]</strong></a> Rootkits are malware that possess the capability to conceal themselves or other malware. They primarily target files, processes, and network communications for their concealment. Reptile&#8217;s concealment capabilities include not only its own kernel module but also files, directories, file contents, processes, and network traffic.</p>
<p>Unlike other rootkit malware that typically only provide concealment capabilities, Reptile goes a step further by offering a reverse shell, allowing threat actors to easily take control of systems. Port Knocking is the most notable feature out of those supported by Reptile. Port Knocking is a method where the malware opens a specific port on an infected system and goes on standby. When the threat actor sends a Magic Packet to the system, the received packet is used as a basis to establish a connection with the C&amp;C server.</p>
<p>This method is similar to that of Syslogk, which was mentioned in a previous report by Avast.&nbsp;<a href="https://decoded.avast.io/davidalvarez/linux-threat-hunting-syslogk-a-kernel-rootkit-found-under-development-in-the-wild/"><strong>[2]</strong></a> One key difference is that Syslogk was developed based on another open-source Linux kernel rootkit called Adore-Ng. However, there are similarities between Syslogk and the features supported by Reptile, such as being on standby in an infected system before being triggered by a Magic Packet, and using a customized TinySHell, known as Rekoobe, as a backdoor for their attack.</p>
<p>After becoming publicly available on GitHub as open-source, Reptile has been used consistently in attacks. For example, a recent report by Mandiant confirmed that a threat group based in China used Reptile in their ongoing attack using the zero-day vulnerability in Fortinet products.&nbsp;<a href="https://www.mandiant.com/resources/blog/fortinet-malware-ecosystem"><strong>[3]</strong></a> Furthermore, in ExaTrack&#8217;s report analyzing the Mélofée malware, Reptile rootkit was also identified. ExaTrack attributes this to the activities of the Winnti attack group based in China.&nbsp;<a href="https://blog.exatrack.com/melofee/"><strong>[4]</strong></a></p>
<p>In this post, we will provide a brief analysis of the basic structure and features of Reptile, followed by a compilation of real-world instances where it was employed in attacks targeting Korean companies. Additionally, it should be noted that ICMP Shell was also used in the attack cases in Korea. At the end, we will summarize the similarities with the Mélofée malware cases based on the installation paths or disguised directory names of the malware.</p>
<h3><strong>1. Analysis of Reptile</strong></h3>
<h4><strong>1.1. Structure of Reptile</strong></h4>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-35.png?resize=916%2C485&amp;ssl=1" alt=""></figure>
<p><strong>1.1.1. Threat Actor&#8217;s System</strong></p>
<p>In addition to providing malware to be installed on the infected system, Reptile also supports tools to be used by the threat actor. Listener is a command line tool that operates by being given the port it has to listen to and its password. It waits for a connection from a reverse shell, which will be executed on the infected system. Depending on the options, Reptile can establish a reverse shell connection to a specified address after installation. In this case, the listener running on the C&amp;C server receives this connection and provides the threat actor with the shell.</p>
<p>Even if the threat actor&#8217;s C&amp;C server is not designated, the Port Knocking method can be used to transmit a specific packet to the infected system to trigger a reverse shell. This is achieved through a command line tool called Packet, which takes arguments such as the address the reverse shell will attempt to connect to and the protocol used in the Port Knocking method. Listener and Packet can be used manually, but they can also be used through a client that provides an interface.</p>
<p><strong>1.1.2. Affected System</strong></p>
<p>If no installation path is designated, the malware strains are installed in the /reptile/ directory with the file names reptile, reptile_shell, and reptile_cmd by default. The loader, known as &#8220;reptile&#8221;, is responsible for decrypting and loading the Reptile rootkit kernel module that is encrypted within the file. This means the Reptile rootkit does not exist directly as a kernel module file. Instead, it is installed through a decryption process by the loader.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-37.png?resize=464%2C214&amp;ssl=1" alt=""></figure>
<p>reptile_cmd is responsible for transmitting commands to the Reptile rootkit, and it communicates with the rootkit by specifying and executing the target to be concealed as an argument. The reverse shell malware, reptile_shell, is capable of receiving command line arguments and is executed by the Reptile rootkit. &nbsp;</p>
<p>If the option to attempt a direct connection to the C&amp;C server during the installation process is specified, the command is assigned to the /reptile/reptile_start script file. The Reptile rootkit operates the reverse shell by executing the mentioned script file after loading the kernel module. Furthermore, it can also execute the reverse shell and transmit the decrypted C&amp;C server address if an address is received through the Port Knocking method.</p>
<p><strong>1.2. Analysis of the Reptile Rootkit</strong></p>
<p>The reptile created in the /reptile/ directory is not a kernel module but a user mode application. When the loader is executed, it decrypts the Reptile kernel module contained within the data section and then loads it into the memory using the init_module() function. Additionally, the algorithm used for encryption and decryption is also used when decrypting the Magic Packet later on. The key value is generated using a random value during the build process, resulting in each generated file having a different value.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-39.png?resize=520%2C263&amp;ssl=1" alt=""></figure>
<p>The loaded Reptile is a kernel module packed using another open-source tool called kmatryoshka.&nbsp;<a href="https://github.com/milabs/kmatryoshka"><strong>[5]</strong></a> kmatryoshka is a Linux kernel module-based packer that is responsible for decrypting the original kernel module, which exists in an encrypted form. It then utilizes the sys_init_module() function to load it. As such, the original Reptile rootkit exists in a packed form within both user-mode and kernel-mode.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-40.png?resize=741%2C431&amp;ssl=1" alt=""></figure>
<p>&nbsp;</p>
<h5><strong>1.2.1. Analysis of Concealment Feature</strong></h5>
<p>Reptile uses a Linux kernel function hooking engine called KHOOK to hook on kernel functions.&nbsp;<a href="https://github.com/milabs/khook"><strong>[6]</strong></a> For example, it hooks the ip_rcv() kernel function to use the Port Knocking method. By doing so, it can monitor the packets it receives.</p>
<p>When delivering commands to the rootkit, Reptile utilizes reptile_cmd, which sends ioctl to the Reptile kernel module. The inet_ioctl() kernel function is hooked in order to monitor this ioctl. Among the data that is sent to ioctl, cmd represents the command number. Like the process concealing command, if additional data such as the PID is required, the argv variable of the control structure is used to transmit the data. During the command delivery process, AUTH and HTUA are random values. Reptile monitors the ioctl and performs corresponding actions when a match is found.</p>

<p style="text-align:center;"><span style="color:#6B7684;font-size:14px;">Table 1. cmd commands</span></p>
<p>Aside from the concealment and Port Knocking features, Reptile provides a feature where the &#8220;root&#8221; command can be used to give the current user root privileges. It also supports persistence through Udev. The following rules file is created in the /lib/udev/rules.d/ directory and the copied path is designated to ensure that it will be executed even after a reboot.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-43.png?resize=620%2C181&amp;ssl=1" alt=""></figure>
<p><strong>A. File and Directory Concealment</strong><br />The Reptile rootkit can hide or show files and directories based on the &#8220;hide&#8221; and &#8220;show&#8221; commands. The targets to be hidden are paths that contain the specified string during the build. To achieve this, it hooks multiple kernel functions including fillonedir(), filldir(), and filldir64(). If the string of the concealment target is present in the path name, the hooking function returns &#8220;ENOENT&#8221; which is the &#8220;no such file or directory&#8221; error.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-44.png?resize=605%2C187&amp;ssl=1" alt=""></figure>
<p><strong>B. Self Concealment</strong><br />The &#8220;hide&#8221; and &#8220;show&#8221; commands support concealment for not only files and directories, but also for the Reptile kernel module itself. When the &#8220;hide&#8221; command is received, the current module is removed from the module list. Accordingly, using the lsmod command will not show the currently installed Reptile kernel module.</p>
<p><strong>C. Process Concealment</strong><br />When a PID is given along with the &#8220;hide&#8221; or &#8220;show&#8221; command, the process of the PID is concealed. There are four main methods used for process concealment. One involves hooking the find_task_by_vpid() function to return NULL for the concealment target&#8217;s process, and another involves hooking the vfs_statx() function to return an &#8220;EINVAL&#8221; (Invalid argument) error. Additionally, the find_task_by_vpid() function is used in system calls like getsid() and getpgid().</p>
<p>Furthermore, hooking next_tgid() makes it so that the concealment target&#8217;s process is skipped, making it invisible in the /proc/ list. Lastly, for the sys_kill and __x64_sys_kill system calls, an &#8220;ESRCH&#8221; (No such process) error is returned, making termination impossible. Aside from these, the load_elf_binary() function is hooked and processes that have the path of reptile_shell are hidden.</p>
<p><strong>D. TCP/UDP Concealment</strong><br />If the &#8220;conn&#8221; command and the concealment target&#8217;s IP address are transmitted as an argument, the TCP/UDP network communication can be concealed. The tcp4_seq_show() and udp4_seq_show() functions are hooked for this purpose.</p>
<p><strong>E. File Content Concealment</strong><br />Reptile provides a file tampering feature, which allows the contents of a file to be hidden. When the tags designated during the build process, like those shown below, are added to the file content, the strings between these tags are concealed. By default, the tags &#8220;#&lt;reptile&gt;&#8221; and &#8220;#&lt;/reptile&gt;&#8221; can be used. The command to activate this feature is &#8220;file-tampering&#8221;, and it involves hooking the vfs_read() function.</p>
<p><strong>1.2.2. PORT KNOCKING</strong></p>
<p>The Reptile rootkit supports the Port Knocking technique. After being installed on the infected system, instead of immediately connecting to the C&amp;C server, it opens a certain port and waits until the threat actor sends a Magic Packet to the system, after which it begins to operate. The data received through the Magic Packet contains the C&amp;C server address. Based on this, a reverse shell connects to the C&amp;C server.</p>
<p>In Reptile&#8217;s defconfig file, there are basic configurations present. By default, the MAGIC_VALUE is set to &#8220;hax0r,&#8221; PASSWORD is set to &#8220;s3cr3t,&#8221; and SRCPORT is set to &#8220;666.&#8221;</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-45.png?resize=219%2C115&amp;ssl=1" alt=""></figure>
<p>The Reptile rootkit on the infected system hooks a kernel function and monitors packets incoming through the TCP, UDP, and ICMP protocols. If a TCP or UDP packet is received, the source port is first checked. The &#8220;666&#8221; port that was designated in the above configuration file is the target.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-46.png?resize=672%2C178&amp;ssl=1" alt=""></figure>
<p>Threat actors can use a client to transmit the Magic Packet to the infected system. To do this, they can first choose one of the protocols, TCP, UDP, or ICMP, to be used as part of the Port Knocking technique. They then have to designate the infected system&#8217;s IP address and configuration data set during the above creation of Reptile, which includes MAGIC_VALUE, PASSWORD, and SRCPORT. Afterward, when the run command is executed, Packet encrypts the data and transmits it to the infected system.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-47.png?resize=717%2C592&amp;ssl=1" alt=""></figure>
<p>If Reptile, which is listening on port 666, receives a packet through this port, it scans the data in the received packet to check for the value &#8220;hax0r,&#8221; which is designated by MAGIC_VALUE and TOKEN. When the process reaches this point, Reptile decrypts the packet and obtains the address and port number of the C&amp;C server. It then uses these values as arguments to execute reptile_shell, which is a reverse shell.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-48.png?resize=486%2C229&amp;ssl=1" alt=""></figure>
<p><strong>1.3. Reverse Shell</strong></p>
<p>The reverse shell executed by the Reptile rootkit connects to the C&amp;C server based on the received address and provides shell access. Additionally, the reverse shell is executed with the argument &#8220;s3cr3t&#8221;, which is specified as PASSWORD in the configuration data. This PASSWORD serves as the session key for communication with the listener that is waiting on the C&amp;C server.</p>
<p>Reverse shell is a command line tool that operates based on the provided arguments, and it can be executed in two different ways, depending on the conditions. The first method is the Port Knocking technique covered above. The second method involves executing during the installation process of the Reptile rootkit kernel module.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-49.png?resize=562%2C117&amp;ssl=1" alt=""></figure>
<p>After installation, the Reptile rootkit executes the startup script named reptile_start during the initialization process. This can contain numerous commands, a notable one being the command to execute reverse shells.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-50.png?resize=738%2C633&amp;ssl=1" alt=""></figure>
<p>The reverse shell of Reptile is based on TinySHell, an open-source Linux backdoor. Rekoobe is a backdoor malware based on TinySHell and is known to be predominantly used by Chinese threat groups.&nbsp;<a href="https://asec.ahnlab.com/en/55229/"><strong>[7]</strong></a> Additionally, according to a report from Avast, the Syslogk rootkit also supports the Port Knocking method triggered by a Magic Packet. It also utilizes a customized version of TinySHell, known as Rekoobe, as a backdoor. Based on these similarities, it is speculated that the Syslogk threat actor might have adopted the structure of Reptile in their malware.</p>
<p>When compared to TinySHell, Reptile&#8217;s reverse shell shows a remarkable similarity, with most of the code being identical along with the supported commands.&nbsp; In particular, the use of the HMAC SHA1 algorithm and AES-128 key to encrypt the communication data with the C&amp;C server and the data used for integrity verification during the communication process are also the same.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-51.png?resize=781%2C537&amp;ssl=1" alt=""></figure>
<p>Specifically, the Reptile reverse shell supports a delay command in addition to file download/upload and command execution. Moreover, it includes a built-in feature to send concealment commands to the Reptile rootkit via ioctl, effectively hiding communications with the C&amp;C server.</p>
<h3><strong>2. Cases of Attacks</strong></h3>

Even if the recent zero-day vulnerability attack case on Fortinet products by a China-based threat group, which was reported on by Mandiant, <a href="https://www.mandiant.com/resources/blog/fortinet-malware-ecosystem"><strong>[8]</strong></a> is excluded, the periodic uploads of Reptile rootkit malware on the VirusTotal platform can still be observed.</p>
<p>While it is not certain whether they were used in actual attacks, numerous Reptile rootkits have been regularly uploaded to VirusTotal over the past few years. In this section, the configuration data from a portion of these Reptile samples were extracted and categorized. <h4><strong>2.2. Attack Cases in Korea</strong></h4>
<p>Reptile has been used in past attacks against Korean companies. The initial method of infiltration remains unidentified, but upon examination, the Reptile rootkit, reverse shell, Cmd, and startup script were all included, allowing the basic configuration to be ascertained.</p>
<p>In this particular attack case, apart from Reptile, an ICMP-based shell called ISH was also utilized by the threat actor. ISH is a malware strain that uses the ICMP protocol to provide the threat actor with a shell. Typically, reverse shells or bind shells use protocols like TCP or HTTP, but it is speculated that the threat actor opted for ISH to evade network detection caused by these communication protocols.</p>
<h5><strong>2.2.1. Analysis of Reptile</strong></h5>
<p>The malware is presumed to be installed in the &#8220;/etc/intel_audio/&#8221; directory, and the threat actor used &#8220;intel_audio&#8221; as their keyword instead of &#8220;reptile&#8221;.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-52.png?resize=774%2C109&amp;ssl=1" alt=""></figure>
<p>Furthermore, the absence of any command lines to execute the reverse shell in the intel_audio_start file suggests that the reverse shell is likely to be used through the Port Knocking method. Alternatively, the threat actor could have used the bind shell, ISH, which will be covered later. Aside from these, the threat actor activated the file-tampering feature.</p>
<p>Next, upon examining the rc.local autorun script, it is evident that a command to ensure persistence exists between the tags &#8220;#&lt;intel_audio&gt;&#8221; and &#8220;#&lt;/intel_audio&gt;&#8221;, which have been marked for concealment with the file-tampering feature. A notable point is the fact that the threat actor used Reptile in the form of a kernel module rather than a loader. As a result, they load &#8220;/etc/intel_audio/intel_audio.ko&#8221; by manually inputting the insmod command.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-53.png?resize=615%2C97&amp;ssl=1" alt=""></figure>
<p>Intel_audio.ko is a kernel module packed with kmatryoshka before being packed by a loader. Upon inspecting the vermagic of the kernel module, &#8220;3.10.0-514.el7.x86_64,&#8221; it is estimated that the infected system was likely Red Hat or a CentOS-based Linux system.</p>
<p>The extracted rootkit contains various hard-coded configuration data. For example, in the reptile_init() function, the path name of the startup script file, &#8220;/etc/intel_audio/intel_audio_start,&#8221; was identified. A notable characteristic is that the threat actor set the MAGIC_VALUE and PASSWORD strings to glibc-related strings to disguise it as a normal program.</p>

<h5><strong>2.2.2. Analysis of ICMP SHELL</strong></h5>
<p>The file executed and targeted to be concealed in the &#8220;rc.local&#8221; autorun script, located at &#8220;/etc/intel_audio/gvfs-gdb-volume-monitor,&#8221; is an ICMP Shell known as ISH. ISH consists of the server module ishd and the client module ish. The &#8220;gvfs-gdb-volume-monitor&#8221; file, operating as ishd, is executed by the Reptile rootkit and kept in a listening state. It is presumed that when the attacker establishes a connection using ish, the ICMP Shell is provided. The command line option identified in &#8220;gvfs-gdb-volume-monitor&#8221; is the same as ishd.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-55.png?resize=405%2C494&amp;ssl=1" alt=""></figure>
<p>Additionally, when the threat actor created the ishd malware, they opted not to use the source code as-is. Instead, they made modifications to disguise it as a normal program so that it could avoid file detection. In the following figure, the left side displays the usage() function identified in the original ishd source code, while the right side shows the usage() function in &#8220;gvfs-gdb-volume-monitor. This allows the malware to be perceived as a normal program instead of a bind shell since it outputs the string &#8220;ICMP Debug Tool&#8221; when executed without any specific arguments.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-56.png?resize=809%2C265&amp;ssl=1" alt=""></figure>
<h3><strong>3. Similarities to Mélofée</strong></h3>
<p>ExaTrack conducted an analysis of the malware strains that have recently been targeting Linux servers and named them Mélofée. Based on the malware and infrastructure used in the attacks, they identified connections to the Winnti (APT41) threat group, which operates from China.&nbsp;<a href="https://blog.exatrack.com/melofee/"><strong>[9]</strong></a></p>
<p>The threat actor also used the Reptile rootkit during their attack process, and a notable characteristic was the installation of the rootkit in the path name &#8220;/etc/intel_audio/intel_audio.ko&#8221;. For reference, the path &#8220;/etc/intel_audio/intel_audio.ko&#8221; is identical to the installation path of the Reptile rootkit in the previously mentioned Linux server attack case that targeted Korean companies.</p>
<p>The use of the Reptile rootkit in the attack process, the identical installation path, and the direct installation of the kernel module through the &#8220;insmod&#8221; command instead of the conventional methods provided by Reptile are common factors between these two attack cases.</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-57.png?resize=368%2C112&amp;ssl=1" alt=""></figure>
<p>However, there are differences as well. In the Mélofée attack case, the Reptile used had only the file concealment feature activated, and the hidden paths were hardcoded to two locations: &#8220;intel_audio&#8221; and &#8220;rc.modules.&#8221;</p>
<figure class="image"><img decoding="async" src="https://image.ahnlab.com/atip/content/atcp/2023/07/image-58.png?resize=442%2C156&amp;ssl=1" alt=""></figure>
<p>Due to the limited available information in the Korean attack case, aside from the malware, there is a limit to how much information can be gathered. However, it is worth noting that the keyword &#8220;intel_audio&#8221;, though only used to disguise itself as a normal kernel module path, is an uncommon string and stands out as a distinctive characteristic in both attack cases.</p>
<h3><strong>4. Conclusion</strong></h3>
<p>Reptile is a Linux kernel mode rootkit malware that provides a concealment feature for files, directories, processes, and network communications. Due to being open-source, Reptile can be easily utilized by various threat actors, which has led to numerous attack cases being discovered. Considering the nature of rootkits, they are often used in conjunction with other malware. However, Reptile itself also provides a reverse shell, making systems with Reptile installed susceptible to being hijacked by threat actors.</p>
<p>To prevent such security threats, systems must be checked for vulnerable configurations and relevant systems must always be kept up to date to protect them from attacks. Also, V3 should be updated to the latest version so that malware infection can be prevented.</p>
