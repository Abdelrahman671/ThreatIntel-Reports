    <h1 class="text-center">Threat Spotlight: WarmCookie/BadSpace</h1>

    <div class="text-center m-1">
        <div class="post-author">

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <ul><li>WarmCookie is a malware family that emerged in April 2024 and has been distributed via regularly conducted malspam and malvertising campaigns.&nbsp;</li><li>WarmCookie, observed being used for initial access and persistence, offers a means for continuous long-term access to compromised environments and is used to facilitate delivery of additional malware such as <a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.csharpstreamer" rel="noreferrer noopener"><u>CSharp-Streamer-RAT</u></a> and <a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.cobalt_strike" rel="noreferrer noopener"><u>Cobalt Strike</u></a>.&nbsp;</li><li>Post-compromise intrusion activity associated with WarmCookie overlaps with previously observed activity we attribute to TA866.&nbsp;&nbsp;</li><li>We <a href="https://blog.talosintelligence.com/on-conveying-doubt/" rel="noreferrer noopener"><u>assess</u></a> that WarmCookie was likely developed by the same threat actor(s) as <a href="https://www.esentire.com/blog/esentire-threat-intelligence-malware-analysis-resident-campaign" rel="noreferrer noopener"><u>Resident</u></a> backdoor, a post-compromise implant previously deployed in intrusion activity that Cisco Talos attributes to <a href="https://blog.talosintelligence.com/highlighting-ta866-asylum-ambuscade/" rel="noreferrer noopener"><u>TA866</u></a>. &nbsp;</li></ul><h1 id="what-is-warmcookie">What is WarmCookie?&nbsp;</h1><p><a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.warmcookie" rel="noreferrer noopener"><u>WarmCookie</u></a>, also known as BadSpace, is a malware family that has been distributed since at least April 2024. Throughout 2024, we have observed several distribution campaigns conducted using a variety of lure themes to entice victims to take actions that result in malware infection.&nbsp;&nbsp;</p><p>These campaigns typically rely on malspam or malvertising to initiate the infection process that results in the delivery of WarmCookie. WarmCookie offers a variety of useful functionality for adversaries including payload deployment, file manipulation, command execution, screenshot collection and persistence, making it attractive to use on systems once initial access has been gained to facilitate longer-term, persistent access within compromised network environments. &nbsp;</p><p>In previously analyzed intrusion activity involving WarmCookie, we have observed that it is used as an initial payload and that <u>CSharp-Streamer-RAT</u> and <a href="https://malpedia.caad.fkie.fraunhofer.de/details/win.cobalt_strike" rel="noreferrer noopener"><u>Cobalt Strike</u></a> were delivered following the initial WarmCookie infection.&nbsp;&nbsp;</p><p>While analyzing the campaigns, intrusion activity, and infrastructure associated with WarmCookie over the course of 2024, we also identified multiple overlaps with activity conducted by <a href="https://blog.talosintelligence.com/highlighting-ta866-asylum-ambuscade/" rel="noreferrer noopener"><u>TA866</u></a> in 2023.&nbsp;</p><h1 id="typical-infection-chains">Typical infection chains&nbsp;</h1><p>As previously mentioned, we have observed WarmCookie campaigns being conducted since at least April 2024. These campaigns rely on malspam or malvertising to facilitate the delivery of malicious content. &nbsp;</p><p>In the case of malspam, we have observed consistent use of invoice-related and job agency themes that entice victims to access hyperlinks present in either the email body, or within attached documents, such as PDFs.&nbsp;&nbsp;</p><p>Examples of common message subjects observed in campaigns conducted between April and August 2024 are listed below.&nbsp;</p><ul><li><code>United Rentals Inc: Invoice# [0-9]{9}\-[0-9]{3}</code>&nbsp;</li><li><code>Invoice and Remittance</code></li></ul><p>In a recent campaign conducted in August, the messages contained PDF attachments. The attachment filenames were randomized but typically use the following format.&nbsp;</p><ul><li><code>Attachment_[0-9]{3}\-[0-9]{3}\.pdf</code></li></ul><p>While there have been variations over time, below is a representative example of one of these emails and the associated PDF attachment.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/10/data-src-image-a9bcabae-965d-49a2-9fa4-f95fff22e06f.png" class="kg-image" alt="" loading="lazy" width="1223" height="888" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/10/data-src-image-a9bcabae-965d-49a2-9fa4-f95fff22e06f.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/10/data-src-image-a9bcabae-965d-49a2-9fa4-f95fff22e06f.png 1000w, https://blog.talosintelligence.com/content/images/2024/10/data-src-image-a9bcabae-965d-49a2-9fa4-f95fff22e06f.png 1223w" sizes="(min-width: 720px) 720px"><figcaption><i><em class="italic" style="white-space: pre-wrap;">WarmCookie emails and attachments.</em></i></figcaption></figure><p>The PDFs contain hyperlinks that direct victims to web servers hosting malicious JavaScript files that continue the infection process.&nbsp;</p><p>We have also observed WarmCookie campaigns leveraging infrastructure associated with traffic distribution and malware delivery systems. In one early campaign, we observed the use of the LandUpdates808 cluster of infrastructure described <a href="https://malasada.tech/the-landupdate808-fake-update-variant/" rel="noreferrer noopener"><u>here</u></a>. In observed cases, malicious JavaScript downloaders were being hosted at the following paths on servers associated with the LandUpdates808 cluster of web servers.&nbsp;</p><p>/wp-content/upgrade/update[.]php</p>
<p>Regardless of whether the delivery stage of the attack was conducted via malspam or malvertising, an obfuscated JavaScript downloader is delivered that is responsible for continuing the infection process. We have observed the use of ZIP archives to compress the JavaScript file and the delivery of the JavaScript file directly from the distribution infrastructure.&nbsp;&nbsp;</p><p>When executed, it deobfuscates and executes a PowerShell command that uses Bitsadmin to retrieve and execute the WarmCookie DLL using syntax, like that shown below.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/10/image-23.png" class="kg-image" alt="" loading="lazy" width="1750" height="556" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/10/image-23.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/10/image-23.png 1000w, https://blog.talosintelligence.com/content/images/size/w1600/2024/10/image-23.png 1600w, https://blog.talosintelligence.com/content/images/2024/10/image-23.png 1750w" sizes="(min-width: 720px) 720px"><figcaption><i><em class="italic" style="white-space: pre-wrap;">PowerShell execution.</em></i></figcaption></figure><p>We have observed a relatively small number of distribution servers hosting WarmCookie DLLs compared to the infrastructure used in earlier stages of the infection chain.&nbsp;&nbsp;</p><h2 id="warmcookie">WarmCookie&nbsp;</h2><p>The main WarmCookie payload has been extensively analyzed in prior reporting <a href="https://www.gdatasoftware.com/blog/2024/06/37947-badspace-backdoor" rel="noreferrer noopener"><u>here</u></a> and <a href="https://www.elastic.co/security-labs/dipping-into-danger" rel="noreferrer noopener"><u>here</u></a>.&nbsp;While performing this research, newly observed WarmCookie samples were <a href="https://x.com/GenThreatLabs/status/1840762181668741130" rel="noreferrer noopener"><u>reported</u></a> on social media during September 2024. We observed significant additions and changes in this latest version that demonstrate the threat actor is continuing to improve their tooling. &nbsp;</p><p>We observed changes to the way the malware is executed and how persistence is achieved on infected systems. As described in prior reporting, the malware is typically delivered and executed as a PE DLL or a PE EXE. If the payload is in the DLL format, it is typically executed with specific command-line parameters that determine whether persistence should be achieved. &nbsp;</p><p>In previous WarmCookie samples the execution was consistent with the following:&nbsp;</p><p>rundll32.exe &lt;DLL_Filename&gt;,Start /p</p>
<p>In the latest samples analyzed, this command-line syntax has been modified as follows:&nbsp;</p><p>rundll32.exe &lt;DLL_Filename&gt;,Start /u</p>
<p>Additionally, the user agent used during C2 communications in previous WarmCookie samples featured extraneous spaces not consistent with normal user agent strings seen in the wild. This allowed for easy detection of WarmCookie C2 activity via network traffic inspection. In the latest WarmCookie samples, this mistake has been corrected. Below is a comparison between the old and new user agent strings used during C2 communications.&nbsp;</p><p>Old User Agent:&nbsp;</p><p>Mozilla / 4.0 (compatible; MSIE 6.0; Windows NT 5.1; SV1;.NET CLR 1.0.3705)</p>
<p>New User Agent:&nbsp;</p><p>Mozilla/5.0 (Windows NT 6.1; Win64; x64; rv:109.0) Gecko/20100101 Firefox/115.0</p>
<p>We also observed the inclusion of a new self-updating mechanism that would enable an attacker to dynamically deliver updates to WarmCookie via the C2 server, however, this functionality did not appear to be fully implemented in the analyzed sample at the time.&nbsp;</p><p>In the latest sample, changes were made to the sandbox detection mechanism present in the malware where some checks present in previous versions have been removed.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/10/data-src-image-497bc96f-0bcd-40fc-9d8b-cb302b2c1ac2.png" class="kg-image" alt="" loading="lazy" width="1265" height="418" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/10/data-src-image-497bc96f-0bcd-40fc-9d8b-cb302b2c1ac2.png 600w, https://blog.talosintelligence.com/content/images/size/w1000/2024/10/data-src-image-497bc96f-0bcd-40fc-9d8b-cb302b2c1ac2.png 1000w, https://blog.talosintelligence.com/content/images/2024/10/data-src-image-497bc96f-0bcd-40fc-9d8b-cb302b2c1ac2.png 1265w" sizes="(min-width: 720px) 720px"><figcaption><i><em class="italic" style="white-space: pre-wrap;">WarmCookie sandbox detection.</em></i></figcaption></figure><p>Several changes to the C2 commands supported by the malware have also been made in the latest WarmCookie samples analyzed. The command to remove persistence and the malware itself has been deleted. New commands have been added as follows:&nbsp;</p><ul><li><strong>Command 0x8:</strong> Supports the creation of a DLL file received from the C2 server that is assigned a temporary filename and then executed by WarmCookie.&nbsp;&nbsp;&nbsp;</li><li><strong>Command 0xA</strong>: Appears to be a prepared update command, it is like Command 0x8, but adds hardcoded parameters to the DLL:&nbsp;&nbsp;<br><code>C:\Windows\System32\rundll32.exe &lt;tmpfilename.dll&gt; Start /update</code></li><li><strong>Command 0xB</strong>: Supports moving the malware to a temporary file name and location and deletes the previously scheduled task. It prepends the string âdatâ to the temporary filename. It also exits the C2 loop, leading to termination of the malware process.&nbsp;</li></ul><p>During the malwareâs initialization and startup phase, the <code>/update</code> parameter of the Command <code>0xA</code> is checked to determine if the parameter was set. Regardless of the result of this check, the same function is executed, as shown below.&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://blog.talosintelligence.com/content/images/2024/10/data-src-image-3691fe35-cde9-472c-87fd-d759fc287719.png" class="kg-image" alt="" loading="lazy" width="873" height="280" srcset="https://blog.talosintelligence.com/content/images/size/w600/2024/10/data-src-image-3691fe35-cde9-472c-87fd-d759fc287719.png 600w, https://blog.talosintelligence.com/content/images/2024/10/data-src-image-3691fe35-cde9-472c-87fd-d759fc287719.png 873w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">WarmCookie update parameter.&nbsp;</span></figcaption></figure><p>Analysis suggests that the malware will continue to evolve moving forward as the threat actor continues to improve on it and adds additional functionality as needed.&nbsp;</p><h1 id="links-to-past-intrusion-activity">Links to past intrusion activity&nbsp;</h1><p>While analyzing the distribution campaigns, infrastructure used, and post-compromise intrusion activity associated with WarmCookie, we identified multiple overlaps with previously observed malicious activity.&nbsp;&nbsp;</p><p>In earlier WarmCookie distribution campaigns, threat actors relied on lures that appear as if they were associated with talent/job search agencies. As mentioned <a href="https://www.elastic.co/security-labs/dipping-into-danger" rel="noreferrer noopener"><u>here</u></a>, the lure documents and landing pages associated with this campaign are like those used by distributors of Ursnif in past campaigns. &nbsp;</p><p>While analyzing intrusion activity associated with WarmCookie, we observed the deployment of CSharp-Streamer-RAT as a follow-on payload following the initial system compromise. CSharp-Streamer-RAT is a full-featured remote access trojan that offers robust functionality as described <a href="https://cyber.wtf/2023/12/06/the-csharp-streamer-rat/" rel="noreferrer noopener"><u>here</u></a>.&nbsp;&nbsp;</p><p>In this case, the sample reached out to a C2 server that was configured to use an SSL certificate that appeared to have been programmatically generated with several fields randomly populated. Using Regular Expressions to identify other servers with similar SSL characteristics, we identified three additional C2 servers, all previously associated with CSharp-Streamer-RAT samples. One of these C2 servers was observed being used by a CSharp-Streamer-RAT sample we identified in a previous intrusion that we assess with high confidence was conducted by TA866.&nbsp;&nbsp;</p><p>The screenshot below shows the relevant fields present within the SSL certificate associated with the CSharp-Streamer-RAT C2 server observed in previous intrusion activity we attribute to TA866.&nbsp;&nbsp;</p><figure class="kg-card kg-image-card kg-card-hascaption"><img system leading up to, during, and after its deployment. In the recent WarmCookie case, we also assess with high confidence that the attacker who deployed WarmCookie also deployed CSharp-Streamer-RAT following the initial compromise.</p><h1 id="warmcookie-vs-resident-backdoor">WarmCookie vs. Resident backdoor&nbsp;</h1><p>As referenced <a href="https://blog.talosintelligence.com/highlighting-ta866-asylum-ambuscade/" rel="noreferrer noopener"><u>here</u></a>, and in <a href="https://www.esentire.com/blog/esentire-threat-intelligence-malware-analysis-resident-campaign" rel="noreferrer noopener"><u>prior reporting</u></a>, TA866/Asylum Ambuscade has been observed delivering a post-compromise implant called Resident backdoor in prior intrusion activity. Prior <a href="https://www.bleepingcomputer.com/news/security/fake-browser-updates-spread-updated-warmcookie-malware/" rel="noreferrer">reporting</a> on WarmCookie has alluded to observed links between Resident backdoor and WarmCookie. </p><p>We performed a code and function level analysis of Resident backdoor samples from previous intrusion activity and WarmCookie samples from September 2024 and observed several notable similarities in the way core functionality has been implemented across both malware families. WarmCookie appears to contain much of the same functionality as Resident backdoor but has been significantly extended to support additional functionality.&nbsp;&nbsp;</p><p>We assess that both were likely developed by the same entity based on the following analysis findings:&nbsp;</p><ul><li>The RC4 implementation is consistent across both malware families.&nbsp;</li><li>The RC4 string decryption function implementation is consistent across both malware families.&nbsp;</li><li>Mutex management is performed consistently across both malware families.&nbsp;</li><li>Both malware families use GUID-like strings for the mutex.&nbsp;</li><li>The way in which various functions were constructed and the coding conventions used is consistent.&nbsp;</li><li>The definition of scheduled tasks to achieve persistence is consistent.&nbsp;</li><li>Both malware families wait one minute before executing the scheduled task.&nbsp;</li><li>The directory, file schema and parameters are similar in both malware families.&nbsp;&nbsp;</li><li>The initial startup logic and command line parameter implementation are similar.&nbsp;</li></ul><h2 id="code-similarity-analysis">Code similarity analysis&nbsp;</h2><p>We conducted a similarity analysis of the code execution flow between both Resident backdoor and a recent WarmCookie sample that was shared on social media. We observed consistent implementation of core functionality across both as well as consistent use of coding conventions across both malware families.&nbsp;</p><h3 id="task-scheduler-implementation">Task Scheduler implementation&nbsp;</h3><p>If the malware is initially executed without supplying any parameters, both Resident and WarmCookie first determine if the initially launched application was a PE DLL or an PE EXE. Depending on the result, they either create a filename with the extension â<code>.dll</code>" or â<code>.exe</code>". Also based on the results of this test, they both create a scheduled task via the Windows Task Scheduler, which spawns a copy of the malware after waiting for 60 seconds.
