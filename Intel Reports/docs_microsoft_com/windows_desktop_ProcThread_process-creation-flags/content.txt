										</nav><!-- <content> --><p>The following process creation flags are used by the <a href="/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-createprocessa" data-linktype="absolute-path"><strong>CreateProcess</strong></a>, <a href="/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-createprocessasusera" data-linktype="absolute-path"><strong>CreateProcessAsUser</strong></a>, <a href="/en-us/windows/desktop/api/WinBase/nf-winbase-createprocesswithlogonw" data-linktype="absolute-path"><strong>CreateProcessWithLogonW</strong></a>, and <a href="/en-us/windows/desktop/api/WinBase/nf-winbase-createprocesswithtokenw" data-linktype="absolute-path"><strong>CreateProcessWithTokenW</strong></a> functions. They can be specified in any combination, except as noted.</p>
<h2 id="example">Example</h2>
<pre><code class="lang-c">     BOOL creationResult;
    
    creationResult = CreateProcess(
        NULL,                   // No module name (use command line)
        cmdLine,                // Command line
        NULL,                   // Process handle not inheritable
        NULL,                   // Thread handle not inheritable
        FALSE,                  // Set handle inheritance to FALSE
        NORMAL_PRIORITY_CLASS | CREATE_NEW_CONSOLE | CREATE_NEW_PROCESS_GROUP, // creation flags
        NULL,                   // Use parent's environment block
        NULL,                   // Use parent's starting directory 
        &amp;startupInfo,           // Pointer to STARTUPINFO structure
        &amp;processInformation);   // Pointer to PROCESS_INFORMATION structure

</code></pre>
<p>Example from <a href="https://github.com/microsoft/Windows-classic-samples/blob/1d363ff4bd17d8e20415b92e2ee989d615cc0d91/Samples/ManagementInfrastructure/cpp/Process/Provider/WindowsProcess.c" data-linktype="external">Windows classic samples</a> on GitHub.</p>
<h2 id="flags">Flags</h2>
<table>
<thead>
<tr>
<th style="text-align: left;">Constant/value</th>
<th style="text-align: left;">Description</th>
</tr>
</thead>
<tbody>
<tr>
<td style="text-align: left;"><span id="CREATE_BREAKAWAY_FROM_JOB"></span><span id="create_breakaway_from_job"></span><dl> <dt><strong>CREATE_BREAKAWAY_FROM_JOB</strong></dt> <dt>0x01000000</dt> </dl></td>
<td style="text-align: left;">The child processes of a process associated with a job are not associated with the job. <br> If the calling process is not associated with a job, this constant has no effect. If the calling process is associated with a job, the job must set the <strong>JOB_OBJECT_LIMIT_BREAKAWAY_OK</strong> limit.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_DEFAULT_ERROR_MODE"></span><span id="create_default_error_mode"></span><dl> <dt><strong>CREATE_DEFAULT_ERROR_MODE</strong></dt> <dt>0x04000000</dt> </dl></td>
<td style="text-align: left;">The new process does not inherit the error mode of the calling process. Instead, the new process gets the default error mode. <br> This feature is particularly useful for multithreaded shell applications that run with hard errors disabled.<br> The default behavior is for the new process to inherit the error mode of the caller. Setting this flag changes that default behavior.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_NEW_CONSOLE"></span><span id="create_new_console"></span><dl> <dt><strong>CREATE_NEW_CONSOLE</strong></dt> <dt>0x00000010</dt> </dl></td>
<td style="text-align: left;">The new process has a new console, instead of inheriting its parent's console (the default). For more information, see <a href="/en-us/windows/console/creation-of-a-console" data-linktype="absolute-path">Creation of a Console</a>. <br> This flag cannot be used with <strong>DETACHED_PROCESS</strong>.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_NEW_PROCESS_GROUP"></span><span id="create_new_process_group"></span><dl> <dt><strong>CREATE_NEW_PROCESS_GROUP</strong></dt> <dt>0x00000200</dt> </dl></td>
<td style="text-align: left;">The new process is the root process of a new process group. The process group includes all processes that are descendants of this root process. The process identifier of the new process group is the same as the process identifier, which is returned in the <em>lpProcessInformation</em> parameter. Process groups are used by the <a href="/en-us/windows/console/generateconsolectrlevent" data-linktype="absolute-path"><strong>GenerateConsoleCtrlEvent</strong></a> function to enable sending a CTRL+BREAK signal to a group of console processes.<br> If this flag is specified, CTRL+C signals will be disabled for all processes within the new process group.<br> This flag is ignored if specified with <strong>CREATE_NEW_CONSOLE</strong>.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_NO_WINDOW"></span><span id="create_no_window"></span><dl> <dt><strong>CREATE_NO_WINDOW</strong></dt> <dt>0x08000000</dt> </dl></td>
<td style="text-align: left;">The process is a console application that is being run without a console window. Therefore, the console handle for the application is not set.<br> This flag is ignored if the application is not a console application, or if it is used with either <strong>CREATE_NEW_CONSOLE</strong> or <strong>DETACHED_PROCESS</strong>.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_PROTECTED_PROCESS"></span><span id="create_protected_process"></span><dl> <dt><strong>CREATE_PROTECTED_PROCESS</strong></dt> <dt>0x00040000</dt> </dl></td>
<td style="text-align: left;">The process is to be run as a protected process. The system restricts access to protected processes and the threads of protected processes. For more information on how processes can interact with protected processes, see <a href="process-security-and-access-rights" data-linktype="relative-path">Process Security and Access Rights</a>.<br> To activate a protected process, the binary must have a special signature. This signature is provided by Microsoft but not currently available for non-Microsoft binaries. There are currently four protected processes: media foundation, audio engine, Windows error reporting, and system. Components that load into these binaries must also be signed. Multimedia companies can leverage the first two protected processes. For more information, see <a href="../medfound/protected-media-path" data-linktype="relative-path">Overview of the Protected Media Path</a>.<br> <strong>Windows Server&nbsp;2003 and Windows&nbsp;XP:</strong> This value is not supported.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_PRESERVE_CODE_AUTHZ_LEVEL"></span><span id="create_preserve_code_authz_level"></span><dl> <dt><strong>CREATE_PRESERVE_CODE_AUTHZ_LEVEL</strong></dt> <dt>0x02000000</dt> </dl></td>
<td style="text-align: left;">Allows the caller to execute a child process that bypasses the process restrictions that would normally be applied automatically to the process.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_SECURE_PROCESS"></span><span id="create_secure_process"></span><dl> <dt><strong>CREATE_SECURE_PROCESS</strong></dt> <dt>0x00400000</dt> </dl></td>
<td style="text-align: left;">This flag allows secure processes, that run in the Virtualization-Based Security environment, to launch.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_SEPARATE_WOW_VDM"></span><span id="create_separate_wow_vdm"></span><dl> <dt><strong>CREATE_SEPARATE_WOW_VDM</strong></dt> <dt>0x00000800</dt> </dl></td>
<td style="text-align: left;">This flag is valid only when starting a 16-bit Windows-based application. If set, the new process runs in a private Virtual DOS Machine (VDM). By default, all 16-bit Windows-based applications run as threads in a single, shared VDM. The advantage of running separately is that a crash only terminates the single VDM; any other programs running in distinct VDMs continue to function normally. Also, 16-bit Windows-based applications that are run in separate VDMs have separate input queues. That means that if one application stops responding momentarily, applications in separate VDMs continue to receive input. The disadvantage of running separately is that it takes significantly more memory to do so. You should use this flag only if the user requests that 16-bit applications should run in their own VDM.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_SHARED_WOW_VDM"></span><span id="create_shared_wow_vdm"></span><dl> <dt><strong>CREATE_SHARED_WOW_VDM</strong></dt> <dt>0x00001000</dt> </dl></td>
<td style="text-align: left;">The flag is valid only when starting a 16-bit Windows-based application. If the DefaultSeparateVDM switch in the Windows section of WIN.INI is <strong>TRUE</strong>, this flag overrides the switch. The new process is run in the shared Virtual DOS Machine.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_SUSPENDED"></span><span id="create_suspended"></span><dl> <dt><strong>CREATE_SUSPENDED</strong></dt> <dt>0x00000004</dt> </dl></td>
<td style="text-align: left;">The primary thread of the new process is created in a suspended state, and does not run until the <a href="/en-us/windows/win32/api/processthreadsapi/nf-processthreadsapi-resumethread" data-linktype="absolute-path"><strong>ResumeThread</strong></a> function is called.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="CREATE_UNICODE_ENVIRONMENT"></span><span id="create_unicode_environment"></span><dl> <dt><strong>CREATE_UNICODE_ENVIRONMENT</strong></dt> <dt>0x00000400</dt> </dl></td>
<td style="text-align: left;">If this flag is set, the environment block pointed to by <em>lpEnvironment</em> uses Unicode characters. Otherwise, the environment block uses ANSI characters.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="DEBUG_ONLY_THIS_PROCESS"></span><span id="debug_only_this_process"></span><dl> <dt><strong>DEBUG_ONLY_THIS_PROCESS</strong></dt> <dt>0x00000002</dt> </dl></td>
<td style="text-align: left;">The calling thread starts and debugs the new process. It can receive all related debug events using the <a href="/en-us/windows/win32/api/debugapi/nf-debugapi-waitfordebugevent" data-linktype="absolute-path"><strong>WaitForDebugEvent</strong></a> function.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="DEBUG_PROCESS"></span><span id="debug_process"></span><dl> <dt><strong>DEBUG_PROCESS</strong></dt> <dt>0x00000001</dt> </dl></td>
<td style="text-align: left;">The calling thread starts and debugs the new process and all child processes created by the new process. It can receive all related debug events using the <a href="/en-us/windows/win32/api/debugapi/nf-debugapi-waitfordebugevent" data-linktype="absolute-path"><strong>WaitForDebugEvent</strong></a> function. <br> A process that uses <strong>DEBUG_PROCESS</strong> becomes the root of a debugging chain. This continues until another process in the chain is created with <strong>DEBUG_PROCESS</strong>.<br> If this flag is combined with <strong>DEBUG_ONLY_THIS_PROCESS</strong>, the caller debugs only the new process, not any child processes.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="DETACHED_PROCESS"></span><span id="detached_process"></span><dl> <dt><strong>DETACHED_PROCESS</strong></dt> <dt>0x00000008</dt> </dl></td>
<td style="text-align: left;">For console processes, the new process does not inherit its parent's console (the default). The new process can call the <a href="/en-us/windows/console/allocconsole" data-linktype="absolute-path"><strong>AllocConsole</strong></a> function at a later time to create a console. For more information, see <a href="/en-us/windows/console/creation-of-a-console" data-linktype="absolute-path">Creation of a Console</a>. <br> This value cannot be used with <strong>CREATE_NEW_CONSOLE</strong>.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="EXTENDED_STARTUPINFO_PRESENT"></span><span id="extended_startupinfo_present"></span><dl> <dt><strong>EXTENDED_STARTUPINFO_PRESENT</strong></dt> <dt>0x00080000</dt> </dl></td>
<td style="text-align: left;">The process is created with extended startup information; the <em>lpStartupInfo</em> parameter specifies a <a href="/en-us/windows/desktop/api/WinBase/ns-winbase-startupinfoexa" data-linktype="absolute-path"><strong>STARTUPINFOEX</strong></a> structure.<br> <strong>Windows Server&nbsp;2003 and Windows&nbsp;XP:</strong> This value is not supported.<br></td>
</tr>
<tr>
<td style="text-align: left;"><span id="INHERIT_PARENT_AFFINITY"></span><span id="inherit_parent_affinity"></span><dl> <dt><strong>INHERIT_PARENT_AFFINITY</strong></dt> <dt>0x00010000</dt> </dl></td>
<td style="text-align: left;">The process inherits its parent's affinity. If the parent process has threads in more than one <a href="processor-groups" data-linktype="relative-path">processor group</a>, the new process inherits the group-relative affinity of an arbitrary group in use by the parent.<br> <strong>Windows Server&nbsp;2008, Windows&nbsp;Vista, Windows Server&nbsp;2003 and Windows&nbsp;XP:</strong> This value is not supported.<br></td>
</tr>
</tbody>
</table>
<h2 id="remarks">Remarks</h2>
<p>On 32-bit Windows, 16-bit applications are simulated by ntvdm.exe, not run as individual processes. Therefore, the process creation flags apply to <em>ntvdm.exe</em>. Because ntvdm.exe persists after you run the first 16-bit application, when you launch another 16-bit application, the new creation flags are not applied, except for <strong>CREATE_NEW_CONSOLE</strong> and <strong>CREATE_SEPARATE_WOW_VDM</strong>, which create a new ntvdm.exe.</p>
