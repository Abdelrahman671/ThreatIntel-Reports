                let docLanguage = document.querySelector('meta[name="language"]').getAttribute('content');
                
                var docSeparator;
                
                if ((docLocale.toLowerCase() == "us") && (docLanguage.toLowerCase() == "en")) {
                    docSeparator="/";
                } else {
                    docSeparator="_";
                }
                
                let anchorURLReplace = docLanguage.toLowerCase() + docSeparator + docLocale.toLowerCase();
                anchorChildHREF = anchorChildHREF.replace("%3Clocale%3E", anchorURLReplace);

class="lia-message-template-content-zone"><p><ul =""=""></ul></p><h2 id="toc-hId--755759320"></h2><h2 id="toc-hId--952272825">Introduction</h2><p>This document describes recommendations to consider against password-spray attacks aimed at Remote Access VPN services in Secure Firewall.</p><h2 id="toc-hId--1148786330">Background Information</h2><p>Password spray attacks are a type of brute-force attack where an attacker attempts to gain unauthorized access to multiple user accounts by systematically trying a few commonly used passwords across many accounts.&nbsp;Successful password spray attacks can lead to unauthorized access to sensitive information, data breaches, and potential compromises of network integrity</p><p>Moreover, these attacks, even when unsuccessful in their attempt to gain access, can consume computational resources from the Secure Firewall and prevent valid users from connecting to the remote access VPN services.</p><p> </p><h2 id="toc-hId--1345299835">Observed Behaviors</h2><p>When your Secure Firewall is targeted by password-spray attacks in Remote Access VPN services, you can identify these attacks by monitoring syslogs and using specific show commands. The most common behaviors to look for include:</p><h3 id="toc-hId--1412730621">Unusual Amount of Rejected Authentication Requests</h3><p>The VPN headend Cisco Secure Firewall ASA or FTD shows symptoms of password-spray attacks with an unusual rate (100-thousands or millions) of rejected authentication attempts.&nbsp;</p><p style="background-image: url('http://www.cisco.com/en/US/i/templates/note.gif'); background-repeat: no-repeat; background-position: 2px 4px; height: auto; width: auto; padding: 10px 5px 10px 35px; margin-top: 10px; margin-bottom: 10px; border-top: 1px solid #ccc; border-bottom: 1px solid #ccc; overflow-x: hidden;"><strong>Note</strong>: These unusual attempts to authenticate can be directed towards either the LOCAL database or external authentication servers.</p><p> </p><p>The best way to detect this is by looking at the syslog. Look for an unusual number of any of the next ASA syslog IDs:</p><p> </p><ul>
<li>%ASA-6-113015</li>
</ul><pre><strong><span>%ASA-6-113015</span></strong>: AAA user authentication Rejected : reason = User was not found : local database : <strong><span>user</span></strong> = <em>admin</em> : user <strong><span>IP</span></strong> = <em>x.x.x.x</em><em><br /></em></pre><p> </p><ul>
<li>%ASA-6-113005</li>
</ul><pre><strong>%ASA-6-113005</strong>: AAA user authentication Rejected : reason = Unspecified : server = x.x.x.x : user = ***** : user IP = x.x.x.x</pre><p> </p><ul>
<li>%ASA-6-716039</li>
</ul><pre><strong>%ASA-6-716039</strong>: Group &lt;DfltGrpPolicy&gt; User &lt;admin&gt; IP &lt;x.x.x.x&gt; Authentication: rejected, Session Type: WebVPN.</pre><p> </p><p>The username is always hidden until the <strong>no logging hide username</strong>&nbsp;command is configured on the ASA.</p><p style="background-image: url('http://www.cisco.com/en/US/i/templates/note.gif'); overflow-x: hidden;"><strong>Note</strong>: This gives insight into verifying if valid users are generated or known by offending IPs however, please be cautious as usernames will be visible in the logs.</p><p> </p><p>To verify, log in to the ASA or FTD Command Line Interface (CLI), run the <strong>show aaa-server&nbsp;</strong>command, and&nbsp;investigate for an unusual number of attempted and rejected authentication requests to any of the configured AAA servers:</p><pre>ciscoasa# show aaa-server<br /><br />Server Group: LDAP-SERVER - - - - - &gt;&gt;&gt;&gt; Enable Logging.</h3><p>Logging is a crucial part of cybersecurity that involves recording events happening within a system. The absence of detailed logs leaves gaps in understanding, hindering a clear analysis of the attack method. It is recommended that you enable logging to a remote syslog server for improved correlation and auditing of network and security incidents across various network devices.</p><p>For information on how to configure logging, see the next platform-specific guides:</p><p> </p><p>Cisco ASA Software:</p><ul>

Configure Threat Detection Features or Hardening Measures for Remote Access VPN.</h3><p>To help mitigate the impact and reduce the likelihood of occurrence of these brute-force attacks on your RAVPN connections,&nbsp;you can review and apply the next configuration options:</p><h3 id="toc-hId-1899669150"></h3><h4 id="toc-hId-1334521269">Option 1 (preferred): Configure Threat Detection for Remote Access VPN Services.</h4><p>Threat detection features for remote access VPN services allow you to protect against these kind of attacks by&nbsp;automatically blocking the host (IP address) that exceeds the configured thresholds, to prevent further attempts until you manually remove the shun of the IP address.</p><p>These threat detection features are currently supported in the Cisco Secure Firewall versions listed next:</p><p> </p><p>ASA Software:</p><ul>

</ul><p style="background-image: url('http://www.cisco.com/en/US/i/templates/note.gif'); background-repeat: no-repeat; background-position: 2px 4px; height: auto; width: auto; padding: 10px 5px 10px 35px; margin-top: 10px; margin-bottom: 10px; border-top: 1px solid #ccc; border-bottom: 1px solid #ccc; overflow-x: hidden;"><strong>Note</strong>: These features are currently <strong>not supported in version trains 7.1 or 7.3</strong>.</p><p> </p><p>For full details and configuration guidance, please refer to the next documents:</p><ul>

</ul><p> </p><h4 id="toc-hId-1138007764">Option 2: Apply Hardening Measures for Remote Access VPN.</h4><p>If the threat detection features for Remote Access VPN services are not supported in your Secure Firewall version, implement all the next hardening measures to mitigate the impact of these attacks:</p><ol>
<li>Disable AAA Authentication in the DefaultWEBVPN and DefaultRAGroup Connection Profiles (step-by-step: <a class="self_link" href="https://www.cisco.com/c/en/us/support/docs/security/secure-client/221880-implement-hardening-measures-for-secure.html#toc-hId--1773400573" target="_blank" rel="noopener nofollow noreferrer">ASA</a> | <a class="self_link" href="https://www.cisco.com/c/en/us/support/docs/security/secure-client/221880-implement-hardening-measures-for-secure.html#toc-hId-613091301" target="_blank" rel="noopener nofollow noreferrer">FTD managed by FMC</a>).</li>
<li>Disable Secure Firewall Posture (Hostscan) from the DefaultWEBVPNGroup and DefaultRAGroup&nbsp;(step-by-step:&nbsp;<a class="self_link" href="https://www.cisco.com/c/en/us/support/docs/security/secure-client/221880-implement-hardening-measures-for-secure.html#toc-hId--1998243892" target="_blank" rel="noopener nofollow noreferrer">ASA</a> | <a class="self_link" href="https://www.cisco.com/c/en/us/support/docs/security/secure-client/221880-implement-hardening-measures-for-secure.html#toc-hId--201292533" target="_blank" rel="noopener nofollow noreferrer">FTD managed by FMC</a>).</li>
<li>Disable Group-aliases and Enable Group-URLs in the rest of the connection profiles&nbsp;(step-by-step:&nbsp;<a class="self_link" href="https://www.cisco.com/c/en/us/support/docs/security/secure-client/221880-implement-hardening-measures-for-secure.html#toc-hId-2100209899" target="_blank" rel="noopener nofollow noreferrer">ASA</a> | <a class="self_link" href="https://www.cisco.com/c/en/us/support/docs/security/secure-client/221880-implement-hardening-measures-for-secure.html#toc-hId--987346553" target="_blank" rel="noopener nofollow noreferrer">FTD managed by FMC</a>).</li>
</ol><p style="background-image: url('http://www.cisco.com/en/US/i/templates/note.gif'); background-repeat: no-repeat; background-position: 2px 4px; height: auto; width: auto; padding: 10px 5px 10px 35px; margin-top: 10px; margin-bottom: 10px; border-top: 1px solid #ccc; border-bottom: 1px solid #ccc; overflow-x: hidden;"><strong>Note</strong>: If you require support with FTD managed through local Firewall Device Management (FDM), please contact the Technical Assistance Center (TAC) for expert guidance.</p><p> </p><p>For further details please refer to the&nbsp;<a class="self_link" href="https://www.cisco.com/c/en/us/support/docs/security/secure-client/221880-implement-hardening-measures-for-secure.html#toc-hId-613091301" target="_blank" rel="noopener nofollow noreferrer">Implement Hardening Measures for Secure Client AnyConnect VPN</a>&nbsp;guide.</p><p> </p><h4 class="p1" id="toc-hId-941494259">Option 3: Manually Block Connection Attempts from Malicious Sources.</h4><p>In order to impede connection attempts from unauthorized sources, you can implement any of the options listed below:</p><ul>
<li class="p1">Use the "shun" command:</li>
</ul><p>This is a straightforward approach to blocking a malicious IP, however, it must be done manually. Please read the section <a class="self_link" href="https://www.cisco.com/c/en/us/support/docs/security/secure-firewall-threat-defense/221531-configure-control-plane-access-control-p.html#toc-hId-792985040" target="_blank" rel="noopener nofollow noreferrer">Alternative configuration to block attacks for secure firewall using the 'shun' Command</a>&nbsp;for further details.&nbsp;</p><ul>
<li>Configure Control-plane ACL:</li>
</ul><p>Implement a control-plane ACL on the ASA/FTD to filter out unauthorized public IP addresses and prevent them from initiating remote VPN sessions.&nbsp;<a href="https://www.cisco.com/c/en/us/support/docs/security/secure-firewall-threat-defense/221531-configure-control-plane-access-control-p.html" target="_blank" rel="noopener nofollow noreferrer">Configure Control Plane Access Control Policies for Secure Firewall Threat Defense and ASA.</a></p><p style="background-image: url('http://www.cisco.com/en/US/i/templates/note.gif'); background-repeat: no-repeat; background-position: 2px 4px; height: auto; width: auto; padding: 10px 5px 10px 35px; margin-top: 10px; margin-bottom: 10px; border-top: 1px solid #ccc; border-bottom: 1px solid #ccc; overflow-x: hidden;" data-unlink="true"><strong>Note</strong>: Cisco Talos has published a list of IP addresses and credentials associated with these attacks. A link to their GitHub repository can be found in the&nbsp;"IOCs" section&nbsp;of their <a href="https://blog.talosintelligence.com/large-scale-brute-force-activity-targeting-vpns-ssh-services-with-commonly-used-login-credentials/" target="_blank" rel="noopener nofollow noreferrer">advisory</a>. <span>It is important to note that the source IP addresses for this traffic are likely to change, therefore, you must review the security logs (syslog) to identify the problematic IP addresses. Upon identification, any of the 3 options can be used to block them.</span></p><p> </p><h2 id="toc-hId-1331786768">Related Behaviors</h2><p>There are certain symptoms that can be experienced as a consequence of the Secure Firewall being targeted by password spray attacks. To resolve these issues, consider implementing the recommendations provided in this document.</p><h3 id="toc-hId-841870256">Symptom 1: Unable to establish VPN connections with Cisco Secure Client (AnyConnect) when Firewall Posture (HostScan) is enabled</h3><p>When attempting to establish a RAVPN connection using Cisco Secure Client (AnyConnect), users can intermittently encounter an error message that states, <strong>"Unable to complete connection. Cisco Secure Desktop not installed on the client."</strong>. This behavior typically arises when there is a failure to allocate a hostscan token by the VPN headend, either a Cisco Secure Firewall ASA or FTD. Notably, this allocation failure correlates with instances of brute-force attacks targeting the Secure Firewall infrastructure and prevents the successful completion of the VPN connection process.. This behavior was tracked and resolved via&nbsp;<a href="https://tools.cisco.com/bugsearch/bug/CSCwj45822" target="_blank" rel="noopener nofollow noreferrer">Cisco bug ID CSCwj45822</a>.</p><p> </p><p><span class="lia-inline-image-display-wrapper lia-image-align-center" image-alt="angelort_0-1711382905110.png" style="width: 400px;"><img src="/c/dam/en/us/support/docs/security/secure-firewall-threat-defense/221806-password-spray-attacks-impacting-custome-00.png" alt="secure_client_error" aria-label="secure_client_error"/></span></p><p> </p><p style="background-image: url('http://www.cisco.com/en/US/i/templates/note.gif'); background-repeat: no-repeat; background-position: 2px 4px; height: auto; width: auto; padding: 10px 5px 10px 35px; margin-top: 10px; margin-bottom: 10px; border-top: 1px solid #ccc; border-bottom: 1px solid #ccc; overflow-x: hidden;"><strong>Note</strong>: This specific behavior occurs only when Firewall Posture (HostScan) is enabled at the headend, regardless of the Secure Client or AnyConnect version used.</p><h3 id="toc-hId-645356751"></h3><p>To confirm if the VPN headend Cisco Secure Firewall ASA or FTD show symptoms of hostscan token allocation failures, run the&nbsp;<strong>debug menu webvpn 187 0</strong> command.<br /><br /></p><pre>ASA# debug menu webvpn 187 0 <br />Allocated Hostscan token = 1000<br /><strong>Hostscan token allocate failure = xxx - - - - &gt; Increments</strong> </pre><h3 id="toc-hId-448843246"></h3><p style="background-image: url('http://www.cisco.com/en/US/i/templates/note.gif'); background-repeat: no-repeat; background-position: 2px 4px; height: auto; width: auto; padding: 10px 5px 10px 35px; margin-top: 10px; margin-bottom: 10px; border-top: 1px solid #ccc; border-bottom: 1px solid #ccc; overflow-x: hidden;"><strong>Note</strong>: The occurrence of this issue is a consequence of the attacks. This behavior was tracked and resolved via&nbsp;<a href="https://tools.cisco.com/bugsearch/bug/CSCwj45822" target="_blank" rel="noopener nofollow noreferrer">Cisco bug ID CSCwj45822</a>.</p><p> </p><p>To resolve this issue, consider implementing the recommendations provided in this document.</p><p> </p><h2 id="toc-hId-545732748">Additional Hardening Implementations for RAVPN</h2><p>You can consider additional countermeasures that require extra changes to your deployments to harden the security of your Remote Access VPN deployment, such as adopting<strong> certificate-based authentication</strong> for RAVPN.
