		<div>
			<h1><span class="body-subhead-title">Summary</span></h1>
<ul>
<li><span class="rte-red-bullet" style="font-weight: normal;">Attackers exploit exposed Docker Remote API servers to deploy  the perfctl malware through probing and payload execution.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">The attack involves creating a Docker container with specific settings and executing a Base64 encoded payload.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Payload execution includes escaping the container, creating a bash script, setting environment variables, and downloading a malicious binary disguised as a PHP extension.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Attackers use evasion techniques to avoid detection, such as checking for similar processes and creating directories and a custom function to download files.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;"> We provide a detailed breakdown of the attack sequence, shedding light on how threat actors leverage vulnerable Docker Remote API servers.</span></li>
</ul>
<p>Recent cyberattacks have leveraged unprotected Docker Remote API servers to deploy malicious code. Attacks targeting the Docker Remote API server are structured, starting with probing for the server's presence and ending with the actual execution of payloads.</p>
<p>We will conduct a detailed analysis of the attack flow, describing how attackers exploit vulnerable Docker Remote API servers. By looking over recent incidents, we will emphasize the importance of securing the Docker Remote API server and the potential consequences of this exploitation.</p>
<p>In a similar previous incident, an unknown threat actor installed a cryptocurrency miner using vulnerable <a href="https://docs.docker.com/engine/api/v1.45/" target="_blank">Docker Remote API</a> servers. The attacker sets up a docker container using the “<i>ubuntu:mantic-20240405”</i> image from Docker Hub, then uses &quot;nsenter&quot; to break out of the container and run the Base64 encoded payload.</p>
<h1><span class="body-subhead-title">The Attack Sequence</span></h1>

		</div>
	

</div>
            <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig1.png" alt="Attack chain"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 1: Attack chain</figcaption>
			
		</div>
	</figure>

</div>
            <div>




    
    
    <div class="richText">
	
    


	
		<div>
			<h2><span class="body-subhead-title">Probing the Docker Remote API Server</span></h2>
<p>The sequence of events in this planned <a href="/en_us/research/24/f/commando-cat-a-novel-cryptojacking-attack-.html">attack campaign begins with a ping to the Docker Remote API server</a>, which is the pivotal starting point for the subsequent chain of actions.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig2.jpg" alt="Ping Request"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 2: Ping Request</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<h2><span class="body-subhead-title">Container Creation with an <a name="_Int_J7wavKV4" id="_Int_J7wavKV4"></a>ubuntu:mantic-20240405 Image</span></h2>
<p>A container named &quot;kube-edagent” was created from the ubuntu:mantic-20240405<b> </b>base image, with “sleep 9955&quot; specified as the command to be executed once the container is up and running. The &quot;sleep&quot; command is used to identify the container process when running the &quot;ps&quot; command.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig3.jpg" alt="Container creation request"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 3: Container creation request</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>In the request mentioned above, the attackers create a container named &quot;kube-edagent,&quot; deliberately giving it a name similar to a legitimate container. They configure the container to operate in privileged mode by setting &quot;Privileged&quot; to &quot;true.&quot; They also utilize &quot;pid mode: host,&quot; allowing the container to share the Process ID (PID) namespace of the host system. This means the processes running inside the container will share the same PID namespace as the processes on the host. As a result, the container's processes will be able to see and interact with all the processes running on the host system in the same way as all running processes, as if they were running directly on the host.</p>
<p>The image below is pulled from Docker Hub, and another attempt is made to create the container if the ubuntu image is not present in the victim machine.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig4.jpg" alt="Ubuntu image creation request"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 4: Ubuntu image creation request</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<h2><span class="body-subhead-title">Payload Execution</span></h2>
<p>Payloads are executed using a Docker Exec API. The payload consists of two parts. The first part tries to escape the container by using the <a href="https://kubehound.io/reference/attacks/CE_NSENTER/" target="_blank">&quot;nsenter</a>&quot; command to enter the specific namespaces of the target process with PID 1 (“target 1”), which is the process ID of &quot;init.&quot; This command runs as root and includes flags such as &quot;<b>--mount, --uts, --ipc, --net, --pid</b>,&quot; indicating that it should enter the target's mount, UTS, IPC, network, and PID namespaces, effectively granting it similar capabilities as if it were running in the host system. The second part contains a Base64 encoded shell script.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig5.jpg" alt="Command execution in container"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 5: Command execution in container</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The decoded Base64 payload executes the following commands:</p>
<p><b>Step 1: Check and Prevent Duplicate Processes</b></p>
<p>It checks for multiple running processes matching the pattern &quot;nsenter.*bash.*base64&quot; to avoid running similar processes simultaneously.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig6.png" alt="Checks for and Terminates Multiple Instances of the Process"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 6: Checks for and Terminates Multiple Instances of the Process</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p><b>Step 2: Create a Bash Script</b></p>
<p>It creates a bash script called &quot;kubeupd&quot; in the &quot;/tmp&quot; directory. This script sets the environment variable &quot;VEI&quot; to &quot;dck_&lt;public IP of docker API&gt;.&quot; It is assumed that a customized payload has been generated based on the IP address identified in earlier stages. The script also includes another Base64 encoded bash payload.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig7.jpg" alt="Kubeupd file creation in &#34;/tmp&#34;"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 7: Kubeupd file creation in &#34;/tmp&#34;</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p><b>Step 3: Unpack the Base64 Payload</b></p>
<p>The Base64 payload located in the &quot;/tmp/kubeupd&quot; file is designed to perform the following actions:</p>
<ol>
<li><span style="font-weight: normal;">It contains a custom &quot;__curl&quot; function that can be used when curl or wget is not present in the system.</span></li>
<li><span style="font-weight: normal;">The script will terminate if the architecture is not &quot;x86_64.&quot; It initially checks for the presence of the &quot;/tmp&quot; directory and creates it if it does not exist. Then, it sets the &quot;/tmp&quot; mount as executable using the &quot;exec&quot; option. Subsequently, it creates &quot;.perfc&quot; and &quot;xdiag&quot; directories within the &quot;/tmp&quot; directory.</span></li>
</ol>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig8.png" alt="Mounts &#34;/tmp&#34; directory"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 8: Mounts &#34;/tmp&#34; directory</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<ol start="3">
<li><span style="font-weight: normal;">It sets the AAZHDE environment variable to localhost if it's not already set.</span></li>
</ol>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig9.png" alt="sets environment variable"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 9: sets environment variable</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<ol start="4">
<li><span style="font-weight: normal;">The environment variable $VEI, which contains &quot;dck_&lt;public_ip&gt;,&quot; is included in &quot;/tmp/.xdiag/vei.&quot;</span></li>
</ol>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig10.png" alt="Checks for VEI environment variable"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 10: Checks for VEI environment variable</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<ol start="5">
<li><span style="font-weight: normal;">It confirms the presence of a malicious process by checking &quot;/tmp/.xdiag/p,&quot; which should contain the PID of the malicious process. It verifies the existence of the same process by checking the &quot;/proc/&quot; directory. The secondary check looks for active TCP connections using ports 44870 or 63582.</span></li>
</ol>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig11.png" alt="Checks if a malicious binary is running "/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 11: Checks if a malicious binary is running </figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<ol start="6">
<li><span style="font-weight: normal;">If it confirms that the process is not running, then it will download the malicious binary, which is disguised as a PHP extension, to avoid file extension-based detection. Downloading the binary file uses the custom __curl function if curl and wget are not in the system. It downloads the file in the “/tmp” directory with the “httpd” name to look like a legitimate name. We were unable to determine the specific payload downloaded in this incident.</span></li>
</ol>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fig12.png" alt="Downloads the malicious binary"/>

    <div class="richText">
	
    


	
		<div>
			<ol start="7">
<li><span style="font-weight: normal;">If the downloaded file exists and its size matches a specific value (e.g., 9301499), it triggers more actions. These actions include killing processes (perfctl), setting permissions, updating the PATH environment variable, and executing a command (KRI=kr httpd) in the background.</span><i></i></li>
</ol>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">

			<ol start="8">
<li><span style="font-weight: normal;">It then deletes the &quot;/tmp/.install.pid33&quot; marker file.</span></li>
</ol>
<p><b>Step 4: Engage a Persistence Strategy</b></p>
<p>To remain active, the malware uses a persistence strategy. If systemd runs as non-offline, it creates a systemd service using &quot;multi-user.target.&quot; Otherwise, it resorts to creating a cron job, which is challenging to eradicate.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">

    <div class="richText">
	
    


	
		<div>
			<h2><span class="body-subhead-title">Base64 Decoded Functions</span></h2>
<p>In addition to the fallback mechanism, the Base64 payload, once decoded, contains four distinct functions:</p>
<ul>
<li><span class="rte-red-bullet"><b>Kill_container</b>: This command kills a process with the &quot;sleep 9955&quot; argument.</span></li>
</ul>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/kill_con.png" alt="Kill_container: This command kills a process with the &#34;sleep 9955&#34; argument."/>
		
		<div class="caption-image-container ">
			
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<ul>
<li><span class="rte-red-bullet"><b>Wait_run</b>: It waits for the &quot;/tmp/k8s.run42&quot; file to exist or until the maximum wait time is reached.</span></li>
</ul>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/wait_run.png" alt="Wait_run: It waits for the &#34;/tmp/k8s.run42&#34; file to exist or until the maximum wait time is reached."/>
		
		<div class="caption-image-container ">
			
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<ul>
<li><span class="rte-red-bullet"><b>Restore_sh</b>: It replaces the &quot;/bin/sh&quot; file with &quot;/bin/kkbush&quot; to bypass the detection (T1036.005).</span></li>
</ul>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/restore_sh.png" alt="Restore_sh: It replaces the &#34;/bin/sh&#34; file with &#34;/bin/kkbush&#34; to bypass the detection (T1036.005)"/>
		
		<div class="caption-image-container ">
			
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<ul>
<li><span class="rte-red-bullet"><b>Fallback</b>: It sets up a persistent backdoor by replacing the original &quot;/bin/sh&quot; shell with a modified version that enables privilege escalation and execution of any command. It then duplicates the original &quot;sh&quot; file as the &quot;kkbush&quot; and &quot;kbush&quot; binaries and later restores it during the cleanup process. In addition, it sets up a background process (kubeupd) for further interaction and maintains the backdoor using the wait_run function.</span></li>
</ul>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/j/attackers-target-exposed-docker-remote-api-servers-with-perfctl-malware/fallback.png" alt="Fallback"/>
		
		<div class="caption-image-container ">
			
			
		</div>
	</figure>

    <div class="richText">
	
    


	
		<div>
			<p>It uses the Tor network to reroute traffic, as confirmed by a Tor relay node (192.121.108.237) in the network packet capture.</p>

		</div>
	

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<h1><span class="body-subhead-title">Recommendations</span></h1>
<ul>
<li><span class="rte-red-bullet" style="font-weight: normal;">We recommend the following steps to enhance the security of Docker Remote API servers and mitigate the risks associated with potential exploitation for malicious activities:</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Secure Docker Remote API servers by implementing strong access controls and authentication mechanisms to prevent unauthorized access.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Regularly monitor Docker Remote API servers for any unusual or unauthorized activities, and promptly investigate and address any suspicious behavior.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Implement container <a href="https://docs.docker.com/engine/security/" target="_blank">security best practices</a>, such as avoiding the use of &quot;Privileged&quot; mode and carefully reviewing container images and configurations before deployment.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;"><a href="https://www.docker.com/resources/trainings/" target="_blank">Educate and train personnel</a> responsible for managing Docker Remote API servers about security best practices and potential attack vectors.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Stay informed about <a href="https://docs.docker.com/security/security-announcements/" target="_blank">security updates</a> and <a href="https://docs.docker.com/desktop/release-notes/" target="_blank">patches</a> for Docker and related software to address any known vulnerabilities that could be exploited by threat actors.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Regularly review and update security policies and procedures related to Docker Remote API server management to align with the latest security best practices and recommendations.</span></li>
</ul>
<h1><span class="body-subhead-title">Conclusion</span></h1>
<p>Exploiting exposed Docker remote API servers has now reached a critical level where the attention of an organization and its security professionals is seriously required. The first step to avoiding such incidents is understanding the attack sequence a threat actor can use. It is essential that every organization's Docker Remote API server is secured, monitored regularly for unauthorized access and suspicious activities to reduce the risk of attacks, and has security patches up to date. Carrying out regular security audits will significantly improve the general security posture against such types of exploits.</p>

		</div>
