<p><em>with Pedro Marrucho and David Alves</em></p><p></p><p></p><p><br>A few weeks ago, while consulting skimming threat intel sources Jscrambler researchers stumbled across a new skimming campaign that, at first glance, stood out because of the JavaScript obfuscation it exhibits. Some people raised the question if this was a new obfuscation technique, probably because the code is using weird accented characters.<br><br>As part of a company that makes a <a target="_blank" href="/reports-and-e-books/javascript-obfuscation-e-book">JavaScript obfuscation tool</a>, the team could tell immediately that it is not. The obfuscation author just used unusual Unicode characters for variables and function names. But that has been done before and it’s hardly an obstacle.<br><br>The Jscrambler researchers went over the obfuscation, retrieved the details of the attack, and affected victims and we even saw a strange interaction between different skimming actors chatting through comments in the code. Read everything below.</p><p><br><br><img src="/images/blog-images/image-1.png" alt="source-code-example"></p><p><br><br><br></p><h2>Mongolian Skimmer</h2><h3><br>Obfuscation</h3><p></p><p><br>Jscrambler researchers learned about this skimming campaign from Sansec threat intel. No details were available on it, but the team did a full analysis. At first glance, the thing that stood out was the script’s obfuscation, which seemed a bit bizarre because of all the accented characters. Some people online were saying it could be a new obfuscation technique. The heavy use of Unicode characters, many of them invisible, does make the code very hard to read for humans.<br> </p><p>But the minute Jscrambler researchers started looking at the obfuscation, they realized immediately that it wasn’t a new fancy technique — it was simply leveraging a well-known JavaScript language capability: the <a target="_blank" href="https://tc39.es/ecma262/#sec-identifier-names"><u>ability to use any Unicode character in identifiers</u></a> (e.g. variable names), as per the ECMAScript Standard.</p><p><br></p><p>The added obfuscation does introduce confusion (we call this obfuscation potency) but it does not add any resiliency (how hard it is to reverse engineer, using manual or automated methods). A simple method to remove these characters is to pass the code through a JS transformer that renames all identifiers to simple names (e.g. ‘a’, ‘b’, ‘c’, …). But that’s only useful if a human needs to look at it. A fully automated JS obfuscation reverse engineering engine will not mind what unicode characters are being used. The tooling has been available and fully automated at Jscrambler for years. It’s one of the things Jscrambler researchers use to find skimmers in our products and research.<br></p><p></p><p><br></p><p>Using Jscrambler&#039;s own Code Integrity product, the team quickly renamed all the identifiers to much shorter and more readable ones.<img src="/images/blog-images/image-2.png" alt="mongolian-skimmer-obfuscation-example">After renaming the variables the code resulted in a bunch of string concatenations, including a few String.fromCharCode(...) statements, which contain a hidden skimmer, to be passed to an eval function call.</p><p><br><img src="/images/blog-images/image-3-1728467505.png" alt="jscrambler-code-integrity"><br></p><p>The eval-ed string reveals a script that was also obfuscated using a common obfuscation tool. Once the eval runs, it loads a skimmer designed to steal sensitive information.</p><p><br></p><p><img src="/images/blog-images/image-4.png" alt="steal-sensitive-information-example"></p><p><br></p><p><br>Once Jscrambler researchers reversed the JS, they quickly confirmed that this was a pretty standard skimmer.</p><p></p><p><img src="/images/blog-images/image-5-1728467352.png" alt="standard-skimmer-revelation"></p><p></p><p></p><h4>Naming the Skimmer</h4><p></p><p><br>When they analyzed the first variable assigned in the script, its initial character was represented by the hex value “E1A0A5”, which corresponds to a Unicode character named &quot;Mongolian Letter OE&quot;. Such an unusual character being used by an identifier inspired the team to name the skimmer the <strong>Mongolian Skimmer</strong>.</p><p><br></p><p><img src="/images/blog-images/image-6.png" alt="Mongolian-Letter-OE"></p><p></p><p></p><p><br></p><p></p><h4>Skimmer Analysis</h4><p></p><p><br>The skimmer follows a typical skimming structure Jscrambler researchers have seen countless times in the past:</p><p><br></p><ul><li><p><strong>DOM Monitoring for Changes</strong><br>The script continuously monitors changes in specific DOM elements like &lt;input&gt;, &lt;select&gt;, and &lt;textarea&gt; fields. These are prime targets for stealing user data, such as personal information or payment details. It extracts the name or ID attributes and fetches the value of these elements when users interact with them.</p></li><li><p><a href="/learning-hub/data-exfiltration"><strong>Data Exfiltration</strong></a><br>The skimmer checks the page URL for keywords such as &quot;checkout&quot; or &quot;admin&quot; to identify pages handling sensitive data. Once identified, it encodes the data using window.btoa() for base64 encoding and sends it to a remote server via a tracking pixel (constructed with new Image()) - a classic skimming technique.</p></li><li><p><strong>Developer Tools Detection</strong><br>The script actively detects whether the browser developer tools (DevTools) are open. If detected, the skimmer disables certain functions—likely an attempt to evade detection during debugging or dynamic analysis phases.</p></li><li><p><strong>Data Collection Before Page Unload</strong><br>The skimmer ensures no data is lost by capturing any last-minute data entries before the page is closed or refreshed using the <em>beforeunload</em> event.</p></li><li><p><strong>Cross-Browser Compatibility</strong><br>The skimmer uses well-known techniques to ensure compatibility across different browsers by employing both modern and legacy event-handling techniques. This guarantees it can target a wide range of users, regardless of their browser version.</p></li><li><p><a href="/learning-hub/anti-debugging"><strong>Anti-Debugging</strong></a><br>The skimmer employs an interesting anti-debugging technique by checking its formatting through string conversion and regex tests to detect tampering. This is a standard trick used by skimmers to prevent debugging by detecting code beautification or any other type of formatting changes.</p></li></ul><p></p><p></p><p></p><h4><br>Loader Analysis</h4><p></p><p><br>Most of the Mongolian skimmer instances the team found used an inline script on the homepage, which asynchronously fetched the skimmer script from an external source. However, they did encounter one peculiar instance where a different loader was being used that is worth highlighting.</p><p><br></p><h4>Unusual Loader Variant </h4><p><br></p><p><img src="/images/blog-images/image-7.png" alt="unsual-loader-variant"></p><p></p><p><br>This loader script appears to have been added to the loader of a Magento plugin named “Magento 2 Google Tag Manager” built by MageFan. It dynamically loads Google Tag Manager and the skimmer script, but only if it detects user interactions, such as:</p><p><br></p><ul><li><p>Scroll</p></li><li><p>Mouse movement</p></li><li><p>Touchstart</p></li></ul><p></p><p><br>This approach may not only serve as an effective anti-bot measure but also as a way to make extra sure that the script loading is not visibly impacting page load performance.</p><p><br></p><p></p><h3>Cyber Chit-Chat Using the Victims&#039; Code</h3><p><br></p><p>In one compromised Magento site, Jscrambler researchers found not just one, but <strong>two</strong> skimmer instances: the familiar &quot;’Google’+’Analytics’+’Object’&quot; loader which we talked about some time ago <a href="https://jscrambler.com/blog/defcon-skimming-a-new-batch-of-web-skimming-attacks"><u>here</u></a>, and the Mongolian skimmer. </p><p><img src="/images/blog-images/image-8.png" alt="cyber-chit-chat-using-the-victim-code">It’s not uncommon for compromised sites to be targeted by multiple malicious actors exploiting the same vulnerabilities. What made this case special was that Jscrambler researchers found the two threat actors casually chatting by leaving code comments in the source code to each other.
