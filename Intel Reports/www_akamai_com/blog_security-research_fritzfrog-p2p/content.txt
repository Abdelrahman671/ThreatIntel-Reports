    <h2>Executive summary<br />
</h2>
<ul>
<li><p>FritzFrog is a peer-to-peer <a href="/glossary/what-is-a-botnet" target="_self">botnet</a> campaign <a href="https://www.guardicore.com/labs/fritzfrog-a-new-generation-of-peer-to-peer-botnets/" target="_blank">uncovered</a> by Guardicore Labs (now Akamai Threat Labs) in August 2020</p>
</li>
<li><p>The decentralized botnet targets any device that exposes an SSH server — cloud instances, data center servers, routers, etc. — and is capable of running any malicious payload on infected nodes</p>
</li>
<li><p>Its peer-to-peer architecture and proprietary code demonstrate a high level of sophistication</p>
</li>
<li><p>It has recently resurfaced and displayed 10x growth in its infection rate within a month, compromising servers in the healthcare, education, and government sectors</p>
</li>
<li><p>1,500 distinct hosts have been infected since the reappearance of the botnet, most of which are in China</p>
</li>
<li><p>The Golang malware being spread adds new functionality to the botnet, including the usage of a proxy network and the targeting of WordPress servers</p>
</li>
<li><p>The recent round of attacks provides more evidence as to the origins of FritzFrog, indicating a possible link to an actor operating in China, or one masquerading as Chinese</p>
</li>
<li><p>Akamai Threat Labs has updated the FritzFrog detection tool to handle the newest version of the malware</p>
</li>
</ul>

    <h2>FritzFrog v1 — a quick recap<br />
</h2>
<p>FritzFrog is a peer-to-peer botnet, which means its command and control server is not limited to a single, centralized machine, but rather can be done from every machine in its distributed network. In other words, every host running the malware process becomes part of the network, and is capable of sending, receiving, and executing the commands to control machines in the network.</p>
<p>FritzFrog propagates over SSH. Once it finds a server’s credentials using a simple (yet aggressive) brute force technique, it establishes an SSH session with the new victim and drops the malware executable on the host. The malware then starts listening and waiting for commands. These commands include exchanging targets, sharing the details of compromised machines, and transferring files, as well as running scripts and binary payloads. The full list of commands is available in our <a href="https://www.guardicore.com/labs/fritzfrog-a-new-generation-of-peer-to-peer-botnets/" target="_blank">previous blog post</a>.</p>
<p>We consider FritzFrog to be a “next generation” botnet due to a combination of properties that make it unique in the threat landscape:</p>
<ul>
<li><p>Constantly updating — Databases of targets and breached machines are exchanged seamlessly</p>
</li>
<li><p>Aggressive — Brute force is based on an extensive dictionary; by comparison, DDG, another recently discovered P2P botnet, used only the username “root”</p>
</li>
<li><p>Efficient — Targets are evenly distributed among nodes</p>
</li>
<li><p>Proprietary — The P2P protocol is completely proprietary, relying on no known P2P protocols such as μTP</p>
</li>
</ul>
<h2>FritzFrog v2 — the resurface</h2>
<p>Right after publishing the details about FritzFrog in August 2020, the Guardicore Labs team (now Akamai Threat Labs) noticed a drop in the number of attack incidents. However, in early December 2021, we started seeing an uptick in attacks toward our global sensor network.</p>
<p>A FritzFrog attack starts with an SSH brute force, and continues with a file being dropped and executed. This file immediately starts listening on port 1234 and scanning thousands of internet IP addresses over ports 22 and 2222.</p>
<p>One difference between the old FritzFrog attacks and new attacks is the name of the malicious process. In the first round of attacks, the malicious process was named ifconfig or nginx; this time the FritzFrog operators chose the name apache2.</p>
<p>We started monitoring the new variant and saw a surprising rise in the number of FritzFrog attacks, peaking at 500 incidents per day.</p>
<p> </p>

<h2>Victim analysis</h2>
<p>As part of our investigation into FritzFrog’s first campaign, we developed a tool called Frogger. Frogger allows us to gather information on infected hosts, including their uptime, hashrate, peers, and hasrate, if a cryptominer is running. This program takes an infected node’s IP address and queries the host over an encrypted communication channel in order to receive information on its status. By doing this, we can find out more about the node, and other nodes it is connected to. This process leverages the botnet's infrastructure to do so.</p>
<p>Our victim analysis was based on two sources: the IP addresses of the machines that attacked our sensors, and information obtained by Frogger. Namely, we started with the list of attacker IPs seen by our sensors, and then expanded that list by querying each one for its peers, using Frogger, recursively.</p>
<p>The graphs below show the daily number of IP addresses that attacked our sensors and the difference in the number of attackers between consecutive days. The rise in the number of attacking nodes — victim machines running the malware — is disturbing.</p>

<h2>New malware capabilities</h2>
<p>The FritzFrog binary is written in Golang and can be compiled to run on many different architectures. It is packed using UPX and is usually executed under one of four process names: ifconfig, nginx, apache2, or php-fpm. We have provided a thorough analysis of the malware in our <a href="https://www.guardicore.com/labs/fritzfrog-a-new-generation-of-peer-to-peer-botnets/" target="_blank">previous publication</a>, and will therefore focus on the new samples and the additions they make.</p>
<p>FritzFrog is updated daily — sometimes multiple times a day. Most new releases involve bug fixes, but some versions add new capabilities to the malware.</p>
<h2>Preparation for WordPress targeting</h2>
<p>FritzFrog released a new version that implements the infrastructure for tracking WordPress servers. It contains functions responsible for adding and removing entries from lists titled Wordpress and WordpressTargetsTTL. The snippet of disassembled code below shows the implementation of a new P2P command, put wordpress, which adds a new entry to the WordPress target list.     <p>The malware does not contain any module capable of cracking or identifying WordPress targets. Based on this, we assume the code is a preparation for new versions, which will be capable of compromising those targets and using them for purposes other than mining, such as information leaks, ransomware, etc.</p>
<h2>Tor proxy chain</h2>
<p>FritzFrog can proxy outgoing SSH connections using the Tor proxy chain by setting the proxy for SSH connections to local port 9050. The Tor proxy chain is a network of nodes that provides its users with better privacy and masking by creating an encapsulation-based path from source to destination; each node is only aware of its direct neighbor nodes. </p>
<p>By proxying requests to local port 9050, FritzFrog uses the Tor proxy chain to connect to owned SSH devices. An owned device would see the incoming request as coming from the last node in the proxy chain. This can be used to conceal the address of current infected nodes. As of today, while the functionality exists, we have not yet seen this capability being used by the malware.</p>
<h2>SCP</h2>
<p>FritzFrog now uses SCP to copy itself to a remote compromised server. This is different from the first version, where the malware executable was dropped on a new victim using the cat command over an established SSH session. The new implementation uses a <a href="https://github.com/povsister/scp" target="_blank">public SCP library written in Golang</a> in GitHub. We could not determine any meaningful advantage for one method over the other. It is, however, notable that the writers of the SCP library are located in China. </p>
<h2>Blocklist</h2>
<p>FritzFrog’s early version implemented a blocklist, to exclude specific machines from being breached by the cracker (brute force) module. While a special P2P command, putblentry, allows for dynamic insertion of entries to this list, the new versions hard-code several entries in advance.</p>
<p>Some of these entries specify a Unix name, and some of them an IP address — but never both.</p>

<h2>Origins and attribution<br />
</h2>
<p>Recent changes and additions to this campaign have allowed us to examine the possible origins of this malware. While we cannot be certain of its true origin, we believe that sharing this information can be valuable. </p>
<p>The first piece of evidence comes from one of the newly added libraries compiled into the FritzFrog malware named scp, which implements the Secure Copy Protocol for file transfers over SSH. The library is written in Go and its code is shared on GitHub, in a <a href="https://github.com/povsister/scp" target="_blank">repository</a> by a user located in Shanghai. One fork exists in this repository, which was done by a second individual <a href="https://github.com/chulinx" target="_blank">in Shanghai</a>.</p>
<p>Another piece of evidence linking to China comes from FritzFrog’s cryptomining activity. Our research team has managed to find new wallet addresses as well as new mining pools used in the cryptomining process. One of the newly observed wallets addresses (given below) was also used as part of the <a href="https://blog.netlab.360.com/the_death_of_mozi_cn/" target="_blank">Mozi</a> botnet campaign, whose authors were recently arrested in China.</p>
<h3>FritzFrog Monero wallet address connected to Mozi</h3>
<pre>
47BD6QNfkWf8ZMQSdqp2tY1AdG8ofsEPf4mcDp1YB4AX32hUjoLjuDaNrYzXk7cQcoPBzAuQrmQTgNgpo6XPqSBLCnfsjaV
</pre>
<p>These points of evidence, while not damning, lead us to believe a possible link exists to an actor operating in China, or an actor masquerading as Chinese.Lastly, monitoring attack data has demonstrated a high level of activity in and around China throughout the lifetime of the campaign. As of now, approximately 37% of infected nodes seem to be located in China. </p>
<h2>Prevention and mitigation</h2>
<p> </p>
<h3>FritzFrog detection tool</h3>
<p>Akamai provides a <a href="https://github.com/guardicore/labs_campaigns/blob/master/FritzFrog/detect_fritzfrog.sh" target="_blank">FritzFrog detection script</a> to run on SSH servers. It looks for the following FritzFrog indicators:</p>
<ul>
<li><p>Running processes named <i>nginx, ifconfig, php-fpm, apache2, or libexec</i>, whose executable file no longer exists on the file system (as seen below)</p>
</li>
<li><p>Listening port 1234</p>
</li>
</ul>

<h2>Recommendations</h2>
<ul>
<li><p>Always Keep systems updated and patched</p>
</li>
<li><p>Implement passwordless login using strong key management and rotation system</p>
</li>
<li><p>Enable system login auditing with alerting</p>
</li>
<li><p>Monitor the authorized_hosts file on Linux</p>
</li>
<li><p>Configure explicit allow list of SSH login</p>
</li>
<li><p>Disable root SSH access</p>
</li>
<li><p>Enable Akamai cloud-based DNS protection with threats and unrelated business applications such as coin mining set to block  </p>
</li>
</ul>
