	<p class="article-details__description">In this blog entry, we will discuss how the Jenkins Script Console can be weaponized by attackers for cryptomining activity if not configured properly.</p>

		<div>
			<p><span class="body-subhead-title">Summary</span></p>
<ul>
<li><span class="rte-red-bullet" style="font-weight: normal;">Attackers can leverage the Jenkins Script Console to execute malicious Groovy scripts, leading to cybercriminal activities such as the deployment of cryptocurrency miners.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Misconfigurations such as improperly setup authentication mechanisms expose the /script endpoint to attackers. This can lead to remote code execution (RCE) and misuse by malicious actors.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Attackers can exploit <a name="_Int_lU5X9Hr3" id="_Int_lU5X9Hr3"></a>Jenkins vulnerabilities to run scripts that can download and execute a miner binary and maintain persistence using <i>cron</i> jobs and <i>systemd</i>-run utilities.</span></li>
</ul>
<p><a href="https://www.jenkins.io/" target="_blank">Jenkins</a> is an open-source solution that enables continuous integration and continuous delivery (CI/CD), allowing for the automation of the various stages of software development such as the test, build, and deployment phases. While it offers many benefits to users, it can also be used as an attack vector by malicious actors that can exploit misconfigured servers and unpatched Jenkins versions to deploy cryptocurrency miners and backdoors, as well as to gather sensitive information. In this blog entry, we will discuss how the <a href="https://www.jenkins.io/doc/book/managing/script-console/" target="_blank">Jenkins Script Console</a> can be weaponized by attackers for cryptomining activity if not configured properly.</p>
<h1><span class="body-subhead-title">Technical details</span></h1>

		</div>

    <div class="richText">
	
    


	
		<div>
			<p>The Script Console in Jenkins is a tool that allows administrators and authorized users to execute , Groovy scripts directly on the Jenkins server. This console runs scripts with Jenkins' access privileges, giving it elevated permissions. Jenkins supports scripting via the <a href="https://www.jenkins.io/doc/pipeline/steps/groovy/" target="_blank">Groovy language</a>.</p>
<p>By default, Jenkins does not enable the script console for anonymous users. Rather, the console, which allows the execution of Groovy scripts (and thus potentially dangerous commands), is restricted to authenticated users with administrative permissions. However, it is possible for a Jenkins instance to be misconfigured — for example, when certain authentication mechanisms are improperly set up — potentially leading to vulnerable instances. If the &quot;<i>Sign Up</i>&quot; or &quot;<i>Registration</i>&quot; option is enabled, gaining access to Jenkins and the <i>/script</i> endpoint can also result in RCE. Therefore, proper protection of this interface is essential. Unfortunately, some Jenkins deployments that are exposed to the internet are misconfigured, leaving them vulnerable to misuse.</p>
<p>A quick search on Shodan reveals the number of Jenkins servers exposed to the public. Note that while not all exposed Jenkins servers might be susceptible to misuse, they can still serve as pivot points for attackers.</p>

			<p>The function takes an argument and creates a temporary shell script named <i>vinars</i> in the specified directory. The script simply prints &quot;<i>ginerd</i>&quot; to the console. It makes this small script executable and captures the exit status with <a href="https://tldp.org/LDP/abs/html/exit-status.html#EXSREF" target="_blank"><i>$?</i></a>, a special variable that contains a number indicating the result of the last executed command. This function is further used to execute a simple script in a dynamically specified location.</p>
<p>The script ensures it has enough system resources to perform the mining effectively (similar to other cryptocurrency miners). To do this, the script checks for processes that consume more than 90% of the CPU’s resources, then proceeds to kills these processes. Furthermore, it will terminate all stopped processes.</p>

After decryption, <i>tar</i> is used to decompress the file, after which the original “cex” file is removed and the executable permission is assigned to the file which is the same file validated earlier in this script with the “sha256sum app” command.</p>

			<h1><span class="body-subhead-title">Using cron job to achieve persistence</span></h1>
<p>The code in Figure 13 creates a cron job:</p>
<ol>
<li><span style="font-weight: normal;">It uses the <i>flock </i>system utility to lock the file, and the<i> /var/tmp/verl.lock</i> file to ensure that only one instance is running at a time</span></li>
<li><span style="font-weight: normal;">It uses<i> </i><a href="https://learnbyexample.github.io/learn_gnuawk/dealing-with-duplicates.html" target="_blank"><i>awk '!a[$0]++'</i></a> to remove all the duplicate entries from the cron job. This is another step to ensure only one instance of cron job executes at a time.</span></li>
<li><span style="font-weight: normal;">Finally, the output from <i>awk </i>is piped into <i>crontab -</i>, which updates the user's <i>crontab</i> with the filtered and unique cron job entries.<br />
</span><br />
</li>
</ol>
<p>The script stores the whole cron job entry in a variable named <i>cronny</i> with all the parameters to run the miner, including the pool and wallet addresses and adds this to the <i>crontab</i>.</p>

This allows applications and services running within the user session to communicate via D-Bus.</p>
<p>The attacker then uses <i>systemd-run</i> to schedule the execution of a cryptocurrency mining application at the beginning of every hour. The <i>flock</i> utility ensures that only one instance runs at a time using a lock.</p>
<p>Even if these methods fail to execute the miner in a desired manner, the attacker has another mechanism to use for execution.</p>

			<p>The script part performs the following actions:</p>
<ol>
<li><span style="font-weight: normal;">Writes the string to a file named <i style="">strptr</i> that sets the rig ID for the mining operation.</span></li>
<li><span style="font-weight: normal;">The “<i>trap '' SIG</i>” command sets a trap to ignore the SIG signal. The purpose of this line could be to prevent the script from being terminated prematurely by certain signals.</span></li>
<li><span style="font-weight: normal;">The <i>“( sleep 40; kill -9 $PPID ) &amp;</i>” command starts a subshell in the background (&amp;). The subshell sleeps for 40 seconds and then sends a SIGKILL signal to the parent process ($PPID), which is likely the script itself (possibly a way to ensure that the script terminates after a certain period as a cleanup measure).</span></li>
<li><span style="font-weight: normal;">Finally, the <i>exec </i>command executes the miner.</span></li>
</ol>
<h1><span class="body-subhead-title">Mitigations</span></h1>
<p>To protect Jenkins servers from the attacks that we discussed here, we recommend the following best practices:</p>
<ul>
<li><span class="rte-red-bullet"><span style="font-weight: normal;">Use the <a href="https://plugins.jenkins.io/script-security/" target="_blank">Script Approval</a> feature provided by Jenkins.</span></span></li>
<li><span class="rte-red-bullet"><span style="font-weight: normal;">Apply proper authentication and authorization policies to access the web console. Note that Jenkins offers specific <a href="https://www.jenkins.io/doc/book/security/access-control/">guidelines</a><u> </u>on Access Control.</span></span></li>
<li><span class="rte-red-bullet"><span style="font-weight: normal;">Use the <a href="https://plugins.jenkins.io/audit-trail/" target="_blank">Audit Logging</a> feature provided by Jenkins.</span></span></li>
<li><span class="rte-red-bullet"><span style="font-weight: normal;">Ensure that Jenkins servers are not accessible from the internet.</span></span></li>
</ul>
<h1><span class="body-subhead-title">Trend solutions</span></h1>
<p><b>Vision One Threat Hunting Query</b></p>
<p>The following text lists potentially useful queries for threat hunting within Vision One:</p>
<ul>
<li><span class="rte-red-bullet"><span style="font-weight: normal;">eventSubId: 2 AND processCmd:* exec * --rig-id *<br />
 </span></span></li>
<li><span class="rte-red-bullet"><span style="font-weight: normal;">eventSubId: 2 AND processCmd:cron AND objectCmd:* exec * --rig-id *</span></span></li>
</ul>
<h1><span class="body-subhead-title">Conclusion</span></h1>
<p>The robust capabilities of the Jenkins software make it an essential tool for modern software development. However, these capabilities can also bring significant risks when misconfigured or left unpatched (also seen in our <a href="/en_us/research/24/c/cve-2024-23897.html">previous Jenkins blog entry</a>). In this blog entry, we discuss how the Jenkins Script Console can be weaponized by attackers to deploy malicious scripts for activities such as cryptocurrency mining.</p>
<p>Ensuring proper configuration, robust authentication and authorization, regular audits, and restricting the internet accessibility of Jenkins servers are all vital steps for organizations to significantly reduce the likelihood of their Jenkins instances becoming attack vectors and safeguard their development environments against exploitation.</p>
