<br />
<br />

<br />
<br />
After I received the files two things became apparent:<br />
<br />
<ul>
<li>the webserver (and thus the website) was infected with C99shell;</li>
<li>the webserver was infected with other PHP backdoors.</li>
</ul>
<br />
<span id="analysis"></span><br />
<div id="analysis">
</div>
<span style="font-size: large;"><b>Analysis</b></span><br />
<br />
<b>PHP/c99shell </b>or simply c99shell should be well known by now - it is a PHP backdoor that provides a lot of functionality, for example:<br />
<br />
<br />
<ul>
<li>run shell commands;</li>
<li>download/upload files from and to the server (FTP functionality);</li>
<li>full access to all files on the hard disk;</li>
<li>self-delete functionality.</li>
<li>...</li>
</ul>
<br />
<br />
In short, it can pretty much do everything you want, which results in end-users getting malware onto their systems and/or data getting stolen and/or personal information compromised.<br />
<br />
There's an excellent blog post over at <b>Malwaremustdie</b> in regards to C99shell, you can read it here:<br />
<a href="http://blog.malwaremustdie.org/2012/10/how-far-phpc99shell-malware-can-go-from.html" target="_blank">How EVIL the PHP/C99Shell can be? From SQL Dumper, Hacktools, to Trojan Distributor Future?</a><br />

After some manual decoding, we turn up with the following interesting line:<br />
<blockquote class="tr_bq">
getenv(HTTP_X_UP_CALLING_LINE_ID);</blockquote>
<br />
Another example:<br />
<blockquote class="tr_bq">
getenv(HTTP_X_NOKIA_ALIAS);</blockquote>
<br />
The "x-headers" HTTP_X_UP_CALLING_LINE_ID and HTTP_X_NOKIA_ALIAS are actually part of WML, the <a href="http://en.wikipedia.org/wiki/Wireless_Markup_Language" target="_blank">Wireless Markup Language.</a><br />
<br />
Thus, this PHP backdoor seems specifically designed to target mobile users. I've put a copy of the script in screenshot above on Pastebin as well:<br />
<a href="http://pastebin.com/a8VFXq9i" target="_blank">Unknown PHP backdoor</a><br />
<br />
Darryl from <a href="https://twitter.com/kahusecurity" target="_blank">Kahu Security</a> has written an excellent post on how to manually decode this kind of PHP obfuscation: <a href="http://www.kahusecurity.com/2015/deobfuscating-a-wicked-looking-script/" target="_blank">Deobfuscating a Wicked-Looking Script</a><link></link><br />
<br />
If you have any information on what kind of PHP backdoor this might be (if not generic), feel free to let me know.<br />

<span id="disinfection"></span><br />
<div id="disinfection">
</div>
<span style="font-size: large;"><b>Disinfection</b></span>
<br />
<br />
What if your website's already been hacked and serving up malware to the unknowing visitor? Best practice is to simply take your website offline and restore from an earlier back-up. (don't forget to verify if your back-up isn't infected as well!)<br />
<br />
If that's not a possibility for whatever reason, you'll first need to find where any malicious code was injected (or created) on your website, or how it was infected in the first place.<br />
<br />
An easy way would be to simply check all recently changed files on your web server. However, those dates can be altered. So what's a better alternative? You can comb over the files one by one, or you can use an online tool to check your website.<br />
<br />
A short overview:<br />
<br />
<a href="http://sitecheck.sucuri.net/">http://sitecheck.sucuri.net/</a><br />
You can use Sucuri's SiteCheck to quickly spot if they detect any malware, see if you're blacklisted and, the most useful part in this case is to check whether or not you have any outdated plugin or CMS running - as well as a list of links.<br />
<br />
<a href="http://aw-snap.info/file-viewer/">http://aw-snap.info/file-viewer/</a><br />
Use Redleg's file viewer to easily see if any malicious iframes have been injected - you can even choose which <i>Referrer</i> and<i> User Agent</i> should be used (some malware requires you to visit the site via a specific Referrer or User Agent).<br />
<br />
<a href="http://www.rexswain.com/httpview.html">http://www.rexswain.com/httpview.html</a><br />
Useful additional tool to Redleg's file viewer. Allows you to only fetch headers of a website, or fetch both header and content.<br />
<br />
<a href="http://jsunpack.jeek.org/">http://jsunpack.jeek.org/</a><br />
Excellent tool in case any malicious Javascript (iframe) is injected into any of your web server files. Less intuitive, but provides a great overview.<br />
<br />
<a href="http://urlquery.net/">http://urlquery.net/</a><br />
Excellent tool and more graphical as opposed to JSunpack - especially useful is to see if any <a href="http://en.wikipedia.org/wiki/Intrusion_detection_system" target="_blank">IDS</a> was triggered as well as JavaScript and HTTP Transactions.<br />
<br />
<a href="https://www.virustotal.com/">https://www.virustotal.com/</a><br />
As usual, VirusTotal is a great resource as well - it can pinpoint which Antivirus (if any) is triggering an alert related to your website.<br />
<br />
<a href="https://hackertarget.com/wordpress-security-scan/">https://hackertarget.com/wordpress-security-scan/</a><br />
Online WordPress Security Scanner to test vulnerabilities of a WordPress installation. Checks include application security, WordPress plugins, hosting environment and web server.<br />
<br />
<a href="https://github.com/nbs-system/php-malware-finder">https://github.com/nbs-system/php-malware-finder</a><br />
NBS System's PHP Malware Finder does its very best to detect obfuscated/dodgy code as well as files using PHP functions often used in malwares/webshells.<br />
<br />
<a href="https://github.com/sullo/nikto">https://github.com/sullo/nikto</a><br />
Nikto web server scanner.<br />
<br />
If nothing is found using any of these tools, but you are still receiving reports from either blacklists (eg. Google) or users, you'll have to manually go over all your files to see if any code was attached.<br />
<br />
If you're hosting a web server yourself, you obviously know where you've installed it, so be sure to check in there. If you're not sure where it's installed, may want to look in any of these default locations, if they exist:<br />
<br />
Linux:<br />
<ul>
<li><b>/var/www/</b></li>
<li><b>/var/www/html</b></li>
<li><b>var/lib/tomcat7/webapps</b></li>
</ul>
<div>
Windows:</div>
<ul>
<li><b>C:\inetpub</b></li>
<li><b>C:\inetpub\wwwroot\</b></li>
<li><b>...</b></li>
</ul>
<br />
<br />
&nbsp;Another method (and obviously not foolproof) is to copy over all your files to a Windows system and scan them with an antivirus. An example of such antivirus, which works on both Linux and Windows, is <a href="https://www.clamav.net/" target="_blank">ClamAV</a>. I think you're starting to realize why back-ups are important.<br />
<br />
If you had any outdated plugins running, chances are very high the backdoor or script was created/added in that specific directory. For example for WordPress this is typically:<br />
<b>/www/wp-content/plugins/</b><br />
<br />
You can also install a plugin for your CMS which can scan your web server for any infected files. (Which is ironic, but might still do the trick should you not be able to find anything manually.)<br />
<br />
Last but not least: check your access logs! See any unauthorized (FTP) logins for example? Take a look in any of these locations:<br />
<br />
<ul>
<li><b>/var/log/httpd&nbsp;</b></li>
<li><b>var/log/nginx &nbsp;</b></li>
<li><b>/var/log/apache</b></li>
<li><b><b>/var/log/apache2</b></b></li>
</ul>
<br />
<b><b><br /></b></b>
You may also want to take a peek in:<br />
<b>/var/log</b><br />
<br />
Contact your hosting provider - they might be able to provide you with assistance.<br />
<br />
If you're still stuck, feel free to shoot me an email or contact me on <a href="https://twitter.com/bartblaze" target="_blank">Twitter</a>. Otherwise, contact one of X companies which can help you assist in clean-up.<br />
<br />
<u>Don't forget</u>: after clean-up, reset <b>all</b> your passwords (and don't use the same for everything) and follow the prevention tips above, or you'll simply get infected again.<br />
Additionally, always install relevant security patches or updates for your operating system if you are hosting the web server yourself.<br />

<br />
<span style="font-size: large;"><b>Prevention</b></span><br />
<br />
This shouldn't be repeated normally, but I will again just for good measure:<br />
<br />
<ul>
<li>Create&nbsp;<b>back-ups</b>&nbsp;regularly! Yes, even for your website.</li>
<li>Keep your CMS up-to-date; whether you use WordPress, Joomla, Drupal, ...&nbsp;</li>
<li>Keep your installed plugins up-to-date. Remove any unnecessary plugins.</li>
<li>Use strong passwords for your FTP account(s), as well as for your CMS/admin panel login.</li>
<li>Use appropriate file permissions - meaning don't use 777 everywhere. (seriously, don't)</li>
<li>Depending on how you manage your website - keep your operating system up-to-date and, if applicable, install and update antivirus software.</li>
<li>Consider using a tool like&nbsp;<a href="http://www.splunk.com/en_us/products/splunk-light.html" target="_blank">Splunk</a>&nbsp;to monitor your access logs.&nbsp;</li>
<li>Consider installing a security plugin. For WordPress, you have a plugin called&nbsp;<a href="https://wordpress.org/plugins/all-in-one-wp-security-and-firewall/" target="_blank">All In One WordPress Security</a>&nbsp;which has a ton of options to better secure your website.Don't forget to keep this one up-to-date as well.</li>
</ul>
<br />
More (extended) tips can be found over at StopBadware:<br />
<a href="https://www.stopbadware.org/prevent-badware-basics" target="_blank">Preventing badware: Basics</a><br />
<br />
There are also guides available on how to harden your specific CMS installation, for example:<br />
<br />
WordPress:&nbsp;<a href="http://codex.wordpress.org/Hardening_WordPress" target="_blank">Hardening WordPress</a><br />
Joomla:&nbsp;<a href="https://docs.joomla.org/Security_Checklist/Joomla!_Setup" target="_blank">Security Checklist/Joomla! Setup</a><br />
Drupal:&nbsp;<a href="https://www.drupal.org/writing-secure-code" target="_blank">Writing secure code</a><br />

<span style="font-size: large;"><b>Conclusion</b></span><br />
<br />
C99shell is obviously not dead and neither are other PHP backdoors - or any other malware for that matter. Securing your website is not only beneficial for you, but also for your customers and other visitors. This blog post should have provided you with the essentials on securing your website and cleaning it up should it ever be infected <br />
<br />
<u>Repeating</u>: best practice is to take your website offline and restore from a back-up.<br />
