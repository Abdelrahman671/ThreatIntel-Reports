																							<h2>New observations on BlackEnergy2 APT activity</h2>											
											<div class="js-reading-content">
												<div class="c-wysiwyg">
													<p>The BlackEnergy malware is crimeware turned APT tool and is used in significant geopolitical operations lightly documented over the past year. An even more interesting part of the BlackEnergy story is the relatively unknown custom plugin capabilities to attack ARM and MIPS platforms, scripts for Cisco network devices, destructive plugins, a certificate stealer and more. Here,we present available data &#8211; it is difficult to collect on this APT.We will also present more details on targets previously unavailable and present related victim profile data.</p>
<p>These attackers are careful to hide and defend their long-term presence within compromised environments. The malware&#8217;s previously undescribed breadth means attackers present new technical challenges in unusual environments, including SCADA networks. Challenges, like mitigating the attackers&#8217; lateral movement across compromised network routers, may take an organization&#8217;s defenders far beyond their standard routine and out of their comfort zone.</p>
<h2 id="brief-history">Brief History</h2>
<p>BlackEnergy2 and BlackEnergy3 are known tools. Initially, cybercriminals used BlackEnergy custom plugins for launching DDoS attacks. There are no indications of how many groups possess this tool. BlackEnergy2 was eventually seen downloading more crimeware plugins &#8211; a custom spam plugin and a banking information stealer custom plugin. Over time, BlackEnergy2 was assumed into the toolset of the BE2/Sandworm actor. While another crimeware group continues to use BlackEnergy to launch DDoS attacks, the BE2 APT appears to have used this tool exclusively throughout 2014 at victim sites and included custom plugins and scripts of their own. To be clear, our name for this actor has been the BE2 APT, while it has been called &#8220;Sandworm Team&#8221; also.</p>
<h2 id="the-plugins-and-config-files">The Plugins and Config Files</h2>
<p>Before evidence of BlackEnergy2 use in targeted attacks was uncovered, we tracked strange activity on one of the BlackEnergy CnC servers in 2013. This strangenesswas related to values listed in newerBlackEnergy configuration files. As described in Dmitry&#8217;s 2010 <a href="https://securelist.com/black-ddos/36309/">Black DDoS&#8217; analysis</a>, a configuration file is downloaded from the server by main.dll on an infected system. The config file provides download instructions for the loader. It also instructs the loader to pass certain commands to the plugins. In this particular case in 2013, the config file included an unknown plugin set, aside from the usual &#8216;ddos&#8217; plugin listing. Displayed below are these new, xml formatted plugin names &#8220;weap_hwi&#8221;, &#8220;ps&#8221;, and &#8220;vsnet&#8221; in a BlackEnergy configuration file download from a c2 server. This new module push must have been among the first for this group, because all of the module versions were listed as &#8220;version 1&#8221;, including the ddos plugin:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071421/1.png" class="magnificImage"><img loading="lazy" decoding="async" width="383" height="506" class="wp-image-67356 aligncenter aligncenter size-full" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071421/1.png" alt="Strange new BE2 config from late 2013 " srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071421/1.png 383w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071421/1-227x300.png 227w" sizes="auto, (max-width: 383px) 100vw, 383px" /></a></p>
<p style="text-align: center"><em>Config downloaded from BE2 server</em></p>
<p style="text-align: left">The &#8216;ps&#8217; plugin turned out to be password stealer. The &#8216;vsnet&#8217; plugin was intended to spread and launch a payload (BlackEnergy2 dropper itself at the moment) in the local network by using PsExec, as well as gaining primary information on the user&#8217;s computer and network.<br />
Most surprising was the &#8216;weap_hwi&#8217; plugin. It was a ddos tool compiled to run on ARM systems:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071415/2.png" class="magnificImage"><img loading="lazy" decoding="async" width="628" height="387" class="wp-image-67358 aligncenter size-full" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071415/2.png" alt="BE2 config for ARM" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071415/2.png 628w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071415/2-300x185.png 300w" sizes="auto, (max-width: 628px) 100vw, 628px" /></a></p>
<p style="text-align: center"><em>Weap_hwi plugin</em></p>
<p style="text-align: left">At first, we didn&#8217;t know whether the ARM plugin was listed intentionally or by mistake, so we proceeded to collect the CnC&#8217;s config files. After pulling multiple config files, we confirmed that this ARM object inclusion was not aone-off mistake. The server definitely delivered config files not only for Windows, but also for the ARM/MIPS platform. Though unusual,the ARM module was delivered by the same server and it processed the same config file.</p>
<h2 id="linux-plugins">Linux plugins</h2>
<p>Over time we were able to collect several plugins as well as the main module for ARM and MIPS architectures. All of these ARM/MIPS object files were compiled from the same source and later pushed out in one config: &#8220;weap_msl&#8221;, &#8220;weap_mps&#8221;, &#8220;nm_hwi&#8221;, &#8220;nm_mps&#8221;, &#8220;weap_hwi&#8221;, and &#8220;nm_msl&#8221;. It&#8217;s interesting that the BE2 developers upgraded the ddos plugin to version 2, along with the nm_hwi, nm_mps, and nm_msl plugins. Theysimultaneously released version 5 of the weap_msl, weap_mps, and weap_hmi plugins. Those assignments were not likely arbitrary, as this group had developed BlackEnergy2 for several years in a professional and organized style:</p>

<p>We are pretty sure that ourlist of BE2 tools is not complete. For example, we have yet toobtain the router access plugin, but we are confident that it exists. Evidence also supports the hypothesis that there is a encryption plugin for victim files (see below).</p>

It suggests that based on a specific USB and BIOS devices, the attackers may upload specificplugins to carry out additional actions. Perhaps destructive, perhaps to further infect devices. We don&#8217;t know yet.</p>

<h2 id="victim-cases">Victim cases</h2>
<p>We gained insight into significant BE2 victim profiles over the summer of 2014. InterestingBE2 incidents are presented here.</p>
<p><strong>Victim #1</strong></p>
<p>The BE2 attackers successfully spearphished an organization with an exploit for which there is no current CVE, and a metasploit module has been available This email message contained a ZIP archive with EXE file inside that did not appear to be an executable. This crafted zip archive exploited a WinRAR flaw that makes files in zip archives appear to have a different name and file extension.</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071340/10.png" class="magnificImage"><img loading="lazy" decoding="async" width="628" height="517" class="wp-image-67371 aligncenter size-full" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071340/10.png" alt="BE2 spearphish example" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071340/10.png 628w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071340/10-300x247.png 300w" sizes="auto, (max-width: 628px) 100vw, 628px" /></a></p>
<p style="text-align: center"><em>BE2 spearphish example</em></p>
<p>The attached exe file turned out to be &#8216;BlackEnergy-like&#8217; malware, which researchers already dubbed &#8216;BlackEnergy3&#8217; &#8211; the gang uses it along with BlackEnergy2. Kaspersky Lab detects &#8216;BlackEnergy3&#8217; malware as Backdoor.Win32.Fonten  naming it after its dropped file &#8220;FONTCACHE.DAT&#8221;</p>
<p>When investigating computers in the company&#8217;s network, only BE2 associated files were found, suggesting BE3 was used as only a first-stage tool on this network. The config files within BE2 contained the settings of the company&#8217;s internal web proxy:</p>
<p><a href="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071336/11.png" class="magnificImage"><img loading="lazy" decoding="async" width="914" height="61" class="wp-image-67372 aligncenter size-full" src="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071336/11.png" alt="BE2 config file contains victim's internal proxy" srcset="https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071336/11.png 914w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071336/11-300x20.png 300w, https://media.kasperskycontenthub.com/wp-content/uploads/sites/43/2014/11/08071336/11-768x51.png 768w" sizes="auto, (max-width: 914px) 100vw, 914px" /></a></p>

<p style="text-align: left">As the APT-specific BE2 now stores the downloaded plugins in encrypted files on the system (not seen in older versions  all plugins were only in-memory), the administrators were able to collect BE2 files from the infected machines. After decrypting these files, we could retrieve plugins launched on infected machines: ps, vsnet, fs, ss, dstr.<br />
By all appearances, the attackers pushed the &#8216;dstr&#8217; module when they understood that they were revealed, and wanted to hide their presence on the machines. Some machines already launched the plugin, lost their data and became unbootable.</p>

<p><span style="font-weight: bold">Victim set #4</span></p>
<p>A set of victims discoveredinstalled Siemens SCADA software in their ICS environment was responsible for downloading and executing BlackEnergy. Starting in March 2014 and ending in July 2014, Siemens &#8220;ccprojectmgr.exe&#8221; downloaded and executed a handful ofdifferent payloads hosted at94.185.85.122/favicon.ico. They are all detected as variants of&#8220;Backdoor.Win32.Blakken&#8221;.</p>
<h2 id="build-ids">Build IDs</h2>
<p>Each config file within BE2 main.dll has a field called build_id which identifies the malware version for the operators. Currently this particular BE/SandWorm gang uses a certain pattern for the build ids containing three hex numbers and three letters, as follows:</p>
<p>0C0703hji</p>
<p>The numbers indicate the date of file creation in the format: Year-Month-Day. Still, the purpose of the letters is unknown, but most likely it indicates the targets.The hex numbers weren&#8217;t used all the time, sometimes we observed decimal numbers:</p>
<p>100914_mg</p>
<p>100929nrT</p>
<p>Most interesting for us was the earliest build id we could find. Currently it is &#8220;OB020Ad0V&#8221;, meaning that the BE2/SandWorm APT started operating as early as the beginning of 2010.</p>
