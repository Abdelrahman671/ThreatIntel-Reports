<div class="cmp-wp-headline" id="headline-en-us-04-19-2024">
    
        <h1>CrowdStrike Launches Free Targeted Log4j Search Tool</h1>
        
        <div class="publish_info">
            <p>December 22, 2021</p>
            <span>|</span>
            <a href="/en-us/blog/author.randy-burton---ian-barton/" title="Posts by Randy Burton - Ian Barton" rel="author">
                Randy Burton - Ian Barton</a>
            <span>|</span>
                
            <a href="/en-us/blog/category.from-the-front-lines/" title="From The Front Lines">From The Front Lines</a>
        </div>
        <div class="post_image">
            <img width="1060" height="698" src="https://assets.crowdstrike.com/is/image/crowdstrikeinc/Blog_1060x698-81-us" class="attachment-post-thumbnail size-post-thumbnail wp-post-image" alt="" decoding="async" fetchpriority="high" srcset="/content/dam/crowdstrike/www/en-us/wp/2024/12/Blog_1060x698-81-us.jpeg 1060w" sizes="(max-width: 1060px) 100vw, 1060px"/>
        </div>
    

    
</div>

    <div style="display: none;">
        
    </div>


    
</div>
<div class="text text--blog-content aem-GridColumn aem-GridColumn--default--12">

  

<div class="cmp-text" id="text-acec2e67eb" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
    
       <span><span style="	font-weight: 400;
">The recently discovered </span><a href="/en-us/blog/log4j2-vulnerability-analysis-and-mitigation-recommendations/"><span style="	font-weight: 400;
">Log4j vulnerability</span></a><span style="	font-weight: 400;
"> has serious potential to expose organizations across the globe to a new wave of cybersecurity risks as threat actors look to exploit this latest vulnerability to execute their malicious payloads using remote code execution (RCE).</span>

<span style="	font-weight: 400;
">An immediate challenge that every organization faces is simply trying to understand exactly where you have applications that are using this very popular Java library — but you are not facing this challenge alone.</span>

<span style="	font-weight: 400;
">The CrowdStrike Services team has been busy developing a community tool that can be used to quickly scan file systems looking for versions of the Log4j code libraries to help organizations understand what they need to patch in order to mitigate their risk.</span><span style="	font-weight: 400;
">​</span>

<span style="	font-weight: 400;
">The free CrowdStrike tool (dubbed the </span><a href="https://github.com/CrowdStrike/CAST"><span style="	font-weight: 400;
">CrowdStrike Archive Scan Tool</span></a><span style="	font-weight: 400;
">, or “CAST”) performs a targeted search by scanning a given set of directories for JAR, WAR, ZIP and EAR files, and then it performs a deeper scan on those file types matching against a known set of checksums for Log4j libraries. We help organizations find any version of the affected Log4j library anywhere on disk, even if it is deeply nested in multiple levels of archive files.</span>

<span style="	font-weight: 400;
">CAST searches for approximately 6,500 SHA256 checksums unique to the known vulnerable releases. It will walk the files or directories scanning inside of ZIP-format archives to find every instance of these. As we developed the tool, we carefully considered the following:</span>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Be mindful of the resource consumption when running a scan to minimize the impact on end-user systems.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Intentionally allow a higher number of false-positive results, leaving the decision in the hands of the system owners whether a given result warrants further investigation.<p> </p></span>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">We may see higher false positives because we identify any trace of vulnerable versions of Log4j, even if the vulnerability has been addressed by removing one or more classes from the deployment.</span></li></ul>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The results should be extremely reliable, as they’re based on cryptographic checksums.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Allow use of the tool with pre-indexed (e.g., “locate”) file systems to avoid scanning and<p> </p> simply pass the paths to known files on the command line.<p> </p></span>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">For example, locate -0 *.jar | xargs -0 ./cast</span></li></ul>
</li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Provide the ability to tune memory usage — for example:<p> </p></span>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">-recursion 0 to disable scanning sub-archives<p> </p></span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">-recursion 1 to scan only 1 sub-archive deep<p> </p><p> </p></span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">-maxmem 1000000 to limit sub-archive scanning to 1MB (compressed)</span></li></ul>
</li></ul>
<span style="	font-weight: 400;
">The tool is intentionally single-threaded as we have to be conscious of resource consumption and allow users or administrators to manage their own resources. One thread will (in our experience) scan a file system quickly enough. One could scan multiple directories simultaneously by executing multiple copies of the tool, but the file system load would likely cause a noticeable user impact.</span>

<span style="	font-weight: 400;
">Staying true to CrowdStrike’s </span><b>cross-platform</b><span style="	font-weight: 400;
"> focus, we developed CAST as a tool that will run on </span><b>Windows, Mac and Linux </b><span style="	font-weight: 400;
">systems, and we are using the tool in CrowdStrike Services engagements to assist our clients who need support to find Log4j instances.<p> </p></span>

<span style="	font-weight: 400;
">The tool is easily deployed by simply downloading the binary to your disk and then executing the binary with the directories or files you want to scan.</span>

<span style="	font-weight: 400;
">For example: “./cast /opt /srv /path/to/java/application”</span>

<a href="/en-us/endpoint-security-products/"><span style="	font-weight: 400;
">CrowdStrike Falcon®</span></a><span style="	font-weight: 400;
"> customers also have the option to deploy and run the tool using the Falcon Real Time Response (RTR) capabilities in the Falcon sensor. A companion PowerShell script &#34;Find-VulnerableLog4J&#34; is included with CAST. This script is designed to be executed on Windows systems via RTR and provide actionable information to systems administrators and incident responders.</span>

<span style="	font-weight: 400;
">Our incident responders know that forensic triage is a continual process of </span><em><b>casting</b></em><span style="	font-weight: 400;
"> increasingly fine nets, and identifying systems that warrant further investigation. Hence, CAST was designed to be a first-cast tool, narrowing investigative scope to a handful of machines (or paths) with known vulnerabilities.</span>

<span style="	font-weight: 400;
">CAST reports back in the form of a JSON file when it locates vulnerable Log4j libraries. Organizations can use this output to get an understanding of where the Log4j libraries exist across their environment so they can prioritize the systems that need to be patched using the latest security updates released by Apache.</span>

<img class="alignnone size-full wp-image-52442" src="/content/dam/crowdstrike/www/en-us/wp/2021/12/Picture1-5.png" alt="" width="2198" height="1107"/>

<span style="	font-weight: 400;
">CrowdStrike investigators use our Humio solution to load and analyze the data, but you can use any visualization solution (such as ELK). You can also work through the data with a programming language or JSON query language of your choice — the events are intended to be portable.</span>

<span style="	font-weight: 400;
">And finally, CrowdStrike recommends that you fully document your Log4j patching process to streamline future patch application repeatability. Since the initial discovery of the Log4j vulnerability, Apache has released three security updates (patches) at the time of this blog. Organizations that patched systems early in the process may need to reapply the latest patches, hence the need to fully document the process.</span>

<span style="	font-weight: 400;
">We hope you find the resources and tools in this blog useful as you cast your own net in your quest to identify Log4j vulnerabilities across your environment. We stand together when it comes to defeating adversaries that try to exploit this vulnerability against us.<p> </p></span>

<span style="	font-weight: 400;
">One team, one fight!</span>
