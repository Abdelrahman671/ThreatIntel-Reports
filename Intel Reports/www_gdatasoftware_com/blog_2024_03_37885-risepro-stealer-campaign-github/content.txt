class="ce-bodytext"><header class=" "><h3 class=" "> Injected RisePro payload </h3></header><p>We identified the payload<sup>[4]</sup> as RisePro stealer version 1.6. The version number stems from the stealer’s logs. We assume that this is the latest version of RisePro, as within the malware authors' publicly accessible Telegram channel, the most recent server updates are referred to as version 1.6.&nbsp;</p><p>The sample resolves its imports dynamically using import hashing with <a href="https://en.wikipedia.org/wiki/Fowler%E2%80%93Noll%E2%80%93Vo_hash_function">Fowler–Noll–Vo hash 1A</a>.</p></div><div class="ce-gallery row small-up-1 medium-up-1 large-up-1"><div class="column column-block"><figure class="image"><img class="image-embed-item " src="/fileadmin/user_upload/Presse/Deutschland/2024/03/GDATA_RisePro_FNVhashing.png" width="652" height="672" loading="lazy" alt="" /><figcaption class="image-caption"> Import hash comparison for LoadLibraryA using FNV-1A </figcaption></figure></div></div></div></div> <div id="c232397" class="fce column large-12 small-12 frame-layout-0 " ><div class="ce-textpic ce-center ce-below"><div class="ce-bodytext"><p>RisePro uses XOR obfuscated stack strings. A previous <a href="https://research.openanalysis.net/risepro/stealer/config/triage/2023/06/15/risepro.html">RisePro report by OALabs</a> states that <a href="https://github.com/JustasMasiulis/xorstr">xorstr library</a> has been used for RisePro's string obfuscation in the past, but it seems the encryption scheme has been changed in the meantime.</p><p>The library xorstr features "heavily vectorized compile time string encryption" and uses vpxor/pxor instructions to perform the XOR operations. A yara rule in the aforementioned OALabs report searches for pxor patterns in the binary. RisePro developers may have reacted to that public rule.</p><p>The new stack string decryption scheme uses one hardcoded decryption function for every string length that the stealer needs. All of these functions use the same underlying decryption algorithm but the length is hardcoded in the decryption loop.</p><p>&nbsp;</p></div><div class="ce-gallery row small-up-1 medium-up-1 large-up-1"><div class="column column-block"><figure class="image"><a href="/fileadmin/user_upload/Presse/Deutschland/2024/03/GDATA_RisePro_decryptstackstring.png" class="fancybox"><img class="image-embed-item " src="/fileadmin/user_upload/Presse/Deutschland/2024/03/GDATA_RisePro_decryptstackstring.png" width="722" height="400" loading="lazy" alt="" /></a><figcaption class="image-caption"> RisePro&#039;s string decryption function for strings of length 12 </figcaption></figure></div></div></div></div> <div id="c232398" class="fce column large-12 small-12 frame-layout-0 " ><div class="ce-textpic ce-center ce-below"><div class="ce-bodytext"><p>We decrypted the strings with a <a href="https://github.com/binref/refinery">Binary Refinery</a> snippet, which prints the decrypted strings and their offsets in a format usable for Python dictionaries.</p><p>&nbsp;</p><pre><code class="language-bash">emit sample | 
	rex "\xC7(\x85|\x45).{4,8}\xC7(\x85|\x45).{50}" [| 
		put o offset | 
		rex "\xC7(\x85....|\x45.)(.{4})" {2} [| 
			nop ]| 
		alu -s "0-101" --inc "B^S" | 
		carve -n 4 printable | 
		resub \\ \\ | 
		resub \" \\\" | 
		cfmt {o} : \"{}\", ]] </code></pre><p>&nbsp;</p><p>The snippet's regular expression matches a few unintended values, but it is not detrimental to set these as comments in the IDA database. We added the the output to the following IDAPython script.</p><p>&nbsp;</p><pre><code class="language-python">import idc 

decrypted_strings_dict = { ... }

for k, v in decrypted_strings_dict.items(): 
    base = 0x400000 
    addr = k + base 
    comment = '"' + v + '"' 
    idc.set_cmt(addr, comment, 0) 
    print("comment", comment, "set at", addr) </code></pre><p>&nbsp;</p><p>Address translation is not necessary here because the file offset and virtual addresses in the .text section are the
