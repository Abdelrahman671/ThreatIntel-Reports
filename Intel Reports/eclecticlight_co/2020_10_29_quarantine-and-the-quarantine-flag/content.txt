window.addEventListener( "message", function( e ) {
	if ( e.origin === "https://r-login.wordpress.com" ) {
		var data = {};
		try {
			data = JSON.parse( e.data );
		} catch( e ) {
			wpcom_remote_login_final_cleanup();
			return;
		}

		if ( data.msg === 'LOGIN' ) {
			// Clean up the login check iframe
			wpcom_remote_login_remove_dom_node_id( "wpcom_remote_login_key" );

			var id_regex = new RegExp( /^[0-9]+$/ );
			var token_regex = new RegExp( /^.*|.*|.*$/ );
			if (
				token_regex.test( data.token )
				&& id_regex.test( data.wpcomid )
			) {
				// We have everything we need to ask for a login
				var script = document.createElement( "script" );
				script.setAttribute( "id", "wpcom_remote_login_validate" );

					+ '//' + window.location.hostname
					+ '&postid=54963'
					+ '&is_singular=1';
				document.body.appendChild( script );

			return;

		// Safari ITP, not logged in, so redirect
		if ( data.msg === 'LOGIN-REDIRECT' ) {
			window.location = 'https://wordpress.com/log-in?redirect_to=' + window.location.href;
			return;
		}

		// Safari ITP, storage access failed, remove the request
		if ( data.msg === 'LOGIN-REMOVE' ) {

			e.parentNode.removeChild( e );

			document.cookie = 'wordpress_com_login_access=denied; path=/; max-age=31536000';

			return;
		}

		// Safari ITP
		if ( data.msg === 'REQUEST_ACCESS' ) {
			console.log( 'request access: safari' );

			// Check ITP iframe enable/disable knob
			if ( wpcom_remote_login_extra_auth !== 'safari_itp_iframe' ) {
				return;

				var opendb = window.openDatabase( null, null, null, null );
			} catch( e ) {
				private_window = true;
			}

			if ( private_window ) {
				console.log( 'private window' );
				return;
			}

			var iframe = document.createElement( 'iframe' );
			iframe.id = 'wpcom_request_access_iframe';
			iframe.setAttribute( 'scrolling', 'no' );
			iframe.setAttribute( 'sandbox', 'allow-storage-access-by-user-activation allow-scripts allow-same-origin allow-top-navigation-by-user-activation' );

			var css = 'html { -webkit-transition: margin-top 1s; transition: margin-top 1s; } /* 9001 */ html { margin-top: 46px !important; } * html body { margin-top: 46px !important; } @media screen and ( max-width: 660px ) { html { margin-top: 71px !important; } * html body { margin-top: 71px !important; } #wpcom_request_access_iframe { display: block; height: 71px !important; } } #wpcom_request_access_iframe { border: 0px; height: 46px; position: fixed; top: 0; left: 0; width: 100%; min-width: 100%; z-index: 99999; background: #23282d; } ';

		<h1 class="entry-title">Quarantine and the quarantine&nbsp;flag
</h1>
	</header><!-- .entry-header -->

	
		<div class="entry-content" data-first_letter="W">
		<p>We all know that a &#8216;quarantine flag&#8217; is attached to files which are downloaded from the Internet, using most but not all apps, and determines whether an app needs to undergo a full first run check by Gatekeeper. In fact, there&#8217;s a great deal more to quarantine and its extended attribute than that. For a start, the majority of items on your Mac which carry a quarantine flag aren&#8217;t apps at all, but non-executable documents. And in most cases, macOS doesn&#8217;t even know why they are there.</p>
<p>Quarantine and the <code>com.apple.quarantine</code> extended attribute (xattr) originated in macOS 10.5 in 2007, although Gatekeeper didn&#8217;t appear until 10.7 in 2011-12, at around the same time that sandboxing was introduced.</p>
<p><strong>Entering quarantine</strong></p>
<p>All files which are downloaded from the Internet, using HTTPS or HTTP, in email messages, over AirDrop, and by other means, can have a quarantine flag attached to them by the app which performs the downloading. Custom app download-installers and most updaters either don&#8217;t set the flag at all, or, when one is set, remove it (for example, Sparkle-based updaters).</p>
<p>The quarantine flag is an opt-in system, not one imposed by macOS itself. Any developer, including malware authors, can download files from the Internet without setting the flag on them, and any app on your Mac can change or strip the quarantine flag on any item to which it has write permission. The use of these flags in security is very much a gentleman&#8217;s agreement, which is easily broken when software doesn&#8217;t behave like a gentleman.</p>
<p>Setting the quarantine flag is normally determined the Info.plist property list which every app is required to contain. The entry there which controls flag behaviour is named LSFileQuarantineEnabled, and you can inspect this in each app to check what should happen when that app creates a new file, for example when downloading something from the Internet. When this is set to true, every new file created by that app should have the quarantine flag set; when false, they won&#8217;t unless macOS overrides that behaviour. If an unsandboxed app&#8217;s Info.plist doesn&#8217;t set LSFileQuarantineEnabled explicitly, then the default is not to set the quarantine flag.</p>
<p>macOS also provides a set of overrides to what appears in the Info.plist of many apps, listed in the Additions item in /System/Library/CoreServices/CoreTypes.bundle/Contents/Resources/Exceptions.plist.</p>
<p>The Exceptions.plist property list contains five dictionaries:</p>
<ul>
<li><strong>Additions</strong>, which assigns a lot of app categories, sets Java version requirements, and determines default settings for quarantine on documents created by apps.</li>
<li><strong>AppNapOverrides</strong>, which sets App Nap behaviours.</li>
<li><strong>HighResolutionOverrides</strong>, which overrides High Res options for apps.</li>
<li><strong>LaunchOverrides</strong>, which can disable specific version ranges of apps from being launched; these prevent many older apps from being run.</li>
<li><strong>MergeDocumentTypes</strong>, which merges some document types such as <em>doc</em> and <em>docx</em> for specific apps.</li>
<li><strong>Overrides</strong>, which can override other settings.</li>
</ul>
<p>For example, the entry in the Additions dictionary for the popular BitTorrent client Transmission reads:<br />
<code>&lt;key&gt;org.m0k.transmission&lt;/key&gt;<br />
&lt;dict&gt;<br />
&lt;key&gt;LSApplicationCategoryType&lt;/key&gt;<br />
&lt;string&gt;public-category.internet&lt;/string&gt;<br />
&lt;key&gt;LSFileQuarantineEnabled&lt;/key&gt;<br />
&lt;true/&gt;<br />
&lt;/dict&gt;</code></p>
<p>Referring to the app by its ID of <code>org.m0k.transmission</code>, that first assigns the app to an app category of <code>public-category.internet</code>, and then sets the app to set the quarantine flag on all documents that it creates, including everything that it downloads.</p>
<p>Among the existing overrides in Catalina, for example, are org.pythonmac.unspecified.BitTorrent and org.xlife.Xtorrent, which ensures that Transmission, Xtorrent and PythonMac BitTorrent clients should write quarantine flags to all their downloaded files. Although this Exceptions property list doesn&#8217;t cover every client, it should ensure that most do protect their downloads with quarantine flags. However, there&#8217;s no method by which you can add or modify these, and they don&#8217;t appear to apply to command tools such as <code>curl</code>, which is often used to bypass quarantine flag attachment.</p>
<p><strong>Behaviour &#8211; apps</strong></p>
<p>The quarantine flag is among the stickiest of all xattrs. When you unZip an archive which has been flagged, the xattr is normally propagated to all items which are saved from that, a behaviour which ensures that compressed apps retain their flag when uncompressed, for example. This isn&#8217;t, though, imposed by macOS, and some tools and utilities which can decompress archives may not follow this behaviour; the bundled Archive Utility does, though.</p>
<p>In macOS Mojave and later, a typical quarantine xattr consists of a Unicode string of the form<br />
<code>0083;5991b778;Safari.app;BC4DFC58-0D26-460D-9688-81D119298642</code><br />
with the components:</p>
<ol>
<li>the quarantine value in hexadecimal,</li>
<li>the time at which the xattr was attached, in hexadecimal,</li>
<li>the app or agent responsible for creating the xattr (normally the downloading app too),</li>
<li>a UUID referring to the entry for this quarantine flag in the QuarantineEvents database</li>
</ol>
<p>separated by semi-colons.</p>
<p>The QuarantineEvents database is an SQLite database at ~/Library/Preferences/com.apple.LaunchServices.QuarantineEventsV2. There appears to be no system-level equivalent, so each user is only able to access details of their own quarantine events, not those of other users. It&#8217;s not known whether spent events are ever removed from there, nor whether maintenance is performed on it.</p>
<p>When an app or other executable is run from the GUI, its quarantine xattr is checked before opening. If the executable code hasn&#8217;t been cleared previously by a full Gatekeeper check, that is deemed to be its first run, and a full check is performed. If that&#8217;s successful, the quarantine value on all checked executable code is changed from (for example)<br />
<code>00000000 10000011 = 83</code><br />
to<br />
<code>00000000 11000011 = C3</code><br />
Subsequent attempts to run that code are then no longer blocked for the first run Gatekeeper check to be performed. Successful completion of that first run check doesn&#8217;t alter the quarantine xattrs attached to non-executable files within an app bundle, though.</p>
<p>Earlier versions of macOS have used other bits in the quarantine value too. For example, in Sierra and earlier an app which has passed first run and been successfully opened could end up with a value of<br />
<code>00000000 11100011 = E3</code><br />
the lower-order bit signifying that the app had also been run. These appear to have fallen into disuse in Mojave.</p>
<p>The QuarantineEvents database contains and retains additional information for these flags, which includes the reason for their attachment in the LSQuarantineType value, which includes: LSQuarantineTypeWebDownload, LSQuarantineTypeEmailAttachment, LSQuarantineTypeOtherDownload, LSQuarantineTypeInstantMessageAttachment, LSQuarantineTypeCalendarEventAttachment, LSQuarantineTypeOtherAttachment, and LSQuarantineTypeSandboxed, which is only attached to documents.</p>
<p>In normal circumstances, quarantine xattrs which are attached to apps and other executables remain in place until that app is removed.</p>
<p>Developers who ship apps which are signed or notarized need to check that those will successfully pass through Gatekeeper first run checks before distributing those apps. I have discussed ways of doing this <a href="https://eclecticlight.co/2019/01/18/code-signing-for-the-concerned-4-results-and-testing/">here</a>.</p>
<p>Executable scripts with quarantine flags may not undergo Gatekeeper checks, because most are unsigned. The presence of a quarantine flag on a script therefore normally results in macOS blocking it from running until that flag has been removed.</p>
<p><strong>Behaviour &#8211; documents</strong></p>
<p>macOS has been attaching quarantine flags to documents for as long as it has been to apps, as part of the the same process. If a webpage or other file is downloaded from the Internet and saved on your Mac by an app which adds quarantine flags, then a normal quarantine xattr will be added to it. When you decompress a flagged Zip archive, quarantine flags are automatically attached to all the files extracted from it.</p>
<p>These non-app flags differ as follows:</p>
<ul>
<li>Opening a quarantined document or non-executable file doesn&#8217;t trigger a Gatekeeper check, which would in any case be meaningless.</li>
<li>Nothing appears to change or remove a flag, unless you use a utility or command to do so. The sole exception to this is with flags attached by sandboxed apps, which can replace one another.</li>
<li>macOS also adds its own quarantine flags to documents which haven&#8217;t been downloaded from the Internet. Oddly, while quarantine of apps is an opt-in behaviour, you can&#8217;t opt out of this behaviour, as it&#8217;s built into the macOS sandbox.</li>
<li>Performing certain operations with quarantined documents may be forbidden. For example, a flagged shell script can&#8217;t normally be executed. Thus using flagged files can result in errors.</li>
</ul>
<p>Document quarantine xattrs attached following download from the Internet have identical content to those attached to apps, including the UUID of an entry in the QuarantineEvents database. Those attached to files which haven&#8217;t been downloaded differ, as they aren&#8217;t associated with entries in the QuarantineEvents database, so lack the UUID. Quarantine values you&#8217;re likely to encounter include:</p>
<ul>
<li><code>00000000 10000001 = 81</code></li>
<li><code>00000000 10000010 = 82</code></li>
<li><code>00000000 10000011 = 83</code></li>
</ul>
<p>each of which has the high-order bit set to indicate that the file is still in quarantine, and are normally given with an LSQuarantineType of LSQuarantineTypeSandboxed.</p>
<p>The addition of quarantine flags to files which have never been downloaded from the Internet appears to be a relatively recent behaviour, but has now been seen to occur in macOS Sierra and later. Apple doesn&#8217;t appear to have documented this for users, and references to this behaviour are buried deep in Apple&#8217;s now outdated but not replaced <a href="https://developer.apple.com/library/archive/documentation/Miscellaneous/Reference/EntitlementKeyReference/Chapters/EnablingAppSandbox.html" target="_blank" rel="noopener">Entitlement Key Reference</a>.</p>
<p>Sandboxed apps, which includes many of those bundled with macOS and all delivered by the App Store, will attach quarantine flags to files which Apple describes as executable unless the app has the com.apple.security.files.user-selected.executable entitlement. Apple explains:<br />
<em>&#8220;By default, when writing executable files in sandboxed apps, the files are quarantined. Gatekeeper prevents quarantined executable files and other similar files (shell scripts, web archives, and so on) from opening or executing unless the user explicitly launches them from Finder.<br />
If those executables are tools that are intended to run from the command line, such as shell scripts, this presents a problem.&#8221;</em></p>
<p>Note that this refers to a sandboxed app writing executable files, which should only account for a very small number of files having a quarantine flag attached.</p>
<p>The role and purpose of these quarantine flags added by sandboxed apps remains obscure, beyond being used to prevent the execution of shell scripts, web archives, etc. Not only that, but sandboxed apps write them, when permissions allow, to all documents which they open, even though the document may not be formally saved by that app.</p>
<p><strong>Access</strong></p>
<p>As extended attributes, you can view and edit quarantine flags using standard tools including the <code>xattr</code> command and my free utility <a href="https://eclecticlight.co/xattred-sandstrip-xattr-tools/">xattred</a>. These flags are easier to work with than many xattrs as they are UTF-8 text, not encoded binary.</p>
<p>As a developer, you can opt to access them as xattrs, which is easiest using extensions to the URL class in Swift, for instance. They&#8217;re also accessible more directly as NSURL Resources, using code such as<br />
<code>var theQuarFlag: AnyObject? = nil<br />
try theNSURL.getResourceValue(&amp;theQuarFlag, forKey: kCFURLQuarantinePropertiesKey as URLResourceKey)</code><br />
which, if <code>theQuarFlag</code> is non-nil, is a dictionary containing key-value pairs for all known fields in the quarantine flag. These not only include the data in the xattr, but where there&#8217;s an extant entry in the QuarantineEvents database, this also returns information from that. Keys are listed in LSQuarantine.h, and include:</p>
<ul>
<li>LSQuarantineAgentNameKey, the name of the downloading agent;</li>
<li>LSQuarantineAgentBundleIdentifierKey, the bundle ID of the downloading agent;</li>
<li>LSQuarantineTimeStampKey, a CFDateRef to the date and time that the quarantine flag was attached;</li>
<li>LSQuarantineTypeKey, one of the values listed above;</li>
<li>LSQuarantineOriginURLKey, a CFURLRef to the original location of the file;</li>
<li>LSQuarantineDataURLKey, a CFURLRef to the data source of the file.</li>
</ul>
<p>Removing a quarantine flag simply requires setting it to nil, as in<br />
<code>try theNSURL.setResourceValue(nil, forKey: kCFURLQuarantinePropertiesKey as URLResourceKey)</code></p>
