    <h2>Windows Pragmatic General Multicast </h2>
<p>Windows Pragmatic General Multicast (PGM) is a protocol designed to deliver packets to multiple network members in a reliable manner. On Windows, the implementation of this protocol is referred to as <a href="https://learn.microsoft.com/en-us/windows/win32/winsock/reliable-multicast-programming--pgm-" target="_blank" rel="nofollow noreferrer">reliable multicast programming</a>. <b>A critical remote code execution vulnerability in this protocol, with a CVSS score of 9.8, was fixed this Patch Tuesday.</b></p>
<p>Since Windows Server 2003, PGM has relied on Windows sockets. In user space, it is implemented in a library called wshrm.dll (Windows Sockets Helper DLL for PGM, where the “<i>rm</i>” stands for <i>remote multicast</i>). In kernel space, PGM is implemented through the driver rmcast.sys.</p>
<p>According to Microsoft’s mitigations on the CVEs, the Message Queuing service must be running for the vulnerability to be exploitable. The service is not installed by default; it must be added via the Features screen in the control panel.</p>
<p>In our observations, <b>we noticed that approximately 50% of environments had servers with the Message Queuing service installed and running, and 25% had servers with PGM installed</b>. Although in most data centers only a few machines had Message Queuing enabled, we noticed some environments in which multiple servers were running the service.</p>
<p>Since the attack complexity for all vulnerabilities is low, we recommend patching relevant servers as soon as possible. If patching is not possible — because of operational continuity or some other reason — then we recommend restricting access to the service by using <a href="/blog/security-research/segmentation-from-a-practical-perspective" target="_self">network segmentation</a> policies, at the very least. </p>
<p><b>Since the Message Queuing service is accessible over port 1801, but isn’t likely to be accessed by very many clients</b> (as it’s mostly used by the enterprise application itself), <b>we recommend restricting arbitrary network access to that port and service</b>. Try to segment it using allowlist policies, granting access only to the machines that actually need it.</p>

    <h2>Microsoft Protected Extensible Authentication Protocol </h2>
<p>The <a href="https://en.wikipedia.org/wiki/Protected_Extensible_Authentication_Protocol" target="_blank" rel="nofollow noreferrer">Protected Extensible Authentication Protocol</a> (PEAP) is a protocol that encapsulates the <a href="https://en.wikipedia.org/wiki/Extensible_Authentication_Protocol" target="_blank" rel="nofollow noreferrer">Extensible Authentication Protocol</a> (EAP) within a TLS session. EAP is used to authenticate devices and connections to networks, and is also used in Wi-Fi network authentication.</p>
<p>Windows servers can be configured with an optional role called Network Policy Server (NPS), and the NPS can be used to authenticate and allow network connection requests with EAP and PEAP. This isn’t enabled by default — the <b>NPS role must be manually installed and configured</b>. From our observations, only 21% of networks had NPS servers.</p>
<p>Since the NPS is supposed to be central to the network, and is also supposed to process authentication requests, it might be problematic to apply segmentation policies on it. Instead, Microsoft says that a possible mitigation would be to block PEAP in protocol negotiations and provides <a href="https://learn.microsoft.com/en-us/windows-server/networking/core-network-guide/cncg/wireless/e-wireless-access-deployment#bkmk_configureprofile" target="_blank" rel="nofollow noreferrer">two</a> <a href="https://learn.microsoft.com/en-us/windows-server/networking/technologies/nps/nps-np-configure" target="_blank" rel="nofollow noreferrer">sources</a> to do so.</p>
<p><b>This month, a critical remote code execution vulnerability (<a href="https://msrc.microsoft.com/update-guide/en-US/vulnerability/CVE-2023-36028" target="_blank" rel="nofollow noreferrer">CVE-2023-36028</a>) with a score of 9.8 was patched.</b></p>
