            <div class="clearfix text-formatted field field--name-body field--type-text-with-summary field--label-hidden field__item">
                          <p>Necurs is a malware that is mainly known for sending large spam campaigns, most notably the Locky ransomware. However, Necurs is not only a spambot, it is a modular piece of malware that is composed of a main bot module, a userland rootkit and it can dynamically load additional modules. </p>
<p>Having been around for a few years, this malware has been subject to a lot of great research that has covered many of its aspects <a href="https://www.virusbulletin.com/virusbulletin/2014/04/curse-necurs-part-1">including</a> <a href="https://www.virusbulletin.com/virusbulletin/2014/05/curse-necurs-part-2">the</a> <a href="https://www.virusbulletin.com/virusbulletin/2014/06/curse-necurs-part-3">rootkit</a>, <a href="https://www.johannesbader.ch/2015/02/the-dgas-of-necurs/">the DGA</a>, <a href="https://www.cert.pl/en/news/single/necurs-hybrid-spam-botnet/">the communication protocol and the spam module</a>. However, there has not been much public information on any modules loaded by Necurs besides the spam module.</p>
<!--more-->
<p>About six months ago we noticed that besides the usual port 80 communications, a Necurs infected system was communicating with a set of IPs on a different port using, what appeared to be, a different protocol. The following image shows an example of this network traffic.</p>
<p><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog1.png" alt="AnubisBlog1.png" width="730" height="555" loading="lazy"></p>
<p>Recently, while decrypting the C2 communication of the a Necurs bot, we observed a request to load two different modules, each with a different parameter list. The following is the decrypted translation of the modules section of the C2 response:</p>
<p><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog2.png" alt="AnubisBlog2.png" width="730" height="222" loading="lazy"></p>
<p>The first one was the spam module for which Necurs is most known, and the parameters are the C2 addresses from which it can receive new spam campaigns. The second one was an unknown module and, judging by the parameter values, it was the one responsible for the communication we were seeing to port 5222.</p>
<p>We noticed this module in September 2016 and the compilation timestamp on the module is “Aug 23 2016”, which suggests the module started being used around then. However, it is possible that another version of the same module had been deployed before, without being noticed.</p>
<p>We downloaded the module and reverse engineered it to try to understand exactly what it was. At first look, it seemed to be a simple SOCKS/HTTP proxy module, but as we looked at the commands the bot would accept from the C2 we realized that there was an additional command, that would cause the bot to start making HTTP or UDP requests to an arbitrary target in an endless loop, in a way that could only be explained as a DDOS attack. This is particularly interesting considering the size of the Necurs botnets (the largest one, where this module was being loaded, has over 1 million active infections each 24 hours, we blogged about it <a href="http://blog.anubisnetworks.com/blog/monitoring-necurs-the-tip-of-the-iceberg">here</a>). A botnet this big can likely produce a very powerfull DDOS attack. </p>
<p>Please notice that <strong>we have not seen Necurs being used for DDOS attacks</strong>, we simply saw that it has that capability in one of the modules that it has been loading.</p>
<p>The rest of this post contains the results of a technical analysis of this module, detailing its C2 protocol, the SOCKS/HTTP proxy features, and the DDOS attack features.</p>
<h2>MODULE TECHNICAL ANALISYS</h2>
<h3><strong>Module start/initialization</strong></h3>
<p>Once the module is loaded by the bot, it performs the following initialization actions:</p>
<ul>
<li>Parses the parameters and stores them in an internal <strong>list of C2 addresses</strong>;</li>
<li><strong>Fills a memory structure </strong>(see <em>botsettings</em> struct definition below) with:
<ul>
<li><strong>The BotID </strong>- Generated through gathering unique system characteristics;</li>
<li><strong>The internal IP address </strong>- Obtained by checking the outbound sockets IP address when connecting to google.com;</li>
<li><strong>The external IP address </strong>- Obtained trough HTTP from ipv4.icanhazip.com or checkip.dyndns.org;</li>
<li><strong>The available bandwidth </strong>- Obtained by measuring the download speed of the Windows 7 Service Pack 1 file from microsoft;</li>
<li><strong>The (socks/http) proxy service port </strong>- The port of the service listening on a random port above 1024;</li>
</ul>
</li>
<li><strong>Checks if the system is behind NAT </strong>- By checking if the outbound socket IP is not a local address and that it matches the external IP;</li>
<li>If the system is not behind NAT, the bot <strong>starts a SOCKS/HTTP proxy service </strong>listening on a random port above 1024.</li>
</ul>
<p>The <em>botsettings</em> struct can be defined as follows:</p>
<p><strong><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog3.png" alt="AnubisBlog3.png" width="730" height="146" loading="lazy"></strong></p>
<h3><strong>C2 communication protocol</strong></h3>
<p>After initialization, the bot will enter the main C2 connection loop, where it will attempt to connect to the current C2 every 10 seconds, unless instructed otherwise. If the connection to the current C2 fails, it will fetch another from the address list and try again. </p>
<p>The communication protocol is binary and encrypted/obfuscated using a custom algorithm. Messages to, and from, the server have a very similar structure (see struct <em>botmsg </em>and <em>c2msg </em>definitions below) and contain the following data:</p>
<ul>
<li><strong>Key </strong>- A 32 bit encryption key;</li>
<li><strong>Encrypted header </strong>- A header structure (see struct <em>botmsgheader</em> and <em>c2msgheader </em>definitions below), encrypted with the key and containing:
<ul>
<li><strong>Message type </strong>- A byte that defines the type of message/command being sent;</li>
<li><strong>Payload length </strong>- The length of the payload being sent;</li>
<li><strong>Header hash </strong>- A hash of the first bytes of the message (key, msgtype, unknown and datalength);</li>
<li><strong>Data hash </strong>- A hash of the payload, used for integrity checking;</li>
</ul>
</li>
<li><strong>Encrypted payload </strong>- An array of data being sent, encrypted with the reverse value of the key.</li>
</ul>
<p> <img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog4.png" alt="AnubisBlog4.png" width="730" height="203" loading="lazy"><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog5.png" alt="AnubisBlog5.png" width="730" height="258" loading="lazy"></p>
<p>There are three types of messages sent by the bot to the C2, that can be distinguished by the <em>msgtype</em> byte in the header:</p>
<ul>
<li><strong>Beacon </strong>(msgtype 0) - This is the main message that is sent by the bot every 10 seconds. It sends the <em>botsettings</em>struct described previously as a payload;</li>
<li><strong>Connectivity check </strong>(msgtype 1) - This is a simple dummy message that contains no data besides the encrypted message header. It is sent in case a timeout occurs to the current C2 to make sure it is no longer available;</li>
<li><strong>Proxybackconnect </strong>(msgtype 2) - This message is sent in case the bot receives the command to start a socks backconnect. This will start a connection to the C2 that will be reused for the SOCKS/HTTP proxy connection, proceeding as if it had been initiated by the proxy client.</li>
</ul>
<p>As a response to the beacon, there are also three types of messages (or commands) sent by the C2 to the bot, that can be distinguished by the msgtype byte in the header:</p>
              </div>

  </div>

<div class="text-wysiwyg
   block-content block-content--type-text block-content--8301 block-content--view-mode-full container" class="block-content block-content--type-text block-content--8301 block-content--view-mode-full container">

            <div class="clearfix text-formatted field field--name-body field--type-text-with-summary field--label-hidden field__item">
                          <ul>
<li><strong>Start Proxybackconnect </strong>(msgtype 1) - This command tells the bot to start a <em>backconnect</em> proxy session by sending a <em>proxybackconnect</em> message from the bot to the C2. This connection’s socket will be reused and allows the bot to be used as a proxy even behind a firewall and without establishing a direct connection to it;</li>
<li><strong>Sleep </strong>(msgtype 2) - This will cause the bot to sleep for 5 minutes;</li>
<li><strong>Start DDOS </strong>(msgtype 5) - This command will start a DDOS attack against the target specified in the message payload using one of two available attack modes:
<ul>
<li><strong>HTTPFlood: </strong>If the first bytes of the message payload are the string “http:/”, the bot will start an HTTP flood attack against the target;</li>
<li><strong>UDPFlood: </strong>If the first bytes of the message payload are <strong>not</strong> the string “http:/”, the bot will start an UDP flood attack against the target.</li>
</ul>
</li>
</ul>
<h3><strong>Proxy features</strong></h3>
<p>The SOCKS/HTTP proxy service and command, allows the botnet owners to use the compromised bots as proxies (HTTP, SOCKSv4 and SOCKSv5 protocols), relaying connections through them in two modes of operation (direct proxy and proxy backconnect).</p>
<p>In direct proxy mode, the client connects to the proxy service, which will forward the communication to the destination, as in the following image:</p>
<p><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog6.png" alt="AnubisBlog6.png" width="625" height="83" loading="lazy"> </p>
<p>This is only possible if the bot is not protected behind NAT or a firewall, which is not the case in the vast majority of the botnet.</p>
<p>In proxy backconnect mode, the client connects to the proxy controller, that will then obtain an outbound proxy from a pool of available proxies and relay the connection through it. The following diagram illustrates the system:</p>
<p><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog7.png" alt="AnubisBlog7.png" width="730" height="674" loading="lazy"></p>
<p>There are several advantages to this operation mode. The main ones being that it works even if the infected system is behind NAT and it will allow a connection to internal network resources as if it came from the infected machine. Another common use for these kind of proxies is to have a constantly changing IP address, by constantly and automatically changing the bot serving as proxy.</p>
<p>Taking a brief view at how this is implemented, the following image shows part of the function that processes the C2 commands.</p>
<p><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/ANubisBlog8.png" alt="ANubisBlog8.png" width="625" height="500" loading="lazy"></p>
<p>In case the bot receives a <em>startproxybackconnect</em> command (msgtype 1), the bot will send a <em>proxybackconnect</em> message to the C2 (msgtype 2) and then the same socket that is used in the C2 communication (<em>globalsocket)</em> to the <em>startprocessincoming</em> function that does the actual proxying work. This means that the same connection used for C2 communication will then be used to proxy the connections.</p>
<p>The <em>processincomming</em> function reads 2 bytes from the incoming connection (direct or through backconnect) and checks if the first has the value 5 (Socks v5 protocol), 4 (Socks v4 protocol) or if it is alphanumeric (HTTP Proxy). It then calls the appropriate function that does the actual proxying work for each of the supported protocols. </p>
<p><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog9.png" alt="AnubisBlog9.png" width="730" height="759" loading="lazy"></p>
<h3><strong>DDOS attack features</strong></h3>
<p>Possibly the most interesting, and unexpected,  feature of this module is the DDOS attack mode. The module contains two very basic DDOS attack modes, that don’t have special features like origin IP address spoofing or amplification techniques. However, given the size of the Necurs botnets (more than 1M IP/24 hours in the largest botnet), even the most basic techniques should produce a very powerful attack.</p>
<p>Taking a brief look at how it is implemented, if the bot receives the <em>startDDOS</em> command (msgtype 5) the payload portion of message is parsed looking for the string “http:/”. If it is found, the <em>HTTPflood</em> function is called, if not, the <em>UDPFlood</em> function is called, as shown in the following image:</p>
<p><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog10.png" alt="AnubisBlog10.png" width="546" height="883" loading="lazy"> </p>
<h2>HTTP Flood attack mode</h2>
<p>The HTTP attack works by starting 16 threads that perform an endless loop of HTTP requests. The following images show the 16 threads being queued and the section of the code that sends the HTTP request: <img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog11.png" alt="AnubisBlog11.png" width="531" height="313" loading="lazy"><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog12.png" alt="AnubisBlog12.png" width="730" height="596" loading="lazy"></p>
<p>The HTTP request is built using the following format string:</p>
<p><img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog13.png" alt="AnubisBlog13.png" width="730" height="237" loading="lazy"></p>
<h2>UDP Flood attack mode</h2>
<p>The UDP flood attack works by repeatedly sending a random payload with size between 128 and 1024 bytes. The function contains a 0.1 second sleep that may be triggered depending on the available bandwidth (calculated during bot initialization), possibly to avoid losing access to the bot during a DDOS attack. The following image shows the UDPFlood main loop: <img src="https://cdn2.hubspot.net/hubfs/277648/images/blog/AnubisBlog14.png" alt="AnubisBlog14.png" width="730" height="799" loading="lazy"><strong> </strong></p>
<h2><strong>Conclusion</strong></h2>
<p>Although known mainly for its spam module, Necurs is a modular malware that can be used for many different purposes. In this blog post we detailed a module that adds SOCKS/HTTP proxy and DDOS capabilities to this malware. Although we have not seen Necurs being used to perform DDOS attacks, this capability is currently deployed in the infected systems and taking into account the size of the botnet it could produce a powerful attack. </p>
