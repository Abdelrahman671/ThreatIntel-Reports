											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>When an application dynamically loads a dynamic-link library without specifying a fully qualified path name, Windows attempts to locate the DLL by searching a well-defined set of directories in a particular order, as described in <a href="dynamic-link-library-search-order" data-linktype="relative-path">Dynamic-Link Library Search Order</a>. If an attacker gains control of one of the directories on the DLL search path, it can place a malicious copy of the DLL in that directory. This is sometimes called a <em>DLL preloading attack</em> or a <em>binary planting attack</em>. If the system does not find a legitimate copy of the DLL before it searches the compromised directory, it loads the malicious DLL. If the application is running with administrator privileges, the attacker may succeed in local privilege elevation.</p>
<p>For example, suppose an application is designed to load a DLL from the user's current directory and fail gracefully if the DLL is not found. The application calls <a href="/en-us/windows/win32/api/libloaderapi/nf-libloaderapi-loadlibrarya" data-linktype="absolute-path"><strong>LoadLibrary</strong></a> with just the name of the DLL, which causes the system to search for the DLL. Assuming safe DLL search mode is enabled and the application is not using an alternate search order, the system searches directories in the following order:</p>
<ol>
<li>The directory from which the application loaded.</li>
<li>The system directory.</li>
<li>The 16-bit system directory.</li>
<li>The Windows directory.</li>
<li>The current directory.</li>
<li>The directories that are listed in the PATH environment variable.</li>
</ol>
<p>Continuing the example, an attacker with knowledge of the application gains control of the current directory and places a malicious copy of the DLL in that directory. When the application issues the <strong>LoadLibrary</strong> call, the system searches for the DLL, finds the malicious copy of the DLL in the current directory, and loads it. The malicious copy of the DLL then runs within the application and gains the privileges of the user.</p>
<p>Developers can help safeguard their applications against DLL preloading attacks by following these guidelines:</p>
<ul>
<li><p>Wherever possible, specify a fully qualified path when using the <a href="/en-us/windows/win32/api/libloaderapi/nf-libloaderapi-loadlibrarya" data-linktype="absolute-path"><strong>LoadLibrary</strong></a>, <a href="/en-us/windows/desktop/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a>, <a href="/en-us/windows/desktop/api/processthreadsapi/nf-processthreadsapi-createprocessa" data-linktype="absolute-path"><strong>CreateProcess</strong></a>, or <a href="/en-us/windows/desktop/api/shellapi/nf-shellapi-shellexecutea" data-linktype="absolute-path"><strong>ShellExecute</strong></a> functions.</p>
</li>
<li><p>Use the LOAD_LIBRARY_SEARCH flags with the <a href="/en-us/windows/desktop/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a> function, or use these flags with the <a href="/en-us/windows/desktop/api/LibLoaderAPI/nf-libloaderapi-setdefaultdlldirectories" data-linktype="absolute-path"><strong>SetDefaultDllDirectories</strong></a> function to establish a DLL search order for a process and then use the <a href="/en-us/windows/desktop/api/LibLoaderAPI/nf-libloaderapi-adddlldirectory" data-linktype="absolute-path"><strong>AddDllDirectory</strong></a> or <a href="/en-us/windows/desktop/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> functions to modify the list. For more information, see <a href="dynamic-link-library-search-order" data-linktype="relative-path">Dynamic-Link Library Search Order</a>.</p>
<p><strong>Windows&nbsp;7, Windows Server&nbsp;2008&nbsp;R2, Windows&nbsp;Vista and Windows Server&nbsp;2008:</strong> These flags and functions are available on systems with <a href="https://support.microsoft.com/kb/2533623" data-linktype="external">KB2533623</a> installed.</p>
</li>
<li><p>On systems with <a href="https://support.microsoft.com/kb/2533623" data-linktype="external">KB2533623</a> installed, use the LOAD_LIBRARY_SEARCH flags with the <a href="/en-us/windows/desktop/api/LibLoaderAPI/nf-libloaderapi-loadlibraryexa" data-linktype="absolute-path"><strong>LoadLibraryEx</strong></a> function, or use these flags with the <a href="/en-us/windows/desktop/api/LibLoaderAPI/nf-libloaderapi-setdefaultdlldirectories" data-linktype="absolute-path"><strong>SetDefaultDllDirectories</strong></a> function to establish a DLL search order for a process and then use the <a href="/en-us/windows/desktop/api/LibLoaderAPI/nf-libloaderapi-adddlldirectory" data-linktype="absolute-path"><strong>AddDllDirectory</strong></a> or <a href="/en-us/windows/desktop/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> functions to modify the list. For more information, see <a href="dynamic-link-library-search-order" data-linktype="relative-path">Dynamic-Link Library Search Order</a>.</p>
</li>
<li><p>Consider using <a href="dynamic-link-library-redirection" data-linktype="relative-path">DLL redirection</a> or a <a href="/en-us/windows/desktop/SbsCs/manifests" data-linktype="absolute-path">manifest</a> to ensure that your application uses the correct DLL.</p>
</li>
<li><p>When using the standard search order, make sure that safe DLL search mode is enabled. This places the user's current directory later in the search order, increasing the chances that Windows will find a legitimate copy of the DLL before a malicious copy. Safe DLL search mode is enabled by default starting with Windows&nbsp;XP with Service Pack&nbsp;2 (SP2) and is controlled by the <strong>HKEY_LOCAL_MACHINE\System\CurrentControlSet\Control\Session Manager</strong>\<strong>SafeDllSearchMode</strong> registry value. For more information, see <a href="dynamic-link-library-search-order" data-linktype="relative-path">Dynamic-Link Library Search Order</a>.</p>
</li>
<li><p>Consider removing the current directory from the standard search path by calling <a href="/en-us/windows/desktop/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a> with an empty string (""). This should be done once early in process initialization, not before and after calls to <a href="/en-us/windows/win32/api/libloaderapi/nf-libloaderapi-loadlibrarya" data-linktype="absolute-path"><strong>LoadLibrary</strong></a>. Be aware that <strong>SetDllDirectory</strong> affects the entire process and that multiple threads calling <strong>SetDllDirectory</strong> with different values can cause undefined behavior. If your application loads third-party DLLs, test carefully to identify any incompatibilities.</p>
</li>
<li><p>Do not use the <a href="/en-us/windows/desktop/api/processenv/nf-processenv-searchpathw" data-linktype="absolute-path"><strong>SearchPath</strong></a> function to retrieve a path to a DLL for a subsequent <a href="/en-us/windows/win32/api/libloaderapi/nf-libloaderapi-loadlibrarya" data-linktype="absolute-path"><strong>LoadLibrary</strong></a> call unless safe process search mode is enabled. When safe process search mode is not enabled, the <strong>SearchPath</strong> function uses a different search order than <strong>LoadLibrary</strong> and is likely to first search the user's current directory for the specified DLL. To enable safe process search mode for the <strong>SearchPath</strong> function, use the <a href="/en-us/windows/desktop/api/winbase/nf-winbase-setsearchpathmode" data-linktype="absolute-path"><strong>SetSearchPathMode</strong></a> function with BASE_SEARCH_PATH_ENABLE_SAFE_SEARCHMODE. This moves the current directory to the end of the <strong>SearchPath</strong> search list for the life of the process. Note that the current directory is not removed from the search path, so if the system does not find a legitimate copy of the DLL before it reaches the current directory, the application is still vulnerable. As with <a href="/en-us/windows/desktop/api/Winbase/nf-winbase-setdlldirectorya" data-linktype="absolute-path"><strong>SetDllDirectory</strong></a>, calling <strong>SetSearchPathMode</strong> should be done early in process initialization and it affects the entire process. If your application loads third-party DLLs, test carefully to identify any incompatibilities.</p>
</li>
<li><p>Do not make assumptions about the operating system version based on a <a href="/en-us/windows/win32/api/libloaderapi/nf-libloaderapi-loadlibrarya" data-linktype="absolute-path"><strong>LoadLibrary</strong></a> call that searches for a DLL. If the application is running in an environment where the DLL is legitimately not present but a malicious copy of the DLL is in the search path, the malicious copy of the DLL may be loaded. Instead, use the recommended techniques described in <a href="/en-us/windows/desktop/SysInfo/getting-the-system-version" data-linktype="absolute-path">Getting the System Version</a>.</p>
</li>
</ul>
<p>The Process Monitor tool can be used to help identify DLL load operations that might be vulnerable. The Process Monitor tool can be downloaded from <a href="https://technet.microsoft.com/sysinternals/bb896645.aspx" data-linktype="external">https://technet.microsoft.com/sysinternals/bb896645.aspx</a>.</p>
<p>The following procedure describes how to use Process Monitor to examine DLL load operations in your application.</p>
<p><strong>To use Process Monitor to examine DLL load operations in your application</strong></p>
<ol>
<li>Start Process Monitor.</li>
<li>In Process Monitor, include the following filters:
<ul>
<li>Operation is CreateFile</li>
<li>Operation is LoadImage</li>
<li>Path contains .cpl</li>
<li>Path contains .dll</li>
<li>Path contains .drv</li>
<li>Path contains .exe</li>
<li>Path contains .ocx</li>
<li>Path contains .scr</li>
<li>Path contains .sys</li>
</ul>
</li>
<li>Exclude the following filters:
<ul>
<li>Process Name is procmon.exe</li>
<li>Process Name is Procmon64.exe</li>
<li>Process Name is System</li>
<li>Operation begins with IRP_MJ_</li>
<li>Operation begins with FASTIO_</li>
<li>Result is SUCCESS</li>
<li>Path ends with pagefile.sys</li>
</ul>
</li>
<li>Attempt to start your application with the current directory set to a specific directory. For example, double-click a file with an extension whose file handler is assigned to your application.</li>
<li>Check Process Monitor output for paths that look suspicious, such as a call to the current directory to load a DLL. This kind of call might indicate a vulnerability in your application.</li>
</ol>
