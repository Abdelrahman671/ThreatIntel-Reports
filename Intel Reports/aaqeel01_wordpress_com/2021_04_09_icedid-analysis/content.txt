		<h1 class="entry-title">IcedID Analysis</h1>				<div class="entry-meta">
			<span class="posted-on"><a href="https://aaqeel01.wordpress.com/2021/04/09/icedid-analysis/" rel="bookmark"><time class="entry-date published" datetime="2021-04-09T18:46:36+03:00">April 9, 2021</time><time class="updated" datetime="2021-04-10T20:44:23+03:00">April 10, 2021</time></a></span><span class="byline"> by <span class="author vcard"><a class="url fn n" href="https://aaqeel01.wordpress.com/author/alialmemar/">Ali Aqeel</a></span></span><span class="cat-links">, posted in <a href="https://aaqeel01.wordpress.com/category/article/" rel="category tag">article</a></span>		</div><!-- .entry-meta -->			</header>
	<div class="entry-content">
		
<p><a rel="noreferrer noopener" href="https://malpedia.caad.fkie.fraunhofer.de/details/win.icedid" target="_blank">IcedID</a> aka (BokBot) is banking malware designed to steal financial information. <a rel="noreferrer noopener" href="https://malpedia.caad.fkie.fraunhofer.de/actor/lunar_spider" target="_blank">Lunar Spider</a> is the threat actor behind IcedID which they&#8217;ve been running campaigns since at least 2017. Beside stealing banking information, some incident show that IcedID is an entry stage to ransomware or RAT attack. It&#8217;s been observed lately that the threat actor has been using new techniques to evade detection by endpoint security, sandbox, and anti-reversing. Which makes it interesting to try dissecting samples to find out indicators and other artifacts that could be missed by security tools. </p>



<p>In this post, will take a look at IcedID sample that&#8217;s been posted on <a rel="noreferrer noopener" href="https://www.malware-traffic-analysis.net/2021/03/19/index.html" target="_blank">Malware-traffic-analysis.net</a>. Will walkthrough each artifact to learn how to unpack the hidden malicious binaries. These techniques would also work on other IcedID samples that has been found lately. </p>



<h2 class="wp-block-heading">behaviour overview</h2>



<p>Threat actor send an email with attached ZIP archived including maldoc either MS Word or Excel spreadsheet. When opening the the maldoc it asks to enable macros. Once enabled two function happens first download a DLL file and run it in a process using &#8216;rundll32.exe&#8217;. The downloaded DLL has unknown extension. After running in process, the DLL file &#8216;Installer&#8217; does mainly two things: download a GZIP compressed binary and install it. The GZIP might have zip extension, but it can&#8217;t be open or extracted with any archived tool. The GZIP mainly a dropper, it&#8217;s packed with two binaries. without further ado let&#8217;s get started with the below artifacts.</p>

<p>&#8216;Kiod.hod&#8217; is the name of the first stage IcedID execution in this sample. <p>There&#8217;s one library &#8216;kernel32.dll&#8217; and no sign of imported APIs to help guide either statically or in debugger in order to unleash any indicators. Simply loading the sample on x64dbg doesn&#8217;t work! There&#8217;re multiple ways to unpack the installer DLL, however, the quick and possible way to unpack the binary by attaching the installer DLL to (~Windows\System32\rundll32) in x64dbg like steps below:</p>



<pre class="wp-block-code"><code>
1. x64dbg -&gt; File -&gt; Open -&gt; C:\Windows\System32\rundll32.exe
2. x64dbg -&gt; File -&gt; Change Command Line -&gt; "C:\Windows\System32\rundll32.exe" "C:\Users\~\lab\Kiod.dll",DllRegisterServer 
* no spaces except the single space between " " and copy the full DLL directory
** DllRegisterServer is the export function
3. After hitting ok, go to Options -&gt; Preferences -&gt; Events tab -&gt; check ✔ DLL Entry
4. Hit F9 (few seconds and pause)
</code></pre>

<h2 class="wp-block-heading">TEMPORARY DLL</h2>



<p>suit_32.tmp, is another 64-bit DLL. It dropped from GZIP with the &#8216;license.dat&#8217; binary. Located in %temp% directory</p>



<p><em>C:\Users[username]\AppData\Local\Temp\suit_32.tmp</em></p>



<p>The main purpose of this temporary DLL to initiate persistent with &#8216;license.dat&#8217; and later copy itself to another directory for persistent. </p>



<p><em>Run method: rundll32.exe [filename],update /i:&#8221;LuxuryQuarter\license.dat&#8221;</em></p>



<p>This artifact is also well packed for evasion and anti-analysis purposes. like the &#8216;installer&#8217; no libraries or API to get hint where to breakpoint. To unpack :</p>



<pre class="wp-block-code"><code>1. Load 'suit_32.tmp' in x64dbg
2. Either single or over stepping till reaching &#091;RtlExitUserProcess] API function
3. Check the stack or RDI register for MZ header. 
4. Dump from Memory Map </code></pre>



<p>The unpacked requires addresses matching because it were mapped to memory. </p>

<h2 class="wp-block-heading"> IcedID (license.dat)</h2>



<p>Leaving the beast for last! Even-though it&#8217;s been summited to VT by early March 1st, 2021. It&#8217;s still unrecognized by many vendors that &#8216;license.dat&#8217; is the IcedID. </p>



<p> </p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img data-attachment-id="475" data-permalink="https://aaqeel01.wordpress.com/figure-22/" data-orig-file="https://aaqeel01.wordpress.com/wp-content/uploads/2021/04/figure-22.png" data-orig-size="1089,197" data-comments-opened="0" data-image-meta="{&quot;aperture&quot;:&quot;0&quot;,&quot;credit&quot;:&quot;&quot;,&quot;camera&quot;:&quot;&quot;,&quot;caption&quot;:&quot;&quot;,&quot;created_timestamp&quot;:&quot;0&quot;,&quot;copyright&quot;:&quot;&quot;,&quot;focal_length&quot;:&quot;0&quot;,&quot;iso&quot;:&quot;0&quot;,&quot;shutter_speed&quot;:&quot;0&quot;,&quot;title&quot;:&quot;&quot;,&quot;orientation&quot;:&quot;0&quot;}" data-image-title="figure-22" data-image-description="" data-image-caption="" data-medium-file="https://aaqeel01.wordpress.com/wp-content/uploads/2021/04/figure-22.png?w=300" data-large-file="https://aaqeel01.wordpress.com/wp-content/uploads/2021/04/figure-22.png?w=685" loading="lazy" width="1024" height="185" src="https://aaqeel01.wordpress.com/wp-content/uploads/2021/04/figure-22.png?w=1024" alt="" class="wp-image-475" srcset="https://aaqeel01.wordpress.com/wp-content/uploads/2021/04/figure-22.png?w=1024 1024w, https://aaqeel01.wordpress.com/wp-content/uploads/2021/04/figure-22.png?w=150 150w, https://aaqeel01.wordpress.com/wp-content/uploads/2021/04/figure-22.png?w=300 300w, https://aaqeel01.wordpress.com/wp-content/uploads/2021/04/figure-22.png?w=768 768w, https://aaqeel01.wordpress.com/wp-content/uploads/2021/04/figure-22.png 1089w" sizes="(max-width: 1024px) 100vw, 1024px" /></figure></div>



<p>Huge credit to BinaryDefense team for their efforts building the decryption <a rel="noreferrer noopener" href="https://github.com/BinaryDefense/IcedDecrypt" target="_blank">tool</a> for this part of IcedID and giving it a way on Github. </p>



<p>The unknown &#8216;license.dat&#8217; encrypted binary is running on Task Scheduler with the persistent DLL. As it turns out the unknown binary is also 64-bit DLL. Unlike the previous DLL, this is different kind of beast and this is what IcedID (BokBot) is all about. The decryption does a good job dissecting readable DLL from encrypted binary. However, due to very complication of this part is only possible to disassemble it in IDA, Cutter and other kind of disassemble tools. It&#8217;s not possible to debug it. </p>



<p>Never the less, it&#8217;s possible to reverse engineer the function with proper disassembler to unleash the behavior which by looking at it&#8217;s API list seems to be detectable by Endpoints. The main functions of &#8216;license.dat&#8217; is collecting host and user information and connecting to C2. </p>

						var setDefaultNotApplicableCookie = function() {
							var value = '1---';
							var domain = '.wordpress.com' === location.hostname.slice( -14 ) ? '.rootDomain' : location.hostname;
							cookieLib.setItem( 'usprivacy', value, 24 * 60 * 60, '/', domain );
						};

						var setCcpaAppliesCookie = function( applies ) {
							var domain = '.wordpress.com' === location.hostname.slice( -14 ) ? '.rootDomain' : location.hostname;
							cookieLib.setItem( 'ccpa_applies', applies, 24 * 60 * 60, '/', domain );
						}

						var maybeCallDoNotSellCallback = function() {
							if ( 'function' === typeof window.doNotSellCallback ) {
								return window.doNotSellCallback();
							}

							return false;
						}

						// Look for usprivacy cookie first.
						var usprivacyCookie = cookieLib.getItem( 'usprivacy' );

						// Found a usprivacy cookie.
						if ( null !== usprivacyCookie ) {

							// If the cookie indicates that CCPA does not apply, then bail.
							if ( '1---' === usprivacyCookie ) {
								return;
							}

							// CCPA applies, so call our callback to add Do Not Sell link to the page.
							maybeCallDoNotSellCallback();

							// We're all done, no more processing needed.
							return;
						}

						// We don't have a usprivacy cookie, so check to see if we have a CCPA applies cookie.
						var ccpaCookie = cookieLib.getItem( 'ccpa_applies' );

						// No CCPA applies cookie found, so we'll need to geolocate if this visitor is from California.
						// This needs to happen client side because we do not have region geo data in our $SERVER headers,
						// only country data -- therefore we can't vary cache on the region.
						if ( null === ccpaCookie ) {

							var request = new XMLHttpRequest();
							request.open( 'GET', 'https://public-api.wordpress.com/geo/', true );

							request.onreadystatechange = function () {
								if ( 4 === this.readyState ) {
									if ( 200 === this.status ) {

										// Got a geo response. Parse out the region data.
										var data = JSON.parse( this.response );
										var region      = data.region ? data.region.toLowerCase() : '';
										var ccpa_applies = ['california', 'colorado', 'connecticut', 'delaware', 'indiana', 'iowa', 'montana', 'new jersey', 'oregon', 'tennessee', 'texas', 'utah', 'virginia'].indexOf( region ) > -1;
										// Set CCPA applies cookie. This keeps us from having to make a geo request too frequently.
										setCcpaAppliesCookie( ccpa_applies );

										// Check if CCPA applies to set the proper usprivacy cookie.
										if ( ccpa_applies ) {
											if ( maybeCallDoNotSellCallback() ) {
												// Do Not Sell link added, so set default opt-in.
												setDefaultOptInCookie();
											} else {
												// Failed showing Do Not Sell link as required, so default to opt-OUT just to be safe.
												setDefaultOptOutCookie();
											}
										} else {
											// CCPA does not apply.
											setDefaultNotApplicableCookie();
										}
									} else {
										// Could not geo, so let's assume for now that CCPA applies to be safe.
										setCcpaAppliesCookie( true );
										if ( maybeCallDoNotSellCallback() ) {
											// Do Not Sell link added, so set default opt-in.
											setDefaultOptInCookie();
										} else {
											// Failed showing Do Not Sell link as required, so default to opt-OUT just to be safe.
											setDefaultOptOutCookie();
										}
									}
								}
							};

							// Send the geo request.
							request.send();
						} else {
							// We found a CCPA applies cookie.
							if ( ccpaCookie === 'true' ) {
								if ( maybeCallDoNotSellCallback() ) {
									// Do Not Sell link added, so set default opt-in.
									setDefaultOptInCookie();
								} else {
									// Failed showing Do Not Sell link as required, so default to opt-OUT just to be safe.
									setDefaultOptOutCookie();
								}
							} else {
								// CCPA does not apply.
								setDefaultNotApplicableCookie();
							}
						}
					} );
				};

				// Kickoff initialization.
				if ( window.defQueue && defQueue.isLOHP && defQueue.isLOHP === 2020 ) {
					defQueue.items.push( setupPrivacy );
				} else {
					setupPrivacy();
				}
