<div id="hs_cos_wrapper_module_1577203992502740" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module">

<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p style="text-align: left;"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Packer-Code-Reuse.jpg?width=700&amp;name=Blog-Packer-Code-Reuse.jpg" alt="Packer Code Reuse" loading="lazy" style="width: 700px;" width="700" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Packer-Code-Reuse.jpg?width=350&amp;name=Blog-Packer-Code-Reuse.jpg 350w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Packer-Code-Reuse.jpg?width=700&amp;name=Blog-Packer-Code-Reuse.jpg 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Packer-Code-Reuse.jpg?width=1050&amp;name=Blog-Packer-Code-Reuse.jpg 1050w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Packer-Code-Reuse.jpg?width=1400&amp;name=Blog-Packer-Code-Reuse.jpg 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Packer-Code-Reuse.jpg?width=1750&amp;name=Blog-Packer-Code-Reuse.jpg 1750w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/Blog-Packer-Code-Reuse.jpg?width=2100&amp;name=Blog-Packer-Code-Reuse.jpg 2100w" sizes="(max-width: 700px) 100vw, 700px"><br>One of the core tenets of computer science is code reuse. Why write something new, when code that already exists can be repurposed or changed slightly and then reused for a different situation. This is no different in the world of malware. SystemBC is a family of remote access trojans used to provide access to the local network of a victim and are a beachhead for lateral movement inside that network <a href="#references" rel="noopener"><sup>1</sup></a>. SystemBC has been observed using a variety of packers <a href="#references" rel="noopener"><sup>2</sup></a>. One specific sample <a href="#references" rel="noopener"><sup>3</sup></a> has multiple stages of unpacking which eventually lead to an unpacker stub that has nearly complete code overlap with the unpacking stub used in DLL loaders that are found to deliver Ursnif, IceID, DanaBot, Dridex, Zloader, HanciTor, Valak, and a single example of TrickBot. What follows is a detailed analysis of the packed SystemBC sample up to the unpacker stub in question. From that stub a large set of DLL loaders is discovered via YARA hunting. Finally, the generalized process for dumping the payload from these DLLs is shown.</p>
<!--more-->
<h2>Packed SystemBC Sample</h2>
<p>The first stage of the packer in this sample has some extraneous code in addition to the code that performs initial unpacking. Other than this extraneous code, there are a few key points of interest. The first one being a mutex, "guessHi", that is checked for near the start of execution in the main function. This mutex loaded from a hard coded string along with the call to OpenMutexW is shown in Figure 1.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_01-1.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_01-1.jpg" alt="Code Block with guessHi String and OpenMutexW" style="width: 900px;"></a>
<p style="text-align: center;"><span style="font-weight: bold; font-size: 16px;"><br>Figure 1: Code Block with "guessHi" String and OpenMutexW</span></p>
<p>Another interesting feature of the file is a cryptographic signature block at the end of the file, but according to the PE header, there is no signature directory content. Because of this missing data in the header, this file is not properly signed. The data directories from the PE header with the empty security directory highlighted is shown in Figure 2.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_02.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_02.jpg" alt="No Security Directory Referenced in PE Header" style="width: 1200px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 2: No Security Directory Referenced in PE Header</span></p>
<p>However, looking at the very end of the file in a hex editor reveals that there is a blob of DER encoded binary that is clearly a cryptographic signature for the file. Because this DER data is not referenced in the header as shown above, this signature may have been copied from a different file. The start of this signature blob is shown in Figure 3.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_03.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_03.jpg" alt="Cryptographic Signature Blob" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 3: Cryptographic Signature Blob</span></p>
<p>Looking more closely at the content in this signature, a Gmail address is revealed: "draskovicnono[@]gmail[.]com". This email address is highlighted in Figure 4.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_04.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_04.jpg" alt="Gmail Address in Signature" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 4: Gmail Address in Signature</span></p>
<p>Additionally, this same email address can be found using the search feature on the extracted strings list in the Titanium Platform. This string search is shown in Figure 5.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_05.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_05.jpg" alt="Gmail Address in Extracted Strings" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 5: Gmail Address in Extracted Strings</span></p>
<p>Another important string from this file is the program database string <a href="#references" rel="noopener"><sup>4</sup></a>. In this file that string is "c:\lawHeart\costforward\pagepushwritten.pdb". One can find this particular string in the A1000 under the CodeViews feature. This string is shown in Figure 6.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_06.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_06.jpg" alt="Code Views with Program Database Path" style="width: 1200px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 6: Code Views with Program Database Path</span></p>
<p>Using any of these strings or by pivoting using the ReversingLabs Hash Algorithm (RHA) reveals one other file that is related to the SystemBC sample being analyzed <a href="#references" rel="noopener"><sup>5</sup></a>. The results of an RHA pivot is shown in Figure 7.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_07.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_07.jpg" alt="RHA Pivoting Results" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 7: RHA Pivoting Results</span></p>
<p>However, on closer inspection comparing the bytes of the two files in Hex Fiend <a href="#references" rel="noopener"><sup>6</sup></a>, the only major difference is an additional 4 kilobytes of data which is just a second copy of the already existing file info data. No other significant differences are found, so these two files are effectively the same file. This difference is shown in Figure 8.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_08.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_08.jpg" alt="Difference in the Two Samples" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 8: Difference in the Two Samples</span></p>
<p>The function that specifically performs the unpacking routine in this file is found at the address 0x414ED0. This function contains a set of three calls to kernel32.Sleep. These are a basic anti-analysis technique and need to be circumvented to make analysis easier. These three API function calls are shown in Figure 9 with one of them shown in the disassembler view.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_09.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_09.jpg" alt="API Calls to Kernel32.Sleep" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 9: API Calls to Kernel32.Sleep</span></p>
<p>In the debugger, the number of milliseconds for each of these Sleeps can be modified to zero them out. This is shown in Figure 10.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_10.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_10.jpg" alt="Zero Out Sleeps" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 10: Zero Out Sleeps</span></p>
<p>After the sleeps are neutralized, the first stage of the unpacker writes the next stub to the .data section of the module in memory. The call into that code is shown in Figure 11.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_11.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_11.jpg" alt="Call into Next Stage of Unpacker" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 11: Call into Next Stage of Unpacker</span></p>
<p>As shown in Figure 12, the destination of this call is in the initialized data section of the module with the name .data.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_12.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_12.jpg" alt="Initialized Data Section" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 12: Initialized Data Section</span></p>
<p>The first set of instructions in this next stage is a small loop that decodes the rest of the stub in place. This loop is highlighted in Figure 13.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_13.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_13.jpg" alt="Decoding Loop" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 13: Decoding Loop</span></p>
<p>After the decoding loop has written out the rest of the stub, the resulting instructions are used to write a YARA rule. The specific instructions used are highlighted in Figure 14.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_14.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_14.jpg" alt="Decoded Instructions" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 14: Decoded Instructions</span></p>
<p>The process to write the YARA rule starts with writing out the exact bytes of these instructions. Here, just the first few instructions are shown, but in the actual process the whole set of instructions all the way to and including the first function call at 0x43F3EE is used. The example instructions are the following.</p>
<p style="font-weight: bold; font-size: 16px; text-align: center;"><span style="font-family: 'courier new', courier;">E8 00000000 5B 81EB FD148000 8D83 00108000 8983 CC148000</span></p>
<p>Next, each of the bytes that are specific to locations in this particular file or values that may be unique to this instance of the packer are converted into wildcards and jumps. The bytes that this applies to are shown in red below.</p>
<p style="font-weight: bold; font-size: 16px; text-align: center;"><span style="font-family: 'courier new', courier;">E8 00000000 5B 81EB <span style="color: #f7143f;">FD148000</span> 8D83 <span style="color: #f7143f;">00108000</span> 8983 <span style="color: #f7143f;">CC148000</span></span></p>
<p>The resulting byte string with these jumps and wildcards in place is the following.</p>
<p style="font-weight: bold; font-size: 16px;"><span style="font-family: 'courier new', courier;">E8 00 00 00 00 5B 81 EB [4] 8D 83 [4] 89 83 [4] 8D B3 [4] 89 B3 [4]</span><br><span style="font-family: 'courier new', courier;">8B 46 ?? 89 83 [4] 8D B3 [4] 56 8D B3 [4] 56 6A ?? 68 [4] 8D BB [4]</span><br><span style="font-family: 'courier new', courier;">FF D7</span></p>
<p>The full YARA rule using this byte string is provided at the end of this blog.</p>
<h2>Related DLLs</h2>
<p>Using the YARA rule written using the process above, a retro-hunt is run in the Titanium Platform. The results of this are a large set of hundreds of malicious DLLs that are all packed and utilize the same unpacker stub found in the second stage of the packed SystemBC sample above. These results in the A1000 are shown in Figure 15. This is a very accurate YARA rule in that there are zero false positives found in the result set.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_15.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_15.jpg" alt="Retro-hunt Results" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 15: Retro-hunt Results</span></p>
<p>This is a moderately large set of files, so unpacking each one to determine what malware family is being delivered would be time consuming. Therefore a strategy for grouping the files into clusters which can then have representative files analyzed is a good idea. One effective strategy for this particular data set is to group the files by import hash. Figure 16 shows all the DLLs that share an import hash in descending size of the groups, but excluding single member groups.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_16.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_16.jpg" alt="Files Grouped by Import Hash" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 16: Files Grouped by Import Hash</span></p>
<p>Each cluster can then be examined to determine if the members of a cluster are in fact all delivering the same unpacked payload. Figure 17 shows one cluster that has two different detection names according to automation. The fuzzy hash, ssdeep, is also shown as a sanity check to make sure that the structure of the files in the cluster are nearly the same.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_17-1.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_17-1.jpg" alt="Cluster of Files Sharing One Import Hash" style="width: 1900px;"></a>
<p style="font-weight: bold; font-size: 16px; text-align: center;"><br>Figure 17: Cluster of Files Sharing One Import Hash</p>
<h2>Two Basic Flavors</h2>
<p>Among all these files, there are two basic flavors of packing. The payload binary is written to allocated memory in all cases, but in one case this payload is uncompressed and the other is compressed. The uncompressed payload can simply be dumped directly and then analyzed. However, in the case of the compressed payload, one needs to determine the compression algorithm and then decompress the data before analyzing the resulting binary. This can be done a few different ways. First, the unpacker itself will decompress the binary and overwrite the original DLL's module. After that, the DLL can be dumped and analyzed. Alternatively, one can, as noted earlier, determine the algorithm and decompress the data. However, there is an easier, more straightforward method using the Titanium Platform. The first step is to open the DLL in x64dbg and run the executable up to the entry point. From there, one sets a breakpoint at the return instruction in kernel32.VirtualAlloc. This breakpoint is shown in Figure 18.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_18.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_18.jpg" alt="Breakpoint on VirtualAlloc" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 18: Breakpoint on VirtualAlloc</span></p>
<p>Once set, run the file until that breakpoint is reached. When execution is on this return instruction, observe the address of the newly allocated memory in the EAX register. An example of this is shown in Figure 19 with the address of the newly allocated memory highlighted.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_19.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_19.jpg" alt="Newly Allocated Memory" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 19: Newly Allocated Memory</span></p>
<p>Once execution has arrived at this point, set one more breakpoint on VirtualProtect. The reason for waiting until execution is deep into the unpacker stubs before setting the breakpoint on VirtualProtect is that often there are benign calls to VirtualProtect that one would need to pass over before even getting to the initial VirtualAlloc, and that can get tedious.<br><br>For each call to VirtualAlloc, follow that new memory address in x64dbg's dump then run to the next call to VirtualAlloc. Each time examine the new data that is written to the allocated memory. This is where eventually the payload binary is written to. In the first flavor of unpacking mentioned above, it will be crystal clear when the MZ magic number along with the DOS stub appear in the allocated memory. However, the second flavor where the payload is compressed will also be quite recognizable, if not a bit garbled. An example of this compressed flavor is shown in Figure 20.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_20.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_20.jpg" alt="Compressed Payload" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 20: Compressed Payload</span></p>
<p>This is where the Titanium Platform comes in handy. Just dump this memory to a file and upload it to the A1000. After it has been analyzed, navigate to the extracted files and drill into the extracted file until the payload DLL is revealed. This extracted DLL is shown in Figure 21.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_21.jpg" class="lightbox-image"><img src="https://www.reversinglabs.com/hubfs/Blog/packer_code_reuse_figure_21.jpg" alt="Decompressed Payload" style="width: 1400px;"></a>
<p style="text-align: center;"><br><span style="font-weight: bold; font-size: 16px;">Figure 21: Decompressed Payload</span></p>
<p>Analysis of all the clusters of DLLs in the retro-hunt results in this manner, along with all the import hashes with only one file, reveals that this unpacker code has been used to deliver many malware families over the past year. These families include Ursnif, IceID, DanaBot, Dridex, Zloader, HanciTor, Valak, and a single example of TrickBot. A full list of file hashes in clusters by payload malware family is provided below.</p>
