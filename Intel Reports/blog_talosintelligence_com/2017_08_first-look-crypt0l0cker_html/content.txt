    <h1 class="text-center">Taking the FIRST look at Crypt0l0cker</h1>

    <div class="text-center m-1">
        <div class="post-author">
            <span>By </span>
                <a href="https://blog.talosintelligence.com/author/alexander-chiu/">Alexander Chiu</a>

        <time class="post-datetime" datetime="August 3, 2017 11:35">

        <div class="m-3">

    <section class="post-content-wrapper mt-5">
        <div class="post-content">
            <p><i>This post is authored by <a href="https://blogs.cisco.com/author/matthewmolyett">Matthew Molyett</a>.</i></p><h2 id="executive-summary-in-march-talos-reported-on-the-details-of-crypt0l0cker-based-on-an-extensive-analysis-i-carried-out-on-the-sample-binaries-binaries-plural-because-as-noted-in-the-original-blog-the-crypt0l0cker-payload-leveraged-numerous-executable-files-which-shared-the-same-codebase-those-executables-had-nearly-identical-functions-in-each-but-identifying-all-of-those-functions-repeatedly-is-tedious-and-draws-time-away-from-improving-the-analysis-enter-first-the-function-identification-and-recovery-signature-tool-released-by-talos-in-december-2016-">Executive Summary In March, Talos reported on the details of <a href="https://blog.talosintelligence.com/crypt0l0cker-torrentlocker-old-dog-new">Crypt0l0cker</a> based on an extensive analysis I carried out on the sample binaries. Binaries -- plural -- because, as noted in the original blog, the Crypt0l0cker payload leveraged numerous executable files which shared the same codebase. Those executables had nearly identical functions in each, but identifying all of those functions repeatedly is tedious and draws time away from improving the analysis. Enter <a href="https://blog.talosintelligence.com/project-first-share-knowledge-speed-up">FIRST</a>, the Function Identification and Recovery Signature Tool released by Talos in December 2016.</h2><p><a href="http://first-plugin.us/">FIRST</a> allowed me to port my analysis from the unpacking dll to the payload file instantly. Once I was satisfied my analysis across both files, I was then handed a suspected previous version of the sample. <a href="https://github.com/vrtadmin/FIRST">FIRST</a> was able to identify similar code across the versions and partially port the analysis back to the older file. When the next version of Crypt0l0cker comes out, I will be able to get a jump on my analysis by using FIRST to port that work forward to the similar code. You can use it to port my work to your sample as well. I will demonstrate doing just that with a Crypt0l0cker sample which appeared on VirusTotal in April 2017, more than a month after the Talos blog about it. There has been no targeted analysis of this file to provide background for this post.</p><h2 id="locating-the-sample-procuring-a-malware-sample-of-a-known-family-without-analyzing-it-can-feel-like-a-heavy-challenge-to-overcome-thankfully-talos-can-leverage-threat-grid-sandbox-reports-of-suspected-malware-samples-that-we-receive-such-reports-can-be-scanned-for-family-iocs-per-our-previous-analysis-into-crypt0l0cker-the-infection-status-of-that-version-is-stored-in-a-file-named-ewiwobiz-by-searching-cisco-threat-grid-telemetry-for-files-which-created-ewiwobiz-i-identified-a-file-which-was-probably-a-crypt0l0cker-executable-">Locating the Sample Procuring a malware sample of a known family without analyzing it can feel like a heavy challenge to overcome. Thankfully, Talos can leverage Threat Grid sandbox reports of suspected malware samples that we receive. Such reports can be scanned for family IOCs. Per our previous analysis into Crypt0l0cker, the infection status of that version is stored in a file named ewiwobiz. By searching <a href="http://www.cisco.com/c/en/us/products/security/threat-grid/index.html">Cisco Threat Grid</a> telemetry for files which created ewiwobiz, I identified a file which was probably a Crypt0l0cker executable.<br></h2><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-euyICRMwsSM/WYIx1ZvKwoI/AAAAAAAABM8/KgkQBcrq5dMhrAhyToEBL4P0xyqFTuFjgCLcBGAs/s640/image2.png" class="kg-image" alt loading="lazy"></figure><p><br> With a report to investigate, I needed to procure the actual sample. My sandbox report shows that the suspected Crypt0l0cker file is nearly 400 kb and likely a <a href="https://en.wikipedia.org/wiki/Nullsoft_Scriptable_Install_System">Nullsoft Installer</a> file, which is a common packager. Static file information gives me the file hash which arms me with the ability to continue my investigation on VirusTotal.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-oKMj1cqh-z0/WYIx5VW2cAI/AAAAAAAABNA/RnUDU6lUjeYAwANTZwTYlJMb2i7ukZWsACLcBGAs/s640/image1.png" class="kg-image" alt loading="lazy"></figure><p><br> While the sample is clearly malicious, my VirusTotal inspection does not suggest that the sample belongs to any known family. No detections refer to Crypt0l0cker, TorrentLocker, a listed alias in the original Talos blog, nor <a href="https://www.bleepingcomputer.com/news/security/crypt0l0cker-ransomware-is-back-with-campaigns-targeting-europe/">Teerac</a>.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-utXvjaRAx8k/WYIx9MHJPRI/AAAAAAAABNE/E2gT686OZAYCmh41Nnxj2cvY1lJT78kjACLcBGAs/s640/image13.png" class="kg-image" alt loading="lazy"></figure><p><br> With a file sample in hand, and no static indication that I have located Crypt0l0cker, I move onto FIRST to discover how similar it is to known files.</p><h2 id="exploring-the-sample">Exploring the Sample</h2><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-Mi4UogqjztU/WYIyBFx9aWI/AAAAAAAABNI/EajhafITVqgVlehSU1RO7B1evUGDe5fvQCLcBGAs/s400/image11.png" class="kg-image" alt loading="lazy"></figure><p><br> As the FIRST client code is an IDA Pro plugin, my first step was opening the file in my local IDA copy and allow auto-analysis. Upon completion, the start function was displayed in front of me at the graph view. I opened up the graph view context menu and requested FIRST lookups for all of the discovered functions.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-u0JSYLtMkOc/WYIyFZl2OkI/AAAAAAAABNM/bBS9V2YI73ksFhz0sC460w9bYkdqBeTNgCLcBGAs/s400/image4.png" class="kg-image" alt loading="lazy"></figure><p><br> After a minute, the FIRST display shows that 13 of the functions have been previously identified and uploaded.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-4OUvJBrEry4/WYIyJfv6i7I/AAAAAAAABNQ/08JWupVSFtcP4HsI5k7rFH1V6xNdUaL-gCLcBGAs/s640/image5.png" class="kg-image" alt loading="lazy"></figure><p><br> Expanding the matched functions displays the metadata associated with that function, including a proposed name and function prototype. Notice that the files detected in this installer have been named to draw attention to the fact that these functions are known to be in NullSoft Installers. I had previously marked up a different NullSoft Installer before and uploaded significant functions from it to assure that I would not do so again. In general, a malware analyst is wasting any time spent inspecting such a file. Identifying when a packer is in use and moving along to the true payload is a much better use of time.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-WwTheAuVFak/WYIyZJv_gSI/AAAAAAAABNU/t4OvEsDlGsAB0Lq1r6-qE63FsC57uq51ACLcBGAs/s640/image9.png" class="kg-image" alt loading="lazy"></figure><p><br> Check the Select Highest Ranked checkbox and click Apply. The function names get applied across the database and we can see clearly that the sandbox analysis was correct. This file is a packer and we need to extract the original.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-wu5Sf5vKmLw/WYIydCGTFaI/AAAAAAAABNY/wC0tPwCcbygH5IKHslQp7yM9aG0o83IuwCLcBGAs/s640/image10.png" class="kg-image" alt loading="lazy"></figure><p><br></p><h2 id="unpacking-the-sample-i-admit-that-at-this-point-i-cheated-to-perform-the-unpack-from-previous-extraction-of-crypt0l0cker-files-protected-with-nullsoft-i-already-knew-that-the-install-script-consisted-of-consuming-multiple-encrypted-blobs-internally-decrypt-the-payload-and-run-it-via-process-hollowing-as-such-allowing-it-to-run-debugged-and-breaking-on-writeprocessmemory-should-present-the-payload-buffer-to-me-">Unpacking the Sample I admit that at this point I cheated to perform the unpack. From previous extraction of Crypt0l0cker files protected with NullSoft I already knew that the install script consisted of consuming multiple encrypted blobs, internally decrypt the payload, and run it via <a href="https://www.endgame.com/blog/technical-blog/ten-process-injection-techniques-technical-survey-common-and-trending-process">Process Hollowing</a>. As such, allowing it to run debugged and breaking on WriteProcessMemory should present the payload buffer to me.</h2><p>There was a complication though, because the install script loaded and unloaded System.dll many times. The ModLoad notification caused the debugger to consume the majority of the process cycles, effectively causing a denial of service against the debugger. I allowed this system to run for over an hour without getting beyond this delay.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-u4ePEhoyPzE/WYIytPwTcYI/AAAAAAAABNc/7h24hqxfyuQzTsMgd-ZQBm2M9DbDx5MZACLcBGAs/s640/image3.gif" class="kg-image" alt loading="lazy"></figure><p><br> By disabling the ModLoad notification via `<a href="http://blog.opensecurityresearch.com/2013/12/getting-started-with-windbg-part-2.html">sxi ld</a>` I could get my debugger to allow the System.dll file to be loaded without triggering the significant extra processing. Crypt0l0cker then was able to spike up to 99% of the CPU use to, rather than the debugger holding on to 80%.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-a8oLld_LJew/WYIy0DKKYoI/AAAAAAAABNg/0AIyISQ4waIw80ib10_c81koPBXbMBCTACEwYBhgL/s640/image7.gif" class="kg-image" alt loading="lazy"></figure><p><br> I dumped out the PE image file and prepared to continue with FIRST.</p><h2 id="exploring-the-real-file">Exploring the Real File</h2><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-BzLDs8blAno/WYIy9xntjeI/AAAAAAAABNk/3fSwxj0RcX4M_WshkNL92Wn9_NnBwfuogCLcBGAs/s640/image6.png" class="kg-image" alt loading="lazy"></figure><p>Again, the first step of using FIRST was opening the extracted file in IDA Pro. This file was built as a Windows GUI file on top of the Visual Studio C runtime. Thus, the runtime was identified during auto-analysis and I was left with a graph view displaying the _WinMain@16 function. Using the FIRST command from the context menu, I checked for the existence of metadata for just that one function. It was discovered as Crytp0l0cker_WinMain@16. Looking pretty likely that this is, in fact, Crytp0l0cker.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-iqMc8pEpYIo/WYIzEaYGJwI/AAAAAAAABNo/kydtR4b91DYUKS6zel9wk5eh1-aVPW_bgCLcBGAs/s640/image12.gif" class="kg-image" alt loading="lazy"></figure><p><br> With confidence that FIRST will be useful, since it had a result for _WinMain@16, I began the search for the full file. At 436 functions this is not a quick lookup, so go get a fresh cup of coffee and let FIRST work. Since this file uses a known runtime, the runtime files are also known to FIRST. You can filter those functions with the Show only "sub_" functions checkbox.</p><figure class="kg-card kg-image-card"><img src="https://blog.talosintelligence.com/content/images/-m2cWfvRY9eQ/WYI0AaZV7lI/AAAAAAAABNw/GXoJMZA_GDUIDV5ZmDIy4vU_7adz7qnMQCLcBGAs/s640/image8.gif" class="kg-image" alt loading="lazy"></figure><p><br> After FIRST completes 78 new function markups are applied out of 295 total known functions. With 78 Crytp0l0cker_* functions identified, you can now dig in on the shoulder of days of professional malware analysis.</p><h2 id="conclusion-first-provides-the-ability-to-share-your-work-from-one-file-to-a-similar-file-whether-that-other-file-is-a-previous-or-future-version-or-even-an-additional-step-in-the-module-execution-when-opening-up-a-new-file-first-can-provide-hints-as-to-whether-the-file-is-interesting-or-just-needs-to-be-unpacked-when-finally-extracting-a-new-embedded-binary-first-can-migrate-your-notes-from-the-current-file-to-the-shared-code-in-the-new-file-use-first-to-save-your-notes-share-your-discoveries-and-speed-up-your-next-analysis-">Conclusion FIRST provides the ability to share your work from one file to a similar file, whether that other file is a previous or future version or even an additional step in the module execution. When opening up a new file, FIRST can provide hints as to whether the file is interesting or just needs to be unpacked. When finally extracting a new, embedded binary, FIRST can migrate your notes from the current file to the shared code in the new file. Use FIRST to save your notes, share your discoveries, and speed up your next analysis.</h2><h2 id="ioc">IOC</h2><h3 id="file-hash-d845e4f2292ba78a993dbbf6f1317894ce1a795c096d7959f3d718e583f1cea3">File Hash d845e4f2292ba78a993dbbf6f1317894ce1a795c096d7959f3d718e583f1cea3</h3>
