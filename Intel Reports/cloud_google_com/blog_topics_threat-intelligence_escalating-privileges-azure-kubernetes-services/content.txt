<h2><span style="vertical-align:baseline">Introduction</span></h2>
<p><span style="vertical-align:baseline">Kubernetes can be difficult to harden. Enforcing authentication for internal services, applying granular NetworkPolicies, and restricting unsafe workloads with Pod Security are now table stakes for preventing post-exploitation activity that can compromise an entire cluster. These security configurations that limit attack surface help prevent against known and unknown attacks alike.</span></p>
<p><span style="vertical-align:baseline">Azure Kubernetes Services clusters using "Azure CNI" for the "Network configuration" and "Azure" for the "Network Policy" were affected by this privilege escalation vulnerability. An attacker with command execution in a Pod running within an affected Azure Kubernetes Services cluster could download the configuration used to provision the cluster node, extract the transport layer security (TLS) bootstrap tokens, and perform a TLS bootstrap attack to read all secrets within the cluster. This attack did not require the Pod to be running with </span><code style="vertical-align:baseline">hostNetwork</code><span style="vertical-align:baseline"> set to </span><code style="vertical-align:baseline">true</code><span style="vertical-align:baseline"> and does not require the Pod to be running as </span><code style="vertical-align:baseline">root</code><span style="vertical-align:baseline">.</span></p>
<p><span style="vertical-align:baseline">Mandiant disclosed this vulnerability to Microsoft via the MSRC vulnerability disclosure program, and Microsoft has fixed the underlying issue.</span></p>
<h2><span style="vertical-align:baseline">Background</span></h2>
<h3><span style="vertical-align:baseline">Cluster Network Access</span></h3>
<p><span style="vertical-align:baseline">Kubernetes clusters are often deployed without consideration for the possibility of an attacker with code execution within a Pod. This can happen in many ways, including via existing vulnerabilities in running workloads, continuous integration build jobs, or a compromised developer account. In these scenarios, </span><a href="https://kubernetes.io/docs/concepts/services-networking/network-policies/" rel="noopener" target="_blank"><span style="text-decoration:underline;vertical-align:baseline">NetworkPolicies</span></a><span style="vertical-align:baseline"> are the first line of defense to prevent post-exploitation activity.</span></p>
<p><strong style="vertical-align:baseline">Without NetworkPolicies in place, you should assume a compromised Pod can access any network resource any other Pod on the cluster can access.</strong><span style="vertical-align:baseline"> This could include the local Redis cache for another Pod, managed databases running in your cloud provider, or even your on-premises network. When these services require authentication and are configured correctly, this is a relatively low-risk configuration —a vulnerability in one of these services would be necessary for an attacker to exploit.</span></p>
<p><span style="vertical-align:baseline">Often overlooked among these accessible services are the internal cloud services used to configure the worker nodes on which these Pods are running. The </span><a href="https://cloud.google.com/compute/docs/metadata/overview"><span style="text-decoration:underline;vertical-align:baseline">metadata server</span></a><span style="vertical-align:baseline">, accessible at </span><a href="http://169.254.169.254" rel="noopener" target="_blank"><span style="text-decoration:underline;vertical-align:baseline">http://169.254.169.254</span></a><span style="vertical-align:baseline"> across cloud providers, provides machine configuration and often the credentials used to identify the machine to the cloud provider. Generally speaking, direct access to the metadata server is equivalent to having the same permissions the machine does.</span></p>
<p><span style="vertical-align:baseline">Metadata server attacks are not new and cloud providers do a lot to limit their attack surface by default. Most cloud providers do a combination of recommending NetworkPolicies to block access to </span><code style="vertical-align:baseline">169.254.0.0/16</code><span style="vertical-align:baseline">, limiting the default privileges assigned to worker nodes, and providing alternatives to allow Pods to have their own credentials separate from the instance they're running on.</span></p>
<h3><span style="vertical-align:baseline">Bootstrapping Kubernetes Nodes</span></h3>
<p><span style="vertical-align:baseline">The difficulty in bootstrapping trust in Kubernetes Nodes is well known among the Kubernetes security community. The kubelet that runs on Kubernetes Nodes needs a TLS certificate signed by the control plane Certificate Authority (CA) to operate safely. But in a large distributed system where nodes (or virtual machines [VMs]) are constantly created and destroyed, how should that certificate be provisioned onto the VM? One option on cloud services is to use the </span><a href="https://cloud.google.com/compute/docs/metadata/overview"><span style="text-decoration:underline;vertical-align:baseline">metadata server</span></a><span style="vertical-align:baseline">, accessible at </span><a href="http://169.254.169.254" rel="noopener" target="_blank"><span style="text-decoration:underline;vertical-align:baseline">http://169.254.169.254</span></a><span style="vertical-align:baseline"> across cloud providers, to deliver a static token to provisioned VMs that can be used to prove the VM should be part of the cluster and issued a kubelet certificate.</span></p>
<p><span style="vertical-align:baseline">The problem with that approach is that these metadata services are network accessible and could cause theft of the token if the attacker has network access, such as through a server-side request forgery (SSRF) vulnerability. The Google Kubernetes Engine (GKE) security team </span><a href="https://www.youtube.com/watch?v=2XCm7vveU5A" rel="noopener" target="_blank"><span style="text-decoration:underline;vertical-align:baseline">presented about this style of attack with Shopify at Kubecon in 2018</span></a><span style="vertical-align:baseline">. With possession of these bootstrap tokens, an attacker can create a kubelet certificate for their own machine and use those credentials to attack the control plane, steal secrets, and interfere with the workloads scheduled on their malicious "node."</span></p>
<p><span style="vertical-align:baseline">While protecting these tokens by denying applications access to the metadata server can help, the managed Kubernetes industry has evolved beyond simple token provisioning as a means for identifying VMs for critical security decisions. </span></p>
<p><span style="vertical-align:baseline">Taking GKE as an example, we can see this evolution happening. GKE first protected clusters against these kinds of bootstrap token-stealing attacks in February 2018 with the launch of the </span><a href="https://cloud.google.com/kubernetes-engine/docs/how-to/protecting-cluster-metadata#concealment"><span style="text-decoration:underline;vertical-align:baseline">metadata concealment proxy</span></a><span style="vertical-align:baseline">, which was </span><a href="https://www.youtube.com/watch?v=2XCm7vveU5A" rel="noopener" target="_blank"><span style="text-decoration:underline;vertical-align:baseline">presented at Kubecon that year</span></a><span style="vertical-align:baseline">. That temporary solution was </span><a href="https://cloud.google.com/blog/products/identity-security/exploring-container-security-bringing-shielded-vms-to-gke-with-shielded-gke-nodes"><span style="text-decoration:underline;vertical-align:baseline">replaced in September 2019</span></a><span style="vertical-align:baseline"> with a cryptographically verifiable virtual Trusted Platform Module (vTPM)-backed trust bootstrap process that operates </span><a href="https://cloud.google.com/kubernetes-engine/docs/how-to/shielded-gke-nodes"><span style="text-decoration:underline;vertical-align:baseline">as part of shielded nodes</span></a><span style="vertical-align:baseline">. Shielded nodes have been enabled by default for all newly created GKE clusters since January 2021. It is enabled for all GKE Autopilot clusters and cannot be disabled.</span></p>
<p><span style="vertical-align:baseline">GKE shielded nodes remove the risk of bootstrap token theft instead of concealing it. Instead of relying on possession of a static token to authenticate and authorize a request for a new kubelet certificate, the VM requests an attestation from the VM's vTPM, which is then verified by the control plane before issuing the kubelet certificate. Generating this attestation requires the attacker to have access to a root-owned device on the VM, which is a significantly higher bar than network access to the metadata server. Even with that access, the attacker can only generate an attestation for that node, not any node in the cluster. A new attestation will need to be produced to obtain a new kubelet certificate when the existing one expires, requiring the attacker to maintain presence on the node.</span></p>
<h2><span style="vertical-align:baseline">Azure WireServer and the HostGAPlugin</span></h2>
<p><span style="vertical-align:baseline">Azure </span><a href="https://learn.microsoft.com/en-us/azure/virtual-network/what-is-ip-address-168-63-129-16" rel="noopener" target="_blank"><span style="text-decoration:underline;vertical-align:baseline">WireServer</span></a><span style="vertical-align:baseline"> is an undocumented component of Azure used internally by the platform for several reasons. At the time of writing, the best official resource for the WireServer's functionality is Azure's </span><a href="https://github.com/Azure/WALinuxAgent" rel="noopener" target="_blank"><span style="text-decoration:underline;vertical-align:baseline">WALinuxAgent</span></a><span style="vertical-align:baseline"> repository, which handles provisioning of Linux instances and interactions with Azure Fabric.</span></p>
<p><span style="vertical-align:baseline">CyberCX published </span><a href="https://cybercx.co.nz/blog/azure-ssrf-metadata/" rel="noopener" target="_blank"><span style="text-decoration:underline;vertical-align:baseline">research</span></a><span style="vertical-align:baseline"> in May 2023 that included an interesting attack path with the undocumented HostGAPlugin. Given access to the WireServer (</span><code style="vertical-align:baseline">http://168.63.129.16/machine/?comp=goalstate</code><span style="vertical-align:baseline">) and HostGAPlugin (</span><code style="vertical-align:baseline">http://168.63.129.16:32526/vmSettings</code><span style="vertical-align:baseline">) endpoint, an attacker could retrieve and decrypt the settings provided to a number of extensions, including the "</span><a href="https://learn.microsoft.com/en-us/azure/virtual-machines/extensions/custom-script-linux" rel="noopener" target="_blank"><span style="text-decoration:underline;vertical-align:baseline">Custom Script Extension</span></a><span style="vertical-align:baseline">," a service used to provide a virtual machine its initial configuration.</span></p>
<h2><span style="vertical-align:baseline">Vulnerability Exploitation</span></h2>
