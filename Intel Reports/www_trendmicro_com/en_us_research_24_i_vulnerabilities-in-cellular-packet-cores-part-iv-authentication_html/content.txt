	<p class="article-details__description">Our research reveals two significant vulnerabilities in Microsoft Azure Private 5G Core (AP5GC), both of which have now been resolved and are discussed in this blog post.</p>

		<div>
			<p><b>Report highlights:</b></p>
<ul>
<li><span class="rte-red-bullet" style="font-weight: normal;">This article explores two vulnerabilities in Microsoft Azure Private 5G Core (AP5GC): <a name="_Hlk177569239" id="_Hlk177569239"></a>CVE-2024-20685 and ZDI-CAN-23960.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Successful exploitation of CVE-2024-20685 can lead to potential service outages, while ZDI-CAN-23960 can lead to the disruption of network operations.</span></li>
<li><span class="rte-red-bullet" style="font-weight: normal;">Our analysis reveals that both exploitation of the two vulnerabilities can result in varying degrees of Denial-of-Service (DoS). Packet cores are critical network infrastructure nodes, which means the impact extends beyond the directly affected device and can potentially disrupt a broader network segment.</span></li>
</ul>
<h1><span class="body-subhead-title">Overview</span></h1>
<p>This article is the latest in a <a href="https://ctone.com/threat-intelligence/threat-research/" target="_blank">series</a> about vulnerabilities in cellular packet cores, using various commercial and open-source products as examples. Our focus throughout these articles has been identifying systemic issues and attack vectors rather than individual vulnerabilities. We’ve discussed how the <a href="https://www.trendmicro.com/vinfo/us/security/news/internet-of-things/plague-private-5g-networks">lack of cross-checks</a> between components can lead to session injection, how user devices could <a href="/en_us/research/23/i/attacks-on-5g-infrastructure-from-users-devices.html">exploit tunneling</a> to attack infrastructure, and how <a href="/en_us/research/23/j/asn1-vulnerabilities-in-5g-cores.html">insufficient isolation</a> between components can result in the exploitation of internal vulnerabilities.</p>
<p>This article explores two vulnerabilities in Microsoft Azure Private 5G Core (AP5GC) discovered by our research team:</p>
<ol>
<li><span style="font-weight: normal;">A crafted signaling message can crash the control plane.</span></li>
<li><span style="font-weight: normal;">A crafted signaling message can disconnect and replace attached base stations.</span></li>
</ol>
<p>These vulnerabilities are specific to the implementation of AP5GC. However, the underlying issue that allows them to be exploited is a systemic weakness: the lack of mandatory authentication procedures between base stations and packet-cores.</p>
<p>Microsoft fixed CVE-2024-20685 (Attack Vector: Network) in AP5GC version 2403.0-2 and assured that ZDI-CAN-23960 (Attack Vector: Adjacent) will be addressed at a future date.</p>
<h1><span class="body-subhead-title">Acknowledgments</span></h1>
<p>We are grateful to Dr. Shumin Chuang of Telecom Technology Center (TTC), Taiwan, for her guidance and support during this research.</p>
<h1><span class="body-subhead-title">Cellular Network Architecture</span></h1>

		</div>
	

</div>
            <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/fig1.png" alt="The end-to-end architecture of cellular networks"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 1. The end-to-end architecture of cellular networks</figcaption>
			
		</div>
	</figure>

</div>
            <div>




    
    
    <div class="richText">
	
    


	
		<div>
			<p>Cellular devices like cell phones and cellular IoT devices connect to a radio base station via RF (Radio Frequency) signals. The base station then converts these RF signals into IP packets. The base stations are connected to a central infrastructure unit, called the packet-core, over IP links. These backhaul IP links are usually wired.</p>
<p>Cellular network architecture is divided into two planes: the control plane and the data plane. The control plane handles signaling between user equipment (UE) and the packet core, managing tasks such as initial UE attachment, authentication, data session setup, etc. The data plane (user plane) deals with user application traffic.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/fig2.png" alt="Cellular network architecture"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 2. Cellular network architecture</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The connection between the base station and packet core also reflects this division. The control plane runs over the SCTP transport protocol. User plane traffic is encapsulated in UDP, with port number 2152.</p>
<p>The Layer 7 equivalent protocol in the 5G control plane is the Next Generation Application Protocol (NGAP). It runs over SCTP port 38412. NGAP defines network messages for various signaling procedures, which are serialized in Abstract Syntax Notation One (ASN.1) format.</p>
<p>Note: In 5G terminology, base stations are called gNBs or gNodeBs. On the packet-core side, the control plane node is called the access and mobility management function (AMF), and the user plane node is called the user plane function (UPF).</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/fig3.png" alt="Control message Stack"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 3. Control message Stack</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Both the base station and the packet core need ASN.1 parsers. ASN.1 parsing of 5G control messages can get complex. We wrote about it <a href="/en_us/research/23/j/asn1-vulnerabilities-in-5g-cores.html">here</a>, describing how user devices can potentially bring down a packet-core by triggering ASN.1 vulnerabilities. In that article, the technique was to trigger control plane ASN.1 vulnerabilities from the user plane. However, the ASN.1 vulnerabilities described in this article are triggered directly from the control plane.</p>
<h1><span class="body-subhead-title">5G Message Flow</span></h1>
<p>An SCTP connection is established between the base station and packet core to carry NGAP messages. Figure 4 shows the typical message exchange after that.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/fig4.png" alt="The typical message exchange after an SCTP connection is established"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 4. The typical message exchange after an SCTP connection is established</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The base station starts a control-connection setup with 5G Core by sending an NGSetUpRequest.</p>
<p>Upon receiving this message, the 5GC validates the parameters and sends ‘NGSetUpResponse’ with success or failure status. Authentication is not mandatory in this step.</p>
<p>Meanwhile, a UE may establish an RF link with the base station. Once the RF link is established, the UE sends a registration request to the base station. The base station adds additional information, encapsulates it into an NGAP packet, and forwards it to the packet core.</p>
<p>After that, mutual authentication between the UE and the packet core takes place (Figure 4, packets 4, 5), followed by security mode negotiation (packets 6, 7), and finally, the data session is set up (packets 10, 11).</p>
<p>The diagram in Figure 5 shows the message flow.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/Fig5.png" alt="UE Registration Message Flow"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 5. UE Registration Message Flow</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p><b>Key points:</b></p>
<p>Authentication is not mandatory between base stations and packet-cores.</p>
<ul>
<li><span class="rte-red-bullet" style="font-weight: normal;">3GPP standards do not have a provision for NGAP-based authentication. Instead, it is <i style="">recommended</i> that the base station and 5GC set up an IPSec link or use certificate-based authentication.</span></li>
</ul>
<p>Registration is the first message from UEs to 5GC, sent <i>before</i> authentication and security capabilities are set up.</p>
<ul>
<li><span class="rte-red-bullet" style="font-weight: normal;">This means the Registration Request is sent in plain text and is processed by 5GC without authentication.</span></li>
</ul>
<p>The vulnerabilities described in this article are triggered within the first three NGAP messages (see Figure 4) before authentication and encryption are set up for UEs.</p>

		</div>
	

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<h1><span class="body-subhead-title">Microsoft Azure Private 5G Core (AP5GC)</span></h1>
<p><a href="https://azure.microsoft.com/en-us/products/private-5g-core">Microsoft Azure Private 5G Core</a> is Microsoft’s 5G packet core designed for private networks. The packet processing functionality runs on Azure Stack Edge (ASE) hardware, while most configuration and management are from Azure cloud. Both ASE hardware and Azure require a subscription.</p>
<p>AP5GC is part of Microsoft’s Azure for Operator’s portfolio.</p>
<p><b>Authentication in AP5GC</b></p>
<p>No authentication is necessary between the base station and AP5GC in the default setting. In fact, we could not find any information related to this authentication in the <a href="https://learn.microsoft.com/en-us/azure/private-5g-core/" target="_blank">documentation</a> or the AP5GC configuration portal.</p>
<h1><span class="body-subhead-title">Attack Preconditions and Preparation</span></h1>
<p>The preconditions for both described attacks are the same.</p>
<p><b>Precondition-1</b></p>
<p>AP5GC requires users to set up a gateway for gNBs to connect with AMF. An attacker must have connectivity to this gateway to be able to communicate with AP5GC.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/Fig6.png" alt="AP5GC Gateway Configuration"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 6. AP5GC Gateway Configuration</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Preparation: Our attack machine is connected to the gateway. We used an SCTP scan to identify the AMF interface.</p>
<p><b>Precondition-2</b></p>
<p>The attacker needs to know PLMN, which is configured in packet-core (AP5GC). Cell towers broadcast PLMN in clear text so that UEs can identify it.</p>
<p>Preparation: Software-defined Radios (SDRs) can be used to capture broadcasted messages from cell towers and find the PLMN.</p>
<p><b>Precondition-3</b></p>
<p>The attacker needs to know the Slice Service Type (SST) and (Slice Differentiator) SD.</p>
<p>SST values are generally standard 1,2, or 3. SD is optional. <a href="https://learn.microsoft.com/en-us/azure/private-5g-core/collect-required-information-for-private-mobile-network#collect-the-required-information-for-a-network-slice" target="_blank">Microsoft Documentation</a><u></u></p>
<p>Preparation: 1 is the most common SST. We used SST=1 and empty SD on our gNB.</p>
<h1><span class="body-subhead-title">CVE-2024-20685: Azure Private 5G Core Denial of Service Vulnerability</span></h1>
<p>By sending a crafted malformed UE Registration message, an attacker can cause a denial-of-service in AP5GC. All cellular devices in the network will lose connectivity.</p>
<p>This vulnerability is assigned CVE ID <a href="https://msrc.microsoft.com/update-guide/vulnerability/CVE-2024-20685" target="_blank">CVE-2024-20685</a>. Its ZDI tags are <a href="https://www.zerodayinitiative.com/advisories/ZDI-24-362/" target="_blank">ZDI-24-362</a> and <a href="https://www.zerodayinitiative.com/advisories/ZDI-24-362/" target="_blank">ZDI-CAN-23397</a>. The figure gives an overview of the attack seen from the packet trace.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/Fig7.png" alt="Packet capture of CVE-2024-20685 in action"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 7. Packet capture of CVE-2024-20685 in action</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p><b>Attack Setup</b></p>
<p>It is possible to manually craft and send the messages in Steps 1, 2, and 3 from a generic SCTP client. However, it is easier to send these messages with software-defined radios (SDRs) designed to connect with 5G cores.</p>
<p><a href="https://www.srslte.com/" target="_blank">srsRAN</a> is an SDR package that supports both software UE (srsUE) and gNB (srsRAN gNB). We installed srsRAN on a Linux machine. srsRAN supports the <a href="https://docs.srsran.com/projects/project/en/latest/tutorials/source/srsUE/source/index.html" target="_blank">simulation</a> of the radio interface, so no radio hardware is needed.</p>
<p>Our srsUE – srsRAN gNB – AP5GC connection setup is as follows.</p>
<p>192.168.1.101 is an actual radio hardware base station.</p>
<p>192.168.1.102 is the Linux machine with srsRAN.</p>
<p>192.168.1.100 is the gateway switch.</p>
<p>192.168.1.10 is AMF.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/Fig8.png" alt="Attack Topology of CVE-2024-20685"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 8. Attack Topology of CVE-2024-20685</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The <i>Modifier</i> module intercepts InitialUEMessage Registration sent from gNB and changes the payload to the one we crafted. An NFQ-based Python script does this job.</p>
<p><b>Results</b></p>
<p>We modified an “InitialUEMessage Registration Request” with a crafted payload. Figure 9 shows the packet capture after sending the modified packet.<i></i></p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/Fig9.png" alt="Details of the crafted packet. "/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 9. Details of the crafted packet. </figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The AP5GC NGAP layer no longer responds after InitialUEMessage (Fig: 9, pkt no: 150). Any NGAP control message to AMF at this point will trigger AMF shutdown/restart. Sometimes, AMF restarts automatically; sometimes, it remains inoperative until manually restarted.</p>
<p>Figure 10 shows the modified flow from Figure 5, with the attack scenario included.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/Fig10.png" alt="UE Registration flow with the crafted packet"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 10. UE Registration flow with the crafted packet</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p><b>Root Cause</b></p>
<ol>
<li><span style="font-weight: normal;">The lack of authentication between the base station and packet core allows attackers to connect their own SDRs to the packet core and mount more attacks.</span></li>
<li><span style="font-weight: normal;">In this case, the crash is either caused by ASN.1 parsing or improperly handling the parser error.</span></li>
</ol>
<h1><span class="body-subhead-title">ZDI-CAN-23960: Microsoft Azure Private 5G Core Un- Authenticated Base Station Override</span></h1>
<p>An arbitrary NGSetupRequest sent to AP5GC, with the ID of a legitimate base station, can eject the original legitimate base station(gNB) from the network. Cellular devices served by the original base station will be forced to disconnect. A malicious base station can replace the original legitimate base station.</p>
<p>This vulnerability is assigned ZDI tag <a href="https://www.zerodayinitiative.com/advisories/upcoming/" target="_blank">ZDI-CAN-23960</a>.</p>
<p><b>Attack Setup</b></p>
<p>192.168.1.102 is a legitimate base station.</p>
<p>192.168.1.101 is a Linux PC with srsRAN.</p>
<p>192.168.1.100 is the gateway switch.</p>
<p>192.168.1.10 is AMF.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/Fig11.png" alt="Attack Topology for ZDI-CAN-23960"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 11. Attack Topology for ZDI-CAN-23960</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>The fuzzer intercepts ‘NGSetupRequest’ from srsRAN gNB, changes gNB-ID, and replays several mutations to AMF.</p>
<p><b>Results</b></p>
<p>The packet trace is given in Figure 12. The ‘NGSetupRequest’ (gNB-ID = 0) in packet 52 is from a legitimate gNB. It gets a successful response in packet 54 and is connected. The attacker sends ‘NGSetupRequest’ with the same gNB-ID (0) in packet 88. AP5GC disconnects the first gNB (packet 90) and responds ‘success’ to the last received request in packet 91.</p>
<p>The root cause is that AP5GC trusts the last ‘NGSetupRequest’ received without verification and disconnects any currently attached gNB with the same gNB-ID.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/Fig12.png" alt="Packet capture of ZDI-CAN-23960 in action"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 12. Packet capture of ZDI-CAN-23960 in action</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		
			<img src="/content/dam/trendmicro/global/en/research/24/i/vulnerabilities-in-cellular-packet-cores-part-iv-authentication/Fig13.png" alt="The message flow of ZDI-CAN-23960"/>
		
		<div class="caption-image-container ">
			<figcaption>Figure 13. The message flow of ZDI-CAN-23960</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p><b>Root cause</b></p>
<ol>
<li><span style="font-weight: normal;">Lack of authentication enables attackers to send setup requests to AP5GC and elicit a response.</span></li>
<li><span style="font-weight: normal;">Lack of verification of received messages allows attackers to trick AP5GC into trusting them as legitimate base stations.</span></li>
</ol>
<h1><span class="body-subhead-title">Impact</span></h1>
<p>Both attacks result in varying degrees of denial-of-service (DoS). Since packet cores are critical network infrastructure nodes, the impact extends beyond the directly affected device, potentially disrupting a broader network segment.</p>
<p>The first attack (CVE-2024-20685) brought down the entire control plane, thus causing a total outage of services. All devices connected to the cellular network lose connectivity.</p>
<p>The second attack (ZDI-CAN-23960) ejects an intermediate node, a base station, from the network. All devices connected to that base station lose connectivity. Moreover, it adds an attacker-controlled base station to the network, which can be used to mount further attacks.</p>
<p>Outages in public networks are not common, but not rare either (E.g., <a href="https://www.lightreading.com/finance/outage-slows-customer-migration-to-1-1-s-new-network" target="_blank">1 &amp; 1</a>, <a href="https://edition.cnn.com/2024/02/22/tech/att-cell-service-outage/index.html" target="_blank">AT&amp;T</a>, <a href="https://www.dailymail.co.uk/yourmoney/consumer/article-12457139/T-Mobile-outages-5G-service-areas-worst-affected.html" target="_blank">T-Mobile</a>). Private 5G networks are widely used in critical infrastructure, and service outages can seriously affect safety and business.</p>
<p>To execute the attacks detailed in this article, the attacker needs access to the control-plane subnet. This somewhat mitigates concerns about immediate exploitation. Both vulnerabilities are classified as “Medium” on the CVSS severity scale.</p>
<p>So far, we are not aware of any exploitation in the wild.</p>
<h1><span class="body-subhead-title">Mitigation</span></h1>
<p>CVE-2024-20685 exploits an ASN.1 parser issue, and ZDI-CAN-23960 exploits insufficient message verification. However, both attacks rely on the lack of authentication between the base station and packet core for execution. Addressing this should be the first step toward mitigation.</p>
<ol>
<li><span style="font-weight: normal;">IPSec between base station and packet-core</span></li>
<li><span style="font-weight: normal;">Certificate-based authentication between base station and packet core</span></li>
<li><span style="font-weight: normal;">Allow-list/Whitelist for authorized base stations</span></li>
<li><span style="font-weight: normal;">Access control and isolation of Control Plane subnet</span></li>
<li><span style="font-weight: normal;">Deep Packet Inspection (DPI) solutions for virtual patching before vendor fixes</span></li>
</ol>
<h1><span class="body-subhead-title">Disclosure and Response</span></h1>
<p>We reported both vulnerabilities to Microsoft through Zero Day Initiative (ZDI).</p>
<p>Microsoft has <a href="https://learn.microsoft.com/en-us/azure/private-5g-core/azure-private-5g-core-release-notes-2403" target="_blank">fixed</a> CVE-2024-20685.</p>
<p>ZDI-CAN-23960 was reported to Microsoft on April 25, 2024. Microsoft responded, “We have shared your report with the team responsible for maintaining the service to ensure our customers are protected they will take the necessary action based on their internal timeline.”</p>
