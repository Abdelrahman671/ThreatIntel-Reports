											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>Microsoft Exchange enables admins to disable email autoforwards and autoreplies to remote domains for external recipients. Exchange does this by using message-type headers, such as Auto Forward received from Outlook and Outlook on web clients.</p>
<p>Microsoft Power Platform has the capability to insert specific SMTP headers in email messages sent through Power Automate and Power Apps. It does this using Microsoft 365 Exchange/Outlook connectors. You can use these SMTP headers to set up appropriate exfiltration rules. These rules are for the unauthorized transfer of data from one device to another in Exchange for outbound emails.</p>
<p>For more details on the Microsoft 365 Outlook connector, go to: <a href="/en-us/connectors/office365/#smtp-headers" data-linktype="absolute-path">SMTP headers</a>.</p>
<div class="NOTE">
<p>Note</p>
<p>This method of exfiltration control is not available in GCC High and DoD environments.  In these clouds the x-ms-mail-* headers are not used.</p>
</div>
<h2 id="block-exfiltration-of-forwarded-emails">Block exfiltration of forwarded emails</h2>
<p>Admins can set up Exchange mail flow rules to monitor or block emails sent by Power Automate and/or Power Apps using the Microsoft 365 Outlook connector. The format of the SMTP header uses a reserved phrase âMicrosoft Power Automateâ or 'Microsoft Power Apps.' It's inserted with the header type: âx-ms-mail-application.â For example:</p>
<pre><code>**x-ms-mail-application: Microsoft Power Automate**; User-Agent: 
azure-logic-apps/1.0 (workflow afa0fb167803450aa650267e95d43287; version 
08586093372199532216) microsoft-flow/1.0 
x-ms-mail-operation-type: Forward 
</code></pre>
<p>To identify the operation ID, a reserved word, such as Forward, Reply or Send, gets inserted with the header type âx-ms-mail-operation-type.â For example:</p>
<pre><code>x-ms-mail-application: Microsoft Power Automate; User-Agent: 
azure-logic-apps/1.0 (workflow afa0fb167803450aa650267e95d43287; version 
08586093372199532216) microsoft-flow/1.0 
**x-ms-mail-operation-type: Forward**
</code></pre>
<p>Exchange admins can use these headers to set up exfiltration blocking rules in the Exchange admin center as shown in the example here. Here the âmail flowâ rule rejects outbound email messages with:</p>
<ul>
<li>âx-ms-mail-applicationâ header set as âMicrosoft Power Automateâ and</li>
<li>âx-ms-mail-operation-typeâ header set as âSendâ or âForwardâ</li>
</ul>
<p>This is equivalent to the Exchange âmail flowâ rule set up for the message type equal to âautoforward.â  This rule uses Outlook and Outlook for web clients.</p>
<p><span class="mx-imgBorder">
<img src="media/block-outbound-forwards80.png" alt="Block outbound forwards." data-linktype="relative-path">
</span>
</p>
<h2 id="exempt-specific-flows-from-exfiltration-blocking">Exempt specific flows from exfiltration blocking</h2>
<p>In addition to the âx-ms-mail-applicationâ messages, Power Platform also inserts the workflow identifier as the new âUser-Agentâ header. This is equal to the app or flow ID. For example:</p>
<pre><code>x-ms-mail-application: Microsoft Power Automate; User-Agent: 
azure-logic-apps/1.0 (workflow afa0fb167803450aa650267e95d43287; version 
08586093372199532216) microsoft-flow/1.0 
**x-ms-mail-operation-type: Forward**
</code></pre>
<p>As an admin, if you want to exempt flows (or apps) from the exfiltration due to a legitimate business scenario, use the workflow ID as part of the user-agent header. All other exception conditions offered by Exchange rules, such as sender address, remain available to exempt the legitimate business use cases from the blocking enforcement. For example:</p>
<p><span class="mx-imgBorder">
<img src="media/block-outbound-forwards-exception85.png" alt="Block outbound forwards exceptions." data-linktype="relative-path">
</span>
</p>
<p>Admins can also use other exception capabilities in Exchange mail rules to exempt flows from the exfiltration blocking rules, For example, a unique sender address allows legitimate business use cases to bypass the control.</p>
