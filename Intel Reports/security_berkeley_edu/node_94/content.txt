    <div class="field field-name-body field-type-text-with-summary field-label-hidden"><div class="field-items"><div class="field-item even"><h2><a name="how" id="how"></a>How secure is Windows Remote Desktop?</h2>
<p>Remote Desktop sessions operate over an encrypted channel, preventing anyone from viewing your session by listening on the network. However, there is a vulnerability in the method used to encrypt sessions in earlier versions of RDP. This vulnerability can allow unauthorized access to your session using a <a href="http://en.wikipedia.org/wiki/Man-in-the-middle" target="_blank" rel="noopener noreferrer">man-in-the-middle attack</a>. </p>
<p><span><span>Remote Desktop can be secured using SSL/TLS in Windows Vista, Windows 7, Windows 8, Windows 10 and Windows Server 2003/2008/2012/2016. <strong>*Some systems listed are no longer supported by Microsoft and therefore do not meet Campus security standards.  If unsupported systems are still in use, a <a href="https://security.berkeley.edu/exception">security exception</a> is required.</strong></span></span></p>
<p>While Remote Desktop is more secure than remote administration tools such as VNC that do not encrypt the entire session, any time Administrator access to a system is granted remotely there are risks. The following tips will help to secure Remote Desktop access to both desktops and servers that you support.</p>
<div id="openberkeley-expand-all"><a class="openberkeley-collapsible-collapse" href="#openberkeley-expand-all">collapse all</a> <a class="openberkeley-collapsible-expand" href="#openberkeley-expand-all">expand all</a></div>
<div class="openberkeley-collapsible-container">
<h2 class="openberkeley-collapsible-controller"><a name="basic" id="basic"></a>Basic Security Tips for Remote Desktop</h2>
<div class="openberkeley-collapsible-target">
<h3><span>1. Use strong passwords</span></h3>
<p dir="ltr"><span>Strong passwords on any accounts with access to Remote Desktop should be considered a required step before enabling Remote Desktop. Refer to the </span><a href="https://security.berkeley.edu/node/326" rel="noreferrer"><span>campus password complexity guidelines</span></a><span> for tips.</span> </p>
<h3><span>2. Use Two-factor authentication</span></h3>
<p dir="ltr"><span>Departments should consider using a two-factor authentication approach. This topic is beyond the scope of this article, but RD Gateways  can be configured to integrate with the Campus instance of DUO. Other unsupported by campus options available would be a simple mechanism for controlling authentication via two-factor certificate based smartcards. </span><span>This approach utilizes  the Remote Desktop host itself, in conjunction with YubiKey and RSA as examples.</span></p>
<h3><span>3. Update your software</span></h3>
<p dir="ltr"><span>One advantage of using Remote Desktop rather than 3rd party remote admin tools is that components are updated automatically with the latest security fixes in the standard Microsoft patch cycle. Make sure you are running the latest versions of both the client and server software by enabling and auditing automatic Microsoft Updates. If you are using Remote Desktop clients on other platforms, make sure they are still supported and that you have the latest versions. Older versions may not support high encryption and may have other security flaws.</span></p>
<h3><span>4. Restrict access using firewalls</span></h3>
<p dir="ltr"><span>Use firewalls (both software and hardware where available) to restrict access to remote desktop listening ports (default is TCP 3389). Using an RDP Gateway is highly recommended for restricting RDP access to desktops and servers (see discussion below). As an alternative to support off-campus connectivity, you can use the campus VPN software to get a campus IP address and add the campus VPN network address pool to your RDP firewall exception rule. <a href="https://security.berkeley.edu/services/bsecure/bsecure-remote-access-vpn">Visit our page </a></span><a href="https://security.berkeley.edu/services/bsecure/bsecure-remote-access-vpn">for more information on the campus VPN service</a>.</p>
<h3><span>5. Enable Network Level Authentication</span></h3>
<p dir="ltr"><span>Windows 10, Windows Server 2012 R2/2016/2019 also provide Network Level Authentication (NLA) by default. It is best to leave this in place, as NLA provides an extra level of authentication before a connection is established. You should only configure Remote Desktop servers to allow connections without NLA if you use Remote Desktop clients on other platforms that don't support it.</span></p>
<ul>
<li>
<p dir="ltr"><span>NLA should be enabled by default onWindows 10, Windows Server 2012 R2/2016/2019.</span></p>
</li>
<li>
<p dir="ltr"><span>To check you may look at Group Policy setting Require user authentication for remote connections by using Network Level Authentication found at Computer\Policies\Windows Components\Remote Desktop Services\Remote Desktop Session Host\Security. This Group Policy setting must be enabled on the server running the Remote Desktop Session Host role.</span></p>
</li>
<li>
<p dir="ltr"><a href="https://docs.microsoft.com/en-us/windows-server/remote/remote-desktop-services/clients/remote-desktop-allow-access" rel="noreferrer"><span>https://docs.microsoft.com/en-us/windows-server/remote/remote-desktop-services/clients/remote-desktop-allow-access</span></a></p>
</li>
</ul>
<h3><span>6. Limit users who can log in using Remote Desktop</span></h3>
<p dir="ltr"><span>By default, all Administrators can log in to Remote Desktop. If you have multiple Administrator accounts on your computer, you should limit remote access only to those accounts that need it. If Remote Desktop is not used for system administration, remove all administrative access via RDP, and only allow user accounts requiring RDP service. For Departments that manage many machines remotely remove the local Administrator account from RDP access at and add a technical group instead.</span></p>
<ol>
<li>
<p dir="ltr"><span>Click Start--&gt;Programs--&gt;Administrative Tools--&gt;Local Security Policy</span></p>
</li>
<li>
<p dir="ltr"><span>Under Local Policies--&gt;User Rights Assignment, go to "Allow logon through Terminal Services." Or “Allow logon through Remote Desktop Services”</span></p>
</li>
<li>
<p dir="ltr"><span>Remove the Administrators group and leave the Remote Desktop Users group.</span></p>
</li>
<li>
<p dir="ltr"><span>Use the System control panel to add users to the Remote Desktop Users group.</span></p>
</li>
</ol>
<p dir="ltr"><span>A typical MS operating system will have the following setting by default as seen in the Local Security Policy:</span></p>

<p><img src="https://security.berkeley.edu/sites/default/files/styles/panopoly_image_original/public/rdp_pic1.png?itok=4wRhf-1N&amp;timestamp=1440183405" alt=""></p>
<p dir="ltr"><span>The problem is that “Administrators” is here by default, and your “Local Admin” account is in administrators.  Although a password convention to avoid identical local admin passwords on the local machine and tightly controlling access to these passwords or conventions is recommended, using a local admin account to work on a machine remotely does not properly log and identify the user using the system. It is best to override the local security policy with a Group Policy Setting.</span></p>
<p><img src="https://security.berkeley.edu/sites/default/files/styles/panopoly_image_original/public/rdp_pic2.png?itok=fJupQNB6&amp;timestamp=1440183457" alt=""></p>
<p dir="ltr"><span>To control access to the systems, even more, using “Restricted Groups” via Group Policy is also helpful.</span></p>
<p dir="ltr"><span>If you use a “Restricted Group” setting to place your group, e.g., “CAMPUS\LAW-TECHIES” into “Administrators” and “Remote Desktop Users,” your techies will still have administrative access remotely, but using the steps above, you have removed the problematic “local administrator account” having RDP access. Going forward, whenever new machines are added in the OU under the GPO, your settings will be correct.</span></p>
<p><img src="https://security.berkeley.edu/sites/default/files/styles/panopoly_image_original/public/rdp_pic3.png?itok=hwexNyIs&amp;timestamp=1440183506" alt=""></p>
<h3>7. Set an account lockout policy</h3>
<p>By setting your computer to lock an account for a set number of incorrect guesses, you will help prevent hackers from using automated password guessing tools from gaining access to your system (this is known as a "brute-force" attack). To set an account lockout policy:</p>
<ol>
<li>Go to Start--&gt;Programs--&gt; Administrative Tools--&gt; Local Security Policy</li>
<li>Under Account Policies--&gt; Account Lockout Policies, set values for all three options. Three invalid attempts with 3-minute lockout durations are reasonable choices.</li>
</ol></div>
</div>

<div class="openberkeley-collapsible-container">
<h2 class="openberkeley-collapsible-controller"><a name="best" id="best"></a>Best Practices for Additional Security</h2>
<div class="openberkeley-collapsible-target">
<h3><span>1. Do not allow direct RDP access to clients or servers from off campus.</span></h3>
<p dir="ltr"><span>Having RDP (port 3389) open to off campus networks is highly discouraged and is a known vector for many attacks.  The options below list ways of improving security while still allowing RDP access to system. </span></p>
<p dir="ltr"><span><span>Once an RDP gateway has been set up, hosts should be configured to only allow RDP connections from the Gateway host or campus subnets where needed.</span></span></p>
<h3><span>2. Use RDP Gateways (Best Option)</span></h3>
<p dir="ltr"><span>Using an RDP Gateway is strongly recommended. It provides a way to tightly restrict access to Remote Desktop ports while supporting remote connections through a single "Gateway" server. When using an RD Gateway server, all Remote Desktop services on your desktop and workstations should be restricted to only allow access only from the RD Gateway. The RD Gateway server listens for Remote Desktop requests over HTTPS (port 443) and connects the client to the Remote Desktop service on the target machine.</span></p>
<ol>
<li>
<p dir="ltr"><span><strong>Utilize Campus RDP Gateway Service.</strong>  This is the best option to allow RDP access to system categorized as UC P2 and lower.  Includes DUO integration. </span>RDP Gateway Service is provided by the Windows Team. Documentation is available here: <a href="https://berkeley.sharepoint.com/sites/calnetad/gateway" rel="noreferrer">https://berkeley.sharepoint.com/sites/calnetad/gateway</a>. </p>
<p dir="ltr"><em>The RDP Gateway Service also supports the new <a href="https://security.berkeley.edu/policy/minimum-security-standards-networked-devices-draft#remoteaccess" rel="noreferrer">Remote Access Services</a> requirement of the draft MSSND update (requirement 8), which requires the use of an approved service (i.e., RDP gateway, dedicated gateway, or bSecure VPN) for access to the UC Berkeley network from the public Internet. </em> </p>
</li>
<li>
<p dir="ltr"><span><strong>Dedicated Gateway Service (Managed). </strong> Needed for rdp access to systems that are UC P4 or higher.  Must also be configured for DUO<br><br></span>Some campus units use an Berkeley IT managed VPS as an RD Gateway. A rough estimate might be that 30-100 concurrent users can use one RD Gateway. The HA at the virtual layer provides enough fault-tolerant and reliable access; however a slightly more sophisticated RD gateway implementation can be done with network load balancing.</p>
</li>
<li><strong>Dedicated Gateway Service (Unmanaged).</strong> Installing and configuring RD Gateway on department run hardware.<br><br>There are many online documents for configuring this embedded Windows 2016/2019 component. The official documentation is here: <a href="https://docs.microsoft.com/en-us/windows-server/remote/remote-desktop-services/rds-deploy-infrastructure" rel="noreferrer">https://docs.microsoft.com/en-us/windows-server/remote/remote-desktop-se...</a><br><br>Installing the configuring, the role service is mostly as described; however, using a Calnet issued trusted Comodo certificate is recommended. Using a self-signed cert is ok for testing, and using a CalnetPKI cert can work if all clients have trusted the UCB root. The Comodo cert is usually better accepted so that your end users do not receive certificate warnings.<br><br>Configuring your client to use your RD Gateway is simple.The official documentation for the MS Client is here: <a href="http://technet.microsoft.com/en-us/library/cc770601.aspx" rel="noreferrer">http://technet.microsoft.com/en-us/library/cc770601.aspx</a> </li>
</ol>
<p>In essence, a simple change on the advanced tab of your RDP client is all that is necessary:</p>
<p><span><br><img src="https://security.berkeley.edu/sites/default/files/styles/panopoly_image_original/public/rdp_pic4.png?itok=LyyPBN5u&amp;timestamp=1440183557" alt=""></span></p>
<h3><a name="SecuringRemoteDesktopforSystemAdministrators-ChangethelisteningportforRemoteDesktop" id="SecuringRemoteDesktopforSystemAdministrators-ChangethelisteningportforRemoteDesktop"></a>3. Change the listening port for Remote Desktop</h3>
<p>Changing the listening port will help to "hide" Remote Desktop from hackers who are scanning the network for computers listening on the default Remote Desktop port (TCP 3389). This offers effective protection against the latest RDP worms such, as <span>Morto</span>. To do this, edit the following registry key (WARNING: do not try this unless you are familiar with the Windows Registry and TCP/IP): HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Terminal Server\WinStations\RDP-Tcp. Change the listening port from 3389 to something else and remember to update any firewall rules with the new port. Although this approach is helpful, it is security by obscurity, which is not the most reliable security approach. You should ensure that you are also using other methods to tighten down access as described in this article.</p>
<h3>4. Tunnel Remote Desktop connections through IPSec or SSH</h3>
<p>If using an RD Gateway is not feasible, you can add an extra layer of authentication and encryption by tunneling your Remote Desktop sessions through IPSec or SSH. IPSec is built-in to all Windows operating systems since Windows 2000, but use and management are greatly improved in Windows 10 (see: <a href="http://technet.microsoft.com/en-us/network/bb531150" rel="noreferrer">http://technet.microsoft.com/en-us/network/bb531150</a>). If an SSH server is available, you can use SSH tunneling for Remote Desktop connections. </p>
<h3>5. Use existing management tools for RDP logging and configuration</h3>
<p>Using other components like VNC or PCAnywhere is not recommended because they may not log in a fashion that is auditable or protected. With RDP, logins are audited to the local security log, and often to the domain controller auditing system. When monitoring local security logs, look for anomalies in RDP sessions such as login attempts from the local Administrator account. RDP also has the benefit of a central management approach via GPO as described above. Whenever possible, use GPOs or other Windows configuration management tools to ensure a consistent and secure RDP configuration across all your servers and desktops.</p>
<p>By enforcing the use of an RDP gateway, you also get a third level of auditing that is easier to read than combing through the domain controller logins and is separate from the target machine so it is not subject to tampering. This type of log can make it much easier to monitor how and when RDP is being used across all the devices in your environment.</p>
</div>
</div>

<h2 class="openberkeley-collapsible-controller"><a name="restrict" id="restrict"></a>Restrict Access to RDP with Windows Firewall</h2>
<div class="openberkeley-collapsible-target">
<p>If you have a campus-managed computer:</p>
<ul>
<li>Contact <a href="https://berkeley.service-now.com/ess" rel="noreferrer">IT Client Services</a> or your departmental IT support for assistance.</li>
</ul>
<p>If you have a personally-managed computer and Administrator access:</p>
<ul>
<li>Follow the instructions in this article to update your Windows Firewall so that only authorized hosts and networks can access your system via Remote Desktop (RDP). </li>
</ul>
<p><strong>Settings &gt; Update and Security &gt; Windows Security &gt; Firewall and Network Protection &gt; Advanced Settings &gt; Inbound Rules &gt; Remote Desktop - User Mode (TCP-In) &gt; Properties &gt; Scope &gt; Remote IP address &gt; Add &gt; This IP address or subnet</strong></p>
<ol>
<li>Settings &gt; Update and Security</li>
</ol>
<p style="margin-left:30px"><img alt="Settings Update and Security " src="https://security.berkeley.edu/sites/default/files/styles/width_300/public/update_and_security.png?itok=pxL5iGet&amp;timestamp=1620840347"></p>
<ol start="2">
<li>Windows Security &gt; Firewall and Network Protection</li>
</ol>
<p style="margin-left:30px"><img alt="Firewall and Network Protection" src="https://security.berkeley.edu/sites/default/files/styles/width_400/public/firewall_and_network_protection.png?itok=QV_2WqqV&amp;timestamp=1620840410"></p>
<ol start="3">
<li>Advanced Settings</li>
</ol>
<p style="margin-left:30px"><img alt="Firewall Advanced" src="https://security.berkeley.edu/sites/default/files/styles/width_300/public/firewall_advanced_border.png?itok=f38rHZ8t&amp;timestamp=1620840526"></p>
<ol start="4">
<li>Inbound Rules &gt; Remote Desktop - User Mode (TCP-In) &gt; Properties </li>
</ol>
<p style="margin-left:30px"><img alt="RDP Inbound" src="https://security.berkeley.edu/sites/default/files/styles/width_400/public/rdp_inbound_zoom.png?itok=M6hWndM3&amp;timestamp=1620840568"></p>
<ol start="5">
<li>Scope &gt; Remote IP address &gt; Add </li>
</ol>
<p style="margin-left:30px"><img alt="RDP Scope" src="https://security.berkeley.edu/sites/default/files/styles/width_400/public/rdp_scope2.png?itok=QQLIMmMt&amp;timestamp=1620840612"></p>
<ol start="6">
<li>Under This IP address or subnet, only add IP addresses and network subnets that should be authorized to connect to your computer’s Remote Desktop (RDP) service. Some common examples of campus IP addresses and subnets are listed in the section below.</li>
</ol>
<p style="margin-left:30px"><img alt="RDP IP Address" src="https://security.berkeley.edu/sites/default/files/styles/width_400/public/rdp_ip_address.png?itok=n7f58am8&amp;timestamp=1620840654"></p>

<h4>Campus IP addresses and subnets</h4>
<p>Based on your needs, choose only authorized campus IP addresses and subnets to connect to your computer’s RDP service. Network Operations &amp; Services maintains the <a href="https://berkeley.service-now.com/kb_view.do?sysparm_article=KB0011960" rel="noreferrer">source list of UC Berkeley Campus Networks</a>, but some common examples are included below for reference.</p>
<p><strong>Berkeley IT RD Gateway</strong><br>To access your system via RDP directly from the Internet, utilize the Campus Remote Desktop Gateway. The RD Gateway will allow you to use your CalNet ID with Duo push notifications to connect. You can authorize the RD Gateway by adding the following subnet to your firewall rule:</p>
<ul>
<li>169.229.164.0/24</li>
</ul>
<p><strong>Campus Remote Access VPN Networks (bSecure Remote Access Services with GlobalProtect)</strong><br>To access your system via RDP via the campus VPN, add one or more, as appropriate, of the following VPN networks to your firewall rule:</p>
<ul>
<li>Split Tunnel Client Networks
<ul>
<li>10.136.128.0/18</li>
</ul>
</li>
<li>Split Tunnel Client Networks
<ul>
<li>136.152.16.0/20</li>
</ul>
</li>
<li>Restricted Tunnel Networks
<ul>
<li>136.152.210.0/23</li>
</ul>
</li>
</ul>
<p><strong>Campus Networks (onsite)</strong></p>
<p>To access your system via RDP while on campus, add the appropriate campus wireless or wired networks to your firewall rule. See <a href="https://berkeley.service-now.com/kb_view.do?sysparm_article=KB0011960" rel="noreferrer">UC Berkeley Campus Networks </a>for the most recent information available.</p>
