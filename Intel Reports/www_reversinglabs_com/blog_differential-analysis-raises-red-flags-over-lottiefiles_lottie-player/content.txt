<h1><span id="hs_cos_wrapper_name" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="text">Differential analysis raises red flags over @lottiefiles/lottie-player</span></h1>
<div id="hs_cos_wrapper_module_1577203992502740" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module">
<h2 class="post-header__secondary-head blog__post-secondary-h2">Three versions of the popular package were infected and used to spread malicious code that was stealing crypto wallet assets. </h2>
</div>
</div>
<div class="blog-author__short">
<div class="blog-author__short-info">
<div class="blog-author__short-info__image-container">

<div id="hs_cos_wrapper_module_1705062190107298" class="hs_cos_wrapper hs_cos_wrapper_widget hs_cos_wrapper_type_module" style="" data-hs-cos-general-type="widget" data-hs-cos-type="module"></div>
<div class="blog__post-content html-content">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-crypto-blog.webp?width=1400&amp;height=732&amp;name=lottie-crypto-blog.webp" width="1400" height="732" loading="lazy" alt="lottie" style="height: auto; max-width: 100%; width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-crypto-blog.webp?width=700&amp;height=366&amp;name=lottie-crypto-blog.webp 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-crypto-blog.webp?width=1400&amp;height=732&amp;name=lottie-crypto-blog.webp 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-crypto-blog.webp?width=2100&amp;height=1098&amp;name=lottie-crypto-blog.webp 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-crypto-blog.webp?width=2800&amp;height=1464&amp;name=lottie-crypto-blog.webp 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-crypto-blog.webp?width=3500&amp;height=1830&amp;name=lottie-crypto-blog.webp 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-crypto-blog.webp?width=4200&amp;height=2196&amp;name=lottie-crypto-blog.webp 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></p>
<p>Malware on public repositories is nothing new. For a couple of years now, ReversingLabs threat researchers have been monitoring npm, PyPI and recently VSCode Marketplace, RubyGems and NuGet for potential malware whose inclusion in the development cycle could cause a supply chain attack. More often than not, malicious packages are published by new accounts and are made from scratch.</p>
<!--more--><p>Sometimes they are very simple — ranging from simple infostealers usually used in red team operations to downloaders delivering malicious second stage payloads. Other times, malicious actors put some effort into making their packages seem trustworthy. They spend significant effort to typosquat or mimic another popular package, or try to build trust (and a community of users) by releasing non-malicious software versions before publishing a malicious one.&nbsp;<br><br>ReversingLabs (RL) researchers published a research blog post in June about a <a href="/blog/a-lurking-npm-package-makes-the-case-for-open-source-health-checks" rel="noopener">malicious npm package that was dormant for a couple of months</a> before the malicious version was published. The team has also observed and documented the emergence of new, malicious packages linked to older, dormant campaigns that we have already seen and recorded. For example, in September, researchers wrote about <a href="/blog/fake-recruiter-coding-tests-target-devs-with-malicious-python-packages" rel="noopener">PyPI packages connected with the VMConnect campaign</a> that was identified almost a year earlier. &nbsp;&nbsp;<br><br>The majority of the malicious packages found on public repositories usually come from new, throwaway accounts. However, there have been some exceptions to this rule. In 2021, an official npm package, <span style="font-style: italic;">ua-parser-js</span>, was <a href="https://x.com/ReversingLabs/status/1451643445123158021" rel="noopener" target="_blank">caught serving malware</a> in versions <a href="https://secure.software/npm/packages/ua-parser-js/0.7.29" rel="noopener" target="_blank">0.7.29</a>, <a href="https://secure.software/npm/packages/ua-parser-js/0.8.0" rel="noopener" target="_blank">0.8.0</a> &amp; <a href="https://secure.software/npm/packages/ua-parser-js/1.0.0" rel="noopener" target="_blank">1.0.0</a>. Compromised versions were using a legitimate Windows certificate management utility called certutil to download and launch a malicious executable.Then, in December 2023, another legitimate package, <a href="https://secure.software/npm/packages/@ledgerhq/connect-kit" rel="noopener" target="_blank"><span style="font-style: italic;">@ledgerhq/connect-kit</span></a>, <a href="https://www.ledger.com/blog/security-incident-report" rel="noopener" target="_blank">was compromised</a>. Using an individual's session token, malicious actors gained access to the associated npm account and published three malicious versions of the connect-kit that were used to steal crypto wallet assets from victims.<br><br>Then, in October, RL and other firms &nbsp;observed something similar happening again: Three versions (2.0.5, 2.0.6 and 2.0.7) of a popular, legitimate package <a href="https://secure.software/npm/packages/@lottiefiles/lottie-player" rel="noopener" target="_blank"><span style="font-style: italic;">@lottiefiles/lottie-player</span></a> were infected and used to spread malicious code that was designed to steal crypto wallet assets from victims.<br><br>Since then, LottieFiles maintainers worked with npm to remove malicious versions from the repository, and they have since published new, clean versions of <span style="font-style: italic;">@lottiefiles/lottie-player</span>. The compromise has already attracted <a href="https://www.theregister.com/2024/10/31/lottiefiles_supply_chain_attack/" rel="noopener" target="_blank">media attention</a>, but ReversingLabs’ research team dug deeper to understand what is going on.&nbsp;</p>
<h2 style="font-size: 24px; font-weight: normal;">Unwanted pop-ups</h2>
<p>First published five years ago, the package <span style="font-style: italic;">@lottiefiles/lottie-player</span> is a legitimate npm package with an estimated 84,000 weekly downloads. It is used for easily embedding and playing Lottie animations and Lottie-based Telegram Sticker animations in websites.<br><br>The package itself has now over 50 versions since being released, and it is being semi sporadically updated. So new versions 2.0.5, 2.0.6 and 2.0.7 being published 8 months after the last one wasn’t something too strange. However, those versions were malicious.<br><br>The malicious changes attracted attention almost immediately after they were pushed out directly to the npm. Developers using <span style="font-style: italic;">@lottiefiles/lottie-player</span> started noticing the package behaving differently then they were used to. Sites that installed the &nbsp;malicious versions of the package <span style="font-style: italic;">@lottiefiles/lottie-player</span>, were displaying a pop-up to connect to web3 wallets. Any user that connected their wallets opened the door to <a href="https://x.com/realScamSniffer/status/1851800628189933806" rel="noopener" target="_blank">attackers draining its assets</a>.<br><br>Little did website visitors know that connecting their wallet would connect it to the attacker's architecture. &nbsp;Developers and users alike started raising alarms and discussing the behavior on <a href="https://forum.lottiefiles.com/t/the-problem-of-someone-elses-popup-appearing/6470/3" rel="noopener" target="_blank">forums</a> and <a href="https://github.com/LottieFiles/lottie-player/issues/254" rel="noopener" target="_blank">GitHub</a>, trying to find more information about new versions in the community. It was apparent very fast that the new versions were malicious.&nbsp;</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/lottie-figure_1.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_1.png?width=1400&amp;name=lottie-figure_1.png" alt="users on GitHub asking if lottiefiles/lottie-player was hacked" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_1.png?width=700&amp;name=lottie-figure_1.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_1.png?width=1400&amp;name=lottie-figure_1.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_1.png?width=2100&amp;name=lottie-figure_1.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_1.png?width=2800&amp;name=lottie-figure_1.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_1.png?width=3500&amp;name=lottie-figure_1.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_1.png?width=4200&amp;name=lottie-figure_1.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>
<p style="font-weight: bold; font-size: 16px; text-align: center;">Figure 1: users on GitHub asking if @lottiefiles/lottie-player was hacked</p>
<p>LottieFiles maintainers were very fast with the response, and after investigation was concluded, Nattu Adnan, co-founder and CTO at LottieFiles, confirmed three versions were published over the course of a few hours using a compromised access token from a developer with the required privileges.&nbsp;</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/lottie-figure_2-1.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_2-1.png?width=1400&amp;name=lottie-figure_2-1.png" alt="Incident response" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_2-1.png?width=700&amp;name=lottie-figure_2-1.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_2-1.png?width=1400&amp;name=lottie-figure_2-1.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_2-1.png?width=2100&amp;name=lottie-figure_2-1.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_2-1.png?width=2800&amp;name=lottie-figure_2-1.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_2-1.png?width=3500&amp;name=lottie-figure_2-1.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_2-1.png?width=4200&amp;name=lottie-figure_2-1.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>
<p style="font-weight: bold; font-size: 16px; text-align: center;">Figure 2: <a href="https://forum.lottiefiles.com/t/the-problem-of-someone-elses-popup-appearing/6470/11" rel="noopener" target="_blank">Incident response</a></p>
<p>Malicious versions were removed from npm, and a new, clean version that was a copy of version 2.0.4, the last-known “good” version, was published on npm. This was important for all developers that didn’t pin dependency to a specific version but were instead using <em>@latest</em>, as they received the patched update automatically.&nbsp;</p>
<h2 style="font-size: 24px; font-weight: normal;">Change in behaviors</h2>
<p>So what exactly happened? What did the malicious actors do to alter versions of <span style="font-style: italic;">@lottiefiles/lottie-player</span>? RL researcher analysis shows that malicious actors took the file <span style="font-style: italic;">lottie-player.js</span> and replaced its code with their own. The malicious code was <a href="https://www.imperva.com/learn/performance/minification/" rel="noopener" target="_blank">minified</a>, but this wasn’t very weird or noticeable at first since the original file had minified code as well. However, even at first glance, the two sets of code looked very different.<br><br>Once our researchers were aware of the malicious nature of those three versions, we marked them as malicious — and they stand as such on RL’s <a href="https://secure.software/npm/packages/@lottiefiles/lottie-player/2.0.5" rel="noopener" target="_blank">Spectra Assure Community page</a>. Additional research using the <a href="/products/software-supply-chain-security" rel="noopener">Spectra Assure CLI</a> and taking the last known non-malicious version of lottie-player, 2.0.4, and the last malicious version, 2.0.7, and doing a differential analysis of them highlights how the malicious actors altered the <em>@lottiefiles/lottie-player</em> package (Figure 3).</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/lottie-figure_3.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_3.png?width=1400&amp;name=lottie-figure_3.png" alt="Spectra Assure scan" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_3.png?width=700&amp;name=lottie-figure_3.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_3.png?width=1400&amp;name=lottie-figure_3.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_3.png?width=2100&amp;name=lottie-figure_3.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_3.png?width=2800&amp;name=lottie-figure_3.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_3.png?width=3500&amp;name=lottie-figure_3.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_3.png?width=4200&amp;name=lottie-figure_3.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>
<p style="font-weight: bold; font-size: 16px; text-align: center;">Figure 3: Spectra Assure scan showing a differential analysis between lottie-player.js file from different versions of <br>@lottiefiles/lottie-player using Spectra Assure CLI</p>
<p>Differential analysis created with <a href="https://docs.secure.software/cli/" rel="noopener" target="_blank">Spectra Assure CLI</a> allows users to compare two versions of the same package and see the changes between them and their files. Changes can include the unexplained introduction of new behaviors; the omission of other behaviors; changes in package size; or behaviors that trigger threat hunting policies. Looking at the comparison between versions 2.0.4 and 2.0.7 of <span style="font-style: italic;">@lottiefiles/lottie-player</span>, some things jump out at us. First is the big difference in size between two versions of a package — with no corresponding features to account for the extra code. That is often an indicator something has changed.&nbsp;<br><br>RL researchers also found that there was an omission of normal <a href="https://docs.secure.software/concepts/behavior-reference?" rel="noopener" target="_blank">behaviors</a> that can usually be found in the file <span style="font-style: italic;">lottie-player.js</span> such as “Enumerates display information” or “Might enumerate information about screen” (Figure 4).<br><br>However, some behaviors that aren’t usually part of <em>lottie-player.js</em> files appeared, such as “Contains URLs related to Bitcoin exchange services”. These behaviors could have pointed to the new malicious functionality of <em>@lottiefiles/lottie-player </em>package.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/lottie-figure_4.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_4.png?width=1400&amp;name=lottie-figure_4.png" alt="changes in package behaviors" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_4.png?width=700&amp;name=lottie-figure_4.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_4.png?width=1400&amp;name=lottie-figure_4.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_4.png?width=2100&amp;name=lottie-figure_4.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_4.png?width=2800&amp;name=lottie-figure_4.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_4.png?width=3500&amp;name=lottie-figure_4.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_4.png?width=4200&amp;name=lottie-figure_4.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>
<p style="font-weight: bold; font-size: 16px; text-align: center;">Figure 4: changes in package behaviors between 2.0.4 and 2.0.7 of lottie-player.js as identified by Spectra Assure CLI</p>
<p>Also very important to note are the <a href="https://docs.secure.software/policies/threat-hunting?" rel="noopener" target="_blank">threat hunting policies</a> that the <span style="font-style: italic;">@lottiefiles/lottie-player</span> package tripped. These policies focus on detecting malicious and suspicious behaviors in software components to prevent tampering incidents and software supply chain attacks. The subset of these threat hunting polices that is of interest here is <a href="https://docs.secure.software/policies/threat-hunting#differential-analysis-policies" rel="noopener" target="_blank">differential analysis policies</a>. They detect behaviors and changes characteristic for known software supply chain attacks. Their focus is detecting issues that closely resemble previously discovered supply chain attacks based on their signatures. For example, there is differential analysis policy <a href="https://docs.secure.software/policies/threat-hunting/TH20101" rel="noopener" target="_blank">TH20101</a>, which detects indicators of tampering that resemble the SolarWinds Orion software compromise. It gets triggered whenever there is a change between two versions of a package that is characteristic to the changes seen between the last-known good SolarWinds Orion software update and the compromised update.<br><br>In the case of lottie-player, RL researchers saw a number of differential analysis policies that were triggered (Figure 5). They include <a href="https://docs.secure.software/policies/threat-hunting/TH16128" rel="noopener" target="_blank">TH16128</a>, which is designed to detect presence of software components that can detect common crypto tokens. Also, policy <a href="https://docs.secure.software/policies/threat-hunting/TH17107" rel="noopener" target="_blank">TH17107</a> was triggered. That policy detected the presence of files containing URLs related to Bitcoin exchange services.</p>
<a href="https://blog.reversinglabs.com/hubfs/Blog/lottie-figure_5.png" class="lightbox-image"><img src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_5.png?width=1400&amp;name=lottie-figure_5.png" alt="Threat hunting policies" width="1400" style="width: 1400px;" srcset="https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_5.png?width=700&amp;name=lottie-figure_5.png 700w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_5.png?width=1400&amp;name=lottie-figure_5.png 1400w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_5.png?width=2100&amp;name=lottie-figure_5.png 2100w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_5.png?width=2800&amp;name=lottie-figure_5.png 2800w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_5.png?width=3500&amp;name=lottie-figure_5.png 3500w, https://www.reversinglabs.com/hs-fs/hubfs/Blog/lottie-figure_5.png?width=4200&amp;name=lottie-figure_5.png 4200w" sizes="(max-width: 1400px) 100vw, 1400px"></a>
<p style="font-weight: bold; font-size: 16px; text-align: center;">Figure 5: Threat hunting policies triggered by the comparison of clean and compromised lottie-player.js file</p>
<p>Due to this incident with the npm package <em>@lottiefiles/lottie-player</em>, we are working on adding a differential analysis policy specifically targeted to this incident, so incidents of a similar type can be flagged as malicious in the future. One of the indicators that will be included in future differential analysis policy is “Contains URLs related to Bitcoin exchange services”.</p>
<h2 style="font-size: 24px; font-weight: normal;">Conclusion</h2>
<p>The reported supply chain attack involving the <span style="font-style: italic;">@lottiefiles/lottie-player</span> underscores the dangers posed by malicious packages and supply chain attacks. These threats can lurk not only in new packages published by new and unknown actors, but in established code bases. The lottie-player hack shows how malicious actors can hijack accounts of maintainers with privileged access and can publish a series of malicious versions of that package to the community of users and developers, polluting their product and causing supply chain attacks in different projects.&nbsp;<br><br>Smart, secure development practices can diminish the risks posed by supply chain attacks. One of those practices is pinning dependencies to a specific, known-good version or range of versions of a file instead of using <span style="font-style: italic;">@latest</span> when specifying dependencies in production. Pinning specific versions in the production environment may deny you the immediate benefit of software updates, but it will allow developers to avoid the risks of a supply chain attack and exert more and better control over the code used in their development cycle. That allows them to vet any updates and avoid introducing breaking or unwanted changes, as well as mitigating incompatibilities.<br><br>It is also very important to remember to regularly update the dependencies as well, to avoid any vulnerabilities and risks older versions might bring &nbsp;(after vetting them for malicious implants of course).<br><br>The lottie-player compromise should serve as a reminder that malicious actors are crafty, trying new techniques to avoid detection and even going so far as to try and hijack and infect popular, widely used open-source packages.&nbsp;<br><br>In the case of the <span style="font-style: italic;">@lottiefiles/lottie-player</span>, the supply chain compromise was detected quickly. However, that doesn’t mean that malicious actors couldn’t work in the future towards being even more secretive and better at hiding their malicious code. That’s why it’s necessary for developers to conduct security assessments that can verify the integrity and quality of public, open source libraries for safety before they are used.&nbsp;<br><br>This is not the only way malicious code can be added and compromise the product. Security assessments should be performed to make sure the build pipeline wasn’t compromised, and some malicious dependency added without notice.It is important to perform security assessments regularly. New versions of existing file dependencies or products can introduce new vulnerabilities, risks and other issues that can be caught with regular checks. Security assessments can also be beneficial in catching hijacked- and infected libraries before they are included in production.&nbsp;</p>
