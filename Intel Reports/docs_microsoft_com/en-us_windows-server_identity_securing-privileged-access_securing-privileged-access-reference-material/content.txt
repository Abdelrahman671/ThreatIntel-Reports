											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>This document describes an overall enterprise access model that includes context of how a <a href="privileged-access-strategy" data-linktype="relative-path">privileged access strategy</a> fits in. For a roadmap on how to adopt a privileged access strategy, see the <a href="security-rapid-modernization-plan" data-linktype="relative-path">rapid modernization plan (RaMP)</a>. For implementation guidance to deploy this, see <a href="privileged-access-deployment" data-linktype="relative-path">privileged access deployment</a></p>
<p>Privileged access strategy is part of an overall enterprise access control strategy. This enterprise access model shows how privileged access fits into an overall enterprise access model.</p>
<p>The primary stores of business value that an organization must protect are in the Data/Workload plane:</p>
<p><img src="media/privileged-access-strategy/data-workload-plane.png" alt="Data/workload plane" data-linktype="relative-path"></p>
<p>The applications and data typically store a large percentage of an organization's:</p>
<ul>
<li><strong>Business processes</strong> in applications and workloads</li>
<li><strong>Intellectual property</strong> in data and applications</li>
</ul>
<p>The enterprise IT organization manages and supports the workloads and the infrastructure they are hosted on, whether it's on-premises, on Azure, or a third-party cloud provider, creating a <strong>management plane</strong>. Providing consistent access control to these systems across the enterprise requires a <strong>control plane</strong> based on centralized enterprise identity system(s), often supplemented by network access control for older systems like operational technology (OT) devices.</p>
<p><img src="media/privileged-access-strategy/control-management-data-workload-planes.png" alt="Control, management, and data/workload planes" data-linktype="relative-path"></p>
<p>Each of these planes has control of the data and workloads by virtue of their functions, creating an attractive pathway for attackers to abuse if they can gain control of either plane.</p>
<p>For these systems to create business value, they must be accessible to internal users, partners, and customers using their workstations or devices (often using remote access solutions) - creating <strong>user access</strong> pathways. They must also frequently be available programmatically via application programming interfaces (APIs) to facilitate process automation, creating <strong>application access</strong> pathways.</p>
<p><img src="media/privileged-access-strategy/user-app-control-management-data-workload-planes.png" alt="Adding user and application access pathways" data-linktype="relative-path"></p>
<p>Finally, these systems must be managed and maintained by IT staff, developers, or others in the organizations, creating <strong>privileged access</strong> pathways. Because of the high level of control they provide over business critical assets in the organization, these pathways must be stringently protected against compromise.</p>
<p><img src="media/privileged-access-strategy/privileged-access-over-underlying-planes.png" alt="Privileged access pathway to manage and maintain" data-linktype="relative-path"></p>
<p>Providing consistent access control in the organization that enables productivity and mitigates risk requires you to</p>
<ul>
<li>Enforce Zero Trust principles on all access
<ul>
<li>Assume Breach of other components</li>
<li>Explicit validation of trust</li>
<li>Least privilege access</li>
</ul>
</li>
<li>Pervasive security and policy enforcement across
<ul>
<li>Internal and external access to ensure consistent policy application</li>
<li>All access methods including users, admins, APIs, service accounts, etc.</li>
</ul>
</li>
<li>Mitigate unauthorized privilege escalation
<ul>
<li>Enforce hierarchy â to prevent control of higher planes from lower planes (via attacks or abuse of legitimate processes)
<ul>
<li>Control plane</li>
<li>Management plane</li>
<li>Data/workload plane</li>
</ul>
</li>
<li>Continuously audit for configuration vulnerabilities enabling inadvertent escalation</li>
<li>Monitor and respond to anomalies that could represent potential attacks</li>
</ul>
</li>
</ul>
<h2 id="evolution-from-the-legacy-ad-tier-model">Evolution from the legacy AD tier model</h2>
<p>The enterprise access model supersedes and replaces the legacy tier model that was focused on containing unauthorized escalation of privilege in an on-premises Windows Server Active Directory environment.</p>
<p><img src="media/privileged-access-strategy/legacy-tier-model.png" alt="Legacy AD tier model" data-linktype="relative-path"></p>
<p>The enterprise access model incorporates these elements as well as full access management requirements of a modern enterprise that spans on-premises, multiple clouds, internal or external user access, and more.</p>
<p><img src="media/privileged-access-strategy/legacy-tier-model-comparison-new.png" alt="Complete enterprise access model from old tiers" data-linktype="relative-path"></p>
<h3 id="tier-0-scope-expansion">Tier 0 scope expansion</h3>
<p>Tier 0 expands to become the control plane and addresses all aspects of access control, including networking where it is the only/best access control option, such as legacy OT options</p>
<h3 id="tier-1-splits">Tier 1 splits</h3>
<p>To increase clarity and actionability, what was tier 1 is now split into the following areas:</p>
<ul>
<li><strong>Management plane</strong> â for enterprise-wide IT management functions</li>
<li><strong>Data/Workload plane</strong> â for per-workload management, which is sometimes performed by IT personnel and sometimes by business units</li>
</ul>
<p>This split ensures focus for protecting business critical systems and administrative roles that have high intrinsic business value, but limited technical control. Additionally, this split better accommodates developers and DevOps models vs. focusing too heavily on classic infrastructure roles.</p>
<h3 id="tier-2-splits">Tier 2 splits</h3>
<p>To ensure coverage for application access and the various partner and customer models, Tier 2 was split into the following areas:</p>
<ul>
<li><strong>User access</strong> â which includes all B2B, B2C, and public access scenarios</li>
<li><strong>App access</strong> â to accommodate API access pathways and resulting attack surface</li>
</ul>
