<div class="cmp-text" id="text-c24609b785" data-cmp-name="cmp-text-" data-cmp-is="text" data-target-location="false">
<span><ul><li><span style="	font-weight: 400;
">CrowdStrike Services identified a novel technique used by threat actors that escapes typical containment practices and permits persistence in victim AWS environments.</span></li><li><span style="	font-weight: 400;
">The technique requires that the adversary first obtain valid AWS API credentials with the necessary security token service (STS) and identity and access management (IAM) permissions, and then use the </span><code><span style="	font-weight: 400;
">sts:GetFederationToken</span></code><span style="	font-weight: 400;
"> API call to create a federated user session.</span></li><li><span style="	font-weight: 400;
">Permissions and access to the federated sign-in session survive the deactivation of the base user’s API credentials.</span></li><li><span style="	font-weight: 400;
">Responders should attach an explicit deny-all IAM policy to compromised IAM users as a containment measure. If a root API key is compromised, containment is much more challenging and may not always be possible. </span><b>Note:</b><span style="	font-weight: 400;
"> Creating an AWS API key (a prerequisite for this technique) for the root user is discouraged by AWS. Organizations should use the root user sparingly, and only via </span><a href="https://docs.aws.amazon.com/accounts/latest/reference/best-practices-root-user.html#bp-root-lock-away-credentials"><span style="	font-weight: 400;
">the AWS console</span></a><span style="	font-weight: 400;
">.</span></li></ul>
<p><span style="	font-weight: 400;
">In recent incident response </span><a href="/en-us/blog/analysis-of-intrusion-campaign-targeting-telecom-and-bpo-companies/"><span style="	font-weight: 400;
">investigations</span></a><span style="	font-weight: 400;
">, CrowdStrike Services has observed adversaries use the </span><code><span style="	font-weight: 400;
">sts:GetFederationToken</span></code><span style="	font-weight: 400;
"> API call to create federated sessions from IAM users. In this scenario, the federated session inherits permissions from the base IAM user. Perhaps surprising to many incident responders, the privileges and access of the federated session are not revoked when the base IAM user’s credentials are deactivated. Many defenders may not realize it, but this is a characteristic of all forms of temporary credentials obtained via the various API calls to AWS STS. The resulting sessions are independent of the long-term credentials of the underlying IAM or root user. The privileges of the federated session are only reduced or revoked when the IAM policies attached to the base IAM user are modified </span><a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_temp_control-access_disable-perms.html"><span style="	font-weight: 400;
">to reduce or revoke those permissions</span></a><span style="	font-weight: 400;
">. (Deleting the IAM user will effectively revoke all privileges but CrowdStrike does not recommend doing so, as detailed below.). The federated session remains active until the session token expires.<sup>1</sup> It is common for incident responders to only deactivate the base IAM user’s credentials. However, this will not be sufficient to contain an attack utilizing this technique. CrowdStrike is posting this blog to raise awareness of the use of this technique by threat actors and how threat actors can expand its use to establish persistence and escalate privileges within AWS environments if the underlying IAM user already had those escalation privileges.</span> <b>Learn how </b><a href="/en-us/services/"><b>CrowdStrike Services</b></a><b> can help your organization respond to and recover from incidents with speed and precision, prepare to defend against sophisticated threats and fortify your cybersecurity practices.</b></p>
<h2><span style="	font-weight: 400;
">What Is a Federated Session?</span></h2>
<p><span style="	font-weight: 400;
">A federated session in AWS involves exchanging long-lived credentials for a temporary set of credentials: an AWS access key, AWS secret key and AWS session token, issued by AWS STS. AWS has stated that federated sessions were predominantly used within custom access brokers before the advent of modern methods using IAM roles instead.</span> </p>
<p><span style="	font-weight: 400;
">AWS’s best practice is to use temporary credentials instead of long-lived credentials (such as API keys and passwords tied to local IAM users) </span><a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html#bp-users-federation-idp"><span style="	font-weight: 400;
">in most use cases</span></a><span style="	font-weight: 400;
">. AWS also recommends using federation-based </span><a href="https://aws.amazon.com/iam/identity-center.html"><span style="	font-weight: 400;
">AWS Identity Center</span></a><span style="	font-weight: 400;
"> for managing access for all human users. Using temporary security credentials enhances security by eliminating the exposure from static credentials that are vulnerable to theft and ensuring that any stolen session tokens have a limited lifespan.</span> </p>
<p><span style="	font-weight: 400;
">A method for federating users that, in CrowdStrike’s experience, is not well known by cloud practitioners is federating directly from IAM user and root principals. All IAM users that have been granted the permission </span><code><span style="	font-weight: 400;
">sts:GetFederationToken</span></code><span style="	font-weight: 400;
"> and the account’s root user are capable of acting as local identity providers and issuing temporary credentials for this type of federation. AWS describes a use case for this capability as:</span></p>
<blockquote><span style="	font-weight: 400;
">A typical use is in a proxy application that gets temporary security credentials on behalf of distributed applications inside a corporate network. You must call the </span><a href="https://docs.aws.amazon.com/STS/latest/APIReference/API_GetFederationToken.html"><span style="	font-weight: 400;
">GetFederationToken operation</span></a><span style="	font-weight: 400;
"> using the long-term security credentials of an IAM user. As a result, this call is appropriate in contexts where those credentials can be safely stored, usually in a server-based application.</span></blockquote>
<p><span style="	font-weight: 400;
">Because it is older and less commonly used, most organizations are unaware of the user-based federation capability and treat IAM users as local service accounts whose permissions are bound to the long-term credentials of that single entity. However, using the </span><code><span style="	font-weight: 400;
">sts:GetFederationToken</span></code><span style="	font-weight: 400;
"> API, a user can pass its privileges for use in a derivative federated session, which operates with distinct credentials.</span> </p>
<p><span style="	font-weight: 400;
">AWS places certain limits on federation-based privilege inheritance: The temporary credentials associated with the federated session cannot have greater privileges than the original identity. In addition, the federated session is blocked from exercising IAM actions from the AWS CLI or API. However, these IAM limits do not apply to console sessions, which is how an adversary with the compromised credentials of an IAM user with certain IAM privileges can abuse user-based federation to persist despite deactivation of the base identity’s compromised credentials.</span> </p>
<p><span style="	font-weight: 400;
">AWS previously indicated in its documentation that IAM limitations for federated sessions were limited to the CLI/SDK and did not apply when using the AWS console,<sup>2</sup> but after being notified of these concerns by CrowdStrike, AWS agreed the documentation could be made clearer and has done so by stating explicitly that IAM limitations do not apply to console usage. This change was made prior to the publication of this blog post.</span></p>
<h2><span style="	font-weight: 400;
">Persistence Technique</span></h2>
<p><span style="	font-weight: 400;
">This persistence technique takes advantage of the following:</span></p>
<ul><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Incident response teams are trained to preserve artifacts and affected resources such as IAM users during an incident. So instead of deleting the affected IAM user, or detaching IAM policies, they often will simply disable an IAM user’s API keys as a method of containment.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">Federated sessions can be used to make IAM changes via the AWS console</span></li></ul>
<p><span style="	font-weight: 400;
">The steps to utilize this technique are outlined below. Note that the technique works for an account’s root user as well as any IAM user with the required permissions, but in the interest of clarity, only the IAM user approach is described.</span></p>
<h3><span style="	font-weight: 400;
">1. Compromise an IAM user with the required permissions</span></h3>
<p><span style="	font-weight: 400;
">The technique begins when an adversary compromises a user who has the minimum permissions set of </span><code><span style="	font-weight: 400;
">sts:GetFederationToken</span></code><span style="	font-weight: 400;
">, </span><code><span style="	font-weight: 400;
">iam:ListAccessKeys</span></code><span style="	font-weight: 400;
"> and </span><code><span style="	font-weight: 400;
">iam:UpdateAccessKey</span></code><span style="	font-weight: 400;
">. This is unfortunately a common scenario, given that many operations teams attach the highly privileged </span><code><span style="	font-weight: 400;
">AdministratorAccess</span></code><span style="	font-weight: 400;
"> IAM policy to IAM users for ease of use, which allows all API actions.</span></p>
<div class="wp-caption" id="attachment_150905"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-8.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-150905 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-8.png" alt width="2416" height="1318" /></a>Figure 1. IAM policy granting permission to call the <code>sts:GetFederationToken</code> API call (click to enlarge)</div>
<p><span style="	font-weight: 400;
">This IAM user </span><b>does not need to have a password (console access) configured. </b><span style="	font-weight: 400;
">The technique will work as long as the adversary has access to API keys associated with the base IAM user.</span></p>
<div class="wp-caption" id="attachment_150906"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-10.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-150906 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-10.png" alt width="2512" height="1335" /></a>Figure 2. IAM user has no password, only an access key (clock to enlarge)</div>
<h3><span style="	font-weight: 400;
">2. Create a federated session from this IAM identity</span></h3>
<p><span style="	font-weight: 400;
">The next step is to use an API key associated with the compromised IAM user to get a federation token derived from the identity of that user. The API call requires the caller to specify a name for the session (in this case, “UnauthorizedUser”) and one or more in-line policies or policy ARNs to limit the privileges that will be granted to the session. The effective permissions of the resulting temporary credentials are the least-privileged subset of the requested permissions and the calling user’s permissions. By specifying </span><code><span style="	font-weight: 400;
">AdministratorAccess</span></code><span style="	font-weight: 400;
">, which allows all API actions, the adversary ensures that the federated session will have all of the permissions of the calling user.</span></p>
<div class="wp-caption" id="attachment_150907"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-11.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-150907 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-11.png" alt width="3987" height="1125" /></a>Figure 3. API call creating the federated session from the base IAM user (click to enlarge)</div>
<h3><span style="	font-weight: 400;
">3. The federated session permissions allow the adversary to perform all actions that the IAM user can perform.</span></h3>
<p><span style="	font-weight: 400;
">In the example in Figure 4, the adversary uses the new federated session tokens to exercise the </span><code><span style="	font-weight: 400;
">ec2:DescribeRegions</span></code><span style="	font-weight: 400;
"> permission inherited from the federating IAM user.</span></p>
<div class="wp-caption" id="attachment_150908"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-12.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-150908 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-12.png" alt width="4064" height="879" /></a>Figure 4. API calls made using the federated session (click to enlarge)</div>
<h3><span style="	font-weight: 400;
">4. The adversary can use the federated session to log into the console, even if the IAM user has no password.</span></h3>
<p><span style="	font-weight: 400;
">Converting the AWS CLI session to a console session is accomplished by using the federated session token to request a sign-in token from </span><code><span style="	font-weight: 400;
">https://signin.aws.amazon&lt;.&gt;com/federation</span></code><span style="	font-weight: 400;
"> and then using that token to authenticate to the AWS console endpoint </span><code><span style="	font-weight: 400;
">https://console.aws.amazon&lt;.&gt;com/console/home</span></code><span style="	font-weight: 400;
">. NetSPI has an easy-to-use tool for this </span><a href="https://github.com/NetSPI/aws_consoler"><span style="	font-weight: 400;
">on GitHub</span></a><span style="	font-weight: 400;
">. CrowdStrike has responded to </span><a href="/en-us/blog/analysis-of-intrusion-campaign-targeting-telecom-and-bpo-companies/"><span style="	font-weight: 400;
">incidents</span></a><span style="	font-weight: 400;
"> in which adversaries used the </span><code><span style="	font-weight: 400;
">aws_consoler</span></code><span style="	font-weight: 400;
"> tool to generate a sign-in token to access the AWS console using AWS API keys.</span></p>
<div class="wp-caption" id="attachment_150909"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-13.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-150909 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-13.png" alt width="2702" height="1337" /></a>Figure 5. Console login using the federated session (click to enlarge)</div>
<h3><span style="	font-weight: 400;
">5. Changing permissions of the originating IAM user immediately affects the federated session, but disabling the access key that was used to issue the session does not.</span></h3>
<p><span style="	font-weight: 400;
">While changes to the permissions of the base IAM user are propagated to the federated session, CrowdStrike found that disabling the access key that produced the federation session leaves the session intact for the remainder of the duration specified when it was created. In an incident response situation involving a compromised IAM user, many incident response teams leave the IAM user in place, but disable all access keys as part of containment. Based on CrowdStrike’s observations, this commonly leveraged remediation action is insufficient. Defenders must instead revoke or override all permissions for that user. CrowdStrike recommends attaching an explicit deny-all IAM policy to the IAM user, since explicit deny statements take precedence </span><a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/reference_policies_evaluation-logic.html"><span style="	font-weight: 400;
">over allow statements</span></a><span style="	font-weight: 400;
">. This would let incident responders preserve the IAM user’s existing permissions instead of having to record them separately and recreate them later.</span></p>
<div class="wp-caption" id="attachment_150910"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-14.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-150910 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-14.png" alt width="2833" height="1418" /></a>Figure 6. Disabling the base user’s access key does not invalidate the federated session (click to enlarge)</div>
<h3><span style="	font-weight: 400;
">6. If the base IAM user has the UpdateAccessKey IAM permissions assigned to it, the adversary can reactivate the deactivated credentials, establishing persistence.</span></h3>
<p><span style="	font-weight: 400;
">Adversaries also have a way to reestablish persistence using a lingering federated session. As noted earlier, </span><a href="https://docs.aws.amazon.com/STS/latest/APIReference/API_GetFederationToken.html"><span style="	font-weight: 400;
">AWS documentation on the </span><code><span style="	font-weight: 400;
">GetFederationToken</span></code><span style="	font-weight: 400;
"> API</span></a><span style="	font-weight: 400;
"> stated prior to the release of this blog post:</span></p>
<blockquote><span style="	font-weight: 400;
">You can use temporary credentials generated by </span><code><span style="	font-weight: 400;
">GetFederationToken</span></code><span style="	font-weight: 400;
"> in any AWS service except the following: You cannot call any IAM operations using the AWS CLI or the AWS API.</span></blockquote>
<p><span style="	font-weight: 400;
">While most IAM actions are denied when using the AWS CLI or the AWS API, these actions are </span><b>allowed</b><span style="	font-weight: 400;
"> when done using the AWS console. A lingering federated session allows the adversary to avoid the additional limitations AWS places on CLI/SDK-based IAM operations by converting to a console session. An adversary using a compromised access key with the sufficient IAM permissions can simply re-enable the key after incident responders have performed containment and then issue itself another session.</span></p>
<div class="wp-caption" id="attachment_150911"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-15.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-150911 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-15.png" alt width="2483" height="1243" /></a>Figure 7. The federated session is able to re-enable the disabled access key of the base user (click to enlarge)</div>
<p><span style="	font-weight: 400;
">In comparison, the IAM actions are denied through the CLI regardless of permissions.</span></p>
<div class="wp-caption" id="attachment_150912"><a href="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-16.png" target="_blank" rel="noopener noreferrer"><img class="wp-image-150912 size-full" src="/content/dam/crowdstrike/www/en-us/wp/2023/01/Picture1-16.png" alt width="6175" height="722" /></a>Figure 8. <h3><span style="	font-weight: 400;
">7. If the IAM user has the AttachUserPolicy or PutUpdatePolicy IAM permissions assigned to it (essentially permitting privilege elevation by policy), the federated session can create, modify or attach additional IAM policies to the base IAM user, potentially escalating their privileges — something that is not possible when using the CLI or API.</span></h3>
<p><span style="	font-weight: 400;
">Adversaries could similarly use the federated session to escalate their privileges if the base user has the requisite IAM permissions (</span><code><span style="	font-weight: 400;
">AttachUserPolicy</span></code><span style="	font-weight: 400;
"> or </span><code><span style="	font-weight: 400;
">UpdateUserPolicy</span></code><span style="	font-weight: 400;
">) assigned to it. (It should be noted that any principal with those unrestricted privileges is de facto a full administrator since they can add administrative privileges to themselves at any time.). Once logged in via the console, the federated session could be used to attach a privileged policy (such as </span><code><span style="	font-weight: 400;
">AdministratorAccess</span></code><span style="	font-weight: 400;
">) to the base IAM user, which would then elevate their own permissions to include </span><code><span style="	font-weight: 400;
">AdministratorAccess</span></code><span style="	font-weight: 400;
">.</span></p>

">Note that just as in the previous example, these actions </span><b>cannot</b><span style="	font-weight: 400;

<h3><span style="	font-weight: 400;
">8. The effective privileges of the federated session are only revoked when the base user’s permissions are detached, or an explicit deny-all IAM policy is applied.</span></h3>
<p><span style="	font-weight: 400;
">The only ways to fully contain an adversary using an IAM user federated session are to fully remove or override the permissions of the IAM user. CrowdStrike recommends overriding the base IAM user’s permissions with an explicit deny-all IAM policy, or if the AWS account is a member of an AWS Organization, an SCP rather than removing attached IAM policies. This would allow incident responders to more easily review the full scope of capabilities an adversary had during an incident and makes it easier for operations teams to restore functionality post-remediation.</span> <span style="	font-weight: 400;
">The example below shows an explicit deny-all IAM policy attached to the compromised IAM user, that results in the lingering federated session losing all permissions, even in the console.</span></p>

<h2><span style="	font-weight: 400;
">Recommendations</span></h2>
<p><span style="	font-weight: 400;
">CrowdStrike recommends that organizations take the following actions for prevention, detection, response and remediation to protect against this technique.</span></p>
<h3><span style="	font-weight: 400;
">Prevention</span></h3>
<ul><li><span style="	font-weight: 400;
">Create an SCP preventing the use of </span><code><span style="	font-weight: 400;
">sts:GetFederationToken</span></code><span style="	font-weight: 400;
"> for all IAM users:</span></li></ul>
<pre>
<code>{
  &#34;Version&#34;: &#34;2012-10-17&#34;,
  &#34;Statement&#34;: &lt;
    {
      &#34;Sid&#34;: &#34;DenyGetFederationToken&#34;,
      &#34;Effect&#34;: &#34;Deny&#34;,
      &#34;Principal&#34;: &#34;*&#34;,
      &#34;Action&#34;: &#34;sts:GetFederationToken&#34;,
      &#34;Resource&#34;: &#34;*&#34;
    }
  &gt;
}</code>
</pre>
<ul><li><span style="	font-weight: 400;
">Exercise good root user hygiene by never configuring </span><a href="https://docs.aws.amazon.com/accounts/latest/reference/best-practices-root-user.html#bp-root-lock-away-credentials"><span style="	font-weight: 400;
">API keys for the root user</span></a><span style="	font-weight: 400;
">, which is a prerequisite for creating federated sessions from the root user.</span></li><li><span style="	font-weight: 400;
">To the extent possible, CrowdStrike recommends avoiding the use of IAM users. Organizations should favor using temporary credentials over long-lived credentials associated with IAM users, by instead using </span><a href="https://aws.amazon.com/iam/identity-center.html"><span style="	font-weight: 400;
">AWS Identity Center</span></a><span style="	font-weight: 400;
"> to manage access for all human users, and IAM instance profiles for EC2 instance-backed workloads. This will avoid the use of long-lived AWS API keys within the AWS environment, which is a prerequisite for creating federated sessions from IAM users.</span></li></ul>
<h3><span style="	font-weight: 400;
">Detection</span></h3>
<ul><li><span style="	font-weight: 400;
">Detect and alert on users calling the </span><code><span style="	font-weight: 400;
">GetFederationToken</span></code><span style="	font-weight: 400;
"> API call. Here’s an example CloudTrail search: </span><code><span style="	font-weight: 400;
">eventName&#61;”GetFederationToken” AND eventSource&#61;”sts.amazonaws.com”</span></code></li><li><span style="	font-weight: 400;
">If federated sessions are not used widely in the AWS account or organization, the following CloudTrail search returning all actions taken using federated sessions will return high fidelity indicators of compromise: </span><code><span style="	font-weight: 400;
">userIdentity.type&#61;”FederatedUser”</span></code></li></ul>
<h3><span style="	font-weight: 400;
">Response and Remediation</span></h3>
<ul><li><span style="	font-weight: 400;
">As shown in the above attack description, deactivating the access key of the compromised base IAM user will not stop abuse using the federated session. Instead, CrowdStrike recommends using an SCP or explicit deny-all IAM policy (described below) to override the permissions assigned to the compromised IAM user. AWS also recently released a detailed</span><a href="https://aws.amazon.com/blogs/security/how-to-revoke-federated-users-active-aws-sessions.html"><span style="	font-weight: 400;
"> blog post</span></a><span style="	font-weight: 400;
"> on revoking role session tokens generated via an external identity provider that can be referenced for additional remediation strategies.</span></li></ul>
<pre>
<code>{
  &#34;Version&#34;: &#34;2012-10-17&#34;,
  &#34;Statement&#34;: &lt;
    {
      &#34;Sid&#34;: &#34;DenyAll&#34;,
      &#34;Effect&#34;: &#34;Deny&#34;,
      &#34;Action&#34;: &#34;*&#34;,
      &#34;Resource&#34;: &#34;*&#34;
    }
  &gt;
}</code>
</pre>
<h2><span style="	font-weight: 400;
">Summary</span></h2>
<p><span style="	font-weight: 400;
">CrowdStrike has observed threat actors leveraging federated users in recent investigations, and has identified how they can be used to establish persistence within AWS environments. The use of federated sessions is a method by which an adversary possessing persistent API credentials can generate temporary credentials that survive revocation of the original persistent credentials. CrowdStrike found that these permissions endure until the pre-established session expiration time, even if the credentials associated with the base user are deactivated. The only way to revoke the permissions of an IAM user’s federated session is to reduce the scope of permissions assigned to the base user. This can be done by either detaching all policies assigned to the base user, or attaching an explicit deny-all IAM policy or SCP to the base IAM user. Furthermore, federated sessions derived from the root user cannot be contained except through an SCP, and only for AWS accounts that are members of an AWS organization.</span></p>
<h2><span style="	font-weight: 400;
">Endnotes</span></h2>
<ol><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The default session duration when acting as an IAM user is 12 hours but can be set as high as 36 hours. When acting as a root user, the maximum duration is 1 hour. </span><a href="https://docs.aws.amazon.com/STS/latest/APIReference/API_GetFederationToken.html"><span style="	font-weight: 400;
">Learn more</span></a><span style="	font-weight: 400;
">.</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The documentation pages in question are </span><a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_temp_request.html#stsapi_comparison"><span style="	font-weight: 400;
">here</span></a><span style="	font-weight: 400;
"> and </span><a href="https://docs.aws.amazon.com/STS/latest/APIReference/API_GetFederationToken.html"><span style="	font-weight: 400;
">here</span></a><span style="	font-weight: 400;
">. Both pages state “You cannot call any IAM operations using the AWS CLI or the AWS API.” Based on input from CrowdStrike, both pages have been updated to add “This limitation does not apply to console sessions.”</span></li><li style="	font-weight: 400;
"><span style="	font-weight: 400;
">The CloudTrail events generated from this technique contain the </span><code><span style="	font-weight: 400;
">userIdentity.type</span></code><span style="	font-weight: 400;
"> field of </span><code><span style="	font-weight: 400;
">FederatedUser</span></code><span style="	font-weight: 400;
"> but it is important to understand that this is not a federated user in the sense of a user who was authenticated in an external directory.</span></li></ol>
