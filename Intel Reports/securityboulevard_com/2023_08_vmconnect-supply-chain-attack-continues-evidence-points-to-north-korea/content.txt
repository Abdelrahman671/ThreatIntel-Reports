As happened with the ReversingLabs team&#039;s earlier VMConnect research, the team was unable to obtain copies of the Stage 2 malware used in this campaign. We include a review of the similarities between this latest tranche of malicious Python packages and the earlier VMConnect packages, and we discuss the possible links to earlier software supply chain campaigns attributed to North Korean threat actors." />

<img style="position:relative;top:4px;left:0px;bottom:2px;width: 35px;" alt="SBN" src="https://securityboulevard.com/wp-content/uploads/2017/09/SBNIcon4_512px.png">
<h1 class="entry-title">
VMConnect supply chain attack continues, evidence points to North Korea </h1>
</header>
<div class="below-entry-meta">
<img alt src="https://secure.gravatar.com/avatar/4ea7a0988ca28899512e67b011b957b8?s=32&#038;d=mm&#038;r=g" srcset="https://secure.gravatar.com/avatar/4ea7a0988ca28899512e67b011b957b8?s=64&#038;d=mm&#038;r=g 2x" class="avatar avatar-32 photo" height="32" width="32" decoding="async" /> <span class="sbyline"><span class="author vcard"> <a class="url fn n" href="https://securityboulevard.com/author/karlo-zanki/"><span class="jcolor">by</span> Karlo Zanki <span class="jcolor">on</span> August 31, 2023</a></span></span>
</div>
<div class="entry-content clearfix">
<p><img fetchpriority="high" decoding="async" src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Supply-Chain-Campaign-Continues.jpg?width=1400&amp;height=732&amp;name=VMConnect-Supply-Chain-Campaign-Continues.jpg" width="1400" height="732" alt="VMConnect Supply Chain Campaign Continues" style="height: auto; max-width: 100%; width: 1400px;"></p>
<p>In early August, ReversingLabs identified a malicious supply chain campaign that <a href="https://www.reversinglabs.com/blog/vmconnect-malicious-pypi-packages-imitate-popular-open-source-modules">the research team dubbed “VMConnect.”</a> That campaign consisted of two dozen malicious Python packages posted to the Python Package Index (PyPI) open-source repository. The packages mimicked popular open-source Python tools, including <span style="font-style: italic;">vConnector</span>, a wrapper module for <span style="font-style: italic;">pyVmomi VMware vSphere</span> bindings; <span style="font-style: italic;">eth-tester</span>, a collection of tools for testing Ethereum-based applications; and <span style="font-style: italic;">databases</span>, a tool that gives asynchronous support for a range of databases.&nbsp;</p>
<p>The research team has continued monitoring PyPI&nbsp;and now has identified three more malicious Python packages that are believed to be a continuation of the VMConnect campaign: <span style="font-style: italic;">tablediter, request-plus, </span>and<span style="font-style: italic;"> requestspro</span>. As happened with the ReversingLabs team&#8217;s earlier VMConnect research, the team was unable to obtain copies of the Stage 2 malware used in this campaign. However, an analysis of the malicious packages used and their decrypted payloads reveals links to previous campaigns attributed to Labyrinth Chollima, an offshoot of Lazarus Group, a North Korean state-sponsored threat group.</p><div class="code-block code-block-15" style="margin: 8px 0; clear: both;">

We include a review of the similarities between this latest tranche of malicious Python packages and the earlier VMConnect packages, and we discuss the possible links to earlier software supply chain campaigns attributed to North Korean threat actors.</p>
<h2 style="font-weight: bold;">Discussion</h2>
<p>Most of the malicious actors pushing malicious open-source packages that ReversingLabs has observed in recent years employed some form of mimicry as part of their attack plan. That includes <a href="https://www.reversinglabs.com/glossary/typosquatting">so-called typosquatting attacks</a> in which malicious packages are given names and descriptions that closely resemble the names of legitimate open-source packages. The hope is that busy developers will enter a typo when they search for the packages and then install the malicious package without looking closely to spot the subtle differences.</p><div class="code-block code-block-12 ai-track" data-ai="WzEyLCIiLCJCbG9jayAxMiIsIiIsMV0=" style="margin: 8px 0; clear: both;">

<p>Other malicious payloads pose as new open-source modules that offer desired functionality while hiding malicious components such as backdoors and information-stealing features that developers are likely to miss.</p>

<h2 style="font-weight: bold;">Tablediter: A (not so) prettytable</h2>
<p>The <span style="font-style: italic;">tablediter</span> (pronounced “table editor”) Python package is a good example of the ways that malicious actors push their code to legitimate applications. Our researchers discovered this package in mid-August after noticing that it was mimicking <a href="https://pypi.org/project/prettytable/" style="font-style: italic;">prettytable</a>, a popular Python tool that developers use for printing tables in an attractive ASCII format. <span style="font-style: italic;">Prettytable</span> has more than 9 million monthly downloads, which makes it a highly attractive target for malicious actors.&nbsp;</p>
<p>Functionally, <span style="font-style: italic;">tablediter</span> is very similar to previously discovered malicious packages in the VMConnect campaign. However, there are some significant differences introduced in this package that make it harder to detect than previous packages. The biggest difference is that the malicious functionality within the package is not executed when the package is installed. Rather, it is triggered when the package gets used in a project. To accomplish this, the malicious actors responsible for creating <span style="font-style: italic;">tablediter</span> did not have the malicious functionality executed through the<span style="font-style: italic;"> __init__.py</span> file, which executes automatically when new packages are imported to a project.&nbsp;</p>
<p>Instead, the malicious code was added to a function called <span style="font-style: italic;">add_row,</span> which is a part of the <span style="font-style: italic;">tablediter</span> class defined in the <span style="font-style: italic;">tablediter.py</span> file. That could be called during testing of the application on a developer’s workstation or during execution by a user working with published software that has incorporated the malicious <span style="font-style: italic;">tablediter</span> dependency.&nbsp;</p>
<p>(As for why the attackers used the <span style="font-style: italic;">add_row</span> function to deliver the malicious code: A review of the documentation for <span style="font-style: italic;">prettytable</span>, which <span style="font-style: italic;">tablediter</span> is mimicking, shows that the <span style="font-style: italic;">add_row</span> function is the most-used function in the <span style="font-style: italic;">prettytable</span> package and the first code sample highlighted in the documentation that explains how to implement <span style="font-style: italic;">prettytable</span>.)</p>
<p><a href="https://blog.reversinglabs.com/hubfs/Blog/VMConnect-Continues-Fig1.png" class="lightbox-image"><img decoding="async" src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Continues-Fig1.png?width=1455&amp;name=VMConnect-Continues-Fig1.png" alt="Malicious code added to add_row function" width="1455" style="width: 1455px;"></a> </p>
<p style="text-align: center;"><span style="font-weight: bold; font-size: 16px;">Figure 1: Malicious code added to add_row function inside the tablediter.py file</span></p>
<p>When executed, the code invokes a method from a file, <em>bounding.py</em>, that is located in the edt subdirectory. The invoked method then receives a parameter that represents a XOR key used to decrypt the content of a long, hex-encoded string enclosed in the package.</p>
<p><a href="https://blog.reversinglabs.com/hubfs/Blog/VMConnect-Continues-Fig2.png" class="lightbox-image"><img decoding="async" src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Continues-Fig2.png?width=1455&amp;name=VMConnect-Continues-Fig2.png" alt="Decryption function from the bounding.py file" width="1455" style="width: 1455px;"></a> </p>
<p style="text-align: center;"><span style="font-weight: bold; font-size: 16px;">Figure 2: Decryption function from the bounding.py file</span></p>
<p>After that hex-encoded string is decrypted, it reveals a template URL that is modified to use the Internet address of the attackers’ command-and-control (C2) server. Specifically, the hex-encoded template contains a placeholder for the C2 server and malicious functionality. The code responsible for decoding that hex code (Figure 2) contains the real C2 address, which overwrites the placeholder value in the hex string. The decoded hex string (Figure 3) includes a placeholder for the C2 domain (ADDR) and for the infected machine (<span style="font-style: italic;">paperpin3902</span>) in communication with the C2 infrastructure. That value gets replaced with a string constructed of the &lt;first letter of the platform&gt;_&lt;login name of the user&gt;_&lt;random 6-character string&gt;). Finally, the decrypted code gets executed.&nbsp;</p>
<p>Research team analysis revealed that the functionality of this decrypted code is almost identical to that extracted from the VMConnect package we <a href="https://www.reversinglabs.com/blog/vmconnect-malicious-pypi-packages-imitate-popular-open-source-modules">wrote about in early August</a>. As observed at the time, that package contained a Base64 string containing an endless execution loop. When that Base64 string was decoded and executed, it contacted the C2 server and attempted to download another Base64-encoded string with additional commands. When successful, that code was executed and the loop repeated, with the malicious C2 server polled by the infected host for new commands after a preconfigured sleep period.</p>
<p><a href="https://blog.reversinglabs.com/hubfs/Blog/VMConnect-Continues-Fig3.png" class="lightbox-image"><img decoding="async" src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Continues-Fig3.png?width=1455&amp;name=VMConnect-Continues-Fig3.png" alt="Content of the decrypted payload string" width="1455" style="width: 1455px;"></a> </p>
<p style="text-align: center;"><span style="font-weight: bold; font-size: 16px;">Figure 3: Content of the decrypted payload string</span></p>
<p>The main differences in the code executed from <span style="font-style: italic;">tablediter</span> and the malware from the VMConnect package is that the former uses a combination of XOR encryption and hex encoding instead of the Base64 encoding used by VMConnect. The <span style="font-style: italic;">tablediter</span> package also does away with the <span style="font-style: italic;">execute-upon-install</span> functionality, which is the most encountered execution method for the malware observed in PyPI packages.&nbsp;</p>
<p>The decision not to execute malware automatically upon the installation of the tablediter package is almost certainly an effort by the attackers to avoid detection by traditional security monitoring tools that rely on dynamic analysis. Waiting until the designated package is imported and its functions called by the compromised application is a way to avoid one form of common behavior-based detection and raise the bar for would-be defenders.&nbsp;</p>
<p>Researchers observed malware design elements similar to what has been seen in previous research. For example, <a href="https://www.reversinglabs.com/blog/sentinelsneak-malicious-pypi-module-poses-as-security-sdk">SentinelSneak</a>, an imposter SentinelOne PyPI package exposed in December 2022, was also designed so that the malware waited for a malicious function to be called on programmatically before it was activated.</p>
<p>For organizations that are looking only for “the usual suspects,” this approach of lying low is often a sufficient ruse to avoid detection. In contrast, <a href="https://www.reversinglabs.com/products/software-supply-chain-security">ReversingLabs Software Supply Chain Security platform</a> is capable of extracting a wide range of behavior indicators during static analysis, allowing the team to detect this type of threat before a malicious package is imported to a legitimate application.</p>
<h2 style="font-weight: bold;">Requests denied: Request-plus and requestspro</h2>
<p>In addition to the tablediter package, ReversingLabs researchers discovered two other malicious packages that targeted a hyper-popular Python package: the <a href="https://requests.readthedocs.io/en/latest/"><em>requests</em></a> HTTP library on PyPI.&nbsp;</p>
<p><span style="font-style: italic;">Requests</span> is a standard HTTP library with thousands of monthly downloads and more than 2.3 million dependencies. The malicious packages we detected use names that are close cousins to the legitimate package: <span style="font-style: italic;">request-plus</span> and <span style="font-style: italic;">requestspro</span>. As with the <span style="font-style: italic;">tablediter</span> package, these packages were constructed with special care to avoid detection both before and after installation.&nbsp;</p>
<p>The evasion techniques employed by the malicious actors behind this campaign include typosquatting and other means of impersonation. For example, attackers copied the description of the <span style="font-style: italic;">requests</span> package on PyPI and pasted it into the description for the phony packages, updating the package name across the documentation references.&nbsp;<br/>Beyond that, the attackers exactly reproduced the files found within the legitimate <span style="font-style: italic;">requests</span> package in their imposter packages, with no new files added as compared to the legitimate <span style="font-style: italic;">requests</span> package.&nbsp;</p>
<p>The only modifications made to the malicious packages were found in the <span style="font-style: italic;">__init__.py</span> file, which was modified to include a few lines of code responsible for launching a thread that executes a function from the cookies.py file.&nbsp;</p>
<p>In addition, the <span style="font-style: italic;">cookies.py</span> file was modified to contain several malicious functions. Those include functions to collect information about the infected machine and send it to a URL referencing the C2 server in the form of a <span style="font-style: italic;">POST HTTP</span> request. The response from the server is a token that gets sent back to a different URL on the same C2 server, this time in the form of a <span style="font-style: italic;">GET HTTP</span> request.&nbsp;</p>
<p><a href="https://blog.reversinglabs.com/hubfs/Blog/VMConnect-Continues-Fig4.png" class="lightbox-image"><img decoding="async" src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Continues-Fig4.png?width=1455&amp;name=VMConnect-Continues-Fig4.png" alt="Code responsible for communication with C2 server" width="1455" style="width: 1455px;"></a> </p>
<p style="text-align: center;"><span style="font-weight: bold; font-size: 16px;">Figure 4: Code responsible for communication with C2 server</span></p>
<p>The received response by the infected host is a double-encrypted (Base64 and XOR) Python module with accompanying execution parameters, including another download URL.&nbsp;</p>
<p><a href="https://blog.reversinglabs.com/hubfs/Blog/VMConnect-Continues-Fig5.png" class="lightbox-image"><img decoding="async" src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Continues-Fig5.png?width=1455&amp;name=VMConnect-Continues-Fig5.png" alt="Code responsible for decryption of the received payload" width="1455" style="width: 1455px;"></a> </p>
<p style="text-align: center;"><span style="font-weight: bold; font-size: 16px;">Figure 5: Code responsible for decryption of the received payload</span></p>
<p>The team believes the module gets executed after decoding and then downloads the next stage of the malware. As was the case in the earlier iteration of the VMConnect campaign, the C2 server associated with the campaign did not provide additional commands by default, but rather waited for a suitable target, making it difficult to assess the full scope of the campaign.</p>
<h2 style="font-weight: bold;">A Lazarus connection?</h2>
<p>In an effort to better understand the origins of the VMConnect campaign, the ReversingLabs research team analyzed the malware samples discovered as part of this extended VMConnect campaign with the goal of linking this campaign to other known malware campaigns. In the process, the team identified clues that point in the direction of Lazarus Group, the North Korean advanced persistent threat (APT) group that has been linked to a number of sophisticated campaigns.&nbsp;</p>
<p>By experimenting with some <a href="https://www.reversinglabs.com/glossary/yara-rule">threat hunting YARA rules</a> based on the samples collected in the latest campaign, for example, our researchers discovered a package, <span style="font-weight: normal; font-style: italic;">py_QRcode,</span> that contains a <em>builder.py </em>file with malicious functionality that is very similar to that found in the VMConnect package.&nbsp;</p>
<p>Figures 6 and 7 show highlighted similarities in the code responsible for decryption of payload, which, in both cases, is executed with a call to subprocess the Popen function. &nbsp;</p>
<p><a href="https://blog.reversinglabs.com/hubfs/Blog/VMConnect-Continues-Fig6-7.png" class="lightbox-image"><img decoding="async" src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Continues-Fig6-7.png?width=1455&amp;name=VMConnect-Continues-Fig6-7.png" alt="VMConnect" width="1455" style="width: 1455px;"></a> </p>
<p style="font-weight: bold; text-align: center; font-size: 16px;">Figure 6: Code responsible for payload decoding and execution from VMConnect package<br/>Figure 7: Code responsible for payload decryption and execution from py_QRcode package</p>
<p>Among the similarities worth noting are:&nbsp;</p>
<p><span style="font-weight: bold; color: #f6143f;">•</span> <span style="font-weight: bold;">Functionality</span> related to the adaptation of file paths based on the architecture, which is determined with the call to <em>platform.system() </em>function in both cases (refer to the blue boxes)<br/><span style="font-weight: bold; color: #f6143f;">•</span> <span style="font-weight: bold;">Use of a variable containing B64-encoded text</span> to identify a next level payload (refer to the yellow boxes)<br/><span style="font-weight: bold; color: #f6143f;">•</span> <span style="font-weight: bold;">The code</span> responsible for B64 decoding the payload and writing it to a local file (refer to the green boxes)&nbsp;<br/><span style="font-weight: bold; color: #f6143f;">•</span> <span style="font-weight: bold;">Creation of a process</span> based on the determined platform via a call to <em>platform.system() </em>with similar branching and creation flags (refer to the red boxes)&nbsp;</p>
<p>More similarities appear when looking at the contents of the decrypted payloads (Figures 8 and 9). Here, nearly identical functionality in both payloads that periodically poll the C2 server for instructions can be observed. In both cases, this happens via an endless loop with 60-second sleep periods between the polls. And, in both cases, the received instructions are Base64-encoded strings representing a series of Python commands that are executed after decoding.</p>
<p><a href="https://blog.reversinglabs.com/hubfs/Blog/VMConnect-Continues-Fig8.png" class="lightbox-image"><img decoding="async" src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Continues-Fig8.png?width=1455&amp;name=VMConnect-Continues-Fig8.png" alt="Code responsible for C2 server polling in payload from VMConnect package" width="1455" style="width: 1455px;"></a> </p>
<p style="text-align: center;"><span style="font-weight: bold; font-size: 16px;">Figure 8: Code responsible for C2 server polling in payload from VMConnect package</span></p>
<p><a href="https://blog.reversinglabs.com/hubfs/Blog/VMConnect-Continues-Fig9.png" class="lightbox-image"><img decoding="async" src="https://www.reversinglabs.com/hs-fs/hubfs/Blog/VMConnect-Continues-Fig9.png?width=1455&amp;name=VMConnect-Continues-Fig9.png" alt="Code responsible for C2 server polling in payload from py_QRcode package" width="1455" style="width: 1455px;"></a> </p>
<p style="text-align: center;"><span style="font-weight: bold; font-size: 16px;">Figure 9: Code responsible for C2 server polling in payload from py_QRcode package</span></p>
<p>Further investigation revealed that the discovered <span style="font-weight: normal; font-style: italic;">py_QRcode</span> package had already been <a href="https://blogs.jpcert.or.jp/en/2023/07/dangerouspassword_dev.html?&amp;web_view=true">described in a report published in July, 2023 by Japan’s Computer Emergency Response Team Coordination Center (JPCERT/CC)</a>. The JPCERT report describes discovering malware samples targeting Windows, macOS, and Linux environments with installed Python and Node.js runtimes.&nbsp;</p>
<p>The report said the starting point of these malware infections on Windows machines was the execution of the mentioned <span style="font-style: italic;">py_QRcode</span> package. However, neither ReversingLabs research nor the research conducted by JPCERT/CC was able to find any evidence of that package ever being published to the PyPI repository. That leaves in question how the malware was distributed to victims.</p>
<p>The JPCERT research identified <span style="font-style: italic;">PythonHTTPBackdoor</span> as a possible Stage 2 malware sample downloaded as part of this campaign. In the macOS environment, <a href="https://www.intego.com/mac-security-blog/jokerspy-backdoor-mac-malware-discovered-in-the-wild/">JokerSpy</a> was the detected Stage 2 malware.&nbsp;</p>
<p>Finally, the JPCERT analysis <a href="https://www.sentinelone.com/blog/jokerspy-unknown-adversary-targeting-organizations-with-multi-stage-macos-malware/">references research published by SentinelOne</a> that talks about QRLog — Java malware with functionality that is identical to that found in the <span style="font-style: italic;">py_QRcode</span> package. That includes code that determines the host device’s operating system and then decodes a large base64 string that is written out to a temporary directory. That code then gets executed. Of note: The same <span style="font-style: italic;">www.git-hub[.]me</span> C2 domain is also found in both the <span style="font-style: italic;">QRLog</span> and <span style="font-style: italic;">py_QRcode</span> malware samples.&nbsp;</p>
<p>As for attribution, ReversingLabs is unable to definitively attribute this campaign to any specific threat actor. However, <a href="https://github.com/MauroEldritch">Mauro Eldritch</a>, the researcher who initially discovered the malicious <span style="font-style: italic;">QRLog</span> package, shared his findings with the cybersecurity firm Crowdstrike. Analysts at Crowdstrike attributed the malware to<span style="font-weight: bold;"> </span>Labyrinth Chollima, a subgroup within the Lazarus Group, a North Korean state-sponsored threat group, with a high degree of confidence.&nbsp;</p>
<p>A similar attribution was made by the JPCERT/CC, which linked the attack it uncovered to DangerousPassword, another subsidiary of the Lazarus Group. &nbsp;</p>
<p>Based on those attributions and the described code similarities between the packages discovered in the VMConnect campaign and the campaign described in the research published by JPCERT/CC, the ReversingLabs research team has reached the conclusion that the same threat actor is behind both attacks and, therefore, that the VMConnect malicious campaign activity can be linked to the North Korean state-sponsored Lazarus Group.</p>
<h2 style="font-weight: bold;">Conclusion</h2>
<p>ReversingLabs discovered three additional malicious packages with links to the VMConnect software supply chain campaign: <span style="font-style: italic;">request-plus</span>, <span style="font-style: italic;">requestspro,</span> and <span style="font-style: italic;">tablediter</span>. The team provides evidence that the VMConnect malicious software supply chain campaign that the team discovered in July and disclosed in early August is ongoing. As part of that, threat actors continue to use the Python Package Index (PyPI) repository as a distribution point for their malware. This is just another in a line of malicious attacks targeting users of the PyPI repository, including the recent campaign connected with the <a href="https://www.reversinglabs.com/blog/more-malicious-npm-packages-found-in-wake-of-jumpcloud-supply-chain-hack">JumpCloud incident</a>.&nbsp;</p>
<p>As with prior software supply chain campaigns, including <a href="https://www.reversinglabs.com/blog/iconburst-npm-software-supply-chain-attack-grabs-data-from-apps-websites">IconBurst</a>, <a href="https://www.reversinglabs.com/blog/sentinelsneak-malicious-pypi-module-poses-as-security-sdk">SentinelSneak,</a> and others, the malicious actors behind VMConnect took steps to disguise their malicious payloads and make their published packages look trustworthy, despite the existence of malicious functionality. Those efforts include standard practices such as typosquatting on the names of popular open-source packages and appropriating package descriptions and other metadata to confuse developers.&nbsp;</p>
<p>Links to the VMConnect campaign are also supported by deep similarities in the code used in the three newly discovered malicious packages, as well as shared C2 infrastructure.&nbsp;</p>
<p>In the latest tranche of packages, the malicious actors also took steps to avoid detection by <a href="https://www.reversinglabs.com/glossary/dast">dynamic application security testing (DAST) tools</a>. They did so by designing packages to execute their malicious payloads only after they had been imported to and called on by legitimate applications rather than immediately upon installation of the package — a technique ReversingLabs previously observed in the SentinelSneak campaign.&nbsp;</p>
<p>The revelations about the ongoing VMConnect campaign are a reminder that organizations need to improve their cyber defensive capabilities to encompass the full range of possible threats and attacks — including software supply chain attacks. That requires firms to invest both the effort and resources needed to detect and prevent supply chain attacks before they cause material damage to their business.</p>
<p>That means investing more heavily in training and awareness campaigns that ensure developers will not fall for typosquatting and other impersonation attacks. It also highlights the need for tools and processes to ensure that any open-source or proprietary code is evaluated for the presence of suspicious or malicious indicators, including hidden (obfuscated) functionality, unexplained communications with third-party infrastructure, and more.</p>
