							<div class="content "><h1 id="domain-controller-allow-server-operators-to-schedule-tasks">Domain controller: Allow server operators to schedule tasks</h1><div class="display-flex justify-content-space-between align-items-center flex-wrap-wrap page-metadata-container">
										<div class="margin-right-xxs">
											<ul class="metadata page-metadata" data-bi-name="page info" lang="en-us" dir="ltr"><li>Article</li><li>
														<time class="is-invisible" data-article-date aria-label="Article review date" datetime="2016-08-31T00:00:00.000Z" data-article-date-source="ms.date">08/31/2016</time>
													</li></ul>

											<h2 id="ms--in-this-article" class="title is-6 margin-block-xs">In this article</h2>
										</nav><!-- <content> --><p>&nbsp;</p>
<p>Applies To: Windows Server 2003, Windows Vista, Windows XP, Windows Server 2008, Windows 7, Windows 8.1, Windows Server 2008 R2, Windows Server 2012 R2, Windows Server 2012, Windows 8</p>
<p>This security policy reference topic for the IT professional describes the best practices, location, values, and security considerations for this policy setting.</p>
<h2 id="reference">Reference</h2>
<p>This policy setting determines whether server operators are allowed to submit jobs by means of the <strong>at</strong> command. If you enable this policy setting, jobs that are created by server operators by means of the <strong>at</strong> command run in the context of the account that runs the Task Scheduler service. By default, that is the Local System account.</p>
<div class="NOTE">
<p>Note</p>
<p>This security option setting affects only the scheduler tool for the <strong>at</strong> command. It does not affect the Task Scheduler tool.</p>
</div>
<p>Enabling this policy setting means jobs that are created by server operators through the <strong>at</strong> command will be executed in the context of the account that is running that serviceâby default, that is the Local System account. This means that server operators can perform tasks that the Local System account is able to do, but server operators would normally not be able to do, such as add their account to the local Administrators group.</p>
<p>The impact of enabling this policy setting should be small for most organizations. Users, including those in the Server Operators group, will still be able to create jobs by using the Task Scheduler Wizard, but those jobs will run in the context of the account that the user authenticates with when setting up the job.</p>
<h2 id="possible-values">Possible values</h2>
<ul>
<li><p>Enabled</p>
</li>
<li><p>Disabled</p>
</li>
<li><p>Not defined</p>
</li>
</ul>
<h2 id="best-practices">Best practices</h2>
<ul>
<li>Best practices for this policy are dependent on your security and operational requirements for task scheduling.</li>
</ul>
<h2 id="location">Location</h2>
<p><em>GPO_name</em>\Computer Configuration\Windows Settings\Security Settings\Local Policies\Security Options</p>

<h2 id="operating-system-version-differences">Operating system version differences</h2>
<p>There are no differences in this policy between operating systems beginning with Windows Server 2003.</p>
<h2 id="policy-management">Policy management</h2>
<p>This section describes features and tools that are available to help you manage this policy.</p>
<h2 id="restart-requirement">Restart requirement</h2>
<p>None. Changes to this policy become effective without a computer restart when they are saved locally or distributed through Group Policy.</p>
<h2 id="command-line-tools">Command-line tools</h2>
<p>The <strong>at</strong> command schedules commands and programs to run on a computer at a specified time and date. The Schedule service must be running to use the <strong>at</strong> command. For more information about the <strong>at</strong> command, see <a href="cc772590(v=ws.11)" data-linktype="relative-path">At [Vista]</a>.</p>
<h2 id="security-considerations">Security considerations</h2>
<p>This section describes how an attacker might exploit a feature or its configuration, how to implement the countermeasure, and the possible negative consequences of countermeasure implementation.</p>
<h2 id="vulnerability">Vulnerability</h2>
<p>Tasks that run under the context of the Local System account can affect resources that are at a higher privilege level than the user account that scheduled the task.</p>
<h2 id="countermeasure">Countermeasure</h2>
<p>Disable the <strong>Domain controller: Allow server operators to schedule tasks</strong> setting.</p>
<h2 id="potential-impact">Potential impact</h2>
<p>The impact should be small for most organizations. Users (including those in the Server Operators group) can still create jobs by means of the Task Scheduler snap-in. However, those jobs run in the context of the account that the user authenticates with when setting up the job.</p>
