        <section class="gh-content gh-canvas is-body">
            <p>In this post we leverage passive DNS analysis tools to expand on an ACTINIUM intelligence report published by <a href="https://www.microsoft.com/en-us/security/blog/2022/02/04/actinium-targets-ukrainian-organizations/?ref=embeeresearch.io">Microsoft</a>. </p><p>This analysis will leverage the initial domains provided in the report to identify new domains of interest that match the reported style and structure detailed in the original report. The initial report was published in 2022, so the idea here is to simply expand and update on the initial intelligence. </p><p>The TLDR of this analysis is that we will leverage passive DNS analysis to identify common patterns (IP, Registration dates, Subdomain Patterns) in the domains provided by Microsoft, we will then leverage this pattern to obtain updated ACTINIUM domains which were not included (or likely have been added after) the publication of the initial report. </p><p><em>It is worth noting that the initial report from Microsoft is nearly 2 years old. The "new" intelligence obtained here simply highlights changes to the ACTINIUM infrastructure since the initial report was published. The initial report is fantastic and Microsoft almost certainly knows everything published here. <br><br>The key point of this blog is to highlight technical methods that can be utilised to obtain the same intelligence with tooling that is accessible to regular analysts. </em></p><h3 id="initial-intelligence-and-iocs">Initial Intelligence and IOC's</h3><p>The initial intelligence and IOC's are taken from this <a href="https://www.microsoft.com/en-us/security/blog/2022/02/04/actinium-targets-ukrainian-organizations/?ref=embeeresearch.io">Microsoft</a> blog on the ACTINIUM threat actor. </p><p>Here, Microsoft shared a total of 124 domains related to the actor. We will be leveraging these domains to identify patterns and identify new infrastructure. </p><figure class="kg-card kg-image-card kg-card-hascaption"><img src="https://www.embeeresearch.io/content/images/2024/03/image-42.png" class="kg-image" alt="" loading="lazy" width="1259" height="1109" srcset="https://www.embeeresearch.io/content/images/size/w600/2024/03/image-42.png 600w, https://www.embeeresearch.io/content/images/size/w1000/2024/03/image-42.png 1000w, https://www.embeeresearch.io/content/images/2024/03/image-42.png 1259w" sizes="(min-width: 720px) 720px"><figcaption><span style="white-space: pre-wrap;">ACTINIUMDomains Shared By Microsoft Blog</span></figcaption></figure><p>Microsoft has also reported that each of these domains has numerous subdomains which are generated from a wordlist. </p><p>The subdomain wordings are relatively random, and we will use this as an indicator to identify new domains. </p><figure class="kg-card kg-image-card"><img src="https://www.embeeresearch.io/content/images/2024/03/image-43.png" class="kg-image" alt="" loading="lazy" width="1249" height="1135" srcset="https://www.embeeresearch.io/content/images/size/w600/2024/03/image-43.png 600w, https://www.embeeresearch.io/content/images/size/w1000/2024/03/image-43.png 1000w, https://www.embeeresearch.io/content/images/2024/03/image-43.png 1249w" sizes="(min-width: 720px) 720px"></figure><h3 id="initial-search-for-ioc-commonalities">Initial Search for IOC Commonalities</h3><p>We can begin our analysis by performing a bulk search on the domains provided by Microsoft. We utilised Validin for this and a link to the bulk search can be found <a href="https://app.validin.com/bulk?job_id=948c1605-9949-4e8e-b08f-d2b344395e98&ref=embeeresearch.io">here</a>. </p><p>This enables us to view the timelines of IP addresses utilised by the 124 domains reported by Microsoft. </p><p>The full results on the page are extremely as each domains has individually resolved to dozens of IP addresses over the last few years. (Hence it is difficult to screenshot the vast amount of information provided in the bulk search)</p><p>Here we can see a short snippet of the IP history for <code>acetica[.]online</code> from the Microsoft report. </p><figure class="kg-card kg-image-card"><img src="https://www.embeeresearch.io/content/images/2024/03/image-44.png" class="kg-image" alt="" loading="lazy" width="2000" height="872" https://www.embeeresearch.io/content/images/size/w1600/2024/03/image-44.png 1600w, https://www.embeeresearch.io/content/images/2024/03/image-44.png 2116w" sizes="(min-width: 720px) 720px"></figure><p>We can also see the same for <code>akowaika[.]ru</code></p><figure class="kg-card kg-image-card"><img src="https://www.embeeresearch.io/content/images/2024/03/image-45.png" class="kg-image" alt="" loading="lazy" width="2000" height="755" srcset="https://www.embeeresearch.io/content/images/size/w600/2024/03/image-45.png 600w, list and utilised CyberChef to sort the results. </p><h3 id="identifying-commonalities-using-cyberchef">Identifying Commonalities Using CyberChef</h3><p>After exporting the results from Validin, we can import them into CyberChef and perform some basic analysis and sorting. </p><p>When accounting for all resolved records of the 124 initial domains, this export produced 20402 indicators (inclusive of duplicates where multiple domains shared the same IP). </p><figure class="kg-card kg-image-card"><img src="https://www.embeeresearch.io/content/images/2024/03/image-46.png" class="kg-image" alt="" loading="lazy" width="1281" height="461" srcset="https://www.embeeresearch.io/content/images/size/w600/2024/03/image-46.png 600w, https://www.embeeresearch.io/content/images/size/w1000/2024/03/image-46.png 1000w, https://www.embeeresearch.io/content/images/2024/03/image-46.png 1281w" sizes="(min-width: 720px) 720px"></figure><p>To better understand the data and identify pivot points, we can duplicate the results and use CyberChef to sort them by most occurrences. </p><p>The goal here is to identify infrastructure that was most re-used across the reported domains, and to use these as pivot points to identify new or unreported domains. </p><p>The sorting can be achieved with a <code>Unique</code> and <code>Sort</code> operation, which reveals the most common resolved IP address was <code>139.180.174[.]234</code>, with 49 occurrences across the 124 initial domains.</p><figure class="kg-card https://www.embeeresearch.io/content/images/size/w1600/2024/03/image-47.png 1600w, https://www.embeeresearch.io/content/images/2024/03/image-47.png 2065w" sizes="(min-width: 720px) 720px"></figure><h3 id="pivoting-on-the-resolved-ip">Pivoting On The Resolved IP</h3><p>We can take this most common IP <code>139.180.174[.]234</code> and search for it in the bulk export. </p><p>The goal here is to look for other patterns we can leverage. </p><p>In this case, we can see that the majority of IPs (not all shown here) were leveraging this IP only on <code>2022-07-27</code></p><figure class="kg-card kg-image-card"><img src="https://www.embeeresearch.io/content/images/2024/03/image-48.png" class="kg-image" alt="" loading="lazy" width="1970" height="395" 720px"></figure><figure class="kg-card kg-image-card"><img src="https://www.embeeresearch.io/content/images/2024/03/image-49.png" class="kg-image" alt="" loading="lazy" width="1986" height="677" id="pivoting-on-usage-dates">Pivoting On Usage Dates</h3><p>Using this new information on the resolution dates, we can pivot on the IP address and specify that we only want DNS resolutions that first
