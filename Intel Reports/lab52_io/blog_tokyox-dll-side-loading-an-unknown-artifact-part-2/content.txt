<p class="entry-meta"><time class="entry-time" itemprop="datePublished" datetime="2022-01-12T09:21:10+01:00">January 12, 2022</time></p></header><div class="entry-content" itemprop="text">
<p>As we mentioned in <a href="https://lab52.io/blog/tokyox-dll-side-loading-an-unknown-artifact/">the previous post</a>, we have performed an analysis of the threat which, lacking further information, we have not been able to identify it as a known threat. Thus, for the moment, we will keep referring to it as TokyoX.</p>



<p>This threat can only be found in memory, since it is encrypted on disk and its loading is performed as described in the previous post.</p>



<p>After the whole loading process, the first thing the threat does is to check for the existence of a mutex in the system with the name &#8220;aftdoslm&#8221; and if it finds it, it terminates its execution in order to avoid multiple instances of the threat running in parallel.</p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img decoding="async" width="993" height="695" src="https://lab52.io/blog/wp-content/uploads/2022/01/1.png" alt="" class="wp-image-1218" srcset="https://lab52.io/blog/wp-content/uploads/2022/01/1.png 993w, https://lab52.io/blog/wp-content/uploads/2022/01/1-300x210.png 300w, https://lab52.io/blog/wp-content/uploads/2022/01/1-768x538.png 768w" sizes="(max-width: 993px) 100vw, 993px" /></figure></div>



<p>It then generates a 32-character string similar to a md5 hash from the computer&#8217;s file system and Volume information, which is used as the victim ID and placed at the beginning of every packet it sends to the command and control server:</p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img decoding="async" width="558" height="441" src="https://lab52.io/blog/wp-content/uploads/2022/01/2.png" alt="" class="wp-image-1219" srcset="https://lab52.io/blog/wp-content/uploads/2022/01/2.png 558w, https://lab52.io/blog/wp-content/uploads/2022/01/2-300x237.png 300w" sizes="(max-width: 558px) 100vw, 558px" /></figure></div>



<p>It then obtains the user name, computer name, local IP and operating system version, with which it builds a string and generates a first packet for the C2:</p>



<p>At the time of the analysis the command and control server was not available. For that reason, some fields of the packets a priori not used by the threat, could not be identified. However, from the analysis of the threat assembly, it has been possible to observe that the response of the C2 is very similar to those sent by the threat, since it most likely contains the first 44 bytes of the threat request or a very similar content: the victim ID (in red) , the first three dwords (in green). After that part, it will have a fourth dword (in blue) that consists of a command code supported by the threat, after which, depending on the command, it may contain one information or another (in yellow).</p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img loading="lazy" decoding="async" width="562" height="117" src="https://lab52.io/blog/wp-content/uploads/2022/01/3.png" alt="" class="wp-image-1220" srcset="https://lab52.io/blog/wp-content/uploads/2022/01/3.png 562w, https://lab52.io/blog/wp-content/uploads/2022/01/3-300x62.png 300w" sizes="(max-width: 562px) 100vw, 562px" /></figure></div>



<p>The command code in question goes in LittleEndian, so for the command 7010, the field should contain &#8220;10 70 00 00&#8221;. Inside the threat code, a switch can be found that identifies 6 different commands:</p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img loading="lazy" decoding="async" width="1024" height="332" src="https://lab52.io/blog/wp-content/uploads/2022/01/4-1024x332.png" alt="" class="wp-image-1221" srcset="https://lab52.io/blog/wp-content/uploads/2022/01/4-1024x332.png 1024w, https://lab52.io/blog/wp-content/uploads/2022/01/4-300x97.png 300w, https://lab52.io/blog/wp-content/uploads/2022/01/4-768x249.png 768w, https://lab52.io/blog/wp-content/uploads/2022/01/4.png 1256w" sizes="(max-width: 1024px) 100vw, 1024px" /></figure></div>



<p>The first one is &#8220;6010&#8221;, which can be seen in the screenshot along with the contents of the network packet. This command is not among the options accepted by the threat, but it is the one that the threat itself uses when it sends the first packet to the command and control server, and we assume that it is the one used as a beacon when it does not intend to do anything, since a command that is not supported by the switch statement will simply cause a 5-minute sleep and a new request to C2 after this time.</p>



<p>Second comes the &#8220;6011&#8221; command which is relatively interesting since it first calls a function that will delete all the infection files and terminate the execution of the threat. The fact that makes it interesting to us is that after this call, inside the switch that controls each command, it appears again as a command to perform no action (in this case it should be the 6010 probably) and make another request to C2, although this code will never be executed because the initial function will close the process.</p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img loading="lazy" decoding="async" width="244" height="283" src="https://lab52.io/blog/wp-content/uploads/2022/01/5.png" alt="" class="wp-image-1222"/></figure></div>



<p>The next command is &#8220;7010&#8221; which scans the disks installed in the computer using the &#8220;GetLogicalDriveStringsW&#8221; API and sends the results to the command and control server.</p>



<p>The command &#8220;7011&#8221; imitates the &#8220;Dir&#8221; command as it receives after the code a folder path, and allows it to list the content of a folder, showing hidden and system files too.</p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img loading="lazy" decoding="async" width="460" height="147" src="https://lab52.io/blog/wp-content/uploads/2022/01/6.png" alt="" class="wp-image-1223" srcset="https://lab52.io/blog/wp-content/uploads/2022/01/6.png 460w, https://lab52.io/blog/wp-content/uploads/2022/01/6-300x96.png 300w" sizes="(max-width: 460px) 100vw, 460px" /></figure></div>



<p>The next command &#8220;7017&#8221; expects, after the command code, a structure with three dwords followed by a path with a filename, and allows to upload a file from the attacking computer to the victim machine. In order to do so, it first creates an empty file and then generates a new thread that will make a second request to the command and control server. Within the response from the C2, this request expects the contents of the file to be uploaded, which the requesting thread will store in the file generated by the main thread.</p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img loading="lazy" decoding="async" width="571" height="337" src="https://lab52.io/blog/wp-content/uploads/2022/01/7.png" alt="" class="wp-image-1224" srcset="https://lab52.io/blog/wp-content/uploads/2022/01/7.png 571w, https://lab52.io/blog/wp-content/uploads/2022/01/7-300x177.png 300w" sizes="(max-width: 571px) 100vw, 571px" /></figure></div>



<p>Similarly, the command &#8220;7018&#8221; also expects after the command code a structure with three dwords, identical to the &#8220;7017&#8221; command, followed by a path with a filename. In this case, it generates a new thread that tries to read that file within the victim computer and makes a second request to the C2 containing the raw content, allowing the attacker to exfiltrate files from the infected computer.</p>



<p>With respect to the three dwords received before the file path, it has only been possible to identify the third one for the moment, which consists of the total size of the file to be sent or received and is used before obtaining the file to reserve a buffer of that size, plus one in the Heap of the application in order to recover the content there before dumping it to disk or reading it from disk.</p>



<p>Finally, there is the &#8220;8001&#8221; command with a command string to be run in a cmd.exe shell, since  it calls CreateProcessW for &#8220;cmd.exe&#8221; with the input and output redirected by anonymous pipes to the executable itself. Then, it executes the command received in the request and replies to the C2 with the output of that command, thus forming a remote Shell of the victim machine.</p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img loading="lazy" decoding="async" width="354" height="68" src="https://lab52.io/blog/wp-content/uploads/2022/01/8.png" alt="" class="wp-image-1225" srcset="https://lab52.io/blog/wp-content/uploads/2022/01/8.png 354w, https://lab52.io/blog/wp-content/uploads/2022/01/8-300x58.png 300w" sizes="(max-width: 354px) 100vw, 354px" /></figure></div>



<div class="wp-block-image"><figure class="aligncenter size-large"><img loading="lazy" decoding="async" width="770" height="279" src="https://lab52.io/blog/wp-content/uploads/2022/01/9.png" alt="" class="wp-image-1226" srcset="https://lab52.io/blog/wp-content/uploads/2022/01/9.png 770w, https://lab52.io/blog/wp-content/uploads/2022/01/9-300x109.png 300w, https://lab52.io/blog/wp-content/uploads/2022/01/9-768x278.png 768w" sizes="(max-width: 770px) 100vw, 770px" /></figure></div>



<p>Although <strong>we do not have enough evidence to make any kind of attribution</strong>, there are some characteristic TTPs in the infection chain: the attacker using a Dll-sideload with a legitimate binary and a library, the fact that the library is responsible for decrypting and executing a third encrypted file in memory, and the fact of removing the MZ characters from the beginning of the binary and replacing them with another string. This last characteristic is very common in APT groups that use PlugX as the main threat, where instead of  using “tokyo” to replace MZ, they usually add “plug” at the beginning, which has motivated the name of the threat.</p>



<p>All of this explains <a href="https://www.virustotal.com/gui/file/382b3d3bb1be4f14dbc1e82a34946a52795288867ed86c6c43e4f981729be4fc/detection" data-type="URL" data-id="https://www.virustotal.com/gui/file/382b3d3bb1be4f14dbc1e82a34946a52795288867ed86c6c43e4f981729be4fc/detection">the attribution of this hash to PlugX</a> by some antivirus engines, although as it has been observed in the post, it is a somewhat smaller threat, which is not developed in Delphi as plugx and that keeps a greater resemblance to HttpBrowser in terms of capabilities. However, HttpBrowser does not share code or style of command codes with this sample, so it is most likely a different tool for the first infection stage.</p>
