class="entry-container"><div class="entry-content-wrap"><div class="entry-content"><p>Our lead security analyst <a href="https://twitter.com/liamsmith86">Liam Smith</a> recently worked on an <a href="https://blog.sucuri.net/2022/05/manually-identifying-an-x-cart-credit-card-skimmer.html">infected X-Cart website</a> and found two interesting credit card stealers there — one skimmer located server-side, the other client-side.</p><p>X-Cart’s e-commerce platform is not nearly as popular as <a href="https://blog.sucuri.net/2022/04/wordpress-overtakes-magento-in-credit-card-skimmers.html">Magento or WooCommerce</a> and as a result we don’t see as many threat actors targeting it. While we do still regularly find skimmers on X-Cart sites, they are usually more customized and don’t look like typical <a href="https://sucuri.net/guides/what-is-magecart/" target="_blank" rel="noopener">Magecart malware</a>.</p><h3>Server-Side Credit Card Stealer</h3><p>The server-side skimmer found in the compromised environment was very simple.</p><p>In the <b>payment/payment_swoosh_cc.php</b> file, hackers had added a single line of code that saved the base64-encoded <b>POST</b> data with payment details into a file on disk.</p><p>To make it less suspicious, the file had a .<b>jpg</b> extension and was placed in a directory with image files. To retrieve the stolen information, hackers simply needed to download their fake JPG file from time to time.</p><figure id="attachment_28846" aria-describedby="caption-attachment-28846" style="width: 650px" class="wp-caption aligncenter"><a href="https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details.png" target="_blank" rel="noopener"><img decoding="async" class="wp-image-28846 size-large" src="https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details-650x37.png" alt="Injected line of code that saves payment details into a fake image file" width="650" height="37" srcset="https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details-650x37.png 650w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details-600x34.png 600w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details-768x43.png 768w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details-390x22.png 390w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details-820x46.png 820w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details-1180x66.png 1180w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details-560x31.png 560w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details-80x4.png 80w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-code-saves-payment-details.png 1246w" sizes="(max-width: 650px) 100vw, 650px" /></a><figcaption id="caption-attachment-28846" class="wp-caption-text">Injected line of code that saves payment details into a fake image file</figcaption></figure><h3>Client-Side Credit Card Stealer</h3><p>The client side skimmer turned out to be much more interesting. The following obfuscated script was found in the <b>skin/common_files/check_cc_number_script.tpl</b> file.</p><figure id="attachment_28847" aria-describedby="caption-attachment-28847" style="width: 1208px" class="wp-caption aligncenter"><a href="https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="wp-image-28847 size-full" src="https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer.png" alt="Injected JavaScript skimmer" width="1208" height="712" srcset="https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer.png 1208w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer-600x354.png 600w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer-650x383.png 650w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer-768x453.png 768w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer-390x230.png 390w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer-820x483.png 820w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer-1180x695.png 1180w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer-560x330.png 560w, https://blog.sucuri.net/wp-content/uploads/2022/05/injected-js-skimmer-80x47.png 80w" sizes="auto, (max-width: 1208px) 100vw, 1208px" /></a><figcaption id="caption-attachment-28847" class="wp-caption-text">Injected JavaScript skimmer</figcaption></figure><p>The first steps of deobfuscation were obvious. We applied the <b>decodeURIComponent </b>function and prettified the resulting JavaScript code.</p><figure id="attachment_28848" aria-describedby="caption-attachment-28848" style="width: 1442px" class="wp-caption aligncenter"><a href="https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code.png" target="_blank" rel="noopener"><img loading="lazy" decoding="async" class="wp-image-28848 size-full" src="https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code.png" alt="Prettified code after the first steps of deobfuscation" width="1442" height="1600" srcset="https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code.png 1442w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-541x600.png 541w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-586x650.png 586w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-768x852.png 768w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-1384x1536.png 1384w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-390x433.png 390w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-820x910.png 820w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-1180x1309.png 1180w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-560x621.png 560w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-80x89.png 80w, https://blog.sucuri.net/wp-content/uploads/2022/05/dobefuscated-code-72x80.png 72w" sizes="auto, (max-width: 1442px) 100vw, 1442px" /></a><figcaption id="caption-attachment-28848" class="wp-caption-text">Prettified code after the first steps of deobfuscation</figcaption></figure><h3>Anti-Debugging Trap</h3><p>The resulting code was still obfuscated and didn’t reveal its functionality. It obviously contained functions to decode itself, so we executed the script to reconstruct its meaningful representation. At this step, we encountered the first obstacle. The script returned errors telling us that certain variables were “undefined”.</p><p>After the analysis, we figured out that the deobfuscation algorithm relied on the length of the code of one of the wrapper functions. The use of <b>decodeURIComponent</b> along with the prettification of the code to improve readability had significantly changed the length of that function and broke the deobfuscation algorithm.</p><p>We regularly come across this “anti-debugging” approach in malware. It can normally be circumnavigated by copying the original script and feeding its content to the function that calculates its length. In this case, because of the additional layer of obfuscation, we used a different approach. We wrote a brute force script that tried all possible numbers unless it found the one that produced meaningful deobfuscation results.</p><h3>Credit Card Stealer Functionality</h3><p>After we had successfully deobfuscated the script, all we needed to do was replace the encrypted values with their plain text versions and rename variables and function names to see how exactly the malicious code worked.</p><p>As we anticipated, the script contained typical skimmer functionality. It looked for the <b>input</b> and <b>select</b> form elements and sent their values to a third-party server as lightly encoded (<b>encodeURIComponent</b>) GET parameters of an injected script. To make sure that only valid payment details are sent to the attacker, the malware checks the values to verify  that at least one of the “<b>input</b>” elements contains a <b>valid credit card number</b> using a publicly available validation algorithm.</p><h3>DOM-based Obfuscation of the Exfiltration Domain</h3><p>At this stage, the only unknown left was the domain name of the exfiltration It is hosted on a server with the IP address <b>162.241.222 .226</b> (UNIFIEDLAYER-AS-1, US), which has a <a href="https://urlscan.io/ip/162.241.222.226">history of being used by phishers</a>.</p><p>The <b>winsiott[.]com</b> domain was registered on <b>June 7, 2021</b>. It is hosted on a server with the IP <b>83.242.96 .149</b> (AHOST-AS, RU).</p><h2>Conclusion &amp; Mitigation Steps</h2><p>Every e-commerce site is a potential target for bad actors trying to steal payment details, regardless of whether they&#8217;re using an ultra-popular CMS or a custom built solution. Less popular platforms usually attract more targeted attacks where hackers may employ more creative attack vectors and injected malware.</p><p>Websites that allow credit card transactions should take security very seriously — <a href="https://sucuri.net/guides/pci-compliance-requirements-checklist/">PCI compliance is mandatory</a> for anyone who operates an e-commerce website, and the implication of a compromise affects not only the personal information and security of the site’s users but also the website&#8217;s ability to accept credit card payments. Site owners may even incur fines or penalties from regulators if customers complain about fraudulent transactions.</p><p>One of the most important things you can do to protect your website against credit card stealing malware is to follow the requirements outlined in the <a href="https://www.pcisecuritystandards.org/document_library?category=pcidss&amp;document=pci_dss">PCI-DSS</a>.
