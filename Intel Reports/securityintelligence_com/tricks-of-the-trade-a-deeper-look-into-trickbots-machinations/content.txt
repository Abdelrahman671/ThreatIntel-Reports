&lt;h2&gt;An Unusual Man-in-the-Browser Technique&lt;/h2&gt;
Nowadays, most modern financial malware families are capable of injecting malicious code into ongoing browser sessions (e.g., MitB or a webinjection attack). The most common way malware developers implement injections is by setting them up locally at the victim&#039;s machine. The malware keeps a local configuration file for the injections, specifying exactly when and how the malware will modify the contents of targeted bank webpages.
&lt;p style=&quot;text-align: center;&quot;&gt;[button link=&quot;https://www-01.ibm.com/marketing/iwm/dre/signup?source=mrs-form-9451&amp;S_PKG=ov54550&quot; color=&quot;orange1&quot; icon=&quot;download&quot; size=&quot;small&quot;]Learn more about how to outsmart Fraudsters with Cognitive Fraud Detection[/button]&lt;/p&gt;
A more advanced and less common method to achieve the same result is to fetch the injection instructions from the attacker&#039;s server in real time. This is the method TrickBot&#039;s developers opted to use. It is also known as serverside injections.

For this purpose, and much like other advanced banking Trojans, TrickBot deploys a browser-hooking engine designed to intercept communications to and from the victim&#039;s internet browser. With the real-time fetching trick, the malicious code injections themselves are kept securely on the attacker&#039;s server, not in a file on the victim&#039;s endpoint.

When a victim browses one of TrickBot&#039;s target URLs, this is what actually happens:
&lt;ol&gt;
 	&lt;li&gt;TrickBot&#039;s financial module intercepts the original HTTP response before it is rendered to the victim. The browser does not display the &quot;clean&quot; response.&lt;/li&gt;
 	&lt;li&gt;TrickBot sends a multipart HTTP packet to its C2 with the following sections:
&lt;ol&gt;
 	&lt;li&gt;&quot;sourcelink,&quot; the complete URL that triggered the attack;&lt;/li&gt;
 	&lt;li&gt;&quot;sourcequery,&quot; the browser&#039;s complete HTTP query; and&lt;/li&gt;
 	&lt;li&gt;&quot;sourcehtml,&quot; the original HTML as would be displayed by an uninfected browser.&lt;/li&gt;
&lt;/ol&gt;
&lt;/li&gt;
 	&lt;li&gt;C2 replies with full HTML content to be rendered by the victim&#039;s browser, including injected parts.&lt;/li&gt;
 	&lt;li&gt;Finally, TrickBot&#039;s financial module replaces the original response that would normally come from the bank with the C2&#039;s response, and the injected page is displayed on the victim&#039;s end.&lt;/li&gt;
&lt;/ol&gt;
The serverside injection method has advantages over the standard, local mechanism used by most financial malware today. Notably, it allows for enhanced obscurity and flexibility. The malware&#039;s author can keep the injection code out of sight until it is needed. The actor can turn the webinjections on or off on the fly, easily modify the injections and then push an update to some or all the infected victims instantaneously.
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;img title=&quot;trickbot research browser hooks&quot; src=&quot;https://static.securityintelligence.com/uploads/2016/11/trickbot_1.png&quot; alt=&quot;TrickBot research IBM Security&quot; /&gt;&lt;/p&gt;

&lt;p style=&quot;text-align: center;&quot;&gt;&lt;em&gt;Figure 2: TrickBot&#039;s Server Side Web-Injects — Top Level Flow&lt;span style=&quot;color: #2f2f2f; font-family: &#039;Helvetica Neue&#039;; font-size: medium;&quot;&gt;&lt;i&gt;.&lt;/i&gt;&lt;/span&gt; &lt;/em&gt;&lt;/p&gt;

&lt;h2&gt;An Elegant Choice for API Obfuscation&lt;/h2&gt;
When it comes to keeping malware alive longer, it is common practice for malware authors to add protection layers to their code to ward off reverse engineering. As expected, we identified one such technique employed by TrickBot: API obfuscation.

Having analyzed TrickBot&#039;s obfuscation method, we found it very similar to — and likely borrowed from — the Carberp Trojan&#039;s API obfuscation. Carberp&#039;s source code was &lt;a href=&quot;https://www.google.co.uk/url?sa=t&amp;rct=j&amp;q=&amp;esrc=s&amp;source=web&amp;cd=2&amp;cad=rja&amp;uact=8&amp;ved=0ahUKEwjK2unIoozQAhXJOxQKHe4FC-8QFggpMAE&amp;url=https%3A%2F%2Fkrebsonsecurity.com%2F2013%2F06%2Fcarberp-code-leak-stokes-copycat-fears%2F&amp;usg=AFQjCNEcgZYIGMN2nxSi1InhoxDtCBL-lQ&amp;bvm=bv.137132246,d.d24&quot;&gt;leaked in 2013&lt;/a&gt;, giving rise to other malware based on its sophisticated DNA.

We found that TrickBot does not apply the API obfuscation to all the APIs; it only applies it to the more sensitive APIs that the developer wants to be hidden. This is a sneaky method, since researchers may believe they already know all the APIs being used, when in reality more APIs are covertly part of the game.

The obfuscation process here is based on precalculated hash values of the APIs. Calling an API function only includes a hash value instead of the function name, making static analysis harder unless the researcher applies an additional method to resolve the APIs.
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;img title=&quot;trickbot research ibm security&quot; src=&quot;https://static.securityintelligence.com/uploads/2016/11/trickbot_2.png&quot; alt=&quot;TrickBot WSAStartup hash&quot; /&gt;&lt;/p&gt;
&lt;p align=&quot;center&quot;&gt;&lt;em&gt;Figure 3: WSAStartup hash from Carberp&#039;s source code.&lt;/em&gt;&lt;/p&gt;

&lt;p align=&quot;center&quot;&gt;&lt;em&gt;Figure 4: Resolving an API by hash — WSAStartup.&lt;/em&gt;&lt;/p&gt;
A simple way of overcoming this obfuscation is by using an Interactive Disassembler (IDA) Python script, especially since the hashed values themselves are already available within Carberp&#039;s leaked source code.
&lt;h2&gt;A Bug in the Wild&lt;/h2&gt;
TrickBot has been in testing since summer 2016, even before it was equipped with financial malware features. Initially, TrickBot&#039;s developers appeared to struggle with the malware&#039;s webinjection mechanism, since we found a few TrickBot samples in the wild that presented strangely erratic behavior. At first, we suspected TrickBot was up to some anti-research wiles, but in reality, it was just bugged.

Per our analysis, TrickBot&#039;s webinjection malfunction caused the malware to constantly inject the same code over and over again, sabotaging the malware&#039;s own functionality. As this behavior was inconsistent across some samples, we had to manually apply a fix to continue researching the mechanism.

We won&#039;t get into further detail at this point, since this bug actually prevented TrickBot from performing fraud. We will say, however, that since the malware is under constant development, the developers may have already addressed the bug and fixed it in the newer samples, enabling TrickBot to operate more smoothly.
&lt;h2&gt;The TrickBot-Dyre Connection&lt;/h2&gt;
Speculation over the &lt;a href=&quot;https://securityintelligence.com/news/trickbot-malware-resurrects-the-ghost-of-dyre/&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;TrickBot-Dyre connection&lt;/a&gt; emerged as soon as the malware was discovered and has been the source of much debate ever since. Even though this subject was mentioned in several other &lt;a href=&quot;http://www.threatgeek.com/2016/10/trickbot-the-dyre-connection.html&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;TrickBot analysis blogs&lt;/a&gt;, we would like to contribute several key points from our own research into this new threat:
&lt;ul&gt;
 	&lt;li&gt;TrickBot&#039;s serverside webinjection method is uncommon in today&#039;s malware. The other malware family that used it, as you can guess, was Dyre.&lt;/li&gt;
&lt;/ul&gt;
&lt;ul&gt;
 	&lt;li&gt;Packets sent to the attack server during the serverside webinjects consist of three parts, titled &quot;sourcelink,&quot; &quot;sourcequery&quot; and &quot;sourcehtml.&quot; These exact names were also used in Dyre&#039;s webinjection mechanism.&lt;/li&gt;
&lt;/ul&gt;
&lt;p style=&quot;text-align: center;&quot;&gt;&lt;img title=&quot;trickbot malware research&quot; src=&quot;https://static.securityintelligence.com/uploads/2016/11/trickbot_5.png&quot; alt=&quot;TrickBot Dyre connection&quot; /&gt;&lt;/p&gt;
&lt;p class=&quot;&quot; style=&quot;text-align: center;&quot; align=&quot;center&quot;&gt;&lt;em&gt;Figure 5: TrickBot and Dyre both use &quot;sourcelink&quot; and &quot;sourcequery&quot; for their communications.&lt;/em&gt;&lt;/p&gt;

&lt;ul&gt;
 	&lt;li&gt;Targeted URLs and command-and-control (C&amp;C) addresses are kept encrypted on the infected machine. &lt;a href=&quot;https://securityintelligence.com/dyre-straights-group-behind-the-dyre-trojan-busted-in-moscow/&quot; target=&quot;_blank&quot; rel=&quot;noopener noreferrer&quot;&gt;Dyre did the same&lt;/a&gt;. While the encryption schemes for TrickBot are not identical, they appear to be too similar to be a mere coincidence.&lt;/li&gt;
&lt;/ul&gt;
&lt;ul&gt;
 	&lt;li&gt;TrickBot passes the target URLs list to its financial module, which is injected into the browser using pipes communication. This, again, is a Dyre hallmark.&lt;/li&gt;
&lt;/ul&gt;
&lt;ul&gt;
 	&lt;li&gt;The structure of the targeted URLs in the configuration is typically consistent for each malware, and TrickBot&#039;s target specification — you guessed it — sure looks a lot like Dyre&#039;s.&lt;/li&gt;
&lt;/ul&gt;
[insert-content id=&quot;381206&quot;]

Although the similarities are there, keep in mind that most of them are relatively simply to imitate. For example, even though the serverside webinjection technique is common to both Trojans, the code implementing this capability and the coding style itself are actually different.

This point is important because we can see how rapidly and effectively new and advanced malware is being developed, either by the same actors or by newcomers inspired by one of the most nefarious gangs in cybercrime history.
&lt;h2&gt;A Last-Minute Update: Redirection Attacks!&lt;/h2&gt;
TrickBot&#039;s developers must be hard at work these days, pushing to enhance the malware for a live campaign targeting banks. Just as we were about to publish this post, we detected a new infection campaign with a new configuration targeting U.K. banks. Until now, TrickBot only targeted banks in Australia. Furthermore, some of these new U.K. targets are set up for redirection attacks, while TrickBot only employed the serverside webinjection attack described above until now.

A redirection attack, in short, means that instead of injecting malicious code into the original webpage, the victim is now redirected to a new site forged by the fraudsters. This site looks and feels exactly like the original website, and the browser indicates a Secure Sockets Layer (SSL) connection based on the original site&#039;s certificate.

To learn more about redirection attacks and their purpose, read our blogs about &lt;a href=&quot;https://securityintelligence.com/dridex-launches-dyre-like-attacks-in-uk-intensifies-focus-on-business-accounts/&quot;&gt;Dridex&lt;/a&gt; and &lt;a href=&quot;https://securityintelligence.com/time-is-money-goznym-launches-redirection-attacks-in-poland/&quot;&gt;GozNym&lt;/a&gt;.
&lt;h2&gt;A Newcomer to the Malware Arena&lt;/h2&gt;
TrickBot is undoubtedly the work of professionals who have been around the banking Trojan scene for some time. These experienced fraudsters are apparently well-versed in the modern features common to the types of malware banks reckon with nowadays. We expect to see this Trojan evolve its anti-security and anti-research techniques and pop up in more infection campaigns as the year comes to a close.

<html><body><p>TrickBot is a new banking Trojan. It appears to be a Dyre successor that emerged in the wild in October 2016. TrickBot&rsquo;s code has been in <a href="http://www.threatgeek.com/2016/10/trickbot-the-dyre-connection.html" target="_blank" rel="noopener nofollow" >progressive testing</a> since August 2016. It continues to see ongoing updates and, now, actual infection campaigns and <a href="http://www.ibm.com/software/products/en/trusteer-fraud-protection-suite"  target="_blank" rel="noopener nofollow" >fraud attacks</a>.</p>
<p>Internally, there is more to <a href="https://securityintelligence.com/an-aggressive-launch-trickbot-trojan-rises-with-redirection-attacks-in-the-uk/"  target="_blank" rel="noopener nofollow" >TrickBot</a> than meets the eye. In this research post, we&rsquo;ll cover some of the most notable points about this malware&rsquo;s capabilities, including:</p>
<ul>
<li>An uncommon method of performing man-in-the-browser (MitB) attacks;</li>
</ul>
<ul>
<li>TrickBot&rsquo;s buggy webinjection mechanism;</li>
</ul>
<ul>
<li>The developer&rsquo;s elegant application program interface (API) obfuscation, borrowed from Carberp; and</li>
</ul>
<ul>
<li>Our two cents about the suspected TrickBot-Dyre connection.</li>
</ul>
<p>For analysis, the sample we used was: <a href="https://www.virustotal.com/en/file/5e363a42d019fc6535850a2867548f5b968d68952e1cddd49240d1f426debb73/analysis/" target="_blank" rel="noopener nofollow" >5e363a42d019fc6535850a2867548f5b968d68952e1cddd49240d1f426debb73</a>.</p>
<h2>An Unusual Man-in-the-Browser Technique</h2>
<p>Nowadays, most modern financial malware families are capable of injecting malicious code into ongoing browser sessions (e.g., MitB or a webinjection attack). The most common way malware developers implement injections is by setting them up locally at the victim&rsquo;s machine. The malware keeps a local configuration file for the injections, specifying exactly when and how the malware will modify the contents of targeted bank webpages.</p>
<p style="text-align: center;"><a class="btn orange1 small" href="https://www-01.ibm.com/marketing/iwm/dre/signup?source=mrs-form-9451&amp;S_PKG=ov54550" >Learn more about how to outsmart Fraudsters with Cognitive Fraud Detection <i class="fa fa-download"></i></a></p>
<p>A more advanced and less common method to achieve the same result is to fetch the injection instructions from the attacker&rsquo;s server in real time. This is the method TrickBot&rsquo;s developers opted to use. It is also known as serverside injections.</p>
<p>For this purpose, and much like other advanced banking Trojans, TrickBot deploys a browser-hooking engine designed to intercept communications to and from the victim&rsquo;s internet browser. With the real-time fetching trick, the malicious code injections themselves are kept securely on the attacker&rsquo;s server, not in a file on the victim&rsquo;s endpoint.</p>
<p>When a victim browses one of TrickBot&rsquo;s target URLs, this is what actually happens:</p>
<ol>
<li>TrickBot&rsquo;s financial module intercepts the original HTTP response before it is rendered to the victim. The browser does not display the &ldquo;clean&rdquo; response.</li>
<li>TrickBot sends a multipart HTTP packet to its C2 with the following sections:
<ol>
<li>&ldquo;sourcelink,&rdquo; the complete URL that triggered the attack;</li>
<li>&ldquo;sourcequery,&rdquo; the browser&rsquo;s complete HTTP query; and</li>
<li>&ldquo;sourcehtml,&rdquo; the original HTML as would be displayed by an uninfected browser.</li>
</ol>
</li>
<li>C2 replies with full HTML content to be rendered by the victim&rsquo;s browser, including injected parts.</li>
<li>Finally, TrickBot&rsquo;s financial module replaces the original response that would normally come from the bank with the C2&rsquo;s response, and the injected page is displayed on the victim&rsquo;s end.</li>
</ol>
<p>The serverside injection method has advantages over the standard, local mechanism used by most financial malware today. Notably, it allows for enhanced obscurity and flexibility. The malware&rsquo;s author can keep the injection code out of sight until it is needed. The actor can turn the webinjections on or off on the fly, easily modify the injections and then push an update to some or all the infected victims instantaneously.</p>
<p style="text-align: center;"><amp-img src="https://static.securityintelligence.com/uploads/2016/11/trickbot_1.png" layout="intrinsic" class="" alt="TrickBot research IBM Security" width="" height="" lightbox="lightbox"></amp-img></p>
<p style="text-align: center;" align="center"><em>Figure 1: TrickBot hooking FireFox&rsquo;s network functions to enable MitB interception.</em></p>
<p style="text-align: center;"><amp-img src="https://static.securityintelligence.com/uploads/2016/11/server-injects-diagram_ibm_pics.jpg" layout="intrinsic" class="" alt="TrickBot Malware Research Reversing" width="" height="" lightbox="lightbox"></amp-img></p>
<p style="text-align: center;"><em>Figure 2: TrickBot&rsquo;s Server Side Web-Injects &mdash; Top Level Flow<span style="color: #2f2f2f; font-family: 'Helvetica Neue'; font-size: medium;"><i>.</i></span> </em></p>
<h2>An Elegant Choice for API Obfuscation</h2>
<p>When it comes to keeping malware alive longer, it is common practice for malware authors to add protection layers to their code to ward off reverse engineering. As expected, we identified one such technique employed by TrickBot: API obfuscation.</p>
<p>Having analyzed TrickBot&rsquo;s obfuscation method, we found it very similar to &mdash; and likely borrowed from &mdash; the Carberp Trojan&rsquo;s API obfuscation. Carberp&rsquo;s source code was <a href="https://www.google.co.uk/url?sa=t&amp;rct=j&amp;q=&amp;esrc=s&amp;source=web&amp;cd=2&amp;cad=rja&amp;uact=8&amp;ved=0ahUKEwjK2unIoozQAhXJOxQKHe4FC-8QFggpMAE&amp;url=https%3A%2F%2Fkrebsonsecurity.com%2F2013%2F06%2Fcarberp-code-leak-stokes-copycat-fears%2F&amp;usg=AFQjCNEcgZYIGMN2nxSi1InhoxDtCBL-lQ&amp;bvm=bv.137132246,d.d24" target="_blank" rel="noopener nofollow" >leaked in 2013</a>, giving rise to other malware based on its sophisticated DNA.</p>
<p>We found that TrickBot does not apply the API obfuscation to all the APIs; it only applies it to the more sensitive APIs that the developer wants to be hidden. This is a sneaky method, since researchers may believe they already know all the APIs being used, when in reality more APIs are covertly part of the game.</p>
<p>The obfuscation process here is based on precalculated hash values of the APIs. Calling an API function only includes a hash value instead of the function name, making static analysis harder unless the researcher applies an additional method to resolve the APIs.</p>
<p style="text-align: center;"><amp-img src="https://static.securityintelligence.com/uploads/2016/11/trickbot_2.png" layout="intrinsic" class="" alt="TrickBot WSAStartup hash" width="" height="" lightbox="lightbox"></amp-img></p>
<p align="center"><em>Figure 3: WSAStartup hash from Carberp&rsquo;s source code.</em></p>
<p style="text-align: center;"><amp-img src="https://static.securityintelligence.com/uploads/2016/11/trickbot_3.png" layout="intrinsic" class="" alt="TrickBot banking Trojan" width="" height="" lightbox="lightbox"></amp-img></p>
<p align="center"><em>Figure 4: Resolving an API by hash &mdash; WSAStartup.</em></p>
<p>A simple way of overcoming this obfuscation is by using an Interactive Disassembler (IDA) Python script, especially since the hashed values themselves are already available within Carberp&rsquo;s leaked source code.</p>
<h2>A Bug in the Wild</h2>
<p>TrickBot has been in testing since summer 2016, even before it was equipped with financial malware features. Initially, TrickBot&rsquo;s developers appeared to struggle with the malware&rsquo;s webinjection mechanism, since we found a few TrickBot samples in the wild that presented strangely erratic behavior. At first, we suspected TrickBot was up to some anti-research wiles, but in reality, it was just bugged.</p>
<p>Per our analysis, TrickBot&rsquo;s webinjection malfunction caused the malware to constantly inject the same code over and over again, sabotaging the malware&rsquo;s own functionality. As this behavior was inconsistent across some samples, we had to manually apply a fix to continue researching the mechanism.</p>
<p>We won&rsquo;t get into further detail at this point, since this bug actually prevented TrickBot from performing fraud. We will say, however, that since the malware is under constant development, the developers may have already addressed the bug and fixed it in the newer samples, enabling TrickBot to operate more smoothly.</p>
<h2>The TrickBot-Dyre Connection</h2>
<p>Speculation over the <a href="https://securityintelligence.com/news/trickbot-malware-resurrects-the-ghost-of-dyre/"  target="_blank" rel="noopener nofollow" >TrickBot-Dyre connection</a> emerged as soon as the malware was discovered and has been the source of much debate ever since. Even though this subject was mentioned in several other <a href="http://www.threatgeek.com/2016/10/trickbot-the-dyre-connection.html" target="_blank" rel="noopener nofollow"  target="_blank" rel="noopener nofollow" >TrickBot analysis blogs</a>, we would like to contribute several key points from our own research into this new threat:</p>
<ul>
<li>TrickBot&rsquo;s serverside webinjection method is uncommon in today&rsquo;s malware. The other malware family that used it, as you can guess, was Dyre.</li>
</ul>
<ul>
<li>Packets sent to the attack server during the serverside webinjects consist of three parts, titled &ldquo;sourcelink,&rdquo; &ldquo;sourcequery&rdquo; and &ldquo;sourcehtml.&rdquo; These exact names were also used in Dyre&rsquo;s webinjection mechanism.</li>
</ul>
<p style="text-align: center;"><amp-img src="https://static.securityintelligence.com/uploads/2016/11/trickbot_5.png" layout="intrinsic" class="" alt="TrickBot Dyre connection" width="" height="" lightbox="lightbox"></amp-img></p>
<p class="" style="text-align: center;" align="center"><em>Figure 5: TrickBot and Dyre both use &ldquo;sourcelink&rdquo; and &ldquo;sourcequery&rdquo; for their communications.</em></p>
<ul>
<li>Targeted URLs and command-and-control (C&amp;C) addresses are kept encrypted on the infected machine. <a href="https://securityintelligence.com/dyre-straights-group-behind-the-dyre-trojan-busted-in-moscow/"  target="_blank" rel="noopener nofollow" >Dyre did the same</a>. While the encryption schemes for TrickBot are not identical, they appear to be too similar to be a mere coincidence.</li>
</ul>
<ul>
<li>TrickBot passes the target URLs list to its financial module, which is injected into the browser using pipes communication. This, again, is a Dyre hallmark.</li>
</ul>
<ul>
<li>The structure of the targeted URLs in the configuration is typically consistent for each malware, and TrickBot&rsquo;s target specification &mdash; you guessed it &mdash; sure looks a lot like Dyre&rsquo;s.</li>
</ul>
<div class="related-links"><b>Related: </b><a href="https://securityintelligence.com/an-aggressive-launch-trickbot-trojan-rises-with-redirection-attacks-in-the-uk/" title="An Aggressive Launch: TrickBot Trojan Rises With Redirection Attacks in the UK" >An Aggressive Launch: TrickBot Trojan Rises With Redirection Attacks in the UK</a></div>
<p>Although the similarities are there, keep in mind that most of them are relatively simply to imitate. For example, even though the serverside webinjection technique is common to both Trojans, the code implementing this capability and the coding style itself are actually different.</p>
<p>This point is important because we can see how rapidly and effectively new and advanced malware is being developed, either by the same actors or by newcomers inspired by one of the most nefarious gangs in cybercrime history.</p>
<h2>A Last-Minute Update: Redirection Attacks!</h2>
<p>TrickBot&rsquo;s developers must be hard at work these days, pushing to enhance the malware for a live campaign targeting banks. Just as we were about to publish this post, we detected a new infection campaign with a new configuration targeting U.K. banks. Until now, TrickBot only targeted banks in Australia. Furthermore, some of these new U.K. targets are set up for redirection attacks, while TrickBot only employed the serverside webinjection attack described above until now.</p>
<p>A redirection attack, in short, means that instead of injecting malicious code into the original webpage, the victim is now redirected to a new site forged by the fraudsters. This site looks and feels exactly like the original website, and the browser indicates a Secure Sockets Layer (SSL) connection based on the original site&rsquo;s certificate.</p>
<p>To learn more about redirection attacks and their purpose, read our blogs about <a href="https://securityintelligence.com/dridex-launches-dyre-like-attacks-in-uk-intensifies-focus-on-business-accounts/" >Dridex</a> and <a href="https://securityintelligence.com/time-is-money-goznym-launches-redirection-attacks-in-poland/" >GozNym</a>.</p>
<h2>A Newcomer to the Malware Arena</h2>
<p>TrickBot is undoubtedly the work of professionals who have been around the banking Trojan scene for some time. These experienced fraudsters are apparently well-versed in the modern features common to the types of malware banks reckon with nowadays. We expect to see this Trojan evolve its anti-security and anti-research techniques and pop up in more infection campaigns as the year comes to a close.</p>
