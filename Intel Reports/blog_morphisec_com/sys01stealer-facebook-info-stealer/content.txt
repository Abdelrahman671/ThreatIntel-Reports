<div class="section post-body">
<span id="hs_cos_wrapper_post_body" class="hs_cos_wrapper hs_cos_wrapper_meta_field hs_cos_wrapper_type_rich_text" style="" data-hs-cos-general-type="meta_field" data-hs-cos-type="rich_text"><p><span data-contrast="auto">Starting in November 2022, Morphisec has been tracking an advanced info stealer we have named “SYS01 stealer.” SYS01 stealer uses similar lures and loading techniques to another information stealer recently dubbed S1deload by the Bitdefender group, but the actual payload (stealer) is different. </span><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span></p>
<!--more-->
<p><span data-contrast="auto">We have seen SYS01 stealer attacking critical government infrastructure employees, manufacturing companies, and other industries. The attack is designed to steal sensitive information, including login data, cookies, and Facebook ad and business account information.</span><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span></p>

<p><span data-contrast="auto">The campaign was first seen in May 2022 and was initially <a href="https://www.zscaler.com/blogs/security-research/new-php-variant-ducktail-infostealer-targeting-facebook-business-accounts" rel="noopener" target="_blank">attributed to the Ducktail operation</a> by Zscaler. <h2>The Infection Chain&nbsp;</h2>
<p><span data-contrast="auto" xml:lang="EN-US" lang="EN-US"><span>The attack </span><span>begins</span><span> </span><span>by luring </span><span>a </span><span>victim </span><span>to </span><span>click on a URL from a fake Facebook profile or advertisement to </span><span>download a </span><span>ZIP </span><span>file th</span><span>at </span><span>pretends to </span><span>have</span><span> </span><span>an application, game, movie, etc</span><span>.</span><span>&nbsp;</span></span><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span></p>
<p style="text-align: left;">The infection chain is divided into two parts: the loader, and the <em>Inno-Setup</em> installer that drops the final payload. The loader is usually a legitimate C# application susceptible to a side-loading vulnerability that comes with a hidden malicious dynamic link library (DLL) file that’s eventually side-loaded to the application. This legitimate application drops the <em>Inno-Setup</em> installer that decompresses to a whole PHP application containing malicious scripts. The PHP scripts are responsible for stealing and exfiltrating information. The scripts are encoded using different techniques, which makes their analysis and detection harder.&nbsp;</p>

<h2>The Loader&nbsp;</h2>
<p>We’ve seen the payload delivered in diverse ways including DLL side-loading, Rust and Python executables, and many others. All methods eventually drop an <em>Inno-Setup</em> installer which, at the next stage, drops and executes the PHP information stealer. In the next sections we elaborate on various delivery techniques and luring themes the attackers use.</p>
<p>Note: these delivery methods are representative samples. There are many other variations of these methods with minor modifications.&nbsp;&nbsp;</p>
<h3>DLL Side-Loading V1&nbsp;</h3>
<p>In this method, the victim downloads a zipped folder with different luring themes such as world cup live streaming, free applications, and more that abuse legitimate applications vulnerable to DLL side-loading attack. The zipped folder usually holds the following file patterns:&nbsp;</p>
<ol>
<li><strong>Executable.</strong> A benign, legitimate executable abused to side-load the malicious DLL</li>
<li><strong>[A-Z]Data.dat (hidden).</strong> A ZIP or self extracting archive (SFX) containing legitimate HTML webpages used as a decoy&nbsp;</li>
<li><strong>[A-Z]License (hidden).</strong> The Inno-Setup installer to be executed (base64 encoded with some string modifications)</li>
<li><strong>DLL (hidden).</strong> The malicious side loaded DLL</li>
</ol>

<p>The malicious DLL has two main goals: displaying the decoy to the victim and executing the <em>Inno-Setup</em> installer. It does this by creating a thread that checks whether the <em>License</em> file exists. If it doesn’t, it downloads the file from its command and control (C2) server, then decodes and executes it. In the main thread the SFX/ZIP file is executed/decompressed, and the victim is shown the decoy HTML files.&nbsp;</p>

<h2>Inno-Setup to PHP Information Stealer&nbsp;</h2>
<p>Once the <em>Inno-Setup</em> installer executes, it drops a <span style="font-weight: bold;">PHP</span> application with additional files, usually to <em>%localappdata\[A-Z]{4}\&lt;version_number&gt;</em>. Between different variants of this information stealer, we saw the following files used to execute the malicious logic:&nbsp;</p>
<ul>
<li><strong>include.php</strong> Responsible for installing persistence via scheduled tasks</li>
<li><strong>index.php</strong> Executes the main logic of the stealing act</li>
<li><strong>version.php (embedded in index.php if it does not exist)</strong> Holds the stealer version</li>
<li><strong>rhc.exe</strong> Hides the console window of started programs (hidec)</li>
<li><strong>rss.txt (other variants have a different name)</strong> Base64 encoded string with several string replacements. Once decoded, this executable, written in <em>Rust</em> and compiled with <em>Cargo</em>, gets the current date and time, and decrypts Chromium-based browsers’ encryption key&nbsp;</li>
</ul>
<p>In older variants the <span style="font-weight: bold;">PHP</span> scripts were not obfuscated in any manner. In newer variants we noticed commercial encoders <a href="https://www.ioncube.com/" rel="noopener" target="_blank">ionCube</a> and Zephir, which are self-written extensions that obfuscate the <span style="font-weight: bold;">PHP</span> scripts.</p>
<p>After dropping the folder, the <em>Inno-Setup </em>executable executes <span style="font-weight: bold;">php.exe include.php</span>, or passes this command line as an argument to <span style="font-weight: bold;">rhc.exe</span>.&nbsp;<br><br><span style="font-weight: bold;">include.php</span> registers two scheduled tasks: &nbsp;</p>
<ol>
<li><span style="font-family: terminal, monospace;">rhc.exe php.exe include.php&nbsp;</span><br><span style="font-family: terminal, monospace;">rhc.exe php.exe index.php </span></li>
<li><span style="font-family: terminal, monospace;">rhc.exe php.exe index.php&nbsp;</span></li>
</ol>
<p>The first task is triggered at log-on. The second task is triggered every two minutes. The attacker must know the time to set it in the scheduled task. It does this by decoding the rss.txt into an executable, adding a uniquid at the end of the executable, and dropping it to <em>%temp%\tmp\&lt;uniqid&gt;.exe</em>. If for some reason the operation fails, this script gets the current date and time running: “wmic os get LocalDateTime /value”.&nbsp;</p>

<h2>How do You Combat SYS01 stealer?&nbsp;</h2>
<p><span data-contrast="auto">DLL side-loading is a highly effective technique for tricking Windows systems into loading malicious code. Dynamic link libraries enable more efficient memory use and system modularity, but&nbsp;because Microsoft doesn’t enforce search order for DLLs by default, it&nbsp;makes applications vulnerable to exploitation.&nbsp;</span><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span></p>
<p><span data-contrast="auto">Microsoft doesn’t enforce search order for a range of reasons, such as enabling things like portability and backwards compatibility—for example, portable browser applications that use older Microsoft libraries. Security-minded developers may enforce search order within their code. But most developers aren't security minded. </span></p>
<p><span data-contrast="auto">This enables threat actors to position a malicious payload alongside a legitimate application. Then when an application loads in memory and search order is not enforced, the application loads the malicious file instead of the legitimate one, allowing threat actors to hijack legitimate, trusted, and even signed applications to load and execute malicious payloads. Adversaries use side-loading attacks for execution, persistence, privilege escalation, and defense evasion. </span><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span></p>
<p><span data-contrast="auto">Microsoft has started to enforce default search order on many of their applications, even if not universally. But attackers can still misuse highly popular Microsoft applications that were created in previous years that are legitimately signed. And they will be able to do so for many years to come.</span><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span></p>
<p><span data-contrast="auto">Basic steps to help prevent SYS01 stealer include implementing a zero-trust policy and limiting users’ rights to download and install programs. And SYS01 stealer at heart relies on a social engineering campaign, so it’s important to train users about the tricks adversaries use so they know how to spot them.&nbsp;</span><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span></p>
<p><a href="https://blog.morphisec.com/infostealer-comparison"><i><span data-contrast="none">Read Your Guide to Top Info Stealers of 2022</span></i></a><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span></p>
<p><span data-contrast="auto">But humans are fallible, and limiting device functionality is not always possible when you need to ensure effective business functions. This is why the best protection is all-of-the-above plus a Defense-in-Depth approach. Security tools like next generation anti-virus (NGAV), endpoint protection platforms (EPP), and endpoint detection and response (EDR, XDR, and MDR) are necessary, but not sufficient to stop stealers like SYS01 stealer.&nbsp;</span><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span></p>
<p><span data-contrast="auto">This is because detection-based tools don’t always flag benign executables used to side-load payloads during delivery and/or execution. And malicious payloads are also sometimes encrypted/packed or obfuscated until loaded into runtime memory, which detection-based tools </span><a href="https://blog.morphisec.com/runtime-attacks-in-memory"><span data-contrast="none">struggle to effectively scan</span></a><span data-contrast="auto">. The most effective way to secure runtime memory is with Moving Target Defense (MTD) technology. MTD morphs—randomizes—the runtime memory environment to create a dynamic attack surface and leaves decoy traps where targets used to be. Any code that attempts to execute on a decoy is immediately terminated and trapped for forensic analysis. The combination of detection-based tools with Moving Target Defense creates the most effective Defense-in-Depth against threats like SYS01 stealer. To learn more, read the white paper: </span><a href="https://engage.morphisec.com/the-ultimate-ransomware-strategy"><i><span data-contrast="none">Zero Trust + Moving Target Defense: The Ultimate Ransomware Strategy</span></i></a><span data-contrast="auto">.&nbsp;</span><span data-ccp-props="{&quot;201341983&quot;:0,&quot;335559739&quot;:160,&quot;335559740&quot;:259}">&nbsp;</span>&nbsp;&nbsp;</p>
