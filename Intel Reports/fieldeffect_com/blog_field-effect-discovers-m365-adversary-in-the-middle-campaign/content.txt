<p style="font-weight: bold;">Original Post:</p>
<p>During a cursory analysis of telemetry related to Microsoft 365 (M365) authentication attempts from unexpected ISPs, Field Effect noticed that the user agent string â€˜axios/1.7.2â€™ was associated with suspicious attempts to log in to various user accounts.</p>
<!--more-->
<p>Although Field Effect MDR customers are automatically alerted to this type of suspicious activity, we eager analysts couldnâ€™t help but dig deeper into this unfamiliar user agent string with the goal of discovering a new trend being used to facilitate business email compromise (BEC) attacks.</p>
<h2>â€˜Axiosâ€™ user agent string</h2>
<p>According to its <a href="https://axios-http.com/" rel="noopener" target="_blank">website</a>, Axios is a promise-based HTTP client for browsers and node.js. It supports multiple features, including XMLHttpRequests from the browser and HTTP requests from node.js. Axios also supports the ability to intercept, transform, and cancel request and response data, which we will see the threat actor leverage later in this investigation.</p>
<p>While the app itself is legitimate, and easily downloaded from its GitHub repository, there is little reason for it to interact with M365 in the manner described above.</p>
<p><img src="https://fieldeffect.com/hs-fs/hubfs/axios%20website.png?width=1912&amp;height=1106&amp;name=axios%20website.png" width="1912" height="1106" loading="lazy" alt="axios website" style="height: auto; max-width: 100%; width: 1912px; border: 1px solid #707070; padding: 20px 0px; margin-left: auto; margin-right: auto; display: block;" srcset="https://fieldeffect.com/hs-fs/hubfs/axios%20website.png?width=956&amp;height=553&amp;name=axios%20website.png 956w, https://fieldeffect.com/hs-fs/hubfs/axios%20website.png?width=1912&amp;height=1106&amp;name=axios%20website.png 1912w, https://fieldeffect.com/hs-fs/hubfs/axios%20website.png?width=2868&amp;height=1659&amp;name=axios%20website.png 2868w, https://fieldeffect.com/hs-fs/hubfs/axios%20website.png?width=3824&amp;height=2212&amp;name=axios%20website.png 3824w, https://fieldeffect.com/hs-fs/hubfs/axios%20website.png?width=4780&amp;height=2765&amp;name=axios%20website.png 4780w, https://fieldeffect.com/hs-fs/hubfs/axios%20website.png?width=5736&amp;height=3318&amp;name=axios%20website.png 5736w" sizes="(max-width: 1912px) 100vw, 1912px"></p>
<p style="text-align: center; font-size: 16px;"><em>Image 4: Screenshot of Axios website (axios-http.com)</em></p>
<p>Upon further review, we observed that many of the login requests with an Axios user agent string originated from two ISPs: Hostinger International Limited and Global Internet Solutions LLC. These two ISPs have previously been flagged by Field Effect as suspicious due to their frequent abuse by threat actors, usually located in Russia.</p>
<p>Searching for all M365 login attempts from these two ISPs revealed the use of user agent strings associated with older versions of Axios, such as 1.6.8, and another user agent string used sparingly, â€˜BAV2ROPCâ€™, which is associated with a legacy Microsoft authentication protocol mostly used by threat actors to enable password spraying attacks.</p>
<p><img src="https://fieldeffect.com/hs-fs/hubfs/axios%202.png?width=3253&amp;height=1735&amp;name=axios%202.png" width="3253" height="1735" loading="lazy" alt="axios 2" style="height: auto; max-width: 100%; width: 3253px; margin-left: auto; margin-right: auto; display: block;" srcset="https://fieldeffect.com/hs-fs/hubfs/axios%202.png?width=1627&amp;height=868&amp;name=axios%202.png 1627w, https://fieldeffect.com/hs-fs/hubfs/axios%202.png?width=3253&amp;height=1735&amp;name=axios%202.png 3253w, https://fieldeffect.com/hs-fs/hubfs/axios%202.png?width=4880&amp;height=2603&amp;name=axios%202.png 4880w, https://fieldeffect.com/hs-fs/hubfs/axios%202.png?width=6506&amp;height=3470&amp;name=axios%202.png 6506w, https://fieldeffect.com/hs-fs/hubfs/axios%202.png?width=8133&amp;height=4338&amp;name=axios%202.png 8133w, https://fieldeffect.com/hs-fs/hubfs/axios%202.png?width=9759&amp;height=5205&amp;name=axios%202.png 9759w" sizes="(max-width: 3253px) 100vw, 3253px"></p>
<p style="text-align: center; font-size: 16px;"><em>Image 5: Example of M365 telemetry showing the â€˜axiosâ€™, â€˜BAV2ROPCâ€™, and a seemingly normal user agent string.</em></p>
<p>We also observed requests that appeared to have legitimate user agent strings. However, further analysis revealed that this was the same user agent string used by the legitimate owner of the targeted account, which is now being used by the threat actor to make the authentication request appear more legitimate. More on this later.</p>
<p>At this point, we were confident we had discovered the previously unreported use of the Axios user agent string to facilitate business email compromise (BEC) attacks on our clients, similar to what has been reported by other researchers on the â€˜AZURECLI/2.47.0â€™ and â€˜BAV2ROPCâ€™ user agent strings.</p>
<p>However, we knew there must be more to this story since it appeared that the threat actor was able to successfully log in to some accounts, despite MFA being enabled. This would have required some interaction with the user to obtain both the password and MFA key, tasks an Axios HTTP client is capable of.</p>
<h2>Deciphering the attack chain</h2>
<p>We zeroed in on a particular login attempt to decipher the wider attack chain. What we discovered is that the targeted user was making requests to some shady-looking domains just as the M365 login attempts occurred.</p>
<p><img src="https://fieldeffect.com/hs-fs/hubfs/domain%20pcap.png?width=1966&amp;height=549&amp;name=domain%20pcap.png" width="1966" height="549" loading="lazy" alt="domain pcap" style="height: auto; max-width: 100%; width: 1966px; margin-left: auto; margin-right: auto; display: block;" srcset="https://fieldeffect.com/hs-fs/hubfs/domain%20pcap.png?width=983&amp;height=275&amp;name=domain%20pcap.png 983w, https://fieldeffect.com/hs-fs/hubfs/domain%20pcap.png?width=1966&amp;height=549&amp;name=domain%20pcap.png 1966w, https://fieldeffect.com/hs-fs/hubfs/domain%20pcap.png?width=2949&amp;height=824&amp;name=domain%20pcap.png 2949w, https://fieldeffect.com/hs-fs/hubfs/domain%20pcap.png?width=3932&amp;height=1098&amp;name=domain%20pcap.png 3932w, https://fieldeffect.com/hs-fs/hubfs/domain%20pcap.png?width=4915&amp;height=1373&amp;name=domain%20pcap.png 4915w, https://fieldeffect.com/hs-fs/hubfs/domain%20pcap.png?width=5898&amp;height=1647&amp;name=domain%20pcap.png 5898w" sizes="(max-width: 1966px) 100vw, 1966px"></p>
<p style="text-align: center; font-size: 16px;"><em>Image 6: PCAP logs of network activity at the time of the M365 login request</em></p>
<p>Additionally, the IP (172.64.80[.]1) to which these domains resolved was observed connecting to the targetâ€™s M365 login page.</p>
<p>Unfortunately, given the U.S. July 4 holiday, we have not yet been able to obtain the full URL that the targeted user would have followed. And, out of the three domains, only lsj.logentr[.]com currently returns any meaningful contentâ€”a default error string used by node.js, an environment in which Axios is capable of running. Curiously, this same default error string also appeared on the IP addresses logging in to the M365 accounts.</p>
<p><a href="#_msoanchor_1"></a><img src="https://fieldeffect.com/hs-fs/hubfs/logen%20error%20code.png?width=1884&amp;height=328&amp;name=logen%20error%20code.png" width="1884" height="328" loading="lazy" alt="logen error code" style="height: auto; max-width: 100%; width: 1884px; margin-left: auto; margin-right: auto; display: block;" srcset="https://fieldeffect.com/hs-fs/hubfs/logen%20error%20code.png?width=942&amp;height=164&amp;name=logen%20error%20code.png 942w, https://fieldeffect.com/hs-fs/hubfs/logen%20error%20code.png?width=1884&amp;height=328&amp;name=logen%20error%20code.png 1884w, https://fieldeffect.com/hs-fs/hubfs/logen%20error%20code.png?width=2826&amp;height=492&amp;name=logen%20error%20code.png 2826w, https://fieldeffect.com/hs-fs/hubfs/logen%20error%20code.png?width=3768&amp;height=656&amp;name=logen%20error%20code.png 3768w, https://fieldeffect.com/hs-fs/hubfs/logen%20error%20code.png?width=4710&amp;height=820&amp;name=logen%20error%20code.png 4710w, https://fieldeffect.com/hs-fs/hubfs/logen%20error%20code.png?width=5652&amp;height=984&amp;name=logen%20error%20code.png 5652w" sizes="(max-width: 1884px) 100vw, 1884px"></p>
<p style="text-align: center; font-size: 16px;"><em>Image 7: Default node.js error string returned from lsj.logentr[.]com</em></p>
<h2>Adversary-in-the-middle (AiTM) BEC</h2>
<p>Based on experience and analyst intuition, we assess that we are looking at an adversary-in-the-middle (AiTM) BEC attack and that these URLs likely led to what appeared to be an M365 login page, but in fact was an Axios-based lookalike designed to harvest the targetâ€™s M365 credentials.</p>
<p>Once the target inputs their credentials and, if necessary, their MFA code, the Axios infrastructure captures and uses them, and possibly the session token, to log the victim into their M365 account, thus completing the AiTM BEC.</p>
<p>The user has no suspicions logging in to their account, meanwhile, the threat actorâ€™s Axios-based infrastructure has logged their credentials for future exploitation.</p>
<p>So given this data, we can logically conclude that the threat actor isnâ€™t just using Axios user agent strings to log in to random M365 accounts, but rather using it to proxy login requests from the legitimate account owner, likely a result of them interacting with a phishing email.</p>
<p>While the phishing domains could also have been served up via an infected website, we believe that it was more likely an email-borne attack since we observed that the user opened Outlook.exe right before the attack, which in turn opened the Edge browser, kicking off the requests to the shady phishing domains.</p>
<p><img src="https://fieldeffect.com/hs-fs/hubfs/Slide1.png?width=720&amp;height=405&amp;name=Slide1.png" width="720" height="405" loading="lazy" alt="Slide1" style="height: auto; max-width: 100%; width: 720px; border: 1px solid #707070; margin-left: auto; margin-right: auto; display: block;" srcset="https://fieldeffect.com/hs-fs/hubfs/Slide1.png?width=360&amp;height=203&amp;name=Slide1.png 360w, https://fieldeffect.com/hs-fs/hubfs/Slide1.png?width=720&amp;height=405&amp;name=Slide1.png 720w, https://fieldeffect.com/hs-fs/hubfs/Slide1.png?width=1080&amp;height=608&amp;name=Slide1.png 1080w, https://fieldeffect.com/hs-fs/hubfs/Slide1.png?width=1440&amp;height=810&amp;name=Slide1.png 1440w, https://fieldeffect.com/hs-fs/hubfs/Slide1.png?width=1800&amp;height=1013&amp;name=Slide1.png 1800w, https://fieldeffect.com/hs-fs/hubfs/Slide1.png?width=2160&amp;height=1215&amp;name=Slide1.png 2160w" sizes="(max-width: 720px) 100vw, 720px"></p>
<p style="text-align: center; font-size: 16px;"><em>Image 8: Flow chart of Axios user agent string attack chain</em></p>
<p>Itâ€™s unclear why the threat actor did not consistently swap the Axios user agent string for the userâ€™s when forwarding the login, since we did observe some instances where this was the case and Axios does have the ability to change its user agent string.</p>
<p>Using the targetâ€™s user agent string would make the request look much more legitimate, but we observed this sparingly. The threat actor may have only changed it when previous attempts using the Axios user agent string failed.</p>
<p>Itâ€™s also possible that the threat actor was not always able to capture the targetâ€™s user agent string via Axios, or it simply could have been a configuration issue.</p>
<h2>Conclusion</h2>
<p>Even though it is trivial for threat actors to mask or change the user agent string associated with their attacks, this case and many others show that some threat actors simply choose not to change it or forget to, making it easier for network defenders to detect and prevent certain BEC attacks.</p>
<p>Field Effect maintains an evergreen list of suspicious and malicious user agent strings that are constantly correlated against our telemetry, resulting in an alert or block when one is detected.</p>
<p>However, detecting and blocking connection requests based on the user agent string alone should not be considered an adequate security control in and of itself. Rather, it should be part of a larger detection regime that includes connections from suspicious IP addresses, ISPs, geographical areas, impossible travel, etc., all of which are employed by Field Effect MDR.</p>
<p>This case also reflects the importance of having a holistic view of an organizationâ€™s IT infrastructure â€“ network, endpoint, and cloud â€“ to enable the type of threat hunting used in this investigation. Without the ability to observe activity on the impacted hosts, it would have been much more difficult to independently identify the other infrastructure associated with the campaign.</p>
