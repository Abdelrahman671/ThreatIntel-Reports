<div class="text text__pt-none text__pb-none aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-61c40da567&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2024-10-09T19:50:19Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>To create effective API security policies, organizations must first uncover the full scope of their API estate and create a comprehensive inventory.&amp;nbsp;&lt;/p>\r\n&lt;p>One thing that sets API security apart from other security domains is that most security teams don’t have a complete understanding of the size and scope of the problem.     <p>To create effective API security policies, organizations must first uncover the full scope of their API estate and create a comprehensive inventory. </p>
<p>One thing that sets API security apart from other security domains is that most security teams don’t have a complete understanding of the size and scope of the problem. APIs often originate from many different development teams within a typical organization. </p>
<p>Although some organizations have attempted to consolidate these functions into centralized tools like API gateways, security teams are still consistently shocked by the amount of previously unknown APIs we discover for them.</p>
<p>This typically includes a blend of:</p>
<ul>
<li><p><b>Shadow APIs</b> that may serve a legitimate purpose but were deployed outside of formal processes, often without appropriate security reviews and documentation</p>
</li>
<li><p><b>Zombie APIs</b> that are no longer being used but remain active as potential attack vectors</p>
</li>
<li><p><b>Rogue APIs</b> created by third-party developers who access the data through a backdoor without the knowledge or approval of the company that owns the data</p>
</li>
</ul>
<p>In our experience, it's common that these unseen APIs represent approximately 40% of a typical organization’s overall API estate. This is an eye-opener for many security teams, and drives home the point that a sound API security strategy must include a method of discovering all API in use and a way of maintaining an up-to-date API inventory at all times. Only then will the security team be able to implement effective protections for all sanctioned APIs and to decommission APIs that should not be providing access to company resources.</p>

      <h2 class="cmp-title__text">The limitations of WAAP platforms as the sole API cybersecurity strategy</h2>
    
</div>

<div class="text text__pt-none text__pb-none aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-e0b7879902&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2024-10-09T19:51:50Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>WAAP platforms are security solutions that focus on distributed denial-of-service (DDoS) protection, bot management, web application firewall (WAF), and base API protection. Through tactics like DDoS mitigation and bot protection, they help organizations by:&lt;/p>\r\n&lt;ul>\r\n&lt;li>&lt;p>Safeguarding web apps and APIs from a range of cyberattacks, including DDoS attacks, cross-site scripting (XSS) attacks, and SQL injections&lt;/p>\r\n&lt;/li>\r\n&lt;li>&lt;p>Addressing the &lt;a href=\&#34;https://owasp.org/www-project-top-ten/\&#34; target=\&#34;_blank\&#34; rel=\&#34;nofollow noreferrer\&#34; referrerpolicy=\&#34;no-referrer\&#34;>OWASP Top 10&lt;/a> web application security vulnerabilities&lt;/p>\r\n&lt;/li>\r\n&lt;/ul>\r\n&lt;p>But, on their own, these platforms aren’t foolproof when it comes to API traffic authentication or modern API threats — meaning sensitive data remains vulnerable to other types of cyberattacks, including API data scraping.&lt;/p>\r\n&lt;p>Although data captured by WAAPs is a useful source of API activity information, it’s far from comprehensive. Rogue, shadow, and zombie APIs can easily bypass the WAAP or other tools like the API gateway. In addition, sanctioned APIs are often designed for internal use only and may not be implemented with protection from external cyberthreats in mind. Unfortunately, an internal API being accidentally exposed to the internet is an all too common occurrence.&lt;/p>\r\n&lt;p>Akamai API Security addresses this challenge by ingesting data from a wide range of possible sources, including WAAPs, content delivery network (CDN) infrastructure, API gateways, reverse proxies, cloud platforms, container/mesh orchestration tools, centralized log management systems, and more. This ensures that sanctioned and unsanctioned APIs are discovered and inventoried continuously in real time.&lt;/p>\r\n&#34;}}">
    <p>WAAP platforms are security solutions that focus on distributed denial-of-service (DDoS) protection, bot management, web application firewall (WAF), and base API protection. Through tactics like DDoS mitigation and bot protection, they help organizations by:</p>
<ul>
<li><p>Safeguarding web apps and APIs from a range of cyberattacks, including DDoS attacks, cross-site scripting (XSS) attacks, and SQL injections</p>
</li>
<li><p>Addressing the <a href="https://owasp.org/www-project-top-ten/" target="_blank" rel="nofollow noreferrer" referrerpolicy="no-referrer">OWASP Top 10</a> web application security vulnerabilities</p>
</li>
</ul>
<p>But, on their own, these platforms aren’t foolproof when it comes to API traffic authentication or modern API threats — meaning sensitive data remains vulnerable to other types of cyberattacks, including API data scraping.</p>
<p>Although data captured by WAAPs is a useful source of API activity information, it’s far from comprehensive. Rogue, shadow, and zombie APIs can easily bypass the WAAP or other tools like the API gateway. In addition, sanctioned APIs are often designed for internal use only and may not be implemented with protection from external cyberthreats in mind. Unfortunately, an internal API being accidentally exposed to the internet is an all too common occurrence.</p>
<p>Akamai API Security addresses this challenge by ingesting data from a wide range of possible sources, including WAAPs, content delivery network (CDN) infrastructure, API gateways, reverse proxies, cloud platforms, container/mesh orchestration tools, centralized log management systems, and more. This ensures that sanctioned and unsanctioned APIs are discovered and inventoried continuously in real time.</p>

      <h2 class="cmp-title__text">Unique characteristics of API threats</h2>

<div class="text text__pt-none text__pb-none aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-17d927c469&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2024-10-09T19:53:26Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>Another key reason that API security must extend beyond the WAAP is that API abuse is much harder to distinguish from legitimate use than other types of API threats. In many scenarios, a malicious API request and a legitimate API request, viewed individually by a WAAP, look nearly identical. In addition, the business logic and data accessible through APIs vary greatly from one application to the next, leaving WAAPs without the necessary context to detect many types of API security threats.&lt;/p>\r\n&lt;p>So, while WAAPs can detect and stop API threats with telltale characteristics, many more sophisticated API threats will likely blend in with the large volume of legitimate use and proceed undetected. This opens the door for potentially devastating attacks like &lt;a href=\&#34;/content/akamai/en/glossary/what-is-bola.html\&#34; target=\&#34;_self\&#34;>Broken Object Level Authorization (BOLA)&lt;/a>, unauthenticated resource access attempts, abnormal JSON property, path parameter fuzzing attempts, impossible time travel, and API data scraping.&lt;/p>\r\n&lt;p>Akamai API Security fills this protection gap in two critical ways: contextualizing API activity and using behavioral analytics to detect API abuse.&lt;/p>\r\n&#34;}}">
    <p>Another key reason that API security must extend beyond the WAAP is that API abuse is much harder to distinguish from legitimate use than other types of API threats. In many scenarios, a malicious API request and a legitimate API request, viewed individually by a WAAP, look nearly identical. In addition, the business logic and data accessible through APIs vary greatly from one application to the next, leaving WAAPs without the necessary context to detect many types of API security threats.</p>
<p>So, while WAAPs can detect and stop API threats with telltale characteristics, many more sophisticated API threats will likely blend in with the large volume of legitimate use and proceed undetected. This opens the door for potentially devastating attacks like <a href="/glossary/what-is-bola" target="_self">Broken Object Level Authorization (BOLA)</a>, unauthenticated resource access attempts, abnormal JSON property, path parameter fuzzing attempts, impossible time travel, and API data scraping.</p>
<p>Akamai API Security fills this protection gap in two critical ways: contextualizing API activity and using behavioral analytics to detect API abuse.</p>

<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-9d2eac5973&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2024-10-09T19:55:14Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>Although the collection of comprehensive data about API activity is critical, it’s only the first step.     <p>Although the collection of comprehensive data about API activity is critical, it’s only the first step. The next challenge is to reveal the stories that are hiding within this data. This starts with identifying the entities involved — including the parties or systems that initiate the action, as well as the business processes that are accessible through APIs. </p>
<p>Partners who are granted API access to certain systems are an example of a potential API actor entity, and API-accessible invoicing functionality is an example of a business process entity. Most organizations have countless examples of actors and business process entities. Identifying them and putting their activity in context makes it possible to distinguish between sanctioned and unsanctioned API use, even if the specific API requests look identical.</p>

      <h3 class="cmp-title__text">Using behavioral analytics to detect API abuse</h3>

<div class="text text__pt-none text__pb-none aem-GridColumn aem-GridColumn--default--12">
<div data-cmp-is="text" class="cmp-text" data-cmp-data-layer="{&#34;text-08c9af4f0a&#34;:{&#34;@type&#34;:&#34;akamai/components/text&#34;,&#34;repo:modifyDate&#34;:&#34;2024-10-09T19:56:18Z&#34;,&#34;xdm:text&#34;:&#34;&lt;p>Once you’ve contextualized data about your API activity, the next step is analyzing it for signs of API abuse.     <p>Once you’ve contextualized data about your API activity, the next step is analyzing it for signs of API abuse. Akamai API Security uses machine learning and behavioral analytics to:</p>
<ul>
<li><p>Create baselines of normal API use</p>
</li>
<li><p>Detect anomalies that represent likely API abuse</p>
</li>
</ul>
<p>Unlike a WAAP, which can only analyze very small samples of point-in-time API requests, Akamai API Security takes a much broader view. This makes it possible to spot API abuse and malicious traffic that would otherwise blend in with authorized activity.</p>
<p>For example, imagine that a hotel chain is granting travel booking sites API access to pricing and room availability information. Individual API requests for pricing from a booking site wouldn’t attract attention from a WAAP, since this is expected behavior. </p>
<p>But if one travel booking site’s overall volume of pricing requests is orders of magnitude higher than all similar travel booking sites — and their own historical volume levels — this may indicate that the site is abusing their API access to exfiltrate data at scale and gain a competitive advantage.</p>
<p>From a WAAP’s perspective, all of these API requests are identical. But Akamai API Security can tell the difference by spotting the anomaly. Although this is a very simple example, it illustrates the power of context and visibility that extends beyond individual API requests.</p>
