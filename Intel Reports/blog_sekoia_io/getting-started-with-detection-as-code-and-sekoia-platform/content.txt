<h2 class="wp-block-heading" id="h-introduction">Introduction</h2>



<p>Managed Security Service Providers (MSSPs) and internal SOC teams face major challenges in the current cybersecurity landscape. Managing different SOC deployments across various clients or business units can lead to inconsistencies, inefficiencies, and vulnerabilities. Traditional approaches often fall short, struggling with scalability, agility, and accuracy. This is where Detection-as-Code can play a crucial role, offering transformative solutions to elevate your security operations.</p>



<p>Whether you&#8217;re an MSSP looking to enhance client offerings or an internal SOC team striving for operational excellence, adopting Detection-as-Code can be a game-changer. Here’s why it matters.</p>



<span id="more-10302"></span>



<h2 class="wp-block-heading" id="h-what-is-detection-as-code">What is Detection-as-Code?</h2>



<p>Detection-as-Code is a modern approach to threat detection that allows security teams to define, manage, and deploy detection logic using code. This methodology enhances accuracy, fosters collaboration, and enables rapid scaling by adopting practices similar to DevOps.</p>



<h2 class="wp-block-heading" id="h-key-benefits-for-mssps-and-internal-soc-teams">Key benefits for MSSPs and internal SOC teams</h2>



<ol class="wp-block-list">
<li><strong>Scalability</strong>: Automate and standardize detection rules to easily scale across different environments and clients.</li>



<li><strong>Consistency</strong>: Ensure uniform application of detection logic, reducing false positives and missed detections.</li>



<li><strong>Flexibility</strong>: Quickly adapt to emerging threats by updating detection logic on-the-fly.</li>



<li><strong>Efficiency</strong>: Streamline collaboration and knowledge sharing within your security team using version control and code repositories.</li>



<li><strong>Quality assurance</strong>: Leverage code reviews, testing, and continuous integration to maintain high standards for detection rules.</li>
</ol>



<h2 class="wp-block-heading" id="h-key-sekoia-capabilities-for-enabling-detection-as-code">Key Sekoia capabilities for enabling Detection-as-Code</h2>



<p>Sekoia SOC Platform has been designed to handle Detection-as-Code natively, leveraging:</p>



<ol class="wp-block-list">
<li><strong>Detection logic with Sigma:</strong> Detection rules are defined using Sigma, a generic and open standard for writing SIEM detections. These can be easily managed, versioned, and shared in git repositories, enabling:</li>
</ol>



<ul class="wp-block-list">
<li><strong>Collaboration: </strong>Allow security teams to collaboratively develop and refine detection logic using familiar tools.</li>



<li><strong>Version control:</strong> Track changes, review updates, and maintain a history of detection rules with git repositories.</li>



<li><strong>Portability:</strong> Facilitate the seamless transfer and sharing of detection logic across different systems and environments.</li>
</ul>



<ol start="2" class="wp-block-list">
<li><strong>API-based configuration:</strong> All aspects of the Sekoia platform can be configured through APIs, ensuring seamless integration and automation with third party code repositories and CI/CD tools. This API-first approach allows for the comprehensive management of detection logic, workflows, and policies without manual intervention. More details on Sekoia APIs on the public documentation here: <a href="https://docs.sekoia.io/xdr/develop/rest_api/quickstart/" target="_blank" rel="noreferrer noopener">https://docs.sekoia.io/xdr/develop/rest_api/quickstart/</a></li>
</ol>



<ol start="3" class="wp-block-list">
<li><strong>Fine-tuned detections: </strong>All built-in and custom detections can be enabled and fine-tuned through the API across several levels, including:</li>
</ol>



<ul class="wp-block-list">
<li><strong>Communities (Tenants):</strong> Manage detection rules at the tenant level, ensuring consistent security policies across all clients or business units.</li>



<li><strong>Entities:</strong> Tailor detection rules to specific entities within your organization for more precise threat detection.</li>



<li><strong>Assets:</strong> Customize detection logic based on the criticality and context of individual assets, enhancing protection where it matters most.</li>



<li><strong>Alert filters: </strong>Ignore specific cases by creating Sigma patterns to be excluded from the detection.</li>
</ul>



<h3 class="wp-block-heading" id="h-first-steps-create-and-deploy-a-sigma-detection-rule-in-sekoia">First steps: Create and deploy a Sigma detection rule in Sekoia</h3>



<p>To illustrate how Detection-as-Code can be implemented with the Sekoia platform, here’s a quick example of creating a Sigma detection rule and deploying it via the Sekoia API.</p>



<h4 class="wp-block-heading" id="h-create-a-sigma-detection">Create a Sigma Detection</h4>



<p>Create a file named <code>my_detection.yml</code> containing a Sigma pattern, such as the following simple content:</p>



<pre class="wp-block-code"><code>name: failed_logon
detection:
    selection:
        event.category: authentication
        action.outcome: failure
    condition: selection
---
correlation: 
    type: event_count
    rules:
        - failed_logon
    group-by:
        - action.properties.TargetUserName
        - action.properties.TargetDomainName
    timespan: 5m
    condition:
        gte: 10</code></pre>



<p>This example was taken from the <a href="https://blog.sigmahq.io/introducing-sigma-specification-v2-0-25f81a926ff0" target="_blank" rel="noreferrer noopener">Sigma v2.0</a> and <a href="https://blog.sigmahq.io/introducing-sigma-correlations-52fe377f2527" target="_blank" rel="noreferrer noopener">Sigma Correlation</a> Sigma_HQ recent blog posts, with mapping the field names to the <a href="https://docs.sekoia.io/integration/develop_integration/formats/taxonomy/" target="_blank" rel="noreferrer noopener">ECS format being used in Sekoia</a>.</p>



<h4 class="wp-block-heading" id="h-post-request-to-sekoia-api">POST Request to Sekoia API</h4>



<p>Use the following <a href="https://docs.sekoia.io/xdr/develop/rest_api/configuration/#tag/rules-catalog/operation/post_rules_resource" target="_blank" rel="noreferrer noopener">Create Rule API endpoint </a>to submit your detection rule via a POST request with a JSON body, containing the detection parameters. Here’s an example using <code>curl</code> and <code>jq</code>:</p>



<pre class="wp-block-code"><code>curl -X POST "https://api.sekoia.io/v1/sic/conf/rules-catalog/rules" \
-H "accept: application/json" \
-H "Authorization: Bearer YOUR_API_KEY" \
-H "Content-Type: application/json" \
-d "$(jq -n --arg payload "$(cat my_detection.yml)" '{
    name: "Multiple failed logons for a single user (possible brute force attack)",
    type: "sigma",
    description: "Detect possible brute force attacks on Windows endpoints and servers",
    payload: $payload,
    severity: 40,
    effort: 3,
    alert_type_uuid: "26b9f04e-65c7-4fcd-b0a4-ab5854cc36e7",
    enabled: true
}')"</code></pre>



<p>The detection rule should then appear in Sekoia UI.</p>

<pre class="wp-block-code"><code>import copy
import glob
import json
import os

import requests

# Sekoia API Base URL depends on the Hosting region https://docs.sekoia.io/getting_started/regions/
API_BASE_URL = "https://api.sekoia.io"  # Base URL for FRA1
API_DETECTION_PATH = "/v1/sic/conf/rules-catalog/rules"

# Sekoia API Key is set by an environment variable
SEKOIA_API_KEY = os.getenv("SEKOIA_API_KEY")

def create_detection(query):
   url = f"{API_BASE_URL}{API_DETECTION_PATH}"
   detection_name = query&#091;"name"]
   response = requests.post(url, json=query, headers={"Authorization": f"Bearer {SEKOIA_API_KEY}"})
   if response.status_code == 200:
      print(f"Rule \"{detection_name}\" has been successfully created.")
      detection_details = response.json()
      return detection_details&#091;"uuid"]
   else:
      error_message = response.text
      print("Error: failed to create the rule \"{detection_name}\".\nStatus code: {response.status_code} {error_message}")
      return None

def update_detection(query):
   uuid = query.pop("uuid")
   url = f"{API_BASE_URL}{API_DETECTION_PATH}/{uuid}"
   detection_name = query&#091;"name"]
   response = requests.put(url, json=query, headers={"Authorization": f"Bearer {SEKOIA_API_KEY}"})
   if response.status_code == 200:
      print(f"Rule \"{detection_name}\" has been successfully updated.")
   else:
      error_message = response.text
      print("Error: failed to update the rule \"{detection_name}\".\nStatus code: {response.status_code} {error_message}")
      return None

def deploy_detections():
  for json_file in glob.glob("detections/*.json"):
    # Check if a sigma content is here
    sigma_file = json_file.replace(".json", "_sigma.yml")
    if not os.path.isfile(sigma_file):
      print(f"Ignoring {json_file} as file {sigma_file} is missing")
      continue

    # Build the query parameters
    detection_description = {}
    with open(json_file, "r") as file:  
      detection_description = json.load(file)
    api_query = copy.deepcopy(detection_description)
    with open(sigma_file, "r") as file:
      api_query&#091;"payload"] = file.read()
    
    # Create the detection if no UUID is there, otherwise update it
    if "uuid" in api_query:
       update_detection(api_query)
    else:
       uuid = create_detection(api_query)
       if uuid:
        detection_description&#091;"uuid"] = uuid
        with open(json_file, "w") as file:
          json.dump(detection_description, file, indent=4)


if __name__ == '__main__':
    deploy_detections()</code></pre>
