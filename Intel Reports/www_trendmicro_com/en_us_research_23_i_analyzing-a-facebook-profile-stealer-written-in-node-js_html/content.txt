		<div>
			<p>During our <a href="/en_us/research/23/h/profile-stealers-spread-via-llm-themed-facebook-ads.html">previous analysis of a campaign</a> involving a Facebook stealer, we discovered another interesting stealer. It was written in <a href="https://nodejs.org/en">Node.js</a>, packaged into an executable, exfiltrated stolen data via both <a href="https://telegram.org/blog/bot-revolution">Telegram bot</a> API and a command-and-control (C&amp;C) server, and employed <a href="https://graphql.org/">GraphQL</a> as a channel for C&amp;C communication. This blog entry investigates this new stealer and provides an in-depth analysis of its routines and capabilities.</p>
<h1><span class="body-subhead-title">Infection vector</span></h1>
<h2><span class="body-subhead-title"><span class="rte-sub-menu-text">Recent campaign</span></span></h2>
<p>Like the earlier campaign, we noticed that this stealer was distributed via malicious Large Language Model (LLM)-themed Facebook ads. These malicious ads contain a link to a page hosted on <a href="https://sites.google.com/">Google Sites</a>, which then contains a link to an archive hosted on <a href="https://trello.com/">Trello</a> (an online project and task management tool).</p>

		</div>
	

</div>
            <div class="image">
	
    


	<figure class="image-figure">

		</a>
		<div class="caption-image-container ">
			<figcaption>Figure 1. A fake LLM-themed website offering a MetaAI package</figcaption>
			
		</div>
	</figure>

</div>
            <div>




    
    
    <div class="richText">
	
    


	
		<div>
			<h2><span class="body-subhead-title"><span class="rte-sub-menu-text">Previous campaigns</span></span></h2>
<p>Searching further back, we managed to find older campaigns that abused Trello to host archives containing stealers. The keywords used in the file names include “marketing specialist,” “marketing recruitment,” “CV sample,” “advertising campaign,” “marketing project,” and the names of services like Google, Facebook, and TikTok.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">

		</a>
		<div class="caption-image-container ">

	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>Another example showed a fake website impersonating the CapCut video editor that was used to trick victims into downloading the package.</p>

		</div>
	

</div>

    <div class="richText">
	
    


	
		<div>
			<h1><span class="body-subhead-title">Node.js packaging</span></h1>
<p>The archive will usually contain a single executable file with a size between 50 – 70MB. Note that the JavaScript code of the stealer itself is quite small (less than 100KB), however, embedding the <a href="https://nodejs.org/en">Node.js</a> binary and all packages with <a href="https://www.npmjs.com/package/pkg">pkg</a> significantly inflates the overall size. As a result of packaging, the developer gets a single file that doesn’t need to download and install any dependencies. During packaging, the pkg uses a <a href="https://firefox-source-docs.mozilla.org/devtools-user/debugger/how_to/use_a_source_map/index.html">source map</a>, where all the sources are combined into one file.</p>
<p>The offsets and lengths of this source map can be found at the end of the packaged executable file. Researchers can then use the provided information to extract the desired components from the package. As shown in figure 5, <i>app.js</i> (the main file) is the starting point of the application.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		<a id="dcc42a" href="/content/dam/trendmicro/global/en/research/23/i/analyzing-a-facebook-profile-stealer-written-in-node-js/fb-stealer5.png" target="_blank" rel="noopener noreferrer">
			<img src="/content/dam/trendmicro/global/en/research/23/i/analyzing-a-facebook-profile-stealer-written-in-node-js/fb-stealer5.png" alt="Figure 5. The source map containing list of all files contained within the package."/>
		</a>
		<div class="caption-image-container ">
			<figcaption>Figure 5. The source map containing list of all files contained within the package.</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<h1><span class="body-subhead-title">The stealer</span></h1>
<p>When the stealer is executed, it runs its main function that steals cookies and credentials from several <a href="https://www.chromium.org/">Chromium</a>-based web browsers, then exfiltrates the data to the C&amp;C server and to the Telegram bot. It also subscribes the client to the C&amp;C server running <a href="https://graphql.org/">GraphQL</a>. When the C&amp;C server sends a message to the client, the stealing function will run again.</p>
<h2><span class="body-subhead-title"><span class="rte-sub-menu-text">The main function routine</span></span></h2>
<ol>
<li>The main function prevents the stealer from running too often. The stealer saves the timestamp of its last run to a text file in the home directory (<i>c:\Users\&lt;username&gt;).</i> If the time elapsed from the last run is less than approximately 30 minutes (or 0x1cd6d0 ms = 1890000 ms = 1890 s = 31.5 mins, to be exact), the stealing routine is skipped for this run.</li>
<li>It checks for the backup C&amp;C server address, which is encoded using Base64, by querying <i>&lt;C&amp;C server&gt;/bk/map.txt</i>. This backup response is also saved in another text file in the user home directory for future use.</li>
<li>It retrieves the external IP address by querying the <i>whoer.net</i> website.</li>
<li>It initializes <a href="https://www.npmjs.com/package/graphql-ws">graphql-ws</a>, <a href="https://www.npmjs.com/package/ws">ws</a>, and other necessary libraries to enable <a href="https://graphql.org/">GraphQL</a> over <a href="https://developer.mozilla.org/en-US/docs/Web/API/WebSockets_API">websockets</a> communication with the C&amp;C server. It then initializes the <a href="https://www.npmjs.com/package/@apollo/client">Apollo Client</a> to access a <a href="https://www.apollographql.com/docs/graphos/quickstart/self-hosted">self-hosted</a> supergraph.</li>
<li>It <a href="https://www.apollographql.com/docs/react/data/subscriptions">subscribes</a> to the C&amp;C server to enable it to push messages to the victim’s machine. It subscribes to observables with callbacks using the <a href="https://github.com/zenparsing/zen-observable">zen-observable</a> library. The subscribed operation contains an external IP address as a parameter, with the response (the content of the push message) returning the string value _id.</li>
<li>It runs the information theft process.</li>
<li>It then copies itself to the home directory and sets autorun persistence.</li>
</ol>
<h2><span class="body-subhead-title"><span class="rte-sub-menu-text">Handling push messages</span></span></h2>
<p>If a threat actor pushes a message to the infected client, the received message has the  JSON format seen in Figure 6. The purpose of the a message is to assign an _id to the infected client.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		<a id="739c69" href="/content/dam/trendmicro/global/en/research/23/i/analyzing-a-facebook-profile-stealer-written-in-node-js/fb-stealer6.png" target="_blank" rel="noopener noreferrer">
			<img src="/content/dam/trendmicro/global/en/research/23/i/analyzing-a-facebook-profile-stealer-written-in-node-js/fb-stealer6.png" alt="Figure 6. The format of the received push message"/>
		</a>
		<div class="caption-image-container ">
			<figcaption>Figure 6. The format of the received push message</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>After the client receives an _id, it then sends another message back to the C&amp;C server to modify some back-end data (called a mutation in GraphQL terminology). The mutation operation contains a previously assigned _id, followed by a status message concatenated with the current date and time. The theft of credentials from all browsers in the targeted machine then follows.</p>

		</div>
	

</div>


    
    
    <div class="image">
	
    


	<figure class="image-figure">
		<a id="8b866a" href="/content/dam/trendmicro/global/en/research/23/i/analyzing-a-facebook-profile-stealer-written-in-node-js/fb-stealer7.png" target="_blank" rel="noopener noreferrer">
			<img src="/content/dam/trendmicro/global/en/research/23/i/analyzing-a-facebook-profile-stealer-written-in-node-js/fb-stealer7.png" alt="Figure 7. The mutation operation in which the client sends the status message to the server"/>
		</a>
		<div class="caption-image-container ">
			<figcaption>Figure 7. The mutation operation in which the client sends the status message to the server</figcaption>
			
		</div>
	</figure>

</div>


    
    
    <div class="richText">
	
    


	
		<div>
			<p>After the stealing process is completed, the client sends another status message to the server stating that the stealing process has been completed. In the case of a server pushing a message during the stealing process, the client responds with a “wait for the completion” message.</p>
<p>The reason for the implementation of handling such messages is the activation of the stealing process. The threat actor maintains an IP address list of infected clients that are just waiting for activation (by receiving a message). After the threat actor pushes a message to the clients, the stealing process restarts.</p>
<h2><span class="body-subhead-title"><span class="rte-sub-menu-text">The information theft process</span></span></h2>
<p>The stealer focuses on the following web browsers:</p>
<ul>
<li><span class="rte-red-bullet">Microsoft Edge</span></li>
<li><span class="rte-red-bullet">Google Chrome</span></li>
<li><span class="rte-red-bullet">Opera / OperaGX</span></li>
<li><span class="rte-red-bullet">Brave</span></li>
</ul>
<p>For each browser, the stealer searches for available profiles in the <a href="https://chromium.googlesource.com/chromium/src/+/master/docs/user_data_dir.md"><i>User Data</i></a> folder. It then collects the user home path, the profile path, the <i>User Data</i> path, and version information from the \<i>User Data\Last Version</i> file. Next, it extracts the encrypted key (used to decrypt Chrome cookies and passwords) from \<i>User Data\Local State</i>  and decrypts it. Finally, it will kill the browser process.</p>
<p>For each available profile, the stealer gets all saved cookies database from <i>&lt;profile&gt;\Network\Cookies</i>. It then checks if a Facebook cookie named <a href="https://cookiedatabase.org/cookie/facebook/xs/"><i>xs</i></a><i> </i>is present. This cookie is a session ID that indicates whether a user is logged in to Facebook for that profile. If this cookie is not found, it skips the profile.</p>
<p>It extracts all cookies belonging to Facebook, Google, and Outlook (live.com) and decrypts all of them with a key previously obtained from <i>\User Data\Local State</i>. It also retrieves the database of all saved login credentials from <i>&lt;profile path&gt;\Login Data</i>.</p>
<p>Furthermore, the stealer decrypts all Facebook, Google, and Outlook logins (usernames, emails and passwords) also using a key obtained from <i>\User Data\Local State</i>.</p>
<p>It then checks if the MetaMask extension exists in <i>&lt;profile path&gt;\Local Extension Settings\</i>, after which it packs the whole MetaMask extension directory into a zip archive and exfiltrates it to the Telegram bot.</p>
<p>It gets Facebook’s <a href="https://cookiedatabase.org/cookie/facebook/c_user/">c_user</a> cookie and additional browser information such as operating system, version, and architecture. The stealer will also attempt to steal <a href="https://developers.facebook.com/docs/facebook-login/guides/access-tokens/">Facebook’s access token</a>. If this is unsuccessful, it will exfiltrate Facebook cookies, browser names, executable paths, saved logins, IP addresses, and country codes. The targeted profile is then skipped, with the stealer proceeding to another profile.</p>
<p>It then exfiltrates the following in order via GET requests to C&amp;C server:</p>
<ol>
<li>Facebook identity numbers, full usernames, email addresses, birthdays, access tokens, Facebook cookies, browser names, executable paths, saved logins, IP addresses, and country codes</li>
<li>Gmail credentials and cookies</li>
<li>Outlook credentials and cookies</li>
<li>Additional Facebook information, such as email addresses and location information</li>
<li>Business account information, such as usernames and identifiers.  For each business account, it will steal the name, ad account limit, creation time, business ID, permitted roles, verification status, and number of business users associated with the business.</li>
<li>Page information, including usernames and <a href="https://developers.facebook.com/docs/pages/access-tokens/">page access token</a>s</li>
<li>Ad account information, which includes usernames. For each ad accounts, the stealer extracts ad account IDs, <a href="https://developers.facebook.com/docs/marketing-api/reference/ad-account/agencies/">ad account agencies</a>, spending limits, extended credits (invoice and how often it is billed), currency ratio to USD, time zones, next billing dates, the creation time, <a href="https://www.facebook.com/business/help/776240779095515">billing threshold</a>s, balances, payment cards, payment card expiration dates, payment card verification status, ad account insights, and account status</li>
</ol>
<p>Most likely for backup purposes, previously extracted information is also saved into a text file and sent to the Telegram bot.</p>
<h2><span class="body-subhead-title">Exfiltration to the C&amp;C server</span></h2>
<p>The exfiltration of stolen data to the C&amp;C server is done via GET request to a randomly generated path (<i>&lt;server&gt;/image/&lt;random 26-character ID&gt;.png</i>). The exfiltrated content is passed inside the <a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Authorization">authorization header</a>.</p>

		</div>
	

</div>

		<div>
			<p>The payload in the authorization header is encrypted and hex-encoded. The author likely intended to use a simple string for the encryption, as suggested by the following snippet.</p>

		</div>
	

</div>


    
    
    <div class="image">

			<p>However, in the code, the encryption and decryption is implemented using the improper/illogical use of map/<a href="https://www.w3schools.com/jsref/jsref_reduce.asp">reduce</a>. This means that all characters in passphrase are XORed, which in turn makes the encryption a single-byte XOR encryption.</p>

		</div>
	

</div>

    <div class="richText">
	
    


	
		<div>
			<h2><span class="body-subhead-title">Anti-analysis features</span></h2>
<p>To make analysis by security researchers more difficult, the application hides its console window by calling the hideConsole command from the <a href="https://www.npmjs.com/package/node-hide-console-window">node-hide-console-window</a> package.</p>
<p>In addition, selected methods from <a href="https://developer.mozilla.org/en-US/docs/Web/API/console">console</a> objects are <a href="https://stackoverflow.com/questions/1215392/how-to-quickly-and-conveniently-disable-all-console-log-statements-in-my-code">overwritten with empty functions</a> to suppress logging error and debug messages.</p>

<p>Within the code, we noticed several comments, status messages and variable names in Vietnamese language, suggesting that the author(s) of this malware may have knowledge of this language.</p>

		<div>
			<h1><span class="body-subhead-title">Conclusion</span></h1>
<p>The sheer amount of information pilfered by stealers such as the one we just analyzed in this blog entry means that users must be extra careful since these kinds of malware often results in significant compromise and could lead to being targeted for follow-up attacks. As always, vigilance and proper cybersecurity hygiene go a long way in ensuring that a person’s online activities are safe and secure.</p>
<p>Here are some steps users can take to minimize the risks associated with malware that use malicious ads for distribution:</p>
<ol>
<li>Avoid clicking on ads, especially if they seem suspicious or offer something that seems too good to be true.</li>
<li>Be cautious when dealing with pop-up windows or ads that ask for personal information. Never provide sensitive information unless absolutely sure of the legitimacy of the app or website being visited.</li>
<li>Invest in reputable antivirus and anti-malware technologies. These tools can help detect and mitigate the impact of malicious software that might be delivered through ads.</li>
<li>Learn how to recognize the most common signs of malicious ads, such as typographical errors, poor grammar, and offers that seem too good to be true.</li>
<li>When downloading or clicking on ads that promote software, ensure they come from reputable sources. Stick to official app stores and trusted websites when downloading tools and apps.</li>
</ol>
