<p style="font-size:12px" class="has-text-align-left">Last October (2019), ESET <a href="https://www.welivesecurity.com/wp-content/uploads/2019/10/ESET_Operation_Ghost_Dukes.pdf">published extensive research</a> regarding additional tooling from the &#8220;Dukes&#8221; adversary, which analysts have traditionally aligned with [APT29/Cozy Bear] operations. While conducting some unrelated research, I came across a LiteDuke sample and decided to take a deeper dive into the mechanics of its loader and its malware. <br><br>The original ESET publication covers the key details at a high level; in addition, this malware is old and has likely been discontinued for several years. This blog post is intended to document some additional lower-level details for operational comparison purposes and general learning.</p>



<span id="more-356"></span>



<p style="font-size:12px" class="has-text-align-left"><strong><span style="text-decoration: underline;">Loading Structure</span><br></strong><br>The loading structure identified by this blog is similar to the one described by ESET (differences may be attributable to a different variant of the malware being examined or more likely just a different analytic perspective). This blog identified a DLL component of the loading chain on VirusTotal with the following properties:<br><br>MD5: 79ac8431f484ab50de137544d88d5c83<br>SHA1: 5c01fcc8c9cf72bd1d5088ee739939353b41b1d6<br>SHA256: b9670993179beb5c1af574c156c0379e058b2f4a6635a83b98766a3339431274<br>Compilation Timestamp: 2014-10-02 11:09:36<br>Export: ShowDialog<br><br>This DLL was likely preceded by another executable, script, or command-line invocation, as it requires that the &#8220;<strong>ShowDialog</strong>&#8221; export be called to deploy the next stage. This DLL allocates a segment of virtual memory and decodes content from a .bmp resource. The decoded memory is a block of <em>four</em> executable files:<br><br>1) A DLL with an export named <strong>SetLibrary</strong> (that expects #2-4 to be present).<br>2) A DLL with several exports, including one named <strong>LoadRegistry</strong> (that expects #3-4 to be present).<br>3) The LiteDuke payload, temporarily decrypted (that expects #4 to be present).<br>4) A SQLite library that is actually a component of the LiteDuke payload.  </p>



<div class="wp-block-image"><figure class="aligncenter size-large"><img fetchpriority="high" decoding="async" width="481" height="272" src="https://norfolkinfosec.com/wp-content/uploads/2020/05/Loading_Structure.png" alt="" class="wp-image-357" srcset="https://norfolkinfosec.com/wp-content/uploads/2020/05/Loading_Structure.png 481w, https://norfolkinfosec.com/wp-content/uploads/2020/05/Loading_Structure-300x170.png 300w, https://norfolkinfosec.com/wp-content/uploads/2020/05/Loading_Structure-210x119.png 210w" sizes="(max-width: 481px) 100vw, 481px" /><figcaption><strong>LiteDuke Loading Structure</strong></figcaption></figure></div>



<p style="font-size:12px" class="has-text-align-left">The SetLibrary export from the top component (#1) performs the following actions:<br><br>1) Creates a directory at C:\ProgramData\User\<br>2) Writes a file named NTUSER.DAT here. This file is #2 from the list above, but with the #3/4 blocks encrypted.<br>3) Calls rundll32.exe [path to NTUSER.DAT],LoadRegistry<br><br>This invokes the LiteDuke loader, which decrypts and runs the LiteDuke payload via the LoadRegistry export.<br><br><strong><span style="text-decoration: underline;">LiteDuke Payload</span></strong><br><br>The LiteDuke payload can be broken down into three core workflows:<br><br>1) A setup routine, which establishes persistence, creates the SQL database file, and launches the C2 routines.<br>2) A &#8220;higher-level&#8221; C2 block, which performs basic maintenance tasks.<br>3) A &#8220;lower-level&#8221; C2 block, which carries out a large number of commands. <br><br><strong><em>Setup Routine</em><br></strong><br>As part of its setup, the malware creates a mutex named &#8220;NtUserRegistryService&#8221; and terminates the process if the mutex already exists. The malware then calls three functions. The first of these creates the encryption key for a local database, the second creates this database (written to the same ProgramData\User folder with the name NTUSER.DAT.BAK), and the third assigns a User Agent used for HTTP communication. <br><br>The SQL database contains three tables: config, modules, and objects. The config table contains information for the C2 routine and can be updated (as this post will later show). The modules field appears to allow for additional code execution (although, as ESET noted, none have been publicly documented). The purpose of the objects field is less clear; a handful of functions in the binary suggest it could be used to hold files, although the phrase &#8220;object&#8221; itself has a distinct meaning in object databases. </p>



<figure class="wp-block-image size-large"><img decoding="async" width="1024" height="535" src="https://norfolkinfosec.com/wp-content/uploads/2020/05/SQL_Browser-1024x535.png" alt="" class="wp-image-362" srcset="https://norfolkinfosec.com/wp-content/uploads/2020/05/SQL_Browser-1024x535.png 1024w, https://norfolkinfosec.com/wp-content/uploads/2020/05/SQL_Browser-300x157.png 300w, https://norfolkinfosec.com/wp-content/uploads/2020/05/SQL_Browser-768x401.png 768w, https://norfolkinfosec.com/wp-content/uploads/2020/05/SQL_Browser-210x110.png 210w, https://norfolkinfosec.com/wp-content/uploads/2020/05/SQL_Browser.png 1028w" sizes="(max-width: 980px) 100vw, 980px" /><figcaption><strong>SQL table and fields set up by the binary</strong></figcaption></figure>



<div class="wp-block-image"><figure class="aligncenter size-large"><img decoding="async" width="475" height="570" src="https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-1.png" alt="" class="wp-image-359" srcset="https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-1.png 475w, https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-1-250x300.png 250w, https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-1-175x210.png 175w" sizes="(max-width: 475px) 100vw, 475px" /><figcaption><strong>Segment of setup workflow leading to the browser query</strong></figcaption></figure></div>



<p style="font-size:12px" class="has-text-align-left">To select a User Agent, the malware uses the AssocStringQuery API to search the root registry for the default program associated with the &#8220;http&#8221; extension. In practical terms, this returns the default browser, which is checked against a hardcoded list. The malware assigns a User Agent corresponding to the default browser and also moves a value into the EAX register based on which browser was chosen. <br><br>If this check sets the EAX value to either 2 or 5 (Firefox and Opera respectively), the malware will <em>also</em> attempt to retrieve and use proxy configuration information stored in the browser settings. <br><br>The first image below shows the browser/User Agent association function. The second image below shows the parent function that calls both this User Agent function and the proxy function. This second image corresponds with the &#8220;BowserCheck_ProxyDecide&#8221; label in the setup routine image above.</p>



<figure class="wp-block-image size-large"><img loading="lazy" decoding="async" width="1024" height="677" src="https://norfolkinfosec.com/wp-content/uploads/2020/05/user_agents-1024x677.png" alt="" class="wp-image-360" srcset="https://norfolkinfosec.com/wp-content/uploads/2020/05/user_agents-1024x677.png 1024w, https://norfolkinfosec.com/wp-content/uploads/2020/05/user_agents-300x198.png 300w, https://norfolkinfosec.com/wp-content/uploads/2020/05/user_agents-768x508.png 768w, https://norfolkinfosec.com/wp-content/uploads/2020/05/user_agents-210x139.png 210w, https://norfolkinfosec.com/wp-content/uploads/2020/05/user_agents.png 1476w" sizes="auto, (max-width: 980px) 100vw, 980px" /><figcaption><strong>User Agent selection (right click and open in a new tab to expand)</strong></figcaption></figure>



<figure class="wp-block-image size-large"><img loading="lazy" decoding="async" width="1024" height="851" src="https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-2-1024x851.png" alt="" class="wp-image-361" srcset="https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-2-1024x851.png 1024w, https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-2-300x249.png 300w, https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-2-768x638.png 768w, https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-2-210x174.png 210w, https://norfolkinfosec.com/wp-content/uploads/2020/05/main_proxy-2.png 1063w" sizes="auto, (max-width: 980px) 100vw, 980px" /><figcaption><strong>Parent function for user agent and proxy selections</strong></figcaption></figure>



<p style="font-size:12px" class="has-text-align-left">Finally, the malware takes two additional setup steps.<br><br>First, the malware creates a persistence mechanism. As ESET previously noted, this takes the form of a .lnk file. In this case, the file is named NTUSER.DAT.lnk and placed inside of the same ProgramData\User\ directory as the database and loader. The .lnk file calls rundll32.exe, passing the .DAT file and the LoadRegistry DLL, forcing the application to run at login. <br><br>The application <em>does</em> have handling for administrative privileges that includes a SID check for default SIDs and the %systemprofile% ProfileImage path, but the difference in outcome from this routine was not immediately obvious during debugging.<br><br>Second, the malware queries the SQL database for any &#8220;autorun&#8221; plugins. Presumably, these are plugins transmitted by the C2 that are then executed when the malware first loads, although this is unconfirmed due the previously mentioned absence of such data. </p>



<p style="font-size:12px" class="has-text-align-left"><em><strong>Command and Control</strong><br></em><br>The malware has an initial C2 check-in that serves an unclear purpose; however, the primary two C2 workflows are described below. <br><br><span style="text-decoration: underline;"><em>&#8220;Higher&#8221; Workflow</em><br></span><br>This workflow is built around a series of &#8220;if-else&#8221; statements in which the &#8216;al&#8217; register is compared to a hardcoded hex value:<br><br>0x01 &#8211; Retrieve an entry from the configuration table in the SQL database OR retrieve the operating system version.<br>0x02 &#8211; Update the configuration via SQL statement.<br>0x15 &#8211; Call &#8220;lower&#8221; command structure.<br>0x04 &#8211; Insert new data into the &#8220;modules&#8221; table via SQL statement.<br>0x0F &#8211; Insert new data into the &#8220;objects&#8221; table via SQL statement.<br><br>As previously mentioned, the purpose of the objects table (and thus the 0x0F) command isn&#8217;t entirely clear given the possible double meaning of the phrase &#8220;objects.&#8221; This routine includes the phrase &#8220;upload,&#8221; which for some malware can mean &#8220;upload to the victim device.&#8221; The phrase &#8220;moved to crypto container&#8221; is also present and is preceded by the &#8220;SaveToCC&#8221; export that inserts data into the objects table. These factors (and traditional C2 routines discussed below) point towards file transmission, but this remains hypothetical.</p>



<div class="wp-block-image"><figure class="aligncenter size-large is-resized"><img loading="lazy" decoding="async" src="https://norfolkinfosec.com/wp-content/uploads/2020/05/higher_C2.png" alt="" class="wp-image-363" width="473" height="720" srcset="https://norfolkinfosec.com/wp-content/uploads/2020/05/higher_C2.png 630w, https://norfolkinfosec.com/wp-content/uploads/2020/05/higher_C2-197x300.png 197w, https://norfolkinfosec.com/wp-content/uploads/2020/05/higher_C2-138x210.png 138w" sizes="auto, (max-width: 473px) 100vw, 473px" /><figcaption><strong>&#8220;Higher&#8221; C2 Workflow</strong></figcaption></figure></div>



<p style="font-size:12px" class="has-text-align-left"><em><span style="text-decoration: underline;">&#8220;Lower&#8221; Workflow</span></em><br><br>When the malware enters the &#8220;lower&#8221; command structure (based on the 0x15 check above), it can execute one of a large number of possible commands. The initial intent of this research was to identify as many commands as possible. So far, these include:<br><br>&#8211; Set current working directory.<br>&#8211; Retrieve data from the &#8220;body&#8221; field of the &#8220;objects&#8221; table and write it to the disk.<br>&#8211; Copy a file from one location to another location.<br>&#8211; Read data from a file.<br>&#8211; Delete a file.<br>&#8211; Save data to the SQL database.<br>&#8211; Create a pipe, create process, open mutex (&#8220;ZWQSI_Mutex&#8221;) and open file mapping (&#8220;ZWQSI_PIDs&#8221;).<br>&#8211; Unknown, possibly terminates process with &#8220;ZWQSI_Mutex.&#8221;<br>&#8211; Retrieve current directory.<br>&#8211; Read data from a named pipe.<br>&#8211; Update a module&#8217;s &#8220;autorun&#8221; configuration value.<br>&#8211; Read the body of an item in the &#8220;object&#8221; table.<br>&#8211; Read the configuration of an item in the &#8220;module&#8221; table.<br>&#8211; Delete a file using the SHFileOperation API, with flags set for Silent, No Confirmation, and No Error.<br>&#8211; Move a file.<br>&#8211; Create a directory. <br>&#8211; Terminate a process (specified by PID).<br>&#8211; Terminate a process (via CreateToolhelp32Snapshot and Process32First/Next).<br>&#8211; Set working directory to user&#8217;s profile (&#8220;C:\Users\[User]\&#8221;).<br>&#8211; Set working directory to the Temp path. <br>&#8211; Retrieve drive and disk space information. <br>&#8211; List installed programs by enumerating the &#8220;DisplayName&#8221; values in the &#8220;\CurrentVersion\Uninstall&#8221; key.<br>&#8211; Retrieve HDD, OS, ProcessorNameString, SystemBiosVersion, net adapter (IP, MAC, DHCP, Description).<br>&#8211; Get values that comprise database key (BIOS, username, computername, processor).<br>&#8211; Retrieve entries in modules table.<br>&#8211; Delete object from SQL database.<br>&#8211; Delete module from SQL database.<br>&#8211; Retrieve entries in objects table.<br>&#8211; Return an error message if an unsupported command is issued. </p>



<div class="wp-block-image"><figure class="aligncenter size-large is-resized"><img loading="lazy" decoding="async" src="https://norfolkinfosec.com/wp-content/uploads/2020/05/uninstall_display_workflow.png" alt="" class="wp-image-364" width="578" height="729" srcset="https://norfolkinfosec.com/wp-content/uploads/2020/05/uninstall_display_workflow.png 770w, https://norfolkinfosec.com/wp-content/uploads/2020/05/uninstall_display_workflow-238x300.png 238w, https://norfolkinfosec.com/wp-content/uploads/2020/05/uninstall_display_workflow-768x969.png 768w, https://norfolkinfosec.com/wp-content/uploads/2020/05/uninstall_display_workflow-166x210.png 166w" sizes="auto, (max-width: 578px) 100vw, 578px" /><figcaption><strong>Workflow for command that enumerates DisplayName values in Uninstall registry key</strong></figcaption></figure></div>



<p style="font-size:12px" class="has-text-align-left">The number of features are extensive. Most notably, several of these commands perform the same or similar functions. For example, there are multiple supported operations for deleting a file that use different APIs. Several commands change the working directory. There are also several commands for interacting with the SQL database.<br><br>While such actions could have been streamlined, one possible explanation for these redundancies is that the authors of the tool intended to minimize user errors during an intrusion that could be caused by issuing invalid SQL commands or invalid command-line entries. From a development standpoint, however, this can become cumbersome: every possible operator action would require a new entry into the malware. <br><br>One additional note includes the &#8220;ZWQSI_Mutex&#8221; mutex. This blog has not yet identified a process associated with this specific value. It may refer to an additional executable or plugin used by the attacker; however, given that this malware has likely not been in use for five years, that data may never be recovered. Google searches for ZWQSI suggest it may be used as a common abbreviation for the &#8220;ZwQuerySystemInformation&#8221; API, and perhaps this implies the attackers tried to hide this mutex by naming it after that.<br><br><strong><span style="text-decoration: underline;">Concluding Thoughts</span></strong><br><br>The LiteDuke malware is extensive, featuring a multi-stage loading process, a SQL table that holds configuration information and plugins, and a wide range of commands that allowed the operators to interact with this database and carry out common backdoor actions on the device. While this workflow has several advantages &#8211; most notably, unique but predictable database encryption keys as well as plugin support &#8211; it also requires the implementation of several redundant commands. This blog suspects that the attackers likely abandoned this in favor of their more flexible tools.<br><br><span style="text-decoration: underline;">Additional hashes</span>:<br><br>&#8220;LoadRegistry&#8221; loader: f00297add1ea84ace5677d710cd68b51 (Compiled 2014-10-02 11:09:35)<br><br>Unpacked LiteDuke payload: 8267a73a144e0f0d9ccf487958050354 (Uploaded to VT by this blog after analysis)<br><br></p>
