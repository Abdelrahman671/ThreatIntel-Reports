    <section class="b-blog b-blog--space-above">

            <div class="l-blog">
                <div class="l-blog__content">

                    
                    <div class="h-wysiwyg-html">
                        <article class="c-article" data-scheme-target>
                            <p>Bad Rabbit is the new bunny on the ransomware scene. While the security community has concentrated mainly on the similarities between Bad Rabbit and EternalPetya, there&#8217;s one notable difference which has not yet gotten too much attention. The difference is that Bad Rabbit&#8217;s disk encryption <strong>works</strong>.</p>
<p>EternalPetya re-used the custom disk encryption method from the original Petya. Although it didn&#8217;t implement the actual ECDH key delivery mechanism, it installed the Petya boot loader, and effectively just rendered the machine useless.</p>
<p>Petya&#8217;s disk encryption had one specific weakness: it only encrypted some parts of the key file system structures, not the whole disk. This design obviously lead to speculations about whether it is possible to recover the disk using a known clear-text attack, and in fact researchers have made significant progress in investigating this <a href="https://www.crowdstrike.com/blog/decrypting-notpetya-tools-for-recovering-your-mft-after-an-attack/">recovery technique</a>.</p>
<p>At least on the surface, things look quite different with Bad Rabbit. Instead of using a custom encryption mechanism, it follows the current trend in the ransomware community of leveraging known legitimate encryption tools.</p>
<p>Bad Rabbit uses <a href="https://diskcryptor.net/wiki/Main_Page">DiskCryptor</a>, a full disk partition encryption software for locking the user disk. The ransomware ships with an unmodified DiskCryptor driver (borrowed from ReactOS) and implements relevant parts of the DiskCryptor user-mode code for communicating with the driver. The ReactOS driver is a signed, valid driver, so it gets loaded by the Windows cleanly with the elevated privileges required by Bad Rabbit&#8217;s fake Flash installer dropper.</p>
<h3>Key generation</h3>
<p>Bad Rabbit uses Windows&#8217; crypto API to generate random key data for the disk encryption. This key data is converted to a human-readable encryption key, which is a 32-bytes long ASCII encoded string, presenting around <a href="https://en.wikipedia.org/wiki/Password_strength">165 bits of entropy</a> for the stream cipher AES used by DiskCryptor.</p>
<p>The following screenshot represents the random disk encryption key as it is being generated by the malware:</p>
<div id="attachment_40752" style="width: 1306px" class="wp-caption alignnone"><img fetchpriority="high" decoding="async" aria-describedby="caption-attachment-40752" class="size-full wp-image-40752" src="https://blog-assets.f-secure.com/wp-content/uploads/2017/10/17162049/asquiring_password.png" alt="" width="1296" height="868" /><p id="caption-attachment-40752" class="wp-caption-text">Running dispci.exe under user-mode debugger</p></div>
<p style="text-align: center;"><span style="color: #808080;">Â </span></p>
<p>Along with the random key, Bad Rabbit packages some other information like the victim computer name and domain, then forms the so-called installation key that will be presented after the reboot. In Petya, this code was relatively short code that was protected by the public EC key. In Bad Rabbit, it is a much longer blob of data that is protected by the RSA public key shipped with the malware installer.</p>
<p>After packaging the installation key, the random key data is just discarded. The installation key is written to the disk so that the Bad Rabbit boot loader can present it on the boot screen:</p>
<div id="attachment_40734" style="width: 1306px" class="wp-caption alignnone"><img decoding="async" aria-describedby="caption-attachment-40734" class="size-full wp-image-40734" src="https://blog-assets.f-secure.com/wp-content/uploads/2017/10/17162042/decrypting_disk.png" alt="" width="1296" height="868" /><p id="caption-attachment-40734" class="wp-caption-text">Bad Rabbit&#8217;s boot loader</p></div>
<p style="text-align: center;"><span style="color: #808080;">Â </span></p>
<p>The user is expected to grab the installation key from the boot screen, paste it to the attacker&#8217;s TOR site, which then will use its own private RSA key to extract the 32-bytes password (typed in the screenshot above).</p>
<h3>After the boot screen</h3>
<p>Because the disk encryption software used is pretty much similar to any other disk encryption used by businesses around the globe, like TrueCrypt or Microsoft <a href="https://en.wikipedia.org/wiki/BitLocker">BitLocker</a>, the disk is in fact *still* encrypted, although it has been mounted by the DiskCryptor driver transparently.</p>
<p>So if the user now reboots the machine, the same password prompt will be presented as long as the disk decryption routine is initiated. The DECRYPT tool referenced by the boot loader is actually just a shortcut to the dispci.exe tool dropped by the malware. This tool borrows code from legit DiskCryptor sources for implementing the relevant parts of the code for communicating with the disk encryption driver.</p>
<p>Even though all this is quite apparent by just looking at the code, we wanted to demonstrate the encryption scheme by catching the password inline, at the time it was generated by Bad Rabbit. When this password is used for unlocking the machine, it is possible to install the real DiskCryptor GUI tool and initiate the disk decryption process.</p>
<div id="attachment_40743" style="width: 1306px" class="wp-caption alignnone"><img decoding="async" aria-describedby="caption-attachment-40743" class="size-full wp-image-40743" src="https://blog-assets.f-secure.com/wp-content/uploads/2017/10/17162045/correct_pw.png" alt="" width="1296" height="868" /><p id="caption-attachment-40743" class="wp-caption-text">Using DiskCryptor to verify encrypted volume</p></div>
<p>DiskCryptor identifies the disk presented by the virtual machine (QEMU HARDDISK in the above screenshot) as an AES-encrypted volume, and accepts the random password.</p>
<h3>It works now</h3>
<p>We have speculated before that the flawed disk encryption in EternalPetya was due to problems in the malware development process. Or it may be that they just didn&#8217;t care. People will pay anyways, right?</p>
<p>Whatever was the reason, they have now fixed this issue (if they are the same group of malware developers, which seems to be the consensus in the research community).</p>
<p>At least the developers of Bad Rabbit have noted the recent developments in research on Petya&#8217;s disk encryption weaknesses and decided to use something different.</p>
<h3>Recovery considerations</h3>
<p>As we have demonstrated in this blog post, Bad Rabbit seems to use a sound principle in its disk encryption, a full disk encryption scheme familiar to all businesses.</p>
<p>We don&#8217;t yet have the full details of this scheme, so there might be bugs in the implementation. But at least its design enables a strong mechanism for locking the machine until the correct password is really typed to the boot screen.</p>
<h3>Disclaimer</h3>
<p>For screenshots used in this blog post, we DID NOT go to the attacker TOR site and pay for the recovery key.</p>
<p>The procedures presented in this text DO NOT mean there&#8217;s an easy way to unlock the disk protected by the Bad Rabbit. We just present them as proof of the encryption scheme. Catching the password inline to the encryption process is not practical in a general sense, because it requires software that is aware of the exact password generation mechanism <strong>prior to the infection</strong>. It is used here just for a relatively easily reproducible proof-of-concept.</p>
